# 反模式（Anti-patterns）

使用本技能时常见的错误模式和正确做法。

---

## 1. 直接更新 `scripts/` 下的原始脚本

**错误现象：**
LLM 或用户为了适配某个特定场景，直接编辑 `scripts/text_crud.py` 等原始脚本，导致：
- 原始基线丢失，无法回滚到干净状态
- 多次适配后脚本变得臃肿、难以维护
- 其他调用方依赖原始行为，意外被破坏

**正确做法：**
参考 `scripts/` 下的原始脚本，创建一个新的适配副本（如 `my_text_crud.py`），在副本中更新。原始脚本保持只读，确保基线可回溯。

---

## 2. 忽略备份直接执行删除/覆盖操作

**错误现象：**
调用 `file_ops.py --action delete` 或 `text_crud.py --action create --overwrite` 时不加 `--no-backup` 以外的备份保护，导致误操作后无法恢复。

**正确做法：**
- 默认不要使用 `--no-backup`（备份是默认开启的）
- 重要操作前可手动额外备份
- 记录操作返回的 `rollback_id`，以便出错时回滚

---

## 3. 在 `SKILL.md` 中写复杂的脚本逻辑说明

**错误现象：**
在 `SKILL.md` 的正文中详细列举每个脚本的函数签名、参数类型、返回值结构，导致主文件超过 230 行限制，且信息冗余、难以维护。

**正确做法：**
- `SKILL.md` 只保留触发场景、核心能力概述、快速开始示例
- 详细的 API 参考、参数说明放到独立的 API 文档文件
- 使用示例放到独立的示例文档文件

---

## 4. 批量操作时 `stop_on_error=false` 导致错误扩散

**错误现象：**
编写 batch.json 时配置 `"stop_on_error": false`，前一个任务失败后仍继续执行后续任务，具体表现：
- 后续任务依赖前置任务的输出文件，因前置失败而全部失败
- 错误日志被大量相似错误淹没，难以定位根因

**正确做法：**
- 默认使用 `"stop_on_error": true`（orchestrator 默认值）
- 确认后续任务不依赖前置结果时，才配置为 `false`
- 用 `--dry-run` 先验证批量配置的正确性

---

## 5. 未检查 `success` 字段就使用返回结果

**错误现象：**
调用脚本后直接解析 stdout JSON，未先检查 `success` 字段，导致：
- 操作失败时仍尝试使用 `result` 中的空数据
- 错误消息丢失，调试困难

**正确做法：**
- 解析 JSON 后先检查 `success` 字段
- `success=false` 时，从 `error` 字段获取错误详情
- 在编排多步骤任务时，前置步骤失败应中止后续步骤

---

## 6. 直接更新 `py_tools.py` 的规范化规则

**错误现象：**
为了调整规范化行为（如放宽行长度限制），直接编辑 `scripts/py_tools.py` 中的硬编码规则，导致：
- 规则与 `references/py_standards.md` 文档不一致
- 更新技能时规则被覆盖，更新丢失
- 其他用户不理解规则来源

**正确做法：**
- 所有规范化规则的依据应记录在 `references/py_standards.md`
- 如需调整规则，更新 `references/py_standards.md` 并同步更新 `scripts/py_tools.py`
- 重大规则更新应在 skill 的 CHANGELOG 中记录

---

## 7. 对超过 600 行的 py 文件直接更新而不 OO 化

**错误现象：**
文件已超过 600 行，仍直接添加新的函数/逻辑，导致：
- 文件进一步膨胀，可维护性急剧下降
- 函数间隐式共享全局状态，容易产生 bug
- 后续 OO 化成本指数级增长

**正确做法：**
- 运行 `py_tools.py review` 检查行数，如超过 600 行先运行 `py_tools.py oo-ify` 查看建议
- 将相关函数封装为类（状态作为实例属性，相关函数作为方法）
- 如文件功能复杂（>1000 行），按功能拆分为多个模块
