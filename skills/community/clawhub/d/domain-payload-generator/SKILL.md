# 领域负载物技能制作器

## 定位

本技能是一个 **元技能（Meta-Skill）**——它的产出不是领域知识本身，而是**领域负载物技能**。当用户说"帮我做一个XX领域的知识参考库"时，本技能负责从零创建完整的、与UTOS完全兼容的领域负载物技能。

| 本技能提供 | 消费方式 |
|-----------|---------|
| 领域分析框架 | 分析新领域并确定域划分和任务类型 |
| 三层结构模板 | 自动生成SKILL.md + references/三层文件 |
| UTOS接口校验清单 | 确保生成的技能与UTOS无冲突 |
| 生成工作流 | 从用户输入到完整技能的标准化流程 |

## 核心能力

```
用户输入："帮我做一个XX领域的知识参考库"
        ↓
  ┌─────────────────────┐
  │ Step 1: 领域分析     │ ← 用领域分析框架拆解新领域
  │   确定域数/任务数     │
  ├─────────────────────┤
  │ Step 2: 结构生成     │ ← 用三层结构模板填充内容
  │   SKILL.md           │
  │   + catalog          │
  │   + requirements     │
  │   + exemplars        │
  ├─────────────────────┤
  │ Step 3: UTOS校验    │ ← 用接口校验清单检查一致性
  │   无冲突 → 输出      │
  │   有冲突 → 修正      │
  └─────────────────────┘
        ↓
  完整的XX领域知识参考库技能（可直接使用）
```

## 与其他技能的关系

### 独立性保证

> **关键设计约束**：本技能**不依赖任何被它创建的领域技能**。它在创建时是自包含的，仅依赖：
> - Universal Task OS（用于理解目标接口）
> - Workflow Refactor（复杂领域先重构工作流，再生成技能）
> - 自身的references文件（框架+模板+校验清单）

这意味着：
- ✅ 创建任何领域技能时，不需要该领域技能已存在
- ✅ 本技能可以独立运行，输出产物后才被其他流程消费

### 职责分工

| 技能 | 管什么 | 不管什么 |
|------|--------|---------|
| **Workflow Refactor** | 流程结构——哪些环节保留、消除、校准 | 领域知识内容（清单/样本） |
| **Domain Payload Generator** | 领域知识内容——catalog(清单)、requirements(要求)、exemplars(范本) | 流程结构 |
| **Universal Task OS** | 三轴执行框架——执行轴编排、内容轴消费清单/样本、创新轴突破 | 具体领域内容 |

### 价值链

```
传统工作流 ──[Workflow Refactor]──→ 重构后IPO基元链
                                          │
                                          ▼
                              [Domain Payload Generator]
                                          │
                                          ▼
                                   领域负载物（简化版）
                                          │
                                          ▼
                              [Universal Task OS] 持续执行
```

转化→创建→执行，是一条价值链，不是替代关系。

### 三条路径对比

| 路径 | 清单/样本来源 | 优势 | 劣势 |
|------|-------------|------|------|
| **Workflow Refactor 单独用** | 无，用户临时提供 | 流程极简 | 内容质量靠用户自身积累 |
| **重构 + Domain Payload + UTOS** | 领域负载物结构化提供 | 流程+内容双保险，系统化 | 首次生成有成本 |
| **重构 + UTOS（无领域负载物）** | 用户手动输入到 IPO 的 I | 灵活 | 每次都要手动准备，覆盖度不稳定 |

### 重构如何简化负载物

领域负载物的复杂度 = 领域本身的复杂度 + 传统工作流遗留的冗余任务类型。

| 场景 | catalog 任务数 | requirements 复杂度 | exemplars 数量 |
|------|---------------|-------------------|---------------|
| 未重构直接生成 | 包含传递/协调/格式环节对应的任务类型 | 大量"人的局限补偿"相关要求 | 范本里嵌套冗余中间产物 |
| 先重构再生成 | 只保留✅核心+🔶校准+⚡关键校验对应的类型 | 要求聚焦事情本身 | 范本干净，无冗余传递物 |

先重构再生成，负载物体积和认知负荷都大幅降低。

### 三层价值属性

| 技能 | 价值类型 | 使用频率 |
|------|---------|---------|
| **Workflow Refactor** | 转化价值——解决"从旧到新"的转化问题 | 低频、脉冲式 |
| **Domain Payload Generator** | 创建价值——解决"从无到有"的创建问题 | 中频、按需 |
| **Universal Task OS** | 运行时价值——解决"执行任务"的运行问题 | 高频、持续 |

Workflow Refactor 的价值不会归零（新领域不断出现、已重构领域会过时需要重新审视、AI能力跃升时重构本身可以被重构），但会随成熟领域完成重构而衰减为低频触发。

## 与UTOS的关系

本技能生成的所有产物都遵循以下UTOS兼容性契约：

| 兼容性维度 | 要求 |
|-----------|------|
| **三层结构一致** | 必须为：第一层(清单+拓扑) + 第二层(要求) + 第三层(范本) |
| **依赖声明格式** | 必须含强依赖UTOS声明 + 加载检查流程 + 降级模式 |
| **元操作映射** | 每个任务必须标注S/C/A/O/I/G映射提示 |
| **五字段Schema** | 每个任务组件必选：ID/名称/说明/依赖/UTOS映射 |
| **域间逻辑流** | 域间必须有明确的价值链逻辑顺序 |
| **依赖拓扑摘要** | 必须有跨域管线链路描述 |
| **Step 0-4接口** | SKILL.md必须含"与UTOS的接口"章节（Step 0~4各条目） |

## 使用规则

1. **生成前置判断**：复杂领域建议先用 Workflow Refactor 重构工作流，再基于重构结果生成技能——这样产出的技能更精准，不会把人的局限补偿层编码进领域知识。简单或标准化领域可直接生成，跳过重构。
2. **触发条件**：用户请求创建新的领域知识参考库/领域负载物技能时激活
3. **首次加载**：读取 `references/domain-analysis-framework.md` 获取领域分析方法
4. **按需深入**：确认领域后，读取 `references/structure-template.md` 获取三层结构模板；读取 `references/utos-interface-checklist.md` 获取校验规则
5. **执行生成**：按 `references/generator-workflow.md` 的步骤逐一执行
6. **输出交付**：将生成的技能写入指定目录
