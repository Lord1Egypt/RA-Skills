# 元规则

## Overview

本文档定义 v3.7-discipline 体系中的**元层面规则**——即关于规则本身的规则。当 Risk Scorecard 的量化阈值与项目上下文产生矛盾时，本文档提供决策依据。本文档的优先级高于所有具体技能文件和 scorecard.yaml。

## 🔴 铁律 0：技能文件是强制指令，不是建议

- 所有 `skills/*.skill.md` 文件中描述的流程、检查项、纪律约束，对子代理 Agent 具有**强制约束力**
- Agent 不能将其视为「最佳实践建议」而降级为「可以参考」
- 违反技能文件要求的行为 = 违反系统指令
- 唯一的例外：本文档定义的「何时无视指标」条件 + 人工 override 流程

## 量化指标上限

| 指标 | 硬上限 | 软上限 | 说明 |
|------|--------|--------|------|
| 单阶段注入技能数 | ≤2 | — | 严格遵守，不做例外 |
| 单文件行数 | ≤200 | ≤180 | 硬上限为 scorecard/discipline-meta 全局文件 |
| 单次修改文件数 | — | ≤5 | 超过触发 🔴 block（手术刀式修改） |
| 无测试新增代码行数 | — | ≤200 | 超过触发 🔴 block |
| 新增抽象层数 | — | ≤1 | 超过触发 🟡 warn（Rule of Three） |
| 连续 Scorecard block | — | ≤1 | 超过 1 次触发连续阻塞升级（见下文） |
| 调试无反馈循环尝试次数 | — | ≤0 | 第一次尝试前必须建反馈循环 |

## 何时无视指标（合理跳过 vs 偷懒跳过）

### ✅ 合理跳过（允许 override）
1. **项目初始化阶段**：首次搭建项目骨架时，测试覆盖阈值不适用
2. **紧急热修复**：生产环境故障修复，可跳过 zoom-out 分析，但必须写回归测试
3. **依赖升级**：仅升级第三方依赖版本，不改业务逻辑，可跳过 TDD 循环
4. **配置文件变更**：纯配置修改（无逻辑变更），可跳过测试阶段
5. **文档/注释变更**：仅修改文档、注释、README，不触发编码纪律检测
6. **手动合并冲突解决**：由人类执行的合并操作，Agent 不参与评分

### ❌ 偷懒跳过（不允许 override）
1. 「太简单不需要测试」→ 所有逻辑变更必须有测试
2. 「后面补」→ 技术债不能延期（只能显式标记为 debt 并设截止日期）
3. 「我一眼就能看出对错」→ 工具验证 > 人类直觉
4. 「之前都通过」→ 每次变更必须重新验证
5. 「顺手改一下」→ 不在需求范围内的变更一律拒绝

**判断原则**：override 的理由必须是**客观环境约束**（时间、类型、阶段），不能是**主观能力判断**（简单、熟练、自信）。

## 人工 override 流程

当 Agent 认为阈值与当前项目上下文矛盾时：

1. **Agent 发出 override 请求**，包含：
   - 触发的规则 ID 和当前信号值
   - override 理由（必须对应「合理跳过」列表中的条目）
   - 风险自评（选择 override 可能带来的风险）
2. **主会话（人类）决策**：
   - `确认` → Agent 记录 override 理由到 scorecard log，继续执行
   - `拒绝` → Agent 执行 block 动作（暂停当前阶段）
   - `调整阈值` → 修改本项目 `.auto-coding/scorecard.yaml` 的对应 threshold
3. **所有 override 记录写入** `.auto-coding/logs/overrides.log`，包含时间戳、理由、决策人
4. **override 不修改全局规则**，仅对当前任务生效

## A 级最低纪律要求

A 级任务跳过了大部分阶段（仅编码→测试→验证），但必须保留以下纪律底线：

1. **需求确认**：即使跳过 Step 1 设计，也必须回答 3 个核心问题：
   - 改什么？（范围明确）
   - 不改什么？（边界明确）
   - 怎么验证？（验收标准）
2. **测试运行**：编码后必须跑测试，不因 A 级而豁免
3. **安全自查**：任何涉及用户输入、文件操作、网络请求的变更，必须检查安全

## 连续阻塞升级规则

如果同一个 A 级任务连续触发 Scorecard 🔴 block：

- **第 1 次 block**：暂停，Agent 修复问题，继续
- **第 2 次 block**：自动升级为 **B 级任务**，强制走完整 8 步流程
- 日志记录 `.auto-coding/logs/escalations.log`
- 升级后的任务不再降级回 A 级

如果同一个 B/C 级任务的同一阶段连续触发 2 次 block：
- 第 2 次 block 时，要求人类介入决策（不自动继续）
- Agent 输出该阶段的完整 Scorecard Report + 修复历史

## 与 Risk Scorecard 的关系

- `risk-scorecard.skill.md` 定义了**什么规则**（11 条五元组）
- `discipline-meta.skill.md` 定义了**如何使用规则**（何时遵守、何时 override、如何升级）
- 两者随第一次技能注入一起加载，后续复用缓存
- 存在冲突时，discipline-meta 的 override 逻辑优先于 scorecard 的 threshold
