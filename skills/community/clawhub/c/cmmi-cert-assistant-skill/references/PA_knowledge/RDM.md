# RDM — 需求开发与管理（CMMI V3.0 Core PA）

## PA 基本信息
- **缩写**：RDM
- **英文**：Requirements Development and Management
- **中文**：需求开发与管理
- **所属**：Core PA（核心实践域）
- **对应 V2.0**：相当于 RDM（需求开发与管理）

---

## CMMI V3.0 要求要点

### 核心目的
确保需求得到规范管理，需求变更受控，需求与设计/测试的可追溯性得到维护。

### 关键实践（Specific Practices）
| 实践编号 | 实践内容 | 说明 |
|---------|---------|------|
| RDM 1.1 | 挖掘和理解干系人需求 | 需求获取、访谈、调研 |
| RDM 1.2 | 开发运行场景 | 用户故事、用例、场景描述 |
| RDM 1.3 | 开发操作概念场景 | 系统交互场景 |
| RDM 2.1 | 建立和维护需求 | 需求规格说明书 |
| RDM 2.2 | 建立和维护需求追溯性 | 需求跟踪矩阵 RTM |
| RDM 2.3 | 管理需求变更 | 变更控制、影响分析 |
| RDM 3.1 | 定义性能目标 | 性能/质量属性需求 |

### 通用实践（Generic Practices）
- GP 1.1 执行特定实践
- GP 2.1 ~ GP 2.10（完整覆盖）

## 用户文档 → PA 匹配规则

### 高概率匹配（主要PA）
- 文档名含"需求"、"Backlog"、"需求规格" → **RDM 主要匹配**
- 文档含需求列表、用户故事 → **RDM 主要匹配**

### 中概率匹配（次要PA）
- 文档含变更记录 → 同时匹配 **CM**（配置管理）
- 文档含评审记录 → 同时匹配 **PR**（同行评审）
- 文档含测试跟踪 → 同时匹配 **VV**（验证与确认）

## 差距检查清单（用于差距分析）

### RDM 1.x — 需求获取
- [ ] 是否有需求获取记录（访谈/问卷/调研）？
- [ ] 是否有用户故事或用例文档？
- [ ] 需求是否条目化（唯一ID、描述、优先级）？

### RDM 2.x — 需求管理
- [ ] 是否有需求规格说明书（或等效文档）？
- [ ] 是否有需求跟踪矩阵（RTM）？
- [ ] 需求变更是否有记录和审批？
- [ ] 需求和设计/测试用例是否有追溯性？

### RDM 3.x — 需求性能
- [ ] 是否有性能/质量属性需求定义？

## 文档转换指导（模式C）

### 转换后的文档结构（Markdown模板）
```markdown
# 需求规格说明书

## 1. 引言
（目的、范围、定义、参考资料）

## 2. 总体描述
（产品概述、用户特征、运行环境、设计约束）

## 3. 功能需求
| 需求ID | 功能模块 | 需求描述 | 优先级 | 验收标准 |
|---------|---------|---------|--------|----------|

## 4. 非功能需求
（性能需求、安全需求、可用性需求）

## 5. 需求跟踪矩阵（RTM）
| 需求ID | 设计条目 | 开发条目 | 测试用例ID | 验收状态 |
|---------|---------|---------|------------|----------|

## 6. 变更记录
| 变更ID | 变更日期 | 变更内容 | 影响分析 | 审批人 | 状态 |
|---------|---------|---------|----------|--------|--------|
```

## 评估访谈常见问题（模式D）

### 针对PM/需求工程师的问题
1. "需求是如何获取的？有记录在案吗？"
2. "需求变更的流程是什么？请展示变更记录。"
4. "如何保证需求到测试用例的追溯性？"
5. "需求评审是怎么做的？有评审记录吗？"

### 评分要点
- **RDM 1.1-1.3**：需求获取是否系统化
- **RDM 2.1**：需求是否得到规范管理（有唯一ID、版本控制）
- **RDM 2.2**：追溯性是否建立（RTM是否存在）
- **RDM 2.3**：变更是否受控（有审批记录）
