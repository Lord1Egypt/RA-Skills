# CPN 建模指导

## 什么是 CPN？

着色 Petri 网（Colored Petri Net）是一种用于描述**并发、同步、资源竞争**的形式化建模语言。

- **库所（Place）**：系统状态，圆形，持有 token
- **变迁（Transition）**：状态变化动作，矩形，消耗输入 token、产生输出 token
- **弧（Arc）**：连接库所和变迁，带注释说明 token 类型
- **Token**：在库所间流动的数据，携带颜色（类型）值
- **颜色集合（Color Set）**：token 的类型定义，相当于枚举

---

## 第一步：识别库所

**规则：名词 + 状态 = 库所**

问自己：*"系统在某个时刻处于什么状态？"*

| 描述片段 | 库所 |
|---------|------|
| 订单已提交 | `订单_已提交` |
| 货物等待发货 | `货物_待发货` |
| 审批通过 | `审批_已通过` |
| 用户未登录 | `用户_未登录` |

**初始标记（Initial Marking）**：流程开始时哪些库所有 token？
- 流程入口库所必须有初始 token，否则没有变迁能触发
- 资源库所（如"库存充足"、"服务员空闲"）也需要初始 token

---

## 第二步：识别变迁

**规则：动词 = 变迁**

问自己：*"系统做了什么动作，导致状态改变？"*

| 描述片段 | 变迁 |
|---------|------|
| 提交订单 | `提交订单` |
| 审批通过 | `审批通过` |
| 发货 | `发货` |
| 扣减库存 | `扣减库存` |

**变迁触发条件**：所有输入库所都有 token 时，变迁才能触发（enabled）。

---

## 第三步：连接弧

**输入弧**：库所 → 变迁（变迁触发时消耗该库所的 token）
**输出弧**：变迁 → 库所（变迁触发时向该库所放入 token）

每个变迁至少有一条输入弧和一条输出弧。弧上的注释说明 token 的颜色值，如 `1\`堂食`。

---

## 第四步：识别依赖关系

依赖关系描述"A 完成后 B 才能开始"，有三种表达机制：

### arc_sequence（链内顺序）
**场景**：同一业务链内，A 的输出直接是 B 的输入。

```
[P_A完成] → T_B → [P_B完成]
```

A 的输出库所就是 B 的输入库所，token 自然流动，无需额外处理。

### guard_condition（跨链守卫）
**场景**：不同业务链，B 需要等待 A 完成，但 A 的 token 不流入 B。

```
T_B 的守卫条件：P_A完成 ∈ marking
```

B 的变迁加守卫条件，检查 A 的输出库所是否有 token，但不消耗它。

### fusion_place（跨页面融合）
**场景**：不同子项目（页面）之间共享同一个状态。

两个页面各自有一个同名库所，CPN Tools 将它们合并为一个，token 共享。

---

## 第五步：划分子项目（并发角色链）

**核心规则：子项目 = 并发运行的角色链，不是顺序阶段。**

| 错误做法（按阶段） | 正确做法（按角色链） |
|---|---|
| 申请阶段 / 审批阶段 / 通知阶段 | 员工链 / 主管链 / HR链 |
| 顺序流程图，没有并发 | 三条链并发运行，通过融合库所同步 |
| 无法回答"多个请求同时到来会怎样" | 能建模资源竞争和并发等待 |

**判断标准**：如果两个"子项目"永远不会同时有 token 在流动，它们不应该是独立子项目——合并成一条链即可。

**审批流程的正确划分示例：**
- **员工链**：员工提交申请、等待结果（主流程）
- **主管链**：主管接收申请、做出决定（主管是资源库所，初始 token = 1）
- **HR链**：HR接收批准、确认记录（HR是资源库所，初始 token = 1）

这样模型能回答：**同时有 3 个员工请假，主管只有 1 个，第 2、3 个请求会在哪里等待？** 这才是 CPN 建模的核心价值。

**资源库所的标准模式：**
```
P_主管空闲（初始 token: 1）
T_主管审批 的输入：申请_已提交 + 主管_空闲
T_主管审批 的输出：主管_已决定 + 主管_空闲（归还）
```

消耗 + 归还 = 占用期间阻塞其他请求，完成后释放。

## 常见建模错误

### ❌ 多输入变迁的输入库所缺少初始 token

```
T5 需要 P5 AND P6 都有 token
如果 P6（资源库所）初始为空 → T5 永远无法触发 → 死锁
```

**修复**：检查所有多输入变迁，确认每个输入库所在流程某个时刻都能获得 token。

### ❌ 把依赖关系当普通弧处理

跨链依赖不能用普通弧（会错误地消耗 token），应用 guard_condition。

### ❌ 颜色集合过于细化

不需要为每个字段建颜色集合。只对**影响流程走向**的属性建模，其余属性放在 token 的数据载荷里。

### ❌ 遗漏资源库所

"库存"、"服务员"、"审批额度"这类资源是隐含的库所，描述中不一定明说，但它们是变迁的前置条件，必须建模。

### ❌ 按顺序阶段划分子项目

```
子项目：申请阶段 / 审批阶段 / 通知阶段
```

这是流程图思维，不是 CPN 思维。CPN 的子项目应该是**并发运行的角色链**。按阶段划分的模型无法建模资源竞争，也无法回答"多个请求同时到来会怎样"。

**修复**：按参与者角色划分：员工链 / 主管链 / HR链，每个人工参与者建模为资源库所。

### ❌ 无意义变迁（Silent Transition）

```
T_通知批准（输入：HR_已确认，输出：假期_已批准）
```

如果变迁不消耗任何资源、不产生新的并发分支，只是搬运 token，它是无意义变迁，应该合并库所或删除。

**修复**：直接将 HR_已确认 作为终态；或者，如果通知是真实业务动作（发邮件、调接口），则建模为消耗"通知系统"资源的变迁。

### ❌ 冗余颜色集合

```
ApprovalStatus: ["批准", "拒绝"]
```

如果库所名称已经编码了状态（主管_已批准 / 主管_已拒绝），颜色集合没有额外信息量，不影响流程走向。

**修复**：只为**影响流程走向**的属性建颜色集合。资源库所用 `ResourceType: ["空闲"]` 即可。

## 建模流程速查

```
1. 读描述，划出所有名词+状态 → 候选库所
2. 划出所有动词 → 候选变迁
3. 问：哪些库所是流程起点？→ 设初始 token
4. 问：哪些人工参与者或资源是隐含前提？→ 补充资源库所 + 初始 token
5. 连弧：每个变迁的前置状态 → 输入弧，后置状态 → 输出弧；资源库所消耗后必须归还
6. 识别依赖：arc_sequence / guard_condition / fusion_place
7. 按角色链划分子项目（不是按顺序阶段）
8. 检查：是否存在永远无法触发的变迁（死锁检查）
9. 检查：是否有无意义变迁（只搬运 token，无资源消耗）→ 合并或删除
```
