## 二、验证方案设计

### 2.1 验证人群选择

#### 2.1.1 目标用户画像

| 维度 | 描述 |
|------|------|
| 角色 | 外贸业务员/小微外贸企业主 |
| 经验 | 1-3年外贸经验为主 |
| 产品 | 小商品（义乌优势品类） |
| 市场 | B2B为主（B端客户） |
| 痛点 | 合规不懂、选品困难、邮件不会写 |

#### 2.1.2 招募方案

| 来源 | 数量 | 方式 |
|------|------|------|
| 义乌商贸城档口 | 20人 | 地推+礼品激励 |
| 外贸社群 | 30人 | 群公告招募 |
| 合作渠道 | 10人 | 合作伙伴引流 |
| **合计** | **60人** | 最小验证规模 |

#### 2.1.3 激励方案

| 激励类型 | 额度 | 条件 |
|----------|------|------|
| 现金激励 | ¥50-200 | 完成深度访谈 |
| 产品激励 | 1个月VIP | 有效使用满20次 |
| 周边礼品 | 义乌小商品 | 提交有效反馈 |

### 2.2 验证方法

#### 2.2.1 定量验证

| 指标 | 收集方法 | 目标值 | 验证假设 |
|------|----------|--------|----------|
| 日活用户(DAU) | 后台埋点 | >100 | H1 |
| 功能使用率 | 后台埋点 | >70%功能被使用 | H1 |
| 问题解决率 | 用户反馈+AI判断 | >50% | H2 |
| 付费转化率 | 支付数据 | >5% | H3 |
| NPS评分 | 问卷调查 | >40 | H4 |

#### 2.2.2 定性验证

| 方法 | 样本量 | 目的 |
|------|--------|------|
| 用户访谈 | 15-20人 | 深度理解需求 |
| 可用性测试 | 10人 | 发现使用障碍 |
| 焦点小组 | 2组(每组5人) | 需求优先级排序 |

### 2.3 验证任务设计

#### 2.3.1 核心任务清单

| 任务ID | 任务描述 | 预期完成率 |
|--------|----------|------------|
| T1 | 询问1039资质办理流程 | >90% |
| T2 | 询问义乌选品建议（饰品） | >80% |
| T3 | 生成一封开发信（英语） | >85% |
| T4 | 了解物流方案（发往美国） | >80% |
| T5 | 询问支付方式选择 | >70% |
| T6 | 询问报价成本计算 | >75% |
| T7 | 了解风险防控措施 | >70% |
| T8 | 询问客户跟进策略 | >70% |

#### 2.3.2 任务完成判定

```
完成标准：
✅ 任务完成 = 用户问题得到有效回答 + 用户理解并认可

判定维度：
1. 回答相关性（1-5分）：回答是否针对用户问题
2. 回答准确性（1-5分）：回答内容是否正确
3. 回答完整性（1-5分）：是否完整回答用户问题
4. 用户满意度（1-5分）：用户对回答的满意程度

综合评分 ≥ 4分 视为"任务完成"
```

```
┌─────────────────────────────────────────────────────────────────┐
│                        反馈收集四通道                             │

│                                                                  │
│  ┌─────────────┐    ┌─────────────┐    ┌─────────────┐          │

│  └──────┬──────┘    └──────┬──────┘    └──────┬──────┘          │
│         │                  │                  │                  │
│     点赞/点踩           1v1访谈           使用路径             │
│     星级评分           焦点小组           功能停留             │

│         └──────────────────┬──────────────────┘                  │

│                   ┌────────────────┐                              │

│                   └────────┬───────┘                              │
```

### 3.2 嵌入式便捷反馈

| 反馈类型 | 触发时机 | 展示形式 |
|----------|----------|----------|
| 点赞/点踩 | 每条回答后 | 👍👎 按钮 |
| 星级评分 | 使用3次后 | ⭐⭐⭐⭐⭐ |
| 快捷反馈 | 回答后3秒 | "有用/没用/改进" |
| 举报问题 | 任意时刻 | 举报入口 |

### 3.3 主动邀约反馈

| 类型 | 频率 | 对象选择 | 奖励 |
|------|------|----------|------|
| 深度访谈 | 每周5人 | 高活+低活各50% | ¥100 |
| 可用性测试 | 每两周5人 | 新用户为主 | ¥150 |
| 焦点小组 | MVP结束时 | 核心用户10人 | ¥200 |

### 3.4 行为数据采集

| 数据类型 | 采集点 | 分析价值 |
|----------|--------|----------|
| 使用路径 | 入口→功能→退出 | 优化流程 |
| 功能停留 | 各功能页停留时长 | 识别热点 |
| 放弃节点 | 离开时的最后操作 | 发现障碍 |
| 复访频率 | 访问间隔分布 | 评估粘性 |
| 功能使用 | 各功能使用次数 | 优先级参考 |

## 四、快速迭代响应流程

### 4.1 问题分级

| 级别 | 定义 | 响应时间 | 解决时间 |
|------|------|----------|----------|
| P0 | 核心功能完全不可用 | 1小时 | 4小时 |
| P1 | 核心功能受损 | 4小时 | 24小时 |
| P2 | 非核心功能问题 | 24小时 | 72小时 |
| P3 | 体验优化类 | 72小时 | 1周 |

### 4.2 迭代节奏

```
┌─────────────────────────────────────────────────────────────────┐
│                       双周迭代机制                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  Week 1                          Week 2                         │
│  ┌──────────────┐                ┌──────────────┐               │
│  │  数据收集   │                │  问题修复   │               │
│  │  反馈整理   │      →        │  功能优化   │               │
│  │  需求分析   │                │  灰度发布   │               │
│  └──────────────┘                └──────────────┘               │
│         ↑                                                        │
│         │ 持续收集                                              │
│  ┌──────┴──────┐                                               │
│  │  用户行为   │                                                │
│  │  使用数据   │                                                │
│  └─────────────┘                                                │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 4.3 问题处理流程

```
问题发现
    ↓
记录 → 分类(P0/P1/P2/P3) → 评估影响
    ↓
P0/P1 → 立即处理 → 紧急发布
P2/P3 → 排入迭代 → 常规发布
    ↓
验证 → 上线 → 监控
```

### 4.4 决策机制

| 决策类型 | 决策者 | 参考依据 |
|----------|--------|----------|
| 功能优先级 | 产品+运营 | 用户反馈+使用数据 |
| Bug修复 | 技术 | 影响范围+紧急程度 |
| 体验优化 | 产品 | 反馈频率+改进价值 |
| 重大变更 | 产品+技术+业务 | MVP验证结果 |

## 五、验证里程碑

### 5.1 里程碑定义

| 里程碑 | 时间 | 完成标志 | 决策依据 |
|--------|------|----------|----------|
| M1: MVP上线 | 第1周 | 60人激活 | - |

```
Day 1-7: MVP上线
    ↓
Day 7 检查点:
    · DAU是否>50（预警线）？
    · 是否有P0级Bug？
    · 用户反馈是否积极？
    ↓
Day 14 检查点:
    · DAU是否>100（目标线）？
    · 解决率是否>40%？
    · 付费意愿如何？
    ↓
Day 21 检查点:
    · 深度访谈完成度？
    · 核心痛点是否命中？
    · 用户主要不满点？
    ↓
Day 28 验证结论:
    · 继续大版本开发
    · 调整方向
    · 或暂停项目
```

## 十、总结

### 10.1 关键成功因素

1. **精准定位用户**：聚焦义乌小商品外贸场景
2. **快速获取反馈**：多渠道、多形式、高频次
3. **敏捷响应**：小步快跑，快速迭代
4. **数据驱动**：用数据说话，而非主观判断

### 10.2 决策树

```
MVP验证通过？
    │
    ├── 是 → 大版本开发（继续现有方向）
    │
    ├── 部分通过 → 调整方向（优化核心痛点）
    │
    └── 否 → 暂停/转向（重新验证假设）
```

### 10.3 下一步行动

1. **立即启动**：组建MVP验证小组
2. **本周完成**：用户招募+埋点方案
3. **下周开始**：正式验证执行
