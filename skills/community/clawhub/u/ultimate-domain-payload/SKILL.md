# 第一章 核心框架

## 1.1 五大基本需求（驱动所有人类活动）

| ID | 需求 | 说明 | 驱动的领域 |
|----|------|------|-----------|
| N1 | 生存 | 存活的基本条件 | 健康、财富 |
| N2 | 安全 | 稳定和可预期的环境 | 财富、法律、健康 |
| N3 | 归属 | 社会连接和情感纽带 | 家庭、社交、教育 |
| N4 | 成长 | 能力提升和自我实现 | 教育、职业、创意、科技 |
| N5 | 传承 | 超越个体的意义延续 | 家庭、公民、精神、创意 |

## 1.2 六大原始操作（全域共享基元）

所有人类活动都可分解为这 6 种操作的不同组合：

| ID | 操作 | 说明 | AI 自治度 |
|----|------|------|----------|
| S | 感知（Search） | 获取领域相关信息：数据、事实、现状 | ⬛ 全自动 |
| C | 认知（Compose） | 处理信息：识别模式、评估风险、发现机会 | ⬛ 全自动 |
| A | 行动（Act） | 在选项中做出选择：方案比较、优先排序 | 🟨 半自动 |
| O | 组织（Output） | 将决策转化为行动：产出具体成果 | 🟨 半自动 |
| I | 交互（Interact） | 贯穿全程的外部协作：沟通、反馈、协调 | 🟨 半自动 |
| G | 守护（Guard） | 贯穿全程的约束验证：质量检查、合规审查 | ⚡ 校验节点 |

**操作链**：S → C → A → O（主线），I 和 G 贯穿全程

## 2.2 子域文件结构

每个子域按 DPG 完整规范分为四个文件（`references/D{XX}-{name}-*.md`）：

| 文件 | 内容 | 参考规模 |
|------|------|---------|
| `{prefix}-SKILL.md` | 子域定位、三层结构、依赖声明、使用规则、UTOS 接口、领域特有维度、域概览 |
| `{prefix}-catalog.md` | 子域划分、任务清单表格（ID/名称/说明/依赖/UTOS映射）、依赖拓扑摘要 |
| `{prefix}-requirements.md` | 每个任务类型的必选组件、可选组件、组装顺序、领域约束、格式 |
| `{prefix}-exemplars.md` | 3 个完整工作流范本（含校准点、跨域链接） |

## 2.4 依赖声明

本技能**强依赖** Universal Task OS (universal-task-os)。没有UTOS，本技能只有参考查阅能力，无法执行任何跨域编排任务。

**加载检查流程**（每次激活时执行）：

1. 检测 `universal-task-os` 技能是否已安装
2. **未安装** → 自动安装 `universal-task-os` 技能
3. **安装成功** → 同时加载UTOS，按本技能"使用规则"执行
4. **安装失败** → 降级为**只读参考模式**：
   - ✅ 允许：查阅领域注册表、任务清单、要求、范本索引
   - ❌ 拒绝：任何涉及跨域编排、管线生成、内容产出的任务，并提示"需先安装 Universal Task OS"

**任务模式判定**：

| 任务类型 | 无UTOS | 有UTOS |
|---------|--------|--------|
| 查阅领域清单/要求/范本 | ✅ 只读参考 | ✅ 完整 |
| 按清单产出领域任务成果 | ❌ 拒绝 | ✅ UTOS编排执行 |
| 跨域价值链编排 | ❌ 拒绝 | ✅ UTOS执行轴 |
| 合规/安全校验点插入 | ❌ 拒绝 | ✅ UTOS守护单元 |

## 2.5 与UTOS的接口

当UTOS处理本技能的跨域任务时：

- **Step 0 三轴判定**：跨域任务通常为复杂+结构化 → 执行轴+内容轴激活；需要创新时额外激活创新轴
- **Step 1 领域校准**：根据涉及的具体领域组合确定校准参数——健康域 R4(规范性)=高、财富域 R1(信息密度)=高、创意域 R2(创造性)=高、法律域 R4(规范性)=极高
- **Step 2 内容轴**：清单法用各子域的 requirements；样本法用各子域的 exemplars
- **Step 3 执行轴**：管线编排基于跨域价值链（第三章）和各子域的依赖拓扑自动推导元操作序列
- **Step 4 交付**：G类守护单元在合规节点（法律/安全/伦理）自动插入校验点

  → D01 健康.信息输入(I)

## 3.2 跨域传递规则

| # | 规则 | 说明 |
|---|------|------|
| 1 | O→I 必须显式映射 | 上游输出必须明确是下游的什么输入 |
| 2 | 校准点不可跳过 | 跨域传递必须经过校准点确认 |
| 3 | 合规节点保留 | 法律/安全/伦理相关的跨域传递必须有合规校验 |
| 4 | 单链基元数 ≤5 | 一条跨域链路的基元数不超过 5 个 |
| 5 | 并行优于串行 | 无依赖关系的跨域链路应并行执行 |

# 第五章 使用指南

## 5.1 使用方式

1. **识别任务涉及哪些生命领域**：查第二章领域总览
2. **加载子域定位**：读取 `references/D{XX}-{name}-SKILL.md` 获取领域特有维度和 UTOS 接口
3. **加载任务清单**：读取 `references/D{XX}-{name}-catalog.md` 获取子域划分和依赖拓扑
4. **深入任务要求**：读取 `references/D{XX}-{name}-requirements.md` 获取每个任务的组件清单
5. **获取范本参考**：读取 `references/D{XX}-{name}-exemplars.md` 获取完整工作流范本
6. **识别跨域链路**：查第三章，确定 O→I 映射和校准点
7. **用 UTOS 编排执行**：三轴处理（执行轴+内容轴+创新轴）
8. **校准点确认**：每个跨域传递必须经过校准点

## 5.2 与三个技能的关系

```
Workflow Refactor → 简化域间和域内流程
Domain Payload Generator → 为子域生成更详细的负载物
Universal Task OS → 消费此负载物，三轴编排执行
此文件 → 注册表 + 跨域价值链 + 12 子域索引 = 跨域关系定义
references/D{XX}-{name}-*.md → 每个子域的 SKILL + catalog + requirements + exemplars = 领域知识层
UTOS → 实际编排引擎，消费此文件的跨域关系定义和子域的领域知识
```

## 5.3 扩展子域

当需要为某个生命领域生成更详细的子域负载物时：
1. 确认任务类型 ID 和边界
2. 用 DPG 的四文件结构模板深入生成：`{task}-SKILL.md` + `{task}-catalog.md` + `{task}-requirements.md` + `{task}-exemplars.md`
3. 文件命名为 `references/D{XX}-{task}-{type}.md`
4. 更新本文件的领域注册表
