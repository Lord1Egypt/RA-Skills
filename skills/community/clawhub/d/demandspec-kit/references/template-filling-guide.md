# DemandSpec 模板填写指南

本指南用于把 `templates/` 中的骨架写成可评审、可开发、可测试的需求资产。模板标题可以按项目调整，但最低完成条件不能用空白、`TODO`、泛化口号或无责任人的“待确认”代替。

## 通用填写规则

1. 每个事实注明来源，例如会议日期、原文链接、现有系统或数据表。
2. 每个假设写明验证方式、负责人和最晚确认时间。
3. 每条功能需求分配稳定 ID，并在规格、验收、测试和任务中复用。
4. 数量、时效、成功率、阈值和容量尽量量化；尚无基线时明确“先测基线再定目标”。
5. “支持、优化、智能、友好、及时”不能单独作为要求，必须改写成可观察行为。
6. 删除不适用章节，或写明“不适用及原因”，不要保留空标题。

## 模板选择与最低完成条件

| 模板 | 何时使用 | 最低完成条件 |
|---|---|---|
| `demand-card.md` | 原始想法、会议纪要或口述需求首次受理 | 明确问题、目标用户、目标场景、期望结果、初步范围/非范围、干系人和风险；未知项带负责人 |
| `clarify-questions.md` | L0-L2 或现有材料存在冲突 | 区分阻塞问题、可后置问题、当前假设和待补材料；每个阻塞问题说明影响 |
| `scope-boundary.md` | 需要确定 MVP、版本边界或防止范围蔓延 | In Scope、Out of Scope、MVP、边界条件、后续版本和变更规则 |
| `ai-fit-assessment.md` | 涉及生成、抽取、分类、预测、推荐、Agent 或多模态 | AI 适配结论、非 AI 替代方案、输入质量、输出定义、模型/业务/风险指标、人工兜底和低置信度处理 |
| `process-model.md` | 涉及角色协作、审批、跨系统或流程改变 | As-Is、To-Be、变化点、异常分支；每步包含角色、输入、动作和输出 |
| `data-fields.md` | 页面、接口、报表、AI 输入输出或数据迁移 | 字段名、类型、来源、必填、生成方式、校验、权限、敏感级别和缺失处理 |
| `business-rules.md` | 存在判断、计算、状态流转、阈值或权限规则 | 规则 ID、触发条件、规则表达、优先级、异常处理、冲突和变更机制 |
| `prototype-spec.md` | 涉及页面或交互 | 页面清单、信息架构、主路径、核心交互，以及空/加载/成功/失败/禁用/权限状态 |
| `figma-mastergo-spec.md` | 需要交付设计工具生成或精细原型 | Frame 尺寸、布局、组件树、交互跳转、数据示例、生成 Prompt 和原型验收清单 |
| `prd.md` | 需求达到 L4，需要形成评审基线 | 背景目标、范围、角色、流程、带 ID 的功能、页面、字段、规则、权限、非功能、验收、风险与依赖相互一致 |
| `acceptance-criteria.md` | 准备评审、测试或研发交付 | 每个 P0/P1 需求至少一个 GIVEN/WHEN/THEN 场景，并覆盖数据、AI、权限、异常和非功能要求 |
| `test-cases.md` | 验收标准需要转为可执行测试草案 | 用例 ID、需求 ID、前置条件、步骤、测试数据、预期结果、优先级；AI 场景含阈值边界与失败回退 |
| `review-checklist.md` | PRD、变更包或交付前质量检查 | 每项给出通过/不通过/不适用、证据链接、问题负责人和整改期限 |
| `dev-handoff.md` | 需求达到 L5，需要拆解研发工作 | 前端、后端、算法、数据、测试任务带需求 ID、输入输出、依赖、完成定义；写明上线和回滚 |
| `archive.md` | 需求上线、取消或变更完成 | 变更记录、上线反馈、指标结果、偏差原因、经验、可复用资产和下一轮动作 |

## 从空泛到可执行

### 目标

空泛：

> 提升拜访单填写效率。

可执行：

> 在销售人员提交前的草稿阶段，从录音转写、现场图片 OCR 和邮件正文中生成拜访单字段；试点期将单次人工填写中位时长从基线降低至少 40%，且所有生成字段均可追溯到来源材料。

### 功能要求

空泛：

> 系统支持智能生成拜访单。

可执行：

> `CRM-VISIT-001` 系统 MUST 在用户选择客户并上传受支持材料后生成拜访单草稿，展示每个生成字段的来源和置信度；未完成人工确认前 MUST 禁止正式提交。

### 待确认项

空泛：

> 录音格式待确认。

可执行：

> 待确认：MVP 是否仅支持 MP3、M4A、WAV；影响上传校验和转写服务选型；负责人：语音平台负责人；截止：技术评审前。

### AI 指标

空泛：

> 模型准确率要高。

可执行：

> 在已人工标注的试点集上，客户诉求字段 micro-F1 不低于 0.85；置信度低于 0.70 的字段必须进入人工确认，系统不得自动提交。

### 原型

空泛：

> 增加 AI 生成页面。

可执行：

> 在现有拜访单编辑页增加“生成草稿”入口、材料上传区、处理进度、字段来源、置信度标签和人工确认状态。处理失败时保留已上传材料，并提供重试和切换为手工填写的入口。

## 跨文件追踪

以 `CRM-VISIT-001` 为例：

```text
proposal.md
  -> specs/crm/spec.md: CRM-VISIT-001
  -> 05_spec/prd.md: FR-CRM-VISIT-001
  -> acceptance.md: CRM-VISIT-001 / AC-001..AC-004
  -> test-cases.md: TC-CRM-VISIT-001-*
  -> tasks.md: FE/BE/ALG/DATA/QA tasks tagged CRM-VISIT-001
```

验收和任务出现没有规格来源的 ID，或规格没有对应验收和任务，都应视为追踪失败。

## 完成判定

文档只有在以下条件满足时才算完成：

- 关键章节包含项目事实或明确假设，不是原模板提示语。
- 范围、流程、字段、规则和验收之间没有明显冲突。
- P0/P1 要求有稳定 ID、验收场景和交付任务。
- AI 和 UI 的专项要求已覆盖。
- 所有阻塞项有负责人，不能确认的内容没有伪装成结论。
- 已执行严格校验，或记录了等价人工检查及证据。
