# AI PR Narrative Factory

**把一个 PR URL 丢进来，还你一份专业的变更叙事报告**

## 业务场景

在现代软件开发中，Pull Request 是代码变更的核心载体。但代码合并只是开始——团队还需要：

- 向非技术同事解释这次变更的价值
- 为发版准备规范的 Changelog
- 让产品经理了解功能影响范围
- 形成团队知识沉淀

**痛点：** 手工编写变更叙事费时费力，且不同人写的质量参差不齐。Code Review 只能帮助开发者理解代码，无法生成面向全团队的沟通文档。

## 痛点分析

| 痛点 | 现状 | 理想状态 |
|------|------|---------|
| 非技术人员看不懂代码 diff | 用大白话解释半天 | AI 自动生成通俗解读 |
| Changelog 靠手工编写 | 发版前临时凑 | PR 合并后自动生成 |
| 团队同步靠口头描述 | 关键信息容易遗漏 | 永久文档可追溯 |
| 跨团队通知成本高 | 每方单独沟通 | 一键生成多版本报告 |

## 协作流程说明

### Step 1：github — PR 数据提取
使用 gh CLI 提取：
- PR 标题、描述、作者、状态
- 涉及文件列表和代码行数变化
- 完整的代码 diff 内容
- 关联的 Issue 和标签

### Step 2：kimi-cli — 代码变更分析
AI 深度分析代码变更：
- 识别新增功能（Feature）
- 识别 Bug 修复（Fix）
- 识别破坏性变更（Breaking Change）
- 识别依赖变更（Dependency Update）
- 评估测试覆盖情况

### Step 3：summarize — 叙事生成
将分析结果转化为：
- **通俗版**：非技术同事能看懂的功能描述
- **技术版**：开发者需要知道的实现细节
- **公告版**：适合发公告/邮件的产品描述
- **Changelog 条目**：符合 Keep a Changelog 规范

### Step 4：feishu-create-doc — 文档发布
生成结构化飞书云文档：
- 永久可分享链接
- 支持评论和协作
- 可嵌入团队知识库

## 使用示例

### 场景 1：Code Review 结果同步给产品经理

**用户：**
> 这个 PR 改了什么？帮我生成说明：https://github.com/owner/repo/pull/456

**AI 执行：**
1. 提取 PR 代码 diff
2. 分析变更内容
3. 生成通俗描述 + 技术说明
4. 发布飞书文档

**产品经理看到的文档：**
> ## 🔍 核心变更
> **新增功能**：用户头像支持裁剪上传
> 现在支持在上传头像时实时裁剪，无需再使用外部工具。
> 预计用户满意度提升 10-15%。

### 场景 2：版本发布 Changelog 生成

**用户：**
> 从 v2.1.0 到 v2.2.0 的所有 PR 生成分享链接

**AI 执行：** 批量分析多个 PR，生成完整版本变更日志

## 技术细节

- **触发方式**：发送 GitHub PR URL 或 `owner/repo#PR号`
- **输出格式**：飞书云文档（永久链接）
- **支持批量**：可输入多个 PR 或版本区间
- **语言**：中文为主（可根据用户偏好调整）
