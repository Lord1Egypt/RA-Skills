## 一、总体架构

### 视角一：端到端流程（横向流水线）

```
  需求方/产品经理              开发经理                    测试人员
       │                          │                           │
       ▼                          ▼                           ▼
  ┌───────────┐  CP1  ┌───────────┐  CP2  ┌───────────┐  CP3  ┌───────────┐  CP4  ┌───────────┐
  │ 需求对话   │──────▶│ 开发编排   │──────▶│ 代码 Review │──────▶│ 测试编排   │──────▶│  代码交付   │
  │ Agent     │ 确认   │ Agent     │ 决策   │ Agent     │ 审查   │ Agent     │ 审查   │           │
  └───────────┘       └───────────┘       └───────────┘       └───────────┘       └───────────┘
        │                   │                   │                   │
        └───────────────────┼───────────────────┼───────────────────┘
                            │
                    ┌───────┴────────┐
                    │   系统知识层    │
                    └───────┬────────┘
                            │
        ┌───────────────────┼───────────────────┐
        │                   │                   │
   ┌────┴────┐        ┌────┴────┐        ┌────┴────┐
   │ 代码库  │        │ 数据库  │        │ 历史经验 │
   │ 分析器  │        │ 分析器  │        │ 提取器  │
   └─────────┘        └─────────┘        └─────────┘
```

**流程说明**：需求方与需求对话 Agent 沟通 → CP1 确认后进入开发编排 → 遇决策点上 CP2 等开发经理选择 → 自测通过进入 CP3 代码 Review → CP3 通过后进入测试编排 → CP4 测试人员审查盲区 → 交付。

### 视角二：角色交互关系

| 角色 | 做什么 | 与智能体的交互方式 |
|------|--------|-------------------|
| **需求方/业务** | 描述业务诉求、确认需求理解 | 直接对话，智能体主动追问缺失信息 |
| **产品经理** | 审查需求文档质量 | 「判断 AI 理解得对不对」，不自己写 PRD |
| **开发经理** | 在 CP1-CP4 做架构/安全/业务决策 | 「AI 给出 2-3 个选项，人选方案」 |
| **开发工程师** | 验证系统知识、必要时介入修改代码 | Phase 1 验证知识文档，Phase 3 兜底 AI 无法收敛的任务 |
| **测试工程师** | 对照盲区检查清单审查测试覆盖、执行探索性测试 | AI 生成用例和报告，人找盲区 |

### 🔧 视角三：知识飞轮（三件事的闭环）

本方案的实质不是"做一个工具"，而是**建立一个运转后自我增强的知识飞轮**：

```
                    ┌─────────────────────┐
                    │     知识体 (3)       │
                    │  源码 + 对话沉淀      │
                    └──────────┬──────────┘
                               │
                    ┌──────────┴──────────┐
                    │                     │
                    ▼                     ▼
          ┌─────────────┐       ┌─────────────┐
          │ 需求沟通 (1)  │       │ 开发测试 (2)  │
          │ 基于已有知识   │──────▶│ 基于需求文档   │
          │ 追问+否定+确认  │       │ 自主推进       │
          │ 输出需求文档    │       │ 遇岔路口等人   │
          └──────┬──────┘       └──────┬──────┘
                 │                     │
                 └──────────┬──────────┘
                            │
                            ▼
              每一次对话和决策都回流到知识体
              （否定了什么、选了什么方案、被纠正了什么）
```

**飞轮运转逻辑**：

```
第一轮需求：知识体只有源码 → 需求对话磕磕绊绊 → 产出需求文档 + 否定/决策知识
     ↓
第二轮需求：知识体 = 源码 + 第一轮沉淀 → 追问更精准 → 需求文档质量更高
     ↓
第三轮需求：知识体 = 源码 + 两轮沉淀 → 智能体已经知道"VIP不用这个""T+1就行"
             → 需求方不再被反复追问同样的问题
     ↓
   ...
     ↓
第 N 轮需求：知识体足够丰富 → 需求方直接描述，智能体秒懂 → 大部分需求自动推进到开发
```

**启动与加速**：

- **冷启动靠源码**：第一轮需求时知识体只有代码分析，智能体靠追问规则弥补知识不足
- **加速靠沉淀**：每多一个需求，知识体就多一批否定/决策/纠正记录，后续需求越来越顺
- **最终稳态**：80% 的常见需求，智能体从知识体就能推断出答案，只问最关键的不确定性

### 核心原则

- 🔧 智能体必须区分「已知事实」与「推断」，不确定时显式声明
- 三层架构：**Agent 层**（对话和编排能力）/ **知识层**（结构化系统知识）/ **Check Point 层**（人类决策嵌入点，控制风险和建立信任）
- 智能体走完 90% 的路，在关键节点停下来等人确认

### 🔧 迭代与反复：流程不是一条直线

方案中的流程图是"理想路径"，但实际执行中每条箭头都可能是双向的。不回避反复，而是预判反复发生的位置、控制反复的代价。

**每个阶段预期的迭代轮次**：

```
Phase 1 知识注入
  AI提取初稿 → 开发者验证 → 发现问题 → AI重新提取
       │                                    │
       └────────── 通常 2-3 轮 ──────────────┘
  最坏情况：AI 一直理解错某个模块的核心逻辑，多轮后仍未收敛
  → 兜底：该模块改由开发者手写知识文档，不再依赖 AI 提取

Phase 2 需求对话
  需求方描述 → AI追问 → 需求方补充 → AI产出文档 → 需求方质疑 → AI修改文档
       │                                                    │
       └──────────── 通常 3-5 轮对话 ────────────────────────┘
  每一轮"反向确认"都可能触发新的追问，回到需求捕获步骤
  最坏情况：需求方说不清自己要什么，AI 也问不出
  → 兜底：产品经理介入，人工引导需求方，AI 旁路记录并产出文档

Phase 3 开发编排
  代码生成 → 测试失败 → AI修复 → 测试通过 → Review不通过 → 补充修改
       │         │                     │              │
       │    ┌────┘              ┌──────┘              │
       │    ▼                   ▼                     │
       │  再生成              修改代码                 │
       │    │                   │                     │
       └────┴───────────────────┴─────────────────────┘
              通常在 2-5 个代码修复循环内收敛
              最多 3 轮自修复（硬性上限），超过则人工接手
  Review 不通过 → 回到开发或测试步骤，最多 2 轮 Review
  CP2 决策点 → 开发经理选方案后 AI 重新执行，可能触发新的决策点

Phase 4 测试
  AI生成用例 → 测试人员审查 → 发现遗漏 → AI补充用例 → 再审查
                                        │
                                 可能发现新遗漏
  通常 1-2 轮补充，极端情况需要 3 轮以上
  每次上线后发现 Bug → 盲区检查清单追加 → 影响后续所有需求的 CP4
```

**跨阶段的回退**：

```
Phase 3 开发中              Phase 3 测试中
  发现需求文档有歧义           发现系统知识与代码不一致
       │                           │
       ▼                           ▼
  回到 Phase 2                回到 Phase 1
  让需求方澄清                AI 重新分析该模块
       │                           │
       ▼                           ▼
  需求文档更新后               知识文档更新后
  重新进入 Phase 3            告知测试人员新发现的风险点
```

**控制反复成本的规则**：

| 场景 | 上限 | 超限处理 |
|------|------|---------|
| 需求对话轮次 | 20 轮 | 自动汇总待澄清项，产品经理人工介入 |
| AI 代码自修复 | 3 轮 | 终止，开发经理接手 |
| Review 打回 | 2 次 | 开发经理和 AI 同步修改，不再单独走 AI 自主循环 |
| 知识文档修正 | 3 轮 | 人工手写该条目 |
| 需求澄清回退 | 不限 | 但每次回退记录原因，用于改进需求对话 Agent 的追问规则 |

**这些反复不是坏事**——每一次回退都在暴露系统知识和追问规则的盲区，持续改进后反复会减少。方案不是假设一路顺利，而是设计好反复发生时的处理路径和止损边界。

## 二、Phase 1：知识注入层（2-3 周）

### 2.1 目标

让智能体拥有系统的"完整理解"——不是人手写的文档，而是 AI 从代码、数据库、历史记录中自动提取并持续更新的结构化知识。

### 2.2 知识提取的四个维度

#### 维度一：代码库知识

| 提取内容 | 方法 | 产出 |
|---------|------|------|
| 项目结构 | 目录树 + 模块依赖分析 | `knowledge/architecture.md` |
| API 清单 | 扫描路由定义、Controller | `knowledge/api-catalog/`（分片） |
| 数据模型 | 扫描 Model/Entity | `knowledge/data-model/`（分片） |
| 业务规则 | 扫描 Service 层逻辑 | `knowledge/business-rules/`（分片） |
| 配置项 | 扫描配置文件、环境变量 | `knowledge/config.md` |
| 公共组件/工具 | 扫描 common/utils 目录 | 补充到 `architecture.md` |

**执行方式**：写一个脚本/Cursor/seazenai Code 任务，逐模块让 AI 阅读并输出结构化摘要。

**关键原则**：
- 不是流水账式的"A 模块有 X、Y、Z 文件"——那是目录结构，不是知识
- 每个知识条目要回答：**这个模块做什么、对外暴露什么、依赖什么、有什么约束**
- 代码中的注释、TODO、FIXME、HACK 标记要特别提取——这些是代码中罕见的"意图表达"

#### 维度二：数据库知识

| 提取内容 | 方法 | 产出 |
|---------|------|------|
| 表结构 | 读取 DDL / migration 文件 | `knowledge/data-model/<module>-tables.md` |
| 表间关系 | 外键 + 代码中的 JOIN 模式 | 补充到 `data-model/` |
| 核心数据流 | 追踪关键业务流程的数据读写 | `knowledge/data-flow/<module>.md` |
| 索引策略 | 读取索引定义 | 补充到 `data-model/` |

**执行方式**：优先从 migration/schema 文件读取，如果没有则连接数据库读取 information_schema。

#### 维度三：接口与集成知识

| 提取内容 | 方法 | 产出 |
|---------|------|------|
| 对内 API | 扫描路由/Controller | `knowledge/api-catalog/<module>.md` |
| 对外接口 | 扫描 Feign/HTTP Client 调用 | `knowledge/api-catalog/external-apis.md` |
| 消息队列 | 扫描 MQ Producer/Consumer | `knowledge/mq-topics.md` |
| 定时任务 | 扫描 Scheduler/Cron 定义 | `knowledge/jobs.md` |

#### 维度四：隐性知识（经验与教训）

| 提取内容 | 方法 | 产出 |
|---------|------|------|
| 历史上出过的 Bug | `git log --grep="fix\|hotfix\|bug"` 遍历 | `knowledge/pitfalls.md` |
| 性能优化记录 | `git log --grep="perf\|optimize\|slow"` | 补充到 `pitfalls.md` |
| 安全修复记录 | `git log --grep="security\|inject\|xss\|auth"` | 补充到 `pitfalls.md` |
| PR 讨论中的关键决策 | 如有 GitHub/GitLab API，读取 PR 评论 | `knowledge/decisions.md` |

#### 🔧 维度五：决策与否定知识（持续沉淀，非一次性提取）

这是 AI 时代特有且最有价值的知识类型。传统文档只记录"最终做了什么"，但**被否定的方案、被澄清的边界、"我们不要什么"——这些否定信息定义了系统的约束边界，比正向描述更有决策价值**。

**知识来源**（贯穿所有 Phase，持续沉淀）：

| 来源 | 提取时机 | 示例 | 沉淀到 |
|------|---------|------|--------|
| 需求对话中的否定 | Phase 2 每次对话结束时（经需求方确认后入库） | "这个功能不对 VIP 用户开放"、"不需要实时，T+1 就行" | `knowledge/negations.md` |
| 需求对话中的决策 | Phase 2 每次对话结束时（经需求方确认后入库） | "选方案 A 而不是 B，因为合规要求" | `knowledge/decisions.md` |
| CP2 架构决策 | Phase 3 每个需求 | "本次不用消息队列异步，量级小直接同步" | `knowledge/decisions.md` |
| Review 中的纠正 | Phase 3 CP3 | "支付回调不能放在事务里，会被长事务锁表" | `knowledge/pitfalls.md` |
| 测试盲区补充 | Phase 4 CP4 | "跨月场景比跨天场景更容易出问题" | `knowledge/pitfalls.md` |

**`negations.md` 的结构**（每个条目记录一次否定）：

```markdown
## [主题/模块]

### N001: [被否定的假设]
- **场景**：需求对话中，智能体假设 X，需求方纠正为 Y
- **来源**：[日期/需求编号/对话轮次]
- **正确的理解**：实际应该是 Y
- **影响范围**：这条否定会影响哪些后续判断
- **关联决策**：见 decisions.md #D00X

### N002: ...
```

**`decisions.md` 的结构**（每个条目记录一次带原因的决策）：

```markdown
## [模块/功能]

### D001: [决策标题]
- **候选方案**：A: xxx / B: yyy / C: zzz
- **选择**：方案 X
- **原因**：为什么选这个（性能/合规/成本/时间）
- **被否定的方案及原因**：为什么不选另外几个
- **日期**：2026-xx-xx
- **关联否定**：见 negations.md #N00X
```

**关键原则**：否定和决策知识不是 Phase 1 一次性提取的，而是在后续每个 Phase 的对话和决策中**持续沉淀**的活文档。这就是"反复不是坏事"的具体体现——每一次回退、每一次否定、每一次决策选择，都在丰富系统的约束知识。

### 2.3 🔧 知识提取执行手册（10 步操作清单）

以下操作清单是 Phase 1 第一周的执行脚本。目标是让执行者不需要从零摸索"怎么让 AI 提取知识"。

```
准备阶段：环境与范围确认
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Step 1: 划定模块边界
  - 操作：让 AI 读取项目根目录 + 构建配置文件（pom.xml/build.gradle/package.json），
          输出"模块/子项目清单 + 模块间依赖关系"
  - 产出：模块边界图（文字版），粘贴到 knowledge/INDEX.md
  - 验证：开发经理扫一眼，纠正模块划分错误

Step 2: 标记代码风格分区
  - 操作：让 AI 扫描各模块的代码风格特征（命名约定、包结构、注解风格），
          标记出风格不一致的区域（"user 模块用 MyBatis，order 模块用 JPA"）
  - 产出：风格分区说明，追加到 INDEX.md
  - 目的：防止 AI 在后续提取中把不同风格的模块混为一谈

Step 3: 建立提取优先级
  - 操作：按"被依赖最多 > 业务核心 > 最近变更频繁 > 边缘模块"排序
  - 产出：优先级列表
  - 原则：先提取被最多模块依赖的核心模块（如用户/权限），
          因为这个模块的知识在提取其他模块时也要用到


提取阶段：逐模块提取
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Step 4: 从入口文件开始（每个模块）
  - 操作：AI 先读取 Controller/路由定义 → 识别该模块对外暴露的所有 API
  - 产出：该模块的 API 摘要（路径 + 参数 + 返回值 + 权限），写入 api-catalog/<module>.md
  - 不做什么：此时不深入 Service 层，不追踪数据库操作

Step 5: 追踪数据流
  - 操作：从每个 API 入口向下追踪 Service → Repository/DAO → 数据表
          记录读写哪些表、什么条件、什么事务边界
  - 产出：数据流描述，写入 data-flow/<module>.md
  - 容易出错点：跨模块调用（Service A 调 Service B）要标记但不深入

Step 6: 提取业务规则
  - 操作：AI 聚焦 Service 层中的 if/switch/状态机/校验逻辑，
          用自然语言重写每条规则
  - 产出：business-rules/<module>.md
  - 特别注意：代码中的注释、TODO、FIXME —— 一起提取，标注为"代码自述"

Step 7: 识别死代码与废弃项
  - 操作：AI 标记以下信号：
          - @Deprecated 注解
          - 注释中有 "废弃" "不再使用" "临时" "workaround"
          - 从未被引用的 public 方法（需 IDE 静态分析辅助）
          - 配置文件中标注为 deprecated 的配置项
  - 产出：在对应知识文档中标注 ⚠️ "可能废弃/待确认"
  - 验证：由模块开发者确认是否真的废弃


验证与沉淀阶段
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Step 8: 开发者 30 分钟快速验证
  - 操作：将该模块的所有知识产出交给负责该模块的开发者，
          要求 30 分钟内标注三色标签：
          🟢 准确 / 🟡 有遗漏 / 🔴 有错误
  - 产出：标注后的知识文档
  - 规则：30 分钟是硬性上限，不要求逐行审查，只凭经验快速标注
  - 开发者发现 🟡/🔴 时必须附一句说明（"漏了 XX 场景"）

Step 9: 定向补充提取
  - 操作：对开发者标注为 🟡/🔴 的部分，AI 重新读代码做定向提取，
          最多 2 轮（按照知识文档修正上限）。
          2 轮后仍有问题 → 该条目改由开发者手写
  - 产出：修正后的知识文档

Step 10: 注册到 INDEX.md
  - 操作：将该模块的知识文档路径、摘要、关键词标签写入 INDEX.md
  - 关键词标签格式：`[用户, 权限, RBAC, 登录, 认证]`
  - 目的：供后续需求对话时 AI 做关键词匹配，定位到对应模块知识
```

**执行节奏建议**：
- Week 1 前半周：Step 1-3（全局准备）
- Week 1 后半周 + Week 2：Step 4-7（核心模块提取，2-3 天/模块，并行）
- Week 3 前半周：Step 8-9（开发者验证 + 补充提取）
- Week 3 后半周：Step 10（全量注册索引）
- Week 4：隐性知识提取（维度四） + 手动触发首次知识刷新

### 2.5 知识保鲜机制 🔧（修订：从 CI 自动化改为手动触发优先）

这是最难也最重要的一环。原方案中的 CI Hook + AI 分析 diff 的轻量级增量更新存在风险：diff 与知识文档段落的精准映射很难做到，错误的增量更新比不更新更危险。

**修订策略**：先稳后快，手动验证 → 定期自动 → 再考虑增量。

```
第一阶段（Phase 1 期间）：手动触发，人工抽查
  代码合入 → 无自动操作 → 每周手动触发一次变更模块的全量刷新
  → 知识文档更新后人工抽查 2-3 个核心模块的准确性
  → 跑稳 3-4 周，建立对知识质量的信心

第二阶段（Phase 2 后期）：定期自动刷新
  每周自动触发一次「上次合入涉及的模块」的全量知识刷新
  → 变更日志记录每次刷新的模块和结果
  → 仍然不接入 CI，作为独立定时任务运行

第三阶段（Phase 3 稳定后）：考虑增量更新
  只有在全量刷新跑稳 ≥ 2 个月、知识质量被验证可靠后
  → 再考虑 CI Hook 增量更新
  → 增量更新必须同时保留每周全量刷新的兜底机制
```

**关键原则**：知识层的错误是系统性风险——后续所有决策都依赖于它。不值得为了自动化而加速，宁可慢而准。

### 2.6 🔧 知识粒度与加载策略（上下文窗口约束）

**问题**：一个中型系统的知识文档（API 清单、数据模型、业务规则等）加起来可能几十万 token。实际 AI 调用时上下文窗口有限，不可能每次全量注入。需要「知识路由」机制。

**方案**：三层粒度，按需加载。

| 层级 | 粒度 | 内容 | 使用场景 | 预估 token |
|------|------|------|---------|-----------|
| **L0：索引入口** | `INDEX.md`（~2K token） | 模块列表 + 各知识文件的摘要 + 关键词索引 | 每次对话/任务启动时必读 | 1-3K |
| **L1：模块摘要** | 每个模块一段（~5K token/模块） | 该模块做什么、对外暴露什么、核心约束 | 需求对话时按关键词匹配 1-3 个模块 | 3-15K |
| **L2：模块详情** | 每个模块完整知识（~15-30K token/模块） | 所有 API 签名、数据表详情、业务规则细节 | 开发编排时读目标模块的完整知识 | 15-30K |

**加载决策逻辑**：

```
智能体收到任务
    │
    ▼
读取 INDEX.md（必读） ──▶ 识别任务关键词
    │
    ▼
匹配相关模块（最多 3 个）──▶ 加载 L1 摘要
    │
    ├── 需求对话场景 → L1 足够，不加载 L2
    │
    └── 开发编排场景 → 根据任务拆解结果，加载 1-2 个目标模块的 L2 详情
```

**知识文档的分片结构**（以 `api-catalog.md` 为例）：

```
knowledge/
├── INDEX.md                    # 总索引
├── api-catalog/
│   ├── _index.md               # API 目录摘要（~3K token）
│   ├── user-module.md          # 用户模块 API（~10K token）
│   ├── order-module.md         # 订单模块 API（~15K token）
│   └── payment-module.md       # 支付模块 API（~8K token）
├── data-model/
│   ├── _index.md               # 数据模型摘要
│   ├── user-tables.md
│   └── ...
```

**关键规则**：
- INDEX.md 和所有 `_index.md` 必须可以通过纯文本匹配定位到对应详情文件
- 每个分片文件的首部包含「关键词标签」，供 AI 快速判断是否需要深入加载
- 知识保鲜更新到分片级别，只刷新变更模块对应的分片文件

### 2.7 知识层的验收标准

- [ ] 任何一个模块，智能体可以回答：它做什么、输入输出是什么、依赖谁、被谁依赖
- [ ] 任何一张表，智能体可以回答：字段含义、关系、哪里读写
- [ ] 任何一个 API，智能体可以回答：路径、参数、返回值、权限要求
- [ ] 业务规则知识覆盖所有核心流程（订单、支付、权限等）
- [ ] `pitfalls.md` 包含近 6 个月内至少 80% 的线上 bug 涉及的模块警示

## 三、Phase 2：需求对话智能体（3-4 周）

### 3.1 目标

让需求方（业务部门/产品经理）通过与智能体对话，产出一份**开发就绪**的需求文档。智能体利用 Phase 1 的系统知识，主动引导、追问、校验。

🔧 对话过程中，智能体不仅记录"确认了什么"，也必须记录"否定了什么"。中文对话中否定表达极其多样（"也不完全是这样""其实我们之前试过不太行""这个倒没必要"），全自动识别既不准确也不可靠——漏提是噪音，错提是误导。因此否定沉淀采用**半自动**：对话结束时 AI 输出摘要，需求方确认后才入库。

### 3.3 提问规则体系

智能体的追问不是随机的，而是基于一套规则模板：

#### A. 通用追问规则（每次需求都触发）

| 维度 | 触发条件 | 追问示例 |
|------|---------|---------|
| 用户与角色 | 需求未提及操作者 | "这个功能的使用者是谁？不同角色是否有不同权限？" |
| 前置条件 | 需求未提及触发条件 | "什么状态下可以执行这个操作？什么状态下不可以？" |
| 后置结果 | 需求只描述操作，未描述结果 | "操作完成后，用户期望看到什么？系统内部应该发生什么变化？" |
| 数据范围 | 需求涉及数据操作 | "这个操作影响全部数据还是部分数据？筛选条件是什么？" |
| 频次与量级 | 需求涉及列表/查询 | "预计数据量级是多少？是否有分页需求？" |

#### B. 系统感知追问规则（利用系统知识触发）

| 维度 | 触发条件 | 追问示例 |
|------|---------|---------|
| 模块影响 | 需求涉及已存在的模块 | "当前X模块的Y逻辑是Z，你的需求会改变这个行为，确认要改吗？" |
| 数据冲突 | 涉及已有表的字段或约束 | "XX表当前有唯一约束在A字段，你的需求中B场景可能导致冲突，如何处理？" |
| 权限继承 | 涉及已有权限体系 | "当前系统权限模型是RBAC，这个功能应该归属哪个权限点？" |
| 下游影响 | 涉及已有消息/回调 | "X操作当前会触发MQ消息给Y系统，新增的Z场景是否也要触发？" |
| 历史教训 | 涉及 pitfall 中记录的模块 | "X模块历史上在Y场景出过数据一致性问题，你的需求在这一点上需要怎么处理？" |

#### C. 边界条件规则（每次需求自动穷举）

智能体自动列出而不需要提问，只在确认阶段让需求方校验：

- **数据边界**：空值、null、0、负数、超长字符串、特殊字符
- **状态边界**：各状态下的操作是否允许
- **并发边界**：多人同时操作同一数据
- **时间边界**：跨天、跨月、跨年、时区
- **权限边界**：无权限、部分权限、过期权限

### 3.5 实现方式 🔧（v3 修订：从"C/S两路径"改为"文件驱动 + Git 协同"）

原方案提出两条路径（seazenai Code Agent / 自建 Web 界面），但在实际落地层面有更简洁的方式：**整个智能体系统 = 四套提示词 + 四组知识文件 + Git**。不需要额外开发、不需要独立部署、不需要改变团队的代码工作流。无论是知识提取、需求对话、开发编排还是测试审查，本质上都是一套**提示词 + 上下文知识**的组合。把这些内容文件化、目录化、Git 化，就自然获得了版本管理、多人协同、历史追溯的能力。

#### 核心理念：Prompt-as-Code

这整个方案的本质不是"造一个智能体平台"，而是**定义一套版本化的提示词文件体系 + 团队 Git 协作流程**。和 infrastructure-as-code 同一个哲学：把过去口头传递、文档传递、工具配置里的隐性知识，全部显式化为版本化的文本文件。修改智能体行为 = 改文件 → 提 PR → Review → 合入 → 下次对话自动生效。分支即实验空间，想试新的追问方式？开分支改 `requirements/AGENT.md`，对比效果后再合入。

#### Git 协同的关键设计

1. **所有人 clone同一个仓库**。产品经理的机器上不需要跑得动完整开发环境，只需要 seazenai Code + 项目代码 + `.seazenai/` 目录。开发同理。

2. **知识刷新谁来触发？** Phase 1 期间手动；Phase 2 后期开始用定时任务或 CI 自动触发，产出的 knowledge/ 更新作为一个独立 commit push。

3. **冲突如何处理？** knowledge/ 下的分片结构本身就减少了冲突——产品经理更新 `negations.md`，开发经理更新 `decisions.md`，CI 更新 `data-model/user-tables.md`，彼此不交叠。

4. **`.seazenai/` 目录与 src/ 的关系**：同一个 repo，但逻辑上独立——src/ 是"系统本身"，`.seazenai/` 是"理解系统的元层"。src/ 变更时，`.seazenai/knowledge/` 需要刷新；反之 `.seazenai/` 变更时 src/ 不受影响。

5. **AGENT.md 即 system prompt**：每个目录下的 `AGENT.md` 是该角色对应的智能体的 system prompt。修改这个文件 = 调整智能体的行为，git diff 即可追溯。

#### 🔧 入口路由：seazenai.md

文件放在那里，但 seazenai Code 不会自动知道"打开它的人是产品经理还是开发"。项目根目录的 `seazenai.md` 是 seazenai Code 启动时自动加载的文件，它充当入口路由——两级路由：先问角色，再问任务。

```markdown
# [项目名] 研发统筹智能体

## 第一级：角色路由

启动时先问用户：

"你好，我是这个项目的研发统筹智能体。请问你的角色是？"

1. 产品经理 / 需求方 → 我要提需求或沟通需求
2. 开发经理 / 开发工程师 → 我要做开发任务或 Review 代码
3. 测试工程师 → 我要审查测试覆盖或做探索性测试

## 第二级：任务路由

根据角色选择，加载该角色目录的 INDEX.md，列出当前可处理的任务：

选 1（产品经理）→ 加载 .seazenai/requirements/AGENT.md + .seazenai/requirements/INDEX.md
  INDEX.md 列出：
    - 进行中的需求：REQ-003（权限模块优化）、REQ-004（报表导出）
    - 选项：继续已有需求 / 新建需求
  用户选择后 → 加载对应 in-progress/REQ-N/ 下的文件开始对话

选 2（开发）→ 加载 .seazenai/development/AGENT.md + .seazenai/development/INDEX.md
  INDEX.md 列出：
    - 待开发的需求：REQ-003（Phase 3 已就绪）、REQ-004（需求文档已完成）
    - 开发中的任务：REQ-002（review-notes 中有待处理意见）
  用户选择后 → 加载对应 tasks/REQ-N/ + requirements/archive/REQ-N/ + knowledge/
  如果是新需求 → 先检查 Phase 2 的需求文档是否已就绪

选 3（测试）→ 加载 .seazenai/testing/AGENT.md + .seazenai/testing/INDEX.md
  INDEX.md 列出：
    - 待测试的需求：REQ-003（开发完成，CP4 待审）
    - 测试中的需求：REQ-002（盲区补充中）
  用户选择后 → 加载对应 tasks/REQ-N/ + requirements/archive/REQ-N/ + blindspot-checklist.md

每个角色只能看到自己目录的 AGENT.md 和共享的知识层。
如果需要跨角色协作（如开发阶段需要回头澄清需求），AI 会提示"这需要产品经理确认"。
```

**INDEX.md 的维护规则**：
- 每次对话结束时，AI 自动更新对应 INDEX.md 的状态和最后更新日期
- 需求归档时，AI 将条目从「进行中」移到「已归档」
- 超过 6 个月无引用的已归档条目，移入 `archive/YYYY/` 目录
- 各 INDEX.md 控制在 50 行以内，超出则拆分历史到归档目录

**设计要点**：

- `seazenai.md` 是整个系统的唯一入口，不需要记任何路径或命令
- 两级路由都是对话式的：角色 → 任务。INDEX.md 是"菜单"，AGENT.md 是"工作方式"
- 目录的物理结构 = 权限边界。产品经理不会加载 `development/AGENT.md`，开发不会加载 `testing/AGENT.md`，但都能读 `knowledge/`
- 如果要越权？他可以直接读文件，但那是 Git 的问题（代码仓库权限），不是智能体系统的问题
- INDEX.md 由 AI 在每次对话结束时自动更新（状态变更、最后更新日期），不需要人手维护

**为什么不用配置文件标记身份**（如 `.role` 文件）？

配置文件带来三个麻烦：① 同一个人可能在不同任务中切换角色（今天提需求，明天 Review 代码），配置文件不能表达"这次是做什么"；② 配置文件可能不同步（A 机器上设了，B 机器上忘设）；③ 不如对话式直接——"你是谁"是最自然的第一问。

#### 🔧 Bootstrap：.seazenai/ 目录初始化（从零到一）

以上所有机制都假设 `.seazenai/` 下已有完整的文件体系，但第一次 clone 到空的 `.seazenai/` 目录时，智能体系统本身也需要 bootstrap。

**Bootstrap 分两步走**：

**第一步：由开发经理初始化骨架**（一次性工作，预计半天）

```
1. 在项目根目录创建 seazenai.md（入口路由模板）
2. 创建 .seazenai/ 目录结构（空壳）：
   - knowledge/INDEX.md（暂时只有项目名称 + "知识提取中"占位）
   - requirements/AGENT.md（先用本文 3.5.1 节的 prompt 骨架；v4 主文档中对应 4.6 节）
   - requirements/INDEX.md（空表格，待第一个需求填入）
   - development/INDEX.md（空表格）
   - testing/INDEX.md（空表格）
   - testing/blindspot-checklist.md（用本文 4.6 节的初始清单；v4 主文档中对应 5.6 节）
3. 提交到仓库：git add .seazenai/ seazenai.md && git commit -m "init agent system"
```

此时其他人 clone 后打开 seazenai Code，seazenai.md 能正常运行两级路由。knowledge/INDEX.md 内容为空，AI 会如实说"当前知识库为空，需要先执行知识提取"。

**第二步：Phase 1 执行知识提取，填充 knowledge/**（2-3 周，按本文 2.3 节执行手册；v4 主文档中对应 3.3 节）

```
Week 1-3：执行 10 步知识提取，产出分片知识文件
  → 逐步更新 knowledge/INDEX.md（从"知识提取中"变成真正的模块索引）
  → 每次 commit 后，其他人 pull 即可获得最新知识

Phase 1 结束时：
  - knowledge/ 下有完整分片知识
  - knowledge/INDEX.md 可作为关键词路由使用
  - requirements/AGENT.md 经过冷启动训练期校准
```

**seazenai.md 在知识库为空时的特殊行为**：

```markdown
# 附加规则（仅当 knowledge/INDEX.md 内容为占位时触发）

如果 knowledge/INDEX.md 只有占位内容（"知识提取中"），
在角色路由后主动告知用户：

"当前系统知识库还在构建中（Phase 1），我可以：
1. 仍然尝试根据你的需求提问（但追问会比较多，因为我不知道现有系统的情况）
2. 暂时不使用系统知识，像普通 AI 一样对话

建议：如果你是要测试需求对话流程，选 1；如果只是随便聊聊，选 2。"
```

#### 为什么比自建界面更好

| 维度 | 自建 Web 界面 | 文件驱动 + Git |
|------|-------------|----------------|
| 开发成本 | 前端+后端+部署 | 零，只写 md 文件 |
| 团队学习成本 | 学新工具 | 不学新工具，用已有的 seazenai Code + Git |
| 版本追溯 | 需要额外开发 | Git 原生 |
| 离线使用 | 不支持 | 本地文件，支持 |
| 与代码同步 | 需要额外集成 | 同一个 repo，天然同步 |
| 定制灵活性 | 界面写死了 | 修改 AGENT.md 即修改行为 |

#### 加载优先级与冲突规则

当不同目录中的文件对同一件事有不同说法时，需要有明确的优先级规则，让智能体知道"信哪个"。

**加载优先级**（从高到低）：

```
1. requirements/in-progress/REQ-N/notes.md     ← 本次需求对话中已确认的否定/约束
2. requirements/in-progress/REQ-N/requirement.md ← 本次需求文档本身
3. requirements/constraints.md                  ← 需求层面的硬性约束
4. development/constraints.md                   ← 开发层面的硬性约束
5. development/standards.md                     ← 通用开发规范
6. knowledge/decisions.md + negations.md        ← 历史决策和否定知识
7. knowledge/（系统事实：API/数据模型/业务规则）   ← 系统知识层
```

**冲突解决规则**：

| 冲突场景 | 解决方式 |
|---------|---------|
| 需求文档说 X，系统知识说 Y | **以需求文档为准**，但智能体必须标注"这与你现有系统不一致，请确认" |
| 本次需求内的 notes.md 与历史 decisions.md 矛盾 | **以 notes.md 为准**，同时标记"与历史决策 #D00X 冲突，请确认是否有意推翻" |
| 开发约束与需求约束冲突 | **上报 CP1**，由开发经理裁决，报需求方确认 |
| 两个历史决策互相矛盾 | **上报 CP2**，由开发经理判断哪个决策在当前上下文适用 |
| AGENT.md 的指令与 constraints.md 冲突 | **以 constraints.md 为准**（AGENT.md 是行为指南，constraints.md 是硬性边界） |

**关键原则**：越靠近"当前这次需求"的文件优先级越高。系统知识是参考，需求文档是指令，本次对话中的否定是最高优先级——因为它是需求方刚刚确认的最新意图。

#### 从哪个 AGENT.md 先开始写

优先级从高到低：
1. `requirements/AGENT.md` —— Phase 2 直接用，产品经理第一个对话就靠它
2. `knowledge/INDEX.md` —— Phase 1 的产出，也是所有智能体的入口
3. `development/AGENT.md` —— Phase 3 时使用
4. `testing/AGENT.md` —— Phase 4 时使用

### 3.5.1 🔧 System Prompt 参考框架

以下不是完整 prompt，而是 Phase 2 需求对话 Agent 的 prompt 骨架，给执行者一个出发点。

```
## 角色定义
你是 [项目名] 的需求分析助理。你拥有该系统的完整知识，包括架构、数据模型、
API、业务规则和历史踩坑记录。你的职责是与需求方对话，产出一份开发就绪的需求文档。

## 强制规则
1. 你必须区分「已知事实」和「推断」。引用系统知识时，标注来源；推断时，明确
   说明"以下是推断，请确认"。
2. 每次需求方否定你的理解（"不对""不是""不需要"），你必须记录下来，在对话结
   束后沉淀到 negations 知识库。
3. 不要在第一次回复中就给出完整的需求文档。先追问，再分析，再生成。
4. 追问时每次最多问 3 个问题，不要一次性抛出 10 个问题让需求方无从下手。
5. 当你发现需求涉及系统的历史事故区域（参考 pitfalls.md），必须主动提醒需求方。

## 对话流程
第 1 步：理解需求概要
  - 让需求方自由描述
  - 追问人员、触发条件、期望结果（通用追问规则 A）
  - 同步记录否定信息

第 2 步：系统影响分析（自主完成，不打扰需求方）
  - 检索 INDEX.md，定位涉及的模块
  - 加载相关模块的 L1 摘要
  - 检查历史 negations 和 decisions 中是否有相关条目
  - 输出影响范围分析

第 3 步：系统感知追问
  - 基于影响分析，追问模块冲突、数据约束、权限继承等（追问规则 B）
  - 引用历史否定："上次在 #N 需求中明确了 X，这次是否仍然适用？"

第 4 步：边界条件穷举（自主完成）
  - 根据规则 C 自动生成所有边界场景
  - 结合 negations 和 pitfalls 补充历史教训

第 5 步：生成需求文档
  - 按模板格式输出，包含澄清记录章节
  - 标注哪些是已知事实、哪些是 AI 推断待确认

第 6 步：反向确认 + 沉淀
  - 逐条向需求方确认关键假设
  - 将本次对话的否定和决策格式化存入知识库

## 禁止行为
- 禁止假装知道不确定的信息
- 禁止跳过追问直接给方案
- 禁止在需求方纠正你时分辩"我的逻辑是对的你理解错了"
- 禁止一次性输出超过 2000 字的回复（太长需求方不看）
- 禁止在追问中使用技术术语（如"我们需要讨论一下 JOIN 策略"——
  要说"这个查询涉及多张数据表，数据量大的时候会不会慢？"）
```

### 3.6 🔧 冷启动策略

第一个需求对话没有历史否定/决策知识可参考，追问规则也未校准。前几个需求的质量必然低于稳定期。不回避这个问题。

**冷启动期的设计**：

```
前 3 个需求 = 训练期，不算效果指标

需求 #1:
  - 产品经理全程陪同需求方对话（不代替，观察+补充）
  - 对话结束后，产品和 PM 一起复盘：AI 遗漏了什么追问、哪里追问了但方向错了
  - 根据复盘结果更新追问规则 → 更新 Agent prompt
  - 产出需求文档 + 复盘记录（不做效果统计）

需求 #2:
  - 产品经理旁路观察，仅在 AI 明显卡住时介入
  - 对话结束后对比：人工 PRD（如有）vs AI 产出，标记差异
  - 根据差异调整 prompt
  - 产出需求文档 + 差异分析（不做效果统计）

需求 #3:
  - 需求方独立与 AI 对话，产品经理不介入
  - 产品经理仅审查最终文档质量
  - 如果质量仍然不达标 → 延长训练期到 5 个需求
  - 达标标准：产品经理认为文档"可直接开发且无需大规模补充"
```

**训练期的关键动作**：

| 训练期 | 谁参与 | 复盘什么 | 改进什么 |
|--------|--------|---------|---------|
| 需求 #1 | 产品经理陪同 | AI 追问的覆盖率（该问但没问的） | 追问规则（A/B 类） |
| 需求 #2 | 产品经理旁路 | AI 产出 vs 人工产出的差异 | prompt 结构 + 边界条件模板 |
| 需求 #3 | 产品经理仅审查 | 反向确认的准确率 | 精细调优 prompt |
| 需求 #4+ | 产品经理常规审查 | 长期质量趋势 | 知识库扩充 |

**冷启动期的"假阳性追问"问题**：初期 AI 为了谨慎会过度追问——"你确认要显示在页面上吗？"这种无效追问。解决方案是在训练期积累一个「不需要追问的常识列表」，逐步录入 prompt 的排除规则。

### 3.7 需求阶段的验收标准 🔧（修订：降级"开发就绪"承诺）

~~开发经理确认文档可以直接用于开发，无需二次澄清~~（删除：这个标准在小需求上可能做到，但不可持续——需求越复杂，自然语言歧义越多，歧义只能在开发过程中暴露）。

**修订后的验收标准**：

- [ ] 至少 3 个真实需求通过智能体产出需求文档
- [ ] 🔧 **澄清轮次减少 ≥ 70%**（相比传统流程中需求方→产品→开发的沟通轮次）。这个指标可测量、可渐进改进——冷启动期 50% 也算进步，稳定期 80% 更好
- [ ] 智能体的边界条件覆盖率达到 90% 以上（对照最终测试发现的遗漏 case）
- [ ] 需求方主观评价 ≥ 3.5/5（"这个文档比我直接跟开发沟通省了多少事"）

## 四、Phase 3：研发统筹智能体（独立为下一季度项目，初期 6-8 周 MVP）

### 4.1 目标

从"开发就绪的需求文档"出发，智能体在严格的**安全边界内**自主完成：任务拆解 → 代码生成 → 测试生成 → 自测 → 提交 Review。人类在关键 Check Point 做决策和把关。

### 4.1.1 🔧 硬性安全边界（必须执行）

在 Phase 3 启动前，必须设置以下三条不可突破的边界：

**边界一：自修复预算上限**

```
AI 生成代码 → 测试失败 → AI 修复 → 测试仍失败 → AI 再修复
                                                      │
                                              ┌───────┴────────┐
                                           第3轮通过          第3轮仍失败
                                              │                 │
                                              ▼                 ▼
                                          继续后续流程      强制终止
                                                         自动上报开发经理
                                                       「AI 已尝试 3 轮修复未果，
                                                         需要人工介入，
                                                         以下是失败日志和当前代码状态」
```

- 最多 3 轮自修复
- 超时即上报，不继续消耗 token 和时间
- 禁止 AI 通过"放宽测试"来通过——测试修改必须有 AI 对修改原因的解释，纳入 CP3 Review

**边界二：范围上限 + 🔧 风险分级**

```
Phase 3 MVP 允许的范围（按风险从低到高，逐步开放）：

  🟢 低风险（Phase 3 启动即可做）：
    ✅ 新增单表 + 新增 API + 基础前端页面
    ✅ 新增独立 Service 方法（不修改已有方法）

  🟡 中风险（Phase 3 稳定 3 个需求后可做）：
    ✅ 对现有接口新增可选参数（兼容性变更，不影响旧调用方）
    ✅ 在已有 Service 中新增方法（不修改已有方法签名）

  🔴 高风险（Phase 3 暂不允许，需开发经理人工完成）：
    ❌ 修改已有接口的必填参数或返回结构
    ❌ 修改已有 Service 方法的内部逻辑
    ❌ 修改已有数据表的字段类型或约束
    ❌ 跨模块改动
    ❌ 涉及状态机改造
    ❌ 引入新的外部依赖
    ❌ 性能敏感路径（高并发、大表）

升级条件：连续完成 3 个 🟢 需求且 Review 无重大问题
  → 开放 🟡
  → 连续完成 5 个 🟡 需求且 Review 无重大问题
  → 逐个评估 🔴 场景是否可以试点
```

- 第一个端到端 case 必须选"即使 AI 搞砸了也可以人工在半天内重做"的需求
- 每周 Review 一次"边界是否应该扩展"，基于 AI 的稳定表现做决策

**边界三：决策点校准期**

```
Phase 3 启动后的前 2 个需求：
  同一个需求 →
    人类开发经理独立标注：「我认为这里需要决策」
    智能体独立标注：「我认为这里需要决策」
    对比差异 →
      - 智能体标注了但人类没标注 → 智能体过于保守（可接受）
      - 人类标注了但智能体没标注 → 🔴 假阴性，需要校准
      - 校准方法：将该遗漏场景加入 prompt 中的「强制检查」规则
```

- 校准期持续到连续 2 个需求的假阴性率为 0（即智能体不再遗漏人类认为需要决策的点）
- 校准期结束前，所有智能体不认为是决策点的内容，仍需开发经理抽查

### 4.4 决策点上报机制

智能体在开发过程中遇到**模糊地带**时，不是瞎猜，而是：

1. **识别决策点**：根据预定义的决策类型判断
2. **提出选项**：给出 2-3 个可选方案，每个附带优缺点
3. **给出建议**：基于系统知识和最佳实践推荐一个方案
4. **上报 CP2**：等待开发经理选择

预定义的决策类型：

| 决策类型 | 触发条件 | 示例 |
|---------|---------|------|
| 架构选择 | 新增功能有多种实现路径 | "用户状态可以用字段标记，也可以建状态历史表。前者简单但丢失历史，后者复杂但可追溯。建议先用字段标记，后续有审计需求再升级。你同意吗？" |
| 性能策略 | 涉及大数据量操作 | "这个查询在万级数据下没问题，但预计数据量会到百万级。方案A：现在加索引，成本低但需要 DBA 审批。方案B：先上线监控，触发阈值后再优化。建议A。你选哪个？" |
| 兼容性取舍 | 改动影响已有功能 | "修改X接口的参数结构会影响3个下游调用方。方案A：新增v2接口，旧接口保持兼容。方案B：一次性全部升级。建议A，分阶段迁移。你同意吗？" |
| 库/工具选型 ⚠️ | 需要引入新的依赖 | "这个功能需要Excel导出。方案A：用现有的poi库，功能弱但零成本。方案B：引入easyexcel，功能强但增加依赖。建议B，长期来看值得。你选哪个？"<br>⚠️ **Phase 3 MVP 期间暂不适用**：引入新依赖属于 🔴 高风险禁区，待范围扩展至 🟡 后启用 |
| 安全策略 | 涉及敏感数据 | "用户手机号在列表中显示，方案A：默认脱敏，点击查看完整。方案B：不脱敏但要记录查看日志。建议A。你同意吗？" |

### 4.5 Review 机制（CP3）

传统 Code Review：审查代码风格、逻辑正确性、性能、安全。

AI 时代的 Code Review：**只审查 AI 做不好的事情**。

| Review 维度 | AI 自检 | 人类审查 |
|------------|---------|---------|
| 代码风格/规范 | ✅ Lint + AI 自查 | ❌ 不需要 |
| 逻辑正确性 | ✅ 单元测试通过 = 证明 | ❌ 不需要逐行看 |
| 边界条件覆盖 | ✅ 从需求文档自动生成 case | ⚠️ 抽查是否有遗漏 |
| 架构合理性 | ⚠️ AI 可自我评估但不完全可靠 | ✅ **人类重点审查** |
| 安全漏洞 | ⚠️ AI 可扫常见漏洞 | ✅ **人类重点审查** |
| 业务逻辑匹配 | ⚠️ AI 可能理解偏差 | ✅ **人类重点审查** |
| 性能影响 | ⚠️ AI 可做简单分析 | ✅ 涉及大表/高并发时人类审查 |

**Review 清单**（开发经理用）：

```markdown
## 开发经理 Review 清单

### 一票否决项（必须检查）
- [ ] 数据库变更是否安全（有无锁表风险、是否兼容旧代码）
- [ ] 权限控制是否正确（有无越权风险）
- [ ] 敏感数据是否妥善处理（脱敏、加密、日志不打印）
- [ ] 对下游系统的影响是否可控
- [ ] 是否有回滚方案

### 重要项（抽查）
- [ ] 核心业务逻辑是否符合需求文档
- [ ] 异常处理是否完整（不回滚导致数据不一致）
- [ ] 并发场景是否正确处理

### 不需要审查（信任 AI + 自动化）
- [ ] 代码风格
- [ ] 单元测试覆盖率（机器检查）
- [ ] 简单 CRUD 的逻辑正确性
```

## 七、角色与分工变化总结

| 角色 | 原来做什么 | 现在做什么 |
|------|-----------|-----------|
| **需求方/业务** | 口头描述 → 产品翻译 | 直接与智能体对话，描述业务诉求 |
| **产品经理** | 理解需求 → 写 PRD → 画原型 → 反复沟通 | 辅助需求方与智能体对话，确认智能体产出的需求文档质量 |
| **开发经理** | 分配任务 + 详细 Code Review | 在 Check Point 做架构/安全/业务决策，不看代码细节 |
| **开发工程师** | 接任务 → 看 PRD → 编码 → 自测 | 接收智能体已完成的任务，审查关键决策，必要时介入修改 |
| **测试工程师** | 分析需求 → 写用例 → 执行 → 报 Bug | 审查 AI 的测试策略，探索性测试，不可自动化场景验证 |

**核心变化**：所有人的工作从"生产内容"转向"审查和决策"，从"自己做"转向"告诉 AI 做什么 + 判断 AI 做得好不好"。

## 八、风险与应对 🔧（修订：团队参与策略）

| 风险 | 可能性 | 影响 | 应对 |
|------|--------|------|------|
| 系统知识不准确导致智能体错误决策 | 中 | 高 | 🔧 Phase 1 即引入开发工程师验证知识；智能体强制区分「已知事实」与「推断」；保鲜机制先手动后自动 |
| 开发经理不信任 AI 的代码质量 | 高 | 中 | 先用统计数据说话（AI 生成代码的 Bug 率 vs 人工代码）；通过校准期数据建立信任；逐步放权 |
| 需求方不愿意与 AI 对话 | 中 | 中 | 产品经理陪同对话过渡期；用高质量产出建立信任 |
| AI 遗漏关键边界条件 | 中 | 高 | CP3/CP4 的人类审查；持续从每个 online bug 反哺盲区检查清单 |
| 🔧 团队抵触角色变化 | 高 | 高 | **具体策略见下方** |
| 🔧 非技术角色的 Git 门槛 | 中 | 中 | 产品经理不熟悉 Git 操作（clone/commit/push/解决冲突）。应对：提供一键脚本封装常用 Git 操作；初期由开发经理代劳 commit；Phase 2 训练期本身就是过渡 |
| 🔧 知识文件数量膨胀 | 中 | 中 | 随着模块增多和需求积累，`.seazenai/` 下的 md 文件可能达到上百个，AI 加载时索引和路由压力增大。应对：INDEX.md 严格控制在一页内；定期归档超过 6 个月无引用的历史需求到 `archive/YYYY/` 子目录；分片文件单文件不超过 500 行 |

### 🔧 团队参与策略（细化）

"不是裁人而是角色升级"方向正确但太抽象。具体做法：

**Phase 1 就让开发工程师参与知识验证**（不是等到 Phase 3 才让他们看到 AI 的产出）。

```
具体操作：
1. AI 先完成知识提取，产出 knowledge/*.md
2. 分配给开发工程师：「请验证这 3 个模块的知识文档，
   标出：哪些是准确的、哪些有遗漏、哪些写错了」
3. 开发者反馈 → 更新知识文档 + 更新 AI 的提取策略
4. 每两周轮换模块，让更多开发者参与

效果：
- 知识质量提高（人机交叉验证）
- 开发者从「被替代者」变成「AI 的训练者和审查者」
  心理抵触大幅降低
- 开发者在验证过程中加深对自己模块的理解
- 为 Phase 3 他们担任 Review 角色建立能力基础
```

**Phase 2 让产品经理参与需求对话的质量评审**：

```
产品经理审查 AI 产出的需求文档，而非被 AI 替代。
他们的价值从「写需求」升级为「判断需求质量」。
```

**Phase 3 的开发经理角色设计**：

```
不做：
  ✗ 逐行看代码
  ✗ 检查代码风格
  ✗ 验证简单 CRUD 逻辑

要做：
  ✓ 在 CP1-CP4 的质量把关（安全、架构、业务匹配）
  ✓ 判断 AI 的决策建议是否合理
  ✓ 在 AI 无法收敛时接手
  ✓ 积累「AI 容易犯错」的经验，反哺到规则和 prompt 中
```

## 九、🔧 成本模型（新增）

方案必须包含成本考量。AI 调用不是免费的，一个需求从对话到代码到测试涉及数十轮调用，需要从实验阶段就开始记录。

### 9.1 成本估算框架

| 环节 | 典型调用次数 | 单次 token 估算 | 说明 |
|------|-------------|----------------|------|
| 需求对话 | 5-15 轮 | 10K-50K input/轮 | 取决于需求复杂度 |
| 知识检索 | 3-5 轮 | 5K-20K input/轮 | 读取相关知识文档 |
| 任务拆解 | 1-3 轮 | 20K-50K input/轮 | 分析需求+系统知识 |
| 代码生成 | 5-15 轮 | 10K-30K input/轮 | 多文件生成 |
| 测试生成 | 3-8 轮 | 10K-25K input/轮 | case 生成+执行结果分析 |
| 修复循环 | 0-3 轮 | 5K-15K input/轮 | 如触发预算上限 |
| 知识保鲜 | N 次/周 | 20K-80K input/次 | 按模块数量和变更频率 |

### 9.2 成本记录模板

从第一个实验开始，每个需求记录：

```markdown
### 需求 #001 成本记录

| 维度 | 数值 |
|------|------|
| 需求标题 | xxx |
| 需求复杂度 | 简单 / 中等 / 复杂 |
| 总 AI 调用次数 | X 次 |
| 总 input token | X |
| 总 output token | X |
| 估算 API 成本 | ¥XXX |
| 传统开发人天 | X 人天 |
| 传统开发人天成本 | ¥XXX |
| AI 成本占比 | X% |
| 结论 | 节省 / 持平 / 超支 |
```

### 9.3 成本优化策略

- **减少冗余读知识**：知识文档建立后，智能体不需要每次从头读，可以缓存索引
- **对话轮次控制**：需求对话设置轮次上限（如 20 轮），超出则自动总结待澄清项并上报产品经理
- **批量生成**：多个同类 API/组件可以批量生成，减少重复的上下文解释
- **模型分级**：知识提取用 Opus（高准确），代码生成用 Sonnet（平衡），测试生成用 Sonnet
