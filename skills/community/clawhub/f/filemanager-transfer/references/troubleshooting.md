# 故障排查

先看脚本返回的具体错误。修复最可能的原因后，重试一次。

## 服务未启动

表现：

- 连接被拒绝。
- 请求超时。
- 无法访问 FileManager API。

处理：

1. 从 `scripts/` 启动当前系统对应的可执行程序；见 `install-and-start.md`。
2. 确认 `FILEMANAGER_BASE_URL` 指向可访问的主机和端口。
3. 如果用户和 agent 不在同一台机器，除非服务就在运行脚本的机器上，否则不要使用 `localhost`。

## 鉴权失败

表现：

- HTTP `401`
- HTTP `403`

处理：

1. 检查 `scripts/.env` 或环境变量中是否存在 `FILEMANAGER_APPKEY`。
2. 确认 AppKey 有效，并拥有上传、分享或下载权限。
3. 排查时不要打印 AppKey。

## 上传失败

表现：

- 本地文件不存在。
- 远程目录不存在。
- HTTP `400`, `404`, or `413`

处理：

1. 确认本地路径是普通文件。
2. 使用已存在的远程目录，通常可先用 `/`。
3. 如果是 HTTP `413`，减小文件体积，或用更大的 `-max-upload-size` 重启 FileManager。
4. 如果是创建分享失败，提供非空 `--password`。

## 下载失败

表现：

- 文件 ID 不存在。
- 输出路径无法写入。

处理：

1. 确认 FileManager 文件 ID。
2. 确认该 ID 指向文件，而不是目录。
3. 使用 agent 有写入权限的输出目录或文件路径。

## 脚本仍然失败

如果 FileManager 可以访问，但封装脚本仍失败：

1. 保留可见错误信息。
2. 用最小测试文件或已知有效文件 ID 重试。
3. 按 `transfer-script.md` 中记录的 API 行为兜底。
4. 告诉用户哪些步骤成功、哪些失败，以及文件当前是通过链接、文件 ID 还是本地路径可用。
