# 版本治理策略

> 如何管理 qa-team-skills 的版本升级、通知和回滚。

## 版本号规范

采用语义化版本（SemVer）：`主版本.次版本.修订`

| 位 | 变化条件 | 示例 |
|----|----------|------|
| 主版本 | 架构变更、指令拆分/合并、不向前兼容 | V1→V2：将 6 指令拆为独立 Skill |
| 次版本 | 新增能力、扩展维度、新增指令 | V1.0→V1.1：新增 7 项 /qa-team 子能力 |
| 修订 | Bug 修复、Prompt 措辞优化、数据更新 | V1.0.1：注入防护、自检清单、数据修正 |

## 升级频率

| 类型 | 频率 | 说明 |
|------|------|------|
| 修订 | 随时 | 静默升级，通过 CHANGELOG 记录 |
| 次版本 | 按需（预计每月 1-2 次） | 团队周知后升级 |
| 主版本 | 慎重（预计半年以上 1 次） | 全员培训后升级 |

## 升级流程

### 次版本/修订升级
```
1. 开发者在 feat 分支完成修改
2. 更新 VERSION 文件 + SKILL.md 版本号 + CHANGELOG.md
3. CI 校验通过
4. 合并到 release/v1 分支
5. 在团队沟通渠道发布升级通知（附 CHANGELOG 摘要）
6. 团队成员执行 git pull 或重新 cp 技能目录
```

### 主版本升级
```
1-5. 同上
6. 组织全员培训（演示新功能、变更点）
7. 提供"变更前后对比示例"
8. 团队成员分批升级，第一批灰度为 2-3 人
9. 灰度无问题后全员升级
```

## 升级通知模板

每次升级后在团队沟通渠道（企业微信/Slack/钉钉）发布：

```
【qa-team-skills 升级通知】
版本：v1.0.0 → v1.0.1
日期：2026-06-22
类型：修订（向前兼容）

变更摘要：
✅ 新增：6 个指令增加注入防护和自检清单
✅ 新增：/qa-agent 扩展至 16 维度（含 RAG）
✅ 优化：/qa-bug 批量模式增加缺陷关联分析
⚠️ 注意：/qa-case 输出增加了"设计方法"必填字段

完整变更：docs/CHANGELOG.md
如有问题：@QA Team
```

## 回滚机制

| 场景 | 操作 |
|------|------|
| 新版本 Prompt 输出质量明显下降 | `git checkout v1.0.0` 回到上一个 tag |
| 新版本与旧用例评审标准冲突 | 项目级部署可锁定版本，不跟随全局升级 |
| 个别项目需要稳定版本 | 在项目 `./.claude/skills/` 下部署指定版本，不跟随全局 |

## 历史版本归档

所有版本通过 git tag 归档：

```bash
git tag -l
# v1.0.0
# v1.0.1
```

## 版本兼容承诺

- 修订升级：100% 向前兼容（输出格式不变）
- 次版本升级：≥95% 向前兼容（新增字段，不删除旧字段）
- 主版本升级：不做兼容承诺，需全员培训
