# /xcheck — 多模型交叉验证

> **作用**: 用另一个 AI 模型对当前方案/代码进行二次审查，
> 发现单一模型容易忽略的问题。
>
> 原始 gstack 用 OpenAI Codex 做二次意见，
> 我们适配为：用已配置的其他国产模型交叉验证。

## 核心理念

**两个模型犯同样错误的概率 << 一个模型犯错的概率**

就像代码审查需要另一个人一样，AI 方案也需要另一个 AI 来验证。

## 可用的模型矩阵（IPO 项目）

| 角色 | 主模型 | 交叉验证模型 |
|------|--------|-------------|
| 编译/摘要 | Kimi K2.6 | Qwen-Max |
| 生成 | DeepSeek V4 | GLM-5.1 |
| 校验 | Qwen-Max | DeepSeek V4 |
| Embedding | Qwen text-embedding-v3 | N/A |

## xcheck 流程

### 何时触发
1. **关键决策点** — 架构选型、技术方案确定前
2. **复杂实现** — 超过200行的核心逻辑完成后
3. **发布前** — Ship 阶段前的最终检查
4. **争议时** — 用户对方案有疑虑时

### 执行流程
```
当前方案/代码
     │
     ├──→ 发送给主模型: "请审查这个方案，找出问题"
     │         ↓
     │      主模型反馈: [问题列表A]
     │
     └──→ 发送给交叉模型: "请审查同一个方案，独立判断"
              ↓
           交叉模型反馈: [问题列表B]
     
     ↓ 对比合并
  ┌──────────────────────┐
  │ 两个模型都指出的 → P0 必须修 │
  │ 只有主模型指出  → P1 建议修 │  
  │ 只有交叉模型指出 → P1 建议修 │
  │ 两个都没指出   → 通过       │
  └──────────────────────┘
```

### Prompt 模板（发给交叉模型的）

```markdown
你是一位独立的代码/方案审查员。请审查以下内容，找出所有问题。

**角色**: 你不是这个方案的编写者。你是来挑刺的。
**原则**: 宁可多报不要漏报。不确定的也标记出来。

## 待审查内容
[这里粘贴代码/方案]

## 审查重点
1. 逻辑错误（逻辑漏洞、边界遗漏）
2. 安全问题（注入、泄露、越权）
3. 性能隐患（N+1查询、内存泄漏、无限循环）
4. 可维护性（过度复杂、硬编码、命名不清）

## 输出格式
按严重程度列出发现：
- 🔴 必须修复: ...
- 🟡 建议改进: ...  
- ℹ️ 提醒注意: ...
```

## 使用示例

```
用户: /xcheck 这个搜索API的设计
```

AI 自动：
1. 取当前方案的摘要
2. 发给 DeepSeek（如果主模型是 Qwen）审查
3. 收集反馈并与自己的 Review 结果对比
4. 输出综合报告：

```markdown
# Cross-Check Report — [目标]

## 参与模型
| 模型 | 角色 | 主要发现数 |
|------|------|-----------|
| Qwen-Max | 主审 | 3个问题 |
| DeepSeek-V4 | 交叉审 | 4个问题 |

## 共识问题（两模型都指出）⚠️ 优先修复
1. **[问题]**: ...

## 主审独有问题
1. **[问题]**: ...

## 交叉独有问题
1. **[问题]**: ...

## 最终判定
- [ ] 通过交叉验证
- [ ] 有共识问题需修复后重验
```
