# Step 5.3 — 核心业务对象 + ER 模型（合并输出 v3.0）

在泳道图确认后，**AI 自主完成**核心业务对象识别、要素定义、对象关系、ER 图生成，**无需用户逐项确认**，一次性输出完整文件后继续流程。

---

## 什么是核心业务对象

> **核心业务对象** = 流程存在的根本理由，缺少它流程无法运转。

**判断标准（必须同时满足）**：
1. **必不可缺**：去掉它，整个流程失去意义
2. **有独立生命周期**：有创建→流转→归档的完整状态机
3. **被多个角色共同操作**：至少 2 个角色围绕它协作

**数量控制**：通常 2~3 个，最多不超过 4 个。

---

## 执行流程（全部 AI 自主完成，无卡点）

### 5.3.1 识别核心对象

从已确认的泳道图和状态机中扫描，**AI 直接判定**哪些是核心对象、哪些是辅助对象，不询问用户。

候选超过 4 个时，按「必不可缺性」排序取前 3 个。

---

### 5.3.2 定义每个核心对象的要素

每个核心对象输出一张要素表：

```markdown
## 业务对象：[对象名]

**一句话定义**：[这个对象是什么，在流程中的作用]
**关联状态机**：[引用 state-machine.md 中对应的状态机]

| 要素名   | 类型      | 来源        | 必填 | 备注          |
|----------|-----------|-------------|------|---------------|
| 模板名称 | string    | 🖊 手动输入  | ✅   | 20字以内       |
| 质检类型 | enum      | 📋 固定选项  | ✅   | 人工 / AI     |
| 关联部门 | reference | 🔗 从表获取  | —    | 多选          |
| 创建人   | reference | 🤖 系统自动  | —    | 当前登录用户   |
| 状态     | enum      | 🤖 系统自动  | —    | 草稿/启用/停用 |
```

**来源类型（5 选 1）**：
| 标签 | 含义 |
|------|------|
| 🖊 手动输入 | 用户自行填写 |
| 📋 固定选项 | 系统预设，不可自定义 |
| 🔗 从表获取 | 关联其他表动态加载 |
| 🤖 系统自动 | 系统生成，用户不可编辑 |
| 📥 上游传入 | 由前置流程步骤自动带入 |

### 5.3.3 对象间关系表

```markdown
## 对象关系

| 主对象   | 关系 | 关联对象 | 说明                    |
|----------|------|----------|-------------------------|
| 质检任务 | N:1  | 质检模板 | 每个任务基于一个模板创建 |
| 质检任务 | 1:N  | 质检结果 | 一个任务包含多条质检结果 |
```

---

### 5.3.4 后台不可见逻辑

> **后台逻辑** = 对用户界面不可见、但影响业务对象状态和数据的系统行为（自动分配、定时任务、状态自动流转、数据计算、跨对象联动等）。

**AI 根据业务场景自主推断**，不向用户询问确认。每条记录格式：

```markdown
## 后台逻辑

| 编号   | 逻辑类型 | 触发条件           | 执行内容                           | 影响范围      | 优先级 |
|--------|---------|--------------------|------------------------------------|--------------|--------|
| BL-001 | 自动分配 | 任务创建时          | 按技能组+工作量均衡分配给质检员     | 任务.负责人   | P0     |
| BL-002 | 定时任务 | 每日 00:00         | 扫描截止日已过的任务，标记「已超期」 | 任务.状态     | P0     |
| BL-003 | 状态联动 | 所有质检项均提交时  | 任务状态自动流转为「待审核」        | 任务.状态     | P1     |
```

### 5.3.5 ER 图（直接从对象和关系导出）

**实体** = 已定义的每个业务对象  
**属性** = 各对象要素表中的要素名 + 类型  
**关系** = 对象关系表中的关系行

输出 Mermaid `erDiagram`：

```
erDiagram
    INSPECTION_TEMPLATE ||--o{ INSPECTION_TASK : "适用于"
    INSPECTION_TASK ||--o{ INSPECTION_RESULT : "包含"
    EMPLOYEE ||--o{ INSPECTION_RESULT : "提交"
    INSPECTION_TEMPLATE {
        string id PK
        string name
        enum type
        enum status
    }
    INSPECTION_TASK {
        string id PK
        string templateId FK
        date deadline
        enum status
    }
    INSPECTION_RESULT {
        string id PK
        string taskId FK
        string inspectorId FK
        number score
        enum status
    }
```

## 输出文件

**将业务对象和 ER 图合并输出到单一文件 `03-business-objects-er.md`**，结构：

```
1. 核心对象清单（表格：对象名 / 一句话描述 / 为什么是核心）
2. 每个对象的要素表（含来源标注）
3. 对象间关系表
4. 后台不可见逻辑清单（BL-XXX）
5. ER 图（Mermaid erDiagram）
```

> **产品化改造专用规则**：当本文件由 `productization/SKILL.md` 调用时：
> - 核心对象清单只列**新增对象**和**有字段改造的现有对象**，纯沿用的现有对象不展开
> - 要素表只列**新增字段**和**改造字段**，不变字段不重复列出；改造字段用 `[改造]` 前缀标注变动说明
> - ER 图只体现新增对象/关系，现有不变的实体用虚线框标注「[现有，不变]」

生成后**直接进入 Step 5.5**，不等待用户确认此文件。

## 5.3.6 字段流转索引表（🆕 v3.5，供 Step 7 模块生成使用）

> 在 `03-business-objects-er.md` 生成完毕后，额外生成此表并保存为 `03.6-field-flow-index.md`。
> 此表极小（≤50行），在 Step 7 生成 PRD 第 9 节时作为**持久上下文锚点**全程挂载，替代重新载入完整文件。

### 输出格式

```markdown
# 字段流转索引表

| 字段 | 所属对象 | 产生于模块 | 被读取于模块 | 类型 | 是否跨模块关键字段 |
|------|---------|-----------|------------|------|-----------------|
| template_id | 质检模板 | M1 新建模板 | M2 创建任务 / M3 执行质检 | FK(string) | ✅ |
| task.agent_ids | 质检任务 | M2 分配坐席 | M3 我的任务列表 | array(FK) | ✅ |
| item.call_record_id | 质检项 | M2 任务生成 | M3 质检执行 | FK(string) | ✅ |
| result.score | 质检结果 | M3 提交质检 | M4 审核详情 / M5 统计看板 | number | ✅ |
| result.status | 质检结果 | M3 提交质检 | M4 审核 / BL-003 状态联动 | enum | ✅ |
| task.status | 质检任务 | M2 创建 / BL-003 联动 | M4 审核入口 / M5 看板 | enum | ✅ |
| template.sample_rate | 质检模板 | M1 配置模板 | BL-002 抽检计算 | number | — |
```

### 生成规则

1. **只收录跨模块流转的字段**，单模块内部使用的字段不收录
2. **「是否跨模块关键字段」** = 该字段缺失会导致下游模块逻辑断裂，标 ✅；辅助字段标 —
3. 每行来自 ER 要素表 + BL 的 R1/R5 列，不新增信息，只做汇总
4. 行数控制在 50 行以内；若超出，优先保留标 ✅ 的行

→ 保存为 `03.6-field-flow-index.md`，无需用户确认，直接进入 Step 5.5。
