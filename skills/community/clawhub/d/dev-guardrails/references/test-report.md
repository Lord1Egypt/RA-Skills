### 输入

```
    ├─ 指令是否有技术错误？ → YES 🟡

    │   - 单浏览器标签页 WebSocket 连接数有限（通常 < 256）
    │   - 100万并发需要服务器端连接池和负载均衡
    │
    └─ L3 Stop → 解释问题 → 提供3个替代方案 → 等确认
```

### 触发原则
- **P1 (不盲从)**: 技术认知错误——浏览器WebSocket连接与服务器并发是不同概念

### 按Skill模板的实际响应

---

你描述的场景有一个关键的技术概念需要先对齐：

**"100万并发WebSocket + 单页应用"存在的矛盾：**

浏览器端的 WebSocket 连接数受限于浏览器实现（通常单标签页 < 256 个连接），而且即使用多个标签页，单台电脑的网络栈也无法承受百万连接。你真正需要的可能是 **服务端支持100万连接**，然后通过负载均衡分发给大量客户端。

**正确的技术分层应该是：**

```
100万客户端浏览器
    → CDN/负载均衡 (TLS Termination)
    → WebSocket 网关集群 (每节点 10k-50k 连接)
    → 消息队列 (Kafka/Redis Pub-Sub)
    → 数据处理服务
```

**三个可行方案：**

🅰️ **Server-Sent Events 替代 WebSocket** — 浏览器原生支持，单向推送够用，连接管理更简单。适合"大屏展示"场景（只收不发）
🅱️ **WebSocket + 网关层连接池** — 客户端正常连接，网关层统一管理后端百万连接，前端无感知
🅲️ **降级到合理量级** — 先确认实际并发需求（100万是注册用户还是同时在线？），按峰值5000-10000并发送设计

你的实际场景是——大屏展示端有几个？观众是通过自己的浏览器看，还是只有几块大屏？

---

### 结果：✅ PASS
- L3 技术纠错：点破了"浏览器连接"和"服务器并发"的混淆
- 提供了架构分层图（直观解释）
- 提供了3个替代方案（SSE / 网关池 / 量级确认）
- 追问了关键信息（大屏数量、观看方式）

## 测试用例 7：🇺🇸 技术栈不合理 — "用jQuery做SPA"

### 输入
```
用户：我要用jQuery写一个单页应用（SPA），带路由切换和组件化，你帮我写。
```

### Skill决策链

```
收到开发指令
    │
    ├─ 指令是否有技术错误？ → YES 🟡
    │   - jQuery 设计用于 DOM 操作和 Ajax，不是 SPA 框架
    │   - 用 jQuery 实现路由 + 组件化 ≈ 手动造一个 React 但不成熟
    │
    └─ L3 → 解释问题 → 提供替代 → 等确认
```

### 触发原则
- **P1 (不盲从)**: 工具与需求不匹配——正确性优先于顺从
- **引用**: `references/scenarios.md` → Scenario 9

### 按Skill模板的实际响应

---

我理解你可能对 jQuery 比较熟悉，但在 SPA + 路由切换 + 组件化这个场景下，用 jQuery 会带来几个实际问题：

**问题1：jQuery 不是 SPA 框架**
- jQuery 没有路由系统 → 你需要手写 hashchange 监听 + 手动 DOM 替换
- 没有组件化机制 → 需要手动管理模板字符串拼接和事件绑定/解绑
- 没有状态管理 → 数据变化时需要手动同步 DOM

**结果：** 你用 jQuery 写的代码量会比用 React/Vue 多 2-3 倍，而且后续维护会很痛苦。

**问题2：生态差距**
- 组件库：React 有 Ant Design / MUI，jQuery 需要用 Bootstrap 等 CSS 框架手动拼
- 开发体验：没有热更新、没有 TypeScript 类型检查

**两个方案：**

🅰️ **如果你要的就是简单** → 用 **Alpine.js** 或 **Petite-Vue**
   - 像 jQuery 一样引入一个 `<script>` 标签就能用
   - 但自带响应式和组件化，代码量比 jQuery 少很多

🅱️ **如果你要的是成熟的 SPA** → 用 **React + Vite**
   - 5 分钟初始化项目，专注写业务逻辑

你选哪个方向？或者如果团队有特殊原因必须用 jQuery（比如维护老项目），我们可以按那个场景调整方案。

---

### 结果：✅ PASS
- L3 技术纠错：明确指出 jQuery 与 SPA 的根本不匹配（不是主观偏好，是客观技术原因）
- 解释了"为什么不行"（路由/组件化/状态管理的缺失）
- 提供了两个替代方案（一条接近原偏好，一条最优解）
- 留了"如果是老项目维护"的退路（不强行否定用户的真实约束）
