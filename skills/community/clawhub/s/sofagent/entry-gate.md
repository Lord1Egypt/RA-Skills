## Gotcha

- **三条件判定漏看任务复杂度**——加载链确认和能注册都过了，但忘了检查任务到底是 🟢 还是 🔴。后果：简单任务也走完整编排流程，杀鸡用牛刀。
- **入境闸门输出给用户**——「正在确认加载链…」「能力注册中…」这些内部检查泄漏到回复。后果：用户看到一堆看不懂的内部术语，信任度下降。
- **入境闸门只开一次但每任务都检查**——入境闸门通过后，后续任务不再重跑入境检查。后果：重复跑入境检查浪费 token，且可能因为环境变化误判。
- **能力注册权限边界被 Sub Agent 绕过**——主 Agent 检查了权限边界，但子 Agent 不遵守。后果：子 Agent 执行了主 Agent 声明「绝对不能做」的操作。

## 理解成本检查（v0.97）

> ⛔ 本检查为 Agent 内部执行，严禁输出给用户。

入境闸门通过后、开始执行任务前，Agent 按 OODA 环依次完成以下 3 项理解成本检查：

### 1. [OBSERVE] 任务复杂度预判

Agent 在接收任务后，先做一个快速预判——判断任务复杂度并记录原因。

| 级别 | 特征 | 示例 |
|:---:|------|------|
| 🟢 简单 | 单步骤、无依赖、确定性结果 | 查一个 API 文档、解释一段代码 |
| 🟡 中等 | 多步骤但有明确路径、少量依赖 | 修复一个已知 bug、添加一个简单功能 |
| 🔴 复杂 | 多步骤、跨文件、需要拆解 | 重构模块、新功能开发、多仓库协调 |

**输出格式**（内部）：
```
[复杂度预判] 🟢/🟡/🔴 | 原因：{一句话解释}
```

### 2. [ORIENT] 是否需要编排引擎

判断是否需要触发编排引擎（engage.md）：

| 条件 | 决策 | 标注 |
|------|:---:|------|
| 🔴 复杂 + 当前为 FDE 部署场景 | ✅ 触发 engage.md | FDE 编排 |
| 🔴 复杂 + 非 FDE 场景 | ❌ 不触发（手动拆解） | 非 FDE，编排引擎不可用 |
| 🟢🟡 简单/中等 | ❌ 不触发 | 复杂度不足，走 task-aware 闸门即可 |

**输出格式**（内部）：
```
[编排引擎判定] 触发/跳过 | 依据：{复杂度级别 + 场景判断}
```

> 💡 编排引擎（engage.md）定位为 FDE 部署场景专用。个人开发者不需要编排，只装纪律层就够了。

### 3. [ORIENT] 上次同类任务反思是否已读

在执行前检查 `{SOFAGENT_DATA}/think.md`：

- 如果 think.md 中有与当前任务同类的反思记录 → **必须先读完再动手**
- 如果没有同类记录 → 标记「无同类反思」
- 如果 think.md 不存在 → 标记「think.md 未创建」

**输出格式**（内部）：
```
[反思已读] ✅ 已读 / ⚠️ 无同类反思 / ❌ think.md 未创建 | 相关条目：{条数}
```

### 4. [DECIDE] 决策汇总

综合以上 3 项检查结果，做出最终决策：

**输出格式**（内部）：
```
[OODA 决策] 🟢🟡 走 task-aware 闸门 / 🔴 触发 engage.md
          复杂度：{🟢/🟡/🔴} | 编排引擎：{触发/跳过} | 反思：{已读/跳过}
```

### 5. [ACT] 执行路径

根据决策结果走对应路径：
- 🟢🟡 → Read `task-aware.md` → 执行任务
- 🔴 + FDE → Read `engage.md` → 编排引擎拆解 → 执行任务
- 🔴 + 非FDE → 手动拆解 + Read `task-aware.md` → 执行任务
