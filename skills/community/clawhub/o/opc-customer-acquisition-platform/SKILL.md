## 一、Skill概述

### 1.1 定位与价值

OPC获客分包平台是胡田OPC导师体系中的**核心协作工具**，定位为社区生态内的智能分包协作平台。它不是猪八戒网那样的外部众包平台（纯商业化、陌生人交易），而是扎根于OPC导师社区的**信任网络**，通过集群接单、智慧分包、协作交付三大核心能力，帮助社区成员突破"一人公司"的业务边界。

**一句话定位**：社区信任背书 + 集群议价能力 + 智能分包匹配的协作平台。

### 1.2 核心解决的问题

| 痛点 | 解决方案 |
|------|----------|
| OPC单打独斗，谈判能力弱 | 集群接单，社区背书，统一对外报价 |
| 能力与需求错配，效率低下 | 智能画像 + 精准匹配 |
| 协作交付难，质量风险高 | 标准化流程 + 分账透明 |
| 信用缺失，信任成本高 | 社区信用双轨制 |

### 1.3 适用场景

- **OPC创业者**：获取更多项目机会，扩大业务边界
- **OPC导师**：发布分包需求，组建团队，管理交付
- **社区内企业**：快速找到靠谱服务商，获得品质保障
- **外部客户**：获得社区背书的性价比服务

## 三、七大环节操作指南

### 环节1：项目挂榜

#### 谁可以发布项目

| 发布者类型 | 权限 | 审核要求 |
|-----------|------|---------|
| **OPC导师** | 可发布任意规模项目 | 需提交客户需求确认 |
| **OPC成员** | 可发布≤5万项目 | 需导师背书 |
| **外部客户** | 可发布需求等待报价 | 需社区审核 |
| **平台管理员** | 代客户发布 | 需客户书面委托 |

#### 项目分级标准

| 等级 | 金额区间 | 复杂度 | 周期 | 审核层级 | 竞标方式 |
|------|---------|--------|------|---------|---------|
| **D级（微项目）** | <5,000元 | 单一任务 | <7天 | 自动通过 | 快速抢单 |
| **C级（小项目）** | 5千-2万 | 简单分包 | 7-15天 | 导师审核 | 3家竞标 |
| **B级（中标）** | 2万-10万 | 常规项目 | 15-30天 | 盟主审核 | 5家竞标 |
| **A级（大标）** | 10万-50万 | 复杂项目 | 30-90天 | 三长老审核 | 联合体竞标 |
| **S级（超大标）** | >50万 | 战略项目 | >90天 | 理事会审批 | 联合体+外协 |

#### 操作流程

```
1. 登录平台 → 发布需求 → 填写项目发布单
2. 提交审核 → 系统自动审核格式
3. 导师/盟主/三长老分级审核
4. 审核通过 → 项目挂榜公示
5. 进入【接榜挂帅】环节
```

**详细文档**：`references/01-项目挂榜指南.md`

### 环节2：接榜挂帅

#### 核心机制

像"打擂台"一样，成员竞争接项目。

```
┌─────────────────────────────────────────────────────────────┐
│                    接榜挂帅 竞标流程                        │
├─────────────────────────────────────────────────────────────┤
│  【揭榜期】           【竞标期】          【挂帅确认】        │
│     │                   │                   │               │
│     ▼                   ▼                   ▼               │
│  项目发布         竞标方案提交        评标+公示              │
│  能力预审        方案PK              竞标结果通知            │
│  资质验证        线上/线下答辩       挂帅人确定             │
└─────────────────────────────────────────────────────────────┘
```

#### 竞标方式对比

| 竞标类型 | 适用场景 | 竞标方数 | 竞标内容 | 时长 |
|---------|---------|---------|---------|------|
| **快速抢单** | D级微项目 | 不限 | 直接报价 | 24小时 |
| **方案竞标** | C级小项目 | 3-5家 | 方案+报价 | 3-5天 |
| **综合竞标** | B级中标 | 5-8家 | 方案+案例+报价+信用 | 7-10天 |
| **联合体竞标** | A级大标 | 2-3组 | 团队方案+分工+报价 | 10-15天 |

#### 评标标准（综合评分制）

| 评分维度 | 权重 | 评分项 | 数据来源 |
|---------|-----|-------|---------|
| **信用评级** | 30% | 信用等级（S/A/B/C/D） | 股权合作Skill-信用台账 |
| **历史业绩** | 25% | 同类项目数量+客户评价 | 股权合作Skill-贡献台账 |
| **方案质量** | 25% | 方案完整性+创新性+可行性 | 评标委员会 |
| **报价合理性** | 15% | 报价与市场指导价对比 | 获客分包平台-指导价体系 |
| **时间承诺** | 5% | 是否能在要求时间内完成 | 竞标书 |

#### 挂帅人权益

- 基础贡献权重 + 10%挂帅补贴
- 优先获得新项目信息
- 可担任三长老角色（需信用A级以上）

**详细文档**：`references/02-接榜挂帅竞标规则.md`

### 环节3：项目分包

#### 拆解四步法

```
Step 1：需求分析
└── 分析客户需求，确定核心交付物

Step 2：模块划分
└── 按功能/流程拆解为独立模块
    • 每个模块有明确边界
    • 每个模块可独立验收

Step 3：技能映射
└── 为每个模块匹配所需技能
    • 核心技能（必备）
    • 辅助技能（可选）

Step 4：分包分配
└── 根据能力画像分配模块
    • 优先匹配度高
    • 考虑负载均衡
```

#### 分包规则

| 规则类型 | 具体内容 |
|---------|---------|
| **分包门槛** | 单模块金额≥1000元才能分包 |
| **分包比例** | 挂帅人保留模块价值≥30% |
| **分包审核** | C级以上项目需三长老审批 |
| **接包限制** | 同一成员承接模块数≤3个 |
| **利益回避** | 挂帅人不得承接自己分包的项目 |

#### 匹配算法

```
匹配得分 = 技能匹配度×40% + 信用等级×30% + 负载状态×20% + 历史好评×10%
```

**详细文档**：`references/03-项目分包机制.md`

### 环节6：分账结算

#### 分账触发条件

- 客户验收通过
- 项目尾款到账
- 无未解决纠纷

#### 分账流程（7步）

```
1. 【回款到账】客户付款给牵头方
2. 【开票给客户】牵头方开全额发票
3. 【成本确认与开票】各方向牵头方开成本发票
4. 【税费核算】牵头方核算应缴税费
5. 【分账执行】
   ├── 扣除税费预留
   ├── 扣除平台服务费（5%）
   ├── 报销实际成本（已开票部分）
   └── 按贡献权重分配税后净收益
6. 【打款】牵头方分别打款给各方
7. 【完税凭证】全员可查
```

#### 分账优先级

1. 税费预留（按比例计提）
2. 平台服务费（5%）
3. 实际成本报销
4. 按贡献权重分配收益

**详细文档**：`references/06-分账结算与纠纷处理.md`

### 环节7：评价沉淀

#### 双向评价机制

| 评价方 | 评价维度 | 权重 |
|--------|----------|------|
| **甲方评乙方** | 交付质量、响应速度、专业能力 | 70% |
| **乙方评甲方** | 配合程度、付款及时、需求清晰 | 30% |

#### 评价维度（5分制）

| 维度 | 说明 | 优秀(5) | 良好(4) | 一般(3) | 较差(2) | 很差(1) |
|------|------|---------|---------|---------|---------|---------|
| **交付质量** | 成果物质量 | 超预期 | 符合要求 | 基本达标 | 部分不达标 | 完全不达标 |
| **时效性** | 是否按时完成 | 提前完成 | 按期完成 | 轻微延期 | 明显延期 | 严重延期 |
| **沟通协作** | 沟通效率 | 非常顺畅 | 比较顺畅 | 一般 | 沟通困难 | 无法沟通 |
| **专业能力** | 解决问题的能力 | 游刃有余 | 能解决问题 | 基本能解决 | 解决困难 | 无法解决 |
| **服务态度** | 配合意愿 | 非常积极 | 比较积极 | 一般 | 比较消极 | 非常消极 |

#### 信用更新规则

| 评价综合分 | 信用分变化 | 备注 |
|-----------|-----------|------|
| 4.5分以上 | +3分/次 | 高质量完成 |
| 3.5-4.5分 | +1分/次 | 正常完成 |
| 2.5-3.5分 | -2分/次 | 存在问题 |
| 2.5分以下 | -5分/次 | 严重问题 |

**详细文档**：`references/07-评价沉淀与信用更新.md`

## 六、快速检查清单

启动一个获客分包项目前，确认以下事项：

```
□ 项目已挂榜发布并通过审核
□ 竞标已完成，挂帅人已确定
□ 项目已拆解为若干分包模块
□ 各分包方已匹配并确认
□ SPV主协议已签署
□ 保密协议已签署（全体成员）
□ 质量承诺书已签署（各执行方）
□ 里程碑验收标准已确认
□ 分账方案已约定
□ 三长老角色已指定
□ 共管账户已开设（如需）
□ 各方联系方式已确认
□ 协作工具已就绪（飞书群/文档）
□ 项目启动会议已召开
```

## 八、合规底线

| 红线 | 正确做法 |
|------|---------|
| ❌ 向不特定对象公开募集资金 | ✅ 只对接特定的已有合作基础的人 |
| ❌ 承诺固定收益或保本 | ✅ 只说按实际项目收益分配，风险自担 |
| ❌ 平台归集资金形成资金池 | ✅ 资金走第三方托管或共管账户 |
| ❌ 层级裂变、拉人头返利 | ✅ 每个项目独立核算，不搞层级 |
| ❌ 发行代币、积分分红 | ✅ 不用任何代币化，只做合同约定 |
| ❌ 服务包没有定价依据 | ✅ 每个服务项都有市场价格参考和验收标准 |

### Pre-Check（执行前检查）
| 序号 | 检查项 | 检查方法 | 失败处理 |
|------|--------|---------|---------|
| PC1 | references目录完整性 | 检查9个references文件是否存在 | 提示缺失文件 |
| PC2 | contracts目录完整性 | 检查11个contracts文件是否存在 | 提示缺失文件 |
| PC3 | scripts脚本可用性 | 检查2个Python脚本是否存在且语法正确 | 提示脚本错误 |
| PC4 | 股权合作Skill可用 | 验证股权合作Skill的SPV协议模板可访问 | 提示依赖缺失 |

| PC7 | 挂帅人信用记录可查 | 验证挂帅人信用等级≥A级 | 提示信用不足 |

### Self-Heal（自修复策略）
| 故障场景 | 修复策略 | 重试上限 |
|---------|---------|---------|
| 竞标方案数量不足 | 延长竞标期或降低竞标门槛 | 2次 |
| 评标脚本执行失败 | 检查Python环境，重新执行 | 3次 |
| 协议模板缺失章节 | 自动补充标准章节模板 | 2次 |
| 里程碑验收标准不明确 | 自动补充量化验收标准模板 | 2次 |
| 分账计算错误 | 重新读取权重数据重新计算 | 2次 |
| 评价数据缺失 | 自动补充默认值并标记 | 1次 |
