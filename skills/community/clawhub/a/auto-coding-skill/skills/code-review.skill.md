# Code-Review：代码审查

## 1. Overview（概述）

代码审查不是走过场。本技能定义 Reviewer 的三级否决权（🔴 阻塞 / 🟡 警告 / 💭 建议）、审查边界和 Risk Scorecard 集成，确保审查结果可量化、可追溯。

**核心原则**：
- Reviewer 说了算 — 审查不通过必须打回重做
- 不审查实现细节 — 审查契约、安全、架构、可维护性
- 每个 review comment 必须关联一个 Risk Scorecard 条目

## 2. When to Use（使用条件）

**触发条件**：
- Step 5 反思阶段自动注入（与 zoom-out 配对）
- B 级 / C 级任务强制执行
- A 级任务简化版（仅检查安全 + 需求对齐）

**可跳过**：
- 纯配置修改（无逻辑变更）
- 文档修改

## 3. Process（执行流程）

### 3.1 Reviewer 三级否决权

```
🔴 阻塞（Blocking）：不允许合入，必须修复
  ├─ 安全漏洞：SQL 注入、XSS、未验证用户输入、密钥泄露
  ├─ 不符合需求：与 CONTEXT.md / ADR 中的决定矛盾
  ├─ 严重过度设计：500 行完成的功能用了 1500 行
  ├─ 破坏现有功能：回归测试失败
  └─ 动作：打回重做，不进入下一阶段

🟡 警告（Warning）：强烈建议修改，但不阻塞
  ├─ 不必要的抽象（Rule of Three 违反）
  ├─ 命名不一致（但不破坏功能）
  ├─ 缺少必要注释（复杂逻辑）
  ├─ 代码重复（同一概念 ≥ 2 处）
  └─ 动作：记录，下次审查检查是否处理

💭 建议（Informational）：仅供参考，可选
  ├─ 代码风格微调
  ├─ 替代方案建议（不改变功能）
  ├─ 性能优化机会（非关键路径）
  └─ 动作：记录，不跟踪处理状态
```

### 3.2 审查边界

```
✅ 审查这些（必须）：
  - 接口契约：输入/输出是否与设计一致
  - 安全：所有用户输入是否经过验证
  - 架构：是否违反分层、引入不当耦合
  - 测试质量：测试是否覆盖边界和错误路径
  - 需求对齐：代码是否实现（且只实现）了需求

❌ 不审查这些：
  - 实现方式的选择（除非明显错误）— 尊重工程师判断
  - 代码风格（除非团队有明确规范）— 不因个人偏好打回
  - 命名偏好（除非误导性命名）— 不因"我更喜欢 X"打回
  - 未来扩展性（YAGNI）— 不因"以后可能需要"要求加抽象
```

### 3.3 Risk Scorecard 集成

```
审查流程：
1. 每个 review comment → 寻找对应的 Scorecard 条目
2. 命中条目 → 引用条目编号，启用量化信号检测
3. 未命中条目 且 level = 🔴 → 检查是否需要在 Scorecard 中新增该条目
4. 输出：Scorecard Report（阻塞/警告/通过统计）
```

### 3.4 审查输出格式

```
## Code Review Report

### 🔴 阻塞项（{n} 项）
- [ ] {描述} → 关联 Scorecard: {条目编号}

### 🟡 警告项（{n} 项）
- [ ] {描述} → 关联 Scorecard: {条目编号}

### 💭 建议项（{n} 项）
- {描述}

### 迭代统计
- 当前迭代: {n}/{3}
- 上次阻塞: {n} 项 → 本次: {m} 项
```

## 4. Risk Scorecard（反借口表）

| # | 纪律 | 常见借口 | 反驳 |
|---|------|---------|------|
| C1 | 🔴 阻塞项必须修复 | "这个安全问题是理论上的，实际不会被利用" | 安全问题不分理论和实际 |
| C2 | 不审查实现方式 | "这个设计不好，换个写法" | 能 work + 可读 + 有测试 = 够好 |
| C3 | 尊重需求决策 | "技术方案可以更优雅" | 需求指定的做法就是对的 |
| C4 | 不因未来需求加复杂度 | "以后可能需要支持..." | YAGNI — 未来的需求有未来的工程师 |
| C5 | 每个评论关联 Scorecard | "这个不用引用，一看就知道" | 不量化的审查 = 不可审计 |

## 5. Red Flags（危险信号）

- 🚩 审查时间 < 30 秒（对任何非 A 级任务）— 可能跳过了关键检查
- 🚩 只有 💭 项没有 🔴/🟡 — 可能过于宽松（除非代码确实完美）
- 🚩 只有一个 🔴 但涉及安全 — 安全相关的 🔴 通常不止一个
- 🚩 同一 🔴 项连续两次迭代出现 — 修复未生效或 root cause 未被解决
- 🚩 🔴 项 = 0 但 🟡 ≥ 5 — 考虑提升部分 🟡 到 🔴
- 🚩 审查后代码行数增加 > 20% — 可能有过度设计

## 6. Verification（自检清单）

**必检（≤5 项）**：
- [ ] zoom-out 分析已完成（在 code-review 之前）
- [ ] Risk Scorecard 量化检测已运行
- [ ] 🔴 阻塞项已全部解决（无未解决的 🔴）
- [ ] Scorecard Report 已写入阶段日志
- [ ] 没有过度批评（需求要求的做法已被尊重）

**抽检（≤3 项）**：
- [ ] 🟡 警告项有处理计划（接受/修复/推迟，每项标注）
- [ ] 跨模块问题已识别（zoom-out 中的 🔴 项在 review 中被引用）
- [ ] 迭代次数统计正确（≤3 次上限）

**免检**：审查措辞、审查顺序、审查模式（AI vs 人工偏好）
