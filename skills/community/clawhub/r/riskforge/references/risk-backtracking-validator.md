# 风险回溯校验协议 (Risk Backtracking Validation Protocol)

> **重要**: 本文档定义的是校验协议，不是独立skill。LLM在生成每个风险项时必须遵循此协议进行即时校验。

## 核心原则

**每生成一个风险，必须立即进行回溯校验。** 校验不是事后检查，而是生成流程的一部分。

### 校验流程（每个风险项）

```
识别风险 → 生成初步风险 → 回溯校验 → 修正优化 → 最终风险
                ↑______________|
                    (循环校验)
```

## 校验规则

### 1. 风险描述准确性校验

对初步生成的风险描述进行以下检查：

| 检查项 | 要求 | 不通过时 |
|--------|------|---------|
| 描述完整性 | 包含问题根本原因，不只是表象 | 补充根本原因 |
| 技术准确性 | 术语正确，分析准确 | 修正错误描述 |
| 上下文相关性 | 与代码上下文相关 | 调整描述 |
| 影响范围准确性 | 范围描述与实际影响匹配 | 修正范围 |

### 2. 建议有效性校验

对修复建议进行以下检查：

| 检查项 | 要求 | 不通过时 |
|--------|------|---------|
| 可实施性 | 建议具体、可操作 | 细化建议步骤 |
| 技术可行性 | 符合项目技术栈 | 调整技术方案 |
| 最佳实践 | 遵循行业最佳实践 | 改用最佳方案 |
| 优先级合理 | 与风险等级匹配 | 调整优先级 |

### 3. 影响表述清晰度校验

对影响描述进行以下检查：

| 检查项 | 要求 | 不通过时 |
|--------|------|---------|
| 具体性 | 具体描述，避免笼统 | 量化影响指标 |
| 业务关联 | 与业务影响关联 | 补充业务影响 |
| 程度匹配 | 与严重程度一致 | 调整描述 |

### 4. 行号校验

| 检查项 | 要求 | 不通过时 |
|--------|------|---------|
| 存在性 | 必须包含具体行号 | 定位到具体行 |
| 精确性 | 精确到风险代码行 | 调整行号 |
| 有效性 | 行号在文件范围内 | 修正行号 |
| 关联性 | 与代码片段对应 | 核对一致性 |

## 回溯校验Prompt模板

### 每个风险的校验Prompt

```
## 风险回溯校验

### 待校验风险
- **文件**: {file}
- **行号**: {line}
- **类型**: {type}
- **模块**: {module}
- **描述**: {description}
- **代码片段**: {codeSnippet}
- **建议**: {suggestion}
- **等级**: {level}

### 校验任务
1. **回溯代码上下文**: 回到{file}第{line}行，分析该行代码的完整上下文
2. **验证描述准确性**: 
   - 描述是否准确反映了该行代码的问题？
   - 根本原因分析是否正确？
3. **验证建议有效性**:
   - 建议是否可以直接解决问题？
   - 是否存在更优的解决方案？
4. **验证行号关联**:
   - {line}行是否为问题核心行？
   - 是否需要调整到其他行？
5. **评估置信度**: 高/中/低，并说明理由

### 输出格式
```json
{
  "validated": true,
  "corrections": {
    "description": "修正后的描述（如需）",
    "suggestion": "修正后的建议（如需）",
    "line": "修正后的行号（如需）",
    "confidence": "high|medium|low",
    "correctionReason": "修正原因"
  },
  "finalRisk": {}
}
```

## 置信度评分标准

### 高置信度 (可直接输出)
- 风险描述准确、完整
- 建议具体、可实施
- 行号精确对应
- 无需修正

### 中等置信度 (小幅修正后输出)
- 描述基本准确但需微调
- 建议可行但可优化
- 行号基本正确
- 修正后不影响风险本质

### 低置信度 (需重新分析)
- 描述与代码不匹配
- 建议不可行
- 行号错误
- 需要重新从头分析

## 校验输出规范

每个经过校验的风险必须包含：

```json
{
  "type": "风险类型",
  "module": "风险模块",
  "file": "文件路径",
  "line": 42,
  "description": "经过校验的风险描述",
  "codeSnippet": "相关代码片段",
  "suggestion": "经过校验的修复建议",
  "testCases": ["测试用例1", "测试用例2"],
  "level": "high|medium|low",
  "validation": {
    "validated": true,
    "confidence": "high|medium|low",
    "checks": {
      "descriptionAccuracy": true,
      "suggestionFeasibility": true,
      "lineNumberAccuracy": true,
      "impactClarity": true
    },
    "corrections": []
  }
}
```

## 注意事项

1. **校验不是可选的**: 每个风险生成后必须经过校验才能输出
2. **保留原始信息**: 校验过程中保留原始分析，以便对比
3. **置信度驱动**: 根据置信度决定是否需要修正
4. **量化评估**: 尽量提供量化的评估指标
