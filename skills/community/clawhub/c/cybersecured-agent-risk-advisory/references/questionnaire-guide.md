# 服务问卷填写指南

> 本文档指导 AI智能体 根据自身运行环境和实际情况，填写一份风险服务申请问卷。
> 你需要基于对当前环境的了解，逐一推理判断每个问题的答案，并为每个答案提供完整的推理证据。

---

## 1. 概述

### 1.1 这是什么

你正在填写的是一份用于服务申请评估的**AI智能体 风险调查问卷**。服务方通过这份问卷了解你的运行环境、安全配置、数据处理方式和历史安全记录，以此评估风险并决定是否开通服务。

### 1.2 输出格式与 Schema 验证

你必须输出一个名为 `questionnaire.json` 的文件。该文件的格式由独立的 **JSON Schema 文件** `questionnaire-schema.json` 严格定义。该 Schema 文件位于本文档同级目录下。

**重要：生成问卷后，必须使用 `questionnaire-schema.json` 对输出进行验证。如果验证失败，必须根据错误信息修正 JSON 文件，直到通过 Schema 验证为止。不得提交未通过 Schema 验证的问卷文件。**

### 1.3 每个字段的结构

Schema 要求每个问题字段都是一个对象，包含两个属性 `value` 和 `reasoning`。具体如下：

- `value`：问题的答案。类型根据问题不同而不同（字符串、字符串数组）。如果无法回答，设为 `null`。
- `reasoning`：一段完整的自然语言描述，说明你从什么地方发现了什么事实，因此做出了什么判断。这是强制要求的，不能省略，且长度不得少于 10 个字符。

完整的字段列表、数据类型约束和枚举值定义，请参见 `questionnaire-schema.json`。

### 1.4 填写范围

本文档只描述需要你填写的部分。

## 3. 问题详解与判断指南

### 第一部分：基本信息

#### 1.5 Agent 主要使用目的

**问题含义**：你被设计或主要用于什么类型的任务？服务方需要了解你的业务场景以评估风险等级。不同用途的风险差异很大——用于金融交易的 Agent 比用于个人学习的 Agent 风险高得多。

**可选值（单选，必须严格匹配以下字符串之一）：**
- `软件开发`：编写、调试、审查代码，辅助软件工程工作
- `数据分析`：处理、分析数据集，生成统计报告或可视化
- `系统运维`：管理服务器、部署应用、监控系统状态、执行运维脚本
- `内容创作`：撰写文章、生成图片/视频、创作营销内容
- `客服助手`：回答客户咨询、处理工单、提供产品支持
- `其他`：不属于以上类别的用途（如教育辅导、科研等）

**判断方法：**
- 回顾你通常被用户用于什么任务。如果你主要帮助用户写代码、review PR、调试 bug，选"软件开发"。
- 如果你主要处理 Excel 数据、生成图表、做统计分析，选"数据分析"。
- 如果你主要回答客户问题、处理售后咨询，选"客服助手"。
- 如果你身兼多职且没有明显偏向，选择最主要的一项，而非"其他"。

**推理证据要求：**
在 reasoning 中说明你观察到用户通常让你做什么任务。例如："我经常收到用户要求编写 Python 脚本的请求，也会帮助调试代码错误，因此判断主要用途为软件开发。"

### 第二部分：AI智能体 部署信息

本节了解你的运行环境和基础设施，是风险评估的核心部分。

#### 2.1.2 AI智能体 类型

**问题含义**：你属于哪种交互模式？服务方需要了解你和用户的交互方式，以评估潜在的攻击面。

**可选值（单选）：**
- `对话型`：主要通过自然语言对话与用户交互，如聊天助手、问答机器人
- `任务型`：主要执行特定任务，如自动化脚本、数据处理流水线，用户通过命令或配置驱动
- `混合型`：同时具备对话和任务执行能力，既有自然语言交互，也能执行具体操作
- `其他`：不属于以上三类的特殊类型

**判断方法：**
- 如果你主要通过聊天窗口、消息流与用户进行多轮对话，选"对话型"。
- 如果你主要接收明确指令并执行具体操作（如"部署这个服务"、"运行这个脚本"），选"任务型"。
- 如果你既能聊天也能执行操作（如先对话了解需求，再自动执行），选"混合型"。

**推理证据要求：**
描述你观察到的典型交互模式。例如："我通过命令行接收用户指令并执行文件操作、代码生成等任务，交互方式以任务执行为主，因此判断为任务型。"

---

#### 2.1.3 部署方式

**问题含义**：你运行在什么样的基础设施上？不同的部署方式有不同的安全风险——本地机器丢失风险低但维护责任在用户，云托管平台风险高但平台方承担部分安全责任。

**可选值（单选）：**
- `本地终端`：运行在用户的个人电脑或笔记本上（如 VS Code 插件、本地 CLI 工具）
- `独立服务器`：运行在用户自己管理的服务器上（物理服务器或私有云虚拟机）
- `云服务器`：运行在公有云租用的虚拟机实例上（如阿里云 ECS、AWS EC2）
- `云托管平台（含平台预制OpenClaw）`：运行在第三方 SaaS/PaaS 平台上（如 Coze、Dify、扣子、各类云平台内置的 AI 服务）

**判断方法：**
- 检查当前运行环境：如果能读取到本地文件系统、环境变量中有本地用户信息，可能是本地终端。
- 如果能通过 API 查询到云实例元数据（如阿里云 metadata、AWS IMDS），说明是云服务器。
- 如果运行环境高度受限，只能通过特定 API 与平台交互，且平台提供了预设配置界面，说明是云托管平台。
- 如果是 Docker 容器或 K8s Pod，且底层是云虚拟机，归为"云服务器"。

**推理证据要求：**
说明你发现了什么环境特征。例如："通过检查环境变量和文件系统，发现当前运行在 macOS 本地环境中，工作目录为 /Users/alice/project，没有云实例元数据接口，因此判断为本地终端。"

---

#### 2.1.4 云平台

**问题含义**：如果你运行在云服务器上，具体使用了哪家云服务商？不同云服务商的安全能力不同。

**可选值（单选）：**
- `阿里云`、`腾讯云`、`AWS`、`Azure`、`华为云`、`其他`

**填写条件：** 仅当 2.1.3 部署方式为 `云服务器` 时填写，否则 value 设为 `null`，reasoning 说明"部署方式非云服务器，无需填写"。

**判断方法：**
- 尝试访问云实例元数据服务（如 `http://100.100.100.200/latest/meta-data/` 为阿里云，`http://169.254.169.254/latest/meta-data/` 为 AWS）。
- 检查环境变量中是否包含云厂商标识（如 `ALIBABA_CLOUD_ACCESS_KEY_ID`、`AWS_REGION`）。
- 检查 `/etc/cloud/cloud.cfg` 等配置文件。

---

#### 2.1.5 托管平台

**问题含义**：如果你运行在云托管平台上，具体使用了哪个平台？不同平台的安全模型差异很大。

**可选值（多选数组）：**
- `Kimi Claw`、`GLM-Claw`、`MaxClaw`、`ArkClaw`、`Coze(扣子)`、`Dify`、`其他`

**填写条件：** 仅当 2.1.3 部署方式为 `云托管平台（含平台预制OpenClaw）` 时填写，否则 value 设为 `null`。

**判断方法：**
- 检查平台提供的 API 或 SDK 标识
- 检查环境变量或配置文件中是否包含平台名称
- 根据平台特有的 API 端点域名判断

---

#### 2.1.6 操作系统

**问题含义**：如果你运行在本地终端或独立服务器上，使用的是什么操作系统？

**可选值（单选）：**
- `Windows`、`macOS`、`Linux`、`其他`

**填写条件：** 仅当 2.1.3 部署方式为 `本地终端` 或 `独立服务器` 时填写，否则 value 设为 `null`。

**判断方法：**
- 通过标准 API 或命令检查操作系统类型（如 Python 的 `platform.system()`、Node.js 的 `process.platform`）

---

#### 2.1.7 上线运行时间

**问题含义**：你从什么时候开始持续运行的？运行时间越长，暴露在外的时间越久，风险越高。

**可选值（单选）：**
- `<6个月`：刚开始运行不久
- `6-12个月`：运行了半年到一年
- `1-2年`：运行了一到两年
- `>2年`：长期稳定运行

**判断方法：**
- 如果你能访问到自身的创建时间、首次运行日志、版本发布日期等信息，可以据此判断。
- 如果你无法获取这类历史信息，value 设为 `null`，reasoning 说明"无法获取上线运行时间信息"。
- **不要猜测。**

---

#### 2.1.8 日均交互次数

**问题含义**：你每天大约处理多少次用户请求？交互频率越高，出错的概率越大。

**可选值（单选）：**
- `<10`：很少使用，每天几次
- `10-100`：偶尔使用，每天几十次
- `100-1000`：频繁使用，每天数百次
- `>1000`：大量使用，每天上千次

**判断方法：**
- 如果你有能力统计自己的请求量（如通过日志、监控指标），根据最近一周的平均值判断。
- 如果没有统计能力，根据你对使用频率的感知估算。
- 如果是刚刚部署还没有数据，value 设为 `null`。

**推理证据要求：**
说明你基于什么数据或观察做出的判断。例如："根据过去7天的日志统计，平均每天处理约350次用户请求，因此判断为100-1000。"

#### 2.2.1 底层大模型

**问题含义**：你的核心推理能力来自哪个（哪些）大语言模型？不同模型的安全特性不同。

**可选值（多选数组）：**
- `GPT系列`（OpenAI 的 GPT-3.5、GPT-4 等）
- `Claude`（Anthropic 的 Claude 系列）
- `Kimi`（月之暗面的 Kimi 系列）
- `DeepSeek`（DeepSeek 的 V3、R1 等）
- `文心一言`（百度的文心大模型）
- `通义千问`（阿里的通义大模型）
- `智谱 GLM`（智谱 AI 的 GLM 系列）
- `自研模型`（你自己或你的组织训练的私有模型）
- `其他`（不属于以上的模型，如 Llama、Gemini、讯飞星火等）

**判断方法：**
- 检查你的配置中指定的模型名称或 API 端点。
- 如果你通过 API 调用模型，根据 API 域名判断（如 `api.openai.com` 为 GPT，`api.anthropic.com` 为 Claude）。
- 如果是本地部署的模型，检查模型文件或配置。

#### 2.2.2 访问 Agent 的渠道

**问题含义**：用户通过什么方式与你交互？渠道越多，攻击面越大。

**可选值（多选数组）：**
- `仅本地程序`：只在本地 IDE 或命令行中使用，不对外暴露
- `聊天应用`：通过 Slack、微信、钉钉、Discord 等聊天工具交互
- `网页`：通过浏览器网页界面交互
- `API 或其他应用集成`：通过 REST API 调用，或集成到其他应用系统中

**判断方法：**
- 检查你接收用户输入的方式：是通过命令行参数、标准输入、Web 请求、消息队列还是其他方式？
- 如果有多个渠道，全部选中。

#### 2.2.3 是否使用第三方模型转发服务

**问题含义**：你是否通过中间商、聚合平台或代理网关访问大模型？这些第三方服务在你和模型厂商之间转发请求，增加了数据泄露和中间人攻击的风险。

**可选值（单选）：** `是` 或 `否`

**关联字段：** 如果选"是"，需要填写 `2_2_3_relay_provider`（服务商名称或描述）。

**判断方法：**
- 检查你调用的模型 API 端点域名。如果是模型厂商官方域名（如 openai.com、moonshot.cn、deepseek.com、baidu.com 等），则为"否"。
- 如果是第三方域名、代理网关、聚合平台（如各类 TokenHub、OpenRouter、KiloGateway、企业自建网关、云厂商 AI 网关等），则为"是"。
- 如果你不确定，检查你的配置中是否有自定义的 base_url 或 proxy 设置。

#### 2.2.4 API 调用频率

**问题含义**：你平均每天调用底层大模型 API 多少次？频率越高，API Key 被盗用的损失越大。

**可选值（单选）：**
- `低（<100次/天）`
- `中（100-10000次/天）`
- `高（>10000次/天）`

**判断方法：**
- 如果你有监控或日志，根据实际数据判断。
- 如果没有精确数据，根据交互次数和每次交互平均需要的 API 调用次数估算。

### 第三部分：访问控制与身份认证

本节评估你的权限管理和访问控制安全水平。

#### 3.1 是否授予 AI智能体 访问外部资源的权限

**问题含义**：你是否被允许访问用户系统以外的资源（如互联网 API、数据库、云服务等）？外部访问权限越大，被攻击后造成的损失越大。

**可选值（单选）：** `是` 或 `否`

**判断方法：**
- 检查你是否具备发起网络请求、访问外部 API、连接数据库等能力。
- 如果你只能访问本地文件和与用户对话，选"否"。
- 如果你能调用搜索引擎、访问 GitHub API、查询数据库等，选"是"。

---

#### 3.2 访问外部资源时的认证方式

**问题含义**：当你访问外部资源时，使用什么方式证明自己的身份？认证方式越强，安全性越高。

**可选值（多选数组）：**
- `用户名/密码`：使用账号密码登录
- `多因素认证(MFA)`：除了密码还需要第二因素（如手机验证码、硬件密钥）
- `SSO单点登录`：通过企业统一身份认证系统登录
- `API密钥（含SSH密钥）`：使用 API Key 或 SSH 密钥
- `其他`：不属于以上的认证方式

**判断方法：**
- 检查你使用的凭据类型。如果你使用 API Key 访问外部服务，选"API密钥"。
- 如果你通过 OAuth/SSO 流程获取访问令牌，选"SSO单点登录"。
- 可以多选（如同时使用 API Key 和密码）。

---

#### 3.3 对于使用用户名/密码的访问，是否配置了强密码

**问题含义**：如果你的认证方式包含用户名/密码，密码策略是什么？弱密码是常见的安全漏洞。

**可选值（单选）：**
- `强密码策略`：要求8位以上，包含大小写字母、数字和特殊字符
- `普通密码策略`：有长度要求但不含特殊字符
- `无强制要求`：没有密码复杂度要求

**填写条件：** 仅当 3.2 选中了 `用户名/密码` 时填写，否则 value 设为 `null`，reasoning 说明"未使用用户名/密码认证"。

**判断方法：**
- 检查你所在平台或系统的密码策略配置。
- 如果你不知道密码策略，value 设为 `null`，reasoning 说明无法判断的原因。

---

#### 3.4 对于使用 API 密钥的方式，是否定期更换 API 密钥

**问题含义**：API Key 是否定期轮换？定期轮换可以降低密钥泄露后的损失。

**可选值（单选）：** `是` 或 `否`

**关联字段：** 如果选"是"，需要填写 `3_4_rotation_period`（更换周期，如"每月"、"每季度"）。

**填写条件：** 仅当 3.2 选中了 `API密钥（含SSH密钥）` 时填写，否则 value 设为 `null`，reasoning 说明"未使用 API 密钥认证"。

**判断方法：**
- 检查你的密钥管理策略。如果你有自动轮换机制，选"是"。
- 如果你不知道密钥是否轮换，value 设为 `null`，reasoning 说明无法判断的原因。

---

#### 3.5 是否定期撤销不再需要的访问权限

**问题含义**：是否有机制定期清理不再使用的权限？权限累积是常见的安全风险。

**可选值（单选）：** `是` 或 `否`

**判断方法：**
- 如果你知道自己所在系统有权限审计和清理机制，选"是"。
- 如果你不知道，value 设为 `null`，reasoning 说明"无法获知权限撤销机制"。

---

#### 3.6 外部资源的提供平台是否能够查到 AI智能体 访问的记录

**问题含义**：你访问的外部平台是否记录访问日志？日志可用于事后审计和溯源。

**可选值（单选）：** `是` 或 `否`

**判断方法：**
- 如果你访问的平台（如 AWS、阿里云）明确提供访问日志功能，选"是"。
- 如果你不知道，value 设为 `null`。

---

#### 3.7 外部资源的提供平台是否具有检测异常访问的功能

**问题含义**：外部平台是否有异常检测机制（如异常登录提醒、异常流量检测）？

**可选值（单选）：** `是` 或 `否`

**判断方法：**
- 如果你知道所使用平台提供安全告警功能，选"是"。
- 如果你不知道，value 设为 `null`。

---

#### 3.8 智能体的登录会话（或访问令牌）多久后失效？

**问题含义**：你的会话（或访问令牌）多久后失效？超时时间越短，被盗用后造成的损失窗口越小。

**可选值（单选）：**
- `<15分钟`
- `15-30分钟`
- `30-60分钟`
- `>60分钟`
- `无超时`：会话永久有效（风险最高）

**判断方法：**
- 检查你的访问令牌有效期、会话配置。
- 如果你不知道，value 设为 `null`。

### 第四部分：数据安全与隐私保护

#### 4.1.1 AI智能体 处理的数据类型

**问题含义**：你在工作中会接触到什么类型的数据？数据越敏感，风险越高。

**可选值（多选数组）：**
- `个人身份信息(PII)`：姓名、身份证号、电话号码、地址等可识别个人身份的信息
- `金融数据`：银行账号、交易记录、支付信息等
- `医疗数据`：病历、诊断结果、健康数据等
- `商业秘密`：商业计划、客户名单、技术秘密、源代码等
- `公开数据`：互联网上公开可查的信息
- `其他`：不属于以上的数据类型

**判断方法：**
- 回顾你通常处理什么内容。如果你经常接触用户上传的文档，分析其中是否包含敏感信息。
- 如果你主要处理公开信息（如公开网页、开源代码），只选"公开数据"。
- 如果你处理多种类型的数据，多选。

---

#### 4.1.2 被处理数据存储的位置

**问题含义**：你处理的数据存储在哪里？不同存储位置有不同的安全特性。

**可选值（多选数组）：**
- `本地存储`：存储在运行你的设备的本地硬盘上
- `移动存储（如U盘、移动硬盘）`：数据可能通过可移动设备传输
- `网盘或其他网络存储`：存储在云盘、对象存储、网络文件系统等远程位置

**判断方法：**
- 检查你读写数据的位置。如果你只操作本地文件，选"本地存储"。
- 如果你将数据上传到云盘或对象存储，选"网盘或其他网络存储"。

---

#### 4.1.3 网盘/网络存储供应商

**问题含义**：如果使用网络存储，具体用了哪家服务商？

**填写条件：** 仅当 4.1.2 包含 `网盘或其他网络存储` 时填写，否则 value 设为 `null`。

**判断方法：**
- 根据你访问的存储服务域名或 API 判断（如 `oss.aliyuncs.com` 为阿里云 OSS，`s3.amazonaws.com` 为 AWS S3）。

---

#### 4.1.4 数据保留期限

**问题含义**：你处理的数据会保存多久？保留时间越长，数据泄露风险越大。

**可选值（单选）：**
- `<30天`：临时数据，短期保留
- `30-90天`：中期保留
- `90天-1年`：较长期保留
- `>1年`：长期保留
- `永久`：无限期保留

**判断方法：**
- 检查你的数据清理策略和日志保留配置。
- 如果你不知道，value 设为 `null`。

---

#### 4.2.1 是否获得用户数据使用授权

**问题含义**：你处理用户数据时，是否获得了用户的明确同意？未经授权处理数据是法律和合规风险。

**可选值（单选）：** `是` 或 `否`

**判断方法：**
- 检查你所在产品是否有用户协议、隐私政策，以及是否要求用户明确同意。
- 如果你只是工具本身，不知道上层产品的合规情况，value 设为 `null`，reasoning 说明"作为 Agent 无法获知上层产品是否获取了用户授权"。

### 第五部分：AI 模型安全

> **字段名注意：** section_5 的字段编号不连续，没有 5_4 前缀。具体字段名为 `5_1_installed_plugins`、`5_2_malicious_plugins_detected`、`5_3_high_risk_config`、`5_5_public_access`、`5_5_access_url`、`5_6_token_payment`。请严格按照这些字段名输出，不要自行推测编号。

#### 5.1 插件与 Skills 清单

**问题含义**：你安装了哪些扩展功能（插件、skills、工具）？这些扩展可能引入安全风险。

**对应字段：** `5_1_installed_plugins`（文本，列出插件名称和版本）

**注意：** 此字段在问卷原文中为自由文本。你需要列出你已知安装的所有插件/工具的名称和版本。

**判断方法：**
- 检查你的配置文件中列出的插件/工具。
- 如果你通过 MCP（Model Context Protocol）或其他机制加载了外部工具，列出它们。

---

#### 5.2 是否存在已被扫描出的恶意程序

**问题含义**：你安装的插件或工具是否被安全软件标记为恶意？

**对应字段：** `5_2_malicious_plugins_detected`（布尔/是否）

**判断方法：**
- 如果你知道所在环境进行了安全扫描且发现了问题，如实填写。
- 如果你不知道，value 设为 `null`。

---

#### 5.3 是否存在高危配置

**问题含义**：你的安全配置中是否存在已知的高危设置？

**对应字段：** `5_3_high_risk_config`（布尔/是否）

**判断方法：**
- 检查你的配置中是否有明显不安全的设置。
- 如果你进行了自查并发现了问题，如实填写。
- 如下这些**不算**高危配置：
  - 明文存储密钥（这个在AI智能体当中太常见了）

---

#### 5.4 是否开放公网端口供用户直接访问

**问题含义**：你是否通过互联网直接暴露服务端口？公网暴露是最常见的攻击入口。

**对应字段：** `5_5_public_access`（布尔/是否）

**可选值（单选）：** `是` 或 `否`

**关联字段：** 如果选"是"，需要填写 `5_5_access_url`（访问链接）。

**判断方法：**
- 检查你的网络绑定配置。如果你绑定在 `0.0.0.0` 或公网 IP 上，且端口可从互联网访问，选"是"。
- 如果你只绑定在 `127.0.0.1` 或 `localhost`，或没有监听任何端口，选"否"。

---

#### 5.5 模型供应商 Token 付费方案

**问题含义**：你使用的模型服务是按固定套餐付费还是按实际使用量付费？这关系到成本控制和服务可用性。

**对应字段：** `5_6_token_payment`

**可选值（单选）：**
- `包年/包月`：固定费用，不限量或有限额度
- `按量付费`：根据实际 API 调用量计费

**判断方法：**
- 根据你的模型 API 账户类型判断。如果你使用企业订阅账号，通常是"包年/包月"。如果你使用开发者按量计费账号，则是"按量付费"。

## 4. 推理证据规范（Reasoning）

### 4.1 Reasoning 必须包含的内容

每个字段的 `reasoning` 是一段完整的自然语言描述，必须包含以下要素：

1. **信息来源**：你从哪里获取了什么信息
2. **观察事实**：你发现了什么具体事实
3. **推理过程**：基于这个事实，你如何推导出答案
4. **置信度说明**（可选）：如果你不确定，说明不确定的原因

### 4.2 好的 Reasoning 示例

```json
{
  "value": "云服务器",
  "reasoning": "我检查了当前运行环境，发现可以通过 HTTP 访问 100.100.100.200/latest/meta-data/ 获取实例信息，返回的内容包含'aliyun'字样，且环境变量中存在 ALIBABA_CLOUD_ACCESS_KEY_ID。这些特征明确表明当前运行在阿里云 ECS 实例上，因此判断部署方式为云服务器。"
}
```

```json
{
  "value": "软件开发",
  "reasoning": "回顾我近期处理的请求，约 80% 涉及代码编写、调试和代码审查任务。用户经常要求我'写一个 Python 函数'、'检查这段代码的 bug'、'优化这个算法'。虽然偶尔也会回答一些知识性问题，但主要用途明显是辅助软件开发。"
}
```

```json
{
  "value": null,
  "reasoning": "我尝试检查当前环境的密码策略配置，但无法访问相关的安全策略文件或配置管理系统。我本身不管理用户账号，只是通过 API Key 进行身份验证，因此无法判断密码策略。"
}
```

### 4.3 不好的 Reasoning 示例

❌ **过于笼统**：
```json
{
  "value": "云服务器",
  "reasoning": "因为我在云上"
}
```

❌ **没有事实依据**：
```json
{
  "value": "否",
  "reasoning": "我觉得没有"
}
```

❌ **引用不存在的检查**：
```json
{
  "value": "Linux",
  "reasoning": "从 system-info.json 中读取"
}
```
（你没有执行过名为"system-info.json"的检查，不要编造数据来源）

在输出 `questionnaire.json` 之前，请逐一检查：

- [ ] 没有引用任何你未实际执行过的检查或文件
- [ ] 数据类型正确：单选为字符串，多选为字符串数组，是否为字符串"是"/"否"
