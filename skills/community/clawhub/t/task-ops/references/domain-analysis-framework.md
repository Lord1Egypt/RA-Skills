# 领域分析框架

当用户提出"帮我做一个XX领域知识参考库"时，用此框架从零分析新领域并推导出完整的域划分和任务类型。

---

## Step A：领域定义

### A-1 领域名称规范化

| 输入 | 处理 | 输出 |
|------|------|------|
| 用户原始描述 | 提取核心领域关键词 + 确定边界 | 标准化的领域英文名 + 中文名 |

**示例映射**：

| 用户输入 | 规范化领域名 | 域代号前缀 |
|---------|------------|-----------|
| "写小说的" | Novel Writing (小说创作) | N |
| "做智能硬件的" | Smart Hardware (智能硬件) | H |
| "写HTML/Python/Shell/SQL单文件工具的" | Single-File Output (单文件产出) | S |
| "做会计代账的" | Bookkeeping Agency (代理记账) | B |
| "教人减肥健身的" | Metabolic Healing (代谢慢病) | M |

### A-2 领域分类定位

将目标领域在以下坐标系中定位，决定后续生成的参数倾向：

| 维度 | 低 | 高 | 影响什么 |
|------|----------|----------|---------|
| **R1 信息密度(S/C权重)** | 操作类/手工类 → S轻 | 数据类/研究类 → S重C深 | 感知和认知单元占比 |
| **R2 创造性(A权重)** | 流程类/合规类 → A标准 | 艺术类/研发类 → A极高 | 内容轴创新轴激活频率 |
| **R3 交互性(I权重)** | 独立产出类 → I少 | 服务类/协作类 → I多 | 交互单元数量 |
| **R4 规范性(G权重)** | 创作类/设计类 → G偏松 | 法规类/工程类 → G偏严 | 守护单元密度、合规约束数量 |
| **R5 迭代性(循环)** | 一次性交付→循环少 | 持续运营/连载→循环多 | 管线中↻模式的使用 |

**快速判定表**：

| 典型领域 | R1信息密度 | R2创造 | R3交互 | R4规范 | R5迭代 | 推导结果 |
|---------|-----------|--------|--------|--------|--------|---------|
| 医药文档 | 高 | 中 | 高 | 高 | 中 | S重C深I高G高A标准 |
| 网络小说 | 中 | 极高 | 中 | 低 | 极高 | A极高G松循环多 |
| 智能硬件 | 高 | 高 | 高 | 高 | 高 | 全维度高+G极高 |
| 单文件代码 | 中 | 中 | 低 | 高 | 中 | G高A标准结构清晰 |
| 代账记账 | 高 | 低 | 中 | 高 | 周期性 | G高A低C深O强 |
| 教学培训 | 中 | 中 | 高 | 中 | 低 | I高A中G中 |

### A-3 R评分到UTOS参数映射

将A-2的R1-R5评分映射为具体的UTOS执行参数：

| R评分 | 元操作权重调整 | AI自治度偏好 | 守护(G)密度 | 管线模式倾向 |
|-------|--------------|-------------|------------|-------------|
| **R1信息密度=高** | S↑C↑（感知和认知占比高） | 🟨半自动偏多（需人确认数据源） | 数据准确性校验 | P3并行汇聚（多源数据） |
| **R1信息密度=低** | S↓C↓（轻感知轻认知） | ⬛全自动偏多 | 低 | P1基础闭环 |
| **R2创造性=高** | A↑（行动/产出占比高） | ⬜辅助偏多（人定方向） | 风格一致性校验 | P7发散收敛 |
| **R2创造性=低** | A标准 | ⬛全自动偏多 | 低 | P1基础闭环 |
| **R3交互性=高** | I↑（交互占比高） | ⬜辅助偏多（人对外沟通） | 沟通合规校验 | P5交互驱动 |
| **R3交互性=低** | I↓ | ⬛全自动偏多 | 低 | P1基础闭环 |
| **R4规范性=高** | G↑O↑（守护和组织占比高） | 🟨/⬜（合规节点必须人工） | 高密度，每阶段嵌入G | P6全守护 |
| **R4规范性=低** | G↓ | ⬛全自动偏多 | 低 | P1基础闭环 |
| **R5迭代性=高** | 循环↑（短链高频） | 🟨半自动（每轮需确认） | 每轮迭代末尾G | P2迭代精炼 |
| **R5迭代性=低** | 循环↓（长链低频） | ⬛/🟨 | 终末校验 | P1基础闭环 |

**使用方式**：对每个R维度取评分后，叠加各维度的参数调整。多个维度同时为高时，参数叠加（如R1高+R4高 = S↑C↑G↑O↑ + 高密度守护）。

## Step B：域划分（Domain Partitioning）

### B-1 价值链拆解法

**核心方法**：沿领域的**价值链或生命周期**拆分域。每个域代表价值链上的一个阶段。

**通用模板——按生命周期型**（适用于产品/项目/创作类领域）：

```
阶段0：输入/素材阶段   →  需求/调研/灵感/原料
阶段1：规划/架构阶段   →  设计/架构/大纲/方案
阶段2：执行/生产阶段   →  开发/写作/制造/编码（可能有多条并行线）
阶段3：验证/测试阶段   →  测试/审查/校验/评审
阶段4：交付/发布阶段   →  发布/部署/上线/出版
阶段5：运营/维护阶段   →  运营/迭代/客服/优化
```

**通用模板——按职能分工型**（适用于组织/流程类领域）：

```
职能1：前端/面向用户    →  销售/服务/展示/内容
职能2：中台/核心能力    →  产品/技术/专业服务
职能3：后台/支撑保障    →  合规/财务/人力/IT
职能4：横向/协同管理    →  项目/质量/战略/风险
```

### B-2 域数量确定原则

| 因素 | 少域(5-7) | 中等域(8-10) | 多域(11+) |
|------|----------|-------------|----------|
| 领域复杂度 | 低 | 中 | 高 |
| 任务同质化程度 | 高（任务类型相似） | 中 | 低（各域差异大） |
| 依赖链长度 | 短(<3层) | 中(3-5层) | 长(>5层) |
| 参考案例 | — | 单文件产出、网文、硬件 | 医药 |

**经验公式**：域数 = ⌈ln(预估总任务数) × 2⌉，取值范围[5, 15]

### B-3 域命名规范

每个域必须包含：
- **域代号**：单个大写字母 + 数字编号（如 N1, H3, S5）
- **域名称**：2-6个中文词，准确描述域的范围
- **对标说明**（可选）：如果是从已有技能迁移，标注对应关系

**命名禁忌**：
- ❌ 域名之间有包含关系（如"开发"和"软件开发"）
- ❌ 域名过于抽象（如"管理"、"支持"）
- ✅ 域名具体且互斥（如"原型验证" vs "量产制造"，不重叠）

## Step C：任务类型推导

### C-1 任务枚举法

对每个域，使用以下问题清单枚举任务类型：

```
1. 这个域要"产生"什么？        → 产出的文档/制品/代码是什么？
2. 产出之前需要"准备"什么？    → 依赖哪些前置输入？
3. 产出之后需要"检查"什么？    → 有哪些质量/合规要求？
4. 这个域能"帮助"其他域做什么？ → 输出被谁消费？
5. 这个域内是否有"子流程"？    → 是否需要进一步拆分为多个任务？
```

### C-2 任务ID命名规范

```
{域代号}-{两位序号}  如: N1-01, H4-05, S7-03
```

### C-3 每个任务的必填字段

| 字段 | 说明 | 示例 |
|------|------|------|
| ID | 任务ID | D5-03 |
| 名称 | 任务类型的简短名称 | DA故事线文档 |
| 说明 | 一句话描述这个任务产出什么 | 页序、每页核心信息、视觉建议 |
| 依赖 | 前置任务ID列表，无依赖写"无（入口）" | D5-01 |
| UTOS映射提示 | S/C/A/O/I/G + 组合符号 | C→A（先认知后行动） |

### C-4 UTOS映射提示的推导规则

| 任务特征 | 映射提示 |
|---------|---------|
| 收集信息/调研/扫描 | S (感知) |
| 分析/评估/推理/决策 | C (认知) |
| 产出/创建/编写/制作 | A (行动) |
| 存储/归档/索引/分类 | O (组织) |
| 协调/沟通/确认 | I (交互) |
| 校验/审核/合规/约束 | G (守护) |
| 先收集再分析 | S→C |
| 先分析再产出 | C→A |
| 分析+产出+需审核 | C→A→G |
| 采集后直接产出 | S→A |

## Step D：依赖拓扑构建

### D-1 域间逻辑流

域间按价值链顺序排列，形成主线流：

```
D1 → D2 → D3 → ... → Dn
```

同时标注**并行分支**和**反馈回路**。

### D-2 跨域管线链路

识别3-8条关键跨域管线，每条格式为：

```
[链路名]: 任务A → 任务B → 任务C → ...
         [对应已有技能的某条链路（如为迁移）]
```

**管线发现启发式**：
1. **主链路**：从入口到最终交付的最短路径
2. **数据链路**：数据/信息的流转路径
3. **质量链路**：从生产到检验到修正的闭环
4. **资源链路**：物料/供应链的依赖路径
5. **加速链路**：MVP/快速版本的特殊路径

### D-3 依赖拓扑摘要格式

```markdown
## 依赖拓扑摘要

**[链路名1]**: X-01 → X-02 → X-03 → ...
**[链路名2]**: Y-01 → Y-02 ‖ Y-03 → Y-04 （并行）
**[闭环链]**: Z-01 → Z-02 → Z-03 → [回到Z-01]

更多组合由UTOS根据具体任务动态推导。
```
