# 洞察具体度标准

用途：防止人生岔路报告变成通用提示词输出。

## 1. 每个判断必须有机制链

抽象判断不能直接交付。每个关键判断必须补一条机制链：

```text
变量 A 改变 -> 行为 / 资源 / 评价方式改变 -> 结果范围改变
```

示例：

```text
去北京如果进入内容增长中台，日常会接触投放、推荐、热点和复盘数据；
这些数据会改变你判断内容的方式；
三年后更可能沉淀成增长策略简历。
```

## 2. 每个假设结果必须写反证条件

假设结果只有在条件成立时才有意义。必须写出让这个结果失效的条件。

```text
如果北京岗位只是品牌执行、离核心增长团队很远，这条线的收益会下降。
如果房租和通勤吞掉恢复时间，平台优势会被身体成本抵消。
```

## 3. 每个强判断必须写证据债

证据债指：缺哪一个事实，当前判断就只能停留在中等置信度。

常见证据债：

- 当年 offer 的团队位置。
- 直属老板的资源分配权。
- 税后收入、房租、通勤、储蓄率。
- 已经做出的作品或项目样本。
- 家庭支持强度和照护责任。
- 睡眠、通勤和长期恢复能力。

## 4. 少用抽象词，必须落地

以下词可以使用，但不能单独作为洞察：

- 机会密度
- 平台网络
- 职业上限
- 可见度
- 核心项目
- 圈层
- 资源
- 作品
- 稳定

必须写成：

```text
抽象词 + 具体场景 + 评价指标 + 可验证事实
```

例子：

```text
平台网络
```

应改成：

```text
是否能进入一线增长项目的周会，看到预算、转化、推荐策略和复盘数据。
```

## 5. 报告必须给出一个“不舒服但有用”的判断

每份报告至少有一句会让用户停顿的判断，要求具体、克制、可验证。

示例：

```text
你反复回看的对象可能不只是一座城市，真正缺口可能是当年没有把任何项目变成外部可见的作品资产。
```

写作时避免审判用户，重点指出可迁移变量。

## 6. 低成本动作必须能验证判断

行动建议不能写成“多了解、多思考、多输出”。每个动作都要留下可保存记录。

合格动作必须有三层：

- 具体动作：找谁、写什么、发给谁、问什么。
- 低风险边界：不让用户立刻辞职、搬家、投资、买房、分手或承担大额成本。
- 可观察结果：别人是否看懂、能否说出优势、哪个假设被证实或下调。

合格格式：

```text
| 动作 | 怎么做 | 看什么 |
| --- | --- | --- |
| 找一个目标岗位从业者聊一次 | 只问岗位职责、收入、通勤、压力、晋升和能否接触核心数据 | 如果对方长期碰不到复盘和预算，北京线收益下调 |
```

## 7. 回写检查

交付前逐条问：

- 这个判断有没有机制链？
- 这个结论有没有反证条件？
- 这个洞察能不能被换到任意城市仍然成立？
- 用户看完会知道下一步先做什么吗？
- 抽象词是否被具体事实承托？
