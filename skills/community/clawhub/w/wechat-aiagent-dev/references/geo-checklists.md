# GEO 数据、测试与上线清单

## 什么时候读取

用户问 GEO、结构化数据、自动模式兜底、测试、上线检查、怎么提高 AI 命中时，读取本文件。

---

## 一、GEO 的定义

这里的 GEO 不是传统搜索 SEO。它关注的是：小程序的商品、服务、门店、页面和接口，是否以 AI 容易理解、容易调用、容易确认的方式提供。

目标：

- 让 AI 知道你有什么
- 让 AI 知道什么时候应该推荐你
- 让 AI 知道调用哪个接口、填什么参数
- 让用户可以在对话里完成确认和转化

## 七、GEO 字段质量评分

每个字段对 AI 召回的贡献不同，优化时优先补高权重字段。

| 字段 | 权重 | 理由 |
|---|---|---|
| 别名 | ★★★★★ | 直接影响"用户说了能不能命中" |
| 场景词 | ★★★★ | 影响"模糊需求能不能推荐到你" |
| Agent 描述 | ★★★★ | 影响 AI 对商品/服务的理解准确度 |
| 标签 | ★★★ | 辅助分类和筛选 |
| 规格 | ★★★ | 影响参数匹配准确度 |
| 价格 | ★★ | 影响预算筛选 |
| 库存 | ★★ | 影响可售性判断 |
| 图片 | ★ | AI 不读图片内容（需文字描述） |

优化优先级：先补别名和场景词（ROI 最高），再补 Agent 描述和标签。

## 八、原子接口 description 检查

### 好接口描述的特征

- 首句说明业务对象
- 明确什么时候调用
- 明确不适合调用的情况
- ID 字段说明取值来源
- 缺省逻辑写在字段 description 里

### 反例

```json
{
  "name": "search",
  "description": "搜索数据"
}
```

### 推荐

```json
{
  "name": "searchDrinks",
  "description": "搜索饮品。用户明确说出饮品名称、品类或口味时调用；用户只表达场景如'清爽一点'时，应优先调用 recommendDrinks。"
}
```

### 官方最佳实践要点

- 接口名语义化：searchDrinks 优于 search
- 首句声明业务对象，而非入参
- 不写内部实现细节
- 职责单一，描述互不重叠
- 同名字段统一命名（drinkId 不要混用 itemId）
- ID 类字段声明取值来源接口
- 普通字段多举例 + 缺省处理

## 九、测试矩阵

### 通用测试问题（20条）

每个商家至少测试以下 20 类问题：

1. 明确商品："帮我点一杯冰美式。"
2. 模糊需求："想喝点清爽的。"
3. 场景需求："下午开会前喝点提神的。"
4. 价格约束："20 元以内有什么推荐？"
5. 规格修改："少冰、无糖。"
6. 门店/距离："附近能送吗？"
7. 优惠："我有什么券能用？"
8. 复购："再来一单上次买的。"
9. 缺货："这个没有了还有类似的吗？"
10. 售后/状态："我的订单到哪了？"
11. 语音输入："（语音）帮我点一杯拿铁。"
12. 多轮对话："换个大的。"（上下文：上轮选了中杯）
13. 跨店对比："A店和B店哪个便宜？"
14. 优惠叠加："这张券能和会员价一起用吗？"
15. 会员价："会员价多少？"
16. 库存不足替代推荐："M码没了怎么办？"
17. 改约取消："帮我改约到下周三。"
18. 评价投诉："服务不好怎么投诉？"
19. 退款："帮我退掉这个订单。"
20. 多模态："（发图片）这个款有吗？"

### 测试输出表

| 用户问题 | 期望调用 | 期望结果 | 是否通过 |
|---|---|---|---|
| 想喝点清爽的 | recommendDrinks | 推荐带场景标签的饮品 | 待测 |

### 升级版测试矩阵

| 用户表达 | 期望意图 | 期望接口 | 必填参数 | 期望返回 | 失败兜底 | 风险点 |
|---|---|---|---|---|---|---|
| 想喝点清爽的 | 场景推荐 | recommendDrinks | scene=清爽、storeId 如有 | 饮品推荐卡片 | 无结果时返回热销饮品 | 无 |
| 帮我点一杯冰美式少冰 | 下单前确认 | searchDrinks → getSkuOptions → createDrinkOrder | drinkId、ice、storeId、addressId | 订单确认卡 | 缺地址时引导补地址 | 地址、支付 |

## 十、上线检查

| 检查项 | 要点 |
|---|---|
| 用户确认 | 下单、预约、支付前必须让用户确认 |
| 支付 | 不能绕过微信支付确认 |
| 地址 | 获取地址前需要符合隐私授权要求 |
| 手机号 | 不明文展示敏感手机号，必要时脱敏 |
| 位置 | 获取位置要有明确用途 |
| 内容安全 | 用户输入和可生成内容要做安全处理 |
| 库存 | 下单前必须校验库存和门店状态 |
| 失败处理 | 无结果、缺地址、库存不足要有替代出口 |
| 自动模式 | 不把核心交易流程全部依赖文字链 |
| 数据更新 | 商品、库存、营业时间、优惠要能更新 |
| 文件大小 | SKILL.md ≤ 16KB、mcp.json ≤ 24KB、AGENTS.md ≤ 10KB |
| 接口注册 | index.js 中的接口名与 mcp.json 一致（含大小写） |
| 组件关联页 | 原子组件必须配置 relatedPage |
| content 写法 | 事实+动作两段式，失败时三要素（事实+出口+禁令） |

## 十一、自动模式兜底优化

没有开发资源时，先做：

1. 页面标题清楚，包含商品/服务对象
2. 商品和服务信息用文本展示，不只放图片
3. 价格、规格、库存、门店、营业时间、配送范围可被读取
4. 页面元数据写清页面用途和 query
5. FAQ、规则、服务说明整理为知识库（支持 PDF/DOC/TXT/MD/XLSX）
6. 关键页面之间路径清晰，避免用户跳转断裂

---

## 十二、复盘判断

如果测试命中差，优先检查：

- Skill 描述是否太泛
- 接口 description 是否重叠
- 字段是否缺少取值来源
- 商品是否缺少别名和场景标签
- 接口返回 content 是否只写动作，不写事实
- 无结果时是否没有兜底出口
- 信息源是否错位（接口功能写在 SKILL.md 而非 mcp.json description）
- ID 字段是否缺少来源声明（导致模型编造 ID）
