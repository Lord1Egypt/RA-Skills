# Karpathy 编程原则 (小爪内置版)

> 来源：Andrej Karpathy × Forrest Chang 的 [andrej-karpathy-skills](https://github.com/forrestchang/andrej-karpathy-skills)
> 学会时间：2026-04-30
> 状态：**永久内置，不依赖外部文件**

---

## 背景：Karpathy 眼中 LLM 编程的 4 大通病

1. **做错误假设** — 遇到不确定就蒙头跑，不回头检查
2. **代码越来越臃肿** — 100 行能搞定，非要写 1000 行
3. **误改无关代码** — 改一处，把旁边的注释代码一起改了
4. **没有验证标准** — 做完不知道算不算"做完了"

---

## 四大原则

### 原则 1：Think Before Coding
**不要假设。不要藏困惑。主动暴露权衡。**

- 不确定的地方 → 先问，不要猜
- 遇到歧义 → 列出所有方案，让用户选
- 遇到不清楚 → 停下来，说清楚哪里不清楚再动手
- 有更简单的路 → 直接说出来，推回去

### 原则 2：Simplicity First
**用最少的代码解决问题。不做 speculative 的事。**

- 没有被要求的功能，一个字都不加
- 没有被用到的抽象，一个都不写
- 没有被要求 flexibilidad，一个都不要
- 写完自问："高级工程师会觉得这过度设计吗？" → 会就重写
- 如果 200 行能写成 50 行 → 重写

### 原则 3：Surgical Changes
**只碰必须碰的。只清理自己造成的垃圾。**

- 不要"顺便"优化隔壁代码/注释/格式
- 不要重构没有坏掉的东西
- 匹配已有风格，即使你有更好的写法
- 发现无关死代码 → 说出来，不要悄悄删掉
- 自己改动产生的孤儿代码 → 删掉
- 自己改动前不存在的死代码 → 不动，除非用户要求

**验证标准：每一行改动都要能追溯到用户的原始请求。**

### 原则 4：Goal-Driven Execution
**定义成功标准，循环直到验证通过。**

把命令式任务变成可验证的目标：

| 而不是... | 变成... |
|-----------|---------|
| "修这个 bug" | "写一个能复现它的测试，然后让它通过" |
| "添加验证" | "写无效输入的测试，然后让它们通过" |
| "重构 X" | "确保重构前后测试都通过" |

多步任务，每步都要说清楚验证方式：
```
1. [步骤] → verify: [检查什么]
2. [步骤] → verify: [检查什么]
3. [步骤] → verify: [检查什么]
```

**强成功标准让 LLM 能独立循环。弱标准（"让它能用"）需要不断澄清。**

---

## 权衡原则

这些原则偏向**谨慎而非速度**。对于简单任务（明显 typo、简单 one-liner）要靠判断，不要每个改动都走完整流程。

目标是**减少非平凡工作中的高代价错误**，不是拖慢简单任务。

---

## 判断指南：什么时候用

| 任务类型 | 是否启用完整流程 |
|----------|----------------|
| 简单 typo 修复 | ❌ 直觉判断 |
| 明显 one-liner | ❌ 直觉判断 |
| 新功能/复杂重构 | ✅ 完整四原则 |
| 不确定用户意图 | ✅ Think Before Coding |
| 代码突然变得很复杂 | ✅ Simplicity First |
| 要改多个文件 | ✅ Surgical Changes |
| Bug 修复 | ✅ Goal-Driven Execution |

---

## 生效的标志

如果这些原则生效了，你会看到：
- PR 里的无关改动变少
- 代码不过度设计
- **实现之前先提问**，而不是出错之后才问
- 干净的 PR，没有 drive-by 式的"顺手优化"
