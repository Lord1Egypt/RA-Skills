# 拆解模式参考

本文档记录常见的任务拆解模式，供 Skill Factory 选择合适的拆解策略。

---

## 模式一：数据流水线（Pipeline）

**适用场景**：任务涉及数据采集 → 处理 → 输出的线性流程

**拆解方式**：
```
数据采集模块 → 数据处理模块 → 结果输出模块
```

**特点**：
- 模块间有顺序关系，但数据独立
- 每个模块输出是下一个模块的输入
- 适合数据分析、报告生成类任务

**示例**：
- 市场数据采集 → 数据分析 → 报告生成
- 用户行为收集 → 行为分析 → 个性化推荐

---

## 模式二：功能模块（Functional）

**适用场景**：任务包含多个可独立完成的功能点

**拆解方式**：
```
功能A模块 ┐
功能B模块 ├─ 并行执行
功能C模块 ┘
```

**特点**：
- 模块完全独立，无依赖
- 可并行执行
- 结果汇总由 Orchestrator 处理

**示例**：
- 天气查询 + 新闻推送 + 日程提醒
- 代码审查 + 文档生成 + 测试建议

---

## 模式三：阶段划分（Phase-based）

**适用场景**：任务有明显的时间或逻辑阶段

**拆解方式**：
```
阶段1（准备） → 阶段2（执行） → 阶段3（验收）
```

**特点**：
- 阶段间有依赖，但阶段内独立
- 每个阶段有明确的验收标准
- 适合项目管理、学习规划类任务

**示例**：
- 需求分析 → 方案设计 → 实施规划
- 基础学习 → 项目实战 → 技能认证

---

## 模式四：角色分工（Role-based）

**适用场景**：任务需要多个专业角色的协作

**拆解方式**：
```
角色A模块（专业领域A）
角色B模块（专业领域B）
角色C模块（专业领域C）
```

**特点**：
- 每个模块代表一个专业角色
- 模块间可能有协作关系（由 Orchestrator 协调）
- 适合多领域交叉任务

**示例**：
- 技术评估 + 商业分析 + 风险评估
- UI设计 + 前端开发 + 后端开发

---

## 模式五：层次分解（Hierarchical）

**适用场景**：任务有明确的层次结构

**拆解方式**：
```
顶层目标
  ├─ 子目标1模块
  ├─ 子目标2模块
  └─ 子目标3模块
```

**特点**：
- 子模块共同支撑顶层目标
- 子模块间相对独立
- 适合战略规划、系统设计类任务

**示例**：
- 产品规划 → 用户研究 + 竞品分析 + 功能规划
- 企业战略 → 市场战略 + 技术战略 + 组织战略

---

## 模式选择指南

| 任务特征 | 推荐模式 |
|---------|---------|
| 数据处理流程 | Pipeline |
| 多功能聚合 | Functional |
| 有明确阶段 | Phase-based |
| 多专业领域 | Role-based |
| 层次化目标 | Hierarchical |

---

## 组合模式

复杂任务可能需要组合多种模式：

**示例**：职业规划系统
```
阶段1：数据采集（Pipeline）
  └─ 市场数据 + 用户画像

阶段2：分析规划（Role-based）
  └─ 市场分析 + 能力评估 + 路径规划

阶段3：报告输出（Pipeline）
  └─ 报告生成 + 可视化
```

---

## 拆解检查清单

- [ ] 是否选择了合适的模式？
- [ ] 模块数量是否在 3~6 个？
- [ ] 模块间是否有不必要的依赖？
- [ ] 是否覆盖了用户目标的所有方面？
- [ ] 每个模块是否可独立验收？
