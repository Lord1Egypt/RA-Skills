# Token 消耗分析

## 各操作 Token 消耗表

| 操作 | Token 消耗 | 说明 |
|------|-----------|------|
| LLM 标题判断 | ~100 token/条 | 标题独立调用，单条非常轻量 |
| LLM 摘要生成 | ~1500 输入 + 500 输出 | BATCH_SIZE=1，每条公告独立调用 |
| 定期报告跳过 | 0 | 使用固定摘要 `【类型】标题`，不调 LLM |
| Digest 输出 | 0 | 从 DB 读取已有摘要，不调 LLM |
| 无新公告 | 0 | 直接跳过，无 API 调用 |

## 优化策略

### 1. 正则预过滤大幅减少 LLM 调用

13 条 `SKIP_CONTENT_PATTERNS` 在 Phase 1a 顺序执行，极速过滤掉公司章程、信用评级、募集说明书等纯模板文档。只有通过正则检查的公告才进入 Phase 1b 的 LLM 判断。

### 2. LLM 标题判断成本极低

每条公告仅用标题做判断（~100 token），不涉及正文。20 workers 并发处理，延迟极低。

### 3. 低价值公告不下载正文

LLM 判断为 non-valuable 的公告直接跳过 Phase 2（正文下载），节省 PDF 下载带宽和后续 LLM 摘要 token。

### 4. 摘要阶段跳过定期报告

`SKIP_LLM_TYPES` 包括 6 种定期报告类型：业绩预告、业绩快报、季度报告、半年报告、年度报告、补充更正。这些公告正文极长（年度报告可达数万字），直接用 `【类型】标题` 作为摘要，完全不调用 LLM。

### 5. BATCH_SIZE=1 避免超长上下文

BATCH_SIZE 固定为 1，每次 LLM 调用处理一条公告。虽然单次调用效率较低，但避免了超长上下文带来的 token 浪费和摘要质量下降。

### 6. Digest 不消耗 Token

`--digest` 模式直接从 DB 读取已有摘要做格式化输出，无任何 LLM 调用。

### 7. 按需增量处理

`--hours` 参数只处理最近 N 小时入库的公告，避免全量回溯。

### 8. 每日场景估算

假设每天 200 条新公告，其中 50% 通过正则过滤：
- Phase 1b LLM 标题判断：100 条 × ~100 token = ~10K token
- 其中 ~30% 被判断为 valuable（30 条）
- Phase 3 LLM 摘要生成：30 条 × ~2000 token = ~60K token
- 其中 6 条定期报告跳过 LLM：24 条 × ~2000 token = ~48K token

**日总消耗：~58K token**
**月总消耗（22 交易日）：~1.28M token**
