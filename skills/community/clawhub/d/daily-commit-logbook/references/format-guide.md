# Indonesian Logbook Formatting Guide

This guide provides standards for generating professional internship logbook entries in Indonesian.

## Entry Structure

Standard logbook entries follow this format:

```
**Tanggal:** [DD Month YYYY]

**Aktivitas:**
[Brief overview paragraph]

[Bulleted list of specific tasks]

**Hasil:**
[Bulleted list of outcomes/achievements]

**Jam Kerja:** [N] jam
```

## Activity Descriptions

### Backend Development

**Good:**
- Melakukan pengembangan fitur autentikasi pada sistem backend
- Perbaikan bug pada modul pelaporan untuk penanganan data
- Implementasi API endpoint untuk integrasi sistem

**Avoid:**
- "Coding" (too vague)
- "Fix bugs" (not descriptive enough)
- Technical jargon without context

### Common Commit Message Translations

English → Indonesian for common commit patterns:

- `fix:` → "Perbaikan pada..."
- `feat:` → "Pengembangan fitur..."
- `refactor:` → "Refaktorisasi kode pada..."
- `test:` → "Penambahan pengujian untuk..."
- `docs:` → "Pembaruan dokumentasi..."
- `chore:` → "Pemeliharaan dan optimasi..."

### Project-Specific Vocabulary

**moncube-backend:**
- report module → modul pelaporan
- cubicle authorization → otorisasi cubicle
- scope handling → penanganan scope data
- user entity context → konteks entitas pengguna

## Results Section

Standard results to include:

**Technical:**
- Code review dan testing berhasil dilakukan
- Implementasi berjalan sesuai rencana
- Unit test dan integration test berhasil
- Dokumentasi teknis diperbarui

**Process:**
- Koordinasi dengan tim pengembang
- Review requirement dan spesifikasi
- Diskusi solusi teknis

## Work Hours

Standard entry: **8 jam** (full day)

Adjust based on actual internship requirements.

## Examples by Project Type

### Backend API Development

```
**Tanggal:** 31 Maret 2026

**Aktivitas:**
Melakukan pengembangan dan perbaikan pada sistem backend. Aktivitas yang dilakukan meliputi:

- Perbaikan handling scope data untuk fitur pelaporan cubicle

**Hasil:**
- Bug authorization berhasil diperbaiki
- Fitur report berjalan stabil untuk semua level pengguna
- Code review dan dokumentasi diselesaikan

```

```
- Review dokumentasi requirement sistem

**Hasil:**
- Requirement system terdokumentasi dengan baik

```

## Tone Guidelines

- **Professional but not overly formal** - University internship context
- **Active voice** - "Melakukan...", "Mengimplementasikan..." not passive constructions
- **Specific but concise** - Include enough detail to be meaningful without being verbose
- **Achievement-oriented** - Focus on what was accomplished, not just what was attempted

## Common Mistakes to Avoid

❌ Too technical: "Refactored the authentication middleware to use JWT tokens"
✅ Professional: "Perbaikan sistem autentikasi untuk meningkatkan keamanan"

❌ Too vague: "Worked on the project"
✅ Specific: "Pengembangan fitur pelaporan pada modul backend"

❌ English mixed in: "Fix bug pada report system"
✅ Full Indonesian: "Perbaikan bug pada sistem pelaporan"
