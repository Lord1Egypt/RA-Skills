# Conversation History Summarization Spec

## Overview

When the chat assistant's session exceeds 10 conversation turns, automatically compress earlier history into a summary message, replacing the original message sequence. This reduces token usage for subsequent LLM calls while preserving enough context to prevent agent behavior degradation.

## Glossary

- **Conversation turn (Round)**: one user input + the agent's complete response (including all tool calls and their results) constitutes one turn.
- **History messages**: the message sequence persisted in the storage layer, read/written via `StoreHistoryMessages` / `GetHistoryMessages`.
- **Summary message**: a compressed text generated by an LLM, injected as a System-role message to replace the compressed original messages.
- **Retention window**: the most recent N turns of messages are excluded from compression and always kept in full.
- **Anchor messages**: system-level messages (e.g., user profile, environment context) that are never compressed and always retained.

## Behavioral Contracts

### Scenario 1: Turn count has not exceeded the threshold

- **Precondition**: the current session has ≤ 10 turns (default threshold).
- **Input**: user sends a new message, triggering the conversation flow.
- **Expected behavior**: no summarization is performed; history messages are passed to the agent unchanged, identical to current system behavior.
- **Postcondition**: message sequence is unchanged.

### Scenario 2: Turn count first exceeds the threshold — summarization triggered

- **Precondition**: the current session has > 10 turns, and no summary has been generated yet (or enough new turns have accumulated since the last summary).
- **Input**: user sends a new message; the message-building stage fetches history messages.
- **Expected behavior**:
  1. Count the number of conversation turns in history (by counting User-role messages).
  2. If turns > threshold, split the message sequence into two parts:
     - **Compression zone**: the first `(total turns − retention window)` turns.
     - **Retention zone**: the most recent `retention window` turns (default: 5), kept in full.
  3. Extract all non-anchor messages from the compression zone and call an LLM to generate a summary.
  4. Replace all non-anchor messages in the compression zone with a single System-role summary message.
  5. Final message sequence = anchor messages + summary message + retention zone messages + current user input.
- **Postcondition**:
  - The number of messages passed to the agent is significantly reduced.
  - The summarized message sequence is persisted back to the storage layer.
- **Error cases**:
  - Summarization LLM call fails: fall back to no compression, use full history, log the error, do not affect the main flow.
  - Summarization LLM call times out: set an independent timeout (recommended: 15s); fall back on timeout.

**Example**:

```
Original message sequence (12 turns, retention window = 5):
  [System: user profile]
  [System: environment context]
  [User: turn 1 input] [Assistant: turn 1 reply] [Assistant: tool_call] [Tool: tool result]
  [User: turn 2 input] [Assistant: turn 2 reply]
  ...
  [User: turn 7 input] [Assistant: turn 7 reply]  ← end of compression zone
  [User: turn 8 input] [Assistant: turn 8 reply]  ← start of retention zone
  ...
  [User: turn 12 input] [Assistant: turn 12 reply]

After compression:
  [System: user profile]                          ← anchor message, untouched
  [System: environment context]                   ← anchor message, untouched
  [System: Summary of earlier conversation: ...]  ← new summary message
  [User: turn 8 input] [Assistant: turn 8 reply]  ← retention zone, kept as-is
  ...
  [User: turn 12 input] [Assistant: turn 12 reply]
```

### Scenario 3: Conversation continues after an existing summary

- **Precondition**: history already contains a summary message; the user continues the conversation and the retention zone accumulates enough new turns to exceed the threshold again.
- **Input**: user sends a new message.
- **Expected behavior**:
  1. Identify the existing summary message.
  2. Feed the old summary + messages after the old summary but before the new retention window into the LLM to generate a merged summary.
  3. Replace the old summary + compressed messages with the new summary.
- **Postcondition**: there is always exactly one summary message in the message sequence.

### Scenario 4: Interaction with context cache

- **Precondition**: the current agent uses a session-level context cache.
- **Input**: summarization changes the message sequence.
- **Expected behavior**:
  1. After replacing messages, invalidate the context cache.
  2. The next LLM call rebuilds the cache from the updated message sequence.
- **Postcondition**: cache is consistent with the message sequence.

### Scenario 5: Interaction with middleware

- **Precondition**: summarization occurs during the message-building stage.
- **Input**: the pre-chat middleware needs to locate profile messages and sync tool messages.
- **Expected behavior**:
  1. Summarization does not compress anchor messages, so the pre-chat middleware's lookup logic is unaffected.
  2. Compressed tool messages no longer appear in the message sequence.
  3. The post-chat middleware persists the final message sequence (including the summary message) as usual.
- **Postcondition**: middleware behavior does not degrade.

### Scenario 6: Resource references in the summary

- **Precondition**: messages in the compression zone contain resource references (e.g., image variable names like `img_001.jpg`).
- **Input**: the summarization LLM processes messages containing resource references.
- **Expected behavior**:
  1. The summarization prompt instructs the LLM to preserve all resource variable names.
  2. The summary text records referenced resources in a list: "Resources referenced in earlier history: img_001.jpg, img_002.jpg, ..."
  3. The storage mapping from variable names to real URIs is unaffected and always available.
- **Postcondition**: the agent can still reference historical resources by variable name.
- **Error cases**: if the summarization LLM drops a resource variable name, the agent may be unable to reference that resource. This is an acceptable degradation — the user can re-upload.

## Constraints & Boundaries

### Performance
- The summarization LLM call should use a lightweight model to control cost.
- Summarization timeout cap: 15 seconds; fall back to no compression on timeout.
- Summarization must not block user-perceived response time — it should complete synchronously during the message-building stage.

### Business rules
- **Configurable thresholds**: the turn threshold (default: 10) and retention window (default: 5) should be delivered via a config center and adjustable at runtime.
- **Feature flag**: provide a global on/off switch via the config center.
- **No incremental summarization**: each trigger performs a full re-summarization of the compression zone — no incremental appending (keeps implementation simple and avoids quality degradation). [TBD: is incremental summarization needed to further reduce token usage?]

### What we don't do
- **No client protocol changes**: summarization is purely server-side; the client is unaware.
- **No impact on raw message storage**: the complete record of original user messages is unchanged; summarization only affects the history messages fed to the LLM.
- **No partial-turn summarization**: all tool calls and results within a single turn are treated as a unit — either fully retained or fully compressed.
- **No real-time token counting**: summarization is triggered by turn count, not exact token count, for simplicity. [TBD: is a token-count-based trigger strategy needed?]

## Dependencies

### Depends on
- **LLM service**: generates the summary text; requires a dedicated model endpoint.
- **Persistent storage**: reads/writes history messages; existing interface is sufficient.
- **Config center**: delivers threshold, retention window, and feature flag.

### Depended on by
- **Conversation main flow**: calls summarization logic during the message-building stage.
- **Agent middleware**: pre/post middleware must be compatible with the presence of a summary message.
