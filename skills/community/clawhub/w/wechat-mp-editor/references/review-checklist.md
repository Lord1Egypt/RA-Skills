# 编辑评审机制 (Editorial Review Checklist)

> 每次创建或更新草稿后，在**发布之前**逐项检查。这是从实战教训中沉淀的流程。

---

## 一、内容准确性检查（核心）

### □ 1.1 事实归因检查
- 文章里描述的特性/功能，**真的属于**正在介绍的 skill/项目吗？
- ❌ 反面案例：把 OpenClaw 内置的"做梦机制"、自己的实验"发呆机制"归因到了 Self-Improving Agent skill
- ✅ 问自己："这个功能是这个 skill 自带的，还是平台/其他工具的能力？"

### □ 1.2 概念边界检查
- skill 的核心是什么？文章有没有偏离主题去讲无关的能力？
- 如果涉及类比（如"带新人"），类比是否贯穿全文不跑题？

### □ 1.3 不存在的内容检查
- 检查全文是否包含**不应出现的关键词**（如做梦、发呆、REM、DMN）
- `grep -n "做梦\|发呆\|REM\|DMN" 文件名` — 跑一次确认

---

## 二、草稿同步检查

### □ 2.1 API 返回码验证
- API 返回 `{"errcode": 0, "errmsg": "ok"}` 只是**更新请求成功**，不保证内容正确
- 必须额外做**回读验证**

### □ 2.2 回读验证流程
更新草稿后，调用 `draft/get` 接口回读草稿内容，检查：
```python
# 必须执行的验证步骤
checks = {
    "title": "预期的标题",
    "做梦": False,      # 不应存在
    "发呆": False,      # 不应存在
    "预期出现的关键词": True  # 应该存在
}
```

### □ 2.3 本地文件与 API 一致性
- 如果修改了本地 HTML 后提交 API，确认修改确实生效了
- ❌ 反面案例：Python `str.replace(old, new)` 因为 `old` 字符串的 `line-height` 值与文件实际不符，替换失败但未报错
- ✅ 替换后立即 `grep` 验证新旧内容都符合预期

---

## 三、配图检查

### □ 3.1 配图清单
- 标题区插图（1张）：点明主题
- 正文插图（1-2张）：段落间隔，视觉化
- 结尾/升华插图（可选）

### □ 3.2 即梦出图流程
1. Agent 生成提示词 → 给用户
2. 用户在即梦官网生成（5.0 / 16:9 / 2K）
3. 用户给图片本地地址
4. Agent 上传微信 CDN → 获取 URL
5. 嵌入 HTML

### □ 3.3 配图提示词自检
- 提示词是否包含**不想让 AI 生成的内容**（如文字、标签）？
- 场景描述是否足够具体，避免抽象概念？
- 是否指定了画风（摄影/3D/插画/水彩等）？

---

## 四、排版检查

### □ 4.1 间距一致性
- 全文段落 `line-height` 统一（推荐 1.8，略紧凑）
- 段落间距 `margin-bottom` 统一（推荐 16px）
- section 间的 padding 一致

### □ 4.2 金句/强调格式
- 左对齐 + 左侧竖线，不要居中（避免晃眼）
- 引用卡片 `background:#f8f6f3; border-left:3px solid #d4a574;`

### □ 4.3 HTML 大小检查
- 保持在 15KB 以内（WeChat 限制约 64KB，但移动端越小越好）

---

## 五、检查清单执行流程

```
文章内容定稿
    ↓
□ 事实归因检查（1.1）
□ 概念边界检查（1.2）
□ 不存在内容 grep（1.3）
    ↓
提交 API 更新草稿
    ↓
□ API 返回 errcode 检查（2.1）
□ 回读验证（2.2）← 必须做！
□ 本地文件一致性（2.3）
    ↓
配图全部就位（3.1-3.3）
    ↓
排版检查（4.1-4.3）
    ↓
✅ 可发布
```

---

## 六、错误快速修复流程

如果发布后发现错误：

1. **立即回读草稿** — 调用 `draft/get` 确认问题范围
2. **修正本地文件**
3. **grep 验证** 新旧关键词
4. **调用 `draft/update`** 更新
5. **再次回读验证**
6. 如果已发布：调用 `/freepublish/delete` 删除 → 修正 → 重新提交
