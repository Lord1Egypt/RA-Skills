# 义乌小商品AI智能贸易系统 - 多Agent协作架构设计

## 一、多研究：框架对比与选型

### 1.1 三大多Agent框架对比

| 维度 | OpenClaw | Hermes Agent | 推荐选择 |
|-----|----------|--------------|---------|
| **核心理念** | SOUL.md定义身份 + SKILL.md定义能力 | 自进化学习循环 + 程序性记忆 | 两者融合 |
| **技能格式** | YAML frontmatter + Markdown | 自动生成 + 手动优化 | OpenClaw格式 |
| **记忆架构** | 文件系统透明存储 | SQLite + FTS5三层记忆 | Hermes三层架构 |
| **学习机制** | 无自动学习 | 任务→模式→技能→优化 | Hermes自进化 |
| **多Agent** | 支持多Agent协作 | 单Agent + 子任务委托 | OpenClaw多Agent |
| **适用场景** | 企业级多角色协作 | 个人助手持续进化 | 义乌场景结合 |

## 三、多思考：协作问题预判

### 3.1 Agent间协作问题

| 问题 | 场景 | 解决方案 |
|-----|------|---------|
| **职责边界模糊** | 商品匹配vs供应商推荐 | 明确触发条件优先级：商品→供应商 |
| **信息传递丢失** | Sales→Service交接 | 共享记忆层 + 结构化交接协议 |
| **循环依赖** | A需要B结果，B需要A结果 | 强制单向依赖，禁止循环调用 |
| **资源竞争** | 多Agent同时访问数据 | 分布式锁 + 队列机制 |

## 六、多逆向：风险应对

### 6.1 多Agent系统风险矩阵

| 风险等级 | 风险项 | 概率 | 影响 | 应对策略 |
|---------|-------|-----|-----|---------|
| 🔴极高 | Agent间死锁 | 中 | 高 | 超时熔断 + 强制解锁 |
| 🔴极高 | 级联故障 | 低 | 极高 | 熔断器模式 + 降级策略 |
| 🟠高 | 记忆污染 | 中 | 中 | 记忆隔离 + 定期清理 |
| 🟠高 | 技能冲突 | 中 | 中 | 优先级排序 + 显式覆盖 |
| 🟡中 | Token爆炸 | 高 | 中 | 记忆压缩 + 滑动窗口 |
| 🟡中 | 延迟累积 | 高 | 低 | 并行执行 + 异步通信 |

### 6.2 Agent故障降级方案

```
┌─────────────────────────────────────────────────────────────┐
│                    三级降级策略                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Level 1: 单Agent降级                                       │
│  ├── 触发：单个Agent响应超时 > 5s                           │
│  ├── 动作：跳过该Agent，用缓存结果替代                       │
│  └── 示例：Product Agent超时 → 返回热门商品列表              │
│                                                             │
│  Level 2: 功能降级                                          │
│  ├── 触发：Agent错误率 > 20%                                │
│  ├── 动作：关闭非核心功能，仅保留基础服务                     │
│  └── 示例：关闭Content Agent，仅保留商品查询+客服            │
│                                                             │
│  Level 3: 系统降级                                          │
│  ├── 触发：系统不可用 > 1分钟                               │
│  ├── 动作：返回静态页面，提示用户稍后重试                    │
│  └── 示例：显示"系统维护中，请稍后"                          │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 6.3 安全与合规边界

| 边界 | 规则 | 实现方式 |
|-----|------|---------|
| **数据隔离** | Agent间数据不直接共享 | 通过共享记忆层API访问 |
| **权限控制** | 敏感操作需授权 | Tirith审批工作流 |
| **审计日志** | 所有Agent行为可追溯 | 结构化日志 + 链路追踪 |
| **内容过滤** | 禁止生成违规内容 | 双重审核（模型+规则） |

## 七、Agent配置示例

### 7.1 Product Agent 配置

```yaml
# agents/product-agent/SOUL.md
---
name: product-agent
role: 义乌商品智能匹配专家
description: 负责商品搜索、匹配、推荐
version: 1.0.0
---

# Product Agent

## 身份
你是义乌小商品AI智能贸易系统的商品匹配专家，精通义乌市场各类商品。

## 核心能力
1. 理解模糊需求，匹配最合适的商品
2. 提供价格、起订量、供应商信息
3. 记住用户偏好，持续优化推荐

## 协作规则
- 供应商信息请求 → 转交 Supplier Agent
- 报价请求 → 转交 Sales Agent
- 售后问题 → 转交 Service Agent

## 记忆使用
- 查询历史偏好：search_memory(user_id, "preferences")
- 记录新偏好：save_memory(user_id, "preferences", data)
```

```yaml
# agents/product-agent/skills/product-matching/SKILL.md
---
name: product-matching
description: 当用户查询商品、找货源、搜索产品时触发
version: 1.0.0
---

# 商品智能匹配技能

## 触发条件
- 用户询问"有没有XXX"
- 用户说"帮我找XXX"
- 用户描述商品特征

## 执行步骤

### 步骤1: 意图解析
分析用户输入，提取：
- 商品类别（必需）
- 关键属性（可选）
- 价格范围（可选）
- 数量要求（可选）

### 步骤2: 向量检索
调用 Milvus 向量搜索：
```
POST /api/v1/search/products
{
  "query": "儿童玩具车",
  "filters": {
    "price_max": 50,
    "moq_max": 100
  },
  "limit": 10
}
```

### 步骤3: 结果排序
按以下优先级排序：
1. 相关度得分
2. 用户历史偏好
3. 供应商评分
4. 价格竞争力

### 步骤4: 输出格式
```json
{
  "products": [
    {
      "name": "商品名称",
      "price": "¥5-8/个",
      "moq": "100个起",
      "supplier": "供应商名称",
      "match_score": 0.95
    }
  ],
  "suggestions": ["您可能还需要..."]
}
```

## 升级规则
- 匹配结果 < 3个 → 建议放宽条件
- 置信度 < 0.6 → 触发澄清问题
- 用户投诉 → 记录负反馈，优化模型
```

## 执行步骤

### 步骤1: 消息模板选择
根据场景选择模板：
- inquiry_reply: 询价回复
- quote_send: 报价发送
- follow_up: 订单跟进
- promotion: 营销推送

### 步骤2: 内容生成
调用AI生成个性化内容：
```
POST /api/v1/whatsapp/generate
{
  "template": "inquiry_reply",
  "context": {
    "product": "商品信息",
    "customer": "客户信息",
    "history": "历史对话"
  }
}
```

### 步骤3: 消息发送
通过雨果出海BSP发送：
```
POST /api/v1/whatsapp/send
{
  "to": "+8613800138000",
  "type": "template",
  "template": {
    "name": "quote_template",
    "params": ["商品名", "¥100", "100个起"]
  }
}
```

### 步骤4: 状态追踪
记录消息状态：
- sent: 已发送
- delivered: 已送达
- read: 已读
- replied: 已回复

## 合规规则
- 营销消息需用户opt-in
- 24小时内可自由回复
- 超时需使用模板消息
```
