# 六大写作原则详解

本文档是 `SKILL.md` Phase 4 中"六大写作原则"的补充参考资料，包含更详细的说明和示例。

---

## 1. 以自我为中心

论文考核的是考生做了什么，不是项目多牛。必须清楚说明：
- 针对具体项目，自己所做事情的**由来**
- 遇到的**问题**
- **解决方法**和**实施效果**

### 具体要求

- 体现实际经验，不罗列课本内容
- 条理性地说明实际经验
- 写明项目开发体制和规模
- 明确"我"的工作任务和所起的作用
- 以"我"在项目中的贡献为重点
- 以"我"的努力（怎样做出贡献的）为中心

> **注意**：虽然强调"以我为中心"，但不要到处都是"我"。稍大的项目是集体劳动，建议适当用"我们"替代部分"我"。

### 示例对比

| ❌ 错误写法 | ✅ 正确写法 |
|-------------|-------------|
| 本系统采用了微服务架构 | 我主导设计了微服务拆分方案，将单体系统按业务边界拆分为 8 个微服务 |
| Redis 可以用来缓存数据 | 我在商品详情页引入了 Redis 缓存，将数据库查询压力降低了 70% |
| 系统需要考虑安全性 | 我设计了基于 JWT 的认证方案，并引入了 RBAC 权限模型 |

## 2. 站在架构师角度

从全局角度把握：
- 架构的优点及缺点
- 架构设计的方法和过程
- 各层次之间的接口设计
- 而非专注于某个具体的实现细节

### 具体体现

| 层次 | 需要关注的内容 |
|------|----------------|
| 业务架构 | 业务目标与战略、核心业务流程、业务能力模型 |
| 应用架构 | 子系统/模块职责及交互关系、服务划分 |
| 数据架构 | 数据存储策略、数据流动路径、读写分离方案 |
| 技术架构 | 技术选型及理由、质量属性权衡 |

> **避免**：花大量篇幅描述某个具体算法的实现，而忽略了架构层面的设计决策。

## 3. 忠实于论点

- 仔细阅读试题要求，正确理解题意
- 根据试题问题提取论点加以阐述
- 阐述时绝对服从论点，就试题问题展开
- 不要节外生枝，不要偏离论点

### 常见偏离论点的错误

1. **背范文**：拿到题目后不分析子问题，直接写准备好的内容
2. **过度展开技术细节**：花了 800 字介绍某个开源框架，但与试题问题关联不大
3. **遗漏子问题**：试题有 3 个子问题，只回答了 2 个

### 检查方法

在提纲阶段，将每个子问题映射到正文的一个或多个段落，确保：
- 子问题 1 → 段落 A、B
- 子问题 2 → 段落 C、D、E
- 子问题 3 → 段落 F

## 4. 条理清晰，开门见山

- 选定题目后，迅速整理素材，列出提纲
- 提纲不要求全面，关键列出**自己做过的工作**
- 项目概述要精练——背景、意义、规模、开发过程、自己的角色
- 每个自然段不超过 8 行，避免阅卷疲劳

### 项目概述模板（150 字以内）

```
某省银行委托我公司开发"智能风控系统"，该项目于 2023 年 3 月启动，
2024 年 1 月上线。我担任系统架构师，负责整体架构设计。
系统采用微服务架构，包含实时规则引擎、批处理分析、告警服务等
6 个子系统，日处理交易量约 500 万笔。
```

## 5. 标新立异，要有主见

- 不要刻意追求新奇，但要有自己的见解
- 让阅卷专家有耳目一新的感觉

### 如何体现"主见"

| 方法 | 说明 | 示例 |
|------|------|------|
| 提出权衡理由 | 说明为什么选 A 而不是 B | "虽然 Kafka 吞吐量更高，但考虑到团队对 RabbitMQ 更熟悉，且消息可靠性要求高于吞吐量，最终选择了 RabbitMQ" |
| 指出架构的不足 | 客观分析并说明改进方向 | "微服务架构提升了系统可维护性，但也引入了分布式事务的复杂性，我们采用了 Saga 模式来应对这一挑战" |
| 结合项目特点 | 不套用通用方案 | "考虑到本项目的强一致性要求，我们没有采用最终一致性的 Eventual Consistency 方案，而是采用了 TCC 事务" |

## 6. 表达书面化，段落转承自然

论文是正式书面文体，必须使用规范的技术写作语言。

### 书面化表达要点

- 使用完整句式，避免过于口语化的短句堆砌
- 以名词化结构替代动词短语，例如"我们对系统进行了重构"优于"我们重构了系统"
- 使用技术术语和行业惯用语，体现专业水准
- 避免"然后""所以""就是说"等口语化连接词
- 避免感叹号、省略号等非正式标点
- 引入概念时使用"所谓""是指""即"等书面引导词

### 段落转承技巧

- 段落之间使用过渡句承上启下，如"解决了上述问题后，另一个挑战随之而来""在确定了总体架构后，接下来需要关注模块间的接口设计"
- 每个段落开头先用一句话点明本段主旨，再展开论述
- 转折关系使用"然而""但与此同时""值得注意的是"
- 递进关系使用"在此基础上""更进一步""不仅如此"
- 因果关系使用"基于此""鉴于此""正是出于这样的考虑"
- 总结关系使用"综上所述""总的来看""从上述实践可以看出"

### 转承示例

```
（上一段讲了微服务拆分的背景）
然而，微服务架构在提升系统可维护性的同时，也引入了分布式事务这一
新的技术挑战。特别是在跨服务的数据一致性要求较高的场景下，
如何保证事务的 ACID 特性成为了架构设计的关键考量。
```

> 好的段落转承让文章读起来流畅自然，阅卷专家不会感到"跳跃感"。过渡句不需要华丽，关键在于逻辑顺畅。
