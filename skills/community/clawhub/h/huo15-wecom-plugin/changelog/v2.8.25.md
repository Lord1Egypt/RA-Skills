# v2.8.25 (2026-05-05)

**GUIDANCE 优先级翻转：MEDIA: 直发优先 + enhance_share_file 仅大文件兜底**

## 背景

赵博实测 v2.8.23 + v2.8.24 后说："让它优先直接发文件。如果文件过大再发链接给我。"

v2.8.22 当时为了规避 stream 截断 + 86008 群权限风险，把 GUIDANCE 改成"任何文件大小都优先 enhance_share_file 链接"。但：

- v2.8.23 修了群聊场景：`uploadAndReplyBotWsMedia` 在 `chattype=group` 走 `sendMediaMessage`（aibot_send_msg 主动推送通道），不再撞 86008 权限错误
- v2.8.24 修了 UI 锁：placeholder 120s timeout 用 `last=true` 终结 stream，企微 client 解锁交互

MEDIA: 路径现在**稳定可靠**。+ 用户体验上"群里直接收到附件" > "群里收到链接"——这是用户偏好而不仅是技术取舍。**翻转回 MEDIA: 优先**。

## 改动

仅 `index.ts` 的 `WECOM_BOT_WS_MEDIA_GUIDANCE` 文案重写——零代码逻辑变动，零 src/ 文件改动外的影响。

### 1. 决策表（核心）

```
| 文件类型/大小                          | 路径                          | 用户体验     |
|--------------------------------------|-----------------------------|------------|
| 图片 ≤ 10MB                          | MEDIA: 直发 ← 默认           | 群里直接收到 |
| 视频 ≤ 10MB                          | MEDIA: 直发 ← 默认           | 群里直接收到 |
| 语音 ≤ 2MB                           | MEDIA: 直发 ← 默认           | 群里直接收到 |
| 其他文件 ≤ 20MB                      | MEDIA: 直发 ← 默认           | 群里直接收到附件 |
| 超过上面阈值                          | enhance_share_file ← 兜底    | 群里收 markdown 链接 |
| 用户明确说「不要发链接」/「直接发文件」 | 强制 MEDIA: 直发 即使大文件   | 试试看，企微拒了再降级链接 |
```

### 2. ✅ 正确格式 + ❌ 错误反例（继承 v2.8.21）

正确（MEDIA: 独立成行）：
```
你要的文件已经准备好。

MEDIA: ~/.openclaw/media/outbound/report.pdf
```

错误（parser 抽不出来）：
- `📎 MEDIA: ~/foo.zip` ← 行首有 emoji
- `请查收 MEDIA: ~/foo.zip` ← 行首有正文
- `MEDIA: ~/foo.zip 已准备` ← 路径后有正文
- `> MEDIA: ~/foo.zip` ← markdown 引用前缀
- 整体被引号包住

### 3. 用户偏好覆盖

新增段落明示：当用户明确说「不要发链接」/「直接发文件」/「以后都直接发文件」时，**即使大文件也尝试 MEDIA: 直发**——企微 server 拒了 wecom 自动 share-fallback 降级到链接。这样用户至少看到了 LLM 在尝试遵守偏好。

**GUIDANCE 优先级随技术能力演进而变**：

- v2.8.x 早期（无 stream 解码）：MEDIA: 是唯一发文件方式，必须用
