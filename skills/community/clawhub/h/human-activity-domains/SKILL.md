# 人类活动领域穷尽分类体系

## 定位

本技能是一个**领域分类与负载物索引系统**，基于6×6矩阵穷尽分类所有人类活动领域，并为每个领域提供独立的负载物技能。

**核心理念**：人类活动可以按两个维度穷尽分类——活动性质（创造/转化/传递/维持/评估/消除）和作用对象（人/物/信息/资金/能量/权利），形成36个大域×3=108个母域。

## 核心能力

### 领域分类能力

- **6×6矩阵分类**：活动性质×作用对象=36个大域
- **四级层次结构**：总域→大域→母域→子域
- **108个母域**：每个大域细分3个母域

### 领域索引能力

- **快速定位**：根据需求快速定位到具体领域
- **负载物加载**：加载对应领域的负载物技能
- **跨域协作**：支持多个领域的组合使用

### 领域评估能力

- **R1-R5维度评估**：信息密度、创造性、交互性、规范性、迭代性
- **领域适合度评估**：边界清晰度、功能内聚性、独立价值
- **负载物生成建议**：适合度评分、生成优先级

## 人类活动领域特有维度

| 维度 | 说明 | 影响 |
|------|------|------|
| 活动性质 | 创造/转化/传递/维持/评估/决定 | 决定领域的核心动词和任务类型 |
| 作用对象 | 人/物/信息/资金/能量/权利 | 决定领域的处理对象和专业知识 |
| R1 信息密度 | 领域内信息量、知识密集程度 | 高信息密度→S和C权重高 |
| R2 创造性 | 对创新、原创性的需求 | 高创造性→A权重高 |
| R3 交互性 | 与人、外部系统的交互频率 | 高交互性→I权重高 |
| R4 规范性 | 受规则、标准、法规约束程度 | 高规范性→G权重高 |
| R5 迭代性 | 需要频繁迭代、改进的程度 | 高迭代性→循环多 |

## 四层结构

```
第一层：总域SKILL.md（本文件）→ 108母域索引
第二层：大域负载物 → 36个大域的分类定义
第三层：母域负载物 → 108个母域的具体技能
第四层：子域 → 每个母域内部的3个子域
```

## 分布式结构说明

本技能采用**分布式结构**而非标准三层结构，原因如下：

### 设计理由

1. **规模因素**：108个母域×4文件=432个文件，统一管理会导致单文件过大
2. **独立性需求**：每个母域是独立的领域技能，可单独使用和维护
3. **模块化更新**：单个母域的修改不影响其他母域
4. **按需加载**：只需加载目标母域的文件，减少内存占用

### 文件组织方式

```
references/
├── {大域序号}-{大域名称}-{母域序号}-{母域名称}-SKILL.md      # 母域技能定义
├── {大域序号}-{大域名称}-{母域序号}-{母域名称}-catalog.md     # 任务清单
├── {大域序号}-{大域名称}-{母域序号}-{母域名称}-requirements.md # 方法论要求
└── {大域序号}-{大域名称}-{母域序号}-{母域名称}-exemplars.md    # 范本库
```

### 索引机制

- **本文件（SKILL.md）**：作为总索引，包含108母域的完整列表
- **文件前缀**：每个母域有唯一的文件前缀，用于定位相关文件
- **命名规则**：`{大域序号}-{大域名称}-{母域序号}-{母域名称}-{文件类型}.md`

### 与标准三层结构的对应关系

| 标准三层结构 | 本技能分布式结构 | 说明 |
|--------------|------------------|------|
| catalog.md | 各母域独立的catalog.md | 按母域拆分，每个母域有自己的任务清单 |
| requirements.md | 各母域独立的requirements.md | 按母域拆分，每个母域有自己的方法论要求 |
| exemplars/ | 各母域独立的exemplars.md | 按母域拆分，每个母域有自己的范本库 |

## 使用规则

### 执行流程

1. **领域识别**：根据用户需求识别目标领域
2. **领域定位**：在6×6矩阵中定位到大域和母域
3. **负载物加载**：加载对应母域的SKILL.md文件
4. **任务执行**：按照母域的catalog和requirements执行任务

### 分类规则

- **穷尽性**：6×6矩阵覆盖所有人类活动，无遗漏
- **互斥性**：每个领域由"活动性质×作用对象"唯一确定
- **层次性**：总域→大域→母域→子域四级结构

### 命名规则

- **文件命名**：`{大域序号}-{大域名称}-{母域序号}-{母域名称}-{文件类型}.md`
- **示例**：`01-人才培养-01-教育培训-SKILL.md`

## 降级模式

本技能支持三种运行模式，根据用户权限、系统状态以及是否搭配 Meta Skill System 自动切换：

### 完整模式（默认）

**权限要求**：读写权限
**可用功能**：
- ✅ 领域识别与定位
- ✅ 负载物加载与执行
- ✅ 范本读取与参考
- ✅ 文件修改与创建
- ✅ 跨域协作

**搭配 Meta Skill System 时额外可用**：
- ✅ 执行框架（三轴正交：执行轴6大元操作+管线编排、内容轴清单法+样本法、创新轴4种模式+10种元框架）——最重要，可显著提升任务执行质量
- ✅ 领域负载物生成（生成新的母域技能）
- ✅ 领域评估（评估领域是否需要存在）
- ✅ 工作流重构（重构领域工作流）

**适用场景**：正常使用，所有功能可用

### 只读模式

**权限要求**：只读权限
**可用功能**：
- ✅ 领域识别与定位
- ✅ 负载物加载（只读）
- ✅ 范本读取与参考
- ❌ 文件修改与创建
- ⚠️ 跨域协作（只读）

**适用场景**：
- 系统维护期间
- 权限受限的用户
- 只需查询不需要修改的任务

### 受限模式

**权限要求**：部分权限
**可用功能**：
- ✅ 领域识别与定位
- ⚠️ 负载物加载（仅指定母域）
- ⚠️ 范本读取（仅指定母域）
- ❌ 文件修改与创建
- ❌ 跨域协作

**适用场景**：
- 特定领域的专家用户
- 项目隔离环境
- 安全敏感场景

### 模式切换规则

1. **自动降级**：当检测到权限不足时自动降级到相应模式
2. **手动切换**：用户可通过命令显式切换模式
3. **状态保持**：模式状态在整个会话期间保持
4. **恢复机制**：权限恢复后自动升级到完整模式

## 依赖关系

### 与 Meta Skill System 的关系

本技能是一个**领域分类与负载物索引系统**，而 Meta Skill System 是一个**元技能系统**，提供领域评估、工作流重构、领域负载物生成和通用任务执行能力。

**搭配使用效果最佳**：
- **执行框架**：Meta Skill System 提供三轴正交的执行框架（执行轴6大元操作+管线编排、内容轴清单法+样本法、创新轴4种模式+10种元框架），可显著提升任务执行质量。
- **领域负载物生成**：如需生成新的领域负载物（如新的母域技能），需要 Meta Skill System 的领域负载物生成能力。
- **领域评估**：如需评估某个领域是否需要存在，需要 Meta Skill System 的领域评估能力。
- **工作流重构**：如需重构某个领域的工作流，需要 Meta Skill System 的工作流重构能力。

**降级模式**：如果没有 Meta Skill System，本技能将仅提供领域分类和负载物索引功能，无法生成新的负载物或进行领域评估。

### 使用建议

1. **完整模式**：搭配 Meta Skill System 使用，可发挥全部能力。
2. **降级模式**：单独使用本技能，仅提供领域分类和负载物索引功能。

## 执行框架

### Step 0：领域识别

识别用户需求涉及的领域：
- 确定活动性质（创造/转化/传递/维持/评估/消除）
- 确定作用对象（人/物/信息/资金/能量/权利）
- 定位到大域和母域

### Step 1：负载物加载

加载对应母域的SKILL.md文件：
- 文件路径：`{大域序号}-{大域名称}-{母域序号}-{母域名称}-SKILL.md`
- 读取该领域的任务清单和执行框架

### Step 2：任务执行

按照母域的catalog和requirements执行任务：
- 读取catalog.md获取任务清单
- 读取requirements.md获取任务要求
- 按依赖关系执行任务

### Step 3：范本参考

如需范本，读取exemplars.md获取参考：
- 范本状态：待填充/可用
- 范本使用：模仿结构和风格，替换内容

### Step 4：整合交付

整合各母域的执行结果，形成完整交付物：

#### 整合内容

1. **任务执行结果**：汇总各母域的任务执行结果
2. **领域分析报告**：整合各领域的R1-R5评估结果
3. **跨域协作成果**：合并跨域协作的产出
4. **质量检查报告**：汇总各母域的质量检查结果

#### 整合流程

1. **结果收集**：收集各母域的执行结果
2. **冲突检测**：检测跨域结果中的冲突或不一致
3. **优先级排序**：根据领域重要性和紧急程度排序
4. **统一格式**：将结果转换为统一的输出格式
5. **最终审核**：对整合结果进行最终审核

#### 交付物格式

根据任务类型选择合适的交付物格式：

| 任务类型 | 交付物格式 | 说明 |
|----------|------------|------|
| 领域定位 | 定位报告 | 包含活动性质、作用对象、大域和母域定位结果 |
| 负载物加载 | 技能引用 | 包含目标母域的SKILL.md路径、catalog、requirements、exemplars |
| 跨域协作 | 协作报告 | 包含各域贡献、协作关系、整合结果 |
| 问题解决 | 解决方案 | 包含问题分析、涉及领域、解决方案、实施建议 |

#### 质量检查清单

整合交付前必须完成以下检查：

- [ ] 所有母域任务已完成
- [ ] 跨域结果无冲突
- [ ] 输出格式统一
- [ ] 敏感信息已脱敏
- [ ] 交付物完整性验证

## 依赖拓扑摘要

### 一、大域间依赖关系

#### 1. 活动性质维度依赖链

```
创造（01-06）→ 转化（07-12）→ 传递（13-18）→ 维持（19-24）→ 评估（25-30）→ 消除（31-36）
```

**依赖说明**：
- **创造→转化**：创造的产出是转化的输入（如产品研发→制造加工）
- **转化→传递**：转化的产出需要传递到目的地（如制造加工→物流运输）
- **传递→维持**：传递过程需要维持保障（如物流运输→设施维护）
- **维持→评估**：维持效果需要评估（如设施维护→质量检测）
- **评估→消除**：评估结果决定是否消除（如质量检测→废弃处理）

#### 2. 作用对象维度依赖链

```
人（01,07,13,19,25,31）↔ 物（02,08,14,20,26,32）↔ 信息（03,09,15,21,27,33）
资金（04,10,16,22,28,34）↔ 能量（05,11,17,23,29,35）↔ 权利（06,12,18,24,30,36）
```

**依赖说明**：
- **人↔物**：人需要物来生存和发展（如人才培养需要教育设施）
- **信息↔资金**：信息驱动资金流动（如金融信息→资金融通）
- **能量↔物**：能量驱动物的转化（如能源开发→制造加工）

## 跨域协作

### 跨域协作原则

1. **边界清晰**：明确每个母域的职责边界，避免职责重叠
2. **接口规范**：定义标准的输入输出接口，确保数据一致
3. **依赖明确**：明确跨域依赖关系，避免循环依赖
4. **协调机制**：建立跨域协调机制，解决冲突和问题

### 跨域协作模式

#### 1. 顺序协作模式

**适用场景**：任务有明确的先后顺序
**协作方式**：上游母域的输出作为下游母域的输入

**示例**：
```
工业设计（输出：设计方案）→ 技术研发（输入：设计方案，输出：技术方案）→ 机械制造（输入：技术方案）
```

**协作要点**：
- 上游必须在下游开始前完成
- 输出格式必须符合下游输入要求
- 建立中间检查点，确保质量

#### 2. 并行协作模式

**适用场景**：多个母域可以同时执行
**协作方式**：各母域独立执行，最后汇总结果

**示例**：
```
数据分析（并行）→ 数据可视化
信息处理（并行）→ 信息整合
知识管理（并行）→ 知识图谱
```

**协作要点**：
- 任务分解要合理，避免依赖冲突
- 建立统一的时间节点
- 设计好结果汇总机制

#### 3. 迭代协作模式

**适用场景**：需要反复迭代优化
**协作方式**：多个母域轮流执行，逐步优化

**示例**：
```
产品设计 ↔ 技术研发 ↔ 用户测试（循环迭代）
```

**协作要点**：
- 明确迭代终止条件
- 建立版本管理机制
- 记录每次迭代的改进点

#### 4. 聚合协作模式

**适用场景**：需要多个母域的成果汇总
**协作方式**：各母域独立执行，最后聚合为统一成果

**示例**：
```
人才评估（能力测评）┐
背景调查（背景核实）├─→ 综合评估报告
健康体检（健康状况）┘
```

**协作要点**：
- 明确各母域的贡献内容
- 设计统一的聚合规则
- 处理冲突和不一致

### 跨域协作流程

#### 阶段1：协作准备

1. **需求分析**：明确跨域协作的目标和范围
2. **母域识别**：识别需要协作的母域
3. **依赖分析**：分析母域间的依赖关系
4. **资源规划**：规划协作所需的资源

#### 阶段2：协作执行

1. **任务分配**：将任务分配给各母域
2. **并行执行**：各母域按计划执行任务
3. **进度同步**：定期同步执行进度
4. **问题协调**：及时协调解决协作问题

#### 阶段3：成果整合

1. **结果收集**：收集各母域的执行结果
2. **质量检查**：检查各母域成果的质量
3. **冲突解决**：解决结果中的冲突和不一致
4. **成果汇总**：将各母域成果汇总为统一成果

#### 阶段4：评审优化

1. **成果评审**：对整合成果进行评审
2. **问题识别**：识别协作中的问题
3. **改进措施**：制定改进措施
4. **经验总结**：总结协作经验

## 事实纪律

1. 分类必须基于6×6矩阵的逻辑穷尽性，不可遗漏或重叠
2. R1-R5评估必须基于客观标准，不可凭感觉
3. 负载物加载必须准确定位到目标领域
4. 跨域协作时必须明确领域边界和依赖关系
5. 范本内容必须脱敏，移除所有敏感信息
