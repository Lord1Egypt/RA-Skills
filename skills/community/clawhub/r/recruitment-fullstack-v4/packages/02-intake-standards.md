# 🚦 阶段 0：需求确认（强制准入）

## 设计哲学

> 不理解需求就出方案 = 算命
> 必须先对齐，再执行
> **阶段 0 不通过，禁止进入后续任何阶段**

---

## 0.1 准入检查清单

### 6 项必填项（缺一不可）

| # | 必填项 | 说明 | 验证标准 |
|---|--------|------|---------|
| 1 | 岗位名称 + 级别 | 如「高级Java工程师 P6」 | 必须含级别 |
| 2 | JD 全文或核心要求 | 至少 5 条 | 含硬技能 + 软技能 |
| 3 | 候选人简历完整文本 | PDF/文本均可 | 时间线连续 |
| 4 | 重点考察维度 | 技术/管理/文化/潜力 | 至少 3 个 |
| 5 | 决策场景 | 初筛/复试/终面/背调 | 影响输出深度 |
| 6 | 候选人数量 | 单个/批量 | 影响输出格式 |

### 缺失任何一项 → 强制追问

**禁止行为**：
- ❌ 用户说"帮我筛简历"就直接开始
- ❌ JD 模糊就自行补全
- ❌ 简历不全就推测内容
- ❌ 跳过任何一项必填检查

## 0.2 标准追问话术模板

### 单项缺失追问

```
为了给出可靠的评估结论，我需要先确认以下信息：

【缺失项 1】：[具体追问]
 示例：你提到要招"产品经理"，能否补充：
 - 级别是初级（P4）/中级（P5）/高级（P6）/总监级？
 - 主要负责 C 端还是 B 端产品？

【缺失项 2】：[具体追问]
 示例：JD 中提到"5 年经验"，能否说明：
 - 是大厂经验优先，还是创业公司经验也行？
 - 是否要求特定行业背景？

【缺失项 3】：[具体追问]

请补充后我再开始分析。
```

### 全部缺失时的开场追问

```
您好，要给您一份可靠的招聘分析报告，我需要先了解以下基础信息：

1. 【岗位信息】
 - 岗位名称是？
 - 级别是？
 - 直接汇报给谁？

2. 【岗位要求】
 - 您能分享完整 JD 吗？或者列出至少 5 条核心要求？
 - 这个岗位 12 个月内必须达成的关键成果是什么？

3. 【候选人信息】
 - 是单个候选人评估还是批量筛选？
 - 简历能否完整提供？

4. 【使用场景】
 - 当前处于哪个阶段？（简历初筛 / 面试准备 / 面试后评估 / 最终决策）
 - 您最希望我帮您解决什么具体问题？

5. 【重点关注】
 - 您最看重候选人的哪几个维度？
 - 有什么"一票否决"的红线？

请按以上信息回复，我会基于完整信息为您输出可靠分析。
```

## 0.3 岗位类型识别决策树（关键路由）

```
岗位类型识别流程：

┌─ Q1：岗位核心产出是"销售业绩/客户签约"？
│ YES → 【销售岗】→ 调用销售题库
│ NO → 下一题
│
├─ Q2：岗位核心产出是"代码/系统/技术方案"？
│ YES → 【技术岗】→ 调用技术题库
│ NO → 下一题
│
├─ Q3：岗位核心产出是"产品方案/用户增长/功能迭代"？
│ YES → 【产品岗】→ 调用产品题库
│ NO → 下一题
│
├─ Q4：岗位职责包含"直接管理 ≥3 人 + 业绩负责"？
│ YES → 【管理岗】→ 调用管理题库（可叠加专业题库）
│ NO → 下一题
│
├─ Q5：岗位属于 HR/财务/法务/行政/IT支持？
│ YES → 【职能岗】→ 调用职能题库
│ NO → 下一题
│
├─ Q6：候选人来自不同行业/不同职能？
│ YES → 【转行候选人】→ 叠加转行题库
│ NO → 下一题
│
└─ Q7：候选人是在校/应届/无全职经验？
 YES → 【实习/应届】→ 调用实习应届题库
 NO → 默认按【主岗位类型】处理
```

### 复合岗位处理

| 岗位示例 | 主题库 | 叠加题库 | 权重处理 |
|---------|--------|---------|---------|
| 销售总监 | 管理 | 销售 | 管理 60% + 销售 40% |
| 技术总监 | 管理 | 技术 | 管理 50% + 技术 50% |
| 产品总监 | 管理 | 产品 | 管理 50% + 产品 50% |
| 技术转产品 | 产品 | 转行 | 产品 70% + 转行 30% |
| 应届技术 | 实习应届 | 技术 | 应届 60% + 技术 40% |

## 0.4 职级识别决策树（关键路由）

```
职级识别流程：

┌─ 输入信号
│ • JD 中的级别标识（P4/P5/P6/M1...）
│ • 工作年限要求
│ • 是否需要带团队
│ • 决策权范围
│
├─ 判定为【低职级】的条件（满足任一即可）
│ • 应届/实习
│ • 工作年限 < 3 年
│ • 初级/中级标识（P4-P5、Junior、Mid）
│ • 不带团队 + 执行型岗位
│ • 薪酬区间在公司初中级
│ → 触发【2 类面试官矩阵】
│   - 专业面试官（60%）
│   - 通用面试官（40%）
│
├─ 判定为【高职级】的条件（满足任一即可）
│ • 高级/资深/专家标识（P6+、Senior、Staff、Principal）
│ • 工作年限 ≥ 5 年
│ • 需要带团队 ≥ 3 人
│ • 决策权 ≥ 部门级
│ • 总监/VP/C-level
│ → 触发【3 类面试官矩阵】
│   - 专业面试官（40%）
│   - 通用面试官（30%）
│   - 文化与人才发展面试官（30%）
│
└─ 模糊地带处理
 • 高级工程师（P6）：默认按高职级
 • 资深员工但无管理职责：按高职级 + 专业权重加重
 • 跨级招聘（外部 P5 招进来做 P6）：按高职级处理
 • 用户主动指定：以用户指定为准
```

## 1.2 单项评分标准（0-3 分制 + BARS 锚定）

> 简历评分用 0-3 分制（信息量有限）
> 面试评分用 1-5 分制（行为信号丰富，见阶段3）

| 得分 | 行为锚点 |
|------|---------|
| **3** | 简历有**明确证据 + 量化数据/具体案例** ✅<br>示例："2019-2024 在 BAT 做 Java 后端，主导 Spring Cloud 微服务改造，QPS 从 1000 提升至 8000" |
| **2** | 简历**有提及但缺少具体证据/数据** ⚠️<br>示例："熟悉微服务架构"（无具体项目说明） |
| **1** | **有相关经验但非直接对口**（可迁移）🟡<br>示例：要求 Java 微服务，候选人有 Python 微服务经验 |
| **0** | **完全缺失** ❌<br>示例：要求英语流利，简历无任何英语相关信息 |

## 1.7 阶段 1 局限性声明（强制）

```markdown
【本筛选报告的局限性】

1. 评估基础限制
 • 仅基于简历文本，工作经历真实性需背调验证
 • 简历是候选人自述，未经第三方核实

2. 评估准确率限制
 • 硬技能匹配准确率：约 70%（仍需面试技术验证）
 • 软技能评估准确率：约 30%（必须面试验证）
 • 文化匹配评估：无法仅凭简历判断

3. 评分置信度依赖
 • 简历信息完整度 → 影响置信度
 • 简历过简 → 置信度低 → 评分参考意义降低

4. 未做以下评估
 • 实际代码能力 / 实操能力
 • 沟通风格 / 抗压能力
 • 价值观与文化匹配
 • 历史业绩真实性

5. 建议补充验证
 • 阶段 2：针对风险点设计面试题
 • 阶段 3：通过 BARS 评分卡验证
 • 【背调已从V4.0主流程移除，如需请参考行业TORC框架】
```
