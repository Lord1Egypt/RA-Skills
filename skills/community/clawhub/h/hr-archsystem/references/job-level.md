# 职级体系设计

## 输入与输出

**前置依赖：** org-design 产出的职类序列
**输入：** 组织规模、行业特性、职类序列、现有职级体系（如有）
**输出：** 《职级序列表》—— 职级数量、级间距、晋升通道、宽带薪酬映射

---

## Step 1：职级数量决策

职级数量由两个因素决定：

```
组织成熟度：
  创业期/快速增长 → 职级少（L1-L3），快速晋升通道
  稳定期 → 职级多（L1-L9），细分层级
  衰退期/转型期 → 重新定义

岗位复杂度：
  单一业务 → 4-6级足够
  多元业务/职能复杂 → 7-9级
```

**参考标准（知识型组织）：**

| 组织类型 | 建议职级数 | 典型使用场景 |
|---------|----------|------------|
| 扁平型 | 3-4级 | 小型创业团队（<50人）|
| 标准型 | 5-6级 | 成长期公司（50-500人）|
| 精细型 | 7-9级 | 大型组织/集团（500人以上）|

## Step 2：典型职级序列表（L1-L9）

### M 管理序列

| 级别 | 通用职级名称 | 核心标志 |
|------|------------|---------|
| M1 | 主管/Team Lead | 带1-5人，负责小组执行 |
| M2 | 经理 | 带5-15人，负责部门/职能模块 |
| M3 | 高级经理 | 带10-30人，跨职能协调 |
| M4 | 总监 | 带20-50人，对业务线负责 |
| M5 | 高级总监 | 负责多条业务线或重要职能 |
| M6 | VP / 副总裁 | 负责大区/事业部，战略参与 |
| M7 | SVP / 高级副总裁 | 负责集团核心板块 |
| M8 | CRO/COO/CFO等CXO | 向CEO汇报，集团决策层 |
| M9 | CEO | 最高决策者 |

### P 专业序列（以科技/产品为例）

| 级别 | 通用职级名称 | 核心标志 |
|------|------------|---------|
| P1 | 初级专员 | 在指导下执行任务 |
| P2 | 中级专员 | 独立完成分配任务 |
| P3 | 高级专员/Senior | 独立承担完整模块 |
| P4 | 专家/Lead | 技术/专业决策，辅导P1-P3 |
| P5 | 高级专家 | 跨模块影响，方法论建设 |
| P6 | 首席专家/Principal | 领域级影响力，标准制定者 |
| P7 | 资深首席/Distinguished | 公司级影响力 |
| P8 | Fellow | 行业级影响力 |
| P9 | 学术 Fellow | 行业标杆，如ACM Fellow |

## Step 3：双通道模型

建立管理序列（M）和专业序列（P）的等价值映射：

```
P4 = M1（专业影响力范围 ≈ 主管管理幅度）
P5 = M2（专业深度 = 经理管理幅度）
P6 = M3（跨模块 = 高级经理）
P7 = M4（业务线级 = 总监）
```

**为什么要双通道：**
- 不是所有人都想做管理
- 顶级专家的价值可以远超管理者
- 防止"彼得原理"（晋升到不胜任位置）

**设计检查：**
```
□ P通道顶端是否等于M通道顶端？（否则P到顶了就只能转M）
□ 同一级别内，P和M的薪酬范围是否大致对齐？
□ 是否有明确的P↔M转换机制（通常需要申请+评估）？
```

## Step 4：宽带薪酬设计

每个职级的薪酬带宽公式：

```
带宽 = (最大值 - 最小值) / 最小值

建议带宽：
- L1-L3（基础层）：带宽 40-60%（技能学习期，差异小）
- L4-L6（骨干层）：带宽 50-80%（能力分化期）
- L7-L9（战略层）：带宽 80-120%（高度差异化）

重叠度：相邻两级之间建议重叠 10-20%
（重叠使晋升不因薪酬倒挂而卡死）
```

## Step 5：职级名称标准化

原则：内部职级名称 ≠ 市场上流通的Title（避免市场溢价/折价干扰）

```
内部职级体系（实际管理用）
    ↓ 翻译
市场Title（对外招聘用，对标市场）
```

## Step 6：晋升机制

### 晋升条件（通用标准）

| 条件类型 | 说明 |
|---------|------|
| **时间门槛** | 在当前级别至少N年（建议：L1-3至少1年，L4-6至少2年，L7+至少3年）|
| **绩效门槛** | 连续2年绩效在Top X%（建议：前15-20%）|
| **能力门槛** | 满足下一级别胜任力要求（证据支撑）|
| **编制门槛** | 下一级别有HC空缺 |

### 晋升评估频率
- L1-L4：每半年评估
- L5-L7：每年评估
- L8-L9：不定期，通常配合重大业务变化

## Step 7：AI对职级体系的冲击

### 7.1 新增"AI杠杆率"维度

每个岗位的职级现在都要考虑"AI杠杆率"：

```
AI杠杆率 = 该岗位用AI放大的产出 / 该岗位总产出

- 低杠杆（<20%）：执行层，AI作为工具  → 传统职级
- 中杠杆（20-50%）：分析师/策划层  → 职级不变，但胜任力标准变化
- 高杠杆（>50%）：设计/策略层  → 新职级命名可能产生
- 极高杠杆（>80%）：创新/架构层  → AI原生岗位，传统无法对标
```

### 7.2 典型AI原生职级

| 新职级 | 定义 | 对标传统 |
|--------|------|---------|
| Prompt Engineer | 专精AI交互设计 | UX设计师 |
| AI Trainer / Finetuner | 领域模型微调 | 算法工程师 |
| AI Operator | AI工具链管理+效能优化 | 运维工程师 |
| AI Ethics Specialist | AI伦理审查 | 法务/合规 |
| Head of AI Productivity | 组织AI效能管理 | COO |

### 7.3 职级压缩风险

AI加速了低级别工作，导致：
- 部分L1-L2岗位消失或被合并
- L3-L4的能力要求整体上移
- 晋升通道"缩短"但"加宽"
