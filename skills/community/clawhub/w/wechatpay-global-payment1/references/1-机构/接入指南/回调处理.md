# 机构模式 回调处理（境外）

> 本文档**完全按照境外（跨境）官方文档独立编写**，不沿用境内模板。
>
> 文档来源：[境外回调报文解密](https://pay.weixin.qq.com/doc/global/v3/zh/4012354990.md)、[回调通知注意事项](https://pay.weixin.qq.com/doc/global/v3/zh/4016271743.md)、各业务「设置事件通知地址」与「回调通知」API 文档。
>
> ‼️ 本skill以"机构"角色视角描述，回调中必带 `sp_mchid` + `sub_mchid`，业务侧**必须**按 `sub_mchid` 路由。

## 一、机构维度的核心约束

| 约束 | 说明 |
|---|---|
| **回调地址机构维度配置** |  下单时传入；机构维度，所有子商户共用同一回调地址 |
| **必须按 `sub_mchid` 路由** | 同一机构下 N 个子商户共用一个回调，业务侧必须按 `sub_mchid` 路由到对应子商户上下文，**否则会出现"A 子商户的订单被 B 子商户业务处理"的串单事故** |
| **APIv3 密钥机构维度** | 机构维度配置一把 32 字节密钥，机构所有子商户产生的回调都用这把密钥 AEAD 解密 |
| **平台证书机构维度** | 用机构 API 证书 + APIv3 密钥下载 `GET /v3/global/certificates` 得到平台证书，缓存复用 |

## 二、前提条件

### 2.1 必须先做的 5 件事

1. **设置 APIv3 密钥**（32 字节）：登录境外商户平台（机构入口）→ 「商户密钥 → APIv3 密钥」设置；**未设置不会收到任何回调**。
2. **配置业务通知地址**：在下单接口的 `notify_url` 字段传入；机构维度，所有子商户共用。
3. **下载并缓存平台证书**：调用 `GET https://apihk.mch.weixin.qq.com/v3/global/certificates`，按响应头 `Wechatpay-Serial` 自动刷新缓存。
4. **设计 `sub_mchid` 路由表**：维护子商户档案表（`sub_mchid` ↔ 业务上下文 ）。
5. **检查代理 / CDN**：CloudFront / Cloudflare 等会过滤 `Wechatpay-*` 4 个头 → 显式放行 / 应用层直连。

### 2.2 通知地址限制

| 限制 | 说明 |
|---|---|
| **协议** | HTTPS（不接受 HTTP） |
| **网络** | 公网可达，**不允许内网地址** |
| **ICP 备案** | **不需要**（境外服务器在中国大陆境外） |
| **URL 前后空格** | 不允许，会导致 DNS 解析失败 |
| **登录态拦截** | 通知 URL 不能挂在登录鉴权中间件后面 |

## 三、回调报文结构

### 3.1 整体结构

```json
{
  "id": "EV-2018022511223320873",
  "create_time": "2025-08-02T00:00:00+08:00",
  "event_type": "TRANSACTION.SUCCESS",
  "resource_type": "encrypt-resource",
  "summary": "支付成功",
  "resource": {
    "original_type": "transaction",
    "algorithm": "AEAD_AES_256_GCM",
    "ciphertext": "BASE64(...)",
    "associated_data": "transaction",
    "nonce": "..."
  }
}
```

### 3.2 解密 `resource.ciphertext` 后的明文

以 JSAPI 支付结果通知为例：

```json
{
  "sp_mchid": "1230000109",
  "sub_mchid": "1900000001",
  "sp_appid": "wx426a3015555a46be",
  "sub_appid": "wx2421b1c4370ec43b",
  "out_trade_no": "1217752501201407033233368018",
  "transaction_id": "4200000349202508025069555683",
  "trade_type": "JSAPI",
  "trade_state": "SUCCESS",
  "trade_state_desc": "支付成功",
  "bank_type": "OTHERS",
  "success_time": "2025-08-02T10:43:39+08:00",
  "payer": {
    "sp_openid": "...",
    "sub_openid": "..."
  },
  "amount": {
    "total": 100,
    "payer_total": 100,
    "currency": "USD",
    "payer_currency": "USD"
  }
}
```

‼️ 机构回调的关键字段：

- **`sp_mchid` / `sub_mchid`**：必须按 `sub_mchid` 路由
- **`payer.sp_openid` / `payer.sub_openid`**：取下单时使用的那个（业务侧应保持下单与回调使用同一个 OpenID 字段）
- **`amount.currency` 必带**：按子商户 + 币种维度入账

## 四、处理流程（顺序不能错）

```
HTTP 请求到达 notify_url
  ↓
① 取请求头 Wechatpay-Timestamp / Nonce / Signature / Serial
  ↓
② 用平台证书（按 Serial 匹配）验签：3 行验签串（Timestamp\n + Nonce\n + Body\n）
  ↓
③ 验签失败 → 直接 401 / 5xx 拒绝
  ↓
④ 验签成功 → 用机构 APIv3 密钥 AEAD-AES-256-GCM 解密 resource.ciphertext
  ↓
⑤ 取明文里的 sp_mchid 校验是本机构；取 sub_mchid 路由到子商户上下文（必备步骤）
  ↓
⑥ 用 (sub_mchid, id) 或 (sub_mchid, out_trade_no, event_type) 做幂等查询：
   - 已处理过 → 直接回 200
   - 未处理 → 加锁，按子商户维度记录订单状态 → 提交事务 → 回 200
  ↓
⑦ 处理超时（>5s）或异常 → 立即回 200，告警 + 异步重试
```

## 五、关键代码点

> ‼️ **本版 Skill 不内置代码文件**，下面只描述实现要点。落地代码请用官方 SDK，仅需把 baseUrl 改为 `https://apihk.mch.weixin.qq.com`。

### 5.1 验签

拼接 3 行串 → 用平台公钥（按 `Wechatpay-Serial` 匹配）做 RSA-SHA256 验签。

### 5.2 解密

用机构 APIv3 密钥 AEAD-AES-256-GCM 解密。**机构维度的 APIv3 密钥与境内 / 任何其他环境都不同，须独立保存**。

### 5.3 子商户路由（机构必做）

```
1. 解密得到明文 plaintext
2. 校验 plaintext.sp_mchid == 本机构 sp_mchid（不一致 → 拒绝）
3. sub_mchid = plaintext.sub_mchid
4. 查子商户档案表：merchant_ctx = SELECT * FROM sub_merchants WHERE sub_mchid = ?
5. 如果 merchant_ctx 为空 → 告警 + 拒绝（可能是新进件子商户尚未落本地表）
6. 用 merchant_ctx.account_currency 与 plaintext.amount.currency 比对
   - 不一致 → 告警，按 plaintext.amount.currency 入账
7. 入账到 (sub_mchid, currency) 维度账户
```

‼️ **未做按 `sub_mchid` 路由 = 串单事故**。这一步是机构回调最关键的安全约束。

### 5.4 幂等

- **主键候选**：`id`（顶层通知 ID）/ `(sub_mchid, transaction_id)` / `(sub_mchid, out_trade_no, event_type)`。
- 实现：DB 唯一索引 + `INSERT IGNORE`，或 Redis `SET NX EX`。

## 六、处理要求

| # | 要求 | 说明 |
|---|---|---|
| 1 | **必须返回 HTTP 2XX**（200 / 204）| 否则按指数退避重试 |
| 2 | **必须 5 秒内应答** | 建议先入消息队列再异步处理 |
| 3 | **必须做幂等去重 + 子商户路由** | 双重保障 |
| 4 | **必须先验签后解密再路由** | 顺序不能错；解密后**必须校验 `sp_mchid` 是本机构** |
| 5 | **探测流量直接 200** | 签名以 `WECHATPAY/SIGNTEST/` 开头是探测流量 |
| 6 | **业务异常也建议 200** | 非 200 会触发回调重试，可能放大问题 |
| 7 | **多币种入账（按子商户）** | 按 (sub_mchid, currency) 维度路由账户 |
| 8 | **回调与查单双保险** | 不依赖回调一次性到达，必须结合查单接口确认 |

## 七、各业务回调汇总

| 业务 | 配置方式 | 解密后明文关键字段 |
|---|---|---|
| 支付结果通知 | 下单时 `notify_url` 字段 / 设置事件通知地址 API | `sp_mchid` / `sub_mchid` / `out_trade_no` / `transaction_id` / `trade_state` / `amount.{total,currency}` |
| 退款通知 | 通过【申请退款API】中提交的参数`notify_url`设置 | `sp_mchid` / `sub_mchid` / `out_trade_no` / `out_refund_no` / `refund_status` |
| 委托代扣签约 / 解约通知 | 业务设置 | `sp_mchid` / `sub_mchid` / `contract_id` / `contract_state` |
| 委托代扣扣款通知 | 业务设置 | 同支付结果通知 + `sub_mchid` |
| 跨境分账退款通知 | 设置事件通知地址 API | `sp_mchid` / `sub_mchid` / `out_order_no` / `refund_state` |

## 八、跨境部署 8 大踩雷

1. **代理 / CDN 过滤 `Wechatpay-*` 4 个头**：CloudFront / Cloudflare 默认清理 → 显式放行 / 应用直连。
2. **签名头大小写**：HTTP 头不区分大小写，取值用你 HTTP 库的方式即可。
3. **请求体编码不一致**：raw body 必须与 `Wechatpay-Signature` 签名时完全一致；不要先解析 JSON 再序列化。
4. **APIv3 密钥混用**：机构境内 / 境外 / 不同环境的 APIv3 密钥必须分开管理，混用会"解密出乱码"。
5. **服务器时区**：跨境节点时差超 5 分钟问题（虽然回调验签不强校验时间，但 `success_time` 比对会出问题）。
6. **`notify_url` 加了登录态校验**：必须在路由层把 `notify_url` 排除登录鉴权。
7. **未做 `sub_mchid` 路由 = 串单事故**：单租户思维写代码 → A 子商户的订单到 B 子商户系统。
8. **多业务共用同一通知地址但缺乏分发**：不同业务的明文结构不同（如 H5 权限申请回调 vs 支付回调），通知地址里要先按 `event_type` / `original_type` 分发到不同处理器。

## 九、与境内的差异速查

| 项 | 境内（机构 / 服务商） | 境外（机构）|
|---|---|---|
| APIv3 密钥 | 商户维度 / 服务商维度 | **机构维度**，全局唯一 |
| 平台证书路径 | `/v3/certificates` | `/v3/global/certificates` |
| `notify_url` ICP 备案 | 必须 | **不需要** |
| 多币种 | 默认 CNY | **必带 `amount.currency`**，取值默认为sp_mchid的结算币种或CNY |
| 解密算法 | AEAD-AES-256-GCM | 同 |
| 幂等键 | `(out_trade_no)` 等 | **必须加 `sub_mchid` 维度**：`(sub_mchid, out_trade_no, event_type)` |
