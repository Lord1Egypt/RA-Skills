# Activity Type Catalog — Dataverse Classic Workflows

A reference of every activity type you can encounter in classic workflow XAML,
grouped by category, with usage guidance and a minimal template for each.

> All examples use the OOB `account` entity or the synthetic `sample_widget`
> entity. Replace these with your real entities when applying.
>
> For the WF4-vs-Dataverse subset breakdown (which `System.Activities.*` types
> are usable in Classic Workflows and which are not), see
> [`web-research.md` §1](./web-research.md#1-engine-substrate--whats-wf4-and-whats-not).
> For the binding rules custom `CodeActivity` workflow assemblies must follow
> (.NET Framework 4.6.2 target, **stateless** `Execute`, `IWorkflowContext`
> access), see [`web-research.md` §9](./web-research.md#9-custom-workflow-activities--the-binding-rules).

## 1. Workflow structure

### `mxswa:Workflow` (root)

The single child of `mxswa:Activity`. Carries trigger / scope / mode /
run-as / primary entity. See [xaml-anatomy.md §1](./xaml-anatomy.md#1-the-root-element).

### `Sequence`

Plain WF4 sequence container. Holds a list of activities executed in order.
The body of `mxswa:Workflow` is always a single `Sequence`. Each branch of a
`ConditionBranch` is also a `Sequence`.

```xml
<Sequence>
  <!-- ordered child activities -->
</Sequence>
```

## 2. Data operations

### `mxswa:CreateEntity` — Create a new record

```xml
<mxswa:CreateEntity
    DisplayName="Create follow-up account"
    EntityName="account"
    EntityId="{x:Null}">
  <mxswa:CreateEntity.Entity>
    <!-- An InArgument<Entity> built from SetEntityProperty steps -->
  </mxswa:CreateEntity.Entity>
</mxswa:CreateEntity>
```

Pattern: precede with one `mxswa:SetEntityProperty` per field to assign,
then `CreateEntity` commits the in-memory entity.

### `mxswa:UpdateEntity` — Update an existing record

```xml
<mxswa:UpdateEntity
    DisplayName="Mark account as reviewed"
    EntityName="account">
  <mxswa:UpdateEntity.Entity>
    <!-- Reference to the entity variable being updated -->
  </mxswa:UpdateEntity.Entity>
</mxswa:UpdateEntity>
```

The `EntityName` is the logical name. The entity reference is typically the
`PrimaryEntity` of the workflow (the triggering record) but can be any
related record retrieved via `GetEntityProperty` traversal.

### `mxswa:AssignEntity` — Change owner

```xml
<mxswa:AssignEntity
    DisplayName="Assign to manager"
    EntityName="account"
    EntityId="[primaryEntityId]"
    OwnerId="[managerSystemUserId]"
    OwnerEntityName="systemuser" />
```

`OwnerEntityName` is `systemuser` or `team`.

### `mxswa:SetState` — Change statecode/statuscode

```xml
<mxswa:SetState
    DisplayName="Set status to Inactive"
    EntityName="account"
    EntityId="[primaryEntityId]"
    State="1"
    Status="2" />
```

`State` is the statecode integer (0=Active, 1=Inactive for most entities).
`Status` is the statuscode integer (entity-specific). Always look up the
real values for the user's entity — guessing produces broken workflows.

### `mxswa:RetrieveEntity` — Look up by ID (internal helper)

Used to load a record into a workflow variable so subsequent steps can read
its properties. Almost always wrapped or generated by the designer; you
rarely write this by hand.

### `mxswa:GetEntityProperty` — Read a single field

Loads a single field value into a variable for use in expressions. Frequently
chained: traverse a lookup, then read a property of the related record.

```xml
<mxswa:GetEntityProperty
    EntityName="account"
    Attribute="primarycontactid"
    Entity="[triggerEntity]"
    Value="[primaryContactRef]" />
```

### `mxswa:SetEntityProperty` — Write a single field on the in-memory entity

Stages a value into an in-memory entity (for use by a later `CreateEntity`
or `UpdateEntity`). Does **not** commit — the commit happens in the
`CreateEntity` / `UpdateEntity` activity that follows.

### `mxswa:Persist` — Commit checkpoint

Background workflows only. Forces a persistence checkpoint so a long-running
workflow can be safely paused/resumed. **Not allowed in real-time mode.**

## 3. Communication

### `mxswa:SendEmail` — Send an email

Two patterns:

**Pattern A — send an email created by a previous CreateEntity step:**
```xml
<mxswa:SendEmail
    DisplayName="Send approval email"
    EmailId="[createdEmailId]" />
```

**Pattern B — create + send in one step (designer macro):**
The designer expands "Send Email" into a CreateEntity for an `email` record,
sets `from`, `to`, `subject`, `description`, then calls `SendEmail` with the
new email id.

When generating SendEmail steps, prefer Pattern B (the designer-natural
shape) — that's what users will see when they open the workflow in the
classic designer.

### `mxswa:StartChildWorkflow` — Trigger another workflow

```xml
<mxswa:StartChildWorkflow
    DisplayName="Trigger approval subprocess"
    WorkflowId="aaaaaaaa-aaaa-aaaa-aaaa-aaaaaaaaaaaa"
    EntityId="[primaryEntityId]"
    EntityName="account" />
```

The `WorkflowId` is the GUID of the child workflow. The child workflow must
be on the same primary entity (or accept the entity reference as input).

## 4. Conditions and branching

> All of these are wrapped in `<mxswa:ActivityReference AssemblyQualifiedName="…">`.
> See [xaml-anatomy.md §2b](./xaml-anatomy.md#2b-internal-control-flow-activities--wrapped-in-activityreference).

### Canonical class names you will see in `AssemblyQualifiedName`

Match the class portion (everything before the first comma) case-sensitively.
The `Version`, `Culture`, and `PublicKeyToken` portions vary across Dataverse
builds and on-premises versions — do not match on those.

| Class name (match this part) | What it represents | User-visible? |
|---|---|---|
| `Microsoft.Crm.Workflow.Activities.ConditionSequence` | Whole `if/elseif/else` block (or Wait Condition when `Wait=[True]`) | Yes — "Check Condition" or "Wait Condition" |
| `Microsoft.Crm.Workflow.Activities.ConditionBranch` | Single branch (True / ElseIf / Else) of a ConditionSequence | No — collapse into parent |
| `Microsoft.Crm.Workflow.Activities.Composite` | Named subtree / step group | Yes when it has a `DisplayName`; otherwise no |
| `Microsoft.Crm.Workflow.Activities.EvaluateCondition` | Single comparison (`field op value`) | No — collapse into parent ConditionBranch |
| `Microsoft.Crm.Workflow.Activities.EvaluateLogicalCondition` | And/Or combiner over child conditions | No — collapse into parent ConditionBranch |
| `Microsoft.Crm.Workflow.Activities.EvaluateExpression` | Compute / coerce a value (`CreateCrmType`) | No — collapse into parent step |
| `Microsoft.Crm.Workflow.Activities.ConvertCrmXrmTypes` | Type-coercion helper between CRM and .NET types | No — collapse silently |
| `Microsoft.Crm.Workflow.Activities.TerminateWorkflow` (some builds: `….StopWorkflow`) | Stop the workflow with `Succeeded` or `Canceled` | Yes — "Stop Workflow" |
| `Microsoft.Crm.Workflow.Activities.SendEmailNotification` | Backing activity for SendEmail (in addition to the direct `mxswa:SendEmail` element) | Yes — "Send Email" |
| `Microsoft.Crm.Workflow.Activities.PerformAction` | Invocation of a third-party / custom workflow activity registered as an Action | Yes — "Run custom step" (surface the action name) |

### BPF-internal activity classes you should skip

If you see any of these AQN class names, the file is (or contains parts of)
a **Business Process Flow**, not a classic workflow. Skip the activity, do
not try to render it, and — if it's the root — stop processing and tell the
user this isn't a classic workflow:

- `Microsoft.Crm.Workflow.Activities.EntityComposite`
- `Microsoft.Crm.Workflow.Activities.StageComposite`
- `Microsoft.Crm.Workflow.Activities.StepComposite`
- `Microsoft.Crm.Workflow.BusinessProcessFlowActivities.StageRelationshipCollectionComposite`

### `ConditionSequence` — `if / elseif / else` block

Contains an ordered list of `ConditionBranch` children. The first branch
whose condition evaluates true runs; the others are skipped. The optional
final branch (no condition) is the `else`.

When `Wait="True"` on the ConditionSequence, this becomes a **Wait
Condition** — the workflow pauses until the condition becomes true.
Background mode only.

### `ConditionBranch` — single branch

Holds a condition tree (`EvaluateLogicalCondition` / `EvaluateCondition`)
and a `Sequence` of child step activities to run if the condition matches.

### `EvaluateCondition` — single comparison

Evaluates `LeftOperand <Operator> RightOperand`. Operators include
`Equal`, `NotEqual`, `GreaterThan`, `LessThan`, `GreaterEqual`, `LessEqual`,
`Contains`, `DoesNotContain`, `BeginsWith`, `EndsWith`, `In`, `NotIn`,
`OnOrAfter`, `OnOrBefore`, `Last7Days`, `Next7Days`, etc.

### `EvaluateLogicalCondition` — combine sub-conditions

Contains `Type="And"` or `Type="Or"` and a list of child `EvaluateCondition`
or nested `EvaluateLogicalCondition` nodes.

### `EvaluateExpression` — compute a value

Computes the value used as the LeftOperand or RightOperand of a comparison
(or the right-hand-side of a `SetEntityProperty`). Often wraps a single
VB expression like `[GetVariableValue(…)]`.

## 5. Flow control

### `Composite` — step group container (wrapped)

A named subtree of activities that the designer treats as a single
"step group". Used when the user creates a labelled group of steps under
one heading. Carries a `DisplayName`.

### `TerminateWorkflow` — stop the workflow (wrapped)

```xml
<mxswa:ActivityReference
    AssemblyQualifiedName="Microsoft.Crm.Workflow.Activities.TerminateWorkflow, …"
    DisplayName="Stop workflow as canceled">
  <mxswa:ActivityReference.Properties>
    <InArgument x:TypeArguments="x:String" x:Key="StatusMessage">
      [&quot;Cancellation reason text&quot;]
    </InArgument>
    <InArgument x:TypeArguments="OperationStatus" x:Key="Status">
      <Literal>Canceled</Literal>
    </InArgument>
  </mxswa:ActivityReference.Properties>
</mxswa:ActivityReference>
```

`Status` is `Succeeded` or `Canceled`. `StatusMessage` is the message
recorded on the system job.

## 6. Form-side (real-time only)

These activities only function when the workflow runs synchronously
(`Mode="RealTime"`) attached to a form. They have no effect in background
workflows.

| Element | Effect |
|---------|--------|
| `<mcwc:SetVisibility>` | Show/hide a form control |
| `<mcwc:SetDisplayMode>` | `ReadOnly` / `Disabled` / `Normal` for a control |
| `<mcwc:SetFieldRequiredLevel>` | `Required` / `Recommended` / `None` |
| `<mcwc:SetAttributeValue>` | Set a field on the in-memory form record (does not save) |

```xml
<mcwc:SetDisplayMode
    DisplayName="Lock priority field"
    Control="prioritycode"
    Mode="ReadOnly" />
```

If you encounter `mcwc:*` activities in a workflow with `Mode="Background"`,
flag this as a misconfiguration — they will silently do nothing.

## 7. Internal helpers (parser-only)

These are activities you'll see in the XAML but should rarely render to the
user as separate steps in summaries or diffs:

- `EvaluateExpression`, `EvaluateCondition`, `EvaluateLogicalCondition` — roll
  up into the parent `ConditionBranch`.
- `GetEntityProperty`, `SetEntityProperty`, `RetrieveEntity` — roll up into
  the parent `Composite` step (Update/Create record).
- `Persist` — show as a "Checkpoint" annotation, not a top-level step.
- `ConvertCrmXrmTypes` — roll up silently (it's a type-coercion helper).

The `read-workflow` skill collapses these into the user-visible step they
belong to.

## 8. Custom CodeActivity

Third-party or in-house code activities appear in classic workflow XAML in
**two** shapes. Recognize both.

### 8a. As a direct XAML element (assembly registered with custom namespace)

```xml
<sample-asm:DoSomethingActivity
    xmlns:sample-asm="clr-namespace:Sample.Workflow.Activities;assembly=Sample.Workflow.Activities"
    DisplayName="Run my custom logic"
    InputParam1="[somevalue]"
    OutputParam1="[outputVariable]" />
```

Identifying details:
- The XML namespace `clr-namespace:<Namespace>;assembly=<AssemblyName>` tells
  you which assembly provides the activity.
- The element name is the C# class name.
- Input/output parameters are XAML attributes named after the C#
  `[Input("…")]` / `[Output("…")]` attributes on the class.

When summarizing, treat custom activities as opaque "Run custom step" entries
and surface the `AssemblyQualifiedName` so the user can locate the source.

### 8b. As an `ActivityReference` to `PerformAction`

When the custom logic is registered as a Custom API or workflow Action
(rather than a custom CodeActivity assembly), the designer wraps it in:

```xml
<mxswa:ActivityReference
    AssemblyQualifiedName="Microsoft.Crm.Workflow.Activities.PerformAction, …"
    DisplayName="Run my action">
  <mxswa:ActivityReference.Properties>
    <InArgument x:TypeArguments="x:String" x:Key="ActionName">
      <Literal>sample_DoSomething</Literal>
    </InArgument>
    <!-- one InArgument per Action input, plus OutArguments for outputs -->
  </mxswa:ActivityReference.Properties>
</mxswa:ActivityReference>
```

When summarizing, render as `Run action: "sample_DoSomething"` plus the
input parameter mapping. The user can find the action's contract in their
solution explorer or via `pac admin list-message --message-name <ActionName>`.

When editing a workflow that uses a custom activity, **do not modify**
the activity's parameter shape — the consuming assembly's contract may
break. Only modify input expressions.
