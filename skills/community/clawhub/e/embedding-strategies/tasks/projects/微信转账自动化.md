## 📋 项目概述

**一句话描述：** 语音指令触发微信自动发消息 + 转账红包

**核心场景：**
```
用户说："给我老婆转一个 3 块钱的红包"
  ↓
阿福机器人 → 微信聊天框自动打开 → 发送消息 → 发红包
```

---

## 🔬 技术验证结论（2026-03-08）

### 验证方案

**方案 1：PC 版微信（原生 Windows 应用）**
- ✅ 消息自动发送（成功）
- ✅ 联系人自动搜索（成功）
- ⚠️ 红包按钮快捷键（Alt+H 无效）
- ❌ UI Automation 无法定位红包按钮

**方案 2：手机连接镜像（Phone Link 投屏）**
- ✅ 窗口可激活
- ❌ UI Automation 无法访问内部元素（投屏）
- ❌ 无原生红包按钮

**方案 3：微信支付 API**
- ❌ 仅开放商家转账给用户 API
- ❌ 需要企业资质 + 应用上架
- ❌ 个人用户无法使用

### 核心障碍

1. **微信未开放红包/转账 API** - 个人用户无法使用
2. **UI Automation 无法可靠定位红包按钮** - PC 版无标准控件
3. **手机连接是投屏** - 无法抓取内部控件
4. **商家 API 门槛过高** - 需要企业资质 + 应用上架

### 最终结论

**❌ 技术不可行（当前条件下）**

建议：等待微信开放个人转账 API，或考虑其他支付方式（如支付宝）

- **便捷性：** 语音一句话完成转账

- **情感表达：** 随时给老婆发小红包

## ⚠️ 安全约束

### 红线原则
- ❌ **不直接集成微信支付 API**（高风险）
  - 需要企业资质（个人无法申请）
  - 私钥泄露风险（资金可能被盗）
  - 语音误识别风险（说错话就转错钱）
  - 合规问题（可能违反微信支付协议）

### 推荐方案
- ✅ **半自动化方案**（用户最后确认）
  - 阿福自动打开微信、定位聊天窗口、输入消息
  - 阿福打开红包界面、填入金额
  - **用户手动点"发红包"**（最后确认）

## 🔧 技术方案

### 方案 A：半自动化（推荐）⭐

**流程：**
```
1. 语音指令："给我老婆转 3 块钱"
   ↓
2. 阿福生成飞书确认卡片：
   ┌─────────────────────┐
   │ 转账确认            │
   │ 收款人：老婆        │
   │ 金额：3 元          │
   │ [确认转账] [取消]   │
   └─────────────────────┘
   ↓
3. 用户点击"确认转账"
   ↓
4. 阿福自动打开微信 PC 版
   ↓
5. 阿福定位到老婆聊天窗口
   ↓
6. 阿福输入消息："老婆，给你发个红包~"
   ↓
7. 阿福打开红包界面，填入 3 元
   ↓
8. 用户手动点"发红包"（最后确认）
```

**技术栈：**
- PowerShell + UI Automation
- 微信 PC 版 API（或图像识别）
- 飞书交互式卡片

**优点：**
- ✅ 资金安全（最后确认在用户手里）
- ✅ 便捷（前面步骤都自动化）
- ✅ 可追溯（飞书有审批记录）
- ✅ 容错（说错了可以取消）

**缺点：**
- ⚠️ 需要微信 PC 版
- ⚠️ 最后一步需要手动

---

### 方案 B：RPA 全自动（高风险）⚠️

**流程：**
```
语音指令 → 阿福 → 模拟人工操作 → 打开微信 → 输入金额 → 自动发红包
```

**技术栈：**
- AutoHotkey / PowerShell
- 图像识别（定位按钮位置）
- 模拟键盘鼠标操作

**优点：**
- ✅ 完全自动化

**缺点：**
- ❌ 资金风险高（误操作可能转错钱）
- ❌ 微信版本更新可能导致脚本失效
- ❌ 可能被微信识别为外挂

---

### 方案 C：飞书审批流（最安全）✅

**流程：**
```
语音指令 → 阿福 → 飞书审批 → 用户确认 → 飞书通知 → 用户手动发红包
```

**优点：**
- ✅ 完全可控
- ✅ 有审批记录
- ✅ 零资金风险

**缺点：**
- ⚠️ 步骤最多
- ⚠️ 没有自动化打开微信
