## Phase 0 —— 语义拆分完整规则（增强版 v5.16.0）

> **违反 F-01：如果 AI 没有输出语义拆分就开始规划或执行，属于严重违规。必须回滚并先输出拆分结果。**
> **v5.16.0 增强**：参考 semantic-split 的 5W2H 维度提取与约束强度标注（🔴🟡⚪），增强第一步理解能力。

### 拆分规则（增强版）

**块（Block）识别**：按主语划分，每个主语构成一个块。

| 主语类型 | 识别方式 |
|---------|---------|
| 用户（"我"） | 说话者本人 |
| AI 助手（"你"） | 执行者、帮助者 |
| 第三方（"他/她/它"） | 提及的其他角色 |

**每个块提取三类信息：**

| 元素 | 定义 | 增强标注 |
|-----|------|---------|
| **目的** | 最终要达成的目标 | 标注🔴🟡⚪约束强度 |
| **诉求** | 需要执行的具体操作或交付物 | 硬约束🔴/软约束🟡/无约束⚪ |
| **动机** | 为什么会产生这个需求 | 背景/原因/情绪 |

### v5.16.0 新增：5W2H 维度提取（参考 semantic-split）

**步骤 0.5**：任务类型识别与 5W2H 初始化（内部执行，不输出）

1. **任务类型识别**：制作PPT / 填写周报 / 安排行程 / 写邮件 / 策划活动 / 其他
2. **5W2H 维度初始化**：按任务类型从默认映射表填入默认值
3. **约束强度预标注**：🔴硬约束 / 🟡软约束 / ⚪无约束

**步骤 0.7**：注意力锚定（内部执行，不输出）

1. `[CRITICAL]` — 硬约束，后续不可违反
2. `[CORE]` — 核心目标（动词+宾语）
3. `[ENTITY]` — 数字、时间、人名
4. `[EXAMPLE]` — 举例内容，降权不作为约束
5. `[RESISTANCE]` — 「但是/担心/难」后内容
6. 核心句重构：「[主语] 要 [CORE] 于 [时间/场景]」（≤20字）

### v5.16.0 新增：自我反查（内部执行，不输出）

| 反查 | 检查内容 | 修正 |
|------|---------|------|
| 1 | 硬约束遗漏：`[CRITICAL]` 是否全在结果中？ | 补充 |
| 2 | 约束误判：🟡 词被标为 🔴？ | 修正 |
| 2b | 隐式升级：🟡 约束属组织规范/法律/安全？→ 🟡🔴 或 🔴 | 升级 |
| 3 | 举例干扰：`[EXAMPLE]` 被当核心诉求？ | 降级 |
| 4 | 5W2H 缺失：有明显可推断维度？ | 补充 `[推断]` |

### 判断流程（增强版）

```
收到任务
  │
  ├─ [MANDATORY] 第零步：语义拆分（F-01，必做）[增强：5W2H + 约束标注]
  │   → 输出块分析：主语/目的/诉求/动机 + 5W2H维度 + 约束强度
  │
  ├─ [MANDATORY] 第一步：任务规划（F-02，必做，所有任务）[增强：参考skill-sub规划逻辑]
  │   → 明确：任务目的、具体要求、使用工具、预期结果、潜在风险
  │   → [增强] 步骤排序（参考skill-sub的意图理解→步骤排序逻辑）
  │   → [增强] 执行计划生成（参考skill-sub的执行计划生成逻辑）
  │
  ├─ 第二步：判断是否需要拆分
  │   ├─ 单步任务（1个操作可完成）→ 直接进入执行
  │   └─ 多步任务（2个以上操作）→ 任务拆分，列出子步骤
  │
  ├─ [MANDATORY] 第三步：创建进度文件（F-03，规划确认后立即执行）
  │   → 调用 task_progress.py init
  │
  └─ 第四步：进入 Execute→Review→Advance 循环
```

## Phase 2~4 —— 执行/审查/推进循环详细规则（增强版 v5.16.0）

### [MANDATORY] EXECUTE 阶段规则

> **违反 F-04：如果 AI 执行步骤前没有重述本步骤任务目的，属于违规。**
> **v5.16.0 增强**：明确空转计数规则。

- **[MANDATORY]** 执行前必须输出：`🔧 执行前重述目的：[本步骤目的]`
- 实际运行命令、调用工具、测试代码
- 只跑一个主要命令（可并行多个独立命令）
- 不提前分析结果，不跳到审查阶段
- **[v5.16.0 新增]** 空转判断：如果执行阶段没有实际执行任何操作（仅输出描述、未调用工具），则视为空转，空转计数+1

### [MANDATORY] REVIEW 阶段规则

> **违反 F-05：如果 AI 执行步骤后没有输出 REVIEW，属于违规，不得继续执行下一步。**
> **v5.16.0 增强**：明确空转计数、重试计数、换思路到达的处理规则。

- **[MANDATORY]** 执行后必须紧跟 REVIEW，给出明确判断：✅成功 / ❌失败 / ⚠️部分
- 列出具体证据（数据、错误码、状态）
- **[MANDATORY]** 对照任务目的判断是否达成
- 不执行新操作
- **[v5.16.0 新增]** 空转未执行 → 空转计数+1，不画√
- **[v5.16.0 新增]** 重试失败 → 重试计数+1
- **[v5.16.0 新增]** 换思路到达 → 正确记录，不需要确认执行，直接推送到 ADVANCE

### [MANDATORY] ADVANCE 阶段规则

> **违反 F-06：如果 AI REVIEW 后没有输出 ADVANCE，属于违规，不得跳到下一步。**
> **v5.16.0 增强**：明确成功/失败/空转/换思路的完整流转规则，禁止三步骤内部直接循环。

- **[MANDATORY]** REVIEW 后必须紧跟 ADVANCE，明确：继续 / 换方向 / 结束 / 向上返回
- **[MANDATORY]** 更新进度追踪表，标记当前步骤状态
- **[MANDATORY]** 调用 `task_progress.py update` 持久化进度（F-07）
- 不直接执行，只规划
- **[v5.16.0 新增]** 成功：根据步骤确定接下来要执行什么，推进执行；**坚决不倒回已画√的步骤**
- **[v5.16.0 新增]** 失败：根据重试次数明确重试 或 换思路 或 求助
- **[v5.16.0 新增]** 空转：重述之前规划并告知执行必须执行 或 主动截断
- **[v5.16.0 新增]** 换思路：原封不动推送到 REVIEW（经 REVIEW→ADVANCE→EXECUTE 循环处理）

### v5.16.0 新增：空转/重试/换思路/求助 完整流转规则

> **核心原则：无任何三步骤内部直接循环，每一步都是最小单元。**

#### 执行阶段（EXECUTE）流转规则

```
执行阶段（EXECUTE）
  ├─ 实际执行成功 → 进入 REVIEW
  ├─ 实际执行失败 → 重试计数+1
  │    ├─ 重试≤3次 → 继续重试执行
  │    └─ 重试>3次 → 换思路 或 求助用户
  ├─ 空转（未实际执行）→ 空转计数+1
  │    ├─ 空转≤3次 → 重述规划，明确要求必须执行
  │    └─ 空转>3次 → 主动截断，请求用户输入固定触发词打破空转
  └─ 换思路 → 原封不动推送到 REVIEW（不确认执行）
```

#### 审查阶段（REVIEW）流转规则

```
审查阶段（REVIEW）
  ├─ 确认上一步执行逻辑无误 → 步骤画√（✅）
  ├─ 空转未执行 → 空转计数+1，不画√
  ├─ 重试失败 → 重试计数+1
  ├─ 换思路到达 → 正确记录，不需要确认执行，直接推送到 ADVANCE
  └─ 导入分支结果 → 记录成功或失败
```

#### 推进阶段（ADVANCE）流转规则

```
推进阶段（ADVANCE）
  ├─ 成功 → 根据步骤确定接下来要执行什么，推进执行；坚决不倒回已画√的步骤
  ├─ 失败 → 根据重试次数明确重试 或 换思路 或 求助
  │    ├─ 重试次数<3 → 推进重试执行
  │    ├─ 重试次数≥3 → 换思路 或 求助用户
  │    └─ 求助 → 明确发出交互请求，主动截断并等待用户输入
  ├─ 换思路 → 原封不动推送到 REVIEW（经 REVIEW→ADVANCE→EXECUTE 循环处理）
  │    → 哪怕是换思路，也要由 ADVANCE 推到 EXECUTE 再推到 REVIEW 再推到 ADVANCE 来进行换思路思考
  │    → 没有任何三步骤内部直接循环的情况，每一步都是最小单元，不能自循环闭环
  └─ 空转 → 重述之前规划并告知执行必须执行 或 主动截断
       ├─ 空转次数<3 → 重述之前规划并告知执行必须执行，返回 EXECUTE
       └─ 空转次数≥3 → 主动截断，请求用户输入固定触发词打破空转，停止并发出交互请求
```

#### 固定触发词打破空转规则

> **v5.16.0 新增**：当空转次数≥3次时，AI 必须主动截断当前流程，请求用户输入固定触发词来重新激发执行。

**触发词**：`继续执行`（用户可自定义，见配置）

**截断后行为**：
1. AI 输出：`⚠️ 空转次数已达到3次，主动截断。请输入触发词"继续执行"以重新激发执行。`
2. AI 停止当前执行循环，等待用户输入
3. 用户输入触发词后，AI 重新进入 EXECUTE 阶段，重置空转计数（可选），继续执行

#### 求助规则

> **v5.16.0 新增**：当超出重试或空转次数时，AI 必须明确发出交互请求，主动截断并等待用户输入。

**触发条件**：
- 同一步骤失败，重试次数≥3次
- 同一步骤空转次数≥3次
- 换思路后仍无法继续执行

**求助输出格式**：
```
⚠️ 求助请求
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

**问题描述**：[描述当前遇到的问题，为什么无法继续]

**已尝试方案**：
  1. [方案1]（失败原因）
  2. [方案2]（失败原因）
  3. [方案3]（失败原因）

**需要帮助的内容**：
  - [具体问题1]
  - [具体问题2]

**建议下一步**：
  - [建议1]
  - [建议2]

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
请提供指导或输入触发词"继续执行"以继续。
```

### 状态标记规则

| 标记 | 含义 | 说明 |
|------|------|------|
| ⏳ | 待执行 | 初始状态 |
| 🔄 | 执行中 | 正在执行 |
| ✅ | 成功 | 审查通过 |
| ❌ | 失败 | 需要重试或换方案 |
| ⏭️ | 跳过 | 经评估无需执行 |
| 🔁 | 重试中 | 正在重试（重试计数<N） |
| 🔀 | 换思路中 | 正在换思路（经 REVIEW→ADVANCE→EXECUTE 循环处理） |
| ⚠️ | 空转中 | 未实际执行（空转计数<N） |
| 🛑 | 已截断 | 空转/重试超次，等待触发词 |
