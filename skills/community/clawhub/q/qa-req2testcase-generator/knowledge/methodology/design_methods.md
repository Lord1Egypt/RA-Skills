# 用例设计方法库（6种核心方法）

> 适用范围：券商交易系统、账户管理、风控系统等金融业务场景的测试用例设计。

## 1. 等价类划分法

### 定义

将输入数据的值域划分为若干等价类（有效等价类 + 无效等价类），从每个等价类中选取一个代表值作为测试数据。基于假设：同一等价类中的任意值，对程序行为的影响是等价的。

### 适用场景

- 输入参数有明确的取值范围或类型约束
- 需要用最少用例覆盖最多输入组合
- 接口字段校验、表单输入验证

### 操作步骤

1. 分析需求，识别所有输入条件
2. 为每个输入条件划分有效等价类和无效等价类
3. 为每个等价类编号
4. 设计用例：每个用例尽量覆盖多个未被覆盖的有效等价类
5. 为每个无效等价类单独设计一个用例（保证其他输入有效）

### 券商业务示例：股票委托下单

**输入条件分析：委托数量**

| 等价类编号 | 类型 | 等价类描述 | 代表值 |
|-----------|------|-----------|--------|
| EC1 | 有效 | 100的整数倍，100~99999900 | 1000 |
| EC2 | 无效 | 非100的整数倍 | 150 |
| EC3 | 无效 | 小于最小委托量（<100股） | 50 |
| EC4 | 无效 | 超过最大委托量（>99999900） | 100000000 |
| EC5 | 无效 | 非正整数（负数） | -100 |
| EC6 | 无效 | 非数字类型 | "abc" |

**输入条件分析：委托价格（限价委托，沪A）**

| 等价类编号 | 类型 | 等价类描述 | 代表值 |
|-----------|------|-----------|--------|
| EP1 | 有效 | 跌停价~涨停价之间，精度≤2位小数 | 15.50 |
| EP2 | 无效 | 低于跌停价 | 0.01 |
| EP3 | 无效 | 高于涨停价 | 999.99 |
| EP4 | 无效 | 小数位超过2位 | 15.555 |
| EP5 | 无效 | 非正数 | -10.00 |

### 测试点模板

| 编号 | 测试点 | 优先级 | 预期结果 |
|------|--------|--------|---------|
| TC-EC-001 | 委托数量=1000股（有效等价类EC1），价格=15.50（有效EP1），验证下单成功 | P0 | 委托单创建成功，返回委托编号 |
| TC-EC-002 | 委托数量=150股（无效EC2，非100整数倍），验证数量校验 | P0 | 返回错误：委托数量必须为100的整数倍 |
| TC-EC-003 | 委托数量=50股（无效EC3，低于最小委托量），验证下限校验 | P1 | 返回错误：委托数量不得低于100股 |
| TC-EC-004 | 委托价格=15.555（无效EP4，精度超限），验证价格精度校验 | P1 | 返回错误：价格小数位不得超过2位 |
| TC-EC-005 | 委托数量="abc"（无效EC6，非数字），验证类型校验 | P2 | 返回错误：委托数量必须为数字 |

## 2. 边界值分析法

### 定义

在等价类划分的基础上，重点测试等价类边界上及边界附近的值。经验表明，大量缺陷发生在输入范围的边界处，而非中间区域。典型取值：min、min+1、nom（正常值）、max-1、max。

### 适用场景

- 输入有明确的数值范围（最大值、最小值）
- 涉及精度截断的金融计算
- 数组/列表长度限制
- 时间区间边界

### 操作步骤

1. 确定输入条件的边界值（上界、下界）
2. 取边界值本身、刚好超出边界的值、刚好在边界内的值
3. 对每个边界设计测试用例
4. 多个输入条件时，固定其他输入为正常值，逐一测试每个边界

### 券商业务示例：沪A股票委托数量边界

**边界分析：**
- 最小委托量：100股
- 最大委托量：99,999,900股（上交所单笔限制）
- 步长：100股

| 边界点 | 测试值 | 预期 |
|--------|--------|------|
| min-1 | 0股 | 拒绝 |
| min | 100股 | 接受 |
| min+step | 200股 | 接受 |
| nom | 10000股 | 接受 |
| max-step | 99,999,800股 | 接受 |
| max | 99,999,900股 | 接受 |
| max+step | 100,000,000股 | 拒绝 |

**边界分析：涨跌停价格（假设昨收10.00元，涨跌幅±10%）**

| 边界点 | 测试值 | 预期 |
|--------|--------|------|
| 跌停价-0.01 | 8.99 | 拒绝 |
| 跌停价 | 9.00 | 接受 |
| 跌停价+0.01 | 9.01 | 接受 |
| 涨停价-0.01 | 10.99 | 接受 |
| 涨停价 | 11.00 | 接受 |
| 涨停价+0.01 | 11.01 | 拒绝 |

### 测试点模板

| 编号 | 测试点 | 优先级 | 预期结果 |
|------|--------|--------|---------|
| TC-BV-001 | 委托数量=100股（最小边界），验证最小委托量可下单 | P0 | 下单成功 |
| TC-BV-002 | 委托数量=0股（低于最小边界），验证拒绝 | P0 | 返回错误：委托数量不得低于100股 |
| TC-BV-003 | 委托价格=涨停价+0.01（超出涨停边界），验证价格上限 | P0 | 返回错误：委托价格不得高于涨停价 |
| TC-BV-004 | 委托数量=99,999,900股（最大边界），验证最大委托量 | P1 | 下单成功 |
| TC-BV-005 | 委托价格=跌停价（下限边界），验证跌停价可下单 | P1 | 下单成功 |
| TC-BV-006 | 委托数量=100,000,000股（超出最大边界），验证拒绝 | P2 | 返回错误：超过最大委托数量限制 |

## 3. 判定表法

### 定义

将多个输入条件的所有可能组合及其对应的动作列成表格（判定表），系统化地设计测试用例。判定表由条件桩、动作桩、条件项、动作项四部分组成。适用于多条件组合决策的业务逻辑。

### 适用场景

- 业务规则涉及多个条件的组合判断
- 不同条件组合导致不同处理结果
- 需要穷举或覆盖关键条件组合
- 权限控制、费率计算、风控规则

### 操作步骤

1. 列出所有条件桩（输入条件）和动作桩（输出动作）
2. 计算条件组合数（2^n，n为条件数）
3. 填写条件项（每列一种组合）
4. 根据业务规则填写动作项
5. 合并相同动作的规则（简化判定表）
6. 每条规则对应一个测试用例

### 券商业务示例：融资融券开户资格判定

**条件桩：**
- C1：账户资产≥50万（连续20个交易日）
- C2：开户满6个月
- C3：风险评估等级≥C4（积极型）
- C4：无重大违规记录

**动作桩：**
- A1：允许开通融资融券
- A2：拒绝并提示原因

| 规则 | C1 | C2 | C3 | C4 | A1 | A2 |
|------|----|----|----|----|----|----|
| R1 | Y | Y | Y | Y | ✓ | |
| R2 | N | Y | Y | Y | | ✓（资产不足） |
| R3 | Y | N | Y | Y | | ✓（开户时间不足） |
| R4 | Y | Y | N | Y | | ✓（风险等级不足） |
| R5 | Y | Y | Y | N | | ✓（存在违规记录） |
| R6 | N | N | Y | Y | | ✓（资产不足+时间不足） |
| R7 | N | Y | N | Y | | ✓（资产不足+风险等级不足） |
| R8 | N | N | N | N | | ✓（全部不满足） |

### 测试点模板

| 编号 | 测试点 | 优先级 | 预期结果 |
|------|--------|--------|---------|
| TC-DT-001 | 资产≥50万 + 开户≥6月 + 风险≥C4 + 无违规（R1），验证正常开通 | P0 | 融资融券开通成功 |
| TC-DT-002 | 资产<50万 + 其他条件满足（R2），验证资产不足拒绝 | P0 | 拒绝，提示"账户资产不满足50万要求" |
| TC-DT-003 | 存在违规记录 + 其他条件满足（R5），验证违规拒绝 | P0 | 拒绝，提示"存在重大违规记录" |
| TC-DT-004 | 开户不满6月 + 其他条件满足（R3），验证时间不足拒绝 | P1 | 拒绝，提示"开户时间不满6个月" |
| TC-DT-005 | 全部条件不满足（R8），验证多条件拒绝提示 | P2 | 拒绝，提示所有不满足条件 |

## 4. 场景法

### 定义

通过模拟用户的实际操作场景（基本流 + 备选流 + 异常流），设计端到端的测试用例。关注业务流程的完整性，而非单个输入的正确性。基本流是最常见的成功路径，备选流是分支路径，异常流是错误处理路径。

### 适用场景

- 业务流程涉及多个步骤的串联操作
- 需要验证端到端的业务完整性
- 用户操作有多种分支路径
- 涉及状态流转的业务（订单、审批）

### 操作步骤

1. 分析业务需求，识别基本流（Happy Path）
2. 在基本流的每个步骤上识别备选流和异常流
3. 绘制场景流程图
4. 为每条路径（基本流 + 各备选流/异常流）设计一个测试用例
5. 补充数据驱动的变体用例

### 券商业务示例：股票限价委托下单全流程

**基本流（S1）：**
登录 → 选择股票 → 输入价格和数量 → 确认下单 → 委托成功 → 收到委托回报

**备选流：**
- S2：登录后发现账户被冻结 → 提示无法交易
- S3：输入价格超出涨跌停 → 提示价格超限，返回修改
- S4：确认下单时资金不足 → 提示可用资金不足
- S5：委托成功后撤单 → 撤单成功，资金释放
- S6：委托成功后部分成交 → 显示部分成交状态

**异常流：**
- E1：下单过程中网络断开 → 重连后查询委托状态
- E2：下单接口超时 → 提示稍后查询，不重复下单
- E3：非交易时间下单 → 提示当前非交易时段

### 测试点模板

| 编号 | 测试点 | 优先级 | 预期结果 |
|------|--------|--------|---------|
| TC-SC-001 | 基本流S1：正常限价委托下单全流程，从登录到收到委托回报 | P0 | 委托成功，状态为"已报"，收到回报推送 |
| TC-SC-002 | 备选流S4：可用资金不足时下单，验证资金校验 | P0 | 下单失败，提示"可用资金不足" |
| TC-SC-003 | 备选流S5：委托成功后撤单，验证撤单流程及资金释放 | P0 | 撤单成功，冻结资金释放回可用余额 |
| TC-SC-004 | 异常流E2：下单接口超时，验证不重复提交 | P1 | 提示"请稍后查询委托状态"，不产生重复委托 |
| TC-SC-005 | 异常流E3：非交易时间（15:01）提交委托，验证时段校验 | P2 | 提示"当前非交易时段，无法委托" |

## 5. 正交法

### 定义

利用正交表（Orthogonal Array）从大量参数组合中科学地选取具有代表性的测试用例子集。正交表保证任意两个因素的所有水平组合至少出现一次（两两覆盖），在大幅减少用例数量的同时保持较高的缺陷发现率。

### 适用场景

- 输入参数多（≥3个），每个参数有多个取值
- 全组合测试用例数量爆炸（如4因素×3水平=81种组合）
- 配置项组合测试（浏览器×操作系统×分辨率）
- 多条件筛选/查询功能

### 操作步骤

1. 识别所有因素（参数）及其水平（取值）
2. 选择合适的正交表 L_n(k^m)，n=用例数，k=水平数，m=因素数
3. 将因素和水平映射到正交表的列
4. 根据正交表生成测试用例
5. 补充重要的特殊组合（如全默认值、全极端值）

### 券商业务示例：交易查询条件组合测试

**因素与水平：**

| 因素 | 水平1 | 水平2 | 水平3 |
|------|-------|-------|-------|
| A-市场 | 沪A | 深A | 北交所 |
| B-委托类型 | 限价 | 市价 | 条件单 |
| C-时间范围 | 当日 | 近一周 | 近一月 |
| D-状态 | 全部 | 已成交 | 已撤单 |

全组合：3^4 = 81种，使用 L9(3^4) 正交表只需9条用例：

| 用例 | A-市场 | B-委托类型 | C-时间范围 | D-状态 |
|------|--------|-----------|-----------|--------|
| 1 | 沪A | 限价 | 当日 | 全部 |
| 2 | 沪A | 市价 | 近一周 | 已成交 |
| 3 | 沪A | 条件单 | 近一月 | 已撤单 |
| 4 | 深A | 限价 | 近一周 | 已撤单 |
| 5 | 深A | 市价 | 近一月 | 全部 |
| 6 | 深A | 条件单 | 当日 | 已成交 |
| 7 | 北交所 | 限价 | 近一月 | 已成交 |
| 8 | 北交所 | 市价 | 当日 | 已撤单 |
| 9 | 北交所 | 条件单 | 近一周 | 全部 |

### 测试点模板

| 编号 | 测试点 | 优先级 | 预期结果 |
|------|--------|--------|---------|
| TC-OA-001 | 沪A + 限价 + 当日 + 全部状态，验证默认组合查询 | P0 | 返回当日沪A所有限价委托记录 |
| TC-OA-002 | 深A + 市价 + 近一月 + 全部状态，验证跨市场+长时间范围 | P1 | 返回近一月深A市价委托，数据量和分页正确 |
| TC-OA-003 | 北交所 + 条件单 + 近一周 + 全部状态，验证北交所特殊品种 | P1 | 返回北交所条件单记录，字段与沪深格式一致 |
| TC-OA-004 | 沪A + 条件单 + 近一月 + 已撤单，验证撤单状态过滤 | P2 | 仅返回已撤单的条件单记录 |

## 6. 状态迁移法

### 定义

将系统中对象的所有状态及状态之间的转换关系建模为状态迁移图，针对每条状态转换路径设计测试用例。重点验证：合法的状态转换是否正确执行，非法的状态转换是否被正确拒绝。

### 适用场景

- 业务对象有明确的生命周期和状态流转
- 状态之间的转换有条件约束
- 需要验证非法状态转换被拒绝
- 订单系统、审批流程、账户状态管理

### 操作步骤

1. 识别对象的所有状态
2. 识别触发状态转换的事件/条件
3. 绘制状态迁移图（状态为节点，转换为有向边）
4. 列出状态迁移表（当前状态 × 事件 → 下一状态）
5. 设计用例覆盖所有合法转换路径
6. 设计用例验证非法转换被拒绝

### 券商业务示例：股票委托单状态迁移

**状态集合：**
- S0：未报（客户端已提交，未发送交易所）
- S1：待报（已发送柜台，等待报盘）
- S2：已报（交易所已确认接收）
- S3：部分成交
- S4：全部成交（终态）
- S5：已撤单（终态）
- S6：废单（终态）

**状态迁移图：**

```
S0(未报) --报盘发送--> S1(待报)
S1(待报) --交易所确认--> S2(已报)
S1(待报) --报盘失败--> S6(废单)
S2(已报) --部分成交回报--> S3(部分成交)
S2(已报) --全部成交回报--> S4(全部成交)
S2(已报) --撤单确认--> S5(已撤单)
S3(部分成交) --剩余成交回报--> S4(全部成交)
S3(部分成交) --撤单确认--> S5(已撤单)
```

**状态迁移表：**

| 当前状态 | 事件 | 下一状态 | 合法性 |
|---------|------|---------|--------|
| S0 未报 | 报盘发送 | S1 待报 | ✓ |
| S1 待报 | 交易所确认 | S2 已报 | ✓ |
| S1 待报 | 报盘失败 | S6 废单 | ✓ |
| S2 已报 | 部分成交 | S3 部分成交 | ✓ |
| S2 已报 | 全部成交 | S4 全部成交 | ✓ |
| S2 已报 | 撤单确认 | S5 已撤单 | ✓ |
| S3 部分成交 | 剩余成交 | S4 全部成交 | ✓ |
| S3 部分成交 | 撤单确认 | S5 已撤单 | ✓ |
| S4 全部成交 | 撤单请求 | — | ✗（终态不可撤） |
| S5 已撤单 | 成交回报 | — | ✗（终态不可成交） |
| S6 废单 | 任何事件 | — | ✗（终态不可变） |

### 测试点模板

| 编号 | 测试点 | 优先级 | 预期结果 |
|------|--------|--------|---------|
| TC-ST-001 | 合法路径：S0→S1→S2→S4（正常全部成交），验证完整成交流程 | P0 | 状态依次变为 未报→待报→已报→全部成交 |
| TC-ST-002 | 合法路径：S0→S1→S2→S3→S5（部分成交后撤单），验证部分撤单 | P0 | 部分成交金额正确，剩余部分撤单，资金释放 |
| TC-ST-003 | 非法转换：S4（全部成交）→ 尝试撤单，验证终态不可撤 | P0 | 撤单请求被拒绝，提示"已成交委托不可撤单" |
| TC-ST-004 | 合法路径：S0→S1→S6（报盘失败变废单），验证废单处理 | P1 | 状态变为废单，冻结资金释放 |
| TC-ST-005 | 非法转换：S6（废单）→ 尝试重新报盘，验证终态不可变 | P2 | 操作被拒绝，提示"废单不可重新委托" |
