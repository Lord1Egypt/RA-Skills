# Code Debug Companion - AI 智能代码调试助手

## 🎯 业务场景

开发者在日常工作中遇到代码报错或异常，通常需要：
1. 手动搜索错误信息（复制粘贴到 Google/StackOverflow）
2. 阅读大量无关结果找到类似问题
3. 手动编写修复代码
4. 手动写测试用例验证

整个过程琐碎耗时，且容易遗漏边界情况。

## 💥 痛点分析

| 痛点 | 描述 |
|------|------|
| 信息碎片化 | 错误信息分布在终端、截图、日志文件中，分散处理效率低 |
| 搜索成本高 | 需要在多个平台（Google/StackOverflow/GitHub）反复搜索类似问题 |
| 测试遗漏 | 修复后往往不写测试，导致相同 Bug 反复出现 |
| 根因不明 | 表面错误掩盖真正原因，修补式修复治标不治本 |

## 📦 编排的 Skills

| Skill 名称 | 来源 | 作用 |
|------------|------|------|
| code-review | clawhub | 分析错误类型、定位根因、识别代码问题 |
| brave-search | 内置 | 搜索类似问题的解决方案 |
| setup-unit-test | clawhub | 为修复生成单元测试用例 |

## 🚀 使用示例

### 示例 1：终端错误调试

**用户输入：**
```
调试以下错误：
TypeError: Cannot read property 'map' of undefined
  at Array.reduce (src/utils.ts:42:18)
```

**助手响应：**
1. 分析错误类型（undefined 引用）
2. 定位根因（`data` 可能在某些路径为 undefined）
3. 搜索类似问题（找到 3 条相关 StackOverflow 讨论）
4. 生成修复代码 + 2 个测试用例
5. 输出完整修复报告

### 示例 2：截图调试

直接发送错误截图，助手自动 OCR 解析错误信息并启动调试流程。

### 示例 3：批量错误堆栈

粘贴完整的错误堆栈，助手按优先级排序后逐个修复，并生成总体报告。

## 🔍 工作原理

1. **错误捕获** → 解析错误类型、堆栈信息、相关代码上下文
2. **根因分析** → 使用 code-review 分析代码逻辑，找出真正问题
3. **解决方案搜索** → 使用 brave-search 搜索类似问题的成熟解决方案
4. **修复生成** → 结合分析结果和搜索结果，生成修复代码
5. **测试覆盖** → 使用 setup-unit-test 生成测试用例，确保修复不破坏其他功能
6. **报告输出** → 生成完整的调试报告，便于复查和记录
