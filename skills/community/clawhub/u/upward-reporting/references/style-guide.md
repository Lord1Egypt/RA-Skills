# 通用风格规范

适用于所有汇报场景的语气、结构和用词规范。

---

## 核心语气定位

**目标**：自信而不浮夸，有主见但不武断，承认不确定性但给出方向。

不同场景的语气微调：
- 规划汇报 → 略偏战略，有格局感，但要接地气
- 故障复盘 → 克制、客观、直面，不能有辩护感
- 述职 → 有判断力，敢于自评，不夸大也不贬低
- 横向KO → 协作感强，照顾到多方立场

---

## 结论前置原则

所有章节用「结论/洞察先行」，细节跟随。

```
❌ 先铺垫3段背景，最后才给结论

✓ 先给结论（💡 或 📌 标注），再展开论据
```

---

## 量化优先原则

能量化的地方尽量量化，不用「很多」「大幅」「效率提升」等模糊表达。

| 模糊表达 | 替换方向 |
|---------|---------|
| 很多团队遇到这个问题 | X个团队、主要团队 |
| 效率大幅提升 | 从X小时降到Y小时 / X%的case不再需要人工 |
| 影响较大 | 影响X个服务 / 持续Y小时 / 损失Z |
| 覆盖面广 | 覆盖X%的核心链路 |

---

## 问题归因原则（尤其适用于复盘场景）

根因分析要指向**机制/系统**，而不是**个人失误**。

```
❌ 「某同学操作失误导致」
✓ 「[变更流程] 缺少 [某类校验]，导致高风险操作可以在未经审批的情况下执行」
```

---

## 承认局限原则

主动承认局限比回避更可信。对过渡态要诚实：

```
✓ 「理想态是X，但现阶段[约束]，因此过渡策略是Y，方向不变。」
```

---

## 高级表达技法库

以下技法不是某一场景的专属，而是增强表达力的通用工具。根据受众选择是否使用。

### 辩证转折

制造张力、展示判断力。用"不难/难"、"表面/本质"、"不是/而是"结构制造对比。

```
✓ 「引入工具不难，难的是看清路径、想清体系、做对落地」
✓ 「表面看是度量问题，本质是转化问题」
✓ 「不是AI不够强，而是协作范式没跟上」
```

**适用受众**：高层（展示判断力）、直属上级（展示思考深度）
**慎用受众**：风控/财务（辩证转折是修辞而非事实，可能被质疑）、细节控审核者

---

### 挑战+机遇双 framing

不只讲痛点，而是将挑战和机遇并行呈现，制造紧迫感和可能性。

```
✓ 「一方面，业务复杂度持续攀升，交付韧性面临考验；
    另一方面，AI爆发带来组织效能非线性增长的机遇。
    关键在于：能否将局部探索升级为规模化体系落地？」
```

**适用受众**：高层（双 framing 展示战略视野）、平级（让协作方看到共同的挑战和机遇）
**慎用受众**：风控/财务（framing 可能被视为立场先行而非客观分析）

---

### 反直觉声明

主动挑战读者的默认假设，制造认知冲击，抓住注意力。

```
✓ 「AI确实让个人变快了，但组织效能提升在哪？」
✓ 「行业普遍关注代码生成率，但聚焦局部最优往往掩盖了端到端的真实瓶颈」
✓ 「增加人力投入并没有缩短交付周期——瓶颈不在编码，在协作」
```

**适用受众**：高层（反直觉声明是最有效的注意力抓手）、技术评审（展示对问题的深层理解）
**慎用受众**：风控/财务（反直觉声明需要完整数据支撑才能使用）、平级同步（可能显得过于尖锐）

---

### 行业定位声明

明确声明自己在行业中的位置，建立先行者/创新者的定位。

```
✓ 「行业内尚未形成体系化的建设思路与规模化的落地实践」
✓ 「我们在这个问题上的实践，目前未见同类规模的组织公开分享」
```

**适用受众**：高层评审（定位声明帮助其理解为什么这个投入有战略价值）、技术评审
**慎用受众**：平级协作（定位声明可能显得"自我中心"）、风控/财务（定位声明属于主张而非事实）

---

### 命题拆解式表达

将一个大问题拆为2-3个子问题，每个子问题有"本质洞察"句。

```
✓ 核心命题：如何将AIDevOps的体系建设规模化落地？
   命题拆解：
   ● 如何看清演进路径，避免盲目探索？
   ● 面对庞杂研发链路，优先建设哪些环节？
   ● 如何在大型组织中规模化应用，解决渗透难、效果稀释？
```

**适用受众**：高层（命题拆解展示体系化思考）、技术评审（拆解展示方法论严谨性）
**慎用受众**：平级同步进度（拆解过于复杂，简单的现状→计划更清晰）

---

### 技法使用原则

1. **不要堆砌技法**：一份材料用2-3种技法足够，过多会让读者觉得在"炫技"
2. **技法与受众匹配**：每使用一种技法，先问"这个受众会怎么理解它"
3. **技法服务于内容**：技法是增强表达的工具，不能替代实质内容。判断力来自真实的洞见，不是来自修辞

---

## 禁用词

| 禁用 | 替换 |
|------|------|
| 赋能 | 具体说AI/工具在该场景做什么 |
| 打造 | 建设、构建（明确交付物） |
| 探索 | 明确阶段目标和验收标准 |
| 弯道超车 | 不够严谨，避免使用 |
| 新范式 | 具体说与旧模式的差别在哪里 |
| 宝贵经验 | 具体说学到了什么 |

## 图表使用规范

| 场景 | 推荐格式 |
|------|---------|
| 概念对比 | 对比表格（2-3列） |
| 时间线 | 有序列表（带时间戳） |
| 指标体系 | 表格（维度/内容/指标） |
| 打法方向 | 方向/为什么做/怎么做 三列表格 |
| 时间规划 | Q1/Q2 分栏，含 Milestone + 子项 |
| 能力/风险分层 | 递进列表 |
| 命题拆解 | 命题→子问题→解法→实施步骤 递进格式 |
| 归因链路 | 分层表格（能力层→个体层→协同层→组织层） |
