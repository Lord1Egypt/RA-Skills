# 报告交付风格指南

用于把人生岔路模拟结果整理成终端用户能保存、复读和继续追问的交付版本。

## 默认交付

- 基础对话内给 Markdown 报告。
- 有文件能力时优先给可浏览 HTML 用户报告。
- 用户需要留档、转发或打印时，优先交付 HTML，并提示用户用浏览器保存。

## 用户版 HTML 阅读流

HTML 默认读者是真实用户。页面要像在对这个人说话，先命名遗憾，再解释分叉，最后给今天能补回的一部分。

### 前台阅读预算

折叠附录前的用户可见区域需要保持轻量：

- 可见文字总量控制在约 8200 字符以内，允许事件冲击卡承担解释。
- 单个自然段控制在约 190 字符以内。
- 前台必须保留 1 个 SVG 分叉图、3 张结果卡、4-6 张事件冲击卡和 4 张行动卡。
- 分叉图用短标签解释三种结果，图后再用三张结果卡说明给什么、拿走什么、最大误判。
- 前台不要放大表格；依据、外部环境、多角度解释和评分进入后置层。
- 如果内容超出预算，优先压缩解释句，保留判词、三种结果和今天可做动作。

### 首屏

首屏只保留四块：

1. 报告名：人生岔路复盘报告。
2. 用户问题。
3. 一句强判词。
4. 下滑提示。

首页不放：

- 置信度。
- 今日行动。
- 按钮组。
- 表格。
- Agent。
- 变量。
- 产物。
- 判断门槛。
- 后台视角。

### 第二屏

第二屏必须解释分叉，标题固定：

`一次选择，后来分出了三种结果`

三种结果固定用：

- 现实结果：留深圳。
- 假设结果：去北京。
- 回补结果：今天补回可见度。

图表左侧写已经发生的人生，中间写现实分岔点，右侧写三种结果。每个结果只配一句解释，不做复杂数据图，不写分数。

图表优先使用简单分叉曲线，不要只堆卡片。移动端用户扫一眼要能看出：左侧已经发生，中间分岔，右侧分成三种结果。

### 三种结果解释

默认展开区必须有三张结果卡。每张卡包含：

- 结果名称。
- 一句短判词。
- 它给你的。
- 它拿走的。
- 最大误判。

结果卡要写具体场景和代价，避免只写“机会更多、风险更高”。

### 事件冲击卡

`事件冲击卡`必须用六段结构：

1. 事件锚点。
2. 生活场景。
3. 现实代价。
4. 情绪代价。
5. 误判点。
6. 今天验证。

每张卡合计 160-260 字。优先写具体年份、具体场景、现实代价、情绪代价、误判点和低成本验证动作。

所有主题都必须从 2008-2026 事件库里筛 4-6 个相关事件或周期，再转成用户能感受到的生活冲击。写城市、海外、买房、职业、AI 时，先读取 `event-shock-engine.md` 和 `life-scene-library.md`。

### 30 天验证实验

最后给 4 周实验：

1. 第 1 周：找一个真实人或真实材料。
2. 第 2 周：写一页案例或做一个最小流程。
3. 第 3 周：拿给 3 个可信对象看。
4. 第 4 周：根据反馈决定加码、缩小或换题。

这一页禁用“产物、判断门槛、交付、试运行”等项目管理式表达。

每周动作必须同时满足：

- 30 天内能完成。
- 不要求用户辞职、搬家、投资、买房、分手或承担高风险。
- 动作里有明确对象或留下的记录，例如找谁聊、写哪一页、发给谁看。
- 动作结束后能观察一个结果，例如对方能否说出优势、是否看懂案例、哪类机会被证实。

### 后半部分

默认展开保留真实用户会关心的部分：

- 你真正放不下的是什么。
- 三种结果，各自给什么和拿走什么。
- 事件冲击卡。
- 30 天验证实验。
- 四个每周动作。

默认折叠区用人话标题：

- 你当年没算进去的事。
- 哪些是你选的，哪些是时代推的。
- 我为什么会这样判断。
- 换几个角度看这件事。
- 那几年，外部环境也在变。
- 还有哪些信息会改写结论。
- 这份报告不能替你决定什么。

折叠内容也要像继续解释给用户听。每个折叠区打开后，第一句先给人话判断，再放依据、表格或卡片。折叠区内部小标题也要用用户能理解的话，避免后台式摘要、事件清单、材料评估和边界说明标题。质量评分和修订记录只保留在源 Markdown 或开发者注释里，不进入用户界面。

折叠层也要尽量图形化：

- `哪些是你选的，哪些是时代推的`：先给归因小图，再给解释卡。
- `那几年，外部环境也在变`：先给外部时间线图，再给事件卡。
- `你当年没算进去的事`：优先用短卡片，不回到大段表格。
- 不把折叠层写成后台目录或长表格。

## 语言要求

- 少用“变量、路径、矩阵、框架、Agent、系统、模型、置信度”等系统词。
- 多用“你当年以为、你后来才发现、它给你的、它拿走的、你现在还能补的”。
- 每段都要像在解释一个人的人生执念。
- 结论先行，证据后置。
- 前台每段尽量 1-3 句，能折叠的依据放附录。
- 每种结果同时写收益、代价和误判。
- 每个强判断都给机制链、反证条件和证据债。
- 抽象词后面必须跟具体年份、具体场景和现实代价。
- 不把报告写成营销文案。
- 不替用户下结论。
- 不鼓励冲动搬迁、离职、买房、分手或投资。

## 用户可见附录

用户可见附录承接：

- 当年没算进去的事。
- 选择和外部环境的拆分。
- 依据解释。
- 多角度提醒。
- 那几年外部环境的变化。
- 会改写结论的补充信息。
- 5 个补充问题。
- 报告不能替用户决定什么。

质量评分和修订记录不在 HTML 用户界面展示。

## 交付前评分

- 读取 `report-quality-rubric.md`。
- 使用 7 个维度打分：首屏命中感、分叉图、事件冲击卡、反车轱辘、时代事件、反浪漫化、30 天实验。
- 任一维度低于 4 分，先回写一次。
- 用户反馈“太像报告”“车轱辘话”“没有被说中”时，优先重写首屏判词、第二屏分叉图、事件冲击卡和 30 天验证实验。
