## 前置检查

调用任何命令前，按以下顺序检查：

### 1. 确认 CLI 已安装

```bash
kd100 
```

| 结果 | 处理 |
|------|------|
| 输出版本号（如 `0.0.1`） | 已安装，进入下一步 |
| 命令不存在 / 无法执行 | **停止流程**，引导用户执行 `npm install -g @kuaidi100-user/cli` 安装，安装后重新验证 |

安装命令：

```bash
npm i -g @kuaidi100-user/cli
```

### 2. 确认登录状态

```bash
kd100 auth status
```

| 结果 | 处理 |
|------|------|
| `hasKey: true` | 已进入登录模式，继续业务流程 |
| `hasKey: false` | 需要登录，执行 SSO 授权（见下方） |

**未登录时发起 SSO 授权：**

```bash
kd100 auth login --open
```

**授权流程：**
1. 执行 `kd100 auth login --open`——CLI 调用授权接口，取后端返回的 `verificationUriComplete`，**自动用系统外部浏览器**打开该授权页，并阻塞等待用户完成授权
2. **Agent 只执行上述命令并等待返回**；**禁止** Agent 用 `start`/`open`/`xdg-open` 或 Cursor 内置浏览器重复打开链接
3. 告知用户：「已在系统浏览器打开授权页，请在外部浏览器中完成 SSO 登录」
4. 命令返回 `hasKey: true` 后，方可继续业务流程
5. **授权成功前不得执行任何其他业务命令**（寄件、查单、物流等）

**降级（仅 `--open` 无法打开浏览器时）：**

```bash
kd100 auth login --no-wait --json
# → 文本展示 data.verificationUriComplete，提示用户自行复制到外部浏览器打开
kd100 auth login --device-code <deviceCode> --json
```

### 3. 寄快递登录门禁

用户表达**寄快递 / 下单**意图时，在进入寄件流程前必须检查登录态：

```bash
kd100 auth status
```

| 结果 | 处理 |
|------|------|
| `hasKey: true` | 进入寄件流程，**仅使用服务端** `kd100 sender` / `kd100 receiver` |
| `hasKey: false` | **停止寄件流程**，执行 SSO 授权登录；**禁止**读取 `kd100 data sender` / `kd100 data receivers` 本地缓存 |

## 运行模式对比

| 功能 | 未登录（仅辅助） | 已登录 |
|------|----------------|--------|
| 地址解析 | ✅ | ✅ |
| 重量查询 | ✅ | ✅ |
| 快递比价 | ✅ | ✅ |
| **寄快递 / 预下单** | **❌ 需先登录** | **✅** |
| **寄件人获取** | ❌ 不适用 | **服务端 → 重新登录 / 手动** |
| **收件人获取** | ❌ 不适用 | **服务端地址簿** |
| 物流查询 | ❌ 仅本地 orders | ✅ 仅服务端 |
| 订单管理 | ❌ 仅本地 orders | ✅ 仅服务端 |
| 取消订单 | ❌ | ✅ |

## 决策逻辑（自动 vs 询问用户）

### 分步交互（最重要）

寄件流程是**多轮对话**，不是一次表单。严格遵守：

- **每轮只问一件事**：等用户回答后再进入下一步
- **禁止一次性列出所有问题**：不要说"请提供寄件人姓名、电话、地址，收件人姓名、电话、地址，物品名称……"
- **自动填充的步骤可以跳过询问**：如默认寄件人已获取，直接展示确认即可
- **每步之间可以静默调用 CLI**：查地址簿、查重量、查价格等对用户透明，用户只看到"请告诉我收件人"这样的单一问题

正确的交互节奏：
```
你: 请问要寄给谁？（如果寄件人已自动获取，同时展示寄件人信息让用户确认）
用户: 寄给张三，北京朝阳区
你: [调用 kd100 receiver 张三 → 命中] 找到张三的地址，电话 139xxxx，地址北京朝阳区XX号，对吗？
用户: 对的
你: 要寄什么物品？
用户: 手机
你: [调用 kd100 weight 手机 → 0.5kg] 手机参考重量 0.5kg，没问题的话继续？
用户: 可以
你: [调用 kd100 companies] 以下快递可选：1. 顺丰 18元 2. 中通 12元 3. 圆通 10元，选哪个？
...
```

### 可自动执行（无需询问）

| 场景 | 决策 | 命令 |
|------|------|------|
| 寄件人 | 优先 `kd100 sender`，成功则自动填充 | 服务端（寄快递必登录） |
| 收件人 | 调用 `kd100 receiver <姓名>` 模糊匹配，命中则自动填充 | 服务端（寄快递必登录） |
| 物品重量 | 调用 `kd100 weight <物品>` 推荐参考值，用户可改 | 服务端 |
| 快递公司 | 默认推荐最便宜的，用户可换 | 服务端 |

> **寄快递禁止本地缓存**：寄件流程中不得使用 `kd100 data sender` / `kd100 data receivers`。未登录须先 SSO 授权。
> **服务端查无数据**：`kd100 sender` / `kd100 receiver` / `kd100 order list` / `kd100 order track` 失败或空时，**不得展示本地缓存**，提示 `kd100 auth login --open` 重新登录并重试。

### 必须询问用户

| 场景 | 询问内容 |
|------|----------|
| 服务端无寄件人（已重试登录后） | 完整寄件人信息（姓名、手机、地址） |
| 物品类型 | 必须用户提供 |
| 收件人地址簿无匹配 | 完整收件人信息 |
| 快递选择 | 确认或更换 |
| 最终下单 | 确认所有信息后才能提交 |

### 绝不自动执行

- 不猜测用户想寄什么
- 不自动选择收件人（地址簿匹配后必须用户确认）
- 不跳过最终确认直接下单

## 一、寄件人获取

> 寄快递必须先登录（见 [寄快递登录门禁](#3-寄快递登录门禁)）。

### 已登录（寄快递唯一路径）

采用 **仅服务端查询** 策略；查不到数据时不回落本地缓存：

```
Level 1: kd100 sender (服务端)
    ↓ 成功 → 直接使用，跳过询问
    ↓ 失败 / 无数据 (401/404/空/网络异常)
    → 禁止读取或展示 kd100 data sender 本地缓存
    → 提示用户执行 `kd100 auth login --open` 重新登录
    → 重新登录成功后重试 kd100 sender
    ↓ 重试仍无数据
Level 2: 手动询问
    → 询问姓名、手机号、详细地址
```

保存本地缓存命令（**仅用于 CLI 数据管理，寄快递流程中不使用**）：
```bash
kd100 data sender --set '{"name":"张三","phone":"13800138000","province":"广东省","city":"深圳市","district":"南山区","address":"科技园XX路XX号"}'
```

**注意**：地址簿字段 `mobile` → `--sender-phone`，`addr` → `--sender-address`。

## 二、收件人获取（三分支）

> 寄快递必须先登录；**禁止**使用 `kd100 data receivers` 本地匹配。

根据用户提供的信息量，走不同的处理分支：

### 分支 A：用户仅提供收件人姓名

> 例："寄给张三"、"帮我寄个东西给李四"

**寄快递（已登录）：**
```
1. kd100 receiver <姓名> → 服务端地址簿查询（仅用服务端，不查本地）
2. 命中?
   ├── 1条 → 直接展示，让用户确认
   ├── 多条 → 列表展示，让用户选择
   └── 0条 / 失败(401/404/空) → 不展示本地缓存，提示 `kd100 auth login --open` 重新登录并重试；重试仍无匹配 → 走分支 C
```

### 分支 B：用户提供姓名 + 部分地址信息

> 例："寄给张三，北京朝阳区"、"寄给李四，电话139xxxx"

**寄快递（已登录）：**
```
1. kd100 receiver <姓名> → 尝试匹配验证
2. 命中?
   ├── 是 → 用服务端数据补全缺失字段，展示确认
   └── 否 / 失败 → 不查本地缓存；接口异常时提示重新登录；仍无法匹配 → 走分支 C
```

### 分支 C：用户提供完整地址 / 无任何匹配

> 例："寄到北京市朝阳区三里屯XX号"、"第一次寄，收件人是..."

```
1. 走原有流程：询问完整收件人信息
   - 姓名（必填）
   - 手机号（必填）
   - 详细地址（必填，自由文本）
2. kd100 address <地址> → 解析为省市区
3. 保存到本地缓存 kd100 data sender --set 不适用，收件人自动保存
```

## 四、地址解析与快递查询

### 地址字段映射

服务端地址簿（`kd100 sender` / `kd100 receiver`）返回的字段名与下单参数不一致，需要映射：

| 地址簿字段 | 下单参数 | 说明 |
|-----------|---------|------|
| `mobile` | `--sender-phone` / `--receiver-phone` | 手机号 |
| `addr` | `--sender-address` / `--receiver-address` | 详细地址 |

其余字段（`name`/`province`/`city`/`district`）名称一致。

### 地址解析

`kd100 address <地址>` 将自由文本地址解析为结构化地址：

| 返回字段 | 说明 |
|---------|------|
| `province` | 省份 |
| `city` | 城市 |
| `district` | 区县 |
| `subArea` | 详细地址（不含省市区） |
| `resultCode` | 是否解析成功 |

### 快递公司字段映射

`kd100 companies` 返回的字段名与下单参数不一致：

| 比价返回字段 | 下单参数 | 说明 |
|------------|---------|------|
| `name` | `--kuaidi-name` | 快递公司名称 |
| `com` | `--kuaidi-com` | 快递公司编码 |
| `sign` | `--company-sign` | 签名标识（下单必传） |
| `totalprice` | `--estimated-amount` | 运费 |

### 物品重量注意

`kd100 weight` 返回的 `spec_weight` 是字符串类型，使用时需 `parseFloat()` 转换。

### 调用顺序

```
1. 确定寄件人和收件人（省市区已就绪）
2. 调用地址解析按需结构化（用户自由文本 → kd100 address）
3. kd100 companies <寄件城市> <收件城市> <重量> → 快递列表
4. 展示快递列表（按 totalprice 升序）→ 用户选择
5. 映射字段：name→--kuaidi-name, com→--kuaidi-com, sign→--company-sign, totalprice→--estimated-amount
```

## 五、完整下单流程总览

> **Step 0**：`kd100 auth status`，未登录则 SSO 授权，**禁止本地缓存**。

```
0. [门禁] 确认已登录 → 未登录则 `kd100 auth login --open`
       |
       v
1. [服务端] 获取寄件人
   kd100 sender → 失败则重新登录并重试 → 仍无则手动询问
       |
       v
2. [服务端] 获取收件人
   kd100 receiver → 失败则重新登录并重试 → 仍无则手动输入
       |
       v
3. [服务端] 物品信息
   用户输入 → kd100 weight → 确认重量
       |
       v
4. [服务端] 地址解析
   kd100 address 结构化地址
       |
       v
5. [服务端] 查询快递公司
   kd100 companies → 展示列表 → 用户选择
       |
       v
6. [服务端] 确认下单
   展示完整订单信息 → 用户确认 → kd100 order create
       |
       v
7. [展示] 展示结果
   下单链接 + 二维码 + 订单号
       |
       v
8. [服务端] 物流查询（可选）
   kd100 order list / kd100 order track（不合并本地缓存）
```

## 六、下单成功展示格式

`kd100 order create` 返回嵌套结构，需从正确路径提取数据：

```
data.orderInfo.orderNo       → 订单号
data.orderInfo.status        → 订单状态
data.orderInfo.createTime    → 下单时间
data.url                     → 微信小程序链接（用于自动打开）
data.qrCode                  → 二维码链接（用于扫码下单）
data.markdownInfo            → Markdown 格式信息（可直接展示给用户）
```

**展示方式**：文本展示为必选项，浏览器打开为可选项。无论浏览器打开是否成功，用户都能通过文本信息完成下单：

1. **链接展示**（必须）：提供 `data.url` 纯文本链接，确保用户可直接复制或点击
2. **二维码展示**（必须）：展示 `data.qrCode` 二维码图片
3. **自动打开**（可选）：用浏览器工具自动打开 `data.url`，仅作为便捷补充

**核心原则**：先完成文本展示（链接+二维码），确认用户能看到后再选择性地调用浏览器打开。浏览器打开失败时不得影响已展示的链接信息。

## 七、物流查询

### 未登录模式

```
用户要查物流?
└── 仅查询本地 kd100 data orders
    └── 展示本地订单列表
```

### 登录模式

```
用户要查物流?
├── 有运单号 → kd100 order track <运单号>
│   └── 查不到 / 401/404 → 不展示本地缓存，提示 `kd100 auth login --open` 重新登录
└── 无运单号 → 仅查服务端 kd100 order list
    ├── 有数据 → 展示服务端订单列表
    └── 无数据 / 失败 → 不展示本地 kd100 data orders，提示重新登录并重试
        └── 用户选择 → kd100 order track <运单号>
```

## 八、取消订单

### 未登录模式

❌ 不支持，提示用户登录

### 登录模式

```
1. 确认用户要取消的订单号（仅从服务端 kd100 order list 获取，不用本地缓存）
2. 询问取消原因
3. kd100 order cancel <订单号> --reason <原因>
4. 成功后从本地缓存标记状态（如有）
```

## 九、异常降级矩阵

| 命令 | 异常场景 | 未登录模式降级 | 登录模式降级 |
|------|---------|--------------|-------------|
| kd100 sender | 401/404 / 空 / 网络错误 | ❌ 无此命令 | 不查本地 → 提示 `kd100 auth login --open` 重新登录并重试 → 仍无则手动询问 |
| kd100 receiver | 401/404 / 空列表 / 网络错误 | ❌ 无此命令 | 不查本地 → 提示重新登录并重试 → 仍无则手动输入 |
| kd100 order list | 401/404 / 空列表 / 网络错误 | ❌ 无此命令 | 不展示本地 orders → 提示重新登录并重试 |
| kd100 order track | 401/404 / 查不到 | ❌ 无此命令 | 不展示本地 orders → 提示重新登录并重试 |
| kd100 data sender | 本地无数据 | 手动询问 | 不用于登录模式服务端失败时的降级 |
| kd100 data receivers | 本地无匹配 | 手动输入 | 不用于登录模式服务端失败时的降级 |
| kd100 weight | 404 / 未识别 | 询问用户手动输入重量 | 询问用户手动输入重量 |
| kd100 address | 404 / 解析失败 | 提示用户检查地址 | 提示用户检查地址 |
| kd100 companies | 404 / 空列表 | 提示暂无可用快递 | 提示暂无可用快递 |
| kd100 order create | 401 未登录 | 引导 SSO 登录，禁止本地缓存寄件 | 检查必填字段，提示补充 |

**原则**：
- **寄快递**：必须先登录，禁止 `data sender` / `data receivers` 本地缓存
- **辅助查询**（address/weight/companies）：未登录可用
- **已登录服务端查不到数据**：禁止展示本地缓存，提示重新登录并重试；仍无数据再手动询问或告知无记录

## 十、命令调用策略

### 查询优先级

| 场景 | 未登录 | 已登录（寄快递） |
|------|--------|-----------------|
| 获取寄件人 | ❌ 须先登录 | 仅 `kd100 sender`；失败 → 重新登录 → 手动询问 |
| 查找收件人 | ❌ 须先登录 | 仅 `kd100 receiver`；失败 → 重新登录 → 手动输入 |
| 查找订单 | `kd100 data orders`（仅查单，非寄件） | 仅 `kd100 order list`；失败 → 重新登录 |

### 单次寄件命令调用次数

| 步骤 | 命令 | 调用次数 |
|------|------|---------|
| 获取寄件人 | kd100 sender | 0-1 |
| 获取收件人 | kd100 receiver | 0-1 |
| 查物品重量 | kd100 weight | 1 |
| 地址解析 | kd100 address | 0-2 |
| 查快递公司 | kd100 companies | 1 |
| 提交订单 | kd100 order create | 1 |
| **合计** | | **4-7 次** |

频率限制：每分钟 10 次 / 每天 100 次，单次寄件流程留有充足余量。
