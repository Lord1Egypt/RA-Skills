# 项目分包机制

> 本文档详细说明项目拆解方法论、分包规则、匹配算法和分包协议要点。

---

## 一、拆解方法论

### 1.1 拆解原则

| 原则 | 说明 | 示例 |
|------|------|------|
| **模块化** | 按功能/流程拆解为独立模块 | 电商项目：前端+后端+数据库+运维 |
| **边界清晰** | 模块之间接口明确 | API接口文档+数据字典 |
| **独立交付** | 每个模块可单独验收 | 交付物清单明确 |
| **能力匹配** | 分配给最适合的人/团队 | 前端匹配前端专家 |

### 1.2 拆解四步法

```
┌─────────────────────────────────────────────────────────────┐
│                    项目拆解四步法                             │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  Step 1：需求分析                                            │
│  └── 分析客户需求，确定核心交付物                            │
│      • 梳理需求清单                                          │
│      • 确定需求优先级                                        │
│      • 识别关键需求                                          │
│                                                              │
│  Step 2：模块划分                                            │
│  └── 按功能/流程拆解为独立模块                                │
│      • 每个模块有明确边界                                    │
│      • 每个模块可独立验收                                    │
│      • 模块之间定义好接口                                    │
│                                                              │
│  Step 3：技能映射                                            │
│  └── 为每个模块匹配所需技能                                  │
│      • 核心技能（必备）                                      │
│      • 辅助技能（可选）                                      │
│      • 识别技能缺口                                          │
│                                                              │
│  Step 4：分包分配                                            │
│  └── 根据能力画像分配模块                                    │
│      • 优先匹配度高                                          │
│      • 考虑负载均衡                                          │
│      • 确认各方接受                                           │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 1.3 拆解维度

| 拆解维度 | 适用项目类型 | 说明 |
|----------|-------------|------|
| **功能维度** | 软件开发、产品设计 | 按功能模块拆解 |
| **流程维度** | 营销项目、服务项目 | 按业务流程阶段拆解 |
| **交付维度** | 内容项目、创作项目 | 按交付物拆解 |
| **时间维度** | 长周期项目 | 按时间阶段拆解 |
| **混合维度** | 复杂项目 | 综合以上维度 |

## 二、拆解示例

### 2.1 案例：品牌全案营销项目（30万）

**项目背景**：某企业委托进行品牌全案营销，包括品牌策略、视觉设计、官网开发、内容营销、社交媒体运营。

**拆解过程**：

```
Step 1：需求分析
├── 核心交付物
│   ├── 品牌策略报告
│   ├── VI设计系统
│   ├── 企业官网
│   ├── 内容营销方案+内容
│   └── 社交媒体运营方案+执行
└── 关键需求
    ├── 品牌一致性
    ├── 技术可行性
    └── 落地执行

Step 2：模块划分
├── M1：品牌策略（3万）
│   ├── 品牌调研
│   ├── 品牌定位
│   ├── 价值主张
│   └── 品牌故事
├── M2：视觉设计（6万）
│   ├── Logo设计
│   ├── VI系统
│   └── 设计规范
├── M3：官网开发（10万）
│   ├── 前端开发
│   ├── 后端开发
│   ├── 数据库
│   └── 运维部署
├── M4：内容创作（4万）
│   ├── 公众号文章
│   ├── 短视频脚本
│   └── 其他内容
├── M5：社交媒体运营（4万）
│   ├── 运营策略
│   ├── 账号运营
│   └── 数据分析
└── M6：项目管理（3万）
    ├── 项目统筹
    ├── 进度管控
    └── 质量管理
```

**分包方案表**：

| 模块 | 预算 | 所需技能 | 推荐分包 |
|------|------|---------|---------|
| M1 | 3万 | 品牌策划/市场分析 | 策略导师 |
| M2 | 6万 | Logo设计/VI设计 | 设计OPC |
| M3 | 10万 | 前端+后端+运维 | 开发团队 |
| M4 | 4万 | 文案/编辑/排版 | 内容OPC |
| M5 | 4万 | 运营/投放/客服 | 运营OPC |
| M6 | 3万 | 项目统筹/质量管理 | 挂帅人 |

## 三、分包规则

### 3.1 基本分包规则

| 规则类型 | 具体内容 | 适用范围 |
|---------|---------|----------|
| **分包门槛** | 单模块金额≥1000元才能分包 | 全部项目 |
| **分包比例** | 挂帅人保留模块价值≥30% | 全部项目 |
| **分包审核** | C级以上项目需三长老审批 | C级及以上 |
| **接包限制** | 同一成员承接模块数≤3个 | 全部项目 |
| **利益回避** | 挂帅人不得承接自己分包的项目 | 全部项目 |

### 3.2 分包审核规则

| 项目等级 | 审核要求 | 审核内容 | 审核人 |
|----------|----------|----------|--------|
| **D级** | 无需审核 | - | - |
| **C级** | 事后报备 | 分包方案合理性 | 导师 |
| **B级** | 事前审批 | 分包方案+接包人资质 | 盟主 |
| **A级** | 事前审批 | 完整分包计划+团队配置 | 三长老 |
| **S级** | 理事会审批 | 战略分包方案+风险管理 | 理事会 |

### 3.3 分包禁止情形

| 禁止类型 | 具体内容 | 后果 |
|----------|----------|------|
| **转包禁止** | 未经批准将项目整体转包 | 取消资格 |
| **层层分包** | 分包层级超过2层 | 违规处理 |
| **资质挂靠** | 借用他人资质承接项目 | 双方处罚 |
| **利益输送** | 分包过程中存在利益输送 | 严重处罚 |

## 四、匹配算法

### 4.1 匹配维度

| 维度 | 权重 | 数据来源 | 说明 |
|------|------|----------|------|
| **技能匹配度** | 40% | 能力画像：技能标签 | 与任务所需技能的匹配程度 |
| **信用等级** | 30% | 信用台账 | S/A/B/C/D等级 |
| **负载状态** | 20% | 任务状态 | 当前项目数 |
| **历史好评** | 10% | 交易评价 | 最近5个项目好评率 |

### 4.2 匹配得分计算

```
匹配得分 = 技能匹配度×40% + 信用等级×30% + 负载状态×20% + 历史好评×10%
```

**各维度计算方法**：

| 维度 | 计算公式 | 说明 |
|------|----------|------|
| **技能匹配度** | 匹配技能数 / 所需技能总数 × 100% | 精确匹配技能标签 |
| **信用等级** | S=100, A=85, B=70, C=55, D=40 | 直接换算 |
| **负载状态** | 负载系数 = max(0, 100 - (当前项目数-1)×20) | 项目数越少得分越高 |
| **历史好评** | 最近5个项目好评率的平均值 | 直接使用 |

### 4.3 匹配流程

```
┌─────────────────────────────────────────────────────────────┐
│                    智能匹配流程                              │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  步骤1：任务发布                                             │
│  └── 需求方发布任务，标注所需技能                            │
│                                                              │
│  步骤2：能力解析                                             │
│  ├── 解析任务所需技能                                        │
│  ├── 分析任务难度等级                                        │
│  └── 评估工期要求                                            │
│                                                              │
│  步骤3：候选人筛选                                           │
│  ├── 按技能标签匹配候选人                                    │
│  ├── 过滤不在线/忙碌状态                                    │
│  └── 按匹配度排序                                            │
│                                                              │
│  步骤4：智能推荐                                             │
│  ├── 优先推荐：高匹配度+高好评                               │
│  ├── 次选推荐：高匹配度+低经验                               │
│  └── 兜底方案：发布公开任务池                               │
│                                                              │
│  步骤5：双向选择                                             │
│  ├── 候选人接受任务                                          │
│  ├── 或候选人主动投标                                        │
│  └── 需求方确认最终人选                                      │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 4.4 匹配优先级

| 优先级 | 条件 | 说明 |
|--------|------|------|
| **P0** | 匹配度≥90% + 信用A级以上 | 直接推荐 |
| **P1** | 匹配度≥70% + 信用B级以上 | 优先考虑 |
| **P2** | 匹配度≥50% | 备选方案 |
| **P3** | 匹配度<50% | 需要培训或外协 |

## 五、分包协议要点

### 5.1 分包协议与主协议的关系

```
┌─────────────────────────────────────────────────────────────┐
│                    协议层级关系                              │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  【主协议】SPV合作协议                                        │
│  ├── 签约方：全体项目成员                                     │
│  ├── 有效期：整个项目周期                                     │
│  └── 核心内容：项目治理、分账规则、争议解决                    │
│                                                              │
│  【子协议】分包协议                                           │
│  ├── 签约方：挂帅人+各接包人                                 │
│  ├── 有效期：分包模块周期                                     │
│  └── 核心内容：分包范围、交付标准、付款条件                    │
│                                                              │
│  ┌─────────────────────────────────────────────────────────┐ │
│  │ 子协议受主协议约束                                        │ │
│  │ • 子协议不能与主协议冲突                                  │ │
│  │ • 子协议争议按主协议的争议解决条款处理                     │ │
│  │ • 子协议的签署不影响主协议的效力                          │ │
│  └─────────────────────────────────────────────────────────┘ │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 5.2 分包协议必备条款

| 条款类型 | 必须包含 | 说明 |
|----------|----------|------|
| **分包范围** | ✅ | 明确分包的具体内容和工作边界 |
| **交付标准** | ✅ | 明确交付物的验收标准和形式 |
| **时间节点** | ✅ | 明确各交付物的时间要求 |
| **费用约定** | ✅ | 明确分包金额和支付方式 |
| **质量要求** | ✅ | 明确质量标准和违约责任 |
| **知识产权** | ✅ | 明确交付物的知识产权归属 |
| **保密条款** | ✅ | 明确保密范围和期限 |
| **争议解决** | ✅ | 明确争议解决方式和适用条款 |

## 六、快速检查清单

分包前，确认以下事项：

```
□ 项目已拆解为若干独立模块
□ 每个模块有明确的交付物和验收标准
□ 所需技能已梳理清楚
□ 接包人匹配已完成
□ 分包协议已准备
□ 分包方案已通过必要审批
□ 接包人已确认接受分包
□ 主协议与子协议关系已明确
□ 各方联系方式已确认
□ 协作工具已就绪
```
