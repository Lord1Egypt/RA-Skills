```

│
├─ 回群「收到，开始处理」
│
├─ 解析排期 → 提取所有「内容运营」项目（A 列含「内容运营」才处理）

│   ├─ 已预占 + 素材齐 + 时段端口一致 → 补配（走 phase-two §3.5 配置）→ 已配置

│   ├─ 时段/端口不一致              → 交接③ 复制推广+改时段端口 → 副本有素材则配

│
└─（不发催料/催审批，灵渠助手负责）
```

> 💡 绝大多数项目在阶段二素材齐时已配好并发审批（N=已配置），排期来只是**核对一致性**。真正要动手的只有：①漏配的（已预占+素材齐）补配 ②时段端口不一致的复制改。催审批/催素材全归灵渠助手，数字员工不再发提醒。

**关键决策：**
- §2 解析排期 — 忽略帧位数字，提取项目名/时段/端口组；`↳` 项目继承预占时段
- §3 比对 — 时段+端口 vs 追踪表 E/F/G 列；按 N 列分流
- §4 补配/不一致 — 漏配走 phase-two §3.5；不一致走 §5 交接③
- §5 不一致处理 — 交接③ 复制推广+改时段端口

**红线：**
- 只处理「内容运营」项目，其他跳过
- **一次只完整处理一个项目**（配置/复制→门禁→写表，做完再取下一个）
- **配完/复制后必过收尾门禁**回查后台真实态，不过不通知 PM
- 复制推广后由技术那套点预占，不依赖 PM
- **不发催料/催审批**（灵渠助手负责）
- 所有 @PM 的值取自追踪表 B 列，不凭记忆构造

## 2. 解析帧位排期

### 2.1 提取规则

| 提取 | 规则 |
|------|------|
| 项目名 | 每行 `时段 项目名 等级 数值` 提取项目名 |
| 时段 | 每行开头 `HH点-HH点` |
| 端口组 | 当前段落标题（移动端&iPad&新iPad&WebPC / 荣耀白牌 / Mac&旧iPad） |

**忽略：** 帧位数字、等级、数值。

### 2.2 折行处理

折行（无数字开头）属上一帧的同帧替换：
```
1、0点-12点 郑润泽 单曲
   12点-24点 泰勒 玩具总动员ost   ← 折行，12-24切泰勒
```
提取为两个时段：郑润泽(0-12)、泰勒(12-24)。

### 2.3 `↳` 标记

格式 `↳ 有票/空降 项目名`：继承**预占阶段**的时段（追踪表 E/F 列），不继承父行时段。

### 2.4 端口组映射

| 帧位排期端口组 | 对应客户端 |
|---------------|-----------|
| 移动端&iPad&新iPad&WebPC | 移动端(iPhone&Android), iPad(新版), AndroidPad, Web, PC |
| 荣耀白牌 | 荣耀白牌 |
| Mac&旧iPad | Mac, iPadHD(旧版) |

### 2.5 过滤非内容运营项目

每条排期项目搜追踪表 A 列 → 推广名不含「内容运营」→ 跳过。

## 3. 比对追踪表

### 3.1 全量交叉比对

每次以最新排期为准。扫表按 SKILL.md §3.1 的 4 状态判断。

**比对 A：排期项目 → 追踪表**

排期里是裸项目名（如「王可空降测试」），A 列存的是完整推广名（如`【潜力/全网/其他/其他】王可空降测试 –【内容运营部】`）。搜表时用**包含匹配**（A列推广名是否含排期项目名），仅搜 N=「已预占 / 已配置」（终态跳过）。

| 搜索结果 | 处理 |
|---------|------|
| 1 条 | ✅ 继续比对时段/端口 |
| 0 条 | 记异常列表，通知王可：「排期中有项目 {项目名} 搜不到，可能未预占」 |
| 2+ 条 | 跨天兄弟（核心名去 `_MMDD` 相同）：按排期时段**精确匹配**对应 `_MMDD` 行 |

**比对 B：追踪表 → 排期项目**

追踪表中 **E 列日期（精确日期）= 本次排期所属日期**、N=「已预占/已配置」的项目，在排期中**没找到** → 按以下步骤处理：
1. N→「已取消」（终态）
2. O 列→「排期中消失，本期不上，{MMDD}」
3. **DM 通知 PM**（B 列）：
```
「{项目名}」在本次帧位排期中未出现，已标记为本期不上。
如需上线，请重新告诉我预占时段。
另：后台预占位请你去灵渠后台手动取消。
```

> ⚠️ 精确日期匹配：只检查"E列日期 = 本次排期日期"的行，不做区间重叠检查，避免误杀跨天兄弟（`_0618` 的行不在 0617 排期日期的比对范围内）。
> ⚠️ 本期不上即取消（无独立延期态）。PM 后续要上需重新预占。grp.bjoperation 无删除权限，后台预占需 PM 自行取消。

### 3.2 比对内容

| 比对项 | 排期来源 | 追踪表列 | 判断 |
|--------|---------|---------|------|
| 时段 | 排期提取 | E（开始）/ F（结束） | 完全一致才通过 |
| 端口 | 排期提取 | G（端口） | 排期端口 ⊆ 预占端口为一致（见下方特判） |

**空降/有票端口特判**（H 列 = 空降 / 有票时）：

「移动端&iPad&新iPad&WebPC」section 在 §2.4 映射包含 Web 和 PC，但空降/有票天然仅4端（不上 Web/PC）。比对时从排期端口中**去掉 Web 和 PC** 再与 G 列比对，避免永远误判"不一致"。

```
排期端口（去掉Web/PC后）⊆ 预占端口 G列 → 一致
```

> ⚠️ 仅对 H 列 = 空降 / 有票 的行做此特判；标准/数专/H5 不特判。

### 3.3 比对结果分流

| 结果 | 处理 |
|------|------|
| N=「已配置」+ 时段端口一致 | 不动（已配好并发审批了） |
| N=「已预占」+ 素材已齐 + 时段端口一致 | → §4 补配（走 phase-two §3.5） |
| N=「已预占」+ 素材未齐 | 不动，灵渠助手催料 |
| 时段/端口不一致（无论已配/已预占） | → §5 交接③ 复制改时段端口 |
| 追踪表有、新排期没有 | N→「已取消」 |

> 💡 怎么判"素材已齐"（按 H 列配置方式速查，无需翻 phase-two）：
>
> | 配置方式 | 图片齐 | 链接/ID 齐 | 角标齐 |
> |---------|--------|-----------|--------|
> | 标准 | 大卡607 + 小卡400（蓝框420可选） | 系统自动，不用配 | I/K 列有角标值 |
> | 数专 | 大卡607 + 小卡400（蓝框420可选） | J 列数专ID存在（数字员工拼链接用） | K 列数字专辑/数字单曲 |
> | H5 | 大卡607 + 小卡400（蓝框420可选） | J 列 H5 链接存在 | K 列活动/独家策划/演出 |
> | 空降 | 大卡607（只要大卡） | J 列 orpheus 链接已拼好 | 固定「活动」，不用 PM 提供 |
> | 有票 | 大卡607（只要大卡） | J 列购票链接存在 | 固定「演出」，不用 PM 提供 |
>
> 已齐但 N 还在「已预占」= 阶段二漏配，§4 补配兜底。

## 4. 补配（已预占 + 素材齐）

时段端口一致，N=「已预占」，但 I/J/K/L 物料已齐（阶段二漏配的兜底）：

**直接复用 `phase-two-material.md §3.5` 配置逻辑**：
1. 算/确认业务正确值（J 列拼好的链接或端维度摘要等）。若后台详情缺 `fullType/resourceTypes/resourceInfo`，先补问实际投放资源类型、标准资源 ID、全量/非全量和非全量人群。标准资源补配时必须刷新默认值；Web 端必须 http/https。
2. 交接② 委托技术那套 `promotion-update-creative`（带 businessRemark 直接发审批）
3. 过收尾门禁（phase-two §3.6 同款：查 detail 对账推广ID/物料/已发审批）
4. 写表 N→「已配置」

> 配物料动作与阶段二完全相同，不因在阶段三而不同。审批后续由灵渠助手催。

---

## 5. 不一致处理（交接③ 复制推广 + 改时段端口）

时段或端口不一致。委托技术那套复制推广并调整。

### 5.1 交接③ 委托技术那套

**输入：**
- 原 promotionID（追踪表本行 M 列）
- 算好的新时段（E/F）、新端口（G，转 relatedPositionCodes 由技术那套做）

**技术那套负责：** `promotion-copy`（入参仅 `{id}`，后端自动继承源推广全部字段，含 5 前缀/项目类型/已写入的初始化字段，数字员工无需重传）→ `promotion-update` 改时段端口（草稿状态，投放端才可改）→ 点预占（`promotion-transition` PREOCCUPY）→ `admin-add` 补协作者。

**拿回：** 新 promotionID。

> ⚠️ 复制是否自动带物料由技术那套处理；若未带，副本需在 §5.3 重新配物料（走交接②）。

### 5.2 同步追踪表（复制后立即）

**原始推广：** N→「已替代」（终态）；O 列→「已被副本替代，副本 promotionID={副本ID}，{MMDD}」

**每个副本：** 写独立新行；E/F/G 填排期新值；M 列→新 promotionID；I/J/K/L 继承原始；N→「已预占」

### 5.3 副本配物料

副本 I/J/K/L 已从原始行继承（§5.2 写入）。按 §3.3 速查表判断是否已齐：齐则走 §4 补配（交接② 配物料 → 过门禁 → N=已配置）；不齐 → 保持「已预占」，灵渠助手催。

### 5.4 通知 PM（DM）

```
「{项目名}」因帧位排期调整，已复制推广并预占/配置：
📋 原始推广 → 已由副本替代
📋 新推广 → 时段 {新时段}，状态 {已预占/已配置}
```

通过收尾门禁后才通知。

---

## 6. 收尾门禁（每条配置/复制后必做）

委托技术那套调 `promotion-detail`（id = **当前操作行的 M 列**）回查对账：
- 复制场景 = **副本新行的 M 列**（§5.2 写入的新 promotionID）
- 补配场景 = 当前行 M 列（原推广 ID）
- 物料已填且与该行 I/J/K/L 一致
- 状态符合预期（配物料的已进审批 / 复制的已预占）

全过才写表 + 通知。配错行/状态异常 → 停，报警王可 + 写 O 列，**不通知 PM**（hard-rules 原理六）。

## 8. 边界情况

| 情况 | 处理 |
|------|------|
| 排期中有、追踪表搜不到 | 通知王可：「排期中有项目 {项目名} 搜不到，可能未预占」 |
| 排期含非内容运营项目 | 跳过 |
| 配置中收到新排期 | 用新排期重新比对 |
| 终态项目出现在新排期 | 跳过（终态永久不处理） |
| 取不到群 ID | 不静默丢，通知王可 |
