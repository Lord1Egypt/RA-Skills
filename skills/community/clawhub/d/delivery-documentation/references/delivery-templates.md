# 交付文档模板参考

## 一、技术方案模板要点

### 标题规范
- 格式：`{客户名} - {项目名} - 技术方案 - v{版本号}`
- 示例：`XX公司 - 智能客服 Agent - 技术方案 - v1.0`

### 必须包含的章节
1. 项目背景（为什么要做）
2. 技术方案（怎么做）
3. 实施计划（什么时候做）
4. 验收标准（怎么算做完）
5. 风险与应对（可能出什么问题）

### 架构图规范
使用 Mermaid 语法，包含：
- 系统边界（哪些是项目范围，哪些是外部依赖）
- 核心组件（每个组件的职责）
- 数据流向（箭头标注数据类型）
- 外部接口（API、数据库、第三方服务）

## 二、部署手册模板要点

### 标题规范
- 格式：`{项目名} - 部署手册 - v{版本号}`

### 必须包含的章节
1. 环境要求（硬件 + 软件）
2. 安装步骤（可执行的命令）
3. 配置说明（关键配置项）
4. 启动与验证（怎么确认部署成功）
5. 常见问题（FAQ）
6. 运维指南（日志、监控、备份）

### 部署步骤规范
- 每个步骤附带完整的命令
- 命令后面注释预期输出
- 标注容易出错的地方
- 提供回滚方案

示例：
```bash
# 1. 安装依赖
pip install -r requirements.txt
# 预期输出：Successfully installed ...

# 2. 初始化数据库
python init_db.py
# 预期输出：Database initialized successfully
# 注意：首次部署才需要执行，重复执行会清空数据

# 3. 启动服务
python main.py
# 预期输出：Server running on port 8000
```

## 三、交付报告模板要点

### 标题规范
- 格式：`{项目名} - 交付报告 - {日期}`

### 必须包含的章节
1. 项目概要（一页纸总结）
2. 交付内容（功能 + 文档）
3. 测试结果（数据说话）
4. 已知问题（诚实列出）
5. 后续支持（维护计划）
6. 签字确认（双方签字）

### 测试结果展示
用数据而非文字描述质量：
```markdown
| 测试类型 | 用例数 | 通过 | 失败 | 通过率 |
|---------|-------|------|------|-------|
| 功能测试 | 25 | 24 | 1 | 96% |
| 边界测试 | 10 | 9 | 1 | 90% |
| 性能测试 | 5 | 5 | 0 | 100% |
| **总计** | **40** | **38** | **2** | **95%** |
```

### 已知问题处理
每个已知问题必须包含：
- 问题描述
- 影响范围
- 规避方案
- 修复计划

## 四、文档格式规范

### Markdown 规范
- 标题层级：H1（文档标题）→ H2（章节）→ H3（小节）
- 表格：所有列对齐，内容简洁
- 代码块：标注语言类型
- 图表：优先使用 Mermaid，便于版本管理

### 命名规范
- 文件名：`{项目名}_{文档类型}_{版本或日期}.md`
- 示例：`智能客服_技术方案_v1.0.md`
- 示例：`智能客服_交付报告_2026-06-25.md`

### 版本管理
| 版本类型 | 格式 | 示例 |
|---------|------|------|
| 技术方案 | v{主版本}.{次版本} | v1.0, v1.1, v2.0 |
| 部署手册 | v{主版本}.{次版本} | v1.0, v1.1 |
| 交付报告 | 日期 | 2026-06-25 |
