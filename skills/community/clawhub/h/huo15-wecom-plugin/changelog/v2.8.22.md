## v2.8.22 真根因

trajectory `fa74353b-eab2-49d8-9dad-f0def6db4129` 04:42:13 round：

```
provider=minimax modelId=deepseek/deepseek-v4-pro
sessionKey=agent:main:wecom:group:wrgzumeqaadcsaffbvgfobppv-_6ccwg
assistantTexts: ['\n\n打包好了，10KB，发送中。\n\nMEDIA: ~/.openclaw/media/outbound/huo15-rustdesk-deploy.zip']
```

LLM **完整 emit 了含 MEDIA: 行的文本**（assistantTexts 含完整 75+ 字符）。但 gateway.log 显示 wecom-runtime 看到的：

```
04:42:13.502 deliver-start kind=block messageId=0b63... textLen=14 mediaCount=0
04:42:13.503 outbound summary="bot-ws 打包好了，10KB，发送中。"
04:42:13.504 deliver-done block durationMs=2
04:42:13.906 deliver-start kind=final textLen=0 mediaCount=0
04:42:13.906 outbound summary="bot-ws final"
04:42:13.906 deliver-done final durationMs=1
```

**block textLen=14**（"打包好了，10KB，发送中。" 14 字符）+ **final textLen=0**。**MEDIA: 行（35+ 字符）整段没到 wecom**。

也就是 **OpenClaw 的 stream → channel 链路上，含 `MEDIA:` 的 chunk 被某环节吞了**——可能是：

1. **OpenClaw core 4.x 的 streaming 把双换行 `\n\n` 当 paragraph break，切成两个 chunk，第二个 chunk 在某个 hook/wrapper 上被丢弃**
2. **enhance v6.1.4 model-router 重构副作用**：trajectory 显示 `provider=minimax` 但 `modelId=deepseek/deepseek-v4-pro` —— **跨 provider 错配**，stream 可能在 minimax 接口侧异常终止
3. **某个 middleware 识别 `/Users/...` 路径当敏感信息过滤掉**

不是 wecom 的 bug ——v2.8.21 的 reply.ts MEDIA: parser 没机会触发（接收不到含 MEDIA: 行的 payload，自然 detected/warn log 都不出）。

stream 截断的 root cause 调查留另一个 task（spawn 出去深挖 enhance v6.1.4 与 openclaw core stream 链路）。**v2.8.22 不修 root cause，绕开它**。

## 改动：`index.ts` 重写 `WECOM_BOT_WS_MEDIA_GUIDANCE`

### 优先级翻转

| 路径 | v2.8.21 优先级 | v2.8.22 优先级 |
|---|---|---|
| `enhance_share_file` 工具 | 备用（≥ 20MB） | **首选**（任意大小）|
| `MEDIA: <path>` 字面量 | **首选** | 备用（仅 `enhance_share_file` 不可用时）|

### 为什么 share_file 路径不会被截断

| 维度 | MEDIA: 路径 | enhance_share_file 路径 |
|---|---|---|
| 数据通道 | LLM emit 在 reply text 里，**走 stream** | structured **tool call**，不走 stream text |
| 截断风险 | 任何 stream 中间环节都可能丢 chunk | 工具调用是单次 RPC，要么成要么失败，不会中间丢 |
| 群权限 | 走 `replyMedia`，撞 86008 时降级 share-fallback | 内部直接 `wsClient.sendMessage` 主动推送链接，**完全绕开 86008 路径** |
| 大文件 | > 20MB 触发 share-fallback 落盘 | 任意大小直接落盘 |
| 失败可见性 | 用户看不到，沉默吞掉 | 工具返回 error 字段，LLM 能 retry |

### GUIDANCE 文案要点

1. **明确告诉 LLM：必须优先调 `enhance_share_file`**，不要 emit MEDIA: 字面量
2. **解释原因**：stream 偶发吞 MEDIA: 行（实测）
3. **三步流程**：调工具 → 拿 url → markdown 链接发回
4. **MEDIA: 备用路径**：保留但降级，仅在 enhance 不可用 / 工具列表里没有 share_file 时才用
5. **强约束**：原 ❌ 错误示例对照保留（如果真用 MEDIA: 备用路径，格式仍要严格）

## 这就解决问题了吗？

**预期 95% 解决** —— LLM 在新 GUIDANCE 引导下大概率走 share_file。即使 stream 截断，链接通过 enhance share-fallback 的 `sendMessage` 主动推送进群（不依赖 reply 通道）。

**剩 5% 风险**：
- LLM 仍坚持选 MEDIA:（少见，但不能完全杜绝）→ 落到 v2.8.21 的 detected/warn 诊断 log 路径
- enhance_share_file 链接 baseUrl 公网不可达 → 同样降级到 finalText 文本（同样可能撞 stream 截断）

第一个不做防御编程，第二个 baseUrl 已配 `https://keepermac.huo15.com` 没问题。
