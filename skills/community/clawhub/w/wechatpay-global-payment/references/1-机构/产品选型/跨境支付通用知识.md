# 微信支付境外（跨境）业务世界观

> 本文档面向**机构**，建立对境外微信支付业务的完整认知：业务定义、钱包体系、机构分类、资金流转。
>
> ‼️ **两条硬性规则**（贯穿全文）：
> 1. **接入主体限制**：全球范围内**仅中国香港地区**允许商户以"直联商户"身份直接对接微信支付；**其他所有国家和地区（含美国、日本、欧洲、东南亚等）必须通过"机构"接入**，机构再为下游子商户进件。详见 § 3。
> 2. **结算币种限制**：机构 / 子商户的**结算币种由其注册所在地强制决定**，必须是当地法定本地货币（如美国主体 → USD、日本主体 → JPY、英国主体 → GBP、欧元区主体 → EUR、泰国主体 → THB）。一个号仅对应一种结算币种，不可跨地区选择。详见 § 4 与 § 6。

## 1. 什么是境外支付

### 1.1 定义

**境外支付（Cross-border Payment / Overseas Payment）** 是指微信支付面向中国大陆以外地区的商户和机构提供的支付收单服务。其核心场景是：**持有微信支付（大陆人民币钱包）的用户，在境外商户处进行消费支付**。

从本质上说，这是一种**跨境支付**行为——用户以人民币（CNY）付款，商户以外币（如 USD、HKD、GBP、EUR 等）结算收款，中间涉及换汇和跨境资金清算。

### 1.2 业务背景

微信支付跨境业务始于 2015 年前后，随着中国出境游和跨境电商的蓬勃发展而迅速扩大。截至目前：

- **覆盖区域**：78 个境外国家和地区
- **境外商户数**：超过 600 万
- **支持结算币种**：37 种（详见 [§ 6](#6-支持的结算币种共-37-种)）
- **支持标价币种**：132 种外币标价（不含人民币）
- **核心用户群**：持有中国大陆身份证并完成微信支付实名认证的用户

### 1.3 两大业务方向

境外支付业务可细分为两大方向：

| 方向 | 说明 | 典型场景 |
|------|------|----------|
| **内包外用**（Outbound） | 大陆用户出境消费，在海外商户使用微信支付（大陆钱包）付款 | 出境旅游购物、海外餐饮、跨境电商 |
| **外包内用**（Inbound） | 境外用户使用境外电子钱包，在中国境内商户处扫码消费 | 入境旅游、商务访问，境外钱包扫境内聚合码 |

> **"外包内用"** 的官方定义：非居民个人使用境外电子钱包，在境内进行移动支付的业务。

## 2. 钱包类型与可用范围

### 2.1 微信支付钱包体系

微信支付目前面向境外支付场景提供两种自有钱包，两个钱包之间相互独立，不可互通：

| 钱包类型 | 币种 | 发行地区 | 实名要求 |
|----------|------|---------|---------|
| **大陆钱包（WeChat Pay CNY）** | 人民币 CNY | 中国大陆 | 中国大陆身份证 + 大陆银行卡 |
| **香港钱包（WeChat Pay HK）** | 港币 HKD | 中国香港 | 香港身份证/护照 + 香港银行卡 |

#### 2.1.1 大陆钱包（CNY）

- **境内使用**：全场景可用（线上/线下），覆盖中国大陆几乎所有商户
- **境外使用**：
  - 可在 78 个国家/地区的超 600 万境外商户处消费
  - 支持场景：线下扫码（付款码/扫码支付）、线上 H5/小程序/APP 支付
  - **限制**：受中国外汇管理政策约束，个人年度便利化额度为等值 5 万美元
  - 实时换汇，用户以人民币支付，商户以当地外币结算
- **这是境外跨境支付业务的核心用户钱包**

#### 2.1.2 香港钱包（WeChat Pay HK）

- **香港使用**：覆盖香港本地商户
- **跨境使用**：
  - ✅ 可在 **中国大陆** 扫码消费（跨境付款，自动换汇 HKD → CNY）
  - ✅ 可在 **中国澳门** 扫码消费
  - ✅ 可在部分海外地区消费（日本、韩国、泰国、马来西亚等，持续扩展中）
  - 大陆使用覆盖数千万商户
- **限制**：
  - 与大陆钱包完全独立，不可互相转账
  - 需单独进行实名认证（香港身份证/护照）
  - 跨境消费有单笔和单日限额
  - 未完成高级认证时功能受限

### 2.2 外部合作钱包（外包内用场景）

除微信自有钱包外，微信支付还与多个境外钱包建立了合作关系，支持这些钱包的用户在中国境内扫码消费（"外包内用"）：

| 合作钱包 | 地区 | 货币 |
|---------|------|------|
| **LINE Pay** | 泰国 | THB |
| **TrueMoney** | 泰国 | THB |
| **GCash** | 菲律宾 | PHP |
| **KakaoPay** | 韩国 | KRW |
| **NAVER Pay** | 韩国 | KRW |
| **Touch'n Go** | 马来西亚 | MYR |
| **Boost** | 马来西亚 | MYR |
| **Kplus** | 泰国 | THB |
| **Bakong** | 柬埔寨 | KHR/USD |
| **HiPay** | 蒙古 | MNT |
| **DANA** | 印尼 | IDR |
| **Changi Pay** | 新加坡 | SGD |
| 等（共 16+） | ... | ... |

> 2025 年 9 月，微信支付宣布接入"**跨境二维码统一网关**"，将支持更多境外钱包在境内扫码消费。

### 2.3 外卡内绑

除钱包合作外，微信支付还支持 **"外卡内绑"** 功能——持有 VISA、Mastercard、JCB、Discover、银联国际等七大国际卡组织银行卡的外籍用户，入境后可在微信内绑定银行卡进行扫码消费。

## 3. 接入模式

> 来源：[微信支付境外接入指引 §1](https://pay.weixin.qq.com/doc/global/v3/zh/4012354049)。
>
> 微信跨境支付提供 **2 种主流接入模式**（直连模式 / 机构模式），接入方需根据**公司注册区域**选择适用的模式。

### 3.1 直连模式（Direct Connection）

**定义**：商户与微信跨境支付接口**直接对接**，微信支付直接将交易资金结算给商户，商户独自承担收单责任。

**资金流**：
```
消费者 → 微信支付 ─────► 商户（直接结算）
```

**适用区域**：⚠️ 目前**仅中国香港、英国**两个地区开放此模式。

**特点**：
- 接入方 = 资金接收方 = 商户本身
- 信息流和资金流均由商户自己处理
- 适合具备独立技术能力 + 受当地金融牌照覆盖的大型商户

### 3.2 机构模式（Acquirer Mode）⭐

**定义**：由**收单机构**对接微信支付 API 并二次封装接口提供给商户使用。微信支付将交易资金结算给收单机构，机构再二次结算给商户；机构同时处理完整的交易信息流和资金流。

**资金流**：
```
消费者 → 微信支付 ─► 收单机构 ─► 商户（机构二次结算）
```

**适用区域**：在**所有微信跨境支付开放接入的区域均可使用**，是**微信跨境支付主要的对接方式**。

**特点**：
- 接入方 = 收单机构（如当地银行 / 持牌 PSP / 跨境支付机构）
- 资金接收方 = 收单机构 → 商户
- 机构承担 API 对接、二次封装、子商户进件、跨境分账、海关报关等扩展能力
- 商户作为子商户（`sub_mchid`）经营，请求体统一带 `sp_mchid` + `sub_mchid`
- 适合无境外金融牌照 / 无技术对接能力 / 长尾小微商户

> 💡 可通过[官方合作机构查询页](https://pay.weixin.qq.com/index.php/public/wechatpay_en/partner_search#/)查询所在区域已与微信支付合作的收单机构。

### 3.3 接入模式对比

| 对比维度 | 直连模式 | 机构模式 ⭐ |
|---------|---------|-----------|
| **对接方** | 商户自身 | 收单机构 |
| **资金接收方** | 商户 | 收单机构 → 商户 |
| **资金流向** | 微信支付 → 商户 | 微信支付 → 机构 → 商户 |
| **信息流处理** | 商户处理 | 机构处理 |
| **资金流处理** | 商户处理 | 机构处理 |
| **请求体参数** | `mchid` + `appid` | `sp_mchid` + `sp_appid` + `sub_mchid` |
| **开放地区** | ⚠️ 仅香港、英国 | ✅ 所有跨境支付开放区域 |
| **是否主流方式** | ❌ | ✅ **主要方式** |

### 3.4 选型建议

| 你的情况 | 推荐模式 |
|---------|---------|
| 公司注册在 **香港** 或 **英国**，且有独立技术 | 可选直连模式 |
| 公司注册在**香港 / 英国之外**的所有其他国家和地区 | **必须**走机构模式（找一家已与微信支付签约的当地机构作为接入方）|
| 公司虽在香港 / 英国但无技术 / 资源 | 推荐走机构模式 |
| 长尾小微商户（菜市场 / 餐馆 / 小商铺等） | **必须**走机构模式 |

> 📚 本 Skill 所有文档均以**机构模式**为视角撰写。

## 4. 境外机构的类型

境外接入统一以**机构**为主体，机构再向下接入子商户。机构本身按业务属性可进一步分类。

> ‼️ **接入主体硬性规则（再次强调）**：
> - **仅中国香港地区**支持"直联商户"接入模式（请求体只带 `mchid` + `appid`，无 `sp_mchid` / `sub_mchid`）。
> - **其他所有国家和地区**（美国、日本、欧元区、英国、东南亚、中东、大洋洲等）的商户**必须通过机构接入**，自身只能以子商户（`sub_mchid`）形式经营。无法独立直连签约。
> - 这一限制由微信支付的全球合规与持牌策略决定，不可绕过。境外商户在选择接入路径时，请根据所在地：
>   - **香港** → 可选「直联商户」或「通过机构接入」两种路径
>   - **非香港** → 唯一路径：寻找一家已与微信支付签约的境外机构（PSP / 持牌支付机构 / 当地银行）作为接入方，自己作为该机构旗下的子商户接入

### 4.1 按业务范围分类

#### 4.1.1 纯跨境机构

- **定义**：仅处理跨境支付业务的机构，其机构号只能受理大陆钱包用户的跨境交易
- **特点**：
  - 专注于跨境支付收单
  - 主要服务于需要接受中国大陆游客/消费者付款的境外商户
  - 一个机构号仅对接跨境支付 API，只能收取大陆钱包用户的单
  - 不涉及当地本地钱包的支付业务
- **典型代表**：专业做中国移动支付出海的 PSP（Payment Service Provider）

#### 4.1.2 融合机构

- **定义**：使用**一个机构号**，对接**一套 API**，即可同时受理**大陆钱包（CNY）** 和**香港钱包（HKD）** 用户的交易
- **核心价值**：
  - 一个机构号同时支持两种钱包的收单，无需分别申请两套机构号和对接两套系统
  - 大大降低了机构的对接和运营成本
  - 子商户侧无感知，统一对接机构即可同时接受大陆游客和香港本地用户的支付
- **开通限制**：
  - ⚠️ 目前仅支持 **香港、澳门、日本** 地区的机构开通融合机构号
  - 其他地区的机构暂不支持，仍需通过纯跨境机构模式接入
- **典型代表**：香港本地银行、澳门持牌支付机构、日本当地收单机构

#### 4.1.3 对比

| 维度 | 纯跨境机构 | 融合机构 |
|------|-----------|---------|
| **机构号** | 仅跨境支付机构号 | 一个融合机构号 |
| **对接 API** | 跨境支付 API | 一套 API 同时支持两种钱包 |
| **可受理钱包** | 仅大陆钱包（CNY） | 大陆钱包（CNY）+ 香港钱包（HKD） |
| **开通区域** | 所有跨境支付合规区域 | 仅限香港、澳门、日本 |
| **系统复杂度** | 相对简单 | 一套对接、两种能力 |
| **对子商户价值** | 专精跨境 | 一站式解决大陆 + 香港用户收单 |

### 4.2 机构-子商户层级关系

```
┌────────────────────────────────────────────┐
│   机构（Institution / Acquirer）           │
│   sp_mchid: 10000001（机构号）             │
│   持有：商户证书 + APIv3 密钥 + 私钥         │
│                                             │
│   ├── 子商户 A（sub_mchid: 20000001）       │
│   │    实际经营 / 收款主体                   │
│   │                                         │
│   ├── 子商户 B（sub_mchid: 20000002）       │
│   │                                         │
│   └── 子商户 C（sub_mchid: 20000003）       │
└────────────────────────────────────────────┘

调用接口时统一带：sp_mchid + sub_mchid
签名主体：机构（用机构的私钥签名）
资金归属：按 sub_mchid 路由到子商户账户
```

| 角色 | 说明 |
|------|------|
| **机构（Institution / Acquirer）** | 与微信支付直接签约的银行或持牌支付机构，承担资金结算责任，持有私钥与证书 |
| **子商户（Sub-merchant）** | 在机构下入驻的实际经营商户，是最终的收款主体；由机构通过【子商户进件】API 创建 |

### 4.3 子商户进件能力

机构通过【子商户进件】能力为下游商户创建 `sub_mchid`，详见 [📄 产品介绍 § 5.1](./产品介绍.md#51-子商户进件)。

## 5. 资金流与换汇体系

### 5.1 整体资金流转图

```
用户支付（CNY）                              境外机构收款（外币）
    │                                              ▲
    ▼                                              │
┌────────────┐   T日入账   ┌────────────────┐   T+1换汇   ┌────────────────┐   达标打款   ┌──────────────┐
│ 用户微信   │ ──────────► │  CNY 资金池    │ ──────────► │ 结算币种资金池  │ ──────────► │ 机构  │
│ 零钱/银行卡 │            │ （人民币池）    │             │ （外币池）      │             │ 境外银行账户  │
│            │◄────────── │                │             └────────────────┘            └──────────────┘
└────────────┘   退款CNY  └────────────────┘
                       （仅当日未换汇资金可退）
```

### 5.2 双资金池机制

每个机构入驻后，微信支付侧会自动创建两个资金池：

| 资金池 | 币种 | 用途 |
|--------|------|------|
| **人民币资金池（CNY Pool）** | CNY | 存储 T 日用户支付的人民币资金，也是退款的资金来源 |
| **结算币种资金池（Settlement Pool）** | 对应外币 | 换汇后的外币资金暂存池，达到门槛后打款到机构 / 子商户银行账户 |

### 5.3 正向资金流（用户支付 → 商户收款）

#### 第一阶段：用户支付（T 日）

- 用户在境外子商户处使用微信支付（大陆钱包）消费，以**人民币**从用户零钱或绑定银行卡扣款
- 扣款后资金实时进入该子商户对应机构的 **CNY 资金池**
- 此时资金仍为人民币形态，尚未发生换汇
- 同一天内的所有交易资金都会汇集在 CNY 池中

#### 第二阶段：换汇（T+1 日）

- T+1 日（自然日），系统自动将 T 日积累在 CNY 池中的资金进行**锁定**
- 使用**交易发生时的汇率**（非换汇当天的即期汇率）将锁定的人民币资金兑换为结算外币
- 换汇后的外币资金转入 **结算币种资金池**
- 此操作 365 天每日执行，不受节假日影响
- 换汇完成后，同步生成 T 日的交易对账单

#### 第三阶段：结算打款（满足条件时）

- 系统持续监测结算币种资金池的余额
- 当同时满足以下两个条件时，触发自动打款：
  - ① 结算币种资金池总额 ≥ **800 USD**（或等值其他外币）
  - ② 当日为 **工作日**
- 打款时将资金池内**全部资金**一次性汇入机构的境外银行账户
- 到账时效通常 1-3 个工作日，取决于中间行和收款行的处理速度

#### 第四阶段：机构结算子商户

- 机构根据与子商户约定的结算周期和规则结算资金给子商户

### 5.4 换汇规则

> ‼️ **结算币种硬性规则**：每个机构号的**结算币种由其注册所在地强制决定，必须是该国家/地区的法定本地货币，不可自选**。
> - 例如：美国主体 → 只能 USD；日本主体 → 只能 JPY；英国主体 → 只能 GBP；欧元区主体 → 只能 EUR；泰国主体 → 只能 THB；澳大利亚主体 → 只能 AUD。
> - 一个机构号仅对应一种结算币种，**不允许跨地区选择币种**（例如美国机构不能选择以 EUR 结算）。
> - 若同一集团有多个国家/地区的实体且需多币种收款，需在每个地区分别申请独立的机构号或子商户号。

| 项目 | 说明 |
|------|------|
| **换汇时间** | T+1 日（自然日），365 天每日执行，不受节假日影响 |
| **换汇操作** | T 日存入人民币池的资金在 T+1 日被锁定并自动换汇，转入结算币种资金池 |
| **汇率来源** | 采用**交易发生时的汇率**（即用户付款时锁定的汇率） |
| **结算币种** | **由机构 / 子商户注册所在地强制决定，须为当地法定本地货币** |
| **账单生成** | 换汇操作触发 T 日交易账单的生成 |

### 5.5 结算规则

结算（打款到机构境外银行账户）需同时满足两个条件：

| 条件 | 说明 |
|------|------|
| **金额门槛** | 结算币种资金池总额达到 **800 USD**（或等值其他外币） |
| **日期要求** | 当日为 **工作日** |

- 满足两个条件：微信支付将资金池内 **全部** 资金一次性打款到银行账户
- 任一条件不满足：不触发打款，资金继续在池中累积
- 到账时效：通常 1-3 个工作日

### 5.6 负向资金流（退款、撤销）

#### 5.6.1 退款机制

**这是境外支付中最容易踩坑的点之一：**

| 要点 | 说明 |
|------|------|
| **退款币种** | 退款必须退人民币（从 CNY 资金池扣款） |
| **退款资金来源** | 只能使用 **当日交易产生的人民币资金**（即当天还未被换汇的资金） |
| **NOT_ENOUGH 错误** | 当 CNY 池中可用资金不足时，退款会返回 `NOT_ENOUGH` 错误 |
| **退款时效** | 支持对近 **一年内** 的交易进行退款 |
| **部分退款** | 支持部分退款，累计退款金额不超过订单总金额，最多 **50 次** |
| **退款汇率** | 无论人民币标价还是外币标价，均按**订单交易时的汇率**进行退款 |
| **手续费退还** | 以轧差方式退还，退款本身不额外收取手续费 |

> ⚠️ **资金流向不可逆**：一旦人民币资金池的资金换汇进入结算币种资金池，无法再退回人民币池。这意味着已换汇的资金不能用于退款，退款只能依赖新交易带来的 CNY 资金。

#### 5.6.2 退款资金不足（`NOT_ENOUGH`）的解决方案

‼️ **官方推荐优先级（不可颠倒）**：

| 优先级 | 方案 | 适用场景 | 是否需开通 |
|------|------|--------|---------|
| ⭐ **首选** | **机构 / 商户层定期重试** —— 由机构（或商户）封装重试队列，遇到 `NOT_ENOUGH` 后**保持原 `out_refund_no`** 等待新 CNY 交易资金入池后自动重试 | 绝大多数场景：偶发退款、跨日退款、CNY 池暂时不足 | ❌ 无需 |
| 兜底 1 | **充值退款余额（Recharge Funds）** —— 机构预先充值一笔 CNY 备用金到充值余额账户，退款请求里指定从充值余额扣（`source = REFUND_SOURCE_RECHARGE_FUNDS`，字段名以官方文档为准）| 长时间重试无效、CNY 池长期无新交易、对退款时效要求极高的特殊业务 | ✅ 需向 TS 申请开通 |
| 兜底 2 | **联系微信支付运营手动处理** | 上述方案均无法解决的极端场景 | ❌ 无需 |

**关键约束（所有方案通用）**：
- 重试**必须保持原 `out_refund_no`**，**严禁换单号**——否则会被认为是新退款单据，**可能造成重复退款资金损失**（来源：境外退款 FAQ 原话）。
- 同一笔订单多次退款请求需**间隔 ≥ 1 分钟**；并发退款建议串行/分批。
- 退款最长可对 **1 年内** 的交易发起。

**为什么"定期重试"是首选而不是充值余额**：
- 微信支付的**底层资金清结算逻辑期望商户使用真实交易回流的 CNY 资金做退款**，这是设计原则，符合跨境支付资金合规要求；
- 充值余额需要机构预先垫资，会占用机构现金流；
- 充值余额属于特殊能力，需逐机构申请，不是默认能力；
- 大多数有持续经营流水的商户，CNY 池每天都有新流入，等待重试通常 12~48 h 内可成功。

#### 5.6.3 撤销

撤销仅支持 **付款码支付（Quick Pay）** 产品：

| 要点 | 说明 |
|------|------|
| **适用范围** | 仅 Quick Pay，其他产品无撤销能力 |
| **撤销窗口** | 仅支付当天可撤销，跨日后需走退款流程 |
| **资金来源** | 同退款，从 CNY 资金池扣款 |

### 5.7 对账体系

| 项目 | 说明 |
|------|------|
| **对账单生成时间** | T+1 日，中国时间早上 10:00 前生成 |
| **获取方式** | API 接口下载（文件流）或商户平台手动下载 |
| **包含内容** | 正向交易 + 负向退款/撤销记录 |
| **对账维度** | 按 **子商户 + 币种** 维度对账 |
| **结算查询** | 支持查询当前未结算金额、历史结算日期、结算金额及对应交易日期 |

详细对账问题排查参见 [📄 排障手册](../问题排查/排障手册.md)。
