# 方法论原文参考

本文是方法论文档的核心概念汇编，供 `数据关联` 技能调用。当技能执行过程中需要更深层的方法论依据时阅读本节。

---

## 一、核心理念：以决策为中心

传统做法只存"数据是什么"，不存"为什么这么决策"和"决策后发生了什么"。

**核心转变**：不是简单地表示数据，而是表示**决策**。

AI 理解：分析数据时不要只问"这是什么数据"，要问"这个数据要支撑什么决策"。每个发现的终点应该是一个决策选项。

---

## 二、四维集成框架

| 维度 | 本质 | 在数据分析中的含义 |
|------|------|-------------------|
| **Data** | 整合所有数据形态为完整语义表示 | 把表格字段映射为有业务含义的对象、属性、链接 |
| **Logic** | 决定何时以及如何做决策的推理 | 用规则、计算、模型处理数据，不只靠"看着像" |
| **Action** | 数据是"名词"，行动是"动词" | 分析结论必须指向可执行的操作 |
| **Security** | 谁在什么情况下能看什么 | 多维度控制数据访问，不只标颜色 |

### Data 层面的关键做法

"代替扁平的 golden table，以**对象、属性、链接**的形式呈现企业的全貌。"

- **对象（Objects）**：语义世界中的实体（订单、客户、设备）
- **属性（Properties）**：实体的特征（金额、状态、时间）
- **链接（Links）**：实体之间的关系（外键、引用、血缘）

### Logic 层面的关键做法

"确定性函数、算法和常规统计过程必须作为'工具'，补充大模型的非确定性推理。"

这意味着：
- 能用数学算的不要靠猜（算相关系数而不是说"看起来有关"）
- 明确区分"计算出来的"和"推断出来的"
- 规则、模型、函数是独立于 LLM 的"逻辑资产"

### Action 层面的关键做法

"闭环行动是区分**运营系统**和**分析系统**的关键。"

- 分析结束时必须指向行动
- 行动可以安全地暂存为"场景"（scenario），审核后再执行
- 写回源系统形成闭环

### Security 层面的关键做法

标敏感等级，想给谁看不给谁看：

- 🔴 高风险：姓名、身份证、手机号、银行卡 → 提醒脱敏
- 🟡 中风险：成本、客户名单、内部备注 → 对外剥离
- 🟢 公开：产品名、日期、品类 → 无限制

## 三、教实习生方法论（核心）

这是方法论中最具操作性的部分——AI 分析数据时应该像一个**聪明但没有上下文背景的实习生**。

### 五步流程

| 步骤 | 原文 | 在数据分析中的对应 |
|------|------|-------------------|
| **① 选对象和定义参数** | 从本体中选择对象，把特定属性存为参数 | 确定分析目标对象和关键字段 |
| **② 给高级指令** | 用自然语言给出高级指令 | 确定分析的主线和方向 |
| **③ 解释每个属性的含义** | 向 LLM 解释每个属性在业务中的含义 | 对每个字段做业务解释，不只是列名直译 |
| **④ 提供评分标准** | 给一组要检查的规则——评分标准 | 明确什么算异常、什么算正常、阈值在哪 |
| **⑤ 要推理过程** | 要求 AI 给出得出推荐结论的推理 | 每个发现附带思维链 |

**外加调试验证：** 用实际数据验证分析函数是否按预期工作。

## 四、OAG（Ontology Augmented Generation）

OAG 是 RAG 的升级版，以决策为中心：

- **RAG**：从外部来源检索数据给 LLM
- **OAG**：不仅检索，还使用**确定性逻辑工具**（预测、优化、计算）和**行动**（写回系统）来闭环

关键点：
- "把 LLM 锚定在本体上，就是锚定在业务现实中"
- "AI 可以有效地'展示它的工作过程'"
- 逻辑工具补充了 LLM 不擅长的精确计算

AI 理解：分析时不仅要"查到"数据，还要"算清楚"数据，最后"落地"行动。

## 五、思维链（Chain of Thought）

> "应用展示 AI 正在执行的 CoT 推理步骤，包括访问本体中的对象和运行相关函数。我们不仅对 AI 的推理有完整的可见性，还能看到它如何部署不同的数据和逻辑工具。"

这意味着：
- 展示**每一步**推理过程
- 明确标注用了什么数据、什么工具、什么规则
- 用户可以验证推理的正确性

---

## 六、新团队成员模式

> "每个部署的 AI 助手都可以被视为一个新团队成员——随着团队成员对其表现建立信心，逐步授予更广泛的权限范围。"

- 先理解场景和上下文
- 不懂就问，不装懂
- 逐步加深分析层次，而不是一次全量输出
- 跨会话积累记忆

---

## 七、代理记忆系统

| 记忆类型 | 存储内容 | 数据分析中的对应 |
|---------|---------|----------------|
| **Working Memory** | 当前循环中的即时信息 | 当前数据集、变量、中间状态 |
| **Episodic Memory** | 跨会话的历史信息，带时间标记 | 之前分析过的类似数据集、问题模式 |
| **Semantic Memory** | 按类别组织的知识和技能 | 领域知识（指标含义、行业标准） |
| **Procedural Memory** | 驱动稳定执行的代码和工具使用模式 | 分析模板、计算脚本、规则集 |

AI 理解：在同一次会话内，要积累并复用先前分析的经验，不要每次都从零开始。但跨会话时没有可靠的自动记忆——除非用户当场提供或能从文件中查阅到历史记录，否则就当从零开始，不要凭印象编造。如果用户反复问同一类问题，可以把它加到"程序记忆"里（如 skill 规则或模板），让下次执行自动沿用。

## 八、决策血缘

> "决策血缘端到端地自动捕获——某个决策在何时做出，基于哪个版本的数据，通过哪个应用。"

> "聚合决策可以安全地利用为训练数据来微调模型，并可以蒸馏为目标原则。"

AI 理解：每次分析都要记录：
1. 基于哪个版本的数据？
2. 用了什么逻辑/规则？
3. 得出了什么结论/建议？
4. 用户是否采用了这个建议？
5. 采用后的效果如何？

## 九、评估方法论（AIP Evals）

设计目的是处理 AI 的**非确定性**。

| 概念 | 说明 |
|------|------|
| **测试用例** | 定义的输入和预期输出 |
| **评估器** | 比较实际输出和预期输出的方法 |
| **实验模式** | Grid search 遍历 模型×Prompt 组合找最优 |
| **反馈闭环** | 负面反馈 → 新的测试用例 → 同样的失败可复现 |

**生产准入：** 所有指标严格通过才能上线。

**评估器类型：**

| 类型 | 适用场景 |
|------|---------|
| 精确匹配 | 布尔/分类输出 |
| 字符串相似度 | 文本输出 |
| 文本生成质量 | 报告/描述类输出 |
| AI-as-a-Judge | 主观判断（逻辑是否正确、分析是否完整） |

## 十、Bootcamp 方法论

> "hands-on-keyboard 的沉浸式培训，从零到用例只需要 1-5 天。"

> "参与者 100% 的时间在创造价值，而不是组装零件。"

AI 理解：分析要**快速上手**。不用追求完美，先跑通一条分析线，拿到初步结果再迭代深化。

---

## 十一、行动闭环

> "数据要素是'名词'，行动是'动词'。通过本体驱动的工作流，名词和动词通过人或 AI 的推理组合成完整的句子。"

行动的类型：
- **同步**：直接通过函数集成，实时生效
- **异步**：通过自动化流程，审核后执行
- **提案模式**：AI 创建提案→用户审批→执行

AI 理解：建议行动时要区分"直接可以做的"和"需要审批的"。
