# 语音转文字问题总结

## 问题 1：ffmpeg 缺失

**症状：**
```
Transcription failed: [Errno 2] No such file or directory: 'ffmpeg'
```

**原因：** Whisper 依赖 ffmpeg 处理音频文件，但本机未安装。

**解决：**
```bash
brew install ffmpeg
```

**经验：** ffmpeg 是 Whisper 的必要依赖，音频格式转换、编解码都靠它。

---

## 问题 2：pip 路径未加入 PATH

**症状：**
```
WARNING: The script whisper is installed in '/Users/gg/Library/Python/3.9/bin' which is not on PATH.
```

**原因：** Homebrew 安装 Python 包时，脚本安装到了用户目录，不在系统 PATH 里。

**解决：** 不影响 Python 模块导入，只是不建议直接用 `whisper` CLI 命令。用 `python3 -m whisper` 或直接调用 Python 模块更可靠。

---

## 问题 3：CPU 不支持 FP16

**症状：**
```
UserWarning: FP16 is not supported on CPU; using FP32 instead
```

**原因：** Whisper 默认用 FP16（半精度浮点）加速，但 Apple Silicon CPU 不支持。

**影响：** 只是警告，不影响转录结果，只是速度稍慢。

---

## 问题 4：音频源质量问题（最关键）

### 案例 1：手机扬声器录制电视剧 → 几乎无法转录

**音频来源：**
- 影石360相机录制手机播放的《新白娘子传奇》
- 音频经过两次降质：手机扬声器 → 360相机麦克风

**结果：**
- base 模型：只能捕捉碎片（如"你这瓜多少钱一斤"）
- large 模型：只能识别出视频打赏提示语
- 真正内容几乎完全丢失

**结论：** 二次录制的音频信噪比极低，Whisper 能力再强也无法恢复丢失的信息。

### 案例 2：直接朗读文章 → 转录效果好

**音频来源：**
- 直接对着设备朗读《知更鸟》文章
- 音质清晰，无额外噪音

**结果：**
- small 模型：准确率较高（有少量错字如"茅鹿"应为"忙碌"）
- large 模型：几乎完美，语义完整准确

**结论：** 原始清晰录音的质量直接决定转录效果。

---

## 问题 5：模型选择影响准确度

| 场景 | 推荐模型 | 原因 |
|------|----------|------|
| 快速预览 | `tiny` | 速度快，精度低 |
| 一般用途 | `base` | 平衡 |
| 清晰录音 | `small` 或 `large` | 追求准确 |
| 嘈杂环境 | `large` | 最强降噪能力 |

---

## 核心经验总结

### 1. 音频质量 > 模型选择
- 再好的模型也救不回二次录制的音频
- 录音时距离音源越近越好
- 避免通过扬声器再次录制

### 2. 前置检查清单
- [ ] ffmpeg 已安装
- [ ] openai-whisper 已安装
- [ ] 音频文件可正常读取
- [ ] 磁盘空间充足（模型文件较大）

### 3. 模型大小与速度
- `small` 模型：~470MB，约 20-60 秒转录
- `large` 模型：~2.9GB，约 2-5 分钟转录
- 本机 M1/M2/M3 芯片转录速度较快

### 4. 常见错误处理
| 错误 | 解决方案 |
|------|----------|
| ffmpeg not found | `brew install ffmpeg` |
| whisper not found | `pip3 install openai-whisper` |
| 转录失败 | 检查音频文件是否损坏 |
| 准确率低 | 音频质量差，换 better 模型也无解 |

---

## 建议

1. **录音时尽量用麦克风直接录制**，不要通过扬声器二次录制
2. **会议/访谈场景**：用领夹麦或高质量录音设备
3. **已有二次录制音频**：先做音频增强处理（如降噪、声音分离）再转录
4. **长音频**：分段处理更稳定，避免内存问题
