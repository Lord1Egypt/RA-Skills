# 客户域（Customer Domain）

> 本文档覆盖券商客户域核心业务对象的状态机、操作规则、异常模式与测试点，涵盖CRM、机构客户、零售客户、适当性管理等子域。

---

## CRM客户全生命周期管理

### 客户状态全集
- 潜在客户（Prospect）：已录入系统但未开户的意向客户
- 开户中（Opening）：已提交开户申请，审核流程进行中
- 正常客户（Active）：已完成开户，账户状态正常
- 休眠客户（Dormant）：连续一定期限（通常12个月）无交易的客户
- 冻结客户（Frozen）：因司法冻结、合规要求等暂停交易权限
- 销户中（Closing）：已提交销户申请，清算流程进行中
- 已销户（Closed）：账户已注销（终态）
- 黑名单（Blacklisted）：因违规被列入黑名单，禁止开户（终态，可申诉解除）

### 状态转移规则
| 当前状态 | 触发条件 | 目标状态 | 是否允许 |
|---------|---------|---------|--------|
| 潜在客户 | 提交开户申请 | 开户中 | ✅ |
| 潜在客户 | 命中黑名单 | 黑名单 | ✅ |
| 开户中 | 审核通过+三方存管签约 | 正常客户 | ✅ |
| 开户中 | 审核不通过 | 潜在客户 | ✅ |
| 正常客户 | 连续12个月无交易 | 休眠客户 | ✅ |
| 正常客户 | 司法冻结/合规冻结 | 冻结客户 | ✅ |
| 正常客户 | 提交销户申请 | 销户中 | ✅ |
| 休眠客户 | 客户激活（发起交易/登录） | 正常客户 | ✅ |
| 休眠客户 | 提交销户申请 | 销户中 | ✅ |
| 冻结客户 | 解冻（司法解除/合规通过） | 正常客户 | ✅ |
| 销户中 | 清算完成+资金转出 | 已销户 | ✅ |
| 销户中 | 客户撤回销户申请 | 正常客户 | ✅ |
| 已销户 | 重新开户 | — | ❌（需新建客户号） |
| 冻结客户 | 直接销户 | — | ❌（需先解冻） |

### 允许操作
- 开户：客户提交身份证明、风险测评、三方存管签约，通过审核后开通账户
- 信息变更：客户修改联系方式、地址等非核心信息（实时生效）；修改姓名、证件号等核心信息需人工审核
- 销户：客户无持仓、无负债、无在途交易，资金全部转出后方可销户
- 客户分级：根据资产规模、交易频率、持有产品等维度自动/手动分级（如钻石/金/银/普通）
- 客户分类：按投资者类型分类（专业投资者/普通投资者）

### 禁止操作（反例知识）
- 未完成风险测评即开户：监管要求开户前必须完成投资者风险测评
- 有持仓或在途交易时销户：必须先清仓、撤单、结算完毕
- 冻结状态下发起交易：冻结客户禁止一切交易操作
- 未经客户授权修改核心信息：姓名、证件号变更需客户本人申请+影像资料
- 休眠客户直接交易：部分券商要求休眠客户先完成激活流程（重新风险测评）

### 异常模式
- 开户审核超时：开户申请提交后超过T+3未完成审核 → 测试点：验证超时提醒机制和自动升级处理
- 重复开户：同一身份证号在同一券商重复申请开户 → 测试点：验证一人一户限制（沪市）或多户限制（深市最多20户）的校验
- 信息变更冲突：客户同时在线上和柜台发起信息变更 → 测试点：验证并发变更的冲突检测和处理
- 销户资金残留：销户时账户存在小额资金（如利息）→ 测试点：验证小额资金处理规则（归集或转出）
- 三方存管解约失败：销户流程中银行端解约失败 → 测试点：验证销户流程回滚和重试机制

### 测试点模板
- TP-CUS-001：新客户线上开户，验证流程：身份验证→风险测评→协议签署→三方存管→审核通过，状态变为正常客户（优先级：P0）
- TP-CUS-002：同一身份证号重复开户，验证系统拦截并提示已有账户（优先级：P0）
- TP-CUS-003：有持仓客户申请销户，验证系统拒绝并提示需先清仓（优先级：P0）
- TP-CUS-004：客户连续12个月无交易，验证自动标记为休眠客户（优先级：P1）
- TP-CUS-005：冻结客户尝试下单，验证系统拦截并返回错误码 E_ACCOUNT_FROZEN（优先级：P0）
- TP-CUS-006：核心信息变更（姓名），验证需人工审核+影像资料上传（优先级：P1）

### 中间态处理规则
- 「开户中」为中间态，通常持续T+1至T+3个工作日
- 「销户中」为中间态，需等待清算完成（T+1）和银行解约确认
- 开户中期间客户不可发起交易，但可查询申请进度
- 销户中期间禁止新的交易和资金转入

### 幂等/重试规则
- 开户申请使用 idNo + idType + branchCode 作为幂等键
- 信息变更使用 customerId + changeType + applyTime 作为幂等键
- 销户申请使用 customerId + applyDate 作为幂等键，同一日不可重复申请
- 三方存管签约失败后自动重试3次，间隔5min/15min/30min

## 机构客户管理

### 业务背景
机构客户包括企业法人、事业单位、社会团体、基金公司、保险公司等非自然人客户。机构客户管理涉及尽职调查、资质审核、协议管理等环节，合规要求高于零售客户。

### 核心业务对象

#### 尽职调查（KYC/CDD）
- 基础尽调：营业执照、法人身份、实际控制人、股权结构
- 增强尽调（EDD）：高风险客户需额外调查资金来源、关联关系
- 持续尽调：定期（年度）更新客户信息，监控异常交易

#### 资质审核
- 专业投资者认定：净资产≥2000万或金融资产≥1000万+2年投资经验
- 合格投资者认定：私募产品投资门槛（净资产≥1000万或金融资产≥300万）
- 特殊业务资质：融资融券、股票质押、场外衍生品等需单独审批

#### 协议管理
- 框架协议：与机构客户签署的总协议（有效期、费率、服务范围）
- 补充协议：针对特定业务的补充条款
- 协议续签：到期前30天提醒，逾期未续签自动冻结相关业务权限

#### 集团客户
- 集团架构：母公司+子公司的层级关系管理
- 统一授信：集团层面的授信额度管理和分配
- 关联交易监控：集团内部关联交易的识别和报告

### 异常模式
- 尽调信息过期：机构客户年度尽调未按时完成 → 测试点：验证尽调过期后的业务限制（禁止新开业务，存量业务不受影响）
- 实控人变更：机构客户实际控制人发生变更未及时更新 → 测试点：验证工商信息变更监控和预警机制
- 协议到期未续签：框架协议到期后客户仍发起交易 → 测试点：验证协议到期后的自动冻结和提醒
- 集团授信超限：子公司用信超过集团分配额度 → 测试点：验证集团授信的实时额度校验

### 测试点模板
- TP-CUS-010：机构客户开户，验证尽调流程完整性（营业执照+法人身份+实控人+股权结构）（优先级：P0）
- TP-CUS-011：机构客户年度尽调到期未更新，验证系统限制新业务开通（优先级：P0）
- TP-CUS-012：专业投资者认定，验证净资产≥2000万或金融资产≥1000万+2年经验的校验（优先级：P0）
- TP-CUS-013：框架协议到期前30天，验证系统自动发送续签提醒（优先级：P1）
- TP-CUS-014：集团客户子公司用信，验证不超过集团分配额度（优先级：P1）

## 零售客户管理

### 业务背景
零售客户（个人投资者）是券商最大的客户群体，业务涵盖线上开户、风险测评、客户画像、精准营销等环节。

### 核心业务流程

#### 线上开户流程
1. 手机号注册+短信验证
2. 身份证OCR识别+活体检测
3. 个人信息填写（职业、收入、投资经验）
4. 风险测评问卷
5. 协议阅读+签署（电子签名）
6. 三方存管银行卡绑定
7. 视频见证（部分券商要求）
8. 提交审核

#### 风险测评
- 测评维度：年龄、收入、投资经验、风险承受能力、投资目标
- 风险等级：C1（保守型）/ C2（稳健型）/ C3（平衡型）/ C4（积极型）/ C5（激进型）
- 有效期：风险测评结果有效期为2年，到期需重新测评
- 强制测评：首次开户必须完成；购买高风险产品时若测评过期需重新测评

#### 客户画像
- 基础画像：年龄、性别、地域、职业、资产规模
- 行为画像：交易频率、持仓偏好、登录习惯、资讯阅读偏好
- 价值画像：AUM（管理资产规模）、佣金贡献、产品持有量
- 画像更新：T+1批量更新+实时事件触发更新

#### 精准营销
- 营销规则：基于客户画像和标签的自动化营销触达
- 触达渠道：APP推送、短信、电话、企业微信
- 频控规则：同一客户同一渠道每日最多N次触达
- 营销合规：禁止向风险不匹配客户推荐高风险产品

### 异常模式
- 活体检测失败：人脸识别与身份证照片不匹配 → 测试点：验证活体检测失败后的重试次数限制（通常3次）和人工审核通道
- 风险测评过期：客户购买产品时风险测评已过期 → 测试点：验证系统拦截并引导重新测评
- 营销频控失效：系统故障导致同一客户被重复触达 → 测试点：验证频控规则的原子性和降级策略
- OCR识别错误：身份证信息识别有误 → 测试点：验证OCR结果的人工确认环节和纠错机制

### 测试点模板
- TP-CUS-020：线上开户全流程，验证从注册到审核通过的完整链路（优先级：P0）
- TP-CUS-021：活体检测失败3次，验证系统锁定并引导至柜台办理（优先级：P0）
- TP-CUS-022：风险测评过期客户购买R4产品，验证系统拦截并提示重新测评（优先级：P0）
- TP-CUS-023：客户画像T+1更新，验证交易行为变化后画像标签正确更新（优先级：P1）
- TP-CUS-024：营销触达频控，验证同一客户同一渠道每日不超过3次推送（优先级：P1）
- TP-CUS-025：OCR识别身份证，验证姓名、证件号、有效期的识别准确率和异常处理（优先级：P1）

## 适当性管理

### 业务背景
适当性管理是证券行业核心合规要求，核心原则为"将适当的产品销售给适当的投资者"。依据《证券期货投资者适当性管理办法》，券商必须建立完善的投资者分类、产品分级、匹配规则体系。

### 投资者分类

#### 专业投资者
- 法定专业投资者：金融机构、上市公司、净资产≥2亿的法人/组织
- 认定专业投资者（自然人）：金融资产≥500万+2年投资经验+满足风险承受能力要求
- 认定专业投资者（法人）：净资产≥2000万+2年投资经验
- 专业投资者可申请转为普通投资者（书面申请）

#### 普通投资者
- 风险承受能力等级：C1-C5（保守型到激进型）
- 普通投资者可申请转为专业投资者（需满足条件+书面申请+审核）
- 普通投资者享有更高的保护义务（信息告知、风险揭示、适当性匹配）

### 产品/服务分级

| 风险等级 | 产品类型示例 | 适配投资者 |
|---------|------------|-----------|
| R1（低风险） | 国债、货币基金、保本理财 | C1-C5 |
| R2（中低风险） | 债券基金、银行理财 | C2-C5 |
| R3（中风险） | 混合基金、股票基金 | C3-C5 |
| R4（中高风险） | 股票、期权、分级基金B | C4-C5 |
| R5（高风险） | 期货、场外衍生品、私募 | C5 |

### 匹配规则

#### 正向匹配
- C1投资者只能购买R1产品
- C2投资者可购买R1-R2产品
- C3投资者可购买R1-R3产品
- C4投资者可购买R1-R4产品
- C5投资者可购买R1-R5产品
- 专业投资者可购买所有风险等级产品

#### 不匹配处理
- 风险等级不匹配时：系统弹出风险揭示书，要求客户确认
- 客户确认后可继续购买（"超风险购买"），但需留痕
- 部分高风险产品（如场外衍生品）严格禁止不匹配销售，不可通过确认绕过

#### 双录要求
- 向普通投资者销售高风险产品时需录音录像
- 双录内容：产品介绍、风险揭示、客户确认
- 双录文件保存期限：产品存续期+20年
- 线上销售：通过电子签名+风险确认弹窗替代双录（部分场景）

### 状态全集（适当性测评）
- 未测评（NotAssessed）：客户未完成风险测评
- 测评有效（Valid）：测评结果在有效期内（2年）
- 即将过期（Expiring）：测评结果将在30天内过期
- 已过期（Expired）：测评结果已超过有效期
- 测评中（InProgress）：客户正在进行测评问卷

### 状态转移规则
| 当前状态 | 触发条件 | 目标状态 | 是否允许 |
|---------|---------|---------|--------|
| 未测评 | 客户开始答题 | 测评中 | ✅ |
| 测评中 | 提交问卷 | 测评有效 | ✅ |
| 测评中 | 中途退出 | 未测评 | ✅ |
| 测评有效 | 距过期≤30天 | 即将过期 | ✅ |
| 即将过期 | 超过有效期 | 已过期 | ✅ |
| 即将过期 | 重新测评 | 测评有效 | ✅ |
| 已过期 | 重新测评 | 测评有效 | ✅ |
| 测评有效 | 客户主动重新测评 | 测评中 | ✅ |
| 已过期 | 直接购买高风险产品 | — | ❌ |

### 异常模式
- 测评结果与实际不符：客户故意选择高风险选项以获取更高等级 → 测试点：验证测评结果与客户实际资产/经验的交叉验证
- 产品风险等级变更：产品运作中风险等级上调 → 测试点：验证存量持有客户的适当性重新匹配和通知
- 双录设备故障：录音录像设备异常导致双录失败 → 测试点：验证双录失败后的业务阻断和重试机制
- 批量适当性检查：新产品上线时需对所有潜在客户进行适当性预检 → 测试点：验证批量检查的性能和结果准确性
- 投资者类型转换：普通投资者申请转为专业投资者 → 测试点：验证转换审核流程和权限变更的即时生效

### 测试点模板
- TP-CUS-030：C2投资者购买R3产品，验证系统弹出风险揭示并要求确认（优先级：P0）
- TP-CUS-031：C1投资者购买R5产品（场外衍生品），验证系统严格拦截不可绕过（优先级：P0）
- TP-CUS-032：风险测评过期客户购买任意产品，验证系统强制要求重新测评（优先级：P0）
- TP-CUS-033：向普通投资者销售R4产品，验证双录流程触发（优先级：P0）
- TP-CUS-034：专业投资者认定，验证金融资产≥500万+2年经验的条件校验（优先级：P1）
- TP-CUS-035：产品风险等级从R3上调至R4，验证存量C3客户收到通知（优先级：P1）
- TP-CUS-036：双录文件上传，验证文件完整性校验和存储期限标记（优先级：P1）

### 中间态处理规则
- 「测评中」为短暂中间态，问卷超时（通常30分钟）自动作废
- 「即将过期」为提醒态，不影响当前业务操作，但需触发提醒通知
- 适当性匹配检查应在委托/申购提交前同步完成，不可异步
- 双录流程为同步阻断流程，双录未完成不可进入下一步

### 幂等/重试规则
- 风险测评使用 customerId + assessDate 作为幂等键，同一日多次测评以最后一次为准
- 适当性匹配检查为无状态纯函数，相同输入产生相同结果
- 双录文件上传使用 recordId + fileHash 作为幂等键，防止重复上传
- 投资者类型转换申请使用 customerId + applyDate + targetType 作为幂等键

## 监管要求

### 证监会适当性管理办法核心要求
1. **了解客户**：全面了解客户的基本信息、财务状况、投资经验、风险偏好
2. **了解产品**：对产品进行风险评级，建立产品信息披露机制
3. **适当匹配**：将适当的产品推荐给适当的投资者
4. **风险揭示**：向投资者充分揭示产品风险
5. **留痕管理**：适当性管理全过程留痕，保存期限不少于20年

### 客户信息保护（个人信息保护法）
1. **最小必要原则**：仅收集业务必需的客户信息
2. **知情同意**：收集使用客户信息需明确告知并获得同意
3. **数据脱敏**：展示客户敏感信息时必须脱敏（手机号、身份证、银行卡）
4. **访问控制**：客户信息访问需基于角色的权限控制
5. **数据出境**：客户信息不得未经审批跨境传输
6. **删除权**：客户有权要求删除其个人信息（需平衡监管留存要求）

### 反洗钱（AML）要求
1. **客户身份识别**：开户时验证客户真实身份
2. **大额交易报告**：单笔或累计超过阈值的交易需报告
3. **可疑交易报告**：异常交易模式需主动识别和报告
4. **客户风险等级**：基于洗钱风险对客户进行分级管理
5. **名单筛查**：开户和交易时需筛查制裁名单、PEP名单

## 常见缺陷模式

1. **适当性匹配绕过**：前端隐藏了购买按钮但接口未做适当性校验，可通过API直接购买
2. **风险测评过期未拦截**：测评过期后系统仍允许购买高风险产品
3. **数据权限越界**：客户经理A能查看客户经理B名下的客户详细信息
4. **敏感信息未脱敏**：手机号/身份证在列表页、导出文件中全量展示
5. **开户重复提交**：快速双击提交按钮产生两个开户申请
6. **销户资金校验遗漏**：未校验利息、红利等小额资金即允许销户
7. **双录文件缺失**：高风险产品销售完成但双录文件未关联
8. **营销合规违规**：向C1客户推送R4/R5产品营销信息
9. **信息变更未同步**：客户在A系统变更信息后B系统未同步更新
10. **休眠激活未测评**：休眠客户激活后未要求重新风险测评（测评已过期场景）

## 测试数据构造要点

- 客户数据归属：造数时必须明确归属关系（客户经理、营业部、分公司）
- 权限测试：预置不同角色账号（客户经理/主管/合规/运营）
- 适当性测试：预置不同风险等级客户（C1-C5）和不同风险等级产品（R1-R5）
- 敏感数据：使用模拟数据，禁止使用真实客户信息
- 时间相关：风险测评过期场景需构造2年前的测评记录
- 机构客户：需构造完整的集团架构和关联关系数据
- 开户流程：需准备OCR测试图片、活体检测视频素材
