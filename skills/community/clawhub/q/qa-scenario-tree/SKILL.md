# 场景树构建

## 核心原则

你是一位测试设计专家，擅长将需求转化为结构化测试场景。
**核心原则**：用树结构组织测试场景，确保覆盖完整、层次清晰。
本技能通过主路径/分支路径/异常路径/数据流四类场景实现全面覆盖。

## 场景数量要求（参考值）

**关键指标**：根据项目复杂度调整

| 复杂度 | 倍数 | 示例（显性5条） |
|--------|------|----------------|
| 简单项目 | ×3 | 场景15个 |
| 中等项目 | ×5 | 场景25个 |
| 复杂项目 | ×7 | 场景35个 |

**复杂度判断标准**：
- 简单：单模块、低风险、无并发
- 中等：多模块、中风险、少量并发
- 复杂：跨模块、高风险、高并发

## 场景树结构

```text
根节点：功能模块
├─ 主路径（Happy Path）          → 数量：需求数量 × 1
│   ├─ 场景1：正常流程1
│   ├─ 场景2：正常流程2
│   └─ ...
├─ 分支路径（Alternative Path）   → 数量：需求数量 × 2
│   ├─ 场景A：分支条件1
│   ├─ 场景B：分支条件2
│   └─ ...
├─ 异常路径（Exception Path）     → 数量：需求数量 × 1.5
│   ├─ 场景X：异常情况1
│   ├─ 场景Y：异常情况2
│   └─ ...
└─ 数据流（Data Flow）           → 数量：需求数量 × 0.5
    ├─ 场景α：数据创建
    ├─ 场景β：数据流转
    └─ ...
```

## 构建方法

### 第1步：识别主路径
**问**：用户从开始到结束，最正常的操作路径是什么？

```
主路径识别：
1. 用户起点：用户从哪里开始？
2. 核心步骤：必须执行的步骤有哪些？
3. 用户终点：用户在哪里结束？
4. 成功标志：怎么算完成？

示例（电商下单）：
用户浏览商品 → 加入购物车 → 选择地址 → 选择支付 → 确认支付 → 支付成功 → 生成订单
```text

### 第2步：识别分支路径
**问**：在主路径的每一步，有什么条件分支？

```
分支识别方法：
1. 条件判断：if-else条件是什么？
2. 用户选择：用户有哪些选择？
3. 状态分支：不同状态走不同路径？
4. 配置分支：不同配置走不同路径？

示例（电商下单）：
- 选择地址：有地址/无地址/新增地址
- 选择支付：微信支付/支付宝/银行卡
- 使用优惠券：有/无/过期
```text

### 第3步：识别异常路径
**问**：在每个步骤，可能出什么问题？怎么恢复？

```
异常识别方法：
1. 输入异常：用户输入错误
2. 系统异常：系统故障
3. 网络异常：网络问题
4. 依赖异常：第三方服务问题
5. 并发异常：并发操作问题

示例（电商下单）：
- 加入购物车：库存不足
- 选择地址：地址不存在
- 支付：支付失败
- 支付：网络中断
```text

### 第4步：识别数据流
**问**：数据在模块间怎么流转？在哪创建/更新/删除？

```
数据流识别：
1. 数据创建：在哪里创建数据？
2. 数据读取：在哪里读取数据？
3. 数据更新：在哪里更新数据？
4. 数据删除：在哪里删除数据？
5. 数据一致性：数据在各模块间是否一致？

示例（电商下单）：
- 订单数据：创建订单 → 更新库存 → 生成支付单
- 用户数据：读取地址 → 使用优惠券 → 更新积分
```text

## 场景类型速查表

| 场景类型 | 覆盖目标 | 数量参考 | 典型遗漏 |
|---------|---------|---------|---------|
| 主路径 | 核心业务流程完整性 | 需求数 ×1 | 多角色主路径不同 |
| 分支路径 | 条件分支、用户选择 | 需求数 ×2 | 配置项组合、权限分支 |
| 异常路径 | 各类异常处理 | 需求数 ×1.5 | 第三方超时、并发冲突 |
| 数据流 | 数据CRUD + 一致性 | 需求数 ×0.5 | 跨模块数据同步 |

## 输出示例

**场景：测试电商下单流程**
→ 主路径：浏览→加入购物车→下单→支付→完成
→ 分支路径：未登录下单→跳转登录→继续支付
→ 异常路径：下单时库存不足、支付超时、并发下单
→ 数据流：商品信息→订单数据→支付记录→物流信息

**场景：测试用户注册**
→ 主路径：填写信息→验证手机→注册成功
→ 分支路径：手机号已注册→提示已有账号
→ 异常路径：验证码错误、手机号格式不对、网络中断

场景树构建完成后检查：
- [ ] 主路径是否完整？
- [ ] 分支路径是否覆盖所有条件？
- [ ] 异常路径是否考虑全面？
- [ ] 数据流是否清晰？
- [ ] 场景间是否有重叠？
- [ ] 场景优先级是否标注？

## 场景漂移与处理

场景树常见失控现象及对策：

| 漂移类型 | 表现 | 处理方式 |
|---------|------|---------|
| 场景爆炸 | 分支路径衍生过多，树过于庞大 | 按优先级裁剪，只保留P0-P1分支 |
| 路径遗漏 | 某条分支的关键场景被跳过 | 用需求解构表逐一对照检查 |
| 深度失衡 | 某条分支挖很深，其他分支太浅 | 设定每条分支最大深度 = 主路径深度 ×1.5 |
| 层次混乱 | 主路径里混了异常路径的节点 | 强制按四类重新归类，不允许混排 |
