# Decomposition Lenses

Skill2Team should evaluate multiple decomposition lenses before recommending a team.

| Lens | Plain-language question | Typical use |
|---|---|---|
| Skill package / context package | What reusable capabilities exist? | Keep skills reusable and assign them to owners |
| Human role | What would a real team look like? | PM, researcher, analyst, writer, reviewer |
| Workflow / SOP | What steps happen in order? | Intake, research, verification, analysis, composition, review |
| Task / DAG | Which subtasks depend on others? | Parallel research, sequential gates, local reruns |
| Data source / tool / capability | Which tools or sources drive work? | Web, files, database, code, APIs |
| Local resource allocation | Which bundled resources are evidence, tools, templates, assets, examples, or release material? | Assign local files to owner agents without turning every resource into an agent |
| Quality risk | What must be independently checked? | Producer/verifier/reviewer separation |
| Dynamic collaboration | What appears only mid-task? | Work requests, loops, reruns |
| Skill use category | What market-style category does each skill serve? | coding, research, writing, education, lifestyle, QA |

## Rule

The final recommendation should name the dominant lens and explain why other lenses were used as secondary constraints.
