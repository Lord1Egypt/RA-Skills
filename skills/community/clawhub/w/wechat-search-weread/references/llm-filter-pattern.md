# LLM 语义筛选模式

适用于 cron 作业中需要对搜索结果做语义筛选的场景：搜索返回大量噪音 → LLM 逐条判断 → 仅保留真正相关的内容。

## 为什么不用关键词

关键词规则（黑名单/白名单）有两个致命缺陷：
1. **今天能用，明天过时**：今天的噪音类型（电网拓扑、电源纹波）不一定是明天的噪音
2. **无法理解语义**：同一词语在不同上下文中含义不同。比如"拓扑优化"在结构力学和网络科学中是两个完全不同领域

LLM 判断能理解上下文语义，适应任何新出现的噪音类型。

## 流程

```
搜索 → 提取全部文章 → 打印标题/摘要供 LLM 判断 → LLM 输出 KEEP 列表 → 程序筛选 → 后续处理
```

## 实现模板

### Step A：打印全部候选项（仅标题，不看正文省 token）

```python
import json

with open('/tmp/urls.json') as f:
    articles = json.load(f)

for i, art in enumerate(articles):
    print(f"[{i}] {art.get('title', '')}  |  {art.get('source', '')}  |  {art.get('date', '')}")

print()
print("===== 以上是全部文章，请逐条判断 =====")
print("请列出应保留的序号，格式: KEEP=[0,2,5,7,...]")
```

### Step B：LLM 阅读并输出 KEEP 列表

Agent 阅读所有标题+摘要，按领域知识逐条判断是否属于目标领域。在回复中输出 KEEP 列表。

### Step C：执行筛选

```python
import json

keep_indices = [0,2,5,7]  # 替换为 LLM 判断结果

with open('/tmp/urls.json') as f:
    articles = json.load(f)

filtered = [articles[i] for i in keep_indices if i < len(articles)]

with open('/tmp/urls_filtered.json', 'w') as f:
    json.dump(filtered, f, ensure_ascii=False, indent=2)

print(f"筛选结果: {len(articles)} → {len(filtered)} 篇")
```

## 判断标准编写要点

在 cron prompt 中给出清晰的判断标准：
- ✅ 什么是「属于」（正例特征）
- ❌ 什么是「不属于」（反例特征+细分类型）
- 🎯 核心区分问题（一句话帮 LLM 抓住本质）

示例（拓扑优化领域）：
```
- ✅ 属于：讨论结构拓扑优化（SIMP/BESO/水平集/变密度法），工程应用，软件教程
- ❌ 不属于：
    - 网络拓扑优化、通用优化、数学拓扑学、广告招聘、景观艺术
    - 卖课/课程推广/付费培训：标题含「课程」「专题课程」「一起学习」「训练营」等推销口吻，目标是卖课而非分享知识
- 🎯 核心区分：「对结构做拓扑优化」还是「某系统/网络的拓扑结构优化」
```

### 卖课内容识别特征

| 类型 | 特征词/句式 | 示例 |
|------|-----------|------|
| 课程推销 | 「XX课程」「专题课程」「带你发掘」「一起学习」 | 变密度法拓扑优化课程 / COMSOL专题课程 |
| 培训广告 | 「训练营」「速成班」「从入门到精通」「限时优惠」 | XX软件7天训练营 |
| 引流卖课 | 结尾带二维码/微信号、正文讲一半留一半 | 「扫码获取完整教程」 |

> 💡 判断核心：文章目的是**分享知识/案例**还是**推销付费课程**？如果标题/内容有明显的课程推销属性，即使技术内容相关也要排除。

## 两轮筛选（关键）

当候选文章 > 100 篇时，**单轮筛选必然有漏网之鱼**。必须做两轮：

### 第一轮：粗筛（排除明显噪音）

在主干流程的 Step 4.5 完成。排除标准针对**明显的类别错误**（卖课、电源拓扑、光子学、招聘广告、视觉设计等）。排除目标是**高置信度错误**，宁可保留边界案例。

### 第二轮：精筛（去重后审查新文章）

第一轮粗筛后仍有漏网噪音——尤其是这些类型：
- **行业报告**：标题不涉及 TO，正文只将"拓扑优化"列为众多趋势之一（如「汽车热管理电子风扇行业迎结构性变革」）
- **综合概览**：AI/工具/行业 overview 文章，仅在列表中提一句 TO（如「AI生成CAD-BIM方法很多」「一个机械工程师眼中的人工智能」）
- **非结构优化**：管道/电缆布线路由优化、物流路径优化（用"拓扑"指网络布局，非结构设计）
- **无关领域**：水泥窑工艺、沸石转轮化工、液流电池 CFD（TO 非核心，仅是边缘工具）

**第二轮流程**：
1. 去重后得到"新文章"列表
2. **先写入临时文件**（`/tmp/new_articles.json`），暂不更新持久化历史
3. 逐条审查新文章的标题+摘要
4. 如发现误保留文章，**从临时文件移除 + 不将其 dataId 加入 seen_ids**
5. 审查通过后再更新持久化 JSON

> 🔴 如果第二轮发现误保留文章且已写入持久化 JSON，必须**回退**：从 `articles` 数组末尾移除对应条目，从 `seen_ids` 中移除对应 dataId。

## 自主模式（Cron/无人作业）

当无人交互时，交互式的 `KEEP=[]` 流程不适用。改用**单次编程式排除**：LLM 阅读全部标题后，直接在 `execute_code` 中构建 `exclude` 集合并过滤。

```python
# 一次性完成：打印标题 → LLM 审查 → 编码 exclude 集合 → 过滤
exclude = {
    3,   # 技术邻招聘
    56,  # KNX 网络拓扑
    87,  # 光子结构
    # ... 逐个编号
}

kept = [a for i, a in enumerate(articles) if i not in exclude]
```

**两轮过滤在自主模式下的执行顺序**：
1. 第一轮：全量粗筛（对 `/tmp/urls.json` 全部文章）→ 写入 `/tmp/urls_filtered.json`
2. 去重：从 `/tmp/urls_filtered.json` 中排除已见 dataId → 写入 `/tmp/new_articles.json`
3. 第二轮：精筛仅对新文章 → **先审查再写持久化**（避免回退操作）
4. 确认无误后一次性追加到持久化 JSON

> 💡 第二轮精筛在去重**之后**执行（而非之前），这样只需审查少数新文章，大幅节省 token。

### 第二轮精筛的额外排除类型

以下类型常在第一轮漏网（标题含「拓扑优化」但实际非结构 TO）：
| 类型 | 特征 | 示例 |
|------|------|------|
| 3D打印趋势 | 标题讲3DP/增材制造，TO只是手段之一 | 「航空航天融合3D打印与AI，未来真的不用画图了？」 |
| 行业报告 | 标题广泛，TO仅为文中一小节 | 「汽车零部件产业技术发展与制造趋势」 |
| 活动/会议通知 | 论文学术交流会、工作营招生 | 「中国航发涡轮院召开…学术交流会」 |
| 期刊目录 | 期刊当期目录汇总 | 「期刊目录丨《推进技术》2026年第五期」 |
| 尺寸优化（非拓扑） | 明确写「尺寸优化」而非「拓扑优化」 | 「基于OptiStruct的电池包壳体尺寸优化」 |
| 纳米/生物医学 | 用了「拓扑」指纳米结构，非力学TO | 「人工拓扑纳米结构…生物医学应用」 |
| 通用设计优化 | 不特定于TO的轻量化/结构设计概览 | 「在保证强度前提下减轻零件重量的几种思路」 |

## 关键词排除陷阱

| 陷阱 | 说明 | 修复 |
|------|------|------|
| 裸 `GPU` 匹配 | `GPU` 在 TO 文章中常指 GPU 加速计算（如「基于 CPU-GPU 异构并行…拓扑优化」），不是网络拓扑 | 用更具体模式：`NCP-AIN\|英伟达.*网络认证\|面向AI.*网络` |
| `通信` 匹配 | 同上，可能匹配到「多物理场耦合通信」等工程描述 | 限定为 `通信网络\|网络通信协议\|5G通信` |
| `路由` 匹配 | 可能匹配「传力路径」的工程语境 | 限定为 `路由协议\|网络路由\|IP路由` |

> 💡 编写第一轮关键词排除时，测试数据集中的前 20 篇标题确认无假阳性。宁可保留边界案例，不要在粗筛阶段过度排除。

## 注意事项

- **仅根据标题+摘要判断**，不读正文，省 token
- 候选项很多时（>100篇），打印会被截断。分两次 execute_code 调用：前半段 + 后半段
- 筛选后写入新文件（如 `/tmp/urls_filtered.json`），不动原始数据
- 后续去重/格式化从筛选后的文件取数据
- **输出格式化必须用 execute_code 从 JSON 读 URL**，terminal 输出会截断长 URL
- **自主模式第二轮必须先审查再写持久化**：先写 `/tmp/new_articles.json`，审查通过后再追加到持久化文件，避免回退操作
- **持久化 dedup 用 dataId 做 key**：每篇微信文章有唯一 `data-id`，首次运行时初始化 `seen_ids` 集合，后续追加时检查
