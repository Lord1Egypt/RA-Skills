# 测试用例设计细则

> 来源：兴业证券股份有限公司测试用例设计细则（正式文件）
> 适用范围：所有测试阶段的用例设计
> 依据：《证券基金经营机构信息技术管理办法》、《证券期货业软件测试规范》

---

## 一、用例基本结构要求

### 必须包含的三个部分

每个测试用例必须包含：**基本信息 + 主体信息 + 执行结果**

### 基本信息（必填）

| 字段 | 说明 | 要求 |
|------|------|------|
| 用例编号 | 唯一标识，系统/项目内唯一 | 具有唯一性和易识别性 |
| 创建者 | 用例编写人 | 必填 |
| 执行人 | 用例执行人 | 必填 |

### 主体信息（必填）

| 字段 | 说明 | 关键要求 |
|------|------|---------|
| 用例简述 | 测试内容概述 | 简洁易懂，不含具体操作步骤 |
| 菜单 | 对应菜单路径及名称 | 必填 |
| 预置条件 | 执行前必备条件 | 原则上所有用例都有预置条件 |
| 操作步骤 | 执行步骤 | 清晰、无歧义、无假设性词汇 |
| 期望结果 | 预期输出结果 | 不能为空，不含步骤，无歧义 |
| 优先级 | 核心/高/中/低 | 见优先级定义 |
| 用例类型 | 功能/性能/安全/可靠性/兼容性 | 必填 |

---

## 二、预置条件编写规范

预置条件必须包含以下三个要素：

1. **明确入口和角色/权限**：包含同一功能的不同入口等详细说明
2. **明确账号类型和权限**：如注册用户/非注册用户，普通账号/信用账号
3. **明确数据准备方案**：对于不便具体列示的数据，要详细描述各项特性；涉及数据库表的，应写明库表字段及对应脚本

### 测试点模板
- TP-PRE-001：预置条件中账号类型未明确 → 用例不合格（P0）
- TP-PRE-002：预置条件中数据准备方案缺失 → 用例不合格（P0）
- TP-PRE-003：预置条件中入口未明确 → 用例不合格（P1）

---

## 三、操作步骤编写规范

### 规范要求
1. 页面名称、按钮名称和操作方式等需明确说明
2. 多个步骤需标注序号，操作步骤和结果应一一对应
3. **禁止出现假设性词汇**：假如、或许、可能、有时候等
4. **禁止出现二义性语句**

### 测试点模板
- TP-STEP-001：步骤中出现"可能"等假设性词汇 → 用例不合格（P0）
- TP-STEP-002：步骤序号与期望结果不对应 → 用例不合格（P1）
- TP-STEP-003：步骤描述中页面/按钮名称不明确 → 用例不合格（P1）

---

## 四、期望结果编写规范

### 规范要求
1. 每个用例必须有期望结果，**结果不能为空**
2. 期望结果中应包含需验证的结果，**不应有步骤**
3. 预期结果中**不应有歧义或二义的语句**
4. 一个结果有多个检查点时，应确保检查点完整，并标注序号
5. 结果应包含需验证的所有结果输出：**页面检查、存储检查、消息检查**等

### 测试点模板
- TP-RESULT-001：期望结果为空 → 用例不合格（P0）
- TP-RESULT-002：期望结果中包含操作步骤 → 用例不合格（P0）
- TP-RESULT-003：多检查点未标注序号 → 用例不合格（P1）
- TP-RESULT-004：缺少存储检查（数据库验证）→ 用例不合格（P1）

---

## 五、优先级定义

| 优先级 | 定义 | 冒烟测试 |
|--------|------|---------|
| 核心 | 该用例无法执行成功，系统基础功能无法执行 | ✅ 必须是冒烟测试用例 |
| 高 | 验证重要功能、主流程功能的正确性及稳定性 | 可选 |
| 中 | 全面验证各分支流程功能，包含界面测试等 | 不包含 |
| 低 | 包含异常测试用例等 | 不包含 |

**强制要求**：所编写功能测试用例中**必须包含核心功能用例**（冒烟用例）

> **冒烟用例数量控制（三档分档，与 B7 对齐）**：总用例≤ 15 条时20%~35%；16~30 条时15%~25%；>30 条时10%~20%。详见 `specs/B7_测试点→用例映射规则.md` §6。

## 六、功能测试用例设计四步法

### 第一步：场景验证
- 正常路径验证：一个功能正常流程，编写 **1个** 测试用例
- 分支路径验证：
  - 等价类分析法：每个有效等价类对应1个用例（可覆盖多个有效等价类）；每个无效等价类对应1个用例
  - 场景法：每条场景或独立路径对应1个用例

### 第二步：功能验证
- 功能内组合测试：等价类分析法，每个有效等价类对应1个用例
- 功能间组合测试：正交实验法，每个组合覆盖1个用例

### 第三步：界面测试
- 边界值分析法：每个边界值划分对应1个用例（可覆盖多个边界值划分）

### 第四步：异常验证
- 异常流程验证：一个功能中多个异常流程，应**分开编写多个**测试用例
- 异常操作验证：错误推测法+边界值分析法
  - 异常数据：空数据、超长数据、特殊字符，每个类型至少1个用例
  - 异常操作：省略关键步骤、重复操作步骤、插入多余步骤，每个类型至少1个用例

---

## 七、测试方法选用规则

| 场景 | 必须使用的方法 |
|------|--------------|
| 任何情况 | 边界值分析法、等价类分析法 |
| 输入条件较少 | 因果图分析法 |
| 输入条件较多 | 正交实验分析法 |
| 完善补充、异常测试 | 场景分析法、错误推测法 |

---

## 八、测试数据设计规范

### 数据引入方式
1. **建议自建数据**：以正常业务流程操作建立数据，无特殊情况不得使用直接变更库表方式制造的数据
2. 如申请引入生产数据，应符合生产数据使用、管理安全制度要求

### 数据集设计要求（必须考虑）
1. **无数据**：使用空白数据，检查是否有适当提示信息
2. **有效数据集**：符合需求设计文档与用户使用场景的数据，结合等价类法、边界值法和场景法
3. **无效数据集**：非法数据、特殊字符、超长数据等，检查是否有适当报错或容错机制

---

## 九、非功能测试用例设计

### 兼容性测试
- **Web类系统**：必须包含浏览器兼容性测试；可选屏幕分辨率兼容性测试
- **客户端类系统**：必须包含操作系统兼容性测试、软件向前向后兼容性测试
- 被测目标选取优先级：客户占有率 > 市场占有率 > 行业标准 > 业内规范

### 可靠性测试（以下场景必须进行）
- 系统初次建设
- 基础功能的增加或优化
- 用户反馈的可靠性问题修复

必须考虑的场景：进程崩溃或挂起恢复后 / 网络断开及延迟 / 弱网环境 / 应用间切换

### 性能测试（以下场景必须进行）
- 新系统上线或重要模块变更
- 变更性能或容量
- 系统定期性能测试
- 项目组评估必须进行

**性能指标体系**：
- 处理能力：TPS（推荐统一使用，包括极限TPS和最佳稳定TPS）
- 处理效率：指定并发压力下的平均响应时间
- 资源消耗：CPU使用率、内存使用率、网络及磁盘消耗、中间件/数据库性能指标

## 十、用例维护规范

| 场景 | 要求 |
|------|------|
| 需求新增 | 严格按项目排期制定测试计划，新增测试用例并建立用例库 |
| 版本迭代 | 版本上线后**五个工作日内**，将测试用例归入用例库 |
| 新发现缺陷/生产故障 | 若基于现有用例无法发现，需及时转化为测试用例并归入用例库 |
| 需求变更 | 及时修改测试用例，**至少每季度**定期维护和检查用例库 |

---

## 十一、用例评审检查清单

评审时逐项检查，不合格项需返工：

```
□ 用例编号唯一且易识别
□ 预置条件包含：入口/角色/账号类型/数据准备
□ 操作步骤无假设性词汇、无二义性
□ 期望结果非空、不含步骤、无歧义
□ 多检查点已标注序号
□ 期望结果包含页面检查+存储检查+消息检查
□ 优先级标注正确（核心用例已标注）
□ 用例类型已填写
□ 冒烟用例已包含（核心优先级用例）
□ 异常场景已覆盖（空数据/超长数据/特殊字符）
□ 边界值已覆盖
```

## 十二、AI 用例生成字段映射表（P6 输出 → 公司模板）

> 本节说明 P6 Prompt 输出的 JSON 字段与公司测试用例模板（19列）的对应关系，供 RAG 召回时建立语义关联，避免字段名不一致导致的映射错误。

| P6 输出字段（英文） | 公司模板字段（中文） | 说明 |
|-------------------|-------------------|------|
| `case_id` | 用例编号 | 格式：TC-{模块}-{序号} |
| `title` | 标题 | 简洁描述测试内容，不含具体步骤 |
| `menu_path` | 菜单 | 对应菜单路径，如"交易/委托下单" |
| `preconditions` | 预置条件 | 必须包含三要素：①账号/权限准备 ②测试数据构造路径 ③环境/配置要求 |
| `steps` | 操作步骤 | 有序步骤，每步一行，无假设性词汇 |
| `expected_results` | 期望结果 | 非空、不含步骤、无歧义，含页面+存储+消息三层检查 |
| `priority` | 优先级 | P0/P1/P2/P3 → 导出时自动映射为公司字典 Highest/High/Medium/Low（字典值：Highest/High/Medium/Low/Lowest，务必匹配） |
| `is_smoke` | 是否冒烟用例 | 是/否 |
| `case_type` | 类型 | 固定值"测试用例" |
| `test_case_type` | 用例类型 | 正向验证/分支验证/异常处理/边界验证/逻辑组合/集成异常/歧义验证/权限验证/状态迁移/兼容回归/性能验证/安全验证/接口验证 → 导出时自动映射为公司字典「正例」或「反例」（字典值只有这两个，正向类→正例，异常/边界/安全类→反例） |
| `test_category` | 测试类别 | 功能/性能/安全/兼容性 → 导出时自动映射到公司字典（功能测试/性能测试/安全性测试/兼容性测试/可靠性测试/可维护性测试/可移植性测试，无特殊情况默认「功能测试」） |
| `status` | 执行结果 | 默认置空，执行时填写 |
| `creator` | 创建者 | AI生成时固定填写"AI生成" |
| `assignee` | 经办人 | 由测试负责人填写 |
| `remarks` | 备注 | 按功能模块分类标注，格式：`[一级分类/二级分类] 关联规则/风险点/PCI编号`，用于思维导图评审模块划分 |
| `source_test_point` | （内部字段，不导出） | 关联的测试点ID，用于追踪 |
| `automation_score` | （内部字段，不导出） | 自动化适用性评分，参见 automation_spec.md §12 |

**使用说明**：
- P6 输出的 JSON 通过 `export_excel.py` 转换为公司 Excel 模板格式
- 字段名映射在 `export_excel.py` 的 `COLUMNS` 常量和 `normalize_value` 函数中维护
- 如公司模板字段有变更，只需更新 `export_excel.py`，无需修改 P6 Prompt
