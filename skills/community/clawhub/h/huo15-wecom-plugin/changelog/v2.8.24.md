# v2.8.24 (2026-05-05)

**企微 UI 锁交互修复 — placeholder 120s timeout 用 last=true 终结 stream**

## 背景

赵博反复反馈："企微发过来的消息我不能复制、不能点链接，要等下面出现「内容由AI生成」标识后才能交互。但**有时候很久都无法出现「内容由AI生成」**，导致没法复制回复内容。"

## 根因

企微智能机器人（aibot）走流式协议：

| `replyStream(frame, streamId, content, last)` 参数 | 企微 client UI 行为 |
|---|---|
| `last=false`（partial / placeholder） | "AI 生成中…"，**禁用所有交互**（防止用户点了不完整的链接/复制半截文本） |
| `last=true`（final / 终结） | 显示「内容由AI生成」标识 + **解锁交互**（复制 / 点链接 / 长按） |

reply.ts 之前的状态机：

- `sendPlaceholder()` 一直用 `last=false` 推 "⏳ 正在思考中..."（每 3s 一次）
- `placeholderTimeout` 在 120s（`MAX_KEEPALIVE_MS`）后触发 → 调 `stopPlaceholderKeepalive()` 仅 clear timer **没发 last=true**
- 结果：streamId **永远停在最后一帧 last=false**，企微 client UI 锁死

LLM 长任务场景（reasoning 链长 / 多轮工具调用 > 120s）触发频率高：

```
[wecom-ws] frame ... reqId=A
[wecom-ws] uploaded placeholder "⏳ 正在思考中..."  (last=false, t=0)
[wecom-ws] uploaded placeholder "⏳ 仍在处理中（已 30s）..."  (last=false)
[wecom-ws] uploaded placeholder "⏳ 任务较复杂（已 1m）..."  (last=false)
[wecom-ws] uploaded placeholder "⏳ 任务仍在执行（已 2m+）..."  (last=false, t=120s)
[wecom-ws] stopPlaceholderKeepalive  ← 这里只 clear timer，没发 last=true!!!
... LLM 还在思考更久 ...
... 用户看到这条 placeholder 但 UI 锁，不能复制不能点 ...
```

## 修复

[`src/transport/bot-ws/reply.ts:323-360`](src/transport/bot-ws/reply.ts:323) — `placeholderTimeout` 触发的回调改成 async，先发一次 `replyStream(streamId, finalText, true)` 终结 stream，再 stopKeepalive：

```diff
  placeholderTimeout = setTimeout(async () => {
+   // v2.8.24 ⭐ 关键修复：120s 触发时用 last=true 终结当前 stream，让企微 UI 解锁
+   // 交互（"内容由AI生成"标识 + 可复制 / 可点链接）。
+   if (!streamSettled && !isEvent && streamId) {
+     const finalPlaceholderText =
+       progressMode === "progress"
+         ? buildProgressPlaceholder(Date.now() - startedAt)
+         : computeCurrentPlaceholder();
+     try {
+       await params.client.replyStream(
+         params.frame,
+         streamId,
+         finalPlaceholderText,
+         true, // ← 关键：last=true 终结 stream
+       );
+       streamSettled = true;
+       console.log(
+         `[wecom-ws] placeholder timeout 120s: streamId=... 已用 last=true 终结，企微 UI 解锁交互`,
+       );
+     } catch (e) {
+       // 终结失败（reqId 失效 / WS 断）不致命；企微 client 长 idle 后自己解锁
+       console.warn(`[wecom-ws] placeholder timeout finalize 失败（${...}）— 企微 UI 会等长 idle 自己解锁`);
+     }
+   } else if (progressMode === "progress" && !streamSettled) {
+     // 兼容：streamId 还没生成（极小概率）→ 走原行为
+     sendPlaceholder();
+   }
    stopPlaceholderKeepalive();
  }, MAX_KEEPALIVE_MS);
```

终结时用最终档 progress 文案（"任务仍在执行（已 X m+），完成后会主动推送结果"）—— 用户看到的最后一帧明确告知"仍在跑，结果会另起一条消息"，且 UI 解锁可复制 + 点链接。

## 后续 LLM 真出 final 的行为

- 旧 streamId 已被 last=true 终结 → LLM 出 final 时 reply.ts deliver 的 `replyStream(streamId, finalText, true)` 会撞 `errcode 846605/846608`（reqId 失效 / 流过期）
- v2.8.17 已有处理：catch 块走 sendMessage 主动推送 → **新消息**进群（默认可交互）
- 用户体验：群里看到两条消息——
  1. "⏳ 任务仍在执行..."（解锁可复制 / 含可点链接）
  2. 真正的 final 结果（独立消息，正常可交互）

跟之前**第一条永远锁住**比，明显改善。

## 兼容性

完全向下兼容。

- 行为只在 placeholder timeout 触发时多发一次 `replyStream(...,true)`
- 老 LLM 短任务（< 120s 出 final）路径**不受影响**——deliver 的 final 自己会发 last=true 终结
- 失败回退到原行为（stopKeepalive + idle 等长 idle 解锁）

## 残留 / 后续可能

**短任务场景**：LLM 偶尔出 final 失败（ack timeout / reqId 失效）走 fallback sendMessage—— 此时原 streamId 仍卡 `last=false` 状态。**这种边界场景本版没修**，因为：

- v2.8.17 catch 块发的是 sendMessage（新消息），用户看新消息可交互
- 原 stream 卡死的"思考中..."消息企微 client 在长 idle 后自己解锁
- 修这个 case 需要在 fallback 之前先尝试 last=true（risk: race between fallback path and finalize）

观察 v2.8.24 部署后赵博"复制不了"投诉是否消失。如果仍偶发就上 v2.8.25 修这个 edge。

## 经验沉淀

**流式协议必须显式终结**：任何 stream-based 通道（企微 aibot、OpenAI streaming completion、SSE 等）都有"流式中"和"流终结"两个状态。客户端 UI 通常根据流终结信号解锁交互。**永远不要让 stream 中途卡在 partial 状态结束**——即使内容不完整，也要发一次终结信号让 client UI 解锁。

之前 v2.8.x 的 reply.ts 设计假设"deliver final 会及时来"——但实际 LLM 长任务 / 异常退出 / 工具链卡死时这个假设破裂。本版用 `placeholderTimeout` 兜底强制终结，是更稳的状态机。
