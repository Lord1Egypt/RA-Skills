# Research Worker Prompt Template

用于第一步的三路信息收集。支持 subagent 的环境可并行执行；不支持时按三轮顺序执行。

---

## 分工

- **Agent 1 — 时间线信息**：起源、创始人背景、发展历程、关键事件、版本迭代、融资、战略转向、危机
- **Agent 2 — 同期格局信息**：竞品识别、各竞品的特点和用户口碑、行业对比评测、市场份额、商业数据
- **Agent 3**（复杂对象才启用）：补充信息，创始人深度背景、行业环境变化、最新数据校验、用户社区讨论（GitHub issues、Reddit、Twitter/X、知乎）

---

## 联网指引（直接贴进每个 research worker 的 prompt）

> 你需要联网获取信息。使用以下工具：
>
> - 使用当前 Agent 可用的 web search、browser、fetch、crawl 或 CLI 工具
> - 搜索策略：先搜索发现信息来源和线索，找到具体 URL 后定向读取原文
> - 多次搜索、多个关键词组合，不要只搜一次就放弃
> - **必须查最新数据**：在完成常规资料后，用当前日期、近 12-24 个月、最新财报/公告/监管文件/原始研究/权威报道等关键词再查一轮；没有完成最新数据校验，不得建议进入写作
> - **一手来源优于二手来源**：优先找官方、原始文件、原始论文、原始访谈；媒体报道、行业报告和用户口碑默认按二手来源处理
> - **学术类研究对象必查 arxiv**：涉及学术概念、算法、AI 模型、技术范式时，必须通过 arxiv API 获取相关论文
> - **公司研究必须补定量材料**：优先找收入/ARR、用户或客户、利润率、现金流、融资、估值、可比公司倍数、市场份额、增长率。上市公司优先年报、季报、招股书、交易所文件；非上市公司优先公司披露、投资方公告、可信访谈和权威媒体原创报道。找不到时必须列为缺口，不要用叙事替代数字。
>
> arxiv 调用格式：
> ```bash
> 通过当前 agent 的网页读取能力访问 arXiv API 查询页，例如 `https://export.arxiv.org/api/query?search_query=all:关键词1+AND+all:关键词2&max_results=10`
> ```
> 或用 WebFetch 访问同一 URL。返回 XML 格式，包含标题、作者、摘要、发布日期、PDF 链接。可按需调整关键词组合和结果数量。找到关键论文后，用 WebFetch 读取 `https://arxiv.org/abs/{论文ID}` 获取细节。

## prompt 撰写守则

**描述目标**，不暗示手段。用「获取」「调研」「了解」这类动词，**不用**「搜索」「爬取」这类暗示具体工具的动词 —— 让子 Agent 自主判断最佳获取方式。

**示例**：

- ✅ 「请调研 DeepSeek 从 2023 年成立到 2026 年 V4 Pro 发布的完整技术演进路径，特别关注版本更迭节点和背后的技术决策」
- ❌ 「请搜索 DeepSeek 的发布时间线并爬取每个版本的 GitHub release notes」

## 信息来源优先级与类型判定

一手来源优于二手来源。多个媒体引用同一个错误会造成循环印证假象。每条来源在返回时必须标注 `来源类型` 和 `判定理由`。

### 来源类型

- **一手**：事实的原始发布者、当事方、原始文件或原始数据。
- **二手**：对一手材料的报道、整理、转述、分析、评论或聚合。
- **存疑**：找不到原始出处、只有转载/聚合、来源自相矛盾、页面不可访问，或无法判断是否为原始材料。

### 常见判定表

| 信息类型 | 优先找的一手来源 | 常见二手来源 |
|---------|----------------|-------------|
| 产品更新/技术决策 | 官方博客、产品文档、GitHub Release Notes、代码仓库、创始人/团队原帖 | 媒体报道、测评文章、Newsletter、聚合站 |
| 融资/商业数据 | 公司公告、投资方公告、SEC/招股书/年报、工商文件、交易所文件 | 数据库摘要、媒体报道、行业报告 |
| 政策/监管 | 政府官网、监管机构原文、法律法规原文、政策 PDF | 政策解读、媒体报道、咨询报告 |
| 用户口碑 | 不适用；用户口碑不纳入一手来源统计 | GitHub Issues、Reddit 原帖、Twitter/X 原帖、知乎原帖、应用商店原评论、媒体摘编、榜单摘要、第三方评论聚合 |
| 行业分析 | 原始统计口径、公司财报、协会/机构原始数据表、调研原文 | 权威媒体原创报道、券商/咨询/行业研究报告 |
| 学术/技术原理 | arXiv/论文官网、会议论文集、作者主页、代码仓库 | 论文解读、技术博客、媒体报道 |
| 人物观点 | 原始采访全文、播客/演讲实录、本人社交媒体原帖 | 媒体转述、剪辑片段、二次评论 |

### 判定规则

- 权威媒体原创报道通常是高质量二手来源，不要因为「原创报道」就标成一手。
- 媒体可标为一手的少数情况：它发布了自己的原始采访全文、现场调查、独家获取的原始文件或可复核数据；判定理由里必须写明。
- 行业报告、券商报告、数据库条目默认是二手来源；其中引用的原始公告、财报、政策原文才是一手来源。
- 用户口碑一律标 `二手`。具体原帖、评论、issue、讨论串可以作为高质量用户声音材料，但不纳入一手来源统计。
- 学术结论优先找论文原文；只找到论文解读时标二手，并在 `未搜到 / 存疑` 写明未读到原论文。
- 来源类型不确定时标 `存疑`，不要硬判一手或二手。

## 事实核查的反向实例（避免踩坑）

二次引用的信息要逆向核查一手来源。前一次实战验证过的几条陷阱：

- 媒体文章中「某机构鼓励/encouraged」类表述 → 一定要找到原始报道（不同媒体会互相抄错归属和日期）
- 「某公司衰退」「某产品失败」类判断 → 找近期财务数据、融资信息反向验证
- 名人讲话断章取义 → 找原始采访/播客的完整上下文（正面表态还是警示警告可能相反）

## 返回格式规范（必守）

每个 research worker 返回时**必须**按以下结构化 markdown 产出。主会话会直接落盘到配置的中间资料目录，默认路径为 `./output/intermediate/{topic-slug}-{YYYY-MM-DD}/0N-{维度}资料.md`，用于交叉验证和重新组织写作。中间资料不是公开交付物。

```markdown
# {维度名}资料 —— {研究对象}

> 搜集时间：YYYY-MM-DD HH:MM | 搜集者：Agent N | 子任务：{时间线 / 同期格局 / 补充}

## 关键信息清单

[按时间线 / 按竞品 / 按主题分组。每条一句话 + 时间 + 来源短标注 + 来源类型]

- 2023-07-17 DeepSeek 正式成立，由幻方量化孵化（来源：官方公告；类型：一手）
- ...

## 原始引用与来源

[每条关键事实配 2-3 句直接摘录 + URL + 访问时间。用于后续核查]

### {事实 1 的一句话标题}
- **摘录**：「{原文 2-3 句直接引用}」
- **来源**：{URL}
- **访问时间**：YYYY-MM-DD
- **来源类型**：一手 / 二手 / 存疑
- **判定理由**：{为什么这样判定，如「公司官网原始公告」「媒体转述官方发布」「聚合页未给原始出处」}
- **建议分级**：a / b / c / d（按 `source-citation-rules.md`：官方/学术一手为 a，政府机构、监管机构、公司年报和原始研究报告归入 a；权威媒体原创报道为 b；行业媒体/工具商博客为 c；软文、服务商自述或不可验证材料为 d）
- **是否需要逆查**：是 / 否

### {事实 2 ...}

## 未搜到 / 存疑

[诚实列出搜不到、只有单一来源、或跨来源矛盾的项。供关卡二审计]

- 2024-2025 融资细节：多家媒体未报，疑为非公开
- 创始人 X 的早期经历：仅一家博客提及，未找到其他佐证

## 最新数据校验

- **最新可确认资料**：{来源标题 + 发布日期 + URL + 类型}
- **近 12-24 个月是否覆盖**：是 / 否；缺口：{缺什么}
- **公司对象定量数据**：{收入/ARR、用户/客户、利润率、现金流、融资、估值、可比公司指标；非公司对象写「不适用」}
```

**四条硬规则**：

1. **不得合并小节**：四个小节必须齐全。「关键信息清单」≠「原始引用」—— 前者是速查版，后者带原文和 URL 可追溯
2. **不编造补位**：搜不到的内容一律进「未搜到 / 存疑」，不用「业内普遍认为」这类模糊兜底
3. **URL 必真**：不确定的 URL 宁可不写，也不要生成看起来合理但打不开的链接
4. **最新性必查**：返回结果没有「最新数据校验」时，主会话必须要求该 worker 补查后再进入写作
