## Use Case: <br>
Developers and architects use this skill to evaluate client-server, peer-to-peer, and offline-first architecture choices, then document trust boundaries, API contracts, version-skew strategies, and rollout concerns for web or mobile systems. <br>

### Deployment Geography for Use: <br>
Global <br>

## Known Risks and Mitigations: <br>
Risk: Broad architecture trigger wording may activate the skill in conversations where client-server or peer-to-peer guidance is only partly relevant. <br>
Mitigation: Review the generated guidance against the actual system context and use it only when the task involves client-server, peer-to-peer, trust-boundary, or offline-first design. <br>
Risk: Documentation-only architecture guidance can still introduce incorrect or misleading recommendations into ADRs, API contracts, or runbooks. <br>
Mitigation: Have a qualified reviewer check recommendations before using them for design decisions, implementation plans, or operational procedures. <br>
