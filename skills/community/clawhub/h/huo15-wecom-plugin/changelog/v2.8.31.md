# v2.8.31 (2026-05-05)

**P0 真根因：bot-ws sendMessage 一直用错 msgtype，30 个版本都治标**

## 背景

赵博 5/4-5/5 反复反馈 BOT 在企微 direct/group 长任务"思考中..."不出回复，或显示"已发送"但企微 client 不到。

之前的修复栈一直在治标——每次都改流相关：

- v2.8.17 placeholder send  
- v2.8.20 MEDIA: parser  
- v2.8.22 GUIDANCE 链接优先  
- v2.8.23 群聊 sendMediaMessage  
- v2.8.24 placeholder timeout 终结  
- v2.8.25 GUIDANCE MEDIA: 直发优先  
- v2.8.26 streamFinalized 跟踪 + sendMessage 40008 降级 text  
- v2.8.27/28 deliver final 早退 + finalizeStreamIfNeeded helper  
- v2.8.29 finalize 用 accumulatedText 不覆盖  
- v2.8.30 streamId 9min auto-rotate  

但 5/5 全天 gateway.log 仍出 13 条 errcode=40008 "invalid message type"，全部是 sendMessage(markdown_v2) 路径，且 v2.8.26 的 catch 降级 text 也撞 40008——但 `console.warn` 走 stderr 没 redirect 到 gateway.log，调试时看不见。

## 根因

`@wecom/aibot-node-sdk` v1.0.6 的 `node_modules/@wecom/aibot-node-sdk/dist/types/api.d.ts`：

```ts
export interface SendMarkdownMsgBody {
  msgtype: 'markdown';                        // ← 字面量类型，不是 'markdown_v2'
  markdown: { content: string };
}

export type SendMsgBody =
  | SendMarkdownMsgBody                       // markdown
  | SendTemplateCardMsgBody                   // template_card
  | SendMediaMsgBody;                          // file/image/voice/video
```

**`client.sendMessage` 这条 SDK 通道根本不接受 `'markdown_v2'`，也不接受 `'text'`**。两个都是 v2.8.26 fallback 时尝试的不存在 msgtype。

只有 Agent API HTTP 路径（`/cgi-bin/message/send`）的 server 端支持 `markdown_v2`——参见 `agent-api/core.ts:343`，那个 endpoint 跟 SDK WebSocket 完全两套协议。

之前用 `as unknown as Parameters<typeof client.sendMessage>[1]` 把类型穿透了，TypeScript 检查不出来；运行时撞企微 server 的 40008 又被 catch + console.warn（stderr）吞掉，gateway.log 看不见报错——等于完美隐身 30 个版本。

## 修复

3 个文件 6 处 sendMessage 全部 `'markdown_v2'` → `'markdown'`，删掉 v2.8.26 错误的 `text` fallback：

```diff
  // src/transport/bot-ws/sdk-adapter.ts (sendMarkdown handler + sendMedia caption)
- await client.sendMessage(chatId, {
-   msgtype: "markdown_v2",
-   markdown_v2: { content },
- } as unknown as Parameters<typeof client.sendMessage>[1]);
- // ... 40008 catch → text 降级（同样不支持） ...
+ await client.sendMessage(chatId, {
+   msgtype: "markdown",
+   markdown: { content },
+ });
```

```diff
  // src/transport/bot-ws/reply.ts (3 处 sendMessage 主动推送)
- await params.client.sendMessage(peerId, {
-   msgtype: "markdown_v2",
-   markdown_v2: { content: toWeComMarkdownV2(finalText) },
- } as unknown as Parameters<typeof params.client.sendMessage>[1]);
+ await params.client.sendMessage(peerId, {
+   msgtype: "markdown",
+   markdown: { content: toWeComMarkdownV2(finalText) },
+ });
```

```diff
  // src/transport/bot-ws/media.ts (share-fallback 链接 sendMessage)
- const messagePayload = {
-   msgtype: "markdown_v2",
-   markdown_v2: { content: fullText },
- } as unknown as Parameters<typeof params.wsClient.sendMessage>[1];
+ const messagePayload = {
+   msgtype: "markdown" as const,
+   markdown: { content: fullText },
+ };
```

去掉 `as unknown as Parameters<...>` 强转——恢复 TypeScript 类型检查能力，下次再写错 msgtype 编译期就报错，不让运行时 40008 又隐身。

## 影响范围

bot-ws 通道（默认通道）的所有主动推送：

1. **inbound 已接收但 reply 通道死了的兜底主动推送**（reply.ts:675/734） → 之前撞 40008 降级 text 又撞 40008 → 用户感知"已收到企微但没回复"
2. **Event 类型消息回复**（reply.ts:793） → 同上
3. **大文件 share-fallback 链接推送**（media.ts:340） → 链接 sendMessage 失败，落到 finalText 兜底文本回复
4. **subagent 主动 announce 推送**（sdk-adapter.ts sendMarkdown） → 完成通知撞 40008，最终走 announce-give-up

agent-api/core.ts 的 HTTPS 路径**不动**——那条 channel server 端确实支持 markdown_v2，是不同的 transport。bot-webhook/active-reply.ts 同理。

## 经验沉淀

**SDK 调研要先于行为修复**：v2.8.26 在没读 SDK 类型定义前，凭"server 返 40008 + 看着像 markdown_v2 strict 校验"猜根因，加了 text fallback。结果 fallback 同样不支持，问题没解决——只是把症状换了个地方藏。30 个版本都没去看 SDK type definition：

```bash
# 5 秒就能确认的事，拖了 30 个版本：
grep -A 3 "SendMsgBody\|msgtype:" node_modules/@wecom/aibot-node-sdk/dist/types/api.d.ts
```

**强类型穿透 = 自废类型武装**：`as unknown as Parameters<...>` 把 SDK 的字面量类型字段全 narrowed 掉了。SDK 用 `msgtype: 'markdown'` 作为 discriminated union 的 tag，TS 能检查；用 `as unknown` 强转一律绕过。今后插件代码**绝对不要用 `as unknown as Parameters<>` 强转 SDK 调用参数**——要么 SDK 类型不对（提 issue / 改 SDK），要么我们参数就是错的。

**stderr 不进 gateway.log = 调试盲区**：v2.8.26 的 catch 块里 `console.warn` 撞 40008 的降级路径，这条日志走 stderr，gateway 启动时 stderr 没被 redirect 到 `~/.openclaw/logs/gateway.log`。30 个版本里赵博发"BOT 没回复"，我看 gateway.log 看不到任何 sendMessage 错误——以为是 streamId / placeholder timeout 问题。后续教训：插件里所有 catch 块的 warn/error 必须 `console.error` 不是 `console.warn`，且关键错误必须打 `errcode/errmsg/raw` 三件套（v2.8.21 已经在 media.ts 落地，但 sdk-adapter / reply 没跟上）。
