# 项目准备与素材库建设详细指导

本文档提供 Phase 0 的详细指导，包括项目选择标准、项目准备方法、素材库建设示例。

---

## 一、项目选择详解

### 选择标准

一个合适的项目应同时满足以下四个条件：

#### 1. 熟悉度

优先选择自己亲身参与或非常熟悉的项目。只有深入了解项目的背景、需求、实施过程和效果，才能提供真实、详细的案例分析。如果你对项目的细节一知半解，写出来的内容就会空洞、缺乏说服力。

#### 2. 时效性

优先选择最近两三年内的项目，确保项目反映当前的技术趋势和行业发展状况。过旧的项目可能使用了已经淘汰的技术，不利于展示你的技术视野。

#### 3. 符合当前信息化建设潮流

项目最好能够体现以下热门技术方向：
- 数字化转型
- 智能化升级
- 人工智能应用
- 大数据分析
- 5G 技术应用
- 区块链

#### 4. 复杂度

**业务复杂度**是指软件系统所承载的业务目标、流程、规则以及用户需求的复杂性。高业务复杂度的项目通常具有以下特点：
- 涉及多部门、多角色、多业务流程的协作
- 涉及多个较深的专业领域知识
- 业务场景多样性
- 业务需求可能不断调整和细化，需求变更频繁

> **示例**：某省级医院的智慧医疗平台，需要整合门诊、住院、手术、检验、药房等多个科室的业务流程，支持医生、护士、患者多角色操作，并处理医保报销、药品管理、病历管理等复杂业务规则。

**技术复杂度**源于实现特定质量属性的技术手段本身的难度。

> **示例**：为提高系统的可扩展性和提升开发效率，采用了微服务架构，却额外引入了服务治理、分布式事务等问题，导致系统的复杂度增加。多个质量属性要求之间存在冲突，需要进行权衡和取舍，怎么权衡取舍依赖于架构师的经验和对业务的理解。

### 必须避免的项目类型

| 避免类型 | 原因 |
|----------|------|
| 小型系统 | 功能单一、规模小，难以体现复杂的业务逻辑和技术挑战，不利于展示解决方案的深度和广度 |
| 过时技术项目 | 采用淘汰或即将淘汰的技术，不利于展示技术视野和对行业发展趋势的把握 |
| 纯硬件项目 | 难以体现信息系统的整体架构和解决方案 |
| 纯技术项目 | 缺乏明确业务背景和应用场景，难以展示信息系统如何服务于实际业务需求 |

## 二、项目准备详解

### 1. 收集项目文档

#### 需求文档

设计解决方案的出发点和依据一定是项目的需求，否则解决方案就成了"空中楼阁"。通过需求文档明确项目的功能性需求和非功能性需求，清晰了解系统要解决的核心问题和用户期望，这是论文中阐述问题背景和解决方案价值的基础。

#### 设计文档

记录了系统的整体架构和详细设计，包括技术选型、架构设计、数据模型、接口规范等。通过设计文档了解系统是如何实现需求的，以及为什么要采用这种实现方式，这对于在论文中说清楚解决方案非常重要。

#### 测试报告

记录了系统的测试过程、测试结果和发现的问题和缺陷。测试报告中的性能数据和质量指标可以作为论文中方案评估部分的有力支撑。

#### 用户反馈

包含用户对系统的评价和建议。用户的正面评价可以作为方案价值的佐证，负面反馈可用于分析解决方案的不足和未来改进方向。

#### 故障报告

记录了系统运行过程中出现的各类故障、异常情况及其解决过程。通过故障报告可以发现潜在的改进点，让解决方案更加完善。

### 2. 梳理项目背景模板

以下模板可用于梳理项目背景，确保信息完整：

### 3. 项目难点与挑战梳理

从业务和技术两个维度梳理，以下问题可帮助引导思考：

#### 业务难点引导问题

- 业务流程中有哪些复杂的决策点或特殊规则？
- 涉及哪些部门的协作？沟通协调有哪些困难？
- 需求变更有多频繁？如何应对？
- 项目中有哪些业务创新点？在缺乏成熟经验时如何探索？

#### 技术难点引导问题

- 技术选型时考虑了哪些方案？为什么选择最终方案？
- 系统有哪些性能瓶颈？采取了什么优化措施？效果如何？
- 数据安全和系统安全面临什么挑战？如何防护？
- 与遗留系统或第三方系统集成时遇到了什么问题？如何解决？
- 项目中应用了哪些创新技术？解决了什么关键问题？

> 注意：这些难点与挑战不是"世界未解之谜"级别的，而是项目建设过程中遇到的实际问题和挑战。只要符合"现状与期望目标之间的矛盾"这个标准，都可以算作项目的难点与挑战。

## 三、素材库建设详解

### 核心思路

论文考察范围广，但可以**以有限的素材应对无限的考察范围**，达到以少胜多、以不变应万变的效果。

### 建设步骤

#### Step 1：选择 3-7 个核心业务场景

不管遇到什么主题，都写这些业务场景。选择业务场景时考虑：
- 覆盖面广：能适配多种论题方向
- 有深度：场景中包含足够的技术挑战和解决方案
- 真实熟悉：有第一手经验和详细数据

#### Step 2：按软件工程生命周期梳理

针对每个业务场景，梳理每个阶段的内容：

| 生命周期阶段 | 梳理内容 |
|-------------|---------|
| 需求分析 | 任务、面临的问题和挑战 |
| 系统设计 | 采用了哪些工具/方法/手段/步骤 |
| 编码实现 | 解决过程中遇到的问题和解决方案 |
| 测试 | 最终效果 |
| 部署运维 | 做得好的地方及待改进之处 |

#### Step 3：按 SCQA + 5W2H 组织素材

每个素材按照**金字塔原理和 SCQA 结构**组织，并运用 **5W2H** 补充必要的细节信息以增强真实性。

### 素材组织模板

```
## 素材：{素材名称}

### SCQA 结构

**S（情境）**：描述业务场景的背景和现状
**C（冲突）**：现状与期望目标之间的矛盾
**Q（问题）**：需要解决的核心问题是什么
**A（回答）**：我们采取了什么解决方案

### 5W2H 细节

- **Who**：涉及哪些角色/部门/团队
- **What**：具体做了什么
- **When**：什么时候做的（项目阶段）
- **Where**：在哪个模块/子系统中
- **Why**：为什么选择这个方案（权衡过程）
- **How**：具体怎么实施的（步骤、方法、工具）
- **How much**：效果如何（数据指标、用户反馈）

### 效果与反思

- 取得的效果：______________
- 做得好的地方：______________
- 待改进之处：______________
```

### 素材示例：适应性维护

```
## 素材：适应性维护——支付模块接口升级

### SCQA 结构

**S（情境）**：项目上线运营半年后，合作银行发布新的支付接口规范，
要求所有接入方在3个月内完成接口升级，否则将停止服务。

**C（冲突）**：现有支付模块与银行接口深度耦合，升级将影响多个业务流程，
且系统需要保持7×24小时运行，不能长时间停机。

**Q（问题）**：如何在不影响业务连续性的前提下，完成支付接口的适应性维护？

**A（回答）**：采用适配器模式设计兼容层，实现新旧接口的无缝切换，
通过灰度发布策略逐步切换流量，确保业务零中断。

### 5W2H 细节

- **Who**：我作为架构师主导技术方案，支付团队3人负责实施
- **What**：设计并实现支付接口适配层，完成灰度发布
- **When**：2024年6月~7月（项目运维期）
- **Where**：支付服务模块
- **Why**：银行强制要求升级接口，且需保障业务连续性
- **How**：
  1. 采用适配器模式，新建适配器实现新接口，保留旧接口
  2. 配置化切换开关，支持按比例分流
  3. 灰度发布：5% → 20% → 50% → 100%
  4. 全量切换后保留旧适配器2周作为回滚兜底
- **How much**：升级过程零故障，切换期间响应时间波动<5%，用户无感知

### 效果与反思

- 取得的效果：按期完成接口升级，业务零中断，获得银行技术对接好评
- 做得好的地方：适配器模式使变更隔离在支付模块内部，未扩散到业务层
- 待改进之处：灰度发布依赖手动配置，后续可引入自动化流量调度平台
```

### 素材复用策略

同一个素材可以从不同角度适配不同论题：

| 素材 | 论题角度 | 侧重点 |
|------|----------|--------|
| 支付模块接口升级 | 适应性维护 | 适配器模式+灰度发布 |
| 同一素材 | 软件架构风格 | 分层架构中适配层的解耦作用 |
| 同一素材 | 系统可靠性设计 | 灰度发布+回滚兜底保障连续性 |
| 同一素材 | 设计模式应用 | 适配器模式的具体实践 |

通过这种方式，3-7 个核心业务场景可以覆盖绝大多数论文主题。
