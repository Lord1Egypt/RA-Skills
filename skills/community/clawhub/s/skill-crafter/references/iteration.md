# 迭代改进指南

Skill 很少一次写对。本文覆盖"怎么判断 Skill 不好用"和"怎么改"。

---

## 症状 → 诊断 → 修复

### 症状 1：Skill 不触发

**表现：** 用户说了相关的话，但 Agent 没有选用这个 Skill。

**诊断：**
1. description 的触发短语是否覆盖了用户实际的说法？
2. description 是否写成了功能摘要，缺少 `Use when user asks to`？
3. 不适用边界是否过于宽泛，把本该匹配的场景排除了？

**修复：**
- 打开 SKILL.md，在 description 中补充用户实际使用的表达
- 确保触发短语覆盖中英文、正式/口语、简称/全称
- 缩小不适用边界，只排除真正不适用的场景

**示例：**

```yaml
# 修复前（不触发）
description: 知识库管理工具，支持文件导入和权限设置。

# 修复后（触发）
description: 管理知识库的完整操作工具。
  Use when user asks to 导入文件到知识库、创建知识库、设置知识库权限、
  移动知识库文件、重命名知识库.
  不适用于搜索知识库内容——那是 search 工具的工作.
```

---

### 症状 2：触发后出错

**表现：** Agent 选中了 Skill，但执行时报错或产出不正确。

**诊断：**
1. 操作指南中的命令/步骤是否能直接执行？
2. 依赖是否在补充说明中有兜底方案？
3. 是否引用了不存在的工具或路径？

**修复：**
- 在操作指南中用实际可执行的命令替换示例
- 在补充说明中补充依赖缺失的处理
- 移除对不存在工具的引用，或替换为实际可用的工具

---

### 症状 3：输出不稳定

**表现：** 同样的输入，不同次执行产出差异大。

**诊断：**
1. 操作指南是否给了足够的场景示例？
2. 输出格式是否有明确定义？
3. 是否缺少默认值和优先级说明？

**修复：**
- 增加操作指南中的场景示例，覆盖更多典型情况
- 在操作指南中明确输出格式和存放路径
- 在补充说明中明确默认值、优先级和冲突处理

---

### 症状 4：Agent 走岔路

**表现：** Agent 用了这个 Skill，但执行方向和预期不符。

**诊断：**
1. 概述的功能范围是否含糊（如用"等"概括）？
2. 操作指南是否缺少确认门？
3. 补充说明是否没覆盖已知的分支情况？

**修复：**
- 在概述中用穷举替代"等"，明确能力边界
- 在操作指南的长流程中加入用户确认点
- 在补充说明中补充分支判断：如果 X 则走路径 A，如果 Y 则走路径 B

---

## 迭代流程

```
1. 复现问题：确认症状属于哪一类
2. 读取 SKILL.md：定位问题所在的层级（头部/概述/操作指南/补充说明）
3. 用 file_edit 精确修改：只改有问题的部分，不动其他层
4. 用 checklist 做回归检查：确保修改没有破坏其他层级
5. 重新注册：ima_skill_create -d /sandbox/workspace/skills/{name}/
```

## 回归检查

修改后，用以下精简版 checklist 确认没有引入新问题：

| 检查项 | 标准 |
|--------|------|
| 触发词 | 修改后 description 的触发短语仍然覆盖用户主要说法 |
| 不适用边界 | 修改没有意外扩大或缩小适用范围 |
| 场景示例 | 修改后的操作指南仍有至少 2 个完整场景示例 |
| 补充说明 | 新增的兜底方案和原有方案不冲突 |
| TODO 残留 | 修改后无新增 `[TODO` 占位符 |
| 行数控制 | 修改后 SKILL.md ≤ 400 行 |

## 沉淀机制

每次修复一个问题，都把它沉淀到补充说明中：

- 如果是新的依赖问题 → 补充依赖兜底方案
- 如果是新的输入异常 → 补充异常处理方案
- 如果是新的边界情况 → 补充分支判断规则

这样 Skill 会随着使用越来越健壮。补充说明就是 Skill 的"踩坑记录"。
