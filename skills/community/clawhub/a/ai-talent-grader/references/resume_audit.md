# 简历漏洞穿透审计指南

## 审计目标

识别AI生成简历的致命弱点：AI善于构建宏大逻辑，但无法编造完全自洽的商业细节。本审计不是"判断简历写得好不好"，而是"判断简历中有多少真实可信的细节"。

---

## 审计指标1：高阶含金量审计

### 目标
横向对比商业常识，判断简历中声称的增幅/指标是否成立。

### 检查项
- [ ] 指标是否有行业对比基线？
  - 例："用户增长300%" → 同期行业平均多少？竞品多少？
- [ ] 增幅是否符合产品生命周期规律？
  - 例：成熟期产品声称"月活增长200%" → 高度可疑
- [ ] 数据是否过于"漂亮"？
  - 例：所有项目指标全部正向且幅度一致 → AI生成特征
- [ ] 是否有失败的案例或负向数据？
  - 真实工作必然有失败和妥协，全正向记录可疑

### 判定标准
- 有基线对比 + 有失败案例 → 低嫌疑
- 有基线对比但无失败案例 → 中嫌疑
- 无基线 + 全正向 → 高嫌疑

---

## 审计指标2：高势能低细节断层

### 目标
挑出"框架完美但缺乏独特长尾细节"的段落。AI擅长写出漂亮的结构，但无法编造只有亲身经历才有的"长尾细节"。

### 检查项
- [ ] 项目描述是否有"只有做过才知道"的细节？
  - 例："因XX原因，选择了YY方案而非ZZ方案，过程中遇到AA问题，通过BB方法解决"
- [ ] 是否有独特的数据异常或意外发现？
  - 例："原本以为是XX问题，深入后发现其实是YY，调整策略后..."
- [ ] 是否出现标准化模板语言？
  - 例："负责xxx项目，使用xxx技术，取得xxx成果" → 三段式模板
- [ ] 技术选型是否有代价说明？
  - 真实决策必有代价，"用了A方案"但没有"牺牲了什么" → 可疑

### 判定标准
- 有意外发现 + 有代价说明 → 低嫌疑
- 有代价说明但无意外发现 → 中嫌疑
- 三段式模板 + 无代价 → 高嫌疑

---

## 审计指标3：因果链断裂检测

### 目标
检查Action和Result是否漂亮，但Why和How语焉不详。

### 检查项
- [ ] 从"做了什么"到"取得什么成果"中间的逻辑是否自洽？
  - 例："优化了数据库查询 → 营收增长50%" → 因果链断裂
- [ ] 是否有中间步骤的说明？
  - 例："通过A方案解决B问题" → 为什么是A？尝试过其他方案吗？
- [ ] 结果是否超出合理预期？
  - 例：一个前端优化带来"业务转化率翻倍" → 因果失当
- [ ] 是否有他人协作的痕迹？
  - 单枪匹马完成需要多角色协作的任务 → 可疑

### 判定标准
- 因果链完整 + 有协作痕迹 → 低嫌疑
- 因果链部分缺失 → 中嫌疑
- 因果链断裂 + 结果夸大 → 高嫌疑

---

## 审计指标4：AI生成痕迹识别

### 目标
识别AI写作的典型特征：宏大词藻、缺乏个人特质、标准化模板语言。

### AI生成特征清单
- [ ] 过度使用"赋能""驱动""重构""闭环""抓手""底层逻辑"等词汇
- [ ] "通过AI技术大幅提升效率" → 无具体技术名称
- [ ] "主导了从0到1的..." → "0到1"是AI高频词组
- [ ] "实现了业务价值的最大化" → 空洞的万能结尾句
- [ ] 每段开头都是"负责...""主导...""参与..." → 高度模板化
- [ ] 缺乏人称代词，描述像产品说明书而非个人经历
- [ ] 技能列表"精通Python、Java、C++、Go、Rust..." → 全栈精通不现实

### 判定标准
- 无上述特征 → 低嫌疑
- 1-3个特征 → 中嫌疑
- 4个及以上 → 高嫌疑

---

## 审计指标5：逻辑一致性校验

### 目标
验证简历中时间线、职责范围、技术栈是否自洽。

### 检查项
- [ ] 时间线是否有重叠或空档？
  - 例：2019年6月毕业但2019年3月开始全职工作
- [ ] 职责范围与职级是否匹配？
  - 例：工作2年声称"主导公司技术架构重构"
- [ ] 技术栈时间线是否合理？
  - 例：2018年使用GPT-4 → GPT-4于2023年发布
- [ ] 项目规模与团队规模是否匹配？
  - 例：3人团队声称"支撑百万DAU"
- [ ] 多个项目之间的角色是否一致？
  - 例：每个项目都是"主导"，没有任何执行角色

### 判定标准
- 全部自洽 → 低嫌疑
- 1处不一致 → 中嫌疑
- 2处及以上不一致 → 高嫌疑

---

## 综合审计结论

| 高嫌疑指标数 | 结论 |
|-------------|------|
| 0-1 | 低嫌疑：简历真实度较高 |
| 2-3 | 中嫌疑：建议重点验证 |
| 4-5 | 高嫌疑：存在大量AI生成或夸大嫌疑 |

## 测谎面试题生成指南

### 生成原则
1. **追问细节而非结论**：问"你是怎么做的"而非"你做了什么"
2. **追问边界条件**：问"什么情况下会失败"而非"怎么成功的"
3. **要求还原决策过程**：问"为什么选A不选B"而非"A是什么"
4. **追问意外和失败**：问"遇到最大的坑是什么"而非"最大的成就"
5. **追问协作关系**：问"你和谁一起做的，怎么分工"而非"团队有几个人"

### 题量建议
- 低嫌疑：2-3题
- 中嫌疑：4-5题
- 高嫌疑：6-8题

### 题型模板

**类型A：细节追问**
```
在[项目X]中，你提到[某技术选型/某成果]。
请描述：
1. 当时有哪几个备选方案？为什么最终选择了这个？
2. 实现过程中最大的技术挑战是什么？具体怎么解决的？
3. 如果重新做一次，你会改变什么？
```

**类型B：边界条件**
```
关于[技术Y]，请描述：
1. 在什么场景下它表现最好？什么场景下会失效？
2. 你遇到过的最严重的线上问题是什么？排查过程是怎样的？
3. 如果数据量/用户量扩大10倍，你设计的技术方案需要做什么调整？
```

**类型C：决策还原**
```
关于[成果Z]，请描述：
1. 这个目标是怎么定出来的？为什么是这个数字？
2. 推进过程中，有没有调整过方向？为什么？
3. 团队内部有没有不同意见？你是怎么处理的？
```

**类型D：协作验证**
```
关于[项目W]，请描述：
1. 你在团队中的具体角色和日常做什么？
2. 和你协作最紧密的是谁？你们之间的工作界面是怎样的？
3. 项目中哪个部分是你独立完成的，哪个部分需要依赖他人？
```
