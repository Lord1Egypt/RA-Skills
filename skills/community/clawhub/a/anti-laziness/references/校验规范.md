# 校验工作防偷懒专项规范

> 校验是质量把关的最后一道防线，必须严谨、全面、不放过任何问题

---

## 一、校验态度（心态）

### 1.1 正确认识校验

- 校验不是找茬，是确保质量
- 校验是保护交付者，也是保护使用者
- 发现问题 = 负责任，不丢人

### 1.2 绝对禁止的心态

| 禁止 | 说明 |
|-----|------|
| 「大概看了一遍」 | 必须逐条核对，不能凭印象 |
| 「应该没问题」 | 每个检查点必须有明确结论 PASS/FAIL |
| 「差不多就行」 | 差多少都不行 |
| 「人家说改好了」 | 必须自己验证 |
| 「格式对就行」 | 内容也要检查 |

### 1.3 必须有的心态

- **怀疑一切**：除非自己验证过，否则不信
- **锱铢必较**：细节决定成败
- **独立判断**：不因为对方说"没问题"就放过
- **发现问题 = 好事**：早发现早修复

---

## 二、校验流程（步骤）

### 2.1 校验前准备

1. **明确校验标准**
   - 需求文档
   - 设计文档
   - 用户原始需求
   - 行业规范

2. **准备校验清单**
   - 逐项列出要检查的内容
   - 每项明确检查方法

3. **准备记录工具**
   - 表格记录问题
   - 截图/引用定位问题

### 2.2 校验执行

**核心原则：逐条核对，不能跳过**

#### 第一步：格式校验

| 检查项 | 方法 | 通过标准 |
|-------|------|---------|
| 文件格式 | 对比模板 | 完全一致 |
| 命名规范 | 检查文件/函数名 | 符合规范 |
| 目录结构 | 对比要求 | 一致 |
| 编码格式 | 检查文件编码 | UTF-8 |
| 语法正确 | 运行/编译 | 无错误 |

#### 第二步：内容校验

| 检查项 | 方法 | 通过标准 |
|-------|------|---------|
| 需求覆盖 | 逐条对照需求文档 | 全部覆盖 |
| 功能完整性 | 实际运行测试 | 功能正常 |
| 数据准确性 | 对比源数据/计算 | 数值正确 |
| 逻辑正确性 | 走查/测试 | 符合预期 |
| 边界条件 | 测试边界值 | 处理正确 |

#### 第三步：引用校验

| 检查项 | 方法 | 通过标准 |
|-------|------|---------|
| 外部链接 | 点击测试 | 全部有效 |
| 内部引用 | 检查跳转 | 全部正确 |
| 图片资源 | 检查路径/显示 | 全部正常 |
| 代码引用 | 对比源码 | 行号正确 |

#### 第四步：一致性校验

| 检查项 | 方法 | 通过标准 |
|-------|------|---------|
| 术语统一 | 全文搜索 | 术语一致 |
| 格式统一 | 对比段落 | 格式一致 |
| 数字一致 | 交叉核对 | 多处一致 |
| 前后一致 | 对比章节 | 逻辑自洽 |

### 2.3 校验后处理

1. **汇总问题**
   - 按严重程度分类
   - 按模块分组

2. **输出报告**
   - 问题描述
   - 位置定位
   - 修复建议

3. **跟踪修复**
   - 确认修复
   - 重新验证
   - 关闭问题

## 三、校验标准（要求）

### 3.1 问题分级

| 级别 | 符号 | 定义 | 处理方式 |
|-----|------|------|---------|
| 致命 | 🔴 | 导致功能失效/数据错误 | 必须修复 |
| 严重 | 🟠 | 功能有明显缺陷 | 必须修复 |
| 一般 | 🟡 | 不影响功能的小问题 | 建议修复 |
| 提示 | 🔵 | 优化建议 | 可以忽略 |

### 3.2 问题描述规范

每个问题必须有：

```
1. 问题标题（简洁）
2. 问题描述（详细）
3. 位置定位（文件/行号/截图）
4. 严重程度（🔴🟠🟡🔵）
5. 修复建议（具体可操作）
```

### 3.3 禁止的描述

| 禁止 | 原因 | 替代 |
|-----|------|------|
| 「有点问题」 | 不具体 | 「第5行缺少分号」 |
| 「不太对」 | 不明确 | 「计算结果少1」 |
| 「应该改」 | 不确定 | 「建议改为...」 |
| 「差不多」 | 差不多=差很多 | 给出具体数值 |

## 四、自检清单（强制）

### 开始校验前

- [ ] 明确校验标准和范围
- [ ] 准备好记录工具
- [ ] 了解原始需求

### 校验中

- [ ] 每项检查都有明确结论（PASS/FAIL）
- [ ] FAIL 项都有位置定位
- [ ] 记录了所有发现的问题

### 校验后

- [ ] 问题按严重程度分类
- [ ] 每个问题有修复建议
- [ ] 输出了校验报告
- [ ] 确认了修复时间

## 六、常见问题模式

### 6.1 容易漏掉的问题

1. **边界值**：0、1、-1、空值
2. **特殊字符**：Unicode、emoji、换行
3. **大小写**：Windows/Linux、路径
4. **隐形依赖**：未声明的库/环境

### 6.2 容易误判的问题

1. **经验假设**：以为没问题其实有问题
2. **功能正常**：但不符合需求
3. **表面正确**：但逻辑有漏洞

### 6.3 校验者常见错误

1. **光环效应**：因为某部分做得好就放松其他部分
2. **证实偏见**：只看到想看到的
3. **疲劳效应**：检查久了就松懈

---

## 七、校验工具推荐

### 7.1 自动化工具

- 语法检查：ESLint, Pylint, ShellCheck
- 格式检查：Prettier, Black
- 链接检查：linkinator, broken-link-checker
- 拼写检查：typos, spellcheck

### 7.2 人工检查技巧

1. **朗读法**：朗读出来更容易发现问题
2. **对比法**：与标准/模板对比
3. **逆向法**：从结果倒推逻辑
4. **角色扮演**：假设自己是使用者
