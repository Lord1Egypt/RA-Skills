# 编码标准 — Ponytail 决策阶梯 + 最佳实践

> 所有编码任务必须通过此标准检查。

## Ponytail 决策阶梯（编码前必过）

停止在第一个能 hold 住的层级：

1. **这需要存在吗？** → 推测性需求 = 跳过（YAGNI）
2. **代码库已有？** → 复用 helper/util/type/pattern
3. **标准库能做？** → 用它
4. **平台原生功能？** → CSS 优于 JS，数据库约束优于应用代码
5. **已安装依赖能解决？** → 用它，不新增依赖
6. **一行搞定？** → 一行
7. **最小可行实现** → 最后才写完整代码

## 不简化的边界

以下不应被精简：
- 输入验证（信任边界处）
- 防数据丢失的错误处理
- 安全措施
- 可访问性基础
- 用户明确要求的完整实现

## 输出规范

代码优先。然后最多三行短说明：跳过了什么，什么时候加。

格式：`[code] → skipped: [X], add when [Y].`

## 规则

- 无未请求的抽象：只有一个实现的 interface、只有一个产品的 factory、从不改变的 config
- 无样板代码，无"以后用"的脚手架
- 删除优于添加。无聊优于聪明
- 尽可能少的文件。最短的工作 diff 胜出
- 复杂请求？先交付简化版本，然后在同一回复中质疑它
- 两个标准库选项，同样大小？取边界情况正确的那个
- 用 `ponytail:` 注释标记有意简化：`// ponytail: global lock, per-account locks if throughput matters`

## Bug 修复

报告命名的是症状。修复前，grep 即将修改的函数的所有调用者。

懒修复 = 根因修复：共享函数中的一个 guard 比每个调用者中的 guard 更小 diff。

## 强度级别

| 级别 | 变化 |
|------|------|
| lite | 构建请求的内容，但一行说明更懒的替代方案 |
| full | 执行阶梯。标准库和原生优先。最短 diff，最短说明。默认。 |
| ultra | YAGNI 极端主义。删除先于添加。交付一行代码并在同一呼吸中质疑其余需求。 |

## 代码质量检查

### 命名
- 变量/函数名应表达意图
- 布尔值用 is/has/can/should 前缀
- 避免缩写（除非是行业通用）

### 函数
- 单一职责
- ≤50 行为佳
- 参数 ≤4 个
- 圈复杂度 ≤10

### 错误处理
- 不吞异常
- 提供有意义的错误消息
- 在适当的层级处理错误

### 测试
- 非平凡逻辑留一个可运行的检查
- 最小的 assert-based demo/self-check
- 无框架，无 fixtures（除非要求）
- 简单一行代码不需要测试
