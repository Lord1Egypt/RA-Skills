# 工程提示词流程智慧

> 从Eric的14个自定义工程专家提示词中提炼的**通用工程流程模式**——这些不是教科书知识，是你亲手设计的工程智能体工作方法论。

---

## 一、工程专家的通用工作流模式

### 模式A：需求驱动型（适用于设计/开发类问题）

从多个提示词中提取的通用流程：

```
第一步：需求澄清与边界锁定
  ├── 明确应用场景（客户是谁、用在什么环境）
  ├── 收集关键参数（负载/空间/性能/成本/年产量）
  └── 锁定约束条件（安全法规/功能安全等级/材料限制）
  
第二步：方案设计与可行性评估
  ├── 生成2-3个备选方案
  ├── 评估各方案的风险与收益
  └── 锁定最优技术路线
  
第三步：详细设计与仿真验证
  ├── CAE/仿真验证（静强度/模态/疲劳/碰撞）
  ├── 样件制作与测试
  └── 迭代优化
  
第四步：风险与应对预案
  ├── 识别关键风险点
  ├── 制定备选方案
  └── 设置决策关口（Go/No Go）
```

**共性原则**：所有提示词都要求"先问再答"（先收集足够信息再开始设计）、"分阶段输出"（不一次性给全部答案）、"验证闭环"（每个建议都需可验证）。

---

### 模式B：故障诊断型（适用于失效分析类问题）

```
第一步：症状采集
  ├── 主观感受（客户描述）
  ├── 客观数据（测试结果/频谱）
  └── 工况记录（何时/何地/什么条件下出现）

第二步：根因分析
  ├── 按概率排序常见根因
  ├── 交叉验证（多个证据指向同一原因）
  └── 排除法（排除低概率因素）

第三步：方案制定
  ├── 短期遏制方案（快速止血）
  └── 长期根治方案（根因消除）

第四步：验证闭环
  └── 方案实施后复测验证 → 归档
```

**共性原则**：所有诊断类提示词都强调"安全第一"、"数据说话"、"不推荐未经验证的高风险方案"。

## 二、救火与布局双模式（来自汽车非金属零部件专家）

### 核心洞察
工程问题分两类，需要不同的思维模式：

| 维度 | 救火模式（Diagnosis Mode） | 布局模式（Selection Mode） |
|:-----|:--------------------------|:--------------------------|
| 场景 | 产线危机/批量失效/紧急问题 | 新项目选材/技术路线规划/前瞻研发 |
| 目标 | 快速止血，恢复生产 | 最优方案，长期收益 |
| 节奏 | 48小时内出临时方案 | 2-4周完成评估 |
| 思维 | 最可能→验证→锁定 | 多方案对比→评估→选择 |
| 输出 | 根因+短期遏制+长期根治 | 技术路线图+风险提示+分阶段验证计划 |

**核心能力**：优秀的工程专家能在这两个模式间**无缝切换**，且能识别当前问题属于哪种模式。

## 三、系统级最优设计哲学（来自智能执行器总监）

### 核心原则
不从单一维度寻找最优解，而是从**系统层面**寻找整体最优：

```
单一维度最优 × → 系统级最优 √

案例：电机选型时
  ❹ 电机本身效率最高 ×
  ❹ 齿轮箱传动效率最优 ×
  ✅ 整个执行器的效率/成本/体积/寿命综合最优 √
```

### 实施方法
1. **跨学科协同**：机械+电子+软件+热管理+成本工程同步考虑
2. **多目标约束**：在所有约束条件的交集内寻找最优解
3. **系统级思维**：改变一个组件会影响整个系统，需跟踪副作用

## 四、敏捷开发模式（来自极速响应型骨架专家）

### 核心指标
- 24小时输出概念方案
- 72小时锁定技术路线
- 7天完成B样试制

### 并行工程流程
```
传统串行：需求→概念→设计→仿真→样件→测试→修改（数月）
并行模式：需求确定的当天 → 概念+设计+仿真+供应商同时启动（7天）
```

### 模块化快速架构
- 预先建立已验证的模块库
- 新项目直接从模块库中选择+组合
- 仅对差异化部分进行新开发

## 五、技术深度分级输出（通用）

所有提示词都遵循的技术输出规范：

| 技术深度 | 适合对象 | 输出内容 |
|:---------|:---------|:---------|
| L1：入门讲 | 初级工程师 | 原理+概念+标准方法 |
| L2：经验说 | 中级工程师 | 实践经验+注意事项+常见陷阱 |
| L3：专家谈 | 高级工程师 | 权衡分析+深层次耦合+非标方案 |
| L4：架构论 | 技术总监 | 系统架构+技术路线+平台策略 |

**原则**：知道什么时候给什么深度的答案，是高级工程师的标志。

---

## 六、安全优先与验证闭环（通用约束）

所有工程提示词的共同**强制约束**：
1. **安全第一**：任何建议不能牺牲安全
2. **正向指令**：以可行性为导向，不纠结"不可能"
3. **可验证性**：每个技术建议需具备可验证性
4. **时效性**：了解当前技术趋势（不推荐已被淘汰的方案）
5. **不编造参数**：不知道就说不知道，绝不编造

---

## 七、知懂分离原则（从Eric的工程判断延伸）

**知** = 知道知识树（书本能查到的）
**懂** = 走过探路路径（实践验证过的）

高级工程师与普通工程师的核心区别：
- 普通工程师：知道标准答案 → "知"
- 高级工程师：知道什么时候该用什么方案 → "懂"
- 总工：知道什么时候打破标准答案 → "悟"

这与碧树西风的"探树vs探路"模型在工程领域的直接映射。
