# Good vs. Bad Specification Examples

Use these examples to calibrate output quality. The "Bad" examples show
common mistakes; the "Good" examples show the corrected form.

---

## Example 1: Interface Definition

### ❌ Bad — Implementation Leak

```typescript
// BAD: Contains implementation logic, not just signature
async function createUser(data: CreateUserDTO): Promise<User> {
  const hashedPassword = await bcrypt.hash(data.password, 12);
  const user = await db.users.insert({
    email: data.email.toLowerCase().trim(),
    passwordHash: hashedPassword,
    createdAt: new Date()
  });
  await sendWelcomeEmail(user.email);
  return user;
}
```

**Problems:**
- Complete implementation — implementer has nothing to do
- Hardcoded salt rounds (12)
- Calls specific external services (sendWelcomeEmail)
- Business logic embedded in spec

### ✅ Good — Contract Only

```typescript
// GOOD: Signature, types, constraints, no implementation

/**
 * Create a new user account with validated credentials.
 *
 * @param data — user registration data, validated by [RegistrationValidator]
 * @returns created User with generated id
 *
 * @throws ValidationError if email format invalid or password < [MIN_PASSWORD_LENGTH]
 * @throws DuplicateKeyError if email already exists (idempotent with same idempotencyKey)
 *
 * Side effects:
 * - Persists user to [USER_TABLE]
 * - Triggers welcome email workflow (async, non-blocking)
 *
 * Security: password hashed with bcrypt, salt rounds configured via [BCRYPT_ROUNDS]
 * Thread: safe for concurrent calls; duplicate email handled by unique constraint
 */
async function createUser(data: CreateUserDTO): Promise<User>;
// implementation left to implementer

interface CreateUserDTO {
  email: string;        // validated email format, normalized to lowercase
  password: string;     // plaintext, hashed by implementation, min length in [AUTH_CONFIG]
  idempotencyKey?: string;  // optional, for retry safety, UUID v4
}

interface User {
  id: UUID;             // generated by system on creation
  email: string;        // normalized lowercase
  createdAt: DateTime;  // UTC, set by system
  // passwordHash and sensitive fields intentionally omitted from return type
}
```

## Example 2: Constraint Layer

### ❌ Bad — Vague and Incomplete

```markdown
## Constraints

- Should be fast
- Handle errors gracefully
- Use caching for performance
- Secure authentication
```

**Problems:**
- "fast" — no measurable target
- "gracefully" — undefined behavior
- "caching" — no strategy, invalidation, or fallback specified
- "secure" — no mechanism specified

### ✅ Good — Precise and Measurable

```markdown
## 4. Constraint Layer

### 4.1 Performance

| Metric | Target | Measurement |
|--------|--------|-------------|
| API response latency | < 100 ms at p95 | Endpoint load test with k6 |
| Database query | < 20 ms (simple), < 100 ms (complex) | Query analyzer |
| Cache hit latency | < 5 ms | Redis INFO latencystats |

### 4.2 Degradation Strategies

| Dependency | Healthy | Degraded | Failed |
|------------|---------|----------|--------|
| PostgreSQL | Normal query | Read from replica | Circuit breaker open, queue writes to DLQ |
| Redis | Cache read/write | Cache miss → DB read | All reads go to DB, no write caching |
| Email service | Async send | Queue for retry (max 3) | Log failure, do not block user creation |

### 4.3 Boundary Conditions

| Scenario | Input | Behavior |
|----------|-------|----------|
| Empty user list | `[]` | Return `[]` with HTTP 200 |
| Null DTO | `null` | `400 ValidationError` with field detail |
| Duplicate email | existing@example.com | `409 Conflict`, return existing user id |
| Concurrent create | same email, same ms | Unique DB constraint prevents duplicate; first wins |
| Password too short | "123" | `400 ValidationError`, min length defined in [AUTH_CONFIG] |
| Rate limit exceeded | > 100 req/min per IP | `429 Too Many Requests`, `Retry-After: 60` |

### 4.4 Threading

| Operation | Context | Notes |
|-----------|---------|-------|
| HTTP handler | Event loop | Do not block; use async I/O |
| DB query | Connection pool | Max 20 connections; queue if exhausted |
| Email send | Background worker | Fire-and-forget; failures logged, not retried inline |
| Password hash | Background thread | CPU-intensive; off main thread |

### 4.5 Security

| Concern | Requirement |
|---------|-------------|
| Password storage | bcrypt with [BCRYPT_ROUNDS], never store plaintext |
| Email validation | RFC 5322 regex + confirmation email before activation |
| Rate limiting | 100 req/min per IP, 10 req/min per email |
| Input sanitization | Strip HTML tags, validate max lengths per field |
| Audit logging | Log all create/delete operations with actor ID, retention 90 days |
```

## Example 3: Test Case Definition

### ❌ Bad — Vague Test Descriptions

```markdown
## Tests

- Test creating a user
- Test error handling
- Test performance
```

**Problems:**
- No input specified
- No expected output
- No error condition defined
- Not implementable by a developer

### ✅ Good — Input/Output Specified

```markdown
## 5. Verification Checklist

### Unit Tests

- [ ] `createUser` — valid input `{email: "new@example.com", password: "securePass123"}`
  → returns `User` with valid UUID `id`, normalized email `"new@example.com"`,
  `createdAt` within last 1000ms

- [ ] `createUser` — duplicate email `{email: "existing@example.com", password: "pass123"}`
  where email already exists
  → throws `DuplicateKeyError` with code `"EMAIL_EXISTS"`, status 409

- [ ] `createUser` — invalid email `{email: "not-an-email", password: "pass123"}`
  → throws `ValidationError` targeting field `"email"`, status 400

- [ ] `createUser` — short password `{email: "valid@example.com", password: "123"}`
  → throws `ValidationError` targeting field `"password"`,
  message includes minimum length requirement

- [ ] `createUser` — null input `null`
  → throws `ValidationError` with message "Request body required", status 400

- [ ] `createUser` — empty email `{email: "", password: "pass123"}`
  → throws `ValidationError` targeting field `"email"`, status 400

### Integration Tests

- [ ] End-to-end: POST /users with valid payload → 201 Created,
  Location header contains `/users/{id}`, welcome email queued

- [ ] Idempotency: POST /users twice with same `idempotencyKey`
  → second request returns 200 with same `id` as first (not 201)

- [ ] Degradation: create user when email service down → user still created (201),
  email failure logged, no error returned to client

### Performance Tests

- [ ] 100 concurrent createUser calls → p95 latency < 100ms, 0 errors
- [ ] Sustained 500 req/s for 5 minutes → memory usage stable (< 256MB),
  DB connections never exhausted
```

## Example 4: Module Boundaries

### ❌ Bad — Ambiguous Responsibilities

```markdown
## Modules

- src/
  - utils/       (helper stuff)
  - services/    (business logic)
  - models/      (data things)
```

**Problems:**
- "helper stuff" — meaningless
- "business logic" — which business? what logic?
- "data things" — too vague
- No dependency direction specified

### ✅ Good — Clear Single Responsibilities

```markdown
## 2. Module Structure

```
src/
├── auth/                    # Authentication & authorization
│   ├── handlers/            # HTTP route handlers (parse request, call service, format response)
│   ├── service.ts           # Business logic: validate, hash, persist, emit events
│   └── repository.ts        # Data access: SQL queries, transaction management
├── users/                   # User profile management (read-only for this spec)
│   ├── handlers/
│   ├── service.ts           # Profile retrieval, update, deletion
│   └── repository.ts        # User CRUD operations
├── email/                   # Email delivery abstraction
│   ├── client.ts            # Provider-agnostic interface (SendGrid, SES, etc.)
│   └── workers/             # Background job processors
└── shared/                  # Cross-cutting concerns
    ├── errors/              # Domain-specific error classes
    ├── middleware/          # Auth, rate limiting, request ID, logging
    └── types.ts             # Shared type definitions
```

| Module | Responsibility | Depends On |
|--------|---------------|------------|
| `auth` | Registration, login, credential validation | `shared`, `email` (event only) |
| `users` | Profile CRUD, search | `shared`, `auth` (for permission checks) |
| `email` | Async email delivery with retry | `shared` |
| `shared` | Error types, middleware, common utilities | — |

**Dependency Rule:** Domain modules (`auth`, `users`) may depend on `shared`
and `email`, but never on each other. Cross-domain operations use events or
the shared middleware layer.
```

## Example 5: Complete Mini-Spec (Good Example)

A compact but complete spec for a rate limiter module.

```markdown
# Rate Limiter Implementation Specification

## 1. Overview

### 1.1 Goal
Token bucket rate limiter supporting per-key limits with Redis backend,
usable across HTTP APIs, background jobs, and WebSocket connections.

### 1.2 Scope

**IN:**
- Token bucket algorithm with configurable capacity and refill rate
- Per-key limiting (IP, user ID, API key)
- Redis-based distributed storage
- Synchronous and async check interfaces

**OUT:**
- Sliding window algorithm (deferred to v2)
- Rate limit response headers (not in this spec)
- UI dashboard for limit configuration

### 1.3 Tech Stack

| Layer | Technology |
|-------|-----------|
| Language | TypeScript |
| Backend | Redis 7+ |
| Algorithm | Token bucket |

---

## 2. Module Structure

```
rate-limiter/
├── src/
│   ├── token-bucket.ts      # Core algorithm
│   ├── redis-store.ts       # Redis storage adapter
│   ├── limiter.ts           # Public API (simplified interface)
│   └── types.ts             # Shared type definitions
└── tests/
    ├── token-bucket.test.ts
    └── limiter.test.ts
```

| Module | Responsibility | Depends On |
|--------|---------------|------------|
| `token-bucket` | Algorithm logic: consume, refill, status | `types` |
| `redis-store` | Redis operations: get, set, Lua scripts | `types` |
| `limiter` | Public API, key resolution, error mapping | `token-bucket`, `redis-store`, `types` |
| `types` | Interfaces and type aliases | — |

---

## 3. Interface Definitions

### 3.1 Types

```typescript
// RateLimitConfig: parameters defining a single limit bucket
interface RateLimitConfig {
  key: string;              // unique identifier for this limit (e.g., "ip:1.2.3.4")
  capacity: number;         // maximum tokens, must be > 0
  refillRate: number;       // tokens added per second, must be > 0
  windowMs?: number;        // optional: time window for display purposes
}

// RateLimitStatus: result of a rate limit check
interface RateLimitStatus {
  allowed: boolean;         // true if request permitted
  remainingTokens: number;  // tokens left after this request (0 if denied)
  resetAt: DateTime;        // UTC time when bucket will be full
  retryAfterMs?: number;    // present and > 0 when allowed === false
}
```

### 3.2 Public API

```typescript
/**
 * Check if a request is allowed under the configured rate limit.
 * Consumes 1 token if allowed; does not modify state if denied.
 *
 * @param config — rate limit configuration for this key
 * @returns RateLimitStatus with allowance decision and metadata
 *
 * @throws RateLimitError if Redis unavailable (caller decides: fail open or fail closed)
 *
 * Performance: < 10 ms (Redis round-trip + Lua execution)
 * Thread: safe for concurrent calls; Redis Lua script is atomic
 */
async function checkLimit(config: RateLimitConfig): Promise<RateLimitStatus>;
// implementation left to implementer
```

---

## 4. Constraint Layer

### 4.1 Performance

| Metric | Target |
|--------|--------|
| Check latency | < 10 ms (Redis on same network) |
| Throughput | 10,000 checks/sec per Redis instance |
| Memory per key | ~100 bytes (bucket state in Redis hash) |

### 4.2 Degradation

| Dependency | Healthy | Degraded | Failed |
|------------|---------|----------|--------|
| Redis | Normal token bucket | N/A (stateless fallback impossible) | Fail open (allow request) or fail closed (deny) based on [FAILURE_MODE] config |

### 4.3 Boundary Conditions

| Scenario | Behavior |
|----------|----------|
| `capacity <= 0` | Throw `ValidationError` on config creation |
| `refillRate <= 0` | Throw `ValidationError` on config creation |
| Key does not exist in Redis | Treat as full bucket (capacity tokens available) |
| Concurrent checks on same key | Atomic via Redis Lua; no race conditions |
| Token fractional value | Floor to integer (0.9 tokens = 0 usable tokens) |
| Clock skew between instances | Tolerated: Redis server time is authoritative |

### 4.4 Threading

All operations are async/non-blocking. Redis Lua scripts execute atomically
on the Redis server; no client-side locking needed.

### 4.5 Security

- Keys must be sanitized: max length 256 chars, alphanumeric + `:.-_`
- Config values (capacity, refillRate) validated at creation time, never from user input
- Redis connection uses AUTH password, TLS in production

---

## 5. Verification Checklist

### Unit Tests

- [ ] `checkLimit` — new key, 1 capacity → first call allowed (remaining: 0),
  second call denied (retryAfterMs > 0)
- [ ] `checkLimit` — key with 10 capacity, 1/sec refill → 10 rapid calls all allowed,
  11th denied, wait 1s, 1 more allowed
- [ ] `checkLimit` — invalid config (capacity: 0) → throws `ValidationError`
- [ ] `checkLimit` — Redis unavailable, FAILURE_MODE=open → returns allowed=true
- [ ] `checkLimit` — Redis unavailable, FAILURE_MODE=closed → throws `RateLimitError`

### Integration Tests

- [ ] 100 concurrent `checkLimit` calls on same key with capacity=50 →
  exactly 50 allowed, 50 denied, no negative remaining tokens
- [ ] Refill accuracy: consume all tokens, wait `capacity/refillRate` seconds,
  check bucket is full again (within 1 token tolerance)

### Performance Tests

- [ ] 10,000 checks/sec sustained for 60s → p95 < 10ms, 0 errors,
  Redis CPU < 50%

---

## 6. Decisions

| Decision | Chosen | Rejected | Rationale |
|----------|--------|----------|-----------|
| Algorithm | Token bucket | Sliding window | Simpler implementation, good burst handling, Redis Lua friendly |
| Storage | Redis hash + Lua | In-memory + gossip | Required for distributed deployments |
| Failure mode | Configurable | Always fail closed | Some endpoints require availability over rate limiting |
```

## Common Patterns Summary

When in doubt, prefer:

| Aspect | Prefer | Avoid |
|--------|--------|-------|
| Specificity | `"< 200 ms at p95"` | `"fast"` |
| Test cases | Input + expected output + error condition | Vague descriptions |
| Interfaces | Signature + thrown exceptions + constraints | Complete implementation |
| Constraints | Tables with scenarios | Prose paragraphs |
| Responsibilities | One sentence per module | Vague grouping words |
| Decisions | Explicit rationale for rejection | Stating choice without context |
