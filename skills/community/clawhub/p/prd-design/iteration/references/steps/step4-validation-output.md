# Step 4: 合理性检查 + 原型自检 + 输出

## 目标
在输出前执行三层检查：信息完整性 → 系统一致性 → 原型交互自检，全部通过后再输出差量PRD和修改后原型。

---

## 4.1 合理性检查

### 检查项1：信息完整性

逐一检查每个调整模块的材料是否完整：

**新增界面检查**：
- [ ] 有需求描述（功能、用户、场景、目标）
- [ ] 有界面字段清单（字段名、类型、说明、必填）
- [ ] 有前后跳转说明或截图

**调整界面检查**：
- [ ] 有调整说明（调整内容、原因、期望效果）
- [ ] 有当前界面截图或详细描述
- [ ] 有变更位置标注

**删除界面检查**：
- [ ] 有流程影响分析（基于 Step 2 数据流简图）
- [ ] 有数据影响分析（废弃字段、下游影响）
- [ ] 有依赖检查分析（其他模块的依赖关系）

如果发现缺失，创建 `project/05-validation-issues.md` 列出缺失项，要求用户补充后重新检查。

---

### 检查项2：流程闭环性

**步骤1：重建调整后的主流程**

从 Step 1 的现有主流程出发，应用所有调整：
- 新增模块 → 在流程中插入新步骤，标注 `[NEW]`
- 调整模块 → 修改相应步骤描述，标注 `[MODIFIED]`
- 删除模块 → 从流程中移除步骤，标注删除原因

检查调整后流程：
- [ ] 有明确的开始点
- [ ] 有明确的结束点
- [ ] 每个步骤都有执行角色
- [ ] 步骤之间衔接合理（无跳跃、无断裂）

**步骤2：检查界面跳转闭环**

对所有新增和调整的界面，验证跳转路径是否闭环：
- [ ] 所有「进入路径」有来源（不是孤岛）
- [ ] 所有「离开路径」有去处（不是死胡同）
- [ ] 保存/提交后有成功反馈 + 合理跳转
- [ ] 取消/关闭后能回到前序页面

**步骤3：数据流完整性检查**（基于 Step 2 数据流简图）

- [ ] 新增模块的输出数据对象是否接入了正确的下游
- [ ] 删除模块后，其下游模块的输入是否有替代来源
- [ ] 调整模块的字段变更是否影响下游的数据读取

---

### 检查项3：命名一致性检查

**目的**：确保新增/调整界面中的字段名、模块名、状态值与现有系统保持一致，避免出现同一概念在原型和 PRD 中叫法不同的问题。

**检查基础**：Step 1 收集的主流程描述 + Step 2 推断/用户确认的模块列表和数据流简图。

**检查方法**：

1. **提取命名基准表**

从 Step 1 和 Step 2 已确认的信息中，整理命名基准：

```markdown
## 命名基准（来自 Step 1-2）

| 类型 | 已有命名 | 来源 |
|------|---------|------|
| 模块名 | 回访任务管理 | Step 2 确认 |
| 数据对象 | 回访记录 | Step 2 数据流 |
| 状态值 | 待执行 / 执行中 / 已完成 | Step 1 主流程描述 |
| 操作动词 | 分配 / 审核 / 提交 | Step 1 主流程描述 |
| 角色名称 | 客服专员 / 业务部门 | Step 1 收集 |
```

2. **逐项扫描本次新增/调整界面**

对 `project/02-add/` 和 `project/03-modify/` 下每个文件：

```
检查：字段名、按钮名、状态值、模块名、角色名
对比：命名基准表
结果：
  ✅ 一致：继续
  ⚠️ 差异：记录到命名差异表，等待决策
```

**命名差异表**（示例）：

| 位置 | 本次使用的命名 | 现有系统命名 | 建议 |
|------|-------------|------------|------|
| M6 规则引擎 - 字段 | 「执行状态」 | Step1中称「任务状态」 | 统一为「任务状态」 |
| M2 任务管理 - 按钮 | 「自动创建」 | 无现有命名（新增功能） | 保持「自动生成任务」，与用户需求描述一致 |

3. **差异决策规则**

- **有现有命名的**：以现有命名为准，修改本次设计中的差异命名
- **全新功能的**：以本次 Step 3 收集时用户的描述为准，在 PRD 中明确定义
- **AI 推断的命名**：在 PRD 附录「术语表」中注明，提示用户确认是否符合团队惯用叫法

**检查结果输出**：将命名差异表追加到 `project/05-validation-issues.md`（如有差异），或在 `project/05-validation-passed.md` 中注明「命名一致性：通过」。

---

### 检查项4：角色权限覆盖

- [ ] 每个新增界面都在需求描述中明确了操作角色
- [ ] 调整模块未导致某角色失去执行其职责的必要功能
- [ ] 删除模块后，相关角色的工作链路仍然完整
- [ ] 新增权限项已列入后续 PRD 第8节「角色权限变更」

---

### 检查未通过处理

如果任意检查项未通过：
1. 创建/更新 `project/05-validation-issues.md`，列出所有问题
2. 用 `present_files` 呈现，明确要求用户补充哪些信息
3. 用户补充后**重新执行全部4项检查**
4. 全部通过后创建 `project/05-validation-passed.md`，进入 4.2

## 4.2 原型生成 + 自检

### 原型生成规范

**原型技术栈**：React + Tailwind CSS，单文件 HTML

**原型结构要求**：
- **导航菜单**：左侧侧边栏，按「✨ 新增」「🔄 调整」「🗑️ 删除」「📋 流程对比」分组
- **新增界面**：完整交互原型，含所有字段、表单校验、提交/取消跳转
- **调整界面**：Before/After 对比视图（Tab 切换），After 视图中用高亮色标注变更区域
- **删除界面**：「已删除」通知页，包含删除原因和影响说明
- **流程对比**：并排展示调整前 vs 调整后主流程（每步标注 NEW / MODIFIED / DELETED）

生成原型后，**立即执行以下自检，不得跳过**。

---

### 🔍 原型交互自检（全量执行，共16项）

> **核心原则**：原型生成完毕后必须执行完整自检，未通过的项必须修复后才能交付，**禁止带已知缺陷交付**。

#### 第1-9项：基础自检

**第1项：所有页面可达性**
- [ ] 导航菜单中的每个入口都能正确渲染对应页面
- [ ] 无点击后白屏的菜单项

**第2项：Before/After Tab 切换**
- [ ] 每个调整模块的 Before Tab 和 After Tab 都能正常切换
- [ ] 切换后 active 状态正确转移
- [ ] 切换后内容区域实际渲染对应内容（不是空白）

**第3项：表单字段完整性**
- [ ] 新增界面的所有字段均已渲染（与 Step 3 字段清单逐一核对）
- [ ] 必填字段有星号标注

**第4项：表单提交校验**
- [ ] 必填字段为空时，点击提交有提示信息，不直接提交
- [ ] 提交成功有 toast 提示 + 合理跳转

**第5项：取消 / 关闭**
- [ ] 所有表单的「取消」按钮能关闭弹窗或返回前序页面
- [ ] 不存在「进得去出不来」的死胡同页面

**第6项：跳转路径验证**
- [ ] Step 3 中收集的所有前序→后续跳转路径在原型中均可模拟
- [ ] 跳转目标页面存在（不跳转到空白或报错）

**第7项：变更高亮标注**
- [ ] After 视图中新增的按钮/字段有明显高亮（颜色区分 + NEW 标签）
- [ ] 删除的元素在 Before 视图中有删除线或高亮标注

**第8项：数据流标注**
- [ ] 新增模块的输入来源在原型中有「来自XX」的说明
- [ ] 新增模块的输出去向在原型中有「提交后流入XX」的说明

**第9项：PRD 与原型一致性**
- [ ] PRD 第4节「新增功能」的字段表与原型表单字段一致
- [ ] PRD 第5节「调整功能」的 Before/After 描述与原型对比视图一致
- [ ] PRD 第3节「调整后流程」与原型「流程对比」页面一致

---

#### 第10-14项：交互自检强化（v3.6）

**第10项：建立可点击元素清单（逐页逐元素）**

扫描原型中**所有**以下类型的元素，逐一列出并标注 ✅/❌：
- 所有 `<button>` 元素
- 所有带 `onclick` 属性的元素
- 所有 Tab 切换控件
- 所有有联动逻辑的 `<select>` 下拉
- 所有表单提交路径

输出格式：
```
页面/模块：[模块名]
  元素1：[元素描述] → 绑定函数：[函数名] → ✅/❌
  元素2：[元素描述] → 绑定函数：[函数名] → ✅/❌
```

清单中出现任何 ❌ 必须修复后才能继续。

**第11项：Tab 专项检查**

对每一组 Tab 控件逐一验证：
```
□ 所有 Tab 按钮均有 onclick 绑定
□ 切换函数能正确找到目标内容区域（getElementById 不返回 null）
□ 切换后 active 状态正确转移
□ 默认激活的 Tab 对应内容可见，其余内容隐藏
□ 切换后内容区域实际渲染了对应内容，不是空白
```

**第12项：表单交互专项检查**

对每个表单/抽屉/弹窗：
```
□ 「提交」按钮有必填校验，为空时有提示不直接提交
□ 「取消」/「×」按钮能关闭弹窗/抽屉
□ 联动字段（选A显示X，选B显示Y）已正确实现
□ 提交成功有 toast + 弹窗关闭 + 列表刷新反馈
```

**第13项：禁止空函数和未定义函数**
```
□ 禁止 onclick="function(){}" 空函数
□ 禁止绑定在 JS 中未实现的函数名
□ 所有函数必须有实际逻辑和可见反馈
```

**第14项：修复后必须重新自检**

发现任何缺陷修复后，必须重新执行第10-13项完整自检，全部通过后才能继续。

---

#### 第15-16项：代码质量验证

**第15项：JS 语法强制验证**

提取 `<script>` 块内容，用 `node --check` 验证语法，returncode≠0 时**禁止保存输出，必须先修复**。

高危场景强制规则：在 JS 函数中动态构建含 `onclick` 属性的 HTML 字符串时，**禁止字符串拼接方式**，必须使用 DOM API：

```javascript
// ❌ 禁止
row.cells[4].innerHTML = '<button onclick="fn(\''+val+'\')">' + label + '</button>';

// ✅ 必须
const btn = document.createElement('button');
btn.textContent = label;
btn.onclick = function() { fn(val); };
row.cells[4].appendChild(btn);
```

检查项：
```
□ node --check 返回 returncode=0（无语法错误）
□ 全文无内嵌 onclick 的字符串拼接
□ 反引号数量为偶数（模板字符串全部闭合）
□ 花括号总数 opens == closes
□ 所有 getElementById 调用的 ID 在 DOM 中存在
```

**第16项：HTML 结构完整性验证**

用字符计数逐字符遍历验证 `<div>` 开闭平衡：

```python
depth = 0
for i in range(len(html)):
    if html[i:i+4] == '<div' and html[i+4] in ' \n\t>':
        depth += 1
    elif html[i:i+6] == '</div>':
        depth -= 1
# depth == 0 则平衡
```

检查项：
```
□ 受影响页面的 <div> 深度计算结果 == 0
□ 全局 body 的 <div> 开闭总数平衡
□ 被删除模块的特征关键词在全文中出现次数为 0
□ 相邻页面的内容未被污染
```
