# v2.8.30 (2026-05-06)

**修长任务 (>10min) streamId 撞企微过期窗口 errcode=846608**

## 背景

赵博 5/6 私聊 OpenClaw "几次显示'⏳ 任务仍在执行（已 2m+），完成后会主动推送结果'，没有后续推送"。

复盘 messageId=`a5106948af` 的完整时间线：

```
11:11:45.840 inbound "把这个 word 信息脱敏，发我一份"
11:11:46.357 dispatch-done duration=507ms ← !!!! 极速 done
11:11:48.854 placeholder 第一帧 send (streamId=stream_X)
11:11:49.069 ack received ✅
11:11:50.071 deliver-start kind=block messageId=d26faa3ff（另一条 reply）← 关键点
... a5106948af 的 placeholder keepalive 停了 ...
... 14 分钟过去 ...
11:26:24.878 deliver-start kind=block messageId=a5106948af textLen=537
11:26:25.483 errcode=846608 "stream message update expired (>10 minutes)" ❌
11:32:09.895 又一次 deliver block textLen=528 → 同样 846608 ❌
```

## 根因

**v2.8.26 的 streamFinalized 跟踪没覆盖这个场景**：

`reply.ts` 的 `notifyPeerActive()` 设计：每条 reply 收到 deliver / fail 时，会**停掉同 peerId 但不同 reqId 的所有 keepalive**（防止旧消息的 placeholder 跟新消息的 reply 撞）。

但副作用：

1. a5106948af inbound → 创建 reply handle (reqId=A)，placeholder keepalive 启动
2. d26faa3ff（另一条 reply，reqId=B，同 peerId=ZhaoBo）的 deliver block 来 → 调 `notifyPeerActive()` → **停掉 a5106948af 的 placeholder keepalive 和 timeout**
3. a5106948af 的 placeholder timeout **永远不触发** → `streamFinalized` 永远 false
4. 14 分钟后 a5106948af 的 deliver block 来 → `resolveStreamId()` 返回旧 streamId（streamFinalized=false 跳过重启） → `replyStream(...)` → 撞 errcode 846608 "stream message update expired (>10 minutes)"

**企微 stream 协议规则**：同 streamId 距首次创建 10 分钟后，partial 更新一律拒（errcode 846608）。

## 修复

`resolveStreamId()` 加 `streamStartedAt` 时间戳跟踪 + 9 分钟 auto-rotate：

```diff
  let streamId: string | undefined;
  let streamFinalized = false;
+ let streamStartedAt = 0;
+ const STREAM_MAX_AGE_MS = 9 * 60 * 1000; // 9 分钟，企微 10min 过期前主动重启

  const resolveStreamId = () => {
+   const now = Date.now();
    if (streamFinalized) {
      streamId = generateReqId("stream-restart");
+     streamStartedAt = now;
      streamFinalized = false;
      console.log(`[wecom-ws] streamId 已终结 → 重启新 streamId=${streamId}`);
+   } else if (streamId && now - streamStartedAt > STREAM_MAX_AGE_MS) {
+     // v2.8.30 ⭐ 接近 10 分钟过期窗口 → 主动 rotate
+     const oldStreamId = streamId;
+     const ageMs = now - streamStartedAt;
+     streamId = generateReqId("stream-rotate");
+     streamStartedAt = now;
+     console.log(
+       `[wecom-ws] streamId 已用 ${Math.round(ageMs/1000)}s（>9min）接近企微 10min 过期 → 主动 rotate ${oldStreamId} → ${streamId}`,
+     );
    }
-   streamId ||= generateReqId("stream");
+   if (!streamId) {
+     streamId = generateReqId("stream");
+     streamStartedAt = now;
+   }
    return streamId;
  };
```

## 现在长任务路径

```
11:11:45 inbound → streamId=A, streamStartedAt=11:11:45
11:11:50 notifyPeerActive 停掉 placeholder timeout（外因）
... 14 分钟后 ...
11:26:24 deliver block → resolveStreamId() 检测 elapsed=14min>9min → rotate
   → streamId=B (stream-rotate)，streamStartedAt=11:26:24
   → replyStream(B, blockText, false) → 企微 client 显示**新一条消息**含完整内容 ✅
```

老消息（streamId=A 的"正在思考..."）保留 last=false 状态，企微 client 长 idle 后自动解锁。新消息（streamId=B 的真实回复）正常显示。

## 经验沉淀

**stream 协议时间窗口必须主动管理**：企微 stream 10 min 过期是硬限制，wecom 这层必须**主动 rotate 不等过期**。

之前的修复栈一直关注"流如何终结"（v2.8.24 timeout finalize、v2.8.27/28 deliver final 早退、v2.8.29 用 accumulatedText 不覆盖），都是处理"流要不要 last=true"。但**长任务 + notifyPeerActive 副作用**让 stream 既不 last=true 终结也不 rotate，硬撞过期。

修法是在**底层 streamId resolver** 加时间窗口检测，所有调用方（placeholder send / partial / final / finalize）都自动用新 streamId。

**今后任何带过期窗口的协议**（SSE / WebSocket / OAuth token / API session）都要在最底层 resolver 加 elapsed check + auto-rotate。
