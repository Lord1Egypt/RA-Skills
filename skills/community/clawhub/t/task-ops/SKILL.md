# Task Ops — 四模块价值链任务操作系统

## 核心理念

传统工作流中，很多环节不是因为事情本身需要，而是因为人的局限需要。Task Ops 基于"区分事情本身的需要和人的局限需要"这一核心理念，提供四个正交模块，形成完整的价值链。

## 四模块概览

| 模块 | 核心问题 | 价值类型 | 使用频率 | 定位 |
|------|---------|---------|---------|------|
| Domain Elimination Assessor | 这个领域需要独立存在吗？ | 评估价值——判断存在必要性 | 低频、战略级 | 评估层 |
| Workflow Refactor | 流程怎么简化？ | 转化价值——从旧到新 | 低频、脉冲式 | 转化层 |
| Domain Payload Generator | 知识怎么结构化？ | 创建价值——从无到有 | 中频、按需 | 创建层 |
| Universal Task OS | 任务怎么完成？ | 运行时价值——执行任务 | 高频、持续 | 执行层 |

## 四模块关系

```
传统领域/工作流
      │
      ▼
┌─────────────────────────┐
│ Domain Elimination       │ 评估层：这个领域需要存在吗？
│ Assessor (领域消除评估)    │ → 消除 / 保留 / 重构
└─────────────────────────┘
      │ (保留/重构的领域)
      ▼
┌─────────────────────────┐
│ Workflow Refactor        │ 转化层：流程结构怎么优化？
│ (工作流重构)               │ → 拆解→消除人的局限补偿层→重整为IPO基元链
└─────────────────────────┘
      │
      ▼
┌─────────────────────────┐
│ Domain Payload Generator │ 创建层：领域知识怎么结构化？
│ (领域负载物生成)           │ → 生成catalog+requirements+exemplars
└─────────────────────────┘
      │
      ▼
┌─────────────────────────┐
│ Universal Task OS        │ 执行层：任务怎么完成？
│ (通用三轴任务操作系统)      │ → 执行轴+内容轴+创新轴
└─────────────────────────┘
```

**价值链顺序**：评估→转化→创建→执行，不可跳步（除非明确不需要）。四个模块是正交互补关系，不是替代关系。

## 执行编排

收到任务后，按三阶段编排：

### 第一阶段：模块选择判定

| 判定 | 条件 | 动作 |
|------|------|------|
| 领域消除评估 | 领域边界模糊、功能离散、存在历史遗留问题 | 读取 `references/domain-elimination-assessor.md`，按 D0-01→D0-05 执行 |
| 工作流重构 | 用户要求重构/简化工作流；或满足重构判断标准 | 读取 `references/workflow-refactor.md`，按 R0-01→R0-06 执行 |
| 领域负载物生成 | 用户请求创建领域知识参考库/领域负载物技能 | 读取 `references/domain-payload-generator.md` 及4个子参考文件，按 Step1→Step4 执行 |
| 常规任务 | 不满足上述条件 | 直接进入第二阶段 |

### 第二阶段：三轴执行判定

任务通过三轴正交框架处理，执行轴必选，内容轴和创新轴按需激活：

**执行轴（How / 必选）**：管怎么做
- 6大元操作 + 管线编排 + 能力单元Schema + 通用管线模式（P1-P7） + 自适应身份叠加 + IPO基元
- 读取 `references/capability-pipeline-os.md` 获取完整定义
- **复杂度判定**（执行轴内部）：
  - **简单**（可直接完成）→ 单一IPO基元
  - **中等**（3-7步）→ 单层管线编排
  - **复杂**（8+步或多层嵌套）→ 多层管线+IPO递归

**内容轴（What / 按需）**：管造什么
- 清单法（组件组合） + 样本法（模仿产出）
- 方法选择机制：由用户决定，AI不得自动选择
- 结构化成品（文档、方案、报告等）时激活 → 读取 `references/compose-methods.md`

**创新轴（Why Different / 按需）**：管怎么不一样
- 4种模式处理能力 + 10种创新元框架 + 通用四维评估 + 迁移类型
- 需要突破默认路径时激活 → 读取 `references/innovation-os.md`

**判定组合**：

| 执行轴复杂度 | 创新轴 | 内容轴 | 加载文件 |
|-------------|--------|--------|---------|
| 简单 | 不激活 | 不激活 | 无 |
| 中等 | 不激活 | 激活 | `compose-methods.md` |
| 中等 | 激活 | 不激活 | `innovation-os.md` |
| 复杂 | 激活 | 激活 | `compose-methods.md` + `innovation-os.md` |

### 第三阶段：产出沉淀判定

| 判定 | 条件 | 动作 |
|------|------|------|
| 领域消除评估报告 | 前置为领域消除评估 | 产出评估报告 |
| 工作流重构产物 | 前置为工作流重构 | 产出IPO基元链+验证清单 |
| 生成新领域技能 | 前置为领域负载物生成 | 按生成工作流产出技能目录 |
| 沉淀为技能 | 8+工具调用且有复用价值 | 保存为新技能 |
| 常规交付 | 不满足上述条件 | 直接交付结果 |

## 使用规则

1. **模块选择优先**：先确定需要哪些模块，再执行
2. **按需激活**：只激活需要的模块和轴，避免过度复杂化
3. **加载指令明确**：激活某轴/模块时，必须先读取对应参考文件，再按参考文件中的工作流执行
4. **内容轴方法选择**：清单法与样本法由用户决定，AI不得自动选择
5. **创新轴模式选择**：AI自选模式/框架/源领域时，必须向用户展示所选内容及理由
6. **用户主权**：AI产出是起点不是终稿，用户可要求修改任何环节

## 事实纪律

1. AI能力描述必须基于实际能力，不得夸大
2. 评估结果标注为"参考范围"，实际效果取决于具体场景
3. 法规、标准、合同等外部约束必须明确标注
4. 创新方案基于实际可行性，不确定的信息标注"待验证"
5. 任务执行基于实际数据，不可凭感觉
