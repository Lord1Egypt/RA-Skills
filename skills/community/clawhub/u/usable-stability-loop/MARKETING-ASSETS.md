Help agents ship usable, stable solutions faster for non-technical users, then improve them through real-world usage instead of over-engineering upfront.

Designed for agents serving non-technical but goal-clear users: prioritize usability and stability first, then iterate the rest in production use.

## 2) 3 个高转化 use cases

### Use Case 1｜给非技术用户做内部工具 MVP
A user wants a tool, but not a six-week build cycle.
Use this skill to force the agent to ship the smallest usable version first, keep it stable, and avoid wasting time on fancy UI or architecture before the workflow is proven.

**Why it converts:**
This is a universal pain point. Many builders know the trap of overbuilding internal tools before the real usage pattern exists.

### Use Case 2｜给知识库 / 工作流项目防止“产品化过度”
When an agent is building a database, knowledge workflow, or operational assistant, this skill acts like a scope governor: keep the data and workflow usable first, and postpone dashboards, abstractions, and heavy interfaces until real usage proves the need.

**Why it converts:**
This speaks directly to people building agent systems, research pipelines, and personal operating systems.

### Use Case 3｜给多 agent 系统加一层执行约束
Use this skill as a portfolio-level execution rule for a team of agents: usability first, stability first, stop polishing once the system is genuinely usable, and only improve through recurring friction in production.

**Why it converts:**
This elevates the skill from a single-task helper into an operating philosophy for agent builders.

## 3) 对外宣传文案

### 版本 A｜适合发社区 / 群聊
我最近发了一个 ClawHub skill：
**Usable Stability Loop · The 99/80 Loop**

它解决的不是“怎么做得更炫”，而是另一个更常见的问题：
**很多 agent 在给非技术用户交付时，会过度设计、过度产品化，最后迟迟交不出一个真正可用的版本。**

这个 skill 的原则很简单：
- 可用性优先
- 稳定性优先
- 到 80 分就收手
- 剩下 20 分在真实使用中持续进化

更适合什么场景？
- 给非程序员做工具/工作流/MVP
- 做知识库、数据库、内部系统
- 给 agent 系统加一层“别过度折腾”的执行约束

如果你也在做 agent 交付，我觉得这类 skill 可能比“更复杂的框架”更实用。

### 版本 B｜适合发朋友圈 / 轻分享
做 agent 这段时间，我越来越觉得：
