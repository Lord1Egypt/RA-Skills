# 缺陷写作指南

用于编写正式 UAT 问题。

## 必填字段

- 问题编号
- 模块或业务流程
- 页面/功能点
- 页面 URL
- 角色/会话类型
- 业务场景
- 问题描述
- 问题类型
- 严重级别
- 优先级
- 环境/版本
- 测试数据类型
- 复现步骤
- 预期结果
- 实际结果
- 证据编号
- 验收影响
- 建议修复或确认负责人
- 状态

## 规则

- 复现步骤要写成人能理解的页面/业务步骤。
- `页面 URL` 与 `接口 URL` 必须分开。
- 不包含凭证、Token、Cookie、Authorization 头、个人隐私数据或涉密载荷。
- 如果预期行为不清楚，标记为 `ToConfirm`，不要擅自定义为缺陷。
- 如果问题阻断验收，说明它阻断了哪条准则或哪个用例。

## 严重级别参考

- `Blocker`：核心流程、安全边界、数据边界失败，或无法继续验收。
- `Critical`：高影响失败，规避方案有限。
- `Major`：重要功能错误，但可能有规避方式。
- `Minor`：低影响的易用性、文案或显示问题。
- `Trivial`：轻微视觉或文档差异。
