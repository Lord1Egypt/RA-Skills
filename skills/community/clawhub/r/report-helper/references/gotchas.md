# 踩坑记录

保持本文件短而实用。只有当新问题会改变后续行为时，才补充新条目。

---

## 1. 报告结构

### 1.1 分析工具不能变成读者可见标题

**现象**：报告把内部分析标签直接写成章节标题，读起来像方法练习，而不是完成稿。

**处理**：从 `report-template.md` 选择面向读者的正文框架。写作判断工具只用于生成内容和结论，不作为正文目录。

### 1.2 发展研究报告需要正式结构

**现象**：用户要求产业、政策、趋势或发展研究报告，但正文写成了产品故事型文章。

**处理**：除非用户明确要求其他结构，否则使用 `report-template.md` 的 A. 正式研究报告结构。

### 1.3 公司研究不能只写叙事

**现象**：公司报告写得像故事复盘，讲清了发展历程和战略转向，但缺少成长性、盈利质量、融资、现金流、估值或可比公司等定量分析。

**处理**：公司对象必须读取 `adaptations-by-type.md` 的公司定量要求，并在正文加入成长性与估值分析。没有公开收入、利润或估值数据时，明确写缺口和替代观察指标。

## 2. 证据与来源

### 2.1 不把媒体摘要当一手来源

**现象**：媒体文章引用了公司公告或政策文件，但报告把媒体摘要当成原始来源。

**处理**：重要事实必须反查原始公告、财报、论文、政策原文或采访全文。

### 2.2 用户反馈属于二手材料

**现象**：把用户评论、issue、社区讨论当成一手来源统计。

**处理**：用户反馈可以作为有价值的定性材料，但来源类型统计中归为二手材料。

### 2.3 缺失信息必须保持缺失

**现象**：用「业内人士认为」「普遍认为」之类模糊说法填补资料缺口。

**处理**：标注「未搜到 / 存疑」，并解释这个缺口为什么重要。

### 2.4 来源编号不能只在末尾补参考文献

**现象**：正文没有来源上标，只在末尾堆一组链接，读者无法判断哪条事实来自哪里。

**处理**：先建来源台账，再写正文。每条关键事实判断后用 `<sup>a1</sup>` 这类上标，末尾「信息来源与分级」按 a/b/c/d 分组；没有 c 和 d 就不写对应小节，并保证上标和来源条目一一对应。

### 2.5 写作前必须查最新数据

**现象**：资料搜集停留在旧报道或旧数据，Agent 没有查找最新财报、公告、监管文件、原始研究或权威报道，就开始写正文。

**处理**：关卡二前必须做最新数据校验。若近 12-24 个月可能有新数据，先补搜并更新判断；没有完成最新数据校验，不得进入写作。

## 3. 未来推演

### 3.1 不编造官方目标

**现象**：把分析师预测、地方政策措辞和媒体摘要揉成一个看似精确的国家或公司目标。

**处理**：区分官方目标、分析师预测和情景假设。没有找到官方目标时，明确写没有找到。

---

## 4. 交付

### 4.1 路径必须可配置

**现象**：共享 skill 里写死本地路径。

**处理**：使用 `./output` 默认路径、环境变量、`config.local.json` 或用户传参。不要提交本地配置文件。

### 4.2 私有工作流名称不要写进共享代码

**现象**：共享脚本里出现本地路径、私有日志标题、账号标识或个人工作流名称。

**处理**：移动到已忽略的 `config.local.json` 或环境变量。报告署名由用户首次安装时填写；工具作者联系方式只放在最终 PDF 末尾的固定工具签名里。

### 4.3 被指出的新坑必须及时入库

**现象**：用户指出流程问题后，报告继续推进，但可复用的踩坑记录没有补进 `gotchas.md`，下次仍可能重复。

**处理**：用户指出事实核查、最新数据、结构、交付或 skill 维护问题时，先判断是否可复用；可复用就立刻补一条短 gotcha，再继续后续任务。
