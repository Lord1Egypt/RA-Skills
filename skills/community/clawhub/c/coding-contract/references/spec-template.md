with project-specific content. Annotated comments (like this one) should NOT
appear in the final output — they are instructional for the spec author only.

## Usage Notes for Spec Authors

1. **Replace all `[bracketed placeholders]`** with project-specific content.
   Do not leave placeholders in the final spec.

2. **Section 4 is the most important section.** Spend the most time here.
   The implementer can figure out code; they cannot figure out constraints
   that live only in your head.

3. **Pseudocode syntax:** Use the target language's type notation but replace
   function bodies with `// implementation left to implementer`.

4. **Tables over prose:** Prefer tables for boundary conditions, degradation
   strategies, and test cases. They are scannable and unambiguous.

5. **One spec per bounded context.** If a feature spans multiple independent
   subsystems, split into separate spec files — one per subsystem.
