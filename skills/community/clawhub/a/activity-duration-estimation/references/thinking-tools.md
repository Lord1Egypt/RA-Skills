# 思维工具与方法论（Thinking Tools）

> 本文件收录项目管理/文档撰写中常用的思维工具和方法论框架。
> 供 LLM 在生成具体章节内容时参考运用，使输出结果更加结构化、专业和有说服力。
>
> 每个工具标注了：适用章节、用法说明、示例输出。

---

## 一、SWOT分析

> **适用章节**: 项目背景、风险评估、相关方分析

### 说明

从四个维度分析项目所处环境：
- **S**trength（优势）— 内部有利因素
- **W**eakness（劣势）— 内部不利因素
- **O**pportunity（机会）— 外部有利因素
- **T**hreat（威胁）— 外部不利因素

### 用法

撰写"项目背景"时，按 SWOT 四象限组织内容，形成内外部分析结论。

### 示例输出

```
## 项目背景（SWOT分析）

**优势（S）**
- 团队具备3年同类系统开发经验
- 已积累标准化的前端组件库

**劣势（W）**
- 缺少大数据量高并发场景经验
- 运维人员尚未到岗

**机会（O）**
- 公司数字化转型战略支持
- 市场上无同功能竞品

**威胁（T）**
- 核心依赖的第三方API即将升级
- 项目周期跨越年底封网期
```

## 二、SMART原则

> **适用章节**: 项目目标

### 说明

确保目标清晰可衡量：
- **S**pecific（具体的）— 明确做什么
- **M**easurable（可衡量的）— 有量化指标
- **A**chievable（可实现的）— 目标切合实际
- **R**elevant（相关的）— 与业务目标一致
- **T**ime-bound（有时限的）— 有截止日期

### 用法

每个项目目标按 SMART 五项逐一检验，在文档中结构化呈现。

### 示例输出

```
## 项目目标（SMART）

| 维度 | 目标描述 |
|------|---------|
| Specific | 开发电商后台管理系统，覆盖商品/订单/用户/报表四大模块 |
| Measurable | 订单处理效率提升50%，页面加载<2s，日活>1000 |
| Achievable | 基于现有中台架构扩展，团队6人，4个月工期 |
| Relevant | 支撑公司2026年电商业务线上化战略 |
| Time-bound | 2026年9月30日前交付V1.0 |
```

## 三、5W2H分析法

> **适用章节**: 项目背景、范围定义、需求澄清

### 说明

七个维度系统化思考：
- **W**hat — 要做什么
- **W**hy — 为什么要做
- **W**ho — 谁来做/为谁做
- **W**here — 在哪里做/用到哪
- **W**hen — 什么时间做/完成
- **H**ow — 怎么做
- **H**ow much — 多少成本/资源

### 用法

在"项目背景"或"范围定义"章节，用5W2H检查是否覆盖了所有关键问题。遗漏任何一个W都可能成为后续风险。

### 示例输出

```
## 项目背景（5W2H）

What: 建设电商后台管理系统
Why: 现有系统为Excel手工管理，错误率高、效率低
Who: 运营团队是最终用户，技术团队负责开发
Where: 部署在公司云服务器，运营团队远程使用
When: 立项2026年6月，目标9月上线
How: 采用前后端分离架构，React+Spring Boot
How much: 预计投入6人×4个月，预算约80万
```

## 四、PDCA循环

> **适用章节**: 质量管理、过程改进、经验教训

### 说明

持续改进的四步闭环：
- **P**lan（计划）— 制定目标和方案
- **D**o（执行）— 实施计划
- **C**heck（检查）— 评估结果
- **A**ct（处理）— 标准化或改进

### 用法

撰写"质量评估"或"经验教训"章节时，按 PDCA 四阶段回顾项目执行过程。

### 示例输出

```
## 质量评估（PDCA回顾）

Plan: 计划每周代码评审+月功能验收
  → 实际执行中发现评审频率过高难以坚持

Do: 执行了前3次代码评审，第4周起改为双周
  → 覆盖率下降但参与质量提升

Check: 双周评审缺陷检出率20%，月度验收通过率95%
  → 标准评审流程有效

Act: 将双周评审+月验收标准化写入团队规范
  → 后续项目复用该流程
```

## 五、头脑风暴（Brainstorming）

> **适用章节**: 需求调研、风险识别、问题分析

### 说明

无约束地发散想法，再收敛整理。分两阶段：
1. **发散阶段**：不评判，只管列出所有想法
2. **收敛阶段**：归类、排序、筛选

### 用法

在"需求调研"或"风险识别"章节，先列出候选清单，再用其他工具（如 Pareto、优先级矩阵）筛选。

### 示例输出

```
## 需求调研（头脑风暴产出）

发散阶段共收集 37 条需求，经收敛归类为 4 类：

**P0（必须实现）— 8条**
- 商品管理（增删改查+批量导入）
- 订单管理（下单/支付/退款流转）
- 用户权限管理（角色+菜单）
- 数据看板（核心指标）

**P1（重要）— 12条**
- 采购入库管理
- 供应商管理
- 消息通知（站内信+企微）
...
```

## 六、思维导图（Mind Map）

> **适用章节**: WBS构建、需求分析、知识梳理

### 说明

围绕中心主题，向四周放射性展开子主题。天然适合WBS的层级分解结构。

### 用法

在构建WBS时，以思维导图的放射性结构指导分解——父节点是中心，子节点是辐射分支。注意遵守100%规则。

### 示例输出

```
中心: 电商后台系统
├── 商品管理
│   ├── 商品列表（CRUD）
│   ├── 分类管理
│   ├── 品牌管理
│   └── 库存管理
├── 订单管理
│   ├── 订单列表
│   ├── 订单详情
│   ├── 售后处理
│   └── 物流跟踪
├── 用户管理
│   ├── 用户列表
│   ├── 角色权限
│   └── 操作日志
└── 数据报表
    ├── 销售报表
    ├── 用户分析
    └── 库存预警
```

## 七、自上而下 vs 自下而上（Top-down & Bottom-up）

> **适用章节**: WBS构建、需求分析、进度规划

### 说明

两种分解策略：
- **自上而下（Top-down）**: 从顶层目标开始，逐步分解到细节。适合有清晰目标的场景
- **自下而上（Bottom-up）**: 从具体任务开始，逐步归纳到顶层。适合需求不清晰、需要先探索的场景

### 用法

撰写"范围与交付物"或"工作分解"章节时：
- 如果目标明确 → 用 Top-down：先写L1层次，再逐层细分
- 如果需求模糊 → 用 Bottom-up：列出所有具体任务，再归纳归类

两者可以结合使用：Top-down 出骨架，Bottom-up 补充细节，交叉验证。

### 示例输出

```
## 范围定义（Top-down + Bottom-up结合）

Top-down 骨架:
  电商后台 → 商品/订单/用户/报表

Bottom-up 补充:
  一线运营提出的具体需求37条
  → 归类到4大模块中发现:
     - 商品模块覆盖了21条（56%）
     - 订单模块覆盖9条（24%）
     - 跨模块需求5条 → 新增"系统配置"模块
     - 无法归类2条 → 确认是否在范围内

  ✅ 交叉验证后确认范围完整
```

## 八、Pareto分析（80/20法则）

> **适用章节**: 风险排序、需求优先级、问题分析

### 说明

80% 的效果来自 20% 的原因。找出那关键的 20% 优先处理。

### 用法

在"风险登记册"或"需求优先级"章节，列出所有项并按影响排序，取前20%作为最高优先级。

### 示例输出

```
## 风险排序（Pareto分析）

共识别 15 项风险，按影响程度排序：

Top 3（占影响总值65% → 关键控制）:
1. 第三方支付API更新 —— 影响订单支付全流程
2. 核心开发人员离职风险 —— 知识断层
3. 数据库性能瓶颈 —— 影响全部模块

其余12项（占35% → 常规管理）
```

## 九、RACI矩阵

> **适用章节**: 相关方登记册、沟通计划

### 说明

明确每个任务的参与角色：
- **R**esponsible — 执行人
- **A**ccountable — 决策人/最终责任人
- **C**onsulted — 需要征求意见的人
- **I**nformed — 需要知晓结果的人

### 用法

在"相关方登记册"章节，使用RACI矩阵定义各角色的权责边界。

### 示例输出

```
## 相关方权责（RACI矩阵）

| 任务 | 项目经理 | 业务方 | 开发团队 | 运维 | 审批人 |
|------|---------|--------|---------|------|-------|
| 需求确认 | A | R | C | I | I |
| 方案设计 | R | C | A | C | I |
| 开发实现 | I | I | R/A | I | - |
| 测试验收 | R | A | C | - | I |
| 部署上线 | C | I | R | A | I |
| 结项审批 | R | C | I | - | A |

R=执行  A=负责  C=咨询  I=告知
```

## 十、鱼骨图分析（因果分析/Ishikawa）

> **适用章节**: 经验教训、根因分析、风险应对

### 说明

从六个维度追溯问题根因：
- 人（Man）
- 机（Machine）
- 料（Material）
- 法（Method）
- 环（Environment）
- 测（Measurement）

### 用法

在"经验教训"章节，对关键问题使用鱼骨图分析根因，避免表面归因。

### 示例输出

```
## 项目延期根因分析（鱼骨图）

问题: 项目延期2周交付

人: 前端开发请假10天，未及时补人
法: 代码评审流程过重，执行率不足50%
机: 测试环境配置落后生产环境，上架前才发现差异
料: 第三方SDK版本未锁定，升级引发兼容问题
环: 年底封网期前突击完成，测试不充分
测: 自动化测试覆盖率仅30%，回归测试靠人工

→ 根因: 测试环境治理 + 代码评审流程优化
```

## 十一、风险概率-影响矩阵

> **适用章节**: 风险登记册、风险排序

### 说明

矩阵评估风险等级 = 发生概率 × 影响程度。

| 概率\影响 | 低 | 中 | 高 |
|-----------|----|----|----|
| 高 | 中 | 高 | 极高 |
| 中 | 低 | 中 | 高 |
| 低 | 低 | 低 | 中 |

### 用法

在"风险登记册"中，对每条风险评估概率和影响等级，计算风险等级。

### 示例输出

```
## 风险矩阵评估

| 风险 | 概率 | 影响 | 等级 | 策略 |
|------|------|------|------|------|
| API更新 | 中 | 高 | 🔴高 | 减轻：预研替代方案 |
| 人员离职 | 低 | 高 | 🟡中 | 转移：签订合同约束 |
| 性能瓶颈 | 中 | 中 | 🟡中 | 减轻：提前做压力测试 |
| 需求更新 | 高 | 低 | 🟡中 | 接受：预留20%缓冲 |
| 部署失败 | 低 | 低 | 🟢低 | 接受：备份回滚方案 |
```

## 十二、优先级矩阵（MoSCoW）

> **适用章节**: 需求范围、任务排序

### 说明

四象限优先级分类：
- **M**ust have — 必须实现
- **S**hould have — 应该实现
- **C**ould have — 可以实现但非必需
- **W**on't have — 本次不做

### 用法

在"范围与交付物"章节，用 MoSCoW 明确哪些功能在范围内、哪些在范围外。

### 示例输出

```
## 功能优先级（MoSCoW）

M（Must — 必须）:
  商品CRUD、订单流转、用户权限

S（Should — 应该）:
  数据看板、批量导入、操作日志

C（Could — 可以）:
  消息推送、移动端适配、多语言

W（Won't — 不做）:
  支付系统对接（二期）、AI推荐（三期）
```
