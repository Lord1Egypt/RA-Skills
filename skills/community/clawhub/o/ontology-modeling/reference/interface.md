# Interface — 描述共同能力和属性的"身份卡"

**核心本质：** 只定义"要什么"，绝不定义"怎么实现"。

> **继承是血缘，接口是护照。血缘决定你是谁，护照证明你能做什么。**

## Interface vs 继承

| 维度 | 继承 | 接口 |
|------|------|------|
| 关系本质 | **是**——猫是动物 | **能**——猫能被领养 |
| 设计意图 | 复用基类代码，形成层级分类 | 定义跨类型的共同能力契约 |
| 适用场景 | 对象间有清晰父子层级 | 毫不相关的对象需展现相同行为 |
| 实现限制 | 通常单继承 | 多实现（一个对象可实现多个接口） |

## Interface 结构（4个部分）

```
元数据：API Name、Display Name、Description
属性定义（Shared Properties）：所有实现者必须提供数据映射
Link 定义（Shared Links）：接口层定义，所有实现者共享
实现者（Implementations）：多个 Object Types，一个对象可实现多个接口
```

## 四大价值

1. **统一界面**：一个页面统治所有实现者，新增实现者无需改代码
2. **分析加速**：直接对接口做聚合分析，跨部门数据自动对齐
3. **语义统一**：从源头消灭跨部门字段命名混乱
4. **AI 可理解**：AI Agent 通过接口理解"这类对象能做什么"，是 Palantir AIP 的基础

## 设计原则

- **至少两个对象类型需要共享行为时才定义接口**——过早抽象是最常见的错误
- 保持单一职责，一个接口只做一件事
- 避免超过 2-3 层的接口继承层次
- Interface 查询底层是 union 查询，超大规模时需要分页或预聚合
