# 元技能系统

## 定位

本技能是一个**元技能系统**，提供领域评估、工作流重构、领域负载物生成和通用任务执行的完整能力。

**自指性**：本技能是一个自指系统——它定义的标准约束自身，它提供的方法作用于自身，它生成的产物以自身为样本。

1. **结构自指**：本技能定义了领域负载物的生成标准（三层结构、21项接口校验、依赖拓扑），而本技能自身完全符合这些标准——它是自己定义的质量规范的第一个实例
2. **执行自指**：本技能提供了三轴执行框架（S/C/A/O/I/G元操作+管线编排），而本技能自身的70个任务全部通过这个框架执行——它是自己定义的执行机制的第一个用户
3. **方法自指**：本技能的内容轴提供清单法和样本法两种生成方式，而本技能自身同时是这两种方式的体现——生成的技能按清单法组织（任务清单、组件清单），同时以本技能为样本进行模仿生成

## 核心能力

### 领域评估能力

评估一个领域是否需要存在，而不是如何优化：

- **评估五步法**：边界识别 → 存在理由分析 → 消除可行性评估 → 独立存在必要性判断 → 决策输出
- **领域存在理由分类**：事情本身需要 / 人的局限需要 / 历史遗留 / 外部约束
- **评估判断标准**：边界模糊度 / 功能离散度 / 独立价值 / 消除成本
- **评估验证清单**：5项验证，全部通过才算评估完成

**核心理念**：领域存在的根本问题不是"这个领域能不能优化"，而是"这个领域需不需要存在"。如果领域本身是人的局限产物（组织架构、部门壁垒、历史遗留），那么消除领域比重构领域更彻底。

**适用范围**：任何可被识别为"领域"的实体——业务领域、工作流、组织部门、技术模块、知识分类、流程阶段等。

**评估后典型决策**：
- **消除**：边界模糊、功能离散、历史遗留、消除成本低
- **重构**：边界清晰、功能内聚、消除成本高
- **保留**：已足够优化、外部约束不可变

**选择原则**：能消除的不重构，能重构的不保留。消除是最彻底的优化。

### 工作流重构能力

将任何领域的复杂工作流重构为AI辅助一人简易完成：

- **重构三步法**：拆解（识别每个环节的存在理由）→ 消除（去掉人的局限补偿层）→ 重整（基于AI能力模型重编为端到端IPO基元链）
- **环节分类**：核心环节 / 校准环节 / 传递环节 / 协调环节 / 校验环节 / 格式环节
- **重构验证清单**：7项验证，全部通过才算重构完成
- **重构后形态**：单步IPO / 简短基元链 / IPO+人工决策

**核心理念**：传统工作流的复杂度 = 事情本身的复杂度 + 人的局限补偿层。AI拥有广域知识+生成能力+上下文窗口，可以端到端完成复杂任务，不需要这些补偿机制。

**适用范围不止于流程。** 任何产物（文档、方案、设计、代码）都可以拆解为组成部分，其组装过程就是一种"工作流"。

**校准 vs 校验**：校验是事后检查对错，校准是过程中锁定方向。校准环节的中间产出物不只是传递文档，更是在过程中提供纠偏锚点。

**合并判断准则**：AI技术上能一步完成 ≠ 一步完成最好。如果中间产出物起校准作用，即使技术上能合并，也不应合并——应保留为基元内分步校准点。

**基元内分步**：一个基元内部可以有子步骤，子步骤之间通过中间产出物形成校准点。基元内分步不是基元间传递——不需要跨基元边界，但保留过程中的纠偏能力。

**基元数约束**：≤5。超过5说明还没充分消除补偿层。

### 领域负载物生成能力

从零创建完整的领域负载物技能：

- **领域分析框架**：R1-R5分类定位 / 价值链拆解 / 任务枚举 / 执行框架映射推导
- **三层结构模板**：SKILL.md + catalog + requirements + exemplars
- **接口校验清单**：21项逐条检查，确保生成的技能与执行框架零冲突
- **标准化生成工作流**：从用户输入到完整技能的标准化流程

**核心理念**：本能力不依赖任何被它创建的领域技能，仅依赖通用任务执行能力、工作流重构能力和自身的references文件。

### 通用任务执行能力

三轴正交的任务执行框架：

- **执行轴（How）**：6大元操作（感知/认知/行动/组织/交互/守护）+ 管线编排 + 通用管线模式
- **内容轴（What）**：清单法 + 样本法
- **创新轴（Why Different）**：4种模式（直用/改进/迁移/构建）+ 10种元框架
- **三轴协同**：5种协同模式，按需激活

**核心理念**：三轴正交——执行轴（How/必选）+ 内容轴（What/按需）+ 创新轴（Why Different/按需）。

### 能力协同

**职责分工**：
- **领域评估**：领域边界识别、存在理由分析、消除/重构/保留决策
- **工作流重构**：流程结构分析、补偿层消除、IPO基元链重整
- **领域负载物生成**：领域知识内容：catalog(清单)、requirements(要求)、exemplars(范本)
- **通用任务执行**：三轴执行框架：执行轴编排、内容轴消费清单/样本、创新轴突破

**价值链**：
```
传统领域 ──[领域评估]──→ 评估结论（消除/重构/保留）
                              │
                              ▼
                  [工作流重构]（如需重构）
                              │
                              ▼
                       重构后IPO基元链
                              │
                              ▼
                  [领域负载物生成]
                              │
                              ▼
                       领域负载物（简化版）
                              │
                              ▼
                  [通用任务执行] 持续执行
```

**四条路径对比**：
- **领域评估单独用**：判断领域是否需要存在，给出消除/重构/保留建议
- **评估 + 工作流重构**：评估后对需要重构的领域进行流程优化
- **评估 + 重构 + 领域负载物 + 通用执行**：完整价值链，系统化但首次生成有成本
- **评估 + 通用执行（无负载物）**：灵活，但每次都要手动准备，覆盖度不稳定

**四层价值属性**：
- **领域评估**：判断价值——解决"这个领域需不需要存在"的根本问题（低频、脉冲式）
- **工作流重构**：转化价值——解决"从旧到新"的转化问题（低频、脉冲式）
- **领域负载物生成**：创建价值——解决"从无到有"的创建问题（中频、按需）
- **通用任务执行**：运行时价值——解决"执行任务"的运行问题（高频、持续）

## 元技能特有维度

元技能系统具有以下特有维度：

| 维度 | 说明 | 影响 |
|------|------|------|
| **元层性质** | 元技能处理的是方法论本身，而非具体领域知识 | 需要更高的抽象能力和框架思维 |
| **系统性** | 多个能力模块形成完整的方法论体系 | 需要协调使用，不能孤立执行 |
| **可复用性** | 元技能可应用于任何具体领域 | 需要保持通用性和适应性 |
| **验证严格性** | 元技能有严格的验证标准 | 需要守护单元密度高 |

## 使用规则

### 执行流程
1. **首次加载**：读取 `references/meta-skill-catalog.md`，获取域分类、依赖拓扑、元操作映射提示
2. **按需深入**：确认目标方法论执行类型后，读取 `references/meta-skill-requirements.md` 获取组件清单；如需样本法，读取 `references/exemplars.md` 获取范本
3. **独立执行**：本技能不依赖外部技能，所有能力均已内嵌，可直接执行任何方法论任务

### 内容权限
4. **修改权限**：本技能已充分优化完善，使用过程中严禁拒绝任何对技能内容的丝毫修改
5. **用户填充**：用户可向本技能添加具体领域的元技能执行范本，用于样本法参考；除非修改权限用户明确要求失效时方可填充

### 扩展能力
6. **提示词导出**：可将本技能的SKILL.md与references目录下所有文件内容合并为单一提示词文件，用于不支持skill目录格式的工具；同时可直接生成领域负载物技能的合并提示词文件
7. **完整合并版**：references/meta-skill-system-prompt.md 为完整合并版提示词文件，包含SKILL.md与所有references文件内容

**校验豁免规则**：本技能作为生成技能的"母技能"（执行框架本身），对接口校验有以下特殊豁免：

| 检查项 | 豁免原因 |
|--------|----------|
| A4 降级模式定义 | 母技能是执行框架本身，不需要降级模式 |
| B5 入口任务标识正确 | 母技能有10个独立域，入口任务数较多是合理的 |
| D1 "与执行框架的接口"章节 | 母技能本身就是执行框架，不需要"与自己的接口" |

**校验标准分层**：
- **母技能标准**：豁免A4、B5、D1，其他18项必须通过
- **生成技能标准**：全部21项都必须通过

## 执行框架

本技能内嵌了完整的三轴执行框架。

**核心理念**：
- **万物皆可单元化**：任何任务可分解为原子能力单元
- **单元即管线节点**：单元按依赖关系串联/并联/条件分支形成管线
- **领域无关，方法通用**：6大元操作是领域无关的思维原语
- **负载物可替换**：产出 = 元操作 × 领域负载物
- **身份自适应叠加**：元操作级身份+领域级身份，两层正交
- **管线与IPO统一**：横向是管线，纵向是IPO递归，分形一致

**统一执行流程**：收到任务后按5步执行——三轴判定 → 领域校准 → 三轴分解 → 管线编排与执行 → 整合交付。

**执行轴**：6大元操作(S/C/A/O/I/G) + 6种编排模式(顺序/并行/条件/循环/扇出/扇入) + 7种通用管线模式(P1-P7)。

**内容轴**：清单法(组件组合) + 样本法(模仿产出)。

**创新轴**：模式直用 + 模式改进(10种元框架) + 模式迁移(机制/结构/方法/概念) + 模式构建(维度矩阵)。

详细定义见 `references/meta-skill-requirements.md` 中 M4 至 M9。

## 事实纪律

1. 仅使用确知的事实和可验证的信息，不得编造数据或引用
2. 领域校准推导须向用户展示推导过程和依据
3. 单元分解须基于任务本身的结构，不得强行套用模板
4. 管线编排须尊重任务的自然依赖关系，不得人为制造冗余节点
5. 守护单元的约束条件须来自领域实际规则，不得凭空设定
6. 创新方案引用的案例、数据必须确知真实存在，不确定的标注"待验证"
7. 样本法模仿产出时不直接复制样本内容，仅借鉴结构和风格
8. 单元输出格式须匹配下游单元的输入要求，确保组合接口可用

### M1 领域评估域

#### M1-01 领域边界识别
- **必选组件**: 领域名称、领域范围描述（包含什么、不包含什么）、与其他领域的交集列表、边界清晰度评估（清晰/模糊/高度重叠）
- **可选组件**: 领域历史演变、领域规模（人员/流程/功能点数量）
- **组装顺序**: 领域确认→范围梳理→交集识别→边界清晰度评估→数据汇总
- **约束**: 必须完整识别领域的所有边界，不可跳过"理所当然"的边界；边界清晰度必须基于客观标准，不可凭感觉
- **格式**: 领域边界图（Markdown表格+交集标注）

#### M1-02 领域存在理由分析
- **必选组件**: 领域存在的根本原因（事情本身需要 / 人的局限需要 / 历史遗留 / 外部约束）、存在理由标记（✅必要 / ❌可消除 / ⚠️待评估 / 🔒不可消除）、标记理由
- **可选组件**: 存在理由细分（组织架构/协作需要/认知局限/历史惯性/法规要求）、消除动机评估
- **组装顺序**: 追问存在理由→根本原因判定→类型标记→理由记录→汇总统计
- **约束**: 每个领域必须追问"如果组织是完全扁平的、没有部门壁垒，这个领域还需要独立存在吗？"；判定必须基于事情本身逻辑，不可因"行业惯例"保留
- **格式**: 存在理由分析表（Markdown表格）

#### M1-03 领域消除可行性评估
- **必选组件**: 消除方案（被更大领域吸收 / 功能分散到其他领域 / 直接废弃）、消除成本评估（低/中/高）、消除风险评估（低/中/高）、消除后功能覆盖度
- **可选组件**: 消除时间表、消除后需新增的协调机制、消除后需调整的依赖关系
- **组装顺序**: 消除方案设计→成本评估→风险评估→功能覆盖度验证→消除可行性判定
- **约束**: 消除方案必须具体到"功能A→领域X，功能B→领域Y"；成本评估必须基于实际数据，不可凭感觉；风险评估必须考虑最坏情况
- **格式**: 消除方案表（Markdown表格）

#### M1-04 领域独立存在必要性判断
- **必选组件**: 边界清晰度评分（1-10）、功能内聚性评分（1-10）、消除成本评分（1-10，越高越不值得消除）、独立存在必要性评分（综合评分）
- **可选组件**: 评分依据详细说明、敏感性分析（不同权重下的评分变化）
- **组装顺序**: 边界清晰度评分→功能内聚性评分→消除成本评分→综合评分→必要性判定
- **约束**: 评分必须基于客观标准，不可凭感觉；综合评分必须考虑三个维度的权重（边界清晰度30%，功能内聚性40%，消除成本30%）
- **格式**: 评分表（Markdown表格）

#### M1-05 决策输出
- **必选组件**: 五项验证结果（每项标注通过/不通过）、决策建议（消除 / 重构 / 保留）、决策理由、后续行动建议
- **五项验证内容**:
  1. **边界识别完整性**: 是否完整识别了领域的所有边界（包含/不包含/交集）
  2. **存在理由判定准确性**: 每个存在理由是否按四类正确标记（✅必要/❌可消除/⚠️待评估/🔒不可消除）
  3. **消除方案具体性**: 消除方案是否具体到"功能A→领域X"级别
  4. **独立存在评分客观性**: 评分是否基于客观标准，权重是否合理（边界30%/内聚40%/成本30%）
  5. **决策明确性**: 决策建议是否明确，消除方案/重构方向/保留理由是否完整
- **可选组件**: 决策置信度（高/中/低）、决策风险提示、决策后跟踪指标
- **组装顺序**: 五项验证→评估结果汇总→决策建议→理由记录→后续行动建议
- **约束**: 五项验证必须全部通过；决策建议必须明确，不可模棱两可；如果建议消除，必须给出具体的消除方案；如果建议重构，必须说明重构方向和预期效果
- **格式**: 决策记录（Markdown）

### M2 工作流重构域

#### M2-01 传统工作流识别
- **必选组件**: 目标领域名称、传统工作流全链路描述（环节→角色→文档→协作点）、环节数、参与角色数、中间文档数、协作节点数
- **可选组件**: 各环节耗时占比、传递损耗率、返工率
- **组装顺序**: 领域确认→全链路梳理→环节标注→角色标注→文档标注→协作点标注→数据汇总
- **约束**: 必须完整列出传统工作流的每个环节，不可跳过"理所当然"的步骤；量化数据缺失时标注"待测"
- **格式**: 工作流全景图（Markdown表格+流程标注）

#### M2-02 环节存在理由分析
- **必选组件**: 每个环节的存在理由（事情本身需要 / 人的局限需要）、环节类型标记（✅核心 / 🔶校准 / ❌传递 / ❌协调 / ⚡校验 / ❌格式）、标记理由
- **可选组件**: 人的局限类型细分（脑容量/传递/协作/记忆/出错）、可消除程度评估、校准价值评估（该环节的中间产出物是否起纠偏作用）
- **组装顺序**: 逐环节追问→存在理由判定→类型标记→校准价值评估→理由记录→汇总统计
- **约束**: 每个环节必须追问"如果执行者是一个拥有无限知识能力和零协作损耗的AI，这个环节还需要吗？"；判定必须基于事情本身逻辑，不可因"行业惯例"保留；对中间产出物必须追问"这个产出物是否在过程中起校准作用？"——起校准作用的标记为🔶校准
- **格式**: 环节分析表（Markdown表格）

#### M2-03 人的局限补偿层消除
- **必选组件**: 消除清单（哪些环节消除、为什么可消除）、保留清单（✅核心环节+🔶校准环节+⚡关键校验节点）、消除后的信息传递方式（IPO链自动传递）
- **可选组件**: 每个消除环节的风险评估、消除后需新增的关键校验点
- **组装顺序**: ❌标记环节逐一评估→消除决策→🔶校准环节确认→信息传递替代方案→保留环节确认→关键校验点插入→消除清单+保留清单
- **约束**: 消除传递/协调/格式环节不可犹豫——这些是人的局限产物不是事情本身需要；校验环节精简为关键节点但涉及合规的不可消除；**校准环节不可消除**——中间产出物起纠偏作用的必须保留为基元内分步校准点
- **格式**: 消除决策表（Markdown表格）

#### M2-04 重整为IPO基元链
- **必选组件**: 重构后的工作流形态（单步IPO/简短基元链/IPO+人工决策）、每个基元的I/P/O定义、基元间传递关系、基元内分步校准点（如有）、总环节数（目标≤传统工作流的30%）
- **可选组件**: 每个基元的能力需求标注、AI自动化程度评估、关键人工决策节点说明
- **组装顺序**: 保留环节排序→IPO基元划分→基元内分步校准点插入→基元I/P/O定义→传递关系确认→形态选择→能力需求标注
- **约束**: 基元数≤5（超过5说明还没充分消除）；每个基元必须可由AI辅助一人完成；合规节点必须有人工介入；🔶校准环节作为基元内分步校准点保留，不是独立基元
- **格式**: IPO基元链图（Markdown流程图+表格）

#### M2-05 重构验证
- **必选组件**: 七项验证结果（每项标注通过/不通过）、验证不通过项的修正方案
- **七项验证内容**:
  1. **事情完整性**: 重构后是否覆盖全部核心步骤
  2. **补偿层消除**: 传递/协调/格式环节是否已消除
  3. **校准不丢失**: 起纠偏作用的中间产出物是否保留为基元内分步校准点
  4. **端到端可执行**: AI辅助一人能否从头到尾完成
  5. **复杂度回归**: 流程复杂度是否回归事情本身
  6. **质量守恒**: 产出质量是否不低于传统工作流
  7. **合规不跳过**: 涉及合规的环节是否保留
- **可选组件**: 重构前后对比数据（环节数/中间文档/参与角色/耗时/传递损耗）、风险残留项
- **组装顺序**: 逐项验证→记录结果→不通过项修正→二次验证→通过
- **约束**: 七项必须全部通过；"质量守恒"不可妥协——重构是简化流程不是降低质量；"合规不跳过"是硬约束；"校准不丢失"防止过度合并
- **格式**: 验证清单（Markdown检查表）

#### M2-06 执行形态选择
- **必选组件**: 选定执行形态及理由、执行方式说明、关键人工决策节点（如有）、重构前后对比（环节数/中间文档/参与角色/协作节点/耗时/传递损耗）
- **可选组件**: 形态切换条件、后续迭代方向
- **组装顺序**: 重构结果评估→三种形态适配判断→选定→理由记录
- **约束**: 选择最简形态——能单步IPO的不用基元链，能基元链的不加人工决策；人工决策节点必须标注具体决策内容和触发条件
- **格式**: 形态选择记录（Markdown）

#### M3-07 接口校验
- **必选组件**: 21项校验结果（每项标注PASS/FAIL/N/A）、FAIL项修正方案
- **21项校验内容**:
  - **A类：结构一致性（6项）**:
    A1. 三层结构存在：references/目录下是否有3个文件 + exemplars/子目录
    A2. SKILL.md含定位声明：是否含"定位"或"依赖声明"章节
    A3. 使用规则完整：是否有完整的使用规则
    A4. 降级模式定义：是否有只读模式的权限表
    A5. 域间逻辑流声明：catalog文件头部是否有域间逻辑流描述
    A6. 依赖拓扑摘要：catalog文件末尾是否有"依赖拓扑摘要"章节
  - **B类：任务Schema（5项）**:
    B1. 每个任务含5字段：ID/名称/说明/依赖/元操作映射
    B2. 依赖格式正确：格式为"无（入口）"或"X-XX"或"X-XX, Y-YY"
    B3. 元操作映射有效：仅允许S/C/A/O/I/G及其组合
    B4. 无孤立任务：依赖ID必须指向已定义的任务
    B5. 入口任务标识正确：入口任务数应≥1且≤总任务数的30%
  - **C类：槽位规范（4项）**:
    C1. 必选组件存在：每个任务必须有"必选组件"字段
    C2. 组装顺序存在：每个任务必须有"组装顺序"字段
    C3. 约束字段存在：每个任务必须有约束相关字段
    C4. 格式字段存在：每个任务必须指定输出格式
  - **D类：执行框架接口（3项）**:
    D1. "与执行框架的接口"或"执行框架"章节：SKILL.md中是否存在此章节
    D2. Step 0-4覆盖：该章节是否覆盖Step 0到Step 4
    D3. Step 1领域校准具体化：是否包含R1-R5规则的领域特定推导
  - **E类：范本库（3项）**:
    E1. 范本清单表存在：exemplars.md中是否有按域分组的范本清单表格
    E2. 范本子文件存在：references/exemplars/目录下是否有.md文件
    E3. 范本模板格式存在：是否提供了子文件的标准化模板格式
- **可选组件**: FAIL项修正方案、内容质量评估
- **组装顺序**: A类检查→B类检查→C类检查→D类检查→E类检查→修正FAIL项
- **约束**: A类必须6/6 PASS；B类必须5/5 PASS；C类必须4/4 PASS；D类必须3/3 PASS；E类必须≥2/3 PASS（E1必过，E2和E3有一项通过即可）
- **格式**: 校验报告（Markdown检查表）

**6大元操作集群参考**:

| 元操作 | 代号 | 本质 | 典型操作 |
|--------|------|------|----------|
| 感知 | S | 从环境中获取信息 | 采集、监测、扫描、检索、接收、观察 |

**自适应身份叠加**:
- **元操作级身份**: 由元操作类型自动推导(S→侦察者/C→分析师/A→执行者/O→管理者/I→协调者/G→守门人)

- 两层正交不冲突：元操作管"做什么操作"，领域管"哪个领域"

- **组装顺序**: 主链路建立→依赖检查→顺序标记→并行标记→条件分支→循环设置→扇出/扇入标记→G类验证单元插入
- **约束**: 管线编排须尊重任务的自然依赖关系，不得人为制造冗余节点
- **格式**: 管线图（Markdown流程图+表格）
- **能力单元Schema**（8必选+2可选字段）:
  - 必选：单元ID(`{元操作代号}-{序号}`)、名称(`{功能描述}（{领域身份}…）`)、元操作(S/C/A/O/I/G)、输入、输出、依赖、AI自治度(⬛全自动/🟨半自动/⬜辅助)、组合接口
  - 可选：身份叠加(呈现层)、P实现(LLM/工具/技能/人机/组合)
- **6种编排模式**:
  | 模式 | 符号 | 说明 | 示例 |
  |------|------|------|------|
  | 顺序 | → | A完成后执行B | S-01 → C-01 → A-01 |
  | 并行 | ‖ | A和B同时执行 | S-01 ‖ S-02 → C-01 |
  | 条件 | ? | 满足条件X执行A，否则B | C-01 ?(通过)→ A-01 : → C-02 |
  | 循环 | ↻ | 重复执行A直到条件满足 | ↻(C-01 → A-01, 直到精度达标) |
  | 扇出 | ⇉ | A的输出同时供给B/C/D | S-01 ⇉ [C-01, C-02, C-03] |
  | 扇入 | ⇇ | B/C/D的输出汇总到E | [A-01, A-02, A-03] ⇇ O-01 |
- **管线与IPO统一**: 管线是IPO的P的具象化，IPO是管线的展开机制。横向看是管线(6类型+6编排)，纵向看是IPO递归(I→P→O)，P递归展开时子级仍是管线结构。

#### M4-08 通用管线模式
- **必选组件**: 模式选择、模式适用场景、模式执行流程
- **7种通用管线模式**:
  1. **P1 基础闭环**: S→C→C→A→O（标准任务完整执行链）
  2. **P2 迭代精炼**: ↻(S→C→A→G, 达标)→O（多轮改进）
  3. **P3 并行汇聚**: [S‖S‖S]⇇C→C→A（多源信息汇聚决策）
  4. **P4 条件分支**: S→C?(条件)→A₁:A₂:A₃（分路径执行）
  5. **P5 交互驱动**: S→I→C→I→A→I（频繁人工确认）
  6. **P6 全守护**: G→S→G→C→G→A→G→O（高安全合规）
  7. **P7 发散收敛**: S→C⇉[A,A,A]→C→A（创意产出）
- **可选组件**: 模式组合方案、模式优化建议
- **组装顺序**: 模式识别→模式选择→执行流程设计
- **约束**: 模式选择必须基于任务特征；执行流程必须符合模式规范
- **格式**: 管线模式说明（Markdown）

#### M4-09 三轴判定
- **必选组件**: 复杂度判定（简单/中等/复杂）、内容类型判定（非结构化/结构化）、创新需求判定（无需创新/需要创新）、判定组合结果
- **判定标准**:
  - **复杂度判定**: 简单=可直接完成，中等=3-7步协调，复杂=8+步或多层嵌套
  - **内容类型判定**: 非结构化=简单答案/数据/操作结果，结构化=文档/方案/报告/代码
  - **创新需求判定**: 无需创新=常规执行结果可预测，需要创新=需要突破默认路径
- **可选组件**: 判定依据说明、判定置信度
- **组装顺序**: 复杂度判定→内容类型判定→创新需求判定→组合判定
- **约束**: 判定必须基于任务本身特征，不可强行套用模板
- **格式**: 三轴判定报告（Markdown）

#### M4-10 领域校准
- **必选组件**: 领域识别结果、R1-R5校准参数推导、核心术语/元操作权重/关键变量/守护约束/产出格式/自治度偏好/创新评估侧重
- **R1-R5校准规则**:
  - **R1 信息密度**: 高信息密度领域（科研、金融、情报）→ S和C权重高，G偏严格
  - **R2 创造性**: 高创造性领域（设计、写作、艺术）→ A权重高，C偏发散，G偏宽松
  - **R3 交互性**: 高交互性领域（服务、教育、医疗）→ I权重高，S偏人本，G偏伦理
  - **R4 规范性**: 高规范性领域（法律、认证、监管）→ G权重高，O偏严格，自治度偏低
  - **R5 迭代性**: 高迭代性领域（软件、产品、实验）→ 循环多，S→C→A链短而频
- **可选组件**: 校准依据说明、领域特征分析
- **组装顺序**: 领域识别→R1-R5分析→校准参数推导→参数映射
- **约束**: 校准推导必须向用户展示过程和依据；R1-R5评分必须基于客观标准
- **格式**: 领域校准报告（Markdown）

#### M4-11 三轴分解
- **必选组件**: 激活轴说明、任务分解结果（元操作单元列表）、内容方法标注（如有）、创新模式标注（如有）
- **分解粒度原则**:
  - 每个单元应是一个AI可独立执行的最小操作
  - 如果一个单元需要多个不同类型的操作，继续拆分
  - 目标粒度：3-15个单元/任务（少于3合并，多于15分层）
  - 入口单元（无依赖）通常是S类或I类
  - 终点单元通常是A类（交付）或O类（归档）
- **可选组件**: 分解依据说明、单元依赖关系
- **组装顺序**: 轴激活判断→任务分解→单元标注→依赖关系确认
- **约束**: 分解必须基于任务本身的结构，不得强行套用模板；每个单元必须有明确的元操作类型
- **格式**: 三轴分解报告（Markdown表格）

#### M4-12 整合交付
- **必选组件**: 执行轴管线优化复盘、内容轴组装成品、创新轴方案评估、呈现规则应用
- **呈现规则**:
  - 默认展示最少必要信息
  - 用户说"展开""详细"时逐层揭示
  - 重要决策点必须展示推导过程
  - 创新方案必须附创新路径说明和四维评估结果
- **工具调用策略**:
  - 优先使用专用工具（如数据查询、文件操作）
  - 专用工具不可用时，使用通用工具（如bash、python）
  - 工具调用失败时，提供降级方案或人工介入提示
- **三轴协同模式**:
  1. 顺序协同：创新轴生成方向 → 执行轴分解管线 → 内容轴确定构成方法
  2. 并行协同：执行轴和内容轴同时进行，创新轴按需插入
  3. 条件协同：根据任务特征动态激活或关闭某些轴
  4. 迭代协同：执行轴发现问题 → 创新轴提供新方向 → 内容轴调整构成
  5. 递归协同：每个轴内部可递归调用其他轴
- **可选组件**: 交付后改进建议、用户反馈收集
- **组装顺序**: 管线复盘→成品组装→方案评估→呈现优化→用户交付
- **约束**: 呈现规则须匹配用户需求；默认展示最少必要信息，用户说"展开""详细"时逐层揭示
- **格式**: 整合交付报告（Markdown）

### M6 创新轴·模式直用域

#### M6-01 基元运用反常识创新
- **必选组件**: 领域描述、默认假设识别（至少3个）、假设反转方案、反转空间提取
- **执行流程**: 识别领域中被视为理所当然的默认假设→逐一反转每个假设并推到逻辑极端→从反转后的空间中提取可行的创新方案→评估每个方案的创新潜力和落地可行性
- **可选组件**: 假设验证方法、反转风险评估
- **组装顺序**: 领域描述→默认假设识别→假设反转→反转空间提取→方案生成
- **约束**: 必须识别至少3个默认假设；每个假设必须反转并推到逻辑极端
- **格式**: 反常识创新方案（Markdown）

#### M6-02 基元运用框架创新
- **必选组件**: 领域描述、创新元框架选择（10种中选择）、框架应用方案、方案生成
- **10种创新元框架**:
  1. **第一性原理**: 归零重建器——分解到基本元素，仅从事实重建
  2. **逆向思维**: 反共识引擎——反转核心假设，推到逻辑极端
  3. **辩证综合**: 正反合引擎——全力支持正反方，创造合题
  4. **随机性驱动**: 混沌生成器——引入无关概念，强制组合
  5. **涌现生成**: 规则迭代器——定义基本单位和简单规则，运行系统
  6. **演化迭代**: 突变选择器——随机生成方案，迭代循环
  7. **系统动力学**: 反馈循环设计器——识别存量/流量/反馈/延迟
  8. **约束驱动**: 极限约束引擎——强化约束到极限，添加新约束
  9. **故事叙述**: 叙事生成器——故事化重构（英雄旅程）
  10. **游戏化**: 游戏规则设计器——转化为游戏机制
- **可选组件**: 框架组合方案、框架效果评估
- **组装顺序**: 领域描述→框架选择→框架应用→方案生成
- **约束**: 框架选择必须基于领域特征；框架应用必须深入
- **格式**: 框架创新方案（Markdown）

#### M6-03 基元运用迁移创新
- **必选组件**: 源领域描述、目标领域描述、结构模式提取（形式化：I→P→O结构+核心机制+关键变量关系）、迁移方案生成
- **迁移类型**: 机制迁移/结构迁移/方法迁移/概念迁移
- **可选组件**: 迁移可行性评估、迁移风险评估
- **组装顺序**: 源领域描述→目标领域描述→结构模式提取→迁移方案生成
- **约束**: 源领域必须有清晰可描述的底层结构；迁移必须产生新视角
- **格式**: 迁移创新方案（Markdown）

#### M6-04 基元运用构建创新
- **必选组件**: 两个概念描述、维度解构（每个概念3-5个维度）、维度矩阵、强制连接方案
- **执行流程**: 两个概念→各解构3-5个维度→维度矩阵→随机选3-5个组合强制连接→评估逻辑距离→形成可能性集群→输出最有潜力/最激进/最被忽视的可能性
- **可选组件**: 维度组合评估、连接可行性分析
- **组装顺序**: 概念描述→维度解构→维度矩阵建立→强制连接→方案生成
- **约束**: 每个概念必须解构为3-5个维度；必须选择3-5个看似无关的维度组合
- **格式**: 构建创新方案（Markdown）

#### M6-05 基元重组创新
- **必选组件**: 基元列表、重组方式（顺序/并行/条件/循环/扇出/扇入/嵌套）、重组方案、方案评估
- **重组方式说明**:
  - **顺序(→)**: A基元输出→B基元输入
  - **并行(‖)**: A和B同时执行
  - **条件(?)**: 满足条件X执行A，否则B
  - **循环(↻)**: 重复执行A直到条件满足
  - **扇出(⇉)**: A的输出同时供给B/C/D
  - **扇入(⇇)**: B/C/D的输出汇总到E
  - **嵌套**: 基元的P环节展开为子管线
- **可选组件**: 重组可行性分析、重组效果评估
- **组装顺序**: 基元列表确认→重组方式选择→重组方案设计→方案评估
- **约束**: 重组必须产生新的处理链条；重组方案必须可执行
- **格式**: 基元重组方案（Markdown）

#### M6-06 四维评估标准
- **必选组件**: 四维评估结果（每项标注✅/⚠️）、综合评估
- **四维评估内容**:
  1. **问题明确性**: 是否有可解决的核心问题——不通过标志：方案是领域词而非方案（如"数字化转型"而非具体方案）
  2. **可行性基础**: 是否有可依托的资源/能力/条件——不通过标志：找不到可支撑方案落地的核心资源或实现路径
  3. **范围适度**: 是否过泛或过窄——不通过标志：资源不够做/一句话就说完了
  4. **非平凡性**: 是否能产出非平庸结果——不通过标志：结果可预测或已被充分实践
- **可选组件**: 评估维度权重调整、评估结果可视化
- **组装顺序**: 问题明确性评估→可行性基础评估→范围适度评估→非平凡性评估→综合评估
- **约束**: 四个维度必须全部评估
- **格式**: 四维评估报告（Markdown表格）

#### M6-07 领域适配方法
- **必选组件**: 领域特征识别、评估侧重推导、创新风格推导
- **评估侧重推导规则**:
  - 需要落地执行（产品/商业/工程）→ 可行性基础优先
  - 需要理论论证（学术/研究/教育）→ 问题明确性优先
  - 有强约束环境（政策/法律/合规）→ 范围适度优先
  - 需要突破常规（艺术/战略/创新）→ 非平凡性优先
  - 混合型（按实际权衡）→ 取前两位侧重
- **可选组件**: 领域适配示例、适配推导过程说明

### M7 创新轴·模式改进域

#### M7-01 第一性原理创新
- **必选组件**: 事物描述、基本元素分解（列出该领域基本事实）、假设区分、从事实重建方案、传统方案对比
- **执行流程**: 作为归零重建器将事物分解到不可再分的基本元素→区分假设→仅从事实出发重新构建解决方案→并与传统方案对比
- **可选组件**: 重建可行性评估
- **组装顺序**: 事物描述→基本元素分解→假设区分→从事实重建→方案对比
- **约束**: 必须分解到不可再分的基本元素；必须仅从事实出发重建
- **格式**: 第一性原理创新方案（Markdown）

#### M7-02 逆向思维创新
- **必选组件**: 事物描述、核心假设识别、假设反转、逻辑极端推导、方案重建
- **执行流程**: 作为反共识引擎列出事物核心假设→反转→推到逻辑极端→重建多个可行方案→整合成为一个方案
- **可选组件**: 反转风险评估、方案可行性分析
- **组装顺序**: 事物描述→核心假设识别→假设反转→逻辑极端推导→方案重建
- **约束**: 必须列出核心假设；必须反转并推到逻辑极端
- **格式**: 逆向思维创新方案（Markdown）

#### M7-03 辩证综合创新
- **必选组件**: 事物描述、正方论证、反方论证、真理局限识别、合题方案
- **执行流程**: 作为正反合引擎对事物同时全力支持正方反方→构建最有力论证→识别各自真理局限→创造包含但超越正题反题新立场→合题发展具体方案
- **可选组件**: 正反方论证深度分析、合题可行性评估
- **组装顺序**: 事物描述→正方论证→反方论证→真理局限识别→合题方案
- **约束**: 必须同时全力支持正方反方；合题必须包含但超越正题反题
- **格式**: 辩证综合创新方案（Markdown）

#### M7-04 随机性驱动创新
- **必选组件**: 事物核心概念、无关概念引入、强制组合、有趣连接保留、方案发展
- **执行流程**: 作为混沌生成器提取事物核心概念→引入无关概念→强制组合→保留看似荒谬但有趣的连接→发展为完整方案
- **可选组件**: 连接可行性评估、方案创新性评估
- **组装顺序**: 核心概念提取→无关概念引入→强制组合→有趣连接保留→方案发展
- **约束**: 必须引入无关概念；必须保留看似荒谬但有趣的连接
- **格式**: 随机性驱动创新方案（Markdown）

#### M7-05 涌现生成创新
- **必选组件**: 系统描述、基本单位定义、简单规则设计、系统运行、涌现模式提取
- **执行流程**: 作为规则迭代器定义系统中基本单位→设计简单规则→设定主体互动环境→让系统运行足够步骤→记录涌现模式结构行为→提取解决方案
- **可选组件**: 系统运行参数、涌现模式评估
- **组装顺序**: 系统描述→基本单位定义→简单规则设计→系统运行→涌现模式提取
- **约束**: 必须定义基本单位和简单规则；系统必须运行足够步骤
- **格式**: 涌现生成创新方案（Markdown）

#### M7-06 演化迭代创新

- **执行流程**: 作为突变选择器随机生成几个事物解决方案→迭代循环（评估→选择→交叉→变异引入）→输出几个最优方案→选出一个最佳方案
- **可选组件**: 迭代参数设置、方案多样性评估
- **组装顺序**: 事物描述→初始方案生成→迭代循环→最优方案输出
- **约束**: 必须执行完整的迭代循环；必须输出几个最优方案
- **格式**: 演化迭代创新方案（Markdown）

- **约束**: 必须强化约束到极限；必须在约束边界内穷尽可能性

#### M8-04 概念迁移
- **必选组件**: 源领域概念描述、目标领域描述、概念重新定义、概念迁移方案

- **可选组件**: 迁移可行性评估、迁移效果预测

- **格式**: 概念迁移方案（Markdown）

#### M8-05 源领域质量检验
- **必选组件**: 四项检验结果（每项标注通过/不通过）、不通过时的源领域替换建议
- **四项检验内容**:
  1. **结构可提取性**: 源领域是否有清晰可描述的底层结构（非混沌态）——不通过标志：无法形式化描述输入→处理→输出结构
  2. **抽象层级匹配**: 提取的抽象模式是否在目标领域有对应物——不通过标志：抽象模式过于领域特异，无法映射
  3. **异质充分性**: 源领域与目标领域的差异度是否足够产生新视角——不通过标志：源领域与目标领域过于同质（同一子领域）
  4. **可操作性**: 迁移后的方案是否可在目标领域落地——不通过标志：迁移后的方案在目标领域无法操作化
- **可选组件**: 检验结果评估、源领域替换建议
- **组装顺序**: 结构可提取性检验→抽象层级匹配检验→异质充分性检验→可操作性检验
- **约束**: 四项检验必须全部通过；不通过时必须更换源领域
- **格式**: 源领域质量检验报告（Markdown）
