# Step 4 — 用户故事

格式严格遵守：
```
[US-XXX]
作为 <角色>，
我希望 <做什么动作>，
以便 <达到什么业务目的>。

界面标题：<该故事对应的页面/模块在界面左上角显示的名称>

验收条件：
- 当 <前置条件> 时，<角色> 可以 <操作>
- 操作完成后，系统应 <结果>
- 异常情况：<边界处理>
```

**「界面标题」填写规则**：
- 填写该用户故事对应页面在界面左上角显示的**完整文字**，例如「我的任务」「质检模板管理」
- 同一个页面的多个用户故事必须填写**完全相同**的界面标题（一字不差）
- 如果一个故事跨多个页面（如弹窗、抽屉），填写**触发入口所在页面**的标题
- 此字段将作为原型页面标题和 PRD 模块命名的**唯一来源**，后续步骤禁止自行改名

## 规则
- 每个角色至少一个故事
- 每条期望至少映射一个故事
- 业务流三段（输入/中间/输出）都有故事覆盖
- 编号 US-001, US-002...
- **界面标题不允许留空**，信息不足时用 `[TODO: 确认页面名称]` 占位并在 Step 7 前补齐

→ `01-user-stories.md`，**STOP** 等用户确认。修改循环至确认。

> **产品化改造专用规则**：当本文件由 `productization/SKILL.md` 调用时，**只为 Step2 改造清单中的改造点和新增功能编写用户故事**。现有系统已有且无需改动的功能不编写故事，避免重复描述原有系统。
