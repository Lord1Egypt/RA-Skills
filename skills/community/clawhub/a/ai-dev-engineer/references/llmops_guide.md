## 二、Prompt 管理平台选型

| 工具 | 类型 | 核心功能 | 适合规模 |
|------|------|---------|:------:|
| **LangFuse** | 开源 LLMOps | Trace + Eval + Prompt管理 + 成本 | 中小团队首选 |
| **LangSmith** | 商业 SaaS | 全功能 LLMOps | 有预算的团队 |
| **Weights & Biases Prompts** | 商业 | Prompt 版本 + A/B + 实验追踪 | 已有 W&B 的团队 |
| **自建 (Git + 注册中心)** | 自建 | 完全控制, 零成本 | 追求灵活性 |

> **推荐**: 初期用 LangFuse (开源+功能全) → 规模化后按需自建部分组件

## 三、推理网关架构

```
客户端请求
    │
    ▼
┌─────────────┐
│  API Gateway │  ← 统一入口: 鉴权/限流/日志
└──────┬──────┘
       │
       ▼
┌─────────────┐     ┌──────────────┐
│ Semantic     │────▶│ Model Router │  ← 复杂度判断 → 选模型
│ Cache        │     └──────┬───────┘
└─────────────┘            │
                            ▼
                   ┌────────────────┐
                   │ Circuit Breaker │  ← 熔断检测
                   └───────┬────────┘
                           │
              ┌────────────┼────────────┐
              ▼            ▼            ▼
         ┌────────┐  ┌────────┐  ┌────────┐
         │Primary │  │Fallback│  │Fallback│  ← 多模型降级链
         │gpt-4o  │  │deepseek│  │qwen    │
         └────────┘  └────────┘  └────────┘
```

### 网关关键功能

1. **智能路由**: 简单问题→小模型, 复杂→强模型, 省钱30-70%
2. **熔断降级**: 主模型不可用→自动切换备用模型
3. **A/B测试**: 按流量比例分配不同模型/Prompt, 数据驱动决策
4. **速率限制**: 按用户/token维度限流, 防止成本爆炸
5. **成本追踪**: 每次调用实时记录模型+token+费用

## 五、反馈闭环实施

### RLHF 轻量版 (适合中小团队)

```
用户交互 → 点赞/点踩/修正 → 日志收集
                                    ↓
                            每周 Bad Case 分析
                                    ↓
                            ┌───────┴───────┐
                            ▼               ▼
                      Prompt 优化      知识库补充
                            │               │
                            └───────┬───────┘
                                    ↓
                            A/B 测试验证
                                    ↓
                              灰度上线
```

### 实施要点

1. **收集什么**: 点赞/点踩、重新生成、用户修正、对话中断
2. **分析频率**: 每周 Review Bad Case Top 20
3. **改进策略**: Prompt问题(40%) > 知识库缺失(30%) > 模型能力(20%) > 架构(10%)
4. **验证方式**: 修复后在 Golden Dataset 上回归 + 灰度 A/B 测试

## 六、成本优化实战

### 三级优化策略

| 级别 | 手段 | 预期节省 | 实施难度 |
|:---:|------|:------:|:------:|
| L1 | Prompt 精简 + max_tokens 限制 | 10-20% | 低 |
| L1 | 语义缓存 (Redis + 向量DB) | 30-50% | 中 |
| L2 | 模型智能路由 (简单→小模型) | 30-50% | 中 |
| L2 | 批处理 (非实时场景合并请求) | 20-40% | 中 |
| L3 | Prompt Caching (重复前缀复用) | 50%+ | 低(API原生) |
| L3 | 模型蒸馏/量化 (小模型替代) | 50-70% | 高 |

### 典型成本结构分析

```
月成本 = 输入Token费用 + 输出Token费用 + Embedding费用 + 向量库费用

示例: 日活1000, 每人日均10次对话, GPT-4o-mini
├─ 输入: 500 tokens × 10次 × 1000人 × 30天 × $0.15/1M = $22.5
├─ 输出: 200 tokens × 10次 × 1000人 × 30天 × $0.60/1M = $36
├─ Embedding: 1000 docs × 1536维 × $0.02/1M = 极小
└─ 合计: ~$60/月 (优化前)

启用缓存(50%命中) + 模型路由(30%走更便宜模型):
└─ 优化后: ~$25/月 (节省 58%)
```

## 七、常见故障排查

| 症状 | 可能原因 | 排查步骤 |
|------|---------|---------|
| 成本突增 | 死循环/长Prompt/缓存失效 | 1.查Token消耗趋势 2.查Agent步数 3.查缓存命中率 |
| 延迟飙升 | 模型排队/网络/上下文过长 | 1.查P99延迟 2.查输入token长度 3.检查API限流 |
| 质量下降 | Prompt漂移/模型更新/数据污染 | 1.跑 Golden Dataset 回归 2.对比新旧版本 3.查模型变更日志 |
| Agent跑飞 | 工具描述歧义/无step限制 | 1.查Agent Trace 2.检查工具描述 3.添加step硬限制 |
