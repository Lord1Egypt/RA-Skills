# Phase 1: 需求分析与测试策略 (6 Modules)

---

## M01: 需求可测性分析

### 核心方法论
需求的"可测性三要素"：**可观测性** × **可控制性** × **可判定性**。一项需求只有同时满足三个要素才能进行有效测试。

### 专家操作流程
1. **需求解构**：将需求拆解为功能点列表，每个功能点标注输入/处理/输出
2. **可观测性检查**：
   - 输出是否可直接观察？（UI/API/日志/数据库）
   - 是否有时序依赖导致观察窗口有限？
   - 是否需要Mock/Stub才能观察？
   - 评分：1-5分（<3分标记为测试风险）
3. **可控制性检查**：
   - 输入是否可模拟？（手动/脚本/工具）
   - 是否有外部依赖不可控？（第三方服务/硬件）
   - 状态是否可重置？
   - 评分：1-5分（<3分标记为测试风险）
4. **可判定性检查**：
   - 预期结果是否明确？（模糊需求=不可判定）
   - 是否有Oracle问题？（无明确基准判断对错）
   - 多条件组合下是否可判定？
   - 评分：1-5分（<3分标记为需求缺陷）
5. **生成可测性报告**：
   - 总评分 = （可观测+可控制+可判定）/3
   - ≥4分：可测，按标准流程
   - 2.5-4分：需增加测试桩/环境改造
   - <2.5分：需求需返工，标记为阻塞

### 关键检查点
- [ ] 每个功能点都完成了三要素评估
- [ ] 识别了所有外部依赖并评估替代方案
- [ ] 模糊需求已标记并反馈给产品
- [ ] Oracle问题有明确的解决方案（参考实现/专家判定/交叉验证）

### 常见陷阱
| 陷阱 | 对策 |
|------|------|
| 把"需求文档完整"等同于"需求可测" | 即使文档完整，仍需验证可观测/可控制/可判定 |
| 忽略异步逻辑的可测性 | 异步操作需考虑回调/事件/消息队列的观察点 |
| 默认第三方服务可用 | 每个外部依赖都需设计Mock方案 |

### 产出模板
```
## 需求可测性分析报告
- 需求ID/名称：
- 功能点总数：N
- 可测功能点数：M（占比：M/N%）
- 风险项列表：
  | 功能点 | 可观测 | 可控制 | 可判定 | 综合 | 风险描述 | 建议 |
- 阻塞项：（需返工的需求点）
- 测试桩需求：（需开发的Mock/Stub清单）
```

### AI增强技巧
- 输入需求文档，让AI先做初步的可测性评分
- 针对AI识别的风险点，人工确认是否有遗漏
- 用AI生成Mock接口的代码骨架

## M02: 测试策略制定

### 核心方法论
**测试四象限模型**（适配自Agile Testing Quadrants）：
- Q1（技术面·面向团队）：单元测试、组件测试 → 自动化
- Q2（业务面·面向团队）：功能测试、场景测试、用例测试 → 自动化+人工
- Q3（业务面·面向产品）：探索性测试、易用性测试、验收测试 → 人工为主
- Q4（技术面·面向产品）：性能测试、安全测试、兼容性测试 → 专项工具

### 专家操作流程
1. **产品分析**：
   - 产品类型（Web/APP/API/SDK/嵌入式）
   - 技术架构（微服务/单体/Serverless）
   - 发布节奏（周/双周/月/按需）
   - 用户规模与可用性要求（99.9%/99.99%）
2. **四象限占比分配**：
   - Q1：30-40%（基础质量保证）
   - Q2：30-35%（业务正确性）
   - Q3：15-20%（用户体验）
   - Q4：10-15%（非功能质量）
3. **测试金字塔适配**：
   - 单元测试：70%（快速反馈）
   - 接口/集成测试：20%（接口契约验证）
   - UI/端到端测试：10%（关键路径）
4. **方法选择矩阵**：
   | 测试类型 | 适用方法 | 自动化程度 | 执行频率 |
   |---------|---------|-----------|---------|
   | 新功能 | 场景法+探索性 | 半自动 | 每迭代 |
   | 回归 | 等价类+边界值 | 全自动 | 每次提交 |
   | 性能 | 负载模型 | 工具辅助 | 每版本 |
   | 安全 | OWASP检查表 | 半自动 | 每版本 |
5. **资源策略**：人力分配、环境需求、工具选型、时间线

### 关键检查点
- [ ] 四象限都有覆盖，没有遗漏象限
- [ ] 自动化/人工比例适合产品的发布节奏
- [ ] 高风险区域（支付、安全、核心流程）有额外的测试深度
- [ ] 策略文档经过开发+产品+测试三方评审

### 产出模板
```
## 测试策略文档
### 1. 产品概况
### 2. 质量目标（可用性/缺陷率/覆盖率）
### 3. 测试范围（in/out scope）
### 4. 四象限策略
### 5. 测试金字塔分配
### 6. 自动化策略（工具/框架/覆盖率目标）
### 7. 环境策略（环境数量/数据策略/配置管理）
### 8. 风险与应对
### 9. 时间线与里程碑
```

## M03: 测试范围与优先级划分

### 核心方法论
**基于风险的测试（RBT, Risk-Based Testing）**：
风险优先级 = 失效概率 × 失效影响

### 专家操作流程
1. **功能清单梳理**：列出所有功能点，标注所属模块
2. **失效概率评估（1-5分）**：
   - 复杂度：逻辑分支多、数据流复杂 → 高分
   - 变更频率：新增/修改频繁 → 高分
   - 历史缺陷：该模块缺陷密度高 → 高分
   - 技术难度：新技术栈/复杂集成 → 高分
   - 团队经验：团队不熟悉的技术 → 高分
3. **失效影响评估（1-5分）**：
   - 用户影响：影响核心用户路径 → 高分
   - 业务影响：涉及资金/合规/数据安全 → 高分
   - 可恢复性：恢复困难/时间长 → 高分
   - 品牌影响：用户可见/引发舆情 → 高分
4. **风险矩阵**：
   - 风险分 = 失效概率 × 失效影响
   - ≥15分：P0（必须测，深度覆盖）
   - 10-14分：P1（应该测，标准覆盖）
   - 5-9分：P2（可以测，基本覆盖）
   - <5分：P3（时间允许时测，冒烟覆盖）
5. **测试深度定义**：
   - P0深度：等价类+边界值+异常+组合+探索
   - P1深度：等价类+边界值+关键异常
   - P2深度：等价类+关键边界值
   - P3深度：Happy Path冒烟

### 关键检查点
- [ ] 所有利益相关者对风险评分达成共识
- [ ] P0功能点的测试深度达到"深度覆盖"标准
- [ ] 没有"高影响低概率"的功能被遗漏
- [ ] 范围文档明确标注了 out-of-scope

## M04: 风险识别与评估

### 核心方法论
**FMEA（失效模式与影响分析）** 适配测试版：
1. 识别失效模式（What can go wrong?）
2. 分析失效影响（What happens if it fails?）
3. 评估失效原因（Why would it fail?）
4. 设计检测手段（How to detect it?）
5. 制定预防措施（How to prevent it?）

### 专家操作流程
1. **技术风险识别**：
   - 新引入的技术栈/框架
   - 第三方服务集成（API/SDK/支付/地图）
   - 数据迁移/数据库变更
   - 并发/分布式场景
   - 性能瓶颈点（高QPS接口/大数据量查询）
   - 安全攻击面（用户输入/文件上传/权限）
2. **业务风险识别**：
   - 资金流转（支付/退款/提现）
   - 合规要求（GDPR/个人信息保护法）
   - 核心业务流程（下单/登录/数据同步）
   - 用户数据安全（密码/手机号/身份证）
3. **进度风险识别**：
   - 需求变更频率
   - 环境不稳定/不可用
   - 第三方联调延迟
   - 人员变动
4. **风险登记册（Risk Register）**：
   | 风险ID | 描述 | 类型 | 概率 | 影响 | RPN | 缓解措施 | 负责人 |
5. **RPN计算**：Risk Priority Number = 概率(1-5) × 影响(1-5) × 可检测性(1-5)
   - 可检测性：1=极易检测，5=极难检测
   - RPN>60：立即行动
   - RPN 30-60：制定计划
   - RPN<30：持续监控

## M05: 测试估算与资源规划

### 核心方法论
**三点估算法 + 测试用例点（Test Point）**：
工作量 = (最乐观 + 4×最可能 + 最悲观) / 6

### 专家操作流程
1. **测试用例点数估算**：
   - 简单用例（单步骤+单数据+单验证）：1 TP
   - 中通用例（多步骤+多数据+多验证）：2 TP
   - 复杂用例（状态迁移+边界组合+异常）：4 TP
   - 自动化用例（需编写脚本）：3-5 TP
2. **人天换算**：
   - 新手：5-8 TP/天
   - 熟练：10-15 TP/天
   - 专家：15-25 TP/天
3. **阶段工时分配**：
   - 需求分析+策略：15%
   - 测试设计：25%
   - 用例编写：30%
   - 执行+缺陷：20%
   - 报告+总结：10%
4. **缓冲系数**：
   - 需求稳定项目：×1.2
   - 需求变化项目：×1.5
   - 新团队/新技术：×1.8
5. **资源规划清单**：
   - 人力：测试人数 × 技能匹配度
   - 环境：测试环境数量 × 数据准备时间
   - 工具：License费用 + 学习成本
   - 设备：真机/浏览器/OS版本矩阵

## M06: 需求评审检查清单

### 核心方法论
**需求质量6维度评审**：
完整性 × 正确性 × 可行性 × 必要性 × 优先级 × 可验证性

### 测试视角的需求评审清单（50项）

#### A. 完整性 (10项)
- [ ] 所有功能点都有明确的输入定义
- [ ] 所有功能点都有明确的输出定义
- [ ] 异常场景有处理逻辑
- [ ] 边界条件有明确说明
- [ ] 权限控制有明确规则
- [ ] 数据格式有明确定义
- [ ] 依赖的第三方服务有明确说明
- [ ] 并发场景有处理策略
- [ ] 数据持久化需求明确
- [ ] 国际化/多语言需求明确

#### B. 正确性 (10项)
- [ ] 业务规则与业务方确认一致
- [ ] 计算公式与领域专家确认
- [ ] 数据流向与架构设计一致
- [ ] 状态机定义完整且无死锁
- [ ] 时间相关逻辑考虑了时区
- [ ] 金额计算精度有明确定义
- [ ] 数据校验规则与后端一致
- [ ] 排序/分页逻辑有明确规则
- [ ] 缓存策略有明确说明
- [ ] 消息推送逻辑有明确触发条件

#### C. 可验证性 (10项)
- [ ] 每个验收标准都可量化
- [ ] 没有"用户体验好""性能好"等模糊描述
- [ ] 性能指标有具体数值（响应时间<200ms）
- [ ] UI交互有明确的预期行为
- [ ] 错误提示有明确的文案
- [ ] 日志/埋点需求有明确的字段定义
- [ ] 兼容性要求有明确的浏览器/OS版本
- [ ] 安全要求有明确的认证/授权规则
- [ ] 数据迁移有明确的校验规则
- [ ] 灰度发布有明确的流量规则

### 评审结论模板
```
## 需求评审报告
- 需求ID/名称：
- 评审日期：
- 参评人员：产品/开发/测试/设计/运营
- 问题总数：N
  - 阻塞（Blocker）：M个（需解决才能开发）
  - 严重（Major）：P个（需解决才能测试）
  - 一般（Minor）：Q个（可排期解决）
  - 建议（Suggestion）：R个
- 评审结论：[通过/有条件通过/不通过]
```
