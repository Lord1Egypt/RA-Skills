# Decomposition：任务分解

## 1. Overview（概述）

复杂需求不拆 = 估计偏差 + 执行混乱。本技能将需求按依赖关系拆解为无环的子任务 DAG，每个子任务有明确的 Done 标准和可验证的输出。

**核心原则**：
- 粒度 = 一个子任务只改一个关注点（通常 ≤1 个文件）
- 依赖无环（DAG）— 子任务间不能有循环依赖
- 关键路径优先 — 阻塞其他任务的先做

## 2. When to Use（使用条件）

**触发条件**：
- Step 2 分解阶段自动注入
- C 级任务强制执行
- B 级任务简化版（仅做依赖分析 + 关键路径）
- 需求涉及 ≥ 2 个可独立交付的子功能

**可跳过**：
- A 级任务（micro fix，不需要分解）
- 单一函数/文件的修改

## 3. Process（执行流程）

### 3.1 分解四步法

```
步骤 1: 功能点枚举
  ├─ 从 Step 1 的范围声明中提取所有功能点
  ├─ 每个功能点问：能不能独立交付？不能 → 继续拆
  └─ 输出：功能点列表（无遗漏、无重复）

步骤 2: 依赖分析
  ├─ 绘制依赖矩阵：功能 A → 需要 功能 B 先完成？
  ├─ 检查循环依赖：A 依赖 B → B 依赖 C → C 依赖 A？→ 重新设计
  ├─ 标识关键路径：从起点到终点的最长依赖链
  └─ 输出：DAG 图（节点 = 子任务，边 = 依赖）

步骤 3: 粒度检查
  ├─ 每个子任务 ≤ 1 个文件的改动 → ✅ 粒度合适
  ├─ 每个子任务 > 3 个文件 → 🟡 可能还需拆分
  ├─ 每个子任务 > 5 个文件 → 🔴 必须拆更细
  └─ ❌ 反例："写所有 Model"（太粗）vs "写 User Model + 测试"（刚好）

步骤 4: Done 标准定义
  ├─ 每个子任务必须有可验证的 Done 标准
  ├─ Done 标准三要素：「什么完成」+「怎么验证」+「交付什么」
  │   例：✅ "User 认证 API 完成：/login 返回 JWT，/register 创建用户"
  │   ❌ "写完认证功能"
  └─ 输出：子任务清单 + Done 标准
```

### 3.2 输出格式

```yaml
# decomposition-output.yaml
tasks:
  - id: T1
    description: "创建 User Model 和迁移脚本"
    files: ["models/user.py", "migrations/001_user.py"]
    dependencies: []
    done_criteria: "User 表可创建，字段包含 id/email/password_hash"
    critical_path: true
    estimated_complexity: "B"

  - id: T2
    description: "实现 POST /register 端点"
    files: ["api/auth.py"]
    dependencies: ["T1"]
    done_criteria: "POST /register 接收 email+password，返回 201 + user_id"
    critical_path: true
    estimated_complexity: "B"
```

## 4. Risk Scorecard（反借口表）

| # | 纪律 | 常见借口 | 反驳 |
|---|------|---------|------|
| P1 | 每子任务 ≤1 文件 | "这些 Model 都差不多，一起写效率高" | 粒度粗 → 估计偏差 → 中间出错全白费 |
| P2 | 无循环依赖 | "它们互相需要，循环依赖没办法" | 有循环 = 设计问题 = 应该提取共享接口 |
| P3 | 每个子任务有 Done 标准 | "写完就知道了" | 不知道 Done 长什么样 = 不知道何时停 |
| P4 | 关键路径优先 | "哪个简单先做哪个" | 关键路径任务延期 = 全部延期 |
| P5 | 非关键路径可并行 | "我先全做完再让别人介入" | 可并行不并行 = 浪费等待时间 |

## 5. Red Flags（危险信号）

- 🚩 任何子任务涉及 > 5 个文件 — 需要拆得更细
- 🚩 DAG 中有循环依赖 — 重新设计接口
- 🚩 Done 标准含"完成"一词（"完成认证"）— 不可验证
- 🚩 关键路径 > 3 个子任务串行 — 关注单点阻塞风险
- 🚩 子任务编号不连续或跳跃 — 有意外的遗漏
- 🚩 预估总子任务 < 功能点数量 — 有需求被遗漏

## 6. Verification（自检清单）

**必检（≤5 项）**：
- [ ] 任务粒度合理：每个子任务的核心文件 ≤ 3 个（含测试 ≤ 5 个）
- [ ] 依赖关系无环（DAG 合法，遍历检查无回边）
- [ ] 每个子任务有明确的 Done 标准（可验证、无歧义）
- [ ] 关键路径已标识且 ≥ 1 个

**抽检（≤3 项）**：
- [ ] 子任务编号连续、可追溯（T1, T2, ... 无跳跃）
- [ ] 预估工作量合理（≤ 函数数 × 2 个子任务）
- [ ] 可并行的非关键路径任务已标注

**免检**：编号格式、YAML 缩进风格、预估时间精度
