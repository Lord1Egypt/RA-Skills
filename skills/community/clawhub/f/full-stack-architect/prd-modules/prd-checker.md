# PRD校验模块

## 1. 概述

PRD校验模块负责检查生成的PRD文档是否符合规范，确保需求理解准确，故事粒度合理，验收标准可验证，依赖关系明确。

## 2. 校验标准

### 2.1 基本结构检查

1. **文档结构**：PRD是否包含所有必要部分（项目概述、技术栈、用户故事、非目标、成功指标）
2. **格式规范**：是否符合标准Markdown格式
3. **内容完整性**：每个部分是否填写完整，无空白或占位符

### 2.2 用户故事检查

1. **故事粒度**：
   - 每个故事是否足够小，能在一个上下文窗口内完成
   - 故事描述是否清晰简洁，不超过200字
   - 是否避免了过于复杂的多功能故事

2. **验收标准**：
   - 每个故事是否至少有2个验收标准
   - 验收标准是否可验证、可测试
   - 是否避免了模糊的描述（如"界面美观"、"用户体验良好"等）

3. **依赖关系**：
   - 是否明确记录了故事间的依赖关系
   - 依赖关系是否合理，避免循环依赖
   - 依赖的故事是否存在

4. **优先级**：
   - 是否为每个故事指定了优先级（高/中/低）
   - 优先级分配是否合理

5. **故事类型**：
   - 是否为每个故事指定了类型（scaffold/backend/frontend/fullstack）
   - 类型分配是否合理

### 2.3 技术栈检查

1. **技术栈完整性**：是否指定了前端、后端、数据库等技术栈
2. **技术栈可行性**：技术栈选择是否合理、可行
3. **技术栈一致性**：技术栈之间是否相互兼容

### 2.4 其他检查

1. **非目标明确**：是否明确列出了项目的非目标
2. **成功指标可衡量**：成功指标是否可衡量、可追踪
3. **语言表达**：是否使用清晰、准确的语言表达

## 3. 校验流程

1. **接收PRD**：从PRD生成模块接收PRD文档
2. **结构检查**：检查PRD的基本结构是否完整
3. **故事检查**：逐个检查用户故事的各个方面
4. **技术栈检查**：检查技术栈的选择和描述
5. **综合评估**：综合评估PRD的质量和完整性
6. **生成反馈**：生成详细的校验反馈报告
7. **返回结果**：将校验结果返回给PRD生成模块

## 4. 校验反馈格式

### 4.1 反馈报告结构

```markdown
# PRD校验反馈报告

## 1. 总体评估
- **状态**：[通过/需要修正/拒绝]
- **评分**：[0-100分]
- **主要问题**：[主要问题摘要]

## 2. 详细反馈

### 2.1 结构问题
- [问题1]
- [问题2]

### 2.2 故事问题

#### 故事1：[故事标题]
- [问题1]
- [问题2]

#### 故事2：[故事标题]
- [问题1]
- [问题2]

### 2.3 技术栈问题
- [问题1]
- [问题2]

### 2.4 其他问题
- [问题1]
- [问题2]

## 3. 修正建议
- [建议1]
- [建议2]
- [建议3]
```

### 4.2 校验状态定义

- **通过**：PRD符合所有校验标准，无需修改
- **需要修正**：PRD基本符合要求，但存在一些需要修正的问题
- **拒绝**：PRD存在严重问题，需要重新生成

## 5. 最佳实践

1. **严格标准**：坚持严格的校验标准，确保PRD质量
2. **具体反馈**：提供具体、可操作的反馈，避免模糊的建议
3. **迭代优化**：通过多轮校验和修正，不断优化PRD质量
4. **知识积累**：将常见问题和修正方案记录下来，形成校验知识库
5. **持续改进**：根据实际项目经验，不断完善校验标准和流程

## 6. 与其他模块的交互

### 6.1 与PRD生成模块的交互
- 接收PRD生成模块的输出
- 生成校验反馈并返回给PRD生成模块
- 支持多轮迭代修正

### 6.2 与故事执行模块的交互
- 向故事执行模块提供经过校验的PRD
- 确保执行模块接收到的PRD符合规范
- 提供必要的上下文信息

## 7. 自动化校验

为了提高效率，可以考虑实现自动化校验功能：

1. **结构校验**：自动检查PRD的基本结构
2. **故事粒度分析**：自动分析故事的复杂度和粒度
3. **依赖关系检查**：自动检测循环依赖和不合理依赖
4. **标准检查**：自动检查验收标准的可验证性

通过自动化校验，可以快速识别PRD中的问题，提高校验效率和准确性。
