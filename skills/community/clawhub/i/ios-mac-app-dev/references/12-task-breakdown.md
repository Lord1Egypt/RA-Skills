# 任务拆解工作流

> 基于 matt pocock `to-issues` 改写，将 PRD/需求拆解为可独立开发的垂直切片任务。

## 使用时机

- PRD 已生成，需要拆解为具体开发任务
- 用户说「拆任务」、「分 Issue」、「怎么排期」

## 核心原则：垂直切片（Vertical Slices）

**❌ 错误：水平切片（按层拆解）**
```
Issue 1: 数据模型
Issue 2: API 服务层
Issue 3: ViewModel
Issue 4: UI 层
```
→ 每个 Issue 都无法独立验证，必须全部完成才能看到效果。

**✅ 正确：垂直切片（端到端路径）**
```
Issue 1: 用户能查看笔记列表（空状态 → 列表展示）
Issue 2: 用户能创建新笔记（输入框 → 持久化 → 列表刷新）
Issue 3: 用户能编辑笔记（点击 → 编辑页 → 保存 → 列表更新）
```
→ 每个 Issue 都是一条完整可验证的路径。

## 流程

### 1. 收集上下文

- 读取当前对话上下文中的 PRD 或需求
- 如用户传入 Issue 编号/URL，先从 Issue Tracker 拉取完整内容

### 2. 探索代码库（可选）

- 如尚未探索代码库，现在做
- Issue 标题和描述使用项目的领域术语表词汇
- 尊重已有 ADR

寻找可以预先重构的机会：**「让变更变简单，然后做简单的变更」**

### 3. 起草垂直切片

每条切片的规则：

```
✅ 每条切片交付一条狭窄但完整的端到端路径（数据层 → 服务层 → UI 层）
✅ 完成的切片可演示或可验证
✅ 预先重构应先行完成
```

### 4. 向用户展示并确认

以编号列表展示拆解方案，每条显示：

```
1. [切片标题]
   阻塞于：无 / Issue #42
   覆盖用户故事：#1, #3

2. [切片标题]
   阻塞于：Issue #41
   覆盖用户故事：#2
```

向用户确认：
- 粒度是否合理？（太粗 / 太细）
- 依赖关系是否正确？
- 是否需要合并或进一步拆分？

迭代到用户批准。

### 5. 发布到 Issue Tracker

按依赖顺序（阻塞者优先）发布每条切片到 Issue Tracker。

Issue 模板：

```
## 父 Issue
[如来源是已有 Issue，引用它]

## 构建内容
此垂直切片的简要描述。描述端到端行为，不是逐层实现。

## 验收标准
- [ ] 标准1
- [ ] 标准2
- [ ] 标准3

## 阻塞于
- [引用阻塞 Ticket] 或 「无 — 可立即开始」
```

发布时打上正确的 triage 标签（如 `ready-for-development`）。
