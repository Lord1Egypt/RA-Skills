# iOS Bug 诊断工作流

> 基于 matt pocock `diagnosing-bugs` 改写，适配 iOS/macOS 开发场景。

## 适用场景

- 用户说「诊断一下」、「这个 bug 帮我查一下」、「为什么崩溃」
- 报告功能异常 / 崩溃 / 性能回退

## 六阶段诊断法

---

## 阶段 1：建立反馈循环（最关键）

**这是整个技能的核心。** 如果你有一个对这个 bug **紧密的** 通过/失败信号——一个在这个 bug 上变红的信号——你**会**找到原因。

在此阶段投入不成比例的精力。**要积极。要有创造力。拒绝放弃。**

### 建立反馈循环的方法（按优先级尝试）

| # | 方法 | iOS 场景示例 |
|---|------|--------------|
| 1 | **失败测试** | 写一个 XCTest 复现 bug |
| 2 | **curl/HTTP 脚本** | 对有网络请求的功能，用 curl 重放 |
| 3 | **CLI 调用** | 用固定输入调用 CLI 工具，diff 输出 |
| 4 | **无头浏览器脚本** | 对 WebView 内容，用 Playwright |
| 5 | **重放捕获的 trace** | 保存网络请求/负载到磁盘，隔离重放 |
| 6 | **一次性 harness** | 起一个最小系统子集，单函数调用触发 bug |
| 7 | **Property/Fuzz 循环** | bug 是「有时输出错」，跑 1000 个随机输入 |
| 8 | **二分 harness** | bug 在两个已知状态间出现，自动化 `git bisect run` |
| 9 | **差分循环** | 同一输入跑旧版本 vs 新版本，diff 输出 |
| 10 | **HITL bash 脚本** | 最后手段，人类点击时用脚本驱动循环 |

### 收紧循环

把循环当产品对待。一旦有了**一个**循环，**收紧**它：
- 能让它更快吗？（缓存 setup、跳过无关初始化、缩小测试范围）
- 能让信号更清晰吗？（断言在具体症状上，不是「没崩溃」）
- 能让它更确定吗？（固定时间、seed RNG、隔离文件系统、冻结网络）

一个 30 秒的 flaky 循环几乎等于没有；一个 2 秒的确定性循环是超能力。

### 非确定性 bug

目标不是干净复现，而是**更高的复现率**。循环触发 100×，并行化，加压力，缩小时间窗口，注入 sleep。50% 复现率的 bug 可调试；1% 的不行。

### 完成标准

阶段 1 完成的标志：**紧密且能变红的循环**：
- [ ] **能变红** — 驱动实际的 bug 代码路径并断言**用户描述的确切症状**
- [ ] **确定性** — 每次运行结果相同（flaky bug：固定在高复现率）
- [ ] **快速** — 秒级，不是分钟级
- [ ] **可无人运行** — 你可以不插手就跑它

如果发现自己在没有这个命令的情况下读代码来建立理论，**停下来**——这就是此技能防止的典型失败模式。

---

## 阶段 2：复现 + 最小化

跑循环。看到它变红——bug 出现了。

确认：
- [ ] 循环产生的是**用户**描述的失败模式——不是碰巧在附近的不同失败
- [ ] 失败在多次运行中可复现
- [ ] 已捕获确切症状（错误信息、错误输出、慢的计时）

### 最小化

一旦变红，把复现缩小到**仍然变红的最小场景**。逐个削减输入、调用者、配置、数据和步骤，每次削减后重新跑循环——只保留对失败有负载的部分。

为何费这个劲：最小复现缩小了阶段 3 的假设空间，并成为阶段 5 的干净回归测试。

完成标志：**每个剩余元素都是负载 bearing 的**——再移除任何一个就会让循环变绿。

---

## 阶段 3：提出假设

在测试任何假设之前，生成 **3-5 个排名的假设**。单假设生成会锚定在第一个看似合理的想法上。

每个假设必须是**可证伪的**：陈述它做出的预测。

> 格式：「如果 <X> 是原因，那么 <改变 Y> 会让 bug 消失 / <改变 Z> 会让它更严重。」

如果你无法陈述预测，假设就是感觉——丢弃或 sharpen 它。

**在测试前把排名列表展示给用户。** 他们通常有领域知识能瞬间重新排名。

---

## 阶段 4：插桩

每个探针必须映射到阶段 3 的具体预测。**一次改变一个变量。**

工具优先级：
1. **调试器 / LLDB** — 如果环境支持。一个断点胜过十个日志。
2. **针对性日志** — 在区分假设的边界处。
3. 永远不要「记录一切然后 grep」。

**给每个调试日志打唯一前缀**，如 `[DEBUG-a4f2]`。结束时 grep 掉。不打标签的日志会存活下来。

**性能分支：** 对性能回退，日志通常是错误的。改为：建立基准测量（计时 harness、Instruments、查询计划），然后二分。先测量，后修复。

---

## 阶段 5：修复 + 回归测试

**在修复之前写回归测试**——但仅在存在**正确接缝**的情况下。

正确接缝是测试在调用点发生的**真实 bug 模式**的接缝。如果唯一可用接缝太浅（单调用者测试，但 bug 需要多个调用者），那里的回归测试给虚假信心。

**如果没有正确接缝，这本身就是发现。** 记录下来。代码库架构正在阻止 bug 被锁定。标记给下一阶段。

如果有正确接缝：
1. 把最小复现变成该接缝处的失败测试
2. 看着它失败
3. 应用修复
4. 看着它通过
5. 对原始（未最小化的）场景重跑阶段 1 循环

---

## 阶段 6：清理 + 事后分析

在宣布完成前必须完成：
- [ ] 原始复现不再复现（重跑阶段 1 循环）
- [ ] 回归测试通过（或接缝缺失已记录）
- [ ] 所有 `[DEBUG-...]` 插桩已移除（grep 前缀）
- [ ] 一次性原型已删除
- [ ] 正确假设已在 commit/PR 消息中陈述

**然后问：什么能防止这个 bug？** 如果答案涉及架构变更（没有好测试接缝、调用者耦合、隐藏耦合），交给 `/improve-codebase-architecture` 技能，带上具体信息。**在修复之后提建议**——你现在比开始时信息更多。
