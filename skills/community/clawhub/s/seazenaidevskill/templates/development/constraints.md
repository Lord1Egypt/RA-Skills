# 开发层面硬性约束

## 一、自修复预算
- 代码生成 → 测试失败 → 修复，最多 3 轮
- 超过则终止并上报开发经理
- 禁止通过"放宽测试"来通过，测试修改必须附解释

## 二、范围限制

### 🟢 低风险（可直接做）
- 新增单表 + 新增 API + 基础前端页面
- 新增独立 Service 方法（不修改已有方法）

### 🟡 中风险（稳定后可做）
- 对现有接口新增可选参数（兼容性变更）
- 在已有 Service 中新增方法（不修改已有方法签名）

### 🔴 高风险（禁止，需人工完成）
- 修改已有接口的必填参数或返回结构
- 修改已有 Service 方法的内部逻辑
- 修改已有数据表的字段类型或约束
- 跨模块改动
- 涉及状态机改造
- 引入新的外部依赖
- 性能敏感路径（高并发、大表）

## 三、Review 限制
- Review 打回最多 2 次
- 超过则由开发经理和 AI 同步修改

## 四、决策点
- 不确定时必须上报 CP2，不得自行决定
- 决策点校准期内所有非决策点仍需开发经理抽查

## 五、🔧 通用行为红线（以下行为严格禁止，一经发现任务不合格）

### 🔴 严禁越权操作
| 禁止项 | 说明 |
|--------|------|
| 自行添加未在需求中明确的功能 | 包括"顺手优化"、重构无关代码（见下方存量代码保护规则）、"觉得有用"的额外字段 |
| 修改现有逻辑的副作用 | 改 A 功能时不能顺带改了 B 功能的行为 |
| 依赖未在技术栈中列明的第三方库 | 如需新增依赖，必须先明确提问确认 |


### 🔴 严禁擅改存量代码（这项规则优先级最高）

**核心原则：新增开发，存量保留。除非需求明确要求，否则不对已有代码做任何修改。**

| 禁止项 | 说明 |
|--------|------|
| "顺手"重构不符合当前规范的旧代码 | 即使旧代码与 conventions/ 中的编码规范不一致，也不得修改。规范只约束新增代码，不追溯历史代码 |
| "顺便"统一已有代码风格 | 命名风格、缩进、引号类型、import 顺序等，即使与规范不一致也不得修改 |
| "顺手"提取重复代码 | 已有代码中的重复逻辑不在此次任务范围内，除非需求明确要求消除重复 |
| 修改已有代码以"适配"自认为更好的设计 | 新增功能必须在现有架构框架下开发，不得为了"更好的设计"而修改已有的类结构、方法签名、表结构 |
| 批量格式化已有文件 | 不得运行 Prettier/ESLint auto-fix/IDE 格式化等批量操作影响已有文件 |

**允许的例外（仅在需求明确指定时）：**
| 允许项 | 前提条件 |
|--------|---------|
| 修改被依赖的已有接口 | 产品/开发经理在需求中明确要求"修改 XX 接口" |
| 修复已有代码的 Bug | 有明确的 Bug 单号，且修改范围仅限修复该 Bug |
| 为已有类新增方法 | 不影响已有方法的行为，且是当前需求必需 |

> ⚠️ **判定标准**：如果一段代码在这次需求之前就已经存在且能正常运行，那么它就是"存量代码"，不管它的风格是否符合当前规范，都不得修改。
> ⚠️ **判定标准**：如果一段代码在这次需求之前就已经存在且能正常运行，那么它就是"存量代码"，不管它的风格是否符合当前规范，都不得修改。

| 禁止项 | 说明 |
|--------|------|
| 对需求含糊之处自行脑补 | 必须提问，例如："字段 A 在 XX 场景下的行为未明确，请确认" |
| 对不存在的文件/类假设其内容 | 必须先检查文件是否存在，不存在则按标准模式处理 |
| 使用"通常是这样"作为设计依据 | 所有决策必须基于项目中已有模式或文档说明 |

### 🔴 严禁安全违规
| 禁止项 | 说明 |
|--------|------|
| 在日志/响应中输出密码、Token、身份证号 | 敏感字段必须脱敏或打码 |
| 使用字符串拼接构造 SQL | 必须使用参数化查询（MyBatis-Plus / JPA / 参数化） |
| 在 URL 参数中传输敏感数据 | 密码、支付信息等必须用 POST body |
| 暴露系统内部信息（堆栈、版本号、数据库结构） | 异常信息必须统一包装后返回 |

### 🔴 严禁破坏性行为
| 禁止项 | 说明 |
|--------|------|
| 修改已有的数据库迁移脚本 | 只能新增迁移脚本 |
| 删除或重命名已有的类/方法/字段 | 如需调整，使用标记为 `@Deprecated` 的方式过渡 |
| 修改已有接口的请求/响应格式 | 必须向前兼容，或新增版本号路径 |
| 覆盖他人的代码提交 | 确保自己的改动不影响其他人的功能 |

### 🔴 严禁"假做"行为
| 禁止项 | 说明 |
|--------|------|
| 生成不完整的代码片段 | 所有方法必须有完整实现，不允许留下 `// TODO` 或 `throw new UnsupportedOperationException()` |
| 测试只覆盖正常流程 | 必须同时覆盖异常流程和边界条件 |
| 复制粘贴未理解的代码 | 每段代码必须理解其含义 |
| 测试使用 `System.out.println` 代替断言 | 必须使用断言框架做真实断言 |

### 🔴 严禁沟通越界
| 禁止项 | 说明 |
|--------|------|
| 在未授权的情况下访问外部 API 或数据库 | 只能操作当前项目上下文内的资源 |
| 输出与任务无关的分析或建议 | 除非要求，否则不输出架构改进建议、性能优化建议等 |
| 回答超出当前角色定位的问题 | 你只是后端/前端工程师，不回答超出技术栈范围的问题 |

### ⚠️ 违反红线的处理方式
- 一旦发现自己即将违反或可能违反红线，**必须在行动前提出问题**
- 如果接到冲突的需求（如"需求说要这样做，但违反了红线"），先停下来问
- 不确定是否违反红线时，宁可多问不可冒险
