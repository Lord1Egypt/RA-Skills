# 生成工作流

从用户输入"帮我做一个XX领域知识参考库"到完整技能产出的标准化操作流程。

---

## 总览

```
输入: 用户描述的领域名称
  │
  ├─ Step 1: 领域分析（30-40%时间）
  │   ├─ 1A: 领域定义与分类定位
  │   ├─ 1B: 域划分
  │   └─ 1C: 任务类型枚举 + 依赖推导
  │
  ├─ Step 2: 文件生成（40-50%时间）
  │   ├─ 2A: SKILL.md（用模板填充）
  │   ├─ 2B: references/catalog.md（任务清单+拓扑）
  │   ├─ 2C: references/requirements.md（槽位定义）
  │   └─ 2D: references/exemplars.md（范本框架）
  │
  ├─ Step 3: UTOS校验（10-15%时间）
  │   └─ 21项逐条检查 → 修正FAIL项
  │
  └─ 输出: 完整可用的领域负载物技能目录
```

---

## Step 1：领域分析

### 1A：领域定义与分类定位

**操作**：
1. 请用户描述目标领域的核心工作内容
2. 使用 `domain-analysis-framework.md` 的 **Step A** 进行规范化
3. 输出：领域英文名、域名代号前缀、R1-R5定位结果

**输出物**：

```
领域名: {English Name}
代号前缀: {X}
R1(信息密度): 高/中/低 → S/C权重: ...
R2(创造性): 高/中/低 → A权重: ...
R3(交互性): ... → I权重: ...
R4(规范性): ... → G权重: ...
R5(迭代性): ... → 循环模式使用频率: ...
```

### 1B：域划分

**操作**：
1. 根据领域特征选择拆解方法（价值链型 vs 职能分工型 vs 混合型）
2. 使用 `domain-analysis-framework.md` 的 **Step B** 进行域划分
3. 确定域数量和每个域的范围边界

**输出物**：

```
共 N 个域:
{X}1 {域名1} — {一句话说明范围}
{X}2 {域名2}
...
{X}N {域名N}

域间逻辑流: {X}1 → {X}2 → ... → {X}N
```

### 1C：任务类型枚举与依赖推导

**操作**：
1. 对每个域，使用 **Step C** 的问题清单枚举任务
2. 为每个任务分配ID（{X}{域号}-{序号}）和UTOS映射提示
3. 使用 **Step D** 构建依赖拓扑

**输出物**：

```
总任务数: M 种
各域任务分布: {X}1(n1种) / {X}2(n2种) / ...

关键跨域管线:
- {管线名1}: {ID}→{ID}→{ID}
- {管线名2}: ...
（共 K 条）
```

## Step 2：文件生成

### 2A：SKILL.md

**操作**：
1. 打开 `structure-template.md` 中的 SKILL.md 模板
2. 逐一替换所有 `{变量}` 为 Step 1 的分析结果
3. 特别注意以下字段必须**定制化**而非照搬模板：
   - "领域特有维度"章节 —— 必须是该领域独有的
   - "与Task Ops的接口" Step 1 领域校准 —— 必须有具体参数
   - "域概览"表格 —— 必须是实际划分的结果

### 2B：Catalog文件

**操作**：
1. 按1C的枚举结果，为每个域创建表格
2. 每个任务一行，格式严格遵循模板
3. 文件末尾追加"依赖拓扑摘要"章节
4. 至少列出3条跨域管线链路

**质量检查**：
- [ ] 所有任务ID格式正确
- [ ] 所有依赖引用有效
- [ ] UTOS映射提示使用了正确代号
- [ ] 域间逻辑流声明清晰

### 2C：Requirements文件

**操作**：
1. 对每个任务创建槽位定义
2. **必填**：必选组件 + 组装顺序 + 格式
3. **推荐填写**：可选组件 + 约束字段
4. 当任务数>30时，可将部分域的要求迁移至 `references/structure-requirements-{domain}.md` 子文件

**槽位编写原则**：
- 必选组件回答"这个产出物的最小必要元素是什么？"
- 组装顺序回答"按什么逻辑顺序填充这些元素？"
- 约束回答"这个领域对这个产出的特殊限制是什么？"

### 2D：Exemplars文件

**操作**：
1. 创建范本清单主文件（按域分组，每个任务一行，用任务ID索引）
2. 创建范本标准格式模板（子文件模板）
3. 如果有现成可用范本，存入 `references/exemplars/` 子目录并在主文件注册
4. 否则创建 `[待填充]` 占位符框架

**范本策略选择**：

| 领域类型 | 范本策略 | 说明 |
|---------|---------|------|
| 代码产出类 | 范本=可运行代码 | 如单文件技能，嵌入完整.html/.py/.sh/.sql |
| 文档产出类 | 范本=结构化文档模板 | 如医药/代账，嵌入脱敏范例 |
| 创作产出类 | 范本=技法标注片段 | 如网文，嵌入经典片段+拆解 |
| 流程产出类 | 范本=流程示例 | 如硬件，嵌入脱敏项目案例 |

## Step 3：UTOS校验

### 执行校验

1. 打开 `utos-interface-checklist.md`
2. 逐项执行 A→B→C→D→E 五类检查
3. 记录每项结果（PASS/FAIL/N/A）

### 处理FAIL项

| FAIL类别 | 处理策略 |
|---------|---------|
| A类(结构) | 通常是因为模板复制遗漏——全局搜索替换修正 |
| B类(Schema) | 缺失字段或无效值——补充/修正 |
| C类(槽位) | 覆盖率不足——至少为核心域补全 |
| D类(接口) | 照搬无定制——重写该领域特定内容 |
| E类(范本) | 未创建——建立空框架 |

### 校验通过标志

当满足以下条件时，校验通过：
- A类: 6/6 PASS
- B类: 5/5 PASS
- C类: ≥3/4 PASS
- D类: 3/3 PASS
- E类: ≥2/3 PASS

### 内容质量评估（结构校验通过后）

结构校验只验证"有没有"，以下维度评估"好不好"：

| 维度 | 评估方法 | 合格标准 |
|------|---------|---------|
| **任务覆盖度** | 核对领域核心工作流是否都被catalog中的任务覆盖 | 核心工作流无遗漏，可接受少量边缘任务缺失 |
| **域间边界清晰度** | 检查各域的任务是否有重叠或归属模糊 | 无同一任务出现在两个域中，域间边界互斥 |
| **依赖拓扑连通性** | 从入口任务出发，能否沿依赖链到达所有非入口任务 | 无孤立子图，无循环依赖 |
| **要求可操作性** | requirements中的必选组件和组装顺序是否具体到可直接执行 | 不出现"待定""其他"等模糊项 |
| **范本可用性** | 状态为"可用"的范本是否确实可作为样本法参考 | 范本内容完整，四维特征摘要存在 |

## Step 4：输出与交付

### 目录结构

生成的技能应具有以下目录结构：

```
{skill-name}/
├── SKILL.md                              # 技能定义
└── references/
    ├── {catalog-filename}.md              # 第一层：清单+拓扑
    ├── {requirements-filename}.md         # 第二层：槽位定义
    ├── exemplars.md                       # 第三层：范本清单主文件
    └── exemplars/                         # 第三层：具体范本子文件
        ├── {X}-01-{task-name}.md
        ├── {X}-01-{task-name}.summary.md
        └── ...
```

### 交付检查

在交付给用户前，最终确认：

- [ ] 所有文件存在且非空（SKILL.md + references/ 下的 catalog、requirements、exemplars 主文件）
- [ ] SKILL.md可在WorkBuddy中被识别和加载
- [ ] references/下的文件路径在SKILL.md中引用正确
- [ ] 无遗留的 `{变量}` 占位符
- [ ] 通过UTOS接口21项校验

### 后续维护与迭代

**首次交付后**：
1. 范本库(exemplars.md)中的`[待填充]`占位符需要后续补充真实范本才能发挥样本法的完整能力
2. 结构要求(requirements.md)中的`[待补充]`约束字段需要根据实际需求定义
3. 技能生成后可通过实际使用持续迭代完善

**领域演进时的增量更新**：
1. 重新跑 `domain-analysis-framework.md` 的 Step A-B，确认域划分是否需要调整
2. 对比新旧 catalog 的 diff：哪些任务新增、删除、合并、拆分
3. 仅更新变化的域和任务，不全量重建——保持已有范本和要求不丢失
4. 更新后重新跑 UTOS 接口校验（21项）
5. 版本记录：在 SKILL.md 末尾记录变更摘要

**范本扩充流程**：
1. 从实际使用中收集高质量产出作为候选范本
2. 脱敏后存入 `references/exemplars/` 目录
3. 在 exemplars.md 主文件中更新状态（空→可用）
4. 为每个新范本撰写四维特征摘要（结构/风格/逻辑/格式）

## 快速生成速查卡

对于熟练用户，以下是快速生成的心智模型：

```
听到新领域 → 
  ① 它的价值链是什么？（拆成5-12个域）→ 
  ② 每个域要产出什么？（列成3-8个任务/域）→ 
  ③ 任务间谁先谁后？（画依赖箭头）→ 
  ④ 每个任务的"零件清单"是什么？（写必选组件+顺序）→ 
  ⑤ 有没有好的例子可以放进去？（收集或标记待填充）→ 
  ⑥ 和UTOS对得上吗？（跑21项校验清单）
```
