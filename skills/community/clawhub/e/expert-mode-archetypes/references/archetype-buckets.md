# Archetype Buckets v0.2.0-draft

Expert Mode uses `archetype` deliberately. An archetype is broader than a job title. It is a reusable lens of expertise, behaviour, incentives, taste, risk perception, and field wisdom.

Use archetypes when a project needs judgement from a perspective that may not map cleanly to one profession.

## Core idea

An expert archetype can represent:

- a profession
- a domain
- a stakeholder
- a decision style
- a lifecycle stage
- a risk posture
- a craft tradition
- an institution
- an adversary
- a translator between worlds
- a taste or quality standard
- a lived operational position

This lets Expert Mode build richer panels than “ten job titles”.

## Bucket 1: Professional role archetypes

These are recognizable jobs or functions.

Examples:

- Software architect
- Privacy specialist
- Textile merchandiser
- UX researcher
- Support lead
- Incident commander
- Accessibility specialist
- Technical editor

Use when the project needs conventional specialist competence.

## Bucket 2: Domain archetypes

These represent deep familiarity with a field, industry, scene, or operating environment.

Examples:

- Veteran fabric wholesaler
- Rural healthcare operator
- Enterprise procurement insider
- Indie game economy designer
- Community radio station operator
- Local council grants officer
- Small warehouse inventory manager

Use when field-specific judgement matters more than generic professional advice.

## Bucket 3: Stakeholder archetypes

These represent people affected by the project.

Examples:

- First-time confused user
- Power user
- Budget-constrained buyer
- Internal admin who does the boring work
- Accessibility-dependent user
- Angry customer after a failed process
- Contractor with incomplete context
- Executive sponsor who only has five minutes

Use when the project needs empathy, adoption, workflow fit, or communication clarity.

## Bucket 4: Decision-making archetypes

These describe how a lens makes tradeoffs.

Examples:

- Optimizer
- Simplifier
- Risk minimizer
- Systems thinker
- Evidence purist
- Pragmatic shipper
- Maintainer
- Skeptic
- Customer advocate
- Cost controller

Use when the project is stuck because it needs a different judgement mode, not a different technical field.

## Bucket 5: Lifecycle archetypes

These are experts defined by project phase or organizational moment.

Examples:

- Zero-to-one founder
- Launch operator
- Turnaround specialist
- Maintenance engineer
- Migration lead
- Post-incident reviewer
- Scale-up operator
- Sunset/deprecation planner

Use when the same object requires different expertise at different stages.

## Bucket 6: Institutional archetypes

These represent how institutions see the work.

Examples:

- Regulator
- Auditor
- Standards-body contributor
- Procurement committee member
- Insurance assessor
- Grant assessor
- Board risk committee member
- School administrator

Use when the work must survive formal review, policy constraints, approvals, or institutional incentives.

## Bucket 7: Risk archetypes

These look for harm, failure, exposure, misuse, and brittle assumptions.

Examples:

- Security paranoiac
- Privacy guardian
- Legal realist
- Safety engineer
- Failure-mode hunter
- Abuse-case reviewer
- Reliability pessimist
- Compliance tripwire spotter

Use when the project has downside risk that normal builder energy might miss.

## Bucket 8: Adversarial archetypes

These model hostile, negligent, confused, or competitive behaviour.

Examples:

- Red teamer
- Scammer
- Bad-faith user
- Competitor
- Negligent operator
- Curious but careless user
- Data exfiltration attacker
- Troll/community disruptor

Use to test assumptions, guardrails, policies, incentives, and abuse paths.

## Bucket 9: Craft archetypes

These represent skilled practice, taste, and embodied standards.

Examples:

- Master copy editor
- Senior pattern maker
- Production foreman
- Clinical explainer
- Investigative researcher
- Museum cataloguer
- High-end buyer
- Minimalist craftsperson

Use when quality depends on feel, judgement, standards, tradition, or subtle errors.

## Bucket 10: Translation archetypes

These bridge audiences or domains.

Examples:

- Expert-to-novice explainer
- Policy-to-engineering translator
- Customer-language translator
- Executive briefer
- Technical-to-creative mediator
- Cross-cultural localization reviewer
- Field-operator-to-management interpreter

Use when the problem is not only knowing the answer, but making it legible to the right people.

## Bucket 11: Quality and taste archetypes

These judge whether the work feels good, premium, coherent, trustworthy, elegant, or culturally right.

Examples:

- Design critic
- Brand guardian
- Editorial taste-maker
- Luxury retail buyer
- Plain-language advocate
- Minimalist interface reviewer
- Community norm keeper

Use when correctness is not enough and the output must feel right to a specific audience.

## Bucket 12: Historical and field-wisdom archetypes

These represent memory of repeated patterns.

Examples:

- Old hand who has seen this fail before
- Maintainer of legacy systems
- Operator who knows the unofficial process
- Retired inspector
- Long-time community moderator
- Veteran bookkeeper
- Former helpdesk lead

Use when formal docs miss practical reality.

## Bucket 13: Constraint archetypes

These embody one dominant constraint.

Examples:

- Tiny-budget operator
- One-person maintenance team
- Low-bandwidth rural user
- Privacy-first buyer
- Time-poor parent
- Low-literacy user
- Offline-first field worker
- Legacy-system prisoner

Use when a design must be stress-tested against a harsh real-world limitation.

## Bucket 14: Evidence archetypes

These differ by what they trust.

Examples:

- Standards-and-specifications reader
- Empirical tester
- Practitioner case-study collector
- Quantitative analyst
- Qualitative interviewer
- Source-code reader
- Regulator-guidance reader

Use when a project needs clarity about what kind of evidence should guide the decision.

## How to combine buckets

A strong expert archetype often combines several buckets.

Examples:

- “Veteran fabric wholesaler” = domain + field wisdom + commercial operator.
- “Accessibility-dependent first-time user” = stakeholder + constraint + quality lens.
- “Security paranoiac” = risk + adversarial + evidence archetype.
- “Executive briefer” = translation + institutional + decision-making archetype.
- “Maintenance engineer for neglected legacy systems” = lifecycle + craft + field wisdom + constraint.

## Selection rule

When selecting up to 10 expert archetypes, pick for coverage, not quantity.

A healthy panel often includes:

1. One domain expert.
2. One builder/implementer.
3. One stakeholder or user archetype.
4. One risk or adversarial archetype.
5. One operator/maintainer.
6. One translator or communicator if adoption matters.
7. One quality/taste archetype if judgement matters.

Use fewer when the task is simple.

## Naming rule

Prefer specific, vivid, non-fake names for archetypes.

Good:

- “Veteran fabric wholesaler”
- “Privacy guardian”
- “One-person maintenance team”
- “Executive briefer”

Weaker:

- “Business expert”
- “Tech person”
- “User”
- “Reviewer”
