# 领域专属信源 skill 注册表

研究院维护一份**领域→信源 skill** 的内置注册表，作为冷启动时推荐的依据。**只有用户研究方向匹配时才推荐安装**，避免打扰其他领域的用户。

## 调用规范（以 aihot-skill 为例）

```
调用前提：
  · 用户研究方向匹配领域关键词
  · 用户已确认安装该 skill
  · 配置写入 lab-config.json 的 domainSpecificSkills

调用方式：
  · 严格按照该 skill 自身的 SKILL.md 协议（端点、UA、参数）
  · 如 aihot-skill 要求带浏览器 User-Agent，必须遵守
  · 遵守限流（如 600 req/min/IP）

调用时机：
  · 每日发现的步骤 2"联网采集"中，与 web_search 并行调用
  · 如有 since 参数，传入"上次采集时间"避免重复

结果处理：
  · 该 skill 返回的内容进入研究院的统一质检流程（内容质检+摘要质检+关联质检）
  · 不能直接采信该 skill 的"精选"结果，要按研究院自己的标准重新评分
  · 失败 → 静默降级，仅用 web_search

向下兼容：
  · 用户研究方向不匹配 → 注册表不推荐，研究院照常运转
  · 已安装但本次调用失败 → 自动降级，不影响其他流程
  · 注册表本身可由用户在记忆中扩展（添加自定义领域→skill映射）
```

## 询问话术（仅当领域匹配且未安装时）

```
"你的研究方向是 [AI 领域]，发现一个领域专属信源 skill：[skill名]
 它能提供 [简要说明能力，如：实时AI资讯、模型/产品/论文分类等]
 是否安装？安装后会作为每日发现的额外信源（不替代通用搜索）。
 不装也完全不影响研究院运转。"
```

## 关键原则

- 领域专属信源 skill 是**第一层采集的补充**，不是必装项
- 只在领域匹配时推荐，不匹配时绝不打扰用户
- 调用时遵循该 skill 自身的协议（端点/UA/限流），不破坏其规范
- 调用失败时自动降级回 web_search，研究院照常运转
- 该 skill 返回的内容必须过研究院的三层质检，不直接采信"精选"
- 命中率纳入信源调权统计

## 给社区开发者的扩展指引

如果你开发了某个垂直领域的信源 skill，希望被研究院识别，需提供：
1. SKILL.md 的访问地址
2. 领域关键词列表（用于匹配用户研究方向）
3. 调用接口规范（端点、参数、限流、UA要求）
4. 数据分类（便于研究院归类展示）

**研究院的角色：调度方+质量把关方，不替代信源 skill 自身的内容生产逻辑。**

## Skill 安全策略（涉及推荐安装 Skill 时）

```
准入门槛：
  · 下载量 ≥100 + 发布时间 ≥3个月
  · 有 GitHub 仓库且可审查源码

安装前检查：
  · 如有 Skill Vetter 工具 → 先扫描再安装
  · 审查 SKILL.md 声明的权限是否合理

三色评级：
  🟢 绿灯：权限合理、来源可信 → 推荐安装
  🟡 黄灯：权限较高、需审查 → 告知用户风险，由用户决定
  🔴 红灯：权限异常、来源不明 → 不推荐，明确告知风险

原则：院长绝不静默安装任何 Skill，必须告知用户做了什么、为什么需要。
```
