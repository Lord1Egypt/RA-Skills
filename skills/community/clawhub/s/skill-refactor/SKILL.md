# Skill Refactor — 技能改造方法

## 核心理念

**技能存在的根本问题**：不是"这个技能能不能优化"，而是"这个技能需不需要存在"。

传统思维是"技能存在→优化技能"，但更根本的问题是：这个技能是否本就不该存在？如果技能覆盖的领域是人的局限产物（历史遗留、重复功能、过时需求），那么消除技能比重构技能更彻底。

**改造十一步法**：
- **阶段一：领域消除评估**
  - 第一步：边界识别——识别技能覆盖的领域边界
  - 第二步：存在理由分析——追问技能存在的根本原因
  - 第三步：消除可行性评估——评估技能是否可以完全消除
  - 第四步：独立存在必要性判断——判断技能是否值得独立存在
  - 第五步：决策输出——消除/重构/保留
- **阶段二：内容重构**
  - 第六步：组成部分识别——识别技能的每个组成部分
  - 第七步：组成部分存在理由分析——逐部分追问存在理由
  - 第八步：补偿层消除——去掉传递/协调/格式环节
  - 第九步：重整——基于需求重编技能内容
  - 第十步：重构验证——10项验证清单
  - 第十一步：改造形态选择——选择最终形态

**适用范围**：任何已存在的技能——用户技能、项目技能、市场技能等。

**单独存在合理性**：
- **对象特定**：专门针对"技能"这个对象，技能有独特的结构（SKILL.md、范本、验证清单）
- **两阶段设计**：先判断"值不值得存在"，再执行"怎么重构"
- **技能特有规则**：技能的组成部分（YAML frontmatter、章节结构、范本）有其特殊性

## 改造十一步法

| 阶段 | 步骤 | 操作 | 要点 |
|------|------|------|------|
| **领域消除评估** | 1. 边界识别 | 识别技能覆盖的领域边界 | 明确技能包含什么、不包含什么 |
| | 2. 存在理由分析 | 追问技能存在的根本原因 | 事情本身需要 / 人的局限需要 / 历史遗留 |
| | 3. 消除可行性评估 | 评估技能是否可以完全消除 | 被其他技能吸收 / 功能分散 / 直接废弃 |
| | 4. 独立存在必要性判断 | 判断技能是否值得独立存在 | 边界清晰度 / 功能内聚性 / 消除成本 |
| | 5. 决策输出 | 给出明确的消除/重构/保留建议 | 消除/重构/保留三选一 |
| **内容重构** | 6. 组成部分识别 | 识别技能的每个组成部分 | 章节、范本、规则等 |
| | 7. 组成部分存在理由分析 | 逐部分追问存在理由 | 标记为核心/校准/传递/协调/校验/格式 |
| | 8. 补偿层消除 | 去掉不必要的组成部分 | 去掉传递/协调/格式，保留核心+校准 |
| | 9. 重整 | 基于需求重编技能内容 | 保留部分→重新编排为完整SKILL.md |
| | 10. 重构验证 | 10项验证清单 | 全部通过才算完成 |
| | 11. 改造形态选择 | 选择最终形态 | 消除/重构/保留 |

## 技能存在理由分类

| 类型 | 标记 | 说明 | 处理建议 |
|------|------|------|---------|
| 事情本身需要 | ✅必要 | 技能覆盖的领域有清晰的边界和内聚性 | 保留→重构技能内容 |
| 人的局限需要 | ❌可消除 | 技能覆盖的领域是因为人的认知局限、历史遗留 | 消除→删除技能 |
| 历史遗留 | ⚠️待评估 | 技能覆盖的领域是因为历史原因，当前必要性存疑 | 评估后决定 |
| 外部约束 | 🔒不可消除 | 技能覆盖的领域是因为法规、标准、合同等外部约束 | 保留→重构技能内容 |

---

## 技能组成部分类型

| 类型 | 标记 | 说明 | 处理建议 |
|------|------|------|---------|
| 核心环节 | ✅核心 | 事情本身的逻辑步骤 | 保留 |
| 校准环节 | 🔶校准 | 过程中纠偏的中间产出物 | 保留为基元内分步校准点 |
| 传递环节 | ❌传递 | 人之间传递信息的中间步骤 | 消除 |
| 协调环节 | ❌协调 | 管理多人协作的步骤 | 消除 |
| 校验环节 | ⚡校验 | 防止出错的审核步骤 | 精简为关键节点 |
| 格式环节 | ❌格式 | 满足组织流程的形式要求 | 消除 |

---

## 改造判断标准

满足任一即建议消除技能：

| 条件 | 阈值 | 示例 |
|------|------|------|
| 功能重叠度 | 技能功能≥50%与其他技能重叠 | "文档转换"与"格式处理"高度重叠 |
| 领域过时度 | 技能覆盖的领域已不再需要 | "传真发送"技能在现代办公中已过时 |
| 使用频率 | 技能近6个月使用次数<3 | 长期未使用的技能可能已无价值 |
| 维护成本 | 技能维护成本>收益 | 技能依赖的API已废弃，维护困难 |

## 改造后典型形态

| 形态 | 适用场景 | 执行方式 |
|------|---------|---------|
| **消除** | 技能领域不再需要、功能重叠、历史遗留 | 删除技能，功能合并到其他技能 |
| **重构** | 技能领域需要存在，但内容需要优化 | 重构技能结构、内容、范本 |
| **保留** | 技能已足够优化，无需改动 | 维持现状 |

**选择原则**：能消除的不重构，能重构的不保留。消除是最彻底的优化。

## 改造验证清单

改造完成后必须逐项验证，十项全部通过才算改造完成：

| # | 验证项 | 说明 |
|---|--------|------|
| 1 | ⬜ 领域评估准确 | 是否准确评估了技能领域的存在必要性 |
| 2 | ⬜ 消除可行性评估合理 | 是否合理评估了消除的可行性 |
| 3 | ⬜ 重构内容完整 | 如果重构，是否覆盖了技能的全部核心内容 |
| 4 | ⬜ 结构一致性 | 技能结构是否合理完整 |
| 5 | ⬜ 改造建议明确 | 是否给出了明确的消除/重构/保留建议 |
| 6 | ⬜ 领域清晰度 | 技能覆盖的领域边界是否清晰 |
| 7 | ⬜ 内容完整性 | 技能是否覆盖了领域的全部核心内容 |
| 8 | ⬜ 结构规范性 | 技能结构是否合理（YAML frontmatter 建议保留 name/author/description，章节结构可灵活调整） |
| 9 | ⬜ 可执行性 | 技能是否可直接使用，范本是否完整 |
| 10 | ⬜ 独立性 | 技能是否独立，不依赖其他技能 |

## 领域要求清单

### S0-01 边界识别

- **必选组件**: 技能名称、技能覆盖领域描述（包含什么、不包含什么）、与其他技能的交集列表、边界清晰度评估（清晰/模糊/高度重叠）
- **可选组件**: 技能创建时间、技能使用频率、技能维护成本
- **组装顺序**: 技能确认→领域梳理→交集识别→边界清晰度评估→数据汇总
- **约束**: 必须完整识别技能覆盖的领域，不可跳过"理所当然"的功能；边界清晰度必须基于客观标准，不可凭感觉
- **格式**: 技能领域边界图（Markdown表格+交集标注）

### S0-02 存在理由分析

- **必选组件**: 技能存在的根本原因（事情本身需要 / 人的局限需要 / 历史遗留 / 外部约束）、存在理由标记（✅必要 / ❌可消除 / ⚠️待评估 / 🔒不可消除）、标记理由
- **可选组件**: 存在理由细分（组织架构/协作需要/认知局限/历史惯性/法规要求）、消除动机评估
- **组装顺序**: 追问存在理由→根本原因判定→类型标记→理由记录→汇总统计
- **约束**: 每个技能必须追问"如果组织是完全扁平的、没有部门壁垒，这个技能还需要独立存在吗？"；判定必须基于事情本身逻辑，不可因"行业惯例"保留
- **格式**: 存在理由分析表（Markdown表格）

### S0-03 消除可行性评估

- **必选组件**: 消除方案（被其他技能吸收 / 功能分散到其他技能 / 直接废弃）、消除成本评估（低/中/高）、消除风险评估（低/中/高）、消除后功能覆盖度
- **可选组件**: 消除时间表、消除后需新增的协调机制、消除后需调整的依赖关系
- **组装顺序**: 消除方案设计→成本评估→风险评估→功能覆盖度验证→消除可行性判定
- **约束**: 消除方案必须具体到"功能A→技能X，功能B→技能Y"；成本评估必须基于实际数据，不可凭感觉；风险评估必须考虑最坏情况
- **格式**: 消除方案表（Markdown表格）

### S0-04 独立存在必要性判断

- **必选组件**: 边界清晰度评分（1-10）、功能内聚性评分（1-10）、消除成本评分（1-10，越高越不值得消除）、独立存在必要性评分（综合评分）
- **可选组件**: 评分依据详细说明、敏感性分析（不同权重下的评分变化）
- **组装顺序**: 边界清晰度评分→功能内聚性评分→消除成本评分→综合评分→必要性判定
- **约束**: 评分必须基于客观标准，不可凭感觉；综合评分必须考虑三个维度的权重（边界清晰度30%，功能内聚性40%，消除成本30%）
- **格式**: 评分表（Markdown表格）

### S0-05 决策输出

- **必选组件**: 决策建议（消除 / 重构 / 保留）、决策理由、后续行动建议
- **可选组件**: 决策置信度（高/中/低）、决策风险提示、决策后跟踪指标
- **组装顺序**: 评估结果汇总→决策建议→理由记录→后续行动建议
- **约束**: 决策建议必须明确，不可模棱两可；如果建议消除，必须说明消除理由；如果建议重构，必须进入阶段二执行
- **格式**: 决策记录（Markdown）

### S0-06 组成部分识别

- **必选组件**: 技能的每个组成部分（章节、范本、规则等）
- **可选组件**: 组成部分间依赖关系、组成部分重要性评估
- **组装顺序**: 技能内容梳理→组成部分列表→依赖关系标注→数据汇总
- **约束**: 必须完整识别技能的所有组成部分，不可跳过"理所当然"的部分
- **格式**: 技能组成部分清单（Markdown表格）

### S0-07 组成部分存在理由分析

- **必选组件**: 每个部分的存在理由（事情本身需要 / 人的局限需要）、部分类型标记（✅核心 / 🔶校准 / ❌传递 / ❌协调 / ⚡校验 / ❌格式）
- **可选组件**: 部分间依赖关系、部分重要性评估
- **组装顺序**: 逐部分追问→存在理由判定→类型标记→理由记录→汇总统计
- **约束**: 每个部分必须追问"如果执行者是一个拥有无限知识能力和零协作损耗的AI，这个部分还需要吗？"；判定必须基于事情本身逻辑，不可因"行业惯例"保留
- **格式**: 技能组成部分分析表（Markdown表格）

### S0-08 补偿层消除

- **必选组件**: 消除清单（哪些部分消除、为什么可消除）、保留清单（✅核心部分+🔶校准部分+⚡关键校验部分）、消除后的内容传递方式
- **可选组件**: 每个消除部分的风险评估、消除后需新增的关键校验点
- **组装顺序**: ❌标记部分逐一评估→消除决策→🔶校准部分确认→保留部分确认→关键校验点插入→消除清单+保留清单
- **约束**: 消除传递/协调/格式部分不可犹豫——这些是人的局限产物不是事情本身需要；校验部分精简为关键节点但涉及合规的不可消除；**校准部分不可消除**——起纠偏作用的必须保留
- **格式**: 消除决策表（Markdown表格）

### S0-09 重整

- **必选组件**: 重构后技能结构（YAML frontmatter + 章节结构）、重构后技能内容
- **可选组件**: 重构前后对比、重构理由说明
- **组装顺序**: 保留部分排序→技能结构设计→内容填充→范本编写→规则制定→验证清单
- **约束**: 内容必须基于实际能力，不可夸大；范本中 `________` 为待用户提供的内容，不可AI编造；YAML frontmatter 建议保留 name、author、description 三个字段；章节结构可根据用户风格规范灵活调整
- **格式**: 完整 SKILL.md 文件

### S0-10 重构验证

- **必选组件**: 10项验证清单逐项检查结果
- **可选组件**: 验证不通过项的修正方案
- **组装顺序**: 逐项验证→记录结果→不通过项修正→二次验证→通过
- **约束**: 10项必须全部通过；"内容完整性"不可妥协；"结构规范性"是硬约束
- **格式**: 验证清单（Markdown检查表）

### S0-11 改造形态选择

- **必选组件**: 选定改造形态及理由、执行方式说明
- **可选组件**: 形态切换条件、后续迭代方向
- **组装顺序**: 验证结果评估→三种形态适配判断→选定→理由记录
- **约束**: 选择最简形态——能消除的不重构，能重构的不保留
- **格式**: 形态选择记录（Markdown）

**范本要点**:
- 改造的核心是"追问存在理由"——每个技能都必须回答"这个技能覆盖的领域是否还需要存在"
- 消除技能比重构技能更彻底——如果技能领域本不该存在，消除是最佳选择
- 评分必须基于客观标准，不可凭感觉——功能重叠度、领域过时度、使用频率、维护成本都需要量化
- 改造建议必须明确——消除/重构/保留三选一，不可模棱两可
- 范本中 `________` 为待用户提供的内容，不可AI编造

## 使用规则

1. **判断是否需要改造**：当技能功能重叠、领域过时、使用频率低时，触发改造
2. **按十一步法执行**：S0-01 → S0-11，不可跳步
3. **阶段一必执行**：无论决策结果如何，都必须完成领域消除评估
4. **阶段二仅重构时执行**：只有决策为"重构"时，才执行内容重构
5. **产出交付**：按领域要求清单逐项填充，或按SR-01范本结构替换实际内容
6. **用户主权**：AI按技能框架产出的内容是起点，不是终稿。用户对任何环节有独特的校准点、质量标准或业务约束，都可以也应当要求修改——尤其是评分标准的权重，只有用户知道哪些维度对他的场景最重要。用户还可以主动提供清单（该技能应包含的组件列表）和样本（高质量的同类技能作为参考）作为校准参考，让AI的产出更贴合实际需求

---

## 事实纪律

1. AI工具能力描述必须基于实际能力，不得夸大
2. 改造结果必须标注为"参考范围"，实际效果取决于具体技能
3. 涉及法规、标准、合同等外部约束的技能必须明确标注，不可因改造而忽略
4. "技能存在必要性判断"必须基于客观数据，不可凭感觉
5. "技能内容重构"必须基于实际能力，不可夸大
