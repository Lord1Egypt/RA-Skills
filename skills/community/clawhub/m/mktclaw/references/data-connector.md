# 数据连接器规范 (v5.1)

> **本文档定义用户如何将真实业务数据接入 MktClaw**，特别是 data / media / strategy 等 Agent 的数据需求。
> **解决问题**：评估报告指出的"Skill 要求用户提供数据，但未设计'从用户数据源自动拉取'的机制。对 500 强 CMO 而言，手动填入 2 年多渠道投放数据不现实"。

---

## 🎯 设计目标

| 痛点 | 协议对策 |
|------|---------|
| 手动填入多渠道投放数据不现实 | 提供 CSV / JSON 标准模板 + 解析规范 |
| 缺乏 BI 工具对接 | 支持 Tableau / Power BI / 帆软 等导出格式 |
| 缺乏电商平台数据格式 | 支持天猫 / 京东 / 抖音 / 小红书等导出 |
| 缺乏数据质量校验 | 内置数据健康度自检 |
| 缺乏隐私安全边界 | 明确 PII / 敏感数据处理规范 |

### 方式 1：标准 CSV 模板（推荐）

#### 1.1 投放数据 CSV 模板

```csv
date,platform,campaign_id,campaign_name,ad_group,spend,impressions,clicks,conversions,revenue,currency,attribution_window

```

**字段规范**：

| 字段 | 类型 | 必填 | 说明 |
|------|------|:--:|------|
| `date` | YYYY-MM-DD | ✅ | 投放日期 |
| `platform` | enum | ✅ | 抖音/小红书/天猫/京东/百度/B站/微信/快手/微博/其他 |
| `campaign_id` | string | ✅ | 项目 ID |
| `campaign_name` | string | ✅ | 项目名 |
| `ad_group` | string | ⚠️ | 广告组（如有） |
| `spend` | number | ✅ | 投放消耗（元） |
| `impressions` | number | ✅ | 曝光 |
| `clicks` | number | ✅ | 点击 |
| `conversions` | number | ✅ | 转化（按平台定义） |
| `revenue` | number | ⚠️ | 收入（如可追踪） |
| `currency` | enum | ✅ | CNY/USD/... |
| `attribution_window` | string | ⚠️ | 归因窗口（1d/7d/15d/30d） |

## 🔍 数据健康度自检

数据接入后，**自动执行健康度自检**：

```markdown
## 📊 数据健康度报告

### 1. 完整性
- 时间范围: [起始日期] - [结束日期] (共 [N] 天)
- 缺失天数: [X] 天 (⚠️ [日期])
- 字段完整率: [spend] 100% / [revenue] 85% / [conversions] 100%

### 2. 异常值检测
- spend 异常: 检测到 [3] 处异常峰值（>3σ）
- CVR 异常: 检测到 [2] 处异常低值（<1%）⚠️ 可能归因窗口错配
- 季节性: 检测到 [618/双11] 大促节点

### 3. 数据质量评分
| 维度 | 得分 | 说明 |
|------|:---:|------|
| 时间覆盖度 | 95/100 | 缺失 18 天 |
| 字段完整性 | 88/100 | revenue 缺失 15% |
| 数据一致性 | 92/100 | 跨平台定义统一 |
| 异常值处理 | 80/100 | 需用户确认 [X] 处 |
| **总分** | **89/100** | 🟢 良好 |

### 4. 建议处理
- 缺失日期 [X-X]: 建议补充或用插值
- revenue 缺失: 建议补充电商后台数据
- 异常值 [日期]: 请确认是否为大促
```

## 🛡️ 隐私与安全边界

### 5.1 数据脱敏（必做）

接入数据前，**必须脱敏**以下字段：

| 字段类型 | 脱敏方式 |
|---------|---------|
| 用户姓名 | 替换为 `user_xxx` |
| 手机号 | 替换为 `phone_xxx` |
| 邮箱 | 替换为 `email_xxx` |
| 地址 | 替换为 `city_xx` |
| 身份证 | 替换为 `id_xxx` |
| IP 地址 | 替换为 `ip_xxx` |
| 设备 ID | 替换为 `device_xxx` |

### 5.2 数据范围声明

| 数据类型 | 接入策略 |
|---------|---------|
| **聚合数据**（推荐） | ✅ 平台级 / 渠道级 / 项目级聚合 |
| **匿名用户级** | ⚠️ 需脱敏后接入，仅用于 RFM 等分析 |
| **真实 PII** | ❌ 禁止接入 |
| **支付/金融数据** | ❌ 禁止接入（仅聚合后金额） |
| **第三方竞品数据** | ⚠️ 仅限公开数据 |

### 5.3 数据生命周期声明

> **v5.2.1 新增**：明确区分"原始上传数据"与"Vault 沉淀数据"的生命周期，消除语义张力。

MktClaw 处理两种不同生命周期的数据：

| 数据类别 | 定义 | 生命周期 | 持久化 | 关联协议 |
|---------|------|---------|:------:|---------|
| **原始上传数据 (Raw Data)** | 用户上传的 CSV/JSON 原始文件 | 临时——分析完成后 30 天内自动删除 | ❌ 不在服务器端持久化 | data-connector（本文档） |
| **Vault 沉淀数据 (Processed Vault)** | 从交付物中提取的结构化 Campaign 摘要 | 持久化——用户本地文件，24 个月保留 | ✅ 用户本地选择性地保留 | campaign-vault.md |

### 5.3.1 原始上传数据（临时）

当用户上传 CSV/JSON 数据用于分析时：

```
上传 → Agent 读取 → 执行分析 → 输出报告
                         ↓
                 30 天后自动删除（原始文件不保留）
```

**Agent 告知模板**：

```markdown
## 📋 数据接入告知

1. **数据类别**：本次上传的数据属于"原始上传数据"（Raw Data）
2. **生命周期**：分析完成后 30 天内自动删除，不会持久化保留
3. **用途限制**：仅用于本次会话的分析和方案产出
4. **脱敏要求**：上传前请确保不包含 PII（姓名/邮箱/手机号/身份证号）
5. **数据范围**：建议上传聚合级数据（平台/渠道/项目级）
6. **第三方分享**：本 Skill 不会将您的数据分享给任何第三方
7. **Vault 沉淀建议**：分析完成后，[结果摘要会自动生成 Vault Entry](./campaign-vault.md#Agent-交付时的自动捕获模板)。这是可选的——您可以选择性保留。

> **参考**：如需持久化保留分析结果（而不是原始数据），请使用 **Campaign Vault**（[campaign-vault.md](./campaign-vault.md)）。Vault 只保留聚合后的 KPI 和洞察，不保留原始数据文件。
```

### 5.3.2 Vault 沉淀数据（持久化）

当用户选择将 Campaign 结果沉淀到 Campaign Vault 时：

```
Agent 交付方案 → 自动抽取结构化 Entry → 用户复制 JSON 到本地
                                               ↓
                            下次会话粘贴 → Agent 解析 → 个性化 benchmark
```

> **Vault 与原始数据的区别**：Vault 只保留**聚合后的 KPI 和洞察**（ROAS 3.2x、CPA 35 元、关键发现等），不保留原始投放数据文件。用户本地选择性保留，随时可删除。

**数据留存对比**：

| 维度 | 原始上传数据 (Raw) | Vault 沉淀数据 (Processed) |
|------|------------------|--------------------------|
| 内容 | 原始 CSV/JSON 投放数据 | 聚合 KPI + 文本洞察 |
| 大小 | 大（千行级） | 小（Campaign 一条 JSON） |
| 留存 | 30 天自动删除 | 24 个月，用户可删除 |
| 隐私风险 | 高（含广告组级粒度） | 低（仅聚合 KPI） |
| 格式 | CSV/JSON | 结构化 JSON |
| 位置 | 临时会话上下文 | 用户本地文件 |
| 下次可用 | ❌ 不可用 | ✅ 可引用做 benchmark |

## 🔗 与各 Type Agent 的集成

### data Agent（主要使用者）

```markdown
## Data Agent 数据接入流程

1. 检测 Brief 是否含 `data_source` 字段
2. 若有 → Read 对应文件，执行数据健康度自检
3. 若无 → 询问用户："是否方便上传历史数据？支持 [CSV/JSON/BI 导出]"
4. 数据接入后，自动应用 Adstock / Hill 函数分析
5. 输出 MMM 报告 + 数据来源声明
```

### media Agent

```markdown
## Media Agent 数据接入

1. 检测 Brief 是否含历史投放数据
2. 若有 → 基于实际 ROAS / CPM / CVR 优化预算分配
3. 若无 → 退回 benchmark-database.md + 行业专属数据
4. 数据接入后，输出"基于实际数据 vs 基于基准"对比
```

### strategy Agent

```markdown
## Strategy Agent 数据接入

1. 检测 Brief 是否含业务数据（销售/客户/市场）
2. 若有 → 基于实际数据做诊断分析（不再依赖基准）
3. 若无 → 退回行业 benchmark + 标注"基于基准"
4. 数据接入后，所有假设都基于真实数据验证
```

## 🚫 反模式（禁止）

| 反模式 | 为什么禁止 |
|--------|----------|
| 接受未脱敏的 PII 数据 | 隐私风险 |
| 直接接受 Excel 公式 / VBA | 解析复杂、易出错 |
| 接受超过 100MB 的单文件 | 性能问题 |
| 接受非结构化 PDF / 图片 | 无法解析 |
| 不做数据健康度自检直接使用 | 错误数据 = 错误结论 |
| 接入竞品非公开数据 | 法律风险 |
