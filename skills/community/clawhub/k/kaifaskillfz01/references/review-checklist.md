## 逐项检查指南

### 1. Plan 一致性（对照设计方案）
- [ ] 目录结构和 Plan 一致？
- [ ] API 接口签名和 Plan 一致？
- [ ] 数据模型和 Plan 一致？
- [ ] 如果有偏差，偏差是否合理？

### 2. 逻辑正确性
- [ ] 核心逻辑是否正确实现？
- [ ] 条件分支是否覆盖所有情况？
- [ ] 循环是否有终止条件？
- [ ] 有无死代码或不可达路径？
- [ ] 异步/并发处理是否正确？

### 3. 边界处理
- [ ] 空值/undefined/null 是否处理？
- [ ] 输入边界（最大值、最小值、空字符串）是否处理？
- [ ] 错误状态是否有合适的处理？
- [ ] Plan 中列出的边界情况是否全部覆盖？

### 4. 安全性
- [ ] 有无注入风险（SQL/XSS/命令注入等）？
- [ ] 敏感信息（密钥、密码、token）是否暴露？
- [ ] 文件路径是否做了遍历防护？
- [ ] 用户输入是否做了校验和消毒？
- [ ] 权限检查是否到位？

### 5. 代码风格
- [ ] 命名是否清晰、一致？
- [ ] 关键逻辑是否有注释？
- [ ] 函数/方法是否单一职责？
- [ ] 代码结构是否清晰可读？

### 6. 冗余检查
- [ ] 有无重复代码？
- [ ] 有无不必要的抽象层？
- [ ] 有无 Plan 之外的过度实现？

## 严重程度判定标准

| 等级 | 判定标准 | 举例 |
|------|---------|------|
| 🔴 BLOCKER | 架构错误、安全漏洞、核心逻辑错误、Plan 严重偏离 | SQL 注入未防护、数据流方向错了、关键接口缺失 |
| 🟡 MAJOR | 逻辑缺陷、性能隐患、边界遗漏、错误处理缺失 | 未处理 null、潜在死循环、缺少超时 |
| 🟢 MINOR | 命名不规范、缺少注释、代码风格、轻微冗余 | 变量名拼写、函数可拆分但功能正确 |
