# Grill-With-Docs：需求对齐

## 1. Overview（概述）

在写任何代码之前，把需求问穿。模糊需求 = 返工债。本技能通过结构化追问和 CONTEXT.md 维护，确保每个需求在编码前达到「可执行」精度。

**核心原则**：
- 没问清楚不动手 — 假设是万恶之源
- CONTEXT.md 是活的 — 每个新术语、新决策都要写入
- ADR 只记录三类决策 — 不可逆、令人惊讶、有真实权衡

## 2. When to Use（使用条件）

**触发条件**：
- Step 1 设计阶段自动注入
- A 级任务至少注入需求确认部分（精简版）
- 需求描述含歧义词（"应该"、"大概"、"类似"、"参考"）时强制执行

**不适用**：
- 明确的 bug fix（有清晰的期望行为）
- 已有完整 CONTEXT.md 且需求没有新领域概念

## 3. Process（执行流程）

### 3.1 结构化追问（决策树）

```
收到需求
  ├─ 有 CONTEXT.md？ → 否 → 创建 CONTEXT.md（从需求中提取术语、领域模型）
  │   └─ 是 → 读取 CONTEXT.md（最新版）
  │
  ├─ 需求有歧义？
  │   ├─ 是 → 追问 5 个标准问题：
  │   │   1. 输入是什么？格式、来源、边界值？
  │   │   2. 输出是什么？精确的返回格式、成功/失败表现？
  │   │   3. 错误怎么处理？每个可能的错误路径？
  │   │   4. 有什么不做的？明确的范围外事项？
  │   │   5. 验收标准是什么？如何验证「做对了」？
  │   └─ 否 → 进入方案设计
  │
  └─ 所有追问已回答 → 更新 CONTEXT.md → 进入方案设计
```

### 3.2 方案设计输出

```
要求输出（作为阶段产物）：
1. 范围声明：做什么 + 不做什么
2. 接口定义：输入/输出/错误（精确到类型和格式）
3. 验收标准：可量化、可执行
4. ADR 判断：是否需要创建 ADR → 检查三条件
```

### 3.3 CONTEXT.md 维护规则

- 每个新领域术语写入（首次出现时）
- 每个架构决策写入（摘要 + 关联 ADR 编号）
- 每个外部依赖写入（名称 + 版本）
- 格式：`## 术语 / ## 决策 / ## 依赖`
- 反例 ❌：不记录实现细节、不记录临时方案、不重复 ADR 全文

### 3.4 ADR 创建标准

三条件（缺一不创建）：
1. **不可逆** — 决策后很难回退（如数据库选型、协议选择）
2. **令人惊讶** — 非惯例做法，未来的维护者会问"为什么"
3. **真实权衡** — 存在 ≥2 个可选方案，各有优劣

ADR 格式：`## ADR-{序号}: {标题}\n### 背景 / ### 方案 / ### 权衡 / ### 决策`

反例 ❌："用 Python 不用 Ruby"（团队标准语言，不令人惊讶）
反例 ❌："函数名用驼峰"（可逆，风格选择）

## 4. Risk Scorecard（反借口表）

| # | 纪律 | 常见借口 | 反驳 |
|---|------|---------|------|
| R1 | 需求不明确不动手 | "我大概理解，边做边调整" | 模糊理解 → 错误实现 → 返工成本 3x |
| R2 | CONTEXT.md 必须维护 | "这些都是常识，不用记" | 明天的你不记得今天的决定 |
| R3 | ADR 三条件缺一不创建 | "这个决策很重要，得记录" | 重要性 ≠ 需要 ADR，多数决策可逆 |
| R4 | 范围必须明确"不做什么" | "不需要的东西不用说" | 没说 = 可能被误解为该做 |
| R5 | 验收标准量化 | "做完就知道了" | 不知道 Done 长什么样 = 不知道何时停 |

## 5. Red Flags（危险信号）

- 🚩 需求描述含"简单"二字（"写个简单的 API"）— 最危险的需求
- 🚩 CONTEXT.md 不存在且项目 > 1 周 — 知识丢失风险
- 🚩 接口定义缺错误路径 — 实现时会凭空发明错误处理
- 🚩 验收标准含"应该"（"应该返回正确结果"）— 不可执行
- 🚩 跳过追问直接给方案 — 可能解决错误的问题
- 🚩 ADR 数量 > 项目复杂度 2x — 过度文档化（多数可逆决策不需要 ADR）

## 6. Verification（自检清单）

**必检（≤5 项）**：
- [ ] CONTEXT.md 已读取（若存在）或已创建
- [ ] 所有追问已回答（决策树走完，无悬挂问题）
- [ ] 范围明确：做什么 + 不做什么均已列出
- [ ] 接口定义完整：输入/输出/错误均已明确类型
- [ ] 验收标准可量化、可执行（不含"应该"类词汇）

**抽检（≤3 项）**：
- [ ] ADR 创建条件已检查（三条件逐一判定）
- [ ] CONTEXT.md 已更新（新术语、新决策已写入）
- [ ] 技术方案有备选方案对比（至少 1 个备选 + 对比理由）

**免检**：文档格式、用词风格、示例代码细节
