# 电子签名与 SDLC 参考

## 电子签名 (Electronic Signatures)

### 基本概念

电子签名在 GxP 环境中替代传统手写签名，具有法律效力且必须满足监管要求。

### 电子签名类型

| 类型 | 要求 | 典型应用 |
|------|------|----------|
| Basic Electronic Signature | 单要素（知识） | 简单审批流程 |
| Advanced Electronic Signature (AES) | 两要素（知识+持有） | 法规要求的GxP签名 |

### 两要素认证

1. **知识要素**: 密码、PIN码
2. **持有要素**: 智能卡、硬件令牌、生物识别

### 电子签名链接要求

电子签名必须满足"链接原则"：
- 签名与记录不可分割
- 签名后记录不可更改
- 签名元数据必须包含：签署人、签署时间、签署含义

### 审计追踪要求

每个电子签名操作必须记录：
```
- 签署人身份
- 签署日期/时间
- 签署含义（如"Approved", "Reviewed"）
- 计算机系统标识
- 签名对应的电子记录
```

## 软件开发生命周期 (SDLC)

### GxP 环境下的 SDLC 特殊要求

在 GxP 监管环境（特别是21 CFR Part 11和EU Annex 11）中，SDLC必须满足：

### 阶段模型

```
┌─────────────────────────────────────────────────────┐
│  1. 计划与需求定义                                   │
│     - 用户需求规格 (URS)                            │
│     - 法规影响评估                                   │
├─────────────────────────────────────────────────────┤
│  2. 设计阶段                                        │
│     - 功能规格 (FS)                                 │
│     - 技术/设计规格 (DS)                            │
│     - 风险评估 (RA)                                 │
├─────────────────────────────────────────────────────┤
│  3. 开发/采购                                       │
│     - 代码开发或软件采购                             │
│     - 单元测试                                       │
│     - 集成测试                                       │
├─────────────────────────────────────────────────────┤
│  4. 验证                                            │
│     - IQ: 安装确认                                   │
│     - OQ: 运行确认                                   │
│     - PQ: 性能确认                                   │
├─────────────────────────────────────────────────────┤
│  5. 部署与维护                                       │
│     - 切换到生产环境                                 │
│     - 变更控制                                       │
│     - 持续监控                                       │
├─────────────────────────────────────────────────────┤
│  6. 退役                                            │
│     - 数据迁移或归档                                 │
│     - 系统退役文档                                   │
└─────────────────────────────────────────────────────┘
```

### 文档化要求

| 阶段 | 必需文档 |
|------|----------|
| 需求 | URS, 法规影响评估 |
| 设计 | FS, DS, RA, DR |
| 开发 | 源代码控制记录, 测试脚本 |
| 验证 | IQ/OQ/PQ 协议与报告 |
| 运维 | SOP, 变更记录, 事故报告 |
| 退役 | 退役计划, 数据迁移记录 |

### 变更控制 (Change Control)

GxP 系统任何变更必须经过正式变更控制流程：

1. **变更申请 (Change Request)**
2. **影响评估 (Impact Assessment)**
3. **审批 (Approval)**
4. **实施 (Implementation)**
5. **验证 (Verification)**
6. **关闭 (Closure)**

### 配置管理

- 所有系统配置必须文档化
- 配置变更必须可追溯
- 基线配置必须建立并维护

### 供应商管理 (适用于外购系统)

| 评估维度 | 评估内容 |
|----------|----------|
| 质量体系 | ISO 9001, ISO 27001 |
| 开发能力 | SDLC 文档完整性 |
| 验证支持 | IQ/OQ 配合能力 |
| 支持服务 | 维护响应时间 |
| 财务稳定 | 公司财务状况 |
| 客户案例 | 行业参考客户 |

### 数据迁移

系统更换时数据迁移必须：
1. 制定数据迁移计划
2. 执行数据映射
3. 执行试迁移
4. 验证数据完整性
5. 执行正式迁移
6. 保留迁移文档

## 数据完整性 (Data Integrity / ALCOA+)

### ALCOA+ 原则

| 字母 | 含义 | 要求 |
|------|------|------|
| A | Attributable | 可归属 - 数据可追溯到操作者 |
| L | Legible | 可读 - 数据清晰可读 |
| C | Contemporaneous | 同时性 - 数据在操作时记录 |
| O | Original | 原始性 - 使用原始记录 |
| A | Accurate | 准确性 - 数据真实准确 |
| + | Complete | 完整性 - 所有数据完整 |
| + | Consistent | 一致性 - 数据逻辑一致 |
| + | Enduring | 持久性 - 记录可长期保存 |
| + | Available | 可用性 - 数据可随时调取 |

### 数据完整性检查要点

1. 审计追踪是否启用？
2. 数据修改是否有完整日志？
3. 是否存在数据删除风险？
4. 访问权限是否适当分离？
5. 是否有未授权的系统时钟修改？
