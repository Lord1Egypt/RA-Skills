# 故事执行模块

## 1. 概述

故事执行模块负责将PRD中的用户故事分解为可执行的任务，并通过子代理并行执行这些任务，确保项目从需求到代码的顺利落地。

## 2. 用户故事分解

### 2.1 分解原则

1. **粒度适中**：每个故事必须足够小，能在一个上下文窗口内完成
2. **功能独立**：每个故事应具有相对独立的功能
3. **可测试性**：每个故事应有明确的验收标准
4. **依赖清晰**：明确记录故事间的依赖关系

### 2.2 分解流程

1. **接收PRD**：从PRD校验模块接收经过校验的PRD
2. **故事分析**：分析每个用户故事的复杂度和依赖关系
3. **分解决策**：决定是否需要进一步分解故事
4. **任务定义**：为每个故事定义具体的执行任务
5. **优先级排序**：根据依赖关系和优先级对故事进行排序

### 2.3 分解示例

**原始故事**：实现用户登录功能

**分解后的任务**：
1. 设计登录接口API
2. 实现后端登录逻辑
3. 设计登录页面UI
4. 实现前端登录功能
5. 测试登录流程

## 3. 子代理执行机制

### 3.1 子代理分配

1. **类型匹配**：根据故事类型（scaffold/backend/frontend/fullstack）分配相应的子代理
2. **并行执行**：无依赖的故事可以并行执行
3. **串行执行**：有依赖的故事按照依赖关系串行执行

### 3.2 执行环境

1. **上下文隔离**：每个子代理在独立的上下文窗口中执行
2. **文件系统传递**：通过文件系统在子代理间传递上下文
3. **执行日志**：记录每个故事的执行过程和结果

### 3.3 执行流程

1. **初始化**：创建执行环境和上下文
2. **故事执行**：按顺序执行用户故事
3. **结果验证**：验证每个故事的执行结果是否符合验收标准
4. **状态更新**：更新项目的执行状态
5. **模式沉淀**：记录执行过程中的可复用模式

## 4. 模式沉淀机制

### 4.1 模式定义

**可复用模式**：在项目执行过程中发现的、可在其他项目中复用的解决方案或最佳实践。

### 4.2 沉淀流程

1. **模式识别**：在执行过程中识别可复用的模式
2. **模式提取**：提取模式的核心要素和实现细节
3. **模式文档**：将模式记录到`conventions/agents.md`文件中
4. **模式分类**：按照技术领域和功能类型对模式进行分类
5. **模式检索**：建立模式检索机制，方便后续项目使用

### 4.3 模式示例

**模式名称**：JWT认证实现
**技术领域**：后端认证
**适用场景**：需要无状态认证的Web应用
**实现细节**：
- 使用jsonwebtoken库生成和验证token
- 设置合理的token过期时间
- 实现token刷新机制
- 集成到Express中间件中

## 5. 执行状态管理

### 5.1 状态定义

- **待执行**：故事已就绪，等待执行
- **执行中**：故事正在执行
- **已完成**：故事执行完成，验证通过
- **执行失败**：故事执行失败，需要修正
- **已跳过**：故事因依赖问题或其他原因被跳过

### 5.2 状态转换

```
待执行 → 执行中 → 已完成
待执行 → 执行中 → 执行失败 → 待执行
待执行 → 已跳过
```

### 5.3 状态监控

1. **执行进度**：实时监控故事的执行进度
2. **错误处理**：及时处理执行过程中的错误
3. **状态报告**：生成详细的执行状态报告

## 6. 与知识库的集成

### 6.1 知识查询

1. **技术选型**：根据PRD中的技术栈选择，查询知识库中的最佳实践
2. **问题解决**：在执行过程中遇到问题时，查询知识库中的解决方案
3. **代码示例**：查询相关的代码示例和模板

### 6.2 知识更新

1. **新实践记录**：将执行过程中发现的新实践记录到知识库中
2. **问题解决方案**：将解决的问题和方案记录到知识库中
3. **代码质量提升**：将优化后的代码示例更新到知识库中

### 6.3 知识反馈循环

```
执行过程 → 发现新实践 → 更新知识库 → 指导后续执行
```

## 7. 最佳实践

1. **并行执行**：充分利用子代理的并行执行能力，提高效率
2. **依赖管理**：合理管理故事间的依赖关系，避免执行顺序错误
3. **模式沉淀**：重视模式的识别和沉淀，提高代码复用率
4. **知识更新**：及时更新知识库，保持知识的时效性
5. **执行监控**：建立有效的执行监控机制，及时发现和解决问题

## 8. 输出格式

### 8.1 执行日志

执行过程中的日志应保存到`prd-context/execution-logs/development-log.txt`文件中，格式如下：

```
[2024-01-01 12:00:00] 开始执行故事：实现用户登录功能
[2024-01-01 12:05:00] 故事执行完成：实现用户登录功能
[2024-01-01 12:05:00] 验证结果：通过
[2024-01-01 12:05:00] 发现可复用模式：JWT认证实现
```

### 8.2 模式文档

可复用模式应记录到`prd-context/conventions/agents.md`文件中，格式如下：

```markdown
# 可复用模式

## 后端认证

### JWT认证实现
- **适用场景**：需要无状态认证的Web应用
- **技术栈**：Node.js, Express, jsonwebtoken
- **实现细节**：
  1. 安装jsonwebtoken库
  2. 生成token时包含用户信息和过期时间
  3. 验证token时检查签名和过期时间
  4. 实现token刷新机制
- **代码示例**：
  ```javascript
  // 生成token
  const token = jwt.sign({ userId: user.id }, process.env.JWT_SECRET, { expiresIn: '1h' });
  
  // 验证token
  const verifyToken = (req, res, next) => {
    const token = req.headers.authorization?.split(' ')[1];
    if (!token) return res.status(401).json({ message: 'Access denied' });
    
    try {
      const decoded = jwt.verify(token, process.env.JWT_SECRET);
      req.user = decoded;
      next();
    } catch (error) {
      res.status(401).json({ message: 'Invalid token' });
    }
  };
  ```
```

## 9. 与其他模块的交互

### 9.1 与PRD校验模块的交互
- 接收PRD校验模块的输出
- 基于经过校验的PRD执行故事

### 9.2 与知识库的交互
- 查询知识库获取技术指导
- 向知识库反馈新的实践和解决方案

### 9.3 与用户的交互
- 向用户报告执行进度和结果
- 遇到问题时向用户寻求指导
- 执行完成后向用户提供最终结果

## 10. 执行质量保障

1. **代码质量**：确保生成的代码符合最佳实践和编码规范
2. **测试覆盖**：确保每个故事都有相应的测试
3. **文档完整性**：确保生成的代码有完整的文档
4. **性能优化**：考虑代码的性能和可扩展性
5. **安全考虑**：确保代码的安全性，避免常见的安全问题

通过以上机制，故事执行模块能够高效、高质量地将PRD转化为可运行的代码，同时不断积累和更新知识库，为后续项目提供更好的指导。
