# 能力7：自动化接入协议（Beta）

> 本文档定义能力7 "自动化接入助手" 的执行规范与安全边界。
>
> ‼️ 本能力是对原有 6 大能力的**增量扩展**，不替代原有流程。
> 默认情况下仍应使用能力2（接入开发指引，只展示不写入），只有用户**明确主动**请求"帮我接入"时才启动本能力。

---

## 一、启动条件（四选一均须满足）

1. ✅ 用户**主动**发出"帮我接入 SDK""自动接入""生成接入代码"等明确请求
2. ✅ 用户提供了**可写的项目根目录**（非只读、非生产环境）
3. ✅ 用户已提供或愿意提供：`user_action_set_id`、`secret_key`、`appid`（后两者按端差异）
4. ✅ 用户**确认**项目已使用版本控制（git）或已备份

任一不满足 → **降级到能力2**（只展示代码示例，由用户自行复制）。

---

## 二、执行流程（严格按顺序）

```
┌────────────────────────────────────────────────────────────────┐
│                                                                │
│  Step 1  场景识别                                               │
│    运行 scripts/detect_framework.py --json                      │

│  Step 2  扫描接入点                                             │
│    运行 scripts/scan_integration_points.py --sdk-end <X> --json│
│    ↓                                                           │
│  Step 3  展示扫描结果 + 征得用户同意   ⚠️ 强制确认节点           │
│    - 将要改动的文件列表                                         │
│    - init 插入位置的上下文                                      │
│    - 检测到的已有 SDK / 冲突警告                                │
│    - 用户必须回复"确认 / approve / 继续"才能进入 Step 4         │
│    ↓                                                           │
│  Step 4  收集接入参数                                           │
│    - user_action_set_id（必填）                                 │
│    - secret_key（必填）                                         │
│    - appid（小程序/小游戏必填）                                 │
│    运行 scripts/generate_init_patch.py 产出方案 JSON            │
│    ↓                                                           │
│  Step 5  展示方案 + 再次确认             ⚠️ 强制确认节点        │
│    - 展示完整 code_snippet                                      │
│    - 展示 warnings（如有）                                      │
│    - 用户必须回复"确认写入 / apply / 继续"才能进入 Step 6       │
│    ↓                                                           │
│  Step 6  Agent 调用自身 edit 工具写入代码                        │
│    - 使用 replace_in_file / edit_file                           │
│    - 不要使用 write_to_file 覆盖整文件                           │
│    - 改动后输出 git diff 供用户审阅                              │
│    ↓                                                           │
│  Step 7  自动验证                                               │
│    运行 scripts/validate_integration.py --sdk-end <X>           │

│    ↓                                                           │
│  Step 8  引导后续流程                                           │
```

## 三、安全边界（必须遵守）

### 3.1 代码改动边界

| 允许做的 | 禁止做的 |
|---|---|
| ✅ 在 Application/AppDelegate/EntryAbility/App() 内插入 SDK init 代码 | ❌ 修改客户的业务逻辑代码（付费、登录、业务流转等） |
| ✅ 在 AndroidManifest.xml / module.json5 中添加 SDK 所需权限 | ❌ 删除客户已有的任何代码 |
| ✅ 在 Podfile / build.gradle / package.json 中添加 SDK 依赖 | ❌ 修改客户现有的三方库依赖版本 |
| ✅ 在 proguard-rules.pro 中追加 SDK 混淆白名单 | ❌ 清理或重写客户项目结构 |
| ✅ 新建 DN SDK 相关的辅助文件（如 sdk-config.ts） | ❌ 在未获同意下写入任何环境变量、敏感信息 |

### 3.2 参数处理边界

- `secret_key` **严禁**提交到 git 仓库或任何对外可见位置
  - 生成的代码中可以出现，但 Agent 必须提示用户"请将 secret_key 改为从环境变量/配置中心读取"
  - 默认生成的是字面量，方便 Demo；**生产环境的安全改造交给客户自己决定**
- 所有参数不能被 Skill 收集、上报到任何外部服务
- Agent 不应在对话历史中重复完整的 `secret_key`（避免敏感信息泄露到 IDE 对话记录）

### 3.3 降级规则

以下场景**必须降级**到能力2（只展示不写入）：

1. `detect_framework.py` 返回 `auto_integration_supported: false`
2. 检测到项目是 monorepo 且有多个候选 SDK 端（需人工澄清）
3. 用户未明确同意（任一确认节点用户回复"不"/"取消"）
4. `generate_init_patch.py` 产出 `success: false`
5. Agent 在 Step 6 调用 edit 工具连续失败 3 次

### 3.4 不可逆操作防护

- 改动前必须**运行 git status** 确认工作区是否干净
- 改动前提示用户："建议先 git stash 或新建分支再继续"
- 改动后立刻运行 validate 并展示结果
- 如果 validate 有 FAIL，主动提示用户："可通过 `git diff` 查看改动，或 `git restore .` 回滚"

## 四、与其他能力的关系

| 能力 | 关系 |
|---|---|
| **能力1（SDK 选型）** | 能力7 的 Step 1 会自动完成选型识别，不需要重复走能力1 |
| **能力2（接入开发指引）** | 能力7 是能力2 的自动化版；降级场景下回退到能力2 |
| **能力3（事件埋点速查）** | 能力7 只插 init；业务事件埋点仍走能力3 |
| **能力4（联调对账）** | 能力7 完成后推荐进入能力4 |
| **能力5（质量评估）** | 能力7 的 Step 7 等同于能力5 的自动化版；人工全面检查仍可走能力5 |
| **能力6（问题排查）** | 能力7 执行期间若 validate 失败，自动引导到能力6 对应端 FAQ |

## 五、脚本调用速查

```bash
# 1. 识别场景
python3 scripts/detect_framework.py <project_root> --json > /tmp/detect.json

# 2. 扫描接入点
python3 scripts/scan_integration_points.py <project_root> --sdk-end <X> --json > /tmp/scan.json

# 3. 生成方案
python3 scripts/generate_init_patch.py \
  --detect-json /tmp/detect.json \
  --scan-json /tmp/scan.json \
  --user-action-set-id <ID> \
  --secret-key <KEY> \
  --appid <APPID> \
  --output /tmp/plan.json

# 4. 由 Agent 读取 /tmp/plan.json 并调用 edit 工具执行

# 5. 校验
python3 scripts/validate_integration.py <project_root> --sdk-end <X>
```

退出码约定：
- `detect_framework.py`：0 识别成功，1 未知端
- `scan_integration_points.py`：0 正常，1 错误
- `generate_init_patch.py`：0 方案就绪，2 无法生成
- `validate_integration.py`：0 全通过，1 有 WARN，2 有 FAIL

## 六、Beta 说明

本能力为**首期 Beta**，当前白名单支持：

| SDK 端 | 支持的框架/语言 | 限制 |
|---|---|---|
| mini-game | 原生 / Cocos Creator / LayaAir | Unity、Egret 暂不支持自动写入 |
| mini-program | 原生 / Taro / uni-app | WePY、mpvue 等暂不支持自动写入 |
| android | Java / Kotlin | 需是标准 Gradle 项目；RN/Flutter 暂不支持 |
| ios | Objective-C / Swift + CocoaPods | 手动集成、SwiftPM 暂不支持 |
| harmony | ArkTS | 需是标准 DevEco 项目结构 |

白名单外的场景**自动降级**到能力2，不会强行执行。

Beta 期间，每次执行建议用户：
- 使用 git 确认干净的工作区
- 建议新建分支 `feat/dn-sdk-integration`
- 完成后人工 review git diff 再合并

## 七、FAQ

**Q1：为什么不用 AST 解析而用 regex？**
A：Python 标准库没有 JS/Swift/ArkTS 的 AST 解析器；引入 tree-sitter 等依赖会大幅提升 Skill 部署成本。regex + 上下文 3 行足以识别生命周期入口点，实测准确率 > 95%。对识别失败的场景，自动降级处理。

**Q2：为什么让 Agent 调用自身 edit 工具而不是 Python 脚本直接写文件？**
A：
- 让客户在 IDE 内亲眼看到 git diff，体验更透明可控
- 客户可随时 Ctrl+Z 撤销
- Python 脚本写文件需要处理权限、换行符、编码等琐碎问题，交给 IDE Agent 更稳
- **安全审计**：所有改动都经过 IDE 审计，而非 Skill 脚本"偷偷"写文件

**Q3：如果客户的 Application 入口不在项目根目录第一层怎么办？**
A：`scan_integration_points.py` 会递归扫描整个项目，只要不在 IGNORE_DIRS 内都能找到。多入口时 `generate_init_patch.py` 只会选择优先级最高的那一个，其他作为候选返回由 Agent 决策或向用户确认。

**Q4：生成的代码里 secret_key 是明文，有安全风险吗？**
A：
- 对个人开发者/原型项目：可接受
- 对正式生产项目：生成后 Agent 必须主动提示用户迁移到环境变量/配置中心
- 未来可扩展支持 `--secret-key-from-env` 参数，默认生成 `process.env.DN_SECRET_KEY` 而非明文

**Q5：如果我就想用原来的手动方式接入呢？**
A：直接问 Skill"如何手动接入 Android SDK"即可，会走能力2，严禁自动写入。
