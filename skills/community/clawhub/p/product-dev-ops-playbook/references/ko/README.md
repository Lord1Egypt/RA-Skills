# 제품 × 개발 × 운영 협업 SOP

> 이 SOP는 실제 제품 전략 회의록, 사용자 피드백 입력 템플릿, 베타 사용자 인터뷰 체계를 통합하여 **제품·개발·운영 3자 협업**의 범용 워크프레임워크로 정리한 것입니다.
>
> **핵심 원칙:** 모든 것은 상업화를 위해. 모든 것은 수익을 위해.
>
> **사용법:** 본 문서는 범용 템플릿입니다. 모든 `[제품명]` / `[프로젝트 코드]` / `[시스템명]`을 실제 정보로 바꿔서 바로 사용하세요.

---

## 1. 핵심 문제 진단: 왜 제품·개발·운영은 항상 협업이 안 될까?

### 1.1 거의 모든 성장기 제품이 직면하는 3가지 모순

| 모순 | 증상 | 근본 원인 |
|------|------|----------|
| **신기능 vs 사용자 피드백** | 개발팀은 항상 신기능을 쫓고, 사용자가 보고한 버그는 무기한 방치 | 통합된 우선순위 결정 메커니즘이 없음 |
| **개발이 만들고 싶은 것 vs 운영이 필요한 것** | 개발은 "운영이 기술을 모른다"고 생각하고, 운영은 "개발이 사용자 말을 안 듣는다"고 생각 | 공통 목표가 없음 |
| **속도 vs 안정성** | 제품 형태가 안정되기 전에 성장을 서두르고, 기술 부채가 쌓임 | 명확한 제품 단계 판단이 없음 |

### 1.2 해결 접근법

> **실제 회의에서 얻은 인사이트:** 수익화를 잘하는 회사(예: Manus, DeepSeek)는 COO나 운영 책임자가 직접 상업화를 담당하며, 제품·개발·운영이 고도로 연계됩니다.

**4대 핵심 메커니즘:**

| # | 메커니즘 | 설명 |
|---|---------|------|
| 1 | **통합 칸반** | 사용자 피드백과 제품 요구사항을 하나의 풀에 넣고 단일 태그 체계로 관리 |
| 2 | **공통 목표** | 매 이터레이션마다 명확한 비즈니스 지표 설정(예: "가입→결제 전환율 4.5%→7%") |
| 3 | **운영의 거부권** | 사용자 경험이나 결제 전환에 직접 해를 끼치는 문제에 대해, 운영이 릴리스 연기를 제안할 수 있음 |
| 4 | **매 이터레이션 전 3자 얼라인먼트** | 제품·개발·운영 책임자가 공동으로 무엇을 만들지, 무엇을 먼저 할지, 무엇을 다음으로 미룰지 결정 |

## 2. 통합 협업 기반: 칸반 시스템 설계

### 2.1 2단계 칸반 구조

> **실전 경험에서:** 이터레이션별 작은 보드만 두지 마세요 — 반드시 마스터 백로그가 있어야 합니다. 모든 원시 이슈는 먼저 마스터 풀에 들어간 뒤 하위 이터레이션으로 흘러갑니다.

| 레이어 | 이름 | 내용 | 담당자 |
|--------|------|------|--------|
| **1단계: 마스터 백로그** | `[제품명] 마스터 백로그` | 모든 출처의 원시 이슈(사용자 피드백/경쟁 분석/전략적 요구) | 제품 책임자 |
| **2단계: 이터레이션 보드** | `[버전] - [목표 설명]` | 이번 이터레이션에서 확정된 요구사항 + 필수 수정 버그 | 이터레이션 오너 |

**플로우 로직:**

```
마스터 백로그(모든 출처의 이슈)
    ↓ 리뷰 후 솔루션화 가능 판단
솔루션화된 요구사항
    ↓ 스프린트 플래닝에서 결정
특정 이터레이션에 투입
    ↓ 이터레이션 내 개발
출시 / 백로그로 반환 / 다음 이터레이션으로 연기
```

### 2.2 이슈 입력 템플릿(범용 포맷)

> **핵심 원칙:** 운영에서 올리는 모든 이슈에는 반드시 재현 정보가 포함되어야 합니다. 재현할 수 없으면 개발이 고칠 수 없습니다.

| 필드 | 설명 | 예시 |
|------|------|------|
| **이슈 제목** | 문제의 간결한 설명 | "iOS 버전 차트 결과가 Android와 불일치" |
| **소스 채널** | 사용자 피드백 / 경쟁 / 전략 | 사용자 피드백 |
| **신고 횟수** | 같은 문제가 몇 회 신고되었는지 | 3회 이상 → severe 태그 부여 |
| **시스템/플랫폼** | Web / iOS / Android / 셀프호스트 | iOS |
| **버전** | 구체적인 버전 번호 | v2.3.1 |
| **재현 단계** | 재현 가능해야 수정 가능 | 1. 시너스트리 열기 2. XX 선택 3. 결과 확인 |
| **기대 동작** | 사용자가 기대하는 것 | 양 플랫폼에서 결과가 일치해야 함 |
| **실제 동작** | 현재 실제로 일어나는 것 | iOS의 표시 순서가 Android와 반대 |
| **스크린샷/녹화** | 증거 첨부 | [첨부파일] |
| **영향 평가** | 경험/결제에 미치는 영향 | 1명의 사용자가 환불 요청 |
| **태그** | Bug / 기능 요청 / UX 개선 | Bug |
| **우선순위 제안** | 운영 측 판단(높음/중간/낮음) | 높음 |

> **운영 특별 규칙:** 같은 이슈가 **3회 이상** 신고되면 자동으로 `severe` 태그가 붙고, 직근 이터레이션에서 반드시 해결해야 합니다.

### 2.3 Bug vs 기능 요청 분류

| 유형 | 정의 | 처리 방법 |
|------|------|----------|
| **Bug** | 기존 기능이 기대와 다르게 동작 | Bug 보드에 직접 투입. 개발이 공수 산정 후 수정 |
| **기능 요청** | 사용자가 새 기능을 원함 | 마스터 백로그로 이동. 제품이 스펙을 작성한 뒤 이터레이션 투입 |
| **UX 개선** | 핵심 기능은 정상이지만 경험이 나쁨 | Bug와 동일 레벨. 운영이 우선순위를 높일 수 있음 |

## 3. 이터레이션 케이던스 설계

### 3.1 이터레이션 기간 선택

| 제품 단계 | 권장 주기 | 근거 |
|----------|----------|------|
| **초기 PMF 검증** | 1~2주 | 빠른 페이스로 신속 실험 |
| **성장기** | 3~4주 | 속도와 안정성의 균형 |
| **안정기** | 6~8주 | 안정성 우선, 대형 릴리스 대응 가능 |

> ⚠️ **참고:** 현대 웹 개발에서는 **10일~2주**로 압축을 권장합니다. 이터레이션이 너무 길면 긴박감이 사라집니다.

### 3.2 10일 이터레이션 표준 프로세스

| 일차 | 개발 액션 | 운영 액션 | 산출물 |
|------|----------|----------|--------|
| **Day 1** | 개발 시작 | — | — |
| **Day 6** | **테스트 가능 빌드** 제출 | 빌드 수령, **반나절 내 테스트 완료** | 테스트 리포트(버그 목록 + 우선순위) |
| **Day 7** | 우선순위에 따라 P0/P1 수정 | 운영×개발 **얼라인먼트 콜**(20~30분) | 이터레이션 스코프 확정 |
| **Day 8** | 수정 계속 + 2차 테스트 빌드 제출 | 2차 테스트(필요 시) | — |
| **Day 9** | 최종 수정 | **릴리스 자료** 준비(스크린샷/웹사이트/SNS) | 자료 준비 완료 |
| **Day 10** | **프로덕션 출시** | 모든 채널에 콘텐츠 배포 | 릴리스 완료 |

### 3.3 릴리스 케이던스 레드라인

> **실전 교훈에서:** 절대로 금요일이나 주말에 릴리스하지 마세요.

## 4. 3자 얼라인먼트 미팅 체계

### 4.1 데일리 스탠드업(매일, 15분)

**참석자:** 제품·개발·운영 전원

- 서서 진행. 15분 엄수.
- 각자 3가지만 말함: ①어제 한 것 ②오늘 할 것 ③블로커 유무
- **운영은 반드시 보고:** 전일 사용자 불만 핫스팟, 새로 발견된 고빈도 버그
- **1시간 넘는 미팅은 누군가 딴짓하고 있다는 증거**

### 4.2 이터레이션 플래닝(각 이터레이션 시작 시, 30~60분)

**참석자:** 제품·개발·운영 각 책임자

```
의제1: 전 이터레이션 회고(20분)
  - 핵심 지표 달성 현황(목표 대비)
  - 완료되지 않은 것은? 이유는?
  - 사용자 피드백이 어디에 집중되어 있는가?

의제2: 이번 이터레이션 목표 설정(20분)
  - 운영 측: 이번에 반드시 해결할 문제(비즈니스 목표에 연결)
  - 제품 측: 이번에 밀 신기능/최적화
  - 개발 측: 기술 부채/리팩토링 요구
  - 공동 결정: 이번 이터레이션의 Top 3 목표

의제3: 리소스 및 일정 확인(20분)
  - 각 측에서 몇 인일 투입?
  - 어떤 기능이 확정? 어떤 것이 홀드?
  - 리팩토링이 필요한 모듈이 있는가?
```

### 4.3 이터레이션 리뷰(각 이터레이션 종료 시, 30분)

**핵심 질문:**
1. 이번 이터레이션의 **목표 지표**를 달성했는가?
2. 미수정 P0 버그가 있는가? 릴리스를 블로킹하는가?
3. 다음 이터레이션으로 **연기하는** 기능은?
4. 다음 이터레이션의 운영 중점은?

### 4.4 미팅 빈도 개요

| 미팅 | 빈도 | 시간 | 목적 |
|------|------|------|------|
| **데일리 스탠드업** | 매일 | 15분 | 진행 동기화, 블로커 발견 |
| **이터레이션 플래닝** | 이터레이션 시작 시 | 30~60분 | 이번에 무엇을 만들지 결정 |
| **이터레이션 리뷰** | 이터레이션 종료 시 | 30분 | 회고 + 다음 준비 |
| **월간 리뷰** | 매월 | 60~90분 | 핵심 지표 리뷰 + 방향 조정 |

## 5. 협업에서 운영의 역할

### 5.1 운영은 사용자와 개발 사이의 "번역자"

| 책임 | 구체적 액션 | 빈도 |
|------|-----------|------|
| **사용자 목소리 수집** | 사용자 피드백 집계, 고빈도 이슈 추출, Bug/기능으로 분류 | 매일 |
| **Bug 사전 스크리닝** | 각 이슈 재현 확인, 재현 단계 보완 | 매일 |
| **우선순위 제안** | UX와 상업화 관점에서 우선순위 부여 | 매 이터레이션 |
| **버전 수락 테스트** | Day 6부터 시작, **반나절 내 테스트 완료** + 리포트 작성 | 매 이터레이션 |
| **릴리스 자료 준비** | App Store 업데이트, 웹사이트 업데이트, SNS 콘텐츠 | 매 이터레이션 |

### 5.2 운영의 릴리스 거부권

> **발동 조건:** 릴리스 전 **사용자 경험이나 결제 전환에 영향을 미치는** P0 버그가 발견되었고 미수정 상태라면, 운영은 **릴리스 연기**를 제안할 수 있습니다.

**P0 버그 기준(운영 측 판단):**

| 버그 유형 | 예시 | P0? |
|----------|------|-----|
| 결제 관련 | 결제 실패, 이중 청구 | ✅ P0 |
| 핵심 기능 사용 불가 | 가입 불가, 결과 생성 불가 | ✅ P0 |
| 데이터 오류 | 차트 결과가 다른 플랫폼과 다름 | ✅ P0 |
| UI 깨짐 but 기능은 정상 | 버튼 위치 틀어짐 | ⚠️ P1 |
| 카피/번역 오류 | 영어 오타 | ❌ P2 |

## 6. 사용자 피드백으로 제품 이터레이션 추진

### 6.1 피드백 → 최적화 클로즈드 루프

```
사용자가 제품 사용 → 문제/제안 발생
        ↓
운영이 수집(정리/분류/사전 스크리닝)
        ↓
마스터 보드에 이슈 등록(재현 단계 포함)
        ↓
같은 문제 3회 이상 신고 → severe 태그 부여
        ↓
이터레이션 플래닝에서 우선 의제로 논의
        ↓
이터레이션 개발에 투입
        ↓
출시 후 운영이 수정 검증
        ↓
사용자에게 문제 해결 알림(소중히 여긴다는 느낌 강화)
```

### 6.2 베타 사용자 인터뷰 체계

| 단계 | 시점 | 액션 | 담당자 |
|------|------|------|--------|
| **사용자 선정** | 베타 시작 1주 전 | 활성 사용자 중 10~15명 선발 | 운영 |
| **테스트 계정 배포** | 베타 시작 3일 전 | 테스트 계정 + 크레딧 지급 | 운영 |
| **사용자 셀프 테스트** | 베타 기간 중 | 실제 필요에 기반하여 제품 사용 | 사용자 |
| **심층 인터뷰** | 베타 기간 중 | 1세션 30~40분 | 운영 |
| **일일 요약** | 매일 종료 후 | 고빈도 버그 + 핵심 니즈 정리 | 운영 |
| **주간 리뷰** | 매주 | 릴리스 가능 여부 판단 | 제품+개발+운영 |

**인터뷰 구성(30~40분):**

| 시간 | 모듈 | 핵심 질문 |
|------|------|----------|
| 5분 | 사용자 배경 | 평소 무엇을 하시나요? 어떤 도구를 쓰시나요? 가장 큰 어려움은? |
| 20분 | 제품 테스트 리뷰 | 어떤 단계가 가장 매끄러웠나요? 어디서 막혔나요? 버그가 있었나요? |
| 10분 | 개선 제안 | 가장 먼저 개선되었으면 하는 것은? |
| 5분 | 마무리 | 계속 참여하실 의향이 있으신가요? 다른 분께 추천하실 의향이 있으신가요? |

> **핵심 원칙:** 추상적인 감상은 논의하지 않습니다. 실제 사용 경험만 논의합니다.

## 7. 핵심 지표 체계

### 7.1 이터레이션별 목표 지표 템플릿

```
이번 이터레이션 목표: [구체적 설명]
- 기준값: XX%
- 목표값: XX%
- 검증 방법: [데이터 소스 / 쿼리 방법]
- 담당자: 제품 / 운영 / 개발
```

### 7.2 상업화 퍼널 지표

| 단계 | 지표 | 설명 |
|------|------|------|
| **획득** | 신규 가입 사용자 수 | 채널별 분리(오가닉/SNS/유료/KOL) |
| **활성화** | 온보딩 완료율 | 모듈별: 정보 입력/기능 체험/결과 확인 |
| **유지** | 1일/7일/30일 리텐션율 | 사용자 소스별 분리 |
| **수익화** | 가입→결제 전환율 | 채널/사용자 속성별 분리 |
| **추천** | K팩터(NPS/추천 의향) | 초대 메커니즘 참여율 |

### 7.3 이터레이션 건강도 셀프 체크

| 점검 항목 | 건강 값 | 경고 신호 |
|----------|---------|----------|
| P0 버그 평균 수정 시간 | ≤2일 | > 3일 |
| 기능 정시 출시율 | ≥80% | <60% |
| 사용자 피드백 평균 응답 시간 | ≤24시간 | > 48시간 |
| 동일 버그 재신고 횟수 | ≤2회 | ≥3회 |
| 테스트 리포트 기한 내 제출 | Day 6.5까지 100% | 반나절 이상 지연 |

## 8. 기술 부채 및 제품 리팩토링 관리

### 8.1 리팩토링을 발동하는 신호

| 신호 | 설명 |
|------|------|
| 동일 모듈에서 반복적으로 버그 발생(관련 이슈 ≥5건) | 해당 모듈에 근본적인 설계 문제 |
| 신기능 개발 비용이 계속 상승 | 기반 아키텍처가 확장성을 제한 |
| 크로스 플랫폼 데이터 불일치 | 엔티티 추상화가 통일되지 않음 |
| 기술 스택이 낡음 | 스택 업그레이드 필요 |

### 8.2 리소스 배분 가이드라인

| 유형 | 권장 비율 | 비고 |
|------|----------|------|
| **신기능 개발** | 50~60% | 상업화 목표에 직접 기여 |
| **버그 수정** | 20~30% | 기본 경험 보장 |
| **기술 부채/리팩토링** | 20~30% | 장기 지속가능성 보장 |

### 8.3 리팩토링 작업 처리

- 리팩토링은 **별도 프로젝트**로 관리, 1~3개월 타임라인
- 리팩토링 기간 중에도 **일반 이터레이션 릴리스를 블로킹하지 않음**(2개 트랙 병행)
- 리팩토링 완료 후 **전체 리그레션 테스트** 필수(코어 사용자 3~5명 참여 권장)
