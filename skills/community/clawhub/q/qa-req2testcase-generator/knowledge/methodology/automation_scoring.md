# 自动化适用性6维度评分模型

> 适用范围：评估券商业务测试用例是否适合自动化，提供量化决策依据，避免"为了自动化而自动化"。

## 1. 评分维度总览

| 维度 | 权重 | 核心问题 |
|------|------|---------|
| D1-稳定性 | 20% | 被测功能是否稳定，近期是否频繁变更？ |
| D2-可重复性 | 15% | 用例能否在不同环境/时间重复执行并得到一致结果？ |
| D3-数据可控 | 20% | 测试数据能否自动准备和清理？ |
| D4-步骤明确 | 15% | 操作步骤和预期结果是否可精确描述？ |
| D5-执行频率 | 15% | 该用例需要多频繁执行？ |
| D6-人工成本 | 15% | 手工执行该用例的时间和复杂度如何？ |

---

## 2. 各维度详细评分标准

### D1 稳定性

#### 定义

衡量被测功能模块的变更频率和需求稳定程度。功能越稳定，自动化脚本的维护成本越低，投资回报率越高。

#### 适用场景

评估一个功能模块是否值得投入自动化。频繁变更的功能自动化后维护成本高，可能得不偿失。

#### 评分标准

| 分数 | 标准 | 说明 |
|------|------|------|
| 5分 | 核心稳定功能 | 近6个月无需求变更，如登录认证、基础行情查询 |
| 4分 | 基本稳定 | 近3个月无变更，偶尔有小调整，如账户查询 |
| 3分 | 中等稳定 | 每1-2个月有小幅调整，如委托下单流程 |
| 2分 | 较不稳定 | 每月有变更，如新上线的理财产品模块 |
| 1分 | 频繁变更 | 每周都在调整，如正在迭代的新功能 |

#### 券商业务示例

| 功能模块 | 评分 | 理由 |
|---------|------|------|
| 用户登录/认证 | 5 | 核心基础功能，接口稳定，极少变更 |
| 股票行情查询 | 5 | 标准化接口，数据格式固定 |
| 普通限价委托下单 | 4 | 核心交易功能，偶尔有规则微调 |
| 新股申购 | 3 | 每年规则可能调整，但流程基本固定 |
| 智能选股推荐 | 1 | 算法频繁迭代，UI和逻辑持续变化 |

### D2 可重复性

#### 定义

衡量测试用例能否在不同时间、不同环境下重复执行，且每次执行结果一致（不受外部因素干扰）。

#### 适用场景

判断用例是否依赖特定时间窗口、外部系统状态或人工判断，这些因素会导致自动化执行不稳定。

#### 评分标准

| 分数 | 标准 | 说明 |
|------|------|------|
| 5分 | 完全可重复 | 任何时间执行结果一致，无外部依赖 |
| 4分 | 基本可重复 | 需简单前置条件（如登录态），但可自动化准备 |
| 3分 | 有条件可重复 | 依赖特定数据状态，需数据准备脚本 |
| 2分 | 较难重复 | 依赖交易时段、外部行情等实时数据 |
| 1分 | 不可重复 | 依赖人工判断、视觉验证或一次性操作 |

#### 券商业务示例

| 功能模块 | 评分 | 理由 |
|---------|------|------|
| 账户余额查询 | 5 | 任何时间可查，结果确定 |
| 历史委托查询 | 5 | 查询历史数据，结果固定 |
| 限价委托下单 | 3 | 需在交易时段执行，或需模拟撮合环境 |
| 打新中签查询 | 2 | 依赖特定申购周期和中签结果 |
| K线形态识别 | 1 | 需人工判断图形是否正确 |

### D3 数据可控

#### 定义

衡量测试所需数据能否通过自动化手段准备（构造）和清理（回滚），而不依赖手工操作或生产数据。

#### 适用场景

评估自动化测试的数据准备难度。数据越可控，自动化脚本越独立，越不容易因数据问题导致误报。

#### 评分标准

| 分数 | 标准 | 说明 |
|------|------|------|
| 5分 | 完全可控 | 可通过API/SQL自动构造和清理所有测试数据 |
| 4分 | 大部分可控 | 核心数据可自动准备，少量需预置 |
| 3分 | 部分可控 | 需要预置基础数据（如账户、持仓），但可脚本化 |
| 2分 | 较难控制 | 依赖第三方数据（如实时行情、银行回调） |
| 1分 | 不可控 | 依赖生产数据或无法模拟的外部系统 |

#### 券商业务示例

| 功能模块 | 评分 | 理由 |
|---------|------|------|
| 用户注册/开户 | 5 | 可通过API创建测试账户，测试后清理 |
| 委托下单（模拟撮合） | 4 | 模拟环境可控，但需预置账户和资金 |
| 银证转账 | 2 | 依赖银行侧回调，需Mock或联调环境 |
| 实时行情推送 | 2 | 依赖交易所实时数据，需行情模拟器 |
| 年度对账单生成 | 1 | 需要一整年的交易数据积累 |

### D4 步骤明确

#### 定义

衡量测试用例的操作步骤和预期结果是否可以精确、无歧义地描述为可执行的自动化脚本。

#### 适用场景

判断用例是否可以转化为确定性的自动化脚本。步骤越明确，脚本编写越简单，维护成本越低。

#### 评分标准

| 分数 | 标准 | 说明 |
|------|------|------|
| 5分 | 完全明确 | 每步操作和预期结果都可精确编码，无主观判断 |
| 4分 | 基本明确 | 大部分步骤明确，少量需参数化处理 |
| 3分 | 中等明确 | 步骤明确但预期结果需动态计算 |
| 2分 | 较模糊 | 部分步骤依赖上下文判断或条件分支复杂 |
| 1分 | 不明确 | 需要人工判断（如"界面是否美观"、"响应是否流畅"） |

#### 券商业务示例

| 功能模块 | 评分 | 理由 |
|---------|------|------|
| 接口字段校验 | 5 | 输入→输出完全确定，断言明确 |
| 委托下单+查询 | 5 | 步骤固定：POST下单→GET查询→断言字段 |
| 手续费计算验证 | 3 | 步骤明确但费率规则复杂，需动态计算预期值 |
| 交易报表格式验证 | 2 | 需验证PDF/Excel格式，部分需视觉比对 |
| 用户体验评估 | 1 | 主观判断，无法自动化 |

### D5 执行频率

#### 定义

衡量该测试用例在项目周期中需要执行的频率。执行频率越高，自动化的投资回报率越高。

#### 适用场景

评估自动化投入的性价比。高频执行的用例自动化后节省的人力最多。

#### 评分标准

| 分数 | 标准 | 说明 |
|------|------|------|
| 5分 | 每日执行 | 每次构建/部署都需要执行（冒烟测试、回归测试） |
| 4分 | 每周执行 | 每个迭代/Sprint都需要执行 |
| 3分 | 每月执行 | 每个版本发布前执行 |
| 2分 | 季度执行 | 重大版本或季度回归时执行 |
| 1分 | 一次性 | 仅在特定场景执行一次（如数据迁移验证） |

#### 券商业务示例

| 功能模块 | 评分 | 理由 |
|---------|------|------|
| 登录接口冒烟测试 | 5 | 每次部署必须验证 |
| 核心交易链路回归 | 5 | 每日CI/CD流水线执行 |
| 新股申购功能验证 | 3 | 每月有新股发行时验证 |
| 年报生成功能 | 2 | 每年年底集中验证 |
| 系统迁移数据校验 | 1 | 迁移完成后执行一次 |

### D6 人工成本

#### 定义

衡量手工执行该测试用例所需的时间、复杂度和出错概率。人工成本越高，自动化替代的价值越大。

#### 适用场景

评估自动化对人力的释放效果。耗时长、步骤多、易出错的用例最适合自动化。

#### 评分标准

| 分数 | 标准 | 说明 |
|------|------|------|
| 5分 | 极高成本 | 手工执行>30分钟，步骤>20步，易出错 |
| 4分 | 高成本 | 手工执行15-30分钟，步骤10-20步 |
| 3分 | 中等成本 | 手工执行5-15分钟，步骤5-10步 |
| 2分 | 低成本 | 手工执行<5分钟，步骤<5步 |
| 1分 | 极低成本 | 手工执行<1分钟，1-2步操作 |

#### 券商业务示例

| 功能模块 | 评分 | 理由 |
|---------|------|------|
| 全品种委托回归（沪A/深A/北交所/港股通/期权） | 5 | 手工需逐个品种下单验证，>1小时 |
| 多账户并发下单验证 | 5 | 需同时操作多个账户，手工无法真正并发 |
| 单接口字段校验 | 2 | Postman手工测试几分钟即可 |
| 登录功能验证 | 1 | 输入账号密码点击登录，<1分钟 |

## 3. 综合评分与决策树

### 评分计算公式

```
总分 = D1×0.20 + D2×0.15 + D3×0.20 + D4×0.15 + D5×0.15 + D6×0.15
```

满分 = 5.0 分

### 决策树

```
总分 ≥ 3.8 → 【强烈建议自动化】
  ├── 优先纳入自动化回归套件
  ├── 建议在当前迭代内完成脚本编写
  └── ROI预估：3个月内回本

3.0 ≤ 总分 < 3.8 → 【建议自动化，需评估ROI】
  ├── 评估低分维度是否可改善（如数据可控性可通过Mock提升）
  ├── 若低分维度可改善 → 改善后自动化
  └── 若低分维度不可改善 → 半自动化（自动化数据准备+手工验证）

2.0 ≤ 总分 < 3.0 → 【暂不建议自动化】
  ├── 保持手工测试
  ├── 关注低分维度变化，定期重新评估
  └── 可考虑辅助工具（如数据生成脚本）降低手工成本

总分 < 2.0 → 【不适合自动化】
  ├── 纯手工测试
  └── 考虑是否可以简化测试方案本身
```

### 决策流程图

```
开始评估
  │
  ▼
[计算6维度加权总分]
  │
  ├── 总分 ≥ 3.8 ──→ 自动化 ──→ 纳入CI/CD回归套件
  │
  ├── 3.0~3.8 ──→ 检查低分维度
  │                  ├── 可改善 → 改善后自动化
  │                  └── 不可改善 → 半自动化
  │
  ├── 2.0~3.0 ──→ 手工测试 + 定期重评
  │
  └── < 2.0 ──→ 纯手工测试
```

## 4. 实战评估示例

### 示例1：股票限价委托下单接口测试

| 维度 | 评分 | 理由 |
|------|------|------|
| D1-稳定性 | 4 | 核心交易功能，偶有规则微调 |
| D2-可重复性 | 3 | 需模拟撮合环境，交易时段限制可通过Mock解决 |
| D3-数据可控 | 4 | 可通过API准备账户和资金，模拟环境可控 |
| D4-步骤明确 | 5 | POST请求→断言响应→DB验证，完全确定 |
| D5-执行频率 | 5 | 每日CI必跑 |
| D6-人工成本 | 3 | 单次手工5-10分钟，但多品种组合耗时长 |

**总分** = 4×0.20 + 3×0.15 + 4×0.20 + 5×0.15 + 5×0.15 + 3×0.15 = 0.80 + 0.45 + 0.80 + 0.75 + 0.75 + 0.45 = **4.00**

**决策：强烈建议自动化** ✅

### 示例2：K线图形态视觉验证

| 维度 | 评分 | 理由 |
|------|------|------|
| D1-稳定性 | 3 | 图表组件偶有UI调整 |
| D2-可重复性 | 2 | 依赖实时行情数据，不同时间结果不同 |
| D3-数据可控 | 2 | 需要特定行情数据构造特定K线形态 |
| D4-步骤明确 | 1 | 需人工判断"K线是否正确显示" |
| D5-执行频率 | 3 | 每个版本发布前验证 |
| D6-人工成本 | 2 | 打开页面目视检查，几分钟 |

**总分** = 3×0.20 + 2×0.15 + 2×0.20 + 1×0.15 + 3×0.15 + 2×0.15 = 0.60 + 0.30 + 0.40 + 0.15 + 0.45 + 0.30 = **2.20**

**决策：暂不建议自动化** ❌（可考虑截图对比工具辅助）

### 示例3：银证转账全流程

| 维度 | 评分 | 理由 |
|------|------|------|
| D1-稳定性 | 4 | 银证转账流程稳定 |
| D2-可重复性 | 3 | 需银行侧配合，可通过Mock解决 |
| D3-数据可控 | 2 | 依赖银行回调，需Mock银行接口 |
| D4-步骤明确 | 4 | 流程明确，但需处理异步回调 |
| D5-执行频率 | 4 | 每周回归 |
| D6-人工成本 | 4 | 手工需登录银行端确认，15-20分钟 |

**总分** = 4×0.20 + 3×0.15 + 2×0.20 + 4×0.15 + 4×0.15 + 4×0.15 = 0.80 + 0.45 + 0.40 + 0.60 + 0.60 + 0.60 = **3.45**

**决策：建议自动化，需先搭建银行Mock服务提升D3评分** → 改善后预估总分可达3.85 ✅
