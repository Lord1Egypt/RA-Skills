# 决策清单 — Harness 闭环系统的硬性追问维度

> 用途：Layer 3 Socratic 追问阶段，AI 按本清单逐条对照用户系统类型补齐决策。
> 加载时机：Layer 0 验收通过、进入 Socratic 之前。
> 调用规则：①按系统类型选 A / B / C；②在每节内严格按编号顺序问，不跳序；③每节问完做一句小复述（"这块锁了 X / Y / Z，对齐 HP-N"）；④全部问完后再补"通用维度"。

---

## 系统类型选择

| 类型 | 判定特征 | 必须对齐的 HP |
|---|---|---|
| **A** AI Agent 自优化循环（Ralph / Harness 类） | 系统自动评测 + 自动改进 + 人类掌舵 | 全部 HP-1 ~ HP-7 |
| **B** 评测驱动改进（无完整闭环） | 已有 AI 系统，加评测 + 改进，但改进仍由人触发 | HP-1 / HP-7 强制；HP-3 / HP-4 / HP-5 弱化 |
| **C** 人机协作治理（Human-in-the-Loop Governance） | AI 决策 + 关键点人工审批 | HP-2 / HP-6 / HP-7 是绝对核心 |

---

## 类型 A · AI Agent 自优化循环

### A.0 方法论与演化资产
- [ ] 参考方法论锚点是什么？（OpenAI Harness / LangGraph / AutoGPT / 其他）
- [ ] 循环演化的对象是什么？（能力资产 / 模型参数 / Prompt 库 / 知识库 / Skill 目录）
- [ ] 输入原料是什么？（工单 / 日志 / 用户反馈 / 代码提交）

### A.1 · 对齐 HP-1（Eval is foundation）· 评测层

> 不问"要不要评测"，只问"评测从哪来"。

- [ ] **Gold Set 来源**：A. 人工标注 / B. 规则自动提名 / C. 历史高质量样本（必选）
- [ ] **冷启动规模**：最低多少条才能启动循环？
- [ ] 评测频率：每次变更 / 每日 / 每周
- [ ] 打分方式：LLM-as-judge / 规则 / 人工 / 混合
- [ ] 通过阈值：硬门禁 / 软评估
- [ ] LLM-judge 稳定性保证：多次打分 + 方差阈值
- [ ] 评测集更新机制：如何避免过拟合评测集本身

**红线**：用户说"先上线，评测集以后再补" → 拦住，违反 HP-1。

### A.2 · 对齐 HP-2（Gates not code）· 治理闸门层

> 不问"需不需要人审"，只问"每级闸门的边界与升级路径"。

- [ ] **闸门分级**（典型：L1 自动 / L2 Owner / L3 Manager / 红线熔断）
- [ ] 每级闸门的审批边界：什么类型的变更归哪一级
- [ ] 每级闸门的超时策略：多久没人审自动做什么（拒绝 / 降级 / 自动过）
- [ ] 升级路径：未通过 → 升到哪个人工队列；再未通过 → 再升到哪
- [ ] 审批入口：企微 / 邮件 / Dashboard / CLI
- [ ] 审计留痕：谁在什么时间做了什么决定，能追溯多久

**红线**：任何让人类"亲自下场改代码 / Prompt / 数据"的决策 → 违反 HP-2，建议改为"加 / 调 / 审闸门"。

### A.3 · 对齐 HP-3（Idempotent & resumable）· 循环执行层

> 不问"要不要幂等"，只问"每一步如何幂等"。

- [ ] 循环每一步是否幂等（重跑结果一致）
- [ ] 中间态存哪：断点文件 / DB / 对象存储
- [ ] 来源数据是否有稳定的 ID（避免同一条输入被当成两条）
- [ ] 断点恢复机制：从哪一步续跑 / 需要哪些上下文
- [ ] 并发冲突避免：锁 / 乐观并发 / 单例执行

**红线**：用户说"每次跑结果不一样也行" → 拦住，追问不可复现 / 无法调试 / 无法回滚的代价。

### A.4 · 对齐 HP-4（Small reversible steps）· 改进器层

> 不问"改进器能做什么"，只问"改进粒度 / 观察期 / 回滚条件"。

- [ ] **改进粒度**：一条 Prompt / 一个子模块 / 整条 Skill（越小越好）
- [ ] **观察期**：改完立即生效 vs 灰度 N 天再全量
- [ ] **自动回滚触发条件**：评测分跌 X% / 红线事件 / 人工一键
- [ ] 改进器档位：L1 改措辞 / L2 改结构 / L3 改能力
- [ ] 最多尝试次数
- [ ] 禁区清单：哪些字段 / 文件 / 模块不允许自动改

**红线**：用户说"改完直接上线" → 违反 HP-4，讨论弱一致性 / 影响爆炸半径。

### A.5 · 对齐 HP-5（Automation tiers with escalation）· 分级自动化

> 不问"要不要自动化"，只问"分几档 + 档间如何升级"。

- [ ] 自动化分几级（典型：L1 全自动 / L2 Owner 必审 / L3 Manager 必审）
- [ ] 每级的权限边界：哪些动作归 L1，哪些必须升到 L2
- [ ] 升级触发条件：哪些事件 / 异常 / 阈值触发从 L1 升到 L2/L3
- [ ] 降档触发条件：什么情况允许从 L3 降到 L2，避免永久升档

### A.6 · 对齐 HP-6（Asset versioning）· 资产版本化

> 不问"要不要版本化"，只问"版本化策略是什么"。

- [ ] 演化资产的版本化方式：git / 数据库版本表 / 对象存储快照
- [ ] 版本元数据：时间 / 改动者 / 触发事件 / 前后 diff
- [ ] 保留时长：90 天 / 6 个月 / 永久
- [ ] 一致性约束：Skill / Prompt / Knowledge 的版本是否要对齐
- [ ] 回滚粒度：单资产 / 整个快照

### A.7 · 对齐 HP-7（Human time budget）· 人类参与预算

> 不问"要不要人类参与"，先列预算，再看能支撑多少自动化。

- [ ] **管理者每周可投入小时数**（先列预算）
- [ ] **Owner 每天可投入小时数**（先列预算）
- [ ] 标注人力 / 审批人力（先列预算）
- [ ] 预期 ESCALATE 次数 / 周
- [ ] 预期红线事件次数 / 周
- [ ] **算账**：当前设计下预算够吗？
- [ ] 周报 / 日报频率
- [ ] 告警渠道：企微 / 邮件 / Dashboard

**红线**：预算算下来不够 → 必须回头降自动化 / 提高闸门阈值 / 引入更多角色，**禁止默认"人不够就加班"**。

### A.8 红线层（系统级安全边界）
- [ ] 红线分级：critical / warning / behavior
- [ ] 红线来源：人工定义 / 自动挖掘 / 人工录入
- [ ] 红线命中动作：熔断 / 告警 / 降级
- [ ] 红线治理权限：谁能加 / 谁能删 / 谁能豁免

### A.9 熔断层
- [ ] 哪些故障需要熔断
- [ ] 哪些故障不熔断（交给现有重试机制）
- [ ] 恢复方式：自动 / 手动
- [ ] 熔断粒度：单个组件 / 整个循环

### A.10 交付层
- [ ] MVP 范围
- [ ] MVP 时间线
- [ ] 验收标志（可观测、二元判定）
- [ ] V1.1 / V2 路线

---

## 类型 B · 评测驱动改进（无完整闭环）

> 与类型 A 大量重叠，但**改进不自动**，HP-3 / HP-4 / HP-5 强制程度降低；核心仍是 HP-1 与 HP-7。

### B.1 评测设计（同 A.1，对齐 HP-1）
- [ ] 评测什么：准确率 / 覆盖率 / 时延 / 成本 / 用户满意度
- [ ] 评测集规模：冷启动数量 / 稳态数量
- [ ] 评测集更新频率
- [ ] 评测集与线上数据的分布一致性如何保证

### B.2 改进触发
- [ ] 触发条件：评测分下降 / 达到周期 / 人工发起
- [ ] 改进执行人：AI agent / 工程师 / 混合
- [ ] 改进边界
- [ ] 改进后影响评估机制

### B.3 对照组
- [ ] 是否需要 A/B 测试
- [ ] 影子对比（shadow run）
- [ ] 基线版本如何保留（与 HP-6 对齐）

### B.4 人类预算（同 A.7，对齐 HP-7）
- [ ] 改进工程师每周投入
- [ ] 评测标注人力
- [ ] 评审会议频率

---

## 类型 C · 人机协作治理（Human-in-the-Loop Governance）

> AI 决策 + 关键点人工审批。**HP-2 是绝对核心**。

### C.1 · 对齐 HP-2 · 角色与权限
- [ ] 角色分类：auto / owner / manager / admin
- [ ] 角色之间的权限边界
- [ ] 同一人可身兼多角色吗？如何避免"自审自批"

### C.2 · 对齐 HP-2 · 审批矩阵
- [ ] 什么类型的变更走哪类角色
- [ ] 自动合并的边界
- [ ] 升级路径
- [ ] 超时策略

### C.3 · 对齐 HP-6 · 通知与追溯
- [ ] 审批请求送达方式
- [ ] 审批结果记录方式
- [ ] 审计链路 + 保存时长

### C.4 紧急情况
- [ ] 紧急审批通道
- [ ] 绕过审批的条件（绕过必须留痕并事后复盘）
- [ ] 红线事件处理流程

### C.5 · 对齐 HP-7 · 人类预算
- [ ] 每类审批人每周可投入小时数
- [ ] 预期审批数量 / 周
- [ ] 预算够吗

---

## 通用维度（A / B / C 都要补齐）

### 成本
- [ ] LLM API 单日成本预估
- [ ] 人力成本（审批 / 维护 / 标注）
- [ ] 基础设施成本
- [ ] 开发时间成本
- [ ] 预期 ROI / 回收周期

### 风险
- [ ] 技术风险：限流 / 不稳定 / 依赖故障
- [ ] 业务风险：错误建议 / 误导用户
- [ ] 合规风险：数据脱敏 / 隐私
- [ ] 组织风险：Owner 变动 / 团队调整

### 命名（每一项给 2–3 个候选让用户选，禁止擅自决定）
- [ ] 系统名
- [ ] 仓库名
- [ ] 核心组件名
- [ ] 关键抽象概念名

## 提问格式硬约束

| 项 | 必须 |
|---|---|
| 候选数量 | A/B/C/D 共 3–5 个 |
| 推荐 | 必须给出明确倾向 + 一句理由 |
| 退出口 | 永远保留"以上都不是，我选 D：……" |
| 顺序 | 严格按 A.0 → A.10 / B.1 → B.4 / C.1 → C.5 |
| 复述 | 每节问完，一句话锁定该节决策 + 标注对齐的 HP |
| HP 引用 | 任何决策违反 HP-1~HP-7 时，必须显式指出（"这违反 HP-N，因为 ……"） |
