# Link Type — 知识图谱的"骨架"，不是简单的外键

**设计原则：语义优先**——先想"这个关系叫什么"，再想"数据怎么存"。

## 8个配置项

| 配置项 | 说明 | 注意 |
|-------|------|------|
| API Name | `snake_case`，如 `employee_participates_in_project` | 一旦定义不建议修改 |
| 基数约束 | ONE_TO_ONE / ONE_TO_MANY / MANY_TO_MANY | 也是性能优化依据 |
| 方向性 | DIRECTED / UNDIRECTED / BIDIRECTED | 决定遍历方向 |
| 左/右对象类型 | 关系两端的 Object Type | 同一对对象可有多种链接 |
| 左/右展示名称 | 面向用户的语义描述 | 提升可读性 |
| 数据来源 | 外键映射 / 关联表 / 跨数据集推断 | 见下方 |
| 安全控制 | 链接级独立 ACL | 可独立于两端对象 |
| 属性 | 链接自身携带的信息（如角色、开始日期） | 属于链接的属性不要放到对象上 |

## 三种数据实现方式

- **外键映射**（80% 场景）：直接映射数据库外键，性能最高。适用：1:N、1:1，无需链接属性
- **关联表**（多对多必备）：适用：M:N，或链接需要携带属性
- **跨数据集推断**：源数据无明确外键，需要复杂逻辑推断关系

## 图遍历性能基准

| 场景 | 典型性能 | 建议 |
|------|---------|------|
| 单对象沿 1:N 链接 | < 50ms | 正常使用 |
| 单对象沿 M:N 链接 | 100ms~500ms | 必须加分页 |
| 多跳遍历（2-3跳） | 1s~5s | 用 Function 缓存中间结果 |
| 全图遍历 | 分钟级 | 离线预计算 |

## 常见错误

| 错误 | 正确做法 |
|------|---------|
| 直接映射外键为链接 | 先想语义，再考虑数据实现 |
| M:N 链接无限制 fetchAll | 必须加分页 |
| 把链接属性放到对象上 | 角色、开始日期等属于链接本身 |
