# 质量门禁

> 每个阶段必须通过对应门禁才能进入下一阶段。

---

## 阶段 1 → 2：规划门禁

- [ ] spec.md 包含目标、范围、验收标准
- [ ] spec.md 明确排除非目标
- [ ] tasks.md 每个任务有描述、输入、输出、验收标准
- [ ] summary.md 已创建并填入基本信息表
- [ ] 变更目录命名符合 `{type}-{name}-{date}` 格式

## 阶段 2 → 3：编码门禁

- [ ] 每个任务都有对应的代码实现
- [ ] 已读取并理解现有代码模式
- [ ] coding_report.md 已产出
- [ ] self_review.md 所有检查项通过

## 阶段 3 → 4：独立评审门禁

- [ ] Evaluator 是独立的 Agent（不是 Generator）
- [ ] code_review.md 问题已分级（MUST FIX / SHOULD FIX / LOW / INFO）
- [ ] 每个问题有位置、描述、当前代码、建议
- [ ] 健康检查/API 测试通过

## 阶段 4 → 5：修复门禁

- [ ] 所有 MUST FIX 问题已修复
- [ ] SHOULD FIX 问题已修复或说明不修复理由
- [ ] revision_report.md 逐条对应评审意见
- [ ] 健康检查再次通过

## 阶段 5 → 交付：交付门禁

- [ ] Evaluator 确认修复通过（状态 APPROVED）
- [ ] git commit 已执行
- [ ] summary.md 状态更新为 DELIVERED
- [ ] 用户已收到交付报告

---

## 问题分级定义

| 级别 | 含义 | 处理要求 |
|------|------|---------|
| **MUST FIX** | 安全漏洞、功能缺陷、数据损坏 | 必须修复，否则不能交付 |
| **SHOULD FIX** | 边界情况、可维护性问题 | 强烈建议修复，不修复需说明理由 |
| **LOW** | 代码风格、小优化 | 建议修复 |
| **INFO** | 观察、建议、未来改进 | 记录即可 |
