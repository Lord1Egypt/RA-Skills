# 约束强度标注规则

> **加载时机**：步骤 2.5、步骤 3、步骤 6（模型思考）时加载

---

## 一、约束强度定义

| 强度 | 标记 | 关键词 | 处理规则 |
|------|------|--------|---------|
| 硬约束 | 🔴 | 必须、只能、截止、指定、强制、不允许、不得 | 聚焦方案严格遵守；发散方案可轻微突破但必须标注风险 |
| 软约束 | 🟡 | 最好、尽量、如果可能、建议、通常、希望 | 聚焦方案遵守；发散方案可尝试其他方法 |
| 无约束 | ⚪ | 未提及或使用随意、大概、大概就行 | 两个方案均可自由处理 |

---

## 二、约束强度自动识别规则

扫描用户任务文本，对每个 5W2H 维度独立判断：

1. 该维度的描述附近是否有 🔴 关键词？→ 🔴 硬约束
2. 否则，是否有 🟡 关键词？→ 🟡 软约束
3. 否则 → ⚪ 无约束

**就近原则**：约束关键词只修饰其直接关联的内容，不跨句传播。

**示例**：
- "本周五前**必须**提交PPT，**尽量**用公司模板"
  → When 维度：🔴 硬约束（"必须" + "本周五前"）
  → How 维度：🟡 软约束（"尽量" + "用公司模板"）
  → What 维度：⚪ 无约束（"PPT" 无约束词）

## 二-B、隐式约束升级规则

> ⚠️ 关键词初判只是第一步。很多约束的**真实强度**被用户的口头弱化表达掩盖了，需要通过上下文推断还原。

### 核心逻辑

```
关键词初判 → 上下文复核 → 可能升级 → 双标注输出
```

### 升级判断依据

初判为 🟡 的约束，检查其内容是否属于以下领域：

| 领域 | 典型场景 | 升级结果 | 理由 |
|------|---------|---------|------|
| **组织规范** | 公司模板、品牌色、审批流程、命名规范 | 🟡🔴 软表达+硬语义 | 组织规范名义上"建议"实则强制违反有后果 |
| **法律合规** | 合同条款、数据保护、财务审计 | 🔴 | 法律领域无"尽量"，应直接升级 |
| **安全底线** | 权限控制、加密要求、操作规程 | 🔴 | 安全无弹性 |
| **协作依赖** | 接口格式、交付物标准、会议时间 | 🟡＋"影响下游"标注 | 违反会影响他人，但不一定不可协商 |
| **个人偏好** | 颜色选择、工具选择、格式偏好 | 🟡（维持） | 纯个人偏好无需升级 |

### 双标注格式

升级后的约束使用**双标注**：`🟡🔴` — 软约束表达 + 硬约束语义

含义：
- 🟡（用户口头说的）：尽量、最好、建议...
- 🔴（上下文推断的）：实际上可能不可违反

### 处理规则

| 标注 | 聚焦方案 | 发散方案 |
|------|---------|---------|
| 🟡🔴 | **按🔴执行**（保守处理，假设不可违反） | 按🟡探索（可以尝试突破，但必须标注风险+提供备用方案） |
| 🟡＋影响下游 | 按🟡遵守，步骤中注明协作影响 | 可协商替代，但需列出受影响方 |
| 🟡（维持） | 按🟡遵守 | 可自由替代 |

### 示例

- "尽量用公司模板" → 初判🟡 → 上下文复核："公司模板"属于组织规范 → **升级为 🟡🔴**
  - 聚焦方案：严格遵守公司模板
  - 发散方案：🌟可尝试其他模板，但标注"风险：违反品牌规范"

- "最好用蓝色" → 初判🟡 → 上下文复核：纯个人偏好 → **维持🟡**
  - 聚焦方案：使用蓝色
  - 发散方案：可自由选色

- "尽量符合数据保护要求" → 初判🟡 → 上下文复核：法律合规 → **升级为🔴**
  - 两个方案都必须严格遵守

## 三、注意力锚定规则

> ⚠️ 在步骤 3（提取块内元素）之前，模型必须先执行以下内部锚定步骤。本步骤不向用户输出，但必须执行。

### 3.1 强制关注点提取

1. **硬约束标记**：扫描文本中的「必须、只能、截止、指定、强制、不允许、不得」，将其后内容标记为 `[CRITICAL]`，任何后续步骤不可违反
2. **核心目标提取**：识别主要任务动词（制作/安排/填写/规划/写）及其直接宾语，标记为 `[CORE]`
3. **实体提取**：提取所有数字、时间、人名/项目名称，标记为 `[ENTITY]`
4. **降权标记**：「比如/例如/像/假如」后面的内容，标记为 `[EXAMPLE]`，后续提取时权重降低，不作为约束
5. **阻力提取**：「但是/不过/担心/怕/难/卡」后面的内容，标记为 `[RESISTANCE]`，在诉求聚合时优先保留
6. **核心句重构**：用一句话（≤20字）重述任务，格式：「[主语] 要 [CORE] 于 [时间/场景]」

### 3.2 自我反查

拆分结果输出前，模型强制执行以下反查：

| 反查 | 检查内容 | 修正动作 |
|------|---------|---------|
| 反查 1 | 硬约束遗漏：所有 `[CRITICAL]` 内容是否体现在拆分结果中？ | 如否 → 补充遗漏 |
| 反查 2 | 约束强度误判：是否有将 🟡 软约束词误标为 🔴？ | 如是 → 修正为 🟡 |
| 反查 2b | 隐式约束升级：是否有 🟡 约束属于组织规范/法律合规/安全底线，应升级为 🟡🔴 或 🔴？ | 如是 → 按升级规则处理 |
| 反查 3 | 举例干扰：是否有 `[EXAMPLE]` 内容被当作核心诉求？ | 如是 → 降级为参考信息 |
| 反查 4 | 5W2H 完整性：各维度是否有明显可推断但缺失的？ | 如是 → 补充 `[推断]` 默认值 |

✅ 全部通过后才进入用户确认环节。

## 四、双方案对约束的处理差异

| 维度属性 | 聚焦方案 | 发散方案 |
|---------|---------|---------|
| 🔴 硬约束 | 严格遵守，不可违反 | 可轻微突破，但必须：1) 标注 🌟 风险警告 2) 提供备用方案 |
| 🟡🔴 软表达+硬语义 | 按🔴执行（假设不可违反） | 按🟡探索，但必须标注风险+提供备用方案 |
| 🟡 软约束 | 遵守（按默认处理） | 可尝试替代方法，注明"如果不用[软约束]会怎样" |
| ⚪ 无约束 | 采用最有效率的处理方式 | 可尝试非常规方式 |

---

## 五、整合时的约束处理规则

发散方案的创新点若涉及约束突破，整合时按约束类型区分处理：

**🔴 硬约束突破**：
- 该突破**风险可控** → 作为 🌟 增强步骤，附风险说明
- 该突破**风险高** → 仅标注为「可选尝试」，不嵌入主步骤

**🟡🔴 软表达+硬语义突破**：
- 整合时**默认按🔴处理**（聚焦方案骨架中不突破）
- 发散方案的创新点可保留为🌟步骤，但必须标注"⚠️ 风险：可能违反[组织规范/品牌要求等]，执行前需确认"
- 同时提供备用步骤（按🔴执行的方案）

## 六、约束清单输出格式

```
【约束清单】
🔴 硬约束（不可违反）：
  - When：本周五前必须提交
  - What：PPT格式

🟡🔴 软表达+硬语义（用户弱化表达，但上下文推断可能不可违反）：
  - How：使用公司模板（"尽量"，但涉及品牌规范）

🟡 软约束（尽量遵守）：
  - （本例无）

⚪ 无约束：
  - Why / Who / Where / How much：未指定
```
