# 行为类型枚举表

## ⚠️ 数据正确性红线（务必先看）

以下行为的"是否触发"判断**必须由服务端完成**，严禁客户端用 `wx.storage` / 本地缓存自判：

| 事件 | 服务端判断依据 | 客户端自判的风险 |
|---|---|---|
| `REGISTER` | openid 是否首次出现在 DB | 清缓存 / 换机 → 同一 openid 反复算成新注册，注册数严重虚高 |
| `RE_ACTIVE` | DB 中 last_login_ts 距今天数 | 清缓存丢判定依据 → 漏报；多设备无法聚合 → 重复报 |
| `LOGIN`（手动场景） | 业务定义的"登录成功" | 客户端事件未必代表登录成功 |

**正确架构**：客户端 `wx.login` → 用 `code` 调业务后端 → 后端按 openid 查 DB
→ 返回 `{ openid, isNew, isReActive, backFlowDay }` → 客户端按字段分流上报。

详见排障文档 [客户端自判 RE_ACTIVE/REGISTER 的反面案例](../排障/客户端自判沉默注册的反面案例.md)。

## 四、自定义行为

当预定义的 action_type 无法满足需求时，可使用自定义行为。命名规范：

- 仅允许字母、数字和下划线
- 必须以字母开头
- 最大长度 64 字符（APP 端）/ 255 字符（小程序/小游戏端）
- 不可使用 `ams_reserved_` 前缀（系统保留）

也可使用 `CUSTOM` + `custom_action` 组合方式：

| 字段 | 值 | 说明 |
|------|---|------|
| `action_type` | `"CUSTOM"` | 固定值 |
| `custom_action` | 自定义字符串 | 最大 128 字节 |

**常用自定义行为**：

| custom_action | 含义 |
|---------------|------|
| `UV_CORE_ACTION` | 关键行为 |

## 五、组合行为说明

| 业务场景 | 可选 action_type |
|---------|-----------------|
| 综合线索收集 | `CLAIM_OFFER` / `ONLINE_CONSULT` / `MAKE_PHONE_CALL` / `RESERVATION` / `SCANCODE` 任选 |
| 7日留存 | `START_APP` + `length_of_stay=7` |
| 1日留存 | `START_APP` + `length_of_stay=1`（iOS 端） |
| 关键行为 | `CUSTOM` + `custom_action="UV_CORE_ACTION"` |
| 回流用户激活 | 任意事件 + `audience_type=1`（iOS 端） |

## 六、通用上报方法

各端语法略有差异，但结构一致：

| 端 | 方法签名 | 示例 |
|----|---------|------|
| Android | `GDTAction.logAction(actionType, actionParam)` | `GDTAction.logAction("PURCHASE", json)` |
| iOS | `[GDTAction logAction:actionName actionParam:dict]` | `[GDTAction logAction:@"PURCHASE" actionParam:@{@"value":@600}]` |
| 鸿蒙 | `logAction(actionType, actionParam)` | `logAction("PURCHASE", {"value": 600})` |
| 小游戏 | `sdk.track(action_type, action_param)` | `sdk.track("PURCHASE", { value: 600 })` |
| 小程序 | `sdk.track(action_type, action_param)` | `sdk.track("PURCHASE", { value: 600 })` |

> 各端专用上报方法请查阅对应端的必报事件文件或接口索引。

## 七、常见误区

| 误区 | 正确做法 |
|------|---------|
| START_APP 和 ACTIVATE_APP 是同一事件 | **不是**。SDK 接入只需关注 START_APP 的上报 |
| 所有端都需要手动上报 START_APP | 小程序/小游戏/鸿蒙已自动采集；Android/iOS 需手动上报 |
| 开启自动采集后还手动上报 START_APP | 会导致重复。自动采集的事件不再需要手动上报 |
| PURCHASE 不传金额 | **必须**在 action_param 中传入 `value` 字段 |
| 用字符串传金额 | `value` 必须是 **number** 类型 |
| 所有事件都用通用方法 | 有专用方法的事件**建议优先使用专用方法** |
| 鸿蒙端有专用上报方法 | 鸿蒙端**仅有通用方法** `logAction()`，无专用方法 |
| SHARE / ADD_TO_WISHLIST 有自动采集就不用手动上报 | 自动采集仅用于校验，**必须手动上报** |
