## 一、Skill概述

### 1.1 定位与价值

**项目全流程管理**是胡田OPC导师体系的**中枢神经系统**，定位为OPC社区所有项目的统一档案管理系统。它不是独立的项目管理工具，而是贯穿所有其他Skill的"总调度台"和"档案馆"——记录每个项目从"发现机会"到"交付退出"的完整旅程。

**一句话定位**：你每一次启动Skill的输入输出，都会被记录在项目档案里，形成可追溯、可考评、可复盘的"游戏化成就系统"。

### 1.2 核心解决的问题

| 痛点 | 解决方案 |
|------|----------|
| 项目进度靠记忆，缺乏全局视图 | 统一项目档案自动追踪，实时感知状态 |
| Skill之间衔接断裂，输入输出丢失 | 定义清楚每个阶段的Skill触发链和数据传递 |
| 做完项目不知道怎么迭代 | 存档+考评体系，每个项目产出量化指标 |
| 多个项目并行时管理混乱 | 项目组合看板，优先级和资源分配一目了然 |
| 没有成就感，做完了就忘了 | 游戏化成就系统，每个里程碑有徽章和评级 |

### 1.3 与其他Skill的关系——"OPC项目全生命周期Skill调用链"

这是本Skill最核心的设计——定义了OPC项目的**标准生命周期阶段**，以及**每个阶段应该调用哪些Skill**：

```
阶段0: 项目孵化（机会发现）
  触发条件: 新机会/新资源出现
  调用Skill: 三会治理系统（私董会立项讨论）
  产出: 项目立项书（ONE-PAGER）

阶段1: 需求明确
  触发条件: 立项通过
  调用Skill: 用户画像与入驻管理
  产出: 用户画像 + 项目需求文档

阶段2: 能力尽调
  触发条件: 需求明确
  调用Skill: 技术尽调 / 投资方尽调 / 文化商业尽调 / 服务尽调（根据项目类型匹配）
  产出: 尽调报告（含评估评分）

阶段3: 沙盘推演
  触发条件: 尽调完成且评估通过
  调用Skill: 沙盘推演
  产出: 推演报告（路径规划 + 风险评估）

阶段4: 商业规划
  触发条件: 推演确认路径
  调用Skill: 商业计划书 + 三会治理系统（理事会决策）
  产出: 完整商业计划书 + 决策记录

阶段5: 资源对接
  触发条件: 商业计划通过
  调用Skill: 股权合作机制 / 获客分包平台 / 供应链对接
  产出: 合作协议 / 分包方案 / 供应商清单

阶段6: 执行交付
  触发条件: 资源到位
  调用Skill: 协作交付机制 + 市场销售渠道对接
  产出: 交付物 + 验收报告

阶段7: 市场推广
  触发条件: 产品/服务交付
  调用Skill: 投资路演 / IP工具包 / 公众号运营 / 视频号运营
  产出: 路演材料 / 营销方案 / 品牌资产

阶段8: 退出复盘
  触发条件: 项目交付完成或终止
  调用Skill: 三会治理系统（复盘评审）
  产出: 项目复盘报告 + 成就解锁
```

## 二、OPC项目档案体系

### 2.1 项目档案结构

每个OPC项目创建一个独立档案文件，存储在 `./胡田-OPC导师-项目管理/项目档案/[项目编号]-[项目名称]/` 目录下。

**档案文件结构：**

```
项目档案/
  └── [项目编号]-[项目名称]/
      ├── README.md                  ← 项目总览（概要、状态、成就）
      ├── 1-立项申请.md              ← 阶段0
      ├── 2-需求文档.md              ← 阶段1
      ├── 3-尽调报告.md              ← 阶段2
      ├── 4-推演记录.md              ← 阶段3
      ├── 5-商业计划书.md            ← 阶段4
      ├── 6-资源对接记录.md          ← 阶段5
      ├── 7-执行日志.md              ← 阶段6
      ├── 8-市场推广记录.md          ← 阶段7
      └── 9-复盘报告.md              ← 阶段8
```

### 2.2 项目编号规则

```
格式：OPC-[年份]-[序号]-[三位缩写]
示例：OPC-2026-001-GMF  （韩国GMF医疗助眠被子项目）
      OPC-2026-002-LXQ  （量旋科技项目）
      OPC-2026-003-FR   （菲瑞药业项目）
```

### 2.3 项目状态标签

| 状态 | 含义 | 动作 |
|------|------|------|
| 🥚 孵化中 | 刚刚发现机会，尚未立项 | 阶段0进行中 |
| 🐣 立项 | 已通过私董会立项 | 阶段0完成 |
| 🔍 尽调中 | 正在做技术/商业尽调 | 阶段2进行中 |
| 🧪 推演中 | 沙盘推演进行中 | 阶段3进行中 |
| 📋 规划中 | 商业计划书撰写中 | 阶段4进行中 |
| 🔗 对接中 | 资源对接进行中 | 阶段5进行中 |
| 🛠 执行中 | 执行交付进行中 | 阶段6进行中 |
| 📢 推广中 | 市场推广进行中 | 阶段7进行中 |

## 三、阶段触发与Skill衔接机制

### 3.1 阶段触发条件矩阵

每个阶段的进入需要满足前置条件，以及触发执行的Skill。

| 阶段 | 触发条件 | 需要调用的Skill | 输入数据 | 输出数据 |
|------|---------|----------------|---------|---------|
| **0-孵化** | 新机会出现 / 用户主动发起 | 三会治理系统（私董会模块） | 机会描述 + 初步资料 | 立项书(ONE-PAGER) |
| **1-需求** | 立项通过 | 用户画像与入驻管理 | 立项书 | 用户画像 + 需求文档 |
| **2-尽调** | 需求明确 | 技能选择器（根据项目类型自动匹配：技术尽调/投资方尽调/文化商业尽调/服务尽调） | 需求文档 + 项目资料 | 尽调报告 + 评分 |
| **3-推演** | 尽调通过（评分≥3.0/5.0） | 沙盘推演 | 尽调报告 | 推演路径 + 风险清单 |
| **4-规划** | 推演完成 | 商业计划书 + 三会治理系统（理事会） | 推演报告 | 商业计划书 + 决策记录 |
| **5-对接** | 商业计划通过理事会决策 | 股权合作机制 / 获客分包平台 / 供应链对接 | 商业计划书 | 合作协议 / 供应商清单 |
| **6-执行** | 资源到位 | 协作交付机制 + 市场销售渠道对接 | 合作协议 | 交付物 + 验收报告 |
| **7-推广** | 产品/服务可交付 | 投资路演 / IP工具包 | 交付验收 | 路演材料 / 营销方案 |
| **8-复盘** | 交付完成或终止 | 三会治理系统（复盘评审） | 全部阶段档案 | 复盘报告 |

### 3.2 Skill选择器（阶段2专用）

根据项目类型自动匹配尽调Skill：

| 项目类型 | 匹配Skill | 说明 |
|---------|----------|------|
| 科技创新项目 | 技术尽调 | 含专利验证、TRL评估、团队背景 |
| 投资/融资项目 | 投资方尽调 | 资金实力、投资偏好、决策链 |
| 文化/内容项目 | 文化商业尽调 | MCN/短剧/文创行业评估 |
| 服务型项目 | 服务尽调 | 餐饮/电商等服务行业尽调 |
| 综合型项目 | 技术尽调+投资方尽调 | 双重检测 |

### 3.3 跨阶段数据传递契约

下游只能读取上游的输出，不可修改；如果需要回溯修改，必须记录变更日志。

```
上游输出 ──→ 下游输入
                ↓
        下游当前阶段执行
                ↓
        更新下游输出
                ↓
        传递给再下游
```

**变更日志格式：**

```markdown
### 变更记录
| 时间 | 变更内容 | 变更原因 | 变更人 |
|------|---------|---------|-------|
| 2026-05-11 14:00 | 阶段2尽调评分从2.8调整为3.2 | 补充专利检索数据后重新评估 | 胡田 |
```

## 四、阶段性考评与成就体系

### 4.1 阶段完成度评分标准

| 维度 | 权重 | 评分标准（1-5分） |
|------|------|-----------------|
| 产出完整性 | 30% | 1=有缺失, 3=基本完整, 5=完整无缺 |
| 数据质量 | 25% | 1=数据不全, 3=关键数据完整, 5=数据交叉验证通过 |
| 决策效率 | 15% | 1=反复拖延, 3=按时完成, 5=提前完成 |
| 团队配合 | 15% | 1=推诿扯皮, 3=正常协作, 5=高效协同 |
| 风险控制 | 15% | 1=风险爆发, 3=风险识别充分, 5=风险全部可控 |

**阶段得分 = 各维度加权和（满分5分）**

### 4.2 项目总评分公式

```
项目总分 = Σ(各阶段得分 × 阶段权重)
```

| 阶段 | 权重 |
|------|------|
| 0-孵化 | 5% |
| 1-需求 | 10% |
| 2-尽调 | 20% |
| 3-推演 | 15% |
| 4-规划 | 15% |
| 5-对接 | 10% |
| 6-执行 | 15% |
| 7-推广 | 5% |
| 8-复盘 | 5% |

## 五、项目全流程SOP

### 5.1 项目启动流程

当一个新的机会/项目进入视野时：

```
Step 1: 创建项目档案目录
  ├── 生成项目编号（自动规则）
  └── 创建README.md模板

Step 2: 填写立项申请
  ├── 调用三会治理系统（私董会模块）
  └── 完成立项评审

Step 3: 更新项目总览看板
  ├── 添加新项目记录
  └── 设置初始状态为"🐣 立项"
```

### 5.2 阶段推进流程

每个阶段推进的标准操作：

```
Step 1: 检查前置条件
  ├── 确认上一阶段完成
  └── 确认输入数据完整

Step 2: 执行本阶段
  ├── 调用对应Skill
  ├── 记录输入数据到项目档案
  └── 记录执行过程日志

Step 3: 记录输出数据
  ├── 产出物保存到项目档案
  └── 更新对应阶段文件

Step 4: 阶段性考评
  ├── 填写评分（按4.1标准）
  ├── 检查成就解锁条件
  └── 生成考评报告

Step 5: 状态更新
  ├── 更新README.md状态
  ├── 更新总览看板
  └── 通知用户
```

### 5.3 项目终止/休眠流程

当项目需要终止或暂停时：

```
Step 1: 填写终止/休眠原因
Step 2: 标记最终状态（✅已完成/🚫已终止/💤休眠中）
Step 3: 执行阶段8复盘（如果有足够数据）
Step 4: 如果终止，整理可复用资产到案例库
Step 5: 更新总览看板
```

### 5.4 项目恢复流程

当休眠项目被激活时：

```
Step 1: 确认当前断点在哪个阶段
Step 2: 检查该阶段前置条件是否仍然有效
Step 3: 如果前置条件过期（如尽调报告超过30天），重新执行对应阶段
Step 4: 恢复推进
```

## 九、SOP操作手册（快速查表）

### 9.1 日常操作三件事

每天打开这个Skill后的标准操作：

```
1️⃣ 查看项目总览看板
   → 检查所有项目状态
   → 识别需要推进的项目

2️⃣ 检查阶段触发条件
   → 哪个项目满足下一阶段条件？
   → 需要调用哪个Skill？

3️⃣ 执行推进并更新档案
   → 调用对应Skill
   → 记录输入输出
   → 生成考评报告
   → 更新看板
```

### 9.2 常见操作问答

**Q：新项目来了怎么办？**
A：按5.1流程 → 创建档案 → 私董会立项 → 更新看板

**Q：项目做了一半，下一步该做什么？**
A：看README.md的"阶段进度"表 → 找到第一个"⏳进行中"或"⚪待开始"的阶段 → 按3.1触发条件检查前置 → 推进

**Q：怎么查看项目的完整历史？**
A：打开项目档案目录 → README.md看概览 → 各个阶段文件看详细记录

**Q：考评评分谁来打？**
A：AI自动初步评分 + 用户最终确认 + 三会治理体系复核
