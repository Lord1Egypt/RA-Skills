# Verification：交付验证

## 1. Overview（概述）

每个阶段自己通过了验证 ≠ 整体交付合格。本技能在交付前聚合所有阶段的验证结果，确认验收标准全部满足，生成交付清单。这是最后一道防线。

**核心原则**：
- 阶段独立通过 ≠ 整体通过（跨阶段遗漏可能）
- 验收标准是唯一的 Done 定义（不是"我觉得做完了"）
- 所有 🔴 阻塞项必须在交付前清零
- 给出交付物清单而不是"完成了"三个字

## 2. When to Use（使用条件）

**触发条件**：
- Step 7 验证阶段自动注入
- 所有级别的任务（A/B/C）必须执行
- 所有阶段完成后，final handoff 前

**不可跳过**：
- 永远不可跳过 — 这是最后的交付质量门

## 3. Process（执行流程）

### 3.1 聚合验证流程

```
步骤 1: 阶段验证结果聚合
  ├─ 读取每个阶段的 verification report
  ├─ 汇总所有 🔴 阻塞项状态
  │   - 全部解决 ✅ → 进入步骤 2
  │   - 有未解决的 🔴 → 打回对应阶段
  ├─ 汇总 🟡 警告项数量和分布
  │   - 🟡 数量异常多（>10）→ 报告风险
  └─ 输出：聚合验证状态表

步骤 2: 验收标准核对
  ├─ 逐条对比 Step 1 的验收标准 vs 当前交付物
  │   - 标准 A："登录 API 支持 email+password 认证" → API 实际行为？
  │   - 标准 B："密码使用 bcrypt 哈希" → 检查代码是否使用 bcrypt
  │   - 标准 C："错误返回 JSON {error: '...'}" → 检查错误处理
  ├─ 不满足 → 🔴 阻塞（流程没有真正走完）
  └─ 输出：验收标准核对表

步骤 3: 交付物完整性检查
  ├─ 代码文件：全部存在且路径正确
  ├─ 测试文件：每个功能模块有对应测试
  ├─ 配置文件：如有新增配置，有默认值和文档
  ├─ 文档更新：CONTEXT.md 更新 / README 更新 / 新 ADR 创建
  ├─ 临时文件：无调试代码残留、无 .tmp 文件
  └─ 输出：交付物清单

步骤 4: 状态持久化
  ├─ 更新 .auto-coding/state.json（当前阶段 = completed）
  ├─ 写入交付摘要到 .auto-coding/logs/{order}-delivery.log
  └─ 确认所有中间产物（测试结果、scorecard report）已写入日志
```

## 4. Risk Scorecard（反借口表）

| # | 纪律 | 常见借口 | 反驳 |
|---|------|---------|------|
| V1 | 逐条核对验收标准 | "所有测试都过了，肯定满足需求" | 测试通过 ≠ 验收标准满足（可能有没测的功能） |
| V2 | 🔴 必须清零 | "这个 🔴 项影响不大，下个版本处理" | 未解决的 🔴 = 已知缺陷 = 不可交付 |
| V3 | 检查交付物完整性 | "代码都在仓库里，不会丢" | 文件存在 ≠ 文件正确（可能 0 字节/未保存） |
| V4 | 更新状态文件 | "下次跑会自动恢复" | 不更新的状态 = 不可靠的恢复 |
| V5 | 🟡 过多要报告风险 | "5 个 🟡 还好" | 🟡 是技术债，10+ 个 🟡 = 质量信号 |

## 5. Red Flags（危险信号）

- 🚩 任何阶段 verification_report 缺失 — 阶段可能根本没有执行验证
- 🚩 🔴 阻塞项计数 > 0 — 不可交付，必须打回
- 🚩 🟡 警告项 > 10 — 代码质量有系统性问题，需要关注
- 🚩 验收标准有未覆盖项 — 需求有遗漏（即使所有测试通过）
- 🚩 交付文件包含注释掉的调试代码 — grep 检查
- 🚩 state.json 状态 != "completed" — 流程可能被中断
- 🚩 跨阶段的相同错误模式 — 技能注入或验证规则可能失效

## 6. Verification（自检清单）

**必检（≤5 项）**：
- [ ] 所有阶段必检项全部通过（聚合报告确认）
- [ ] 所有 🔴 阻塞项已解决（计数 = 0）
- [ ] 验收标准全部满足（逐条核对 + 实际行为确认）
- [ ] CONTEXT.md 最终版已确认（包含本次所有新增术语和决策）
- [ ] .auto-coding/state.json 状态正确（current_phase = completed）

**抽检（≤3 项）**：
- [ ] 🟡 警告项汇总（总数、分布、处理状态 — 接受/修复/推迟）
- [ ] 交付物清单完整（文件数 = 预期文件数，无缺失）
- [ ] 无调试代码残留（grep 检查）

**免检**：交付报告格式、文档措辞风格
