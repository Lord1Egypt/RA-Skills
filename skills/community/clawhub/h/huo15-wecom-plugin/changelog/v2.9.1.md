# v2.9.1 (2026-05-07)

**v2.9.0 hotfix — writeConfigFile mergePatch 算空：必须 structuredClone 再 mutate**

## 复盘

v2.9.0 部署后实测：`gateway.log` 看到 `[wecom] dynamic agent registered: wecom-default-dm-wanghui ...` ✅，但 `~/.openclaw/openclaw.json`：

- `agents.list` 还是空（没新派生 entry）
- `bindings` 只有原本的 2 条 default，**没新加路由**

## 根因

OpenClaw 核心 `writeConfigFile`（`io-Z7dwPjtk.js`）的实现：

```js
// 简化版
nextCfg = applyMergePatch(
  runtimeConfigSourceSnapshot,
  createMergePatch(runtimeConfigSnapshot, cfg)   // ← 这里
);
```

它对比 **`runtimeConfigSnapshot`（runtime 内部 cached cfg）** 与传入的 `cfg`，算 JSON Merge Patch，再 apply 到 source snapshot。

v2.9.0 的代码：

```ts
const latestConfig = configRuntime.loadConfig!();   // ← 拿 runtime snapshot 引用
upsertAgentEntryAndBinding(latestConfig, ...);       // ← 直接 mutate 这个引用
await configRuntime.writeConfigFile!(latestConfig);  // ← 传同一个引用
```

`loadConfig()` 返回的对象**就是** `runtimeConfigSnapshot` 本体（或同 by-reference）。我直接 mutate 它，到 `createMergePatch(runtimeConfigSnapshot, cfg)` 时两者是同一个对象，patch 自然算成空 → 文件没变。

## 修复

```diff
- const latestConfig = configRuntime.loadConfig!();
- upsertAgentEntryAndBinding(latestConfig, ...);
- await configRuntime.writeConfigFile!(latestConfig);
+ const baseConfig = configRuntime.loadConfig!();
+ const cloned = typeof structuredClone === "function"
+   ? structuredClone(baseConfig)
+   : JSON.parse(JSON.stringify(baseConfig));
+ upsertAgentEntryAndBinding(cloned, ...);
+ await configRuntime.writeConfigFile!(cloned);
```

`structuredClone`（Node 18+ built-in）拿独立副本 → mergePatch 能正确算出 diff → 文件实际写入。

## 经验沉淀

**`writeConfigFile` 是 patch-based, 不是 overwrite-based**。

任何插件代码用 `runtime.config` API 改 cfg 都要：
1. `const cfg = runtime.config.loadConfig()`
2. **`const cloned = structuredClone(cfg)`** ← 必须！
3. mutate `cloned`
4. `await runtime.config.writeConfigFile(cloned)`

未来 OpenClaw 核心可能改 `writeConfigFile` 的语义（支持直接 overwrite），但**目前是 patch 模式**，clone 是必须步骤。
