# 第三篇：CASS11.0 软件体系

## 3.1 软件概述

**CASS**（Computer Aided Surveying System）是南方数码基于 AutoCAD 平台研发的**数字化地形地籍成图软件**，广泛应用于：
- 地形成图
- 地籍成图
- 工程测量应用
- 空间数据建库和更新
- 市政监管

**CASS11.0 版本特性：**
- 基于 CAD 平台（AutoCAD/浩辰/中望 OEM）
- 优化土方计算模式
- 增加裸眼3D采集、点云立采、立面采集
- 二三维一体化数据采集
- 行业版本：勘测定界版、矿山版、工程版、管线版、水上版、竣工测量版、农村地籍版

## 3.3 文件结构体系

### 核心文件格式

| 文件类型 | 扩展名 | 用途 |
|---|---|---|
| **坐标数据文件** | .DAT | 最基础数据文件：点号,编码,Y,X,H |
| **编码引导文件** | — | 引导编码赋给坐标点 |
| **权属引导文件** | — | 宗地权属信息引导 |
| **权属信息文件** | — | 宗地的完整权属信息 |
| **原始测量数据** | — | 全站仪传回的原始数据 |
| **断面里程文件** | — | 断面测量数据 |
| **公路曲线要素** | — | 公路平曲线参数 |
| **横断面设计** | — | 道路横断面设计参数 |
| **CASS交换文件** | .CAS | **核心交换格式**，含全部图形+属性 |
| **配置文件** | cassconfig.db | 符号定义、系统参数 |
| **野外操作码** | JCODE.DEF | 野外采集的简码定义 |
| **地类定义** | Dilei.def | 土地利用地类编码 |

### DAT 坐标文件格式

```
1点点名,1点编码,1点Y(东)坐标,1点X(北)坐标,1点高程
...
N点点名,N点编码,N点Y(东)坐标,N点X(北)坐标,N点高程
```

**注意：**
- 每行代表一个点
- 坐标单位：米
- 编码不能含逗号，即使为空逗号也不能省略
- 所有逗号必须在半角方式下输入

### CAS 交换文件格式（★核心★）

**文件结构：**
```
CASS11.0                    ← 文件头
西南角坐标                   ← 图形范围左下角
东北角坐标                   ← 图形范围右上角
[层名]                      ← 图层开始
实体类型                     ← POINT/LINE/ARC/CIRCLE/PLINE/SPLINE/TEXT/SPECIAL
  ... 实体数据 ...
nil                         ← 该类型结束
[层名]                      ← 下一图层
  ...
END                         ← 文件结束
```

**8种实体类型：**

| 实体类型 | 说明 | 示例编码 | 地物类型 |
|---|---|---|---|
| **POINT** | 独立点状地物 | 143702,旋转角 | 门墩、井盖、路灯等 |
| **LINE** | 双点线段 | 143301,线型 | 围墙短线、行树等 |
| **ARC** | 弧线 | 编码,线型,半径,始角,终角 | 弧形道路、弧形围墙 |
| **CIRCLE** | 圆形地物 | 编码,半径 | 圆形井、圆形花坛 |
| **PLINE** | 复合线（多节点） | 编码,线宽,拟合方式,附加值 | 房屋、道路、陡坎、围墙 |
| **SPLINE** | 样条曲线 | 编码,线型,拟合方式,容差 | 等高线 |
| **TEXT** | 文字注记 | 编码,字高,旋转角 | 地名、高程注记 |
| **SPECIAL** | 程序控制复杂实体 | 依地物类型而变化 | 控制点、高程点、骨架线、界址线 |

**PLINE 拟合方式：**
- `N`：不拟合
- `S`：样条拟合（B-Spline）
- `F`：过点曲线拟合

**PLINE 附加值的含义：**
- 一般情况：0
- 依比例围墙/铁路：宽度（米）
- 陡坎：坎高（米）
- 房屋：层数

## 3.4 CASS 符号编码体系

### 六位编码规则

骨架线编码格式：`1 + 国标图式序号 + 顺序号 + 0或1`

**示例：**
- 三角点：`131100` = 1 + 311（国标序号3.1.1）+ 0 + 0
- 一般房屋：`141101` = 1 + 411（国标序号4.1.1）+ 0 + 1
- 砼房屋：`141111` = 1 + 411 + 1 + 1

### 辅助符号编码

格式：`骨架线编码 + "-" + 顺序号`
- 围墙短线：`144301-1`
- 围墙长线：`144301-2`

# 第四篇：GIS 数据格式与互操作

## 4.1 CASS 与 GIS 的接口

### 接口方法汇总

| 接口格式 | 目标系统 | 特点 |
|---|---|---|
| **CAS 交换文件** (.CAS) | 通用（需二次开发转换） | 文本格式，含全部图形+属性信息，双向转换 |
| **SHP 文件** | ArcGIS | 直接输出/导入，自动处理面状区域扣岛 |
| **MDB 文件** | ArcGIS Personal Geodatabase | 直接输出到 Access 格式地理数据库 |
| **MIF/MID 文件** | MapInfo | 通用交换格式 |
| **国家空间矢量格式** | 国家基础地理信息数据库 | 国家标准格式 |

### 数据入库检查要点

- 地物是否放错图层
- 代码值是否正确
- 面状地物是否封闭（无悬挂点）
- 是否存在伪节点
- 图层正确性

## 5.6 GDB→CASS→DWG 转换核心要点

基于 CASS11 实际工作流程：

1. **GDB 读取** → ArcGIS/ArcPy 读取要素类和属性表
2. **属性映射** → GIS编码 ↔ CASS六位编码 对照表
3. **几何转换** → 投影转换（如需要）
4. **CAS 文件生成** → 按 CASS 交换格式写入
5. **DWG 生成** → CASS 打开 CAS 文件，生成为 DWG

**关键注意事项：**
- 面状要素需闭合
- 属性值必须填入正确的字段位置
- 图层名需符合 CASS 标准
- 编码不能与 CASS 内置编码冲突

> **神经连接**：三种坐标系参数 → 第一篇 坐标系基础（椭球体Krasovsky/CGCS2000椭球参数对照）。高斯投影带号 → 第二篇 地图投影（3°/6°带中央子午线计算公式）。七参数/四参数转换 → 第二十一篇 GNSS测量（Bursa-Wolf七参数/平面四参数实战）、第六篇 CASS工作流（转换后坐标验证）。CGCS2000推广 → 第十四篇 新型基础测绘（CGCS2000作为统一空间基准）。

# 第六篇：CASS 实用工作流程（含编码体系深度解析）

> 来源：CASS11 安装目录逆向分析（layer.def / text.def / CodeField.db），2026-05-24
> 参考：GB/T 20257.1-2017 国家基本比例尺地图图式

## 6.1 CASS 编码体系深度解析（⭐核心知识）

### 6.1.1 layer.def 文件格式

文件位置：`C:\Program Files\CASS11 For AutoCAD 2022\sme\layer.def`

每行格式：
```
编码,图层,图块/线型,线宽,说明,CASS内部码,实体类型后缀
```
示例：
```
141101,JMD,continuous,0,建成房屋,310301003,JMD_M
141101-1,JMD,text,2,房屋结构注记,310301004,JMD_ZJ
141101-2,JMD,X2,0,地下房屋,310301006,JMD_X
```

### 6.1.2 编码后缀命名规则

| 后缀形式 | 含义 | DWG实体类型 | 示例 |
|---------|------|-------------|------|
| 无后缀（如`141101`） | 地物主体 | LWPOLYLINE面 / INSERT点块 | 建成房屋 |
| `-1`（如`141101-1`） | 属性注记 | TEXT/MTEXT | 房屋结构注记 |
| `-2`/`-3`/`-8`等 | 辅助线 | LWPOLYLINE线 | 地下房屋边界线 |
| 无后缀+第3列=`gcXXX` | 点符号块 | INSERT | 三角点(gc113) |

**实测验证**（2026-05-25，参考DXF `DLG373601498.dxf` 解剖）：
- **`141101-2` = 层数注记**（TEXT实体，文本如"2"/"7"/"8"），787个，非地下房屋边界线！
  - layer.def 中写"地下房屋"仅定义了该编码在表结构中的位置，实际数据中用于层数
- **`141101-1` = 结构注记**（TEXT实体，文本如"建"/"砖"/"钢"），3个
- 房屋面 LWPOLYLINE 的 xdata **仅含编码 `[(1000,'141101')]`**，不包含属性值
- CASS V 查询从附近的 TEXT 注记实体读取属性，**不是从 xdata**

### 6.1.3 实体类型后缀含义（第7列）

| 后缀 | 含义 | CASS识别方式 |
|------|------|-------------|
| `_M` | 面实体 | LWPOLYLINE closed=1，参与面查询 |
| `_D` | 点块实体 | INSERT，有定位点 |
| `_X` | 线实体 | LWPOLYLINE closed=0 |
| `_ZJ` | 注记实体 | TEXT，可被V查询读取属性 |
| `ASSIST` | 辅助线图层 | 不参与CASS查询，只用于绘图 |

### 6.1.4 text.def 注记规范

文件位置：`C:\Program Files\CASS11 For AutoCAD 2022\sme\text.def`

每行格式：
```
编码,图层,23(固定),字体样式,字高(1:1000比例尺下),说明
```
示例：
```
141101-1,JMD,23,XDXT,2,房屋结构注记
```
含义：编码`141101-1`的注记使用`XDXT`字体，字高2.0（1:1000下），实际字高需乘以比例尺因子。

### 6.1.5 CodeField.db 属性字段定义（⭐最重要）

文件位置：`C:\Program Files\CASS11 For AutoCAD 2022\system\CodeField.db`

这是CASS的**属性字典数据库**，V查询时弹出的属性对话框就是查的这里。

**关键表：**
| 表名 | 对应地物类型 |
|------|-------------|
| `RESPY` | 居民地及设施（含房屋） |
| `HYDPY` | 水系设施 |
| `TRAPY` | 交通设施 |
| `RESLN`/`RESPT`/`RESPY` | 居民地相关 |
| `VEGPY` | 植被土质 |
| `ZJ` | 注记类 |

**RESPY表关键字段（房屋类）：**

| 字段 | 含义 | 值格式 | 说明 |
|------|------|--------|------|
| `CASSCODE` | CASS编码 | `'141101'` | 对应layer.def主编码 |
| `FWCS` | 房屋层数 | `'M'`或`NULL` | **M=必填**，V查询时强制填写 |
| `DXFWCS` | 地下房屋层数 | `'M'`或`NULL` | 同上 |
| `JGLX` | 结构类型 | `'M,钢,钢筋混泥土,混合结构,砖(石)木'` | **M=必填**，逗号后接枚举值 |
| `YT` | 房屋用途 | `'C,住宅,工业交通仓储,...'` | C=必填 |
| `GD` | 高程 | `'C'` | C=有此属性字段 |

**字段值格式说明：**
- `NULL`：无此属性
- `'C'`：C=Chinese，有此属性字段（V查询对话框中显示）
- `'M'`：M=Must，必填字段（V查询时强制要求填写）
- `'M,值1,值2,...'`：必填，且值为逗号分隔的**枚举列表**（下拉选择）

### 6.1.6 xdata 格式规范（已验证）

CASS通过实体的扩展数据（xdata）识别地物编码：

```
(1001, "SOUTH")        ← AppID，必须
(1000, "141101")       ← 主编码（字符串）
(1000, "砖")           ← 可选：结构类型属性值
(1000, "2")            ← 可选：层数值
...
```

**注意：**
- 每个实体只能有一个`(1001, "SOUTH")`组
- 后面跟若干个`(1000, ...)`字符串组
- 编码必须带引号，是字符串类型
- **注记TEXT实体**的xdata编码也填**主编码**（如`141101`），不是`141101-1`

## 6.2 GDB → CASS DWG 转换规则

### 6.2.1 整体转换流程

```
GDB读取(ArcPy) → 编码映射 → 几何转换 → 属性注记生成 → DXF输出(ezdxf) → CASS打开
```

### 6.2.2 各要素类型转换规则

**点要素（GDB Point FC）→ DXF INSERT实体**
- 块名：查layer.def第3列（如`gc200`=测量控制点）
- 图层：查layer.def第2列（如`KZD`）
- xdata：`(1001,"SOUTH"), (1000, "131100")`

**线要素（GDB Line/Polyline FC）→ DXF LWPOLYLINE实体（closed=0）**
- 线型：查layer.def第3列（continuous / X2 / X7 等）
- 图层：对应layer.def第2列

**面要素（GDB Polygon FC）→ DXF LWPOLYLINE实体（closed=1）**
- 房屋面：continuous线型，LAYER=JMD，xdata编码=141101等
- 水系面：continuous线型，LAYER=SXSS
- **必须闭合**，且高程值正确

**属性注记（✅ 参考DXF实测验证 v7）**
以房屋（141101）为例：
1. **房屋面 LWPOLYLINE 的 xdata 仅含编码** `[(1000,'141101')]`，**不包含属性值**
2. **层数是独立的 TEXT 注记实体**，编码 `141101-2`，文本为层数值（如"2"）
3. **结构类型是独立的 TEXT 注记实体**，编码 `141101-1`，文本为结构类型（如"砖"）
4. 注记位置在房屋面内部（重心附近），字高约 2.0
5. **CASS V 查询从附近的 TEXT 注记实体读取属性**，不是从 LWPOLYLINE 的 xdata

**GDB字段到CASS注记的映射：**
| GDB字段 | 含义 | → CASS注记 | TEXT编码 | 值转换 |
|---------|------|-----------|----------|--------|
| `MATRL` | 原材料/结构 | 结构注记 | `141101-1` | 砖→砖(石)木, 砼→钢筋混泥土, 混→混合结构, 钢→钢 |
| `LAY` | 层数(SmallInteger) | 层数注记 | `141101-2` | 直接取整数字符串 |

**MATRL→CASS标准值映射表：**
```
砖/砖混/砖木 → 砖(石)木
砼/混凝土/RC → 钢筋混泥土
混/混合 → 混合结构
钢/钢结构 → 钢
石木/土木/木 → 砖(石)木
```
未知值保持原值，CASS V查询时会要求用户手动填写。

## 6.3 测制地形图标准流程（CASS操作）

```
准备定显示区 → 展点 → 绘制平面图 → 绘制等高线(DTM) → 编辑与整饰
```

1. **准备**：设置比例尺、坐标系、图幅范围
2. **展点**：将DAT坐标文件中的点展绘到图上

4. **绘制等高线**：建立DTM → 生成等高线 → 等高线注记 → 编辑
5. **编辑与整饰**：添加图廓、图例、比例尺

#### 一、ezdxf 库相关坑（5个）

| # | 坑 | 现象 | 根因 | 正确做法 |
|---|-----|------|------|----------|
| **1** | `.saveas('xxx.dwg')` 输出假DWG | CASS打开报"图形文件无效" | ezdxf 不管后缀名，输出永远是DXF文本格式（文件头 `0\nSECTION`），不是DWG二进制（AC1015） | ①输出后缀直接写 `.dxf`，CASS 10.1 支持直接打开 ✅ ②如需真DWG：用 ODA File Converter 批量转 |
| **2** | `doc.encoding` 不设GBK | CASS 打开后中文乱码/图层名乱码 | ezdxf 默认编码不是GBK | **必须** `doc.encoding = 'gbk'`，CASS 10.1 只认 GBK |
| **3** | `doc.header.hdrvars['$DWGCODEPAGE'] = 'ANSI_936'` | 部分 AutoCAD 版本中文仍乱码 | 仅设 encoding 不够，需同时设 DWGCODEPAGE | 两行都要写 |
| **4** | 跨文档 `ent.copy()` | entitydb 损坏，DXF 打不开 | ezdxf 实体不能跨 doc 复制 | 在目标 doc 中重新创建实体，不要 copy |
| **5** | 输出文件164KB空壳 | 转换完成但文件无实际内容 | ezdxf 在某些错误后输出空 DXF 最小骨架 | 转换后检查 `os.path.getsize() < 500KB` 视为异常 |

#### 二、CASS 编码体系暗坑（6个）

| # | 坑 | 现象 | 根因 | 正确做法 |
|---|-----|------|------|----------|
| **6** | 房屋属性写进xdata → CASS不读 | V 查询时属性对话框空白 | **CASS LWPOLYLINE xdata 只存编码**，属性值在**独立的 TEXT 注记实体**上 | 房屋面 xdata：仅 `[(1000,'141101')]`；另外两个 TEXT：`141101-2`=层数，`141101-1`=结构 |
| **7** | `141101-2` 含义误解 | layer.def 写"地下房屋"让人以为是辅助线 | 实测DXF：787个 `141101-2` TEXT实体全是层数注记（"2"/"7"/"8"），非地下房屋边界 | **以实测 DXF 为准，不要盲信 layer.def 的说明文字** |
| **8** | 编码后缀 `-1`/`-2` 规则有例外 | 以为 `-2` 都是辅助线 | `141101-2` 实际是 TEXT 注记（层数），不是辅助线 | **后缀规则是惯例非铁律**，必须参考 text.def + 实测DXF 验证 |
| **9** | xdata 写入空字符串或0编码 | CASS 不识别该实体 | xdata 中编码必须是非空字符串 | `if not code or code == '0': return` |
| **10** | CodeField.db 字段值格式解析错误 | 以为 `M` 是属性值 | `M` = Must（必填），不是实际值；`C` = Chinese（有此字段）；`M,砖,砼,混` = 必填+枚举列表 | 见 6.1.5 节字段值格式说明 |
| **11** | 房屋属性字段存在性不确定 | 不确定哪些 GDB FC 有 MATRL/LAY | 实测：仅 `JMDJSS_JMD_A` 有完整 MATRL+LAY（529/777条），`GGFWJSS`/`GKJSS`/`NYJSS`/`QTJZWJSS` 仅有 TYPE | 先用 arcpy 检查字段是否存在再读取 |

#### 三、线型系统暗坑（7个）

| # | 坑 | 现象 | 根因 | 正确做法 |
|---|-----|------|------|----------|
| **12** | X22_L(围墙)/X3_L(陡坎)未创建 | 围墙显示为实线，无齿状 | ACAD.LIN 有 X22/X3/X6 定义，但 layer.def 引用的是 `X22_l`（小写）变体 | ①从 ACAD.LIN 解析 pattern ②去掉 `_L`/`_1` 等后缀匹配基础线型 ③ezdxf 线型名大小写不敏感 |
| **13** | X41-X78 系列线型缺失 | 55个线型"无定义" | ACAD.LIN 仅含 X0-X40，X41+ 是 CASS 后续版本添加的自定义线型，存储在工作空间的 `.lin` 或 `.dwg` 模板中 | ①用 continuous 兜底 ②从 CASS 模板 `.dwt` 中导出 ③接受部分管线/境界/交通线型降级 |
| **14** | 线型后缀变体匹配失败 | `X22_l`、`X17_1`、`X41_2` 等匹配不到 pattern | `replace('_L', '')` 不够用，有 `_1`/`_2`/`_10` 等数字后缀 | 用后缀列表逐条尝试剥离：`['_L','_l','_1','_2',...,'_13']` |

| **16** | layer.def 线型列包含 gc 开头值 | `gc200` 等被当成线型处理 | 块名写在 layer.def 第3列时用 `gcXXX` 格式 | 解析 layer.def 时过滤 `linetype.startswith('gc')` 的条目 |
| **17** | ACAD.LIN 含 shape 定义无法直接用于 ezdxf | pattern 创建失败 | ezdxf 的 `pattern` 只接受纯数字 dash/gap 序列，不接受 `[SHAPENAME,shx]` 格式 | 解析时跳过 shape 段，只保留 `('dash', float)` 类型的段 |
| **18** | 线型在运行时创建但未传递 context | process_feature_classes 内 `ensure_linetype` 找不到 acad_patterns | 函数签名没传 `acad_patterns` 参数 | `setup_linetypes` 返回 `acad_patterns`，逐级传递到 `process_feature_classes` → `ensure_linetype` |
