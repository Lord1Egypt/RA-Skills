# Dispute Response Quality Checklist

Run before sending any dispute package. Unchecked = fix or disclose.

## Triage
- [ ] Dispute type classified (INR / INAD / damaged / wrong item / return condition / chargeback / escalation)
- [ ] Deadline recorded in bold at top of package, with timezone
- [ ] Stage identified (first contact / claim / appeal / representment)
- [ ] Buyer history checked (orders, prior disputes, account age)
- [ ] Double-dipping checked (marketplace claim + chargeback simultaneously)

## Evidence
- [ ] Evidence inventory table complete (have / can get / doesn't exist)
- [ ] Every cited artifact actually attached or precisely referenced
- [ ] Every artifact tied to THIS order (order #, label, serial)
- [ ] Live listing screenshotted now (INAD cases)
- [ ] Return unboxing video continuous, label visible (return-condition cases)
- [ ] No asserted-but-unavailable evidence anywhere in the text

## Strategy
- [ ] Decision matrix run: economics, evidence grade, LTV, precedent
- [ ] The "why" paragraph includes actual numbers
- [ ] Partial refund percentage justified, not haggled
- [ ] Extortion/fraud reported through the separate channel where applicable
- [ ] Metric/account-health interaction considered

## Platform submission
- [ ] One-line summary first
- [ ] Facts numbered, dated, each tied to an exhibit
- [ ] Platform's own policy cited
- [ ] Requested outcome explicit
- [ ] ≤200 words before exhibits; zero emotion, zero buyer-blaming adjectives
- [ ] Evidence formatted per platform requirements (file types, PDF for representment)

## Buyer message
- [ ] Separate text from the platform submission
- [ ] Empathy line specific to this buyer's situation
- [ ] Facts without accusation; verdict left to the platform where fighting
- [ ] One clear resolution path (two max) with concrete next steps and dates
- [ ] 60-150 words, customer reading level
- [ ] No banned phrases (per response-templates.md §4)
- [ ] No mention of extortion threats (those went to the platform report)

## Prevention
- [ ] Log entry drafted (type | SKU | carrier | buyer | outcome | cost)
- [ ] Root cause flagged where a pattern exists
- [ ] Operational action proposed (QC step, signature confirmation, block-list, listing fix)
- [ ] Separate filings listed with their own deadlines (carrier claim, fraud report)

## Final
- [ ] Submission planned at ≤80% of the deadline window
- [ ] All dates absolute; currency explicit
- [ ] Platform policy references tagged "verify current" where procedures change
