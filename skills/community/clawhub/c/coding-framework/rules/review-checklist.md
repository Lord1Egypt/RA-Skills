# 审查清单（6 维度）

> 多代理审查时，每个代理对应一个维度。可单独使用，也可并行执行。

## 维度 1：代码质量（code-reviewer）

- [ ] 命名是否清晰表达意图
- [ ] 函数是否单一职责
- [ ] 是否有重复代码（DRY）
- [ ] 是否遵循 KISS 原则
- [ ] 边界条件是否处理
- [ ] 空值/异常是否正确处理
- [ ] 代码精简度（Ponytail 阶梯）

## 维度 2：安全性（security-auditor）

- [ ] 是否有注入漏洞（SQL/XSS/命令）
- [ ] 是否有硬编码凭证
- [ ] 敏感数据是否加密/脱敏
- [ ] 权限检查是否完整
- [ ] 依赖是否有已知漏洞
- [ ] 加密算法是否安全
- [ ] 是否存在路径遍历/SSRF/XXE

## 维度 3：测试（test-engineer）

- [ ] 核心逻辑是否有单元测试
- [ ] 边界条件是否覆盖
- [ ] 错误路径是否有测试
- [ ] 测试是否独立可运行
- [ ] 测试命名是否清晰
- [ ] 覆盖率是否达标

## 维度 4：架构（architecture-critic）

- [ ] 模块职责是否单一
- [ ] 耦合度是否合理
- [ ] 是否存在循环依赖
- [ ] 是否遵循项目架构模式
- [ ] 接口是否面向抽象
- [ ] 是否考虑扩展需求

## 维度 5：性能（performance-analyst）

- [ ] 是否存在不必要的 O(n²)
- [ ] 是否有资源泄漏
- [ ] 数据库是否有 N+1 问题
- [ ] 是否有不必要的同步/阻塞
- [ ] 大数据集是否流式处理
- [ ] 是否有并发安全问题

## 维度 6：文档（documentation-checker）

- [ ] 公开 API 是否有文档
- [ ] 复杂逻辑是否有注释
- [ ] README 是否完整
- [ ] 文档是否与代码一致
- [ ] 是否有使用示例
- [ ] 错误码是否有说明

## 使用方式

### 单维度审查

```
请用 code-reviewer 审查这段代码的代码质量
```

### 多维度并行

```
请并行审查：代码质量 + 安全性 + 性能
```

### 全量审查

```
请对所有 6 个维度进行完整审查
```

## 置信度过滤

- ≥80：纳入最终报告
- 60-79：标记为"待确认"
- <60：丢弃

## 冲突解决

当多个代理对同一位置有矛盾意见时：
1. 严重性优先
2. 置信度加权
3. 标注分歧
4. 标记需人工裁决的冲突项
