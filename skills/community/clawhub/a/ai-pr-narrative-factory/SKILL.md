# AI PR Narrative Factory

> 把一个 PR URL 丢进来，还你一份专业的变更叙事报告

## 🎯 解决痛点

- ❌ Code Review 看完代码，却不知道怎么向非技术同事解释
- ❌ 发布说明（Changelog）靠手工编写，费时易错
- ❌ 团队成员不理解某个 PR 的价值和影响范围
- ❌ 每次发版前临时凑变更日志，质量参差不齐

## 💡 解决方案

```
你：丢一个 GitHub PR URL
        ↓
┌─────────────────────┐
│  github             │ → gh CLI 提取 PR 元数据 + 代码 diff
└────────┬────────────┘
         │ PR 信息 + 代码变更
   ┌─────▼─────┐
   │  kimi-cli │ → AI 分析代码变更：新增功能 / 修复 / 破坏性变更
   └─────┬─────┘
         │ 变更分析结果
   ┌─────▼─────┐
   │ summarize │ → 生成人类可读的变更叙事（非技术人员也能看懂）
   └─────┬─────┘
         │ 叙事报告
   ┌─────▼──────────┐
   │ feishu-create-doc │ → 发布至飞书文档，生成永久分享链接
   └─────┬──────────┘
         │ 完成
   ← 回复 PR 链接 + 飞书报告链接
```

## 📦 包含 Skills

| Skill | 作用 | 调用顺序 |
|-------|------|---------|
| github | gh CLI 获取 PR 元数据、代码 diff、审查意见 | 1 |
| kimi-cli | AI 分析代码变更，识别功能点、风险点、技术亮点 | 2 |
| summarize | 将分析结果转化为人类可读的多层次变更叙事 | 3 |
| feishu-create-doc | 将报告发布为飞书云文档，生成永久可分享链接 | 4 |

## 🚀 使用方法

### 方式一：直接触发
发送 GitHub PR URL 或 PR 编号，Skill 自动完成全流程。

### 方式二：提供更多上下文
```
分析 https://github.com/owner/repo/pull/123
重点关注：是否破坏现有 API？需要迁移吗？
```

### 方式三：批量生成
提供多个 PR 或版本号区间，自动生成版本变更日志。

## 📋 输出报告结构

生成的飞书文档包含以下章节：

### 1. 📌 PR 基本信息
- PR 标题、作者、状态
- 涉及文件数量、代码行数变化
- 关联的 Issue /标签

### 2. 🔍 核心变更解读
- 通俗易懂的功能描述（非技术人员也能看懂）
- 关键技术决策说明
- 依赖变更（新增/升级/移除）

### 3. ⚠️ 破坏性变更（Breaking Changes）
- 明确标注需要迁移的 API
- 提供最小化破坏范围的建议

### 4. 🧪 测试与验证
- 测试覆盖情况
- 手动验证清单（推荐 QA 关注点）

### 5. 📝 发布说明（Changelog 条目）
- 符合 [Keep a Changelog](https://keepachangelog.com/) 规范的条目
- 可直接复制到项目 CHANGELOG.md

### 6. 💬 团队沟通建议
- 非技术同事能理解的功能描述（用于公告/邮件）
- 产品经理需要知道的关键变化

## 🔧 适用场景

| 场景 | 输入 | 输出 |
|------|------|------|
| Code Review 同步 | PR URL | 变更叙事报告 → 分享给产品/运营 |
| 版本发布准备 | Release PR 或 commit range | 发布说明（Changelog） |
| 团队知识沉淀 | 重要架构变更 PR | 技术决策记录 → 入知识库 |
| 跨团队沟通 | 依赖方 PR | 影响分析报告 → 通知相关方 |

## 💡 使用示例

**用户输入：**
> 帮我分析这个 PR：https://github.com/vercel/next.js/pull/65000

**Agent 行为：**
1. 调用 `github` 提取 PR 元数据和 diff
2. 调用 `kimi-cli` 分析代码变更内容
3. 调用 `summarize` 生成变更叙事
4. 调用 `feishu-create-doc` 发布报告
5. 回复：「已生成变更报告 ✅」+ 飞书文档链接

**输出飞书文档节选：**
> ## 🔍 核心变更解读
> **新增功能：** 支持在 `next/image` 组件中使用 AVIF 格式自动优化图片加载
> - 当浏览器支持 AVIF 时自动降级为 WebP/JPEG
> - 图片加载速度预计提升 30-40%
> - 无需修改任何现有代码，完全向后兼容
>
> ## ⚠️ 破坏性变更
> 无
>
> ## 📝 发布说明（Changelog 条目）
> ```markdown
> ### 新增
> - `next/image` 支持 AVIF 格式自动优化 (#65000)
> ```
