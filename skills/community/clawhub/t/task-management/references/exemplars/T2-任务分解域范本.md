## T2 任务分解域 范本

### 范本概述
- **域ID**: T2
- **域名称**: 任务分解域
- **任务数量**: 6
- **范本状态**: 可用

### 任务执行范本

#### T2-01 复杂度评估
**执行步骤**:
1. 识别任务主要步骤
2. 分析任务依赖关系
3. 识别参与角色
4. 估算任务时间
5. 判定任务复杂度

**输入示例**:
- 任务：创建个人博客系统
- 任务描述：开发一个包含用户管理、文章管理、评论管理等功能的个人博客系统

**输出示例**:
- 主要步骤：需求分析、系统设计、数据库设计、前端开发、后端开发、测试、部署
- 依赖关系：设计依赖需求分析，开发依赖设计，测试依赖开发
- 参与角色：产品经理、设计师、前端开发、后端开发、测试工程师
- 时间估算：4周
- 复杂度判定：复杂（7个主要步骤，多个角色参与）

**注意事项**:
- 步骤识别要全面，避免遗漏
- 依赖分析要准确，避免错误
- 复杂度判定要客观，避免主观臆断

#### T2-02 步骤识别
**执行步骤**:
1. 分析任务目标
2. 识别主要执行步骤
3. 确定步骤顺序
4. 分析步骤依赖关系
5. 验证步骤完整性

**输入示例**:
- 任务：创建个人博客系统
- 复杂度：复杂

**输出示例**:
- 主要执行步骤：
  1. 需求分析：收集和分析用户需求
  2. 系统设计：设计系统架构和数据库
  3. 前端开发：开发用户界面
  4. 后端开发：开发服务器端逻辑
  5. 测试验证：进行系统测试
  6. 部署上线：部署到生产环境
- 步骤顺序：需求分析 → 系统设计 → 前端开发/后端开发 → 测试验证 → 部署上线
- 依赖关系：系统设计依赖需求分析，前端/后端开发依赖系统设计，测试验证依赖开发完成

**注意事项**:
- 步骤识别要覆盖任务全流程
- 步骤顺序要合理，避免逻辑错误
- 依赖关系要清晰，避免循环依赖

#### T2-03 子任务创建
**执行步骤**:
1. 将复杂任务分解为子任务
2. 为每个子任务设置元数据
3. 建立子任务间的依赖关系
4. 验证子任务完整性
5. 确认子任务创建

**输入示例**:
- 任务：创建个人博客系统
- 主要步骤：需求分析、系统设计、前端开发、后端开发、测试、部署

**输出示例**:
- 子任务列表：
  1. 需求分析子任务：收集用户需求，分析功能需求
  2. 系统设计子任务：设计系统架构，设计数据库
  3. 前端开发子任务：开发用户界面，实现交互功能
  4. 后端开发子任务：开发API接口，实现业务逻辑
  5. 测试验证子任务：进行功能测试，性能测试
  6. 部署上线子任务：部署到服务器，配置域名
- 依赖关系：系统设计依赖需求分析，前端/后端开发依赖系统设计，测试依赖开发完成

**注意事项**:
- 子任务要可独立执行，避免耦合
- 元数据要完整，避免信息缺失
- 依赖关系要准确，避免错误

#### T2-04 依赖识别
**执行步骤**:
1. 分析子任务间的关系
2. 识别依赖类型
3. 绘制依赖关系图
4. 验证依赖关系
5. 确认依赖识别结果

**输入示例**:
- 子任务列表：需求分析、系统设计、前端开发、后端开发、测试、部署

**输出示例**:
- 依赖类型：blockedBy（被阻塞）
- 依赖关系图：
  ```
  需求分析 → 系统设计 → 前端开发 → 测试 → 部署
                    ↓
                后端开发 → 测试
  ```
- 依赖验证：无循环依赖，依赖关系合理

**注意事项**:
- 依赖类型要准确，避免混淆
- 依赖关系图要清晰，避免混乱
- 依赖验证要严格，避免错误

#### T2-05 关键路径识别
**执行步骤**:
1. 分析依赖关系
2. 识别所有可能路径
3. 计算路径时间
4. 确定关键路径
5. 识别关键任务

**输入示例**:
- 依赖关系图：需求分析 → 系统设计 → 前端开发 → 测试 → 部署
- 时间估算：需求分析2天，系统设计3天，前端开发5天，后端开发4天，测试3天，部署1天

**输出示例**:
- 所有路径：
  1. 需求分析 → 系统设计 → 前端开发 → 测试 → 部署（14天）
  2. 需求分析 → 系统设计 → 后端开发 → 测试 → 部署（13天）
- 关键路径：需求分析 → 系统设计 → 前端开发 → 测试 → 部署（14天）
- 关键任务：需求分析、系统设计、前端开发、测试、部署

**注意事项**:
- 路径分析要全面，避免遗漏
- 时间估算要准确，避免偏差
- 关键路径要明确，避免混淆

#### T2-06 并行识别
**执行步骤**:
1. 分析依赖关系
2. 识别无依赖关系的任务
3. 确认并行条件
4. 规划并行执行
5. 确认并行识别结果

**输入示例**:
- 依赖关系图：需求分析 → 系统设计 → 前端开发/后端开发 → 测试 → 部署
- 并行条件：前端开发和后端开发无依赖关系

**输出示例**:
- 可并行任务：前端开发、后端开发
- 并行条件：系统设计完成后，前端和后端可同时开发
- 并行限制：需要足够的开发资源
- 并行执行计划：系统设计完成后，同时启动前端和后端开发

**注意事项**:
- 并行任务要无依赖关系
- 并行条件要满足
- 并行限制要考虑

### 范本特征摘要
- **结构特征**: 按任务分解流程组织，从复杂度评估到并行识别，步骤清晰
- **风格特征**: 专业、详细、实用，使用图表和示例展示复杂关系
- **逻辑特征**: 从任务分解到依赖管理，逻辑严密，步骤明确
- **格式特征**: 使用Markdown格式，包含流程图、表格、依赖关系图
