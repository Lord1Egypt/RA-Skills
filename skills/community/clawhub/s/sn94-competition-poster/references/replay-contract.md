# SN94 Replay Contract

SN94 reward-active competitions should be replayable by an independent validator from a public task repo plus backend-supplied hidden eval parameters.

## Required Task Fields

- `repository_url`: public Git repo for the task.
- `base_ref`: immutable commit, tag, or protected release branch.
- `allowed_patch_paths`: exact file paths or narrow globs miners may change.
- `setup_command`: repo-owned setup/build command.
- `benchmark_command`: repo-owned replay command.
- `result_path`: JSON output path produced by the benchmark.
- `max_patch_bytes`: hard cap for submitted patch size.
- `timeout_seconds`: practical upper bound for replay.
- `hardware`: CPU, CUDA, ANE, or other expected target.

## Validator Flow

1. Clone `repository_url`.
2. Checkout `base_ref`.
3. Apply miner patch.
4. Reject any changed path outside `allowed_patch_paths`.
5. Reject generated bytecode, cache files, binary blobs, secrets, and oversized patches.
6. Fetch backend-directed hidden data on the validator host, if any.
7. Run `setup_command`.
8. Run `benchmark_command`.
9. Parse `result_path`.
10. Submit observed metrics back to the backend.

## Result JSON

Recommended minimum:

```json
{
  "score": 0.0,
  "metric_name": "reward",
  "higher_is_better": true,
  "valid": true,
  "details": {
    "runtime_seconds": 0.0,
    "program_size": 0,
    "public_split_score": 0.0
  }
}
```

For frontier competitions, include the objective components separately so reviewers can explain why a new best result won.

## Security Rules

- Benchmark containers should not need internet.
- Wallets, keys, API tokens, and host credentials must stay outside the replay sandbox.
- The task repo must not require direct production database access.
- Hidden eval data can be fetched by the validator host before Docker starts, then passed into the sandbox as local files.
- A task repo should be disposable. Rebuilding from clone plus patch should reproduce the replay.
