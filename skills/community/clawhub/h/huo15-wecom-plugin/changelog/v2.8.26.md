# v2.8.26 (2026-05-06)

**修双失败链消息丢失：sendMessage 40008 自动降级 + streamId finalize 后开新流**

## 背景

赵博报告："⏳ 任务仍在执行（已 2m+），完成后会主动推送结果。然后就没有然后了。"

v2.8.24 的 placeholder timeout finalize 修复让"任务仍在执行..."消息 UI 解锁可见 ✅，但 LLM 真的出 final 时**消息静默丢失**。完整复盘 5/6 00:52-00:55 群聊事故：

```
00:52:58.021 [wecom-ws] placeholder timeout 120s: streamId=stream_177...96_51fd6b11
             已用 last=true 终结，企微 UI 解锁交互  ← v2.8.24 ✅
00:55:02.606 [wecom-outbound] Sending Bot WS active message len=41
00:55:02.736 [wecom-ws] Reply ack error: aibot_send_msg_..., errcode=40008,
             errmsg=invalid message type ❌ 路径 A 失败
00:55:02.740 [wecom-ws] Reply message sent reqId: 07zwrwElQHurYpg4...
00:55:03.033 [wecom-ws] Reply ack received reqId: 07zwrwElQHurYpg4... ← ack 成功
00:55:02.872 [wecom-ws] Reply ack error: aibot_send_msg_..., errcode=40008 ❌ 又一次
```

**两条路径都失败**：

| 路径 | 调用 | 结果 |
|---|---|---|
| **A**: outbound.sendText → `client.sendMessage(chatId, msgtype:markdown_v2)`（aibot_send_msg 主动推送）| 撞 **errcode 40008 "invalid message type"** | 消息被企微 server 拒 |
| **B**: reply.ts deliver → `replyStream(frame, streamId, text, false)` 用**已 finalize 的旧 streamId** | 企微 server ack 成功，但 client UI 不更新已 last=true 终结的流 | 消息被企微 client 静默吞掉 |

5/5 全天 13 条 errcode=40008 全部来自 `aibot_send_msg`（sendMessage / sendMediaMessage with markdown_v2 类型）—— 企微 server 对 markdown_v2 在主动推送通道做了 strict 校验，某些内容（特定 emoji / 长 URL / 嵌套结构）不通过。

## 修复

### A. `src/transport/bot-ws/sdk-adapter.ts` — sendMessage 撞 40008 自动降级 text

`sendMarkdown` handler 和 `sendMedia` 的 caption send 路径加 try-catch：

```diff
+ try {
    await client.sendMessage(chatId, {
      msgtype: "markdown_v2",
      markdown_v2: { content },
    } as unknown as Parameters<typeof client.sendMessage>[1]);
+ } catch (err) {
+   const errcode = (err as { errcode?: number })?.errcode;
+   if (errcode === 40008) {
+     console.warn(
+       `[wecom-ws] sendMessage markdown_v2 撞 40008（invalid message type）→ 降级 text 重试 chatId=${chatId} len=${content.length}`,
+     );
+     await client.sendMessage(chatId, {
+       msgtype: "text",
+       text: { content },
+     } as unknown as Parameters<typeof client.sendMessage>[1]);
+     console.log(`[wecom-ws] sendMessage text 降级成功 chatId=${chatId} len=${content.length}`);
+   } else {
+     throw err;
+   }
+ }
```

降级到 plain text 后用户看到的是 markdown 字面量（`**bold**` / `[text](url)` 不渲染），UX 略差但**消息不丢**。

### B. `src/transport/bot-ws/reply.ts` — streamFinalized 跟踪 + 自动重启新 streamId

新增 `streamFinalized: boolean` flag。`resolveStreamId()` 检测到旧 streamId 已 finalize 时自动 generateReqId 新 streamId：

```diff
  let streamId: string | undefined;
+ let streamFinalized = false;
  ...
  const resolveStreamId = () => {
+   if (streamFinalized) {
+     streamId = generateReqId("stream-restart");
+     streamFinalized = false;
+     console.log(`[wecom-ws] streamId 已终结，重启新 streamId=${streamId}`);
+   }
    streamId ||= generateReqId("stream");
    return streamId;
  };
```

placeholder timeout finalize 路径设 `streamFinalized = true`（v2.8.24 只设了 streamSettled，没解决 streamId 重用问题）：

```diff
  await params.client.replyStream(params.frame, streamId, finalPlaceholderText, true);
  streamSettled = true;
+ streamFinalized = true;
  console.log(`[wecom-ws] placeholder timeout 120s: streamId=... 已终结 + 标记 streamFinalized`);
```

之后 LLM 出 final/block 调 `resolveStreamId()` 自动返回新 streamId，企微 client 显示**新一条独立消息**。原"任务仍在执行..."老消息保留 last=true 状态可交互。

## 兼容性

完全向下兼容。

- markdown_v2 默认仍优先（UX 更好），仅撞 40008 时才降级 text
- streamFinalized 标记仅在 v2.8.24 placeholder timeout finalize 路径设——deliver final 路径自己 finalize 后**没有后续 block/final**（dispatch 已结束），不需要重启新流
- 旧 LLM 行为 / 短任务 (< 120s) 路径**完全不变**（不会触发 streamFinalized）
- `resolveStreamId` 公开 API 行为对 caller 透明（caller 不感知 streamId 是否被换）

## 经验沉淀

**stream 协议必须配套"重启策略"**：任何"流终结"操作（last=true / SSE close / WS unsubscribe）后，**后续同 streamId 的发送都是无效的**——企微 server 要么 ack 但 client 静默吞，要么直接 errcode 拒。状态机必须显式区分"流活跃"和"流已终结"，终结后的发送必须走新 streamId（或新 stream / 新 channel）。

v2.8.24 部署后只验证了"UI 解锁"是否生效，没验证"finalize 后的 final 消息能不能到"——这是测试盲点，5/6 00:55 用户实测才暴露。今后类似 stream 状态机改动必须**端到端验证**：finalize → 之后发的消息 → 用户实际收到。
