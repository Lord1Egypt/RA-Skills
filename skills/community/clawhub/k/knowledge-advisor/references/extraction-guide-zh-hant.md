# 萃取指南（繁體中文）

從書籍和學習材料中萃取結構化知識的方法論。

## 萃取目標

萃取可操作的結構化知識，使其能夠與現實情境匹配。目標不是摘要整本書，而是識別離散的、具名的、可重複使用的知識單元。

每個萃取項目必須：
- **忠於原著** — 不重新詮釋
- **可操作** — 使用者能立即運用
- **足夠具體** — 有明確步驟或判斷標準
- **可引用** — 連結到特定章節

## 1. 框架萃取

框架是一個具名的方法、流程或模型，包含明確的步驟或組成要素。

**如何識別：**
- 作者明確命名的方法（例如：「STATE 方法」、「刺蝟原則 (Hedgehog Concept)」）
- 作者規定的步驟流程
- 具有命名組件或軸線的模型（例如：2x2 矩陣）
- 編碼流程的縮寫

**萃取格式：**

```markdown
## [框架名稱] ([English Name])

**來源：** 第[N]章，「[章節標題]」
**類型：** [溝通 | 決策 | 領導力 | 策略 | ...]
**領域：** [以逗號分隔的領域標籤]

### 描述
[2-3句：這是什麼，為什麼重要。使用作者的語言。]

### 步驟
1. [步驟名稱] — [具體做什麼]
2. ...

### 應用情境
- [適用這個框架的情境]
- [另一個情境]

### 反模式（何時不該使用）
- [不適用的情境]

### 相關框架
- [[其他書籍/框架名稱]]

### 關鍵引文
- 「[原文直接引用]」（第[N]頁）
```

**規則：**
- 只萃取作者明確命名或定義的框架
- 如果是流程，必須包含所有步驟 — 不可縮減
- 應用情境至關重要 — 這些將成為搜尋索引
- 關鍵引文使原文有據可查

## 2. 原則萃取

原則是核心教訓、經驗法則或格言 — 比框架更簡短。

**如何識別：**
- 作者反覆強調的陳述
- 經驗法則（「先做X再做Y」、「絕對不要Z」）
- 具名原則（例如：「史托克戴爾弔詭 (Stockdale Paradox)」）
- 作者從證據中得出的結論

**萃取格式：**

```markdown
## [原則名稱] ([English Name])

**來源：** 第[N]章

### 描述
[1-2句，捕捉原則的核心。]

### 適用時機
- [情境]
```

## 3. 心智模型萃取

心智模型是一種思考問題的方式 — 不是要遵循的流程，而是觀看問題的透鏡。

**如何識別：**
- 「把它想成...」或「想像...」的表述
- 作者發展成推理工具的隱喻
- 框架化方式（例如：「沉默 vs. 暴力」作為兩種失敗模式）

**萃取格式：**

```markdown
## [模型名稱] ([English Name])

**來源：** 第[N]章

### 描述
[這個模型如何框架化問題。]

### 如何使用
[何時及如何運用這個思考透鏡。]
```

## 4. 反模式萃取

反模式是書中明確警告不要做的事。

**如何識別：**
- 「不要做X」、「避免Y」、「大多數人犯的錯誤是...」
- 具名反模式（例如：「乍看二選一 (Sucker's Choice)」）
- 作者用作警示的失敗故事

**萃取格式：**

```markdown
## [反模式名稱] ([English Name])

**來源：** 第[N]章

### 描述
[這個錯誤是什麼。]

### 應該怎麼做
[作者建議的替代方案。]
```

## 5. 案例研究萃取

案例研究是作者用來說明框架或原則的真實案例。

**萃取格式：**

```markdown
## [案例標題]

**來源：** 第[N]章
**相關框架：** [列表]

### 摘要
[2-3句：發生了什麼。]

### 關鍵教訓
[作者希望讀者從中學到什麼。]
```

**規則：**
- 只萃取能說明特定框架或原則的案例
- 每個案例必須連結到它所說明的框架
- 摘要保持事實性 — 教訓是作者的，不是你的

## 6. 應用觸發點對應

萃取完所有項目後，建立應用觸發點索引。

**流程：**
1. 為每個框架和原則列出3-5個適用情境
2. 使用使用者描述情境時會用的自然語言
3. 包含用語變化（例如：「給回饋」、「績效面談」、「困難對話」）
4. 按主題分組

**格式：**
```markdown
### [情境描述]
→ [框架名稱]（[書名]），[原則名稱]（[書名]）
```

## 7. 品質標準

好的萃取：
- 使用作者的專有名詞
- 包含完整步驟（不是「詳見原書」）
- 每個框架有3-5個應用觸發點
- 每個框架至少有1個關鍵引文
- 反模式連結到相關框架
- 交叉引用知識庫中其他書籍的相關框架

## 8. 常見錯誤

1. **過度萃取**：不是每個有趣的段落都是框架。只萃取具有名稱或明確結構的項目。
2. **描述太模糊**：「當個好領導者」不是原則。「在分享結論前先聽取數據」才是。
3. **遺漏步驟**：如果框架有5個步驟，就萃取全部5個。不要把5步流程縮減為3步。
4. **混淆案例與框架**：關於某位CEO的故事是案例研究，不是框架 — 除非作者從中推導出具名模型。
5. **跳過應用觸發點**：沒有觸發點，顧問就無法配對情境與框架。這是最常被跳過但最重要的步驟。

## 9. 中英對照命名

萃取中文材料時：
- 所有內容以原文繁體中文儲存
- 每個框架/原則名稱後加括號標註英文翻譯
- 範例：「刺蝟原則 (Hedgehog Concept)」
- 應用觸發點使用繁體中文
- 關鍵引文保持原文
