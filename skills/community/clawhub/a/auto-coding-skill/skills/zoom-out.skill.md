# Zoom-Out：全局视角

## 1. Overview（概述）

不看全局就审查细节 = 盲人摸象。本技能强制在代码审查前先 zoom-out — 理解改动在整体系统中的位置、跨模块的交互关系、上游调用者和下游依赖。只有理解全局，才能判断局部是否正确。

**核心原则**：
- 不理解的代码不能有效审查
- 多文件改动（>3 文件）必须 zoom-out
- 跨模块影响是最高风险点

## 2. When to Use（使用条件）

**强制触发**：
- modified_file_count > 3 → 自动触发 zoom-out
- Step 5 反思阶段（code-review 之前）
- 涉及核心模块（data model、API 契约、配置入口）

**可跳过**：
- 单文件 ≤ 50 行改动且无跨文件调用
- A 级任务（micro fix）

## 3. Process（执行流程）

### 3.1 Zoom-Out 四步检查

```
1. 系统位置
   ├─ 这个模块在系统中是什么角色？（入口/中间层/基础设施）
   ├─ 这个文件的职责边界在哪？
   └─ 它被哪些上游模块依赖？

2. 交互识别
   ├─ 改动影响了哪些跨模块接口？
   ├─ 数据流经过哪些模块？
   └─ 时序依赖是什么？（A 必须在 B 之前调用？）

3. 调用者/依赖分析
   ├─ 谁调用了被改动的函数/类？
   ├─ 被改动的代码依赖了什么？
   ├─ 调用者是否需要同步修改？
   └─ 依赖的行为是否受改动影响？

4. 边界检查
   ├─ 改动是否引入了新的模块间耦合？
   ├─ 有没有打破现有的抽象边界？
   └─ 错误会传播到哪一层？
```

### 3.2 Zoom-Out 输出格式

```
## Zoom-Out 分析

### 改动范围
- 文件数: {n}
- 核心文件: {列表}

### 系统位置
{模块角色 + 职责边界}

### 跨模块影响
- 上游（调用者）: {列表 + 影响评估}
- 下游（依赖）: {列表 + 影响评估}

### 风险点
- 🔴 {高风险项}
- 🟡 {中风险项}
```

## 4. Risk Scorecard（反借口表）

| # | 纪律 | 常见借口 | 反驳 |
|---|------|---------|------|
| Z1 | 多文件改动必须 zoom-out | "这段代码我熟，不用看上下文" | 熟悉 ≠ 理解当前改动的影响 |
| Z2 | 不只看 diff，看完整文件 | "就看 diff 够了，改哪看哪" | diff 隐藏了调用者期望和上下文 |
| Z3 | 调用者必须检查 | "这个方法只有测试调用" | 除非 grep 确认，否则不假设 |
| Z4 | 模块边界不能破 | "就加一个 import 而已" | 跨层 import = 架构腐化的起点 |

## 5. Red Flags（危险信号）

- 🚩 改动跨 3 个以上模块（层）— 可能是架构问题而非编码问题
- 🚩 新增 import 来自不应该依赖的方向（controller → model 内的方法 → 违反分层）
- 🚩 调用者检查显示 ≥ 2 个上游需要同步修改 — 接口契约可能设计不当
- 🚩 zoom-out 分析无法在 2 分钟内完成 — 改动可能太大，应拆分为多个 PR
- 🚩 修改了入口文件（main.py / app.py）但没改对应测试 — 回归风险

## 6. Verification（自检清单）

**必检（≤5 项）**：
- [ ] zoom-out 分析已完成（系统位置、交互模块、调用者、依赖）
- [ ] 所有跨模块影响的调用者已检查
- [ ] 没有异常方向的依赖引入（违反分层）
- [ ] 边界检查已通过（无新耦合，无抽象泄漏）

**抽检（≤3 项）**：
- [ ] zoom-out 在 2 分钟内完成（否则改动太大）
- [ ] 风险点排序合理（🔴 在 🟡 之前）
- [ ] 分析结果已写入阶段产物

**免检**：分析格式、措辞风格
