───────────────────────────────────────
【第 1 层：缺口 GAP】核心问题 + 既往方案不足
───────────────────────────────────────

▎核心问题
问题：大语言模型自回归推理的串行瓶颈（每个 token 必须等前一个生成）
重要性：LLM 推理成本占总成本 80%+，推理速度直接影响产品可用性
不解决的后果：实时对话、代码补全等场景延迟过高，商业化受阻

▎既往方案
方案1：模型蒸馏 → 训练小模型模拟大模型 → 牺牲精度，且需重新训练
方案2：模型量化（INT8/INT4）→ 减少单次计算量 → 精度下降，硬件依赖
方案3：Early Exit → 浅层预测后提前退出 → 适用场景有限，长文本失效
方案4：剪枝 → 移除部分参数 → 精度损失大，结构不灵活
方案5：并行解码（多分支）→ 多个候选并行生成 → 一致性难保证，质量不稳

▎本文战略位置
在"大模型推理速度"问题上，针对"自回归串行生成的本质瓶颈"，
提出"用小模型草稿 + 大模型验证"的投机解码方案，**不损失精度**地实现 2-3 倍加速

───────────────────────────────────────
【第 2 层：增量 DELTA】作者挪动了什么
───────────────────────────────────────

▎一句话结构化
在大模型推理问题上，将生成方式从「自回归逐 token 串行」改为「小模型草稿并行
 + 大模型批量验证」，通过"投机+验证"机制实现 2-3 倍速度提升且零精度损失

▎核心机制
1. 草稿阶段：用小型近似模型（Speculator）快速生成 K 个候选 token
2. 验证阶段：用大模型对这 K 个 token 做一次并行 forward，验证接受/拒绝
3. 接受准则：基于概率比 r = p_large(x) / p_small(x)，若 r ≥ 1 则接受，否则拒绝
4. 关键洞察：小模型生成"接近"大模型分布的 token 时，验证通过率高，整体加速

▎对比维度
- 速度：2-3 倍（取决于草稿模型质量）
- 精度：完全保持（数学上等价于自回归采样）
- 成本：略增（小模型推理 + 大模型并行验证，但总时间减少）
- 适用性：广泛（任何自回归模型都适用）

关键差异标注：
- * 草稿并行生成 K 个候选：利用小模型速度优势
- * 大模型批量验证：单次 forward 验证多个 token，摊销串行成本
- * 数学等价性：接受准则保证分布完全一致

───────────────────────────────────────
【第 4 层：启发 INSPIRATION】跨域迁移
───────────────────────────────────────

▎Inversion（反转）
Inversion 1：把"小模型草稿 → 大模型验证"反转为"大模型草稿 → 小模型验证"
  - 应用场景：模型输出的"自信度审计"——用小模型反向筛选大模型的"自信错误"
  - 推理链：大模型自信输出但小模型拒绝 → 可能是大模型过拟合/幻觉信号
  - 价值：可作为幻觉检测的辅助信号

Inversion 2：把"加速生成"反转为"加速压缩"——用大模型验证哪些信息可以丢弃
  - 应用场景：长文本摘要、关键信息提取
  - 价值：小模型先草拟"看似重要"的内容，大模型验证是否真的重要

▎Mapping（迁移）
Mapping 1：搬到「代码评审」领域
  - 抽象机制：用"快速近似"+"精确验证"的两阶段流水线
  - 套用方式：lint 工具快速检查（草稿） + 人工 review（验证）
  - 价值：不是所有代码都需要深度 review，先快速筛选再重点 review

Mapping 2：搬到「投资决策」领域
  - 抽象机制：用"低成本筛选"+"高质量验证"加速决策
  - 套用方式：技术指标快速初筛（草稿） + 基本面深度研究（验证）
  - 价值：避免对每个标的都做深度研究，提升投资研究效率

Mapping 3：搬到「学习认知」领域
  - 抽象机制：先建立"快速直觉"，再"慢速验证"深化理解
  - 套用方式：先读综述建立框架（草稿） + 重点精读核心论文（验证）
  - 价值：避免一开始就陷入细节，提升学习效率
