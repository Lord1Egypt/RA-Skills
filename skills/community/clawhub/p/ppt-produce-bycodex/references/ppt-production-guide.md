# 企业应用软件解决方案 PPT PNG 制作通用接力提示词

你现在接手一套“企业应用软件解决方案 PPT / PNG 页面 / 后续 PPTX 组装”的制作或延续任务。请先完整阅读本提示词，再检查当前工作区文件状态，以用户提供的最新资料、当前文件和最新口径为准继续执行。

本提示词适用于任意企业应用软件解决方案场景，例如保险、车险、金融、政企、制造、医疗、运营平台、AI 中台、数据治理、风控、营销、客服、低代码、知识库、业务工作台等。它不是某一个具体行业的模板，使用时请把行业内容替换为当前项目的真实业务。

## 0. 使用前先替换的项

开始新项目时，先确认并替换下面占位符：

- 项目名称：`{项目名称}`
- 工作目录：`{项目根目录}`
- 方案主题：`{解决方案主题}`
- 客户对象：`{客户/行业/业务部门}`
- 当前版本：`{v1/v2/v3/...}`
- PPT PNG 目录：`{项目根目录}/workspace/deliverables/{版本}/png`
- PPTX 输出路径：`{项目根目录}/workspace/deliverables/{版本}/{方案名}.pptx`
- 接触表路径：`{项目根目录}/workspace/deliverables/{版本}/contact-sheets/{接触表名}.png`
- 方案 MD：`{项目根目录}/workspace/deliverables/输出成果/{方案名}.md`
- PPT 大纲 MD：`{项目根目录}/workspace/deliverables/输出成果/{PPT大纲名}.md`
- 生成脚本目录：`{项目根目录}/workspace/scripts/`
- 默认标题栏样板：`~/.codex/skills/ppt-produce-bycodex/assets/titlebar-template/enterprise-solution-titlebar-4k.png`

如果发现当前项目里已经存在更高版本目录，例如 `v4` 高于用户口头提到的 `v3`，不要直接沿用旧路径，应先向用户确认以哪个版本为准。

## 1. 工作目录与文件规范

所有新生成、重绘、整理、临时、脚本、备份、接触表、验证结果，都必须放到项目根目录下的 `workspace/`。

推荐分类：
- `workspace/deliverables/`：最终交付物
- `workspace/assets/`：可复用素材、截图、参考图、客户原始图副本
- `workspace/runs/`：生成批次、staging 输出、备份、接触表
- `workspace/tmp/`：临时 prompt、OCR、解包验证、中间脚本
- `workspace/scripts/`：可复用生成脚本、PPTX 组装脚本、检查脚本

不要新增根目录级 `output/`、`outputs/`、`tmp/`、`scripts/`、`设计图与原型/` 等杂乱目录。

创建或修改任何文件后，最终回复必须告诉用户完整绝对路径，避免只说文件名。

## 2. 解决方案内容主线

企业应用软件方案不能只堆功能点。整套 PPT 必须有一条客户能听懂的业务主线。

推荐结构：
1. 背景与挑战：行业变化、业务痛点、现有流程瓶颈、建设必要性。
2. 方案定位：本方案解决什么问题，不解决什么问题，边界在哪里。
3. 总体架构：业务架构、应用架构、系统架构、数据/AI/集成架构。
4. 核心能力：围绕业务流程展开能力，不孤立罗列模块。
5. 场景落地：以用户旅程、业务流程、作业链路或管理闭环呈现。
6. 价值收益：效率、质量、风控、成本、体验、运营、资产沉淀。
7. 实施路径：试点范围、阶段推进、验收指标、风险控制、合作建议。

表达原则：
- 先讲客户问题，再讲方案。
- 先给整体，再下钻能力。
- 先讲业务闭环，再讲技术支撑。
- 每个能力都要讲清输入、处理、输出、业务价值。
- 对复杂内容，优先用流程、矩阵、分层架构、证据链、生命周期、能力地图表达。
- 不做大段文字堆叠，不做空泛口号页。

## 3. 业务边界与过度承诺控制

企业方案要稳，尤其涉及 AI、风控、自动化决策、第三方数据、客户数据、安全合规时，不能把演示效果写成确定承诺。

通用边界表述：
- AI/算法负责识别、抽取、摘要、推荐、生成初稿、辅助判断。
- 业务规则负责确定性校验、阈值判断、流程路由、权限控制。
- 人负责例外处理、高风险判断、争议决策、最终确认。
- 系统负责留痕、审计、版本管理、权限控制、结果回流。

避免表达：
- “AI 直接替代人工最终决策”
- “模型保证 100% 准确”
- “所有数据都可以直接获取”
- “上线即可全流程无人化”
- “自动承诺收益、节省比例、合规结论”

涉及接口、第三方数据、模型效果、监管合规、客户数据使用时，要写清“以实际授权、数据质量、接口开放、试点评估和合同约定为准”。

## 4. PPT PNG 生成硬性要求

如果使用图片模型生成或重绘 PPT 页面：

- 使用 `imagegen` 技能：`~/.codex/skills/.system/imagegen/SKILL.md`。
- 最终 PNG 必须通过 `gpt-image-2` 或用户指定的图像模型整页生成/整页重绘。
- 不允许用本地 HTML/CSS/Canvas/PPT 元素编排最终页面内容。
- PPTX 只负责把最终 PNG 全幅铺满页面，不做本地重排，不加本地文本框、形状、页码、备注。
- 生成前检查 `~/.zshrc` 中 `OPENAI_API_KEY` 和 `OPENAI_BASE_URL`，并调用 `/models` 确认 `gpt-image-2` 可用。
- 不允许使用 low 质量。
- 普通页面至少使用 `2048x1152 high`。
- 复杂架构页、信息密集页、客户重点页建议使用 `3840x2160 high`，必须模型原生输出。
- 禁止本地 2K 放大成 4K。
- 页面不要出现固定页码，方便后续在 PPT 中拖拽调整顺序。
- 页面不要出现水印、prompt 字段名、任务要求、内部讨论、备注、草稿标记。
- 如果在没有明确允许低分辨率的情况下，模型生成图片小于 2K，应提醒用户可能是中转服务商模型能力或参数异常，停止批量生成，避免 token 浪费。

### 413 / Too-Large 硬停止策略

把以下报错视为同一类失败：`413`、`Payload Too Large`、`Request Entity Too Large`、`response too large`、`upstream_status: HTTP 413`、`CC Switch local proxy failed`、Codex `/responses` 请求过大。

处理策略：

- 4K 原生生成或 4K edit 只要出现 1 次 too-large 类报错，就不要再重试 4K。
- 立即降级到原生 2K high，并保持同等内容密度和业务表达丰富度。
- 禁止为了让 4K 成功而压缩内容、删掉关键结构、改成极简页。这样会让页面失去方案表达力。
- 如果 2K 承载不下原内容，应拆成多页，而不是把内容抽空。
- 降级到 2K 后，同一页最多尝试 2 次；不管是什么报错，只要 2 次都失败，就终止当前生成进程。
- 终止后不要继续调用 imagegen，不要继续重发大图、大 prompt 或大量上下文。
- 如果工具还能写文件，在项目 `workspace/` 下生成紧凑接力提示词；如果工具不可用，就在最终回复中直接给紧凑接力提示词，提醒用户开新会话继续。
- 接力提示词只保留：项目路径、当前任务、失败页面、最后报错类型、已成功产物、待完成产物、下一步建议。不要嵌入大图、base64、长 prompt 或大量文件内容。

检查 API 参考命令：
```bash
cd {项目根目录}
zsh -lc 'source ~/.zshrc >/dev/null 2>&1 || true; printf "OPENAI_BASE_URL=%s\n" "${OPENAI_BASE_URL:-}"; if [ -n "${OPENAI_API_KEY:-}" ]; then echo "OPENAI_API_KEY=present"; else echo "OPENAI_API_KEY=missing"; fi; curl -sS -H "Authorization: Bearer ${OPENAI_API_KEY}" "${OPENAI_BASE_URL%/}/models" | python3 -c "import sys,json; data=json.load(sys.stdin); ids=[m.get(\"id\",\"\") for m in data.get(\"data\",[])]; print(\"gpt-image-2 available=\" + str(any(i==\"gpt-image-2\" for i in ids))); print(\"model_count=\" + str(len(ids)))"'
```

检查残留进程：
```bash
ps -ef | rg -i 'image[_-]?gen|gpt-image|openai|run-one|longrun' | rg -v 'rg -i'
```

## 5. 局部修图硬性要求

如果只是修复某页局部问题，例如去页码、修标题、修坏字、调整局部图表：

- 优先基于原图做 `gpt-image-2 image edit`。
- 不要在坏图上本地半透明盖遮罩。
- 不要用本地色块、文字贴片、截图贴图硬盖问题区域。
- 不要直接用本地画图工具补字，否则容易出现底图漏边、覆盖生硬、字体不一致、布局漂移。
- 应以源图为内容基准，用中/高质量 2K+ 修复问题区域。
- 模型输出后，只取修复区域，与源图未改区域像素级拼回。
- 最后校验尺寸、RGB 不透明、无残影、未改区域与源图逐像素一致。
- 不要直接覆盖正式文件。先输出到 `workspace/runs/<task>/` 或 `workspace/tmp/<task>/`，生成接触表/预览，通过后再覆盖正式 PNG。

如果修复标题统一风格：
- 不要把整页参考图直接喂给模型，否则模型可能把参考页正文串到目标页。
- 应裁出“只包含头部标题条”的参考区域，例如只含图标、标题、细横线的头部参考图。
- 标题修复看似小改，但 image edit 可能整页重绘，必须检查正文是否被破坏。

## 6. PPTX 组装策略

不要每次修改一两页 PNG 后都立刻重建 PPTX。

推荐策略：
- 单页或少量 PNG 连续调整阶段：只生成/修复 PNG、接触表和必要预览。
- 多页已经确认，或准备发给领导/客户前：再阶段性重建 PPTX。
- 用户明确要求“转成 PPTX”“生成最终 PPTX”“打包交付”时，再重建 PPTX。

PPTX 只做一件事：
- 每页插入一张最终 PNG。
- 图片从页面左上角开始，全幅铺满 16:9 页面。
- 不添加本地文本框、形状、页码、备注、占位符。

组装后必须验证：
- 页数正确。
- 页序正确。
- 每页正好 1 张全幅图片。
- 无本地文字框、形状、页码。
- 无 notesSlides、notesMasters、备注关系残留。
- PNG 尺寸、色彩空间符合要求。

## 7. 视觉风格要求

默认视觉风格：
- 企业商务风 + 互联网产品方案感。
- 浅色底，蓝白主色调。
- 少量金色、橘黄色、青绿、紫色作为重点、风险、价值、能力点缀。
- 高端、干净、克制、可信，有技术感但不过度炫技。
- 页面不能像普通 Word 截图，也不能像套模板的营销海报。

背景建议：
- 柔和渐变
- 浅色科技纹理
- 低透明数据流
- 细连接线
- 轻玻璃拟态层次
- 局部科技光带
- 业务相关的浅色抽象背景

谨慎使用：
- 大面积网格线。网格背景容易显乱，除非非常轻、非常淡。
- 大面积深蓝。深色页可用于封面或少数章节页，但不要全篇压暗。

避免：
- 赛博霓虹
- 强发光
- 廉价 3D 堆叠
- 大面积深色背景
- 单一蓝色铺满全篇
- 过度紫蓝渐变
- 过花的装饰元素
- 标题超大压屏
- 背景元素抢正文重点

信息密集页：
- 优先保证可读。
- 图表/截图/系统界面作为主内容时，除标题和必要标签外应尽量撑满页面。
- 不要把两张关键截图硬塞一页导致不可读，必要时拆页。
- 表格和矩阵页使用短词短句，避免小号中文长段。

## 8. 标题系统统一要求

整套 PPT 必须建立统一标题系统。

新 PPT 绘制任务的第一步，不只是生成样张，还必须确认标题栏样板：

- 默认使用技能内置标题栏样板：`assets/titlebar-template/enterprise-solution-titlebar-4k.png`。
- 标题栏样板说明见：`references/titlebar-template.md`。
- 第一轮样张应生成 3-5 页，并同步呈现/确认主标题栏样式。
- 可以提示用户是否希望更换标题栏样式；如果用户没有更换诉求，使用默认样板。
- 标题栏样板确认后，所有正常内容页必须使用同一份主标题栏样式生成。
- 标题栏必须只有一行标题文字，禁止出现两行或多行标题。
- 单行标题必须在标题栏垂直方向居中，不能偏上或偏下。
- 如果标题过长，优先压缩文案、在既定样式内适度缩小字号，或请用户确认更短标题；禁止直接换行。
- 封面、目录页、章节过渡页可以有适度变化，但普通内容页不要混用多个标题系统。
- 技能内置标题栏来自一个历史项目，但作为通用样式资产使用，生成新项目时不得出现历史项目行业词或客户名。

正常内容页：
- 标题左对齐，不要居中。
- 标题建议使用深蓝粗体。
- 左侧可配统一小图标，例如蓝色斜切双条、短竖条、圆形能力图标、简洁科技图标。
- 标题下可配浅蓝细横线或淡色装饰线。
- 标题不要超大，不要压屏。
- 同一批页面不要混用太多标题图标、字体、字号、位置。

封面、目录页、章节页：
- 可以特殊处理。
- 同类页面之间仍需统一。
- 章节页应简单克制，主标题 + 少量关键词即可，不要放复杂架构图。

标题文案：
- 面向客户。
- 简洁、有判断、有业务含义。
- 不要把“设计要求”“章节规划”“评审意见”“任务说明”当标题。
- 不要出现“本页需要说明”“用于引出”“客户可能会问”“内部准备”等过程话术。
- 副标题可以概括本页业务含义，但不要像项目组备注。

## 9. 页面类型设计建议

封面：
- 方案名清楚，背景应与业务主题相关。
- 高端、克制、有业务感，不要只做抽象渐变。

目录：
- 章节简洁，结构不要过密。
- 目录文案应能体现汇报逻辑。

章节页：
- 承担节奏切换，不讲太多内容。
- 主标题 + 关键词即可。

背景/挑战页：
- 用对比、流程痛点、行业变量、现状差距表达。
- 不要写成政策或新闻摘要堆砌。

定义/定位页：
- 讲清方案是什么、解决什么、边界是什么。
- 可用四要素、三层定位、能力地图表达。

架构页：
- 先总后分，层级清楚。
- 可按“用户/渠道层、应用层、能力层、数据层、集成层、安全治理层”组织。
- 避免所有东西挤在一页。

流程页：
- 从真实业务流程出发。
- 清楚标注系统处理、AI 辅助、人工确认、外部协同、结果回流。
- 不要让 AI 看起来越权替代最终决策。

能力页：
- 每个能力讲清输入、处理、输出、价值。
- 与业务流程对应，不孤立讲技术名词。

场景页：
- 从用户角色、业务动作、系统呈现、价值收益出发。
- 用工作台、流程链、操作界面、闭环图表达。

试点/落地页：
- 讲清范围、阶段、指标、验收、风险边界。
- 不宜过早暴露过细工作量，除非用户明确要求。

总结页：
- 回应客户关键问题。
- 给出下一步合作建议。

## 10. Prompt 写法要求

必须严格区分“可见文字”和“设计说明”。这是避免模型把任务要求画进页面的关键。

推荐 prompt 结构：
```text
STRICT VISIBLE TEXT ONLY:
只列页面上允许出现的客户可见短词短句。

DESIGN BRIEF - DO NOT RENDER AS TEXT:
碎片化设计指令，不写完整可复制副标题。

LAYOUT BRIEF - DO NOT RENDER AS TEXT:
布局、层级、元素关系。

VISUAL STYLE - DO NOT RENDER AS TEXT:
颜色、质感、背景、标题风格。

CONTENT EMPHASIS - DO NOT RENDER AS TEXT:
这一页要强调的业务含义。

FORBIDDEN VISIBLE TEXT:
描述式禁令，不要把敏感内部词原文塞太多，避免模型误抄。
```

可见文字原则：
- 短
- 准
- 客户可讲
- 有业务含义
- 不写内部过程
- 不写 prompt 字段名
- 不写“要求”“说明”“设计意图”
- 不写无法承诺的绝对化表达

设计说明原则：
- 碎片化
- 不写成长句副标题
- 不写可能被复制到页面的完整句子
- 对禁用词尽量描述式表达，避免大量原文诱导误抄

禁止可见内容示例：
- `STRICT VISIBLE TEXT ONLY`
- `DESIGN BRIEF`
- `prompt`
- `任务要求`
- `设计说明`
- `内部沟通`
- `客户可能会问`
- `提前准备`
- `本页用于引出`
- `严格基于某某草图`
- `不要画成...`

## 11. 批量生成工作流

不要一口气跑完整套。建议小批量推进。

推荐流程：
1. 读取用户资料、现有大纲、方案 MD、参考图。
2. 明确页序、每页标题、页面类型、尺寸、生成方式。
3. 确认主标题栏样板，默认使用技能内置 `assets/titlebar-template/enterprise-solution-titlebar-4k.png`，并询问用户是否需要更换标题样式。
4. 检查 API、模型、网络、残留进程。
5. 先生成 3-5 页关键样张，样张必须体现已确认标题栏样式。
6. 检查风格、标题栏一致性、文字、尺寸、RGB、是否有页码。
   - 标题栏必须只有一行标题文字，禁止多行标题。
   - 标题栏文字必须垂直居中，不能偏上或偏下。
7. 用户确认样式后，再分批生成。
8. 每批保留 prompt、manifest、源图备份、staging 输出。
9. 每批生成接触表做整体观感审核。
10. 只有审核通过后才覆盖正式交付 PNG。
11. 多页确认后再询问是否需要阶段性生成 PPTX。
12. 最后生成全量接触表和 PPTX。
13. 解包验证 PPTX 结构。

并发建议：
- 不要并发过高。
- 4K high 通常耗时较长，2 页并行相对稳妥。
- 如果 4K 已出现 too-large 类报错，后续不要通过“更极简的 4K prompt”换取成功；改用同内容密度的 2K 或拆页。
- 每批生成前检查是否有残留 `image_gen.py`、`gpt-image`、`openai` 相关进程。

## 12. 质量检查清单

每页检查：
- 文件在正确目录下。
- PNG 尺寸符合要求，通常为 `2048x1152` 或模型原生 `3840x2160`。
- RGB 或不透明输出，无 alpha 透明异常。
- 无固定页码。
- 无水印。
- 无 prompt 字段名。
- 无内部讨论或任务要求。
- 无乱码、错字、别字。
- 正常内容页使用同一份已确认主标题栏样板。
- 标题栏只有一行标题文字，未出现换行、两行标题或多行标题。
- 标题栏文字在垂直方向居中。
- 标题/副标题没有把要求句照搬。
- 内容与本页主题一致。
- 主体布局没有漂移。
- 图标风格统一。
- 颜色不过深、不过花。
- 文字不太小。
- 卡片、箭头、流程对齐。
- 无低清、糊字、变形。
- 无参考图内容串页。
- 主图/截图/架构图在播放场景下足够大。

全套检查：
- 章节逻辑递进。
- 同类页面统一。
- 过渡页克制。
- 架构页总分清楚。
- 能力页能回到主线。
- 场景页有业务代入感。
- 边界说明清楚。
- 页序和文件名匹配。
- 接力提示词、方案 MD、PPT 大纲、PNG 口径一致。
- PPTX 只有全幅图片。

检查尺寸参考命令：
```bash
cd {项目根目录}
python3 - <<'PY'
from pathlib import Path
from PIL import Image

png_dir = Path("{PPT PNG 目录}")
for p in sorted(png_dir.glob("*.png")):
    im = Image.open(p)
    print(p.name, im.size, im.mode)
PY
```

关键词检查可结合 OCR、脚本或人工放大，但不要只依赖 OCR。中文小字 OCR 容易误识别，关键页必须放大人工看。

## 13. 常见踩坑与规避

坑 1：把整页参考图作为风格参考，导致参考图正文串到新页。
规避：只截取需要参考的局部，例如标题条、色板、卡片样式，而不是整页。

坑 2：prompt 写了完整要求句，模型直接抄成标题。
规避：可见文字单独列，设计说明碎片化，不写可复制长句。

坑 3：标题样式每页不同。
规避：开工先确认标题栏样板，默认使用技能内置标题栏；确认后所有正常内容页都引用同一份主标题栏样式。

坑 4：过渡页做得过复杂。
规避：过渡页只承担节奏切换，不讲架构，不堆信息。

坑 5：局部错字用本地覆盖。
规避：走 image edit 重绘，并备份源图，修复后像素级拼回。

坑 6：生成后只看缩略图。
规避：接触表看整体，关键页放大看文字。

坑 7：PPTX 里混入本地文字框。
规避：PPTX 只铺 PNG，最后解包检查。

坑 8：页面过度承诺。
规避：涉及第三方数据、接口、模型效果、合规边界时写清“以实际开放、授权、验证结果为准”。

坑 9：截图或架构图太小。
规避：图表/截图页除标题外尽量撑满整页，必要时拆成两页。

坑 10：背景统一网格导致商务感下降。
规避：背景保持浅色层次，网格线只作为极淡纹理，不能成为主视觉。

坑 11：页码画进 PNG。
规避：所有 prompt、脚本、PPTX 组装都禁止页码。

坑 12：文件名、页序、manifest 残留旧标题。
规避：标题或页序变化后，同步清理文件名、manifest、接力提示词、PPT 大纲。

## 14. 方案 MD 与 PPT PNG 一致性

如果涉及方案内容调整，而不只是视觉修图，需要同步检查并按需更新：
- 方案 MD
- PPT 大纲 MD
- PNG 页标题和内容
- 接力提示词中的当前页序/已确认事项
- 后续 demo 或原型说明

不需要每次小修图都更新全部文档。可以在一批任务完成后统一更新，避免频繁消耗时间和 token。

判断是否需要更新方案 MD：
- 能力名称变了：需要。
- 业务流程变了：需要。
- 架构分层变了：需要。
- 合规或责任边界变了：需要。
- 实施阶段或验收口径变了：需要。
- 只是去页码、修错字、调布局：通常不需要。

## 15. Demo / 系统原型延展要求

如果从 PPT 延展到系统原型 demo：
- 不要做营销落地页，应做成可操作的企业应用工作台。
- 默认展示“业务对象进入 -> 数据/材料处理 -> 智能分析 -> 规则/流程判断 -> 人工确认 -> 结果输出 -> 反馈回流”的闭环。
- 风格与 PPT 保持一致：浅色底、蓝白主调、少量橙/金/绿/紫点缀，企业级工作台风格。
- 信息密度适中，控件完整，状态清楚。
- 常见控件包括筛选、列表、详情面板、步骤流、风险提示、证据卡、摘要卡、规则依据、人工复核按钮、结果回流记录。
- 不要引入真实敏感数据，使用高质量 mock data。
- 如果是前端开发，默认使用 `/frontend-design`、`/ui-ux-pro-max`、`design-taste-frontend` 技能要求保证 UI 质量。
- 如需本地运行，启动 dev server 并给出 URL。
- 代码和资产仍放在项目 `workspace/` 下合适目录，避免污染根目录。

## 16. 新项目启动工作流

接手一个新企业应用软件解决方案 PPT 时，先做：
1. 读取用户提供的业务资料、会议纪要、旧 PPT、截图、方案文档。
2. 提炼业务主线，不急着生成页面。
3. 输出 PPT 大纲和页序，确认章节逻辑。
4. 定义视觉系统，包括封面、标题系统、配色、图标、卡片、流程、架构页风格。
5. 先确认主标题栏样板，默认使用技能内置样板，并提示用户是否有更换诉求。
6. 先生成 3-5 页样板：普通内容页、复杂图/架构页、截图/数据页或章节页等。
7. 用户确认后批量生成。
8. 过程中保存 prompt、staging、接触表、备份。
9. 阶段性提醒用户是否需要生成 PPTX，不要每次小改都重建。
10. 最终输出 PNG、PPTX、接触表、方案 MD 或大纲 MD。
