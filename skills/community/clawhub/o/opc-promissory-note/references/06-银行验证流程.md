# 银行验证流程

> 开证行核实、SWIFT验证、确认函流程的完整指南

---

## 一、开证行核实步骤

### 1.1 SWIFT代码查询

**SWIFT代码定义**：
- SWIFT代码又称BIC代码（Bank Identifier Code）
- 标准格式：8位或11位字符
- 例如：中国银行北京分行 = BKCHCNBJ300

**查询渠道**：

| 查询渠道 | 网址 | 说明 |
|----------|------|------|
| SWIFT官网 | swift.com/ 참조 | 官方权威渠道 |
| 中国银监会 | cbirc.gov.cn | 国内银行信息 |
| 银行官方网站 | 各银行官网 | 核实SWIFT代码 |
| 银行年鉴 | 纸质/电子版 | 传统核实方式 |

**查询步骤**：
1. 登录SWIFT官方网站或银行官网
2. 输入银行名称或SWIFT代码
3. 核实银行信息是否匹配
4. 记录查询结果

**核实要点**：
| 核实项 | 标准 | 异常情况 |
|--------|------|----------|
| SWIFT代码格式 | 8位或11位规范格式 | 格式错误 |
| 银行名称匹配 | 与票面一致 | 不匹配 |
| 分行信息 | 与票面一致 | 不匹配 |
| 状态 | 正常运营 | 已撤销/不存在 |

### 1.2 银行年鉴核实

**银行年鉴来源**：
- 《银行年鉴》（国际版）
- 《Bankers' Almanac》
- 《The Banker Yearbook》

**核实内容**：
| 核实项 | 说明 |
|--------|------|
| 银行存在性 | 年鉴中有该银行记录 |
| 银行基本信息 | 成立时间、总部地址 |
| 业务资质 | 是否具有开证资质 |
| 信用评级 | 评级机构评级情况 |

**注意事项**：
- 无法在年鉴中查到的银行需高度警惕
- 注意核实银行的具体分支机构和网点

### 1.3 信用评级核查

**主要评级机构**：

| 机构 | 名称 | 评级符号 |
|------|------|----------|
| 穆迪 | Moody's | Aaa-C |
| 标普 | S&P | AAA-D |
| 惠誉 | Fitch | AAA-D |

**评级参考标准**：

| 评级 | 含义 | 风险等级 |
|------|------|----------|
| AAA/AA/A | 高级别，偿债能力强 | 🟢低风险 |
| BBB | 中等，偿债能力尚可 | 🟡中等风险 |
| BB/B | 投机级，存在违约风险 | 🟠较高风险 |
| CCC/C | 高投机，极高违约风险 | 🔴高风险 |
| D | 违约 | 🔴无法合作 |

### 1.4 官方网站核实

**核实步骤**：
1. 登录银行官方网站（不要点击对方提供的链接）
2. 查找"联系我们"或"网点查询"
3. 核实官方SWIFT代码
4. 核实客服电话
5. 如有疑问，通过官方电话咨询

**核实内容**：

| 核实项 | 说明 |
|--------|------|
| SWIFT代码 | 与票面是否一致 |
| 联系方式 | 官方电话与票面是否一致 |
| 公告信息 | 是否有票据相关公告 |
| 业务范围 | 是否包含票据业务 |

## 二、SWIFT验证流程

### 2.1 SWIFT报文类型

| 报文类型 | 用途 | 适用范围 |
|----------|------|----------|
| **MT799** | 自由格式查询 | 票据真实性查询、付款确认 |
| **MT760** | 保函/备用信用证格式 | SBLC开证确认 |
| **MT999** | 加密信息格式 | 敏感信息传递 |

### 2.2 MT799查询格式

**标准格式**：
```
:20: 查询编号
:21: 相关业务编号
:79: 查询内容（自由格式）
     请确认以下本票的真实性：
     - 票据号码：[号码]
     - 金额：[金额]
     - 出票行：[银行名称]
     - 出票日期：[日期]
```

**查询内容**：
| 查询项 | 说明 |
|--------|------|
| 票据号码 | 票面记载的号码 |
| 金额 | 票面金额 |
| 出票行 | SWIFT代码 |
| 出票日期 | 票面日期 |
| 收款人 | 票面收款人 |

### 2.3 SWIFT验证流程图

```
┌─────────────────────────────────────────────────────────────┐
│                    SWIFT验证完整流程                         │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ┌──────────────┐                                           │
│  │ 1.发起查询   │ 通过开户行发起SWIFT查询                   │
│  └──────┬───────┘                                           │
│         ↓                                                  │
│  ┌──────────────┐                                           │
│  │ 2.填写报文   │ MT799格式，包含票据信息                    │
│  └──────┬───────┘                                           │
│         ↓                                                  │
│  ┌──────────────┐                                           │
│  │ 3.发送报文   │ 通过SWIFT网络发送至开证行                  │
│  └──────┬───────┘                                           │
│         ↓                                                  │
│  ┌──────────────┐                                           │
│  │ 4.等待回复   │ 通常1-3个工作日                           │
│  └──────┬───────┘                                           │
│         ↓                                                  │
│  ┌──────────────┐                                           │
│  │ 5.核对密押   │ 确认是否为银行加押回复                      │
│  └──────┬───────┘                                           │
│         ↓                                                  │
│  ┌──────────────┐                                           │

│  └──────────────┘                                           │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 2.4 SWIFT回复类型

| 回复类型 | 含义 | 后续处理 |
|----------|------|----------|
| **加押确认** | 票据真实，银行承认 | 可继续推进 |
| **加押否认** | 票据不存在或伪造 | 终止交易 |
| **无法确认** | 需进一步核实 | 补充材料后再查 |

## 三、银行确认函流程

### 3.1 确认函类型

| 类型 | 说明 | 效力 |
|------|------|------|
| **加押证实** | SWIFT加押报文确认 | 高 |
| **普通邮件** | 非加密邮件确认 | 低 |
| **电话确认** | 口头确认 | 需书面佐证 |

### 3.2 确认函核实步骤

```
┌─────────────────────────────────────────────────────────────┐
│                   银行确认函核实流程                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 发送查询请求                                            │
│     ↓                                                       │
│  2. 等待银行回复                                            │
│     ↓                                                       │
│  3. 核对加押密押                                           │
│     ↓                                                       │
│  4. 交叉验证信息                                           │
│     ↓                                                       │
│  5. 综合判断                                                │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 3.3 密押核对要点

| 核对项 | 标准 | 异常情况 |
|--------|------|----------|
| 密押格式 | 符合SWIFT规范 | 格式错误 |
| 密押位数 | 与预留一致 | 位数不符 |
| 有效期 | 在有效期内 | 密押过期 |
| 来源 | 来自官方SWIFT地址 | 非官方地址 |

## 四、假银行识别方法

### 4.1 识别要点

| 识别项 | 真实银行 | 假银行 |
|--------|----------|--------|
| SWIFT代码 | 可查询、标准格式 | 无法查询或格式错误 |
| 银行年鉴 | 有记录 | 无记录 |
| 开证资质 | 有资质 | 无资质或不明 |
| 通知行 | 银行为通知行 | 非银行金融机构 |
| 通讯方式 | SWIFT MT报文 | 普通邮件/电话 |

### 4.2 52D格式规避识别

**52D格式特点**：
- 52D格式可指定具体分行
- 可规避SWIFT系统的部分验证
- 是诈骗分子常用手法

**识别方法**：
| 特征 | 说明 |
|------|------|
| 使用52D格式 | 指定分行而非总行 |
| 多机构转通知 | 经多个机构转通知 |
| 不具备直接发送资格 | 无法直接发送加押报文 |

**风险提示**：
> 使用52D格式代替52A格式，可能是在避开SWIFT系统验证，需高度警惕！

### 4.3 非银行开证识别

| 特征 | 说明 |
|------|------|
| 开证机构非银行 | 企业或非银行金融机构 |
| 资质不明 | 无法核实业务资质 |
| 监管不明 | 不受金融监管 |

## 六、电话核实要点

### 6.1 核实步骤

1. **获取电话**：通过银行官方网站获取客服电话
2. **准备信息**：记录票据号码、金额等关键信息
3. **正式询问**：询问票据真实性
4. **记录核实**：记录核实时间和回复内容

### 6.2 核实话术示例

> "您好，我是[公司名称]，我司收到一张贵行开立的本票，票号为[号码]，金额为[金额]，出票日期为[日期]。请帮忙核实该本票是否真实存在。"

### 6.3 注意事项

| 注意事项 | 说明 |
|----------|------|
| 使用官方电话 | 不要使用对方提供的电话 |
| 记录核实过程 | 记录时间、接线人、回复内容 |
| 多渠道核实 | 建议同时使用SWIFT查询 |
| 保密信息 | 不要泄露过多交易细节 |
