# 你的项目，真的需要跑 QCSD 质量门禁吗？

> 本文作者：ai-acheng | 首发：赛博鸭子技术周刊

---

## 🤔 一个灵魂拷问

前段时间发了 QCSD Development Swarm 的介绍，很多同学问：

"这个看起来好厉害，我能不能用在我的个人项目上？"

我一般会反问三个问题：
1. 你的项目上线后如果出 bug，会有人赔钱吗？
2. 你的团队超过 10 个人了吗？
3. 你需要向合规部门提交审计报告吗？

如果答案都是 NO，相信我 —— 你不需要 QCSD。

**杀鸡不用牛刀，但很多人总想着用 CT 扫描仪治感冒。**

---

## 🏥 两个工具的本质区别

我打个最直白的比方：

### QCSD Development Swarm = 医院的 CT 扫描仪

- ✅ 极其精准，能看到毫米级的问题
- ✅ 输出 12 份专业报告，附医生诊断
- ✅ 可以检测出早期癌变（缺陷预测）
- ❌ 很贵，做一次几千块
- ❌ 很慢，排队 + 扫描 + 等报告 = 大半天
- ❌ 要专业医生才能看懂报告

**你不会因为有点头疼就去做 CT。**

---

### Harness Dev Standards = 家里的体温计

- ✅ 简单，拿起来就用，谁都会
- ✅ 快速，量一下几秒钟出结果
- ✅ 便宜，几块钱一个，家家都有
- ✅ 告诉你"发烧了，该去医院了"
- ❌ 不能告诉你具体是什么病
- ❌ 不能给你开处方

**你也不会拿着体温计去做手术。**

---

## 📊 一张表看懂适用场景

| 对比项 | QCSD Development Swarm | Harness Dev Standards |
|-------|-----------------------|----------------------|
| **定位** | 🏭 企业级质量工厂 | 🛡️ 个人质量标准 |
| **目标用户** | 大型企业开发团队 | 单人/小型团队 |
| **核心问题** | 能不能上生产？ | 能不能交付？ |
| **Agent 数量** | 10 个 specialist 并行 | 0 个（自己检查） |

| **Token 消耗** | 💰 极高（几万 token） | 💸 极低（几百 token） |
| **输出物** | 12 份专业报告 + 高管摘要 | 1 页检查报告 + 修复建议 |
| **裁决机制** | SHIP / CONDITIONAL / HOLD | 通过 / 需要修复 |

## 🔬 QCSD 到底有多重型？

很多人对 QCSD 的复杂度没有概念。

我给你列一下它的配置：

### 10 个专业 Agent，各司其职

| Agent | 职责 |
|-------|------|
| qe-tdd-specialist | TDD 合规性审计 |
| qe-code-complexity | 圈复杂度分析 |
| qe-coverage-specialist | 测试覆盖率深度分析 |
| qe-security-scanner | 安全漏洞扫描（OWASP Top 10） |
| qe-performance-tester | 性能基准测试 |
| qe-mutation-tester | 突变测试（测试用例质量） |
| qe-message-broker-tester | 消息中间件健康检查 |
| qe-sap-idoc-tester | SAP IDoc 接口审计 |
| qe-sod-analyzer | 职责分离合规审计 |
| qe-defect-predictor | AI 缺陷预测 |

### 9 条强制执行规则

| 规则 | 内容 |
|-----|------|
| E1 | 必须同时启动全部 3 个核心 agent，不能例外 |
| E2 | 所有并行任务必须在同一个消息中发起 |
| E3 | 每批任务完成后必须等全部结束才能下一步 |
| E4 | 条件满足时必须启动所有条件 agent，不能跳过 |
| E5 | 必须严格按阈值给出 SHIP/CONDITIONAL/HOLD 裁决 |
| E6 | 必须生成完整报告结构，不能缩写 |
| E7 | 每个 agent 必须先读参考文件才能开始分析 |
| E8 | 必须对所有代码变更运行缺陷预测，永远 |
| E9 | 必须执行学习持久化，不能跳过 |

**违反任何一条，整个 Swarm 直接终止。**

这哪是工具啊，这分明是一套军事化管理体系。

## 💡 我的真实使用建议

这两个工具我天天用，我的标准流程是：

### 日常开发：Harness 全程护航

- 每次提交 PR 前，跑一遍 `quality-scan.sh`
- 30 秒，把能自动修复的问题都修了
- 保证基础质量不滑坡

### Sprint 结束：QCSD 深度审计

- 上线前，跑一遍完整的 QCSD Swarm
- 30 分钟，做深度质量分析
- 拿到 SHIP 裁决才敢上线

**就像你在家自己量体温，觉得不对再去医院做 CT。**

## ❌ 这些场景千万别用 QCSD

我见过太多人滥用重型工具，最后反而影响效率：

1. **个人 side project** —— 等 QCSD 跑完，你都能重写三遍了
2. **内部工具/脚本** —— 出 bug 修一下就行，犯不上花几千块做审计
3. **快速迭代的 MVP** —— 产品方向都没确定，要啥质量门禁
4. **少于 5 人的小团队** —— 流程成本大于质量收益

**记住：工具是为了解决问题的，不是为了制造仪式感。**

---

## ✅ 这些场景必须用 QCSD

同样，有些场景省不了：

1. **涉及资金交易的核心系统** —— 出一个 bug 可能损失几百万
2. **监管严格的行业（金融/医疗）** —— 合规比什么都重要
3. **超过 20 人的开发团队** —— 人多了，必须有统一的质量门槛
4. **SaaS 产品生产环境** —— 宕机一小时就是几十万的损失

**这些场景，QCSD 跑出来的 HOLD 裁决，能帮你省七位数的损失。**

## 🎉 最后总结

AI 时代，我们不缺工具。

**缺的是知道什么时候用什么工具的判断力。**

- 想要"快"，用 Superpowers
- 想要"好"，用 Harness Dev Standards
- 想要"稳"，用 QCSD Development Swarm

**没有最好的工具，只有最适合场景的工具。**

不要拿着锤子看什么都是钉子。

也不要因为 CT 扫描仪厉害，感冒了就去做 CT。
