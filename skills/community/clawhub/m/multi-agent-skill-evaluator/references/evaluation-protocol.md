# 子 agent 评估协议

每次 spawn 子 agent 评估时，将以下内容完整作为 task 发送。

## 评估技能信息

（由主 agent 填充：目标 skill 名称、版本、简要描述及全部文件内容）

## 评估标准

### 8 个维度，每项 1-10 分

| # | 维度 | 高分标准（9-10） | 低分信号（1-3） |
|---|------|-----------------|-----------------|
| 1 | **功能完整性** | 宣称的功能全部实现，scope清晰，覆盖边缘场景 | 功能缺失、scope模糊、只覆盖happy path |
| 2 | **代码质量** | 逻辑清晰、错误处理完善、复用合理、注释得当 | 冗余代码、缺少关键检查、逻辑混乱 |
| 3 | **健壮性** | 有降级/回退策略、处理网络/API失败、输入校验 | 无降级、无失败处理、脚本遇到错误直接崩 |
| 4 | **安全性** | 无凭据硬编码、无危险操作、依赖声明完整 | 硬编码token、curl命令不安全、未声明依赖 |
| 5 | **文档质量** | 结构清晰、首次设置步骤完整、注意事项到位 | 缺少必要说明、用户无法独立完成设置 |
| 6 | **依赖合理性** | 依赖必要且最小化、有备选方案 | 依赖过多、可替代方案未提及、无检查 |
| 7 | **预估运行效果** | 按文档配置后大概率正常运转 | 有明显坑、配置复杂、容易出问题 |
| 8 | **总评** | 综合判断 | 综合判断 |

### 打分要求

1. 每个维度必须附具体理由，**引用文件中具体代码或章节**
2. 必须分别列出加分项和扣分项
3. 不允许只写分数不给理由
4. 分数必须是 **整数**（1-10），不允许小数
5. 如果发现重大缺陷，明确指出

## 输出格式

按以下两部分输出：

### 第一部分：详细评语（自然语言）

#### 维度1 - 功能完整性：7/10
**加分：** ...
**扣分：** ...

#### 维度2 - 代码质量：7/10
**加分：** ...
**扣分：** ...

（重复上述格式，覆盖全部 8 个维度）

### 第二部分：分数摘要（固定格式）

```json
===SCORE_SUMMARY===
{"功能完整性":7,"代码质量":7,"健壮性":7,"安全性":7,"文档质量":7,"依赖合理性":7,"预估运行效果":7,"总评":7}
===END_SCORE_SUMMARY===
```

**规则：**
- 用 JSON 对象，一行写完
- key 用中文维度名（与上表完全一致）
- value 只写数字，不写 /10
- 必须包含 `===SCORE_SUMMARY===` 和 `===END_SCORE_SUMMARY===` 标记行
