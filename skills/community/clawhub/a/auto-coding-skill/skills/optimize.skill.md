# Optimize：优化纪律

## 1. Overview（概述）

优化不是"顺手改一下"。必须先证明行为不变，再证明改进真实存在。无基准的优化 = 赌博。

**核心原则**：先测后改、行为不变为第一前提、每处修改有据可查、不加需求。

## 2. When to Use（使用条件）

**触发条件**：
- Step 6 优化阶段自动注入
- C 级任务强制执行
- code-review 阶段提出了 🟡 警告需要处理

**可跳过**：
- A 级任务（micro fix 不需要单独优化阶段）
- 审查无 🟡/💭 项且代码行数/性能无明显问题

## 3. Process（执行流程）

### 3.1 优化三步纪律

```
步骤 1: 建立基线（Baseline）
  ├─ 运行全量测试 → 记录通过/失败状态 + 耗时
  ├─ 量化当前状态：
  │   - 代码行数（按模块）
  │   - 测试耗时
  │   - 关键路径性能（如有 benchmark）
  ├─ 输出：基线报告
  └─ ❌ 禁止：没测基线就上手改

步骤 2: 逐项优化（Incremental + Traceable）
  ├─ 每个优化必须有来源：
  │   ✅ "Review 🟡 #3: 合并 validate_email() 和 check_email_format() 重复逻辑"
  │   ❌ "这个函数可以用列表推导式"
  ├─ 每个优化缩小范围：
  │   ✅ 只改一个文件的一个函数
  │   ❌ 全局搜索替换（碰了其他不相关的代码）
  ├─ 每改一个优化 → 立即跑测试 → 确认 GREEN
  └─ ❌ 禁止：一次性做多个优化（不知道哪个引入了 bug）

步骤 3: 对比验证（Before/After）
  ├─ 对比优化前后：
  │   - 代码行数：减少 / 持平 + 可读性提升？
  │   - 测试耗时：未退化？
  │   - 功能行为：全部测试通过？
  ├─ 输出：before/after 对比报告
  └─ ❌ 禁止：只说"优化了"但不给数据
```

### 3.2 优化范围约束

```
✅ 可以做：
  - 消除重复（同一概念 ≥ 2 处）
  - 简化条件逻辑（complexity > 5）
  - 改善命名（从 Review 意见）
  - 删除死代码（确认无调用者）

❌ 不可以做：
  - 改变公共 API（破坏契约）
  - 引入新的抽象层（"以后会用到"）
  - 更换算法（除非性能数据证明需要）
  - 重新组织文件结构（"这样更好" — 没有功能性理由）
  - 添加新的功能（"顺便加个默认值"）
```

### 3.3 反例

- ❌ 优化=重写：100 行用自己的风格重写成 90 行 → 引入 3 个 bug
- ❌ 优化=加功能："顺便加了环境变量支持" → 这是功能变更，不是优化
- ❌ 优化=换风格：for 循环改 map/reduce → 可读性没提升，性能没变化

## 4. Risk Scorecard（反借口表）

| # | 纪律 | 常见借口 | 反驳 |
|---|------|---------|------|
| O1 | 先测基线再优化 | "变化很小，不用跑测试" | 小的改变 = 小的引入 bug 机会 ≠ 没有 |
| O2 | 逐项优化 | "这几个问题都在这文件里，一起改效率高" | 一起改 = 不知道是哪个改动引入的问题 |
| O3 | 优化有据可查 | "这个写法性能更好" | 没有数据的"更好" = 主观偏好 |
| O4 | 不加功能 | "顺便加个边界检查更安全" | 边界检查是功能变更，要单独走需求 |
| O5 | 对比验证 | "代码少了几行，肯定更好" | 行数少 ≠ 行为正确 ≠ 性能提升 |

## 5. Red Flags（危险信号）

- 🚩 优化后测试失败 — 破坏了原有行为（最高优先级修复）
- 🚩 修改文件数 > 编码阶段修改文件数 — 可能在重构而非优化
- 🚩 新增代码行数 > 删除代码行数 — 可能是加功能或在过度抽象
- 🚩 没有任何代码变化但耗时增加 — 可能引入了性能退化
- 🚩 优化涉及公共 API 签名变更 — 这是重构需求不是优化
- 🚩 优化理由含"更优雅"/"更现代"等主观词汇 — 可能是风格偏好

## 6. Verification（自检清单）

**必检（≤5 项）**：
- [ ] 测试全部通过（优化后，与基线一致）
- [ ] 功能行为无变化（公共 API 输出与优化前一致）
- [ ] 未被要求的功能没有被添加（diff 只含优化改动）
- [ ] 每个优化有关联的审查意见或量化证据

**抽检（≤3 项）**：
- [ ] 代码行数减少（或持平 + 可读性提升有依据）
- [ ] 性能关键路径无退化（测试耗时 ≤ 1.1x 基线）
- [ ] 重构理由可追溯（每个改动关联到具体的 Review 意见编号）

**免检**：性能优化的具体数值目标、代码风格偏好
