## B.1 通用基础类（全软件通用坑）

### B.1.1 路径/文件名含中文、空格、特殊字符

| 维度 | 说明 |
|------|------|
| **现象** | 程序闪退、文件读取失败、转换中断、输出文件为空、工具报错无日志、批量脚本终止 |
| **原因** | GIS/CAD底层内核多为海外开发，字符解析引擎对中文、全角符号、空格、特殊字符兼容性差 |
| **解决方案** | 数据源、工程、模板、输出目录统一使用纯英文+数字命名；路径层级控制在5级以内；禁止使用 @ # & （） ，。 、 等特殊符号 |
| **预防措施** | 统一项目目录规范：盘符/项目编号/数据分类/文件，全员强制执行 |

### B.1.2 字符编码不统一

| 维度 | 说明 |
|------|------|
| **现象** | 中文乱码、属性显示问号、备注丢失、字段映射错位、文本注记残缺 |
| **原因** | GIS系列软件默认UTF-8编码，CAD/CASS、老测绘软件默认GBK(936)，跨软件流转编码不匹配 |
| **解决方案** | GDB→DWG：输出编码强制选择GBK(936)；DWG→GDB：导入编码选择UTF-8；文本、注记、属性全部使用半角字符 |
| **预防措施** | 跨格式转换前，提前确认两端编码规则，固化转换模板编码参数 |

### B.1.3 内存/资源不足、后台程序抢占资源

| 维度 | 说明 |
|------|------|
| **现象** | 软件卡顿、运算中途卡死、大文件转换崩溃、要素缺失、计算结果不完整、导出中断 |
| **原因** | GIS、三维建模、遥感影像处理属于高内存、高IO负载业务，浏览器、网盘、杀毒、视频软件抢占硬件资源 |
| **解决方案** | 作业前关闭浏览器、云盘同步、实时杀毒、视频类软件、虚拟机；大数据量按图层、分类分批次处理；常规作业内存≥16G，海量数据/三维建模≥32G |
| **预防措施** | 制定作业环境标准，超大运算任务使用专用作业设备，错峰运行批量任务 |

### B.1.4 复用旧工程/旧任务，残留配置干扰

| 维度 | 说明 |
|------|------|
| **现象** | 修改参数后运行结果无变化、历史错误反复复现、配置莫名重置、模板自动回退 |
| **原因** | 软件自动缓存上一轮工程参数、映射规则、样式模板、临时文件、校验规则 |
| **解决方案** | 重大参数调整后新建工程/新建任务，禁止直接点击"重新运行"；定期清理缓存、临时文件、过期历史任务 |
| **预防措施** | 一项目对应一个独立工程，失败任务直接删除，不重复复用旧任务 |

### B.1.5 文件只读/权限不足

| 维度 | 说明 |
|------|------|
| **现象** | 无法保存文件、转换后无法写入、覆盖失败、提示"拒绝访问"、临时文件创建失败 |
| **原因** | 文件被其他软件占用、系统权限受限、文件属性设为只读、杀毒软件拦截写入 |
| **解决方案** | 关闭所有占用该文件的软件；右键文件→属性，取消「只读」勾选；以管理员身份启动作业软件 |
| **预防措施** | 成果文件分区单独管理，作业目录开放读写权限，避免文件被多人同时编辑 |

### B.1.6 磁盘空间不足

| 维度 | 说明 |
|------|------|
| **现象** | 转换到一半报错终止、输出文件体积异常偏小、临时文件写入失败、三维建模中断 |
| **原因** | GIS、三维建模会生成大量临时缓存文件，占用数GB~数十GB磁盘空间 |
| **解决方案** | 清理磁盘垃圾、回收站、过期临时文件；更换剩余空间充足的分区作为输出目录 |
| **预防措施** | 大数据作业前检查磁盘剩余空间，保证目标分区剩余空间≥源文件5倍 |

### B.1.7 版本混用（软件高低版本交叉打开）

| 维度 | 说明 |
|------|------|
| **现象** | 高版本文件低版本软件打不开、功能丢失、参数重置、数据结构损坏 |
| **原因** | 软件迭代后数据格式、功能接口发生变更，向下兼容性存在缺陷 |
| **解决方案** | 项目组统一软件版本，全员使用同一版本作业；对外交付前，另存为低版本兼容格式 |
| **预防措施** | 建立软件版本管控清单，禁止私自升级/降级作业软件 |

## B.2 ArcGIS Pro & GDB 数据层坑点

### B.2.1 几何错误（空几何、多部件、悬挂节点、碎图元）

| 维度 | 说明 |
|------|------|
| **现象** | 分析工具报错、转换丢失要素、拓扑检查不通过、空间统计结果异常、裁剪/相交工具运行失败 |
| **原因** | 外业采集、数据编辑、图幅拼接、拓扑修复产生无效畸形几何，工具无法正常识别 |
| **解决方案** | 使用「检查几何」工具批量排查问题；执行「修复几何」完成批量修复；删除空几何、零长度线、零面积面；多部件要素拆分为单部件，清理碎面、短悬挂线 |
| **预防措施** | 数据入库、对外流转前，强制做几何质检与修复 |

### B.2.2 坐标系/投影不一致、未知坐标系

| 维度 | 说明 |
|------|------|
| **现象** | 图形整体偏移、飞图、图层叠加错位、距离/面积计算错误、图幅接边失败、坐标转换偏差 |
| **原因** | 数据源混用WGS84/CGCS2000/北京54/西安80等坐标系，未统一投影、分带 |
| **解决方案** | 测绘类工程统一坐标系：CGCS2000 3度带高斯投影；未知坐标系数据先定义投影、配准校正；跨坐标系数据使用「投影」工具批量转换 |
| **预防措施** | 项目立项阶段确定统一坐标系与高程基准，全程不随意变更 |

### B.2.3 地理坐标系与投影坐标系混淆

| 维度 | 说明 |
|------|------|
| **现象** | 坐标数值异常（经纬度和平面坐标混用）、比例尺错乱、空间分析结果错误 |
| **原因** | 误将地理坐标系（经纬度）当作投影坐标系（平面米制坐标）使用 |
| **解决方案** | 平面测绘作业必须使用投影坐标系；经纬度数据先投影转为平面坐标再做矢量化、成图 |
| **预防措施** | 区分两类坐标系用途，底图、矢量、地形数据统一为平面投影坐标 |

### B.2.4 字段名中文、超长、含特殊符号

| 维度 | 说明 |
|------|------|
| **现象** | 转出CAD后字段丢失、字段名乱码、属性映射失败、第三方工具读取异常、批量赋值报错 |
| **原因** | CAD及老旧GIS工具限制：字段名长度≤8字符，原生不支持中文字段名 |
| **解决方案** | 字段名统一改为英文/英文缩写，长度控制在8字符以内；清除字段名内空格、特殊符号；中文释义填写在「字段别名」中，不改动原始字段名 |
| **预防措施** | 建立标准化字段命名规范，禁止新建中文名称字段 |

### B.2.5 标注(Label)与注记(Annotation)混淆

| 维度 | 说明 |
|------|------|
| **现象** | 转出CAD后注记漂移、丢失、缩放错位、批量叠压、样式无法固化 |
| **原因** | Label为动态渲染标签，无实体几何，依附图层动态计算位置；Annotation是实体注记要素，支持跨软件流转 |
| **解决方案** | 对外转换、出图、跨格式流转，必须将Label转为Annotation；统一注记对齐方式、字高、字体参数 |
| **预防措施** | 正式成果全部使用Annotation，Label仅用于临时查看、内部预览 |

### B.2.6 M值/Z值冗余、高程值异常

| 维度 | 说明 |
|------|------|
| **现象** | 转CAD后高程错乱、等高线变形、三维显示异常、CASS无法识别高程、断面计算错误 |
| **原因** | 要素携带多余M测量值，Z值为空、存在极值或高程跳变 |
| **解决方案** | 清理无用M值；无高程要素统一将Z值赋值为0；检查等高线高程连续性，删除高程跳变数据 |
| **预防措施** | 地形类数据严格管控Z值，禁止随意赋值、批量错误填充 |

### B.2.7 数据集锁定、无法编辑/删除

| 维度 | 说明 |
|------|------|
| **现象** | GDB要素类无法编辑、删除、重命名，提示"数据集正在被占用" |
| **原因** | ArcGIS Pro图层未关闭、其他进程/软件挂载该GDB、数据集开启版本管理 |
| **解决方案** | 移除地图中对应图层，关闭所有关联软件；关闭数据集版本管理、离线编辑 |
| **预防措施** | 编辑完成后及时移除图层，多人协作使用文件锁管控GDB |

### B.2.8 字段类型不匹配（文本/数值混用）

| 维度 | 说明 |
|------|------|
| **现象** | 属性计算报错、排序异常、转出CAD后数值变文本、编码位数错乱 |
| **原因** | 地物编码、高程、点号等字段类型定义错误，文本与数值混用 |
| **解决方案** | 编码、点号使用文本型（防止前置0丢失）；高程、面积、长度使用浮点型/双精度 |
| **预防措施** | 建库时严格定义字段类型，统一字段数据格式 |

### B.2.9 数据版本、副本混杂

| 维度 | 说明 |
|------|------|
| **现象** | 多人编辑后数据冲突、成果版本混乱、旧数据覆盖新数据 |
| **原因** | 未做版本管理，多人直接编辑原始GDB，无中间备份 |
| **解决方案** | 划分原始库、过程库、成果库三级目录；编辑使用副本，成果定期备份归档 |
| **预防措施** | 执行数据版本管理规范，禁止直接修改原始数据源 |

## B.3 GDB ↔ DWG/CASS 双向转换坑点（核心业务）

### B.3.1 GDB→DWG：未映射CASS标准XDATA

| 维度 | 说明 |
|------|------|
| **现象** | CASS打开图形无地物编码、无法自动符号化、属性空白、地籍信息丢失、入库失败 |
| **原因** | CASS依靠扩展数据XDATA识别测绘语义，默认转换规则不生成SouthMap应用名及标准字段 |
| **解决方案** | 转换工具开启写入XDATA，应用名固定为SouthMap；必映射核心字段：CASS_CODE(4/6位字符串)、ELEV(浮点型，无高程填0)、POINT_NO(字符串，唯一无空格) |
| **预防措施** | 固化转换模板，强制校验三大核心XDATA字段 |
| **神经连接** | → `30_GIS↔CAD数据转换.md` §五 CASS XDATA 深度解析（SOUTH/骨架线/XRecord完整架构） |

### B.3.2 GDB点要素转为普通POINT，非CASS属性块

| 维度 | 说明 |
|------|------|
| **现象** | 点符号、点号、高程注记相互分离，移动点位后注记滞留原地，图层杂乱、无法批量编辑 |
| **原因** | CASS标准测绘地物为属性块，普通CAD点无法绑定附属注记与属性 |
| **解决方案** | 点要素转换类型选择：点→CASS属性块；加载官方模板CASS安装目录/system/cass.dwg；块名绑定CASS_CODE，块属性分别绑定点号、高程 |
| **预防措施** | 所有测绘测点、地物点统一转换为CASS属性块 |

### B.3.3 注记锚点不统一，文字漂移错位

| 维度 | 说明 |
|------|------|
| **现象** | 注记偏离点位、整体偏移、批量叠压、放大缩小后位置错乱 |
| **原因** | ArcGIS注记默认左下对齐，CAD默认中心对齐，锚点规则冲突 |
| **解决方案** | GDB新增对齐字段：H_ALIGN=1、V_ALIGN=1（中心对齐）；转换工具内注记对齐方式强制设置为中心/中心 |
| **预防措施** | 对外输出的所有注记，统一使用中心对齐规则 |

### B.3.4 DWG版本选择不当

| 维度 | 说明 |
|------|------|
| **现象** | CASS打不开文件、图形残缺、高版本DWG在低版本CAD中报错、部分实体丢失 |
| **原因** | DWG版本向下兼容有限，主流CASS基于AutoCAD2014-2018内核开发 |
| **解决方案** | 固定输出DWG版本：DWG R2018，兼顾兼容性与功能完整性 |
| **预防措施** | 所有成果DWG统一输出为R2018版本 |

### B.3.5 DWG→GDB：块、属性、XDATA丢失

| 维度 | 说明 |
|------|------|
| **现象** | 导入GDB后仅保留图形，CASS编码、高程、点号全部丢失，属性表为空 |
| **原因** | 默认导入规则忽略CAD块属性与XDATA扩展数据 |
| **解决方案** | 开启「导入块属性」「读取XDATA扩展数据」选项；将SouthMap下的扩展字段批量映射为GDB标准字段 |
| **预防措施** | DWG入库GDB前，先使用LIST命令校验XDATA数据完整性 |

### B.3.6 圆弧、样条曲线转后变形

| 维度 | 说明 |
|------|------|
| **现象** | 圆弧变为折线、样条曲线失真、地物轮廓变形、周长/面积计算偏差 |
| **原因** | GIS内核对CAD复杂曲线实体支持度低，无法完整解析曲线参数 |
| **解决方案** | 转换前将圆弧、样条曲线批量转折线，设置合理简化容差（地形数据0.1m） |
| **预防措施** | 矢量流转前统一曲线转折线，固化简化容差参数 |

### B.3.7 XDATA字段顺序错乱

| 维度 | 说明 |
|------|------|
| **现象** | CASS读取属性串位，编码显示为高程、高程显示为点号，所有语义失效 |
| **原因** | CASS解析XDATA严格按照存储顺序读取，顺序颠倒直接导致取值错误 |
| **解决方案** | 强制固定字段顺序：CASS_CODE → ELEV → POINT_NO → 扩展字段 |
| **预防措施** | 转换模板锁定XDATA字段顺序，禁止随意调整 |

### B.3.8 图层名超长/含中文/特殊字符

| 维度 | 说明 |
|------|------|
| **现象** | 转换后图层丢失、图层名乱码、CASS图层分类失效、批量图层管理混乱 |
| **原因** | CAD图层名最大长度31字符，不支持中文及特殊符号 |
| **解决方案** | 图层名使用英文/拼音缩写，长度≤31字符；提前按照CASS图层规范分层（GCD、YSD、DLSS、ZJ等） |
| **预防措施** | GIS端提前按CASS标准分层命名，再执行转换 |

### B.3.9 多部件面要素转换后碎裂

| 维度 | 说明 |
|------|------|
| **现象** | 一个宗地/地块拆分为多个零散面，拓扑关系破坏、面积统计重复 |
| **原因** | CAD无原生多部件面概念，多部件要素会自动拆分为独立单部件 |
| **解决方案** | GIS端提前将多部件面拆分、合并处理，保证一要素一图形 |
| **预防措施** | 转换前批量拆解多部件要素，重构拓扑 |

### B.3.10 高程Z值全局偏移

| 维度 | 说明 |
|------|------|
| **现象** | 转换后整体地形高程统一偏高/偏低，等高线整体错位 |
| **原因** | 两端高程基准不一致、转换时Z值缩放/偏移参数未归零 |
| **解决方案** | 统一高程基准（1985国家高程）；转换工具关闭Z值偏移、缩放功能 |
| **预防措施** | 转换前核对高程基准，锁定Z值参数 |

### B.3.11 注记比例因子异常

| 维度 | 说明 |
|------|------|
| **现象** | 注记字体超大/超小，同图内字高大小不一，不符合出图规范 |
| **原因** | GIS地图单位与CAD绘图单位不统一，注释比例未锁定 |
| **解决方案** | 统一注释比例为1:1；固定字高参数，关闭自动缩放 |
| **预防措施** | 转换模板固化字高、注释比例参数 |

## B.4 CASS 地形/地籍/成图专项坑点

### B.4.1 XDATA应用名非SouthMap/CASS

| 维度 | 说明 |
|------|------|
| **现象** | CASS完全识别不到扩展属性，DD命令无法自动符号化 |
| **原因** | CASS仅识别应用名为SouthMap或CASS的扩展数据 |
| **解决方案** | 重新转换，强制XDATA应用名设置为SouthMap |
| **预防措施** | 转换工具强制锁定应用名，禁止修改 |

### B.4.2 标准SHX字体缺失

| 维度 | 说明 |
|------|------|
| **现象** | 中文注记全部显示为问号、方框，样式加载失败 |
| **原因** | 缺少CASS标配字体gbenor.shx、gbcbig.shx，使用TrueType字体无法兼容 |
| **解决方案** | 拷贝标准SHX字体到CASS字体目录；文字样式绑定gbenor.shx + gbcbig.shx |
| **预防措施** | 作业电脑统一配齐标准字体，样式模板固化字体配置 |

### B.4.3 等高线类型标识缺失

| 维度 | 说明 |
|------|------|
| **现象** | 计曲线、首曲线无法区分，高程注记错乱、剖面绘制错误 |

## B.5 空间统计 & 空间分析

### B.5.1 空间权重矩阵设置不合理

| 维度 | 说明 |
|------|------|
| **现象** | Moran's I指数失真、聚类结果偏差、热点分析Z/P值无显著性 |
| **原因** | 邻域距离、权重规则（距离/邻接）与研究区尺度不匹配 |
| **解决方案** | 根据研究地物尺度调试邻域阈值；结果必须结合Z得分、P值判断显著性，不单独依靠指数数值 |
| **预防措施** | 同数据集多组权重参数对比，选取最优结果 |
| **神经连接** | → `33_空间分析与统计.md` 空间自相关/权重矩阵构建 |

### B.5.2 GWR带宽选择错误

| 维度 | 说明 |
|------|------|
| **现象** | 回归系数异常、模型拟合度低、空间非平稳性分析失效 |
| **原因** | 人工指定带宽过大/过小，未使用科学准则优选 |
| **解决方案** | 优先使用AICc准则自动优选带宽，保留最优模型 |
| **预防措施** | 禁止人工随意指定带宽，以算法自动计算结果为准 |

### B.5.3 空间插值方法选型错误

| 维度 | 说明 |
|------|------|
| **现象** | 插值表面失真、极值异常、成果不符合地物真实分布规律 |
| **原因** | 不了解各插值算法适用场景，盲目选用工具 |
| **解决方案** | IDW：操作简单但易受极值干扰，不适用于突变地形；Kriging：地统计算法需拟合变异函数，适合气象/土壤/地质；Spline：平滑效果强适合连续渐变表面 |
| **预防措施** | 按数据类型选型，同场景做多算法交叉验证 |

### B.5.4 插值采样点分布不均

| 维度 | 说明 |
|------|------|
| **现象** | 局部插值结果畸变、大片区域无参考点导致表面失真 |
| **原因** | 采样点集中在局部区域，测区边缘、空白区缺少样本 |
| **解决方案** | 补充均匀分布的采样点，剔除异常极值点 |
| **预防措施** | 采样阶段保证点位均匀覆盖全域 |

### B.5.5 全局回归混淆空间非平稳性

| 维度 | 说明 |
|------|------|
| **现象** | 传统全局回归无法体现区域差异，分析结论脱离地理实际 |
| **原因** | 忽略地理现象"空间非平稳"特性，误用全局模型 |
| **解决方案** | 存在区域差异时，优先使用GWR地理加权回归 |
| **预防措施** | 分析前判断数据是否存在空间分异，选择对应模型 |

### B.5.6 聚类分析无空间约束

| 维度 | 说明 |
|------|------|
| **现象** | 纯属性聚类，空间分布逻辑混乱，聚类单元地理上相互隔离 |
| **原因** | 仅依据属性聚类，未结合空间位置关系 |
| **解决方案** | 使用空间约束聚类，结合位置+属性双重条件 |
| **预防措施** | 地理类聚类必须加入空间约束规则 |

### B.5.7 异常值未剔除，统计结果失真

| 维度 | 说明 |
|------|------|
| **现象** | 指数、回归、聚类结果被少数极值带偏，整体结论错误 |
| **原因** | 数据存在错误极值、野值，未做预处理 |
| **解决方案** | 使用箱线图、标准差法识别并剔除异常值 |
| **预防措施** | 空间分析前统一做异常值检测 |

## B.6 遥感影像处理与解译

### B.6.1 跳过预处理直接分类/计算指数

| 维度 | 说明 |
|------|------|
| **现象** | NDVI/NDWI数值异常、分类精度极低、地物大面积错分 |
| **原因** | 大气散射、辐射畸变、几何偏差干扰像素灰度值 |
| **解决方案** | 严格遵循标准流程：辐射定标→大气校正→几何校正→裁剪镶嵌→指数计算/分类 |
| **预防措施** | 固化影像预处理流水线，禁止省略关键步骤 |
| **神经连接** | → `24_遥感与GEE.md` 遥感预处理六步法 |

### B.6.2 波段组合使用错误

| 维度 | 说明 |
|------|------|
| **现象** | 目视解译困难，植被、水体、建筑、裸地无法有效区分 |
| **原因** | 未根据解译目标选择对应波段组合 |
| **解决方案** | 标准假彩色：植被高亮显示通用解译；真彩色：贴近人眼视觉成果出图；水体专用组合：增强水体边界 |
| **预防措施** | 按业务场景固定波段组合方案 |

### B.6.3 分类样本质量差

| 维度 | 说明 |
|------|------|
| **现象** | 分类结果杂乱，总体精度OA、Kappa系数不达标，错分漏分严重 |
| **原因** | 样本分布不均、同类样本离散度大、样本混入混合像素 |
| **解决方案** | 样本均匀覆盖全测区，远离地物边界；分类后用混淆矩阵校验精度，不合格重新采集样本 |
| **预防措施** | 样本采集执行规范，预留独立检验样本 |

### B.6.4 变化检测基准不一致

| 维度 | 说明 |
|------|------|
| **现象** | 伪变化过多，大量无变化区域被误判为变化，成果失效 |
| **原因** | 两期影像坐标系、分辨率、预处理标准、时相差异过大 |
| **解决方案** | 两期影像统一坐标系、分辨率、预处理参数；优先使用分类后比较法，降低伪变化概率 |
| **预防措施** | 变化检测选用同时相、同传感器影像，统一预处理流程 |

### B.6.5 影像镶嵌接边痕迹明显

| 维度 | 说明 |
|------|------|
| **现象** | 镶嵌后图面存在色差、条带，视觉效果差，影响解译 |
| **原因** | 相邻影像辐射值不一致、镶嵌线穿越地物主体 |
| **解决方案** | 执行辐射均衡、色彩匹配；镶嵌线避开房屋、道路、水体等地物 |
| **预防措施** | 镶嵌前统一辐射参数，人工优化镶嵌线路径 |

### B.6.6 裁剪范围溢出、黑边过多

| 维度 | 说明 |
|------|------|
| **现象** | 影像四周大面积黑边，占用存储空间、干扰分析 |
| **原因** | 裁剪矢量范围大于影像实际范围 |
| **解决方案** | 按影像外轮廓精确裁剪，批量去除黑边像素 |
| **预防措施** | 裁剪范围贴合影像边界 |

### B.6.7 多光谱/无人机影像指数公式误用

| 维度 | 说明 |
|------|------|
| **现象** | NDVI、NDWI计算结果为负数/极值，语义完全错误 |
| **原因** | 波段序号对应错误、公式套用混淆 |
| **解决方案** | 核对传感器波段编号，匹配对应指数计算公式 |
| **预防措施** | 按传感器类型整理指数公式对照表 |

### B.10.9 子进程桥接在 ArcGIS Pro 环境中的风险

| 维度 | 说明 |
|------|------|
| **现象** | C# 启动 Python 子进程（`Process.Start("python", "bridge.py")`），stdout 管道永不返回，进程挂起。ArcGIS Pro 沙箱环境中的子进程桥接失败率极高 |
| **原因** | ① ArcGIS Pro 进程环境的权限/路径/环境变量与独立进程隔离；② stdout 管道可能死锁（缓冲区满未读取）；③ Pro 内置 Python（`arcgispro-py3`）环境与系统 Python 冲突 |
| **解决方案** | **彻底放弃子进程桥接方案**。改用 C# `HttpClient` 直连 Ollama/WorkBuddy API，或使用文件桥接 + HTTP 回调架构 |
| **预防措施** | 在 ArcGIS Pro AddIn 开发中，优先选择线程内 HTTP 通信或文件 IPC，避免任何子进程依赖 |

### B.10.10 fire-and-forget 异步调试黑洞

| 维度 | 说明 |
|------|------|
| **现象** | 异步方法调用后无任何反馈——无崩溃、无日志、无异常弹出，行为如"什么都没发生" |
| **原因** | `_ = SomeAsync()` 模式将异步任务 fire-and-forget，Task 中的所有异常被静默吞掉，`try-catch` 也无济于事 |
| **解决方案** | ① 使用 `Task.ContinueWith(t => { if (t.IsFaulted) Log(t.Exception); })` 捕获异常；② 改用 `await` 并在调用链中正确使用 `try-catch`；③ 在 ViewModel 中持有 Task 引用，通过 `IsFaulted` 属性检测状态 |
| **预防措施** | 禁止在 ArcGIS Pro AddIn 中使用 fire-and-forget 模式；所有异步操作必须有异常处理路径 |

### B.10.11 ArcGIS Pro WPF 环境与标准 WPF 的差异

| 维度 | 说明 |
|------|------|
| **现象** | `RelayCommand` 的 `CanExecute` 在标准 WPF 测试应用中正常刷新，但在 Pro 中不触发。`IsBusy` 属性变化后 UI 未更新按钮状态 |
| **原因** | ArcGIS Pro 的 WPF 宿主环境在命令绑定、`CommandManager.RequerySuggested`、Dispatcher 线程上下文方面与标准 WPF 有差异 |
| **解决方案** | ① 在 `IsBusy` setter 中手动调用 `CommandManager.InvalidateRequerySuggested()`；② 考虑使用 ArcGIS Pro SDK 自带的 `DelegateCommand` 替代自定义 `RelayCommand`；③ 用同步 `HttpClient` 调用排除异步上下文问题 |
| **预防措施** | 所有 WPF 绑定和命令逻辑必须在 Pro 实际环境中验证，不能仅依赖独立 WPF 测试 |

### B.10.12 ArcGIS Pro AddIn 编译部署完整 SOP

| 维度 | 说明 |
|------|------|
| **现象** | MSB4175 错误：`The task factory "CodeTaskFactory" could not be loaded...`，环境块超过 64KB 限制（实际 ~283KB） |

| **预防措施** | 使用 `build.bat` 脚本封装上述逻辑，避免手动操作遗漏 |

## B.16 ArcGIS Pro 深度学习全流程（失败案例复盘）

> **项目背景**：workbuddy测试工程，UAV DOM (0.05m, EPSG:4546) + 45个建筑物样本面，目标用 MaskRCNN 提取建筑轮廓。经历两轮完整尝试后失败。
> **结论**：ArcGIS Pro 深度学习工具链 **无法通过 CLI/外部脚本可靠运行**，必须 100% 在 ArcGIS Pro 内部 Python 窗口执行。

### B.16.1 训练数据导出格式错误（PASCAL VOC ≠ RCNN_Masks）

| 维度 | 说明 |
|------|------|
| **现象** | 标注工具导出的训练数据为 PASCAL VOC 格式（`<bndbox>` 边界框），直接喂给 MaskRCNN 训练失败 |
| **原因** | ArcGIS Pro 标注对象工具默认导出 PASCAL VOC（目标检测格式），但 MaskRCNN 需要 **RCNN_Masks** 格式（像素级掩码）。两者本质不同：PASCAL VOC 是矩形框，RCNN_Masks 是多边形掩码 |
| **解决方案** | 使用 `Export Training Data For Deep Learning` 工具时，`metadata_format` 参数必须设为 `RCNN_Masks`，而非 `PASCAL_VOC_rectangles` |
| **预防措施** | 导出前确认模型类型：实例分割（MaskRCNN）→ RCNN_Masks；目标检测（FasterRCNN/YOLO）→ PASCAL_VOC；语义分割（UNet）→ Classified_Tiles |

### B.16.2 arcpy CLI 无法运行 ExportTrainingDataForDeepLearning

| 维度 | 说明 |
|------|------|
| **现象** | 命令行执行 `python export_rcnn_masks.py` 报 `ERROR 999999: Something unexpected caused the tool to fail`，多次重试均失败 |
| **原因** | 该工具是 ArcGIS Pro 核心 geoprocessing 工具，依赖 Pro 内部运行时环境（code page 936.txt GBK 编码表）。CLI 模式下缺少 `C:\Program Files\ArcGIS\Pro\locale\codepage\936.txt`，导致内部字符编码异常 |
| **解决方案** | 在 ArcGIS Pro 内部打开 **Analysis → Python 窗口**，粘贴脚本运行。不可用外部 Python 或 PowerShell 触发 |
| **预防措施** | 所有 ArcGIS Pro geoprocessing 深度学习工具（`ExportTrainingDataForDeepLearning`、`TrainDeepLearningModel`、`DetectObjectsUsingDeepLearning`）**强制在 Pro 内部 Python 窗口执行** |

### B.16.3 WorkBuddy 沙箱与 arcpy 的冲突

| 维度 | 说明 |
|------|------|
| **现象** | 沙箱阻止访问 `D:\YourGISData\`（写入被拒）、`C:\Program Files\ArcGIS\Pro\`（读取被拒），所有 CLI 脚本均无法完成 geoprocessing 操作 |
| **原因** | WorkBuddy 沙箱限制了非工作区路径的读写权限，而 arcpy 工具需要访问 ArcGIS Pro 安装目录下的运行时文件 |
| **解决方案** | 深度学习全流程不通过 WorkBuddy CLI 执行，仅在 ArcGIS Pro 内部 Python 窗口操作。WorkBuddy 仅用于：脚本编写、参数设计、结果分析 |
| **预防措施** | 涉及 arcpy geoprocessing 工具的任务，识别为"不可 CLI 执行"类别，直接生成 Pro 内部脚本交付用户 |

### B.16.4 C盘空间被 conda 缓存耗尽

| 维度 | 说明 |
|------|------|
| **现象** | 训练/推理过程中 C 盘被写满，系统报警，任务中断 |
| **原因** | (1) conda clone 时生成 `arcgispro-py3-clone.tar.bz2` (~686MB)；(2) conda package cache (`pkgs/`) 累积 3-5GB；(3) 训练过程 scratch 数据写入临时目录 |
| **解决方案** | 清理：`conda clean --all`、删除 `clone.tar.bz2`、确保训练数据输出到 D 盘 |
| **预防措施** | 深度学习任务前检查 C 盘剩余空间 ≥ 20GB；设置 `arcpy.env.scratchWorkspace` 指向 D 盘 |

### B.16.5 样本量不足导致训练不可行

| 维度 | 说明 |
|------|------|
| **现象** | 45 个建筑物样本面，导出 RCNN_Masks 后有效训练芯片极少，无法支撑 MaskRCNN 训练 |
| **原因** | MaskRCNN 是数据密集型模型。45 个面要素，按 256×256 tile + stride 64 导出，有效芯片数可能不足 50 个（取决于建筑物分布密度）。而 MaskRCNN 通常需要 ≥ 500 个带标注的训练芯片才能收敛 |
| **解决方案** | (1) 扩大标注样本至 ≥ 200 个面；(2) 减小 stride（如 32）增加芯片重叠率；(3) 考虑使用预训练模型做迁移学习；(4) 评估是否可用更简单的语义分割方案（UNet）替代 |
| **预防措施** | 项目开始前评估样本量-模型匹配度：实例分割 ≥ 200 样本面、目标检测 ≥ 100 样本面、语义分割 ≥ 50 样本面（粗略阈值，视地物复杂度调整） |

### B.16.6 GPU 显存不足

| 维度 | 说明 |
|------|------|
| **现象** | batch_size=8 时 CUDA OOM，训练中断 |
| **原因** | RTX 2000 Ada 8GB VRAM，256×256×3 芯片 + MaskRCNN backbone 内存占用大 |
| **解决方案** | batch_size 降至 2-4；使用 `chip_size=224` 替代 256 |
| **预防措施** | 训练前估算显存：MaskRCNN + ResNet50 ≈ 2-3GB 基础 + batch × chip 内存。8GB GPU 建议 batch_size ≤ 4 |

### B.16.7 训练输出路径必须为纯英文

| 维度 | 说明 |
|------|------|
| **现象** | 输出路径含中文（如"标注对象_RCNN_Masks"）时，arcpy 工具内部路径解析失败 |
| **原因** | ArcGIS Pro geoprocessing 工具在路径处理时对非 ASCII 字符的容错性差 |
| **解决方案** | 所有训练数据输出、模型保存、推理输出路径使用纯英文目录 |

### B.16.8 完整失败流程复盘

| 阶段 | 发生了什么 | 根因 |
|------|-----------|------|
| 第一轮 | 1 epoch 训练 + 无 padding 推理 + 无 NMS 后处理 → 结果不可用 | 参数配置严重不足 |
| 第二轮 | 环境验证通过 → 导出 RCNN_Masks 阻塞 → 多轮重试均 ERROR 999999 | CLI 无法运行 geoprocessing 工具 |
| 清理 | D 盘旧文件 + C 盘 conda 缓存被沙箱阻止删除 | 沙箱权限限制 |
| 最终 | 用户中止任务 | 全流程阻塞，无法推进 |

> **核心教训**：ArcGIS Pro 深度学习工具链设计为 **GUI 驱动 + 内部 Python**，不适合外部脚本自动化。任何试图从 CLI/SDK 绕过的方案都会遇到 code page、沙箱、权限三重阻断。

## B.19 LiDAR360 点云分类算法避坑（V3.3新增）

> 来源：CSDN实战专栏（PTD vs CSF深度对比）/ GreenValley官方文档 / 项目实战经验 / 学术论文
> 收录时间：2026-06-03 | 验证级别：Expert
> **⚠️ 本节为用户明确要求的重点内容——分类算法是LiDAR360最易踩坑的领域**

### B.19.1 PTD低矮植被被误分为地面（高频坑）

| 维度 | 说明 |
|------|------|
| **现象** | 大片灌木/草地被错误分类为地面点，DEM出现"绿色凸起" |
| **原因** | max_angle参数设得过大（>15°），使得倾斜角度大的植被也被纳入TIN表面 |
| **解决方案** | 减小max_angle至8-10°；增大cell_size至5m以上；必要时设置高程变化率阈值 |
| **预防措施** | 每次PTD执行后必须做剖面检查；植被覆盖区建议先CSF预处理 |

### B.19.2 PTD采坑边缘系统性高程偏差（矿山项目致命）

| 维度 | 说明 |
|------|------|
| **现象** | 采坑边缘/悬崖处地面点高程系统性偏高0.5-0.8米，方量计算严重失真 |
| **原因** | PTD在陡峭突变地形处过度平滑，TIN面"跨越"了陡坎 |
| **解决方案** | 局部CSF补偿策略（对边缘区域用CSF重新提取）；或手动精修边缘区域 |
| **预防措施** | 矿山项目必须执行三级精度验证（§8.1），特别是边界区域RTK实测比对 |

### B.19.3 CSF陡坡（>40°）地面点大面积丢失（致命坑）

| 维度 | 说明 |
|------|------|
| **现象** | 山区陡坡段地面点完全消失，形成"数据空洞"，DEM断裂 |
| **原因** | CSF布料在陡坡处因重力分量不足无法下垂到真实地面位置 |
| **解决方案** | 降低rigidity至0.5-1.0（可能损失精度）；或改用PTD（max_angle=15°+） |
| **预防措施** | **山区项目首选PTD而非CSF！** 这是最常见的选错算法导致的灾难 |

### B.19.4 CSF高密度点云内存溢出崩溃

| 维度 | 说明 |
|------|------|
| **现象** | 执行CSF时软件无响应或直接崩溃，Windows任务管理器显示内存100% |
| **原因** | 点密度>50点/㎡ + 布料分辨率<0.3m → 布粒子数量爆炸式增长 |
| **解决方案** | 先体素化降采样至<20点/㎡再执行CSF；或增大布料分辨率至≥0.5m |
| **预防措施** | 执行CSF前先检查点密度统计（数据管理→统计模块），高密度必先降采样 |

### B.19.5 CSF路缘石/田埂等小地形特征被抹平

|维度|说明|
|------|------|
|**现象**|道路路缘石、田埂、台阶等精细地形在CSF处理后变平消失|
|**原因**|rigidity过大(>3.0)使布料过硬无法贴合微小起伏|
|**解决方案**|减小rigidity至1.0-2.0；减小cloth_resolution以增加细节捕捉|
|**预防措施**|测绘工程如需保留精细地物特征，优先选用CSF并严格控制rigidity<2.5|

### B.19.6 算法选择错误——平坦城区用了PTD / 陡峭山区用了CSF

|维度|说明|
|------|------|
|**现象**|分类效果明显差于预期，大量错分，反复调参仍无法改善|
|**原因**|算法与地形不匹配——PTD适合陡峭复杂地形，CSF适合平坦规则地形|
|**解决方案**|严格遵循决策树（见36号文件§3.4）：坡度<25°→CSF；25°-35°→混合；>35°→PTD|
|**预防措施**|项目开始前先分析全域坡度分布直方图，基于数据特征选算法而非凭感觉|

### B.19.7 不同测区PTD/CSF参数不一致导致拼接处阶梯状不连续

|维度|说明|
|------|------|
|**现象**|两块相邻测区处理后的地面高程在边界处存在明显阶梯/跳跃|
|**原因**|各测区分别独立运行PTD/CSF，使用了不同的参数组合|
|**解决方案**|全局统一一套参数；或在拼接带做50m缓冲区距离加权融合|
|**预防措施**| 批处理前制作标准参数模板文件，全项目共用同一套参数|

### B.19.8 建筑物提取漏掉低矮建筑（农村地区常见）

|维度|说明|
|------|------|
|**现象**|农村地区的平房、棚屋等低于3米的建筑物全部漏提|
|**原因**|建筑物提取的最小高度阈值默认值过高（通常默认3-5m）|
|**解决方案**|根据项目区域实际最小建筑高度调整min_height参数（农村可设为2.0m）|
|**预防措施**| 提取前先统计点云高程直方图，了解建筑物高度分布范围|

### B.19.9 LiDAR360中文路径导致批处理异常

|维度|说明|
|------|------|
|**现象**|单个文件正常处理但批量处理脚本中途报错，部分文件跳过或输出路径异常|
|**原因**|工作目录或文件路径包含中文字符，内核解析引擎兼容性问题|
|**解决方案**| 统一使用纯英文路径作为项目根目录(如D:/LiDAR360_Workspace/)|
|**预防措施**| 项目初始化模板强制英文路径规范（同Global Mapper坑G13）|

### B.19.10 AI自动分类结果不可直接使用——必须人工审核

|维度|说明|
|------|------|
|**现象**| 使用32类AI自动分类后直接出图，验收时发现大量错分（尤其是类别边界处）|
|**原因**| AI分类依赖训练样本，与当前数据分布不一致时会产生系统偏差|
|**解决方案**| AI分类结果仅作为"预分类"参考，必须经过人工抽检+修正后才可用于生产|
|**预防措施**| 建立AI分类QA流程：AI预分类→人工抽检5%→修正→全检关键类别→入库|
