## M07: 等价类划分法

### 核心方法论
将输入域划分为若干等价类，每个等价类中的数据对程序的揭露缺陷能力等价。从每个等价类中选取一个代表值即可代表整个等价类的测试效果。

### 专家操作流程
1. **识别输入条件**：从需求中提取所有输入参数及其约束
2. **划分有效等价类**：
   - 数值范围：如 1-100 → {1-100}
   - 数据类型：如正整数 → {正整数集合}
   - 集合元素：如性别 → {男, 女, 其他}
   - 布尔条件：如是否会员 → {true, false}
   - 必须遵守的规则：如手机号11位 → {11位数字}
3. **划分无效等价类**（每个条件至少一个）：
   - 数值范围：小于最小值 / 大于最大值 / 非数字
   - 数据类型：字符串 / 空值 / null / 特殊字符
   - 集合元素：不在枚举中的值
   - 格式约束：不符合格式的值
4. **等价类编号与表**：
   | 输入条件 | 有效等价类 | 编号 | 无效等价类 | 编号 |
   |---------|-----------|------|-----------|------|
   | 年龄(1-120) | 1≤age≤120 | EC-01 | age<1 | EC-02 |
   | | | | age>120 | EC-03 |
   | | | | 非数字 | EC-04 |
   | | | | 空值 | EC-05 |
5. **设计测试用例**：
   - 原则：覆盖尽可能多的有效等价类（减少用例数）
   - 每个无效等价类单独一个用例（避免屏蔽效应）
   - 用例数 ≤ 有效等价类数 + 无效等价类数

### 关键检查点
- [ ] 每个输入条件都有有效和无效等价类
- [ ] 无效条件没有遗漏（null/空/特殊字符/超长/类型错误）
- [ ] 无效等价类设计为独立用例（防止前一个无效条件已拦截后续无效条件的测试）
- [ ] 对数值范围额外做了边界值分析（M08）

### 常见陷阱
| 陷阱 | 对策 |
|------|------|
| 把多个无效等价类放在一个用例中 | 一个用例只测一个无效等价类（屏蔽效应） |
| 只划分了数值的等价类，忽略了类型 | 类型也是等价类的重要维度 |
| 等价类划分太粗（把0和负数放在一起） | 0和负数在业务上可能完全不同 |

## M08: 边界值分析法

### 核心方法论
缺陷倾向于聚集在输入域的边界上。对每个等价类的边界值（边界本身+边界两侧）进行测试。

### 专家操作流程
1. **识别边界**：对每个数值范围识别：
   - 最小值（min）
   - 刚好大于最小值（min+1）
   - 刚好小于最小值（min-1）
   - 最大值（max）
   - 刚好小于最大值（max-1）
   - 刚好大于最大值（max+1）
   - 特殊边界：0, 空字符串, null, -1, 最大值+1
2. **边界值测试点**：
   - 数值范围：min-1, min, min+1, max-1, max, max+1
   - 字符长度：minLen-1, minLen, maxLen, maxLen+1
   - 集合大小：空集合, 1个元素, 最大容量, 最大容量+1
   - 时间范围：开始时间前一秒, 开始时间, 开始时间后一秒, 结束时间前一秒, 结束时间, 结束时间后一秒
3. **边界值 × 等价类组合**：
   - 先按等价类划分所有有效/无效类
   - 对有效等价类中的数值范围，补充边界值测试
   - 对无效等价类（如小于最小值），直接取 min-1
4. **次边界值**：
   - 数据类型边界：int32最大值(2^31-1)、超出int32范围
   - 精度边界：浮点数精度（0.1+0.2≠0.3）
   - 编码边界：GBK/UTF-8字符边界、emoji长度边界
   - 时间边界：闰年2月29日、跨时区边界、夏令时切换

### 关键检查点
- [ ] 每个数值范围都覆盖了6点边界值（min-1/min/min+1/max-1/max/max+1）
- [ ] 考虑了数据类型边界（int溢出、浮点精度）
- [ ] 字符串字段覆盖了长度边界
- [ ] 时间字段覆盖了时区和特殊日期边界

### 产出示例
```
字段：订单金额（0.01-99999.99元）
边界值用例：
- BV-01: 0.00（低于最小值，期望拒绝）
- BV-02: 0.01（最小值，期望通过）
- BV-03: 0.02（最小值+0.01，期望通过）
- BV-04: 99999.98（最大值-0.01，期望通过）
- BV-05: 99999.99（最大值，期望通过）
- BV-06: 100000.00（超过最大值，期望拒绝）
- BV-07: -0.01（负数边界，期望拒绝）
- BV-08: 空值（期望拒绝）
- BV-09: 非数字（期望拒绝）
```

### AI增强技巧
- 对每个数值字段，用AI自动生成完整的边界值组合表
- 提示：识别复杂对象（数组、嵌套JSON）的边界条件

## M09: 判定表与因果图

### 核心方法论
当输入条件之间存在逻辑组合关系时，使用判定表（Decision Table）系统化地列出所有条件组合及其对应的动作/结果。因果图（Cause-Effect Graph）是判定表的前置可视化步骤。

### 专家操作流程
1. **识别条件（Cause）**：提取所有影响输出的输入条件
2. **识别动作（Effect）**：提取所有可能的输出结果
3. **画因果图**（可选，复杂业务逻辑时推荐）：
   - 因果关系：Cause → Effect
   - 逻辑关系：AND, OR, NOT, NAND, NOR
   - 约束关系：E（互斥）, I（包含）, O（唯一）, R（要求）, M（屏蔽）
4. **构建判定表**：
   - 条件行 = 输入条件数
   - 规则列 = 2^条件数（全组合）
   - 动作行 = 输出结果数
   - 对每列填入条件取值（Y/N/无关）和对应的动作（√/×）
5. **化简判定表**：
   - 合并动作相同且只有一个条件不同的列（该条件标记为"-"无关）
   - 去除不可能出现的组合（约束关系排除）
6. **生成测试用例**：每条规则 → 一个测试用例

### 判定表示例
```
条件：
C1: 会员等级（普通/银卡/金卡）
C2: 订单金额≥100
C3: 使用优惠券

动作：
A1: 免运费
A2: 9折优惠
A3: 赠送积分（金额×1）

| 规则     | R1  | R2  | R3  | R4  | R5  | R6  |
|----------|-----|-----|-----|-----|-----|-----|
| C1:会员   | 普通 | 普通 | 银卡 | 银卡 | 金卡 | 金卡 |
| C2:≥100  | N   | Y   | N   | Y   | -   | -   |
| C3:优惠券 | -   | N   | -   | N   | N   | Y   |
| A1:免运费 |     | √   | √   | √   | √   | √   |
| A2:9折    |     |     |     | √   | √   |     |
| A3:积分   | √   | √   | √   | √   | √   |     |
```

### 关键检查点
- [ ] 条件之间没有遗漏的组合
- [ ] 约束关系（互斥/包含/要求）已在表中排除
- [ ] 每个规则产生唯一确定的动作组合
- [ ] 化简后的判定表仍覆盖所有有效组合

## M10: 状态迁移测试

### 核心方法论
将被测系统抽象为有限状态机（FSM），对每种状态下的每个事件导致的每次状态迁移进行测试。目标是实现状态覆盖 + 迁移覆盖 + 路径覆盖。

### 专家操作流程
1. **识别状态**：从需求中提取所有可能的状态
   - 显式状态：文档中明确定义的（如订单状态）
   - 隐式状态：系统隐含的（如登录态、加载中、网络异常）
2. **识别事件**：触发状态变化的操作/条件
3. **构建状态迁移图**：
   - 节点 = 状态
   - 边 = 事件 → 目标状态
   - 标注：边上的守卫条件（Guard）和动作（Action）
4. **设计测试覆盖**：
   - 0-switch覆盖（状态覆盖）：访问每个状态至少一次
   - 1-switch覆盖（迁移覆盖）：触发每条迁移至少一次
   - N-switch覆盖（路径覆盖）：N+1个连续迁移的路径组合
   - 非法迁移：触发每个状态的非法事件（负面测试）
5. **状态迁移表**：
   | 当前状态 | 事件 | 守卫条件 | 动作 | 目标状态 | 用例编号 |
6. **设计测试用例**：每条迁移 → 一个测试用例

### 关键检查点
- [ ] 识别了所有隐式状态（加载/错误/超时/空数据）
- [ ] 对每个状态的所有可能事件都设计了测试（包括非法事件）
- [ ] 验证了状态迁移的原子性（不能出现中间状态）
- [ ] 并发状态迁移有覆盖（两个事件同时触发）

## M11: 正交实验设计

### 核心方法论
使用正交表安排多因素多水平的测试，用最少的试验次数来覆盖所有因素的两两组合。

### 专家操作流程
1. **确定因素与水平**：
   - 因素：影响测试结果的变量（如浏览器、OS、分辨率、网络）
   - 水平：每个因素的取值（如Chrome/Firefox/Safari/Edge）
2. **选择合适的正交表**：L_n(t^q)
   - n = 实验次数（行数）
   - t = 水平数
   - q = 因素数
   - 常用正交表：L4(2^3), L8(2^7), L9(3^4), L16(4^5), L18(3^7), L27(3^13)
3. **因素映射**：将正交表的列映射到实际因素，水平映射到实际取值
4. **补齐与调整**：
   - 因素数不足：删除多余列
   - 水平数不足：使用拟水平法（重复某个水平）
   - 混合水平：使用混合正交表（如L18(2^1×3^7)）
5. **生成测试用例**：正交表的每行 → 一个测试用例（测试条件组合）
6. **补充用例**：
   - 高风险组合：对业务关键组合补充全组合测试
   - 已知缺陷模式：补充历史高缺陷率的组合

### 关键检查点
- [ ] 正交表的选择正确（n≥因素数，t≥最大水平数）
- [ ] 因素之间确实相互独立（否则应用判定表）
- [ ] 高风险组合已额外补充全组合测试
- [ ] 没有遗漏重要因素（如数据库版本、中间件版本）

## M12: 场景法测试设计

### 核心方法论
模拟用户使用产品的真实场景（Happy Path + Alternative Path + Exception Path），通过事件流（Event Flow）来描述端到端的业务流程。

### 专家操作流程
1. **场景识别**：
   - **基本流（Happy Path）**：用户正常完成目标的最简路径
   - **备选流（Alternative）**：合法但非最简的路径（如使用优惠券、选择不同支付方式）
   - **异常流（Exception）**：用户操作错误或系统异常（如支付失败、网络中断、库存不足）
2. **场景建模**：
   - 画出场景流程图
   - 在每个决策点标注备选流和异常流
   - 场景编号：S01（基本流）、S02-S0N（备选流）、S0E1-S0EN（异常流）
3. **场景矩阵**：
   | 场景编号 | 场景名称 | 前置条件 | 步骤序列 | 预期结果 | 优先级 |
   |---------|---------|---------|---------|---------|--------|
   | S01 | 正常下单 | 已登录+有货 | 搜索→加购→下单→支付 | 订单创建 | P0 |
   | S02 | 使用优惠券下单 | 有优惠券 | S01+选择优惠券 | 金额减免 | P1 |
   | SE1 | 支付失败重试 | S01到支付步骤 | 支付失败→重试→成功 | 订单创建 | P1 |
4. **场景优先级**：
   - P0：基本流 + 核心异常流（支付失败、登录过期）
   - P1：常用备选流 + 重要异常流
   - P2：低频备选流 + 边界异常流
   - P3：极端场景（如跨天、大促高并发）

## M13: 组合测试设计

### 核心方法论
对具有交互效应的多个参数，使用Pairwise（两两组合）或N-wise覆盖来设计测试用例，在有限测试成本下捕获参数交互引起的缺陷。

### 专家操作流程
1. **参数与取值建模**：
   - 识别有交互效应的参数
   - 列出每个参数的所有可能取值
   - 标注参数的约束关系（互斥/依赖）
2. **选择覆盖强度**：
   - t=1（单因素）：等价于等价类划分
   - t=2（Pairwise）：两两组合覆盖，最常用
   - t=3（3-wise）：三三组合，高风险系统
   - t=4+（N-wise）：关键安全/生命攸关系统
3. **生成组合**：使用工具（PICT/ACTS/AllPairs）
   - 指定模型文件（参数+取值+约束）
   - 运行生成器获得最小测试集
4. **添加约束**：
   - IF [条件] THEN [约束]
   - 示例：IF [OS=Windows] THEN [Browser != Safari]
5. **补充高风险组合**：对核心功能的关键参数组合，增加人工设计的确定性用例

### Pairwise示例
```
参数模型：
OS: Windows, macOS, Linux, Android, iOS
Browser: Chrome, Firefox, Safari, Edge
Screen: 1920x1080, 2560x1440, 1366x768, 375x667
Network: 4G, WiFi, 3G, Offline

全组合：5×4×4×4 = 320个用例
Pairwise：约20-25个用例（覆盖所有两两组合）
```

## M14: 错误推测法

### 核心方法论
基于测试人员的经验、直觉和对系统的深入理解，推测可能存在的缺陷并进行针对性测试。虽然是"推测"，但专家的推测基于结构化的缺陷模式知识库。

### 专家操作流程
1. **缺陷模式库检索**：
   - 输入类型相关：SQL注入/XSS（用户输入）、精度丢失（浮点计算）、溢出（大数值）
   - 状态相关：并发竞态、死锁、状态残留
   - 资源相关：内存泄漏、连接池耗尽、文件句柄泄漏
   - 时间相关：时区错误、闰秒、夏令时、时间倒流
   - 网络相关：超时、断网重连、慢网络、DNS解析失败
   - UI相关：遮挡、重复点击、快速滑动、横竖屏切换
2. **历史缺陷分析**：
   - 查阅该产品/模块的历史缺陷
   - 分析根因分布（逻辑错误/边界遗漏/并发/环境）
   - 识别高频缺陷类型
3. **技术栈缺陷模式**：
   - Java：NullPointer、ConcurrentModification、内存溢出
   - JavaScript：异步陷阱、this绑定、浮点精度、闭包泄漏
   - Python：可变默认参数、GIL、编码问题
   - SQL：N+1查询、死锁、索引失效
   - 微服务：分布式事务、服务降级、数据一致性
4. **攻击性测试清单**：
   | 攻击类型 | 测试手段 | 适用场景 |
   |---------|---------|---------|
   | 脏数据注入 | SQL/JS/HTML/特殊Unicode | 所有输入框 |
   | 极限值轰炸 | 最大/最小/超长/负数/null | 所有数值字段 |
   | 时序破坏 | 快速连点/并发请求/乱序操作 | 所有带状态的流程 |
   | 资源耗尽 | 大数据量/高频请求/长连接 | 列表/搜索/上传 |
   | 环境突变 | 断网/杀进程/切后台/低电量 | 移动端 |

### 关键检查点
- [ ] 缺陷模式库覆盖了输入/状态/资源/时间/网络/UI六大类
- [ ] 针对当前系统的技术栈选择了对应的缺陷模式
- [ ] 参考了该产品的历史缺陷数据
- [ ] 错误推测用例被标记为"经验性测试"，便于追溯
