## 重要声明硬规则

- 所有功能 1/2/3/4 的聊天答复、Markdown 报告和 HTML 页面，只允许使用下面这一句作为重要声明：`重要声明：CoopLens Skill 产出的内容由 AI 生成，真实性需要使用者自行核实，不代表任何官方意见，不能作为任何决策依据或参考。`。
- 不得另写“位次预估专用声明”“数据来源声明”“权威来源声明”或其他以 `重要声明：` 开头的句子。
- `位次预估重要声明` 只能作为小标题；其下方必须逐字放置固定重要声明。
- 完整报告末尾必须有 `## 附录：重要声明与人工确认提醒`，并再次逐字放置固定重要声明。
- 不得写兜底式来源承诺，例如声称本报告全部数据/全部信息/全部数字结论已经核验、均来自某类渠道、或都带有官方可验证链接。来源只能逐条列在对应数据项或人工确认清单里。

## 分析执行顺序

1. **获取运行时日期**：优先使用用户本地日期工具；没有时运行 `python scripts/cooplens_core.py runtime-date`。日期用于判断当前年份、最新招生材料和最近完成招生年度。
2. **解析问题**：提取学校、合作方、学院/项目、专业或大类、所在省份、科类/选科、招生批次、专业组、分数/位次、预算、家庭费用压力、出国偏好、留服认证需求、未来路径、英语/GPA 承受度、城市/校区偏好和具体疑问。
3. **实时搜索最新官方资料**：检索并打开当年招生简章、招生章程、招生计划、省考试院专业目录、招生批次/专业组/计划类型、当年是否招生/是否停招/是否暂停招生、学费、证书、CRS/监管记录、教育部留学服务中心/CSCSE 认证指南、认证服务大厅、培养方案、分数线、一分一段、批次控制线、就业质量报告、本科教学质量报告，以及排名数字涉及的 QS/TopUniversities 最新官方榜单页面。若项目含境外阶段或用户表达出国意愿，还必须检索外方大学所在城市的最新生活成本资料。
4. **当年是否招生入场闸门**：在写任何结论、排序、冲稳保或位次预估前，先用教育部中外合作办学监管工作信息平台/CRS 或教育部官网核对项目身份，再用学校本科招生网/招生办官网、当年招生章程/招生简章、目标省招生计划/专业目录核对今年是否仍招生。CRS 有效只证明项目身份或审批状态，不等于今年仍招生；若学校或省级当年计划未出现该项目/专业，或官方材料显示停招/暂停招生，必须标注 `未在当年招生计划中` 或 `当年停招/暂停招生`，不得进入本年位次公式、冲稳保判断或硬排序，只能放入 `仅关注/待核验，不参与排序` 或 `暂不建议填报`。
5. **核验证据再写事实**：每个关键事实必须有可打开且内容对应的来源；空链接、打不开的链接、只到官网首页的链接、内容不对应的链接都不能支撑结论。
6. **统一入模分析**：把办学身份、计划性质、证书组合、留服认证路径、外方质量、四年总投入、语言/GPA/延毕压力、校区资源、转专业/调剂、未来路径、公开讨论问题和官方核验边界合并判断。
7. **留服认证分析**：凡涉及外方学位、境外阶段、4+0、2+2、3+1、1+3 或计划外路径，必须写 `留服认证路径与风险`：认证对象、认证路径、项目模式、境外学习记录、计划内/计划外、国内证书兜底、认证结果预期、传统海归/留学生待遇边界、风险等级和向学校书面确认的问题。不能承诺一定认证；证据不足时降为观察、慎选或仅关注/待核验。
8. **位次预估章**：只要分析涉及分数线、投档线、最低位次、候选可冲稳保或新项目门槛，必须单独写 `位次预估：统计、分析与预测`。章内先写 `位次预估结论先行`，给参考区间、冲/稳/保判断、置信度和会推翻结论的条件；随后顺序固定为统计可验证数据、分析位差/线差/样本、预测区间。已有历史线用直接历史线；新项目、首次招生或无历史线项目用 B 普通招生基线 + A 同省同档样本 + 三情景位次参照；证据不足时放入“仅关注/待核验，不参与排序”。凡给出本年位次估计，必须展示“输入数据 → 基线选择 → 差值样本 → 调整项 → 计算过程 → 置信度 → 推翻条件”的完整可核验推导，并附人工确认清单。`计算过程` 必须和结论分离，建议用 fenced code block 放公式链；移动端 HTML 中这块必须显示为黑底红字。

9. **招生批次口径锁定**：凡涉及招生人数、招生计划、投档线、专业线、最低分、最低位次、线差、位差或候选排序，必须先锁定 `年份 + 省份 + 科类/选科 + 招生批次 + 院校专业组 + 专业/大类`。同一项目若出现在提前批、本科批、综合评价批、专项计划、中外合作单列批或不同专业组中，必须分批次列出和分析；不同批次不得合并招生人数、分数线或位次。批次未核到时只能列为弱证据或待核验，不能进入公式和硬排序。
10. **功能 3 阅读顺序**：当用户输入以 `3，` 或 `3,` 开头时，Markdown 正文必须先给 `## 目录`，然后给 `## 一、最重要建议（先看这里）`。这一节要直接告诉家长优先/备选/观察/慎选/仅关注待核验、当前位次与预算是否卡住、下一步怎么填；候选项目卡片和横向对比放在后面；统计口径、来源边界、方法解释和人工确认清单继续后置。不要让功能 3 先铺长篇方法、来源说明或解释性材料。
11. **公开讨论分流**：匿名整理正面主题、负面担忧和澄清；能由官网、教育主管部门或权威原文回答的质疑直接回答，不能回答的转成向学校书面确认的问题。
12. **培养成果检查**：对已有毕业生的机构或项目，查找世界名校深造、知名企业就业、本科生一作高水平论文、竞赛和科研训练等公开成果；不展示姓名和个人身份。
13. **双推荐度判断**：先给 `项目综合实力推荐度评价（项目综合实力角度）`，再给 `个性化推荐度评价（学生/家庭适配）`。项目综合实力看办学身份、证书/留服、专业、师资、城市、费用、毕业成果和风险；个性化适配看家庭预算、出国偏好、留服认证需求、未来路径、英语/GPA 承受度、城市/校区偏好和学生分数/位次。用户给出预算或位次时，必须围绕预算差额和位次安全边际加做对比；用户未提供家庭信息时，用“预算宽裕/中等/紧张”和“愿意出国/可接受/不愿出国”等情景分别判断。
14. **非中庸结论**：必须给出优先、备选、观察、慎选或总体排序，并说明什么条件会推翻当前结论。
15. **关键数据硬闸门**：招生人数、分数线、分位数、升学率、出国率、就业率、保研率等关键数据必须逐项带来源超链接并写成 `约...`。必须打开链接确认页面里能找到对应数据；如果检索不到、链接打不开、链接内容不含对应值，找不到就写未知，格式为 `未知（未核到可打开且内容对应的来源链接）`，不得乱猜、不得用经验值或相近项目补齐。Markdown 完成后必须运行 `critical-data-source-check`，需要联网核对链接内容时运行 `critical-source-evidence-check`。
15. **生成交付物**：默认生成直接聊天答复、带目录的 Markdown 报告、移动端原生 HTML/CSS 静态页面，不默认生成其他格式。HTML 必须更适合手机阅读：色块区分章节、目录锚点可点击跳转、重要声明清晰可见，且报告末尾保留 `附录：重要声明与人工确认提醒`。文件名只能根据用户任务生成，不能使用运行平台、通用采集工具或页面标签作为文件名。HTML 交付必须走安全渲染闸门：先用 `safe-html-build` 从 Markdown 生成，再依次用 `html-syntax-check`、`html-report-check`、`html-render-gate-check` 校验；若失败，必须反馈 HTML 渲染失败原因，修正 Markdown/目录/锚点/标题/链接后重新生成。不通过就重新生成，直到三项校验全部通过；不交付失败 HTML。正常交付不得只给聊天分析或只给 Markdown；最终答复必须含任务命名的、可点击的、直接指向单个纯 `.md` 文件和单个纯 `.html` 文件的下载链接，否则视为未完成。`.zip`、`.rar`、`.7z`、`.tar.gz`、压缩包、打包文件、文件夹链接或“打包下载”不能替代 `.md`/`.html` 直链；功能报告不得生成或交付 ZIP 包。内容像 HTML 不等于交付的是 HTML 文件：外层是 `.zip`、文件无 `.html` 后缀、或字节头是 ZIP/压缩包，都必须判失败并重做。
16. **最终交付闸门**：写最终答复前，先运行 `pure-report-file-check <task-name>.html <task-name>.md` 确认本地文件真的是纯 `.html` 和纯 `.md`，再把最终答复草稿保存为 `final-answer.md`，并运行 `strict-final-delivery-check`，再运行 `html-delivery-check`；功能 3 还必须运行 `function3-delivery-gate-check`。检查不通过时，不允许发送最终答复，必须继续修正并补齐通过校验的 `.html` 文件，并补齐 `.html` 与 `.md` 两个可点击直链；若出现 `.zip`/压缩包/打包合集/文件夹链接作为交付链接，视为交付失败，必须重做直至 `strict-final-delivery-check`、`html-delivery-check` 或 `function3-delivery-gate-check` 全部通过；错误报告只能用于定位原因，不能替代 HTML 交付。

## HTML 渲染失败恢复流程

所有 4 个入口功能都必须遵守同一套 HTML 恢复流程，功能 3 不得例外。尤其是用户输入以 `3，` 或 `3,` 开头时，属于功能 3 候选项目检索，不能只输出聊天正文或 Markdown；必须尝试生成 HTML，并在最终答复中提供直接指向单个纯 `.html` 文件和单个纯 `.md` 文件的下载链接，不能给 ZIP/压缩包/打包合集来替代。

1. 先完成 Markdown 报告，并通过目录、重要声明、位次、功能深度和来源核验。
2. 使用 `python scripts/cooplens_core.py safe-html-build <report.md> --out <task-name>.html --error-out <task-name>_html_audit.json` 生成静态 HTML。
3. 使用 `python scripts/cooplens_core.py html-syntax-check <task-name>.html` 做 HTML 语法校验，检查文档壳、标签栈、闭合顺序、重复 shell 和解析顺序。
4. 使用 `python scripts/cooplens_core.py html-report-check <task-name>.html` 和 `python scripts/cooplens_core.py html-render-gate-check <report.md> <task-name>.html` 做最终闸门校验，重点检查结构闭合、目录目标、重复 id、重要声明、可见文本长度、事实一致性、ASCII-only 锚点、WebView-safe 外部链接、任务标题和 safe-renderer 标记。
5. 如果任何一步失败，先把失败原因翻译成用户能看懂的一句话，例如“目录锚点没有对应章节”“重要声明未显示”“HTML 标签栈不合法或闭合顺序错误”“可见正文过短”“标题或目录锚点用了中文 id，移动端容器可能跳转失败”“链接里混入了中文标点或解释文字”“HTML 不是由 safe-html-build 生成”“浏览器标题或首页大标题变成了 `## 目录`，说明 Markdown 缺少有效任务标题”。随后只修正 Markdown 或目录并重新运行 safe-build，不手工改 HTML。
6. 不通过就重新生成：重复第 2-5 步，直到 `html-syntax-check`、`html-report-check`、`html-render-gate-check` 全部通过。脚本单次有防死循环上限；达到上限仍失败时，必须继续修 Markdown 后再次运行，而不是交付失败 HTML。

功能 3 常见失败原因是候选项目多、同名小标题多、目录锚点不匹配、重复 id、中文 id、带中文标点的外部链接或手写 HTML 片段。因此功能 3 的项目卡片标题应使用清晰唯一名称，例如 `### 候选A：学校 + 项目/专业`；HTML 里的 id/href 由脚本统一生成，不从标题文本手动复制。


### 标题与目录解析失败防线

移动端容器对标题解析更严格。任何 HTML 的 `<title>` 或 hero `<h1>` 不得是 `## 目录`、`目录`、`正文`、`重要声明` 或任何带 `#` 的 Markdown 标题语法。若 Markdown 第一部分只有重要声明、日期和 `## 目录`，必须先补入任务型一级标题，例如 `# 广东港中深理科试验班位次预估报告`，或让 `safe-html-build` 从任务文件名推导标题。`html-report-check` 必须能拦截 `## 目录` 标题；拦截后重新生成，不交付该 HTML。

## 最终下载链接格式

每个入口功能执行完后，最终答复末尾必须出现 `交付文件下载链接`，且至少包含两行：

```markdown
### 交付文件下载链接

- 静态网页（HTML）下载链接：[下载/打开 {任务命名}.html](当前环境提供的可点击文件链接)
- Markdown 文件下载链接：[下载 {任务命名}.md](当前环境提供的可点击文件链接)
```

规则：

- 两个链接都必须是用户可点击的下载链接；纯文件名、普通路径、隐藏附件或只说“已生成”都不算完成。
- `.html` 与 `.md` 必须使用同一任务命名主干，不能使用 `report`、`html`、`markdown`、`output` 等通用文件名。
- 链接行必须清楚写出 `下载链接`；如果当前平台会把文件卡片自动展示，仍要在最终文字里显式给出这两条下载链接。
- 交付区块里只能有两个报告下载目标：一个 `.html`、一个 `.md`；不得出现第三个 ZIP、压缩包、打包合集、文件夹或“全量下载”链接。
- 写完最终答复草稿后必须运行：`python scripts/cooplens_core.py strict-final-delivery-check final-answer.md`。不通过就重做，直到脚本返回 `ok: true`。
- 若最终答复缺少任意一个下载链接，或脚本提示出现 ZIP/archive/bundle，必须继续补齐/重做后再发送。

## HTML 结果缺失处理

如果最终答复里没有 `.html` 文件下载链接，或没有 `.md` 文件下载链接，这不是成功交付。必须按以下顺序处理：

1. 检查是否已经生成 Markdown 报告。没有 Markdown 时先生成 Markdown。
2. 运行 `safe-html-build`。若通过，重新写最终答复并附上 `.html` 与 `.md` 两个下载链接。
3. 若失败，读取 `*_html_audit.json`，修正 Markdown 标题、目录、重复章节、链接和重要声明后继续运行 `safe-html-build`。
4. 不通过就重新生成，直到 HTML 语法校验和渲染闸门全部通过；不能把“没有 HTML”或“有错误报告”包装成成功。
5. 先运行：`python scripts/cooplens_core.py pure-report-file-check <task-name>.html <task-name>.md>`，再运行：`python scripts/cooplens_core.py strict-final-delivery-check <final-answer.md>`。
6. 再运行：`python scripts/cooplens_core.py html-delivery-check <final-answer.md>`；功能 3 额外运行：`python scripts/cooplens_core.py function3-delivery-gate-check <final-answer.md>`。
7. 任一脚本失败都必须重做最终交付区块，直到 `.html` 与 `.md` 两个单文件下载链接通过校验；不得用 ZIP/压缩包临时替代。
