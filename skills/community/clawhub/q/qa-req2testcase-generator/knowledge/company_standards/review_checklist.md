# 用例评审检查清单

> 文档编号：L3-REV-001  
> 版本：v1.0  
> 生效日期：2026-04-18  
> 适用范围：全公司测试用例评审活动

## 1. 目的

规范测试用例评审流程，提供可操作的检查清单，确保评审结果一致、可追溯，提升用例质量。

## 2. 评审维度与检查项

### 2.1 完整性（Completeness）

评审目标：用例是否覆盖了需求文档中的所有功能点和业务场景。

- [ ] 每条需求均有对应的测试用例，需求-用例追溯矩阵完整
- [ ] 正向流程（Happy Path）用例完整覆盖
- [ ] 反向流程（异常/错误输入）用例已编写
- [ ] 边界值用例已覆盖（最大值、最小值、临界值）
- [ ] 前置条件和后置条件已明确描述
- [ ] 测试数据需求已列出（含正常数据和异常数据）
- [ ] 涉及的接口/模块间交互场景已覆盖
- [ ] 兼容性场景已考虑（浏览器/操作系统/设备）
- [ ] 权限相关场景已覆盖（不同角色的操作权限）
- [ ] 并发/性能相关场景已识别并标注（如适用）

### 2.2 准确性（Accuracy）

评审目标：用例描述是否准确反映需求意图，预期结果是否正确。

- [ ] 用例标题准确概括测试目的，无歧义
- [ ] 测试步骤与需求描述一致，无偏离
- [ ] 预期结果明确、可量化，避免"正常显示"等模糊描述
- [ ] 预期结果与需求文档/原型图/接口文档一致
- [ ] 测试数据符合业务规则（如股票代码格式、金额精度）
- [ ] 涉及计算的用例，预期结果经过手工验算
- [ ] 状态流转用例的前后状态描述正确
- [ ] 时间相关用例的时间窗口标注正确（参照交易时间窗口规范）
- [ ] 错误码/提示信息与接口文档定义一致

### 2.3 可执行性（Executability）

评审目标：用例是否能被任意测试人员独立执行，无需额外解释。

- [ ] 测试步骤描述清晰，操作粒度适当（非过粗或过细）
- [ ] 每个步骤均为可执行的具体操作，非抽象描述
- [ ] 测试环境要求已明确（环境地址、配置要求）
- [ ] 测试数据准备步骤已说明（含造数方式或数据来源）
- [ ] 依赖的前置操作已列为前置条件或关联用例
- [ ] 涉及第三方系统的用例已说明Mock/桩方案
- [ ] 用例执行顺序无隐式依赖（或已明确标注依赖关系）
- [ ] 验证点明确，测试人员可判断通过/失败
- [ ] 数据清理/环境恢复步骤已说明

### 2.4 覆盖度（Coverage）

评审目标：用例集整体是否达到充分的测试覆盖。

- [ ] 等价类划分合理，每个等价类至少一条用例
- [ ] 边界值分析完整（开区间/闭区间/临界点）
- [ ] 状态迁移覆盖所有有效状态转换路径
- [ ] 组合场景已使用正交法或Pairwise方法精简
- [ ] 业务规则的每个分支条件均有用例覆盖
- [ ] 异常场景覆盖：网络超时、服务不可用、数据库异常
- [ ] 安全测试场景：SQL注入、XSS、越权访问
- [ ] 回归测试用例已标注（核心功能的冒烟用例集）
- [ ] 脱敏规范已遵守（参照测试数据脱敏规范）

## 3. 评审通过标准

### 3.1 必须满足条件（全部通过方可评审通过）

| 序号 | 条件 | 说明 |
|------|------|------|
| 1 | 完整性检查项通过率 ≥ 90% | 未通过项需标注原因并限期补充 |
| 2 | 准确性检查项通过率 = 100% | 预期结果不准确的用例必须修正 |
| 3 | 可执行性检查项通过率 ≥ 95% | 不可执行的用例必须修正 |
| 4 | 覆盖度检查项通过率 ≥ 85% | 未覆盖项需评估风险并记录 |
| 5 | 无P0/P1级评审问题遗留 | 严重问题必须当场解决或限期修正 |
| 6 | 需求追溯矩阵完整 | 每条需求至少关联一条用例 |

### 3.2 评审结果分类

| 结果 | 条件 | 后续动作 |
|------|------|----------|
| 通过 | 所有必须满足条件均达标 | 用例基线化，进入执行阶段 |
| 有条件通过 | 存在P2/P3级问题，数量≤5个 | 限期2个工作日内修正，修正后无需重新评审 |
| 不通过 | 存在P0/P1级问题，或P2/P3级问题>5个 | 修正后重新组织评审 |

### 3.3 评审问题分级

| 等级 | 定义 | 示例 |
|------|------|------|
| P0 | 用例与需求严重不符或遗漏核心场景 | 交易下单功能缺少撤单用例 |
| P1 | 预期结果错误或关键步骤缺失 | 融资买入保证金比例计算预期值错误 |
| P2 | 描述不清晰或覆盖不充分 | 边界值用例缺少最大持仓量场景 |
| P3 | 格式/命名不规范等轻微问题 | 用例编号未按规范命名 |

## 4. 常见问题清单（高频不合格项）

### 4.1 完整性类

| 序号 | 常见问题 | 改进建议 |
|------|----------|----------|
| 1 | 只写了正向用例，缺少异常场景 | 每个功能点至少编写1条异常用例 |
| 2 | 缺少权限校验用例 | 参照敏感操作权限校验标准补充 |
| 3 | 未覆盖并发场景 | 涉及共享资源的功能必须考虑并发 |
| 4 | 接口间交互场景遗漏 | 绘制接口调用链路图辅助识别 |
| 5 | 缺少数据清理/回滚用例 | 涉及数据变更的用例必须包含清理步骤 |

### 4.2 准确性类

| 序号 | 常见问题 | 改进建议 |
|------|----------|----------|
| 1 | 预期结果写"正常"、"成功"等模糊描述 | 明确具体返回值、状态码、页面元素 |
| 2 | 金额计算未考虑精度（如佣金四舍五入规则） | 预期结果精确到分，标注计算公式 |
| 3 | 时间相关用例未标注交易时段 | 参照交易时间窗口规范标注 |
| 4 | 错误提示信息与接口文档不一致 | 评审前核对最新接口文档 |

### 4.3 可执行性类

| 序号 | 常见问题 | 改进建议 |
|------|----------|----------|
| 1 | 测试步骤跳跃，缺少中间操作 | 新人视角审视步骤完整性 |
| 2 | 未说明测试数据准备方式 | 标注造数SQL或数据准备接口 |
| 3 | 依赖其他用例但未标注 | 使用前置条件字段明确关联 |
| 4 | 验证点不明确，无法判断通过/失败 | 每条用例至少一个可量化的断言 |

### 4.4 覆盖度类

| 序号 | 常见问题 | 改进建议 |
|------|----------|----------|
| 1 | 等价类划分不合理，用例冗余 | 使用等价类划分表辅助分析 |
| 2 | 缺少边界值用例 | 对每个输入域进行边界值分析 |
| 3 | 状态迁移路径覆盖不全 | 绘制状态迁移图，逐路径编写用例 |
| 4 | 安全测试场景缺失 | 参照OWASP Top 10检查清单补充 |

## 5. 评审流程

### 5.1 评审前准备

1. 用例作者提前1个工作日将用例文档发送给评审参与人
2. 评审参与人提前阅读用例并记录问题
3. 准备需求文档、接口文档、原型图等参考资料

### 5.2 评审会议

1. 用例作者介绍用例设计思路（5~10分钟）
2. 逐维度检查（按本清单逐项确认）
3. 记录评审问题并分级
4. 确定评审结论和修正计划

### 5.3 评审后跟踪

1. 用例作者按评审意见修正
2. 修正完成后通知评审负责人确认
3. 评审记录归档至项目文档库

## 6. 附则

- 每次评审须使用本清单逐项检查，评审记录中标注每项的通过/不通过状态。
- 评审清单可根据项目特点增加检查项，但不得删减本文档中的必选项。
- 评审通过的用例集作为测试基线，后续变更需走变更评审流程。

## 7. 与 AI 用例质量自检（P7）的关系说明

> 本节说明本清单与 P7 Prompt 质量自检的定位差异，避免 RAG 召回时产生冲突指令。

### 7.1 定位区分

| 维度 | 本清单（人工评审） | P7 质量自检（AI 自检） |
|------|-----------------|-------------------|
| 执行时机 | 用例设计完成后，人工评审阶段 | P6 生成用例后，自动执行 |
| 执行者 | 测试负责人/评审专家 | AI（P7 Prompt） |
| 检查重点 | 业务理解准确性、规范符合性、可执行性 | 格式完整性、字段合规性、测试点覆盖率 |
| 输出 | 评审意见，需人工修正 | 通过/不通过结论，自动触发 P6 修正 |

### 7.2 互补关系

- **P7 先行**：AI 生成用例后先经过 P7 自检，过滤格式问题和明显遗漏
- **本清单后置**：P7 通过后，再由人工按本清单做深度评审（业务逻辑、边界场景等）
- **不替代**：P7 通过不等于本清单通过，两者检查维度不同，互补而非替代

### 7.3 重叠项处理原则

本清单与 P7 C1-C8 在"字段完整性"维度有交叉，判断标准略有差异：
- **P7 标准**（AI 可判断）：字段非空、禁止模糊动词（如"正确处理"）、步骤与期望一一对应
- **本清单标准**（人工判断）：无歧义、无二义性、符合业务语义

两者标准不冲突，P7 做机器可判断的格式校验，本清单做需要业务理解的语义校验。
