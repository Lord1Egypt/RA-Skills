# 查单逻辑设计指引（境外机构）

在微信支付接入中，**查单**（Query Order）是保证订单状态**最终一致性**的核心手段。本文档说明何时查单、如何设计查单策略——尤其针对**机构 + 子商户**多租户场景。

---

## 1. 为什么需要查单

### 1.1 支付系统中的不确定性
以下场景都可能导致**机构不知道订单真实状态**：

| 场景 | 问题 |
|------|------|
| 下单请求超时 | 不知道微信是否创建订单 |
| 用户支付后网络中断 | 同步响应未收到，不知是否成功 |
| 异步回调丢失 | 网络故障、notify_url 不通、机构服务器临时宕机 |
| 付款码支付 USERPAYING | 用户还在输密码，需轮询 |
| 跨境网络抖动 | 境外网络比境内更不稳定，超时 / 5xx 概率更高 |

### 1.2 查单的作用
- **最终一致**：不管上面哪种情况，查单可以拿到**权威状态**
- **兜底机制**：异步回调丢失时的备选方案
- **幂等确认**：重试前确认实际状态

---

## 2. 查单接口

### 2.1 V3 查单（境外）

```
GET /v3/global/transactions/out-trade-no/{out_trade_no}?sp_mchid={sp_mchid}&sub_mchid={sub_mchid}
```

⚠️ 境外机构查单必须**同时传 `sp_mchid` + `sub_mchid`**，缺一不可。

### 2.2 返回字段

| 字段 | 说明 |
|------|------|
| `out_trade_no` | 商户订单号 |
| `transaction_id` | 微信订单号（支付成功才有） |
| `trade_state` | 订单状态（见下方）|
| `trade_state_desc` | 状态描述（如"支付成功"、"已关闭"）|
| `amount.total` | 总金额 |
| `amount.currency` | 币种 |
| `payer.sub_openid` | 子商户 AppID 下的支付用户 OpenID |
| `success_time` | 支付成功时间 |
| `sub_mchid` | 子商户号（用于路由） |

### 2.3 订单状态（trade_state）

| 状态 | 说明 | 处理建议 |
|------|------|---------|
| `SUCCESS` | 支付成功 ✅ | 最终状态，更新订单为已支付，按 `sub_mchid` 入账 |
| `REFUND` | 已退款 | 查退款单了解详情 |
| `NOTPAY` | 未支付 | 中间状态，可继续等待或关单 |
| `CLOSED` | 已关闭 | 最终状态，订单终结 |
| `REVOKED` | 已撤销（仅付款码）| 最终状态 |
| `USERPAYING` | 用户支付中 | 继续轮询 |
| `PAYERROR` | 支付失败 | 最终状态，通常是银行返回失败 |

**最终状态**（不会再变化）：`SUCCESS`、`REFUND`、`CLOSED`、`REVOKED`、`PAYERROR`
**中间状态**（可能变化）：`NOTPAY`、`USERPAYING`

## 3. 典型场景下的查单策略

### 3.1 下单接口超时 / 无响应（原单重入）

下单类接口（JSAPI / Native / APP / H5 / 小程序的 `transactions/...`）只生成预支付凭证（`prepay_id` / `code_url` / `h5_url`），**不真正扣款**，**接口本身是幂等的**——同一个 `out_trade_no` 重复请求，微信会返回同一笔订单的同一个凭证。

因此处理超时 / 无响应的最佳方式是 **原单重入**，无需先查单：

```
调用下单 API
    │
    ├── 正常返回：走正常流程
    │
    └── 超时 / 网络异常 / 5xx：
          │
          ├── ✅ 直接用原 out_trade_no 重试下单（最多 2-3 次，指数退避）
          │     ├── 微信侧若已创建订单：返回同一笔订单的凭证（幂等）
          │     └── 微信侧未创建：本次正常创建
          │
          └── 重试仍失败 → 告警 + 视为下单失败，前端展示重试入口
```

**关键原则**：
- ⚠️ **必须用同一个 `out_trade_no`** 重试，**严禁换新订单号**——换新订单号才会创建出"两笔订单"
- 下单接口幂等，重入是最简单稳妥的兜底，**不需要先查单**
- 退出循环后，若用户最终未支付，由订单超时机制 + 闭环操作（关单 / 撤销）兜底（详见 § 3.5）

> ⚠️ **付款码（Quick Pay）是例外**：付款码支付接口**真正扣款**，不是简单生成凭证，超时不能盲目重入（可能重复扣款）。付款码超时 / 无响应的正确处理是**先查单**确认实际状态，详见 § 3.2。

### 3.2 付款码支付 - USERPAYING / 网络超时

付款码支付（`/v3/global/transactions/micropay`）**直接扣款**（非生成凭证），与其他下单接口不同——超时必须**先查单确认**，不能盲目重入。

#### 3.2.1 网络超时 / 无响应（不确定是否扣款）

```
调付款码支付，未收到响应（连接超时 / 读超时 / 5xx）
    │
    ▼
⚠️ 切勿用同一参数直接重试（可能重复扣款）
    │
    ▼
用原 out_trade_no 查单（带 sp_mchid + sub_mchid）
    │
    ├── ORDERNOTEXIST → 微信侧未创建，可原参数重试 micropay（仍用同一 out_trade_no）
    │
    ├── NOTPAY → 微信已收到但未扣款，可视为失败 / 引导用户重新出码
    │
    ├── USERPAYING → 进入 § 3.2.2 USERPAYING 轮询流程
    │
    ├── SUCCESS → 实际已扣款成功，更新订单为 PAID
    │
    └── PAYERROR → 支付失败，结束
```

#### 3.2.2 接口同步返回 USERPAYING（用户在输密码）

```
调付款码支付返回 USERPAYING
    │
    ├── 立即启动轮询查单
    │
    ├── 间隔：3s, 5s, 10s, 15s, 30s, 30s...
    │
    ├── 每次查单
    │     │
    │     ├── SUCCESS：支付成功，退出
    │     │
    │     ├── PAYERROR：支付失败，退出
    │     │
    │     └── USERPAYING：继续轮询
    │
    └── 总时间超过 30 秒仍为 USERPAYING：
          │
          ├── 调用撤销接口（详见 § 3.5.1）
          │
          └── 告知收银员"用户未完成支付"
```

**示例代码（Java）**：
```java
long[] intervals = {3000, 5000, 10000, 15000, 30000, 30000}; // ms
for (long interval : intervals) {
    Thread.sleep(interval);
    Order order = queryOrder(outTradeNo, spMchid, subMchid);
    if ("SUCCESS".equals(order.getTradeState())) {
        return order;
    }
    if ("PAYERROR".equals(order.getTradeState())) {
        return order;
    }
    // USERPAYING 继续轮询
}
// 超时，撤销订单
reverseOrder(outTradeNo, spMchid, subMchid);
```

### 3.3 异步回调丢失（兜底）

微信的异步通知可能因各种原因未送达：
- notify_url 临时故障
- 防火墙拦截
- 服务器宕机
- 跨境网络抖动

**兜底方案**：定时任务查单

```
定时任务（每 5-10 分钟）
    │
    ├── 查询：所有状态为"待支付"、创建时间 > N 分钟的订单
    │
    ├── 循环查单（带 sub_mchid）
    │     │
    │     ├── SUCCESS：补偿处理（按 sub_mchid 路由 → 更新状态、发货等）
    │     │
    │     ├── CLOSED / REVOKED：更新为关闭
    │     │
    │     └── NOTPAY：继续等待（或超期关单）
```

⚠️ **机构特别注意**：兜底查单结果回填时，必须按 `sub_mchid` 路由到正确的子商户上下文，**避免串单**（详见 [📄 接入质量检查](./接入质量检查.md)）。

### 3.4 退款状态查询

退款也是**异步处理**，建议：
- 发起退款后返回"处理中"给用户
- 退款通知到达时更新状态
- 同时定时任务查退款单兜底

查询接口：
```
GET /v3/global/refund/domestic/refunds/{out_refund_no}?sub_mchid={sub_mchid}
```

### 3.5 订单闭环操作（订单超时 / 状态不明）⭐

当订单出现以下情况，必须主动做"闭环"操作，避免悬挂订单或资金风险：
- 订单超过有效期（如 Quick Pay 30 秒、其他产品 2 小时 / H5 5 分钟）
- 多次查单仍为 `USERPAYING` / `NOTPAY`，状态不明
- 网络异常下不确定订单是否成功

**根据产品类型分两种闭环路径**：付款码支付走**撤销**，其他产品走**关单**。

---

#### 3.5.1 付款码支付（Quick Pay）—— 撤销（Reverse）

```
订单超时 / 状态不明
    │
    ▼
调用撤销接口
POST /v3/global/transactions/out-trade-no/{out_trade_no}/reverse
    │
    ├── 撤销成功（HTTP 200）→ 订单已撤销，闭环完成
    │
    └── 撤销失败
          │
          ├── 错误码 = "订单支付中"（USERPAYING / SYSTEMERROR 类，表示用户还在输密码）
          │     │
          │     └── ⚠️ 必须重试撤销
          │           延迟 5-10 秒后再次发起撤销
          │           最多重试 3 次（间隔递增：5s、10s、20s）
          │           直到撤销成功 或 查单显示最终状态（SUCCESS/REVOKED/PAYERROR）
          │
          └── 其他错误码 → 查单确认实际状态，按状态决定后续动作
```

**为什么要重试**：付款码支付下，用户可能正在手机上输密码（USERPAYING），此时撤销请求和用户支付动作"赛跑"。微信侧会在用户密码确认前/后短暂返回"订单支付中"，需要给微信几秒时间内部决断。

**伪代码**：
```java
public void reverseQuickPayOrder(String outTradeNo) {
    int maxRetry = 3;
    long[] delays = {5000, 10000, 20000};

    for (int i = 0; i < maxRetry; i++) {
        try {
            ReverseResponse resp = reverse(outTradeNo, spMchid, subMchid);
            if (resp.isSuccess()) {
                log.info("撤销成功: {}", outTradeNo);
                return;
            }
            // 错误码"订单支付中" → 重试
            if (resp.isOrderInPaying()) {
                log.warn("撤销返回订单支付中，{}ms 后重试 (第 {}/{} 次)",
                         delays[i], i + 1, maxRetry);
                Thread.sleep(delays[i]);
                continue;
            }
            // 其他错误 → 查单兜底
            Order order = queryOrder(outTradeNo);
            handleByState(order);
            return;
        } catch (Exception e) {
            log.error("撤销异常", e);
        }
    }
    // 重试用尽仍未撤销成功，告警 + 加入人工处理队列
    alarmReverseFailed(outTradeNo);
}
```

> ⚠️ **撤销限制**：仅 Quick Pay 支持、仅**当天**可撤销，跨日后必须改走退款流程。

#### 3.5.2 其他支付产品（Native / JSAPI / APP / H5 / 小程序）—— 关单（Close）

```
订单超时 / 状态不明
    │
    ▼
【建议】先查一次单（最后兜底确认）
GET /v3/global/transactions/out-trade-no/{out_trade_no}?sp_mchid&sub_mchid
    │
    ├── trade_state = SUCCESS
    │     │
    │     └── ✅ 订单实际已支付 → 标记订单状态为成功，按 sub_mchid 入账
    │           （不要关单！否则会让用户产生"已扣款但订单关闭"的困扰）
    │
    └── trade_state ∈ {NOTPAY, USERPAYING, 不明}
          │
          ▼
       调用关单接口
       POST /v3/global/transactions/out-trade-no/{out_trade_no}/close
          │
          ├── 关单成功 → 订单关闭，闭环完成
          │
          └── 关单失败
                │
                ├── 错误码 = ORDERPAID
                │     │
                │     └── ✅ 该订单实际已支付成功（被竞态用户支付完了）
                │           → 标记订单状态为成功（同上一分支）
                │           → 不要再尝试关单
                │
                └── 其他错误 → 重试关单（短间隔即可，关单是幂等的）
                      │
                      └── 若前端 POS 状态已不可逆转
                          （如店员已告知顾客"未支付"、商品已下架）
                          → 发起退款补偿用户
```

**为什么先查单再关单**：避免"用户其实已支付成功 → 商户还是关单"的悲剧。先查单是最后一道兜底。

**`ORDERPAID` 关单失败 → 视为支付成功**：这是微信侧的并发保护。当微信几乎同时收到"用户支付成功"和"商户关单"两个请求时，会优先认定支付成功，关单返回 `ORDERPAID`。商户必须把这种情况识别为"支付成功"。

**伪代码**：
```java
public void closeOrCompensate(String outTradeNo) {
    // Step 1: 先查单兜底
    Order order = queryOrder(outTradeNo);
    if ("SUCCESS".equals(order.getTradeState())) {
        markPaid(order);
        return;
    }

    // Step 2: 关单
    try {
        close(outTradeNo, spMchid, subMchid);
        log.info("关单成功: {}", outTradeNo);
        markClosed(outTradeNo);
        return;
    } catch (WechatPayException e) {
        if ("ORDERPAID".equals(e.getErrCode())) {
            // 关单失败 = 实际已支付
            log.info("关单返回 ORDERPAID，订单实际已支付: {}", outTradeNo);
            Order finalOrder = queryOrder(outTradeNo);
            markPaid(finalOrder);
            return;
        }
        // 其他错误 → 短间隔重试关单（最多 2 次）
        retryClose(outTradeNo, 2);
    }

    // Step 3: 若前端状态已不可逆（如店员已告知顾客未支付、商品已下架），
    //        且订单已支付，则发起退款补偿用户
    if (orderActuallyPaid(outTradeNo) && frontendStateIrreversible(outTradeNo)) {
        refund(outTradeNo, "前端状态不可逆，自动退款");
    }
}
```

#### 3.5.3 闭环操作总结对照表

| 维度 | 付款码（Quick Pay） | 其他产品（Native/JSAPI/APP/H5/小程序） |
|------|-------------------|---------------------------------------|
| **闭环手段** | 撤销（Reverse） | 关单（Close） |
| **接口路径** | `POST /v3/global/transactions/out-trade-no/{}/reverse` | `POST /v3/global/transactions/out-trade-no/{}/close` |
| **闭环前是否先查单** | 可选（撤销失败时再查）| ✅ **强烈建议**先查一次单 |
| **失败可重试错误码** | "订单支付中"（用户输密码竞态）→ 重试 | 一般幂等可重试；`ORDERPAID` 不重试 |
| **`ORDERPAID` 含义** | -（撤销路径不会返回）| ⭐ **该订单已支付成功**，标记为成功，不要再关单 |
| **跨日处理** | ⚠️ 不可撤销，改走退款 | 关单不受跨日限制 |
| **前端不可逆兜底** | 重试用尽 → 告警 / 退款 | 前端 POS 状态不可逆 + 实际已支付 → 主动**退款**补偿用户 |

## 4. 查单频率控制

### 4.1 避免过度查单

**问题**：
- 高频查单会触发限流
- 对微信服务器造成压力
- 跨境带宽成本高

**建议**：
| 场景 | 频率 |
|------|------|
| USERPAYING 轮询 | 指数退避：3s, 5s, 10s, 15s, 30s... |
| 回调兜底查单 | 每 5-10 分钟一次批量任务 |
| 用户主动查询 | 限流：同一订单 10 秒内最多 1 次 |

### 4.2 订单终止条件

查单到**最终状态**时**停止查询**，不要继续轮询：
- `SUCCESS` / `REFUND` / `CLOSED` / `REVOKED` / `PAYERROR`

## 5. 查单与回调的配合

### 5.1 数据一致性原则

```
订单状态 = 取最新的最终状态

  回调通知 ← → 查单结果 ← → 同步响应
       │            │              │
       └────────────┴──────────────┘
              幂等合并
```

### 5.2 幂等设计

不管是回调还是查单，处理逻辑都要**幂等**：
```java
// 伪代码
synchronized (orderLock(outTradeNo)) {
    Order order = findOrder(outTradeNo);
    if ("SUCCESS".equals(order.getStatus())) {
        return; // 已处理过，跳过
    }

    // 校验 sp_mchid 是本机构
    if (!notify.getSpMchid().equals(MY_SP_MCHID)) {
        log.warn("Unexpected sp_mchid: {}", notify.getSpMchid());
        return;
    }

    // 校验金额一致
    if (notify.getAmount() != order.getAmount()) {
        alarm("Amount mismatch!");
        return;
    }

    // 更新订单为已支付（按 sub_mchid 入账）
    order.setStatus("SUCCESS");
    order.setTransactionId(transactionId);
    saveOrder(order);

    // 业务逻辑（按 sub_mchid 路由：发货、通知用户等）
    processBusiness(order, order.getSubMchid());
}
```

### 5.3 机构串单防护

⚠️ **机构核心铁律**：每条入账资金必须能精确归属到唯一的子商户。

- 查单返回中**必须**包含 `sub_mchid`
- 兜底处理时按 `sub_mchid` 路由到对应子商户业务上下文
- 不能仅按 `out_trade_no` 处理，否则可能 A 子商户的订单被 B 子商户业务处理

## 6. 异常情况处理

### 6.1 查单也失败怎么办？

```
查单失败
    │
    ├── SYSTEMERROR：重试（3 次，递增间隔）
    │
    ├── RATE_LIMITED：退避更久，再重试
    │
    └── 仍失败：告警 + 加入人工处理队列
```

### 6.2 订单号不存在

- 可能是下单还没到达微信
- 建议：等待 30 秒再查
- 仍不存在：视为下单失败，可重新下单（**新订单号**）

### 6.3 订单金额不一致

- 查单返回金额与机构订单不一致
- **高风险**：可能是重放攻击或数据错误
- 立即告警，暂停处理，人工介入

### 6.4 sub_mchid 不匹配

- 查单返回的 `sub_mchid` 与机构本地订单的 `sub_mchid` 不一致
- 立即告警（可能串单）
- 不要更新订单状态

---

## 7. 最佳实践

1. **所有支付都要有查单兜底**：不能依赖回调
2. **异步回调为主，查单为辅**：回调即时性好，查单覆盖遗漏
3. **指数退避**：避免密集查单
4. **幂等设计**：同一订单多次处理结果一致
5. **最终状态停止查询**：避免无效请求
6. **订单生命周期有界**：设置最大订单有效期（如 2 小时），超期关单
7. **监控告警**：查单失败率、异步回调丢失率都要监控
8. **机构必带 `sub_mchid`**：所有查询和兜底逻辑都要按子商户路由
9. **币种校验**：跨境多币种场景，查单返回的 `amount.currency` 必须与本地订单一致
