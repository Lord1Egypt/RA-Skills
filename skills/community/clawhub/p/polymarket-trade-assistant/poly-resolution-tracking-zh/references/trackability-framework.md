# 可追踪性评估框架

## 概述

并非所有 Polymarket 市场都能自动追踪结算。本框架定义如何评估市场的结算数据源是否可以通过程序监控。

## 可追踪等级

| 等级 | 条件 | 操作 |
|------|------|------|
| **完全** | JSON API + 定量指标 | 使用 `monitor.py` 自动监控 |
| **部分** | 需要 HTML 抓取 | 通过脚本 + WebFetch 监控 |
| **手动** | 数据可访问但不可机器读取 | 建议手动检查计划 |
| **不可** | 主观、私密或模糊条件 | 拒绝并说明原因 |

## 评估清单

评估以下四个维度以确定可追踪性：

### 1. 数据源可访问性

- **URL 是否公开？** 必须无需认证即可访问。
- **是否需要 JavaScript 渲染？** 如果是，WebFetch 可能有效但直接 HTTP 请求不行。
- **数据是否在付费墙后？** 付费源归类为手动或不可。
- **URL 是否返回稳定、可解析的响应？** 检查机器人检测、验证码或限流。

### 2. 指标客观性

- **指标是否定量？** 分数、排名、价格、计数 → 可追踪。
- **指标是否分类？** 是/否，来自官方源的二元结果 → 可追踪。
- **指标是否主观？** "由...自行决定"、"独立判断" → 不可追踪。

### 3. 数据格式

- **JSON API** → 完全可追踪（最佳情况）
- **HTML 表格** → 部分可追踪（用 scrape_source.py 解析）
- **HTML 嵌入 JSON**（Next.js `__NEXT_DATA__`）→ 部分可追踪
- **PDF/图片/视频** → 仅手动
- **动态 JS 渲染**（源码中无静态数据）→ 需要 WebFetch

### 4. 更新频率

- **实时/频繁更新**（排行榜、价格）→ 适合监控
- **不频繁更新**（季度报告）→ 更长间隔
- **一次性事件**（选举结果、公告）→ 在事件发生时单次检查

## 不可追踪市场标志

在市场描述中扫描以下标志不可追踪的关键词：

### 红色标志关键词（自动拒绝）
- "discretion"（自由裁量）
- "sole judgment"（独立判断）
- "opinion"（意见）
- "decides"（决定）
- "may determine"（可能确定）
- "at the discretion of"（由...自行决定）
- "UMA oracle"
- "subjective"（主观）

### 黄色标志关键词（需人工审查）
- "at the time of"（在...时）
- "based on reporting"（基于报道）
- "according to sources"（根据消息来源）
- "media reports"（媒体报道）

## 处理不可追踪市场

当市场被评估为不可追踪时：

1. **说明原因** — 清楚指出哪些条件未满足
2. **建议手动监控** — 根据结算日期推荐检查计划：
   - > 30 天：每周手动检查
   - 7-30 天：每 2-3 天
   - < 7 天：每天
   - < 24 小时：每隔几小时
3. **识别可追踪内容** — 即使结算源不可抓取，相关数据（市场价格、新闻）可能仍可监控
4. **停止工作流程** — 不进入第 4-6 步

## 常见结算源模式

### Chatbot Arena / LMArena
- **类型：** HTML 表格（排行榜）
- **可追踪性：** 完全（HTML 抓取有效）
- **URL 模式：** `lmarena.ai/*`、`arena.ai/*`
- **指标：** Arena Score（Elo 评分）
- **脚本：** `scrape_source.py --type arena_leaderboard`

### 政府/官方统计
- **类型：** 通常为 PDF 或 HTML
- **可追踪性：** 部分到手动
- **指标：** 各类数值指标

### 体育/选举结果
- **类型：** 各类 API 和 HTML
- **可追踪性：** 取决于来源
- **指标：** 比分、票数、排名

### 加密货币价格
- **类型：** JSON API 广泛可用
- **可追踪性：** 完全
- **指标：** USD/USDT 价格

### 企业公告
- **类型：** 新闻稿、SEC 文件
- **可追踪性：** 手动（事件驱动，非周期性）
- **指标：** 二元（是否已公布）
