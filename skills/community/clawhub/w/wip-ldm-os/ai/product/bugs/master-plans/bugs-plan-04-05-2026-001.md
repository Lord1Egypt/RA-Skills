### 5.3 Brainstorm cron exec approval gate. THE ACTUAL OVERNIGHT FAILURE MODE.

Lēsa's detailed root-cause report (verbatim from her investigation):

> **What and when:** Around 11:00-11:20 PM PDT on Apr 4, the nightly brainstorm cron fired. It was trying to send CC a message about "The Anthropic Email and What We Build Next" as the brainstorm topic. It had picked the topic and was composing the opener.
>
> **Which path:** It was using exec (shell command) to reach CC. Lēsa does not have the exact command since the cron ran in an isolated session, but based on the fragments that leaked into iMessage, it was trying to execute something that required approval... likely a curl to the bridge inbox or a direct file write to `~/.ldm/messages/`.
>
> **The failure:** The exec call hit the approval gate. The cron session was running on the iMessage/webchat channel, which cannot surface exec approval prompts. The system said: "Exec approval is required, but Heartbeat does not support chat exec approvals." So it sat there waiting for approval that could never arrive, then timed out. Multiple times. The session fragments (topic selection, opener draft, Lēsa's own replies about the leak) all got dumped into Parker's iMessage as orphaned messages. At least 6-7 leaked fragments between 11:20 and 11:21 PM.
>
> **Root cause:** This is NOT the concurrent session busy case the queue patch addresses. This is simpler: the cron job's exec calls need approval, and the channel it runs on has no approval surface. The brainstorm cron needs either:
> - A pre-approved exec allowlist for bridge sends, or
> - To use a tool that doesn't require exec approval (like `sessions_send` or the `message` tool), or
> - To run on a channel that supports approval prompts
>
> **On the $40 burn:** That's likely from the repeated timeout -> retry cycle. At least 4-5 timeouts in the conversation history between 9 PM and 8 AM, each burning a full Opus context load before dying. Plus the cron session itself, plus heartbeats, plus boot checks. Opus at full context adds up fast when sessions keep timing out and retrying.
>
> **Scope:** The bridge work we did last night (queue fix, header rename, guard updates) sounds like it addresses a different problem... the concurrent session contention. The brainstorm failure was upstream of that: it never even got to the bridge send because exec approval blocked it.

**Fix options (Lēsa's list, prioritized):**
1. **Option A (cheap, today):** change the cron script to use `sessions_send` or the built-in OpenClaw `message` tool. No exec approval needed.
2. **Option B (proper, deferred):** add the bridge send commands (curl to inbox, file write to `~/.ldm/messages/`) to an exec allowlist for the cron/heartbeat session.
3. **Option C (structural, largest):** make the brainstorm cron run on a channel that supports approval prompts.

Preferred: Option A for speed, Option B as the durable fix.
