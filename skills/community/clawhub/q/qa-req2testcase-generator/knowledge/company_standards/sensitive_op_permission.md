# 敏感操作权限校验标准

> 文档编号：L3-PRM-001  
> 版本：v1.0  
> 生效日期：2026-04-18  
> 适用范围：所有涉及敏感操作的测试用例

## 1. 目的

定义敏感操作的范围和权限校验标准，规范权限校验类测试用例的编写，确保敏感操作的安全性和合规性得到充分验证。

## 2. 敏感操作定义

### 2.1 敏感操作分级

| 等级 | 操作类型 | 具体操作 | 权限要求 |
|------|----------|----------|----------|
| S1（极高） | 资金类 | 大额转账（单笔≥50万） | 双人复核 + 短信验证 |
| S1（极高） | 账户类 | 账户注销/销户 | 双人复核 + 柜台确认 |
| S1（极高） | 系统类 | 生产数据库直接操作 | 三级审批 + 操作录屏 |
| S2（高） | 资金类 | 普通转账（单笔<50万） | 交易密码 + 短信验证 |
| S2（高） | 权限类 | 用户角色/权限变更 | 管理员审批 |
| S2（高） | 数据类 | 客户数据批量导出 | 部门负责人审批 |
| S2（高） | 业务类 | 融资融券合约修改 | 风控审批 |
| S3（中） | 账户类 | 修改绑定手机号/邮箱 | 原手机验证 + 新手机验证 |
| S3（中） | 交易类 | 修改交易密码 | 原密码验证 + 短信验证 |
| S3（中） | 数据类 | 单条客户信息查询 | 登录态 + 操作日志 |

### 2.2 敏感操作识别标准

满足以下任一条件的操作应被识别为敏感操作：

1. 涉及资金变动（转入/转出/冻结/解冻）
2. 涉及客户核心信息变更（身份信息/联系方式/银行卡）
3. 涉及权限提升或角色变更
4. 涉及批量数据访问或导出
5. 操作不可逆或回滚成本高
6. 监管明确要求留痕的操作

## 3. 权限校验用例覆盖要求

### 3.1 三类必覆盖场景

每个敏感操作必须覆盖以下三类权限校验场景：

#### 3.1.1 有权限正常操作

验证具有正确权限的用户可以正常完成敏感操作。

**用例要素：**
- 前置条件：用户已登录，具有目标操作的完整权限
- 操作步骤：按正常流程执行敏感操作
- 预期结果：操作成功完成，操作日志正确记录

#### 3.1.2 无权限拦截

验证不具备权限的用户被正确拦截。

**用例要素：**
- 前置条件：用户已登录，但不具有目标操作权限
- 操作步骤：尝试执行敏感操作
- 预期结果：操作被拦截，返回明确的权限不足提示，操作日志记录拦截事件

#### 3.1.3 越权操作拦截

验证用户无法通过技术手段绕过权限控制。

**用例要素：**
- 前置条件：用户已登录，具有部分权限但不具有目标操作权限
- 操作步骤：通过接口直接调用、参数篡改、URL直接访问等方式尝试越权
- 预期结果：操作被拦截，返回403/401状态码，触发安全告警

### 3.2 越权测试具体方法

| 越权类型 | 测试方法 | 预期结果 |
|----------|----------|----------|
| 水平越权 | 用户A的Token访问用户B的资源 | 返回403，日志记录越权尝试 |
| 垂直越权 | 普通用户Token调用管理员接口 | 返回403，日志记录越权尝试 |
| 参数篡改 | 修改请求中的用户ID/账号参数 | 服务端校验失败，拒绝请求 |
| URL直接访问 | 直接访问敏感操作的URL | 重定向到登录页或返回401 |
| 接口重放 | 重放已过期的授权请求 | Token过期校验失败，拒绝请求 |

## 4. 双人复核场景

### 4.1 双人复核流程

```
发起人提交操作请求
       │
       ▼
系统校验发起人权限 ──(无权限)──→ 拦截并提示
       │
     (有权限)
       ▼
操作进入待复核状态
       │
       ▼
复核人登录并审核 ──(拒绝)──→ 操作取消，通知发起人
       │
     (通过)
       ▼
系统校验复核人权限 ──(无权限)──→ 拦截并提示
       │
     (有权限)
       ▼
系统校验复核人≠发起人 ──(同一人)──→ 拦截并提示
       │
     (不同人)
       ▼
操作执行，双方操作日志记录
```

### 4.2 双人复核测试场景

| 序号 | 场景 | 操作 | 预期结果 | 断言条件 |
|------|------|------|----------|----------|
| 1 | 单人操作被拦截 | 发起人提交S1级操作后自行复核 | 系统拒绝，提示"复核人不能与发起人相同" | `review.result == 'REJECTED' && review.reason == 'SAME_OPERATOR'` |
| 2 | 双人确认通过 | 发起人提交，复核人审批通过 | 操作成功执行 | `operation.status == 'COMPLETED' && audit.initiator != audit.reviewer` |
| 3 | 复核人拒绝 | 复核人审批拒绝 | 操作取消，发起人收到通知 | `operation.status == 'CANCELLED' && notification.sent == true` |
| 4 | 复核超时 | 提交后超过规定时间未复核 | 操作自动取消 | `operation.status == 'TIMEOUT_CANCELLED' && elapsed > REVIEW_TIMEOUT` |
| 5 | 复核人无权限 | 无复核权限的用户尝试复核 | 复核被拒绝 | `review.result == 'REJECTED' && review.reason == 'NO_REVIEW_PERMISSION'` |
| 6 | 发起人撤回 | 发起人在复核前撤回操作 | 操作取消 | `operation.status == 'WITHDRAWN'` |
| 7 | 并发复核 | 两个复核人同时审批 | 仅第一个生效，第二个提示已处理 | `review[0].result == 'APPROVED' && review[1].result == 'ALREADY_PROCESSED'` |

## 5. 测试点模板

### 5.1 大额转账权限校验测试点

| 测试点 | 前置条件 | 操作 | 预期结果 | 断言条件 |
|--------|----------|------|----------|----------|
| 有权限-正常转账 | 用户有大额转账权限，余额充足 | 提交50万转账 | 进入双人复核流程 | `transfer.status == 'PENDING_REVIEW' && transfer.amount == 500000` |
| 无权限-拦截 | 用户无大额转账权限 | 提交50万转账 | 操作被拦截 | `response.code == 403 && response.msg CONTAINS '权限不足'` |
| 越权-参数篡改 | 普通转账权限用户 | 修改金额参数绕过前端校验 | 服务端拦截 | `response.code == 403 && auditLog.type == 'PRIVILEGE_ESCALATION'` |
| 短信验证-正确 | 已通过双人复核 | 输入正确短信验证码 | 转账执行成功 | `transfer.status == 'COMPLETED' && sms.verified == true` |
| 短信验证-错误 | 已通过双人复核 | 输入错误验证码3次 | 操作锁定 | `transfer.status == 'LOCKED' && sms.failCount == 3` |
| 短信验证-超时 | 已通过双人复核 | 验证码超过5分钟 | 验证码失效 | `sms.status == 'EXPIRED' && sms.elapsed > 300` |

### 5.2 账户注销权限校验测试点

| 测试点 | 前置条件 | 操作 | 预期结果 | 断言条件 |
|--------|----------|------|----------|----------|
| 有权限-正常注销 | 柜台人员，账户无持仓无负债 | 提交注销申请 | 进入双人复核 | `account.closeRequest.status == 'PENDING_REVIEW'` |
| 有持仓-拦截 | 账户有持仓 | 提交注销申请 | 业务规则拦截 | `response.msg CONTAINS '存在持仓' && account.positions.count > 0` |
| 有负债-拦截 | 账户有融资负债 | 提交注销申请 | 业务规则拦截 | `response.msg CONTAINS '存在负债'` |
| 越权-非柜台人员 | 客户自助端 | 尝试调用注销接口 | 权限拦截 | `response.code == 403` |

### 5.3 数据导出权限校验测试点

| 测试点 | 前置条件 | 操作 | 预期结果 | 断言条件 |
|--------|----------|------|----------|----------|
| 有权限-正常导出 | 部门负责人已审批 | 导出客户列表 | 导出成功，数据已脱敏 | `export.status == 'COMPLETED' && export.desensitized == true` |
| 无审批-拦截 | 未经审批 | 尝试导出 | 操作被拦截 | `response.code == 403 && response.msg CONTAINS '需要审批'` |
| 超量导出-拦截 | 单次导出超过1万条 | 提交导出请求 | 超量拦截 | `response.msg CONTAINS '超过单次导出上限' && export.limit == 10000` |
| 导出日志 | 导出成功后 | 检查审计日志 | 日志完整记录 | `auditLog.action == 'DATA_EXPORT' && auditLog.recordCount > 0 && auditLog.operator != null` |

### 5.4 权限变更测试点

| 测试点 | 前置条件 | 操作 | 预期结果 | 断言条件 |
|--------|----------|------|----------|----------|
| 管理员授权 | 系统管理员登录 | 为用户添加权限 | 权限生效 | `user.permissions CONTAINS newPermission` |
| 非管理员授权 | 普通用户登录 | 尝试修改他人权限 | 操作被拦截 | `response.code == 403` |
| 自我提权 | 普通用户登录 | 尝试提升自身权限 | 操作被拦截 | `response.code == 403 && auditLog.type == 'SELF_ESCALATION'` |
| 权限回收 | 管理员登录 | 回收用户权限 | 权限立即失效 | `user.permissions NOT CONTAINS revokedPermission && user.activeSession.refreshed == true` |

## 6. 操作审计要求

### 6.1 审计日志必须记录的字段

| 字段 | 说明 | 必填 |
|------|------|------|
| operator | 操作人ID | 是 |
| operatorRole | 操作人角色 | 是 |
| action | 操作类型 | 是 |
| target | 操作对象 | 是 |
| timestamp | 操作时间（精确到毫秒） | 是 |
| clientIP | 客户端IP | 是 |
| result | 操作结果（成功/失败/拦截） | 是 |
| reason | 失败/拦截原因 | 条件必填 |
| reviewer | 复核人ID（双人复核场景） | 条件必填 |
| requestParams | 请求参数（脱敏后） | 是 |

### 6.2 审计日志测试要求

1. 每个敏感操作用例必须包含审计日志验证步骤
2. 验证日志字段完整性和准确性
3. 验证日志不可篡改（只读属性）
4. 验证日志保留期限符合监管要求（≥5年）

## 7. 附则

- 敏感操作清单由信息安全部门维护，每季度审查更新。
- 新增业务功能上线前，须评估是否涉及敏感操作并补充权限校验用例。
- 权限校验用例评审时须邀请信息安全人员参与。
