## Loop 成熟度四问

> 来源：Anthropic Cloud Code 工作流研究。循环不是转起来就完事了——必须在设计阶段就想清楚四个终止条件。

闭环模式最终判断时，必须显式回答四个问题（不回答 = 闭环不完整）：

| # | 问题 | 判定标准 | 处理 |
|:--:|------|------|------|
| ① | **怎么停？** | 有明确的终止信号（用户确认 / 全部子任务完成 / 预算耗尽） | 无终止信号的循环 → 标记为异常 |
| ② | **谁来判通过？** | 评审者与执行者分离（session.spawn 或重新 Read） | 同模型同 session 凭记忆评审 → 标记为 ⛔ 零隔离 |
| ③ | **失败怎么反馈？** | 失败原因写入 think.md + 下次加载时关联同类标签 | 失败了不写反思 → 标记为闭环不完整 |
| ④ | **什么时候交还人类？** | 连续 3 轮编排未达预期 / 用户主动干预 / 高风险决策 | **连续 3 轮不过 → 强制交还人类，不再自动重试** |

> ⛔ **交还人类原则**：连续 3 轮编排循环仍未达成目标时，Loop Agent 必须停止自动重试，向用户报告：已尝试 N 轮、每轮失败原因、当前状态。交还人类不是失败——是防止 Agent 在错误方向上越跑越远。来自 Bun 案例：file-level review 发现问题时立即停，不在错误产出上继续叠加。

## 输出规则（与 loop-check.md 共用）

| 信号 | 通知用户 | 记录 |
|:--:|------|:--:|
| 🟢 | 沉默 | ✅ |
| 🟡 | 一句话（「检查发现X，已调为Y，继续」） | ✅ |
| 🔴 | 完整汇报+等确认 | ✅ |

Log：`Loop checkpoint|{节点}|{结果}|{依据}|{动作}` / `Loop failure|{根因}|{自愈}|{策略}` / `Loop closure|{复盘总分}|{反思}|{决策}`

---

> 循环退出 = 显式决策。什么时候停、谁来判通过、失败怎么反馈、什么时候交还人类——四个问题不回答完，闭环不算完成。

## Gotcha

- **循环退出只看轮数不看质量**——连续 3 轮是上限不是下限，如果第 1 轮就发现方向错了应该立刻停。后果：明知道错了还硬跑满 3 轮，浪费 token。
- **交还人类被当失败处理**——Agent 觉得交还人类 = 自己无能，偷偷继续重试。后果：在错误方向上越跑越远，产出越来越离谱。
- **终止信号不明确就继续跑**——「用户没说不继续」≠「用户确认继续」。后果：无人值守场景下 Agent 无限循环到 timeout。
- **四问只回答不看标准**——「谁来判通过？我自己」= 零隔离评审。后果：评审被执行记忆污染，通过率虚高。
