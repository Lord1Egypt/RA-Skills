# Coze+IMA协同尽调流程

> 本文档是技术尽调Skill的核心子流程规范，定义Coze与IMA协同完成尽调的标准方法。

---

## 一、模式选择判断

### 1.1 执行路径决策树

```
输入：项目信息 + 可用资源

判断1：IMA知识库是否有该项目资料？
    ├─ 是 ──────────────────────→ 协同模式
    │
    └─ 否
        判断2：客户是否提供原始文档（BP/技术文档/专利文件）？
            ├─ 是 ──→ 用import_doc导入IMA → 协同模式
            │
            └─ 否 ──────────────────────→ 纯Coze模式
```

### 1.2 两种模式对比

| 维度 | 纯Coze模式 | Coze+IMA协同模式 |
|------|------------|------------------|
| **适用场景** | 无IMA资料、快速初筛 | 有IMA资料或客户文档 |
| **目标时长** | 15-20分钟 | 25-30分钟 |
| **验证覆盖率** | 依赖搜索，无结构化 | 32项问题65%+已验证 |
| **信息追溯** | 弱 | 强（每项结论有来源） |
| **报告深度** | 初筛级 | 可用于决策参考 |

### 1.3 模式切换条件

- **升级触发**：纯Coze模式发现高风险信号 → 建议客户导入文档 → 切换协同模式深度核实
- **降级触发**：协同模式客户无新文档且IMA资料完整 → 可简化为纯Coze模式快速输出

## 二、Coze+IMA协同流程（4阶段标准流程）

### 阶段1：IMA信息底座搭建

**时长**：3-5分钟  
**目标**：建立可检索的项目资料库

> ⚠️ **重要更新（2026-05-10）**：IMA API现已支持Wiki API（知识库操作），搜索范围从个人笔记扩展到知识库内容。

#### 1.1 Wiki知识库检索（优先）

使用 Wiki API 搜索知识库元数据：

```
API: search_knowledge_base
请求体: {"query": "关键词组合", "limit": 20, "cursor": ""}
返回: data.info_list 含 kb_id, kb_name, content_count, base_type
```

获取到 kb_id 后，搜索知识库内的具体内容：

```
API: search_knowledge
请求体: {"knowledgeBaseId": "kb_id", "query": "搜索词", "limit": 20}
返回: data.info_list 含 media_id, title, highlight_content
```

#### 1.1.1 已知知识库ID（快速路径）

部分知识库ID已预知，可直接调用：

| 知识库名称 | kb_id |
|-----------|-------|
| 案例库 | vnvRiO... |
| OPC运营官 | pTNyuJ... |
| 韩国吴会长 | 06KNK8... |
| 小白说 | Ii2AWP... |
| BNCT | bySgmn... |

```
# 直接搜索指定知识库
result = client.search_knowledge("vnvRiO...", "项目关键词")
```

#### 1.1.2 搜索结果解析

```
返回字段说明：
- kb_id / media_id: 知识库/文档唯一标识
- title: 文档标题
- highlight_content: 包含关键词的片段（可直接引用）
- parent_folder_id: 所属文件夹
- media_type: 文档类型
```

> ⚠️ **已知限制**：`get_knowledge` 读取正文返回404，但可通过 `search_knowledge` 的 `highlight_content` 获取片段。如需遍历文件结构，使用 `get_knowledge_list`。

#### 1.2 遍历知识库文件结构

如需查看知识库中的完整文件列表：

```
API: get_knowledge_list
请求体: {"knowledgeBaseId": "kb_id", "folderId": "", "limit": 20}
返回: data.knowledge_list 含 media_id, title, media_type, tags
      data.current_path, data.is_end, data.next_cursor
```

#### 1.3 Note个人笔记检索（补充）

仍可使用 Note API 搜索个人笔记：

```
API: search_note_book
请求体: {"query_info": {"title": "关键词"}, "start": 0, "end": 20}
```

**关键词策略**（优先级递减）：
1. `项目名` + `核心技术关键词`
2. `公司名` + `产品名`
3. `创始人/核心团队成员` 全名
4. `专利号` + `技术领域`

#### 1.4 新资料导入

如客户提供了新文档，使用 `import_doc` API：

```
API: import_doc
参数: 
  - file: 文档文件（PDF/DOCX/TXT）
  - title: 文档标题（建议格式：项目名_资料类型_日期）
```

**导入优先级**：
1. 商业计划书（BP）
2. 技术白皮书/产品规格书
3. 专利文件
4. 团队介绍/简历
5. 财务数据/测试报告

#### 1.5 阶段1输出

```
输出清单：
□ 可检索的IMA知识库已就绪（Wiki + Note）
□ 相关资料清单（含kb_id、media_id）
□ 核心资料已导入（如有新文档）
□ 关键词：["项目名", "核心技术", "公司名", ...]
□ 已尝试的知识库列表
```

### 阶段2：Coze尽调框架搭建

**时长**：8-10分钟  
**目标**：生成结构化的问题清单

> ⚠️ 此阶段由Coze独立完成，不调用IMA

#### 2.1 四层分析法框架

基于技术尽调Skill的四层金字塔：

| 层级 | 分析维度 | 核心问题类型 |
|------|----------|--------------|
| 第一层 | 材料完整性 | 可查证性、信息一致性 |
| 第二层 | 技术可行性 | 物理定律、TRL等级、学术基础 |
| 第三层 | 商业逻辑 | 商业模式、竞争格局、估值匹配 |
| 第四层 | 风险量化 | 技术/市场/团队/财务/法律风险 |

#### 2.2 5大维度32项核心问题清单

##### 维度A：材料完整性（6项）

| # | 问题 | 验证要点 | 预期信息源 |
|---|------|----------|------------|
| A1 | 项目备案/代号是否可查？ | 备案渠道权威性 | 政府公示平台 |
| A2 | 核心团队成员履历是否可验证？ | 学历/职历一致性 | LinkedIn/学信网 |
| A3 | 专利号是否真实？ | 专利局核查 | 国家知识产权局 |
| A4 | 技术参数是否有原始数据支撑？ | 测试报告/第三方认证 | 原始文档 |
| A5 | 核心论文是否发表在可信期刊？ | 同行评审/引用关系 | Google Scholar |
| A6 | 工商信息与BP描述是否一致？ | 股权结构/注册资本 | 天眼查/企查查 |

##### 维度B：技术可行性（7项）

| # | 问题 | 验证要点 | 预期信息源 |
|---|------|----------|------------|
| B1 | 技术参数是否违反物理定律？ | 对标物理极限 | 行业标准 |
| B2 | 声称的TRL等级是否有证据支撑？ | 原型/测试/量产 | 实地考察/视频 |
| B3 | 核心技术的学术基础是否扎实？ | 高被引论文/顶刊 | 学术数据库 |
| B4 | 技术壁垒是否足够？ | 专利布局/know-how | 专利分析 |
| B5 | 核心技术是否自主可控？ | 知识产权归属 | 专利文件 |
| B6 | 关键材料/设备是否依赖进口？ | 供应链安全 | 供应商清单 |
| B7 | 是否存在技术路线替代风险？ | 竞品技术对比 | 行业报告 |

##### 维度C：商业逻辑（7项）

| # | 问题 | 验证要点 | 预期信息源 |
|---|------|----------|------------|
| C1 | 价值主张是否清晰可量化？ | 痛点-解决方案匹配 | BP内容 |
| C2 | 目标客户是否明确？ | 客户画像 | 市场调研 |
| C3 | 商业模式是否闭环？ | 收入-成本结构 | 财务预测 |
| C4 | 竞争对手有哪些？ | 市场份额 | 行业报告 |
| C5 | 相对竞争优势是什么？ | 差异化分析 | 竞品对比 |
| C6 | 市场规模是否支撑估值？ | TAM/SAM/SOM | 市场报告 |
| C7 | 商业化路径是否可行？ | 销售策略/渠道 | 运营计划 |

##### 维度D：团队评估（6项）

| # | 问题 | 验证要点 | 预期信息源 |
|---|------|----------|------------|
| D1 | 创始人背景是否匹配项目方向？ | 行业经验/资源 | 个人履历 |
| D2 | 核心团队是否完整？ | 技术/市场/运营 | 团队介绍 |
| D3 | 团队稳定性如何？ | 股权结构/激励机制 | 工商信息 |
| D4 | 是否有行业顶尖人才？ | 专家背景/荣誉 | 公开报道 |
| D5 | 顾问团队是否真实？ | 顾问身份核实 | 顾问本人确认 |
| D6 | 团队执行力记录如何？ | 历史项目/里程碑 | 新闻报道 |

##### 维度E：风险评估（6项）

| # | 问题 | 验证要点 | 预期信息源 |
|---|------|----------|------------|
| E1 | 技术风险等级？ | 可行性/成熟度 | 综合判断 |
| E2 | 市场风险等级？ | 需求/竞争/时机 | 市场数据 |
| E3 | 团队风险等级？ | 稳定性/能力 | 背景核查 |
| E4 | 财务风险等级？ | 资金需求/烧钱率 | 财务数据 |
| E5 | 法律风险等级？ | IP/合规/监管 | 法律文件 |
| E6 | 综合风险评级？ | 多维度加权 | 综合判断 |

#### 2.3 阶段2输出

```
输出清单：
□ 32项问题清单（含验证要点）
□ 每项问题的预期信息源
□ 问题优先级排序（高/中/低）
□ 需客户补充的资料清单
```

### 阶段3：IMA信息验证

**时长**：8-10分钟  
**目标**：对每个问题进行验证，标注可信度

#### 3.1 验证流程

对32项问题逐一执行：

```
For each 问题 in 问题清单:
    1. 用 search_knowledge_base 搜索全局知识库
    2. 用 search_knowledge 在指定知识库内搜索（如已知kb_id）
    3. 如有结果 → 提取 highlight_content 片段
    4. 如需遍历结构 → 用 get_knowledge_list
    5. 如无结果 → 尝试 search_note_book（Note API）
    6. 如仍无结果 → 用 search_web 网络搜索
    7. 标注可信度
    8. 如需补充 → 记录补充调研方向
```

#### 3.2 可信度标注规范

| 标注 | 定义 | 颜色代码 | 处理方式 |
|------|------|----------|----------|
| 🟢 已验证 | IMA知识库或二次搜索确认 | 绿色 | 可直接引用 |
| 🟡 部分验证 | 存在矛盾或信息不足 | 黄色 | 标注矛盾点+补充方向 |
| 🔴 未验证 | 无相关资料 | 红色 | 标注"待核实"+建议行动 |
| 🌐 Web补充 | IMA无结果，用网络搜索 | 蓝色 | 标注"非IMA验证" |

#### 3.3 矛盾处理

如IMA资料与BP描述存在矛盾：
1. 记录矛盾点（原文 vs 核查结果）
2. 标注矛盾严重程度（轻微/重大）
3. 建议核实方式（联系项目方/第三方验证）

#### 3.4 阶段3输出

```
输出清单：
□ 32项问题验证结果表
    - 问题编号
    - 验证状态（🟢/🟡/🔴/🌐）
    - 验证依据（来源+原文引用）
    - 补充方向（如需）
□ 矛盾点清单（如有）
□ 验证覆盖率统计
□ 待客户补充资料清单
```

### 阶段4：Coze方案生成

**时长**：3-5分钟  
**目标**：整合验证结果，生成完整报告

#### 4.1 报告结构

```
尽调报告结构：
1. 执行摘要
   - 项目概况
   - 核心结论（风险评级）
   - 主要红旗信号

2. 材料完整性评估
   - 可查证项汇总
   - 红旗信号清单

3. 技术可行性分析
   - 物理定律核查结论
   - TRL评估
   - 学术基础评估

4. 商业逻辑评估
   - 商业模式画布分析
   - 竞争格局
   - 估值匹配

5. 风险评估
   - 各维度风险矩阵
   - 综合评级（🟢/🟡/🟠/🔴）

6. 投资建议
   - 核心结论
   - 建议行动
   - 继续核实清单

7. 附录
   - 验证结果详情
   - 资料来源清单
```

#### 4.2 渠道方案同步生成

如涉及渠道合作（如韩国市场），同步生成：
- 渠道适配性分析
- 渠道方案建议
- 渠道风险提示

#### 4.3 Skill能力短板识别

报告末尾标注：
```
本报告Skill能力范围内可验证项：XX项（XX%）
超出Skill能力需人工介入项：XX项
建议后续人工尽调重点：
1. ...
2. ...
```

#### 4.4 报告写回IMA

使用 `append_doc` API 将最终报告写入IMA知识库：

```
API: append_doc
参数:
  - doc_id: 目标文档ID
  - content: 报告内容（Markdown格式）
```

**建议命名格式**：
```
尽调报告_项目名_YYYYMMDD
```

#### 4.5 阶段4输出

```
输出清单：
□ 完整尽调报告（Markdown）
□ 渠道方案（如适用）
□ Skill能力评估
□ 报告已写入IMA知识库
```

## 三、IMA API调用规范

### 3.1 认证信息

**来源**：`SECRET.md` 或 环境变量

```
ima-openapi-clientid: {your_client_id}
ima-openapi-apikey: {your_api_key}
```

> ⚠️ **关键注意**：API Key ≠ Client Secret  
> - Client Secret：较短的密钥字符串  
> - API Key：**较长的那个**，用于API调用认证

### 3.2 5个核心API

#### API 1：搜索知识库

```
POST /v1/search_note_book
Headers:
  Content-Type: application/json
  ima-openapi-clientid: {clientid}
  ima-openapi-apikey: {apikey}

Request Body:
{
  "query_info": {
    "title": "关键词",
    "offset": 0,
    "limit": 10
  }
}

Response:
{
  "data": {
    "docs": [
      {
        "doc_id": "xxx",
        "title": "文档标题",
        "snippet": "匹配片段",
        "update_time": 1234567890
      }
    ],
    "total": 100,
    "has_more": true,
    "next_cursor": "cursor_token"
  }
}
```

#### API 2：获取文档内容

```
GET /v1/get_doc_content?doc_id={doc_id}
Headers:
  ima-openapi-clientid: {clientid}
  ima-openapi-apikey: {apikey}

Response:
{
  "data": {
    "doc_id": "xxx",
    "title": "文档标题",
    "content": "文档全文（Markdown格式）",
    "create_time": 1234567890,
    "update_time": 1234567890
  }
}
```

#### API 3：列出笔记

```
POST /v1/list_note_by_folder_id
Headers:
  Content-Type: application/json
  ima-openapi-clientid: {clientid}
  ima-openapi-apikey: {apikey}

Request Body:
{
  "folder_id": "root",  // 或指定folder_id
  "cursor": "",  // 分页游标
  "size": 20
}

Response:
{
  "data": {
    "notes": [
      {
        "note_id": "xxx",
        "title": "笔记标题",
        "create_time": 1234567890
      }
    ],
    "has_more": true,
    "next_cursor": "cursor_token"
  }
}
```

#### API 4：导入外部文档

```
POST /v1/import_doc
Headers:
  ima-openapi-clientid: {clientid}
  ima-openapi-apikey: {apikey}
Content-Type: multipart/form-data

Form Data:
  - file: 文件二进制（PDF/DOCX/TXT，最大50MB）
  - title: 文档标题（必填）

Response:
{
  "code": 0,
  "msg": "success",
  "data": {
    "doc_id": "xxx",
    "title": "文档标题"
  }
}
```

#### API 5：追加文档内容

```
POST /v1/append_doc
Headers:
  Content-Type: application/json
  ima-openapi-clientid: {clientid}
  ima-openapi-apikey: {apikey}

Request Body:
{
  "doc_id": "目标文档ID",
  "content": "追加的内容（Markdown格式）"
}

Response:
{
  "code": 0,
  "msg": "success",
  "data": {
    "doc_id": "xxx",
    "word_count": 1234
  }
}
```

### 3.3 常见错误处理

| 错误码 | 含义 | 处理方式 |
|--------|------|----------|
| 401 | 认证失败 | 检查clientid和apikey是否正确 |
| 403 | 权限不足 | 确认API Key有对应权限 |
| 404 | 文档不存在 | 检查doc_id是否正确 |
| 413 | 文件过大 | 分拆文件或压缩后重试 |
| 429 | 请求频率超限 | 降低调用频率，等待后重试 |

## 四、效率基准与实测数据

### 4.1 时间分配基准

| 阶段 | 目标时长 | 允许波动 | 关键里程碑 |
|------|----------|----------|------------|
| 阶段1：IMA底座搭建 | 3-5分钟 | ±1分钟 | 知识库就绪 |
| 阶段2：Coze框架搭建 | 8-10分钟 | ±2分钟 | 32项问题清单完成 |
| 阶段3：IMA验证 | 8-10分钟 | ±2分钟 | 验证覆盖率>60% |
| 阶段4：Coze生成 | 3-5分钟 | ±1分钟 | 报告完成并写回IMA |
| **总计** | **25-30分钟** | ±5分钟 | 完整尽调报告 |

### 4.2 质量基准

| 指标 | 目标值 | 最低容忍 |
|------|--------|----------|
| 问题覆盖率 | 32项 | 28项（87.5%） |
| 验证覆盖率 | ≥60% | ≥50% |
| 报告完整性 | 7个章节 | 5个章节 |
| 红旗信号识别率 | 100% | 80% |

### 4.3 实测案例

#### 案例：韩国GMF项目

| 指标 | 实测值 |
|------|--------|
| 总时长 | 28分钟 |
| 问题总数 | 32项 |
| 已验证（🟢） | 21项（65.6%） |
| 部分验证（🟡） | 8项（25%） |
| 未验证（🔴） | 3项（9.4%） |
| 发现红旗信号 | 2项 |
| 综合评级 | 🟡 中风险 |

#### 效率分析

## 五、迭代建议

### 5.1 Skill能力持续优化

每完成一次尽调，记录：
1. 验证覆盖率是否达标？
2. 哪些问题类型最难验证？
3. 是否有新的验证渠道发现？
4. 报告结构是否需要调整？

### 5.2 IMA知识库建设

建议维护以下知识库：
- **企业数据库**：已尽调企业的完整资料
- **行业参数库**：各技术领域的物理极限/行业标准
- **竞品档案库**：主要竞争对手的核心信息
- **案例库**：典型红旗信号案例

### 5.3 流程标准化

对于高频场景：
1. 建立该行业的专项问题清单模板
2. 定义该领域的典型红旗信号库
3. 预置该行业的验证信息源清单
