# rag：RAG 知识库检索

知识库管理页面在https://www.siluzan.com/knowledge-base/

用于写稿、产品知识问答、三库素材提取等场景；凡回答依赖「本企业已入库素材」且不应凭模型记忆编造时，优先走 RAG。

## 业界检索共识与本 CLI 的对应关系（为何要靠「提示词」补效果）

生产级 RAG 常见组合是：**合理分块** + **混合检索（关键词 BM25 + 向量）** + **重排序**，并对用户问句做 **查询改写 / 多查询扩展 / 拆解**。本 CLI 调用的是平台侧 **向量相似度检索**，AI 无法改分块、无法开关 BM25 或重排。要在当前架构下提高 **召回（命中率）** 与 **精确（噪声少）**，主要靠：

| 业界手段           | 在本项目中的落地方式                                                                                                                |
| ------------------ | ----------------------------------------------------------------------------------------------------------------------------------- |
| 混合检索补「专名」 | 在 `-q` 中显式加入 **素材里更可能出现的词**：品牌全称、产品型号、**官网/画册里的英文类目**、行业术语，减少「口语 ↔ 正文」词面不一致 |
| 查询改写           | 先去掉寒暄与指代，把用户话改成 **2–8 字的检索型短语**；口语转书面、别名转正名                                                       |
| 多查询 / 拆解      | 与 `-q` 内 **空格分检** 对齐：多角度短词并行检索再合并；避免一句长问塞满空格导致「词太碎」                                          |
| 元数据过滤         | 用 **`--folder-id` 锁定品牌库**、必要时 `--tags` 缩小类目标签，降低跨库噪声（全库时易混入其他客户素材）                             |
| 提高候选再筛       | 适当 **`--top-k` 10–15**，在对话里按 `score` 与正文相关性丢弃明显跑题的片段                                                         |

以下「AI 检索策略」与「提示词规范」均按上表设计：**先锁范围、再改写、再拆词、最后控噪**。

## 命令速览

```bash
# 第一步：列出所有已建索引的知识库（--rag-only 只显示可检索的）
siluzan-cso rag list --rag-only

# 第二步：按关键词检索（写稿 / 知识问答建议带 `--partition wiki`，见下文「检索分区」）
siluzan-cso rag query -q "<关键词>" --folder-id <id> --tags ""
siluzan-cso rag query -q "<关键词>" --partition wiki --top-k 12 --folder-id <id>
siluzan-cso rag query -q "<关键词>"                          # 全库；不传 --partition 时为 default
```

## 参数说明

| 选项                | 默认      | 说明                                                                                                                                                                                                  |
| ------------------- | --------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `-q, --query`       | 必填      | 检索词或短句。**若含空白**（如 `卖点 场景 优势`），CLI 会按空白拆成多个词**分别请求**，再按片段 id 去重、取最高相似度、排序后合并返回（`--json-out` 落盘数据中含 `subQueries`）；无空白时仍为单次检索 |
| `--folder-id`       | 全库      | 指定一个或多个文件夹 ID（逗号分隔），锁定检索范围                                                                                                                                                     |
| `--tags`            | 不过滤    | 不传 = 全量检索（无标签限制）；传具体标签 = 只返回含该标签的片段                                                                                                                                      |
| `--partition`       | `default` | **检索分区**，仅允许 **`wiki`** 或 **`default`**（小写）。控制从哪一类索引里取向量检索结果；含义见下一节 **「检索分区」**                                                                             |
| `--top-k`           | 7         | 每个分检请求返回条数（3–30）；多词合并后总条数上限为 `min(30, topK × 词数)`                                                                                                                           |
| `--json-out <path>` | —         | 将完整检索结果（含本次生效的 `partition`）落盘到目录或 \*.json 文件，stdout 仅一行摘要；处理落盘 JSON 见 `references/core/tips.md`                                                                    |

## 检索分区（`--partition`：`wiki` 与 `default`）

**是什么**：`rag query` 多一个检索维度——在同一企业、同一 `-q` / 文件夹 / 标签条件下，从 **wiki** 或 **default** 分区取结果。两条取值互斥；**一次命令只能选一个**，需要两种分区各来一轮时，请**执行两次** `rag query`（改 `--partition` 即可）。

**怎么用（推荐习惯）**：

1. **首轮用 `wiki`**（多数问答、写稿、需要较长正文作依据时）：`--partition wiki`，**`--top-k` 建议 8–15**（常用 **12**）。命中片段里常有更完整的正文；合成答案时以返回里的**正文**为主，不要只看一行摘要。
2. **仍缺原话、表格格内事实或关键信息时，再补一轮 `default`**：保持同一 `--folder-id` / `--tags`，`-q` 相同或略改写，`--partition default`，**`--top-k` 5–10**。**default 用来补漏**，wiki 已经够用就不必再跑 default。
3. **合并**：两轮结果在对话里按片段 **id**（必要时结合标题）去重后再用；说不清的地方如实说明缺口，**不要编造**。

**校验**：只允许 `wiki` 或 `default`；写错会报错。`--json-out` 落盘数据里会带上本次实际使用的 `partition`，便于核对。

## AI 检索策略指南

> 以下规则指导 AI 如何将用户意图转化为高质量的 RAG 检索。

### 0. 提升命中率与准确率：给模型用的「检索提示词」规范（执行前自检）

在调用 `rag query` 前，先在对话内完成下面 **5 步**，再把产物填入 `-q` / `--folder-id` / `--tags` / **`--partition`** / **`--top-k`**。这是本仓库在「仅向量检索、无 BM25/重排」约束下的最佳补偿流程。

**（1）意图归一（查询改写，Rewrite）**

- 去掉寒暄、指代与任务句（「帮我」「查一下」「有哪些」），保留 **实体 + 属性**。
- **品牌 / 公司**：用 `rag list` 里与知识库 **名称最一致** 的写法（如知识库名为「广东三合…」则不要用未入库的「广州三合」作为主检索词，除非用户坚持且已确认无对应库）。
- **一词多义**：若用户词太泛（「包装」「胶水」），补上 **行业或场景**（「食品软包装」「工业胶粘剂」），否则易全库跑偏。

**（2）锁范围再查（元数据优先，控精确）**

- 已识别品牌 → **必须** `--folder-id`；多品牌对比 → `id1,id2`。
- 仅当知识库已打标且用户要某一类素材时传 `--tags`；不确定标签是否存在时 **宁可不传**，避免 0 结果。

**（3）构造「检索型」`-q`（对齐索引词面，补混合检索之缺）**

- 优先 **名词与专名**，少用「如何」「为什么」等泛词单独检索。
- 素材常为 **官网 HTML、多语言导航、OCR/图片描述**：同一概念用 **中 + 英** 各检一轮或写在同一 `-q` 的空格分检里，例如：`垃圾袋 garbage-bag`、`SUN-HO OEM`。
- **必须整句不被拆分**时（如固定英文短语）：不要在该短语内插空格；或拆成两次独立命令各 `-q` 一整串无空格短语。

**（4）拆解与多角检索（Query Decomposition / Multi-query）**

- 与 CLI 行为对齐：**`-q "甲 乙 丙"` = 三次向量检索 + 去重合并**，适合「多角度各 2–6 字」；**不要**把一句话切成十几个单字。
- 复杂问题模板（思路，非固定文案）：
  - 事实类：`「品牌 + 产品名」` + `「型号 规格」` + `「认证 标准」`（各为短词组，空格连接）。
  - 写稿类：`「核心卖点」` + `「使用场景」` + `「对比 差异」` + `「证言 案例」`（视库内是否有案例再打「案例」）。
- **数值 / 条款 / 精确数据**：业界上 HyDE/随意扩写易引入幻觉；此处只做 **改写与专名对齐**，不在检索词里编造数字。

**（5）控噪与复盘（Precision）**

- 首次结果混入明显无关库（如标签带其他项目名）→ **收紧 `--folder-id`**，或略增 `--top-k` 后在对话里只采纳高 `score` 且正文与问题一致的片段。
- 空结果：**同义替换 → 扩大实体（上级类目）→ 去掉过严的 `--tags` → 最后才全库**。

**可复制给「执行检索的子任务」的系统提示片段（中文）：**

```text
你是 RAG 检索规划器。目标：在 siluzan-cso rag query 约束下最大化命中与准确。
约束：仅向量检索；`-q` 中含空格会拆成多次检索再合并；folder-id 不传则全库。
要求：1) 输出 JSON 计划：folderIds（数组）、queries（字符串数组，每个元素将单独作为一次 -q，或合并为一个用空格连接的 -q）、tags（可选）、**检索轮次 rounds**（数组；每项含 `partition`：`wiki` 或 `default`，及 `topK`；**第一轮建议 `wiki`**；证据不足时再加 `default`）、reason（简短中文）。
2) 默认至少一轮 `wiki`（topK 建议 10–12）；若需补检再加一轮 `default`（topK 可略低于 wiki）。
3) 每个 query 为检索型短语文本：2–12 个汉字为宜，或短英文专名；避免空洞词与过长整句。
4) 先锁品牌文件夹；禁止在 query 中编造具体数字、证书号、未确认的产品型号。
```

### 1. 判断是否需要 RAG

**需要 RAG 的场景：**

- 用户询问特定品牌/产品的信息（"海科佳的产品有哪些卖点"）
- 需要基于公司素材库写文案、口播稿、选题
- 执行三库内容工作流（流量因子库/产品资产库/烹调方法库）

**不需要 RAG 的场景：**

- 询问平台功能、操作方法（查 references 文档即可）
- 纯创意写作、无需依赖品牌素材的通用内容
- 用户明确说"不用查知识库"

---

### 2. 选择知识库（`rag list` → `--folder-id`）

先运行 `siluzan-cso rag list --rag-only --json-out ./snap-cso` 落盘列表并脚本读取，再按以下规则匹配：

| 用户意图                       | 文件夹选择策略                                   |
| ------------------------------ | ------------------------------------------------ |
| 明确提到某品牌/产品名          | 找名称最匹配的文件夹，用 `--folder-id <id>` 锁定 |
| 没有指定品牌，做通用内容       | 不传 `--folder-id`（全库检索）                   |
| 需要公司公共素材（总部公共库） | 不传 `--folder-id`，isGlobal 已覆盖公共库        |
| 用户提到多个品牌               | `--folder-id <id1>,<id2>` 同时检索               |

> 文件夹名称匹配优先：语义近似 > 包含关键词 > 不确定时全库。

---

### 3. 标签策略（`--tags`）

**默认行为（不传 `--tags`）= 不过滤标签 = 全量检索**，适用于绝大多数场景。

只有在知识库**已按标签打标**，且用户明确需要**精确筛选某类内容**时，才传具体标签：如果误传不存在的tag会导致查不到任何内容

```bash
# 只检索打了"流量因子库"标签的片段
siluzan-cso rag query -q "抖音爆款钩子" --tags "流量因子库"

# 同时检索两个标签
siluzan-cso rag query -q "产品卖点" --tags "产品资产库,烹调方法库"

# 不限标签（默认行为，通常省略 --tags）
siluzan-cso rag query -q "海科佳产品特点"
```

---

### 4. 查询拆解（Query Decomposition）

单次检索语义受限，**同一检索意图应拆成 2–5 个短关键词**，每个词聚焦一个子维度。

**传给 CLI 的两种方式（等价于「多词检索」时）：**

1. **一条命令、空格分隔**（推荐，省调用次数）：`-q "词甲 词乙 词丙"` — CLI 内部分别请求再合并排序。
2. **多条命令**：每个 `-q` 一条，适合 **不同 `--folder-id` / `--tags`** 等参数组合**无法合并**时仍须多轮。

> 注意：`-q` 里一旦出现空白就会触发拆分；**必须整段当一句查**时不要插入空格（例如英文长句若不想被拆，需写成无内部空格的表述或拆成单次短句检索）。

**示例 1：写产品口播文案**

```
用户意图 → "帮我写海科佳产品的口播文案"

拆解检索（--folder-id 海科佳ID，--tags ""）：
  "产品核心卖点"
  "用户痛点 解决方案"
  "产品使用场景"
  "品牌差异化优势"
```

**示例 2：三库选题（知识库已按三库打标）**

```
用户意图 → "给我找抖音爆款选题角度"

拆解检索（知识库已打三库标签，按标签精确分库；每个标签各一条命令，-q 内可用空格触发 CLI 分检合并）：
  siluzan-cso rag query -q "抖音 爆款 钩子" --tags "流量因子库"
  siluzan-cso rag query -q "产品 卖点 故事" --tags "产品资产库"
  siluzan-cso rag query -q "内容结构 情绪" --tags "烹调方法库"

若知识库未打标签，直接不传 --tags；多个词可写在一行，例如：
  siluzan-cso rag query -q "抖音 钩子 情绪 结构" --folder-id <id>
```

**示例 3：回答品牌知识问题**

```
用户意图 → "某企业是做什么的"

拆解检索（--folder-id 某企业ID，--tags ""）：
  "公司简介 主营业务"
  "产品类型"
  "服务对象 客户"
```

**拆词规则：**

- 每个子查询 **2–6 个汉字** 为宜（英文专名可略长）；过长的整句效果差；执行顺序与控噪见 **「### 0」**。
- 避免重复语义（"卖点"和"优势"只选一个）
- 优先具体名词，避免虚词（不要查"的"、"是"、"如何"）
- 需要回答某个具体问题时，先查核心实体，再查修饰属性

---

### 5. 结果合成与使用

检索完成后，按以下方式处理结果：

1. **去重**：同一 `-q` 空格分检时，CLI 已按片段 id 合并并保留**最高** `score`。若你仍执行了多条 `rag query`，再在对话侧对重复片段做一次合并。
2. **分类归档**：按检索关键词分组（例如：卖点类、场景类、情绪类）
3. **优先级**：`rag query` 的 `--json-out` 落盘数据与 Markdown 中，`score` 已为 **1 − 接口原始分** 并限制在 0–1，**越大越相关**。
4. **引用而非复制**：将片段作为信息来源重新组织表达，不直接粘贴原文
5. **结果为空时**：尝试更换关键词（同义替换、扩大范围），或告知用户知识库中暂无相关内容

siluzan-cso rag query -q "产品核心卖点 用户痛点 使用场景 品牌差异" --folder-id affe64c5-d3d4-4cec-ab33-196035916894 --partition wiki --top-k 12 --json-out ./snap-cso

## 与三库工作流的关系

三库写稿流程（见 `three-lib-content-workflow/content-writer.workflow.md`）可以调用 RAG 检索作为素材来源。若知识库已按三库体系打标，可按标签精确分库：

| 三库       | 对应 `--tags`         | 检索内容                     |
| ---------- | --------------------- | ---------------------------- |
| 流量因子库 | `--tags "流量因子库"` | 平台趋势、爆款钩子、流量密码 |
| 产品资产库 | `--tags "产品资产库"` | 产品特性、卖点、品牌故事     |
| 烹调方法库 | `--tags "烹调方法库"` | 内容结构、创作手法、情绪设计 |

**若知识库未打三库标签，直接不传 `--tags`，全量检索后按内容性质自行归类。**
