# LLM Wiki 架构

基于 Andrej Karpathy 提出的 LLM Wiki 理念，投教知识库采用"编译优先"模式，实现知识的持久沉淀与复利增长。

---

## 核心原理

**传统 RAG**：用户提问 → 向量检索 → 拼接上下文 → 生成回答（每次独立推理，无沉淀）

**LLM Wiki**：原始素材输入 → 编译为结构化 Markdown Wiki → 所有推理基于编译产物（知识越积越厚）

---

## 三层架构

### 第一层：原始素材层（不可变）

所有输入的投教原始资料统一进入 `raw/` 目录，永久不被修改。

**内容**：
- 证监会公告、交易所投教文章、官方法律法规
- 外部法律政策库、内部客户问答记录

**特点**：
- 事实基准，可追溯
- 当 Wiki 信息出现偏差时，可从原始素材重新编译

### 第二层：Wiki编译层（可变化）

Markdown 文件目录，包含：

| 类型 | 示例 | 说明 |
|------|------|------|
| 实体页面 | "市盈率"、"科创综指"、"北交所" | 基础概念定义 |
| 概念页面 | "资产配置"、"定投"、"复利效应" | 投资原理解释 |
| 主题摘要 | "投资者防骗指南"、"家庭财务规划" | 大类汇总 |
| 对比表格 | "低 PE vs 高PE 的误区对比" | 直观对比 |
| 交叉引用 | 各知识点之间的关联链接 | 知识图谱 |

**AI 代理职责**：
- 创建、更新和维护全部内容
- 当知识点更新时，自动检查所有引用页面并联动更新
- 标记新旧信息之间的矛盾之处
- 定期 Lint 检测（孤立页面、链接失效、内容冲突、信息过时）

### 第三层：应用与查询层

基于 Wiki 进行问答、检索和内容生成。

**三层检索策略**：
1. **Wiki 直接查询**：定义型、列典型的明确问题（如"什么是沪深 300？"）
2. **向量检索补充**：开放式、对比型的问题
3. **推理代理型（ReAct 循环）**：需要多步推理或综合多源信息的复杂问题

---

## 双循环机制

### 内循环（为当前客户即时服务）

```
客户提问 → 顾问检索知识库（优先）→ 找到→转译→输出（文字/语音/图文）→ 客户理解
                                 ↓未找到
                           定向搜索 → 转译 → 输出 → 同时反馈入库
```

### 外循环（系统性提升知识库）

```
顾问使用记录/客户常见问题统计 → 识别高频/难点知识点 → 主动策划深度投教内容 → 上线知识库 → 推送至所有相关客户
```

## 工程实践

| 需求 | 推荐方案 | 说明 |
|------|---------|------|
| 知识库构建平台 | Dify / RAGFlow | 开源、可视化、支持混合检索 |
| 向量数据库 | 云原生数据仓库 / Milvus | 稠密 + 稀疏向量混合检索 |
| 知识图谱 | Neo4j / FalkorDB | 构建实体关系网络 |
| 搜索引擎 | Elasticsearch | 关键词检索 + 元数据过滤 |
| LLM 模型 | Claude / GPT-4 / Qwen | 编译和推理核心 |
| 编排框架 | LangGraph | Agentic RAG 流程 |
