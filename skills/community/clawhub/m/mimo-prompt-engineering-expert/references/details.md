# AI提示词工程专家 - 详细内容参考 v1.3.0

## 一、高级提示词模式详解

### 1.1 CoT (Chain-of-Thought) 思维链模式
```
【问题】
{问题描述}

【思考步骤】
请按以下步骤逐步思考，每一步都要输出推理过程：

Step 1 - 问题拆解：
[分析问题的核心要素是什么]
[列出已知信息和未知信息]
[确定问题的类型（分类/推理/计算/比较/设计）]

Step 2 - 要素分析：
[对每个核心要素进行深入分析]
[找出要素之间的关联关系]
[识别关键约束条件和限制]

Step 3 - 关联推理：
[基于分析结果进行逻辑推理]
[检验推理是否符合常理]
[补充可能被遗漏的推理路径]

Step 4 - 方案生成：
[基于以上分析生成可能的方案]
[列出每个方案的优缺点]
[选择最优方案并说明理由]

Step 5 - 评估选择：
[评估各方案优劣]
[考虑可行性和风险]
[给出最终推荐方案]

【最终答案】
[综合以上步骤给出最终答案]
[确保答案完整覆盖问题所有方面]

【自我检验】
- [ ] 答案是否回应了原问题的每个方面？
- [ ] 推理过程是否有逻辑漏洞？
- [ ] 是否有遗漏的关键点？
- [ ] 答案是否符合常识和专业逻辑？
```

### 1.2 ReAct (Reasoning + Acting) 模式
```
【角色】
你是一个{职业}，擅长{领域}，在解决问题时会先思考、后行动、再观察结果。
你遵循"思考-行动-观察-调整"的循环，直到问题解决。

【任务】
{任务描述}

【流程】
请按以下流程处理每个问题：

Thought: [分析当前情况，我需要做什么？为什么？]
        [可能的方案有哪些？选择哪个方案？]

Action: [执行具体行动]
        [搜索/计算/查询/分析/对比/验证]

Observation: [观察行动的结果]
             [结果是否符合预期？]
             [如果不符合，可能的原因是什么？]

... (重复以上循环直到完成或达到最大迭代次数)

【输出格式】
请用以下格式输出：
## Thought
[你的思考]

## Action
[你的行动]

## Observation
[观察结果]

## Final Answer
[最终答案]
```

### 1.3 Self-Consistency 自洽性模式
```
【问题】
{问题}

【要求】
请用3种不同的方法/角度解决此问题，确保答案一致。
如果3种方法得到相同答案，说明解法可靠。
如果答案不一致，分析原因并给出最合理的答案。

【方法1：{方法名}】
[描述使用的方法/思路]
[执行推理步骤]
[得出答案]

【方法2：{方法名}】
[描述使用的另一种方法/思路]
[执行推理步骤]
[得出答案]

【方法3：{方法名}】
[描述使用的第三种方法/思路]
[执行推理步骤]
[得出答案]

【一致性检验】
- 方法1结果：{结果1}
- 方法2结果：{结果2}
- 方法3结果：{结果3}

如果答案一致（{X}/3一致），说明解法可靠。
如果答案不一致，分析原因：
- 哪些方法的前提假设不同？
- 哪个方法的前提假设更合理？
- 最终应选择哪个结果？

【最终答案】
{最终确认的答案及理由}
```

### 1.4 Tree-of-Thought 思维树模式
```
【问题】
{问题}

【目标】
探索多种可能的解决路径，找到最优解。
不要急于给出答案，先构建完整的思维树。

【思维树探索】
请构建一棵思维树，每个分支是一个思考方向：

🌳 分支1: {方向1}
├── 节点1.1: [具体思考/假设]
│   ├── 结果1.1.1: [可能的结果]
│   ├── 结果1.1.2: [可能的结果]
│   └── 评估: [该路径的可行性评分]
├── 节点1.2: [具体思考/假设]
│   └── 结果1.2.1: [可能的结果]
└── 分支评估: [该方向的优劣势分析]

🌳 分支2: {方向2}
├── 节点2.1: [具体思考/假设]
│   └── 结果2.1.1: [可能的结果]
└── 分支评估: [该方向的优劣势分析]

🌳 分支3: {方向3}
├── 节点3.1: [具体思考/假设]
│   └── 结果3.1.1: [可能的结果]
└── 分支评估: [该方向的优劣势分析]

【最优路径选择】
基于以上探索：
- 各分支的综合评分：{方向1: X分, 方向2: Y分, 方向3: Z分}
- 选择最优路径：{方向X}
- 选择理由：{理由}

【最终答案】
{按最优路径得出的答案}
```

### 1.5 Meta-Prompting 元提示模式
```
【角色】
你是一个提示词优化专家，擅长分析和改进AI提示词。
你会从以下维度分析提示词：
1. 清晰度：是否清晰无歧义
2. 完整性：是否包含所有必要信息
3. 结构：是否逻辑清晰层次分明
4. 约束：是否有足够的边界控制
5. 示例：是否提供了有效的Few-shot示例

【任务】
请分析以下提示词，并提供优化建议：

【原始提示词】
{用户提供的问题或目标}

【问题描述】
{用户当前遇到的问题}

【分析】
请从以上5个维度分析这个提示词的问题：

【优化后的提示词】
请根据分析结果提供优化后的提示词，并说明：
- 做了哪些改进
- 为什么要这样改
- 预期的效果

【测试建议】
建议用以下测试用例验证优化效果：
1. {测试用例1}
2. {测试用例2}
3. {测试用例3}
```

## 二、Few-shot 示例构建技巧

```
【任务】判断评论的情感倾向

【示例1：正面情感】

【示例2：正面情感】
输入：产品超出预期，会回购的！
输出：正面
解释：关键词"超出预期"、"回购"表达满意

【示例3：中性情感】
输入：一般般，没有特别的感觉
输出：中性
解释：没有明显的情感倾向词汇

【示例4：负面情感】
输入：等了两个月还没到，客服也不回复
输出：负面
解释：负面时间词"两个月"、负面服务词"不回复"

【待分类】
输入：{用户评论}
输出：
```

### 2.3 链式推理示例
```
【任务】计算商品折扣后的价格

【示例1】
输入：商品原价299元，现在打7折，还有满减活动满200减30
输出：
Step 1: 先算折扣：299 × 0.7 = 209.3
Step 2: 检查是否满足满减：209.3 > 200，满足
Step 3: 应用满减：209.3 - 30 = 179.3
最终价格：179.3元

【示例2】
输入：商品原价520元，店铺活动5折，平台券再减50，但商品标注"不参与满减"
输出：
Step 1: 先算折扣：520 × 0.5 = 260
Step 2: 检查满减：标注"不参与满减"，所以不满足满减条件
Step 3: 应用平台券：260 - 50 = 210
最终价格：210元

【待计算】
输入：{商品信息}
输出：
```

## 三、格式控制技巧

### 3.1 JSON输出控制
```
【要求】
请以JSON格式输出，包含以下字段：
{
  "status": "success|error|warning",  // 处理状态
  "code": 200,                          // 状态码
  "message": "描述信息",                 // 简要描述
  "data": {
    // 具体数据字段
  },
  "meta": {
    // 可选元数据
  }
}

【字段说明】
- status: 必须为success/error/warning之一
- code: 成功200，参数错误400，服务器错误500
- message: 简明扼要的描述
- data: 请求的实际数据，无数据时为null
- meta: 可选的元信息如分页等

【示例】
输入：统计用户增长数据
输出：
{
  "status": "success",
  "code": 200,
  "message": "用户增长统计完成",
  "data": {
    "total_users": 10000,
    "new_users_today": 150,
    "growth_rate": 0.015,
    "active_users_7d": 6500
  },
  "meta": {
    "query_time": "2024-01-15"
  }
}
```

### 3.2 Markdown表格控制
```
【要求】
请用Markdown表格输出，包含以下列：
| 序号 | 名称 | 评分 | 核心亮点 | 适用场景 | 推荐指数 |
|------|------|------|----------|----------|----------|

【格式规范】
- 表头用加粗
- 评分用星级符号 ⭐
- 推荐指数用⭐⭐⭐⭐⭐格式
- 超长内容用...截断

【示例】
输入：推荐5本编程书籍
输出：
| 序号 | 书名 | 作者 | 评分 | 适用人群 | 推荐指数 |
|------|------|------|------|----------|----------|
| 1 | 《代码大全》 | Steve McConnell | ⭐⭐⭐⭐⭐ | 中级开发者 | ⭐⭐⭐⭐⭐ |
| 2 | 《重构》 | Martin Fowler | ⭐⭐⭐⭐⭐ | 所有开发者 | ⭐⭐⭐⭐⭐ |
| 3 | 《算法导论》 | Cormen等 | ⭐⭐⭐⭐ | 进阶学习 | ⭐⭐⭐⭐ |
```

### 3.3 XML输出控制
```
【要求】
请以XML格式输出：
<result>
  <status>success|error</status>
  <data>
    <item>
      <field1>值1</field1>
      <field2>值2</field2>
    </item>
  </data>
</result>

【示例】
输入：返回员工信息
输出：
<result>
  <status>success</status>
  <data>
    <employee>
      <id>10001</id>
      <name>张三</name>
      <department>技术部</department>
      <salary>25000</salary>
    </employee>
  </data>
</result>
```

## 四、角色扮演深度设定

### 4.1 基础角色设定
```
【角色】
你是一位拥有10年经验的{职业}，专业领域是{具体领域}。
持有{资质认证}，曾参与{代表项目}。

【特点】
- 说话风格：{专业严谨/口语化/幽默风趣/简洁直接}
- 思考方式：{逻辑推理/系统思考/创新发散/经验导向}
- 沟通方式：{直接给答案/引导式提问/详细解释}

【专业边界】
- 擅长：{具体擅长领域}
- 不擅长：{明确边界}
- 不确定时：会说"这个我不太确定，建议查询官方文档"

【限制】
- 只回答{领域}相关问题
- 超出范围会礼貌拒绝
- 不提供医疗/法律/财务的具体建议
```

### 4.2 高级角色设定（带背景故事）
```
【角色背景】
你是张医生，在北京协和医院工作20年，见过无数病例。
从住院医师做到主任医师，擅长疑难杂症的诊断。
你擅长用通俗易懂的语言解释复杂的医学问题。
你总是先安慰病人情绪，再给出专业建议。
你的口头禅是："别担心，我们一起来看看。"

【对话风格】
- 开场白："您好，我是张医生。有什么不舒服的地方吗？"
- 解释病情时：用生活化的比喻（"就像水管堵了一样..."）
- 给出建议时：先说最重要的，再说次要的
- 结尾总会问："还有其他问题吗？"
- 不确定时：诚实说"这个我需要查一下资料"

【行为约束】
- 不夸大病情严重程度
- 不给不确定的诊断（"我没有看到检查结果，不敢下定论"）
- 总是建议"如有疑虑请及时就医"
- 不推荐具体药品（只说药物类别）
- 涉及手术等重大决策时会强调"这需要详细评估"
```

## 五、提示词优化迭代流程

```
┌────────────────────────────────────────────┐
│  Step 1: 明确目标                            │
│  - 确定任务类型（问答/创作/分析/代码/翻译）   │
│  - 确定期望输出的具体特征                     │
│  - 确定必须满足的约束条件                     │
│  - 确定使用的模型                            │
└────────────────────┬───────────────────────┘
                     ↓
┌────────────────────────────────────────────┐
│  Step 2: 编写初版                            │
│  - 角色设定（职业+背景+风格+边界）            │
│  - 任务描述（清晰指令+具体要求）              │
│  - 输出格式（指定格式+长度+结构）             │
│  - 示例（可选，但强烈推荐添加）               │
└────────────────────┬───────────────────────┘
                     ↓
┌────────────────────────────────────────────┐
│  Step 3: 测试验证                            │
│  - 用3个不同类型的输入测试                    │
│  - 记录每个输出的质量                        │
│  - 识别问题和改进点                          │
│  - 测试用例：典型输入/边界输入/极端输入        │
└────────────────────┬───────────────────────┘
                     ↓
┌────────────────────────────────────────────┐
│  Step 4: 优化调整                            │
│  - 调整角色设定（更具体/更严格）              │
│  - 补充或改进示例（覆盖更多场景）              │
│  - 强化约束条件（解决发现的问题）              │
│  - 改进输出格式（更精确的定义）               │
│  - 调整prompt长度和结构                      │
└────────────────────┬───────────────────────┘
                     ↓
┌────────────────────────────────────────────┐
│  Step 5: 压力测试                            │
│  - 极端输入测试（空值/超长/特殊字符）         │
│  - 恶意输入测试（对抗性输入）                 │
│  - 一致性测试（同一输入多次运行）              │
│  - 确认稳定性                              │
└────────────────────┬───────────────────────┘
                     ↓
┌────────────────────────────────────────────┐
│  Step 6: 固化文档化                          │
│  - 记录最终版本                             │
│  - 标注适用模型和参数设置                    │
│  - 记录已知局限和workaround                 │
│  - 设定复查周期                            │
└────────────────────────────────────────────┘
```

## 六、实战模板库

### 6.1 翻译优化模板
```
【角色】
你是一位专业翻译，精通源语言和目标语言。
专业领域：{具体领域，如：法律/医学/技术/商业}
持有翻译资质：{CATTI/雅思/其他}

【任务】
翻译以下文本，保持原文风格和语气。
适当调整表达方式，使其符合目标语言的文化习惯。

【要求】
1. 保留专业术语（如无官方译法则保留原文并注释）
2. 符合目标语言的表达习惯（意译优先于直译）
3. 语气和风格保持一致（正式/口语/文学）
4. 段落结构尽量保持（除非需要重组以符合目标语言习惯）
5. 如有歧义，保留原文并加括号注释

【源语言】
{语言}

【目标语言】
{语言}

【原文】
{待翻译文本}

【输出格式】
翻译：
{译文}

注释：
{需要解释的术语或歧义处理}

术语对照表：
| 原文 | 译文 | 说明 |
|------|------|------|
```

### 6.2 代码审查模板
```
【角色】
你是一位资深{语言}工程师，拥有{年数}年开发经验。
专长领域：{架构设计/性能优化/安全审计/代码规范}

【任务】
审查以下代码，找出潜在问题并提供改进建议。

【代码语言】
{语言}

【代码】
```{语言}
{代码}
```

【审查维度】
请从以下维度进行审查：

1. 功能正确性
   - 代码逻辑是否正确实现需求？
   - 边界条件是否正确处理？

## 三、Prompt诊断修复指南

| 问题现象 | 可能原因 | 修复方案 | 优先级 |
|----------|----------|----------|--------|
| AI回答跑题 | 任务描述模糊 | 明确"你需要"+"具体任务" | 🔴 高 |
| 输出格式不稳定 | 缺少格式示例 | 添加1-2个格式示例 | 🔴 高 |
| 回答太笼统 | 缺少约束条件 | 添加具体限制（字数/维度/深度） | 🟡 中 |
| 回答太啰嗦 | 缺少长度控制 | 明确"不超过N字"或"N个要点" | 🟡 中 |
| 角色扮演不持久 | 角色设定太简单 | 增加人格特征和行为约束 | 🟡 中 |
| 代码质量差 | 缺少代码规范 | 指定语言版本+编码规范+注释要求 | 🔴 高 |
| 多语言混乱 | 语言指令不明确 | 明确"全程使用中文/英文" | 🟡 中 |
| 创意不足 | 示例太固定 | 减少示例数量，增加自由度提示 | 🟢 低 |
| 回答前后矛盾 | 缺少一致性约束 | 添加"确保回答前后一致" | 🟡 中 |
| 忽略部分指令 | prompt太长注意力分散 | 精简prompt，核心指令放开头 | 🔴 高 |
| 输出不够专业 | 缺少领域知识引导 | 添加专业术语表和参考框架 | 🟡 中 |
| 温度过高/过低 | 未指定temperature | 创意类0.7-0.9，精确类0.1-0.3 | 🟢 低 |
