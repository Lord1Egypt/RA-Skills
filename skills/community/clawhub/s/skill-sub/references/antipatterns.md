# skill-sub 反模式

> 本文档收录 skill-sub 使用中的常见错误和正确做法。完整反模式示例见 `references/faq.md`。

---

## AP-01：把 skill-sub 编入调用链

**错误做法：**创建调用链时把 `skill-sub` 作为一个步骤加入。

```
步骤1: skill-sub → 规划
步骤2: xxx
```

**为什么错：**skill-sub 是编辑器/规划器，不是执行器。链需要的是"谁干活"，skill-sub 不干活。

**正确做法：**调用链只包含执行型 Skill（如 semantic-split、git-sync），不包含 skill-sub 本身。

---

## AP-02：为一次性任务创建调用链

**错误做法：**用户说「帮我查一下今天天气」，就创建一个"天气查询"调用链。

**为什么错：**调用链的价值是复用。一次性任务下次不会用到，反而积累垃圾链。

**正确做法：**判断任务是否有复用价值。以下信号表明值得创建链：
- 用户说「每次」「以后也」「下次」
- 涉及 ≥3 个 Skill 协作
- 有明确的输入→输出模式

---

## AP-03：跳过摘取阶段，直接写 action

**错误做法：**不读目标 Skill 的 SKILL.md，凭记忆写 action 描述。

```json
{"action": "执行安全审计"}  // 太模糊，实际执行时不知道具体做什么
```

**为什么错：**action 是执行时的唯一指引。不读 Skill 原文会导致遗漏关键步骤或用错指令名。

**正确做法：**执行阶段2摘取——读取目标 Skill 的 SKILL.md「工作流程」章节，摘取步骤名+关键动作+指令名。action 要包含具体的**做什么**（动词+对象+产出）。

---

## AP-04：把所有步骤都标为里程碑

**错误做法：**3 步的链，全标 `is_milestone: true`。

**为什么错：**里程碑=失败即中止。全标等于没有区分，任何一步失败都中止，丧失灵活性。

**正确做法：**用里程碑判断规则（关键词/瓶颈点/最终交付）自动判断，只对关键步骤标里程碑。非关键步骤用 `on_exhaust: ask` 或 `skip` 而非 `abort`。

---

## AP-05：依赖关系写错导致链无法执行

**错误做法：**
```json
{"index": 1, "depends_on": []},
{"index": 2, "depends_on": [3]},  // 依赖了后面的步骤！
{"index": 3, "depends_on": [1]}
```

**为什么错：**步骤2依赖步骤3，但步骤3在步骤2之后执行——死锁。

**正确做法：**`depends_on` 只能引用 `index` 小于自己的步骤。创建后用 `chain_executor.py validate` 检查。
