## 1.1 什么是智能体？

> 任何能够通过**传感器**感知其所处**环境**，并**自主**地通过**执行器**采取**行动**以达成特定目标的实体。

**四个核心要素：**

- **环境（Environment）** — 智能体所处的外部世界，包括物理环境、信息环境、其他智能体等
- **传感器（Sensors）** — 感知环境状态的通道，如摄像头、麦克风、数据接口
- **执行器（Actuators）** — 对环境施加影响的手段，如电机、显示器、API 调用
- **自主性（Autonomy）** — 独立决策以达成目标的能力，无需人类持续干预

## 智能体与环境的交互

<!-- _class: fig-top -->

<div class=fig-container>

![width:850px](1757242319667-0.png)

</div>

<div class=text-container>

- 智能体通过传感器持续感知环境状态，经过内部处理后通过执行器改变环境。这个**感知→决策→行动→观察**的闭环是所有智能体行为的基础模型。

**关键点：** 环境不仅是"背景"，更是智能体行为的"反馈源"——每一次行动都会改变环境，产生新的感知输入。

</div>

## 1.1.1 传统视角下的智能体演进

<!-- _class: tinytext -->

**智能体的五级演进模型：**

1. **反射智能体（Reflex Agent）**
   - 最简单的"条件-动作"规则，无记忆、无预测能力

2. **基于模型的反射智能体（Model-Based Reflex Agent）**
   - 拥有内部世界模型，能维护环境状态的认知

3. **基于目标的智能体（Goal-Based Agent）**
   - 主动规划行动序列以达成特定目标

4. **基于效用的智能体（Utility-Based Agent）**
   - 在多个可能目标间权衡，最大化期望效用

5. **学习型智能体（Learning Agent）**
   - 通过与环境的持续互动，自主改进决策策略

> **演进方向：** 从被动反应到主动规划，从固定规则到自主学习，从单一能力到综合智能

## 反射智能体（Simple Reflex Agent）

<!-- _class: fig-top -->

<div class=fig-container>

![width:700px](1757242319667-1.png)

</div>

<div class=text-container>

**决策机制：** 显式的"条件-动作"规则表（Condition-Action Rules）
- **输入：** 当前感知
- **输出：** 直接的动作
- **特点：** 无记忆、无推理、无预测

**经典案例：** 自动恒温器
- 规则：`IF 温度 > 设定值 THEN 启动制冷`
- 简单可靠，但无法处理复杂场景

</div>

## 反射智能体的局限与改进

<!-- _class: smalltext -->

**局限性分析：**

- **无状态记忆** — 无法利用历史信息做决策
- **无环境模型** — 不理解世界如何运作
- **短视行为** — 只能对当前刺激做反应
- **规则爆炸** — 复杂环境需要大量手工规则

**典型问题场景：**

> 恒温器检测到阳光直射导致室温升高，立刻启动空调。但实际上太阳落山后温度会自然下降，此时的制冷行为造成能源浪费。反射智能体无法考虑这种"未来变化"。

**改进方向：** 引入内部状态 → 基于模型的反射智能体

## 基于模型的反射智能体

**核心改进：引入内部世界模型**

- **世界模型（World Model）** — 对环境如何运作的内部表示
- **状态追踪** — 维护环境的当前状态估计
- **感知解释** — 将原始感知映射到有意义的状态

**决策过程：**

1. 根据感知更新内部状态估计
2. 基于世界模型预测行动结果
3. 选择最合适的动作

> **隧道中的自动驾驶案例：** 摄像头暂时无法看到前方车辆，但内部模型根据之前的感知（车辆位置、速度）维持对其位置的估计，继续做出合理的驾驶决策。

## 基于目标的智能体

<!-- _class: tinytext -->

**核心能力：规划（Planning）**

- **目标（Goal）** — 期望达成的状态描述
- **规划算法** — 搜索从当前状态到目标状态的行动序列
- **前瞻性** — 考虑行动的长远影响，而非即时反应

**经典案例：GPS 导航系统**

| 传统反射式 | 目标驱动式 |
|---|---|
| 根据当前位置选择下一步 | 规划完整的路线方案 |
| 遇到障碍物绕行 | 预先避开拥堵路段 |
| 局部最优决策 | 全局最优路径 |

**关键区别：** "我应该做什么？" vs "我该怎么做才能达成目标？"

## 基于效用的智能体

<!-- _class: smalltext -->

**核心概念：效用函数（Utility Function）**

- **效用值** — 对每种可能状态赋予的"满意度"分数
- **期望效用最大化** — 选择能带来最高期望效用的行动
- **多目标权衡** — 在冲突目标间做出平衡决策

**智能旅行助手示例：**

```
目标：从 A 到 B
需要权衡的因素：
- 时间成本（越快越好）
- 经济成本（越省越好）
- 舒适度（越舒适越好）
- 可靠性（越稳定越好）
```


## 学习型智能体

<!-- _class: tinytext -->

**架构组成：**

- **性能元件（Performance Element）** — 执行实际任务的智能体核心
- **学习元件（Learning Element）** — 观察行为结果，改进决策策略
- **评价标准（Critic）** — 定义什么是"好"的行为
- **问题生成器（Problem Generator）** — 主动探索新策略

**里程碑案例：AlphaGo Zero**

| 特征 | 传统 AI | AlphaGo Zero |
|---|---|---|
| 知识来源 | 人类专家数据 | 自我对弈探索 |
| 学习方式 | 监督学习 | 强化学习 |
| 性能上限 | 受限于人类水平 | 超越人类顶尖棋手 |
| 训练成本 | 需要大量标注数据 | 仅需游戏规则 |

> **核心洞见：** 智能体可以通过"试错-反馈"的循环，在没有人类示范的情况下学会解决复杂问题。

**范式转变的本质：**
- **传统智能体：** 工程师显式编程 → 行为确定但能力受限
- **LLM 智能体：** 模型隐式学习 → 行为灵活且具备泛化能力

**公式化对比：**
- 传统 = "人类写好规则，机器来执行"

## LLM 智能体的三大核心能力

<!-- _class: tinytext -->

以 **"规划一次厦门之旅"** 为例：

**1. 规划与推理能力**
- 分析任务：确定日期、预算、偏好、行程安排
- 分解子任务：查天气 → 订票 → 订酒店 → 规划景点
- 排序优先级：先解决信息依赖，再执行决策

**2. 工具使用能力**
- 调用天气 API 获取厦门周末天气
- 调用机票搜索接口比较价格
- 调用地图 API 规划路线

**3. 动态修正能力**
- 用户反馈"酒店超预算" → 重新搜索更便宜的选项
- 天气预报有雨 → 调整为室内活动为主
- 航班取消 → 自动寻找替代方案

> **核心转变：** 从开发专用自动化工具 → 构建能自主解决问题的通用智能系统

## 1.1.3 智能体的分类

**三个正交的分类维度：**

| 维度 | 关注点 | 典型分类 |
|---|---|---|
| **内部决策架构** | 智能体如何做决策 | 反应式 → 学习型 |
| **时间与反应性** | 决策速度 vs 质量 | 反应式 / 规划式 / 混合式 |
| **知识表示** | 知识如何存储 | 符号主义 / 亚符号主义 / 神经符号主义 |

不同维度反映了智能体设计的不同权衡，实际系统往往是多个维度的组合。

## 决策时间与质量的关系

<!-- _class: fig-top -->

<div class=fig-container>

![width:600px](1757242319667-3.png)

</div>

<div class=text-container>

**核心权衡：速度（反应性） vs 质量（最优解）**

- **快速响应** → 可能不是最优决策
- **深度规划** → 决策质量高但耗时

**应用场景选择：**
- 安全气囊、高频交易 → 必须毫秒级响应
- 商业决策、战略规划 → 可以花时间深思熟虑

</div>

## 反应式 vs 规划式 vs 混合式

**三种架构对比：**

| 特性 | 反应式 | 规划式 | 混合式 |
|---|---|---|---|
| 响应速度 | 快（毫秒级） | 慢（秒级到分钟级） | 兼顾 |
| 决策质量 | 局部最优 | 全局最优 | 平衡 |
| 适用场景 | 安全关键、实时系统 | 复杂决策、战略规划 | 通用场景 |
| 典型案例 | 安全气囊、防抱死系统 | 下棋 AI、商业决策系统 | 自动驾驶、现代 LLM Agent |

**混合式架构的实现：**

> 现代 LLM 智能体在"思考-行动-观察"循环中巧妙融合两种模式：规划阶段深思熟虑，行动阶段快速执行。

## 知识表示：三大范式

<!-- _class: fig-top -->

<div class=fig-container>

![width:750px](1757242319667-4.png)

</div>

<div class=text-container>

**核心问题：** 智能体的"知识"以何种形式存在于其"思想"中？

三大范式代表了 AI 发展史上不同的哲学立场和技术路径：
- **符号主义** — 知识是显式的符号和规则
- **亚符号主义** — 知识是隐式的分布式表示
- **神经符号主义** — 融合两者的优势

</div>

## 符号主义 vs 亚符号主义

<!-- _class: table-right smalltext -->

<div class=ldiv>

**符号主义 — "图书管理员"**

- 知识以显式符号和逻辑规则存储
- 推理过程透明、可解释、可验证
- 问题：知识获取瓶颈、脆弱性

**亚符号主义 — "牙牙学语的孩童"**

- 知识隐式存储在神经网络权重中
- 从数据中自动学习模式
- 问题：黑箱不可解释、需要大量数据

</div>

<div class=rdiv>

| 维度 | 符号主义 | 亚符号主义 |
|---|---|---|
| 知识表示 | 符号 + 规则 | 神经网络权重 |
| 推理方式 | 逻辑演绎 | 模式匹配 |
| 可解释性 | 高 | 低 |
| 学习能力 | 低（需人工编码） | 高（从数据学习） |
| 泛化能力 | 差 | 强 |
| 典型代表 | 专家系统、知识图谱 | 深度学习、LLM |

</div>

## 神经符号主义 AI

**融合之路：兼具学习能力和推理能力**

**卡尼曼双系统理论的类比：**

| 系统 | 特征 | 对应范式 |
|---|---|---|
| **系统 1** | 快速、直觉、自动 | 亚符号主义（模式识别） |
| **系统 2** | 慢速、逻辑、刻意 | 符号主义（推理规划） |

**LLM 智能体的神经符号特性：**

- **神经部分** — LLM 的预训练权重，提供语言理解和常识推理能力
- **符号部分** — 工具调用、结构化输出、显式推理链

> **实践意义：** LLM 智能体不需要在符号和亚符号之间二选一，而是自然地融合了两种范式的优势。

## 1.2 智能体的构成与运行原理

<!-- _class: fig-top -->

<div class=fig-container>

![width:850px](1757242319667-6.png)

</div>

<div class=text-container>

**PEAS 模型** — 描述智能体构成的经典框架：
- **P**erformance（性能度量）— 如何评价智能体的表现？
- **E**nvironment（环境）— 智能体在什么条件下工作？
- **A**ctuators（执行器）— 智能体如何影响环境？
- **S**ensors（传感器）— 智能体如何感知环境？

</div>

## PEAS 模型实例分析

<!-- _class: smalltext -->

**案例：自动驾驶汽车**

| PEAS 要素 | 具体内容 |
|---|---|
| **性能度量（P）** | 安全性、合规性、效率、舒适度 |
| **环境（E）** | 道路、交通、天气、行人、其他车辆 |
| **执行器（A）** | 方向盘、油门、刹车、转向灯、喇叭 |
| **传感器（S）** | 摄像头、激光雷达、雷达、GPS、里程表、IMU |

**案例：智能客服**

| PEAS 要素 | 具体内容 |
|---|---|
| **性能度量（P）** | 问题解决率、用户满意度、响应速度 |
| **环境（E）** | 用户提问、业务知识库、订单系统 |
| **执行器（A）** | 文本回复、工单创建、API 调用 |
| **传感器（S）** | 用户输入文本、点击行为、历史对话 |

## 环境的复杂特性

**环境特性如何影响智能体设计：**

| 特性 | 描述 | 设计影响 |
|---|---|---|
| **完全/部分可观察** | 是否能获取完整环境信息 | 部分可观察需维护状态估计 |
| **确定性/随机性** | 行动结果是否确定 | 随机环境需要概率推理 |
| **单智能体/多智能体** | 环境中是否有其他行动者 | 多智能体需考虑策略交互 |
| **静态/动态** | 环境是否在智能体思考时变化 | 动态环境需要持续感知 |
| **离散/连续** | 状态和动作空间是否有限 | 影响表示和算法选择 |

**现实环境通常是：** 部分可观察 + 随机性 + 多智能体 + 动态 + 连续

## 智能体循环（Agent Loop）

<!-- _class: fig-top -->

<div class=fig-container>

![width:800px](1757242319667-5.png)

</div>

<div class=text-container>

**Agent Loop** — 智能体最核心的运行机制

1. **感知（Perception）** — 通过传感器接收环境输入
2. **思考（Thought）** — 规划、推理、决策
3. **行动（Action）** — 通过执行器执行操作
4. **观察（Observation）** — 感知行动结果

**关键点：** 这是一个持续闭环，行动改变环境 → 新的观察 → 下一轮循环

</div>

## Thought-Action-Observation 协议

<!-- _class: tinytext -->

**协议结构：**

```
Thought: [智能体的内部推理过程]
Action: [要执行的动作及参数]
```

**运行示例：**

```python
# 第 1 轮
Thought: 用户想知道北京天气，我需要调用天气查询工具。
Action: get_weather("北京")

# 环境返回
Observation: 北京当前天气为晴，气温25摄氏度，微风。

# 第 2 轮
Thought: 已经获得天气信息，可以回答用户问题了。
Action: Finish[北京今天天气晴朗，气温25度，有微风，适合外出活动。]
```

**协议优势：** 结构化输出便于外部系统解析执行，同时保留了推理过程的可解释性。

## 1.3 动手体验：实现第一个智能体

**任务场景：**

> 用户请求："你好，请帮我查询一下今天北京的天气，然后根据天气推荐一个合适的旅游景点。"

**智能体需要：**

1. 理解用户意图 — 这是一个需要分步完成的任务
2. 规划执行顺序 — 先查天气，再推荐景点
3. 正确使用工具 — 调用天气 API 和景点推荐 API
4. 综合信息回答 — 将多次工具调用的结果整合成完整答案

## 核心组件设计

<!-- _class: smalltext -->

**系统架构：**

```
┌─────────────────────────────────────┐
│          System Prompt              │
│   （定义角色、工具列表、输出格式）        │
├─────────────────────────────────────┤
│         Tool 1: get_weather         │
│         Tool 2: get_attraction      │
├─────────────────────────────────────┤
│            LLM Client               │
│      （连接大语言模型 API）             │
├─────────────────────────────────────┤
│           Agent Loop                │
│   （驱动 Thought-Action-Observation） │
└─────────────────────────────────────┘
```

**关键设计决策：**
- 如何定义工具接口？
- System Prompt 怎么写才能引导正确行为？
- 如何判断任务完成？

## System Prompt 示例

<!-- _class: tinytext -->

```
你是一个智能旅行助手，请使用可用工具一步步解决问题。

# 可用工具:
- get_weather(city: str) -> str
  功能：查询指定城市的实时天气
  参数：city - 城市名称
  返回：天气描述字符串

- get_attraction(city: str, weather: str) -> list
  功能：根据城市和天气推荐旅游景点
  参数：city - 城市名称, weather - 天气状况
  返回：推荐景点列表

# 输出格式:
Thought: [你的思考过程]
Action: [工具名称(参数)] 或 Finish[最终答案]

# 注意事项:
- 每次只调用一个工具
- 收集足够信息后再给出最终答案
- 使用 Finish 表示任务完成
```

## 运行案例：3 轮循环

| 轮次 | Thought | Action | Observation |
|---|---|---|---|
| **1** | 先获取北京天气信息 | `get_weather("北京")` | 北京今天晴天，25°C |
| **2** | 晴天适合户外活动 | `get_attraction("北京", "Sunny")` | 故宫、颐和园、天坛 |
| **3** | 信息充足，可以回答 | `Finish[北京今天...]` | — |

**最终输出：**

> 北京今天天气晴朗，气温25摄氏度，非常适合户外活动。推荐您游览故宫博物院，感受皇家宫殿的宏伟；或者去颐和园漫步昆明湖畔，享受春日阳光。

## 智能体的四项基本能力

<!-- _class: smalltext -->

通过这个案例，我们总结出 LLM 智能体的四项核心能力：

**1. 任务分解（Task Decomposition）**
- 将复杂目标拆解为可执行的子任务
- 确定子任务之间的依赖关系

**2. 工具调用（Tool Invocation）**
- 理解工具的功能和参数要求
- 正确构造调用参数
- 解析和处理返回结果

**3. 上下文理解（Context Understanding）**
- 维护多轮对话的上下文
- 整合多次工具调用的结果

**4. 结果合成（Result Synthesis）**
- 将碎片化信息整合成完整答案
- 以自然语言形式呈现给用户

## 1.4 智能体应用的协作模式

**两种应用范式：**

| 范式 | 定位 | 典型代表 |
|---|---|---|
| **开发者工具** | 增强人类能力 | Copilot、Cursor |
| **自主协作者** | 独立完成任务 | AgentGPT、CrewAI |

**关键差异：**
- 开发者工具：人在环路中，AI 辅助决策
- 自主协作者：AI 独立决策，人监督结果

## 作为开发者工具的智能体

> **设计哲学：** 增强而非取代开发者，让人类专注于高层次的创造和决策。

## Workflow vs Agent

<!-- _class: fig-top -->

<div class=fig-container>

![width:450px](1757242319667-18.png)

</div>

<div class=text-container-small>

**Workflow（工作流）** — 预定义的执行步骤
- 固定流程，可预测、可控
- 适合规则明确、变化少的场景

**Agent（智能体）** — 自主决策的执行者
- 动态推理，灵活、适应性强
- 适合复杂、不确定的场景

**最佳实践：** 两者结合——关键节点用 Workflow 控制，细节决策给 Agent 自主权

</div>

## Workflow vs Agent 关键差异

<!-- _class: table-right -->

<div class=ldiv-small>

**设计选择指南：**

**选择 Workflow 当：**
- 任务流程固定、可预测
- 需要严格控制和审计
- 错误成本高，不容许偏差

**选择 Agent 当：**
- 任务复杂、不确定性强
- 需要灵活适应变化
- 可以接受一定的探索成本

**混合策略：**
- 整体流程用 Workflow 框架
- 关键决策点嵌入 Agent 模块
- 人类审核关键输出

</div>

<div class=rdiv>

| 维度 | Workflow | Agent |
|---|---|---|
| **核心** | 预定义步骤编排 | 自主决策与推理 |
| **灵活性** | 低 | 高 |
| **可预测性** | 高 | 低 |
| **开发成本** | 需详细设计 | 需调优提示词 |
| **调试难度** | 低 | 高 |
| **典型场景** | 报销审批流程 | 智能旅行助手 |
| **适应变化** | 需人工修改 | 自动调整 |

</div>

  Workflow 的可控性 vs Agent 的灵活性
