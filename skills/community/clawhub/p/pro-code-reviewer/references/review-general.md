# 通用审查维度

适用于非 iOS/Android 的项目（TypeScript、Python、Go 等）。按以下 7 个维度审查。

## 1. 正确性
- 逻辑错误、边界条件缺失
- 空值/nil/undefined 处理是否完善
- 类型安全（强制类型转换、不安全的 any）
- 数组越界风险

## 2. 线程安全
- 是否存在竞态条件（多线程/协程访问共享状态）
- 锁的使用是否正确（死锁风险、锁粒度）
- 主线程/UI 线程操作是否有保证
- 异步并发控制是否合理

## 3. 内存管理
- 循环引用（闭包/回调中的引用捕获）
- 大对象的持有和释放
- 缓存是否有上限控制
- 事件监听器/订阅是否正确取消

## 4. 性能
- 不必要的重复计算
- 大数据量的遍历/排序效率
- 主线程耗时操作
- 不必要的对象创建

## 5. API 设计与可维护性
- 方法命名是否清晰
- 接口是否符合现有代码风格
- 是否有不必要的 public 暴露
- 代码重复

**过度工程化红线（以下情况不要提出建议）：**
- 三行以内的相似代码不算"重复"
- 仅被调用一次的逻辑不需要提取方法
- 不要建议为假想的未来需求做扩展性设计
- 不要要求给未变更的代码加注释、文档或类型标注
- 不要建议为一次性操作创建工具类/抽象层

## 6. 错误处理
- 异常路径是否覆盖
- 错误是否被静默吞掉
- 降级策略是否合理

## 7. 安全性
- 敏感数据处理（密钥/token 硬编码、日志泄露敏感信息）
- 输入校验与注入防护（命令注入、SQL 注入、XSS、路径遍历）
- 不安全的反序列化、不受信任数据的类型强转
- 权限校验缺失（越权访问、未鉴权接口）

## 报告格式

```markdown
# Code Review 报告

## 概览
- **Review 范围**: commit_id..HEAD (N commits)
- **变更文件数**: N
- **总变更行数**: +X / -Y

## Review 详情

### [Critical] 必须修复
> 影响正确性或安全性的问题，必须修复后才能合入

**文件**: `path/to/file:行号`
**问题**: 具体描述
**建议**: 修复方案

### [Warning] 建议修复
> 可能导致潜在问题，强烈建议修复

### [Suggestion] 改进建议
> 代码质量提升建议，非阻塞性

## 总结
- Critical: N 个
- Warning: N 个
- Suggestion: N 个
- 总体评价: ...
```
