# task-record

把一段已完成工作写入飞书任务树。

## 场景

- 用户说"帮我记一下今天做了什么"
- 用户说"记录一下：完成了数据清洗"
- cron 每日提醒触发记录流程
- agent 自动记录进展

## 两种交互模式

### 模式 A：用户提供了内容

触发："记录一下：今天完成了数据清洗工作"、"帮我记：给老师配了 bot，修了几个 bug"

```
1. check attachable 获取候选列表
2. 语义匹配：用户内容 ↔ 候选任务
3. 判断是否可以直接记录（见下方"直接记录 vs 确认"）
4. 记录（单条或多条）
```

### 模式 B：用户未提供内容

触发："帮我记一下最近做的" / cron 提醒 / "用一下记录技能"

```
1. 优先尝试 Cotrace 获取工作数据（见下方）
2. 如果 Cotrace 有数据 → 生成草案 → 按模式 A 继续
3. 如果 Cotrace 无数据/不可用 → 询问用户做了什么
   - 附带 Cotrace 小提示："也可以接入 cotrace 自动采集工作记录哦"
4. 用户回复后 → 进入模式 A
```

Cotrace 小提示仅在 Cotrace 不可用/无数据时出现，不要每次都提。

## 直接记录 vs 确认

### 可直接记录（无需确认）：

- 内容明确，用户语气确定（"帮我记"、"记下来"、"记录一下：xxx"）
- 挂载点唯一匹配（内容明显属于某个任务）
- 不存在歧义或敏感信息

直接记录后告知结果即可。

### 需要确认的情况：

- 匹配不确定：内容可能属于多个候选，不好选
- 内容含模糊信息
- 用户语气带犹豫（"好像做了..."、"大概是..."）

此时展示草案，询问"这样记可以吗？"，确认后记录。

## 涉及多个任务时

用户一次描述了多项工作（如"今天给老师配了 bot，还修了几个 bug，另外处理了额度问题"）：

1. 拆分为多条记录
2. 分别匹配挂载点
3. **一次性展示所有草案**，打包确认
4. 确认后批量记录

示例输出：
```
帮你整理了 2 条记录：
1. → [主线] Auwomo OS 推广运行
   "给老师配置 bot、修复引导 bug、优化未初始化引导流程"
2. → [主线] Claude 使用方案调研
   "处理 Claude 额度问题"

这样记可以吗？
```

## 无匹配候选时

如果用户描述的工作无法匹配到任何现有任务：

1. 告知用户："这条记录目前没有对应的主线任务可以挂载"
2. 建议："要不要先创建一个主线？比如 [主线] xxx"
3. 用户确认后 → `task create` 创建主线 → 然后挂载记录

绝不创建孤立记录（没有父任务的记录）。

## 使用 Cotrace 作为数据源（优先）

在模式 B 中，优先使用 Cotrace 的 work-records 获取实际工作数据，参考 `skills/cotrace/SKILL.md`：

```bash
# 检查 Cotrace 是否可用
ftc health cotrace

# 获取今天的工作摘要
echo '{"tool":"get_workstream_summaries","args":{"created":{"from":"<today>T00:00:00Z","to":"<today>T23:59:59Z"}}}' | ftc call cotrace

# 获取详细内容
echo '{"tool":"get_workstream_summaries_details","args":{"identifiers":["<id1>","<id2>"]}}' | ftc call cotrace
```

如果 Cotrace 不可用（`ftc` 找不到、health check 失败或无数据），退回到直接询问用户。

## 层级契约

- `[主线]`：顶层长期工作主线
- `[子线]`：主线下或子线下的结构任务（支持多级）
- `[记录]`：已完成工作的记录任务（本文档负责创建的对象）

## 确定挂载点

先用 `check attachable` 找到合适的挂载候选：

```bash
auwomo task check attachable --format json
```

输出中 `attachable=true` 的候选按优先级排序（深度越深越优先）。

### 挂载优先级

1. 最细、最贴近当前工作、且合法可挂载的 `[子线]`
2. 若没有更细合法 `[子线]`，回退到上层 `[子线]`
3. 若仍没有，再回退到合法 `[主线]`
4. 如果没有任何合法候选 → 引导创建主线（见上方"无匹配候选时"）

## 创建记录

```bash
auwomo task record --work <parent_guid> --summary "完成了数据清洗工作" --desc "处理了3个数据源的缺失值和异常值，输出清洗后的特征矩阵"
```

- `--work`：父任务的 GUID（从 check attachable 结果中选取）
- `--summary`：记录标题（简洁，一句话）
- `--desc`：补充细节（可选，1-2 句，比标题多一点具体信息）

CLI 会自动：
- 添加 `[记录]` 前缀（如果你没加）
- 以 bot 身份创建
- 创建时标记为已完成状态

### 何时加 --desc

- 标题已经足够清楚 → 不加（大多数情况）
- 涉及具体产出物、数据量、关键决策 → 加 1-2 句补充
- 来自 Cotrace 且有丰富细节 → 提取核心补充

示例：
```bash
# 标题够了，不需要 desc
auwomo task record --work <guid> --summary "配置了 kaiwen 的 bot 环境"

# 有值得记录的细节
auwomo task record --work <guid> --summary "完成实验数据预处理" --desc "使用 PCA 降维到32维，去除了15%的噪声样本，最终数据集 12000 条"
```

## 草案撰写原则

- 简洁明了，一两句话概括做了什么
- 用结果导向的语言，不要写过程细节
- 如果来自 Cotrace 数据，提取核心成果而非罗列所有活动
- 示例："完成实验数据预处理和特征提取"，而非"打开了 Python，写了一些代码..."

## 错误处理

- `--work` 为空 → 必须指定父任务 GUID
- `--summary` 为空 → 必须填写记录内容
- 无可挂载候选 → 引导走 task-create 流程
- 记录命令失败 → 告知用户具体错误原因

## 不要这样做

- 不要跳过 `check attachable` 直接猜挂载点
- 不要把记录挂在树外面（没有父任务的孤立记录）
- 不要把 Cotrace 原始数据大段贴为记录内容
- 不要只写本地文件就告诉用户"已记录"——必须执行 `auwomo task record`
