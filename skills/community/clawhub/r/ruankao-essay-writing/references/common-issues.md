# 论文常见问题及修正建议

以下问题按严重程度排列，前 6 条为致命或高频失分项，必须逐一排除。

---

## 致命/高频问题

### 1. 走题

**表现**：看到试题标题，不读三个问题就按三段论写论文，导致内容与考查侧重点不匹配。或者看到熟悉主题就默写事先准备好的论文，不管是否切合题意。

**修正**：
- 必须逐一回答试题提出的 3 个问题，**圈出每个要点**
- 同一主题，问题不同则组织内容的方式完全不同
- 用试题的 3 个问题作为论文的 3 个核心段落/部分
- **特别注意第二个子问题**，必须每个要点都回答到位，避免不必要的丢分

### 2. 遗漏子问题要点

**表现**：只回答了子问题的部分要点，遗漏了关键内容。

**修正**：
- 分析试题时逐题圈出所有要点
- 撰写提纲时确保每个要点都有对应的论点或段落
- 检查校对时逐一对照试题要点清单

### 3. 缺乏主题项目

**表现**：笼统地说"我是做银行软件的"，没有具体项目。

**修正**：
- 必须具体说明：某年某月参加的某个具体项目
- 介绍项目背景和功能
- 指明自己在项目中的角色
- 每个论文试题的第一个问题一般就是"简述你参与开发过的项目"

### 4. 字数不足

**表现**：摘要少于 300 字，正文少于 2000 字。

**修正**：
- 摘要目标 350 字以上
- 正文目标 2500-3000 字
- 字数按答题纸格子计数（含标点和图形）

### 5. 缺少特色，泛泛而谈

**表现**：把书刊杂志上的知识点罗列，没有结合项目实例。

**修正**：
- 所有措施（方法、技术）紧密结合主题项目
- 阐述措施时以主题项目中的具体内容为例
- 讲"我在某项目中是如何做的"，而非讲理论
- 用具体的案例支撑论述，而非泛泛而谈

### 6. 摘要归纳欠妥

**表现**：摘要没有实质性内容，或加了太多帽子性语句。

**修正**：
- 标准："读者不看正文，就知道文章的全部内容"
- 摘要应简单包含正文的重点词句
- 尽量不加帽子性语句，直接"压缩"正文内容

## 中频问题

### 7. 项目年代久远

**修正**：主题项目应是近 2-3 年内完成的，确保反映当前技术趋势。

### 8. 口语化太重，表达不够书面化

**表现**：
- 使用口语化的短句堆砌，缺乏完整的句子结构
- 频繁使用"然后""所以""就是说""反正"等口语连接词
- 大量使用"我做了""我搞了""我们弄了"等动词短语
- 出现感叹句、反问句等非正式句式

**修正**：
- 使用书面语替代口语词汇，如"处理→处置""搞→实现""弄→处理""搞定→完成"
- 采用名词化结构：将"我们重构了系统"改为"对系统进行了架构重构"
- 使用完整句式，避免"短句轰炸"，适当增加从句和修饰语
- 使用技术术语体现专业性，如"实现→实施""解决→处置""问题→挑战/瓶颈"
- 引入概念时使用"所谓……是指……""即""通常而言"等书面引导词
- 避免感叹号、省略号、波浪号等非正式标点
- 虽然强调"以我为中心"，但不要到处都是"我"
- 适当用"我们"替代部分"我"，体现项目是集体劳动

**正反示例**：

| ❌ 口语化 | ✅ 书面化 |
|-----------|-----------|
| 我们重构了支付模块，然后上线了。 | 我们对支付模块进行了架构重构，并顺利完成了上线部署。 |
| 这个系统很牛，处理速度很快。 | 该系统的吞吐能力显著提升，平均响应时间从5秒降低至0.8秒。 |
| 我们一开始没想到会有这么多问题。 | 在项目初期，我们并未充分预见到如此复杂的技术挑战。 |
| 后来我们又加了个缓存。 | 在此基础上，我们引入了Redis缓存层以缓解数据库访问压力。 |

### 9. 忠实于论点

**修正**：
- 仔细阅读试题要求，正确理解题意
- 阐述时绝对服从论点
- 不要节外生枝，不要偏离论点
- 半天讲不到点子再草草收场是常见失败模式

### 10. 文章结构不清晰，段落太长

**修正**：
- 每个自然段不超过 8 行
- 如果调整段落，很多不合格文章可以变成好文章
- 加上小标题，使结构更清晰
- 中心论点和各分论点的首句加粗高亮显示

### 11. 整篇文章数字条目太多

**修正**：
- 可以用数字标识顺序使文章有条理
- 但全文充满数字条目则太死板，影响得分
- 适当使用自然段落过渡

### 12. 文章深度不够

**表现**：罗列太多措施（方法、技术），每个都蜻蜓点水。

**修正**：
- 选择 5-6 个有特色的措施准备，考试时动态删减至 2-3 个深入展开
- 每个措施说明：为什么选、怎么做、效果如何
- 不要企图面面俱到

### 13. 逻辑不严谨

**表现**：论证过程中逻辑跳跃或断裂，前后矛盾。

**修正**：
- 撰写提纲时梳理清楚论点之间的逻辑关系
- 使用 SCQA 结构确保每个分论点的逻辑完整
- 检查校对时关注项目细节前后一致性

### 14. 段落转承不自然

**表现**：
- 段落之间缺乏过渡句，读起来有明显的"跳跃感"
- 前后段落内容突兀，逻辑关系不清晰
- 大量使用"然后""接着""还有"等口语化连接词
- 句式单调，连续多个句子以相同主语开头

**修正**：
- 每个段落开头先用一句话点明本段主旨，再展开论述
- 段落之间使用过渡句承上启下：
  - 转折：**"然而""但与此同时""值得注意的是"**
  - 递进：**"在此基础上""更进一步""不仅如此"**
  - 因果：**"基于此""鉴于此""正是出于这样的考虑"**
  - 总结：**"综上所述""总的来看""从上述实践可以看出"**
  - 承接：**"在解决了……问题后""在确定了……之后"**
- 避免连续多句以"我们""系统""该项目"开头，交错使用主语和状语

**示例对比**：

| ❌ 生硬转承 | ✅ 自然转承 |
|-------------|-------------|
| 我们采用了微服务架构。然后我们遇到了分布式事务问题。我们最终采用了Saga模式。 | 在采用微服务架构后，一个随之而来的挑战便是分布式事务的一致性保障。针对这一问题，我们最终引入了Saga模式进行事务补偿。 |
| 系统有性能问题。我们加了缓存。加了缓存后速度快了。 | 随着业务量的持续增长，系统在高并发场景下逐渐暴露出性能瓶颈。**基于此**，我们引入了Redis多级缓存策略。经实测，缓存命中率达到92%，接口响应时间从3秒降至0.5秒。 |

## 低频但需注意的问题

### 14. 字数偏多

**表现**：摘要超 400 字，正文超 4000 字。

**修正**：
- 实际考试时间有限，不可能写太长
- 平时练习严格按考试时间约束写作
- 摘要控制在 300-400 字，正文控制在 2500-3000 字

### 15. 文字表达能力差

**修正**：
- 平时多读文章，多写文档
- 注意措辞的准确性和连贯性

### 16. 缺乏独特见解

**表现**：全篇没有自己的思考和体会，只是按部就班地叙述。

**修正**：
- 在"总结与感悟"部分加入对架构权衡的思考
- 表达对技术选型的反思、对沟通协作的体会
- 体现"架构师的敬畏之心"——没有银弹，只有权衡

### 17. 项目选择不当

**表现**：选择了小型系统、过时技术项目、纯硬件项目或纯技术项目。

**修正**：
- 选择具备业务复杂度和技术复杂度的项目
- 项目应体现当前信息化建设潮流（数字化转型、智能化升级、AI应用等）
- 避免功能单一、规模小的系统

## 检查校对完整清单

完成论文后，按以下清单逐项检查：

| # | 检查项 | 检查方法 | ✓ |
|---|--------|----------|---|
| 1 | 完整回应试题 | 是否逐一回答了试题的所有子问题？每个要点是否都回答到位？ | |
| 2 | 走题检查 | 内容是否围绕试题的考查侧重点？是否偏离了论点？ | |
| 3 | 字数检查 | 摘要 ≥300 字且 ≤400 字？正文 ≥2500 字且 ≤4000 字？ | |
| 4 | 摘要质量 | 不看正文，仅凭摘要能否知道全文内容？ | |
| 5 | 结构完整 | 是否符合金字塔原理？是否加了小标题？中心论点和分论点首句是否加粗高亮？ | |
| 6 | 逻辑严谨 | 论证是否环环相扣？是否有逻辑跳跃或断裂？ | |
| 7 | 深度足够 | 选了 5-6 个技术点准备？最终呈现 2-3 个深入展开？还是蜻蜓点水？ | |
| 8 | 具体案例 | 是否用具体的案例支撑论述？还是泛泛而谈罗列理论？ | |
| 9 | 项目一致性 | 项目细节是否前后一致？有无矛盾之处？ | |
| 10 | 独特见解 | 是否有自己独特的见解和体会？ | |
| 11 | 书面化程度 | 句式是否完整规范？是否仍有口语化表达？是否使用了技术术语？ | |
| 12 | 缺乏主题项目 | 是否具体说明了某年某月的某个项目及角色？ | |
| 13 | 项目时效 | 项目是否在近 2-3 年内？ | |
| 14 | 结构死板 | 是否全篇都是数字条目？段落是否过长（>8行）？ | |
| 15 | 段落转承自然 | 段落之间是否有过渡句承上启下？读起来是否顺畅？主语是否单调重复？ | |

---

## 额外提醒

实践部分的内容应该根据论文题目的理论要点进行描述，重点在于描述理论知识要点在项目中的应用，**而不是介绍项目本身功能是什么**。
