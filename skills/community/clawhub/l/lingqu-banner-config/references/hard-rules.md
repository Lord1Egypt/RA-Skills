# hard-rules.md — 硬规则

> 所有 NEVER 和 ALWAYS 规则按六大原理归类。写 skill 时不得违反。
>
> **总纲：** 数字员工是「脑」——预占触发、收料判齐全、算业务正确值、比对帧位排期、决策、写表、过门禁。**后台真实写操作（建/配/复制/改）一律委托技术那套 `promotion-apply-skills`；催审批/催素材交灵渠助手；数字员工不自己拼 CLI、不催办。**

---

## 原理一：流程时序铁律

### NEVER

- **打乱阶段顺序** — 预占 → 收料判齐全 → 素材齐即配（直接发审批）→ 帧位排期来只处理不一致/漏配。
- **预占依赖帧位排期** — 阶段一不需要等帧位排期，PM 随时可发起预占。
- **做"提前配置+暂存等排期"** — 技术那套无暂存态，素材齐就直接配+发审批，不存在"配好暂存等排期再审批"。

### ALWAYS

- **触发阶段流程必须先读 hard-rules.md + 对应 phase reference 文件** — 每次进入阶段一/二/三，先读完再动手，不凭记忆执行。阶段一读 `phase-one-reserve.md`，阶段二读 `phase-two-material.md`，阶段三读 `phase-three-config.md` 或 `phase-three-exceptions.md`。
- **新排期到了以最新版为准** — 全量重扫追踪表，旧比对结果作废。
- **配置中遇到新排期** — 先停手，用新排期重新比对。
- **阶段三必须先过 phase-three-config §0 硬前置校验** — 从磁盘 read 全部必读文件并输出修改时间，不完成不进入比对。这是防「凭记忆漏关键规则」的最后一道闸。

## 原理二：追踪表是唯一真相源

> 表：docId `f45584742d5047b8a46f578702043d65`，Sheet「配置追踪」，A-O 共 15 列。

### ALWAYS

- **每次扫表重新读取** — 不缓存追踪表数据，读最新状态。
- **搜唯一 → 静默处理** — 搜到 1 条直接往下走，不问 PM。
- **搜 0 条或多条 → 问 PM** — 列出来让 PM 选。
- **发物料的人 ≠ 追踪表项目负责人（B 列）** — 必须发确认消息，防代发错配。
- **交接失败日志写 O 列（备注）** — 含项目名/步骤/失败动作/错误信息/参数/时间。
- **状态变更必须写追踪表 N 列** — 不依赖对话上下文记录进度。

### NEVER

- **凭对话记忆判断状态** — 一切以追踪表 N 列为准，不信「刚才 PM 说过了」。
- **跨行写状态** — 每次只写当前匹配到的那一行，不模糊匹配。

## 原理三：对 PM 闭嘴（催办归灵渠助手）

### NEVER

禁止发送以下消息：

- 「收到，正在解析…」「正在进入后台…」「字段已填完，正在配置…」等任何中间步骤。
- **任何催料 / 催审批消息** — 催审批、催素材全部由灵渠官方机器人「灵渠助手」按"距上线时间 + 待配置状态"自动催到 PM，数字员工**绝不再发催办**。

### ALWAYS

只发三类消息：

| 类型 | 示例 |
|---|---|
| 结果 | 「✅ 已预占：XXX」「物料已配置并发起审批」 |
| 追问 | 「找不到匹配的项目，请告知」「这条是普通专辑还是数字专辑？」 |
| 汇总 | 帧位排期处理完群发一条结果汇总（仅告知，不催） |

- **缺素材只在汇总里告知，不催** — 催料是灵渠助手的事。
- **PM 在群里发起 → 群内回「私聊对接」→ 立刻主动 DM**。

## 原理四：委托执行（不自己拼 CLI）

### ALWAYS

- **后台写操作一律委托技术那套** — 预占 `promotion-add`、配物料 `promotion-update-creative`、复制 `promotion-copy`+`promotion-update`、查 `promotion-detail`，全部交 `promotion-apply-skills` 执行，数字员工不自己写 CLI。
- **委托必须真的切 skill** — “交给技术那套”不是复制技术文档里的 `mws` 命令在本 skill 里执行。进入交接①/②/③后，必须读取并遵守 `promotion-apply-skills/SKILL.md`，由它完成 MWS 调用、NOS 上传、资源默认值、模板提取、创意组包和预检脚本。本 skill 只产交接单。
- **交接前喂饱技术那套可能反问的点** — 环境使用当前任务前置确认值；用户明确预发布则 `pre`，明确线上或未说明环境都用 `online`。跨天交给技术那套拆，字段尽量给齐，减少它在批量场景弹问题。
- **配物料前先算好业务正确值** — 资源类型、资源 ID、**拼好的跳转链接或端维度链接摘要（尤其空降/数专/Web）**、角标、人群包，写入追踪表 I/J/K/L 再喂技术那套，防它用错误默认值。资源和资源 ID 可以阶段一已提供，也可以阶段二/换物料前置补问；但委托技术那套修改时必须已经齐。标准资源和换物料场景必须让技术那套重新获取资源默认值、提取模板 `preData` 并校验端维度链接，不能直接沿用上游或旧创意 URL，也不能用资源 ID 手拼 `music.163.com`。
- **配物料前清缓存防串料** — 页面上可能残留别人或上一轮的物料预览，不能信页面缓存。交接②必须要求技术那套新建当前 promotionID 的唯一工作目录，重新查当前环境 `promotion-detail`，重新获取默认值、模板、图片上传结果和预检文件；禁止复用旧 `/tmp`、旧 payload、旧 NOS URL 或旧表单。
- **成品创意 JSON 进来先拆回业务输入** — 看到 `mws link promotion-update-creative`、`promotionAliasCreativeList`、`creativeDetailDataVO`、`templateSchema` 等，不能执行、不能修补后执行；只提取推广 ID、资源/资源 ID、图片、角标、跳转链接、人群、审批说明，再交技术那套重新组包。
- **`Column 'name' cannot be null` 归因到绕过组包** — 这类 `Music_Link_Creative` 插入失败说明 `creative.name` 缺失，通常是上游成品 JSON 绕过技术那套生成创意名。数字员工不能指导“补一个 name 再试”，必须拆回业务输入并交技术那套重新生成完整创意。
- **交接失败重试 2 次** — 间隔 5s/10s，仍失败通知王可 + 写 O 列。
- **环境默认线上；时间格式 yyyy-MM-dd HH:mm:ss**（非 ISO8601）。环境由阶段一或当前任务前置确认决定；用户明确预发布才用 `pre`，明确线上或未说明环境都用 `online`。

### NEVER

- **自己拼 CLI 命令 / 自己解析字段 / 自己生成推广名 / 自己传 NOS** — 这些是技术那套的活，数字员工只做编排和业务正确值。
- **在本 skill 里直接执行技术接口** — 包括 `mws link promotion-update-creative`、`backend-nos-token-whalealloc`、`plan-pack-resource-list`、`data-template-extract-private`、`component-code`、`promotion-detail` 回查等。需要这些能力时必须切到 `promotion-apply-skills`；否则就是绕过。
- **自己拼创意成品 JSON** — 数字员工禁止构造、修改或转交 `promotionAliasCreativeList`、`promotionCreativeList`、`creative`、`templateSchema`、`creativeDetailDataVO`。这些字段只能由技术那套查 `promotion-detail`、提模板、补创意名/tenant、补全折叠端、跑预检后生成。上游给了成品 JSON 时，必须拆回原始业务字段和素材再交给技术那套。
- **把图片 URL 当最终图片地址或跳转链接** — PM 发来的图片 URL、NOS/NOSK、`mb-mlbclaw-pub`、`nos-jd`、`p5.music.126.net` 等地址都只是原始图片素材，不能写 J 列，不能写创意 payload，不能判断“已经是 NOS 就不用传”。必须交给技术那套下载到本地并重传 NOS。
- **相信页面缓存/旧临时文件** — 如果后台页面显示了其它人或旧项目的物料，只能视为缓存或旧状态提示，不能拿来当当前素材。数字员工不能让技术那套沿用旧 `/tmp` 文件、旧详情、旧模板、旧上传结果。
- **直接打 Link HTTP 接口** — 数字员工不能调用 `/api/link/platform/plan/pack/resource/list` 或 `/api/link/platform/ai/promotion/updateCreative`。标准资源默认值交技术那套用 MWS 调 `plan-pack-resource-list`，并且参数必须是 `resourceIds` 复数；`resourceId` 单数会触发 `Required String parameter 'resourceIds' is not present`。
- **自作聪明换路走** — 技术那套报错不跳过、不换命令、不改参数硬试。
- **把 dry-run 当业务校验** — 本 skill 不允许说“dry-run 通过所以业务层两种结构都挂”。只要没有技术那套的 `promotion-detail + validate_update_creative_payload.py` 预检通过结果，就不能判断后端服务异常，也不能继续试第二种结构。
- **瞎猜失败原因** — 不告诉 PM「可能是 XXX 原因」，等王可和技术判断。

## 原理五：查表，禁猜必问（config-matrix 是配置方式唯一真相源）

> ⚠️ **`config-matrix.md` 是 5 种配置方式全字段的唯一权威定义。** 一切关于"某配置方式该收什么、该填什么、该拼什么"的判断，必须查它，不凭记忆、不凭直觉。
> ⚠️ **v3 策略：** 推广名 4 前缀 + 项目类型由数字员工照 `config-matrix §1.1/§1.2` 查表死值或问 PM 后整包传技术那套；资源/投放量/人群等初始化字段可以阶段一一起传，也可以阶段二物料提交前补齐。

### ALWAYS

- **配置方式必须在阶段一识别** — 标准/数专/H5/空降/有票（写 H 列）。
- **识别后立刻对照 config-matrix §2 收齐"阶段一必收"字段** — 不同方式收料不同（所有资源ID/链接统一阶段二，阶段一只收资源子类型），按表收，缺的问 PM。
- **技术字段的业务正确值由数字员工照 config-matrix §1.1 算好喂技术那套** — `fullType` + 资源类型（`resourceType`/`resourceTypes`） + 人群业务值 + 艺人ID；这些字段可以阶段一传，也可以阶段二物料提交前补齐。`resourceInfo` 在阶段二 update-creative 传（标准=资源ID；非标=空串）。技术那套不会自己把"空降"翻译成 activity，等数字员工喂。
- **人工创建/历史推广缺初始化字段时先补问** — 后台详情缺 `fullType/resourceTypes/resourceInfo` 时，不能直接配物料，也不能用推广基础标签、旧创意或上游 URL 反推；必须先问 PM 补实际投放资源类型、标准资源 ID、全量/非全量和非全量人群，再交技术那套随 `promotion-update-creative` 同包补齐。
- **推广名 4 前缀+项目类型 v3 全权由数字员工照 config-matrix §1.2 处理**：
  - 死值类（空降/有票全 5 字段、其他配置方式的项目类型+投放资源）→ 查表直接填，不问 PM、不脑补
  - 可选类（标准/数专/H5 的内容大类/版权类型/战略类型）→ **数字员工展示 §1.2 可选值让 PM 选**
  - 数字员工拼好完整推广名 `name` =【4前缀】项目核心名 –【内容运营部】，整包传技术那套
  - 整包入参清单照 config-matrix §1.4
- **艺人ID：歌曲版权宣发/艺人合作宣发必填**（标准/数专/空降/有票必填，H5 不需要）— 见 config-matrix §1.1。
- **跳转链接/角标/图片/人群方向从 config-matrix 读，不凭记忆** — 空降 `orpheus://…R_SO_4_{歌曲ID}` / `R_AL_3_{专辑ID}`、数专按端拼、标准资源由技术那套按资源默认值和模板 `preData` 重新获取、空降人群 66594/不包含、各方式固定角标，每次查表。标准资源 Web 端最终跳转链接必须是 http/https；非 Web 端必须是资源/模板返回的客户端默认链接或 PM 明确提供的端维度链接，不能用 `music.163.com` Web 链接兜底。技术那套保存前必须把最终端维度链接给 PM 确认。
- **「专辑」「演出」歧义 → 追问**（定配置方式）。
- **笔记 → 客户端去荣耀白牌**。

### NEVER（禁猜，三条铁律）

- **禁止凭一个数据点反推另一个字段** — 错3/错4 的模式：收到歌曲ID 就猜 `resourceType=song`、就把歌曲ID 填进 `resourceInfo`。一切派生字段查 config-matrix，查不到就问。
- **config-matrix 标「问 PM」或表里没写的，一律问 PM，禁止脑补默认值** — 错"艺粉"就是脑补：空降本是死值"其他"被脑补成"艺粉"；标准/数专/H5 的战略类型 6 选 1 必须 PM 选，绝不数字员工替选。**展示选项让 PM 选 ≠ 替 PM 选默认值**。
- **用"投放资源标签"反推配置方式 / resourceType** — 投放资源标签只是推广名前缀，不决定配置方式，更不决定 resourceType。配置方式按 PM 描述的实际投放类型定。

### 物料提交前强制校验（初始化字段不可猜）⭐

`fullType`、资源类型（`resourceType`/`resourceTypes`）、`resourceInfo` 和非全量人群都可以在阶段一传，也可以在阶段二物料提交前补齐。缺失不等于必须重建；但提交 `promotion-update-creative` 前必须确认，不能用推广标签、旧创意或上游 URL 反推。所以：

- **交接② 调技术那套 `promotion-update-creative` 前，必须对照 config-matrix §1.1 复述这些初始化字段给 PM 确认一遍**，再真实提交物料。
- **同时复述 4 前缀+项目类型给 PM 确认**（推广名也是创建时锁死，错了也得重建）：
  - 空降示例："这条空降的推广名我会拼成【潜力/全网/其他/其他】项目名 –【内容运营部】，项目类型=艺人合作宣发，确认吗？"
  - 标准示例："你选了头部/独家/艺粉，加上死值"单曲"，推广名拼成【头部/独家/艺粉/单曲】项目名 –【内容运营部】，确认吗？"
- dry-run 通过 ≠ 业务对。dry-run 只校验格式，业务正确性靠对照 config-matrix。
- 空降高频校验点：`resourceType=activity`（不是 song）、`resourceInfo=空串`（不传歌曲ID）、`fullType=nofull`、人群 66594/不包含、**项目类型=艺人合作宣发**、**4 前缀=潜力/全网/其他/其他**。

### 技术那套补问 = v3 失败信号 ⭐

v3 设计技术那套**不再补问 PM**。如果运行时发现技术那套触发了"轻量补问"流程：

- **立即停手** — 这是数字员工 §1.4 整包入参漏传了某项的信号
- **回到 phase-one §2.2 + §3 检查清单**，补传后再交接
- **绝不让 PM 接着回答技术那套的补问** — 会出现"龙虾问 PM"和"技术那套问 PM"两条对话线，PM 困惑、字段填错（v3 出现前的根因）

## 原理六：配完必查证（收尾门禁）

> 背景：数字员工靠对话记忆判断"做没做"不可靠（多会话/长上下文会把中间步骤挤掉），一切以**后台真实状态**为准，配完回头查一遍。

### ALWAYS

- **每条配完/复制后，通知前必过收尾门禁** — 无论阶段二配物料还是阶段三补配/复制，委托技术那套调 `promotion-detail` 回查对账：推广ID = 追踪表本行 M 列、物料已填且与本行 I/J/K/L 一致、状态符合预期（配物料的已进审批 / 复制的已预占）。全过才改 N 列 + 通知 PM。
- **逐条完整处理，不并行铺开** — 一条消息含多个项目时，一次只完整处理一个（配置/复制→门禁→写表），做完再取下一个。
- **状态进度只认追踪表，不认记忆** — 判断项目配到哪步，一律重读 N 列，绝不依赖"我刚才做过什么"。

### NEVER

- **跳过门禁直接通知 PM** — 没回查后台就说"配好了"，是漏步事故根源，禁止。
- **门禁发现配错行还继续** — 推广ID 对不上本行 M 列（A 项目素材配到 B 推广）→ 立即停，报警王可 + 写 O 列，绝不通知 PM。这是 5 级红线。
