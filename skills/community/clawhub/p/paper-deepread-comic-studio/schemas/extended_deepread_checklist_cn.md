# 扩展精读检查清单（增强版）

在原始 deep-read 基础上，生成每篇论文报告前，至少逐项核对：

1. 是否单独解释了标题中的关键词，以及标题与正文的映射。
2. 是否说明了论文真正解决的问题，以及连续的上位问题阶梯。
3. 是否把论文中反复提到的关键文献拆开讲，并指出它们遗留的空白。
4. 是否逐条核对了创新点和主张，并标出证据类型与强度。
5. 是否倒推了作者的研究思路来源，并区分证据与推测。
6. 是否解释了符号、定义、设定和核心概念。
7. 是否完整保留了关键公式，并逐式解释每个公式项、作用与上下文。
8. 如果有证明，是否把定理 / 引理 / 假设 / 结论与实际算法步骤或实现流程对应起来，并说明二者是严格一致、局部近似，还是仅提供动机。
9. 是否不仅解释了公式和模块“在做什么”，还判断了关键公式、关键模块、关键假设是否存在脆弱点、欠论证之处、优化困难、计算代价问题或可替代设计空间。
10. 是否给出了针对这些公式 / 模块 / 假设的具体改进空间、替代方案与潜在 trade-off。
11. 是否给出了带具体例子的算法流程，而不是只写公式。
12. 是否把实验中的 baseline 与引言 / 相关工作中的方法谱系对应起来。
13. 是否说明了每个实验块要回答什么问题、支撑什么主张。
14. 是否指出了有趣、反常或有争议的实验现象。
15. 是否借鉴热门 GitHub 审稿 skill 的 reviewer 关注点，额外检查了 novelty、significance、soundness、methodology rigor、reproducibility、results-claims alignment、missing baselines/controls、figure/table clarity、limitations honesty 等维度。
16. 如果有 LaTeX 源，是否基于 figure 环境、caption、label、引用正文和图片路径解释了关键图，而不是因为不是 PDF 就跳过图。
17. 如果有 PDF 源，是否也显式解释了关键图（结构图、流程图、定性图、实验曲线图等），而不是只关注文字和公式。
18. 是否对实验中的关键图表、曲线、表格和数据做了解释，并明确它们各自支撑哪个主张。
19. 是否指出了实验图表 / 数据与作者主张之间不一致、部分一致或解释不足的地方，并分析了可能原因。
20. 是否判断了创新是增量、交叉、边界突破，还是混合型。
21. 是否讨论了是否突破学科边界，以及边界突破的限度。
22. 是否明确讨论了论文的不足、未解决问题、失败模式、适用范围边界，以及哪些局限是结构性的。
23. 是否区分了普通后续工作与真正可能推动科学边界的研究方向。
24. 是否提出了新的研究方向，包括原生延展、交叉迁移、混合创新，以及高风险高回报的边界型方向。
25. 是否用一个通俗、生动的故事收尾。
26. 是否优先保证完整性，而不是人为压缩篇幅。

27. 在介绍 idea、理论证明、具体算法或模块时，是否识别并说明了哪些部分可能体现作者的主观判断、经验偏好、启发式选择、研究风格或工程取舍，并区分客观必要性与作者选择。
28. 全文是否结合论文本身的具体公式、模块名、假设、数据集、baseline、图表、定理对象、实验现象和措辞展开，而不是停留在空泛抽象层面。

29. 是否抽取了论文的 research equation：重要设定 + 被打破假设 + 借用工具 + 新约束 + 替代机制。
30. 是否明确指出了 easier setting 中可用但 target setting 中不可用的理想机制 `Y`，以及论文构造的替代机制 `Z`。
31. 是否重建了作者可能如何发现该方向：valuable field、painful assumption、emerging / borrowed tool、unserved setting、blocking constraint、conceptual replacement。
32. 是否建立了 challenge -> failure mode -> design principle -> module -> evidence 的故事桥接表。
33. 是否判断方法是模块堆叠还是闭环贡献，并指出在模块间流动的核心对象。
34. 是否对每个核心模块使用 Failure + Unavailable Ideal + Available Proxy + Design Choice + Assumption + Risk 模板进行作者思路深读。
35. 是否把关键引用解释为叙事功能，而不是只做背景文献罗列，并形成 Citation-to-Module Map。
36. 是否把每个实验解释为 claim + counterfactual + metric + stress condition，并判断实验是否真的支撑故事链条。
37. 是否抽取出可复用的论文造故事模式，如 replacement story、two-axis empty cell、closed-loop contribution。
38. 是否把隐藏假设 `H_i` 系统转化为 `¬H_i` 下的新研究问题，并区分普通工程延展与可能推动科学边界的方向。
39. 是否避免泛泛总结，把关键方法解释到读者能够复现、答辩、讲给别人听。
40. 是否所有关键概念都按“直觉 -> 数学公式或 formal definition -> 具体例子 -> 局限 / failure case”展开。
41. 是否复杂模块都说明了输入、输出、符号、维度、训练参数、固定超参数、训练数据流和推理数据流。
42. 是否严格区分“论文明确写明”“论文没写但合理推测”“参考相近工作推测”“缺失 / 未报告”。
43. 是否显式列出缺失的硬件、耗时、内存、优化器、学习率、随机种子、早停、超参搜索、baseline 来源、baseline 是否重跑或重构、数据划分、标签定义、代码和计算预算。
44. 是否实验部分足够细：数据集规模、标签定义、划分协议、baseline 谱系、指标含义、主结果例外、消融目的、定性证据和复现风险都已说明。
45. 是否最后提供了一个完整数字例子，把训练和推理串起来，并标注哪些部分是论文明确给出的，哪些只是教学用 toy simplification。
46. 是否整体讲解顺序体现知识依赖：先符号和设定，再数据和输入，再模型模块，再训练与推理，最后实验、不足和复现风险。
47. 若生成生图提示，是否要求“多张连续的卡通图”，并明确拆成多张 separate 16:9 images/cards，而不是压缩到一张图里。
48. 若生成连续漫画提示，是否包含运镜计划、风格 bible、角色/色彩/字体/编号一致性、与前序图片的连续性，以及源文和精读报告的事实核对。
49. 若是后续部分生图，是否明确复用前面已生成图片的视觉设定和叙事状态；若前图不在上下文，是否只复用非事实视觉风格而不编造论文内容。
50. 是否在提示用户下一步怎么说时，显式提醒用户使用“生成多张连续的卡通图”这句话。
