# 静默策略 (Reminder Policy)

SkillRouter 应该在何时介入，何时保持静默。

---

## 核心原则

**SkillRouter 的成功在于减少认知负荷，而非增加。**

如果让任务感觉更重，就是路由失败。

---

## 何时介入

### 高价值场景 (应该介入)

1. **用户明确询问技能存在性**
   - "我们有处理 XX 的技能吗？"
   - "是不是已经装了 XX 相关的技能？"

2. **用户询问技能选择**
   - "应该用哪个技能？"
   - "这几个技能哪个更合适？"

3. **多个技能可能适用**
   - 需要帮助选择最优
   - 避免浪费时间在错误的技能上

4. **已安装技能容易被遗忘**
   - 能明显减少重复工作
   - 用户可能不知道已安装

5. **技能发现/安装任务**
   - 用户明确要求找新技能
   - 先检查已安装技能是否已足够

---

## 何时静默

### 低价值场景 (应该静默)

1. **默认触发路径已足够**
   - 任务简单明确
   - 技能选择不需要有意义的决策

2. **正常技能触发已解决**
   - 默认流程已干净解决任务
   - SkillRouter 不增加价值

3. **用户未要求技能建议**
   - 直接执行任务即可
   - 不需要暴露技能选择过程

4. **技能选择是琐碎的**
   - 只有一个明显合适的技能
   - 不需要建议

---

## 输出噪音控制

### 良好输出 (低噪音)

✅ "这是中文搜索，用 `multi-search-engine`。"
✅ "微信文章有反爬，用 `scrapling-fetch`。"
✅ "已安装技能已足够，无需安装新的。"

### 不良输出 (高噪音)

❌ "让我分析一下你的需求...根据能力分类体系，这属于 fetch.capability.anti_bot 类别..."
❌ "你有以下技能可能适用：1. xxx 2. xxx 3. xxx..." (完整列表)
❌ "根据路由决策算法，我推荐..." (暴露框架语言)

---

## 压缩规则

**如果好的路由回答能用 1-2 句话说完，就停在那里。**

- 不展开框架语言
- 不暴露能力分类术语
- 不解释路由理论

除非用户明确要求设计细节。

---

## 本地偏好

本地环境可配置介入阈值：

```yaml
# 静默配置示例
reminder_policy:
  mode: "balanced"  # quiet/balanced/verbose
  
  #  quiet: 只在明确要求时介入
  #  balanced: 高价值场景介入 (默认)
  #  verbose: 更多建议 (调试用)
  
  # 特定能力类别的介入策略
  capabilities:
    search: "quiet"      # 搜索通常静默
    fetch: "balanced"    # 抓取根据情况
    skill-discovery: "verbose"  # 发现时更多建议
```

---

## 决策检查清单

在决定是否介入前，问自己：

1. 用户明确要求技能建议吗？
   - 是 → 介入
   - 否 → 继续检查

2. 技能选择是有意义的决策吗？
   - 是 → 可能介入
   - 否 → 静默

3. 已安装技能能明显减少工作吗？
   - 是 → 介入提醒
   - 否 → 继续检查

4. 默认流程已足够解决吗？
   - 是 → 静默
   - 否 → 介入
