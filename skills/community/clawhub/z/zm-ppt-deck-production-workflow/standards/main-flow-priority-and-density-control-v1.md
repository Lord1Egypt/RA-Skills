# 主流程优先与信息密度控制标准 v1

## 背景

用户反馈：P2/P3 虽然内容饱满，但信息太多、表达点太散，影响主流程。这类问题必须在内容大纲、页面合同和审核阶段提前识别。

## 核心原则

高密度不等于堆信息。

真正的高密度是：

- 主流程更清楚；
- 支撑信息更有层级；
- 观众一眼知道这一页要服务哪一个主动作；
- 细节为主线服务，而不是和主线抢注意力。

## 最高规则

每一页必须先判断：

```text
这页服务主流程的哪一步？
```

如果页面里的信息影响主流程推进，即使内容正确、美感通过，也必须判定为：

```text
NEEDS_FLOW_REDUCTION
```

## 页面信息分层

每页内容分为三层：

### A. 主流程信息

必须被观众第一眼看到，最多 1 条。

例如：
- 四类能力协作；
- 审核决定交付；
- 内容锁定后再进入视觉。

### B. 支撑信息

用于解释主流程，建议 3 个以内，最多 4 个。

### C. 细节信息

只给讲师口述或放备注，不建议全部放页面。

包括：
- 工具名；
- token；
- file_key；
- 过细的审核项；
- 内部流程术语；
- 多层嵌套解释。

## 信息密度门禁

以下情况判为 NEEDS_FLOW_REDUCTION：

1. 页面里同时出现 2 个以上主流程；
2. 页面有太多模块，观众不知道先看哪里；
3. 支撑点超过 4 个且没有明显主次；
4. 细节文字抢了主结论；
5. 工具名、内部术语、流程名过多，影响对外表达；
6. 右侧视觉、底部流程、正文卡片都在表达不同东西；
7. 页面看起来“全都对”，但讲起来会打断主线。

## 高密度正确做法

高密度页面应采用：

```text
1 个主结论
+ 3 个支撑点
+ 1 条轻流程 / 关系线
+ 细节放讲师口述
```

## P2/P3 修正方向

### P2

原问题：四类能力、角色说明、流程链路同时出现，信息偏多。

修正为：

- 主流程：一份 PPT 不是一个模型完成，而是四层协作。
- 页面只保留 4 个角色名 + 每个角色一句话。
- 工具名、细节职责放讲师口述或备注。

### P3

原问题：审核项、文件项、上传记录、知识库归档同时展开，影响“审核决定交付”的主线。

修正为：

- 主流程：没有审核记录，不算完成。
- 页面只保留 3 个门禁：内容、视觉、交付。
- 上传记录、file_token、知识库节点作为讲师口述或备注，不在主页面铺开。

## 审核输出要求

审核报告必须新增：

- 主流程是否清晰：PASS / NEEDS_FIX
- 信息是否抢主线：PASS / NEEDS_FLOW_REDUCTION
- 是否需要减法：YES / NO
- 建议删减到哪些内容
