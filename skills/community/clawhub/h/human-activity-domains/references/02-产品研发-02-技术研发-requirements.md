## 技术研发生成要求

技术研发每种任务类型的"零件清单"——必选/可选组件、组装顺序、领域约束。按清单逐项产出。

---

### 1 技术调研

#### T1-1 技术趋势

- **必选组件**: 调研范围、调研方法（文献/专利/行业报告）、技术趋势分析、技术成熟度评估
- **可选组件**: 专家访谈、技术会议、开源社区调研
- **组装顺序**: 确定调研范围 → 选择调研方法 → 收集技术信息 → 分析趋势 → 评估成熟度
- **约束**: 范围必须明确；来源必须权威；分析必须客观
- **格式**: 技术趋势报告（Markdown）

#### T1-2 可行性分析

- **必选组件**: 技术方案描述、可行性评估（技术/资源/时间）、风险识别、可行性结论
- **可选组件**: 原型验证、专家评审、成本分析
- **组装顺序**: 描述技术方案 → 评估可行性 → 识别风险 → 分析成本 → 形成结论
- **约束**: 评估必须全面；风险必须识别；结论必须明确
- **格式**: 可行性分析报告（Markdown）

#### T1-3 技术选型

- **必选组件**: 候选技术方案、选型标准（性能/生态/成本/学习曲线）、对比分析、选型结论
- **可选组件**: 技术POC、专家咨询、社区调研
- **组装顺序**: 列出候选方案 → 确定选型标准 → 对比分析 → 验证评估 → 形成结论
- **约束**: 方案必须有代表性；标准必须合理；对比必须客观
- **格式**: 技术选型报告（Markdown）

#### T1-4 技术规划

- **必选组件**: 研发目标、里程碑计划、资源需求、风险应对
- **可选组件**: 技术路线图、资源分配方案、应急预案
- **组装顺序**: 确定研发目标 → 制定里程碑 → 评估资源需求 → 识别风险 → 制定应对
- **约束**: 目标必须明确；计划必须可行；资源必须充足
- **格式**: 技术规划文档（Markdown）

---

### 2 技术开发

#### T2-1 原型开发

- **必选组件**: 原型目标、技术方案、开发环境、原型代码
- **可选组件**: 开发工具、技术文档、参考实现
- **组装顺序**: 确定原型目标 → 搭建开发环境 → 实现核心功能 → 验证可行性 → 记录经验
- **约束**: 目标必须聚焦；环境必须稳定；代码必须可运行
- **格式**: 原型代码 + 开发说明（Markdown）

#### T2-2 功能实现

- **必选组件**: 功能需求、设计方案、代码实现、单元测试
- **可选组件**: 代码规范、开发工具、代码审查
- **组装顺序**: 理解功能需求 → 设计实现方案 → 编写代码 → 执行单元测试 → 代码审查
- **约束**: 需求必须清晰；代码必须规范；测试必须通过
- **格式**: 代码文件 + 设计文档（Markdown）

#### T2-3 集成测试

- **必选组件**: 集成方案、测试用例、测试环境、测试结果
- **可选组件**: 自动化测试、测试工具、测试报告
- **组装顺序**: 制定集成方案 → 准备测试环境 → 编写测试用例 → 执行测试 → 记录结果
- **约束**: 方案必须完整；环境必须稳定；用例必须覆盖
- **格式**: 集成测试报告（Markdown）

#### T2-4 性能优化

- **必选组件**: 性能指标、性能测试、优化方案、优化效果
- **可选组件**: 性能分析工具、优化工具、基准测试
- **组装顺序**: 确定性能指标 → 执行性能测试 → 分析瓶颈 → 实施优化 → 验证效果
- **约束**: 指标必须可量化；测试必须准确；优化必须有效
- **格式**: 性能优化报告（Markdown）

---

### 3 技术验证

#### T3-1 测试方案

- **必选组件**: 测试范围、测试策略、测试用例、测试计划
- **可选组件**: 测试工具、测试环境、测试资源
- **组装顺序**: 确定测试范围 → 设计测试策略 → 编写测试用例 → 制定测试计划 → 评审确认
- **约束**: 范围必须全面；策略必须合理；用例必须覆盖
- **格式**: 测试方案文档（Markdown）

#### T3-2 测试执行

- **必选组件**: 测试环境、测试数据、测试记录、缺陷报告
- **可选组件**: 自动化测试、测试工具、测试监控
- **组装顺序**: 准备测试环境 → 准备测试数据 → 执行测试 → 记录结果 → 报告缺陷
- **约束**: 环境必须稳定；数据必须真实；记录必须完整
- **格式**: 测试执行报告（Markdown）

#### T3-3 问题修复

- **必选组件**: 问题分析、修复方案、修复代码、验证测试
- **可选组件**: 问题追踪、修复评审、回归测试
- **组装顺序**: 分析问题原因 → 设计修复方案 → 实施修复 → 验证修复 → 回归测试
- **约束**: 分析必须深入；方案必须可行；验证必须充分
- **格式**: 问题修复记录（Markdown）

#### T3-4 质量保证

- **必选组件**: 质量标准、质量检查、质量报告、改进建议
- **可选组件**: 质量工具、质量评审、质量改进行动
- **组装顺序**: 确定质量标准 → 执行质量检查 → 形成质量报告 → 提出改进建议 → 跟踪改进
- **约束**: 标准必须明确；检查必须全面；报告必须客观
- **格式**: 质量保证报告（Markdown）

### 4 技术交付

#### T4-1 文档编制

- **必选组件**: 设计文档、API文档、用户手册、部署指南
- **可选组件**: 开发指南、运维手册、故障排查
- **组装顺序**: 编写设计文档 → 编写API文档 → 编写用户手册 → 编写部署指南 → 评审确认
- **约束**: 文档必须完整；内容必须准确；格式必须规范
- **格式**: 技术文档集（Markdown）

#### T4-2 技术转移

- **必选组件**: 转移计划、转移材料、培训安排、验收标准
- **可选组件**: 转移会议、现场支持、知识库
- **组装顺序**: 制定转移计划 → 准备转移材料 → 安排培训 → 执行转移 → 验收确认
- **约束**: 计划必须可行；材料必须完整；培训必须有效
- **格式**: 技术转移记录（Markdown）

#### T4-3 提供支持

- **必选组件**: 支持渠道、支持流程、支持记录、问题解答
- **可选组件**: 支持工具、支持团队、支持SLA
- **组装顺序**: 建立支持渠道 → 制定支持流程 → 接受问题 → 解答问题 → 记录归档
- **约束**: 渠道必须畅通；流程必须明确；响应必须及时
- **格式**: 技术支持记录（Markdown）

#### T4-4 知识沉淀

- **必选组件**: 知识分类、知识内容、知识库、知识更新
- **可选组件**: 知识平台、知识社区、知识激励
- **组装顺序**: 确定知识分类 → 整理知识内容 → 建立知识库 → 持续更新 → 推广应用
- **约束**: 分类必须清晰；内容必须准确；更新必须及时
- **格式**: 知识库文档（Markdown）
