# SOP 文档解析指南

> 当用户甩来一份 SOP 文档时，按此指南提取信息

---

## 一、支持的文档格式

| 格式 | 处理方式 |
|------|---------|
| PDF | 直接读取，提取文字和表格 |
| Word (.docx) | 直接读取 |
| Excel (.xlsx) | 读取表格结构，识别步骤列/条件列/责任人列 |
| Markdown (.md) | 直接读取 |
| 图片/截图 | 用视觉能力识别文字和流程 |
| 飞书/钉钉文档链接 | 通过 API 或浏览器读取 |
| 粘贴的文字 | 直接处理 |

---

## 二、解析步骤

### Step 1: 提取结构

从文档中识别以下元素：

| 元素 | 识别标志 | 后续处理 |
|------|---------|---------|
| **步骤** | 编号列表（1. 2. 3.）、流程图节点 | → 映射为 Step 1/2/3 |
| **判断条件** | "如果""当...时""满足...条件" | → 标记为决策节点 |
| **分支** | "否则""不满足则""驳回" | → 标记为条件分支 |
| **表格** | 表格结构 | → 提取为结构化数据 |
| **角色** | "由XX负责""XX审批""XX确认" | → 标记涉及角色 |
| **时间要求** | "X个工作日内""当天""每月X日前" | → 标记时效约束 |
| **模板/表单** | "填写XX表""使用XX模板" | → 记录模板名称和位置 |
| **引用** | "参见XX""详见XX文档" | → 标记为待补充信息源 |

### Step 2: 标记模糊点

以下表述必须标记为"待追问"：

| 模糊表述 | 追问方向 |
|---------|---------|
| "视情况而定" | 什么情况？具体判断标准是什么？ |
| "相关部门" | 具体哪个部门？联系人是谁？ |
| "根据要求" | 谁的要求？什么要求？ |
| "及时处理" | 多久算及时？有没有 SLA？ |
| "核实确认" | 核实什么？怎么确认？ |
| "必要的时候" | 什么算必要？判断标准是什么？ |
| "原则上" | 有没有例外？例外怎么处理？ |
| "适当调整" | 调整范围多大？谁来决定？ |
| "沟通协调" | 什么时候需要协调？找谁？ |

### Step 3: 生成流程图（文字版）

```
┌─────────────┐
│  触发条件    │
└──────┬──────┘
       ▼
┌─────────────┐
│  Step 1     │
└──────┬──────┘
       ▼
  ◆ 决策节点A？
  ├─ 是 → Step 2a
  └─ 否 → Step 2b
       ▼
┌─────────────┐
│  Step 3     │
└──────┬──────┘
       ▼
  ◆ 决策节点B？
  ├─ 条件1 → [找XX部门]
  ├─ 条件2 → [直接通过]
  └─ 条件3 → [退回修改]
       ▼
┌─────────────┐
│  完成/输出   │
└─────────────┘
```

### Step 4: 生成提取摘要

```markdown
## 📋 SOP 文档提取摘要

**文档名称**：[文件名]
**提取时间**：[时间]
**工序名称**：[从文档中提取或推断]

### 基本信息
- 触发条件：[提取]
- 涉及角色：[列表]
- 主要步骤：[N]步
- 决策节点：[N]个
- 预计耗时：[提取或标注"未提及"]

### 提取的流程
[流程图]

### 待追问项
1. [模糊点1]：文档说"视情况而定"，需要确认判断标准
2. [模糊点2]：文档提到"相关部门"，需要确认具体部门
3. [缺口1]：文档没有提到XX情况的处理方式
4. [缺口2]：文档没有明确XX的质量标准

### 信息源
- [文档中引用的其他文档/模板]
```

## 三、常见 SOP 文档结构模式

### 模式 A: 线性流程
```
Step 1 → Step 2 → Step 3 → 完成
```
**处理**：直接映射，追问每步的细节和判断标准

### 模式 B: 条件分支
```
Step 1 → 判断 → 分支A / 分支B → 合并 → 完成
```
**处理**：重点追问判断条件的标准

### 模式 C: 审批链
```
发起 → 审批1 → 审批2 → 审批3 → 执行
```
**处理**：追问每个审批节点的关注点、退回条件、超时处理

### 模式 D: 检查清单
```
检查项1 ✓/✗ → 检查项2 ✓/✗ → ... → 汇总 → 处理
```
**处理**：追问每项的检查方法、不合格的处理方式

### 模式 E: 循环迭代
```
准备 → 执行 → 检查 → 不合格 → 返回准备 → ... → 合格 → 完成
```
**处理**：追问循环的退出条件、最大迭代次数

## 四、解析后的追问清单生成

根据提取结果，自动生成追问清单，按优先级排序：

### P0（必须追问——缺失会导致 Skill 无法执行）
- 决策节点的判断标准
- 触发条件和前置条件
- 输出/交付物的具体要求

### P1（应该追问——缺失会导致 Skill 质量下降）
- 边界条件和例外处理
- 跨部门协作流程
- 质量标准和检查方法

### P2（可以追问——补充隐性知识）
- 常见坑和避坑经验
- 优化建议和改进空间
- 历史案例和典型错误
