# 37_自进化反馈机制.md

> **版本**：V3.4 新增 | 2026-06-03
> **目的**：使 GIS 技能包具备自我进化能力——根据使用反馈驱动知识库持续迭代优化
> **触发方式**：用户提问后自动评估 → 知识缺口检测 → 触发增量收录

---

## 一、反馈驱动迭代机制

### 1.1 反馈来源

| 来源 | 内容 | 处理方式 |
|------|------|----------|
| **用户追问** | "不对/错了/不对吧/是这样吗？" | 立即核查 → 修正条目 → 记录修正历史 |
| **用户纠正** | "应该是XXX，不是XXX" | 定位条目 → 验证 → 覆盖/新增 → 升级版本号 |
| **用户补充** | "相关项目里是XXX做法" | 评估普适性 → A/B级入库 → 标注"用户贡献" |
| **用户打分** | "这个回答很好/没用" | 记录到对应条目的质量评分 → 驱动优先级排序 |
| **回答失败** | 无法回答 / 检索无结果 | 记录知识缺口 → 加入下次搜索队列 |

### 1.2 自动化处理流程

```
用户提问
  → 检索知识库（SKILL.md + references/）
  → 命中？
      ├── 是 → 生成回答
      │     → 用户反馈？
      │         ├── 有纠正 → 触发修正协议（§3）
      │         └── 无反馈 → 记录成功案例
      └── 否 → 记录知识缺口（§2）
                  → 达到一定阈值 → 触发增量搜索（§4）
```

## 二、知识缺口检测规则

### 2.1 触发条件

以下条件任意满足时，记录为**知识缺口**：

| # | 触发条件 | 阈值 | 动作 |
|---|----------|------|------|
| K1 | 同一问题连续回答失败 ≥2次 | 2次 | 高优先级缺口 |
| K2 | 用户追问"还有呢/还有别的吗" | 每次 | 扩展搜索 |
| K3 | 用户说"不对/错了" | 每次 | 紧急修正 |
| K4 | 回答中使用了"可能/大概/建议验证"等不确定表述 | 每次 | 标记待验证 |
| K5 | 新软件版本发布（检测关键词） | 实时 | 触发版本更新搜索 |

### 2.2 缺口分级

| 级别 | 定义 | 响应时效 |
|------|------|----------|
| **P0-紧急** | 用户明确指出错误 | 下次对话立即修正 |
| **P1-高优** | 知识库完全缺失某主题 | 24小时内搜索补充 |
| **P2-中优** | 现有内容不够深入 | 本周内搜索补充 |
| **P3-低优** | 扩展阅读/进阶内容 | 本月内补充 |

## 三、修正协议（用户纠正时触发）

### 3.1 修正步骤

```
用户纠正
  → Step 1: 定位相关条目（搜索关键词 + 检查引用文件）
  → Step 2: 交叉验证（≥2个权威源）
  → Step 3: 确认错误 → 执行修正
  → Step 4: 版本号 +0.0.1（如 V3.4 → V3.41）
  → Step 5: 记录修正历史
  → Step 6: 通知用户"已修正，感谢反馈"
```

### 3.2 修正历史记录格式

> 文件位置：`${USERPROFILE}\.workbuddy\skills\GIS\feedback\revision_history.md`

```markdown

## 四、增量搜索触发规则

### 4.1 自动触发条件

| 条件 | 搜索范围 | 频率 |
|----------|----------|----------|
| 知识缺口达到3个P1级 | 对应主题权威源 | 立即 |
| 每周日凌晨2:00 | 全部软件版本更新 | 每周 |
| 用户主动说"更新知识库" | 用户指定主题 | 立即 |
| 新国标发布（检测openstd.samr.gov.cn） | 测绘标准体系 | 每月1日 |

### 4.2 搜索执行模板

```
1. 从 knowledge_gaps.md 取出最高优先级缺口
2. 构建搜索关键词（中英文各3组）
3. 并行搜索：
   - 官方文档（Esri/CSDN官方/软件官网）
   - 标准平台（openstd.samr.gov.cn）
   - 技术社区（CSDN/知乎/OSGeo）
4. 交叉验证新信息（§五）
5. A/B级内容入库
6. 更新 knowledge_gaps.md 状态
```

## 五、新信息交叉验证规则

### 5.1 验证维度

每一条新信息必须经过以下验证：

| 维度 | 验证方法 | 通过标准 |
|------|----------|----------|
| **一致性** | 与现有知识库对比 | 无直接矛盾（或矛盾已标注） |
| **权威性** | 检查信息来源 | 官方文档 > 学术论文 > 技术博客 > 论坛 |
| **可复现性** | 是否有操作步骤/参数/代码示例 | 有具体参数/步骤 → 高可信度 |
| **时效性** | 检查发布/更新日期 | 18个月内 → 高时效；超过36个月 → 标注"需验证" |
| **普遍性** | 是否特定版本/特定区域限定 | 标注适用范围限制 |

### 5.2 价值评级

| 级别 | 定义 | 入库决策 |
|------|------|----------|
| **A级** | 完全新增 + 高价值 | 立即入库，创建/更新对应文件 |
| **B级** | 补充现有内容细节 | 入库，追加到对应章节 |
| **C级** | 确认已有知识 | 不入库，记录验证日期 |
| **D级** | 低价值/不可靠 | 丢弃 |

### 5.3 冲突处理

```
新信息 vs 现有知识 → 冲突？
  ├── 无冲突 → 按价值评级处理
  └── 有冲突
        ├── 新信息来源更权威 → 以新信息为准，标注版本/日期
        ├── 现有知识更权威 → 丢弃新信息，记录冲突原因
        └── 无法判断 → 两说并存，标注"存在争议，待验证"
```

```

      → 立即触发修正协议（§三）

      → 加入搜索队列（§四）

## 七、定期自检协议

### 7.1 每日自检（自动）

- 检查 feedback_log.md 未处理条目
- 检查 knowledge_gaps.md 新缺口
- 如果累计未处理条目 ≥3 → 提示用户"有N项反馈待处理"

### 7.2 每周自检（自动）

- 检查所有文件最后更新时间
- 如果需要版本更新搜索（见§四）→ 自动执行
- 输出周报：`GIS技能包本周更新摘要.md`

### 7.3 每月自检（自动）

- 全库交叉验证：随机抽取20%条目进行一致性检查
- 检查所有外部链接有效性（HTTP状态码）
- 清理已超过12个月未引用的的大段内容（移至archive/）

## 八、量化触发算法 ⭐ V4.0 新增

### 8.1 知识缺口优先级算法（Weighted Priority Scoring）

每个知识缺口计算加权优先级分数 WPS：

```
WPS = W1×S + W2×F + W3×A + W4×U

其中：
  W1=4.0 (严重度权重) | S = P级基础分 (P0=200, P1=100, P2=50, P3=25)
  W2=2.0 (频率权重)   | F = 同类问题触发次数 (0~10, 封顶10)
  W3=1.5 (时效权重)   | A = 数据时效性评分 (0~100, 基准2026)
                         A = max(0, 100 - (2026-publish_year)*20)
  W4=1.0 (用户权重)   | U = 用户显式提及次数 (0~5)

WPS 区间：
  ≥800  → 立即触发搜索（最高优先级）
  500~800 → 下次对话触发搜索
  200~500 → 本周内搜索
  <200  → 加入待搜索队列，低优先级
```

**计算示例**：

```
缺口G001 "CGCS2000七参数区域差异"：
  P级=P1 → S=100, F=3次, A=90(2025年), U=2次
  WPS = 4.0×100 + 2.0×3 + 1.5×90 + 1.0×2
      = 400 + 6 + 135 + 2 = 543 → 下次对话触发搜索

缺口G003 "ArcGIS Pro 3.7 Embeddings分析":
  P级=P1 → S=100, F=1次, A=100(2026最新), U=1次
  WPS = 4.0×100 + 2.0×1 + 1.5×100 + 1.0×1
      = 400 + 2 + 150 + 1 = 553 → 下次对话触发搜索
```

### 8.2 版本自动升级规则

```
版本号格式：V[MAJOR].[MINOR].[PATCH]
  MAJOR：知识库架构变化（如V3→V4：六群组重构）
  MINOR：≥30%文件内容更新  或  新增≥3个文件
  PATCH：个别条目修正/小补充（<10%内容变化）

自动升级触发条件：
  MAJOR+1：架构重组（用户指令）→ 需人工确认
  MINOR+1：累计修改文件 ≥10个 且 新增条目 ≥50条 → 自动
  PATCH+1：每次修正操作 → 自动

版本号跳转示例：
  V3.40 → V3.41（修正3个条目）
  V3.41 → V3.50（新增3个文件+累计50条修改）
  V3.50 → V4.00（架构重组：五群组→六群组）
```

### 8.3 反馈质量量化评估

```
用户反馈质量评分 FQS：
  显式纠正 +10分（"不对/错了/应该是"）
  追问深入 +5分（"还有呢/能详细点吗"）
  肯定表态 +2分（"很好/有帮助"）
  否定表态 -3分（"没用/不对吧"——待确认）
  沉默跳过  0分（无反馈，可能满意也可能放弃）

累计 FQS：
  FQS ≥30 → 标记"高价值反馈用户"，自动开启主动优化
  FQS -10~29 → 正常模式
  FQS ≤-10 → 标记"需关注"，检查知识库准确性
```

## 十一、使用说明

### 对用户：

**无需任何操作**，自进化机制在后台自动运行：

1. 您正常提问，像往常一样
2. 如果我的回答有问题，直接说"不对/错了"——我会自动修正
3. 如果您有补充信息，直接说——我会评估是否入库
4. 如果想主动触发更新，说"更新知识库"或"搜索最新的XXX"

### 对AI助手：

每次对话结束时，检查并执行：
1. 是否有未处理的反馈？→ 处理
2. 是否有新的知识缺口？→ 记录
3. 是否达到自动搜索触发条件？→ 执行搜索
4. 更新 feedback/ 目录下的追踪文件

---

> **自进化机制核心理念**：
> - 错误不可怕，可怕的是重复错误
> - 用户反馈是最宝贵的知识来源
> - 权威源验证是入库的唯一门槛
> - 版本号永远向前，修正永远可追溯
