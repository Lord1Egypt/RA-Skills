# 微信小程序 AI 入门与接入模式

> 本文是入门概览。完整的 AI 能力清单见 `references/ai-capabilities.md`。

## 什么时候读取

用户问微信 AI 是什么、开发模式和自动模式怎么选、官方 demo 是什么结构、商家为什么要接入时，读取本文件。

## 小程序 AI 是什么

小程序 AI 是微信内的自然语言服务入口。用户用一句话表达需求，AI 理解意图后，通过小程序 MCP 协议调用开发者在 SKILL 中声明的原子接口和原子组件，完成查询、推荐、下单、预约、支付、配送、售后等任务。

给商家的解释：

> 以前用户要先想起你的小程序、打开页面、自己找商品。现在用户可能只说一句"帮我点杯冰美式"，AI 就要判断应该调用哪个小程序、哪个商品、哪个下单流程。

## 当前状态（2026年6月）

- 处于内测 beta 阶段
- 暂未开放代码提审，提审时间另行通知
- 申请入口：微信公众平台 → 基础功能 → AI能力，或小程序「微信开发者助手」→ 管理 → 微信AI管理
- 调试需下载微信开发者工具（Nightly Electron Build 最新版本）

## 用户路径

```
用户打开小程序 AI
→ 用自然语言说需求
→ AI 判断意图，匹配 SKILL
→ 调用原子接口，获取数据
→ 渲染原子组件（GUI卡片）或返回文本
→ 用户在卡片上操作或继续对话
→ 半屏页面补充信息（按需）
→ 用户确认
→ 完成下单、预约、支付、查询
```

## 开发模式

开发模式是商家主动把业务封装为小程序 AI 可调用的 SKILL。

### 核心文件

| 文件 | 作用 | 大小限制 |
|------|------|----------|
| `AGENTS.md` | 全局提示词，说明服务范围、Skill 分工 | 10000 字节 |
| `app.json` | 声明 `agent.skills`、Skill 路径、分包 | — |
| `SKILL.md` | 单个业务 Skill 的流程、接口依赖、约束 | 16000 字节 |
| `mcp.json` | 原子接口声明（name、description、inputSchema） | 24000 字节 |
| `index.js` | 注册原子接口 | — |
| `apis/` | 接口实现 | — |
| `components/` | 原子组件（GUI卡片） | — |
| `page-meta.json` | 页面元数据（文字链兜底） | 8000 字节 |

### 关键限制

- 一个小程序最多 30 个 SKILL
- SKILL 必须封装在独立分包里
- 需开启 `lazyCodeLoading: "requiredComponents"`
- 原子接口超时上限 300 秒
- 原子组件宽高比 4:1 到 1:1，初始化后不可改
- 原子组件仅支持 tap 事件，不支持动画和滚动

### 适合

- 有商品、订单、预约、支付、配送、会员、优惠券、门店等业务流程
- 想长期运营小程序 AI 入口
- 希望 AI 能稳定理解业务边界和调用顺序

## 自动模式

自动模式更依赖平台对页面、内容和已有能力的理解。

### 适合

- 没有开发团队，想先试水
- 小程序主要是展示型页面、FAQ、内容查询
- 业务流程简单，不涉及复杂确认、支付、预约、库存

### 局限

- 能力边界不可控
- 页面信息不结构化时，模型可能理解不准
- 复杂交易链路不适合只依赖自动模式

## 开发模式 vs 自动模式

| 维度 | 开发模式 | 自动模式 |
|---|---|---|
| 商家动作 | 主动定义 Skill、接口、参数、返回和卡片 | 优化页面、内容、商品描述和知识库 |
| 控制力 | 高 | 低 |
| 开发成本 | 中到高 | 低 |
| 适合业务 | 交易、预约、配送、会员、复杂服务 | 展示、查询、轻量试水 |
| GEO 价值 | 更高，数据可结构化回传 | 较弱，依赖页面理解 |
| 推荐策略 | 主策略 | 兜底补充 |

## 官方 demo 可讲的结构

微信官方 ai-mode-demo 是 WeStoreCafe 点单场景。它演示了如何把饮品推荐、搜索、详情、规格选择、地址选择、订单确认、支付、门店状态等能力封装成 Skill。

可复用讲法：

```
用户说"帮我点一杯冰美式"
→ searchDrinks / recommendDrinks
→ selectDrink
→ chooseSku
→ chooseAddress
→ createOrder
→ requestPayment
→ queryOrderStatus
```

重点不是咖啡，而是业务拆法：任何小程序都要把"页面"拆成"AI 可调用的服务能力"。

## 官方 demo 的设计要点

1. **对话承接模糊意图**：用户说"想喝点清爽的"，AI 完成理解和推荐，再引导至卡片确认
2. **自然语言修改内容**：用户可直接对话修改规格、时间、地点等字段
3. **历史数据可查询可操作**："再来一杯上次喝的"、"用积分兑换券"
4. **服务进程动态更新**：订单卡片随状态推进动态更新，非静态展示

## 知识库

开发模式下可添加知识库，AI 在所有 SKILL 都不匹配时兜底检索。

- 文件格式：PDF/DOC/DOCX/PPT/PPTX/TXT/MD/XLSX
- 单文件上限 10MB，文件总数上限 10 个
- 上传入口：微信公众平台 → 基础功能 → AI能力 → 知识库
- 可在开发版/体验版测试完整效果（SKILL + 知识库）

## 文字链拉起

AI 可在回复中带小程序短链，但这是兜底策略，核心流程不应依赖。核心功能使用文字链可能影响审核或触发召回降权。

## 给用户的默认建议

1. 有交易闭环：优先开发模式
2. 没开发资源：先做自动模式兜底，但同步准备商品/服务结构化数据
3. 有多个业务线：先做一个最能成交的 Skill，不要一口气全做
4. 第一版目标：让 AI 能准确理解、推荐、确认和完成一个核心流程
