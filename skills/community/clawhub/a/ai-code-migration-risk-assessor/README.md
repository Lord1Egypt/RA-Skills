# AI 代码变更风险评估器

## 业务场景

在进行大规模代码重构、合并外部开源项目、或升级核心 API 协议时，开发团队往往面临一个核心痛点：**改动的影响范围难以量化，风险等级无法准确评估**。

传统做法依赖 Code Review 经验和人工排查，效率低且容易遗漏关联模块。本工作流将代码变更分析全流程自动化，从变更获取到风险量化再到方案输出，一气呵成。

## 痛点分析

1. **影响范围不清晰**：改了一个基础模块，不知道有多少下游调用方受影响
2. **风险等级凭经验**：没有量化标准，纯靠主观判断，P0 还是 P2 全凭感觉
3. **方案碎片化**：知道有风险，但不知道从哪里开始改，没有系统性方案
4. **验证成本高**：重构后需要手动检查大量相关模块，容易引入新 Bug

## 协同逻辑

| 阶段 | Skill | 输入 | 输出 |
|------|-------|------|------|
| 数据获取 | github | 仓库 URL / PR 链接 | 变更文件列表 + Commit 历史 |
| 深度分析 | code-review-skill | Diff 内容 | 问题清单 + 风险点 |
| 量化评估 | agentic-devops | 问题清单 + 依赖关系 | 量化风险等级 + 影响图谱 |
| 方案生成 | (本工作流) | 风险报告 | 重构方案 + 行动计划 |

## 使用示例

### 示例 1：分析 GitHub PR 风险

**输入**：
```
帮我分析这个 PR 的风险：
https://github.com/owner/repo/pull/123
```

**执行流程**：
1. github 获取 PR diff 和变更文件列表
2. code-review-skill 对每个变更文件进行深度审查
3. agentic-devops 分析模块依赖关系，量化风险等级
4. 汇总输出风险报告 + 分步骤重构方案

**输出示例**：
```
🔴 P0 风险 (1) | 🟠 P1 风险 (3) | 🟡 P2 风险 (7) | 🟢 P3 风险 (12)

受影响模块：
  - auth/token → 核心鉴权，扩散 23 个调用点
  - api/v2    → API 协议变更，波及 5 个客户端

重构方案：
  Step 1: 先重构 auth/token，添加兼容性 Adapter
  Step 2: 旧 API v1 保留 90 天过渡期
  Step 3: 全量测试覆盖后再删除旧代码
```

### 示例 2：批量分析重构影响范围

**输入**：
```
评估 owner/repo 在 v2.3.0 重构中的影响范围
重点关注：API 兼容性、数据库 schema 变更
```

**执行流程**：
1. github 对比 v2.2.0 → v2.3.0 所有变更
2. code-review-skill 识别 Schema 变更和 API 协议变化
3. agentic-devops 追踪所有下游依赖服务
4. 生成影响评估报告，标注必须人工 review 的高风险区
