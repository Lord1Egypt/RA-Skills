# 📖 端到端写作示例 (End-to-End Examples) — v2.3

> 本文档展示 Writing Triadic 从触发到交付的完整实战流程。
> 每个示例标注了**写作类型标签**、**关键决策点**和**进化引擎输出**。

## 示例 1: 技术博客文章 — "如何用好 AI 编程助手"

**写作类型**: `[博客文章]`  
**用户画像**: 有 3 年开发经验的程序员，常用 AI 编程工具，想写一篇有观点的方法论文章  
**历史档案参考**: MEMORY.md 中 `[博客文章]` 已有 5 次使用记录，偏好：口语化/第一人称/至少 2-3 个代码示例/禁止结尾升华

### Phase 0: 读取风格档案

Creator 静默读取 MEMORY.md，提取到：
- 语调偏口语化、用"我"视角
- 代码示例 ≥ 3 个
- 禁止结尾升华式总结
- 偏好痛点提问式开头
- 字数 1500-2500 字

### Phase 1: 需求挖掘

```
Creator: 技术博客 — 主题是用 AI 编程的实战经验。让我确认几个关键点：

1️⃣ 你更想讲"方法论"（怎么用 AI 更高效）还是"踩坑经验"（AI 在哪些场景出过错）？或者两者结合但偏一个方向？

2️⃣ 目标读者是跟你一样有经验的开发者，还是刚入门的新人？这会决定举的例子深到什么程度。

3️⃣ 你之前几次博客都偏好用"我"的第一人称，这次继续这个路线吗？

4️⃣ 有没有一个你特别想讲的「核心观点」？比如"AI 不是替代人，而是把人从重复劳动里解放出来"这一类的主旨。

User: 1. 偏方法论但是通过踩坑案例来讲，更有说服力
      2. 有经验的开发者，不需要解释基础概念
      3. 继续第一人称，习惯这样写
      4. 核心观点是：Prompt 工程不是"会说话就行"，需要对问题域有深度理解
```

### Phase 1 继续

```
Creator: 方向很清晰了。继续确认：

1️⃣ 你手头有没有一两个特别典型的"烂 prompt vs. 好 prompt"的实际例子可以引用？会让文章更有说服力。

2️⃣ 语调方面 — 犀利直击痛点，还是温和分享经验？上次你的博客偏犀利，这次呢？

3️⃣ 长度偏好有变化吗？之前的博客大概在 1500-2500 字区间。

User: 1. 有。有个例子很明显：一个实习生用"帮我写一个登录功能"拿到了一堆不可用的代码，而我加了 3 个边界条件后得到了直接能跑的实现。
      2. 犀利点，就事论事，不讨好读者
      3. 这次可以偏长一点，2500-3000，这个主题值得讲透
```

### Phase 1: 锁定

```
💡 需求与框架锁定报告
- 核心主题：AI 编程助手的高效使用 — Prompt 工程的深度理解
- 目标受众：有 2-5 年经验的开发者
- 预期目的：方法论分享 + 行为改变
- 写作语调：犀利、直接、第一人称、不讨好
- 应用模板/体裁：博客文章
- 附加限制：2500-3000 字，包含 ≥3 个烂 vs. 好的 prompt 对比案例
- 📖 历史参考：已自动应用 — 口语化、第一人称、禁用结尾升华、痛点开头、≥3 个代码示例

🔄 已锁定需求架构，正在呼叫执行团队...
```

### Phase 1.5: 联网调研

Creator 自动搜索：
- `"Prompt Engineering best practices 2025 2026" + writing`
- `"prompt engineering 博客 写法"`
- `"AI coding assistant tips 技术博客"`

发现一个新观点：**"context stuffing"** 是比 prompt 写法更容易被忽视的问题，于是将 "context stuffing" 加入知识库并告知 Executor 可以作为差异化角度。

### Phase 2: 规则与计划制定

**写作规则.md** (经历史偏好注入):

```
## 用户历史偏好 (自动注入)

### 🌐 全局偏好
- 默认语调偏口语化
- 全局禁用词: "值得注意的是"、"在当今...的时代"

### 🏷️ [博客文章] 专属偏好
- 语调: 用"我"的第一人称视角
- 至少 2-3 个代码/案例示例
- 禁止结尾升华式总结
- 偏好痛点提问式开头
- 字数: 2500-3000 (本次上调)
```

### Phase 3: 执行者产出

Executor 产出两个版本：

**版本 A**（痛点切入 + 犀利毒舌语调）:
```
标题: "你那个 Prompt，实习生都不好意思用"

开头: 去年我带的一个实习生，第一次用 AI 写代码，Prompt 是"帮我写一个登录功能"。结果产出了一段没有密码加密、没有 session 管理、CSRF 完全不设防的代码。

他跟我说："AI 好像不太行。"

我说："不是 AI 不行，是你的问题描述不行。"...

[正文包含 4 组 prompt 对比案例，从简到深，逐级展示 context stuffing 的重要性]
```

**版本 B**（反常识数据开场 + 冷静克制语调）:
```
标题: "我测了 50 个 AI 编程 Prompt，90% 的开发者都在犯同一个错"

开头: 花了两个周末，我把 50 个日常工作中用过的 prompt 逐一标注效果。结论比我想的极端：90% 的 prompt 缺少关键上下文。更讽刺的是——那些写得"看起来很专业"的 prompt，反而比简单直接的 prompt 效果更差。

问题不在于 prompt 的长度，而在于"你到底知不知道自己在问什么"。

[正文以数据分析为主线，穿插 4 组对比案例]...
```

### Phase 4: 读者评审

Reader 代入「有经验的开发者」身份评审：

```
📊 逐版多维评分表

| 评分维度 (权重) | 版本 A | 版本 B |
| :--- | :--- | :--- |
| 吸引力 (15) | 14 — 故事开篇瞬间代入 | 12 — 数据有冲击力但略干 |
| 受众契合度 (20) | 19 — 直击痛点，像在看同事朋友圈 | 16 — 偏报告感，少了互联网写作的松弛 |
| 信息价值与深度 (15) | 14 — prompt 对比案例具体可用 | 14 — 数据分析有说服力 |
| 清晰与连贯 (10) | 9 — 故事线流畅 | 10 — 逻辑极清晰 |
| 目标达成率 (15) | 14 — 读完想立刻改自己的 prompt | 13 — 被说服了但推动力稍弱 |
| 真实性与人味 (25) | 24 — 对话感强，段间距自然，没有 AI 味 | 18 — 有几个地方的过渡词偏机械 |
| **基础总分** | **94** | **83** |
| 🚨 AI痕迹扣分 | -0 | -10 (3处"It is worth noting"痕迹 + 结尾偏升华) |
| 🆕 历史禁忌扣分 | -0 | -5 (文末有强行升华倾向) |
| **🏆 最终得分** | **94** | **68** |

🏁 最终选择: 版本 A
```

### Phase 5: 用户审阅

Creator 呈现版本 A 及其评分。用户表示满意，只改了一处用语。

### Phase 5.5: 进化引擎

Evolution Analyst 分析输出摘要：

```
本次写作类型: [博客文章]
关联模板: 博客文章

### 全局偏好更新: 无

### 类型偏好更新:
- [博客文章] 字数偏好上调至 2500-3000 (来源: 用户本次明确要求)
- [博客文章] 确认: 犀利语调效果优于温和 (来源: 本次行为 — 评分 94 vs 68)
- [博客文章] 确认: 故事化开头效果稳定 (来源: 本次 + 历史)

### 模板效果
- 模板: 博客文章 | 评分: 94 | 用户满意度: 高
- 统计更新: 使用次数 5→6, 平均分 82→84
```

## 示例 2: 朋友圈文案 — 个人成就分享

**写作类型**: `[朋友圈文案]`  
**用户画像**: 刚完成一个重要项目，想在朋友圈分享但不想显得在炫耀  
**历史档案**: MEMORY.md 中 `[朋友圈文案]` 无历史记录

### Phase 1: 需求挖掘（压缩展示）

```
Creator: 恭喜完成项目！朋友圈分享这件事，差别全在「分寸感」。
       让我确认几个关键点：

1️⃣ 想分享的重点是项目结果（比如数据、成果），还是过程中的某个瞬间？（后者通常更像真实的朋友圈）

2️⃣ 你的朋友里，大概有多少是同行业的人？这决定了是写成"圈内交流"还是"碎碎念"。

3️⃣ 朋友圈风格你偏好哪种：
   A. 自嘲路线 — "熬了三个月，终于..."
   B. 轻描淡写 — "一个项目，收工"
   C. 小感动 — 讲一个项目过程中的人情细节

User: 1. 过程中的一个瞬间 — 最后一天通宵调试，发现 bug 的原因竟然是一个拼写错误，又气又好笑。
      2. 大部分不是同行的，就是普通朋友，所以不要写技术细节。
      3. A 路线，自嘲比较适合我。

Creator: 好了，最后一个：

1️⃣ 你有没有拍一张能体现那个"通宵 moment"的照片？比如屏幕前的咖啡杯、凌晨的办公室——这种配图比截图更有温度。

User: 有的。拍了凌晨五点的窗外，天刚亮。

💡 需求与框架锁定报告
- 📝 [朋友圈文案] 尚无历史记录，仅应用全局偏好。写作完成后将自动学习。
```

### Phase 3: 执行者产出

**版本 A**（自嘲 + 对话感）:
```
bug 找了一宿。最后发现是 variableName 里多了一个字母。不是逻辑问题，不是架构设计，是拼错了。

我看着屏幕笑了五分钟。天刚好亮，窗外路灯灭了。

图配一张凌晨五点的窗外，不配也行。那就这样吧。
```

**版本 B**（自嘲 + 叙事感）:
```
凌晨四点，我和一个拼写错误面面相觑。

不是代码逻辑的错，不是架构的错，是我自己手滑多打了一个字母。"花了一整夜，就为了这一个字母。"

五点钟把 bug 修了，天亮了。发条朋友圈，回点人间气息。

[窗外照片]
```

### Phase 4: 读者评审

Reader 高效评分：版本 A 更像真正的人类朋友圈 — 没有完整的叙事弧线，只有"几个瞬间切片 + 一句情绪收尾"，更符合朋友圈语体。版本 B 的叙事结构太完整，反而像微小说。

最终得分：版本 A 85 分，版本 B 62 分。

### Phase 5.5: 进化引擎

本次为 `[朋友圈文案]` 首次使用，进化分析师自动初始化该类型标签并记录偏好。

## 示例 3: 求职简历优化 — 从"流水账"到 STAR 句式

**写作类型**: `[求职简历]`  
**用户背景**: 应届生，有一段医院信息系统运维和两个项目经历  
**历史档案**: `[求职简历]` 无记录

### Phase 1: 关键信息提取

```
Creator: 简历优化是写作里最"功利"的一种——每一行要么证明能力，要么删掉。确认几件事：

1️⃣ 目标岗位是什么？（不同的岗位决定了简历里该放什么项目、放多少细节）

2️⃣ 你目前简历最大痛点是什么？（感觉内容太单薄？不知道怎么写量化结果？排版混乱？还是有东西但表达太弱？）

3️⃣ 你的项目经历里，有没有一个能体现"独立负责/推动/决策"的瞬间？这个比"参与"重要十倍。

User: 1. AI 开发工程师 / 软件实习岗
      2. 最大的问题是工作经历里的描述全是"负责XXX"、"参与XXX"，看着像岗位说明书，不像我在做事
      3. 有的。AI Trending 项目是我从 0 独立架构设计的，数据库方案是我自己对比后选的 SQLite
```

### Phase 3: 执行者产出（关键片段）

**改写前**:
```
2025.08-2026.03  巫山红叶医院  信息科实习生
· 负责医院信息系统基础运维工作
· 参与门诊预约系统优化的测试与反馈
· 整理常见问题记录和操作流程
```

**改写后 (STAR 句式)**:
```
2025.08-2026.03  巫山红叶医院  信息科实习生
· 独立处理 HIS/EMR/LIS/PACS 系统的日常登录异常、打印故障、账号权限核对等 20+ 类问题，
  形成《常见故障排查手册》（15 页，覆盖 80% 高频问题），新人查阅后求助频率下降 60%
· 主导门诊预约系统的上线前测试，发现并记录 23 个缺陷（含 3 个高优先级），
  全部在正式上线前修复，上线首周零投诉
· 与心内科、放射科等 6 个科室沟通协调系统优化需求，将临床反馈翻译为技术方案
  文档，推动预约排队逻辑改进
```

### Phase 4: 读者评审

Reader 代入 HR（10 秒速览简历场景）评审：改写后的版本每条都以动词开头、有数字、有因果。HR 不需要读完就能感知到"这个实习生真的干过事"。评分 97 分。

## 设计原则总结 (从三个示例提炼)

1. **Phase 1 是最关键的** — 文章质量 70% 由需求挖掘决定。Creator 的提问质量直接决定执行者拿到的是"精准任务"还是"模糊方向"。
2. **历史档案让每次写作都在变聪明** — 示例 1 中以前记录的口语化偏好、禁止结尾升华全部自动生效，不需要用户再说一遍。
3. **两版差异化真正有用** — 不是敷衍地"换了几个词"，而是从切入点、语调两个本质上不同的维度出发，让用户能看到真正的选择。
4. **Reader 不是为了打分而打分** — 最好的 Reader 点评像真人读者在跟你聊读后感，不是在交作业。
5. **即时纠错才真正价值大** — 用户说的每句话都是免费的训练数据。把"太啰嗦了"变成永久规则，比写 100 行代码更值钱。
