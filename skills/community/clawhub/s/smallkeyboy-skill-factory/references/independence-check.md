# 模块独立性检查清单

本文档用于验证拆解后的子 Skill 是否满足独立性要求。

---

## 独立性定义

**独立模块**：可以单独执行、单独验收、不依赖其他模块执行结果的模块。

---

## 检查清单

### 1. 输入独立性

- [ ] 模块的输入是否来自用户或系统初始数据？
- [ ] 模块是否不依赖其他模块的输出作为必需输入？
- [ ] 如果有可选依赖，是否能在缺失时正常工作？

**违反示例**：
```
模块A：分析用户数据 → 输出 user_profile
模块B：基于 user_profile 生成推荐 ← 依赖模块A输出（违反独立性）
```

**修正方案**：
```
模块A：分析用户数据 → 输出 user_profile
模块B：生成通用推荐规则（不依赖具体用户数据）
Orchestrator：将 user_profile 应用到推荐规则
```

---

### 2. 执行独立性

- [ ] 模块是否可以单独执行？
- [ ] 模块执行是否不需要等待其他模块完成？
- [ ] 模块执行失败是否不会阻塞其他模块？

**违反示例**：
```
模块A：获取认证token
模块B：使用token调用API ← 必须等待模块A完成（违反独立性）
```

**修正方案**：
```
将认证逻辑内置到模块B，或由 Orchestrator 统一管理认证
```

---

### 3. 输出独立性

- [ ] 模块的输出是否是完整、可独立使用的结果？
- [ ] 模块输出是否不需要其他模块的输出才能理解？
- [ ] 模块输出格式是否是标准化的？

**违反示例**：
```
模块A：输出 "步骤1完成"
模块B：输出 "步骤2完成" ← 输出含义依赖模块A的上下文（违反独立性）
```

**修正方案**：
```
模块A：输出完整的第一阶段结果
模块B：输出完整的第二阶段结果
Orchestrator：合并展示
```

---

### 4. 验收独立性

- [ ] 模块是否可以单独验收？
- [ ] 模块的成功标准是否清晰明确？
- [ ] 验收是否不需要参考其他模块的结果？

**违反示例**：
```
模块A：生成报告框架
模块B：填充报告内容 ← 无法单独验收"填充"效果（违反独立性）
```

**修正方案**：
```
模块A：生成完整报告（包含占位符）
模块B：优化报告表达
两个模块都可以单独验收
```

## 常见依赖类型

### ❌ 硬依赖（不允许）

```
模块A输出 → 模块B必需输入
```

这种依赖违反独立性原则，应避免。

### ⚠️ 软依赖（谨慎使用）

```
模块A输出 → 模块B可选输入（用于增强效果）
```

模块B应能在没有模块A输出时正常工作（降级模式）。

### ✅ 无依赖（理想状态）

```
模块A ← 用户输入
模块B ← 用户输入
模块C ← 用户输入
```

所有模块直接从用户输入获取数据，完全独立。

---

## 检查流程

```
1. 列出所有模块
     ↓
2. 绘制模块依赖图
     ↓
3. 检查每条依赖边
     ↓
4. 标记硬依赖/软依赖
     ↓
5. 重构硬依赖
     ↓
6. 验证重构后的独立性
```

---

## 依赖图示例

### ❌ 有问题的依赖图

```
用户输入
   │
   ├→ 模块A
   │     │
   │     ↓
   │   模块B ← 依赖A
   │     │
   │     ↓
   │   模块C ← 依赖B
```

### ✅ 正确的依赖图

```
用户输入
   │
   ├→ 模块A
   ├→ 模块B
   └→ 模块C
```

所有模块从同一数据源获取输入，无模块间依赖。

## 重构技巧

### 技巧一：数据源下沉

将上游模块的数据获取逻辑下沉到需要它的模块。

**Before**:
```
模块A：获取市场数据
模块B：分析市场数据 ← 依赖A
```

**After**:
```
模块A：获取市场数据（缓存/共享数据源）
模块B：获取市场数据（直接访问相同数据源）
```

### 技巧二：输出合并

将多个相关模块合并为一个。

**Before**:
```
模块A：生成框架
模块B：填充内容 ← 依赖A
```

**After**:
```
模块A+B：生成完整报告
```

### 技巧三：引入协调层

将依赖逻辑上移到 Orchestrator。

**Before**:
```
模块A → 模块B → 模块C（链式依赖）
```

**After**:
```
模块A ← 用户输入
模块B ← 用户输入
模块C ← 用户输入
Orchestrator：协调执行顺序和结果合并
```

## 检查表

在输出拆解结果前，必须通过以下检查：

| 检查项 | 通过标准 |
|-------|---------|
| 输入独立性 | 无硬依赖 |
| 执行独立性 | 可单独执行 |
| 输出独立性 | 输出完整可用 |
| 验收独立性 | 可单独验收 |
| 数量限制 | 3~6 个模块 |
| 覆盖完整性 | 覆盖用户目标 |

全部通过后，方可输出 `status: success`。
