# v2.8.29 (2026-05-06) — P0 紧急

**修 v2.8.27/v2.8.28 引入的严重 regression：finalize 用 placeholder 覆盖了 block 完整回复**

## 背景

赵博 5/6 7:51-8:16 三次发语音消息，BOT 都显示"正在思考..."就没下文了。截图显示三次都卡在"正在思考..."而且**带"内容由AI生成"标识** = 这条消息已被 last=true 终结。

但 gateway.log 看 LLM **实际产出了完整回复**：

```
07:52:11 deliver-start kind=block textLen=25
07:52:26 deliver-start kind=block textLen=582  ← LLM 完整回复 582 字
07:52:26 outbound summary="bot-ws Sir，根据最新记录给你汇报蓝火的**贡居宝任务**情况：..."
07:52:27 deliver-start kind=final textLen=0 mediaCount=0
07:52:27 [wecom-ws] finalizeStreamIfNeeded(final-empty-textmedia) ← v2.8.28 触发
```

**LLM 实际推了 582 字汇报，但用户在企微看到的是"正在思考..."**！

## 根因

v2.8.27/v2.8.28 设计 `finalizeStreamIfNeeded` 时**用 `computeCurrentPlaceholder()` 文案作为 last=true 终结内容**：

```typescript
const settleText = computeCurrentPlaceholder();  // ← "正在思考..."
await params.client.replyStream(params.frame, streamId, settleText, true);
```

**忽略了企微流式协议的关键事实**：

> `replyStream(streamId, content, last=true)` 中 `content` 是 **stream 的最终内容**——企微 client 会**用这个内容完全替换**之前 partial 推过的所有内容（同 streamId）。

所以：
1. `block` 推 partial（streamId, "...贡居宝任务汇报 582 字...", false）→ 企微 client 显示这条内容
2. `finalizeStreamIfNeeded` 推 final（streamId, "正在思考...", true）→ 企微 client **覆盖**显示成"正在思考..." + 锁定内容

**LLM 的完整回复被 placeholder 文案覆盖了**！

## 修复

`finalizeStreamIfNeeded` 优先用 `accumulatedText`（之前 block 累积的所有内容），仅当 accumulatedText 为空时才用 placeholder：

```diff
  const finalizeStreamIfNeeded = async (reason: string): Promise<void> => {
    if (streamSettled || isEvent || !streamId) return;
    try {
-     const settleText = computeCurrentPlaceholder();
+     // v2.8.29 ⭐ 优先用 accumulatedText（保留之前 block partial 已推的内容），
+     // 否则覆盖 block 完整回复内容 → 用户只看到"正在思考..."
+     const settleText = accumulatedText
+       ? toWeComMarkdownV2(accumulatedText)
+       : computeCurrentPlaceholder();
+     const settleSource = accumulatedText ? "accumulatedText" : "placeholder";
      await params.client.replyStream(params.frame, streamId, settleText, true);
      ...
+     console.log(`[wecom-ws] finalizeStreamIfNeeded(${reason}): ... source=${settleSource} len=${settleText.length}`);
    }
  };
```

## 触发场景区分

| 场景 | accumulatedText | source | 终结内容 |
|---|---|---|---|
| LLM 推过 block 然后 final 空 | 非空（block 累积）| `accumulatedText` | LLM 完整回复 ✅ |
| dispatch 极速 done 没产出 | 空 | `placeholder` | "正在思考..." 文案 ✅（v2.8.27 原意场景）|

两种场景兼顾。

7:51 场景重现（v2.8.29 修复后）：

```

07:52:27 final 空 → finalizeStreamIfNeeded(final-empty-textmedia) 用 accumulatedText 终结
```

## 经验沉淀（重要）

**stream 终结协议必须保留累积内容**：任何 `last=true / done` 信号都是**整个 stream 的最终态**——客户端会用它替换所有之前的 partial 内容。中间想用"占位文本"做信号是**错误**——会清空真实内容。

正确设计：
- 流式更新（last=false）：每帧是当前快照（覆盖之前），可以用 placeholder
- 流式终结（last=true）：必须包含**所有累积内容**（否则之前内容被清空）

v2.8.27/v2.8.28 设计时只关注"解锁 UI 交互"忘了这条——是经典的"修 A 引入 B"。今后任何 stream 状态机改动必须**端到端跑测试**：用户实际看到的内容 vs 我们以为发的内容。
