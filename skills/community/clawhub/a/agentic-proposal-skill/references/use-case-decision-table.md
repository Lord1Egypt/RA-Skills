# Proposal Use-Case Decision Table

| Situation | Template | Why |
|---|---|---|
| After discovery call | `discovery-recap` | Confirms what was heard before recommending scope. |
| Buyer wants short pilot summary | `one-page-pilot-proposal` | Gives concise business case, scope, and next step. |
| Buyer needs full sales proposal | `full-agentic-development-proposal` | Covers problem, approach, scope, timeline, risks, and next steps. |
| Scope needs to attach to contract | `statement-of-work` | Uses contract-ready SOW sections. |
| Fixed first project | `fixed-fee-pilot-proposal` | Aligns with bounded pilot pricing. |
| Flexible hourly work | `time-and-materials-proposal` | Handles uncertain scope or advisory/development mix. |
| Ongoing support | `monthly-retainer-proposal` | Defines recurring support/monitoring. |
| Advisory only | `advisory-only-proposal` | Avoids implying implementation. |
| Assessment before build | `workflow-assessment-proposal` | Sells discovery/map/risk review as standalone work. |
| Multi-phase plan | `implementation-roadmap` | Shows phases from discovery through handoff. |
| Evaluation is the key concern | `evaluation-plan` | Centers tests, examples, and acceptance evidence. |
| Risk/governance is the key concern | `risk-control-plan` | Centers approval gates, exclusions, logs, and monitoring. |
| Human approval design needed | `human-in-the-loop-approval-model` | Defines when people must approve. |
| Added scope after agreement | `change-order-proposal` | Documents incremental scope and approval. |
| Second workflow after success | `expansion-second-workflow-proposal` | Frames next workflow as expansion. |
| Procurement/internal approval | `procurement-ready-scope-summary` | Gives concise approval language. |
| Executive decision maker | `executive-summary` | Gives non-technical decision summary. |
| Done criteria unclear | `acceptance-criteria` | Defines measurable acceptance. |
| Scope boundaries unclear | `assumptions-and-exclusions` | Clarifies what is and is not included. |
| Post-launch support planning | `support-and-handoff-plan` | Defines documentation, support, and operating handoff. |
| After discovery, recommendation needed | `post-discovery-recommended-pilot-memo` | Converts discovery into a recommended first pilot. |
| Technical architecture buyer | `technical-architecture-proposal` | Covers systems, tools, data, and deployment path. |
| Data/tool access unresolved | `data-and-tooling-readiness-proposal` | Identifies readiness dependencies. |
| Training included | `training-enablement-proposal` | Covers reviewer/admin/user enablement. |
| Proof of concept only | `proof-of-concept-proposal` | Keeps scope experimental and bounded. |
| Moving from pilot to production | `production-rollout-proposal` | Adds operational controls and rollout planning. |

Selection rule: choose by buyer decision need first. If the buyer is still validating the problem, use recap/assessment. If the buyer agrees on the problem and wants scope, use pilot/full proposal/SOW. If the buyer is approving internally, use executive or procurement summary. If the deal is already active, use change-order, expansion, support, or rollout templates.
