# Autoresearch 机制

> 每3个案件触发一次迭代分析，Agent 执行，棘轮进化。

---

## ⚠️ 核心原则

**脚本只负责计数和提醒，迭代分析由 Agent 做。**

- 脚本：创建骨架、更新计数、提醒触发
- Agent 必做：读取 CASE、分析、做决策、更新 SKILL.md、Git commit

---

## 📋 CASE 字段说明

| 字段 | 作用 | 格式 |
|------|------|------|
| **编号** | 唯一标识 | CASE_XXX_YYYYMMDD_HHMMSS |
| **问题** | 判断适用场景 | 一句话 |
| **心智模型** | 判断采用的心智模型 |
| **逻辑公式** | 判断采用的逻辑公式 |
| **结果** | 判断方法是否有效 | ✅成功 / ❌失败 / 🔄部分成功 |
| **结果描述** | 记录结果产生的详细原因 |
| **处理时间** | 效率指标 | X分钟 |
| **实验次数** | 效率指标 | X次 |
| **切换记录** | 分析切换时机 | 有切换才填 |

---

## 🔄 迭代流程（Agent 执行）

### 触发条件
index.md 中案件计数 % 3 == 0

### 第一步：读取最近3个CASE

读取 `.autoresearch/cases/` 下最近3个 CASE 文件。

### 第二步：逐案快速扫描

每个 CASE 回答6个问题：

| # | 问题 | 答案格式 |
|---|------|---------|
| 1 | 用了什么心智模型？ | 阁楼/链条/异常锚定/假设演绎/反向工程/主动验证/压缩推理 |
| 2 | 用了什么逻辑公式？ | 排除法/假言推理/逆否推理/传递规则/归谬法... |
| 3 | 结果如何？ | ✅成功 / ❌失败 / 🔄部分成功 |
| 4 | 效率如何？ | 处理时间 + 实验次数 |
| 5 | 有无关键切换？ | 有/无，有切换则说明时机对不对 |
| 6 | 结果描述说了什么？ | 成功/失败/部分成功的原因是什么？溯因链完整/断裂在哪里？是方法问题还是执行问题？ |

### 第三步：横向对比

对比3个CASE，找出模式：

```
【模式A】
- 出现次数：X
- 成功率：X/X
- 结论：保留/丢弃/加强

【模式B】
- 出现次数：X
- 成功率：X/X
- 结论：保留/丢弃/加强

【新发现】
- 什么？（之前没意识到的模式）
- 什么时候适用？
```

### 第四步：问题定性

**每个失败的 CASE，判断是方法问题还是执行问题：**

| 类型 | 判断标准 |
|------|---------|
| **方法问题** | 思维框架（心智模型/逻辑公式）选错或是本身存在问题，或缺了某类模型/公式 |
| **执行问题** | 方法正确，但执行时跳步骤/跳跃物理路径/没切换 |

**综合3个CASE，输出问题分布：**
```
【问题定性】
- CASE_XXX: [方法/执行]问题 → [具体原因]
- CASE_YYY: [方法/执行]问题 → [具体原因]
- CASE_ZZZ: [方法/执行]问题 → [具体原因]

方法问题：X个 → 更新SKILL.md
执行问题：X个 → 记录到holmes-learnings.md
```

### 第五步：分流处理

根据第四步的定性结果，分别处理：

**方法问题 → 更新 SKILL.md：**
- 强化有效的思维模式（补充适用场景说明）
- 删除或标注无效的（加严触发条件或标记"慎用"）
- 添加新发现（心智模型/逻辑公式/触发器）

**执行问题 → 记录到 `references/holmes-lessons.md`**：
- 格式：`- [时间] [CASE_XXX] [问题描述] → 以后要这样做而不是那样做`
- 写具体，不要写空话

### 第六步：Git Commit

```bash
cd ~/.openclaw/workspace/skills/holmes
git add -A && git commit -m """
迭代 #XXX | 3案件分析

【分析摘要】
- CASE_XXX: [心智模型]+[逻辑公式] → [结果] | [时间]min | [X]次
- CASE_YYY: [心智模型]+[逻辑公式] → [结果] | [时间]min | [X]次
- CASE_ZZZ: [心智模型]+[逻辑公式] → [结果] | [时间]min | [X]次

【问题定性】
方法问题：X个 | 执行问题：X个
- CASE_XXX: [方法/执行] → [具体原因]
- CASE_YYY: [方法/执行] → [具体原因]
- CASE_ZZZ: [方法/执行] → [具体原因]

【分流处理】
✅ 方法问题 → SKILL.md更新：[写了什么]
📝 执行问题 → holmes-lessons.md记录：[写了什么]

【证据】
基于3个案件: CASE_XXX~CASE_ZZZ
"""
```

## 📊 效率监控

| 指标 | 计算 | 意义 |
|------|------|------|
| 平均处理时间 | 总时间/案件数 | 持续下降=好 |
| 平均实验次数 | 总次数/案件数 | 持续下降=好 |

---

## 🛡️ 棘轮原则

> **只进不退**

- 有效的思维模式 → 保留，写入 SKILL.md
- 坏的思维模式 → 丢弃，不写入
- 历史案件永久保存，供回溯

---

## ⏭️ 快速执行流程

```
解决问题
    ↓
执行 holmes_case_logger.py → 创建 CASE + 更新计数
    ↓
计数 % 3 == 0 ？ ──► 否 → 继续
    ↓ 是
🚨 触发 autoresearch → Agent 执行：
    ↓
1. 读取3个CASE（逐案6问）
2. 横向对比（找模式）
3. 问题定性（方法 vs 执行）
4. 分流处理（SKILL.md / holmes-learnings.md）
5. Git commit
    ↓
继续，计数不归零
```
