# PRD Template for AI Coding Agents

This is the canonical structure for PRDs generated by this skill. Every
section marked REQUIRED must appear in the output. Sections marked
CONDITIONAL appear only when the user's context makes them relevant.

The PRD is a single Markdown file. It must be self-contained: an agent
reading only this file should have everything needed to start building.

---

## Document Header

Always start with this metadata block:

```markdown
# PRD: <Feature Name>

| Field          | Value                                       |
|----------------|---------------------------------------------|
| Version        | 1.0.0                                       |
| Status         | Draft | Review | Approved | Implementing     |
| Author         | <user name or handle>                       |
| Date           | <YYYY-MM-DD>                                |
| Target Agent   | Claude Code | Cursor | Codex | Any          |
| Stack          | <language, framework, runtime>              |
| Repository     | <path or URL, or "greenfield">              |
```

---

## Section 1: Executive Summary (REQUIRED)

3 to 5 sentences. What is being built, why, and what success looks like.
An agent or human who reads only this section should understand the
feature's purpose and scope.

```markdown
## 1. Executive Summary

<Product/Feature Name> is <what it does> for <who>. It addresses
<problem> by <approach>. The MVP targets <scope>. Success is defined
as <measurable outcome>.
```

---

## Section 2: Problem Statement (REQUIRED)

Concrete description of the problem. Not "users want X" but "users
currently do Y, which causes Z." Include data if available.

```markdown
## 2. Problem Statement

### Current State
<What happens today without this feature>

### Pain Points
- <Specific, measurable pain point 1>
- <Specific, measurable pain point 2>

### Impact
<What happens if we do not solve this>
```

---

## Section 3: Goals and Success Metrics (REQUIRED)

Quantifiable targets. Every goal has a metric. If the user did not
provide metrics, use reasonable defaults and mark them `[ASSUMPTION]`.

```markdown
## 3. Goals and Metrics

| Goal                          | Metric                    | Target     |
|-------------------------------|---------------------------|------------|
| Reduce manual processing time | Time per task             | < 2 min    |
| Improve data accuracy         | Error rate                | < 1%       |
| User adoption                 | Active users at 30 days   | > 80%      |
```

---

## Section 4: Scope (REQUIRED)

Explicit boundaries. Both what is in and what is out. Out-of-scope
items prevent agents from inventing features.

```markdown
## 4. Scope

### In Scope (MVP)
- <Feature 1>: <one-line description>
- <Feature 2>: <one-line description>

### Out of Scope (v1)
- <Excluded feature 1>: <why excluded>
- <Excluded feature 2>: <why excluded>

### Future Considerations
- <Feature that may come in v2+>
```

---

## Section 5: User Flows (REQUIRED)

Primary user journeys as numbered steps. Include the happy path and
the top 2 to 3 error paths. Use the format below, not prose paragraphs.

```markdown
## 5. User Flows

### 5.1 Happy Path: <Flow Name>
1. User navigates to <location>
2. User inputs <data>
3. System validates <what>
4. System processes <action>
5. User sees <result>

### 5.2 Error Path: <Scenario>
1. User inputs <invalid data>
2. System returns <specific error message>
3. User sees <recovery guidance>
```

---

## Section 6: Functional Requirements (REQUIRED)

The core of the PRD. Each requirement has an ID, priority, and testable
acceptance criteria. Use RFC 2119 language.

Priority levels:
- **P0 (Must Have)**: MVP fails without this.
- **P1 (Should Have)**: Important, but MVP can ship without it.
- **P2 (Nice to Have)**: Only if time permits.

```markdown
## 6. Functional Requirements

### REQ-001: <Requirement Title> [P0]
The system MUST <specific behavior>.

**Acceptance Criteria:**
- GIVEN <context> WHEN <action> THEN <expected result>
- GIVEN <context> WHEN <edge case> THEN <expected handling>

### REQ-002: <Requirement Title> [P1]
The system SHOULD <specific behavior>.

**Acceptance Criteria:**
- GIVEN <context> WHEN <action> THEN <expected result>
```

---

## Section 7: Non-Functional Requirements (REQUIRED)

Performance, security, accessibility, reliability. Each must be testable.

```markdown
## 7. Non-Functional Requirements

### NFR-001: Performance
- Page load time MUST be < 2s on 3G connection
- API response time MUST be < 500ms at p95
- Database queries MUST NOT exceed 100ms at p95

### NFR-002: Security
- All user input MUST be sanitized before storage
- Authentication tokens MUST expire after <duration>
- Sensitive data MUST be encrypted at rest

### NFR-003: Accessibility
- MUST meet WCAG 2.1 AA standards
- All interactive elements MUST be keyboard-navigable
```

---

## Section 8: Technical Architecture (REQUIRED)

Stack, data model, and integration points. Enough for an agent to make
correct architectural decisions, but not so much that it wastes tokens
repeating discoverable information.

```markdown
## 8. Technical Architecture

### 8.1 Stack
- Runtime: <Node 22 / Python 3.12 / etc.>
- Framework: <Next.js 15 / FastAPI / etc.>
- Database: <PostgreSQL 16 / SQLite / etc.>
- ORM: <Prisma / Drizzle / SQLAlchemy / etc.>
- Styling: <Tailwind CSS 4 / etc.>
- Testing: <Vitest / Pytest / etc.>

### 8.2 Data Model

<Entity definitions with fields, types, constraints, and relationships.
Use a table or code block for clarity.>

#### Entity: <Name>
| Field       | Type      | Constraints          | Description           |
|-------------|-----------|----------------------|-----------------------|
| id          | UUID      | PK, auto-generated   | Unique identifier     |
| title       | string    | NOT NULL, max 255    | Display name          |
| created_at  | timestamp | NOT NULL, default NOW | Creation timestamp    |

#### Relationships
- <Entity A> has many <Entity B>
- <Entity B> belongs to <Entity A>

### 8.3 API Contracts (CONDITIONAL: include when the feature has APIs)

For each endpoint, specify method, path, request body, response body,
and error responses with realistic example data.

#### POST /api/v1/<resource>
**Request:**
```json
{
  "title": "Example Resource",
  "category": "demo",
  "metadata": { "key": "value" }
}
```

**Response (201):**
```json
{
  "id": "res_abc123",
  "title": "Example Resource",
  "category": "demo",
  "created_at": "2026-05-12T14:30:00Z"
}
```

**Error (422):**
```json
{
  "error": "validation_error",
  "details": [
    { "field": "title", "message": "Title is required" }
  ]
}
```

### 8.4 File Structure

Explicit directory tree showing where new files go. If working in an
existing repo, show only the additions/changes, not the entire tree.

```
src/
  modules/
    <feature>/
      <feature>.controller.ts    -- Route handlers
      <feature>.service.ts       -- Business logic
      <feature>.repository.ts    -- Data access
      <feature>.types.ts         -- Types and interfaces
      <feature>.test.ts          -- Unit tests
  lib/
    validators/
      <feature>.schema.ts        -- Zod/Joi validation schemas
```
```

<Seed data for development/testing>

## Section 10: Test Plan (REQUIRED)

What to test, how to test it, and when a module is "done."

```markdown
## 10. Test Plan

### Test Strategy
- Unit tests for all business logic (target: >80% coverage)
- Integration tests for API endpoints
- E2E tests for critical user flows

### Test Cases by Module

#### Module: <Name>
| ID      | Scenario                        | Type        | Expected Result       |
|---------|---------------------------------|-------------|-----------------------|
| TC-001  | Create resource with valid data | Integration | Returns 201 + body    |
| TC-002  | Create resource without title   | Integration | Returns 422 + error   |
| TC-003  | List resources with pagination  | Integration | Returns paginated set |

### Definition of Done
A module is complete when:
- [ ] All P0 requirements pass acceptance criteria
- [ ] All test cases pass
- [ ] No linting or type errors
- [ ] Code reviewed (or agent self-review against conventions)
```

---

## Section 11: Build Order (REQUIRED)

Numbered, dependency-sorted phases. Each phase has a clear deliverable
and a Definition of Done. Phase N never depends on Phase N+1.

This is the section agents use to sequence their work. Be precise.

```markdown
## 11. Build Order

### Phase 1: Foundation
**Deliverable:** <what exists when this phase is done>
**Tasks:**
1. <Specific task with file path>
2. <Specific task with file path>
**Definition of Done:**
- [ ] <Testable checkpoint>
- [ ] <Testable checkpoint>

### Phase 2: Core Logic
**Depends on:** Phase 1
**Deliverable:** <what exists when this phase is done>
**Tasks:**
1. <Specific task>
2. <Specific task>
**Definition of Done:**
- [ ] <Testable checkpoint>

### Phase 3: Integration and Polish
**Depends on:** Phase 1, Phase 2
**Deliverable:** <what exists when this phase is done>
**Tasks:**
1. <Specific task>
2. <Specific task>
**Definition of Done:**
- [ ] All P0 requirements pass
- [ ] Test suite green
- [ ] No TypeScript/lint errors
```

## Section 12: Protected Behaviors (REQUIRED)

Explicit list of things the implementation MUST NOT break. Agents lack
institutional knowledge. Without this section, they will confidently
refactor code that other systems depend on.

```markdown
## 12. Protected Behaviors

The following existing behaviors MUST NOT be altered by this
implementation:

- <Endpoint/function>: <why it is protected>
- <Data format>: <downstream consumers>
- <Authentication flow>: <dependent services>

If any protected behavior must change, stop and discuss with the user
before proceeding.
```

## Section 13: Open Questions (CONDITIONAL)

Include when the Discovery Interview surfaced questions that the user
could not answer yet. Mark each with an owner and a deadline.

```markdown
## 13. Open Questions

| # | Question                                | Owner   | Deadline   | Impact if Unresolved      |
|---|-----------------------------------------|---------|------------|---------------------------|
| 1 | Which payment provider for invoicing?   | @user   | 2026-05-20 | Blocks Phase 3            |
| 2 | Retention policy for audit logs?        | @user   | 2026-05-25 | Default to 90 days        |
```

## Section 14: Assumptions (CONDITIONAL)

Include when you made assumptions during the Discovery Interview that
the user did not explicitly confirm. Each assumption is a risk.

```markdown
## 14. Assumptions

- [ASSUMPTION] PostgreSQL 16 is the target database. If different,
  update Section 8.1 and all migration code.
- [ASSUMPTION] Authentication uses Clerk. If self-hosted auth, the
  auth middleware in Phase 1 must be rewritten.
```

## Formatting Rules (always apply)

1. Use ATX-style headings (`#`, `##`, `###`). No setext underlines.
2. Code blocks use triple backticks with language identifiers.
3. Tables use pipe syntax with alignment.
4. Requirements use `MUST`, `MUST NOT`, `SHOULD`, `MAY` per RFC 2119.
5. File paths are always monospaced: `src/modules/auth/auth.service.ts`.
6. JSON examples use realistic data, not `"string"` or `"example"`.
7. No filler prose. Every sentence either constrains, instructs, or
   exemplifies. If it does none of those, delete it.
8. Maximum line length: 80 characters in Markdown source for readability
   in terminals and editors.
