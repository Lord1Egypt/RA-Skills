# 三层记忆架构

## 为什么需要三层

AI 助手（LLM）的记忆天生有三个矛盾：

| 矛盾 | 说明 |
|------|------|
| 上下文窗口有限 | 能记住的只有当前会话的几万 tokens |
| 需要长效记忆 | 你的偏好、系统配置、踩过的坑不能被冲走 |
| 但全量保存不现实 | 每日对话几万字，全塞进系统 prompt 迟早撑爆 |

三层架构就是为了解决这三个矛盾——**各层管各自的事，互不取代**。

---

## 第一层：Sessions（原始运行层）

**位置**：OpenClaw 内部存储

**职责**：记录每一次对话的原始内容，当前会话上下文就在这一层

**为什么存在**：
- 第一手数据源，不加工不压缩
- conversations.db 和日志的数据来源
- 用来回溯对话上下文、分析历史模式

**你需要做的事**：几乎不用管。OpenClaw 自动管理 sessions。只需要确保不主动清理 sessions 数据即可。

---

## 第二层：conversations.db（结构化存档 + 语义检索层）

**位置**：`workspace/conversations.db`

**职责**：把历史对话切块、向量化、建立全文索引，随时可检索

**数据结构**：

| 表 | 内容 | 数量参考 | 说明 |
|----|------|---------|------|
| `chunks` | 对话文本块 | 几千~几万条 | 每次会话被切成若干文本块存储 |
| `embeddings` | 语义向量 | chunks 的一部分 | 用 embedding 模型（如 bge-m3）生成 1024 维向量 |
| `chunks_fts` | FTS5 全文索引 | 同 chunks | 支持关键词精确搜索 |

**它是怎么工作的**：

```
新对话发生
  ↓
写入 sessions
  ↓
import_sessions.py 将 sessions 切块 → 写入 chunks
  ↓
memory-core 插件对 chunks 生成 embedding → 写入 embeddings
  ↓
FTS5 对 chunks 建立全文索引
  ↓
当 AI 需要回忆时：
  1. memory_search() → embedding 语义检索 → 推荐最相关 chunks
  2. FTS5 → 关键词精确匹配
```

**OpenClaw 中的实现**：

1. **memory-core 插件**：负责将 chunks 转为向量，存入 embeddings 表
2. **memory_search**：内置工具，直接从 conversations.db 做语义搜索
3. **conversations skill**（我们另传的）：负责将 sessions 数据导入 conversations.db

**你需要做的事**：
- 安装并启用 memory-core 插件
- 配置 embedding 模型（推荐 bge-m3，可本地 ollama 部署）
- 定期运行 conversations skill 导入新对话

---

## 第三层：workspace 文件（人工提炼层）

**位置**：`workspace/` 下的 `.md` 文件

**职责**：从海量对话中提炼出"真正有用的信息"，以可读的格式长期保存

### 文件清单与职责

| 文件 | 职责 | 更新频率 |
|------|------|---------|
| `USER.md` | 你是谁——基本信息、偏好、忌讳、技术背景 | 有变化时 |
| `SOUL.md` | AI 是谁——性格、说话方式、铁律 | 大版本迭代 |
| `MEMORY.md` | 长期事实记忆——系统配置、重大决策、时间线 | 整理时 |
| `AGENTS.md` | 工作手册——流程规范、铁律、启动规则 | 规则变更时 |
| `IDENTITY.md` | AI 的身份简历——一句话介绍自己 | 几乎不动 |
| `TOOLS.md` | 工具与环境笔记——命令、配置、踩坑 | 变动时 |
| `HEARTBEAT.md` | 自检清单——定期检查项 | 几乎不动 |
| `memory/YYYY-MM-DD.md` | 每日工作日志——今天发生了什么 | 每次会话 |
| `memory/journal-YYYY-MM-DD.md` | AI 私人日记——AI 视角的"今天所思" | 整理时 |
| `self-improving/Mistakes-v2.md` | 纠错基因库 | 实时 + 整理 |
| `self-improving/Habits-v2.md` | 优势强化基因库 | 实时 + 整理 |

### 每层的原则

**USER.md/MEMORY.md 的原则**：
- "简历式精华"，不是"详细履历"
- 只放经你确认的、长期有效的核心信息
- 带置信度标注（已确认 / 推断 / 待验证）

**AGENTS.md 的原则**：
- 铁律必须可落地、可校验
- 规则来源于真实踩坑，不凭空造规则
- 同一问题出现 3 次才升级为铁律

**日志的原则**：
- 全部记录，不删不减
- 可以精简，不能删除
- 写前确认：有没有已有内容？这是追加还是覆盖？

**日记的原则**（可选）：
- "我"的视角 —— AI 看起来不是人类，但它的视角值得写下来
- 有"人味儿"——有画面、有真实情绪、有对用户的观察

## 三层之间的数据流

```
你说了什么
  ↓
【第一层：sessions】— 原始对话，AI 实时在上下文里看到
  ↓
conversations skill 定时导入
  ↓
【第二层：conversations.db】— 切块 + 向量化 + 全文索引
  ↓
memory_search() 在每次会话中召回相关历史
  ↓
你发现 AI 重复犯错 / 有重要结论 → 说"整理一下"
  ↓
【第三层：workspace 文件】— 提炼进 MEMORY + 日志 + 基因库
  ↓
下次会话，AI 加载第三层文件 + 第二层检索结果 → 回答问题
```

**核心要点**：
- 第二层（DB）负责任意查询：AI 需要回忆某件事，靠语义搜索找到相关内容
- 第三层（文件）做精炼沉淀：识别出"这个经常被问到"→ 写进 MEMORY，减少未来每次都要搜索
- 第一层（sessions）是数据源头，留给系统自己管

## 用户控制与数据安全

### 用户拥有所有数据的所有权

- 所有记忆文件（USER/MEMORY/AGENTS/日志）存储在本地 workspace，用户完全控制
- conversations.db 仅用于语义检索，不对外传输
- 用户可随时查看、编辑、删除任何文件

### 数据保留策略

- 默认保留所有对话日志和记忆文件，用户可随时指定例外
- 建议用户定期备份 workspace（首次使用本 skill 时已提醒）
- 用户可要求 AI：不记录某段对话、删除特定日期的日志、清空整个基因库

### 暂停机制

- 会话启动时的文件读取可被跳过（用户说"今天不用看记忆"）
- 实时学习机制可被暂停（AI 停止自动记录）
- 整理流程可被取消（已写入内容不受影响）

## 常见问题

**Q：为什么不直接用第二层代替第三层？**
A：第二层是模糊检索——每次都要搜 8000 个 chunks，结果不一定准，模型消耗也大。第三层是精确提取——你确认过的结论，直接写进 MEMORY，每次加载就能看到，不用搜。两层互相补位，不是谁取代谁。

**Q：文件越来越多会不会撑爆上下文？**
A：会。但每个文件控制在合理大小（MEMORY 通常 <50 行核心内容，日志只加载最近 1-2 天），总量不大。OpenClaw 的 memory-core 也有 compaction 机制回收闲置空间。

**Q：跨平台换模型怎么办？**
A：文件即本体。把 SOUL/USER/MEMORY/AGENTS 这几份核心文件带到新平台，AI 读完就"复活"。对话记忆通过 conversations.db 同步，consistency 由平台 embedding 模型保证。

**Q：数据安全和隐私怎么保证？**
A：所有数据存储在用户本地硬盘，不会上传到第三方。用户在首次安装本 skill 时已收到备份和安全提示。AI 在读取或写入记忆文件前会向用户说明行为。用户可以随时暂停、删除或回滚任何记忆数据。
