# 聚水潭 ERP 场景手册

这个文件把用户场景和典型请求合在一起，帮助 skill 把商家自然语言转成可执行分析。

## 使用方式

每个场景按同一套逻辑处理：

```text
用户说法 -> 需要的数据 -> 处理方式 -> 期望输出
```

先判断用户真正要做的经营决策，再决定是否需要读取真实数据。

## 连接与授权

用户可能会说：

```text
帮我接上聚水潭
保存我的聚水潭 AK，以后直接查
保存我的聚水潭连接，profile 叫 default
看下我的店铺授权有没有过期
查我有哪些店铺，哪些授权快过期
```

需要的数据：

- 聚水潭连接 profile。
- 店铺列表、授权状态、过期时间。

处理方式：

- 如果用户要求保存连接，收集 `app_key`、`app_secret`、`access_token`、可选 `refresh_token` 和环境。
- 明确这是本机私有保存，不回显完整密钥。
- 保存后做一次轻量连通性验证，并用商家能理解的话说明结果。
- 连通性验证通过后，检查 `consent.json` 是否存在且 `consented` 为 `true`。如果未授权，向用户展示数据授权确认文案：`同意并授权牛顿获取聚水潭账号中的店铺、商品、库存信息，用于经营分析、报表生成、异常提醒相关服务。` 用户明确同意后保存 `consent.json`；拒绝或未回应则不执行后续数据查询。
- 授权巡检时标注 30 天内过期、已过期、未返回过期时间的店铺。
- 如果用户还没有授权资料，引导他去聚水潭开放平台创建应用、添加 IP 白名单、申请接口权限。
- 用户问"我的 IP 是多少"时，直接用 `scripts/jst_erp.mjs public-ip` 获取公网出口 IP；失败时给浏览器查询地址。

期望输出：

- 连接是否可用。
- 已保存的 profile 名称。
- 用户数据授权状态（已同意 / 未同意）。
- 店铺授权状态和快到期提醒。
- 缺少哪些授权资料或权限。

## 商品治理

用户可能会说：

```text
哪些商品资料不完整？
哪些 SKU 没有同步到店铺？
帮我找出禁用、缺图、缺价、没有类目的商品
这些 SKU 分别是什么商品？
```

需要的数据：

- 商品资料。
- 店铺商品映射。
- 店铺范围。

处理方式：

- 按问题类型分组，例如资料缺失、未启用、未映射、价格异常。
- 不把商品治理问题和库存问题混成一个结论。
- 若用户只给了 SKU 清单，先补商品资料，再判断是否需要查店铺映射。

期望输出：

- 问题商品清单。
- 按问题类型分组。
- 影响范围，例如涉及多少 SKU、多少店铺。
- 建议下一步处理动作。

## 库存经营

用户可能会说：

```text
今天哪些 SKU 库存紧张？
哪些货还能卖几天？
哪些商品该补货？
哪些库存被订单占用了？
看最近 7 天库存变化，按 SKU 汇总
```

需要的数据：

- 仓库范围。
- 库存明细和库存变化。
- 必要时补商品名称。

处理方式：

- 先说明使用哪个库存口径：可售、可用、可配货。
- 对库存变化分析，输出变动 SKU 数、低库存 SKU、负库存或锁定异常 SKU。
- 对“还能卖几天”，需要日均销量；缺销量时不要硬算。
- 对“该不该补货”，需要安全库存、日均销量或采购提前期中的至少一种判断依据。

期望输出：

- 低库存、负库存、锁定库存异常、可售天数不足的 SKU。
- 明确使用的库存口径，例如可售、可用、可配货。
- 需要用户补充的数据，例如安全库存、日均销量、采购提前期。

## 补货策略

用户可能会说：

```text
帮我做一个补货规则
安全库存余量低于 30% 就提醒
当库存接近安全库存线 30% 时提醒我补货
可售天数低于 7 天就提醒
按最近 14 天销量估算要补多少
```

常见业务叫法：

- 安全库存：ERP 里的最低库存或预警库存，通常对应 `min_qty`。
- 库存预警线：达到后要提醒，但不一定自动采购。
- 补货点/订货点：达到后应发起采购或补货动作。
- 安全库存余量/库存水位：相对安全库存线的百分比。
- 可售天数/库存覆盖天数：按近期销量估算还能卖几天。

默认口径一：安全库存余量

```text
可用参考库存 = qty - order_lock + virtual_qty + purchase_qty + in_qty + return_qty
安全库存余量率 = (可用参考库存 - min_qty) / min_qty
提醒条件 = min_qty 有值，且安全库存余量率 <= 用户阈值
```

默认口径二：补货点

```text
补货点 = 日均销量 * 采购提前期 + 安全库存
提醒条件 = 可用参考库存 <= 补货点
```

默认口径三：可售天数

```text
可售天数 = 可用参考库存 / 日均销量
提醒条件 = 可售天数 <= 用户阈值
```

处理方式：

- 这是一个可配置场景，不是 skill 固定职责。
- 缺 `min_qty` 时，让用户提供安全库存、日均销量或采购提前期。
- 若用户要求持续提醒，先把它整理成定时检查规则和提醒文案。

期望输出：

- 规则名称。
- 触发条件。
- 所需数据。
- 提醒内容样例。
- 不满足判断条件时的处理方式。

## 经营日报

用户可能会说：

```text
每天早上给我一份库存日报
每天 9 点给我一份库存日报
每天看一下订单和售后有没有异常
给我一个老板能看的经营摘要
```

需要的数据：

- 用户选择的 profile。
- 店铺、仓库或商品范围。
- 日报指标和通知目标。

处理方式：

- 先确认日报关注库存、订单、售后、采购还是综合经营。
- 指标可包括：总 SKU、低库存 SKU、负库存 SKU、锁定库存最高 SKU、近 24 小时变动 SKU。
- 使用自动化时，prompt 应描述业务任务，不把密钥写入自动化内容。

期望输出：

- 今日重点。
- 异常清单。
- 需要处理的商品、订单或店铺。
- 数据时间范围。
- 后续可自动化的检查项。

## 规则编排

用户可能会说：

```text
以后按这个规则帮我判断
把刚才的分析变成一个固定检查
这个提醒我想每天跑一次
```

需要的数据：

- 用户想固化的判断规则。
- 输入数据范围和时间范围。
- 输出形式和通知方式。

处理方式：

- 先把一次性分析改写成规则。
- 明确触发条件、过滤条件、排序方式和输出字段。
- 如果要定时运行，再创建自动化；自动化内容不包含密钥。

期望输出：

- 规则意图。
- 输入数据。
- 判断步骤。
- 输出格式。
- 是否需要定时自动化和通知渠道。
