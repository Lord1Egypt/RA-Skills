# OPC领域清单与依赖拓扑

一人公司按价值链组织的任务类型清单，附任务间依赖关系和能力需求代号（S调研/C设计/A执行/O财务/I获客/G合规）。

**域间逻辑流**：P0(工作流重构) → P1(商业定位) → P2(商业模式) → P3(产品与交付) → P4(获客与品牌) → P5(运营与自动化) → P6(财务与合规)

> **P0是元能力域**：任何领域进入OPC之前，先判断传统工作流是否需要重构。需要则先重构，再进入P1-P6运营。

## P0 工作流重构

| ID | 任务类型 | 说明 | 依赖 | 能力需求 | UTOS映射提示 |
|----|---------|------|------|---------|-------------|
| P0-01 | 传统工作流识别 | 列出目标领域的传统工作流全链路：环节数、参与角色、中间文档、协作节点 | 无（入口） | 调研 | S |
| P0-02 | 环节存在理由分析 | 对每个环节追问：存在是因为事情本身需要，还是人的局限需要？标记为核心/校准/传递/协调/校验/格式 | P0-01 | 调研→设计 | S→C |
| P0-03 | 人的局限补偿层消除 | 去掉传递环节、协调环节、格式环节，精简校验环节为关键节点保留 | P0-02 | 设计 | C |
| P0-04 | 重整为IPO基元链 | 将保留的核心环节和关键校验节点，按AI能力模型重编为端到端IPO基元或简短基元链 | P0-03 | 设计→执行 | C→A |
| P0-05 | 重构验证 | 验证重构后：事情完整性、补偿层已消除、校准不丢失、端到端可执行、复杂度回归、质量守恒、合规不跳过 | P0-04 | 调研→合规 | S→G |
| P0-06 | 执行形态选择 | 根据重构结果选择执行形态：单步IPO / 简短基元链 / IPO+人工决策 | P0-05 | 设计 | C |

## 依赖拓扑摘要

以下为任务间的主要依赖链路，可据此组织执行顺序：

**重构链路（任何领域进入前置）**: P0-01 → P0-02 → P0-03 → P0-04 → P0-05 → P0-06
**主链路（0→1启动路径）**: P1-01 → P2-01 → P3-01 → P4-02 → P5-01 → P6-01
**获客闭环**: P4-01 → P4-04 → P2-04 → P4-01(↻优化)
**产品化链路**: P3-02 → P3-03 → P5-02 → P3-02(↻迭代)
**风控链路**: P6-04 → P6-01 → P6-05 → P6-03 → P6-02
**能力定位链路**: P1-04 → P2-01 → P5-01 → P5-02

> **执行策略**：如果目标领域已有成熟的传统工作流，先走P0重构链路，重构后再进入P1-P6。如果是从零构建（无传统工作流可重构），直接进入P1。

更多组合根据具体任务动态推导。
