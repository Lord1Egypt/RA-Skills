# Harness Engineering — OpenAI 智能体工程实战经验

> **来源**：OpenAI 官方博客 (Ryan Lopopolo, 2026-02-11)  
> **URL**：https://openai.com/zh-Hans-CN/index/harness-engineering/  
> **蒸馏者**：Claw, 2026-04-17  
> **实验规模**：3名工程师 × 5个月 × ~100万行代码 × ~1500 PR（无人工编码）

---

## 1. 实验背景与核心成果

OpenAI 内部团队进行了一项极端实验：**完全禁止人工编写代码**，所有代码由 Codex 智能体完成。

**量化成果：**
- 约 1,500 个 Pull Requests
- 3名工程师，平均每人每天 **3.5 个 PR** 的吞吐量
- 完成时间约为传统人工编写的 **1/10**

**核心洞察：** 这不是"AI辅助编程"，而是真正的"智能体优先（Agent-First）"——人类从代码作者转变为**环境设计者**。

---

## 2. 工程师角色的根本转变

### 旧模式（人工编码）
工程师 → 直接写代码 → 提交 PR

### 新模式（智能体优先）
工程师 → 设计环境 + 明确意图 + 构建反馈回路 → 智能体执行

**关键转变点：**

1. **深度优先分解**：将大目标拆解为"设计 → 代码 → 评审 → 测试"等小构建模块，提示智能体逐块完成
2. **受阻时的正确反应**：不是"再努力"，而是问："还需要什么**能力**？如何让这个能力对智能体既清晰又可强制执行？"
3. **评审的演进**：逐渐从人工评审过渡到**智能体对智能体**的评审方式
4. **交互方式**：人类几乎完全通过提示（prompt）与系统交互

## 3. 情境管理（Context Management）

这是整个框架中最关键的工程挑战。

### ❌ 失败模式：巨型 AGENTS.md

常见错误：把所有规则写进一个大的 `AGENTS.md` 文件。

**失败原因分析：**
- 情境（context window）是稀缺资源，大文件挤占任务相关内容
- 规则过多时，智能体进行**本地模式匹配**而非有意识地导航
- 文件迅速腐烂，变成"陈旧规则的坟场"
- 难以核实一致性，漂移不可避免

### ✅ 成功模式：目录式 AGENTS.md

`AGENTS.md` 保持约 **100行**，作为"地图"或"内容目录"，指向更深层的专项文档。

**推荐仓库知识库布局：**

```
AGENTS.md              # ~100行，作为目录/地图
ARCHITECTURE.md        # 系统架构概览
docs/
├── design-docs/
│   ├── index.md       # 设计文档索引（含验证状态）
│   ├── core-beliefs.md
├── exec-plans/
│   ├── active/        # 进行中的计划（含进度+决策日志）
│   ├── completed/     # 已完成的计划
│   ├── tech-debt-tracker.md
├── generated/
│   └── db-schema.md   # 自动生成的数据库schema文档
├── product-specs/
├── references/
├── DESIGN.md
├── FRONTEND.md
├── PLANS.md
├── PRODUCT_SENSE.md
├── QUALITY_SCORE.md
├── RELIABILITY.md
└── SECURITY.md
```

**渐进式披露原则（Progressive Disclosure）：**
- 智能体从小而稳定的切入点开始
- 通过目录结构逐步被引导到深层知识
- 避免在入口文件中过载信息

**文档处理要点：**
- 设计文档需要被编目和索引，包含**验证状态**（已验证/待验证/已废弃）
- 计划（Plans）作为一等公民：有进度追踪、决策日志，提交到代码仓库
- 每次讨论决策后，必须更新仓库文档——否则智能体会像"迟到三个月的新员工"

## 4. 可读性优化：让应用对智能体可见

突破人工QA瓶颈的关键：让应用程序的 UI、日志和指标对智能体**直接可读**。

### UI 层可读性

- 应用可根据 **git worktree** 启动，Codex 可为每次更改启动独立实例
- 接入 **Chrome DevTools 协议**，创建处理 DOM 快照、截图和导航的技能
- 智能体能够：复现错误 → 验证修复 → 直接推理 UI 行为

### 可观测性层

- 日志、指标、追踪记录通过**本地可观测性堆栈**展示给智能体
- 智能体可用 **LogQL** 查询日志，用 **PromQL** 查询指标
- 这使以下提示成为可行任务：
  - "确保服务启动在 800ms 内完成"
  - "关键用户旅程的跨度不得超过两秒"

### 运行时长

单次 Codex 运行可持续工作超过 **6小时**（通常在人类睡眠时间运行长任务）。

## 5. 架构约束：通过不变量强制执行

**核心原则：不微观管理实施过程，而是通过强制执行不变量（invariants）来保证架构健康。**

### 分层架构模型

每个业务域严格按照以下层次划分，依赖方向单向验证：

```
Types → Config → Repo → Service → Runtime → UI
```

- **横切关注点**（认证、连接器、遥测、功能标志）只能通过 **Providers** 这一单一显式接口进入
- 违反分层约束 → 自动化检测 → 立即拒绝
- 这给智能体一个清晰的"地图"，无论它在哪里进行修改

### 品味不变式（Taste Invariants）

通过工具强制执行的"代码品味"：

| 不变量类型 | 实现方式 |
|-----------|---------|
| 结构化日志记录 | 自定义 linter 静态检查 |
| 模式和类型命名约定 | linter 强制 |
| 文件大小限制 | 构建时检查 |
| 平台可靠性要求 | 测试套件 |
| **错误信息中注入修复指令** | 运行时提示 |

> **关键技巧**：在自动化错误信息中嵌入修复指引，智能体看到错误后可直接自我修正，无需人工介入。

### 技术选型偏好

- 偏好对智能体"可读"的技术：**可组合、API稳定、在训练集中表现良好**
- 某些情况下，让智能体**重新实现功能子集**比绕过不透明的上游库更便宜
- 示例：自行实现并发 map 辅助函数（而非使用 `p-limit` 风格包），以便与 OpenTelemetry 紧密集成

### ⚠️ 架构约束必须早期化（反直觉重要结论）

> 原文明确警告：**"这种严格分层架构通常要等到你拥有数百名工程师时才会推迟。对于编码智能体来说，这是一个早期的先决条件。"**

这直接否定了"先快速堆功能再治理"的传统路径：
- 智能体会**放大**代码库中已有的任何模式（好的坏的均会复制）
- 如果早期不强制架构不变量，AI 残渣会以指数级扩散
- **结论：对于智能体优先项目，架构约束是 Day 1 就必须到位的，不是 Day 100 的事**

## 6. 熵与垃圾收集：持续维护策略

### 问题：AI 残渣（AI Residue）积累

智能体会**放大和复制**代码库中已有的不均衡或低质量模式，形成正反馈恶化。

- 初始解法（人工）：每周五花 20% 时间手动清理 → **不可扩展**
- 演进路径：手动发现 → 将规则编码到系统 → 建立自动清理机制（见下方循环）

### 解决方案：黄金原则 + 循环清理

**黄金原则示例（Golden Rules）：**
1. 倾向使用共享实用程序包，而非手工编写的辅助工具（集中管理不变量）
2. 不使用"YOLO式"探测数据，而是验证边界或依赖类型化 SDK

**循环清理流程（Garbage Collection Loop）：**
```
定期任务：后台 Codex 扫描代码库
    ↓
检测偏差 → 更新质量等级
    ↓
发起针对性重构 PR
    ↓
大多数 PR 在 1 分钟内审查 → 自动合并
```

**类比**：小额持续偿还技术债务，就像垃圾回收器一样——不累积，随用随清。

## 7. 合并策略的调整

高吞吐量下，传统工程规范需要根本性调整：

| 传统模式 | 智能体优先模式 |
|---------|--------------|
| 严格阻塞门（CI全绿才合并） | 尽量减少阻塞合并门 |
| PR 生命周期长，充分讨论 | PR 生命周期很短 |
| 偶发测试失败导致长期阻塞 | 通过后续重跑解决，不无限期阻塞 |
| 等待成本被低估 | **纠错成本低，等待成本高** |

> **黄金法则**：在智能体优先系统中，快速迭代 + 持续纠错 > 追求零错误合并。

## 7.5 自我审查闭环：Ralph Wiggum 循环

原文描述了一个关键的**自我改进机制**，是实现"无需人工直接贡献代码"的核心：

```
Codex 在本地审核自身更改
    ↓
请求额外的智能体/人类审查
    ↓
响应反馈并修改
    ↓
循环往复，直到所有审阅人满意
    ↓
合并更改
```

> **意义**：这使智能体真正成为了"能自我完善的执行者"，而不仅仅是"代码生成器"。

## 8. 智能体自主水平的进展路线

随着开发环节被编码到系统中，智能体可驱动的完整端到端流程：

```
1. 验证代码库当前状态
2. 重现已报告的漏洞
3. 录制演示故障的视频
4. 实施修复措施
5. 通过运行应用验证修复
6. 录制第二个视频演示解决方案
7. 打开 Pull Request
8. 回应智能体和人类反馈
9. 检测并修复构建故障
10. 仅在需要人类判断时交由人工处理
11. 合并更改
```

**这是"完全自主"的终态，但需要大量前期系统建设作为基础。**

---

## 9. 文档即记录系统（Documentation as System of Record）

这是智能体工程中最容易被忽视、却至关重要的原则：

> **代码仓库是智能体能看到的全部。**

- 存储在 Google Docs 的内容 → 智能体不可见
- 存储在聊天记录中的决策 → 智能体不可见
- 存储在人们头脑中的判断 → 智能体不可见

**实践要求：**
- 所有情境（架构决策、产品原则、工程规范、团队文化）必须 push 到仓库
- 每次讨论后必须更新仓库文档
- 计划文档是一等公民，有进度和决策日志

---

## 10. 未解决的挑战

团队坦诚以下问题尚无定论：

1. 完全由智能体生成的系统中，**架构连贯性**如何随时间演变？
2. **人类判断力**在哪些方面能发挥最大杠杆作用？如何编码这种判断力？
3. 随着模型能力增强，现有系统设计将如何演变？

> **核心挑战始终集中在**：设计环境 + 构建反馈回路 + 建立控制系统。
