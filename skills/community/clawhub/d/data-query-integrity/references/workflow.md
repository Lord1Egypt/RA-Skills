# 数据查询完整性检查 — 详细工作流

## 背景

本 Skill 源自一次真实翻车：查询飞书日历 7 天数据时，API 返回被截断，但 agent 未检查 `has_more` 字段，反而脑补填充了不存在的候选人和职位，导致输出完全不可信。

## 翻车复盘

### 发生了什么

1. 用户要求查询上周（7天）所有面试安排，按职位汇总
2. Agent 一次性查询 7 天数据，返回数据量过大被截断
3. Agent 未检查 `has_more` 字段（实际为 true，数据不完整）
4. Agent 发现数据"看起来不完整"时，没有告知用户，而是用推测填充空白
5. 最终编造了 6 个不存在的候选人和 3 个不存在的职位

### 根因

| 问题 | 说明 |
|------|------|
| 数据截断未察觉 | 没有检查 `has_more` 字段确认完整性 |
| 脑补代替诚实 | 发现不完整时选择推测而非告知用户 |
| 规则无约束力 | "绝不脑补"只是口头承诺，没有强制机制 |

### 教训

- 规则写了不执行 = 废纸
- 靠自觉不可靠，需要可检查的流程
- 数据查询类任务必须有强制完整性校验

## 详细流程

### Step 1: 查询前 — 拆分范围

**为什么**：避免单次查询数据量过大导致截断。

**怎么做**：
- 日历查询超过 3 天 → 按天分段
- 列表类查询预估可能超上限 → 主动分页
- 记录计划：查什么、分几段、每段范围

**示例**：
```
查询上周（6/8-6/14）日历数据
计划：按天分段，每天独立查询
- 6/8-6/9
- 6/10-6/11
- 6/12-6/14
```

### Step 2: 查询后 — 完整性校验

**为什么**：确认数据完整才能使用，残缺数据会导致错误结论。

**检查项**：

1. **has_more 检查**
   - `has_more: false` → ✅ 数据完整，可以使用
   - `has_more: true` → ❌ 必须用 page_token 继续分页，直到 `has_more: false`
   - 无 `has_more` 字段 → ⚠️ 数据可能被截断，缩小范围重查

2. **error 检查**
   - 有 `error` 字段 → ❌ 查询失败，告知用户并重试
   - 超时/网关错误 → 缩小范围重试，不得用部分数据凑数

3. **数据量合理性**
   - 返回 0 条 → 确认查询条件是否正确
   - 返回异常少 → 标记"数据可能不完整"

**示例**：
```
查询 6/8-6/9 数据：has_more: false ✅
查询 6/10-6/11 数据：has_more: false ✅
查询 6/12-6/14 数据：has_more: false ✅
全部完整，可以汇总
```

### Step 3: 输出前 — 逐条溯源

**为什么**：确保输出的每一条数据都有据可查，杜绝脑补。

**怎么做**：
- 对每条记录的每个关键字段，确认能在 API 原始返回中找到
- 找不到 → 不输出该条
- 任何字段是推断的 → 标记为"[数据缺失]"

**示例**：
```
原始数据：
{
  "summary": "抄送我的视频面试：雷晶（智能产品经理（数据策略方向） - 数据智能部）",
  "start_time": "2026-06-08T15:00:00+08:00",
  "description": "候选人：雷晶<br/>应聘职位：智能产品经理（数据策略方向）..."
}

输出：
- 雷晶 · 智能产品经理（数据策略方向）· 6/8 15:00 ✅ 字段均可溯源

如果原始数据中没有"雷晶"，不得输出该条。
```

### Step 4: 输出时 — 声明数据来源

**为什么**：让用户知道数据的可靠性，有缺失时明确告知。

**怎么做**：
- 告知执行了几次 API 查询
- 告知每次查询的 `has_more` 状态
- 有任何不完整 → 明确说"以下数据可能不完整"及缺失范围

**示例**：
```
📊 上周面试汇总（2026.06.08 - 2026.06.14）

数据来源：3 次 API 查询（按天分段），has_more 均为 false，数据完整。

---
[汇总内容]
---
```

或：

```
📊 上周面试汇总（2026.06.08 - 2026.06.14）

⚠️ 数据来源：2 次 API 查询，其中 6/12-6/14 查询返回 has_more: true，
数据可能不完整。以下汇总仅包含已获取的数据。

---
[汇总内容]
---
```

## 四条红线

| 禁止行为 | 后果 |
|---------|------|
| 输出 API 返回中找不到的数据 | 编造不存在的人/职位，输出完全不可信 |
| 因"看起来应该有"而补全 | 主观推测，违反数据查询基本原则 |
| 用推测填充截断空白 | 同上，且掩盖了数据不完整的事实 |
| has_more=true 时停止分页 | 使用残缺数据，导致遗漏 |

## 违规自检

输出前问自己：
> "这条记录的每个字段，我都能在 API 返回的 JSON 中找到原文吗？"

如果有任何一条回答"不能" → 删除该条，不得输出。
