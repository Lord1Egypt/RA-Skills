# 专家调度协议

> 9 个专家随 guanshi 安装自动下载，运行时降级安全——加载失败不中断流程，Chief 自行填补。

## 流程概览

```
Phase 1: 信号扫描+问题树（Step 1）
    Chief 尝试调用已安装专家（最多 3 个） → 每专家返回 3 个关键信号
    → Chief 产出 MECE 问题树 + 框架适用性判定表（议程设置）
       ↓
Phase 2: 深度填空（Steps 2-5）
    Chief 只加载问题树中标记为"必须"或"部分"的框架
    → 信息缺口优先通过 web_search + web_fetch 填补
    → 已安装专家可作为辅助信息源（每个子问题 ≤ 500 字）
    → Chief 综合嵌入骨架
       ↓
Phase 3: 最终输出（Step 6 + 宪法审计 + Step 6.5 + [可选] 深度穿透）
    Chief 产出主要结论
    → [可选] DeepInsight 对关键结论进行穿透式审查（三层钻取+红队+张力映射）
    → 宪法审计（七维检查） + 情景规划压力测试
    → BLM 重构 + 自检 → 输出报告
```

## Phase 1：信号扫描

**目标**：Chief 自主扫描外部环境和内部状态，完成整体分析并形成报告骨架。

**执行方式**：
1. Chief 先尝试并行调用已安装的领域专家 skill（`use_skill`，最多 3 个），专家返回 3 个关键信号（每条含指标名/数值/趋势方向/来源/置信度，总字数 ≤ 200 字）
2. 未安装或调用失败的专家 → Chief 通过实时搜索自行填补该维度信号
3. 情报/数据维度默认由 Chief 通过 web_search 自行获取

**Chief 聚合后产出**：
- 全维度信号矩阵（一句话格局判断）
- 报告骨架（标题体系 + 每节论点 + 硬约束节点清单：OHI / BCG / 退出路线 / CEO 矛盾分析 / 渠道业绩等）
- 信息缺口清单（骨架中哪些地方数据/论证不够 → 拆为 Phase 2 子问题）

---

## Phase 2：深度填空

**目标**：Chief 将 Phase 1 识别的信息缺口通过实时搜索和框架分析自行填补，或派给已安装的专家 skill 辅助。

**执行方式**：
1. Chief 对每个缺口优先使用 web_search + web_fetch 获取最新数据
2. 已安装的专家 skill 可作为辅助信息源（`use_skill` 加载后提问），但每个子问题 ≤ 500 字回填
3. 若专家不可用 → Chief 自行填补，标注数据来源和置信度

---

## 调度规则

1. **专家优先尝试、不阻塞** → 尝试 `use_skill` 加载，失败则自动降级为 Chief 自行分析
2. **专家未安装** → Chief 用通用分析兜底，标注"⚠️ 缺乏专业专家支持"
3. **结论冲突** → Chief 必须指出冲突点和各自前提假设，不强行统一
