# PRD生成模块

## 1. 概述

PRD生成模块负责将用户的项目需求转化为结构化的产品需求文档（PRD），包含项目概述、技术栈、用户故事、验收标准等内容。

## 2. 需求澄清机制

在生成PRD前，需要向用户提出3-8个关键澄清问题，确保需求理解准确：

### 2.1 澄清问题示例

1. **项目目标**：您希望通过这个项目解决什么问题？预期的业务价值是什么？
2. **目标用户**：这个项目的主要用户群体是谁？他们的核心需求是什么？
3. **技术栈偏好**：您对前端、后端、数据库等技术栈有什么偏好或限制？
4. **项目规模**：这个项目的预计规模如何？是小型工具、中型应用还是大型系统？
5. **功能优先级**：如果需要取舍，哪些功能是必须的，哪些是可选的？
6. **集成需求**：项目需要与哪些现有系统或第三方服务集成？
7. **性能要求**：有哪些性能指标需要特别关注（如响应时间、并发用户数等）？
8. **安全要求**：项目需要考虑哪些安全方面的需求（如数据加密、权限控制等）？

## 3. PRD结构标准

### 3.1 基础结构

```markdown
# [项目名称] 产品需求文档

## 1. 项目概述
- 项目背景
- 项目目标
- 核心功能
- 目标用户

## 2. 技术栈
- 前端：[技术栈]
- 后端：[技术栈]
- 数据库：[技术栈]
- 其他：[技术栈]

## 3. 用户故事

### 3.1 故事1：[故事标题]
- **描述**：[故事描述]
- **验收标准**：
  1. [可验证的验收标准1]
  2. [可验证的验收标准2]
- **依赖关系**：[依赖的其他故事]
- **优先级**：[高/中/低]
- **类型**：[scaffold/backend/frontend/fullstack]

### 3.2 故事2：[故事标题]
...

## 4. 非目标（Out of Scope）
- [非目标1]
- [非目标2]

## 5. 成功指标
- [成功指标1]
- [成功指标2]
```

### 3.2 关键规则

1. **用户故事粒度**：每个故事必须足够小，能在一个上下文窗口内完成
2. **验收标准**：必须是可验证的，不能是模糊的描述
3. **依赖关系**：必须明确记录故事间的依赖关系
4. **故事类型**：必须为每个故事指定类型（scaffold/backend/frontend/fullstack）

## 4. 生成流程

1. **需求收集**：收集用户的项目需求描述
2. **需求澄清**：提出3-8个关键澄清问题
3. **PRD生成**：根据澄清后的需求生成结构化PRD
4. **保存输出**：将PRD保存到`prd-context/prd-files/prd-[功能名称].md`

## 5. 最佳实践

- **具体明确**：避免模糊的需求描述，确保每个功能都有明确的边界
- **可验证性**：验收标准必须可测试、可验证
- **优先级清晰**：明确区分核心功能和次要功能
- **依赖关系明确**：清晰记录故事间的依赖关系，避免执行顺序错误
- **技术可行性**：考虑技术栈的可行性和实现难度

## 6. 输出格式

生成的PRD应该保存为Markdown格式，文件命名为`prd-[功能名称].md`，存储在`prd-context/prd-files/`目录中。

## 7. 与校验模块的交互

生成的PRD将传递给PRD校验模块进行规范性检查，包括：
- 故事粒度是否足够小
- 验收标准是否可验证
- 依赖关系是否明确
- 故事类型是否正确分类

如果校验不通过，需要根据反馈进行修正，直到PRD符合规范。
