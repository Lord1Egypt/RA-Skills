# 机构模式 排障手册（境外）

> 本文档汇总境外（跨境）**机构**的高频报错与解决方案。
>
> ‼️ 用户报告接口报错 / 接口异常 / 回调收不到 / 签名失败 / 退款失败等），按本手册排障流程执行。
>
> ‼️ 排障完成后必须在回复末尾**主动推荐 [接入质量检查](../接入指南/接入质量检查.md)**（趁排障契机一次性排查其他潜在问题）。

## 一、排障流程

**错误码提取**：Request-Id 的格式为 {请求标识}-{错误码}，截取最后一个 - 后面的数字即为错误码。例如 08E68E97CC0610DF0618AC86BCA80120082880CF05-268578704 → 错误码 268578704。

1. **能给 Request-Id？** → 走「二、错误码 TOP（按 Request-Id 末尾错误码段定位）」。
2. **不能给 / 未命中？** → 走「三、HTTP 错误」「四、SIGN_ERROR 详细排查」「五、回调问题」「六、业务问题」按现象定位子节。
3. **两条都没命中？** → 用「七、排障信息收集清单」回收信息再判断。

## 三、错误码详细排查指引

> 本章对上述高频错误码逐一说明**产生原因**和**建议解决方案**，便于快速定位和处理。

### 268512771 — 订单不存在（软POS）

| 维度 | 说明 |
|------|------|
| **触发场景** | 查单 / 撤销 / 退款时，系统找不到对应订单 |
| **产生原因** | ① `out_trade_no` 传错或与下单时不一致；② 订单在另一个 `sub_mchid` 下，查错了子商户；③ 下单请求实际未成功（网络超时，订单未创建）；④ 软 POS 场景下终端与后台订单号不同步 |
| **解决方案** | ① 核对 `out_trade_no` 是否与下单时完全一致（区分大小写）；② 确认 `sp_mchid` + `sub_mchid` 组合正确；③ 若怀疑下单未成功，用原参数重新下单（幂等重入）；④ 软 POS 场景确保终端本地订单号与提交到后台的一致 |

### 270283348 — 错误的签名，验签失败

| 维度 | 说明 |
|------|------|
| **触发场景** | 所有 API 请求均可能触发 |
| **产生原因** | ① 私钥与证书不匹配；② 签名串构造错误（缺末尾 `\n`、URL 路径错、body 不一致）；③ 用了子商户私钥而非机构私钥；④ `serial_no` 填错（如填了平台证书序列号）；⑤ 时间戳偏差 > 5 分钟 |
| **解决方案** | 详见本文 **§ 四、SIGN_ERROR 详细排查**。核心检查：机构私钥 → 签名串 5 行格式 → 路径必须 `/v3/global/` → `truncated_sign_message` 字节级比对 |

### 268501011 — 合规范的授权码，但现在不支持

| 维度 | 说明 |
|------|------|
| **触发场景** | 付款码支付（Quick Pay）提交扣款时 |
| **产生原因** | ① 用户付款码已过期（超过 1 分钟未刷新）；② 付款码前缀不在 10-15 范围内（非微信付款码）；③ 该类型付款码在当前地区/场景暂不支持 |
| **解决方案** | ① 提示用户刷新付款码后重新扫码；② 收银前端增加付款码前缀校验（18 位，前缀 10-15），非微信码不提交；③ 确认该地区是否支持该钱包类型 |

### 268501834 — 刷卡单笔金额达到免密限额，要求用户输入密码

| 维度 | 说明 |
|------|------|
| **触发场景** | 付款码支付扣款时，API 返回 `USERPAYING` 状态 |
| **产生原因** | 单笔金额超过 3000 CNY（或等值外币），触发强制验密 |
| **解决方案** | 这是**正常业务流程**，非错误。① 启动轮询查单（间隔 3s、5s、10s...），等待用户输入密码；② 超过 30 秒未支付则调用撤销接口；③ 收银员提示用户在手机上输入密码 |

### 268443704 — 受理机构发起支付时，必须传入子商户 mchid

| 维度 | 说明 |
|------|------|
| **触发场景** | 机构模式下单/查单/退款等所有接口 |
| **产生原因** | 请求体中缺少 `sub_mchid` 字段，或字段值为空 |
| **解决方案** | 机构模式请求体**必填 4 字段**：`sp_mchid` + `sp_appid` + `sub_mchid`（+ 部分场景的 `sub_appid`）。确保每次请求都携带子商户号 |

### 1000001021 — 微信支付后台判定需要验密（用户查看了付款码原文）

| 维度 | 说明 |
|------|------|
| **触发场景** | 付款码支付，API 返回 `USERPAYING` 状态 |
| **产生原因** | 用户点击付款码展示区域查看了付款码原文（条码数字），触发安全验证机制 |
| **解决方案** | 同 `268501834` 的处理逻辑——这是**正常安全机制**。启动 USERPAYING 轮询查单 + 超时撤销流程 |

### 268464976 — 查询批次码记录不存在

| 维度 | 说明 |
|------|------|
| **触发场景** | 查询批次/批次码相关接口 |
| **产生原因** | ① 批次号 `batch_id` 传错；② 批次尚未创建成功；③ 查询的 `sub_mchid` 与创建时不一致 |
| **解决方案** | ① 核对批次号是否正确；② 确认批次创建接口已返回成功；③ 确保查询时的 `sp_mchid` + `sub_mchid` 与创建时一致 |

### 268522241 — 境外支付，证件类型不是身份证

| 维度 | 说明 |
|------|------|
| **触发场景** | 子商户进件或实名相关接口 |
| **产生原因** | 境外进件时证件类型字段填写了非身份证类型，但该场景要求身份证 |
| **解决方案** | ① 确认进件接口要求的证件类型；② 境外支付对中国大陆用户要求"居民身份证"类型；③ 按文档规范填写正确的证件类型枚举值 |

### 268576361 — 超过码展示后允许免密的时间（一般 60-70s）

| 维度 | 说明 |
|------|------|
| **触发场景** | 付款码支付，API 返回 `USERPAYING` 状态 |
| **产生原因** | 用户展开付款码后超过约 60-70 秒才被扫码，系统判定为安全风险，强制要求验密 |
| **解决方案** | 这是**正常安全机制**。① 提示用户刷新付款码后重新扫码（展开后尽快完成扫码）；② 若已触发验密，按 USERPAYING 标准流程处理（轮询 + 超时撤销）|

### 268435461 — 参数非法

| 维度 | 说明 |
|------|------|
| **触发场景** | 所有接口均可能触发 |
| **产生原因** | 请求参数格式不合法——可能是字段类型错误、必填字段缺失、字段值超出范围、JSON 格式错误等 |
| **解决方案** | ① 检查响应体 `detail` 中的 `field` / `value` / `issue` 字段定位具体参数；② 对照 API 文档逐项检查参数类型和格式；③ 常见错误：金额非整数、币种码错误、日期格式错误、枚举值拼写错 |

### 88430113 — 余额不足，请核实后再试

| 维度 | 说明 |
|------|------|
| **触发场景** | 退款 / 付款码支付（用户端余额不足） |
| **产生原因** | ① 退款场景：子商户 CNY 资金池余额不足（已换汇或当日无新交易入账）；② 支付场景：用户微信零钱 + 绑定银行卡余额不足 |
| **解决方案** | ① **退款余额不足**：机构层封装定时重试机制（保持原 `out_refund_no`），等待新交易 CNY 资金入池后自动重试；兜底方案为开通"商家充值退款余额"；② **用户余额不足**：提示用户更换支付方式 |

### 268443701 — 受理关系不存在

| 维度 | 说明 |
|------|------|
| **触发场景** | 下单 / 查单 / 退款等任何需要 `sub_mchid` 的接口 |
| **产生原因** | ① `sub_mchid` 填写错误；② 该子商户未在此机构下完成进件；③ 进件审核尚未通过或受理关系尚未生效 |
| **解决方案** | ① 核对 `sub_mchid` 是否正确（注意不要与 `sp_mchid` 混淆）；② 在商户平台确认该子商户的进件状态和受理关系；③ 进件刚完成时有短暂生效延迟，等待几分钟后重试 |

### 132021037 — 海关申报记录不存在

| 维度 | 说明 |
|------|------|
| **触发场景** | 海关报关查询接口 |
| **产生原因** | ① 报关单号 `customs_id` 传错；② 该订单未提交过报关申请；③ 查询的 `sub_mchid` / `out_trade_no` / `transaction_id` 与报关时不一致 |
| **解决方案** | ① 确认已对该订单调用过报关推单接口；② 核对三者一致：`out_trade_no` + `transaction_id` + 报关订单号；③ 报关推单与查询使用相同的 `sp_mchid` + `sub_mchid` |

### 268501838 — V3 API 暂不支持境外支付

| 维度 | 说明 |
|------|------|
| **触发场景** | 调用了境内 V3 API 路径而非境外 V3 路径 |
| **产生原因** | 使用了境内路径（如 `/v3/pay/partner/transactions/jsapi`）而非境外路径（`/v3/global/transactions/jsapi`）。此错误表明请求发到了境内 V3 系统，但该系统不处理境外交易 |
| **解决方案** | ① 将所有 API 路径改为 `/v3/global/...` 前缀；② 域名使用 `apihk.mch.weixin.qq.com`（非 `api.mch.weixin.qq.com`）；③ 参见 [📄 API版本对比](../接入指南/API版本对比.md) |

### 268447845 — 查单 sub_mchid 与订单不一致

| 维度 | 说明 |
|------|------|
| **触发场景** | 查单接口 |
| **产生原因** | 查单请求中填写的 `sub_mchid` 与该订单创建时的 `sub_mchid` 不同 |
| **解决方案** | ① 确保查单时传入的 `sub_mchid` 与下单时完全一致；② 检查系统中是否有多个子商户号导致混用；③ 机构侧建议以 `out_trade_no` 反查本地数据库确认该订单归属的 `sub_mchid` |

### 268449386 — 超过订单可撤销时限

| 维度 | 说明 |
|------|------|
| **触发场景** | 付款码支付撤销接口 |
| **产生原因** | 撤销仅支持**支付当天**操作，跨日后不可撤销 |
| **解决方案** | ① 当天的订单用撤销接口；② 跨日后改用**退款接口**处理；③ 开发时确保撤销逻辑在支付当天的超时窗口内触发 |

### 268449304 — 支付场景非法

| 维度 | 说明 |
|------|------|
| **触发场景** | 下单 / 调起支付时 |
| **产生原因** | ① 下单时指定的支付方式与实际调起方式不匹配（如 JSAPI 下单的 `prepay_id` 用在了 APP 中）；② `trade_type` 参数错误；③ 在不支持的环境中调起支付（如微信外调起 JSAPI） |
| **解决方案** | ① 确保下单产品类型与调起方式一致；② JSAPI 必须在微信内调起，H5 必须在微信外调起；③ 确认 `trade_type` 枚举值正确 |

### 268501784 — 已使用的付款码不可再使用

| 维度 | 说明 |
|------|------|
| **触发场景** | 付款码支付提交扣款时 |
| **产生原因** | 同一个 `auth_code`（付款码）已被用于一笔成功的交易，不能再用于另一笔 |
| **解决方案** | ① 提示用户刷新付款码（每分钟自动刷新）；② 同时生成新的 `out_trade_no`；③ **核心规则**：一个 `auth_code` 只能对应一个 `out_trade_no`，重新发起需同时更换两者 |

### 270279174 — 刷卡场景重复支付需要验密

| 维度 | 说明 |
|------|------|
| **触发场景** | 付款码支付，API 返回 `USERPAYING` 状态 |
| **产生原因** | 短时间内对同一用户多次发起付款码扣款，触发风控的"重复支付"验密机制 |
| **解决方案** | 这是**正常风控机制**。① 按 USERPAYING 标准流程处理（轮询 + 超时撤销）；② 检查系统是否有重复提交问题；③ 避免对同一用户短时间内多次提交 |

### 35312011 — 该订单没有申报

| 维度 | 说明 |
|------|------|
| **触发场景** | 海关报关相关查询/修改接口 |
| **产生原因** | ① 该 `out_trade_no` 对应的交易未提交过报关；② 报关推单接口调用失败但未发现；③ 订单号与报关记录不匹配 |
| **解决方案** | ① 先调用报关推单接口完成申报；② 确认推单接口返回成功后再查询；③ 核对 `out_trade_no` 与 `transaction_id` 的对应关系 |

### 268443838 — 子商户需补资料，无支付权限

| 维度 | 说明 |
|------|------|
| **触发场景** | 下单时 |
| **产生原因** | 子商户进件审核通过但需要补充额外资料，在补资料完成前暂时没有支付权限 |
| **解决方案** | ① 登录商户平台查看该 `sub_mchid` 的状态和待补资料项；② 联系子商户补充所需资料；③ 资料审核通过后支付权限自动恢复；④ 如紧急需要可联系技术支持 `WXP_GlobalTS@tencent.com` |

## 四、HTTP 错误（401 / 400 / 403 / 5xx）

| 状态码 | 含义 | 常见原因（机构特有补充）| 排查要点 |
|:---:|---|---|---|
| 401 | 签名验证失败 / 证书问题 | 私钥与证书不匹配；`serial_no` 错（用了平台证书号 / 子商户的）；签名串拼接错；时间戳偏差 > 5 分钟；**用了境内路径调境外接口（`/v3/pay/partner/...` 而非 `/v3/global/...`）**；**用了子商户私钥签名** | 详见第四节；**机构必须用机构自身私钥与序列号** |
| 400 | 请求参数错误 | 必填缺失（含 `sp_mchid` / `sp_appid` / `sub_mchid`）；`amount.currency` 非法；`sp_mchid` 与 `sub_mchid` 字段写反；时间格式错；金额单位错 | 优先检查机构 4 字段是否齐全；**金额单位为最小货币单位** |
| 403 | 权限不足 | 子商户未开通对应跨境支付产品；H5 域名不在子商户白名单；机构状态异常；跨境分账 / 海关报关 / H5 权限等业务未开通 | 商户平台 → 产品中心确认子商户开通状态；H5 看子商户域名管理 |
| 5xx | 服务端异常 | 极少出现，常见上游网络抖动 | 重试（带退避）；退款类**保持原 `out_refund_no`** 重试 |

## 五、401 SIGN_ERROR 详细排查

### 5.1 通用错误描述（来源 [证书与回调验签机制](https://pay.weixin.qq.com/doc/global/v3/zh/4012354982.md)）

| 错误描述（response message） | 原因 | 解决方案 |
|---|---|---|
| 商户未设置 APIv3 密钥 | 机构未设置 APIv3 密钥 | 登录境外商户平台（机构入口）设置 APIv3 密钥 |
| 商户未申请过证书 | 机构未申请 API 证书 | 参考商户 API 证书说明申请 |
| 商户证书序列号有误 | 用了错误 / 已失效的历史证书 | 登录境外商户平台查正确的证书序列号 |
| 商户证书已过期 | 证书过期 | 商户平台续期，使用续期后新证书 |
| 商户证书已作废 | 主动作废后又使用 | 重新申请证书后使用新的 |
| 错误的签名，导致验签失败 | 私钥错或签名串构造不正确 | 见下方"签名串构造五大常见错误" + "机构特有 3 类错误" |
| Http 头 Authorization 值格式错误 | 缺少 / 格式不正确 | 检查 Authorization 头 |
| Http 头 Authorization 认证类型不正确 | 不支持的算法 | 仅支持 `WECHATPAY2-SHA256-RSA2048` |
| timestamp 与发起时间相差超过 5 分钟 | 系统时间不准 | 检查 NTP / 获取时间逻辑 |
| 已更换证书，请使用新证书 | 商户重新申请了 API 证书 | 改用新申请的 |

### 5.2 签名串构造五大常见错误

来源：境外 [签名验证文档](https://pay.weixin.qq.com/doc/global/v3/zh/4012354989.md) `truncated_sign_message` 比对建议。

1. **签名串最后一行漏 `\n`**：GET 等 body 为空时，最后一行也必须保留 `\n`。
2. **URL 与实际请求不一致**：手工拼接 vs HTTP 库构造的差异；境外路径**必须 `/v3/global/...` 开头**。
3. **`timestamp` / `nonce_str` 前后不一致**：签名串与 `Authorization` 头里的值字节级一致。
4. **JSON 请求体序列化前后不一致**：先把 body 字符串保存为变量，签名时和发送时复用同一个；机构请求体字段多，**字段顺序也要稳定**。
5. **文本编码不一致**：必须 UTF-8。

### 5.3 机构特有的 3 类签名错误

1. **签名主体用错**：误用了其他机构号的私钥，必须用**和机构号sp_mchid对应的 API 证书私钥**。
2. **路径前缀错**：直接用了境内 `/v3/pay/partner/...` 或 `/v3/pay/transactions/...`，境外必须 `/v3/global/...`（如 JSAPI 下单 = `/v3/global/transactions/jsapi`，退款 = `/v3/global/refunds`，平台证书 = `/v3/global/certificates`，跨境分账 = `/v3/global/profit-sharing/orders`）。
3. **请求体里 `sp_mchid` / `sub_mchid` 字段顺序与最终序列化不一致**：建议固定字段顺序，或保存序列化后的 body 字符串复用。

## 六、回调问题

### 6.1 收不到回调排查清单（按优先级）

| # | 检查 | 现象 / 解法 |
|---|---|---|
| 1 | 通知地址未配置 | 通过下单时的 `notify_url` 字段 / 业务的「设置事件通知地址」API 配置；机构未配置不会收到任何回调 |
| 2 | 通知地址不可达 | URL 错 / `localhost` / 服务未启动 |
| 3 | URL 前后空格 | 致 DNS 失败 |
| 4 | 防火墙拦截 | 不要按"微信支付回调 IP"做白名单（官方不公开），用平台证书验签替代 |
| 5 | 登录态拦截 | 通知 URL 须从鉴权中间件中排除 |
| 6 | 响应非 200 | 微信支付按指数退避重试 |
| 7 | 处理超时 | 必须 5 秒内应答 |
| 8 | URL 协议 | 必须 HTTPS |
| 9 | 内网地址 | 不允许 `127.0.0.1` / `192.168.x.x` |
| 10 | **`sub_mchid` 路由错** | 用错 `sub_mchid` 把订单查到了别的子商户 → **串单事故**；解密后必须先按 `sub_mchid` 路由再处理 |

### 6.2 回调常见问题

| # | 问题 | 答案 |
|---|---|---|
| 1 | 怎么确认微信发了回调？ | 微信不提供回调日志查询；查自身服务器访问日志 + 用查单接口确认 |
| 2 | 回调会重复收到吗？ | 会，未正确响应时微信会重试，必须做幂等（机构维度幂等键建议加上 `sub_mchid`）|
| 3 | 回调延迟正常吗？ | 数秒到数十秒均属正常；建议回调 + 主动查单双保险 |
| 4 | 能直接将回调当最终结果吗？ | 不能，回调不保证送达，需结合查单接口确认 |
| 5 | 商户平台能查回调状态吗？ | 不支持，需调用查单接口 |
| 6 | 回调缺少 `Wechatpay-*` 几个头怎么办？ | 某些代理 / CDN 会过滤微信扩展 HTTP 头；调整代理配置或直连，并显式放行 4 个头 |
| 7 | 报错"Http 头缺少 Accept 或 User-Agent" | 必须显式设置 `Content-Type: application/json` / `Accept: application/json` / `User-Agent` |
| 8 | 回调里的 `nonce` 和签名串里的 `nonce_str` 是同一个吗？ | **不是**，两个独立的值 |
| 9 | 多业务回调怎么分发？ | 同一通知 URL 接收多业务时，按 `event_type` / `original_type` 路由到对应处理器 |

### 7.2 退款（来源境外 [退款 API 错误码](https://pay.weixin.qq.com/doc/global/v3/zh/4012354441.md) + [退款 FAQ](https://pay.weixin.qq.com/doc/global/v3/zh/4012355040.md) 原话）

| # | 报错 / 现象 | 原因 / 解法 |
|---|---|---|
| 1 | 退款失败后重试是否换 `out_refund_no`？ | **不要换**，否则会被认为是新单据，可能造成资金损失 |
| 2 | `BIZERR_NEED_RETRY` | 退款业务流程错误，**保持原 `out_refund_no`** 用相同参数再次调用 API |
| 3 | `OUT_REFUND_NO_USED` | 退款单号已使用，且与当前请求参数不同；换一个 `out_refund_no` 即可 |
| 4 | `NOT_ENOUGH` 余额不足 | 子商户当日 CNY 资金池余额不足。**首选方案**：机构 / 商户层封装定期重试（**保持原 `out_refund_no`** 等待新交易 CNY 资金入池后自动重试）。**兜底方案**：申请开通"商家充值退款余额"，预充 CNY 备用金后在请求里指定 `source = REFUND_SOURCE_RECHARGE_FUNDS` 从充值余额扣。详见 [📄 跨境支付通用知识 § 4.6.2](../产品选型/跨境支付通用知识.md#462-退款资金不足not_enough的解决方案) |
| 5 | 订单已超过 1 年退款期限 | 自交易完成日 1 年内可退；超期请走特殊申请 |
| 6 | 退款频率限制 | 错误请求 6 qps；一个月前订单 5000/min；同笔订单多次退款相隔 1 分钟 |
| 7 | 同一笔订单多笔并发退款 | 建议串行/分批，避免并发退款失败 |

### 7.3 多币种 / 跨境特有

| # | 问题 | 答案 |
|---|---|---|
| 1 | 下单时 `currency` 怎么填？ | ISO 4217 三字母币种码；按子商户实际收款币种填；**不要从前端传值** |
| 2 | 金额单位是什么？ | **最小货币单位**整数；USD / HKD 是分（100 = 1.00），JPY 是日元（100 = 100 JPY）|
| 3 | 回调里的 `amount.payer_currency` 与 `amount.currency` 不一致？ | 用户付币种与子商户收款币种不同，按 `payer_*` 与 `currency` 分别记账；具体折算见对账文件 |
| 4 | 对账文件按哪个维度？ | 按 **(`sub_mchid`, 币种)** 维度分别对账 |
