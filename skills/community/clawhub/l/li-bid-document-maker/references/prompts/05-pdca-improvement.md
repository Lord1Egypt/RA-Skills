# 阶段6：PDCA 自动改进（3轮循环）

## 目标
对标书执行最多3轮 PDCA（Plan-Do-Check-Act）改进循环，逐级修复问题，确保输出质量达到交付标准后再提交用户终审。

## ⚡ 执行指引（Agent必读）

### 三轮改进策略

| 轮次 | 修复目标 | 判定标准 | 未通过则 |
|------|---------|---------|---------|
| **第1轮** | 修复所有 🔴 错误级问题（遗漏技术要求、法律格式错误、评分项未覆盖） | 质检报告错误数为0 | 继续改进 → 重跑第1轮 |
| **第2轮** | 修复所有 🟡 警告级问题（空泛表述、格式不统一、证据不足） | 警告数 ≤ 3 | 继续改进 → 重跑第2轮 |
| **第3轮** | 修复所有 🟢 建议项 + 全文一致性打磨（术语统一、排版优化、语气一致） | 无未修复项 | 输出当前最佳版本 |

### 执行流程

```
输入: 质检报告 + 标书全文
        │
   ┌────▼────┐
   │  PLAN   │ ← 分析质检报告，列出本轮需修复的问题清单
   └────┬────┘
        │
   ┌────▼────┐
   │   DO    │ ← 逐项修复（不改动无关内容）
   └────┬────┘
        │
   ┌────▼────┐
   │  CHECK  │ ← 重新检查修复项是否已解决，有无引入新问题
   └────┬────┘
        │
   ┌────▼────┐
   │   ACT   │ ← 判断: 通过→进入下一轮; 未通过→重回PLAN
   └────┬────┘
        │
   ┌────▼────┐
   │ 下一轮  │ ← 最多3轮，第3轮完成后无论结果都输出
   └─────────┘
        │
   ┌────▼────────────┐
   │ 输出: 最终标书 + 改进报告 │
   └─────────────────────────┘
```

### 关键约束

1. **不改动无关内容**：每轮只修复该轮目标的问题，不要额外修改已通过的内容
2. **逐项验证**：每次修改后，对照问题清单逐项确认修复
3. **版本记录**：每轮输出一个版本号（V1→V2→V3），记录修改了什么
4. **第3轮不可再循环**：第3轮完成后无论结果如何，直接输出最终文档

---

## 第一部分：第1轮 PDCA（修复 🔴 错误）

### Plan
从质检报告中提取所有 `severity: "error"` 的问题，生成修复清单：

| 编号 | 问题描述 | 所在章节 | 修复方案 | 涉及修改量 |
|------|---------|---------|---------|-----------|
| E1 | [缺少对XXX技术参数的响应] | [3.2节] | [补充响应内容] | [小/中/大] |
| E2 | [投标函缺少保证金承诺] | [第3章] | [补充保证金条款] | [小] |

> ⚠️ 如果问题数量 > 10，按优先级排序：评分项遗漏 > 技术要求遗漏 > 法律格式 > 资质遗漏

### Do
修复原则：
- **评分项遗漏** → 在对应章节补充完整内容，标注评分维度和分值
- **技术要求遗漏** → 在技术参数响应表中补充对应条目
- **法律格式错误** → 严格按照 `templates/chapter-templates.md` 中的标准格式修正
- **资质遗漏** → 使用占位符 `[待补充：XXX证书]` 标注，不编造虚假材料

### Check
修复后逐项检查：
- [ ] E1 对应内容已补充，与招标要求一致
- [ ] E2 法律条款已按规范补充
- [ ] ...（逐项确认）
- [ ] 修改未导致其他章节编号、引用断裂
- [ ] 修改未引入新的错误

### Act
- 所有🔴错误已修复 → ✅ 进入第2轮
- 仍有🔴错误未修复 → 🔄 重新执行第1轮 Plan（标记"重试1"）
- 第1轮重试超过2次仍未修复所有🔴错误 → ⛔ 标记"部分错误无法自动修复"，进入第2轮

---

## 第二部分：第2轮 PDCA（修复 🟡 警告）

### Plan
从质检报告中提取所有 `severity: "warning"` 的问题，生成修复清单：

| 编号 | 问题描述 | 所在章节 | 修复方案 | 类型 |
|------|---------|---------|---------|------|
| W1 | [表述过于空泛] | [3.1节] | [用具体数据替换] | 内容质量 |
| W2 | [章节编号不连续] | [多处] | [统一编号] | 格式 |

### Do
各类警告的标准修复方式：

| 警告类型 | 修复方法 | 示例 |
|---------|---------|------|
| **空泛表述** | 替换为具体量化的描述 | "提供优质服务" → "7×24小时响应，2小时到场" |
| **格式不统一** | 统一为 `templates/format-rules.md` 规范 | 正文全部改为宋体小四号 |
| **证据不足** | 补充案例/数据，标注证据来源 | 添加"[依据：XX项目验收报告]" |
| **术语不一致** | 全文搜索替换为统一术语 | "物联网"与"IoT"统一为"物联网(IoT)" |

### Check
修复后逐项检查：
- [ ] 所有空泛表述已替换为量化描述
- [ ] 格式符合 `templates/format-rules.md` 规范
- [ ] 术语全文一致
- [ ] 修改未破坏内容逻辑

### Act
- 所有🟡警告已修复或剩余 ≤ 3 个 → ✅ 进入第3轮
- 仍有 > 3个🟡警告未修复 → 🔄 重新执行第2轮 Plan

---

## 第三部分：第3轮 PDCA（修复 🟢 建议 + 一致性打磨）

### Plan
从质检报告中提取所有 `severity: "suggestion"` 的问题，并额外扫描以下一致性项：

| 编号 | 优化类型 | 检查内容 |
|------|---------|---------|
| S1 | 语气一致 | 全文是否统一使用"我公司"（未混用"我方""我们"） |
| S2 | 编号连续 | 章节号、表号、图号是否连续无跳跃 |
| S3 | 目录匹配 | 目录标题是否与正文标题完全一致 |
| S4 | 术语统一 | 专业术语是否全文一致 |
| S5 | 排版美观 | 段落间距、表格格式是否统一 |
| S6 | 语言精炼 | 有无冗余表述可精简 |

### Do — 全文一致性打磨清单

逐一检查并修正：

```
□ 第1章-第N章: "我公司"统一（全文替换确认）
□ 第1章-第N章: 章节编号连续（X.Y.Z 无跳跃）
□ 目录: 重建目录，确认与正文一致
□ 表格: 全部使用三线表格式
□ 图表编号: 图X.X / 表X.X 连续无跳跃
□ 术语: [关键术语列表] 全文统一
□ 标点: 全角/半角统一（中文用全角标点）
□ 数字: 数字格式统一（使用半角阿拉伯数字）
□ 页眉页脚: 标注项目名称和页码格式
```

### Check
最终综合检查：
- [ ] 全文逻辑流畅，无明显衔接问题
- [ ] 各章节风格一致
- [ ] 所有承诺有证据支撑或标注了占位符
- [ ] 文件符合 `templates/format-rules.md` 格式规范

### Act
- ✅ 第3轮完成，**强制输出最终文档**（无论结果如何）
- 生成改进报告，记录三轮的修改内容和最终状态

---

## 第四部分：输出

### 改进报告格式

```
## PDCA 改进报告

### 改进轮次摘要
| 轮次 | 修复问题数 | 状态 | 主要修改 |
|------|-----------|------|---------|
| 第1轮 (🔴错误) | X个 | ✅/⚠️ | [简述] |
| 第2轮 (🟡警告) | X个 | ✅/⚠️ | [简述] |
| 第3轮 (🟢建议+打磨) | X个 | ✅ | [简述] |

### 最终质量状态
- 未修复 🔴 错误：X个（如有，列出原因）
- 未修复 🟡 警告：X个（如有，列出原因）
- 全文总字数：[X]字
- 章节数量：[X]章

### 自检结论
[已通过全部自检 / 存在以下已知问题请用户关注...]
```

### 最终文档交付

输出最终标书，包含：
1. 完整标书内容（15章标准结构）
2. PDCA 改进报告
3. 标注"待补充"的占位符清单（如有）

**至此，AI 自动处理阶段全部完成，可以交付用户终审。**
