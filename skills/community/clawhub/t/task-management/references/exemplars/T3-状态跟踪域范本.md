## T3 状态跟踪域 范本

### 范本概述
- **域ID**: T3
- **域名称**: 状态跟踪域
- **任务数量**: 5
- **范本状态**: 可用

### 任务执行范本

#### T3-01 状态定义
**执行步骤**:
1. 识别需要跟踪的状态类型
2. 定义状态类型和含义
3. 设计状态图标和颜色
4. 制定状态转换规则
5. 确认状态定义

**输入示例**:
- 任务类型：软件开发任务
- 跟踪需求：需要跟踪任务从创建到完成的全过程

**输出示例**:
- 状态类型：
  - pending（待执行）：任务已创建，等待执行
  - in_progress（执行中）：任务正在执行
  - completed（已完成）：任务已完成
  - deleted（已删除）：任务已删除
- 状态图标：pending=⏳, in_progress=🔄, completed=✅, deleted=❌
- 状态颜色：pending=灰色, in_progress=蓝色, completed=绿色, deleted=红色

**注意事项**:
- 状态类型要完整，覆盖所有可能状态
- 状态含义要清晰，避免歧义
- 状态图标和颜色要直观，便于识别

#### T3-02 状态转换规则
**执行步骤**:
1. 定义状态转换触发条件
2. 设计状态转换执行动作
3. 制定状态转换验证规则
4. 记录状态转换日志
5. 确认状态转换规则

**输入示例**:
- 状态类型：pending, in_progress, completed, deleted
- 业务场景：任务从创建到完成的全过程

**输出示例**:
- 转换触发条件：
  - pending → in_progress：开始执行任务时
  - in_progress → completed：任务完成时
  - pending/in_progress → deleted：任务不再需要时
- 转换执行动作：
  - 更新任务状态
  - 记录转换时间
  - 发送状态变更通知
- 转换验证规则：
  - 只能从pending转换到in_progress
  - 只能从in_progress转换到completed
  - 可以从pending或in_progress转换到deleted

**注意事项**:
- 触发条件要明确，避免模糊
- 执行动作要具体，便于操作
- 验证规则要严格，避免错误转换

#### T3-03 进度计算
**执行步骤**:
1. 设计进度计算公式
2. 确定进度数据来源
3. 设定进度更新频率
4. 实现进度计算功能
5. 确认进度计算结果

**输入示例**:
- 任务分解结果：6个子任务
- 子任务状态：2个已完成，2个进行中，2个待执行

**输出示例**:
- 进度计算公式：进度 = 已完成子任务数 / 总子任务数 × 100%
- 进度数据来源：任务状态数据
- 进度更新频率：每完成一个子任务更新一次
- 当前进度：2/6 × 100% = 33.3%

**注意事项**:
- 计算公式要准确，避免错误
- 数据来源要可靠，避免虚假数据
- 更新频率要合理，避免过于频繁或稀疏

#### T3-04 阻塞识别
**执行步骤**:
1. 检测被阻塞的任务
2. 分析阻塞原因
3. 评估阻塞影响
4. 提出解决建议
5. 记录阻塞信息

**输入示例**:
- 任务状态：前端开发任务状态为pending
- 依赖关系：前端开发依赖系统设计完成

**输出示例**:
- 阻塞任务：前端开发任务
- 阻塞原因：系统设计任务未完成
- 阻塞影响：前端开发无法开始，影响整体进度
- 解决建议：加快系统设计任务进度，或并行执行部分前端开发工作

**注意事项**:
- 阻塞检测要及时，避免延误
- 阻塞原因要准确，避免误判
- 解决建议要可行，避免空谈

#### T3-05 状态更新
**执行步骤**:
1. 检测状态变化
2. 准备更新内容
3. 执行状态更新
4. 记录更新时间
5. 确认更新结果

**输入示例**:
- 任务：前端开发任务
- 当前状态：pending
- 状态变化：开始执行任务

**输出示例**:
- 更新时间：YYYY-MM-DD HH:mm:ss
- 更新内容：状态从pending更新为in_progress
- 更新原因：开始执行前端开发任务
- 更新确认：状态更新成功

**注意事项**:
- 状态变化检测要及时
- 更新内容要准确
- 更新时间要精确记录
