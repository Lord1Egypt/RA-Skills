## 一、FactScore

### 基本信息
- **仓库**：https://github.com/shmsw25/FActScore
- **论文**：FActScore: Fine-grained Atomic Evaluation of Factual Precision in Long Form Text Generation (EMNLP 2023)
- **PyPI包**：`factscore`
- **本地路径**：`opensource/FActScore/`

### 核心功能
将长文本生成内容**分解为原子事实**（atomic facts），然后逐条验证每个原子事实是否被知识源支持，最终给出FActScore（事实精度得分）。

**验证流程**：
1. 输入：主题实体（topic）+ 模型生成文本（output）
2. 原子事实分解：用LLM将生成文本分解为若干条原子事实
3. 知识检索：对每个原子事实，从知识源（默认Wikipedia 2023/04/01）检索相关证据
4. 事实判断：用ChatGPT或LLaMA判断每个原子事实是否被支持
5. 输出：FActScore = 被支持的事实数 / 总事实数（可选长度惩罚）

### 适用场景
- ✅ 长文本（传记、百科类）的事实性评估
- ✅ 需要细粒度事实分解的场景
- ❌ 数学计算、逻辑推理类问题（不支持）

### MVP适用性分析
| 维度 | 评价 |
|------|------|
| 功能匹配度 | ★★★★☆ 适合事实性验证，但依赖外部知识源 |
| 集成复杂度 | ★★☆☆☆ 需下载Wikipedia dump（18GB），初始化慢 |
| 运行成本 | ★☆☆☆☆ 每次调用需API请求，成本高 |
| 对Workbuddy适配 | ★★★☆☆ 可以封装为验证后端，但需改造成多AI源对比 |

### MVP建议
**参考其原子事实分解思路，自行实现简化版**：
- 用Workbuddy内置AI源做原子事实分解（不依赖外部知识源）
- 用多AI源交叉验证每个原子事实的一致性
- 避免依赖Wikipedia dump，降低部署复杂度

## 二、xVerify

### 基本信息
- **仓库**：https://github.com/IAAR-Shanghai/xVerify
- **论文**：xVerify: Efficient Answer Verifier for Reasoning Model Evaluations (arXiv:2504.10481)
- **HuggingFace**：IAAR-Shanghai/xVerify-0.5B-I（推荐，轻量）
- **本地路径**：`opensource/xVerify/`

### 核心功能
基于**微调LLM**的答案验证工具，专为**有唯一正确答案的客观题**设计。能从超长推理链中精准提取最终答案，并判断与标准答案是否等价。

**支持题型**：
- 数学问题（复杂表达式，如 LaTeX、`\frac{4}{5}` vs `4/5`）
- 选择题（能识别选项内容，不仅匹配字母）
- 分类任务（情感分析等）
- 简答问题（自然语言描述等价性判断）

**等价判断能力**：
- 基础变换：`a -> A`、`alpha -> α`
- 数学表达式：`\frac{4}{5} -> 4/5`、`1.34 x 10^3 -> 1340`
- 自然语言：`one hundred and twenty-three -> 123`

### 适用场景
- ✅ 有标准答案的客观题验证
- ✅ 含复杂推理链的答案提取
- ✅ 数学表达式等价性判断
- ❌ 开放性问题、长文本事实性评估（不是其设计目标）

### MVP适用性分析
| 维度 | 评价 |
|------|------|
| 功能匹配度 | ★★★★☆ 适合验证有标准答案的问题，但别瞎说需要处理开放性问题 |
| 集成复杂度 | ★★★☆☆ 需部署xVerify模型（推荐0.5B版本，轻量），或用vLLM推理 |
| 运行成本 | ★★★☆☆ 本地部署无API成本，但需GPU资源（云服务器2G内存可能不够） |
| 对Workbuddy适配 | ★★★★☆ 可作为验证后端的一个维度（逻辑/答案正确性验证） |

### MVP建议
**作为验证逻辑的一个补充模块**：
- 当用户输入的问题有标准答案时（如数学题、选择题），用xVerify验证
- MVP阶段可先跳过，后续迭代时再加入
- 若加入，推荐用xVerify-0.5B-I（HuggingFace下载），用vLLM部署

## 三、AIQualityKit

### 基本信息
- **仓库**：https://github.com/LambertClark/AIQualityKit
- **本地路径**：`opensource/AIQualityKit/`
- **技术栈**：FastAPI + Pandas + OpenPyXL + uv包管理

### 核心功能
AI应用综合测试框架，自动评估LLM输出的**质量、安全性和合规性**。

**三大核心功能**：
1. **引文分析（Citation Analysis）**：验证AI回答中引用的来源是否真实、相关
2. **一致性评估（Consistency Evaluation）**：检查AI回答中是否存在前后矛盾或逻辑不一致
3. **幻觉检测（Hallucination Detection）**：识别并标记AI回答中可能包含的虚假或捏造信息

**支持API提供商**：阿里云百炼、OpenAI、DeepSeek、NuwaAPI

**最新更新（v1.2.0）**：
- 无需引文的内部一致性检测（只分析问题+答案）
- 自动清理思考过程
- 基础错误检测（如"11.9>13"等低级错误）
- 五大类型：`无问题/前后矛盾/逻辑错误/基础错误/自相矛盾`

### 适用场景
- ✅ AI应用质量评估、CI/CD质量门禁
- ✅ 需要Web界面进行手动分析的场景
- ✅ 批量Excel文件分析
- ❌ 需要轻量级集成的场景（本项目用不到Web界面）

### MVP适用性分析
| 维度 | 评价 |
|------|------|
| 功能匹配度 | ★★★★☆ 幻觉检测 + 一致性评估功能直接相关 |
| 集成复杂度 | ★★★☆☆ 本身是FastAPI服务，可作为验证后端直接集成 |
| 运行成本 | ★★☆☆☆ 依赖外部API（阿里云/OpenAI/DeepSeek），MVP阶段我们希望用Workbuddy内置AI源 |
| 对Workbuddy适配 | ★★★☆☆ 可以改造其检测逻辑，去掉Web界面依赖 |

### MVP建议
**重点参考其"内部一致性检测"逻辑**：
- 不依赖外部引文，只分析问题+答案
- 检测前后矛盾、逻辑错误、基础错误
- 可以将其检测逻辑（Python函数）集成到我们的验证后端中
- MVP阶段不集成其Web界面和Excel批量处理功能

## 四、综合对比与选型建议

### 功能对比表

| 功能维度 | FactScore | xVerify | AIQualityKit |
|---------|-----------|---------|---------------|
| 事实性验证 | ★★★★★ | ★☆☆☆☆ | ★★★☆☆ |
| 逻辑一致性 | ★★☆☆☆ | ★★★☆☆ | ★★★★★ |
| 数学/计算验证 | ☆☆☆☆☆ | ★★★★★ | ★★★☆☆ |
| 开放性问题支持 | ★★★★★ | ★☆☆☆☆ | ★★★★☆ |
| 无需标准答案 | ✅ | ❌ | ✅ |
| 依赖外部知识源 | ✅（Wikipedia） | ❌ | ❌ |
| 部署复杂度 | 高 | 中 | 中 |
| 运行成本 | 高 | 低（本地） | 中 |

### MVP阶段技术方案（推荐）

**不依赖任何开源模块的直接集成**，而是参考其设计思路，自行实现验证逻辑：

#### 核心验证流程（混合模式）

```
输入：用户问题 + AI生成的答案
     ↓
【步骤1：多AI源并行回答】
调用Workbuddy内置的多个AI源，获取它们对同一个问题的回答
     ↓
【步骤2：答案一致性对比】
对比多个AI源的答案，计算一致性得分
     ↓
【步骤3：原子事实分解】（参考FactScore思路）
用Workbuddy内置AI将答案分解为原子事实
     ↓
【步骤4：原子事实交叉验证】（参考AIQualityKit思路）
用其他AI源验证每个原子事实，标记矛盾点
     ↓
【步骤5：逻辑一致性检测】（参考AIQualityKit思路）
检测答案内部是否前后矛盾、是否有基础错误
     ↓
【步骤6：综合评分】
综合一致性得分 + 事实验证结果 + 逻辑检测结果 → 最终可信度评分
     ↓
输出：可信度评分 + 错误明细 + 多AI源对比结果
```

#### 各开源模块的参考要点

| 模块 | 参考要点 | MVP是否直接集成 |
|------|---------|----------------|
| FactScore | 原子事实分解思路 | ❌ 不直接集成，参考思路自行实现 |
| xVerify | 数学表达式等价判断 | ⏸️ 后续迭代时考虑加入 |
| AIQualityKit | 内部一致性检测逻辑 | ⏸️ 可参考其检测函数，改造成Python工具函数 |
