## 一、人机协作总体设计

### 1.1 设计原则

| 原则 | 说明 |
|------|------|
| **AI擅长优先** | 将重复性、分析性任务交给AI |
| **人工决策保留** | 关键决策点必须人工介入 |
| **透明可追溯** | 用户清楚知道AI在做什么、人工需要做什么 |
| **效率与质量平衡** | 避免过度人工干预降低效率 |
| **异常自动升级** | AI无法处理时自动触发人工介入 |

### 1.2 整体人机比例

| 维度 | 比例 | 说明 |
|------|------|------|
| **AI执行** | 87% | 自动执行的任务占比 |
| **人工参与** | 13% | 需要人工介入的任务占比 |
| **决策权归属** | 100% | 所有重大决策由人工做出 |

## 三、特殊场景处理

### 3.1 用户要求跳过阶段

| 场景 | 处理方式 |
|------|---------|
| 用户已有方向，想跳过猎手 | 允许，直接进入策划师 |
| 用户已有完整方案 | 允许，直接进入分析师 |
| 用户想跳过分析师 | 允许，但需签署免责确认 |
| 用户想跳过IP阶段 | 允许，但需明确说明风险 |

### 3.2 用户要求返回上一阶段

| 场景 | 处理方式 |
|------|---------|
| 策划师阶段想返回猎手 | 允许，保留用户已选择的方向 |
| 分析师阶段想返回策划师 | 允许，保留方案，重新分析 |
| IP阶段想返回分析师 | 允许，保留报告，重新做IP |
| 任意阶段想从头开始 | 允许，清空当前进度 |

### 3.3 用户长时间无响应

| 超时时间 | AI行为 |
|---------|--------|
| 5分钟 | 发送提醒 |
| 15分钟 | 提示「是否需要帮助」 |
| 30分钟 | 自动保存进度，发送进度报告 |
| 24小时 | 发送延续提醒 |

### 3.4 异常升级场景

| 异常场景 | 升级触发条件 | 处理方式 |
|---------|-------------|---------|
| 技术矛盾无法解决 | TRIZ分析无推荐原理 | 升级到所长决策 |
| 查新发现直接冲突 | 相似度>90% | 建议调整或放弃 |
| 商业分析否定 | 可行性评分<4 | 建议重新设计 |
| 用户需求超出能力 | 涉及法律/医疗等专业 | 建议咨询专业人士 |

## 四、所长决策模式

### 4.1 触发条件

当以下异常发生时，触发所长决策模式：

```
┌─────────────────────────────────────────────────────────────────────┐
│ 【所长决策】触发条件                                                  │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  1. 技术矛盾无法解决                                                 │
│     条件：TRIZ分析中矛盾矩阵无推荐原理                               │
│     状态：需要综合判断                                               │
│                                                                     │
│  2. 关键风险超出阈值                                                │
│     条件：风险评估显示「高风险」且无法规避                           │
│     状态：需要决策是否继续                                           │
│                                                                     │
│  3. 用户要求终止项目                                                 │
│     条件：用户明确表示放弃当前创意                                   │
│     状态：需要确认并清理进度                                         │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 4.2 所长决策流程

```
异常发生
    │
    ▼
┌─────────────────────────────────────────────────────────────────────┐
│ 生成所长决策报告                                                     │
│ ├─ 问题描述                                                         │
│ ├─ 已尝试的解决方案                                                 │
│ ├─ 当前状态                                                         │
│ ├─ 可选方案（2-3个）                                                │
│ └─ 建议                                                             │
└─────────────────────────────────────────────────────────────────────┘
    │
    ▼
展示给用户
    │
    ▼
用户决策
├─ 选择方案A/B/C
├─ 自定义方案
└─ 终止项目
```

### 4.3 所长决策示例

```
═══════════════════════════════════════════════════════════════════════
【所长决策】

当前问题：健康数据采集算法的技术矛盾无法通过TRIZ矛盾矩阵解决

问题详情：
- 改善参数：数据采集精度
- 恶化参数：设备功耗
- 矛盾描述：提高精度需要更多传感器，但增加功耗

已尝试的解决方案：
1. 原理#15（动态化）：采用自适应采样频率 → 效果有限
2. 原理#35（参数变化）：调整采样参数 → 仍无法平衡

当前状态：
需要综合判断技术可行性和商业价值

可选方案：
A. 降低精度要求，聚焦核心功能
   - 风险：产品竞争力下降
   - 收益：技术可行，开发周期短

B. 接受高功耗方案，优化电池技术
   - 风险：用户体验受影响
   - 收益：保持技术领先

C. 寻找替代技术路线（如第三方API）
   - 风险：核心技术依赖外部
   - 收益：快速上线

建议：基于MVP优先原则，建议选择方案A「降低精度要求，聚焦核心功能」

═══════════════════════════════════════════════════════════════════════

请选择您的决策：[A / B / C / 其他]
```

## 五、用户体验优化

### 5.1 透明度设计

| 设计要点 | 说明 |
|---------|------|
| 阶段状态可视化 | 用户清楚当前所在阶段 |
| 进度提示 | 告知用户还有多少步骤 |
| AI执行说明 | 让用户知道AI在做什么 |
| 人工等待时间 | 预估人工介入的等待时间 |

### 5.2 反馈机制

| 反馈类型 | 时机 | 内容 |
|---------|------|------|
| 阶段开始提示 | 进入新阶段 | 告知当前阶段目标 |
| 执行进度 | AI执行中 | 简单进度条或文字 |
| 结果展示 | 阶段完成 | 清晰的结论和建议 |
| 下一步引导 | 阶段结束 | 明确告知用户需要做什么 |

### 5.3 帮助提示

| 场景 | 提示内容 |
|------|---------|
| 用户犹豫 | 「您可以在方向1和2之间做选择，或者告诉我您的偏好」 |
| 用户不确认 | 「如果您不确定，可以先选择一个方向试试，不满意可以调整」 |
| 用户困惑 | 「如果您不确定如何选择，可以告诉我您最关注的点（如成本/时间/效果）」 |
| 用户想了解更多 | 「我可以详细解释每个方向的区别」 |
