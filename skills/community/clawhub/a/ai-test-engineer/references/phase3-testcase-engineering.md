## M15: 功能测试用例编写

### 核心方法论
**8要素标准模板** + **4层验证深度** =

### 专家操作流程
1. **用例结构化**（8要素）：
   ```
   [用例编号] 模块缩写-功能缩写-序号（如：ORDER-CREATE-001）
   [用例标题] 动词+验证点+条件（如：验证会员用户使用优惠券下单时金额正确减免）
   [前置条件] 数据状态+环境要求+账号权限+依赖服务
   [测试步骤] 操作描述 → 预期中间状态（一步一验证）
   [预期结果] 精确可量化（状态码200、金额精确到分、文案完整匹配）
   [测试数据] 具体值（非"有效用户名"而是"test_vip_001"）
   [优先级] P0/P1/P2/P3 + 判定理由
   [关联需求] REQ-ID + 版本号
   ```
2. **4层验证深度**：
   - **L1 表现层**：UI显示、文案、布局、交互反馈
   - **L2 数据层**：数据库字段值、缓存、搜索索引
   - **L3 接口层**：请求参数、响应体、状态码、Header
   - **L4 日志层**：关键操作日志、埋点事件、审计记录
3. **用例分类矩阵**：
   | 类型 | 占比 | 设计方法 | 示例 |
   |------|------|---------|------|
   | 正向用例 | 40% | 场景法+等价类 | 正常流程完整跑通 |
   | 边界用例 | 25% | 边界值分析 | 最小/最大值、空集合 |
   | 异常用例 | 20% | 错误推测+无效等价类 | 参数缺失、格式错误 |
   | 组合用例 | 10% | 判定表+Pairwise | 多条件组合逻辑 |
   | 探索用例 | 5% | 探索性+SBTM | 自由测试会话 |
4. **用例质量自检表**：
   - [ ] 其他人不看需求只看用例就能执行
   - [ ] 每个步骤只有一个操作
   - [ ] 预期结果不含"正确""正常"等模糊词
   - [ ] 测试数据可直接使用（不需理解后造数据）
   - [ ] 前置条件排除了环境依赖（可独立执行）

## M16: 接口测试用例编写

### 核心方法论
**请求-响应-副作用** 三维验证模型 + **接口六要素**全覆盖

### 专家操作流程
1. **接口六要素提取**：
   ```
   要素1 - URL + Method：GET/POST/PUT/DELETE/PATCH
   要素2 - 请求参数：
     - Query参数、Path参数、Header、Body(JSON/Form/Multipart)
     - 必填/选填、类型、长度、格式、默认值
   要素3 - 认证鉴权：Token/OAuth/API Key/Signature
   要素4 - 响应结构：
     - 状态码（2xx/3xx/4xx/5xx）
     - 响应体Schema（字段+类型+是否必返回）
     - 响应头（Content-Type/Cache-Control/CORS）
   要素5 - 业务规则：校验逻辑、计算逻辑、权限控制
   要素6 - 副作用：数据库变更、消息队列、缓存更新、日志
   ```
2. **请求参数测试矩阵**：
   | 参数 | 必填 | 有效值 | 无效值 | 边界值 | 默认值 |
   |------|------|--------|--------|--------|--------|
   | userId | Y | 正整数 | 0/负数/字符串/null | 1, MAX_INT | - |
   | pageSize | N | 1-100的整数 | 0/101/字符串 | 1, 100, 101 | 20 |
3. **分层测试策略**：
   - **L1 单接口**：参数组合+边界+异常 → 等价类 + 边界值 + 正交
   - **L2 接口链**：A→B→C 依赖链路 → 场景法 + 状态迁移
   - **L3 业务场景**：完整业务流程 → 场景法 + 判定表
4. **鉴权测试清单**：
   - [ ] 无Token访问 → 401
   - [ ] 过期Token → 401
   - [ ] 伪造Token → 401
   - [ ] 低权限访问高权限接口 → 403
   - [ ] 跨租户数据访问 → 403
5. **特殊接口类型**：
   - 文件上传：大小边界/类型白名单/并发/断点续传
   - 文件下载：权限/大文件/断点续传/格式
   - 流式接口：SSE/WebSocket连接/断线重连/心跳
   - 批量接口：最大批量数/部分成功处理/超时
   - GraphQL：查询深度限制/字段权限/批量查询
   - gRPC：proto版本兼容/流式调用/超时/重试

## M17: 性能测试用例设计

### 核心方法论
**负载模型设计** → **场景矩阵** → **指标基线**

### 专家操作流程
1. **负载模型（Workload Model）**：
   - 从生产数据中提取：
     - 各个接口的调用比例
     - 读写比例
     - 峰值QPS与平均QPS比值（突发系数）
     - 数据量分布（小/中/大响应体占比）
   - 用户行为建模：思考时间、操作间隔、并发度
2. **性能测试场景矩阵**：
   | 场景类型 | 目的 | 并发量 | 持续时间 | 通过标准 |
   |---------|------|--------|---------|---------|
   | 基准测试 | 建立性能基线 | 1 | 5min | 建立P50/P90/P99基线 |
   | 负载测试 | 验证预期负载 | 预计峰值×1 | 30min | P99<目标值，0错误 |
   | 压力测试 | 找到性能拐点 | 逐步递增直到崩溃 | 每阶梯5min | 记录拐点和极限值 |
   | 稳定性测试 | 验证长期稳定性 | 预计峰值×0.8 | 8-24h | 无内存泄漏，P99稳定 |
   | 峰值测试 | 验证突发流量 | 预计峰值×2-3 | 10min | 无雪崩，自动恢复 |
   | 容量测试 | 验证数据量影响 | 正常 | - | 大数据量下性能可接受 |
3. **关键性能指标（KPI）**：
   - 响应时间：P50/P90/P95/P99/Max
   - 吞吐量：TPS/QPS
   - 错误率：<0.1%（业务）/ <1%（非关键）
   - 资源利用率：CPU<70%、内存<80%、磁盘IO
   - 并发连接数/数据库连接池/线程池
4. **常见性能缺陷模式**：
   - N+1查询 → 批量查询/Lazy Loading误用
   - 缓存雪崩 → 大量缓存同时过期
   - 连接池耗尽 → 未释放连接
   - 大对象序列化 → 不必要的全量序列化
   - 同步阻塞 → 非核心操作的同步等待

## M18: 安全测试用例设计

### 核心方法论
**OWASP Top 10 (2021)** + **STRIDE威胁模型** + **安全测试检查表**

### 专家操作流程
1. **OWASP Top 10 全覆盖**：
   - A01 Broken Access Control → 水平/垂直越权、IDOR
   - A02 Cryptographic Failures → 明文传输、弱加密、硬编码密钥
   - A03 Injection → SQL/LDAP/OS Command/NoSQL注入
   - A04 Insecure Design → 缺少速率限制、过度数据暴露
   - A05 Security Misconfiguration → 默认密码、详细错误信息
   - A06 Vulnerable Components → 已知CVE的依赖库
   - A07 Auth Failures → 弱密码策略、会话固定、无MFA
   - A08 Software & Data Integrity → 反序列化、CI/CD投毒
   - A09 Logging & Monitoring → 无审计日志、告警缺失
   - A10 SSRF → 服务端请求伪造
2. **STRIDE威胁模型**：
   | 威胁类型 | 安全属性 | 测试重点 |
   |---------|---------|---------|
   | Spoofing(仿冒) | 认证 | 身份伪造、Session劫持 |
   | Tampering(篡改) | 完整性 | 参数篡改、重放攻击 |
   | Repudiation(抵赖) | 不可否认性 | 日志完整性、数字签名 |
   | Info Disclosure(泄露) | 机密性 | 敏感数据暴露、目录遍历 |
   | DoS(拒绝服务) | 可用性 | 资源耗尽、死锁 |
   | Elevation(提权) | 授权 | 权限提升、角色绕过 |
3. **安全测试用例模板**：
   ```
   用例编号：SEC-AUTH-001
   标题：验证登录接口存在暴力破解防护（速率限制）
   前置：创建测试账号 test@example.com
   步骤：
   1. 连续发送50次错误密码登录请求（100ms间隔）
   预期：
   1. 前5-10次返回401 + "密码错误"
   2. 之后返回429 + "请求过于频繁"
   3. 第51次使用正确密码仍被限制（验证限流是针对账号而非IP时）
   ```

## M19: 兼容性测试用例

### 核心方法论
**兼容性矩阵** = 平台 × 浏览器/OS × 分辨率 × 网络环境

### 专家操作流程
1. **建立兼容性矩阵**（根据产品用户画像）：
   | 维度 | 覆盖范围 | 来源依据 |
   |------|---------|---------|
   | 浏览器 | Chrome(最近3版)/Safari(最近2版)/Firefox/Edge | StatCounter/百度统计 |
   | OS | Windows 10/11, macOS, iOS 16/17, Android 12/13/14 | 用户设备分布 |
   | 分辨率 | 1920×1080, 1366×768, 375×812(iPhone), 360×800(Android) | Top 4分辨率 |
   | 网络 | WiFi, 4G, 5G, 弱网(2G/3G) | 用户网络环境 |
2. **优先级策略**：
   - P0：覆盖 > 5% 用户量的组合
   - P1：覆盖 1-5% 的组合
   - P2：覆盖 < 1% 但有战略意义的组合
3. **兼容性测试检查点**：
   - UI层：布局不错乱、文字不重叠、图片不拉伸、滚动正常
   - 功能层：核心流程可完成、弹窗/Toast正常、键盘不遮挡
   - 性能层：加载时间、动画流畅度、内存占用
   - 特殊：暗黑模式、系统字体缩放、横竖屏、分屏模式

## M20: 易用性测试用例

### 核心方法论
**Nielsen十大可用性原则** → **启发式评估** → **用户任务测试**

### 十大可用性原则测试清单
1. **系统状态可见性**：
   - [ ] 加载中有进度提示
   - [ ] 操作结果有明确反馈（成功/失败）
   - [ ] 当前位置有导航标识
2. **系统与现实匹配**：
   - [ ] 使用用户语言而非技术术语
   - [ ] 图标和隐喻符合用户习惯
3. **用户控制与自由**：
   - [ ] 支持撤销/重做
   - [ ] 退出当前流程有明确提示
4. **一致性与标准**：
   - [ ] 相同功能在所有页面行为一致
   - [ ] 遵循平台设计规范（iOS HIG/Material Design）
5. **错误预防**：
   - [ ] 危险操作有二次确认
   - [ ] 输入框有格式提示和实时校验
6. **识别而非回忆**：
   - [ ] 选项可见而非需要记忆
   - [ ] 常用功能不超过3次点击
7. **灵活高效**：
   - [ ] 支持快捷键
   - [ ] 常用操作有快捷入口
8. **简约设计**：
   - [ ] 无冗余信息干扰
   - [ ] 核心操作视觉突出
9. **帮助识别与恢复**：
   - [ ] 错误提示说明原因+解决方案
   - [ ] 空状态有引导操作
10. **帮助文档**：
    - [ ] 帮助入口易于发现
    - [ ] 文档与实际功能一致

## M21: 测试数据构造

### 核心方法论
**数据金字塔** = 基础数据 → 场景数据 → 边界数据 → 异常数据 → 海量数据

### 专家操作流程
1. **数据需求分析**：
   - 标识每个用例的数据依赖性
   - 数据共享 vs 数据隔离（用例间数据污染风险）
   - 数据生命周期（何时创建、何时清理）
2. **数据构造策略**：
   | 数据层级 | 说明 | 构造方式 | 示例 |
   |---------|------|---------|------|
   | 基础数据 | 系统运行必需 | SQL脚本初始化/API创建 | 用户/商品/配置 |
   | 场景数据 | 特定场景所需 | 用例前置步骤/数据工厂 | 待支付订单/过期优惠券 |
   | 边界数据 | 边界值测试 | 脚本批量生成 | 0库存/超长名称 |
   | 异常数据 | 异常场景 | 直接写库/API构造 | 脏数据/脏状态 |
   | 海量数据 | 性能/翻页 | 脚本批量插入 | 10万条记录 |
3. **数据造数技术**：
   - 直接SQL插入（最快，需注意关联数据完整性）
   - API调用（最真实，慢但不会有数据完整性问题）
   - 数据工厂模式（Fixture Factory，可复用）
   - Faker库：生成逼真的随机数据（姓名/地址/手机号/邮箱）
4. **数据隔离策略**：
   - 用例级：每个用例独立数据（最安全，维护成本高）
   - 套件级：同类用例共享数据（平衡）
   - 全局级：所有用例共享数据（需严格管理状态）

### AI增强技巧
- 用AI生成Faker脚本，批量创建符合业务规则的测试数据
- 用AI识别用例的数据依赖关系，生成数据拓扑图

## M22: 测试用例评审与优化

### 核心方法论
**7维度评审模型** + **用例瘦身四步法**

### 专家操作流程
1. **7维度评审**（每项1-5分）：
   | 维度 | 评估标准 | 权重 |
   |------|---------|------|
   | 正确性 | 用例与需求一致，预期结果正确 | 25% |
   | 完整性 | 覆盖所有功能点、边界、异常 | 25% |
   | 清晰性 | 步骤清晰、无歧义、可独立执行 | 15% |
   | 可维护性 | 结构清晰、编号规范、易于更新 | 10% |
   | 可自动化 | 适用于自动化脚本的程度 | 10% |
   | 优先级合理 | P0-P3划分有充分理由 | 10% |
   | 无冗余 | 没有重复或等价用例 | 5% |
2. **评审报告模板**：
   ```
   评审结论：[通过/有条件通过/返工]
   整体评分：__/5
   优点：...
   问题清单：
   | # | 用例编号 | 问题描述 | 严重程度 | 修改建议 |
   统计：Blocker N个, Major M个, Minor P个, Suggestion Q个
   ```
3. **用例瘦身四步法**：
   - Step 1: 合并重复（多个用例验证同一逻辑 → 保留最全面的1个）
   - Step 2: 分解大用例（1个用例测太多 → 拆为多个独立用例）
   - Step 3: 补充遗漏（对照需求/边界清单 → 补充缺失用例）
   - Step 4: 优先级重排（基于风险重新划分P0-P3）
4. **用例评审角色分工**：
   - 测试Peer：方法正确性、覆盖率
   - 开发：技术可行性、接口准确性
   - 产品：业务逻辑正确性、用户体验
   - 架构师：系统级影响、非功能需求
