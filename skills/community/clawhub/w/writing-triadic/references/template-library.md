## 1. 技术文档 (Technical Documentation)

- **核心逻辑**：极简、清晰、面向解决问题。
- **模板骨架**：
  - **TL;DR / 概述**：一句话说明文档用途及能解决的问题。
  - **先决条件 (Prerequisites)**：所需环境、权限或前置知识。
  - **操作步骤 (Step-by-Step)**：使用动宾短语开头（如"打开控制台"），配以代码块。
  - **避坑指南 (Gotchas/Warnings)**：标注常见报错或易踩坑点。
  - **参考链接**：相关 API 或延伸阅读。
- **质量标准**：术语正确性、代码可编译性、章节结构清晰、Flesch 阅读分数 30-50。
- **禁止**：修辞性语言、类比堆砌、无数据支撑的性能断言。

### SEO 优化 (v2.9)

适用完整 SEO 模块 ([seo-module.md](seo-module.md))：关键词提取 → 意图分类 → 标题评分 → 密度预设 → 可读性目标 → 内链策略 → 检查清单。

### AI 疲劳词表 (v2.7)
- "值得注意的是" / "众所周知" / "毋庸置疑" / "在我们的日常工作中"
- "大大提高" / "极大地改善" / "有效地解决"
- "本文旨在" / "接下来我们将" / "让我们深入探讨"
- 机械连接词泛滥：每段以"此外" / "另外" / "与此同时"开头
- 英文: `delve into`, `robust`, `seamless`, `cutting-edge`, `leveraging`

## 2. 博客文章 (Blog Post)

- **核心逻辑**：提供增量信息，注重互动感。
- **模板骨架**：
  - **Hook (钩子)**：反常识现象、痛点场景或个人小故事开场。
  - **Why it matters (重要性)**：迅速点明主题与读者的利益相关性。
  - **Body (正文核心)**：分为 2-3 个子模块，多用小标题、列表，穿插具体案例。
  - **Actionable Takeaway (可落地结论)**：直接告知读者能带走的实际价值，拒绝强行升华。
  - **CTA (行动呼吁)**：抛出能引发评论的具体问题。
- **质量标准**：Flesch 60-70、段落流畅、主动语态与短句比例高、SEO 关键词恰当。
- **禁止**：每段结尾强行升华、空洞的"未来展望"、无具体案例的泛泛而谈。

### SEO 优化 (v2.9)

适用完整 SEO 模块 ([seo-module.md](seo-module.md))：关键词提取 → 意图分类 → 标题评分 → 密度预设 → 可读性目标 → 内链策略 → 检查清单。

### AI 疲劳词表 (v2.7)
- "在当今这个...的时代" / "随着...的不断发展" / "纵观...的发展历程"
- "引发了广泛的关注" / "成为热议的话题" / "开启了新的篇章"
- "如何...成为摆在面前的重大课题" / "既带来了机遇也带来了挑战"
- "完美地诠释" / "深刻地改变" / "极大地丰富"
- 英文: `moreover`, `furthermore`, `additionally`, `it is worth noting that`, `in today's digital age`

## 3. 随笔 / 散文 (Personal Essay)

- **核心逻辑**：重情绪流淌、细节描写和非线性叙事。
- **模板骨架**：
  - **微观切入**：从极小的物理细节或转瞬即逝的动作开始。
  - **延展联想**：由细节牵引出回忆或当前的情绪状态。
  - **内在冲突**：展示矛盾感或困惑，不急于给出答案。
  - **留白结尾**：用意象或未完成的动作结尾，拒绝总结。
- **质量标准**：情绪真实、细节鲜活、避免说教、节奏有呼吸感。
- **禁止**：人生道理总结、强行正能量、机械排比、每个自然段等长。

### AI 疲劳词表 (v2.7)
- "人生就像一场旅行" / "岁月如梭，光阴似箭" / "回首往事"
- "也许这就是成长吧" / "所有的相遇都是久别重逢"
- "时间是最好的答案" / "在岁月的长河中" / "生活教会了我们"
- "静静地流淌" / "温暖而美好" / "简单而纯粹"
- 英文: `a tapestry of`, `the beauty of`, `a gentle reminder`, `in the grand scheme of things`

## 4. 学术 / 行业论文 (Whitepaper / Paper)

- **核心逻辑**：严谨、客观、逻辑推演。
- **模板骨架**：
  - **摘要 (Abstract)**：研究背景、方法、核心发现。独立可读，不含引用文献编号。
  - **引言 (Introduction)**：定义研究边界，指出领域空白 (Gap)，明确研究问题。
  - **文献综述 (Literature Review)**：梳理现有研究，指出前人工作与本文的差异（论文章节可并入引言）。
  - **方法论 (Methodology)**：数据来源、样本描述、分析模型或实验设计。要求可复现。
  - **发现与分析 (Results & Discussion)**：数据支撑结论，与假设对比，解释异常值。
  - **结论 (Conclusion)**：总结核心贡献，指出理论和实践意义。
  - **局限性 (Limitations)**：坦诚承认研究不足与未来方向。
  - **参考文献 (References)**：格式统一（APA/MLA/Chicago 等），每条引用至少包含作者+年份+标题+来源。
- **质量标准**：学术格式规范、引用准确可查、图表编号完整、第三人称客观叙述。
- **禁止**：情绪化语言、未经证实的断言、隐藏研究局限、引用格式混乱、第一人称叙事（除非学科允许）。

### AI 疲劳词表 (v2.7)
- "值得注意的是" / "需要指出的是" / "研究表明" / "众所周知"（如果没有引用就是空话）
- "这一发现具有重要意义" / "对未来的研究提供了新的方向"
- "填补了...领域的空白"（除非确实填补了）/ "在某种程度上" / "相对而言"
- "不仅有助于...也为...提供了"（学术版 not only... but also）
- 英文: `it is worth noting that`, `plays a crucial role`, `sheds light on`, `paves the way for`, `opens up new avenues`

## 5. 商业文案 (Business Copy — PAS 框架)

- **核心逻辑**：煽动情绪，促成转化。
- **模板骨架**：
  - **Problem (痛点场景)**：精准描绘受众正在经历的麻烦。
  - **Agitation (伤口撒盐)**：放大麻烦带来的负面后果。
  - **Solution (引出方案)**：抛出产品/服务作为解药。
  - **Proof (信任背书)**：数据、客户评价或权威认证。
  - **CTA (限时行动)**：明确且具紧迫感的转化指令。
- **质量标准**：语句简洁有力、号召动词突出、符合品牌声调、A/B 测试友好。
- **禁止**：假大空形容词堆砌、无信任背书的承诺、缺乏紧迫感的软 CTA。

### SEO 优化 (v2.9)

当商业文案用于公开落地页、产品页、服务页或品牌官网时，适用完整 SEO 模块 ([seo-module.md](seo-module.md))。重点关注：搜索意图是否偏交易型/商业调查型、主关键词是否自然进入 H1 与首屏文案、Meta Description 是否包含明确价值承诺、CTA 是否与关键词意图一致。若是私域广告、内部提案或短促销语，跳过 SEO 模块。

### AI 疲劳词表 (v2.7)
- "颠覆性" / "革命性" / "前所未有" / "划时代"（无数据支撑不用）
- "让您的生活更加美好" / "为您带来极致体验" / "开启全新生活方式"
- "还不快行动起来" / "错过今天再等一年" / "限时特惠不容错过"
- "品质保证" / "值得信赖" / "行业领先"（不附证据等于没说）
- 英文: `game-changing`, `revolutionary`, `next-generation`, `best-in-class`, `unparalleled`

## 6. 小说 / 故事 (Narrative Fiction — 极简三幕剧)

- **核心逻辑**：冲突驱动，人物弧光。
- **模板骨架**：
  - **破局点 (Inciting Incident)**：打破日常的意外事件，迫使主角反应。
  - **试错与升级 (Rising Action)**：主角尝试解决但不断失败，代价变大。
  - **灵魂黑夜与高潮 (Climax)**：逼入绝境，直面恐惧并做出最终选择。
  - **余波 (Resolution)**：恢复平静，主角发生不可逆的改变。
- **质量标准**：情节连贯、语言生动形象、情感铺垫自然、避免明显 AI 痕迹。
- **禁止**：上帝视角总结人物心理、对话标签单一重复（避免全用"说"、"说道"）、场景描写纯视觉化忽视其他感官。

### AI 疲劳词表 (v2.7)
- "心中涌起一股..." / "一种说不清道不明的感觉" / "内心充满了力量"
- "在这一刻，他终于明白了" / "他深深地意识到" / "他不得不承认"
- "然而事情并没有那么简单" / "就在这时，意外发生了"
- "命运的安排" / "冥冥之中" / "仿佛一切早已注定"
- "他的眼神中闪烁着" / "目光如炬" / "嘴角微微上扬"
- 英文: `little did he know`, `it was at that moment`, `a sense of`, `overwhelmed by`

## 7. 小红书/社交媒体短文 (Xiaohongshu / Social Short)

- **核心逻辑**：轻量种草，人设即信任，短平快。
- **模板骨架**：
  - **标题/封面句**：数字+利益点+情绪 hook（如"用了3年的5个效率神器"）
  - **人设锚点**：一句话建立身份（"作为写了100篇技术博客的程序员"）
  - **分点分享**：emoji 分段，每段2-3句，配图位标注[图]
  - **灵魂总结**：一句话金句或反问引发互动
  - **标签与钩子**：#相关标签 + "你呢？评论区聊聊"
- **质量标准**：短段为主（2-3句一段）、大量留白、emoji适度、每段有信息增量。
- **禁止**：长段落、学术腔、过度排版、无图空文、假大空标题。

### AI 疲劳词表 (v2.7)
- "家人们谁懂啊"（过度用翻车）/ "我不允许还有人不知道"
- "绝绝子" / "yyds" / "一整个爱住了"（过度且塑料）
- "这个小东西真的惊艳到我了" / "太太太好用了"
- "无广自用"（如果确实是广告就别说）/ "按头安利"
- "咱就是说" / "就是说呢" / "就是说一整个"（叠甲太厚）

## 8. 视频脚本 / 口播稿 (Video Script / Voiceover)

- **核心逻辑**：听觉驱动，口语化，节奏感。
- **模板骨架**：
  - **Hook (0-3秒)**：一句话抓住注意力（反问/反常识/数据冲击/痛点场景）
  - **身份+价值承诺 (3-10秒)**："我是谁"+"看完这视频你会得到什么"
  - **正文递进 (10-60秒)**：3个递进观点或步骤，每点配画面提示[画面:]
  - **高潮/转折**：制造意外或情绪波峰
  - **结尾行动号召**：关注/点赞/评论的明确指令
- **质量标准**：每句不超过25字、大量停顿标注[...]、画面提示清晰、节奏感强。
- **禁止**：书面化长句、无画面配合的纯文字、语速均匀无变化。

### AI 疲劳词表 (v2.7)
- "你知道吗？" / "你有没有想过？" / "试想一下"（AI 最爱开场白）
- "其实这个问题的答案很简单" / "真正重要的不是...而是..."
- "改变你对...的认知" / "颠覆你的三观" / "看完你会感谢我的"
- "今天我们就来聊聊/今天给大家分享"（出镜过度翻车软广感）
- 英文: `have you ever wondered`, `what if I told you`, `let's dive in`, `you won't believe`

## 9. 邮件 / 商务信函 (Email / Business Letter)

- **核心逻辑**：尊重时间，开门见山，行动导向。
- **模板骨架**：
  - **主题行**：动宾结构，包含核心信息（如"请确认：XX项目排期调整方案"）
  - **开场**：1-2句寒暄或背景（纯事务性邮件可跳过）
  - **核心信息**：3-5句话说明事由+所需行动+截止时间
  - **补充细节**：如有必要，附链接或附件说明（复杂事项建议拆分多封邮件）
  - **结尾**：礼貌收尾+联系方式
- **质量标准**：日常事务邮件控制在 200 字内、一个邮件只说一件事、行动指令明确。
- **禁止**：长篇大论（将长内容放附件）、情绪化表达、无 Deadline 的请求、废话寒暄。

### AI 疲劳词表 (v2.7)
- "希望这封邮件能找到你" / "展信佳" / "顺颂商祺"（过于书面/古板）
- "如有任何问题，请随时与我联系"（换成"有疑问直接找我就行"）
- "感谢您百忙之中抽空阅读" / "感谢您的耐心"（真诚感谢可以，套话不行）
- "持续沟通" / "协同推进" / "共同探讨"（企微/飞书八股文）
- 英文: `I hope this email finds you well`, `touching base`, `circle back`, `moving forward`, `at your earliest convenience`

## 10. 演讲稿 (Speech / Presentation)

- **核心逻辑**：情感驱动，节奏多变，金句记忆点。
- **模板骨架**：
  - **开篇炸弹**：故事/反问/数据/沉默——制造在场感
  - **主题铺垫**：今天我要讲的三件事
  - **主体展开**：观点A → 故事/案例 → 反转/洞察；观点B → ...
  - **情感高潮**：个人经历或脆弱暴露，拉近距离
  - **金句收尾**：一句能被记住的话，配合行动号召
- **质量标准**：口语化短句、12-18词/句、重复关键句3次、停顿标注。
- **禁止**：论文式演讲、无故事纯观点、无与观众互动的设计。

### AI 疲劳词表 (v2.7)
- "尊敬的各位来宾" / "很荣幸站在这里" / "今天我想和大家分享"
- "在这个充满机遇和挑战的时代" / "这是一个最好的时代，也是一个最坏的时代"
- "让我用一个小故事开始今天的演讲"（除非真的有故事）
- "改变世界" / "创造未来" / "引领变革"（没有具体行动的空话）
- 英文: `I'm honored to be here`, `without further ado`, `let's give a round of applause`, `in conclusion, I'd like to leave you with`

## 11. 产品说明书 (Product Manual / User Guide)

- **核心逻辑**：安全第一，步骤清晰。
- **模板骨架**：
  - **安全警告**：如有安全风险，最先加粗说明
  - **产品概述**：一句话定义 + 功能清单（3-5 条）
  - **快速开始**：最小可行步骤（≤5步），每步一句话
  - **详细操作**：按功能模块分节，每节配"操作步骤 → 预期结果"
  - **故障排除**：常见问题 + 原因 + 解决方案（三列表格更好）
  - **技术规格 / 售后**：参数列表、客服/保修信息
- **质量标准**：小学六年级可读水平、每步一句话、模糊指令必须附具体判断标准。
- **禁止**：废话营销语、"功能强大"等空洞形容词、跳过安全警告。

### SEO 优化 (v2.9)

当产品说明书发布为公开帮助中心、教程指南、官网文档或下载页说明时，适用完整 SEO 模块 ([seo-module.md](seo-module.md))。重点关注：用户问题型长尾关键词、H2/H3 操作层级、FAQ/故障排除段落、内链到相关教程或售后页面。若说明书仅用于包装盒、内部培训或私密交付，跳过 SEO 模块。

- "采用先进技术" / "人性化设计" / "专业品质"

## 🆕 12. 毕业论文 / 学位论文 (Thesis / Dissertation)

- **核心逻辑**：系统性学术论证，体量大（本科 1-3 万字，硕博 3-10 万字），章节独立可读。
- **模板骨架**：
  - **封面与声明**：题目、作者、导师、院系、原创性声明、授权说明。
  - **摘要与关键词**：中英文双语，各 300-500 字。研究问题→方法→主要发现→结论→3-5 个关键词。
  - **目录**：三级标题 + 图表索引 + 附录索引。
  - **第一章 绪论**：
    - 研究背景（宏观→微观，2-3 层递进）
    - 研究问题（明确列出 2-4 个具体问题）
    - 研究意义（理论意义 + 实践意义，各 1 段）
    - 论文结构（各章一句话概述）
  - **第二章 文献综述**：按主题或时间线组织，每个小节结尾必须评述"前人研究不足 → 本文如何不同"。
  - **第三章 研究方法**：数据来源+样本规模+收集流程+分析工具/模型。可复现性为最高原则。
  - **第四章 结果与分析**：按研究问题逐一切入，每节先陈述结果再分析原因。
  - **第五章 讨论**：与文献综述对话（哪些与前人一致/不一致？为什么？），解释异常发现。
  - **第六章 结论**：总结核心发现（与研究问题一一对应）→ 理论贡献 → 实践建议 → 研究局限与展望。
  - **参考文献**：统一格式，每篇在正文中必须有引用。
  - **附录**：问卷、访谈提纲、原始数据表等。
- **质量标准**：章节逻辑闭路（每章开头有导言、结尾有小结）、引用密度适当（关键论断必有引用支撑）、避免"本文首次..."等绝对化表述、目录层级不超过三级。
- **禁止**：研究问题与结论脱节、文献综述变成"罗列清单"、方法论描述不足（复现不了）、结论夸大研究发现、引用二手来源而不查原文。

### AI 疲劳词表 (v2.7)
- "随着社会的不断发展" / "在经济全球化的背景下" / "进入新时代以来"
- "本文采用文献研究法"（方法是做了什么事，不是归类命名）
- "具有一定的理论和实践意义" / "提供了重要的参考价值"（必须说具体什么意义）

- 英文: `this paper aims to`, `fills a gap in the literature`, `to the best of our knowledge`, `future work will address`

## 🆕 13. 求职简历 (Resume / CV)

- **核心逻辑**：一页纸说服，以结果为导向，关键词穿透 ATS（简历筛选系统）。
- **模板骨架**：
  - **头部信息**：姓名（加粗/CV 用大号字）+ 电话 + 邮箱 + 作品集/GitHub/LinkedIn 链接（可选）。⚠️ 不写"个人简历"四个字。
  - **求职意向**：目标职位，不超过一行（可选但强烈建议加，尤其是跨行业投递时）。
  - **教育背景**：学校 + 专业 + 学位 + 起止时间（倒序）。GPA 3.5+ 可写，应届生可加相关课程 3-5 门。
  - **工作/实习经历**（核心区）：
    - 格式：公司 | 职位 | 起止时间
    - 每条用 **动宾短语 + 量化结果** 的 STAR 句式：
      - ❌ "负责公众号运营"
      - ✅ "运营公司公众号（3 万粉丝），通过选题优化使月均阅读量从 2000 提升至 15000（+650%），转化付费用户 120 人"
    - 每段经历写 3-5 条，按影响力排序
  - **项目经历**：项目名 + 一句话介绍 + 你的角色 + 技术栈/工具 + 量化成果
  - **技能**：按类别分组（编程语言 | 工具/框架 | 语言能力），不要写进度条或星级
  - **荣誉/证书**：挑含金量高的写（≤5条），无关荣誉删掉
  - **自我评价**（可选）：3 句以内。第一句身份定位，第二句核心优势证据，第三句与目标岗位的匹配。不写"性格开朗、吃苦耐劳"。
- **质量标准**：一页纸（10 年以上经验可两页）、动词开头、数字密集、无错别字、PDF 格式投递。ATS 友好：单栏布局、无表格、无图标、标准字体（宋体/微软雅黑/Arial）。
- **禁止**：写"个人简历"标题、使用第一人称"我"、假大空形容词（"工作认真负责"等）、标注技能熟练度百分比/星级、放照片（除非岗位要求）、列出与岗位无关的爱好。

### AI 疲劳词表 (v2.7)
- "本人性格开朗，积极向上，具有较强的团队合作精神"（三行字说了一行等于没说的东西）
- "吃苦耐劳" / "认真负责" / "学习能力强" / "适应能力强"（没有证据的形容词都是噪音）

- "取得了良好的效果" / "得到了领导的一致好评" / "获得了显著提升"（数字呢？）
- 英文: `highly motivated`, `team player`, `results-driven`, `excellent communication skills`, `attention to detail`

## 🆕 14. 朋友圈文案 (WeChat Moments / Social Post)

- **核心逻辑**：人格化传播，像真人在说话。朋友圈不是微商广告位，是人设展示窗。
- **模板骨架**：
  - **钩子开头**：反常场景 / 自嘲 / 悬念 / 一句话情绪冲击
    - "完了，手滑把老板的汇报 PPT 删了..."
    - "花了 3 个月，终于敢说自己是会 XX 的人了"
  - **叙事体正文**（3-5 句）：像跟朋友讲一件事，有细节、有画面、允许废话
    - 不要用分点列表（那是工作汇报）
    - 不要写"首先/其次/最后"
  - **情绪落点**：幽默收、温暖收、反问收、留白收都可以——就是不要"教育"收
  - **配图位**：[图1] [图2] ... 或标注"配一张 XX 的图"
- **质量标准**：真实感第一（读出来像在聊天）、有具体的细节锚点（时间/地点/数字/对话）、允许不完美（句子可以口语化到不符合书面语法）。
- **禁止**：分段带序号、强行正能量结尾、"给大家推荐一下"这类推销口吻、复制粘贴的节日祝福模板、超过 5 段以上的长文（那是公众号文章）。

## 🆕 15. 产品评测 (Product Review)

- **核心逻辑**：帮助读者做购买决策，坦诚比文采重要。信任来自你暴露缺点的那一刻。
- **模板骨架**：
  - **一句话结论**（置顶）：直接告诉读者"适合谁、不适合谁"。
    - "如果你预算 3000 以内且有剪辑需求，这款闭眼入。但如果只是日常办公，往下翻看替代品。"
  - **产品速览**：价格 + 规格表（3-5 行）+ 开箱/上手第一印象。
  - **深度体验**（按场景分组，非按参数罗列）：
    - 场景一：日常使用 → 好的地方 + 不够好的地方
    - 场景二：重度使用/极限测试 → ...
    - 场景三：特定人群视角 → ...
  - **横向对比**：与 2-3 个竞品的关键差异（表格最佳），不要只说"完胜"，要说"在这个维度上 X 比 Y 更..."
  - **优缺点汇总**：优点 ≤5 条，缺点 ≤5 条，每条一句话，不藏着掖着。
  - **购买建议**：针对不同预算/需求的分支推荐 + 购买渠道提示。
- **质量标准**：优缺点数量相当（全是优点就是广告）、每个观点附具体使用场景或数据、与竞品对比时注明对比条件和价格基准。
- **禁止**：全程夸爆不提缺点、没有实际使用就评测（"看参数推测"）、把官方宣传语当自己的评价、不标竞品价格进行不公平对比。

### SEO 优化 (v2.9)

当产品评测面向公开平台发布时，适用完整 SEO 模块 ([seo-module.md](seo-module.md))。重点关注：商业调查型搜索意图、"值得买吗/对比/推荐/评测"长尾关键词、竞品名称自然覆盖、标题是否包含产品名与核心判断、横向对比和购买建议是否能承接搜索需求。

- "无论是...还是...都表现出色" / "在XX方面展现了不错的实力"
- "瑕不掩瑜" / "性价比极高" / "闭眼入" / "早买早享受"

## 🆕 16. 开题报告 (Thesis Proposal)

- **核心逻辑**：研究价值论证，突出创新点的前期可行性分析。
- **模板骨架**：
  - **选题背景与意义**：宏观到微观的递进，说明为什么选这个题。
  - **国内外研究现状**：按主题分类梳理前人工作，每类结尾评述不足与本文的切入点。
  - **研究目标与内容**：列出 2-4 个具体目标，每目标对应 1-2 项具体内容。
  - **研究方法与技术路线**：具体到用哪些工具/模型/数据集，各步骤的输入输出关系。
  - **创新点与可行性分析**：创新点 1-3 个，逐条说明"新在哪里"+"为什么可行"。
  - **预期成果**：论文、原型系统、专利、数据集等，含可交付物形式。
  - **进度安排**：以月或半月为粒度的时间轴描述。
  - **参考文献**：近 3 年≥60%，格式统一。
- **质量标准**：研究问题边界清晰（能明确回答"做什么和不做什么"）、创新点可量化可验证（不是"综合创新"）、技术路线有具体操作步骤而非空泛框图、文献引用时效性好。
- **禁止**：立项依据写成科普文（不要从牛顿力学讲起）、研究现状只罗列不评述（每段末尾必须有评述句）、技术路线写成"首先采集数据→然后分析→最后写论文"等空话、进度安排写成流水账（"第1-2周搜集资料，第3-4周阅读文献"）。

## 🧬 跨模板融合指南 (v2.5)

当 Creator 推荐跨模板配方（例如 `技术博客:70% + 产品评测:30%`）时，Executor 需按以下规则融合两个模板的特性。

### 融合优先级规则

| 维度 | 主模板 (≥60%) | 辅模板 (≤40%) | 融合规则 |
|------|---------------|---------------|----------|
| **结构骨架** | ✅ 主导 | — | 以主模板的章节/段落结构为主线 |
| **语调风格** | — | ✅ 调色 | 在主线结构内，融入辅模板的语气特征 |
| **开头钩子** | 50% | 50% | 双模板钩子可混合：主模板场景 + 辅模板表达方式 |
| **例证选择** | 主模板范型 | 辅模板范型 | 按比例分配（如 70:30 → 7成例证来自主模板套路，3成来自辅模板） |
| **词汇库** | 白名单并集 | 黑名单交集 | 白名单：两个模板的偏好词都可用；黑名单：取最严格交集 |
| **AI 痕迹检测** | 主模板清单 | 辅模板追加 | 以主模板的检测清单为主，追加辅模板特有的高危模式 |

### 常见融合配方示例

| 配方 | 典型场景 | 效果 |
|------|----------|------|
| `技术博客:70% + 产品评测:30%` | 技术分享但附带项目/工具安利 | 主技术干货，辅佐自然推荐，不硬推 |
| `博客文章:60% + 演讲稿:40%` | 写得像聊天一样的技术分享 | 口语化博客，像在台上讲给你听 |
| `朋友圈文案:50% + 商业文案:50%` | 社交平台的产品种草 | 人格化文案，像朋友在推荐而非广告 |
| `学术论文:70% + 博客文章:30%` | 学术成果的通俗解读 | 严谨底线不动，但可读性提升 |
| `技术文档:60% + 视频脚本:40%` | 技术教程的录制稿 | 结构化操作流程，适合边看边做 |
| `演讲稿:80% + 随笔散文:20%` | 需要走心的公开演讲 | 演讲骨架不动，但情感层次比纯演讲稿丰富 |

### 自定义配方 (用户自创)

用户可提出不在常见列表中的配方组合。Creator 需在 Phase 2 中明确写出融合规则：

```
🧬 自定义配方: [模板A:X%] + [模板B:Y%]
   融合策略: 结构用A，语调融合B的特征，具体为...
   Executor 注意事项: ...
```

### 配方冲突处理

如果两个模板的某些维度存在根本性冲突（例如学术论文要求"严禁口语"而脱口秀脚本要求"极度口语"），Creator 在 Phase 2 需明确裁定优先级并向用户确认。

冲突示例：
- `学术论文 + 演讲稿` → 冲突：学术禁第一人称，演讲鼓励第一人称。裁定：主模板优先→禁第一人称
- `商业文案 + 吐槽测评` → 冲突：商业文案避免负面，吐槽测评鼓励真实批评。裁定：向用户确认边界
