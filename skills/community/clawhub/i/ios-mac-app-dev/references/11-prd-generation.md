# PRD 生成工作流

> 基于 matt pocock `to-prd` 改写，适配 iOS/macOS App 产品需求文档。

## 使用时机

- 烤问 session 结束后，将讨论结果转为正式 PRD
- 用户说「生成 PRD」、「写个需求文档」、「整理一下需求」

## 流程

### 1. 理解当前代码库

先读取项目现有状态：
- 读取 `CONTEXT.md`（如有） — 获取领域术语表
- 读取 `docs/adr/` — 了解已有架构决策，不重复讨论
- 快速扫描相关模块的代码结构

### 2. 确认接缝（Seams）

找出功能将触及的模块边界：
- 优先复用现有接缝，避免新建
- 如需新接缝，提出在最高层级
- **原则：接缝越少越好，理想情况是 1 个**

与用户确认接缝是否符合预期。

### 3. 按模板写 PRD

---

## PRD 模板

```
## 问题陈述
从用户视角描述当前痛点。

## 解决方案
从用户视角描述解决方案。

## 用户故事
1. 作为 [角色]，我想要 [功能]，以便 [收益]
2. 作为 [角色]，我想要 [功能]，以便 [收益]
...

（列表应详尽，覆盖功能的各个方面）

## 实现决策
- 将构建/修改的模块
- 将修改的接口
- 技术澄清（开发者提供的）
- 架构决策
- Schema 变更（如数据模型）
- API 契约（如有后端交互）
- 特定交互行为

⚠️ 不包含具体文件路径或代码片段（容易过时）
   例外：如果原型产生了精确编码决策的内容（状态机、Reducer、Schema、类型结构），
   在相关决策处内联，并注明来自原型。

## 测试决策
- 良好测试的描述（仅测试外部行为，不测实现细节）
- 哪些模块将被测试
- 测试的先验参考（代码库中类似测试）

## 范围外
明确本次不做的东西。

## 补充说明
关于此功能的任何其他备注。
```

---

### 4. 发布

将 PRD 发布到项目的 Issue Tracker（如配置了 GitHub Issues），或写入项目根目录 `docs/prd/` 下。

打上 `ready-for-implementation` 标签。
