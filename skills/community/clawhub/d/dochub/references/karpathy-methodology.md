# Karpathy LLM Wiki 方法论参考

本文档详细记录 Andrej Karpathy 的 LLM Wiki 方法论，作为 dochub 技能的理论基础。

## 核心理念

**LLM Wiki = 编译器系统**

| 编译器概念 | LLM Wiki 对应 |
|-----------|-------------|
| 源代码 | `raw/` — 不可变源文档 |
| 编译器 | LLM Agent |
| 可执行输出 | `wiki/` — 结构化知识 |
| 编译过程 | Ingest 操作 |
| 测试 | Lint 操作 |
| 运行时 | Query 操作 |

Karpathy 的类比：**Obsidian 是 IDE，LLM 是程序员，wiki 是代码库。** 人类几乎不直接编辑 wiki，那是 LLM 的领地。

## 三层架构详解

### Layer 1: 原始层 (raw/)

**核心原则：不可变。** 这是所有知识的真相来源，LLM 只读不写。

- 手动策划的源文档集合
- 每个声明必须可追溯到 raw/ 中的文件
- 作为验证基线：wiki 中每个声明都可以反向验证

### Layer 2: 知识层 (wiki/)

**核心定位：完全由 LLM 生成和维护。** 人类主要阅读，LLM 主要写入。

**页面类型：**

| 类型 | 目录 | 用途 |
|------|------|------|
| `index` | `wiki/_index.md` | 主编排索引，所有检索的入口 |
| `log` | `wiki/_log.md` | 仅追加操作日志，按时间记录 |
| `overview` | `wiki/_overview.md` | 知识库统计与可视化概况 |
| `source-summary` | `wiki/sources/` | 每个源文档的摘要页 |
| `concept` | `wiki/concepts/` | 跨文档的主题概念页 |
| `entity` | `wiki/entities/` | 人物/组织/系统/地点实体页 |
| `comparison` | `wiki/comparisons/` | 对比分析页 |

**两个关键结构文件：**

1. **`_index.md`** — 内容目录。每次 ingest 更新。LLM 先读这个（~1-2K token），然后定位相关页面。
2. **`_log.md`** — 仅追加操作日志。记录每次摄入、每次更新、每次发现的矛盾。

### Layer 3: Schema 层 (_schema.md)

**Karpathy 认为这是"整个系统中最重要的文件"。**
它将通用 LLM 转变为有纪律的知识工作者。

定义内容：
- Wiki 结构规范
- 命名约定
- 页面模板
- 操作工作流
- 领域标签体系

## 三大操作

### Ingest（摄入）

单次摄入可能触及 10-15 个 wiki 页面：
1. 读取源文档，讨论关键要点
2. 创建 `wiki/sources/{name}.md` 摘要页
3. **级联更新**所有相关页面
4. 必要时创建新的概念/实体页
5. 更新 `_index.md`
6. 追加 `_log.md`

### Query（查询）

渐进式披露策略：
- L0: 项目上下文 (~200 token)
- L1: `_index.md` (1-2K token)
- L2: 匹配页面 (2-5K token)
- L3: 完整源文档 (5-20K token)

输出：综合回答 + 来源引用 + 置信度标注

### Lint（健康检查）

检查维度：
- 矛盾检测
- 孤立页面（无入链）
- 缺失概念（被引用但未创建）
- 过期声明（被新来源取代）
- 置信度异常
- 知识缺口

## 社区实战经验

来自 bluewater8008 团队的生产环境总结：

1. **先分类再提取** — 不同文档类型用不同处理策略
2. **给索引设 Token 预算** — 四级渐进披露，不读完索引不读全文
3. **每种实体类型一个模板** — 定义 7 种实体类型，各有专属必填字段
4. **每个任务产出两个输出** — 答案 + 知识库更新（最关键！）
5. **从第一天就设计跨域标签** — domain 标签后期补很痛苦
6. **人类负责验证** — LLM 是作者，你是主编

## 规模与局限性

- Karpathy 的 wiki 约 100 篇文章、40 万字
- 更大规模推荐 qmd（本地 Markdown 搜索引擎，BM25 + 向量混合搜索）
- 冷启动前 5-10 篇需要深度参与调校
- 验证成本易被低估——严肃场景需额外抽查
