# 量化交易 — 事件驱动的交易智能体系统

## 定位

本分支是 `agentic-engineering` 的量化交易子分支，专注：

- **状态维持型**范式——交易系统是持续运行的，不是"做完就结束"
- 事件驱动（行情变化 → 状态触发 → 执行器唤醒 → 回到 idle）
- 多 Agent 对抗辩论 + 风控审批

## 核心特征：为什么量化交易必须是状态维持型

| 特征 | 解释 |
|------|------|
| 常态运行 | 市场不停，系统不停 |
| 事件触发 | 价格变动、新闻事件触发决策，不是人工发起任务 |
| 状态基线 | 持仓、敞口、风控阈值是持续维护的状态，不是一次性输入 |
| 人工审批 | 大额交易必须经过人机协同断点 |
| 可恢复 | 交易系统崩溃必须从 Checkpoint 恢复，不能丢失状态 |

## Agent 角色设计

### 研究层

| Agent | 职责 | 输出 |
|-------|------|------|
| 数据分析师 | 技术指标、K线形态、量价关系 | 技术面信号 |
| 新闻分析师 | 财经新闻、政策变动、舆情 | 基本面信号 |
| 情绪分析师 | 社交媒体情绪、资金流向 | 市场情绪指标 |

### 辩论层

| Agent | 职责 | 目的 |
|-------|------|------|
| 多头辩手 | 找做多理由 | 暴露看多逻辑漏洞 |
| 空头辩手 | 找做空理由 | 暴露看空逻辑漏洞 |

### 决策层

| Agent | 职责 |
|-------|------|
| 研究经理 | 综合多空观点，形成投资策略 |
| 交易员 | 策略→执行方案（进场、仓位、止损） |
| 风控经理（激进） | 评估上行风险 |
| 风控经理（保守） | 评估下行风险 |
| 风控经理（中性） | 评估整体风险敞口 |
| 投资组合经理 | 最终决策，提取可执行交易信号 |

## 状态机设计

```python
class TradingState(TypedDict):
    # 输入
    market_data: dict           # 行情快照
    news_signals: list          # 新闻信号
    sentiment_score: float      # 情绪分数

    # 研究输出
    bull_arguments: list        # 多头论点
    bear_arguments: list        # 空头论点
    strategy: Optional[dict]    # 投资策略

    # 交易执行
    trade_plan: Optional[dict]  # 交易方案
    risk_assessment: dict       # 风险评估
    pending_approval: bool      # 等待人工审批

    # 系统状态
    position: dict              # 当前持仓
    pnl: float                  # 盈亏
    risk_limits: dict           # 风控阈值
```

## 流转逻辑

```
[市场事件] → 数据采集节点 → 研究分析节点
                                    ↓
                            多空辩论节点 ←→ 辩论循环
                                    ↓
                            策略合成节点
                                    ↓
                            交易方案节点
                                    ↓
                        ┌─── 风控审批节点 ───┐
                        ↓                    ↓
                   [金额 < 阈值]        [金额 ≥ 阈值]
                   自动执行          interrupt() 人工审批
                        ↓                    ↓
                    执行节点 ←────────── Command(resume)
                        ↓
                    状态更新 → 回到 idle
```

## 风控断点策略

| 断点类型 | 触发条件 | 行为 |
|----------|----------|------|
| 仓位断点 | 单笔交易金额 > 阈值 | interrupt() 等待审批 |
| 敞口断点 | 总敞口 > 限额 | interrupt() 强制人工干预 |
| 亏损断点 | 单日亏损 > 止损线 | interrupt() + 告警 |
| 异常断点 | 行情数据异常/延迟 | 自动暂停，等待确认 |

## 回测与实盘切换

| 模式 | 数据源 | 执行 | 断点 |
|------|--------|------|------|
| 回测 | 历史数据 | 模拟成交 | 可跳过人工审批 |
| 模拟盘 | 实时数据 | 模拟成交 | 保留审批 |
| 实盘 | 实时数据 | 真实成交 | 全部审批 |

## 关键原则

1. **Never lose state** — 交易系统崩溃 = 可能的巨大损失，Checkpoint 必须每步持久化
2. **Risk first** — 风控节点是硬性断点，不是可选流程
3. **Debate before trade** — 多空对抗辩论是决策前置条件，不做单边决策
4. **Human at the gate** — 超阈值交易必须 interrupt()，不做全自动大额交易
5. **Idle is the baseline** — 无信号时系统 idle，有事件时唤醒，不需要持续运行计算
