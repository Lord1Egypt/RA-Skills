# 串并行规划规则

> **加载时机**：模型思考生成步骤时加载（决策树分支③）

---

## 一、原子步骤拆解

将 request 中的主要动作拆解为**原子步骤**（不可再分的最小执行单元）。

| 规则 | 说明 | 示例 |
|------|------|------|
| 动作→步骤 | 每个独立动作对应一个步骤 | "做PPT" → 收集资料 / 设计大纲 / 制作PPT / 审核 / 导出 |
| 前置挖掘 | 从 motivation 中提取隐含前置步骤 | "钛合金成分" → 增加"材质专题页"；"研发200天" → 增加"研发故事" |
| 时间倒推 | 根据 purpose 时间约束倒推交付步骤 | "明天要" → 最后一步必须是"导出PPT" |

## 二、串并行标记

- **无数据依赖且不冲突**的步骤标记为可并行，归入同一 `parallel_group`
- `parallel_group` 相同的步骤可同时执行
- 无标记的步骤按序串行执行

**标记方式**：
```
步骤 4a: 制作内容页 ── parallel_group: pg1 ── milestone: ❌
步骤 4b: 设计视觉风格 ── parallel_group: pg1 ── milestone: ❌
```

## 三、关联热度（dependency_heat）

步骤间的**关联热度**表示两个步骤之间的数据依赖强度（0-10），用于指导渐进加载时的步骤拆分和合并决策。

| 热度值 | 含义 | 示例 |
|--------|------|------|
| 0 | 无依赖，完全独立 | "选模板" 与 "定配色" |
| 1-3 | 弱依赖，可延迟或跳过 | "导出" 依赖 "审核" |
| 4-6 | 中等依赖，通常需顺序执行 | "制作内容页" 依赖 "大纲" |
| 7-9 | 强依赖，必须严格顺序 | "大纲" 依赖 "资料收集" |
| 10 | 强耦合，不可拆分 | 同一步骤内的子操作 |

## 四、里程碑标记（milestone）

- 如果某步骤**失败会导致整个任务失败**，标记为 `milestone: true`
- milestone 步骤失败时，**中止**后续步骤，报告失败原因
- 非 milestone 步骤失败时，可选择跳过继续

## 五、规划输出格式

```
【规划结果】

📌 任务：[任务名称]
⏰ 约束：[时间/资源约束]

执行步骤：
  步骤 1: [名称] ── serial ── milestone: ✅/❌
    action: [具体操作]
    depends_on: [依赖步骤]
    dependency_heat: [关联热度]

  步骤 2a: [名称] ── parallel_group: pg1 ── milestone: ✅/❌
    action: [具体操作]
    depends_on: [依赖步骤]
    dependency_heat: [关联热度]

  步骤 2b: [名称] ── parallel_group: pg1 ── milestone: ✅/❌
    ...

  步骤 3: [名称] ── serial ── milestone: ✅/❌
    ...

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
来源：[规则级json / 能力级json / 模型思考]
【询问用户】是否按此规划执行？如有调整请说明。
  1. 确认执行
  2. 有调整（说明调整内容）
  3. 要求重新规划/重新思考
```

**用户选择处理**：
- **确认执行** → 执行任务
- **有调整** → 按用户意见修改规划 → 再次展示 → 再次询问
- **有调整后的额外询问** → 调整是否要：
  - A. 更新原有能力级 json（改写对应步骤）
  - B. 创建新的能力级 json（保留原有）
  - C. 仅本次使用，不更新 json
- **要求重新规划** → 忽略已有 json，启动模型重新思考生成步骤

## 六、双视角推理与整合规则

> 本规则仅在决策树分支③（模型思考）时生效。详见 `constraint_annotation.md` 第四节。

### 6.1 聚焦方案生成规则

1. 对每个 5W2H 维度，选择「最窄/最保守/最直接」的值
2. 严格遵守所有🔴硬约束（任何步骤不可违反）
3. 只使用已验证、低学习成本的方法（不引入新工具）
4. 时间窗口取截止时间的一半（如周五截止 → 周三前完成初稿）
5. 每个步骤不超过 30 分钟可完成
6. 输出格式：步骤列表，每个步骤含 action / depends_on / dependency_heat

### 6.2 发散方案生成规则

1. 对每个 5W2H 维度，选择「最宽/最大胆/最间接」的值
2. 🔴硬约束可轻微突破，但必须在步骤中标注 🌟 和「风险：...」，并提供备用方案
3. 引入至少一个非惯用工具或方法（如 AI 生成、手绘、录音口述）
4. 时间上尝试「反向工作法」（先做最不可能的部分）
5. 包含一个「如果无限资源会怎么做」的变体步骤
6. 输出格式：步骤列表 + 创新点列表

### 6.3 整合规则

1. **以聚焦方案为骨架**：整合后的步骤顺序 = 聚焦方案顺序
2. **嵌入发散创新点**：
   - 遍历发散方案的每个创新点
   - 判断：该创新点对应聚焦方案的哪个步骤？（通过语义相似度）
   - 若找到对应步骤 → 将该创新点作为「🌟增强」嵌入该步骤
   - 若找不到对应步骤 → 作为独立🌟步骤插入最合适位置
3. **标注来源**：所有🌟步骤注明「增强：来自发散方案」
4. **风险说明**：若🌟步骤涉及🔴硬约束突破，附「⚠️ 风险：...」
5. **输出**：单一的整合后步骤列表（不输出双方案对比）

### 6.4 工作包分解规则

对整合后的步骤列表，进行工作包分解：

**聚焦式分解**（适用于大多数任务）：
- 粒度：每个工作包 2-4 小时可完成
- 格式：`WP{n}: [任务描述] (预计耗时，前置：WP{x},WP{y})`
- 每 3-5 个 WP 设置一个检查点：`📍 检查点：验证[XX]`

**探索式分解**（适用于创意/研究类任务）：
- 格式：`模块A/B/C：[探索内容]`
- 不要求严格顺序，可并行探索
- 每个模块标注「探索目标」和「成功标准」

## 七、约束遵循检查

生成整合步骤后、输出前，执行以下检查：

| 检查项 | 修正动作 |
|--------|---------|
| 是否有步骤违反了🔴硬约束？ | 如有 → 重新生成该步骤 |
| 是否有🔴硬约束未被任何步骤覆盖？ | 如有 → 补充步骤 |
| 发散创新点（🌟步骤）是否标注了风险？ | 如涉及🔴突破 → 必须标注 |
| 工作包粒度是否超过4小时？ | 如超过 → 继续拆分 |
