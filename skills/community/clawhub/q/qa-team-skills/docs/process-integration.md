# qa-team-skills 流程嵌入指南

> 如何将 6 大指令嵌入研发流程的标准节点，确保 AI 辅助融入日常而非附加品。

## 版本生命周期中的嵌入点

```
需求评审 ──→ 用例设计 ──→ 开发提测 ──→ 测试执行 ──→ 缺陷分析 ──→ 发布评审
   ↓            ↓                              ↓            ↓
 /qa-prd      /qa-case                       /qa-bug     /qa-report
                                                         /qa-team
                                       /qa-agent (贯穿)
```

## 各阶段详细说明

### 1. 需求评审阶段

| 项目 | 说明 |
|------|------|
| **触发时机** | PRD 初稿完成，评审会前 1 天 |
| **前置条件** | PRD 已由产品经理自检通过 |
| **使用指令** | `/qa-prd` |
| **输出用途** | 作为评审会的问题清单，投屏逐条讨论 |
| **过期条件** | PRD 发生实质性变更（功能增删、流程重设计）时，结果需重新生成 |
| **参与角色** | 测试工程师用 AI 预审 → 评审会上逐条确认 → 产品经理回答"需澄清的问题" |

**工作流示例**：
```
1. 产品经理发出 PRD v1.0
2. 测试工程师粘贴 PRD 到 /qa-prd，生成评审报告
3. 评审会上：AI 发现的问题逐条过 + 人发现的增量问题
4. 会议纪要记录：哪些问题已确认、哪些需要 PRD 修订
5. PRD 修订后（如有实质性变更），重新运行 /qa-prd
```

---

### 2. 用例设计阶段

| 项目 | 说明 |
|------|------|
| **触发时机** | PRD 评审通过，接口文档定稿后 |
| **前置条件** | /qa-prd 的问题清单已闭环（高优问题已确认处理方案） |
| **使用指令** | `/qa-case`（如有 Agent 产品，并行使用 `/qa-agent`） |
| **输出用途** | 生成用例 → 人工审阅 P0 用例 → 提交用例评审 → 导入 Jira/禅道 |
| **过期条件** | 接口文档变更、需求逻辑调整时，受影响的用例需重新生成 |
| **参与角色** | 测试工程师生成 → 测试经理或 peer 审阅 P0 用例 → 确认后导入 |

**工作流示例**：
```
1. 测试工程师输入需求描述 + /qa-prd 的问题清单（手动粘贴）
2. AI 生成 6 类测试用例（功能/边界/异常/安全/性能/兼容性）
3. P0 用例逐条人工审阅，确认测试数据和步骤可执行
4. 用例评审会上讨论
5. 复制追踪标签到 Jira/禅道创建对应 Issue
```

---

### 3. 开发提测阶段

| 项目 | 说明 |
|------|------|
| **触发时机** | 开发自测通过，代码合并到测试分支 |
| **前置条件** | 冒烟用例 100% 通过（由开发在提测前执行） |
| **使用指令** | `/qa-team` — 准入检查 |
| **输出用途** | 提测条件检查清单，不通过则退回开发 |
| **参与角色** | 测试经理或值班测试工程师执行准入检查 |

**工作流示例**：
```
1. 开发在 Jira 中将 Issue 状态改为"待测试"
2. 测试经理运行 /qa-team 准入检查
3. 准入不通过 → 标注不通过项 → 退回开发 → 整改后重新申请
4. 准入通过 → 开始正式测试
```

---

### 4. 测试执行阶段

| 项目 | 说明 |
|------|------|
| **触发时机** | 准入通过后 |
| **前置条件** | 测试环境就绪、测试数据准备完毕 |
| **使用指令** | 执行 /qa-case 生成的用例（AI 已生成，人工执行） |
| **期间辅助** | 发现缺陷时使用 `/qa-bug` 分析根因 |
| **参与角色** | 测试工程师执行用例 → 提交缺陷 → /qa-bug 辅助分析 |

---

### 5. 缺陷分析阶段

| 项目 | 说明 |
|------|------|
| **触发时机** | 发现缺陷时（单个）/ 迭代末尾（批量） |
| **前置条件** | 缺陷复现步骤明确 |
| **使用指令** | `/qa-bug`（单个分析 / 批量模式） |
| **输出用途** | 单个：跟缺陷单一起贴给开发；批量：复盘用 |
| **参与角色** | 测试工程师分析 → 开发确认根因 → 修复 → 回归 |

---

### 6. 版本发布评审阶段

| 项目 | 说明 |
|------|------|
| **触发时机** | 测试执行完毕，准备发布前 |
| **前置条件** | 所有 P0/P1 用例执行完毕 |
| **使用指令** | `/qa-team` — 准出检查 + 版本质量评估；`/qa-report` — 阶段报告 |
| **输出用途** | 准出结论 + 质量评分 → 作为发布评审会的输入材料 |
| **参与角色** | 测试经理输出 → 研发总监/VP 决策是否发布 |

## 指令-角色-时机对照表

| 指令 | 主要角色 | 触发时机 | 输入来源 | 输出去向 |
|------|----------|----------|----------|----------|
| `/qa-prd` | 测试工程师 | 评审会前 | PRD 文档 | 评审会 |
| `/qa-case` | 测试工程师 | PRD 评审后 | 需求描述 + 评审问题 | 用例评审 → Jira/禅道 |
| `/qa-agent` | 测试工程师 | 与 /qa-case 并行 | Agent 描述 | 用例评审 |
| `/qa-bug` | 测试工程师 | 发现缺陷时 | 缺陷描述 + 日志 | 开发 |
| `/qa-report` | 测试工程师 | 每日/周/阶段末 | 测试数据 | 站会/周会/发布评审会 |
| `/qa-team` | 测试经理 | 提测/发布节点 | 团队数据 | 管理决策 |

---

## 常用组合

| 场景 | 指令组合 |
|------|----------|
| 新需求从零开始 | `/qa-prd` → `/qa-case` |
| 迭代测试中 | `/qa-bug`（发现缺陷时）+ `/qa-report` 日报（每天） |
| 迭代结束 | `/qa-report` 阶段报告 + `/qa-team` 准出检查 + 版本质量评估 |
| 周一站会 | `/qa-report` 周报 + `/qa-team` 团队汇总 |
| 新 Agent 上线 | `/qa-agent` + `/qa-case`（Agent 的基础功能部分） |
| 线上出事故 | `/qa-team` 漏测复盘 + `/qa-bug` 根因分析 |
| 季度汇报 | `/qa-report` 季度报告 + `/qa-team` 团队效能 |
