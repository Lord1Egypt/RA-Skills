<p>Use this skill for Apple Notes tasks on macOS.</p>
<p>Typical triggers:</p>
<ul>
<li>Read an existing Apple Note</li>
<li>Search notes by title or content</li>
<li>Create a note</li>
<li>Append or update a note</li>
<li>Create a folder</li>
<li>Move a note into a folder</li>
<li>Organize uncategorized notes</li>
<li>Save project progress into Apple Notes</li>
<li>Save processed content into a specific note</li>
</ul>
<p>Do not use this skill for summarizing or translating itself. Do that work first, then use this skill to operate on Apple Notes.</p>
<p>Classification is allowed when it directly supports organization. If the task is to sort uncategorized notes, the agent may infer a practical destination folder and then move the note there. Keep the classification simple and action-oriented. Do not turn this skill into a general taxonomy design exercise.</p>
<h2>Preferred Method</h2>
<p>Use the already validated Apple Notes access method first. Do not waste time retrying multiple unrelated approaches.</p>
<p>Default rule:</p>
<ul>
<li>Prefer <code>osascript</code> with Apple Notes automation</li>
<li>Reuse the known working command pattern for the current machine</li>
<li>Only switch approach if the preferred method clearly fails</li>
</ul>
<h2>Working Style</h2>
<p>When operating on Apple Notes:</p>
<ul>
<li>Search before creating when there is any chance the note already exists</li>
<li>Prefer append/update over replacing the whole note</li>
<li>Keep formatting clean and readable</li>
<li>Be conservative with move and delete</li>
<li>If the target note is ambiguous, stop and clarify instead of guessing</li>
</ul>
<h2>Practical Workflows</h2>
<h3>Organize Uncategorized Notes</h3>
<ul>
<li>Search or list candidate notes first</li>
<li>Infer a reasonable folder based on title and content</li>
<li>Create the folder if it does not exist</li>
<li>Preview the source note and destination folder before moving in bulk</li>
<li>If classification confidence is low, stop and ask instead of guessing</li>
</ul>
<p>For large messy folders, do not try to solve everything in one pass.</p>
<p>Preferred sequence:</p>
<ul>
<li>First pass: move only high-confidence matches</li>
<li>Second pass: create a few practical flat folders for recurring themes</li>
<li>Final pass: move the leftovers into a deliberate catch-all folder instead of leaving them unorganized forever</li>
</ul>
<p>For Apple Notes specifically, flat folder names are safer than fake hierarchical names.</p>
<p>Prefer:</p>
<ul>
<li><code>系统结构</code></li>
<li><code>觉察与能量</code></li>
<li><code>行动与方法</code></li>
<li><code>对话摘录</code></li>
<li><code>思辨片段</code></li>
</ul>
<p>Avoid creating pseudo-nested names unless you know the current Notes setup supports them well.</p>
<h3>Update Project Progress</h3>
<ul>
<li>Search for the existing project note first</li>
<li>If found, append a dated update section</li>
<li>If not found, create a project note in the best matching folder</li>
<li>Preserve existing history; do not replace the whole note</li>
</ul>
<h3>Save Processed Content Into Notes</h3>
<ul>
<li>Finish the upstream work first, such as extraction, translation, or summary</li>
<li>Search for the target note or decide whether a new one is needed</li>
<li>Write the final content in a clean Apple Notes friendly structure</li>
</ul>
<h2>Core Actions</h2>
<h3>Read Notes</h3>
<p>Use when the user wants existing content, project history, or note contents.</p>
<p>Default behavior:</p>
<ul>
<li>Find the note first by title or nearby keywords</li>
<li>If multiple results match, narrow before reading</li>
<li>Read the target note only after identifying it confidently</li>
</ul>
<h3>Search Notes</h3>
<p>Use search when:</p>
<ul>
<li>The exact note name is unknown</li>
<li>The user refers to a topic, not a title</li>
<li>You need to locate candidate notes before moving or updating</li>
</ul>
<p>Return the best matching notes first, then act on the chosen note.</p>
<h3>Create Note</h3>
<p>Create a new note only when:</p>
<ul>
<li>Search shows no existing target note</li>
<li>The user explicitly wants a new note</li>
</ul>
<p>Before creating:</p>
<ul>
<li>Choose the correct folder if known</li>
<li>Use a clear title</li>
<li>Write content in clean structure, not a text block dump</li>
</ul>
<h3>Edit Or Append</h3>
<p>Prefer append/update when a note already exists.</p>
<p>Default rule:</p>
<ul>
<li>Append new sections rather than rewriting old content</li>
<li>Preserve existing structure unless the user asks for cleanup</li>
<li>Avoid destroying manually written content</li>
</ul>
<h3>Create Folder</h3>
<p>Create folders when:</p>
<ul>
<li>The user is organizing notes</li>
<li>A project needs a stable home</li>
<li>A clear category does not already exist</li>
</ul>
<p>Use simple, stable names. Do not create many near-duplicate folders.</p>
<h3>Move Note</h3>
<p>Move a note only after confirming the destination is correct.</p>
<p>Default rule:</p>
<ul>
<li>Search and identify the note</li>
<li>Confirm the destination folder is the intended one</li>
<li>For bulk moves, preview the mapping before executing</li>
</ul>
<h3>Delete Note</h3>
<p>Delete is high risk.</p>
<p>Default rule:</p>
<ul>
<li>Do not delete unless the user clearly asks</li>
<li>If the target is ambiguous, clarify first</li>
<li>Prefer caution over speed</li>
</ul>
<h2>Formatting Rules</h2>
<p>Never write dense wall-of-text notes if the content is more than a few lines.</p>
<p>Use clean spacing:</p>
<ul>
<li>A clear title</li>
<li>Short intro if needed</li>
<li>Blank lines between paragraphs</li>
<li>Bullets for lists</li>
<li>Headings for sections</li>
</ul>

<pre><code class="language-bash">osascript -e &#x27;tell application &quot;Notes&quot; to get name of every folder&#x27;
</code></pre>
<h3>Read a note body</h3>
<pre><code class="language-bash">osascript -e &#x27;tell application &quot;Notes&quot; to get body of note &quot;Apple Notes Skill Reference&quot; of first folder&#x27;
</code></pre>

</code></pre>

<h3>Append content carefully</h3>
<p>Default pattern:</p>
<ul>
<li>Read the existing body first</li>
<li>Preserve structure</li>
<li>Append a new section instead of overwriting the entire note unless explicitly requested</li>
</ul>
<h2>Error Handling</h2>
<p>When a Notes action fails:</p>
<ul>
<li>Check whether Automation permission is blocked</li>
<li>Check whether the folder exists</li>
<li>Check whether the note title is ambiguous</li>
<li>Retry the same validated method before inventing a new one</li>
</ul>
<p>If a destination folder does not exist:</p>
<ul>
<li>Create it first</li>
<li>Then create or move the note</li>
</ul>
<p>If multiple notes share the same title:</p>
<ul>
<li>Narrow by folder or related keywords before editing or moving</li>
</ul>
<h2>Limitations</h2>
<p>Keep these in mind:</p>
<ul>
<li>Apple Notes access depends on macOS Automation permission</li>
<li>Rich text body may be returned as HTML-like content</li>
<li>Reading may expose HTML tags such as <code>&lt;div&gt;</code> or <code>&lt;br&gt;</code></li>
<li>Writing should use Apple Notes friendly structure and line breaks</li>
<li>Apple Notes organization is simpler and flatter than full document systems</li>
</ul>
<p>When reading note bodies:</p>
<ul>
<li>Be aware that the raw body may include HTML-ish markup</li>
<li>Normalize the content mentally before interpreting structure</li>
</ul>
<p>When writing note bodies:</p>
<ul>
<li>Use headings, paragraphs, and explicit spacing</li>
<li>Do not dump one giant block of text</li>
</ul>
