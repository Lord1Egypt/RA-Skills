## 品蜜规则（自省回路）

蜂巢酿完蜜后自动执行的自检。触发修正时自动记录，形成新的进化规则。

### rule_001: 弱模型霸榜检测

```yaml
trigger: capability < 0.70 的模型进入综合排名 Top 3
diagnosis: 性价比公式对低价弱模型过度偏袒
fix:
  - capability 做指数惩罚（capability² 做乘数）
  - 设能力门槛（综合排名要求 capability ≥ 0.70）
source: glm-4-flash 事件 2026-05-13
confidence: 0.95
status: active
```

**原理**：¥0.1/M 的 0.68 能力模型，在旧公式下 `0.68 × log10(10000) = 2.72`，碾压 ¥1/M 的 0.84 能力模型 `0.84 × log10(769) = 2.42`。这不是性价比，是垃圾便宜。用 capability² 后，`0.68² × 4.0 = 1.85` vs `0.84² × 2.89 = 2.04`，合理。

### rule_002: 超低价对数饱和

```yaml
trigger: 加权价格 < ¥0.5/M 的模型性价比分异常高
diagnosis: 对数缩放在极端低价区仍有过大梯度
fix: 使用 capability² 做乘数后已缓解，暂无额外 fix
source: 规则 001 的衍生分析
confidence: 0.80
status: superseded_by_rule_001
```

### rule_003: 厂商花期衰减

```yaml
trigger: 连续 7 天无新信号的厂商 bloom_score 仍 > 0.5

```

## 防御性架构原则

低能模型理解蜂巢范式时可能产生幻觉。防御方法：**把需要"懂"的事变成"照做"的事**。

### 原则 1：规则执行归代码，感知判断归模型

| 层次 | 硬编码（不依赖模型） | 模型理解（可降级） |
|------|---------------------|-------------------|
| bloom 阈值 | `≥0.7 盛开 / ≥0.4 开花 / ≥0.2 含苞` | 新闻语义分析 |
| 采蜜策略 | bloom 值直接决定采蜜深度 | 计费页解析 |
| 性价比公式 | `capability² × log10(1000/price)` | 无需理解 |
| 厂商识别 | 别名表 + 正则 | 新厂商发现 |
| 异常检测 | 品蜜规则代码化 | 复杂矛盾发现 |

### 原则 2：高能品蜜，低能执行

```
日常运行（99% 时间）：低能模型执行规则 → 成本低
品蜜自检（每天 1 次）：高能模型审查结论 → 质量保证
发现矛盾 → 高能模型归纳规则 → 固化到本文件 → 低能模型照做
```

### 原则 3：结构化记忆，不用自然语言

低能模型读"百度因为文心5.1发布而开花"可能误解。读下面这个不会：

```json
{"vendor": "百度", "delta": +0.30, "reason_code": "PRODUCT_LAUNCH", "detail": "ernie-5.1"}
```

枚举值 > 自然语言。代码 > prompt。
