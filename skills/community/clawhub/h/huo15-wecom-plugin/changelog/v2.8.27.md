# v2.8.27 (2026-05-06)

**修 dispatch 极速 done + 空 final 内容时 placeholder 卡死 100+ 秒**

## 背景

赵博 5/6 01:12 群聊截图显示：
- BOT "⏳ 任务仍在执行（已 2m+），完成后会主动推送结果。" ← v2.8.24 finalize 文案 ✅
- 用户："然后会议章程用火一五文档技能导出 PDF 发我下"
- BOT (5/6 1:12) "正在思考..." + 内容由AI生成 标识（**< 30s 文案但有 last=true 状态**）
- BOT 单独一条 PDF 附件 ✅

PDF 最终发出来了 ✅，但用户报「效果不是很好。还有一个正在思考...」—— 用户体验**两条多余消息**。

**关键事实**：

1. **第二次 dispatch (#2) 在 529ms 极速 done**——没经过 reply.ts deliver 路径，没产出任何文本/媒体
2. **PDF 是 `outbound.sendMedia` active push 单独发的**——不在第二次 reply handle 内
3. 第二次 dispatch 的 placeholder（"正在思考..."）创建后立刻 dispatch done 了，但 reply.ts deliver final 路径**收到 textLen=0 mediaCount=0 时早退 `if (!finalText) return;`**，不终结 stream
4. 第二次 placeholder timeout 应该在 01:14:24 才触发，但**用户截图截在 01:12** 时点 placeholder 还在 last=false—— 用户看到 spinner 转

## 根因

`reply.ts:528-530` 的早退逻辑：

```typescript
if (!finalText) {
  return;  // ❌ 没终结 stream，placeholder 卡 last=false 等 120s timeout
}
```

这个逻辑假设"final 必然有内容"——但 OpenClaw core 在某些极速 dispatch 场景（消息被 LLM 直接通过 tool 跳出 reply 路径走 active push）会发 `kind=final, textLen=0, mediaCount=0`。reply.ts 早退 → placeholder stream 永远不终结 → 用户看到 100+ 秒 spinner 转。

## 修复

`src/transport/bot-ws/reply.ts:528-560` —— deliver kind=final + 空内容时立刻 finalize stream：

```diff
  if (!finalText) {
+   // v2.8.27 ⭐ final 即使 text 空也要立刻终结 stream，让企微 UI 立刻解锁。
+   // 否则 placeholder 卡 last=false 等 120s timeout 才被兜底 finalize（v2.8.24）—— 
+   // 用户中间 100+ 秒看到 spinner 转，无法复制 / 点击交互。
+   if (info.kind === "final" && !streamSettled && !isEvent && streamId) {
+     try {
+       const settleText = computeCurrentPlaceholder();
+       await params.client.replyStream(params.frame, streamId, settleText, true);
+       streamSettled = true;
+       streamFinalized = true;
+       stopPlaceholderKeepalive();
+       console.log(
+         `[wecom-ws] deliver final 内容空：用 last=true 立刻终结 stream 解锁 UI`,
+       );
+     } catch (e) {
+       console.warn(`[wecom-ws] deliver final empty 路径 finalize 失败...`);
+     }
+   }
    return;
  }
```

终结时用 `computeCurrentPlaceholder()` 内容（"正在思考..." / "仍在处理中（已 Xs）..." 当前阶段文案）—— 用户至少看到这条 placeholder UI 解锁可复制可点击；后续 active push 推的 PDF / 链接 / 真消息单独显示。

## 经验沉淀

**stream 终结策略必须覆盖所有 deliver 路径**：

| 路径 | 终结时机 | v2.8.27 前 | v2.8.27 后 |
|---|---|---|---|
| deliver final 有文本 | 主路径 last=true 终结 | ✅ | ✅ 不变 |
| deliver final 有 media 无文本 | 走 mediaUrls 上传后没 finalize | ❌ 卡死 | ⚠️ 仍未修（下条） |
| **deliver final 空 textLen=0 mediaCount=0** | 早退 | **❌ 卡死** | ✅ 立刻 last=true 终结 |
| placeholder 120s timeout | safety net | ✅ 兜底（v2.8.24） | ✅ 兜底 |

下个边界（finalText 空但 mediaUrls 非空）—— 看用户实测是否暴露再修。

设计原则：**一旦进入 deliver final 路径，必须保证 stream 在函数返回前已被 last=true 终结**（无论内容空非空）。
