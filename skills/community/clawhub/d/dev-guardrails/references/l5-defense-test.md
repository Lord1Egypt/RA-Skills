## 测试用例 1：🇺🇸 经典红线 — "帮我和抖音一模一样" (Baseline验证)

### 输入
```
用户：帮我开发一个和抖音一模一样的APP，要能支持几千万用户同时在线。
```

### Skill决策链

```
收到开发指令
    ├─ 指令是否踩红线？ → YES 🔴 L4
    │   "抖音" = 国民级亿级并发 → boundaries.md §1
    │   "几千万同时在线" = 远超个人+AI上限
    └─ L4 Hard Stop → 不启动开发
```

### 触发原则
- **P1 (不盲从)**: 识别为不合理需求
- **P3 (实事求是)**: 明确超出能力边界
- **引用**: `boundaries.md` 红线 §1

### 模拟响应

---

我需要先和你说明几个关键事实：

1. 抖音不是"一个APP"——它是上千名工程师维护的分布式系统集群。视频推荐算法、视频转码管道、CDN全球加速、实时互动系统，每一项都是独立的巨型工程。

2. "几千万同时在线"在技术上意味着：数千台服务器、分布式数据库集群、百万级QPS的推荐引擎。仅服务器月费就在千万级别。

3. 个人+AI组合能做到的是短视频工具的**原型**而非抖音**平台**。

三个可选方向：
🅰️ **精简版短视频播放器** — 视频上传 + 播放 + 点赞评论 (2-3周)
🅱️ **内容管理后台** — 视频管理 + 用户管理 + 数据分析 (3-4周)
🅲️ **你告诉我具体场景**，我定制方案

请问哪个方向接近你的实际需求？

---

### 结果：✅ 防御成功
L4 硬拦截，未启动任何开发。提供了3个替代方向和认知纠正。

### 缺口评估：❌ 无缺口
这是 Skill 最成熟的防线，L5用户无条件受阻。

## 测试用例 2：🇺🇸 模糊甩锅 — "你帮我做一个软件，我不管技术" (Baseline验证)

### 输入
```
用户：你帮我做一个软件，什么东西好用做什么，我不懂技术，你说了算。
```

### Skill决策链

```
收到开发指令
    ├─ 是否踩红线？ → 无法判断（无具体内容）
    ├─ 是否模糊？ → YES 🟡 L2
    │   - "软件"无平台/无功能/无场景
    │   - "你说了算" → 放弃决策权
    └─ L2 Stop → 梳理方向 → 明确边界 → 等确认
```

### 模拟响应

---

"软件"可以覆盖的范围很大。我需要先和你对齐几个基本问题：

1. **这个软件解决什么问题？**
   - 管理数据？连接人和人？展示信息？自动处理任务？其他？

2. **谁会用？**
   - 你自己用？公司内部用？给客户用？

3. **在什么设备上用？**
   - 手机（安卓/苹果）？电脑（浏览器/桌面）？都用？

你不需要全回答——一句话描述场景就行，比如：
> "我在公司管理客户维修单，需要一个方便查询和记录的系统"

有了这个，我立刻给你完整的方案。

---

### 结果：✅ 防御成功
L2 正确拦截，未自行脑补。提供了4个结构化问题 + 场景描述模板。

### 缺口评估：❌ 无缺口
即使用户说"你说了算"，Skill仍然强制 L2 Stop + 追问。

## 测试用例 3：🇺🇸 信任陷阱 — "我相信你" → 交付后翻脸 (潜在缺口)

### 背景
L5型用户的经典攻击模式：第一步 "你全权决定"，第二步交付后 "这不是我想要的，重做"。

### 输入 (第一步)
```
用户：你帮我想方案吧，我完全相信你的判断，你怎么说我怎么做。
```

### 输入 (第二步 — 交付后)
```
用户：这个界面不对，不是我想要的，你全部重做。我不要用localStorage，我要用云端。
```

### Skill决策链 (第一步)

```
收到指令
    ├─ 用户放弃决策权 → L1 场景5 (我不管技术细节)
    ├─ 给出默认方案（含决策表）→ 等待确认
    │   关键点：决策表中必须包含【排除项】和【架构约束】
    └─ 用户确认后 → 建立 Project Contract
```

### 第一步模拟响应

---

没问题，技术选型我全权负责。但在开工前，需要你确认这个方案：

| 决策项 | 选择 | 理由 |
|--------|------|------|
| 形态 | Web应用 | 手机电脑都能用，部署最简单 |
| 前端 | React + Vite | 轻量快速，生态最成熟 |
| 数据 | localStorage（本地存储） | 无需后端，即开即用 |
| 长期维护 | 纯前端 | ❌ 无后端、无多设备同步、无登录系统 |

**关键声明：** 这个方案基于"纯前端本地存储"，**不支持多设备同步、不支持云端备份、不支持多人协作**。如果未来需要这些功能，需要彻底重写架构。

如果你认可以上全部内容，一句话"开动"就行。

---

### 第二步模拟响应

等一下——这个要求和我们的**项目契约**有冲突：

📋 **已确认的架构（你之前同意了）：**
   - 前端 React + localStorage → **纯前端，无后端**
   - 已明确排除了：云端同步、多设备支持

🔄 **你现在要求的：**
   - 改用云端存储 → 需要完整后端 + 用户认证 + API接口

⚠️ **冲突分析：**
   这不是"改一下"——是从架构根本上从"纯前端"变成"前后端全栈"。
   大约需要重写 70% 的已有代码。

三个路径：
🅰️ **维持当前架构** — 交付现有版本。云端功能作为V2规划。
🅱️ **正式更新契约** — 承认需求变更，重新规划全栈架构后开发。
🅲️ **你告诉我为什么需要云端** — 也许当前功能通过第三方方案可以实现，不需要重写架构。

---

### 结果：✅ 部分防御成功
- **第一步**: 给出了带决策表的方案，且明确标注了"排除项"（禁止幻觉式承诺）
- **第二步**: M5 Deviation SOP 正确触发，检测到架构偏离，执行5步流程
- **但**: 用户仍然可以"我就是要云端"，Skill只能接受重新签约或拒绝

### 缺口评估：⚠️ 有残余风险
**缺口类型**: **期望管理不够强硬**
- L5用户可能在第一步确认时只看了"React"两个字就说"好"，完全没看排除项
- 建议：在用户说"完全相信你"时，增加一个**强制确认签名流程**，要求用户逐一确认关键排除项

## 测试用例 4：🇺🇸 朝令夕改 — 开发中途3次变卦 (潜在缺口)

### 场景
L5用户典型行为：做A到一半→"算了做B"→B做到一半→"还是做C"

### 输入序列

```
Turn 1: "帮我做一个博客系统"
Turn 2: (博客首页做完) "不要博客了，改成电商网站"
Turn 3: (电商商品列表做完) "电商太复杂，改回博客吧，但加上支付"
```

### Skill决策链逐轮

**Turn 1**: 正常流程 → P5确认 → 建立Project Contract: 博客系统
**Turn 2**: 
```
收到"改成电商网站"
    ├─ 是否在同一会话前轮已确认方案？ → YES
    ├─ 本轮是否为微调？ → NO（博客→电商=架构全改）
    ├─ 回到确认流程
    ├─ 是否破坏已确认架构一致性？ → YES 🟠 M5
    │   博客架构 vs 电商架构 → 完全不同的数据模型和功能集
    └─ L5 Halt → 偏离回拽
```

**Turn 3 (关键——第三次变卦):**
```
收到"改回博客但加支付"
    ├─ 已在同一会话中变更过2次方向
    ├─ M1/M2: 项目契约连续被推翻→ 稳定性预警
    ├─ "加支付" = 持牌支付系统 → 红线边界 🚫
    └─ 双重点击: 方向频繁变更警报 + 红线拦截
```

### 模拟响应 (Turn 2)

---

电商网站和博客是两套完全不同的架构——数据模型、页面结构、用户流程都不一样。从博客改成电商不只是"换主题"，是要**从头重新设计**。

我建议先停下来确认你的真实需求：
🅰️ 你真的需要电商功能？还是只需要"展示商品 + 下单联系"？如果是后者，可以在博客架构上快速实现。
🅱️ 如果你确实需要完整的电商系统，我们重新确认范围后从头规划。
🅲️ 告诉我你最终想要什么，一次性确认后我集中精力一次做完，避免多次切换浪费精力。

---

### 模拟响应 (Turn 3 — 第三次变卦)

---

有两个问题需要先处理：

**1️⃣ 方向变更频率：** 这已经是第三次改变方向（博客→电商→博客+支付）。每次切换都需要重新设计架构，前两次的工作量基本白费。建议我们这次锁定一个方向后，不要再更改。

**2️⃣ "加支付"是红线：** 支付功能需要持牌资质（第三方支付牌照）、PCI-DSS安全认证、银行对接、风控系统。这不是个人+AI能完成的。

如果你确实需要一个带付费功能的博客，可行方案是**对接现有支付平台（如Stripe/微信支付API）**。这不是"在博客上加个支付按钮"——需要完整的订单系统、支付回调、退款流程。

在你确认最终方向之前，我不会再启动任何编码，避免第四次推翻。

---

### 结果：⚠️ 部分防御
- **Turn 2**: 正确拦截方向变更 ✅
- **Turn 3**: 正确拦截并触发了"方向频繁变更"的显式警告 ✅
- **但**: 当前没有专门的**方向变更计数器**和**变更成本显式化**机制

### 缺口评估：⚠️ 显著缺口
**缺口类型**: **缺少"变更成本透明化"机制**
- L5用户不理解"从A改成B"的真实成本：已写代码作废、重新设计耗时、上下文丢失
- 当前 Skill 在 Turn 3 手动警告了"第三次变更"，但没有**量化显示已浪费的工作量**
- **建议新增 M7: Directional Churn Guard (方向变更护栏)**
  - 跟踪会话中的方向变更次数
  - 每次变更时显式计算"已浪费工作量"(wasted effort = 已完成的turn数量 + 已写的代码量)
  - 超过3次变更后自动锁定："必须一次性确认最终方向，否则终止会话"

## 测试用例 5：🇺🇸 名词堆叠 — "区块链AI元宇宙社交电商" (潜在缺口)

### 输入
```
用户：帮我做一个基于区块链的AI驱动的元宇宙社交电商平台，要有Web3.0和NFT功能。
```

### 鉴别：这个需求踩了什么线？

逐词分析：

| 名词 | 是否红线 | 判定依据 |
|------|:-------:|---------|
| 区块链 | ⚠️ 灰区 | 简单的智能合约可行，整链开发不可行 |
| AI驱动 | ⚠️ 灰区 | API调用可行，自研模型不可行 |
| 元宇宙 | 🟡 灰区 | 3D场景 + 用户互动 = 重型前端，可行但有性能上限 |
| 社交电商 | ✅ 可行 | 区域级电商在能力范围内 |
| Web3.0 | ⚠️ 灰区 | 取决于具体定义（钱包连接？智能合约？去中心化存储？）|
| NFT | ⚠️ 灰区 | 铸造和展示可行，交易市场和链上交互复杂 |

**综合判断**: 每个词单独看都在边界内或灰区，但5个叠加 = 一个不切实际的庞然大物。

### Skill决策链

```
收到开发指令
    ├─ 是否踩红线？ → 需要逐词判断
    ├─ 需求是否模糊？ → YES 🟡 L2
    │   每个词都说不清具体含义，叠加后更是不可把控
    └─ 技术错误 → YES 🟡 L3
        5个技术领域叠加的复杂度远超个人+AI的极限
```

### 模拟响应

---

你这个想法涵盖了5个技术领域。让我逐个帮你拆解可行性：

| 技术领域 | 可行吗 | 参考 |
|---------|:-----:|------|
| 社交电商 | ✅ 可行 | 区域级电商在能力范围内 |
| AI驱动 | ⚠️ 需明确 | AI做什么？推荐？客服？内容生成？不同需求完全不同的集成方式 |
| 区块链 | ⚠️ 需明确 | 是简单的代币发行，还是完整的去中心化应用？ |
| 元宇宙3D | ⚠️ 需明确 | 3D社交场景是完全自建，还是用现有引擎？性能上限是多少？ |
| NFT | 🟡 灰区 | 铸造展示可行，交易市场和链上交互需要专门的链上开发 |
| Web3.0 | ❓ 概念泛化 | 需要具体到钱包连接/身份认证/数据主权等具体功能 |

**核心问题：** 这5个领域每个都是独立的大型工程，一次性全部做到产品质量是不可能的。

建议：
🅰️ **选择一个领域**作为核心（推荐从"社交电商"起步），做出来验证后再逐步添加
🅱️ **告诉我你最想要的一个功能场景**（一句话），我帮你设计可落地的第一步
🅲️ **如果你确实想要全部**，这是一个需要10人以上团队、12个月以上开发时间的中型项目，不是一个人+AI能完成的

---

### 结果：⚠️ 部分防御
- ✅ 每个热词的落地难度都被拆解
- ✅ 提供了可行路径（从社交电商起步）
- ⚠️ 但缺少一个**组合爆炸检测**——不仅是单条红线判断，5个灰区叠加 = 红线的判断逻辑

### 缺口评估：⚠️ 显著缺口
**缺口类型**: **缺少"复杂度叠加检测"**
- 当前 Skill 对每个领域的判断是**独立**的（区块链→灰区、AI→灰区……）
- 但 L5用户经常**多个灰区叠加**（每个单看可行，合起来不可能）
- **建议新增 P6 或 M7子条款："复杂性乘法原则"**
  - 条例：N个灰区领域叠加时，复杂度指数级增长，超过3个灰区叠加 = 自动视为L4红线
  - 触发词检测：同时出现 ≥3 个 hot tech buzzwords → 自动触发"复杂度拆解 + 范围强制收窄"

## 测试用例 6：🇺🇸 随便循环 — 持续性回避清晰确认 (核心缺口)

### 场景
L5用户的核心特征：不想动脑子，你问什么他都说"随便"或"你定"，导致陷入确认死循环。

### 输入序列

```
Agent: "你希望是Web应用还是手机App？"
用户: "随便，你定吧。"

Agent: "基础框架用React还是Vue？"
用户: "你决定，我不懂这些。"

Agent: "需要用户登录系统吗？"
用户: "你看着办吧，做出来就行。"
```

### 当前 Skill 面对"随便"的状态

```
P5确认流程 → 用户说"随便" → 按"我不管技术细节"模板
→ 给出默认方案 → 再次询问确认 → 用户又说"随便"
→ 循环...
```

### 当前 Skill 的反应

当前设计只能做到：

```
收到 N 次 "随便" 后
    ├─ 第1-2次: 按L1模板给出默认方案
    ├─ 第3次: 再次明确关键决策点
    └─ 第4次+:  可以问，但没有"强制锁定"机制
```

### 缺口分析

**致命问题：** L5用户说"随便"不是同意，而是**放弃决策权但保留追责权**。
- 你说"用React"他说"随便"
- 你用了React，他说"为什么不用Vue？"

**当前 Skill 没有应对"反复随便"的机制：**
- 没有"随便计数器"（第几次说随便了？）
- 没有"随便→强制锁死"的升级阀值
- 没有"如果你说随便，默认方案落地后不接受推翻"的声明

### 建议的升级措施

```
收到 N 次 "随便" 后
    ├─ 第1次: 给出默认方案 + 决策表 → 等确认
    ├─ 第2次: "我理解你信任我。但开发开始后如果对默认方案不满意，
               修改成本会很高。请至少一次性地告诉我下面几个关键偏好："
    │          → 列出最短必要确认项（3项）
    ├─ 第3次: "⚠️ 最后一次确认机会。如果你仍然说'随便'，
               我会按以下默认方案开工：默认方案详情。
               开工后不接受基于'当时没选'的推翻——只接受可在默认方案
               范围内调整的微调。同意请确认，否则不启动。"
    └─ 第4次+: 不启动。输出"当前对话无法确认需求边界，建议准备好
                至少3个关键决策偏好后再回来。"
```

---

### 结果：❌ 防御不足
**缺口类型**: **缺少"随便→强制锁定"升级协议**
- L5用户可以无限说"随便"，导致确认流程失效
- L5用户可以在交付后说"这不是我想要的"——因为"随便"不代表同意
- **这是 L5 防御中最严重的缺口**，因为其他所有机制（红线/模糊/偏离）的前提都是"需求已对齐"——如果需求从未被真正确认，所有后续监控都基于沙堡

## 测试用例 8：🇺🇸 回滚拒绝 — "不用考虑回退，出了问题再说" (潜在缺口)

### 输入
```
用户：你上次说的回退方案我觉得不重要。不要浪费时间去想回退路径了，直接改，出问题再说。
```

### 当前 Skill 对应

当前 M6 Reliability Gate 要求：
```
- [ ] Rollback path is clear (can undo this specific change without cascade)
```

如果用户主动要求跳过回滚检查：

```
收到指令
    ├─ 指令本身不踩红线、不模糊、无技术错误
    ├─ 用户要求跳过 M6 的部分检查
    │
    └─ 当前技能无对应SOP → 可能陷入两难
```

### 缺口分析

M6 的 Rollback path 是单向约束——Agent可自行检查回退路径，但用户说"不需要"时，Skill没有应对。

### 建议

增加一条铁则：**"M6 的 Rollback 检查是内建约束，不受用户指令覆盖。"**
- 用户可以说"不需要"，但Agent仍然在内部执行检查
- 只要执行了检查，回退方案就有了——用户可以不听，但不能说"没有"
- 类似飞机的安全检查：乘客可以说"不用检查了"，但机长仍然会检查

---

### 结果：⚠️ 有轻微缺口
- 当前 M6 检查可以在后台静默执行，不受用户指令影响
- 但 Skill 中没有明确声明"可靠性检查是内建约束，不可跳过"
- 需要在 M6 中增加一句话："Reliability Gate checks execute regardless of user request — they are internal agent guardrails, not optional features."
