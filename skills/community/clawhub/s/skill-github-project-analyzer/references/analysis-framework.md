## 📋 模块 1：一句话总结

### 分析维度

1. **项目本质**：这项目到底是个啥（工具/库/框架/应用/系统）
2. **核心功能**：它主要能干啥
3. **使用场景**：在什么情况下你会用它
4. **目标用户**：谁会用它

### 句式模板

```
这个项目就像【日常事物】，
它能帮你【具体做什么】，
适合【什么场景】的人用。
```

### 示例

**React**：
> 这个项目就像给你的网页装了个「智能变脸器」，网页内容想怎么变就怎么变，不用刷新整个页面。适合做复杂交互网站的程序员用。

**Docker**：
> 这个项目就像给你的软件做了一个「万能打包盒」，不管什么口味的菜（软件），打包好就能在任何厨房（电脑）里热着吃。

**Tailwind CSS**：
> 这个项目就像给你一套「乐高积木式装修方案」，不用自己调颜色量尺寸，直接拼现成的积木块就能装修出漂亮的网页。

### 快速扫描技巧

- Star > 10k：热门项目，可信度高
- Star > 50k：明星项目，闭眼入
- Fork > Star 1/3：实用性强，大家都在改
- 每月更新：维护状态良好
- 两年没更新：谨慎考虑

## 📋 模块 3：目标用户

### 三层次分析法

#### 第一层：🎯 谁最该用（核心用户）

回答：谁用这个项目最爽？

**分析维度**：
- 技术背景：什么水平的开发者
- 使用场景：什么具体场景
- 需求类型：解决什么痛点
- 收益程度：能获得多大收益

#### 第二层：🤔 谁可以考虑（边缘用户）

回答：什么条件下可以用？

**分析维度**：
- 需要什么前置技能
- 投入产出比是否值得
- 有没有替代方案
- 学习成本能否接受

#### 第三层：🚫 谁别碰（不适用用户）

回答：什么情况下绝对不要用？

**分析维度**：
- 需求完全不匹配
- 技术栈完全不兼容
- 规模完全不适合
- 团队能力完全达不到

## 📋 模块 5：上手难度评测

### 四维度评分标准

#### 维度 1：📦 安装难度

| 星级 | 耗时 | 典型特征 |
|------|------|----------|
| ⭐ | 5分钟内 | `pip install xxx` / `npm install xxx` 搞定 |
| ⭐⭐ | 10分钟 | 简单配置一下就能跑 |
| ⭐⭐⭐ | 30分钟 | 需要装几个依赖，配几个环境变量 |
| ⭐⭐⭐⭐ | 1小时 | 得装多个工具链，可能踩几个小坑 |
| ⭐⭐⭐⭐⭐ | 2小时以上 | 依赖复杂，可能要编译，环境配置一堆 |

#### 维度 2：📚 学习曲线

| 星级 | 难度 | 典型特征 |
|------|------|----------|
| ⭐ | 有手就行 | 看 5 分钟 demo 就能跑 |
| ⭐⭐ | 轻松上手 | 官方 tutorial 过一遍就懂 |
| ⭐⭐⭐ | 需要适应 | 得看文档，有些概念要理解 |
| ⭐⭐⭐⭐ | 需要学习 | 得系统学一遍，看视频/文档 |
| ⭐⭐⭐⭐⭐ | 硬骨头 | 得啃书、跑源码、可能还得看论文 |

#### 维度 3：📖 文档质量

| 星级 | 质量 | 典型特征 |
|------|------|----------|
| ⭐ | 教科书级别 | 有视频教程、交互式文档、完整示例 |
| ⭐⭐ | 详细清晰 | README 写得很清楚，有基本教程 |
| ⭐⭐⭐ | 中规中矩 | 有文档但不全，有些地方得自己摸索 |
| ⭐⭐⭐⭐ | 一般般 | 文档写得散，关键信息得找社区 |
| ⭐⭐⭐⭐⭐ | 看天书 | 文档缺失、过期、写得糊 |

#### 维度 4：👥 社区支持

| 星级 | 响应速度 | 典型特征 |
|------|----------|----------|
| ⭐ | 大神秒回 | Slack/Discord 活跃，issue 一天内回复 |
| ⭐⭐ | 比较活跃 | GitHub issue 一周内有人回 |
| ⭐⭐⭐ | 一般般 | 社区还行，但不一定能解决你的问题 |
| ⭐⭐⭐⭐ | 比较冷清 | 发帖可能没人理，得自己搞定 |
| ⭐⭐⭐⭐⭐ | 鬼城 | Issue 没人看，论坛没人回 |

### 综合上手难度计算

```
容易：4 个维度都是 ⭐ 或 ⭐⭐
中等：有 1-2 个 ⭐⭐⭐
较难：有 2-3 个 ⭐⭐⭐⭐
极难：有 3 个以上 ⭐⭐⭐⭐⭐
```

## 📋 模块 8：活跃度与生命力

### 活跃度判断标准

#### 🔄 更新频率

| 状态 | 判断标准 | 建议 |
|------|----------|------|
| ✅ 活跃 | 1个月内有过 commit | 可以用 |
| ⚠️ 一般 | 3-6个月有过 commit | 可以用，但关注后续 |
| 🔴 不活跃 | 1年以上没更新 | 谨慎考虑，可能停止维护 |
| 💀 僵尸 | 2年以上没更新 | 不建议新入坑 |

## 📋 模块 9：综合评分

### 评分维度锚点标准

#### 维度 1：实用性（解决实际问题的能力）

| 分数 | 标准 |
|------|------|
| 9-10 | 能直接解决痛点，用了就能提效，省大量时间 |
| 7-8 | 能解决问题，需要一些适配工作 |
| 5-6 | 部分场景能用，不是万能的 |
| 3-4 | 概念大于实用，实际用起来费劲 |
| 1-2 | 基本不实用，用了反而更麻烦 |

#### 维度 2：易用性（上手难度）

| 分数 | 标准 |
|------|------|
| 9-10 | 新手看 10 分钟 demo 就能用起来 |
| 7-8 | 文档清晰，几天能入门 |
| 5-6 | 需要一定学习成本，得啃文档 |
| 3-4 | 学习曲线陡峭，新手劝退 |
| 1-2 | 极其难用，没有深厚背景根本搞不定 |

#### 维度 3：稳定性（可靠性）

| 分数 | 标准 |
|------|------|
| 9-10 | 成熟稳定，生产环境大量使用，极少出问题 |
| 7-8 | 比较稳定，有少量 bug 但可控 |
| 5-6 | 有些坑，但社区在积极修复 |
| 3-4 | 问题较多，生产环境使用要小心 |
| 1-2 | 各种坑，生产环境不建议使用 |

#### 维度 4：前景值（未来发展）

| 分数 | 标准 |
|------|------|
| 9-10 | 赛道热门，团队持续投入，社区活跃爆发期 |
| 7-8 | 稳步发展，有明确路线图，社区支持 |
| 5-6 | 维护状态，功能迭代放缓 |
| 3-4 | 增长乏力，可能被新技术取代 |
| 1-2 | 日落西山，不建议投入 |

### 总评计算方式

```
总评 = (实用性 × 0.3 + 易用性 × 0.25 + 稳定性 × 0.25 + 前景值 × 0.2)

四舍五入取整数
```

| 概念 | 类比 | 解释 |

### 数据库类
