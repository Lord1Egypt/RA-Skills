# 会话模式 — 按学习类型选会话形状

## 用法

诊断结束后、进入里程碑循环之前，从下表先选模式，再按模式条目执行。

## 模式选择表（先用这张）

学习者描述里出现 → | 选模式
---|---
"我不太*理解*为什么…" / "总是迷糊它在干嘛" | **M1 概念密集型**
"我能读懂但写不出来" / "想会用 X 做东西" | **M2 技能密集型**
"下周工作要上手 X" / "在我们代码库里…" | **M3 工具/系统型**
"我老是选错方法" / "不知道什么时候该用 A 还是 B" | **M4 判断力型**
"我知道该这么做，就是做不到" / "想养成 X 习惯" | **M5 思维/习惯型**

混合时按"忽略它损失最大"的那个选主导，其它折叠进去。常见组合：

主导 | 常见折叠
---|---
M2 技能 | 概念在构建需要时即时教，不前置理论
M3 工具 | 技能在真实任务中反复练，不做全景游览
M4 判断 | 概念多为复习，时间花在案例上
M5 思维 | 技能很容易，工作全在练习轮

---

## M1 · 概念密集型（理解*为什么*）

**适用：** 统计推断、设计原则、一致性模型、乐理、递归、经济学推理。

**必做动作：**
1. 诊断重 Q4 微探测，不依赖学习者声称的水平。
2. 围绕**不变量 + 扰动**组织：先定一个简单情况让学习者推理 → 每轮换一个变量 → 让他们预测变化。
3. 主用 Hypothesize（揭示前先猜）+ Challenge（答对后追问）。
4. 微任务必须是**预测**或**解释**，不是"构建"。

**出场产物：**
- 学习者手绘的概念图，或
- 一个带学习者注释的 worked example，或
- 一段用学习者自己话写的 ≤200 字解释。

**必避：**
- 在自然边界处不停下，硬塞"再多解释一个"。
- 用术语列表代替不变量。
- 接受"我懂了"而没让学习者预测一次。

---

## M2 · 技能密集型（构建*怎么做*）

**适用：** 编程语言、框架（React/Vue）、命令行工具（Git）、乐器、盲打。

**必做动作：**
1. 诊断方式：让学习者**直接做一个你估计他能做的小任务**，看产物，不靠口述。
2. 每轮包含一个**微构建**——敲、运行、改的具体动作。学习者动手 ≥70%，AI 解释 ≤30%。
3. 每个微构建只引入一个新元素，禁止"两个小东西打包"。
4. Verify 在做的过程中完成：能做到 = 此刻理解够用。
5. 每 ~3 个微构建后强制一轮 Reflect："你看到什么模式重复出现了？"

**出场产物：** 一个可运行物——能跑的程序 / 完成的曲段 / 可 push 的 diff。

**必避：**
- 让学习者长时间只听不做。
- 跳过 Reflect，让战术不沉淀为原则。
- 在他能跑通之前堆叠下一个新元素。

---

## M3 · 工具/系统学习（导航*存在什么*）

**适用：** 特定代码库、Figma/Blender、某个 API、公司内部平台。

**必做动作：**
1. 诊断必须挂在**他的真实任务**上。没有任务则在开始前共同设计一个最小现实任务。
2. 即时教学：任务碰到一个概念才引入，没碰到不讲。
3. 开场画一句**地形图**："系统主要 N 块，今天碰其中 M 块，其余仅标注存在。"
4. Challenge 用法变形：用"如果改成 X 会怎样"探测**工具行为**，而非学习者推理。

**出场产物：**
- 任务完成的具体产物 + 学习者亲手记的"我注意到的坑"清单（≤10 条）。

**必避：**
- 做"全景游览"展示功能。
- 偏离任务去解释还没遇到的功能。
- 替学习者写"坑清单"——必须他自己写。

---

## M4 · 问题解决学习（培养*判断力*）

**适用：** 算法设计、调试、架构选型、医学诊断、棋类战术、UX 评审。

**必做动作：**
1. 诊断方式：给一个示例问题，让学习者**叙述自己的方法**。方法是数据，最终答案不重要。
2. 准备 3–5 个**逐渐微妙的案例**，每个考察判断力的不同轴。
3. 重度 Challenge：每个案例至少一次魔鬼代言人追问。
4. 每个案例后必做 Reflect："你刚才用了什么启发式规则？"——让隐性规则显化。

**出场产物：**
- 一份学习者亲手写的"决策启发式"清单：3–7 条，每条配一句理由 + 一个边界条件（在哪里不再适用）。

**必避：**
- 直接递规则给学习者（会被遗忘）。
- 在学习者表述启发式前替他总结。
- 让会话停在"答对了"而不显化方法。

---

## M5 · 思维/习惯学习（改变*运作方式*）

**适用：** 统计思维、清晰写作、防御性编码、给反馈、编码前先规划、提更好的问题。

**必做动作：**
1. 诊断方式：找一个学习者最近**用旧方式做事**的真实例子，作为本会话锚点。
2. 围绕**模式打断**组织：展示旧模式 → 展示新模式 → 在锚点例子上练新模式 → 换一个新例子再练。
3. 几乎不引入新词汇/概念——学习者已经同意建议，缺的是反射。
4. Reflect 是主工具："刚才什么让你滑回旧模式？" / "什么本可以更早抓住你？"

**出场产物：**
- 一份**个人化的触发器/检查清单**（≤5 条），由学习者自写，必须绑定到他自身的具体线索，不能是通用建议。

**必避：**
- 讲述新模式的重要性（他已经认同）。
- 给通用建议而非个人触发器。
- 没有真实锚点例子就开始"练习"。
