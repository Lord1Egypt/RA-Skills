# Transit Actionable Output 指南（v4.1.0）

> **目的**：解决 Transit 分析"输出停留在星盘数据陈述层"的问题——每条 Transit 预测必须包含用户可执行的行动方案。

---

## 一、触发条件

任何以下场景自动触发本规范（无需用户要求）：

| 触发词 | 示例 |
|--------|------|
| "被发现"、"被赏识"、"被贵人" | "2026年6月会有贵人赏识，具体怎么被发现？" |
| "什么时候"、"应期"、"时机" | "什么时候有合作机会？" |
| "合作"、"破圈"、"突破"、"升职" | "什么时候能破圈？" |
| "移民"、"搬迁"、"迁居" | "什么时候会搬到大房子？" |
| **任何涉及具体事件类型的 Transit 预测** | — |

---

## 二、三要素强制输出

每条 Transit 预测必须同时输出以下三要素，缺一不可：

### 要素1：时间段（精确到日/周/月）

```
✅ 6月2日—6月18日（峰值6月18日前后72小时）
✅ 2026年5月10日—5月20日
❌ "6月份"（精度不足）
❌ "今年内"（精度不足）
```

### 要素2：具体行动类型

| 行动类型 | 说明 | 触发信号 |
|---------|------|---------|
| **发布型** | 发布/展示已有内容 | Jupiter+Venus同宫、AmK活跃 |
| **跟进型** | 主动联系已在沟通中的人 | Mars在2/11宫、主动出击窗口 |
| **等待型** | 保持活跃但不主动 | Jupiter入12宫前的积累期 |
| **被发现型** | 已有产品/工具被人主动发现 | AmK Moon 7宫机制 |
| **推进型** | 推进已有合作/谈判 | Mercury激活、合同谈判期 |

### 要素3：置信度标注

```
✅ [A] 已验证（倒推历史事件命中）——置信度最高
✅ [B] 高概率（3+独立维度信号一致）——次高置信度
✅ [C] 推断（1-2个维度信号，理论推断）——最低置信度，需标注未验证
```

**每条关键结论必须标注置信度，未验证者必须附注未验证声明。**

---

## 三、案例检索三步法（强制）

当动态预测（被发现/合作/破圈/关键事件型）时，**必须先执行案例检索**：

### Step 1：检索同类行星配置真实案例

```bash
# 使用引擎内置名人案例库
python3 scripts/jyotish_engine.py celebrity --config "Jupiter+Moon+AmK 7宫"

# 或 WebSearch 检索真实创作者发现路径
web_search: "astrologer content creator discovered by mentor collaboration 2024 2025 case study"
```

**检索优先级**：
1. 一楠本人历史验证案例（已有数据优先）
2. Skill 内置名人案例库（`celebrity --config`）
3. WebSearch 真实案例（行业数据、创作者故事）

### Step 2：对比 D10/D9 信号一致性

- D10 3宫 Venus/Rahu（内容/传播工具）→ 确认内容发现路径
- D9 Jupiter 位置（灵魂/使命）→ 确认身份定位
- 7宫 AmK Moon（被发现机制）→ 确认合作/被动发现路径

### Step 3：整合输出带案例参照的结论

**输出格式模板**：

```
## [B] 2026年6月2日—6月18日：Jupiter入Cancer触发事业窗口
**置信度**：⭐ [B]（高概率——基于3个独立维度：Jupiter合相本命Mars+D10 3宫Venus/Rahu+Dasha Saturn/Venus支持）

**Actionable Output**：
- 具体时间段：6月2日起，峰值6月18日前后72小时
- 具体行动：已有内容/产品保持展示活跃度，留意主动联系
- 行动类型：⚠️ 被发现型（AmK Moon 7宫机制：被人使用工具后触发）
- 行动基础：基于真实案例——Starcross App（0广告月入6万，内容获客+情感变现）

**案例参照**：Starcross App；Neda Farr（TikTok 22万粉丝→工具创始人）
**未验证声明**：以上预测基于星盘信号与案例类比推断，未经个人历史事件验证，置信度[B]。
```

## 四、禁用规则

| 禁用行为 | 示例 | 正确做法 |
|---------|------|---------|
| ❌ 纯理论推断行动建议 | "保持开放心态" | "保持在线，留意主动联系（AmK 7宫机制）" |
| ❌ 无时间段行动建议 | "做好内容准备" | "5月14日前发布（Jupiter+Venus同宫窗口）" |
| ❌ 无置信度预测 | "6月肯定会被发现" | "[B] 6月2日—18日高概率被发现（3+维度）" |
| ❌ 无未验证声明 | — | 标注"以上未经历史事件验证" |
| ❌ 案例未检索就给结论 | "你会被发现" | "基于Neda Farr/Starcross案例+Jupiter合相Mars信号，你极可能被贵人发现" |

## 五、典型错误模式

### 错误1：数据陈述替代行动方案

```
❌ 错误：Jupiter 6月2日入Cancer，在1宫形成重大转变...

✅ 正确：[B] 2026年6月2日—6月18日事业窗口期（Jupiter入Cancer+合相本命Mars）
   Actionable Output：保持内容展示活跃，峰值在6月18日前后72小时
   置信度：B（3维度：Transit+Jupiter合Mars+Dasha支持）
   案例参照：Starcross App被发现路径
   未验证声明：未经历史事件验证，仅供参考
```

### 错误2：模糊时间段

```
❌ 错误：大概在6月份...
✅ 正确：6月2日—6月18日（峰值6月18日前后72小时）
```

### 错误3：缺乏具体行动

```
❌ 错误：保持开放，等待机会...
✅ 正确：已有工具/App保持展示活跃度，留意任何主动联系（AmK Moon 7宫=被动被发现机制）
```

## 六、质量自检清单

每次 Transit 分析输出前，检查：

- [ ] 时间段是否精确到日/周/月？
- [ ] 行动类型是否明确（发布/跟进/等待/被发现/推进）？
- [ ] 置信度是否标注（[A]/[B]/[C]）？
- [ ] 是否先检索了案例再给结论？
- [ ] 是否有未验证声明？
- [ ] 结论是否基于3个以上独立维度？
