# TDD：测试驱动开发

## 1. Overview（概述）

TDD 不是"先写测试"，是**用测试驱动设计**。每个功能从一个失败的测试开始（RED），写刚好让测试通过的最少代码（GREEN），然后重构消除重复（REFACTOR）。这保证每个功能有覆盖、每个覆盖有意义。

**铁律**：
- 没有 RED 的 GREEN 可能是假通过（测错了东西）
- 测试只验证 PUBLIC 行为 — 测实现细节的测试在重构时全废
- 垂直切片 > 水平切片 — 一个功能一个功能做完，不是一层一层做完

## 2. When to Use（使用条件）

**触发条件**：
- Step 3 编码阶段自动注入
- 所有 B 级和 C 级任务强制执行
- A 级任务简化版（至少保证每个函数有基本测试）

**可跳过**（需明确理由 + 记录）：
- 纯配置修改（yaml/json/toml）
- 已有完整测试覆盖的微小修改（≤5 行）
- 不可测试的外部系统适配层（需通过集成测试覆盖）

## 3. Process（执行流程）

### 3.1 RED → GREEN → REFACTOR 标准循环

```
功能需求：{功能描述}
  │
  ├─ RED（写失败测试）
  │   1. 在 test 文件中写测试
  │   2. 只测一个行为（一个 assert 概念，可以多个断言同主题）
  │   3. 命名：test_{方法}_{状态}_{期望}
  │   4. 运行 → 确认 RED（失败，且失败消息是对应功能缺失）
  │
  ├─ GREEN（最小实现）
  │   1. 写刚好让测试通过的最少代码
  │   2. 允许 hard-code、复制粘贴（REFACTOR 阶段清除）
  │   3. 运行 → 确认 GREEN（全部测试通过，包括已有测试）
  │   4. ❌ 禁止：写"未来可能需要"的代码
  │   5. ❌ 禁止：同时实现多个功能
  │
  └─ REFACTOR（消除重复，改善结构）
      1. 消除重复代码（同一概念出现 ≥2 次 → 提取）
      2. 改善命名（让意图一目了然）
      3. 简化逻辑（消除不必要的抽象和分支）
      4. 运行全部测试 → 必须保持 GREEN
      5. ❌ 禁止：添加新功能（新功能 = 新循环）
      6. ❌ 禁止：局部重构顺便改其他文件
```

### 3.2 垂直切片 vs 水平切片

```
✅ 垂直切片（正确）：
   功能 A：test → code → refactor → 完成
   功能 B：test → code → refactor → 完成
   功能 C：test → code → refactor → 完成
   每次交付一个可用的功能

❌ 水平切片（错误）：
   所有 model → 所有 service → 所有 controller → 所有测试
   没有测试覆盖所有代码是一次性写完
```

### 3.3 测试公共行为，不测实现

```
✅ 测行为：测试公共 API 的输入 → 输出
   assert add_to_cart(user, item) == CartState.ADDED

❌ 测实现：测试内部方法调用顺序
   mock_validate.assert_called_once()
   mock_save.assert_called_with(item)
   → 重构时这些测试全部失效
```

## 4. Risk Scorecard（反借口表）

| # | 纪律 | 常见借口 | 反驳 |
|---|------|---------|------|
| R1 | 必须先 RED | "我相信这测试会通过" | 没见过 RED 的测试可能测错了 |
| R2 | GREEN 最小实现 | "顺便把这个也实现了" | YAGNI — 没测试的需求 = 没需求 |
| R3 | 测试只测 Public API | "得验证 private 方法被调用了" | 测实现 = 锁死重构 = 测试变成负债 |
| R4 | 垂直切片 | "先把所有 model 写好" | 水平切片 = 最后一起测试 = 没有安全网 |
| R5 | REFACTOR 不加功能 | "这个变量名顺便改一下" | 重构 + 加功能 = 不知道测试失败是因为哪个 |

## 5. Red Flags（危险信号）

- 🚩 同一测试文件没有出现 RED → GREEN 的 commit 记录
- 🚩 测试命名含"mock"（"test_mock_called"）— 在测实现不是行为
- 🚩 一个测试覆盖 > 3 个功能 — 违反单一行为原则
- 🚩 GREEN 阶段新增 > 50 行生产代码 — 潜在过度实现
- 🚩 REFACTOR 后测试失败但生产代码逻辑没变 — 测试测了实现细节
- 🚩 所有测试一次性写完（没有 RED 记录）— 水平切片

## 6. Verification（自检清单）

**必检（≤5 项）**：
- [ ] 每个测试对应一个 RED → GREEN 循环（第一个 commit 是测试失败）
- [ ] 测试只使用公开接口（不测 private/protected 方法）
- [ ] 当前代码为测试所需的最小实现（无投机性代码）
- [ ] 没有添加未被要求的功能
- [ ] 所有测试通过（GREEN）

**抽检（≤3 项）**：
- [ ] 垂直切片（非水平：功能逐个完成，不是逐层完成）
- [ ] 测试命名描述行为而非实现（test_returns_error_when_empty 而不是 test_validates_input）
- [ ] 循环次数 ≤ 功能数 × 2（无频繁 RED→GREEN→RED 反覆）

**免检**：测试工具选择、断言风格、fixture 组织方式
