# 软件开发领域任务组件要求清单

软件开发领域每种任务类型的"零件清单"——必选/可选组件、组装顺序、领域约束。按清单逐项产出。

---

## S01 需求工程

### S01-01 用户故事编写
- **必选组件**: 用户角色定义、用户目标描述、业务价值说明、验收标准
- **可选组件**: 用户故事地图、故事优先级、依赖关系
- **组装顺序**: 角色识别→目标梳理→价值量化→验收标准定义
- **约束**: 用户故事必须遵循INVEST原则（Independent, Negotiable, Valuable, Estimable, Small, Testable）
- **格式**: 用户故事卡片（Markdown）

### S01-02 需求优先级排序
- **必选组件**: 需求列表、评估维度、排序方法选择、排序结果
- **可选组件**: 利益相关者权重、技术依赖约束
- **组装顺序**: 需求收集→维度定义→评分→排序→验证
- **约束**: 排序必须基于可量化的维度，不得凭感觉
- **格式**: 优先级排序表（Markdown表格）

### S01-03 需求规格说明书
- **必选组件**: 项目背景、功能需求列表、非功能需求列表、约束条件、假设和依赖、验收标准
- **可选组件**: 术语表、参考文档、附录
- **组装顺序**: 背景梳理→功能需求→非功能需求→约束→验收标准
- **约束**: 需求必须可测试、可追溯、无歧义
- **格式**: 需求规格说明书（Markdown）

### S01-04 需求评审组织
- **必选组件**: 评审材料、评审人员、评审议程、评审结论
- **可选组件**: 评审检查清单、历史问题参考
- **组装顺序**: 材料准备→人员邀请→会议组织→结论记录
- **约束**: 评审必须有明确的结论和后续行动项
- **格式**: 评审会议纪要（Markdown）

### S01-05 需求变更管理
- **必选组件**: 变更请求描述、变更影响分析、变更审批决策、变更实施计划
- **可选组件**: 变更历史记录、回滚方案
- **组装顺序**: 变更接收→影响分析→审批→实施→验证
- **约束**: 变更必须经过影响分析和审批，不得直接实施
- **格式**: 变更请求单（Markdown）

### S01-06 用户调研执行
- **必选组件**: 调研目标、调研方法、调研对象、调研问题
- **可选组件**: 调研工具、激励方案
- **组装顺序**: 目标确定→方法选择→对象招募→执行调研
- **约束**: 调研问题不得引导，保持中立
- **格式**: 调研计划（Markdown）

### S01-07 竞品功能分析
- **必选组件**: 竞品列表、分析维度、功能对比矩阵、分析结论
- **可选组件**: 竞品截图、用户评价
- **组装顺序**: 竞品选择→功能梳理→维度对比→结论提炼
- **约束**: 分析必须基于实际功能，不得臆测
- **格式**: 竞品分析报告（Markdown表格）

### S01-08 业务流程梳理
- **必选组件**: 流程名称、流程步骤、角色分工、输入输出
- **可选组件**: 流程图、异常分支
- **组装顺序**: 流程识别→步骤分解→角色标注→输入输出定义
- **约束**: 流程必须覆盖正常和异常路径
- **格式**: 业务流程图（Markdown流程图）

### S01-09 需求追溯矩阵
- **必选组件**: 需求ID、需求描述、设计文档、代码模块、测试用例
- **可选组件**: 追溯工具、追溯规则
- **组装顺序**: 需求梳理→追溯关系建立→矩阵填充→验证完整性
- **约束**: 每个需求必须可追溯到设计、代码、测试
- **格式**: 追溯矩阵（Markdown表格）

### S01-10 验收标准定义
- **必选组件**: 需求ID关联、验收条件列表、测试方法、通过标准
- **可选组件**: 边界条件、异常场景
- **组装顺序**: 需求分析→条件提取→测试方法定义→标准量化
- **约束**: 验收标准必须可测试、可量化
- **格式**: 验收标准清单（Markdown）

### S01-11 需求分解
- **必选组件**: 原始需求、分解规则、子需求列表、依赖关系
- **可选组件**: 分解优先级、估算工作量
- **组装顺序**: 需求分析→规则确定→分解执行→依赖梳理
- **约束**: 子需求必须可独立实现和测试
- **格式**: 需求分解树（Markdown）

### S01-12 非功能性需求识别
- **必选组件**: 需求场景、非功能类别、指标定义、验收标准
- **可选组件**: 行业基准、历史数据
- **组装顺序**: 场景分析→类别识别→指标定义→标准确定
- **约束**: 指标必须可测量、有明确的度量方法
- **格式**: 非功能需求清单（Markdown表格）

### S01-13 需求冲突检测
- **必选组件**: 需求列表、冲突检测规则、冲突列表、解决方案
- **可选组件**: 冲突优先级、解决历史
- **组装顺序**: 需求收集→规则定义→冲突检测→方案制定
- **约束**: 冲突必须有明确的解决方案，不得遗留
- **格式**: 冲突检测报告（Markdown）

### S01-14 需求影响分析
- **必选组件**: 变更需求、影响范围、影响评估、风险评估
- **可选组件**: 历史变更记录、回退方案
- **组装顺序**: 变更识别→范围分析→影响评估→风险评估
- **约束**: 影响分析必须覆盖所有相关模块
- **格式**: 影响分析报告（Markdown）

### S01-15 原型需求验证
- **必选组件**: 原型方案、验证目标、验证方法、验证结论
- **可选组件**: 用户反馈、改进建议
- **组装顺序**: 原型准备→目标确定→验证执行→结论分析
- **约束**: 验证必须有明确的通过/失败标准
- **格式**: 验证报告（Markdown）

### S01-16 需求文档版本管理
- **必选组件**: 文档版本、变更记录、审批状态、发布日期
- **可选组件**: 版本对比、变更通知
- **组装顺序**: 版本规划→变更记录→审批→发布
- **约束**: 版本必须有明确的变更说明
- **格式**: 版本记录（Markdown表格）

### S01-17 需求沟通对齐
- **必选组件**: 沟通目标、沟通对象、沟通内容、沟通结论
- **可选组件**: 沟通方式、会议纪要
- **组装顺序**: 目标确定→对象识别→内容准备→沟通执行→结论确认
- **约束**: 沟通必须有明确的结论和行动项
- **格式**: 沟通纪要（Markdown）

### S01-18 需求风险识别
- **必选组件**: 需求场景、风险类型、风险描述、风险等级
- **可选组件**: 风险历史、应对策略
- **组装顺序**: 场景分析→风险识别→等级评估→策略制定
- **约束**: 风险必须有明确的应对策略
- **格式**: 风险登记册（Markdown表格）

### S01-19 需求可行性评估
- **必选组件**: 需求描述、技术可行性、资源可行性、时间可行性
- **可选组件**: 替代方案、风险评估
- **组装顺序**: 需求理解→技术评估→资源评估→时间评估→结论
- **约束**: 评估必须基于实际技术能力和资源
- **格式**: 可行性评估报告（Markdown）

### S01-20 需求池管理
- **必选组件**: 需求来源、需求分类、需求状态、需求优先级
- **可选组件**: 需求标签、需求关联
- **组装顺序**: 需求收集→分类→状态跟踪→优先级管理
- **约束**: 需求必须有明确的状态和负责人
- **格式**: 需求池清单（Markdown表格）

### S01-21 需求复盘
- **必选组件**: 复盘范围、复盘目标、复盘发现、改进措施
- **可选组件**: 复盘参与者、复盘工具
- **组装顺序**: 范围确定→目标设定→发现收集→措施制定
- **约束**: 复盘必须有可执行的改进措施
- **格式**: 复盘报告（Markdown）

### S01-22 需求度量
- **必选组件**: 度量指标、度量方法、度量数据、度量分析
- **可选组件**: 度量基准、度量目标
- **组装顺序**: 指标定义→方法确定→数据收集→分析报告
- **约束**: 度量必须有明确的数据来源和计算方法
- **格式**: 度量报告（Markdown表格）

### S01-23 需求模板维护
- **必选组件**: 模板类型、模板内容、模板版本、使用说明
- **可选组件**: 模板示例、模板变更记录
- **组装顺序**: 类型确定→内容编写→版本管理→说明编写
- **约束**: 模板必须易于使用，包含必要字段
- **格式**: 模板文件（Markdown）

### S01-24 需求工具配置
- **必选组件**: 工具选择、配置参数、使用规范、培训材料
- **可选组件**: 工具对比、集成方案
- **组装顺序**: 工具选择→配置→规范制定→培训
- **约束**: 工具配置必须符合团队工作流程
- **格式**: 配置文档（Markdown）

### S01-25 需求培训
- **必选组件**: 培训目标、培训内容、培训对象、培训方式
- **可选组件**: 培训材料、培训评估
- **组装顺序**: 目标确定→内容准备→对象识别→培训执行
- **约束**: 培训必须有明确的目标和评估方式
- **格式**: 培训计划（Markdown）

## S02 架构设计

### S02-01 技术选型评估
- **必选组件**: 候选技术列表、评估维度、评估矩阵、选型建议
- **可选组件**: POC验证结果、迁移成本评估
- **组装顺序**: 需求分析→候选筛选→维度评估→矩阵对比→决策
- **约束**: 选型必须基于可验证的技术指标，不得仅凭"流行度"
- **格式**: 技术选型报告（Markdown）

### S02-02 架构模式选择
- **必选组件**: 业务特征分析、候选架构模式、模式匹配分析、选择理由
- **可选组件**: 混合架构方案、演进路线
- **组装顺序**: 业务分析→模式库匹配→适配分析→决策
- **约束**: 架构模式必须与业务特征匹配
- **格式**: 架构决策记录（Markdown）

### S02-03 领域模型设计
- **必选组件**: 领域概念、实体关系、聚合根、领域服务
- **可选组件**: 领域事件、值对象
- **组装顺序**: 领域分析→概念提取→关系建模→服务定义
- **约束**: 领域模型必须反映业务本质，不得受技术实现影响
- **格式**: 领域模型图（Markdown表格+关系图）

### S02-04 API设计
- **必选组件**: 资源模型、端点列表、请求/响应格式、错误码定义
- **可选组件**: 分页策略、限流策略、缓存策略
- **组装顺序**: 资源建模→端点设计→格式定义→错误处理
- **约束**: API必须遵循RESTful/GraphQL设计规范
- **格式**: API设计文档（Markdown）

### S02-05 数据库设计
- **必选组件**: 实体关系图、表结构定义、索引设计、约束定义
- **可选组件**: 分区策略、归档策略、读写分离方案
- **组装顺序**: 需求分析→实体识别→关系建模→表设计→索引优化
- **约束**: 数据库设计必须考虑性能和扩展性
- **格式**: 数据库设计文档（Markdown表格+ERD）

### S02-06 系统架构图绘制
- **必选组件**: 系统边界、核心组件、组件关系、数据流向
- **可选组件**: 部署视图、安全视图、性能视图
- **组装顺序**: 组件识别→关系梳理→分层布局→标注说明
- **约束**: 架构图必须清晰表达系统结构和数据流
- **格式**: 系统架构图（Markdown图表）

### S02-07 容量规划
- **必选组件**: 业务量预估、性能指标、资源需求、扩容方案
- **可选组件**: 历史数据、行业基准
- **组装顺序**: 业务分析→指标定义→资源评估→方案制定
- **约束**: 容量规划必须基于实际业务量和增长预期
- **格式**: 容量规划报告（Markdown）

### S02-08 技术风险评估
- **必选组件**: 技术方案、风险识别、风险评估、应对策略
- **可选组件**: 风险监控、应急预案
- **组装顺序**: 方案分析→风险识别→等级评估→策略制定
- **约束**: 风险评估必须覆盖技术、资源、时间三个维度
- **格式**: 技术风险评估报告（Markdown表格）

### S02-09 架构决策记录
- **必选组件**: 决策背景、决策内容、决策理由、决策后果
- **可选组件**: 替代方案、决策置信度
- **组装顺序**: 背景梳理→方案对比→决策制定→后果分析
- **约束**: 决策必须有明确的理由和预期后果
- **格式**: 架构决策记录（Markdown）

### S02-10 性能架构设计
- **必选组件**: 性能目标、性能瓶颈、优化方案、验证方法
- **可选组件**: 性能基准、监控方案
- **组装顺序**: 目标定义→瓶颈分析→方案设计→验证计划
- **约束**: 性能设计必须有明确的指标和验证方法
- **格式**: 性能架构设计文档（Markdown）

### S02-11 安全架构设计
- **必选组件**: 安全需求、威胁模型、安全控制、安全验证
- **可选组件**: 安全标准、合规要求
- **组装顺序**: 需求分析→威胁建模→控制设计→验证计划
- **约束**: 安全设计必须遵循OWASP等安全标准
- **格式**: 安全架构设计文档（Markdown）

### S02-12 可用性设计
- **必选组件**: 可用性目标、故障模式、冗余方案、恢复策略
- **可选组件**: SLA定义、监控告警
- **组装顺序**: 目标定义→故障分析→冗余设计→恢复策略
- **约束**: 可用性设计必须有明确的SLA目标
- **格式**: 可用性设计文档（Markdown）

### S02-13 扩展性设计
- **必选组件**: 扩展需求、扩展策略、扩展接口、扩展验证
- **可选组件**: 扩展历史、扩展成本
- **组装顺序**: 需求分析→策略制定→接口设计→验证计划
- **约束**: 扩展性设计必须考虑未来业务增长
- **格式**: 扩展性设计文档（Markdown）

### S02-14 集成方案设计
- **必选组件**: 集成目标、集成接口、集成协议、集成测试
- **可选组件**: 集成工具、集成监控
- **组装顺序**: 目标确定→接口设计→协议选择→测试计划
- **约束**: 集成方案必须考虑系统兼容性和稳定性
- **格式**: 集成方案设计文档（Markdown）

### S02-15 技术原型开发
- **必选组件**: 原型目标、原型范围、原型实现、原型验证
- **可选组件**: 原型文档、原型演示
- **组装顺序**: 目标确定→范围界定→实现开发→验证测试
- **约束**: 原型必须聚焦核心风险，不得过度开发
- **格式**: 原型开发报告（Markdown）

### S02-16 架构评审
- **必选组件**: 评审材料、评审人员、评审标准、评审结论
- **可选组件**: 评审检查清单、历史评审参考
- **组装顺序**: 材料准备→人员邀请→评审执行→结论记录
- **约束**: 评审必须有明确的通过/不通过标准
- **格式**: 架构评审报告（Markdown）

### S02-17 依赖分析
- **必选组件**: 系统组件、依赖关系、依赖类型、依赖风险
- **可选组件**: 依赖图、依赖版本
- **组装顺序**: 组件识别→关系梳理→类型标注→风险评估
- **约束**: 依赖分析必须覆盖直接和间接依赖
- **格式**: 依赖分析报告（Markdown表格+依赖图）

### S02-18 部署架构设计
- **必选组件**: 部署环境、部署拓扑、部署策略、部署工具
- **可选组件**: 部署监控、部署回滚
- **组装顺序**: 环境分析→拓扑设计→策略制定→工具选择
- **约束**: 部署架构必须考虑环境一致性和可重复性
- **格式**: 部署架构设计文档（Markdown）

### S02-19 日志架构设计
- **必选组件**: 日志级别、日志格式、日志存储、日志分析
- **可选组件**: 日志监控、日志告警
- **组装顺序**: 级别定义→格式设计→存储方案→分析工具
- **约束**: 日志设计必须考虑性能影响和存储成本
- **格式**: 日志架构设计文档（Markdown）

### S02-20 缓存策略设计
- **必选组件**: 缓存场景、缓存层级、缓存策略、缓存失效
- **可选组件**: 缓存监控、缓存预热
- **组装顺序**: 场景分析→层级设计→策略制定→失效机制
- **约束**: 缓存策略必须考虑数据一致性和缓存命中率
- **格式**: 缓存策略设计文档（Markdown）

### S02-21 消息架构设计
- **必选组件**: 消息场景、消息模型、消息队列、消息处理
- **可选组件**: 消息监控、消息重试
- **组装顺序**: 场景分析→模型设计→队列选择→处理逻辑
- **约束**: 消息设计必须考虑可靠性和顺序性
