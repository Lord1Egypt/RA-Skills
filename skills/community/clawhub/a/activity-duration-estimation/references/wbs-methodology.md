# WBS 工作分解结构 — 方法论文档

> 本文件定义 WBS（Work Breakdown Structure）在活动历时估算技能中的完整实现：分解方法、参考模板、递归算法、验证规则以及与估算流程的衔接。
>
> → `scripts/wbs_engine.py` 为代码实现
> → `SKILL.md` 定义了子技能入口和全流程调用方式

---

## 一、WBS 概述

WBS 将项目最终成果逐层分解为更小、更易管理的工作包。它不是算法，而是一种结构化分解方法。

### 四种分解方法

| 方法 | 分解依据 | 最适用 | 风险 |
|------|---------|--------|------|
| **交付成果式** (推荐) | 项目产出物的组成部分 | 大多数项目 | 无 |
| **生命周期式** | 项目自然阶段 | 流程清晰的项目 | 阶段间整合易忽略 |
| **模块组件式** | 系统功能模块划分 | 软件开发/AI项目 | 模块间耦合难定义 |
| **职能领域式** | 部门/专业/技术领域 | 大型多学科项目 | 形成部门墙 |

### 100% 规则

下层工作之和必须完整覆盖上层工作，不多不少：
- **不多**：子节点不在父节点范围外引入额外工作
- **不少**：父节点所有工作都能在其子节点中找到对应

由于项目独特性，100%规则不能数学证明。本技能采用**启发式检查**（见第五节）。

### 分解终止条件

工作包同时满足以下条件即停止分解：

| 条件 | 检查标准 | 示例（满足） | 示例（不满足） |
|------|---------|-------------|---------------|
| **可估算** | 预计工时 ≤ 80h（或用户指定粒度） | "登录模块前端开发" | "前端开发" |
| **可分配** | 可推断单一责任人角色 | "UI设计师" | "一个团队" |
| **可验证** | 有明确的交付物或完成标准 | "登录页面完成开发" | "做好" |
| **可控制** | 不跨越里程碑或更新节点 | "模块测试完成" | "上线交付" |

默认最大分解深度 **4 层**，超限自动提示用户确认。

## 三、递归分解算法

### 算法流程

```
1. 用户输入项目描述
2. LLM分析项目类型 → 选择最接近的参考模板
3. 将模板作为骨架，LLM逐节点定制化（改名称/增减子节点）
4. 对每个节点应用分解终止条件检查
5. 不满足终点的节点递归分解（至多4层）
6. 应用100%规则启发式验证
7. 输出WBS结果
```

### 伪代码

```python
def decompose_project(description, template=None, max_depth=4):
    """递归分解项目描述为WBS树"""
    if template is None:
        template = select_template(description)  # LLM选择
    
    root = WBSNode(name=项目名, level=0)
    
    # 在模板基础上定制化
    root.children = llm_customize(template, description)
    
    # 递归分解
    _recursive_decompose(root, max_depth)
    
    return root

def _recursive_decompose(node, max_depth):
    """递归分解节点直到满足终止条件"""
    if node.level >= max_depth:
        node.is_work_package = True
        return
    
    if meets_termination(node):
        node.is_work_package = True
        return
    
    # 由LLM生成子节点（以模板为参考）
    node.children = llm_decompose(node)
    
    for child in node.children:
        _recursive_decompose(child, max_depth)
```

### 关键约束

- **不可跳过用户确认**：WBS生成后必须展示给用户确认，用户可手动调整
- **不可无限递归**：最大深度 4 层，超出即停止并提示
- **不可空节点**：每个节点必须有名称和描述

## 五、100% 规则启发式验证

由于项目独特性，不能数学证明100%规则。本技能采用以下**结构性+语义性检查**：

| 检查项 | 规则 | 通过条件 |
|--------|------|---------|
| 最小子节点数 | 每个非叶节点至少 2 个子节点 | 全部 ≥ 2 |
| 无"其他"遗漏 | 子节点名称不含"其他/等等/其他工作" | 无匹配 |
| 父节点覆盖 | 父节点名称语义能覆盖子节点合集 | LLM判断 |
| 同层互斥 | 同层子节点间无重叠语义 | LLM判断 |
| 深度合规 | 无超过 max_depth 的路径 | 全部 ≤ 4 |

验证不通过时输出具体问题位置，提示人工复审。**不会自动更新 WBS**——只标记问题，由用户决定是否调整。

## 六、WBS → 估算阶段的自动衔接

WBS 工作包自动映射为估算阶段参数：

### 映射规则

| WBS 工作包 | → | 估算阶段参数 |
|------------|---|-------------|
| 名称 + 描述 | → | `name`（阶段名称） |
| LLM 填充的 OMP 值 | → | `o`, `m`, `p` |
| WBS 层级结构 | → | 紧前关系（同级从左到右 FS 串联） |
| 交付物标记 | → | `deliverable`（报告展示） |

### 说明

- OMP 初始值由 LLM 根据行业知识和工作包描述填充
- 用户可在确认 WBS 时提供/调整 OMP 值
- 跨层级紧前关系：子节点完成后才能开始同层级下一个父节点的子节点
- 用户可手动指定紧前关系覆盖自动推演

## 七、WBS 反模式

| AP 编号 | 反模式 | 正确做法 |
|---------|--------|---------|
| AP-W01 | 不管什么项目都用同一种分解方法 | 根据项目特征选择最适配的方法 |
| AP-W02 | 分解到不可估算的粒度（"写一行代码"） | 到 80h 左右即止，满足可估算条件 |
| AP-W03 | 忽略工作包间的依赖关系 | 明确 FS/SS/FF/SF 关系 |
| AP-W04 | 跳过用户确认直接使用 LLM 生成的 WBS | 必须展示给用户确认，允许手动调整 |
| AP-W05 | WBS 和估算脱节（做完 WBS 重新手填参数） | 自动映射 WBS 工作包为估算阶段 |
| AP-W06 | 分解深度不足（只到 L2 就说完成了） | 至少到 L3，工作包须满足终止条件 |
| AP-W07 | 工作包描述含糊（"做好"、"完成相关工作"） | 必须有明确的交付物或完成标准 |
