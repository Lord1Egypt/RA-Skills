# /qa-bug 缺陷分析示例 — 从被驳回到根因定位

> 本示例展示 `/qa-bug` 的核心价值：描述不清时拒绝分析（避免瞎猜）+ 补全信息后精准定位。

## 场景一：信息不足 — 被拒绝

### 原始输入

```
/qa-bug

缺陷标题：支付成功了，但钱没到账
复现步骤：用户付了钱，商家没收到的样子
期望结果：付了应该到账
实际结果：没到账
报错日志：（无）
产品/模块：支付模块
```

### AI 输出

#### 缺陷描述质量评估

| 维度 | 评分 | 说明 |
|------|------|------|
| 标题清晰度 | ❌ 不达标 | "钱没到账"未指明是商家账户/平台账户/还是用户账户 |
| 复现步骤 | ❌ 不达标 | 缺少：具体支付方式（微信/支付宝/银行卡）、订单号、支付金额、操作时间 |
| 期望 vs 实际 | ❌ 不达标 | "应该到账"不可测试——应描述到账的时间预期和具体表现 |
| 环境信息 | ❌ 不达标 | 缺：App 版本、操作系统、网络环境（WiFi/4G） |
| 日志/截图 | ❌ 不达标 | 无支付成功页截图、无商家后台截图、无商户平台流水截图 |

**结论：信息不足，无法进行根因分析。请补全以下信息后重新提交。**

#### 帮助补全——请回答以下问题

1. **"钱没到账"具体指谁的账户？**
   - A. 商家在平台的余额账户
   - B. 商家的银行卡
   - C. 用户被扣款了但订单状态未更新
   - D. 其他（请描述）

2. **支付方式是什么？**
   - A. 微信支付
   - B. 支付宝
   - C. 银行卡快捷支付
   - D. 其他

3. **请提供以下信息：**
   - 订单号：_______
   - 支付金额：_______
   - 支付时间：_______
   - 是否每次都出现？_______

4. **请提供截图：**
   - 用户端支付成功截图
   - 商家后台该订单的状态截图
   - 支付平台（微信/支付宝）商户后台的流水截图

## 场景二：信息补全后 — 根因分析

### 补全后的输入

```
/qa-bug

缺陷标题：微信支付成功，商家平台余额未增加（订单状态仍为"待支付"）
复现步骤：
  1. 用户下单 ORD-20240622-001，金额 299 元，选择微信支付
  2. 跳转微信完成支付，微信端显示支付成功
  3. 返回 App，订单状态仍显示"待支付"
  4. 商家后台"账户余额"未增加 299 元
期望结果：支付成功后订单状态变为"已支付"，商家余额增加 299 元
实际结果：订单状态未变化，余额未增加
报错日志：
  服务端日志：2026-06-22 14:32:15 [WARN] wxpay_callback_handler: signature verification failed
            2026-06-22 14:32:15 [ERROR] order ORD-20240622-001 payment callback rejected
  微信商户后台：显示该笔订单支付成功，回调 URL 返回 403
产品/模块：支付模块 - 微信支付回调
```

### AI 输出

#### 缺陷分析报告
- 缺陷编号：BUG-2024-015
- 产品/模块：支付模块 - 微信支付回调
- 严重程度：严重（AI 辅助判断，置信度：高）

#### 简明摘要（给开发看）
- 现象：微信支付成功但回调被拒，签名验证失败
- 最可能根因：回调签名验证配置不一致（APIv2→v3 迁移遗留问题）
- 建议修复方向：检查回调 URL 对应的签名算法版本，更新签名密钥配置
- 回归要点：验证支付宝和银行卡支付的回调是否受影响

#### 根因分析
- 分类：配置错误
- 可能根因：微信支付回调签名验证失败。服务端日志明确显示 `signature verification failed`，说明微信发来的回调签名与服务端计算的签名不匹配。可能原因：
  1. 微信支付 APIv3 升级后，回调签名算法从 MD5 变更为 RSA，服务端未同步更新验证逻辑
  2. 微信商户后台配置的回调 URL 签名密钥与代码中的密钥不一致
- 置信度：高（日志明确指向签名验证失败 + 微信商户后台显示回调返回 403）

#### 影响范围
- 受影响功能：所有使用微信支付的订单
- 受影响用户角色：消费者（订单状态不更新，可能重复支付）、商家（余额未到账）
- 是否阻塞：是（微信支付渠道完全不可用）

#### 修复建议
- 建议修复方向：对比微信商户后台的 API 版本和代码中的签名验证逻辑，统一为 APIv3 的 RSA 签名方案
- 建议修复位置：`wxpay_callback_handler` 函数，签名验证逻辑
- 注意事项：修复后先在生产环境的灰度机器上验证，确认回调 200 后再全量

#### 回归测试要点
1. 验证修复后微信支付正常回调、订单状态更新、余额增加
2. 检查支付宝和银行卡支付的回调是否受影响（确认不是通用问题）
3. 同类场景排查：退款回调、企业付款回调是否使用了相同的签名验证逻辑
4. 检查是否有其他商户因同一问题导致回调失败（商户后台可能积压了失败回调）

#### 同类缺陷预防
- 建议补充到 Checklist：第三方 SDK/API 升级时，检查所有回调接口的兼容性
- 建议补充的自动化校验规则：在预发布环境部署后自动发送一笔测试支付验证回调链路

## 场景三：批量模式

### 原始输入

```
/qa-bug

批量分析本周缺陷（W26），以下是缺陷列表（CSV）：
ID,标题,模块,严重级别,状态
BUG-015,微信支付回调签名验证失败,支付模块,严重,待修复
BUG-016,订单列表分页 100 条时超时,订单模块,一般,待修复
BUG-017,支付宝支付成功但短信未发送,支付模块,一般,已修复
BUG-018,客服改价输入框未限制负数,订单模块,严重,已修复
BUG-019,微信支付重复扣款,支付模块,致命,待修复
BUG-020,订单导出日期筛选无响应,订单模块,建议,待修复
```

### AI 输出（精简）

#### 简明摘要（给管理者看）
- 缺陷总数：6 个（致命:1 / 严重:2 / 一般:2 / 建议:1）
- 修复率：33%（2/6）
- 高频模块：支付模块 3 个（含致命重复扣款）占 50%
- 主要根因：第三方对接（3个）、边界遗漏（2个）

#### 高频模块预警
- **支付模块**：缺陷密度最高（3 个），含致命级"重复扣款"。建议下一版本增加支付模块专项回归
- **订单模块**：集中在边界问题（分页超时、负数输入），建议在 /qa-case 中补充边界值 Checklist

#### 缺陷关联分析
- **同源缺陷群**：BUG-015（签名失败）和 BUG-019（重复扣款）均涉及支付回调链路，可能共享签名验证逻辑，建议合并排查
- **批次效应检测**：致命级 BUG-019 出现在本周三微信支付 API 升级的批次中，建议排查该次变更的全部影响范围

## 使用技巧

1. **日志是关键**：有日志→置信度高→开发信服。没有日志的缺陷，AI 只能给出低置信度的推测
2. **先看质量评估，再决定是否追问**：开发关 Bug 说"我这儿没问题"，让测试员补全信息后再提交一次
3. **批量模式适合迭代复盘**：周五下班前把本周全部缺陷 CSV 导出粘贴进去，5 分钟出一份趋势报告
4. **回归要点这部分不要只依赖 AI**：AI 提的关联功能建议和测试人员自己的业务理解结合起来
