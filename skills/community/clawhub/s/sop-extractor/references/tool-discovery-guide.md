# 工具预调研指南

> 在萃取工序之前，先摸清这个岗位/场景有哪些工具可以用。
> 目标：让生成的 Skill 从"说明书"变成"自动化工作流"。

---

## 一、为什么要做工具预调研

**核心问题**：如果 Skill 里没有可调用的工具，它就只是一份文字说明书，不是自动化能力。

**预调研的价值**：
- IT 可以提前把 CLI/MCP/API 包装好，Skill 生成时直接对接
- 能自动化的环节标为"自动"，不能的标为"人工中转"
- 让每个 Skill 的自动化率可量化（目标：≥80%）

---

## 二、调研维度

### 2.1 岗位涉及的系统/平台

向用户确认该岗位日常使用的系统：

| 问题 | 示例 |
|------|------|
| "你日常用哪些系统/平台？" | 猎聘、企业微信、OA、ERP、飞书... |
| "哪些操作你每天/每周都要登录进去做？" | 猎聘搜索候选人、OA审批... |
| "有没有觉得特别麻烦、重复的操作？" | 每次都要手动复制粘贴候选人信息... |

### 2.2 每个系统的自动化可行性

对每个系统，调查以下信息并记录：

```
系统名称：[猎聘]
用途：[搜索候选人、查看简历]
是否有官方API：[无]
是否有CLI工具：[无]
是否有MCP Server：[无]
是否有第三方爬虫/自动化：[有，Python爬虫，但不稳定]
数据导出方式：[无官方导出，只能手动复制]
自动化结论：[❌ 无法自动化，需人工中转]
人工中转方式：[用户截图/复制粘贴 → AI处理]
```

### 2.3 工具矩阵模板

| 系统/平台 | 用途 | API | CLI | MCP | 自动化结论 | 人工中转方式 |
|----------|------|-----|-----|-----|-----------|------------|
| 例：企业微信 | 日程读取+消息推送 | ✅ | ✅ | ✅ | 可自动 | — |
| 例：猎聘 | 搜索候选人 | ❌ | ❌ | ❌ | 需人工 | 截图/粘贴 |
| 例：OA系统 | 审批 | ✅ | ❌ | ❌ | 部分自动 | 复杂审批手动 |

## 三、工具搜索方法

### 3.1 搜索顺序

```
1. 搜官方API文档     → "[系统名] API" / "[系统名] 开放平台"
2. 搜CLI工具         → "[系统名] CLI" / "[系统名] command line"
3. 搜MCP Server      → "[系统名] MCP server" / "github [系统名] MCP"
4. 搜开源自动化       → "[系统名] automation github" / "[系统名] python SDK"
5. 搜ClawHub/Hermes  → clawhub search "[系统名]" / 检查已有skills
```

### 3.2 本地工具检查

在搜索之前，先检查本地已有的能力：

```bash
# 检查已安装的 MCP servers
cat ~/.hermes/config.yaml | grep -A 20 "mcp"

# 检查已有的 skills
ls ~/.hermes/skills/*/SKILL.md

# 检查已安装的 CLI 工具
which [tool-name]
```

### 3.3 常见系统的自动化方案参考

| 系统类型 | 通常可用方案 | 备注 |
|---------|------------|------|
| 企业微信/钉钉/飞书 | API + 机器人 | 消息、日程、审批都有API |
| 招聘平台（猎聘/Boss/拉勾） | 通常无官方API | 需人工中转或浏览器自动化 |
| OA/ERP/CRM | 视具体产品 | SAP/Salesforce有API，国产OA需逐一确认 |
| 文档协作（飞书/腾讯文档） | API | 读写都有API |
| 邮件 | SMTP/IMAP | 标准协议，CLI工具丰富 |
| Git/GitHub | CLI + API | gh CLI, git |
| 数据库 | CLI + API | mysql/psql/redis-cli |

## 四、预配置清单

调研完成后，生成一份"预配置清单"交给 IT：

```markdown
## 工具预配置清单

### 可以立即配置的（有现成方案）
- [ ] 企业微信机器人：注册应用，获取 webhook URL
- [ ] 企业微信日程 API：配置应用权限，获取 access_token
- [ ] 飞书文档 API：创建应用，获取 app_id/app_secret

- [ ] 封装企业微信消息发送为 CLI 工具

### 无法自动化的（需人工中转）
- [ ] 猎聘搜索 → 用户截图/粘贴，AI 处理
- [ ] 电话沟通 → AI 生成话术模板，人执行

### 已有可用的
- [x] 文件读写（PDF/Word/Excel）
- [x] 文本处理和分析
- [x] 企业微信消息模板生成
```

## 五、自动化率计算

每个工序的自动化率按以下公式计算：

```
自动化率 = 可自动化的环节数 / 总环节数 × 100%
```

**评级标准**：
- ≥ 80%：🟢 高自动化，Skill 直接可用
- 60%-79%：🟡 中等自动化，部分环节需人工中转
- 40%-59%：🟠 低自动化，建议先预配置工具再生成 Skill
- < 40%：🔴 不适合做 Skill，建议先解决工具问题

**目标**：每个 Skill 上线前自动化率 ≥ 60%，通过持续补充工具逐步提升到 80%+。

## 六、断点处理策略

当自动化链路中有断点（某环节无法自动化）时：

| 断点类型 | 处理策略 | 示例 |
|---------|---------|------|
| 需要登录外部系统 | 人工操作 → 粘贴结果给 AI | 猎聘搜索 |
| 需要打电话/发消息 | AI 生成内容 → 人工发送 | 电话邀约 |
| 需要审批决策 | AI 建议 → 人工确认 | 费用审批 |
| 需要物理操作 | AI 无法介入，标注为人工 | 实物签收 |
| 需要读取外部数据 | 配置 API/CLI/MCP → 自动读取 | 日程表 |

**原则**：断点应该被显式标注，而不是隐藏在流程中。用户执行到断点时，Skill 应该清晰地告诉他："这一步需要你手动做，以下是你要做的事情。"
