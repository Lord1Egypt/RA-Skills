## 一、平台运营（约1年）

### 工作职责

负责对接公司、合作伙伴、供应商、客户，进行需求与交易的集中管理。核心逻辑：将公司自有产品、伙伴产品、供应商产品统一划入"资源池"，在客户（需求方）提出需求时进行个性化推荐，促成交易撮合。

### 对接角色

| 角色 | 性质 | 说明 |
|------|------|------|
| 公司 | 供给方（内部） | 自有产品直接划入资源池 |
| 合作伙伴 | 供给方（外部） | 产品需审核通过后划入资源池 |
| 供应商 | 供给方（外部） | 移交至职能管理部门进行资质审查，通过后划入资源池 |
| 客户 | 需求方 | 根据需求从资源池中进行个性化推荐 |

### 工作流程

```
确认对方角色
    ├── 供给方路径
    │   ├── 公司自有产品 → 直接划入资源池
    │   ├── 伙伴产品 → 运营审核 → 通过后划入资源池
    │   └── 供应商产品 → 移交职能管理部门 → 资质审查 → 通过后划入资源池
    └── 需求方路径
    │   └── 运营根据资源池内产品 → 个性化推荐 → 促成交易
```

### 常见数字化产品案例（辅助求职者理解）

运营需要非常了解各类产品，求职者至少应能清楚一类具体产品并能说出交易撮合过程。以下是常见数字化产品分类：

| 产品类别 | 典型产品 | 适用场景 | 撮合关键点 |
|----------|----------|----------|------------|
| 云基础设施 | 云服务器、云存储、CDN加速 | 企业数字化转型、网站/App部署 | 客户有IT基础设施需求，运营推荐匹配规格的云产品 |
| 数据服务 | 数据采集工具、数据分析平台、BI报表 | 企业数据治理与决策支持 | 客户有数据分析需求，运营推荐适配的数据产品组合 |
| 安全产品 | 防火墙、入侵检测、安全审计 | 企业网络安全建设 | 客户有合规/安全需求，运营推荐对应的安全方案 |
| 协同办公 | OA系统、即时通讯、项目管理工具 | 企业日常办公效率提升 | 客户有组织协同需求，运营推荐合适的办公工具 |
| 智能客服 | AI客服机器人、呼叫中心系统 | 企业客户服务自动化 | 客户有客服效率需求，运营推荐智能客服方案 |
| 电商服务 | 电商SaaS平台、支付系统、物流接口 | 商家线上交易能力建设 | 客户有线上销售需求，运营推荐电商解决方案 |

**撮合过程示例（云基础设施）：**

1. 客户（某中型制造企业）提出需求："我们要做数字化转型，需要搭建官网和ERP系统"
2. 运营确认需求细节：预估访问量、数据量、预算范围
3. 从资源池中筛选匹配产品：云服务器（4核8G）、云数据库、CDN加速、SSL证书
4. 组织产品组合方案，向客户推荐并讲解各产品价值
5. 客户确认方案，完成交易签约

### 注意事项

1. **产品理解是核心门槛**：运营需要非常了解各类产品，求职者至少要能清楚一类具体产品并能说出交易撮合过程。面试时务必准备1-2个具体案例
2. **专业词汇解释**：
   - **伙伴**：与公司有合作关系的外部企业或组织，双方通过产品互补、渠道共享等方式进行合作，伙伴产品需通过审核才能进入资源池
   - **供应商**：为公司提供产品或服务的外部企业，区别于伙伴——供应商是纯粹的供给方角色，需经过资质审查（如营业执照、行业资质认证等）
   - **资源池**：公司统一管理的产品集合，包含自有产品、伙伴产品和供应商产品，是运营进行推荐撮合的"弹药库"
   - **交易撮合**：运营在需求方和供给方之间起桥梁作用，匹配需求与产品，促成交易达成的过程
3. **管理流程差异提醒**：各公司的管理流程、制度有差别。伙伴审核流程、供应商资质审查标准、交易签约方式等在不同公司可能有不同规范。求职者在面试时应注意辨别，避免将一家公司的流程套用到另一家

## 二、电商产品经理（约1年）

### 工作职责

负责为商家提供线上交易系统的产品设计，实现商品、库存、订单、价格的线上数字化管理。目标：让商家能够通过系统高效管理线上生意。

### 对接角色（5类）

| 角色 | 性质 | 说明 |
|------|------|------|
| 运营 | 需求方 | 汇总市场需求，与产品共同调研 |
| 销售 | 需求方 | 前线接触客户，成单后反馈需求 |
| 商家 | 需求方 | 系统的最终使用者，提出使用需求 |
| 研发 | 开发者 | 负责系统功能的技术实现 |
| 测试 | 开发者 | 确保研发质量，验证功能正确性 |

### 工作流程

```
市场需求（三类客户：3C卖场、生鲜、商超）
    ↓
销售成单
    ↓
运营汇总需求
    ↓
运营 + 产品共同调研
    ↓
产品功能设计
    ↓
组织评审（项目经理发起，各方参与）
    ↓
研发投入
    ↓
测试跟进确保研发质量
    ↓
系统上线
    ↓
运营跟进反馈需求
    ↓
优化迭代（循环）
```

### 三类业态差异（关键知识点）

不同业态的客户对系统的侧重点不同，这是求职者必须掌握的差异化理解：

| 维度 | 3C卖场 | 生鲜 | 商超 |
|------|---------|------|------|
| **核心侧重点** | 供应商资质、线下服务与售后 | 品类管理、供货中心、配送服务 | 传统零售模式（POS、库存、促销） |
| **商品管理** | 重点关注品牌授权、型号管理 | 重点关注保质期、损耗率、品类划分 | 重点关注SKU数量、货架管理 |
| **供应链** | 需要严格的供应商准入和资质审核 | 需要供货中心集中采购、冷链配送 | 需要多级库存管理、配送调度 |
| **售后** | 退换货流程复杂，涉及维修服务 | 保质期短，售后相对简单 | 常规退换货 |
| **价格策略** | 价格波动大，需要灵活定价 | 价格随季节波动，需要快速调价 | 促销活动多，需要促销引擎 |

### 推荐面试案例方向

建议选择**商超**业态作为面试案例，原因：
- 商超是最常见的零售形态，面试官容易理解
- 商超的需求相对通用化，涵盖商品管理、库存管理、订单管理、促销管理等电商核心功能
- 不会涉及太多行业特有的复杂概念（如冷链、保质期管理等）

**商超案例示例——促销引擎设计：**

1. **需求背景**：商超客户希望系统能支持多种促销活动（满减、折扣、限时特价、组合优惠等）
2. **产品方案**：
   - 设计促销规则引擎，支持灵活配置促销条件
   - 促销类型：满减（满XX减XX）、折扣（X折优惠）、限时特价（时间段内特价）、组合优惠（买A+B优惠）
   - 促销优先级设置（多个促销同时生效时的计算规则）
   - 促销效果数据追踪（参与人数、优惠金额、订单增长等）
3. **关键设计决策**：促销计算在购物车环节执行，与订单价格计算解耦，便于后续扩展

### 互联网常见内部术语解释

| 术语 | 解释 |
|------|------|
| PRD | Product Requirements Document，产品需求文档，描述产品功能的详细规范 |
| 需求池 | 产品经理管理所有需求的地方，按优先级排列 |
| 评审 | 产品方案由各方（运营、研发、测试等）共同审查讨论，确认可行性和细节 |
| 迭代 | 产品按版本周期持续优化更新，每个版本是一次迭代 |
| MVP | Minimum Viable Product，最小可行产品，先上线核心功能再逐步完善 |
| 上线 | 产品功能经过测试后正式发布给用户使用 |
| 回归测试 | 修改bug后重新测试，确保修复没有引入新问题 |
| 灰度发布 | 先让部分用户使用新功能，验证没问题后再全量发布 |
| 需求调研 | 产品经理通过访谈、问卷、数据分析等方式深入了解用户需求 |
| 产品路线图 | 产品功能的长期规划，展示各版本的计划和目标 |

## 三、项目管理（约0.5-1年）

### 工作职责

负责对接客户、伙伴、公司团队，协调各方推动项目落地验收。核心是确保项目按计划、按标准、按时交付。

### 对接角色

| 角色 | 说明 |
|------|------|
| 客户 | 项目甲方，验收方 |
| 合作伙伴 | 可能参与项目交付的外部协作方 |
| 公司团队 | 包含销售、产品、研发、设计 |
| 双方领导 | 项目关键节点的审批与决策者 |

### 工作流程

```
项目签订、正式启动
    ↓
确定合同范围和交付标准
    ↓
与客户、团队共同确认项目计划（项目经理撰写）
    ↓
监督项目执行进度
    ├── 协调产品：确保需求明确
    ├── 协调研发：确保每阶段任务完成
    ↓
阶段性汇报进度（向双方领导）
    ↓
项目验收汇报
```

### 风险识别与管理（重要补充）

除了推进流程，项目经理必须能识别和管理项目风险：

| 风险类型 | 具体表现 | 应对策略 |
|----------|----------|----------|
| 人员变动 | 关键人员离职或调岗 | 提前识别关键依赖人，建立AB岗机制，及时调整资源 |
| 成本变动 | 需求变更导致成本超预算 | 建立变更管理流程，重大变更需双方领导审批 |
| 验收延期 | 各方配合不到位导致延期 | 设置里程碑检查点，提前预警，预留缓冲期 |
| 需求蔓延 | 客户不断追加新需求 | 明确合同范围边界，超出范围走变更流程 |
| 技术风险 | 技术方案可行性存疑 | 早期组织技术预研，关键点提前验证 |

### 注意事项

1. **项目经理不参与业务**：项目经理的核心价值在于管理、组织、协调、汇报能力，而非直接参与产品设计或开发
2. **管理能力是核心**：对于一个人的管理、组织、协调、汇报能力要求很高。面试时应突出这些软实力
3. **计划撰写是基本功**：项目计划要自己写，不是简单复制模板，要能根据项目特点定制合理的里程碑、任务分工和时间安排

### 项目管理常见术语解释

| 术语 | 解释 |
|------|------|
| 里程碑 | 项目中重要的阶段性节点，标志着某个阶段的完成 |
| 交付物 | 项目每个阶段需要产出的具体成果（如文档、系统、功能等） |
| 变更管理 | 对项目范围、需求、计划变更的规范化管控流程 |
| 需求蔓延 | 项目范围无序扩展，超出原始合同约定 |
| WBS | Work Breakdown Structure，工作分解结构，将项目任务逐层拆解 |
| 甘特图 | 项目进度可视化工具，展示任务时间线和依赖关系 |
| 验收 | 客户确认项目交付成果符合合同约定的标准和要求 |
| 立项 | 项目正式启动，获得审批和资源分配 |
| 站会 | 团队每天短会同步进度和问题（源自敏捷开发） |
| 复盘 | 项目结束后回顾总结，分析做得好的和需要改进的地方 |

## 四、AI产品经理（当前岗位）

### 工作职责

负责AI中台（模型开发平台）的产品设计、功能上线，并对企业AI落地应用过程进行设计，同时保障项目培训、交流等工作。

### 前置知识门槛（求职者必须掌握）

在使用这段经历之前，求职者必须能回答以下核心问题：

#### Q1：什么是AI中台（模型开发平台）？

**AI中台**是为企业提供一站式AI能力建设的平台，核心功能是让企业能够自主开发、训练、部署和管理AI模型，而不需要从零搭建基础设施。也叫"模型开发平台"或"AI开发平台"。

类比理解：就像云平台让企业不需要自建机房就能用云计算，AI中台让企业不需要自建算法团队就能用AI能力。

#### Q2：什么是模型？

**模型**是通过算法对数据进行学习后得到的"知识压缩体"。它像一个经过训练的专家，能够对新的输入做出预测或判断。例如：
- 图像识别模型：输入一张图片，输出"这是猫"还是"这是狗"
- 文本分类模型：输入一段文字，输出"这是投诉"还是"这是咨询"
- 推荐模型：输入用户行为数据，输出推荐的商品列表

#### Q3：模型开发平台与AI应用有什么区别？

| 对比维度 | AI中台（模型开发平台） | AI应用 |
|----------|----------------------|--------|
| **定位** | 工具平台，帮助开发模型 | 终端产品，直接服务用户 |
| **用户** | 数据科学家、算法工程师、开发者 | 业务人员、终端用户 |
| **核心功能** | 数据管理、模型开发/训练/部署 | 特定场景的智能服务（如智能客服、智能质检） |
| **产出** | 模型本身 | 基于模型的业务解决方案 |
| **类比** | 像工厂的生产线 | 像商店里的成品商品 |

#### Q4：模型开发过程是怎样的？

```
开发 → 训练 → 部署 → 应用
  │      │      │      │
  │      │      │      │
数据    算法    模型    业务
准备    选择    上线    场景
与      与      与      与
特征    参数    接口    效果
工程    调优    封装    评估
```

- **开发**：数据准备、特征工程、算法选型
- **训练**：用数据喂给模型，调整参数使其学习
- **部署**：将训练好的模型封装成可调用的服务接口
- **应用**：将模型能力嵌入具体业务场景，产生实际价值

### 对接角色

| 角色 | 说明 |
|------|------|
| 项目经理 | 提供项目进度和客户需求信息 |
| 销售 | 前线了解客户AI应用场景 |
| 研发（工程） | 负责平台功能的技术实现 |
| 研发（算法） | 负责模型算法的优化和开发 |
| 测试 | 确保平台功能质量 |

### 工作流程

#### 流程一：系统功能迭代

```
产品设计 → 研发 → 项目上线 → 需求反馈 → 优化设计 → 研发迭代（循环）
```

#### 流程二：AI应用落地

```
需求调研 → 方案设计 → AI应用上线 → 运营反馈（循环）
```

### 注意事项

1. **AI产品是前沿岗位**：提醒求职者要多学习和了解AI行业动态，面试时需要展示对AI技术趋势的理解
2. **中台思维**：AI中台的产品设计需要具备"平台思维"，不是做一个具体功能，而是设计一套让其他人能高效工作的工具链
3. **技术与业务的桥梁**：AI产品经理需要既懂技术（模型原理），又懂业务（落地场景），这是最独特的价值
4. **持续学习提醒**：AI行业发展极快，求职者需要持续关注新技术（如大模型、RAG、Agent等）的发展动态

### AI行业常见术语解释

| 术语 | 解释 |
|------|------|
| 中台 | 企业级共享能力平台，将通用能力集中建设，避免各业务线重复开发 |
| 模型 | 通过算法对数据学习后得到的"知识压缩体"，能对新输入做出预测 |
| 算法 | 解决特定问题的一系列计算步骤，是模型训练的"方法" |
| 特征工程 | 从原始数据中提取有用信息（特征）的过程，直接影响模型效果 |
| 训练 | 用数据喂给模型，调整参数使其学习规律的过程 |
| 部署 | 将训练好的模型封装成可调用的服务，让业务系统能使用 |
| 推理 | 模型接收新输入并输出预测结果的过程 |
| 调优 | 通过调整参数、数据等方式提升模型效果 |
| 标注 | 给原始数据打标签（如"这是猫""这是投诉"），是训练数据的来源 |
| RAG | Retrieval-Augmented Generation，检索增强生成，大模型先检索知识再生成回答 |
| Agent | AI代理，能自主感知环境、做出决策并执行动作的智能体 |
| Fine-tuning | 微调，在预训练模型基础上用特定领域数据进一步训练，使其更专业 |
| MLOps | Machine Learning Operations，模型运维管理体系，类似DevOps但针对AI |
