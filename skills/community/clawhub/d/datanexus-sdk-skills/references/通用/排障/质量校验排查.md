# 质量校验排查 — 创角有深无浅

> 适用端：IAP/混变小游戏、Android 游戏
>
> 质量因子：**有深无浅（CREATE_ROLE）**

## 规则说明

质量看板检测到：某用户在近 N 天内上报了**注册（REGISTER）**，且后续有**付费（PURCHASE）**等深层行为，但**缺少创角（CREATE_ROLE）**行为上报。

正常的行为漏斗：`REGISTER → CREATE_ROLE → PURCHASE`

校验逻辑：有 PURCHASE（深）但无 CREATE_ROLE（浅），判定为"有深无浅"不合格。

## 可能原因及诊断流程

```
质量看板"创角有深无浅"不合格
│
├─① 该游戏是否真的有创角行为？
│   ├─ 没有 → 联系运营申请豁免（无需上报）
│   └─ 有 → 继续排查 ↓
│
├─② REGISTER 是否上报正确？（最常见原因）
│   │
│   │  诊断方法：在 DataNexus 日志查询中，
│   │  筛选该 openid/用户 的 REGISTER 行为，
│   │  检查是否存在**多次注册**记录。
│   │
│   ├─ 老用户也上报了 REGISTER
│   │   → 原因：注册判断逻辑有误（客户端自判 / 服务端逻辑缺陷）
│   │   → 表现：系统误将老用户识别为"新注册用户"，
│   │          老用户本身就跳过了创角，直接有付费，
│   │          所以看起来"有深无浅"
│   │   → 修复：确保 REGISTER 仅在服务端确认 isNew=true 时上报
│   │          详见：客户端自判沉默注册的反面案例.md
│   │
│   └─ REGISTER 上报正确（仅新用户有）→ 继续排查 ↓
│
├─③ CREATE_ROLE 是否漏报？
│   │
│   │  诊断方法：
│   │  1. 在日志查询中筛选该用户的全部行为，确认是否有 CREATE_ROLE
│   │  2. 检查代码中创角流程是否调用了上报方法
│   │
│   ├─ 代码中有创角流程但没有上报调用
│   │   → 原因：埋点遗漏
│   │   → 修复：在创角成功回调中添加上报
│   │          小游戏：sdk.onCreateRole(roleName) 或 sdk.track("CREATE_ROLE", {...})
│   │          Android：ActionUtils.onCreateRole(roleName) 或 GDTAction.logAction("CREATE_ROLE")
│   │
│   ├─ 代码中有上报但某些分支未覆盖
│   │   → 原因：条件分支遗漏（如第三方登录跳过了创角流程、
│   │          游客模式自动创角但未上报等）
│   │   → 修复：排查所有创角入口，确保每条路径都有上报
│   │
│   └─ 代码中有上报且逻辑正确 → 继续排查 ↓
│
└─④ 是否存在上报时序问题？
    │
    ├─ 创角上报时 openid 尚未设置
    │   → 表现：SDK 缓存了数据但未能成功上报（无用户标识）
    │   → 修复：确保 setOpenId 在创角上报之前完成
    │
    └─ 创角上报接口返回失败
        → 在开发者工具 Network 或 Logcat 中查看上报接口返回码
        → 参考：错误码速查表.md
```

## 快速自查清单

| # | 检查项 | 预期结果 |
|---|--------|---------|
| 1 | 游戏内是否有创角流程 | 有（若无需联系运营豁免） |
| 2 | DataNexus 日志中，同一用户是否有多条 REGISTER | 仅 1 条（首次进入时） |
| 3 | 创角成功后是否调用了 `onCreateRole` / `track("CREATE_ROLE")` | 是 |
| 4 | 所有创角路径是否都覆盖了上报（含自动创角、游客模式等） | 是 |
| 5 | 创角上报时 openid 是否已设置 | 是（先 setOpenId 再上报行为） |
| 6 | 创角上报接口返回 code=0 | 是 |

## 修复后验证

1. 修复代码后，用新设备/新用户完整走一遍：注册 → 创角 → 付费
2. 在 [DataNexus 日志查询](https://datanexus.qq.com/web/monitor/logs) 中确认该用户依次有 REGISTER、CREATE_ROLE、PURCHASE 三条记录
3. 等待质量看板 T+1 更新后确认校验通过
