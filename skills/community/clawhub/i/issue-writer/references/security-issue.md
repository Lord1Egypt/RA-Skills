# Security Issue Shape

Use this shape for vulnerabilities, hardening gaps, abuse paths, token/secret risks, auth/authorization gaps, proxy/origin risks, SSRF, DoS, and insecure deployment defaults.

## Required Sections

```markdown
# <Action-oriented title>

## Summary

What boundary is weakened, what components are involved, and why the combined impact matters.

## High / Medium / Low

### N. <Finding title>

**Severity:** High | Medium | Low

**Description:**
One paragraph describing the issue and the security boundary affected.

**Evidence chain:**
1. `path:line`: Direct code/config fact.
2. `path:line`: Call chain, deployment exposure, or data flow.
3. `path:line`: Missing control or risk-amplifying context.

**Risk reasoning:**
Because A + B + missing C, an attacker with D precondition can cause E impact.

**Recommended fix:**
- Recommendation:
  - Reason:
  - Options:

## Acceptance Criteria

- [ ] Verifiable security boundary restoration.
- [ ] Tests for bypass/rejection paths.
- [ ] Docs/config examples updated when defaults change.
```

## Security-Specific Rules

1. Include a direct evidence chain for every finding.
2. Include risk reasoning for every finding; do not rely on labels like "SSRF" alone.
3. Recommendations that reject defaults, strip headers, deny targets, restrict origins, or fail startup must include a reason and an alternative.
4. Do not include live secrets, token values, private credentials, or copy-paste exploit payloads beyond what is necessary to explain the issue.

## Recommendation Examples

```markdown
- Recommendation: Strip client-provided `X-Forwarded-*`, `Forwarded`, and `X-Real-IP` before proxying.
  - Reason: These headers describe trusted forwarding metadata and should only be generated by a trusted proxy; client-provided values can corrupt auth, audit, redirect, or origin decisions upstream.
  - Options: Rebuild `X-Forwarded-For`, `X-Forwarded-Host`, `X-Forwarded-Proto`, and `X-Forwarded-Prefix` from trusted proxy state.
```

```markdown
- Recommendation: Reject placeholder secrets such as `change-me-*`, `*-change-in-production`, `REPLACE_ME`, and `<REPLACE_WITH_...>` in production.
  - Reason: Predictable values must not become usable signing keys.
  - Options: Use clearly invalid placeholders in sample YAML and document generation via `openssl rand -base64 32`, Secret Manager, or ExternalSecret.
```

## Severity Guidance

Assign severity from the evidence, not from the vulnerability label alone. Consider:

- Attacker preconditions and reachability.
- Trust boundary crossed.
- Privilege, credentials, or data exposed.
- Blast radius across users, tenants, systems, or infrastructure.
- Default deployment exposure.
- Available mitigations or compensating controls.
