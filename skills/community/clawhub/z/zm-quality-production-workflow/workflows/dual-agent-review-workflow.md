# 双 Agent 审核流程

## 目的

避免同一执行者自写自审导致漏问题。正式产出必须至少经过两个独立视角审核，再由 main 终审。

## 默认角色

### Reviewer A：内容逻辑 / 专业质量审核

关注：

- 目标是否准确；
- 结构是否完整；
- 逻辑是否跳步；
- 论点是否有证据；
- 案例是否真实；
- 数据是否有来源；
- 是否有夸大；
- 是否符合业务 / 课程体系。

### Reviewer B：用户体验 / 场景可用性审核

关注：

- 用户是否看得懂；
- 讲师是否讲得顺；
- 是否有互动；
- 是否能行动；
- 学员会问什么；
- 答案是否准备；
- 交付场景是否可用；
- 是否符合目标人群心理。

## 审核输入

- 标准化输入卡；
- 产物路径；
- 样章标准；
- 任务目标；
- 禁止项；
- 依赖缺口清单；
- 当前版本说明。

## 审核输出

每个 Reviewer 必须输出：

```text
结论：PASS / NEEDS_REVISION / BLOCKED
评分：0-100
最高风险：
必须返工：
建议优化：
证据路径：
是否允许进入下一阶段：YES / NO
```

模板见 `templates/reviewer_report_template.md`。

## 冲突处理

- A PASS，B FAIL：说明逻辑成立但用户体验不行，返工表达 / 互动 / 案例。
- A FAIL，B PASS：说明好懂但不严谨，返工论证 / 数据 / 结构。
- A BLOCKED 或 B BLOCKED：不得放行，main 判断是否补材料。
- 两者都 PASS：进入 main 终审。

## 循环规则

- 默认最多 2 轮返工；
- 同类问题重复出现，升级为流程问题；
- 关键素材缺失，不得继续靠 AI 猜；
- 用户反馈必须扩展到全包同类问题。

## 单 Agent 降级模式

只有轻量内部任务可使用“单 Agent 双视角审核”。

正式任务不得降级，除非用户明确要求或 SubAgent 不可用并标记原因。
