# 机构模式 产品选型与产品介绍（境外微信支付）

> 本文档服务于**机构**接入境外微信支付（跨境）的产品选型。
>
> ‼️ **境外接入主体硬性规则**：目前**全球范围内仅中国香港地区**允许商户以"直联商户"身份直接对接微信支付（请求体只带 `mchid` + `appid`）；**其余所有国家和地区（含美国、日本、欧洲、东南亚等）的商户均必须通过"机构"接入**——由机构与微信支付签约、为子商户进件，子商户在机构旗下经营。境外文档（[llms.txt](https://pay.weixin.qq.com/doc/global/v3/zh/llms.txt)）通过主要参数区分"商户/机构"两种模式：机构既能直接收款（接入 7 大支付产品），也能为子商户进件、做跨境分账、推送海关报关、为子商户申请 H5 支付权限。本 Skill 所有文档以机构视角撰写，请求体统一带 `sp_mchid`（机构号）+ `sub_mchid`（子商户号）。
> 
> ‼️ **接入模式选择规则**：根据商户展业区域决策。只有商户展业区域在香港，才允许选择直连或机构模式接入，其他区域均只能通过机构模式接入。
>
> ‼️ **结算币种硬性规则**：机构 / 子商户的**结算币种由其注册所在地强制决定**，必须是该国家/地区的法定本地货币（例如美国主体只能结算 USD、日本主体只能结算 JPY、欧元区主体只能结算 EUR、英国主体只能结算 GBP、泰国主体只能结算 THB……）。一个机构号 / 子商户号仅对应一种结算币种，**不允许跨地区选择币种**。
>
> ‼️ **境外 ≠ 境内**：境外域名 `apihk.mch.weixin.qq.com`、路径前缀 `/v3/global/`、API 版本包括v2和v3，但推荐使用v3，；如果商户接入的是境内域名（`api.mch.weixin.qq.com`），请引导切换到 `apihk.mch.weixin.qq.com`。

## 一、能力地图

机构在境外可使用的能力分两大类，共 **11 个业务模块**：

### A. 收款类（7 大支付产品）

付款码 / Native / JSAPI / APP / 小程序 / H5 / 委托代扣 —— 全部走 `/v3/global/...` 前缀，签名主体为**机构**自身。

### B. 平台扩展类（4 项）


| #   | 能力            | 用途                                  | 是否需额外资质          |
| --- | ------------- | ----------------------------------- | ---------------- |
| 1   | **子商户进件**     | API 化为子商户递交资料、获取 `sub_mchid`、变更结算账户 | 需机构主体已开通         |
| 2   | **跨境分账**      | 平台 / 连锁品牌 / 多角色分润，发起分账 + 分账退款       | ⚠️ 需单独申请         |
| 3   | **海关报关**      | 跨境电商 / 保税交易场景，向海关推送支付单完成三单合一        | ⚠️ 需海关备案         |



> 详见各模块章节。

## 二、产品选型

> 本章节基于官方文档 [境外支付方案选择](https://pay.weixin.qq.com/doc/global/v2/zh/4013637404) 及子页面汇总，从**实际业务场景**出发，给出推荐的支付产品组合。适合机构在初次选型时快速定位方案。

### 6 大典型接入场景

境外机构的收款业务按**商户形态 + 用户终端 + 扣款模式**可归纳为 6 类主流场景：

| # | 场景名称 | 商户特征 | 用户终端 | 推荐产品 |
|---|---------|---------|---------|---------|
| 1 | **线下 POS 终端方案** | 有专业收银 POS 设备 | 柜台扫枪 / 屏幕 / 打印纸 | **付款码** 或 **Native** |
| 2 | **无 POS 线下门店方案** | 中小微商户无收银机 | 移动设备 / 无任何设备 | **付款码 + Native** 或 **JSAPI 静态码** |
| 3 | **PC 网页方案** | 商户有 PC 官网/商城 | 桌面浏览器 | **Native** |
| 4 | **移动端网页方案** | 商户有移动 H5 | 手机浏览器 / 微信内浏览器 | **JSAPI** 或 **H5** |
| 5 | **移动端应用方案** | 商户有 APP / 小程序 | 手机 APP / 微信小程序 | **APP 支付** 或 **小程序支付** |
| 6 | **周期扣款 / 订阅类业务** | 需周期性 / 免密扣款 | 任意（先签约后扣款） | **委托代扣** ⚠️ 需单独申请开通 |

---

### 场景 1：线下 POS 终端方案

商户已有 POS 机具，按 POS 的**设备能力**递进适配：

| 子场景 | POS 能力 | 推荐产品 | 交互流程 |
|-------|---------|---------|---------|
| 1A | ✅ 有扫码设备（扫码枪）| **付款码** | 用户出示付款码 → 扫码枪扫 → 提交 micropay 扣款 |
| 1B | ✅ 有显示屏 | **Native** | 生成 `code_url` → 屏幕显示二维码 → 用户手机微信扫 |
| 1C | ✅ 可打印小票 | **Native** | 生成 `code_url` → 打印到小票上 → 用户扫纸质码 |
| 1D | ❌ 均无能力 | 走场景 2（无 POS 方案） | — |

**开发成本对比**：1A / 1B 需改造 POS 系统（与 POS 厂商联合开发），1C 实现最简但用户体验较差。

---

### 场景 2：无 POS 线下门店方案

中小微商户无专业收银设备，按**是否有移动设备**分两种方案：

| 子场景 | 商户设备 | 推荐产品 | 关键点 |
|-------|---------|---------|------|
| 2A | 有手机/平板 | **付款码 + Native**（二选一） | 商户开发一个收银的e-POS APP 扫用户付款码 或 APP 生成二维码让用户扫 |
| 2B | **完全无设备** | **JSAPI 静态码** ⭐ | 打印一张固定二维码贴柜台，用户扫码进 H5 输入金额 |

**⭐ 场景 2B 是机构批量接入小微商户的"标准方案"**：
- 机构用**统一的 `sp_appid` + `sp_openid`**（无需每个子商户单独绑 `sub_appid`）
- 机构开发一个统一的 H5 收银页，所有子商户的静态码都指向这个页面（用 URL 参数区分 `sub_mchid`）
- 适用：菜市场 / 路边摊 / 小餐饮 / 夫妻店 等长尾商户
- 详细做法见下方 §4.3 「线下静态码（基于 JSAPI）」章节

---

### 场景 3：PC 网页方案

商户有 PC 官网 / 跨境电商 B2C 站点：

| 项 | 内容 |
|---|------|
| **推荐产品** | **Native 支付**（动态二维码）|
| **典型用户** | 跨境电商、B2B 采购、海外华人官网 |
| **用户流程** | 1) 用户在 PC 选购 → 2) 前端展示二维码（由后端 `code_url` 生成）→ 3) 用户用手机微信「扫一扫」完成支付 → 4) PC 页面轮询/WebSocket 感知支付完成 |
| **关键注意** | 每个订单一个独立的 `code_url`（2 小时有效），**不要**复用二维码 |

---

### 场景 4：移动端网页方案

根据用户访问网页的**环境**分两种：

| 子场景 | 用户环境 | 推荐产品 | 说明 |
|-------|---------|---------|------|
| 4A | **微信内浏览器** | **JSAPI 支付**  | 标准对外产品、无需额外申请；|
| 4B | **微信外浏览器**（Safari/Chrome）| **H5 支付** 推荐 | **境外常用的移动端web解决方案，需额外权限申请** |


**混合方案（同一 URL 兼容两种环境）**：
```javascript
if (/MicroMessenger/i.test(navigator.userAgent)) {
  // 微信内 → JSAPI
  payViaJSAPI();
} else {
  // 外部浏览器 → H5（前提：已获得 H5 权限）
  payViaH5();
}
```

---

### 场景 5：移动端应用方案

商户自有 APP 或小程序：

| 子场景 | 商户形态 | 推荐产品 | 前置配置 |
|-------|---------|---------|---------|
| 5A | 原生 APP（iOS/Android）| **APP 支付** | 微信开放平台注册 APP → 获取 `sub_appid` → 机构侧完成 `sub_appid` ⇄ `sub_mchid` 绑定 |
| 5B | 微信小程序 | **小程序支付** | 小程序平台注册 → 获取 `sub_appid` → 绑定 `sub_mchid` |

**重要**：
- APP / 小程序的**注册主体**必须与 `sub_mchid` 的主体**一致**
- 参数传输规则与其他支付方式略有差异（详见 [API 规则](https://pay.weixin.qq.com/doc/global/v3/zh/4012354960.md)）
- 有的商户想要在APP内使用H5页面完成支付，可以选择h5支付的对接方式，但是支付完成后无法返回APP而是打开浏览器。

### 场景 6：周期性扣款 / 订阅类业务

> 来源：[微信支付境外 委托代扣产品介绍](https://pay.weixin.qq.com/doc/global/v3/zh/4012354183)

**业务定义**：在用户**事先授权**的前提下，商户可在**无需用户输入密码**的情况下，按业务规则或用户行为触发，主动从用户微信支付账户扣款。

#### 适用业务类型

| 大类 | 触发方式 | 典型业务 |
|------|---------|---------|
| **周期扣费** | 按时间规则触发（每月/每周/按账单周期） | 会员订阅、SaaS 续费、水电煤气、话费、保险、宽带 |
| **先享后付** | 按用户行为触发（用完即扣） | 二维码乘车、共享单车 / 充电宝、网约车、按量计费云服务 |

#### 推荐产品

| 产品 | 路径 | 何时使用 |
|------|------|---------|
| **委托代扣（Entrusted Deduction / 自动扣费）** | `POST /v3/global/papay/transactions` | 所有周期性 / 先享后付场景 |

#### 两阶段流程

```
┌──────────────────────────────────────────────────────┐
│ 阶段一：签约（一次性）                                │
│   用户授权 → 商户拿到 contract_id                     │
│   契约长期有效，直到用户主动解约                      │
└──────────────────────────────────────────────────────┘
                       │
                       ▼ （后续按业务节奏触发）
┌──────────────────────────────────────────────────────┐
│ 阶段二：扣款（多次）                                  │
│   商户后台调扣款 API（带 contract_id）                │
│   微信侧无密扣款 → 异步回调结果                       │
│   用户无感知（也可短信/邮件通知）                     │
└──────────────────────────────────────────────────────┘
```

#### 5 种签约通道（根据用户接入端选择）

| 签约通道 | 用户场景 | 调起方式 |
|---------|---------|---------|
| **JSAPI 预签约** | 用户在微信内 H5 / 公众号 | 通过JSAPI在微信内H5完成签约 |
| **H5 签约** | 用户在外部浏览器 | 跳转微信完成签约，再跳回 |
| **APP 签约** | 用户在商户原生 APP | 微信开放平台 OpenSDK 调起 |
| **小程序签约** | 用户在微信小程序 | `wx.requestPayment` 类似调起 |
| **WEB 预签约** | 用户在 PC 网页 | 类似 Native，展示二维码让用户手机微信扫 |

#### 前置条件 / 资质要求

⚠️ **委托代扣需单独申请开通**，并不在标准 7 大支付产品的默认权限内：

1. **机构必须先开通委托代扣业务**（联系商务对接人 / 邮件申请）
2. **行业准入限制** —— 截至当前已开放以下行业：
   - 互联网共享 / 租赁充电宝
   - 公共事业（水、电、燃气、供暖）
   - 智能冰柜
   - 话费充值
   - 交通行业（共享单车 / 网约车 / 公交地铁）
   - 线上内容行业（订阅、会员、SaaS）
3. **其他行业**：需联系商务单独申请并提交资质材料
4. 详见 [接入前准备](https://pay.weixin.qq.com/doc/global/v3/zh/4012354185)

#### 关键 API 路径

| 用途 | API |
|------|-----|
| 预签约 （JSAPI） | `POST /v3/global/papay/contracts/pre-entrust-sign` |
| 查询契约 | `GET /v3/global/papay/contracts/{contract_id}` |
| 扣款 | `POST /v3/global/papay/transactions` |
| 查询扣款单 | `GET /v3/global/papay/transactions/out-trade-no/{out_trade_no}` |
| 解约 | `POST /v3/global/papay/contracts/{contract_id}/terminate` |
| 退款 | 走标准退款接口 `POST /v3/global/refunds`（注意：通知字段与普通支付略不同） |
| 签约 / 解约结果回调 | 商户配置的 `notify_url`（与扣款回调独立） |
| 扣款结果回调 | 商户配置的 `notify_url` |

详细 API 列表见 [API 列表](https://pay.weixin.qq.com/doc/global/v3/zh/4012354194)。

#### 与普通一次性支付的核心差异

| 维度 | 普通支付 | 委托代扣 |
|------|---------|---------|
| **谁主动** | 用户主动发起 | **商户**主动发起 |
| **是否需密码** | 视风控决定 | ❌ **完全免密** |
| **授权模式** | 单次授权（每次都需操作）| 预先授权（一次签约长期有效）|
| **API 调用次数** | 1 次（下单 → 扣款合并）| 2 阶段（签约 1 次 + 扣款 N 次）|
| **凭证** | `prepay_id` / `auth_code` | **`contract_id`**（永久有效直到解约）|
| **回调** | 1 类（支付结果）| 2 类（签约/解约结果 + 扣款结果，独立通知）|
| **退款** | 标准退款接口 | 走标准退款接口，但通知字段略有差异 |

#### 重要业务规则

1. **用户随时可解约** —— 用户可在微信内主动取消授权；机构必须监听 `CONTRACT_TERMINATED` 回调并立即停止后续扣款
2. **每次扣款用新 `out_trade_no`** —— 失败重试也必须换号
3. **扣款失败要有重试与降级策略** —— 推荐 T+1 / T+3 退避，连续失败 3 次则挂起服务并通知用户


详细开发指引见 [开发指引](https://pay.weixin.qq.com/doc/global/v3/zh/4012354187)。

### 场景速查决策树
决策建议：根据商户描述的支付场景，列举所有可行方案，及方案优劣势的对比，不做优先方案的推荐。

```
你的商户长什么样？
│
├── 有 POS 机具
│   └─> 场景 1（线下 POS）：付款码 / Native
│
├── 小微商户 + 无收银机
│   ├── 有手机 ──> 场景 2A：付款码 或 Native
│   └── 无任何设备 ──> 场景 2B：JSAPI 静态码（⭐ 推荐机构统一方案）
│
├── 有 PC 网站
│   └─> 场景 3（PC 网页）：Native 扫码
│
├── 有移动 H5
│   ├── 主要从微信内进入 ──> 场景 4A：JSAPI 
│   └── 主要从外部浏览器进入 ──> 场景 4B：H5（需申请权限，境外常见解决方案）
│
├── 有自研 APP / 小程序
│   ├── APP ──> 场景 5A：APP 支付
│   └── 小程序 ──> 场景 5B：小程序支付
│
└── 周期性扣款 / 订阅业务（叠加场景）
    └─> 场景 6：委托代扣（先签约 → 后无密扣款，需额外申请开通）
```

### 场景与产品对应表

以 11 业务模块视角反向查找：

| 支付产品 | 适用场景 | 是否主力推荐 |
|---------|---------|------------|
| 付款码 | 1A / 2A | ⭐ 线下被扫首选 |
| Native | 1B / 1C / 2A / 3 | ⭐ 线下主扫、PC 首选 |
| JSAPI | 2B / 4A | ⭐ 静态码首选 |
| H5 | 4B | ⚠️ 需单独申请权限⭐境外移动端web解决方案首选 |
| APP 支付 | 5A | ⭐ 原生 APP 唯一选择 |
| 小程序支付 | 5B | ⭐ 小程序唯一选择 |
| 委托代扣 | **场景 6**（订阅 / 周期扣款 / 先享后付） | ⭐ 周期性 / 免密扣款唯一选择，需单独申请开通 |

> 💡 **组合思路**：同一机构/子商户可**同时启用多种产品**（例如一家连锁餐饮：门店用付款码、外卖小程序用小程序支付、官网商城用 Native），产品之间的订单号、对账都是独立的。
> **重要**：
- 微信支付目前没有统一的web支付产品（即对接一套接口，可同时支持移动端和 PC 端支付），需要机构根据不同的用户终端对接不同的产品。

## 四、各产品关键说明

### 4.1 付款码支付（被扫，一步扣款）

**核心差异**：付款码是 7 大产品中唯一的"被扫"模式 —— 由商户侧扫码设备扫描用户付款码，直接 API 扣款，无下单步骤。

**交互流程**：

1. 用户打开微信 →"我"→"服务"→"收付款"，屏幕展示付款码（条码 + 二维码）
2. 收银员在子商户系统生成订单、确认金额
3. 收银员用扫码枪扫用户付款码，子商户系统提交"付款码支付"API 直接扣款
4. 微信后台按风控规则判断是否需要验证支付密码：
  - 免密交易：直接扣款成功
  - 需验密交易：用户微信端弹出密码框，确认后完成

**付款码编码规则**：18 位纯数字，前缀 10~14 = 中国大陆用户，前缀 15 = 中国香港用户。

**两种扣款模式**：
- **小额免密**：金额在免密额度内（境外默认通常 3000 CNY 以内或等值），直接扣款
- **密码确认**：金额超过免密额度或用户启用密码保护，需用户在手机上输密码

**触发验密（USERPAYING）的条件**：

以下任一条件满足时，用户需要输入支付密码确认，付款码支付接口会返回 `USERPAYING` 状态：

| # | 触发条件 | 说明 |
|---|---------|------|
| 1 | **单笔金额 > 3000 CNY**（或等值外币） | 超出免密额度上限 |
| 2 | **单日付款码支付次数 > 10 次** | 当日累计使用付款码支付超过 10 笔 |
| 3 | **用户点击了付款码并展示了其内容** | 用户主动查看付款码详情（条码数字），触发安全验证 |
| 4 | **其他风控原因** | 微信支付风控系统综合判定（如异地交易、设备变更、账户异常等） |

> ⚠️ 以上条件由微信支付风控引擎实时判定，机构/子商户无法绕过或关闭。开发时**必须**完整实现 `USERPAYING` 状态的轮询查单 + 超时撤销逻辑。

**关键开发注意**：

- ⚠️ **境外付款码已是 V3 版本**（JSON + WECHATPAY2-SHA256-RSA2048），与境内仅有 V2 不同；部分迁移自境内的代码需重写
- ⚠️ **付款码识别预拦截**：付款码必须 18 位且前缀 10-15，**非微信付款码须在收银前端拦截**，不要提交到微信接口（这是 checklist 强制项）
- ⚠️ 必须实现 USERPAYING（用户支付中）状态的**轮询查单** + **超时撤销**逻辑，否则会出现"用户已付款但商户未感知"的问题
- 同一付款码（`auth_code`）**只能对应一个订单号**（`out_trade_no`）；如需重新提交，**必须同时更换付款码和订单号**
- 网络异常时**不要直接重试**，使用原 `out_trade_no` 查单确认实际状态

**异常响应处理表**：

| 响应状态 | 处理方式 |
|---------|---------|
| `SUCCESS` | 支付成功，打小票 |
| `USERPAYING` | 用户支付中（密码确认模式），**调用查单接口轮询**（间隔 3s、5s、10s…），**切勿重复提交付款码** |
| `AUTHCODE_INVALID` | 付款码无效/过期，要求用户**刷新付款码**并**生成新订单号**后重试 |
| `NOTENOUGH` | 用户余额不足 |
| 网络超时/无响应 | 使用原 `out_trade_no` **查单**确认实际状态，切勿直接重试（避免重复扣款）|

**撤销与退款**：
- **撤销**：仅 Quick Pay 支持、仅当天可撤销，用于提交时网络异常、不确定是否扣款成功的场景
- **退款**：跨日后必须用退款接口；退款由 CNY 资金池扣款，可能出现 `NOT_ENOUGH`（详见 [📄 排障手册](../问题排查/排障手册.md)）

### 4.2 委托代扣（先签约后扣款）

**核心差异**：委托代扣不是一次性支付，而是"用户授权 → 商户存协议 → 后续按需扣款"的两阶段产品。

**两阶段流程**：

1. **签约阶段**：用户在以下任一签约通道完成授权（5 选 1）：
  - 小程序签约 / H5 签约 / JSAPI 预签约 / WEB 预签约 / APP 签约
  - 签约成功后，机构通过**异步通知**收到 `contract_id`（必须正确接收并存储）
2. **扣款阶段**：机构持有签约成功返回的 `contract_id`，按业务节奏调用「支付扣款」API；扣款结果以异步通知（[扣款结果通知](https://pay.weixin.qq.com/doc/global/v3/zh/4013015485.md)）回到机构 `notify_url`。

**典型应用场景**：
- 订阅服务（视频/音乐/软件月费、年费自动续费）
- 酒店住宿（入住时授权，退房时按实际消费扣款）
- 共享出行（租车/共享单车按使用时长自动扣款）
- 分期付款 / 公共事业缴费 / 教育培训学费分期

**关键开发注意**：

- ⚠️ **委托代扣业务须单独开通**，签约后才能调用扣款 API
- ⚠️ **每次扣款前/后必须通知用户**（短信/邮件/APP 推送），微信支付风控会监测通知行为，未通知可能导致权限下调
- 一个用户对一个机构 + 子商户号只能有**一份有效签约**
- 扣款前建议先调**查询签约状态** API，确认协议仍有效（用户可能在微信侧主动解约）
- 签约 / 解约结果有独立回调（[签约与解约结果回调](https://pay.weixin.qq.com/doc/global/v3/zh/4013015481.md)），需单独处理
- 退款用普通退款接口，但通知字段与基础支付不同
- 失败重试节奏建议：T+1 重试 1 次，T+3 再重试 1 次，仍失败则挂起

### 4.3 JSAPI 支付 与 小程序支付（同接口、不同前端）

- **共用下单接口**：境外 JSAPI 与小程序都走 `/v3/global/transactions/jsapi`（已核实）
- **区别在前端调起方式**：
  - JSAPI：在微信内浏览器中用 `WeixinJSBridge.invoke('getBrandWCPayRequest', ...)`
  - 小程序：在微信小程序中用 `wx.requestPayment(...)`
- **`openid` 获取方式不同**（机构模式可二选一）：
  - JSAPI：通过 `sub_appid`（已认证服务号）的网页授权拿 `sub_openid`，**或**通过 `sp_appid`（机构服务号）的网页授权拿 `sp_openid`（OAuth 2.0，scope=snsapi_base）
  - 小程序：通过 `sub_appid`（小程序）的 `wx.login` + `code2session` 拿 `sub_openid`，**或**通过 `sp_appid`（机构小程序）拿 `sp_openid`
- **`sub_openid` vs `sp_openid` 选择**：
  - 用 `sub_appid` + `sub_openid` → 每个子商户必须有自己的服务号/小程序，且需做 `sub_appid` 与 `sub_mchid` 绑定
  - 用 `sp_appid` + `sp_openid` → **机构提供统一的服务号/小程序**，所有子商户共用，**无需绑定 sub_appid**（**静态码统一收银场景的标准做法** ⭐）
- **共用注意**：调起支付时前端传的 `appId` 必须与 openid 来源 AppID 一致——用 `sub_openid` 就传 `sub_appid`，用 `sp_openid` 就传 `sp_appid`，混用会报 "appid mchid not match"

**JSAPI 用户流程要点**：
1. 用户在微信内点击链接进入商户 H5
2. H5 页面通过公众号 OAuth 授权获取 `sub_openid`
3. 后端调用下单 API 获得 `prepay_id`
4. 后端用商户私钥生成 `paySign`（**严禁前端签名**，会暴露私钥）
5. 前端调 `WeixinJSBridge.invoke` 唤起支付
6. 用户输密码确认 → 异步回调到 `notify_url`

**JSAPI 前置配置**：
- 公众号 AppID 与机构号 / 子商户号绑定
- 在公众平台配置**网页授权域名**（用于 OAuth）
- 在商户平台配置**支付授权目录**（必须以 `/` 结尾，与发起支付页 URL 前缀匹配）

**小程序 用户流程要点**：
1. 用户在小程序内 `wx.login` → 后端用 code 换 `sub_openid`
2. 下单 API 必须传 `payer.sub_openid`
3. 前端 `wx.requestPayment` 调起，无跳转，体验流畅
4. 后端处理异步回调

**小程序 前置配置**：
- 小程序 AppID 与机构号 / 子商户号绑定
- 在小程序后台配置 **request 合法域名**（HTTPS）
- 不需要"支付授权目录"

**线下静态码（基于 JSAPI）** ⭐ 机构统一收银的标准方案：

- 商户打印一张静态二维码贴柜台，用户扫码后跳转**机构提供的统一收银 H5 页面**
- 用户在 H5 页面输入金额 → 走标准 JSAPI 流程

**🔑 静态码场景的 openid 接入推荐方式**：

| 接入方式 | AppID | 取的 openid | 是否需 sub_appid 绑定 sub_mchid | 适用场景 |
|---------|-------|-------------|--------------------------------|---------|
| ⭐ **推荐**：机构统一服务号 | `sp_appid` | `sp_openid` | **❌ 不需要** | 静态码 / 机构提供统一收银台 / 子商户无服务号 |
| 子商户自有服务号 | `sub_appid` | `sub_openid` | ✅ 需要 | 子商户有独立公众号品牌、自建收银页 |

**为什么静态码场景推荐用 `sp_appid` + `sp_openid`**：
- 机构只需准备 **1 个**服务号即可服务**所有子商户**，无需为每个小商户单独申请、备案、绑定
- 进件流程大幅简化：子商户进件时**不用提供 `sub_appid`**，也跳过 `sub_appid`–`sub_mchid` 绑定步骤
- 适合菜市场 / 路边摊 / 小餐饮等大量长尾子商户的批量接入
- 下单时请求体里传 `payer.sp_openid`（不是 `sub_openid`），但 `sp_mchid` + `sub_mchid` 仍照常带上

**下单请求示例（静态码 + `sp_openid` 方式）**：
```json
POST /v3/global/transactions/jsapi
{
  "sp_appid":  "wx_机构服务号_appid",
  "sp_mchid":  "1900000001",
  "sub_mchid": "190000xxxx",
  "description": "订单描述",
  "out_trade_no": "STATIC20260429100001",
  "notify_url": "https://yourdomain.com/notify",
  "amount": { "total": 1500, "currency": "HKD" },
  "payer": { "sp_openid": "oXXXX..." }   ← 注意是 sp_openid
}
```

**适用场景**：菜市场 / 路边小店 / 摊贩 / 咖啡店点单台 / 餐桌台卡

**⚠️ 注意事项**：
- 静态码可能被覆盖/替换，建议大额支付二次校验、店内播报到账
- 调起支付时前端 `appId` 必须传 `sp_appid`（与 `sp_openid` 来源一致）
- 详见：[📄 接入质量检查](../接入指南/接入质量检查.md) 中的静态码风险章节

### 4.4 H5 支付 vs JSAPI 支付（最容易混淆）


| 项                | H5 支付                         | JSAPI 支付                   |
| ---------------- | ----------------------------- | -------------------------- |
| 用户起点             | 微信**外**手机浏览器                  | 微信**内**浏览器（公众号 / 微信内 H5）   |
| 调起方式             | 跳 `mweb_url` → 切到微信完成支付 → 切回浏览器 | 直接 `WeixinJSBridge.invoke` |
| 是否需 `sub_openid` | ❌ 不需要                         | ✅ 必需                       |
| `prepay_id` 时效   | `mweb_url` **5 分钟**             | `prepay_id` 2 小时           |
| 境外特有前置           | ⚠️ 机构必须先为子商户走"H5 支付权限申请"配域名   | 无                          |


> **判断标准**：用户访问网页时所处的环境。在微信内 → JSAPI；在外部浏览器 → H5。

**H5 支付关键开发注意**：
- ⚠️ **场景限制**：微信内不能用 H5 支付（会报错），必须在**系统浏览器**中打开
- 必须配置**支付域名**白名单（在商户平台 / H5 权限申请 API）
- `mweb_url` 有效期 **5 分钟**，必须立即跳转使用
- 必传 `scene_info.payer_client_ip`（**用户真实 IP**，不是服务器 IP）
- `redirect_url` 用于支付完成后用户主动返回商户页面，**不是支付结果判定依据**——必须以异步回调为准
- 跳转方式：`window.location.href = h5_url + '&redirect_url=' + encodeURIComponent(returnUrl)`

### 4.5 Native 支付 vs JSAPI 扫码场景（次易混淆）

- **Native 支付**：商户在 PC 网页 / 自助机展示二维码，用户**手机**微信「扫一扫」直接完成支付（PC 浏览 + 手机扫码付款）
- **JSAPI 扫码场景**：用户扫的是商户公众号的二维码，扫完**在微信内**打开 H5 页面，再走 JSAPI（手机扫 → 手机内付款）

**Native 支付关键开发注意**：
- 使用返回的 `code_url`（格式：`weixin://wxpay/bizpayurl?pr=xxx`）生成二维码
- ⚠️ **不要** URL encode `code_url`
- 二维码尺寸建议 ≥ 200×200，保证扫码识别率
- 不能仅靠前端轮询判断成功，**必须以异步回调为准**
- 用户长时间未支付应主动调**关单 API**，避免订单悬挂
- `out_trade_no` 一旦使用**不可复用**

**Native 在线下场景的应用（动态码）**：
- 连锁门店 / 超市 / 有收银屏幕场景下，每笔订单生成一个动态码
- 收银流程：输入金额 → 调下单 API → 屏幕展示二维码 + 金额 → 启动前端轮询（3 秒/次）→ 收到回调后语音播报 + 打小票 → 用户放弃则关单

### 4.6 APP 支付（境外特殊配置）

**接入流程**：
1. **微信开放平台注册移动应用**（open.weixin.qq.com，不是公众平台），获取 AppID
2. iOS 提供 Bundle ID + Universal Link；Android 提供包名 + 签名 MD5
3. 应用通过审核（3-5 工作日）
4. 商户平台将开放平台 AppID 绑定到机构 / 子商户号
5. 客户端集成 SDK：
   - iOS：`pod 'WechatOpenSDK'` + 配置 URL Scheme + Universal Link
   - Android：`com.tencent.mm.opensdk:wechat-sdk-android` + 创建 `{包名}.wxapi.WXPayEntryActivity`
6. 后端调下单 API（`/v3/global/transactions/app`）→ 拿到 `prepay_id`
7. 后端用商户私钥**生成 5 个调起参数**（appId、partnerId、prepayId、nonceStr、timestamp、sign），返回客户端
8. 客户端 `api.sendReq(req)` 拉起微信支付

**关键开发注意**：
- ⚠️ App AppID 必须在**开放平台**（不是公众平台）注册
- iOS 必须配置 **Universal Link**（HTTPS 域名 + apple-app-site-association 文件），否则支付完不能跳回 App
- Android 注册时签名 MD5 与发布 APK 签名必须一致
- 客户端调起参数必须**后端签名**

## 五、平台扩展能力（4 项）

### 5.1 子商户进件

机构通过子商户进件 API 完成子商户入驻，承担：


| 子能力   | 说明                                              |
| ----- | ----------------------------------------------- |
| 进件子商户 | 提交子商户主体 / 法人 / 经营信息 / 结算账户，审核通过后获得 `sub_mchid`  |
| 查询子商户 | 获取审核进度、查询子商户当前状态                                |
| 修改子商户 | 变更结算账户 / 经营信息（仅部分字段允许，详见官方文档）                   |
| 图片上传  | 营业执照 / 身份证等资质材料，每张返回一个 `media_id`，**每秒最多 10 张** |


**进件流程**：
```
1. 收集子商户资料（营业执照、法人身份、经营地址、结算账户等）
   ↓
2. 机构调【提交进件申请】API
   ↓
3a. 微信支付返回 sub_mchid，若verification_status 若返回Approved，则sub_mchid 马上可用，受理交易
3b. 微信支付返回 sub_mchid，若verification_status 若返回Under Review，则触发人工审核 
   ↓
4. 微信支付人工审核（通常 1-3 工作日）
   ↓
5. 机构通过查询确认子商户审核状态，直至sub_merchant_status变为OPERATING（即审核通过，sub_mchid 可用） 或sub_merchant_status 保持INCOMPLETE_APPLICATION，但application_status 变为REJECTED（表示明确审核失败，入驻被拒绝）
   ↓
6. 对于REJECTED 的申请，机构可登录商户平台确认拒绝原因，重新提交进件请求
```

**关键开发注意**：
- ⚠️ 资料必须真实合法，虚假资料会导致申请被拒 + 机构信誉受损
- ⚠️ 敏感信息（身份证号、银行账号等）必须用**平台证书公钥** + RSA-OAEP 加密，请求头带 `Wechatpay-Serial`（详见 [📄 签名与验签规则](../接入指南/签名与验签规则.md)）
- `business_code` 一次申请绑定一个，被驳回后需**换新**编号重新提交
- 选对 MCC 类目（影响费率和可受理产品）
- 批量进件控制频率 ≤ 10 QPS，避免限流；图片上传每秒 ≤ 10 张

> 详见 [子商户进件介绍](https://pay.weixin.qq.com/doc/global/v3/zh/4012354226.md) / [API 列表](https://pay.weixin.qq.com/doc/global/v3/zh/4012354235.md)。

### 5.2 跨境分账

跨境分账面向"平台 / 连锁品牌 / 资金按比例分配"场景，包含 4 大类共 27+ 个 API：


| 类别      | 核心动作                                                                        |
| ------- | --------------------------------------------------------------------------- |
| 发起交易类   | 付款码 / Native / JSAPI / APP / 小程序 / H5 统一下单（**带分账标记** `profit_sharing=true`） |
| 操作分账类   | 分账标记 / 请求分账 / 解冻剩余资金 / 查询分账结果 / 查询剩余可分账金额 / 添加 / 删除 / 查询分账接收方 / 上传文件        |
| 发起退款类   | 分账退款（独立于普通退款）/ 查询订单剩余可退款金额 / 查询可供垫付退款实时金额                                   |
| 账单及资金查询 | 交易账单 / 分账账单文件下载 / 分账退款账单 / 结算资金明细                                           |


**完整分账流程**：
```
1. 下单（profit_sharing=true）⚠️ 必须在下单时标记，不可后期补
   ↓
2. 用户支付成功 → 资金冻结在子商户 CNY 资金池
   ↓
3. 添加接收方（一次性，可复用）→ POST /v3/global/profit-sharing/receivers/add
   ↓
4. 发起分账 → POST /v3/global/profit-sharing/orders（指定接收方 + 金额）
   ↓
5. 查询分账结果（接收方为商户：进入对方 CNY 池；接收方为个人：实时入零钱）
   ↓
6. 完结分账 ⭐（必做）→ 解冻剩余资金，进入正常结算
```

**关键开发注意**：
- ⚠️ **下单时未标记 `profit_sharing=true` → 该订单永久无法分账**，只能退款重新下单
- ⚠️ **必须调用完结分账**，否则订单资金一直冻结、不结算
- 默认分账上限是订单金额的 **30%**，可申请提升至 100%
- 单笔订单最多 **50 个**接收方
- 接收方需先添加（add-receivers），姓名 / 身份证号等敏感信息需 RSA 加密
- 退款场景需先**分账回退**再退款给用户

> ⚠️ **需先申请开通跨境分账资质**，详见 [开通前指引](https://pay.weixin.qq.com/doc/global/v3/zh/4012355044.md) / [申请开通](https://pay.weixin.qq.com/doc/global/v3/zh/4012355046.md) / [产品介绍](https://pay.weixin.qq.com/doc/global/v3/zh/4012354203.md)。
>
> 已核实接口路径：`POST /v3/global/profit-sharing/orders`（请求分账）。

### 5.3 海关报关

跨境电商 / 保税交易场景，机构向海关推送支付单完成三单合一与合规核验：


| API    | 用途             |
| ------ | -------------- |
| 报关     | 报关推单           |
| 身份信息校验 | 买家身份与订单信息一致性校验 |
| 报关查询   | 查询报关结果         |
| 报关重推   | 失败后重新推单        |
| 报关信息修改 | 修正已推送的报关信息     |


**业务流程**：
```
1. 用户跨境电商下单（海外仓发往中国大陆消费者）
   ↓
2. 用户完成支付
   ↓
3. 机构在收到支付成功回调后，立即调【报关 API】推送支付信息
   ↓
4. 海关完成"三单比对"：订单单（机构推）+ 物流单（物流公司推）+ 支付单（机构推）
   ↓
5. 海关放行/查验 → 商品入境派送
```

**关键开发注意**：
- ⚠️ 商户必须先在**中国海关总署**完成跨境电商企业备案，取得海关注册编号
- ⚠️ 推送给海关的**金额、订单号、用户身份证号**必须与订单 + 物流完全一致，任何字段不一致都会清关失败
- 订单需携带用户**真实身份证号**（用于海关实名核验），身份证号与支付人不一致时清关可能失败
- 报关失败时通过**修改报关 API** 重推（同 `out_trade_no` + 不同 `sub_order_no`）
- 报关时机：**支付成功后立即推送**，最晚不应超过商品发货前

> 详见 [海关报关介绍](https://pay.weixin.qq.com/doc/global/v3/zh/4012354238.md)。


## 六、机构模式：请求体必带字段


| 字段                         | 含义                          | 必填条件                                                                                          |
| -------------------------- | --------------------------- | --------------------------------------------------------------------------------------------- |
| `sp_mchid`                 | 机构号（机构主体的 mchid）            | ✅ 全部接口必填（无论收款还是平台扩展类）                                                                         |
| `sp_appid`                 | 机构 AppID（机构主体名下绑定的 AppID）   | ✅ 大多数接口必填，详见各 API 文档                                                                          |
| `sub_mchid`                | 子商户号（实际收款 / 被代办的子商户）        | ✅ 全部接口必填                                                                                      |
| `sub_appid`                | 子商户 AppID                   | ⚠️ 当需要用子商户 AppID 取 `sub_openid` 时必填（JSAPI / 小程序 / 部分 APP / 委托代扣）；也可用 `sp_appid` 取 `sp_openid` |
| `sub_openid` / `sp_openid` | 用户在子商户 / 机构 AppID 下的 openid | JSAPI / 小程序 / 委托代扣 必填                                                                         |


> 详见 [📄 开发参数获取](../接入指南/开发参数获取.md) 的 `sub_appid` / `openid` 适配表。

## 七、业务规则速查（纯业务，不含开发参数）


| 规则      | 取值 / 说明                                                                                 |
| ------- | --------------------------------------------------------------------------------------- |
| 金额单位    | **最小货币单位的整数**（如美分），杜绝浮点                                                                 |
| 币种      | `amount.currency` **必填**，按子商户实际收款币种填，使用 ISO 4217 三位字母代码                                 |
| 订单有效期   | 各支付产品默认 2 小时（H5 仅 5 分钟），可通过 `time_expire` 收紧                                            |
| 退款期限    | 自交易完成日起 **1 年内**可退；超期需走特殊申请                                                             |
| 重复退款    | 同一笔订单最多 50 次部分退款，`out_refund_no` 每次唯一；同笔订单多次退款须**相隔 1 分钟**                              |
| 退款频率限制  | 错误请求 6 qps；一个月前订单退款 5000/min；同笔订单退款相隔 1 分钟                                              |
| 退款资金来源  | ⚠️ **退款只能从 CNY 资金池扣款**（即只能退当日尚未换汇的资金），CNY 池不足时返回 `NOT_ENOUGH`，需延迟重试 |
| 对账周期    | T+1 提供前一日账单；按子商户 + 币种维度对账 |
| 子商户路由   | 机构所有回调共用同一 `notify_url`，业务侧**必须按 `sub_mchid` 路由**到正确子主体，否则会出现"A 子商户的订单被 B 子商户业务处理"的串单事故 |
| 签名 / 验签 | 见 [📄 签名与验签规则](../接入指南/签名与验签规则.md)                                                      |


## 八、与境内的关键差异（一表速查）


| 维度           | 境内                                                              | 境外（机构）                                            |
| ------------ | --------------------------------------------------------------- | ------------------------------------------------- |
| API 域名       | `api.mch.weixin.qq.com`                                         | `apihk.mch.weixin.qq.com`                         |
| 路径前缀         | `/v3/pay/transactions/...`、`/v3/pay/partner/transactions/...` 等 | **统一** `/v3/global/...`                           |
| 付款码 API 版本   | 仅 V2（XML + MD5/HMAC-SHA256）                                     | **V3**（JSON + RSA2048）                            |
| H5 支付域名      | 必须 ICP 备案 + 商户后台配置                                              | **不需要 ICP 备案**，但走"H5 支付权限申请"API 流程配域名             |
| 分账产品         | 国内分账                                                            | **跨境分账**（独立资质，路径 `/v3/global/profit-sharing/...`） |
| 多币种          | 默认 CNY                                                          | `amount.currency` **必填**（ISO 4217）                |
| 语言           | 中文                                                              | 请求头可加 `Accept-Language` 控制错误信息语言                  |
| 证书与 APIv3 密钥 | 境内体系                                                            | **完全独立的境外体系**，不通用                                 |
| 错误信息字段       | 标准 `code` + `message`                                           | 境外签名错误额外带 `detail.sign_information`（更易定位）         |


## 九、调起支付差异对比


| 支付方式   | 下单产物                  | 调起方式                                                 | 产物有效期      |
| ------ | --------------------- | ---------------------------------------------------- | ---------- |
| JSAPI  | `prepay_id`           | `WeixinJSBridge.invoke('getBrandWCPayRequest', ...)` | 2 小时       |
| 小程序    | `prepay_id`           | `wx.requestPayment(...)`                             | 2 小时       |
| APP    | `prepay_id`           | 微信开放平台 OpenSDK `sendReq`                             | 2 小时       |
| H5     | `h5_url`              | 浏览器跳转链接（带 `redirect_url` 回跳）                         | **5 分钟**   |
| Native | `code_url`            | 转为二维码展示，用户手机微信扫一扫                                    | 2 小时       |
| 付款码    | 无（一步扣款）               | 商户扫枪扫用户付款码后调 API                                     | 用户付款码每分钟刷新 |
| 委托代扣   | 无（凭 `contract_id` 扣款） | 后台主动调"支付扣款"接口                                        | 由签约协议决定    |


> ⚠️ **JSAPI 与小程序的 `appId` 字段差异**：调起支付时前端传的 `appId` 取决于产品类型——
>
> - JSAPI：传**已配置 JSAPI 授权目录的 AppID**（通常是 `sub_appid`，若用机构网页授权则是 `sp_appid`）
> - 小程序：传**小程序 AppID**（即 `sub_appid` 或 `sp_appid` 中是小程序的那个）
>
> 错传会报"appid mchid not match"。详见 [📄 接入质量检查](../接入指南/接入质量检查.md)。
