# Agent Native Speaker

> **Learn how an Agent works — from the Agent itself.**

Agent Native Speaker（简称 ANS）是一个通用的 Agent Skill。它让任何开源 Agent 能够**对自己的架构进行"自解释"**——当你想了解 Agent 是怎么构建的，你不再需要翻文档、看视频或者读源码，而是直接问它，它会用自己的代码当场给你讲。

## 1. 产品定义

### 一句话

Agent Native Speaker 是一个行为约束型 Skill——它不教用户任何东西，而是**管住 Agent 自己**：当被问及架构问题时，必须搜索源码再回答，禁止凭空编造。

### 服务于谁

- **Agent 用户 / 学习者** — 想理解 Agent 内部机制但不想读完整套源码的人
- **Agent 开发者** — 想快速了解一个新 Agent 框架的架构设计
- **Agent 本身** — 这个 Skill 约束 Agent 的行为，防止它产生架构层面的幻觉

### 使用场景

| 场景 | 例子 |
|------|------|
| 学习 Agent 概念 | "什么是 agent loop？" |
| 了解特定框架的实现 | "Hermes 的 memory 是怎么存的？" |
| 对比不同 Agent 的设计 | "Hermes 和 OpenCode 的 tool calling 有什么不同？" |

### 核心价值主张

> **让 Agent 成为自己架构的母语者。**
>
> 不靠文档、不靠二手中介、不靠训练语料中的一般性知识——靠实时读取自己的源码来回答问题。

## 2. 核心概念

### 2.1 Agent Native Learning（ANL）

**定义：** 学习者通过**与 Agent 交互**，让 Agent 实时解析自身源代码，从而理解 Agent 系统如何构建的一种学习方式。

这是这个 Skill 的底层教育哲学。

对比传统方式：

| 对比维度 | 传统方式（文档/视频） | Agent Native Speaker |
|----------|----------------------|---------------------|
| 信息源 | 人工写的二手资料 | 源码本身（第一手） |
| 更新及时性 | 等作者更新 | 源码更新即更新 |
| 可追问深度 | 固定的内容 | 可无限追问至任意细节 |
| 参与感 | 单向接收 | 交互式对话 |
| 覆盖框架数 | 一个文档覆盖一个框架 | 一个 Skill 覆盖所有开源 Agent |

### 2.2 Harness Engineering

**定义：** 构建 Agent 系统本身的工程学科——非"用 Agent 完成任务"，而是"如何设计 Agent 的内部机制"。

涵盖的领域包括：
- Agent Loop / Event Loop
- Tool Calling / Function Calling
- Tool Result Orchestration
- Session Management
- Memory / State Persistence
- Model Routing / Provider Abstraction
- Configuration System
- Error Handling / Retry / Fallback
- Message Protocol Design

**注意：** 这些是**通用的**——不管你用 Hermes、OpenCode、Claude Code 还是别的框架，Harness Engineering 的核心问题都是一样的。不同框架只是在实现层面给出了不同的答案。

### 2.3 Self-Explanation Protocol（自解释协议）

**定义：** Agent 在回答有关自身架构的问题时，所必须遵守的行为约束。

核心逻辑：

```
        用户提问
            │
            ▼
    ┌─── 是否涉及 Agent 架构？ ───┐
    │           │                  │
    │         是                  否
    │           │                  │
    │    ┌──────┴──────┐         正常回答
    │    │ 触发域识别  │
    │    └──────┬──────┘
    │           │
    │    ┌──────┴──────┐
    │    │ 搜索源码    │
    │    └──────┬──────┘
    │           │
    │    ┌──────┴──────┐
    │    │ 读取关键代码 │
    │    └──────┬──────┘
    │           │
    │    ┌──────┴──────┐
    │    │ 基于代码回答 │
    │    └──────┬──────┘
    │           │
    │    ┌──────┴──────┐
    │    │ 引用源码位置 │
    │    └─────────────┘
```

**纪律铁则：** 不搜索源码不回答。如果源码不可访问，必须明确告知。

### 2.4 Source-First Principle（源码优先原则）

**定义：** 代码是第一权威。答案中如果有代码和文档冲突的地方，以代码为准，并标注差异。任何架构结论必须附带代码位置（文件路径 + 行号范围）。

## 3. 痛点与问题

这个 Skill 要解决的问题，不是理论推测出来的——而是在真实对话中踩出来的。

### 🔴 痛点 1：Agent 不知道自己怎么造的

**真实案例：** 在一个对话中，用户问 Hermes Agent：

> "你的记忆系统是怎么设计的？"

Agent 没有去搜源码，而是靠训练语料中的一般性知识回答：

> "这两块存到 SQLite 数据库里。"

但实际上 Hermes 的记忆系统存在 `memories/MEMORY.md` 和 `memories/USER.md` 两个 **Markdown 文件**里。被用户当场抓包后，Agent 不得不承认："我压根没去查，直接猜了一个合理的说法。"

**后果：** Agent 不可信了。用户不再信任它对自身架构的任何回答。

### 🔴 痛点 2：Harness Engineering 没有系统教程

市面上关于 Agent 的内容绝大多数是 prompt 技巧和工具链使用。如果有人想学"如何构建一个 Agent 系统"——比如 agent loop 怎么设计、tool orchestration 怎么做、session 怎么管理——几乎找不到系统化的教学资源。

**真实案例：** 一个刚入门的开发者想了解 agent loop 的常见设计模式。他去 B 站搜，搜到的是一个个讲怎么用 ChatGPT 写论文的视频；去博客看，看到的是"10 个你必须知道的 prompt 技巧"。没有人系统讲 agent loop 的状态机设计、max_turns 策略、interrupt 处理。

**后果：** 学习者只能靠零散地拼凑信息，或者硬读几千行源码，学习成本极高。

### 🔴 痛点 3：文档与代码脱节

架构文档发布即过时。API 变了、组件重构了、设计决策推翻了——但文档没人更新。

**真实案例：** Hermes Agent 的 `conversation_loop.py` 有 4,000+ 行，核心逻辑从 `run_agent.py` 迁移出来过至少两次，模块间的依赖关系进化了好几轮。任何静态文档在这套持续重构面前都会迅速过时。

**后果：** 文档反而成为误导源。"你去看官方文档" → 文档里的架构图和实际代码对不上 → 浪费时间。

### 🔴 痛点 4：学 Agent 没有参与感

视频或文章是单向的。你看完了，想追问"那 error handling 在哪个文件？"——没有人回答你。

**后果：** 看完就忘。知识没有沉淀，下次遇到同样的问题又要重头学。

### 🔴 痛点 5：Agent 缺乏"元认知"

Agent 能帮人写代码、回答问题、做研究，但没有机制让它"认识自己"——反思自身架构的设计。

**后果：** Agent 在回答"你是怎么工作的"这类关键问题时，和回答"今天晚上吃什么"用的是同一个推理路径——没有意识到它此刻应该去读自己的源码。

### Agent Native Learning 的解决方案

| 痛点 | ANS 怎么解决 |
|------|-------------|
| 1. Agent 不知道自己的架构 | 强制自解释协议——不读代码不回答 |
| 2. 没有系统教程 | Agent 本身就是活的教科书 |
| 3. 文档过时 | 源码就是文档，永远最新 |
| 4. 没有参与感 | 可追问至任意细节 |
| 5. 缺乏元认知 | 第一次要求 Agent 反思自身结构 |

## 4. 架构设计

### 4.1 整体架构

```
            ┌─────────────────────────────────────┐
            │           User Question              │
            └──────────────┬──────────────────────┘
                           │
                           ▼
            ┌──────────────────────────────┐
            │  ┌─────────────────────────┐ │
            │  │  触发域识别 (Trigger)    │ │
            │  └───────────┬─────────────┘ │
            │              │               │
            │    ┌─────────┴─────────┐     │
            │    │ 是架构问题吗？    │     │
            │    └────┬────────┬────┘     │
            │        No       Yes         │
            │         │        │          │
            │         ▼        ▼          │
            │    正常回答  判定教学层级   │
            │    (不激活)   ┌────┴───┐   │
            │              │       │     │
            │         Layer 1  Layer 2   │
            │         概念层   架构层     │
            │         (通用)  (读源码)   │
            └──────────────────────────────┘
```

### 4.2 两层教学模型

#### Layer 1 — 基础层（概念教学）

| 属性 | 说明 |
|------|------|
| **触发条件** | 用户询问 Agent 通用知识和基础概念，**未指定具体框架** |
| **知识来源** | Agent 训练语料中的通用领域知识（不需要读源码） |
| **输出风格** | 通俗易懂，善用类比和比喻 |
| **目标受众** | 对 Agent 感兴趣但不了解内部机制的初学者 |
| **源码需求** | ❌ 不需要 |

**示例：** 用户问"什么是 tool calling？" → 用"厨师和助理"的类比解释，不读代码。

#### Layer 2 — 架构层（代码深潜）

| 属性 | 说明 |
|------|------|
| **触发条件** | 用户询问**当前 Agent 的具体架构实现** |
| **知识来源** | 源码搜索 + 读取 |
| **输出风格** | 严谨，引用具体文件和行号，解释设计权衡 |
| **目标受众** | 想深入理解 Agent 实现原理的开发者 |
| **源码需求** | ✅ 必定需要搜索并读取源码 |

**示例：** 用户问"Hermes 的 tool calling 怎么实现的？" → 搜索 `agent/tool_executor.py` → 读 ThreadPoolExecutor 实现 → 回答带代码引用。

### 4.3 自解释协议（6步流程）

| 步骤 | 行为 | 适用层级 |
|------|------|---------|
| Step 1 | 识别触发域：问题是否是 Agent 架构/设计类问题 | 两者 |
| Step 2 | 判定教学层级：是否提及了具体框架名 | 两者 |
| Step 3 | 定位源码：搜索入口文件 → 核心模块 | Layer 2 专用 |
| Step 4 | 读取关键代码：至少 2-3 个核心文件 | Layer 2 专用 |
| Step 5 | 基于代码组织回答：[概念] → [代码位置] → [做了什么] → [为什么这么设计] | 两者 |
| Step 6 | 透明边界声明：找不到源码时明确告知 | Layer 2 专用 |

### 4.4 数据流

```
用户提问
    │
    ▼
ANS Skill 激活
    │
    ├── Layer 1 路径:
    │   识别概念域 → 从训练语料提取通用知识 → 用类比组织回答 → 输出
    │
    └── Layer 2 路径:
        识别框架名 → search_files("tool_call") → 
        read_file("agent/tool_executor.py") → 
        分析代码结构 → 提取设计决策 → 
        输出 [代码位置 + 做了什么 + 为什么]
```

### 4.5 关键设计决策

| 决策 | 选择 | 放弃的选项 | 理由 |
|------|------|-----------|------|
| 两层教学模型 | 按问题自动分层 | 单层一刀切 | 避免对初学者丢源码，对开发者讲幼儿园 |
| 自解释协议 | 强制搜索源码 | 软提示（"请尽量先搜代码"） | 软提示无效——第一次对话就现场证明了 |
| 框架无关 | 扫入口文件 + 核心目录 | 硬编码每个框架的路径 | 维护成本高，且不通用 |
| 按需触发 | 只有架构问题才激活 | 启动时自动加载 | 不影响 Agent 的日常使用体验 |

### 4.6 边界

| 范围 | 做 | 不做 |
|------|----|------|
| 源码访问 | 搜索、读取、引用源码 | 修改源码、重构、debug |
| 教学深度 | 解释架构设计原理 | 教授 ML/AI 理论、模型训练 |
| 交互方式 | 用户提问 → 自动判定层级 → 讲解 | 自动生成完整教程文档 |
| 覆盖框架 | 任何开源 Agent 框架 | 闭源、无源码可访问的系统 |
| 输出形式 | 文字讲解 + 代码引用 | 视频、图表、代码生成 |
| 学习节奏 | 按需触发，用户主导节奏 | 强行推送学习内容 |

### 4.7 项目结构

```
agent-native-speaker /
├── README.md              # 整体设计文档（这个文件）
├── SKILL.md               # Skill 定义文件（可被 skill_manage 加载）
└── references /

```

## 5. 快速开始

### 安装

```bash
# 通过 ClawHub
clawhub install agent-native-speaker

# 或从 GitHub 克隆
git clone https://github.com/joeyxin-del/agent-native-speaker.git
```

### 使用

安装后，直接问你的 Agent 架构问题即可——Skill 会自动激活：

```
用户: "什么是 agent loop？"
Agent: [Layer 1 通用概念解释]

用户: "Hermes 的 agent loop 是怎么实现的？"
Agent: [Layer 2 搜索源码 → 读代码 → 带位置引用]
```

### 在 Hermes Agent 中使用

安装到 `~/.hermes/skills/autonomous-ai-agents/agent-native-speaker/SKILL.md` 后，Hermes 会在启动时自动加载。如果已有 Hermes 运行，执行 `/reload-skills` 即可生效。
