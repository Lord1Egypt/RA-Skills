# 惠迈Skill发布工作流SOP

> 版本: v1.0 | 更新: 2026-04-29
> 适用: 所有ClawHub Skil发布/更新操作

---

## 一、角色与职责

| 角色 | 职责 | 在当前架构中的对应 |
|------|------|-------------------|
| 🏗️ **执行者** | 编写代码、创建文件、执行发布命令 | 香港工程师hk001 或 cn001 |
| 🔍 **审核者** | 逐条核对指令与执行结果，检查漏项 | 惠迈高级工程师 |
| 🚀 **放行人** | 最终确认后放行发布 | 老王 |

> **当前情况**: 单人或两人协作时，执行者完成工作后必须进行自审，自审通过后交由放行人确认。

---

## 二、执行阶段规范

### 2.1 指令接受

收到任务后，执行者必须做三件事：
1. ✅ **拆解为明确条目** — 把模糊的任务拆成可执行的单项
2. ✅ **区分"指令"与"建议"** — 标注每项的来源性质
3. ✅ **确认范围边界** — 确认"只做A"还是"从A推广到同类也做"

### 2.2 执行检查清单

| # | 检查项 | 说明 |
|:-:|--------|------|
| 1 | 是否只改了指名要改的内容？ | 禁止无授权范围扩展 |
| 2 | 修改是否影响关联文件？ | 如改name要同步改description和meta |
| 3 | 修改是否符合统一标准？ | 校准框架、温情模式、联系邮箱 |
| 4 | 不同skill间是否有不一致？ | slug、版本号、tag格式统一 |
| 5 | 是否需要回退的备份？ | 大范围修改前确保可回滚 |

---

## 三、审核阶段规范 ★核心

### 3.1 审核的两重标准

每次审核必须**同时**对照两个维度：

```
维度A：是否符合下达的指令？
    └── 逐条核对，没说的不擅自加
维度B：是否符合已有的SOP规范？
    └── 校准框架、温情模式、邮箱等基础设施标准
```

### 3.2 审核流程

```
第1步：读取原始指令 → 分解为检查条目（只做指令内的事）
第2步：读取SOP规范 → 分解为检查条目（基础设施标准）
第3步：逐文件比对 → 逐项标记通过/不通过
第4步：汇总问题 → 分类紧急程度
第5步：修正问题 → 重新审核修正项
第6步：输出审核报告 → 交给放行人
```

### 3.3 审核检查表

```
【指令符合性】☐ 所有修改是否在指令范围内？
               ☐ 是否有多余的未经授权的修改？
               ☐ 是否遗漏了指令要求的修改？
【SOP规范性】 ☐ 校准框架说明是否存在？
               ☐ 温情模式说明是否存在？
               ☐ 联系邮箱是否存在？
               ☐ 版本号是否正确？
               ☐ slug是否和目录名一致？
               ☐ 描述是否和名称匹配？
               ☐ 是否包含"无需API Key"声明？
               ☐ 是否去掉了"基于DeepSeek v4"类描述？
【文件完整性】 ☐ SKILL.md ✓
               ☐ _meta.json ✓
               ☐ 发布时间戳更新 ✓
               ☐ 已发布版本与本地版本号一致 ✓
```

## 四、放行阶段

### 4.1 放行条件

只有**全部满足**以下条件才能放行：
1. ✅ 审核发现问题已全部修复
2. ✅ 放行人（老王）确认
3. ✅ 非紧急发布 或 紧急发布已获特批
4. ✅ 检查当前时间是否在合理工作时间段内

### 4.2 紧急发布规则

例外情况需要特批：
- 线上严重 Bug 需要紧急修复 → 优先处理，事后补充备案
- 特批人：老王

## 五、持续改进

每次发布后，如果发现：
- 流程有遗漏 → 更新此SOP
- 发现新模式 → 更新准则
- 发现边界 → 更新效能参数

**积跬步，至千里。** 🏔️

## 附录：常见错误列表

| # | 错误类型 | 举例 | 预防措施 |
|:-:|----------|------|----------|
| 1 | 超范围执行 | 用户只让改A，却改了A和B | 审核时核对指令范围 |
| 2 | 关联文件遗漏 | 改了SKILL.md没改_meta.json | 审核时检查文件完整性 |
| 3 | 本地版本不匹配 | 本地记录v0.1.0，实际发布v0.1.1 | 发布后同步更新本地meta |
| 4 | 批量遗漏 | 批量改了7个，漏了其中1个 | 审核时逐文件逐项检查 |
| 5 | 名实不符 | 名称叫A，描述里说是B | 审核时交叉对比名称和描述 |
