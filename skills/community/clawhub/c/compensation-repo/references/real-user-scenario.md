# 定薪与申报检查助手：真实用户场景

## 这个 skill 最适合从哪里开始

第一次打开时，优先从这两个场景开始：

1. `做申报前检查`
2. `做定薪判断`

它们分别承担两种完全不同但都很真实的使用心智：

1. `做申报前检查`
   这是“月底快报了，先帮我排雷”
2. `做定薪判断`
   这是“这个价到底能不能给，怎么解释更稳”

## 推荐第一生产场景

`月末准备做个税、社保、公积金申报，HR 想先做一轮检查`

这非常真实，因为：

1. 月度薪酬数据总会有缺口
2. 申报前最后一轮检查非常耗 HR 时间
3. 真正让 HR 紧张的不是“填表”，而是“怕申报失败或错报”

### 用户第一次可以这样说

1. `这是本月申报名单，帮我看看有没有高风险问题。`
2. `个税、社保、公积金申报前先帮我排一下雷。`
3. `哪些人缺字段，哪些人基数或主体有问题，给我一个待办清单。`

### 用户至少要给什么

1. 本月薪酬或申报名单
2. 个税申报字段
3. 社保、公积金申报字段
4. 员工状态、法人主体和缴纳地信息

### 用户最希望拿到什么

1. 能不能直接报
2. 哪些人最危险
3. 缺了什么字段
4. 今天先追回什么
5. 给内部同事的提醒话术

### 结果页要优先按这个顺序展示

1. `能不能直接报`
2. `高风险问题`
3. `按人列出的缺口`
4. `今天先处理什么`
5. `内部提醒或追回话术`

## 推荐第二生产场景

`准备发 offer 或做关键岗位定薪，HR 想先看 band、市场和内部公平`

这也非常真实，因为：

1. 薪酬判断并不是拍脑袋
2. 定薪最难的是把市场、band、候选人期望和内部公平放在一起看
3. 这也是最适合做“专业感 + 传播感”的薪酬场景

### 用户第一次可以这样说

1. `这个候选人期望 40k，到底能不能给？`
2. `帮我看看这个 offer 是在 band 里什么位置。`
3. `把市场分位、内部参考和定薪建议整理成一版摘要。`

### 用户至少要给什么

1. 岗位级别和 band
2. 市场调研分位点
3. 候选人当前或期望薪资
4. 内部同岗同级参考
5. 可选的预算或审批口径

### 用户最希望拿到什么

1. 建议怎么定
2. 为什么这么定
3. 有没有内部公平风险
4. 怎么和业务解释
5. 还需要谁确认

## 这个 skill 背后的真实薪酬语境

它要默认理解这些情况：

1. 申报前数据往往来自多个表，不会天然干净
2. band、市场和内部公平经常是分开存放的
3. 薪酬负责人最怕的不是“慢一点”，而是“错报”和“解释不清”

## 哪些地方不要装得太确定

如果出现这些情况，要主动收窄判断：

1. 城市口径未提供
2. 只有期望薪资，没有当前薪资或总包口径
3. band 规则缺少中位值、级差或预算信息
4. 个税、社保、公积金数据口径彼此不一致
5. 员工状态或法人主体信息不完整

此时要明确写出：

1. `当前判断依据不足`
2. `哪些结论只是初筛`
3. `还需要谁确认`
4. `建议先补什么数据`
