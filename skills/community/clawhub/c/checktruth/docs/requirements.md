# 别瞎说（BieXiaShuo）需求规格

## 1. 项目背景

AI 回答看起来自信但经常包含错误。用户需要一个工具来**自动验证 AI 回答的正确性**，并给出可信度评分。

## 2. 功能需求

### 2.1 核心功能：回答验证

**输入**：
- `question`：用户提问（字符串）
- `answer`：待验证的 AI 回答（字符串）
- `reference`（可选）：用户提供的参考来源（字符串或URL）

**输出**：
- `verdict`：最终判定（`true`/`false`/`partial`/`unknown`）
- `score`：可信度评分（0-100）
- `reason`：判定理由（字符串）
- `details`：详细验证过程（结构化数据）

### 2.2 验证流程（MVP）

1. **多源答案获取**：向多个 AI 模型询问相同问题，获取参考答案
2. **原子事实分解**：将待验证回答和参考答案都分解为原子事实（atomic facts）
3. **交叉验证**：对比原子事实，判断一致性
4. **内部一致性检测**：检查回答本身是否自相矛盾
5. **综合评分**：基于以上结果给出最终评分

### 2.3 支持的验证维度

| 维度 | 说明 | 优先级 |
|------|------|--------|
| 事实准确性 | 回答中的事实是否与其他AI答案一致 | P0 |
| 内部一致性 | 回答自身是否自相矛盾 | P0 |
| 答案等价性 | 不同表述是否指向相同答案（参考xVerify） | P1 |
| 逻辑推理 | 推理过程是否合理 | P2 |

### 2.4 输入约束

- `question` 长度：≤ 2000 字符
- `answer` 长度：≤ 5000 字符
- 支持的问答类型：
  - 事实型（谁/何时/何地/多少...）
  - 定义型（什么是...）
  - 解释型（为什么/如何...）
  - 推理型（分析/比较...）

## 3. 非功能需求

### 3.1 性能

- 单次验证耗时：≤ 30 秒（MVP目标）
- 并发：不限制（依赖外部API配额）

### 3.2 可靠性

- 外部API失败时给出明确错误提示
- 部分API失败时降级处理（用可用结果继续）

### 3.3 可扩展性

- 易于添加新的AI数据源
- 易于调整验证策略

## 4. 方案A架构（纯Skill实现）

```
用户输入
   │
   ▼
┌─────────────────────────────┐
│     BieXiaShuo Skill       │
│                             │
│  1. 调用多个AI API获取参考答案 │
│  2. 原子事实分解（本地逻辑）  │
│  3. 交叉验证（调用AI判断）   │
│  4. 一致性检测（调用AI判断）  │
│  5. 综合评分并输出结果       │
└─────────────────────────────┘
   │
   ▼
格式化的验证报告
```

**关键决策**：
- 不依赖后端服务，所有逻辑在Skill内完成
- 通过WorkBuddy已有的LLM调用能力完成AI判断
- 原子事实分解使用prompt工程实现（不依赖FActScore的完整pipeline）

## 5. 验收标准

- [ ] 能正确识别明显错误的事实陈述
- [ ] 能检测到回答内部的自相矛盾
- [ ] 对无法验证的内容给出 `unknown` 而非随意猜测
- [ ] 输出格式清晰，便于用户理解验证过程
