# 通用Agent适配指南

## 适用平台

### 支持的操作系统
| 系统 | 兼容性 | 注意事项 |
|------|--------|---------|
| **Windows 10/11** | ✅ 完全支持 | 路径使用 `C:\` 或 `C:/`；换行符 CRLF；字体默认有宋体/黑体 |
| **macOS** | ✅ 完全支持 | 路径使用 `/Users/`；换行符 LF；可能需要安装中文字体 |
| **Ubuntu/Debian** | ✅ 完全支持 | 路径使用 `/home/`；换行符 LF；需安装中文字体：`apt install fonts-noto-cjk` |
| **其他Linux** | ✅ 支持 | 需确保中文字体和PDF/Word处理库可用 |

### 跨平台注意事项
- **路径分隔符**：Agent 应统一使用正斜杠 `/`，所有系统均支持
- **中文字体**：Windows自带宋体/黑体；Mac/Linux可能需要安装 `fonts-noto-cjk`
- **换行符**：最终文档换行符应与目标系统一致（Windows: CRLF, Mac/Linux: LF）
- **文件编码**：统一使用 UTF-8 编码（避免中文乱码）

### 支持的Agent平台
任何支持以下基础的LLM Agent：
- 文件读取能力（PDF/Word）
- 文件写入能力（输出文档）
- 长上下文处理（≥64K tokens）
- 文本生成能力

## 适配方式

### 方式一：纯Prompt注入（零工具依赖）

直接将以下内容注入到Agent的系统提示词中：

```markdown
你现在是「标书制作专家(bid-document-maker)」。
你的职责、工作流和规范定义在以下文件中，请严格遵循：

(1) 角色定义：prompts/system-prompt.md
(2) 工作流：workflows/pipeline.yaml（按 stages 顺序执行）
(3) 每阶段使用对应的 prompts/ 目录下的提示词
(4) 输出格式参考 templates/ 和 schemas/
(5) 格式规范以 templates/format-rules.md 为准
```

### 方式二：分步指令模式

对于不支持系统提示词注入的Agent，使用以下分步指令：

**Step 1 — 加载角色**
> 请以"资深投标顾问"身份工作，精通中国招投标法规，有10年标书编制经验。

**Step 2 — 读取文件**
> 请读取 [path/to/tender.pdf] 文件并解析招标信息。
> 使用 prompts/01-parse-tender.md 中的模板提取结构化信息。
> 注意：Windows路径用 `C:/` 格式，Mac/Linux用 `/Users/` 或 `/home/` 格式。

**Step 3 — 分析策略**
> 基于解析结果，使用 prompts/02-strategy-and-outline.md 分析评分策略并生成大纲。
> 让用户确认大纲后再继续。

**Step 4 — 编写内容**
> 使用 prompts/03-write-section.md 逐章编写标书内容。
> 注意：每个章节独立生成，内容要评分导向、量化具体、有证据支撑。

**Step 5 — 质检**
> 使用 prompts/04-quality-check.md 进行全面质量检查。
> 生成质检报告，列出所有🔴错误、🟡警告、🟢建议。

**Step 6 — PDCA自动改进**
> 使用 prompts/05-pdca-improvement.md 执行最多3轮改进循环：
> - 第1轮：修复所有🔴错误
> - 第2轮：修复所有🟡警告
> - 第3轮：全文一致性打磨
> 完成后再交付用户终审。

### 方式三：工具链模式

若Agent支持工具调用（Function Calling / Tool Use），可构建如下工具链：

```
读取文件 → 解析结构化信息 → 生成大纲 → 逐章写入 → 质检 → 输出最终文档
```

参考 `workflows/pipeline.yaml` 中的 stages 顺序和传递关系。

## 能力要求矩阵

| 能力 | 必需级别 | 备注 |
|------|---------|------|
| PDF解析 | 推荐 | 若无此能力，需用户自行提取文本后提供 |
| Word读取 | 推荐 | 同上 |
| 长上下文 | 必需 | 招标文件通常20-100页，需≥64K context |
| Markdown输出 | 必需 | 标书中间格式推荐Markdown |
| JSON输出 | 推荐 | 结构化解析阶段需要 |
| 代码执行 | 可选 | 可用于图表生成、字数统计等 |
| 跨平台路径处理 | 必需 | 自动适应 Windows/Mac/Linux 路径格式 |

## 质量保证（6阶段体系）

| 阶段 | 保证机制 | 产出 |
|------|---------|------|
| 1. 解析 | 结构化信息提取 + 完整性评估 | 解析报告 |
| 2. 策略 | 权重计算 + 资源缺口识别 | 策略报告 |
| 3. 大纲 | 评分项逐条对应验证 | 评分覆盖大纲 |
| 4. 写作 | 量化公式 + 证据标注 | 各章节草稿 |
| 5. 质检 | 6维度全面检查 | 质检报告（含问题分级） |
| **6. PDCA** | **3轮闭环改进（错误→警告→打磨）** | **改进报告 + 最终文档** |

即使Agent能力有限，以下方法可保证输出质量：
1. **分步确认**：每个阶段输出让用户确认后再继续
2. **模板驱动**：使用 templates/ 中的模板规范输出格式
3. **检查清单**：每阶段都有验证清单（见 prompts/ 各文件末尾）
4. **PDCA兜底**：阶段6的自动改进会修复大部分问题，减少人工审核负担
5. **增量输出**：长篇标书分章节输出，降低单次输出压力
