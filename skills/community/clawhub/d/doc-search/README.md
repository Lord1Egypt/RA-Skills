# doc-search

零外部依赖的本地文档库搜索方案，结合 BM25 倒排索引、LLM 查询扩展与 grep 精确定位。

## 原理

```
用户输入
  ↓
[LLM] 查询扩展：同义词 + 中英文变体 + 相关术语
  ↓
[BM25] 倒排索引查询 → Top-K 文件（毫秒级）
  ↓
[grep -ni] 对 Top-K 文件精确定位行号
```

### BM25 倒排索引

索引构建时对每个文档分词（英文按词、中文按 bigram/trigram），建立 `token → [{文件, 词频}]` 的倒排表，并预计算 BM25 所需的文档长度和语料统计。

查询时只需查找命中 token 的 posting list，对候选文档打分排序，复杂度是 `O(查询词数 × 平均命中文档数)`，与语料库总大小无关。

索引默认保存在 `<docs_dir>/.doc-search/index.json`，与文档库放在一起，便于迁移和版本管理。采用 mtime+size 指纹做增量更新，只重新索引变更文件。

### LLM 查询扩展

在 BM25 查询前，由 LLM 将用户输入扩展为多个术语变体：

```
"获取音色列表" → "获取 查询 list voice 音色 tts ListVoice 音色列表"
```

这一步在 skill workflow 内完成，无需额外模型调用（LLM 本身就在运行）。

### grep 精确定位

BM25 返回文件粒度的 Top-K 结果，再对这几个文件执行 `grep -ni`，获得精确行号。grep 的范围从整个语料库缩小到 3-5 个文件，速度极快。

## 对比纯 grep

| | 纯 grep | doc-search |
|---|---|---|
| 搜索范围 | 全量文件逐行扫描 | 先 BM25 缩小到 Top-K，再 grep |
| 速度（万级文件） | 慢（秒级~分钟级） | 快（BM25 毫秒，grep 仅扫少量文件） |
| 同义词/近义词 | 不支持 | LLM 扩展查询词覆盖 |
| 跨语言查询 | 不支持 | LLM 自动生成中英文变体 |
| 拼写容错 | 不支持 | LLM 纠错后扩展 |
| 精确行号 | 支持 | 支持（最终仍走 grep） |

---

## 对比纯 RAG（向量检索）

| | 纯 RAG | doc-search |
|---|---|---|
| 外部依赖 | embedding 模型 + 向量数据库 | 零依赖（Python stdlib） |
| 语义理解 | 强（向量相似度） | 中（BM25 + LLM 扩展补偿） |
| 技术文档精确术语 | 一般（向量可能漂移） | 好（精确词匹配 + BM25 TF-IDF） |
| 精确行号定位 | 不支持（chunk 粒度） | 支持（grep 二次定位） |
| 索引构建成本 | 高（需调用 embedding API） | 低（本地分词，无网络请求） |
| 内容过多时效果 | 下降（top-K chunk 质量不稳定） | 稳定（BM25 对长尾词仍有效） |
| 可解释性 | 低（黑盒相似度） | 高（可见 BM25 分数和命中词） |

### 为什么技术文档不需要向量语义

技术文档的特点是术语一致性高（函数名、配置项、API 名称都是精确词汇），语义漂移少。BM25 在这类场景下能达到向量检索 85-90% 的效果，而 LLM 查询扩展补上了剩余的同义词/跨语言短板，整体方案在技术文档场景下与 RAG 效果相当，且更轻量、更快、更可控。
