## Step 5.3 补丁 — 业务对象新增要求

在 shared/step5.3-business-objects.md 的标准ER基础上：

**强制包含以下SaaS标准对象**：

```
TENANT {
  string id PK
  string name          -- 企业名称
  string code          -- 租户标识码（唯一）
  enum status          -- 启用/停用/试用
  json feature_flags   -- 功能模块开关
  datetime created_at
}

TENANT_CONFIG {
  string id PK
  string tenant_id FK
  string module        -- 配置所属模块
  string config_key    -- 配置项键名
  json config_value    -- 配置项值（支持复杂结构）
  datetime updated_at
}
```

**所有主要业务表必须包含**：
- `tenant_id FK` — 行级隔离键，所有查询必须带此过滤条件

## Step 5.5 补丁 — 字段决策新增必决策项

在 shared/step5.5-field-decisions.md 的标准决策项基础上，强制新增：

```
[D-SaaS-001] 字段配置化粒度
场景：租户配置工单字段的显示/必填时，粒度是「模块级」还是「服务目录级」
备选：
  A) 模块级 — 同一模块所有场景用同一套字段配置
  B) 服务目录级 — 不同服务目录可有不同字段配置（更灵活，改造量更大）
  C) 两级混合 — 模块级为基础，服务目录级可覆盖
建议：C，理由：不同服务类型（咨询/投诉/救援）的字段需求差异大

[D-SaaS-002] 租户字典与平台字典的继承关系
备选：
  A) 完全独立 — 租户字典与平台字典互不影响
  B) 继承覆盖 — 平台提供默认字典，租户可覆盖
  C) 平台只读 — 平台字典租户不可修改，只能新增租户专属字典
建议：C，理由：平台标准枚举需要统一，防止租户改乱

[D-SaaS-003] 流程提醒节点与服务目录的绑定粒度
场景：不同服务目录需要不同的提醒流程（外呼模板/提醒时机/通知对象）
备选：
  A) 服务目录节点级 — 每个目录节点绑定一套提醒流程
  B) 业务类型级 — 按业务类型（咨询/投诉/服务请求）绑定
  C) 独立流程模板 — 配置提醒流程模板，服务目录选择引用哪个模板
建议：C，理由：模板复用性更高，避免重复配置
```

| 字段名 | 类型 | 来源 | 必填 | 默认值 | 约束 | 备注 | **租户隔离** |
|--------|------|------|------|--------|------|------|------------|
| tenant_id | string | 系统自动 | ✅ | 当前租户 | 所有查询必须带此过滤 | 隔离键 | ✅ 行级隔离 |

**租户隔离标注规则**：
- ✅ 行级隔离：该字段所在表必须在所有查询中加 `WHERE tenant_id = ?`
- 🔧 配置化：该字段的值来自租户配置，不同租户可能不同
- 📋 平台标准：该字段使用平台统一枚举，租户不可修改

**新增必须详述的模块**（在原有业务模块之外）：

每个以下模块必须按7层格式完整输出：

### Step 6 补丁 — 配置类页面形态判断规则（🆕）

在设计 Layer 3 界面形态时，必须先判断页面类型，再选择对应的设计模式：

#### 规则1：全量展示 vs 增量列表

| 配置对象特征 | 页面形态 | 典型场景 |
|------------|---------|---------|
| **有限枚举集合**（对象数量固定且预先已知，如业务类型、服务目录、字典枚举） | **全量展示 + 配置状态区分**：所有对象都显示在列表中，用「已配置/未配置」或颜色区分状态；未配置行提供「去配置」入口 | 跟进坐席配置、审核人配置、各业务类型的规则绑定 |
| **用户自由创建**（对象数量不固定，由用户决定创建几个） | **增量列表 + 新建按钮**：列表只展示已创建的，提供新建入口 | 规则模板、字段模板、自定义角色 |

**全量展示模式的 Layer 3 必须包含**：
```
┌──────────────────────────────────────────────────────┐
│ 页面标题               已配置 X 个 · 未配置 Y 个     │
├──────────────────────────────────────────────────────┤
│ 配置对象  配置状态   当前配置内容   操作              │
│ 对象A    ✅ 已配置   [具体配置]     编辑 / 清除       │
│ 对象B    ✅ 已配置   [具体配置]     编辑 / 清除       │
│ 对象C    ○ 未配置   [系统默认行为说明]  去配置        │  ← 灰色背景区分
│ 对象D    ○ 未配置   [系统默认行为说明]  去配置        │
└──────────────────────────────────────────────────────┘
```

---

#### 规则2：只读数据来源页设计规范

当模块数据来源为**外部系统同步**且**系统内不可写入**时（如坐席组来源外呼平台），必须遵循以下设计规范：

**强制包含元素**：
- 页面顶部橙色提示条，文案格式：「数据来源：[外部系统名称]，只读。变更请在 [外部系统名称] 操作，同步后此处自动更新」
- 右上角提供「手动同步」按钮（异步操作，Toast 提示「同步中，请稍后刷新」）
- 显示上次同步时间

**禁止出现的元素**：
- 「新增」「删除」「编辑」操作按钮
- 表单或输入框
- 任何可能引导用户修改数据的控件

**推荐布局**：左侧列表 + 右侧详情（主从布局），点击列表项右侧展示详情，不使用独立「查看」按钮。右侧默认显示空状态引导「← 点击左侧 [对象名] 查看详情」。

---

#### 规则3：决策型枚举字段设计规范

当一个配置字段存在「**显式配置某个值**」与「**不配置（使用系统默认行为）**」两种状态，且这两种状态在业务上有明确语义差异时，**必须将其设计为显式枚举字段**，不能依赖系统兜底逻辑代替用户决策。

**判断标准**：

```
问：如果用户看不到这个字段，是否能明确知道系统会如何处理？
→ 能明确知道（只有一种合理行为）→ 可以使用系统默认兜底
→ 有多种合理行为，或用户需要主动选择 → 必须设计为显式枚举字段
```

**典型场景**：

| 场景 | 错误设计 | 正确设计 |
|------|---------|---------|
| 工单跟进人配置 | 不配置坐席组时系统默认用创建人（隐式兜底） | 显式字段：「无需指定技能组（负责坐席跟进）」/「指定坐席组跟进」，用户必须主动选择 |
| 审核层级配置 | 不配置则跳过审核 | 显式字段：「无需审核」/「1级审核」/「2级审核」 |

**决策型枚举字段在 Layer 3 中必须用卡片单选形式呈现**（而非普通下拉），每个选项配有描述文案，说明选择后的系统行为：

```
跟进类型 *
┌────────────────────────────────────────┐
│ ○  无需指定技能组                      │
│    当前工单的负责坐席即为跟进人         │
└────────────────────────────────────────┘
┌────────────────────────────────────────┐  ← 已选中时蓝色边框+浅蓝背景
│ ●  指定坐席组跟进                      │
│    工单创建时从指定坐席组按策略分配     │
└────────────────────────────────────────┘
```

**关联字段的显隐规则**：
- 选择某个枚举值后才需要的字段，必须在未选中时**完全隐藏**（不是置灰），选中后才展示
- Layer 5 字段交互生命周期中必须明确写出每个枚举值对应的显隐状态变化

## Step 6.5 补丁 — RBAC三层结构强制要求


| 角色 | 可见数据范围 |

| 租户管理员 | 本租户所有数据 |
