# 用了 100 次 Superpowers 后，我发现大多数人都用错了

> 本文作者：ai-acheng | 首发：赛博鸭子技术周刊

---

## 🔍 一个扎心的发现

这段时间用 Superpowers 做了十几个项目，从简单的工具脚本到复杂的 Next.js 应用。

发现一个很有趣的现象：

**90% 的人用 Superpowers，最后都卡在同一个地方 —— 代码"写完了"，但不敢交付。**

子 agent 拍胸脯说"功能全部实现了"，你跑一下发现：
- TypeScript 飙红 20 几个类型错误
- package.json 里躺着 5 个根本没用的依赖
- .env.example 只有三行配置，连个注释都没有
- README 写得像天书，新人根本跑不起来

Superpowers 很擅长"把代码写出来"，但它不负责"把代码写好"。

这就是为什么我做了 **Harness Dev Standards**。

---

## 🎯 一个负责"过程"，一个负责"结果"

很多人问我：这俩不都是开发工具吗？有啥区别？

我举个最简单的例子：

### 场景 1：你说"帮我做个用户登录系统"

**Superpowers 是这样干活的：**

1. 🤔 先问你 5 个澄清问题：要不要验证码？要不要记住登录？密码强度要求？
2. 📋 给你 3 个技术方案：NextAuth / Lucia / 自研，附优缺点对比
3. 🔨 拆成 8 个小任务，每个任务 2-5 分钟
4. 🤖 派 8 个子 agent 逐个实现，每个写完都有双重评审
5. ✅ 最后交付："功能全部完成，你测一下"

**这个过程中，它根本不关心：**
- 代码风格好不好
- 依赖干不干净
- README 写没写清楚
- .env 配置有没有注释

**它只保证「过程正确」，不保证「结果合格」。**

---

### 场景 2：你说"这个登录系统写完了，帮我检查一下"

**Harness Dev Standards 是这样干活的：**

1. 🔍 运行 `quality-scan.sh`
2. ❌ TypeScript：发现 3 个类型错误，附修复方案
3. ⚠️  依赖：发现 2 个未使用的包，可以安全移除
4. ❌ 配置：.env.example 缺少 2 个配置项的注释
5. ✅ 构建：通过
6. 📋 对照 6 道门禁逐项确认，给出最终交付评分

**这个过程中，它根本不关心：**
- 你是花 1 天还是 1 周写的
- 你是用 TDD 还是想到哪写到哪
- 你拆了多少个任务，用了多少个子 agent

**它只保证「结果合格」，不关心「过程怎么来的」。**

## 📊 一张表看懂本质区别

| 对比项 | Superpowers | Harness Dev Standards |
|-------|------------|----------------------|
| **核心定位** | 🎯 开发工作流框架 | 🛡️ 质量标准体系 |
| **回答的问题** | 怎么开发？ | 什么叫做好？ |
| **关注点** | 过程正确 | 结果正确 |
| **执行者** | 子 agent 分工协作 | 当前 agent 一键检查 |
| **强制程度** | 流程强制，不能跳步 | 标准强制，必须通过 |
| **输出物** | 可运行的功能代码 | 质量检查报告 + 修复建议 |
| **使用阶段** | 开发前 + 开发中 | 开发后 + 交付前 |
| **运行时间** | 5~30 分钟 | < 1 分钟 |

## 💡 它们是最佳拍档，不是竞争对手

我知道很多人会问：那我应该用哪个？

**成年人不做选择，两个都要。**

这是我现在的标准开发流程：

```
用户需求 → Superpowers（组织开发过程） → 代码写好了 → Harness Dev Standards（质量检查） → 交付
```

### Step 1: Superpowers 管"做对的事"
- 确保需求是清晰的，不会做着做着跑偏
- 确保技术方案是经过论证的，不会上来就硬写
- 确保代码是按 spec 实现的，不会漏功能

### Step 2: Harness 管"把事做对"
- 确保代码质量是达标的，类型全对
- 确保依赖是干净的，没有多余的包
- 确保交付是标准化的，README 能看懂

**Superpowers 让你"不会做错事"，Harness 让你"不会把事做坏"。**

## 🚀 为什么每个开发者都需要一套质量标准

我见过太多项目：
- 功能都能用，但没人敢改代码
- 依赖一大堆，根本不知道哪个在用
- 新人上手要折腾三天才能跑起来
- 上线前才发现配置漏了，手忙脚乱

这些问题，都不是"功能问题"，而是"质量问题"。

Superpowers 解决的是"产能问题" —— 让你更快地写出更多代码。

Harness 解决的是"质量问题" —— 让你写的代码敢交付、敢维护、敢给别人用。

**没有质量的产能，都是负债。**

---

## 📦 开箱即用的 6 道质量门禁

Harness Dev Standards 把我踩过的坑，总结成了 6 道硬性门禁：

| 门禁 | 检查内容 |
|-----|---------|
| **需求门禁** | 需求完整清晰，无模糊点 |
| **架构门禁** | 文件结构标准化，依赖最小化 |
| **编码门禁** | 语法类型正确，命名规范 |
| **依赖门禁** | 依赖完整无冗余，无安全漏洞 |
| **环境门禁** | .env.example 完整，敏感信息保护 |
| **交付门禁** | 项目可构建，README 完整 |

每一道门禁都有对应的自动检查脚本和自动修复策略库。

不用你记，运行 `./quality-scan.sh`，一分钟出结果。

## 🎉 最后说两句

AI 写代码的时代，效率已经不是问题了。

真正的问题是：
- 你敢把 AI 写的代码直接上生产吗？
- 三个月后，你还敢改这段代码吗？
- 新人接手，能在 30 分钟内跑起来吗？

Superpowers 给了你踩油门的能力。
Harness Dev Standards 给了你踩刹车的底气。

**一个让你跑得快，一个让你跑得远。**
