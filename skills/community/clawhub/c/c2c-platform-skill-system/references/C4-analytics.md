# C4 交易数据分析 — 数据大脑

C2C本地生活服务平台交易数据分析集群，58个能力单元。以订单状态机、虚拟币"易点"流转、三级裂变推荐、信用分公式为核心数据模型，覆盖从原始采集到预测风控的全链路分析。

**核心数据模型**：
- **订单状态机**：DRAFT → CONFIRMED → PENDING_PAY → PAID → FULFILLING → COMPLETED，含 CANCELLED / REFUNDED / ARBITRATED 分支
- **虚拟币"易点"**：充值(1元=1易点) → 消费(订单支付) → 提现(1易点=0.95元，5%手续费) → 冻结(履约中/争议中)
- **信用分公式**：CreditScore = (AvgRating/5)×100×0.6 + CompletionRate×0.3 + min(100, Reviews×2)×0.1
- **转化漏斗**：浏览 → 下单 → 支付 → 完成
- **留存指标**：D1 / D7 / D30
- **裂变模型**：L1直接推荐 → L2间接推荐 → L3间接推荐，K-factor = 平均每人推荐数 × 接受率

## 步骤4.1 交易数据采集与基础统计

### C4-01 交易数据采集与清洗

- **输入**:
  - [必须] 订单事件流（order_events：含order_id, user_id, provider_id, status, amount_yd, created_at, updated_at）
  - [必须] 支付事件流（payment_events：含payment_id, order_id, channel, amount, status, paid_at）
  - [必须] 易点流水（yd_ledger：含user_id, type[recharge/spend/withdraw/freeze/unfreeze], amount, balance_after, created_at）
  - [必须] 用户行为日志（user_actions：含user_id, action[browse/order/pay/confirm], listing_id, timestamp）
  - [可选] 外部渠道回传数据（utm_source, campaign_id）
- **输出**: 清洗后的事实表（fact_orders, fact_payments, fact_yd_flow, fact_user_actions），字段标准化、空值填充、异常标记，粒度至单条记录
- **依赖**: 无
- **AI自治度**: ⬛
- **组合接口**: → C4-02, C4-03, C4-04, C4-05, C4-09, C4-11, C4-12, C4-14, C4-15, C4-16, C4-17, C4-18, C4-19, C4-20, C4-21, C4-22, C4-23, C4-27, C4-28, C4-30, C4-31, C4-42, C4-43, C4-44, C4-45, C4-48, C4-53

**采集规则**：
1. 订单状态机校验：仅允许合法状态转换（DRAFT→CONFIRMED, CONFIRMED→PENDING_PAY, PENDING_PAY→PAID, PAID→FULFILLING, FULFILLING→COMPLETED；任意状态可转CANCELLED，PAID后可转REFUNDED，FULFILLING后可转ARBITRATED）
2. 易点流水对账：每笔订单的spend必须有对应payment，freeze/unfreeze必须成对
3. 时间戳校验：事件时间不能晚于入库时间，同订单事件按timestamp严格递增
4. 去重策略：基于(order_id, status, timestamp)三键去重

### C4-02 GMV统计与趋势分析

- **输入**:
  - [必须] fact_orders（来自C4-01）
  - [必须] 易点汇率配置（1元=1易点充值，1易点=0.95元提现）
  - [可选] 品类映射表（listing_id → category）
- **输出**: GMV报表，维度=时间[日/周/月/季]×品类×区域×渠道，指标=总GMV(易点)、总GMV(人民币)、订单数、环比增长率、同比增长率；趋势折线数据
- **依赖**: C4-01
- **AI自治度**: ⬛
- **组合接口**: → C4-13, C4-38, C4-39, C4-41, C4-50, C4-51

**计算规则**：
- GMV = SUM(订单金额) WHERE status IN (PAID, FULFILLING, COMPLETED)，不含CANCELLED/REFUNDED
- 易点→人民币换算：GMV_CNY = GMV_YD × 0.95（取提现汇率反映实际经济价值）
- 趋势基线：使用7天移动平均消除日间波动，季节数据按周对齐

### C4-03 订单量与客单价分析

- **输入**:
  - [必须] fact_orders（来自C4-01）
  - [必须] 品类映射表
- **输出**: 订单量与客单价报表，维度=时间×品类×新老用户×订单来源，指标=订单量、客单价(易点)、客单价(人民币)、中位数客单价、客单价P25/P75
- **依赖**: C4-01
- **AI自治度**: ⬛
- **组合接口**: → C4-49

**计算规则**：
- 客单价 = GMV / 有效订单数，有效订单 = status IN (PAID, FULFILLING, COMPLETED)
- 品类差异：家政类客单价通常200-500易点，咨询类50-200易点，维修类100-800易点
- 异常值处理：单笔>5000易点的订单标记为高价值订单，单独统计但不从主指标中剔除

### C4-04 供需比分析

- **输入**:
  - [必须] fact_orders（需求侧：按品类/区域/时段统计订单量）
  - [必须] 服务商在线状态日志（provider_online：provider_id, category, region, online_at, offline_at）
  - [可选] 条目(listing)上下架日志
- **输出**: 供需比报表，维度=品类×区域×时段(小时粒度)，指标=需求量(订单数)、供给量(在线服务商数)、供需比(供给/需求)、供需缺口、缺口等级[严重不足/不足/平衡/过剩/严重过剩]
- **依赖**: C4-01
- **AI自治度**: ⬛
- **组合接口**: → C4-46

**计算规则**：
- 供需比 = 可服务服务商数 / 待服务订单数
- 分级：供需比<0.5严重不足，0.5-0.8不足，0.8-1.2平衡，1.2-2.0过剩，>2.0严重过剩
- 时段切分：以小时为单位，工作日/周末分别计算，识别高峰时段供给瓶颈

## 步骤4.2 转化与留存分析

### C4-05 转化漏斗分析

- **输入**:
  - [必须] fact_user_actions（来自C4-01，action类型含browse/order/pay/confirm）
  - [必须] 品类映射表
  - [可选] 渠道来源（utm_source）
- **输出**: 漏斗报表，维度=品类×渠道×时段，指标=浏览量、下单量、支付量、完成量；各环节转化率[browse→order, order→pay, pay→complete]；整体转化率[browse→complete]
- **依赖**: C4-01
- **AI自治度**: ⬛
- **组合接口**: → C4-06, C4-07, C4-08, C4-46, C4-47

**漏斗定义**：
1. 浏览(browse)：用户查看条目详情页
2. 下单(order)：用户创建DRAFT/CONFIRMED订单
3. 支付(pay)：订单状态到达PAID
4. 完成(complete)：订单状态到达COMPLETED
- 漏斗窗口：同一用户7天内的行为链路归为一次漏斗实例
- 品类差异：咨询类browse→order转化率通常高于维修类（决策周期短）

### C4-06 浏览→下单转化率

- **输入**:
  - [必须] C4-05漏斗数据（browse→order环节）
  - [必须] 条目详情（listing_id, title, price_yd, category, provider_rating）
  - [可选] A/B测试分组标识
- **输出**: browse→order转化率报表，维度=品类×条目价格区间×服务商评分区间×页面版本，指标=转化率、流失率、流失用户画像
- **依赖**: C4-05
- **AI自治度**: ⬛
- **组合接口**: → C4-46

**分析维度**：
- 价格敏感度：按易点价格区间(<50/50-200/200-500/>500)分组统计转化率
- 评分效应：服务商信用分区间(0-60/60-80/80-100)对转化的影响
- 页面优化点：识别高浏览低转化条目，标记为优化候选

### C4-07 下单→支付转化率

- **输入**:
  - [必须] C4-05漏斗数据（order→pay环节）
  - [必须] 支付方式配置（易点余额/微信/支付宝）
  - [可选] 优惠券使用记录
- **输出**: order→pay转化率报表，维度=支付方式×金额区间×是否使用优惠券×时段，指标=转化率、平均支付耗时、未支付取消率
- **依赖**: C4-05
- **AI自治度**: ⬛
- **组合接口**: → C4-46

**分析要点**：
- 易点余额支付 vs 第三方支付的转化率差异
- 未支付订单特征：金额偏高/首次下单/余额不足
- 支付超时：CONFIRMED→PENDING_PAY后30分钟未支付自动关单的比例

### C4-08 支付→完成转化率

- **输入**:
  - [必须] C4-05漏斗数据（pay→complete环节）
  - [必须] 订单状态转换日志（含CANCELLED/REFUNDED/ARBITRATED分支）
- **输出**: pay→complete转化率报表，维度=品类×服务商等级×订单金额区间，指标=完成率、取消率、退款率、仲裁率、平均履约周期
- **依赖**: C4-05
- **AI自治度**: ⬛
- **组合接口**: → C4-27, C4-46

**分支分析**：
- PAID→CANCELLED：用户主动取消，分析取消原因与距支付时间
- PAID→REFUNDED：服务商无法履约或质量争议
- FULFILLING→ARBITRATED：进入仲裁，标记为服务失败
- 完成率 = COMPLETED数 / PAID数，目标>85%

### C4-09 用户留存率分析

- **输入**:
  - [必须] fact_user_actions（来自C4-01）
  - [必须] 用户首次交易日期表（user_id, first_order_date, first_category）
  - [可选] 渠道来源
- **输出**: 留存率报表，维度=首次交易品类×渠道×用户类型[需求方/供给方]，指标=D1/D7/D30留存率、留存人数、留存用户平均交易次数
- **依赖**: C4-01
- **AI自治度**: ⬛
- **组合接口**: → C4-10, C4-39

**留存定义**：
- D1留存：首次交易后第1天有任意行为（浏览/下单/支付）的用户占比
- D7留存：首次交易后第7天有交易行为（下单/支付）的用户占比
- D30留存：首次交易后第30天有交易行为的用户占比
- 分母为当日新交易用户数，不包含注册未交易用户

### C4-10 D1/D7/D30留存计算

- **输入**:
  - [必须] C4-09留存数据
  - [必须] 用户交易频次表
  - [可选] 竞品基准数据
- **输出**: 留存曲线与基准对比，维度=用户群组(cohort)×时间，指标=各日留存率、留存衰减率、与行业基准偏差、流失预警标记
- **依赖**: C4-09
- **AI自治度**: ⬛
- **组合接口**: → C4-39

**计算规则**：
- Cohort分组：按首次交易周分群，观察12周留存曲线
- 衰减模型：拟合幂律衰减 N(t) = N(0) × t^(-α)，α>0.5为高危衰减
- 基准参考：C2C本地生活D1基准15-25%，D7基准8-15%，D30基准3-8%
- 预警：某cohort留存率低于前3个cohort均值的1个标准差时标记

- **依赖**: C4-01
- **AI自治度**: ⬛
- **组合接口**: → C4-46

**覆盖率评级**：

### C4-12 信用分分布统计

- **输入**:
  - [必须] 服务商评价数据（provider_id, avg_rating, total_reviews, completion_rate）
  - [必须] 信用分公式：CreditScore = (AvgRating/5)×100×0.6 + CompletionRate×0.3 + min(100, Reviews×2)×0.1
- **输出**: 信用分分布报表，维度=品类×区域×服务商等级，指标=信用分均值/中位数/P25/P75、各分段占比[0-60/60-70/70-80/80-90/90-100]、公式各因子贡献度
- **依赖**: C4-01
- **AI自治度**: ⬛
- **组合接口**: → C4-33, C4-34

**分布特征**：
- 评分因子(AvgRating/5×100×0.6)：权重60%，范围0-60分，多数服务商4.0+评分即48+分
- 完成率因子(CompletionRate×0.3)：权重30%，范围0-30分，完成率>90%即27+分
- 评价数因子(min(100, Reviews×2)×0.1)：权重10%，范围0-10分，50条评价即满分
- 典型分布：右偏，众数在75-85区间

  - [必须] 优惠券/补贴扣减记录
- **输出**: 佣金收入报表，维度=品类×时段×渠道，指标=总佣金(易点)、实际佣金率、补贴抵扣金额、净佣金收入、佣金占GMV比重
- **依赖**: C4-02

**佣金规则**：
- 家政类：8%佣金
- 维修类：10%佣金
- 咨询类：12%佣金

- 订单取消/退款时佣金原路退回

### C4-14 提现率与资金留存分析

- **输入**:
  - [必须] fact_yd_flow（来自C4-01，易点流水）
  - [必须] 提现规则（5%手续费，T+1到账）
- **输出**: 提现率报表，维度=用户类型[需求方/供给方]×时段，指标=充值总额、消费总额、提现总额、提现率(提现/收入)、资金留存率、平均留存天数
- **依赖**: C4-01
- **AI自治度**: ⬛
- **组合接口**: → C4-26

**分析要点**：
- 供给方提现率：服务商收入中提现占比，反映资金在平台内的循环效率
- 需求方提现率：用户充值后未消费而提现的比例，过高说明充值动机非消费
- 资金留存 = 1 - 提现率，目标>60%（资金在平台内流转）
- 留存天数：从充值到提现的平均时间间隔

### C4-15 裂变系数与推荐链分析

- **输入**:
  - [必须] 推荐关系表（referrer_id, referee_id, level[L1/L2/L3], created_at）
  - [必须] 被推荐人交易记录（referee_id的首笔订单信息）
  - [可选] 推荐渠道（分享链接/二维码/口令）
- **输出**: 裂变系数报表，维度=时段×渠道×层级，指标=K-factor(人均推荐数×接受率)、L1/L2/L3推荐人数、推荐转化率、裂变衰减系数
- **依赖**: C4-01
- **AI自治度**: ⬛
- **组合接口**: → C4-35, C4-37

**K-factor计算**：
- K = i × c，其中i=每人平均发出邀请数，c=每个邀请的接受转化率
- L1：直接推荐，K1 = i1 × c1
- L2：间接推荐(推荐的推荐)，K2 = i2 × c2
- L3：三级间接推荐，K3 = i3 × c3
- 总K = K1 + K1×K2 + K1×K2×K3，K>1时裂变自增长

### C4-16 纠纷率与退款率分析

- **输入**:
  - [必须] 订单状态转换日志（重点关注REFUNDED和ARBITRATED分支）
  - [必须] 仲裁记录表（arbitration_id, order_id, initiator, reason, outcome, created_at）
  - [必须] 退款记录表（refund_id, order_id, amount_yd, reason, status）
- **输出**: 纠纷率报表，维度=品类×服务商等级×时段，指标=退款率、仲裁率、纠纷原因分布[质量/时效/沟通/价格]、平均退款金额、仲裁处理时长
- **依赖**: C4-01
- **AI自治度**: ⬛
- **组合接口**: → C4-53

**纠纷分级**：
- 退款率 = REFUNDED订单数 / PAID订单数，目标<5%
- 仲裁率 = ARBITRATED订单数 / FULFILLING订单数，目标<2%
- 原因分类：质量问题(35%)、时效问题(25%)、沟通问题(20%)、价格争议(15%)、其他(5%)

### C4-17 用户画像构建

- **输入**:

**标签规则**：

- 支付偏好：易点余额支付占比>80%为易点偏好，否则为混合/第三方偏好

## 步骤4.7 用户价值与经济模型

### C4-38 用户获取成本(CAC)计算

- **输入**:
  - [必须] C4-02 GMV数据
  - [必须] 营销支出表（channel, date, spend_amount, new_users_acquired）
  - [必须] 新增交易用户数
- **输出**: CAC报表，维度=渠道×时段，指标=总营销支出、新增交易用户数、CAC、渠道CAC对比、CAC趋势
- **依赖**: C4-02
- **AI自治度**: ⬛
- **组合接口**: → C4-40

**CAC计算**：
- CAC = 渠道营销支出 / 该渠道带来的新增交易用户数
- 分子：广告费+优惠券成本+分润支出
- 分母：7天内完成首笔支付的用户（非注册用户）
- 典型值：自然流量CAC≈0，裂变CAC≈20易点，付费渠道CAC≈80-150易点

### C4-39 用户生命周期价值(LTV)计算

- **输入**:
  - [必须] C4-02 GMV数据
  - [必须] C4-09留存率数据
  - [必须] 平均佣金率
  - [必须] 用户生命周期参数
- **输出**: LTV报表，维度=渠道×品类×用户类型，指标=LTV(佣金贡献)、LTV(含分润)、平均生命周期月数、LTV趋势
- **依赖**: C4-02, C4-09
- **AI自治度**: ⬛
- **组合接口**: → C4-40

**LTV计算**：
- LTV = Σ(Monthly_Commission × Retention_Rate^t)，t=1,2,...,36
- Monthly_Commission = 该用户月均消费 × 平均佣金率
- Retention_Rate使用D7→D30→D90衰减模型拟合
- LTV(含分润) = LTV(佣金) - 该用户带来的分润支出
- 典型值：高价值用户LTV≈500-2000易点，中价值≈100-500，低价值<100

**健康度评估**：
- LTV/CAC > 3：健康，该渠道可持续投入
- LTV/CAC 1-3：边际可行，需优化
- LTV/CAC < 1：不可持续，需暂停或调整
- 回本周期：LTV累计超过CAC的月数，目标<6个月

**预警规则**：

### C4-54 刷单行为识别

- **输入**:
  - [必须] C4-53异常交易预警数据
  - [必须] 订单详情（含IP、设备指纹、GPS位置）
  - [必须] 用户关系图（推荐关系、交易关系）
  - [必须] 服务商-需求方交易对频次
- **输出**: 刷单嫌疑报表，指标=嫌疑等级[疑似/高度疑似/确认]、刷单模式分类[自买自卖/互刷/佣金套利]、涉及金额、涉及账号数、证据链
- **依赖**: C4-53
- **AI自治度**: 🟨
- **组合接口**: → C4-55

**识别模式**：
1. **自买自卖**：需求方与服务商为同一人/同设备/同IP，或收货地址=服务地址
2. **互刷**：A买B的服务，B买A的服务，时间接近、金额相近
3. **佣金套利**：利用新人优惠券+分润机制，首单即完成并提现，实际无服务
4. **信用分刷单**：大量小额快速完成订单，快速提升信用分

**监测规则**：

### C4-58 数据口径与定义标准化

- **输入**:
  - [必须] C4-57指标体系文档
  - [必须] 现有各系统口径差异记录（如：不同团队对"活跃用户"的定义差异）
  - [必须] 数据源元数据
- **输出**: 数据口径标准文档，包含统一定义、计算公式、例外处理、口径变更记录、口径一致性校验规则
- **依赖**: C4-57
- **AI自治度**: 🟨
- **组合接口**: 无（终端单元）

**口径标准化重点**：
1. **GMV口径**：以订单到达PAID状态为准，不含CANCELLED/REFUNDED，易点按0.95汇率折算人民币
2. **活跃用户口径**：统计周期内有支付行为的用户（非浏览/下单未支付）
3. **活跃服务商口径**：统计周期内有完成订单的服务商（非仅上线/接单未完成）
4. **客单价口径**：GMV / 有效订单数，有效订单=PAID及以后状态
5. **留存口径**：首次交易后第N天有支付行为，分母为当日新交易用户
6. **信用分口径**：严格按公式统一计算，不接受手动调整

**口径变更流程**：任何口径变更需经数据委员会审批，同步更新C4-57指标体系文档，并在看板中标注口径变更日期。
