# 跨组件通用反例（强约束）

本文档列举跨组件的通用错误模式。组件专属约束（如 value 类型、cycle 字段、focusPoints 数量等）已内联到 SKILL.md 各组件定义中，此处不再重复。

---

## 反例1：用 TextCard 替代特殊组件

**错误示例**：将"社媒趋势的4个机会（有优先级高低之分）"用 TextCard 展示。

```json
❌ {"type": "TextCard", "data": [
  {"title": "3D打印定制", "text": "探索3D打印技术应用..."},
  {"title": "明星联名款", "text": "推出明星同款或联名款..."}
], "config": {"title": "社媒新兴机会"}}
```

**规则**：当内容包含优先级排序、影响力分级时，必须使用 InfluenceCard；当内容包含风险提示时，必须使用 AlertCard；当内容包含执行步骤时，必须使用 PhaseCard；当内容有3+对象的多维度对比时，必须使用 Chart.List。严格参照 SKILL.md 中"组件选型优先级"章节的映射表。

## 反例2：章节大量丢失，过度压缩

**错误示例**：原文有8个大章节、20+个子主题，输出仅9个组件，丢失了"成本结构明细""细分场景产品矩阵""品牌业绩数据""风险与机会提示"等完整章节。将多个独立章节压缩进同一个组件：

```json
❌ {"type": "TextCard", "data": [
  {"title": "行业转型趋势", "text": "从功能性老年鞋向轻运动生活方式载体转型..."},
  {"title": "供需错配现状", "text": "低端产品严重过剩..."},
  {"title": "技术创新方向", "text": "超临界发泡材料..."}
], "config": {"title": "行业核心洞察"}}
← 将"行业概况""产品创新""供需格局"三个独立章节压缩为一个3条的TextCard
```

**规则**：
1. 生成前必须逐一列出原文所有章节和子主题
2. 生成后必须逐一核对，确保每个章节至少有一个组件对应
3. 禁止将多个独立章节压缩进同一个组件
4. 每个原文大章节（一级标题）应至少对应1-2个组件

## 反例3：基础图表类组件独占一行造成布局浪费

**错误示例**：

```
MODULE 内：
r1: 仅 Chart.Column（基础图表类）  ← 可与 Pie/DataCard 组合，浪费半行
r2: 仅 KeyValueCard（基础图表类）  ← 可与其他基础图表组合，浪费半行
→ 两个基础图表类组件可以合并为 r1-c1 + r1-c2，却各自独占一行
```

**正确示例**：

```
r1: Chart.Column(消费诉求) + Chart.Pie(价格带分布)  ← 两个基础图表组合
r2: Chart.Column(品牌份额) + DataCard(品牌业绩)     ← 两个基础图表组合
```

**规则**：
1. 基础图表类组件（Chart.Line/Chart.Column/Chart.Pie/DataCard/KeyValueCard）必须优先两两组合
2. 生成每个 MODULE 后，必须检查是否存在两个或以上基础图表类组件各自独占一行的情况
3. 如果存在，必须合并到同一行（一个 c1，一个 c2）
4. 仅当 MODULE 内只有一个基础图表类组件且无其他基础图表可组合时，才允许独占一行
5. 常见可组合对：Column+Pie、Column+DataCard、Column+KeyValueCard、DataCard+KeyValueCard、Pie+DataCard

## 反例4：整个章节被完全跳过/省略

**错误示例**：原文有8个大章节，输出仅覆盖其中5个，以下章节被完全跳过：

```
❌ "宏观环境与政策支持"（4个政策方向）→ 无任何组件对应
❌ "社交媒体趋势与新兴机会"（4个机会方向）→ 无任何组件对应
❌ "风险与挑战" → 无任何组件对应（应用 AlertCard）
```

**规则**：
1. 生成前必须列出原文所有一级标题和二级标题，形成章节清单
2. 生成后必须逐一核对章节清单，确保每个一级标题至少有 1 个组件对应
3. 总组件数不得少于原文一级标题数 × 1.5
