# 图片处理规范 

## 图片传输机制 

**插件端**：真实图片文件通过 `background.js` 采集为 Blob，存入 Chrome IndexedDB。不通过 API 传输图片二进制数据。 
**API 请求**：仅传输图片元数据（纯文本），嵌入 `content` 字段末尾，请求体远小于 1MB 限制。 

* * *

## 图片元数据格式 

图片元数据以文本形式嵌入 `content` 字段末尾，用 `---` 与正文分隔： 

```
[IMAGE: img_001.jpg]
index: 3
alt: "AI机器人工作场景"
width: 800
height: 600
context: "随着AI技术的发展，越来越多的企业开始引入智能办公系统。这些系统不仅能自动处理文档，还能辅助决策。"
keywords: "AI办公,自动化文档,机器人流程,智能处理"

[IMAGE: img_002.jpg]
index: 7
alt: "自动化办公示意图"
width: 1200
height: 800
```
**字段说明**  
| 字段 | 类型 | 说明 | 
 |------|------|------| 
 | `name` | string | 图片文件名（来自 `[IMAGE: xxx]` 头部标记，如 `img_001.jpg`），对应占位符 | 
 | `index` | number | 图片在原文中的位置序号，技能据此在正文中定位上下文 | 
 | `alt` | string | 图片 HTML alt 属性，辅助 LLM 理解图片内容 | 
 | `width` | number | 图片宽度 px，辅助判断图片类型（横图/竖图/方图） | 
 | `height` | number | 图片高度 px | 
 | `keywords` | string | （可选）已有的图片关键字，逗号分隔。存在时表示该图片已在首次生成时获得正确关键字，技能应直接使用，不再重新生成 | 
 | `context` | string | （可选）图片在原文中的前文文字。采集时从 DOM 中上溯多个同级元素提取，用于技能关键字生成的上下文参考。存在时优先使用，替代 index 段落匹配 | 

**注意**：不传输 `url`、`blob`。`context` 字段在采集时从 DOM 提取，以纯文本形式嵌入请求。 

* * *

## 图片占位符格式 

生成的文章中，所有图片必须使用以下格式的占位符： 
```
{{IMAGE:img_XXX.jpg}}
```
其中 `XXX` 为三位数字编号（零填充）： 
 - 第 1 张图片：`{{IMAGE:img_001.jpg}}` 
 - 第 2 张图片：`{{IMAGE:img_002.jpg}}` 
 - ... 
 - 第 10 张图片：`{{IMAGE:img_010.jpg}}` 

插件端编辑器预览时，将 `{{IMAGE:img_XXX.jpg}}` 替换为 IndexedDB 中对应图片的 Blob URL。 

* * *

## 图片说明格式 

每张图片占位符后添加一行图片说明，使用 Markdown 引用格式： 

```
{{IMAGE:img_001.jpg}} 
> 图片说明文字，不超过 20 字
```

## 关键字注释输出规范

步骤 2 中为每张图片都生成了关键字，无论该图片最终是否被插入文章，关键字注释都必须出现在输出中：

**被使用的图片**：关键字注释紧跟图片占位符和说明之后

```
{{IMAGE:img_001.jpg}} 
> 图片说明
<!-- kw:img_001.jpg:关键词1,关键词2,关键词3 -->
```

**未被使用的图片**：关键字注释集中追加在文章末尾

```
<!-- kw:img_003.jpg:关键词1,关键词2,关键词3 -->
<!-- kw:img_004.jpg:关键词1,关键词2,关键词3 -->
```

插件端通过全文扫描 `<!-- kw:... -->` 注释回填 IndexedDB，因此所有图片的关键字都必须输出，确保：
- 首次生成后所有图片都有正确关键字
- 改写场景中不会因缺少关键字而重新生成错误关键字

## 关键字生成方法

### 改写场景：关键字保留

当图片元数据中包含 `keywords` 字段时，表示该图片已在首次生成时获得正确关键字（基于原文上下文提取）。改写场景中正文已被 AI 改写，无法再从改写后内容中正确提取关键字，因此：

- **直接使用** `keywords` 字段中的关键字，原样写入输出中的 `<!-- kw:文件名:关键字1,关键字2,... -->` 注释
- **禁止**重新从改写后正文中提取关键字，避免错误关键字覆盖已有正确关键字
- 仅对**没有** `keywords` 字段的图片执行下方正常的关键字生成流程

### 第一步：定位上下文

优先使用图片元数据中的 `context` 字段作为上下文参考（该字段在采集时从页面 DOM 中上溯多个同级元素提取，比通过 `index` 猜测段落位置更准确）。

如果某张图片没有 `context` 字段或为空，则退回到根据 `index` 序号在正文中定位对应段落的方式：

1. 将正文按空行/标题分割为段落序列
2. `index` 表示图片在原文中的位置序号
3. 映射 `index` 到对应段落（段落序号 ≈ index）
4. 取该段落及其前后各一段作为图片上下文

### 第二步：生成关键字

LLM 综合分析以下信息生成 3-5 个中文关键字：

| 分析依据   | 优先级 | 说明                     |
| ------ | --- | ---------------------- |
| 上下文文字  | 最高  | 图片所在段落的文字，直接反映图片在文中的含义 |
| alt 属性 | 中   | 图片原始 HTML 描述           |
| 文章主题   | 基础  | 整体主题方向约束               |

## 生成原则

- 关键字必须具体、有区分度
- 不同图片的关键字必须体现差异性（避免所有图片都是 `["科技", "AI", "办公"]`）
- 禁止使用泛泛词汇：`"图片"`、`"示意图"` 、 `"""配图"""`、`"插图"` 、`"图表"`
- 关键字仅用于内部规划，不得出现在最终输出中

## 示例

| 图片 alt + 上下文主题          | 生成的关键字                         |
| ----------------------- | --------------------------------------- |
| "AI 完成自动化文档处理" + 效率提升话题 | `["AI办公", "自动化文档", "机器人流程", "智能处理"]` |
| "数据可视化大屏" + 实时监控话题      | `["数据大屏", "实时监控", "业务指标", "可视化分析"]`  |
| "产品设计稿" + UI 改版话题       | `["产品设计", "UI改版", "用户界面", "交互体验"]`   |

## 图片插入原则

| 原则    | 说明                           |
| ----- | ---------------------------- |
| 语义相关性 | 关键字与段落内容做语义匹配，匹配度最高的位置优先     |
| 密度控制  | `{mode} ≤ 600` → 最多 2 张；`{mode} > 600` → 每 500-800 字一张 |
| 顺序灵活  | 不强制按 `index` 或文件名顺序，以相关性为准 |

### 位置规则：

| 图片角色 | 插入位置              |
| ---- | ----------------- |
| 引导图  | 引言段之后（通常是第 1-2 段） |
| 分段配图 | 对应小节标题之后、正文之前     |
| 结尾图  | 倒数第二段之后           |

## 字数说明

注意：这里的字数指用户选择的 `{mode}` 参数对应的目标字数，不是元数据中的 `width`/ `height` 。

| 模式 | 目标字数        |
| -- | ----------- |
| `{mode}` | 500-3000 之间的整数，步长 100，默认 1000 |
