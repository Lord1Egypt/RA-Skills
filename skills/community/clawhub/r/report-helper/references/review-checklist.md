# 审核清单（Review Checklist）

交付前必做。**命中即重写该句**，然后再扫一遍直到通过。

---

## 一、禁止套话（违反则重写该句）

### 禁止的句式结构

- 「不是 A，而是 B」「不是 A，更是 B」
- 「不仅仅是 A，更是 B」「不只是 A，还是 B」（作修辞强调时禁止；事实陈述允许）
- 「值得注意的是」「值得一提的是」「值得关注的是」
- 「让我们来看看」「让我们深入探讨」「让我们一起了解」
- 「在这个背景下」「在当今时代」「在这样的大环境下」
- 「某种意义上来说」「从某种角度来看」
- 「事实上」做句首
- 「毫无疑问」「不可否认」（作为句首修饰语时禁止）
- 「总的来说」「综上所述」「总而言之」
- 「这意味着什么？这意味着……」（自问自答式设问）
- 「简单来说」「换句话说」「换言之」
- 「深刻的洞见」「深刻地影响」「深刻地改变」—— 禁止所有带「深刻」的修饰
- 「引发了广泛关注/讨论/思考」
- 「给出了自己的答案」
- 「一场关于……的革命/对话/思考」
- 任何无功能的排比句

### 禁止的高频踩雷词

「说白了」「意味着什么？」「这意味着」「本质上」「换句话说」「不可否认」

### 空洞形容词 / 动词

「赋能」「抓手」「打造闭环」「深度赋能」「生态化反」

### 教科书开头

「在当今 AI 快速发展的时代」「随着技术的不断进步」

### 空泛工具名

不说「AI 工具」「某个模型」，要说具体产品、模型或工具名称。

### 编造场景

搜不到的信息诚实标注「该信息暂缺」，**绝不编造**

---

## 二、标点规范扫描

- 破折号 `——`：全文搜，同段落超 1 处就删
- 感叹号 `！`：全文计数，> 5 个就改成句号
- 双引号 `""`：中文正文里的全部换成「」（代码块和英文原文保留）
- 嵌套引号用 `『』`
- 中文正文里的 `,` `.` `;` `:` `?` `!` 都要改成全角 `，。；：？！`

---

## 三、句式规范扫描

- 平均句长 15-25 字
- 长短句交替，不要连续 3 句都是长句或短句
- 单句不超过 2 个分句

---

## 四、交付前质检清单（逐条打钩）

**内容层面：**

- [ ] 成品体裁是否匹配用户原始要求？如用户要「发展研究报告 / 产业研究报告」，正文是否是正式报告结构，而不是方法论结构？
- [ ] 正文结构是否来自 `report-template.md` 的选定框架或用户指定目录，而不是把写作判断工具当成目录？
- [ ] 涉及发展历程的段落是否有因果逻辑和阶段脉络？不是年表流水账？
- [ ] 创始人/发起者的背景和动机有足够深度？
- [ ] 每个关键节点都展开写了，没有为了压缩而跳过重要细节？
- [ ] 决策逻辑有还原？不只是「发生了什么」，还有「为什么这么选」？
- [ ] 同期格局 / 竞品格局是否判断清楚？代表性对象是否足够？
- [ ] 用户口碑部分是否标为二手来源，并引用具体用户声音？不只是官方宣传？
- [ ] 判断合成是否产出了新的判断，不是前面内容的缩写版？
- [ ] 未来推演的三个剧本都有逻辑支撑？
- [ ] 写作前是否完成最新数据校验？近 12-24 个月的官方披露、监管文件、财报、原始研究或权威报道是否进入判断？
- [ ] 公司对象是否包含成长性和估值的定量分析？收入/ARR、用户/客户、利润率、现金流、融资、估值、可比公司指标等可得数据是否被纳入？
- [ ] 公司对象缺少公开收入、利润或估值数据时，是否明确写出缺口和替代观察指标，而不是用叙事代替数字？
- [ ] 目录之后、第一章正文之前是否有「关于本报告的信息来源」，并说明 `<sup>a1</sup>` / `<sup>b4</sup>` 这类编号规则？
- [ ] 所有关键事实判断是否都带了 `<sup>[a-d]数字</sup>` 格式来源上标？
- [ ] 正文每个来源上标是否都能在末尾「信息来源与分级」找到同级同号条目？
- [ ] 末尾来源是否按 a/b/c/d 分组，且编号不因分组重新从 1 开始；没有 c 和 d 时是否省略对应小节？
- [ ] c、d 级来源是否没有支撑正文判断？
- [ ] 搜不到的信息诚实标注了「暂缺」，没有编造？

**风格层面：**

- [ ] 禁止套话清单全清？
- [ ] 标点规范通过？
- [ ] 句式规范通过？
- [ ] 有节奏感、有可读性，不是咨询报告腔？

**交付物层面：**

- [ ] 内部 Markdown 构建稿存到了配置的 `work_dir`（默认 `./output/work/`）
- [ ] PDF 文件名是 `{研究对象}深度研究报告.pdf`（中文紧连无空格）
- [ ] PDF 封面作者来自 `config.local.json` 的 `author`
- [ ] PDF 最末尾包含工具签名：`本报告由 report-helper skill 工具协助生成`、开源地址、作者联系方式
- [ ] 内部 Markdown 构建稿和 PDF 末尾都**没有**「方法论说明」章节
- [ ] 内部 Markdown 构建稿和 PDF 末尾都保留「信息来源与分级」章节
- [ ] PDF 目录能被目标读者直接理解，不暴露内部工作分层
- [ ] 总字数在 10,000-30,000 字范围内
- [ ] 如果本次被用户指出流程、事实或结构问题，是否已把可复用的新坑补进 `references/gotchas.md`？

---

## 五、审查流程

1. 逐条扫「禁止套话」
2. 标点扫描
3. 句式扫描
4. 打钩清单
5. 发现问题 → 重写 → 再扫一遍直到全部通过
