# task-report

按时间段汇总任务进展，生成人可读的汇报。

## 场景

- 用户说"帮我写日报"、"总结一下本周进展"
- 定时汇报（每日/每周）
- 管理员需要团队进展概览

## 操作步骤

### 1. 确定时间窗口

- 日报：`--duration 1d`
- 周报：`--duration 7d`
- 月报：`--duration 30d`

### 2. 获取上下文

普通用户：
```bash
auwomo task context --duration 7d -d
```

团队视角（管理员/有下属）：
```bash
auwomo task context --duration 7d -d --team
```

### 3. 生成汇报

基于 context 输出，按以下原则编写汇报：

**结构**：
1. 一句话总览 — 整体有没有进展
2. 重点推进 — 当前在做什么
3. 当前状态 — 推进到什么程度

**原则**：
- 用人话写，不写工具日志
- 先结论，再脉络，再补充
- 保持主线 → 子线 → 记录的层级关系
- 记录应放回其所属主线/子线脉络中理解
- 不要把 description 大段原样贴进汇报
- 如果没有新增记录，用自然语言说明

### 4. 团队汇报（管理员视角）

```bash
auwomo task context --duration 7d -d --team --format json
auwomo task missing --range 2d --team --format json
```

团队汇报结构：
1. 整体概览 — 谁有推进，谁暂无
2. 按人简报 — 按 `assignee` 分组，每人一两句
3. 缺失记录 — 基于 `task missing` 结果
4. 停滞主线检查 — 基于 `updated_at` 字段
5. 风险卡点 — 阻塞/逾期/停滞的任务

#### 停滞阈值

| 场景 | 阈值 | 处理 |
|------|------|------|
| 个人主线自提醒 | 3 天 | 温和提醒自己关注 |
| 团队主线提醒 | 7-14 天 | 提醒关注 |
| 团队主线警告 | >14 天 | 建议主动跟进 |
| 缺交记录提醒 | 2 天 | 列出无记录成员 |

#### 同理心表述规则

涉及"某人没做某事"的表述时，**必须**遵循以下规则：

1. **先看 context**：从 `recent_records` 中按该 assignee 筛选，看此人同期在做什么
2. **推测原因**：
   - 有其他活跃记录 → "可能精力集中在 [主线]yyy"
   - 任务本身复杂 → "项目可能有一定难度"
   - 无任何记录 → "可能近期较忙或有其他安排"
3. **格式**：一句原因推测 + 一句建议
4. **示例**：
   - "马子健 近7天无 [主线]xxx 的进展记录，但同期在推进 [主线]yyy（最近3天有5条记录），可能是精力集中在那边。建议确认是否需要调整优先级。"
   - "黄永强 本周无 [主线]xxx 的进展，但 [主线]yyy 有较多记录（本周6条），可能是优先级侧重。建议下周确认 xxx 是否需要推进。"

**绝对不要**：
- 直接说"XXX没做"、"XXX未完成"
- 用指责性语气描述缺失
- 忽略此人在其他任务上的付出

#### 风险识别标准（按优先级）

- 阻塞状态
- 逾期风险（due_at 已过/即将到期）
- 进展停滞（按上述阈值判定，使用 `updated_at` 字段）
- 描述含"风险"、"阻塞"、"卡住"等关键词

### 5. 未初始化状态处理

当 `auwomo identity whoami` 或后续命令返回认证错误时，不要输出技术性报错，改为输出友好引导消息，介绍自身能力并引导用户完成配置。

## 不要这样做

- 不要自己重新扫描任务树（用 context 输出）
- 不要把记录从所属主线/子线中剥离后平铺
- 不要忽略多级子线结构
- 不要把技术性判定直接当汇报内容
- 不要用指责性语气描述团队成员的缺失或停滞
