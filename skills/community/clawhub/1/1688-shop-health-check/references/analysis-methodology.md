# 分析方法论（Step 3 — Step 8）

本文档包含第二阶段各分析方向的详细方法论。Agent 根据用户选择的方向，按需读取对应章节。

> **前置依赖**：所有 Step 均依赖 Step 1（总盘健康度判断）的结果，用于交叉验证。

## Step 3 — 异常商品定位（`seller_import_abnormal_offer`）

**按需调用（用户选择方向 1 或 7 时执行）。**

按 `reason` 分三类识别：

| 异常类型 | 含义 | 优先级 |
|---------|------|--------|
| **访客下跌** | 偏流量获取问题 | 中 |
| **支付下跌** | 偏转化/成交问题 | 中 |
| **访客下跌 + 支付下跌（双跌）** | 流量与转化同时恶化 | **最高** |

**筛查优先级**（按以下顺序排列异常商品）：

1. **影响金额大**：`valueMap.payAmt.value.value` 高且 `valueMap.payAmt.cycleCqc.value` 负向大
2. **跌幅大**：`cycleCrc` 负向显著（如 < -30%）
3. **多重异常**：`reason` 同时包含访客下跌和支付下跌
4. **集中度**：是否集中在某类目或店铺核心商品上

输出时聚焦 **TOP 3-5 个最值得处理的异常商品**，每个需说明：异常类型、当前规模、影响幅度、优先处理理由。

### 下一步操作卡片触发判定（必做）

Step 3 取数完成后，**必须根据异常商品数量决定调用哪个交互组件**，但**具体的 `show_interaction` 调用参数、文案模板、构造规则**统一在主文件「二、多阶段输出格式 — 第二阶段输出模板 §C」集中定义，本节只负责**判定**：

| 判定条件 | 调用 `show_interaction` 的交互名 | 输出形态 |
|---|---|---|
| `abnormal_offer_count > 0` | `select_abnormal_action` | §C 形态一：商品 × 操作合并卡片 |
| `abnormal_offer_count == 0` | `input_offer_for_optimize` | §C 形态二：两步问答（输入 offerId + 选择优化方向） |

**调用时机**：在第二阶段 §B 的 ` ```seller-report``` ` 代码块输出完毕**之后**，通过 `show_interaction` 展示【下一步操作卡片】，不得在 §B 之前或内部调用。

**与方向 7（完整诊断）的衔接**：完整诊断模式下，§C 也只调用一次 `show_interaction`，紧跟在 §B 之后；不要为每个方向重复触发。

**⚠️ 多方向组合场景**：当用户同时选择了多个方向（如方向 1+3），异常商品的 §C 判定和数据准备在 Step 3 完成，但**实际调用 `show_interaction` 的时机**始终在所有方向的 §A + §B 输出完毕之后，不因异常商品结果为空（`abnormal_offer_count == 0`）而提前。即使异常商品无数据，也必须先完整输出 §A（覆盖所有方向的总结）和 §B（覆盖所有方向的可视化报告），再在最后执行 §C。

> 📍 **完整 `show_interaction` 调用参数、文案构造规则、关键词映射、执行优化动作调用、兜底逻辑**全部见主文件「二、多阶段输出格式 — 第二阶段输出模板 §C」。

## Step 4 — 流量趋势分析（`get_traffic_trend`）

**按需调用（用户选择方向 2 或 7 时执行），用于判断流量波动趋势和识别异常。**

**判断维度**：

| 维度 | 判断方式 |
|------|---------|
| **整体趋势** | 近 7 天 / 近 30 天 uv/pv 的整体走向（上升/下降/波动） |
| **异常波动** | 单日 uv/pv 突然大幅下跌或上涨的日期 |
| **UVCTR 趋势** | UV 点击率的变化，判断流量质量是否稳定 |
| **波动幅度** | 计算近 7 天 uv 的标准差或最大/最小值差异 |
| **与总盘关联** | 流量趋势是否与 `seller_trade_code_index` 中的 `payByrCnt` 变化一致 |

**结论分类**：

| 结论 | 特征 |
|------|------|
| **流量稳定** | uv/pv 趋势平稳，无明显波动 |
| **流量下滑** | 近 7 天 uv 持续下降，需排查流量来源 |
| **流量波动** | uv 在某些日期突然下跌，需检查是否有活动结束或渠道中断 |
| **流量回升** | uv 近期开始上升，可能是新活动或渠道生效 |
| **流量质量下降** | uv 稳定但 UVCTR 下降，说明流量质量降低 |

**关键判断逻辑**：
- 若 `get_traffic_trend` 显示近 7 天 uv 持续下滑，且 `seller_trade_code_index` 中 `payByrCnt` 同步下降 → **流量问题为主因**
- 若 uv 稳定但 UVCTR 下降 → **流量质量降低，需优化流量来源或承接**
- 若某日 uv 突然大幅下跌（如跌幅 > 30%） → **识别该日期，排查是否有活动结束、渠道中断或系统问题**

**与异常商品交叉验证**：
- 若流量趋势显示某日期突然下跌，需检查该日期前后是否有异常商品出现
- 若流量整体下滑，需重点排查 `seller_import_abnormal_offer` 中是否有"访客下跌"的商品

## Step 5 — 增长引擎识别（`seller_top_offer`，**必须多次调用**）

**根据分析需要，按需调用以下榜单**（不得只调用 `payAmt` 一种）：

| 调用 | 识别目标 |
|------|----------|
| `--order_by payAmt` | **成交主力**：撑 GMV 的核心商品 |
| `--order_by uv` | **流量主力**：流量大但需结合 `payRate` 看转化 |
| `--order_by payNewByrCnt` | **拉新主力**：贡献新客的商品 |
| `--order_by itemMultiByrCnt` | **复购主力**：承担复购经营的商品 |

**强制最低调用集**：
- 若只做总体诊断 → 至少调用 `payAmt` + `uv` + `payNewByrCnt` 或 `itemMultiByrCnt` 中的一个
- 若用户问"靠什么增长" → 至少调用全部 4 种

**交叉判断逻辑**：

| 特征 | 商品类型 | 行动建议方向 |
|------|---------|------------|
| 高 `payAmt` + 高 `payRate` | **核心成交款** | 重点保护，加大投入 |
| 高 `uv` + 低 `payRate` | **高流量低转化款** | 优化转化（详情/价格/评价） |
| 低 `uv` + 高 `payRate` | **潜力款** | 加流量放量 |
| 高 `payNewByrCnt` | **拉新款** | 投放裂变、首单优惠 |
| 高 `itemMultiByrCnt` + 高 `itemMultiByrPayAmt` | **复购款** | 老客经营、会员激励 |

**关键交叉验证**：
- 优秀商品列表里是否同时出现在异常商品列表中？**若重叠 → 高优先级风险**（核心商品在恶化）
- 异常商品中是否有店铺 TOP 成交品？**若有 → 必须最高优先级修复**

## Step 6 — 活动效果分析（`seller_activity_registered_info`）

**判断维度**：

| 维度 | 判断方式 |
|------|---------|
| 是否带来访客 | `activityItmVisitorUv` / `activityItmVisitorPv` 是否显著 |
| 是否承接转化 | `activityItmOrderCnt` / `activityItmGmv` 与 PV 的比值 |
| 是否优于同行 | `activityItmGmv` vs `peerActivityItmGmv`，`activityItmVisitorPv` vs `peerActivityItmVisitorPv` |
| 是否值得复制 | GMV 高 + 优于同行 + 流量稳定 |

**结论分类**：

| 结论 | 特征 |
|------|------|
| **有效活动** | GMV 高且优于同行基准 |
| **高曝光低成交** | PV 高但订单/GMV 显著低于同行 → 承接差，详情/选品/价格有问题 |
| **低曝光低产出** | PV 和 GMV 均低 → 活动未拿到流量红利或参与不充分 |
| **可复制活动** | 有效活动中可识别共性（类目、玩法、时段）→ 后续放大 |

**注意**：若 `peerActivityItmGmv` 等同行字段缺失，仅做内部对比，不强行下"是否优于同行"结论。

## Step 7 — 地域结构分析（`seller_customer_business_province`）

**判断维度**：

| 维度 | 判断方式 |
|------|---------|
| **集中度** | TOP 1 / TOP 3 省份 `payBuyerNumRate` 之和 |
| **依赖风险** | TOP 1 省份占比 > 40% → 高度依赖；> 60% → 严重依赖 |
| **覆盖面** | 出现的省份数量、长尾省份分布 |
| **潜力区域** | 占比中等（5-15%）但接近核心区域的省份，可作为下一阶段拓展目标 |

**结论必须结合店铺类目特征**：某些类目（如本地服务、生鲜）天然区域集中，不应一刀切判定为"风险"。

## Step 8 — 头部老客稳定性（`seller_customer_detail`）

**判断维度**：

| 维度 | 判断方式 |
|------|---------|
| **依赖度** | TOP N 头部老客 `payAmount` 占店铺总 `payAmt` 的比例 |
| **活跃度** | `lastPayDate` 距今天数，看头部客户是否近期还在买 |
| **流失风险** | 历史 `payAmtAll` 高但本期 `payAmount` = 0 或显著低 → 走弱风险 |
| **客户结构** | `identityName` / `buyerCreditLevel` / `custAreaName` 分布特征 |
| **获客来源** | `fstFromAd = true` 占比高 → 广告获客型；占比低 → 自然沉淀型 |

**结论分类**：

| 结论 | 特征 |
|------|------|
| **基本盘稳** | 头部老客持续活跃，贡献稳定 |
| **依赖风险高** | 少数头部客户占比极高，单客流失即重创 GMV |
| **存在断层风险** | 高累计价值客户近期不活跃，需立即召回 |
| **广告依赖型** | 头部客户多数来自广告，需关注获客成本与生命周期 |
