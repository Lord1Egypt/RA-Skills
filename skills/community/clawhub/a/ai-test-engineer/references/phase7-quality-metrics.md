## M43: 测试覆盖率度量

### 核心方法论
**四维覆盖率模型**：代码覆盖率 × 需求覆盖率 × 风险覆盖率 × 业务覆盖率

### 专家操作流程
1. **代码覆盖率**：
   | 指标 | 含义 | 建议目标 |
   |------|------|---------|
   | 行覆盖率 | 至少执行一次的代码行比例 | ≥80% |
   | 分支覆盖率 | 每个if/else分支至少覆盖一次 | ≥70% |
   | 函数覆盖率 | 至少调用一次的函数比例 | ≥85% |
   | 路径覆盖率 | 所有可能路径的组合覆盖 | ≥60%（复杂系统） |
   - **注意**：覆盖率不等于质量！高覆盖+低断言=虚假安全感
2. **需求覆盖率**：
   ```
   需求覆盖率 = 有对应用例的需求数 / 总需求数 × 100%
   需求深度覆盖 = 对每个需求，有害覆盖(Happy Path)+边界覆盖+异常覆盖
   目标：需求覆盖率≥95%，深度覆盖≥80%
   ```
3. **风险覆盖率**：
   | 风险等级 | 覆盖率目标 | 深度要求 |
   |---------|-----------|---------|
   | P0（致命） | 100% | 所有设计方法全覆盖 |
   | P1（严重） | 100% | 等价类+边界值+关键异常 |
   | P2（一般） | ≥80% | 等价类+边界值 |
   | P3（轻微） | ≥50% | Happy Path |
4. **业务覆盖率**：
   - 端到端业务流程覆盖：每个核心用户旅程有对应的E2E测试
   - 用户角色覆盖：每种角色权限都有对应测试
   - 部署模式覆盖：每种部署架构都有对应测试

### 覆盖率报告模板
```
## 测试覆盖率仪表盘
| 维度 | 当前值 | 目标值 | 差值 | 趋势 |
|------|--------|--------|------|------|
| 代码-行 | 82% | 80% | +2% | ↑ |
| 代码-分支 | 68% | 70% | -2% | → |
| 需求 | 95% | 95% | 0% | → |
| 风险-P0 | 100% | 100% | 0% | - |
| 业务 | 90% | 90% | 0% | ↑ |
```

## M44: 缺陷密度与趋势分析

### 核心方法论
**缺陷度量四指标**：缺陷密度 + 缺陷发现率 + 缺陷清除率 + 缺陷收敛度

### 专家操作流程
1. **缺陷密度（Defect Density）**：
   ```
   缺陷密度 = 发现的缺陷数 / 功能点数（或代码行数×1000）
   对比基准：
   - 行业平均水平：2-5个缺陷/千行代码
   - 高质量目标：<1个缺陷/千行代码
   - 按模块对比，识别高缺陷密度模块
   ```
2. **DDP（Defect Detection Percentage）**：
   ```
   DDP = 测试阶段发现的缺陷 / (测试阶段发现 + 发布后发现) × 100%
   目标：DDP ≥ 95%（即线上逃逸率<5%）
   ```
3. **缺陷发现率趋势**：
   - 理想曲线：测试前期高（大量缺陷暴露）→ 中期降低 → 后期趋近于0
   - 异常模式：
     - 始终高位不降 → 版本质量差，可能需要延期
     - 突然飙升 → 引入新模块/需求变更
     - 提前触底 → 可能测试不够深入
4. **缺陷收敛判断**：
   ```
   收敛条件（全部满足）：
   1. 连续5天新增缺陷数递减
   2. 修复率 > 发现率（未解决缺陷在减少）
   3. P0/P1缺陷=0
   4. Reopen率 < 5%
   5. 最近3天无P0缺陷
   ```

## M45: 测试效率度量

### 核心方法论
**效率三维度**：生产率 × 执行效率 × 自动化ROI

### 专家操作流程
1. **用例生产率**：
   ```
   用例生产率 = 有效用例数 / 投入人天
   有效用例：经过评审且实际执行中找到≥1个缺陷的用例
   参考值：新手5-8个/天，熟练10-15个/天，专家15-25个/天
   ```
2. **缺陷发现效率**：
   ```
   缺陷发现率 = 发现的缺陷数 / 测试人天
   参考值：2-5个/人天（取决于产品成熟度）
   ```
3. **自动化ROI计算**：
   ```
   ROI = (自动化节省的手工执行时间 - 自动化投入时间) / 自动化投入时间 × 100%

   自动化投入时间 = 编写脚本时间 + 维护时间 + 失败分析时间
   节省时间 = 手工执行时间 × 执行次数 - 自动化执行时间 × 执行次数

   示例：
   编写时间：4h，维护时间：1h/月×12月=12h
   手工执行时间：2h/次，自动化执行时间：0.1h/次
   年执行次数：50次

   投入 = 4 + 12 = 16h
   节省 = (2-0.1) × 50 = 95h
   ROI = (95 - 16) / 16 × 100% = 494%
   ```
4. **效率优化方向**：
   - 用例瘦身：减少冗余用例（目标：减少20%用例但保持相同缺陷发现率）
   - 自动化率提升：优先自动化高频执行的用例
   - 并行化：测试数据准备并行、执行并行、结果分析并行

## M46: 质量模型评估

### 核心方法论
**ISO 25010 软件质量模型**：8大质量特性 × 多子特性

### 专家操作流程
1. **ISO 25010 八维质量评估框架**：
   | 质量特性 | 子特性 | 度量方法 | 评分(1-5) |
   |---------|--------|---------|----------|
   | 功能适用性 | 功能完整性/正确性/适当性 | 需求覆盖率/缺陷密度 | |
   | 性能效率 | 时间行为/资源利用/容量 | P99耗时/CPU/内存 | |
   | 兼容性 | 共存性/互操作性 | 兼容性测试矩阵通过率 | |
   | 易用性 | 可识别/可学习/可操作/UI美学 | 可用性测试/SUS量表 | |
   | 可靠性 | 成熟性/可用性/容错性/可恢复性 | MTBF/可用率/故障恢复时间 | |
   | 安全性 | 机密性/完整性/不可否认性/可追溯性 | OWASP扫描/渗透测试 | |
   | 可维护性 | 模块化/可重用/可分析/可修改/可测试 | 代码复杂度/耦合度 | |
   | 可移植性 | 适应性/可安装/可替换 | 跨平台测试通过率 | |
2. **质量评分卡**：
   ```
   加权总分 = Σ(特性评分 × 权重)
   权重建议（通用产品）：
   - 功能适用性：25%
   - 可靠性：20%
   - 性能效率：15%
   - 安全性：15%
   - 易用性：10%
   - 可维护性：10%
   - 兼容性：3%
   - 可移植性：2%
   ```
3. **发布标准判定**：
   - 总分≥4.0 且 无单项<2.5 → 建议发布
   - 总分3.0-4.0 或 有单项<2.5 → 有条件发布（标注风险）
   - 总分<3.0 或 有单项<1.5 → 不建议发布

## M47: 测试ROI分析

### 核心方法论
**全链路成本-收益模型**：缺陷预防价值 + 自动化节省 + 质量信心价值

### 专家操作流程
1. **测试成本核算**：
   ```
   总测试成本 = 人力成本 + 工具成本 + 环境成本 + 培训成本

   人力成本 = (测试工程师人天数 × 日薪) + (开发配合人天数 × 日薪)
   工具成本 = License费 + 云服务费 + 设备租赁费
   环境成本 = 服务器费 + 数据准备费
   培训成本 = 学习时间 × 日薪
   ```
2. **测试收益核算**：
   ```
   A. 缺陷预防价值 = Σ(每个发现的缺陷 × 修复成本差)
      修复成本差 = 上线后修复成本 - 测试阶段修复成本
      参考倍数：需求阶段1× / 设计阶段3× / 编码阶段10× / 测试阶段30× / 上线后100×

   B. 自动化节省价值 = 手工执行时间 × 人工费率 × 执行次数 - 自动化投入

   C. 质量信心价值 = （发布信心增强带来的业务收益 × 信心系数）
      信心系数：0.05-0.15（难以精确，但不可忽略）
   ```
3. **ROI综合判定**：
   ```
   测试ROI = (总收益 - 总成本) / 总成本 × 100%
   ROI > 200%：优秀（测试投入效果显著）
   ROI 100-200%：良好
   ROI 50-100%：合理
   ROI < 50%：需优化测试策略
   ```

## M48: 质量仪表盘设计

### 核心方法论
**实时可视化** + **趋势预测** + **决策建议** + **多层级视图**

### 专家操作流程
1. **仪表盘层级设计**：
   - L1 - 管理层视图：发布就绪状态 / 整体质量评分 / 关键风险 / 趋势摘要
   - L2 - 项目经理视图：进度 / 缺陷趋势 / 阻塞项 / 资源利用
   - L3 - 测试工程师视图：执行详情 / 缺陷分布 / 覆盖率 / Flaky用例
2. **核心可视化组件**：
   | 组件名 | 类型 | 数据源 | 刷新频率 |
   |--------|------|--------|---------|
   | 发布就绪度 | 仪表盘（红黄绿） | 质量门禁状态 | 实时 |
   | 缺陷趋势 | 折线图 | 缺陷管理系统 | 每日 |
   | 测试执行 | 饼图+进度条 | 测试管理平台 | 实时 |
   | 模块质量热力图 | 热力矩阵 | 缺陷密度+覆盖率 | 每迭代 |
   | 自动化健康度 | 状态卡片 | CI/CD流水线 | 每次构建 |
   | 风险雷达 | 雷达图 | 风险管理 | 每迭代 |
   | 效率趋势 | 趋势图 | 测试管理系统 | 每周 |
3. **仪表盘数据流**：
   ```
   测试管理系统 ─┐
   CI/CD Pipeline ─┤
   缺陷管理系统 ──┼──→ 数据聚合层 → 指标计算引擎 → 可视化层
   代码仓库 ──────┤
   项目管理工具 ──┘
   ```
4. **预警规则配置**：
   | 指标 | 黄色预警 | 红色告警 |
   |------|---------|---------|
   | 通过率 | <95% | <90% |
   | P0未解决缺陷 | =1 | ≥2 |
   | 进度偏差 | >15% | >30% |
   | 自动化成功率 | <95% | <90% |
   | 代码覆盖率 | <目标-5% | <目标-10% |
