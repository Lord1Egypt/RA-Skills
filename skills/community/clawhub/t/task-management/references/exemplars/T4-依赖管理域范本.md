## T4 依赖管理域 范本

### 范本概述
- **域ID**: T4
- **域名称**: 依赖管理域
- **任务数量**: 5
- **范本状态**: 可用

### 任务执行范本

#### T4-01 依赖类型定义
**执行步骤**:
1. 分析依赖关系特征
2. 识别依赖类型
3. 定义依赖含义
4. 说明使用场景
5. 确认依赖类型定义

**输入示例**:
- 依赖关系特征：任务A完成才能开始任务B
- 业务场景：软件开发任务依赖关系

**输出示例**:
- 依赖类型：blockedBy（被阻塞）
- 依赖含义：任务B被任务A阻塞，任务A完成后任务B才能开始
- 使用场景：任务间存在前后依赖关系时使用
- 示例：前端开发依赖系统设计完成

**注意事项**:
- 依赖类型要完整，覆盖所有可能类型
- 依赖含义要清晰，避免歧义
- 使用场景要具体，便于应用

#### T4-02 依赖解析
**执行步骤**:
1. 收集依赖关系信息
2. 解析依赖链路
3. 确定可执行任务
4. 验证依赖关系
5. 输出依赖解析结果

**输入示例**:
- 依赖关系：
  - 需求分析 → 系统设计
  - 系统设计 → 前端开发
  - 系统设计 → 后端开发
  - 前端开发 → 测试
  - 后端开发 → 测试

**输出示例**:
- 依赖链路：
  ```
  需求分析 → 系统设计 → 前端开发 → 测试
                    ↓
                后端开发 → 测试
  ```
- 可执行任务：需求分析（无依赖）
- 依赖验证：无循环依赖，依赖关系合理

**注意事项**:
- 依赖收集要全面，避免遗漏
- 依赖解析要准确，避免错误
- 依赖验证要严格，避免问题

#### T4-03 循环检测
**执行步骤**:
1. 设计循环检测算法
2. 执行循环检测
3. 处理循环依赖
4. 制定预防措施
5. 确认循环检测结果

**输入示例**:
- 依赖关系：
  - 任务A → 任务B
  - 任务B → 任务C
  - 任务C → 任务A

**输出示例**:
- 循环检测算法：深度优先搜索（DFS）算法
- 循环检测结果：发现循环依赖 A → B → C → A
- 处理策略：重新设计依赖关系，打破循环
- 预防措施：在添加依赖关系时进行循环检测

**注意事项**:
- 检测算法要准确，避免漏检
- 处理策略要有效，避免问题复发
- 预防措施要可行，避免形同虚设

#### T4-04 依赖变更处理
**执行步骤**:
1. 检测依赖变更
2. 分析变更影响
3. 设计处理流程
4. 执行变更处理
5. 确认变更结果

**输入示例**:
- 依赖变更：新增依赖关系，任务D依赖任务C
- 当前依赖关系：任务A → 任务B → 任务C

**输出示例**:
- 变更检测：检测到新增依赖关系
- 变更影响分析：任务D需要等待任务C完成，可能影响整体进度
- 处理流程：更新依赖关系图，重新计算关键路径
- 变更执行：更新依赖关系，通知相关任务负责人

**注意事项**:
- 变更检测要及时，避免延误
- 影响分析要全面，避免遗漏
- 处理流程要规范，避免混乱

#### T4-05 并行机会识别
**执行步骤**:
1. 分析依赖关系
2. 识别无依赖关系的任务
3. 确认并行条件
4. 规划并行执行
5. 确认并行识别结果

**输入示例**:
- 依赖关系：
  - 需求分析 → 系统设计
  - 系统设计 → 前端开发
  - 系统设计 → 后端开发
- 并行条件：前端开发和后端开发无依赖关系

**输出示例**:
- 可并行任务：前端开发、后端开发
- 并行条件：系统设计完成后，前端和后端可同时开发
- 并行限制：需要足够的开发资源
- 并行执行计划：系统设计完成后，同时启动前端和后端开发

**注意事项**:
- 并行任务要无依赖关系
- 并行条件要满足
- 并行限制要考虑
