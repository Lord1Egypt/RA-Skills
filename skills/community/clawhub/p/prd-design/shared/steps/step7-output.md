## 7.0 生成前链路预检（🆕 v3.3，**所有生成动作的前置门禁**）

> **时机**：在开始写 PRD 任何章节和生成 HTML 之前执行。
> **目的**：提前识别上下文中逻辑不清晰、数据来源不明、链路断裂的地方——能从已有上下文推断的自行补全，推断不了的集中问用户确认，确认完毕后再动笔生成。

### 执行流程

#### 第一步：绘制链路地图

读取已确认的 `02-flows/`、`03-business-objects-er.md`、`04-modules.md`，构建完整链路映射表：

```
操作节点        → 前端触发动作     → 后台逻辑(BL)     → 写入字段              → 下游读取位置
─────────────────────────────────────────────────────────────────────────────────────
M1.1 新建模板   → 点击「提交」      → BL-001 名称校验  → template.name/type等  → M2.1 任务创建下拉
M2.1 创建任务   → 选择模板后提交    → BL-002 自动分配  → task.assigneeId       → M3.1 我的任务列表
M3.1 执行质检   → 提交质检结果      → BL-003 状态联动  → result.score等        → M4.1 审核详情页
M4.1 审核结果   → 点击「通过/驳回」 → BL-004 通知发送  → task.status=完成      → 统计看板
```

#### 第二步：研发视角逐节点深度拆解（🆕 v3.4）

> **颗粒度标准**：每个操作节点的后台逻辑必须拆解到「研发可直接写代码执行」的程度。
> 对每个涉及数据处理/自动逻辑的节点，逐一回答以下 7 个研发关键问题：

| # | 问题维度 | 必须回答的内容 |
|---|---------|--------------|
| R1 | **数据来源** | 这个操作需要的输入数据从哪张表读取？用什么字段过滤？有无关联查询？ |
| R2 | **数据抽取规则** | 如果需要从大量数据中取子集，抽取算法是什么？随机/顺序/按权重/按规则？抽取比例/数量如何确定？ |
| R3 | **分配/路由规则** | 数据或任务分配给人/模块的规则是什么？优先级顺序？负载均衡算法？冲突时怎么处理？ |
| R4 | **计算/转换逻辑** | 涉及分数/金额/状态推导时，计算公式是什么？入参是什么字段？精度/舍入规则？ |
| R5 | **写入规则** | 结果写到哪张表哪些字段？是 INSERT 还是 UPDATE？有无唯一约束需处理？批量还是单条？ |
| R6 | **异常与边界** | 输入数据为空/不足时怎么处理？并发写入怎么防重？超时/失败时是回滚还是部分提交？ |
| R7 | **幂等性** | 该操作能否被重复触发？重复触发时结果是否一致？如何保证幂等？ |

**示例（质检任务生成节点）**：

```
节点：M2.1 生成质检任务（含抽检 + 分配坐席）

R1 数据来源
  - 抽检数据来源：call_record 表，过滤条件：
      call_date BETWEEN task.start_date AND task.end_date
      AND agent_id IN (task.scope_agent_ids)
      AND duration >= template.min_call_duration
      AND status = 'completed'
  - 坐席列表来源：agent 表，过滤条件：
      group_id = task.group_id AND is_active = true

R2 数据抽取规则
  - 抽取算法：按 template.sample_method 分支：
      · 'random'：对符合条件的记录执行 ORDER BY RAND() LIMIT n
      · 'proportional'：按 agent_id 分组，每组各取 ceil(n * group_ratio) 条
      · 'rule'：按 template.sample_rules JSON 中的条件顺序过滤
  - 抽取数量 n = floor(total_count * template.sample_rate)，最小值为 template.min_sample，最大值为 template.max_sample
  - 若符合条件的记录总数 < template.min_sample，抽取全部并在 task.remark 写入「数据不足，已全量抽取」

R3 分配规则
  - 分配算法：轮询 + 工作量上限双约束
      1. 查询每个坐席当日已分配未完成的质检项数量 assigned_count
      2. 过滤掉 assigned_count >= template.max_per_agent 的坐席
      3. 对剩余坐席按 assigned_count ASC 排序
      4. 逐条抽检记录依次分配给排序最前的坐席，每分配一条 assigned_count+1 并重新排序
  - 若所有坐席均已达上限：任务整体进入「待分配」状态，触发 BL-008 通知组长

R4 计算逻辑
  - 无评分计算（评分在 M3.1 执行阶段）

R5 写入规则
  - 写入 inspection_task 表（1条）：status='pending', created_at=now(), sample_count=n
  - 批量写入 inspection_item 表（n条）：
      inspection_item(task_id, call_record_id, agent_id, status='unassigned')
  - 使用事务：inspection_task 和所有 inspection_item 原子提交，任一失败全部回滚

R6 异常与边界
  - call_record 无符合条件数据：返回错误「无可抽检通话记录，请检查时间范围或人员范围」，不创建 task
  - 坐席列表为空：返回错误「当前分组无在岗坐席」，不创建 task
  - RAND() 抽取并发：同一时刻多个任务对同一批数据抽取，用 SELECT ... FOR UPDATE 锁定 call_record 的 inspected 字段防止重复抽取

R7 幂等性
  - 任务创建接口：前端提交时携带 idempotency_key（UUID），服务端以 idempotency_key 为唯一键，重复提交返回已创建的 task_id，不重新执行抽检逻辑
```

#### 第三步：逐行扫描 5 个接驳点，标记疑问

### 🆕 v3.5 分包规则（按字段流转链路分组，替代逐模块独立生成）

> **核心思路**：同一条字段流转链路上的模块共享上下文，打包一起生成；不同链路的模块分包隔离。
> 相比逐模块生成（每模块重新载入全量文件），此方案：
> - **减少 token**：锚点只用字段流转索引表（≤50行），不重复载入 ER/BL/用户故事全文
> - **防止上下文压缩**：同包内模块连续生成，字段引用链路完整；跨包隔离防止无关模块干扰

**分包步骤**：

1. 读取 `03.6-field-flow-index.md`，识别所有标 ✅ 的跨模块关键字段
2. 以关键字段的「产生于模块」→「被读取于模块」为边，构建模块依赖图
3. 将依赖图中强连通的模块归为同一包，无依赖关系的模块归为独立包
4. 每包不超过 3 个模块；若某包超过 3 个，按流转顺序拆为两个子包，子包间通过字段流转索引表传递上下文

**分包示例（质检系统）**：

```
包A：M1(质检模板管理) + M2(创建质检任务)
     共享字段：template_id, template.sample_rate, template.sample_method
     原因：M2 创建任务强依赖 M1 的模板配置字段

包B：M3(执行质检) + M4(审核质检结果)
     共享字段：item.call_record_id, result.score, result.status, task.status
     原因：M4 审核逻辑直接读取 M3 写入的 result 字段

包C：M5(统计看板)
     依赖：读取包A的 task.* 聚合 + 包B的 result.* 聚合
     原因：纯读取，无写入依赖，独立生成
```

**每包生成时的上下文注入规则**：

生成每包前，只注入以下两项，不载入其他文件：

```
注入1：字段流转索引表全文（03.6-field-flow-index.md，≤50行，全程固定）
注入2：当前包的极简锚点摘要，格式如下：

[包A 锚点]
包含模块：M1 质检模板管理 / M2 创建质检任务
用户故事：US-001 / US-002
输入来源：M1 无前置输入 / M2 读取 template_id
写入字段：M1→template(id,name,type,status,sample_rate) / M2→task(id,template_id,agent_ids,status)
下游读取：M3 读取 task.agent_ids + item.call_record_id
BL关联：BL-001(分配坐席) / BL-002(抽检计算)
状态起点：草稿→启用（M1）/ 待执行（M2）
```

**执行规则**：
- **禁止在生成过程中向用户输出任何中间内容**，所有阶段静默执行
- 每包生成时**只注入「字段流转索引表 + 当前包锚点摘要」**，不重新载入 ER/用户故事/BL 全文
- 包内各模块连续生成，中间不插入任何其他操作
- 第 9 节合并时不修改任何模块内容，严格原文拼接；节首加模块索引表：`| 模块编号 | 模块名称 | 关联用户故事 | 所属包 |`
- 全部 13 节合并完毕，写入 `07-output/prd.md`，**一次性用 `present_files` 交付**

### PRD 各节格式要求

#### PRD 第 9 节 功能模块详述（每模块 7 层，独立生成）

每个操作节点必须含 7 层（缺一不可）：
- Layer 1 入口与触发（入口位置/触发控件/可见条件/可用条件/是否二次确认）
- Layer 2 完整交互流：逐步描述「用户动作 → 系统响应」，覆盖主路径+条件分支+异常路径，**页面上每一个可点击元素（含操作列的编辑/删除/查看图标）都必须有对应条目，不允许遗漏**
- Layer 3 界面形态（ASCII 草图 + 形态/尺寸/关闭方式说明）
- Layer 4 字段表（7 列：名/类型/来源/必填/默认值/约束/备注）；下拉/选择器字段必须额外注明「选项来源」（固定枚举 or 动态接口+过滤条件+空态提示）；**每个字段必须标注「流转去向」——该字段在下一环节由谁读取、在哪个页面回显，若本环节是终点则标注「终态存档」**
- Layer 5 字段交互生命周期（每字段 4 阶段：前置填充/输入中/失焦时/提交时）
- Layer 6 提交与反馈（按钮文案/校验顺序/loading/成功/失败/自动关闭）
- Layer 7 边界与异常清单（默认值/必填空/数值/业务逻辑/并发/异常）

```markdown
## 第 6.5 节 — 核心业务对象定义

> 本节列出流程必不可缺的核心业务对象，所有角色操作和流程动作均围绕这些对象展开。

### 核心对象清单

| 对象名 | 一句话描述 | 为什么是核心 |

| 质检任务 | 基于模板创建的一次具体质检工作单 | 是质检员执行的直接载体，也是流程驱动的核心 |

### [对象名] — 要素定义
（每个对象一节，输出要素表，格式同 03-business-objects-er.md）
```

### 第15项：JS语法强制验证（🆕 每次生成/修改原型后必须执行）

> **触发时机**：原型首次生成完毕后；每次对原型进行任何修改后（包括增删页面、修改函数、修改HTML结构）。

**执行方式**：提取 `<script>` 块内容，用 `node --check` 验证语法，returncode≠0 时**禁止保存输出，必须先修复**。

**高危场景强制规则（不需等报错，生成时直接禁止）**：

在 JS 函数中动态构建含 `onclick` 属性的 HTML 字符串时，**禁止字符串拼接方式**，必须使用 DOM API：

```javascript
// ❌ 禁止 — 单引号三层嵌套，必然导致 SyntaxError
row.cells[4].innerHTML = '<button onclick="fn(\''+val+'\')">' + label + '</button>';

// ✅ 必须 — 用 createElement + onclick 赋值
const btn = document.createElement('button');
btn.textContent = label;
btn.onclick = function() { fn(val); };
row.cells[4].appendChild(btn);
```

**自检检查项**：
```
□ node --check 返回 returncode=0（无语法错误）
□ 全文无「innerHTML = '...' + var + '...'」内嵌 onclick 的字符串拼接
□ 反引号数量为偶数（模板字符串全部闭合）
□ 花括号总数 opens == closes
□ 所有 getElementById 调用的 ID 在 DOM 中存在
```

### 第16项：HTML结构完整性验证（🆕 每次修改HTML结构后必须执行）

> **触发时机**：每次对原型进行增删 HTML 区块的操作（增加/删除页面、增加/删除抽屉、删除Tab内容块）后必须执行。

**执行方式**：用字符计数逐字符遍历（不用正则）验证受影响区块的 `<div>` 开闭平衡：

```python
# 标准验证方法（逐字符，防正则漏判SVG等内嵌内容）
depth = 0
for i in range(len(html)):
    if html[i:i+4] == '<div' and html[i+4] in ' \n\t>':
        depth += 1
    elif html[i:i+6] == '</div>':
        depth -= 1
# depth == 0 则平衡，否则报错
```

