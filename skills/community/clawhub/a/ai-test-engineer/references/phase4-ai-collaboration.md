## M23: AI生成测试用例

### 核心方法论
**Prompt工程驱动的用例生成**：结构化输入 → 专家级Prompt → AI生成 → 人工审查 → 迭代优化

### 专家操作流程
1. **构建Prompt四要素**：
   ```
   要素1 - 角色设定：你是一名10年经验的资深测试工程师，精通[领域]测试
   要素2 - 上下文注入：
     - 需求描述/PRD/用户故事
     - 接口文档/API Spec/Proto文件
     - 技术架构说明
     - 历史缺陷数据（可选）
   要素3 - 任务描述：
     - 测试方法指定（等价类/边界值/判定表/场景法）
     - 用例格式要求（8要素模板）
     - 数量要求（每种类型的数量）
   要素4 - 输出约束：
     - 格式：Markdown表格
     - 语言：中文
     - 质量标准：预期结果可量化、步骤可执行
   ```
2. **分层生成策略**：
   - 第一层：生成测试场景清单（让AI先列出要测什么）
   - 第二层：对每个场景生成具体用例（5-10个/轮，避免超出token限制）
   - 第三层：生成边界值和异常值补充用例
   - 第四层：生成数据构造脚本
3. **Prompt模板：功能测试用例**：
   ```
   你是一名资深测试工程师。请为以下功能生成测试用例。

   【需求】
   {粘贴需求文档}

   【测试方法】
   请使用等价类划分 + 边界值分析 + 场景法

   【输出格式】
   | 编号 | 标题 | 前置条件 | 测试步骤 | 预期结果 | 测试数据 | 优先级 |
   |-----|------|---------|---------|---------|---------|--------|

   【要求】
   1. 正向用例覆盖Happy Path
   2. 边界用例覆盖所有数值/长度边界
   3. 异常用例覆盖至少5种异常场景
   4. 预期结果可量化（状态码、文案、数值）
   5. 优先级标注P0-P3及理由
   ```
4. **AI生成后的审查清单**：
   - [ ] 业务逻辑是否正确（AI可能误解业务规则）
   - [ ] 测试数据是否真实（AI可能生成不存在的测试数据）
   - [ ] 是否有遗漏的边界条件
   - [ ] 是否有幻觉（AI编造了不存在的功能/字段/错误码）
   - [ ] 预期结果是否可执行验证

## M24: AI辅助测试脚本

### 核心方法论
**自然语言 → AI生成脚本 → 本地验证 → 调试迭代 → 纳入自动化套件**

### 专家操作流程
1. **脚本生成Prompt模板**：
   ```
   你是一名自动化测试工程师。请为以下测试用例生成[语言/框架]的自动化脚本。

   【测试用例】
   {粘贴用例内容}

   【技术要求】
   - 语言：Python/JavaScript/Java
   - 框架：pytest+requests / Playwright / Selenium / JUnit
   - 风格：遵循AAA模式（Arrange-Act-Assert）
   - 错误处理：包含try-catch和断言信息
   - 数据驱动：使用参数化（不硬编码测试数据）
   - 注释：关键步骤用中文注释

   【环境信息】
   - 接口Base URL：{URL}
   - 认证方式：{Token/JWT/OAuth}

   【依赖库】（已经安装的）
   {列出requirements.txt}
   ```
2. **分步生成策略**（复杂脚本）：
   - Step 1: 生成测试数据构造函数
   - Step 2: 生成前置条件准备代码
   - Step 3: 生成核心测试逻辑
   - Step 4: 生成清理/Teardown代码
   - Step 5: 生成参数化配置
3. **脚本质量自检**：
   - [ ] 独立运行（不依赖其他用例的执行顺序）
   - [ ] 幂等性（多次运行结果一致）
   - [ ] 清晰的断言失败信息
   - [ ] 适当的等待（不使用固定sleep）
   - [ ] 失败时保留现场（截图/日志/响应体）
4. **常见AI脚本问题**：
   | 问题 | 频次 | 对策 |
   |------|------|------|
   | 硬编码测试数据 | 高 | 明确要求参数化 |
   | 缺少断言 | 中 | Prompt中强调"每条Assert包含失败描述" |
   | 过时的API/选择器 | 中 | 提供当前框架版本号 |
   | 无Teardown | 中 | 明确要求包含清理逻辑 |
   | 缺少import | 低 | Prompt中要求完整的可运行脚本 |

## M25: AI缺陷分析

### 核心方法论
**缺陷数据驱动分析**：AI自动分类 → 根因推断 → 模式识别 → 预测建议

### 专家操作流程
1. **缺陷分类分析**（输入一批缺陷，AI自动）：
   - 按类型分类：功能/性能/安全/兼容性/UI/数据
   - 按根因分类：逻辑错误/边界遗漏/并发/环境/需求理解/编码疏忽
   - 按严重度分布：Blocker/Critical/Major/Minor/Trivial
   - 按模块分布：识别高缺陷密度模块
   - 按引入阶段：需求/设计/编码/集成/测试
2. **缺陷趋势分析**：
   - 缺陷发现率曲线（理想：先升后降）
   - 缺陷修复率 vs 发现率
   - 缺陷存活时间分布（均值/中位数/长尾）
   - Reopen率趋势
3. **AI根因分析Prompt**：
   ```
   基于以下缺陷数据，请分析：
   1. 最常见的3个根因类型及占比
   2. 每个根因对应的典型案例（缺陷ID）
   3. 是否有新出现的缺陷模式
   4. 建议的预防措施（按模块/开发人员）
   5. 预测接下来2周可能出现的缺陷类型
   ```
4. **缺陷模式知识库建设**：
   - 从历史缺陷中提取：症状 → 根因 → 修复方案
   - AI辅助打标和索引
   - 新缺陷入库时自动匹配历史模式

## M26: 智能回归测试选择

### 核心方法论
**基于代码变更的精准回归**：代码Diff → 影响面分析 → 选择最小回归集

### 专家操作流程
1. **代码变更影响分析**：
   ```
   输入：代码变更Diff + 调用链/依赖关系图
   输出：受影响的模块/接口/功能点列表
   ```
2. **风险分级**：
   - 高风险变更：核心模块修改、数据库Schema变更、公共库修改 → 全量回归
   - 中风险变更：业务逻辑修改、接口参数变更 → 选择回归
   - 低风险变更：UI调整、文案修改、配置变更 → 冒烟回归
3. **智能选择算法**：
   - 基于调用链：变更方法 → 调用方 → 上游接口 → 关联功能
   - 基于数据流：变更数据表 → 读写该表的所有功能
   - 基于历史缺陷：该模块历史缺陷密集区 → 即使未变更也建议回归
   - 基于需求追溯：变更代码关联的需求 → 所有关联需求的用例
4. **AI辅助选择Prompt**：
   ```
   以下是本次版本的代码变更清单，请分析并推荐回归测试范围：

   【变更文件】
   {git diff --name-only}

   【关键变更摘要】
   {commit messages / PR description}

   【已有测试用例】（共N个）
   {用例列表或路径}

   请输出：
   1. 推荐回归的用例列表（用例ID+推荐理由）
   2. 风险评分（1-10分）和回归深度建议
   3. 可以不回归的用例及理由
   ```

## M27: AI测试数据生成

### 核心方法论
**智能造数工厂**：Schema理解 → 业务规则注入 → 数据生成 → 数据校验

### 专家操作流程
1. **数据需求理解**（向AI提供）：
   - 数据库Schema / API接口定义
   - 数据校验规则（格式/长度/枚举/关联）
   - 业务约束（如：订单金额必须>0，用户年龄必须≥18）
   - 数据量要求
2. **AI生成策略分类**：
   | 策略 | 用途 | Prompt要点 |
   |------|------|-----------|
   | 随机生成 | 通用测试数据 | 指定Faker库、字段类型映射 |
   | 边界生成 | 边界值测试 | 明确min/max/边界±1 |
   | 异常生成 | 异常场景测试 | NULL/空串/超长/错误类型/非法字符 |
   | 关联生成 | 场景测试 | 指定实体间关系（用户-订单-商品） |
   | 脱敏生成 | 安全测试 | 生成符合格式但不泄露真实信息的敏感数据 |
3. **数据生成脚本Prompt**：
   ```
   请生成Python脚本，创建以下测试数据：

   【数据要求】
   - 表/接口：{schema}
   - 数量：100条
   - 要求：
     1. 使用Faker生成逼真数据
     2. 包含10%边界值数据
     3. 包含5%异常数据
     4. 保持关联数据完整性（外键）
     5. 输出为可直接执行的SQL脚本

   【边界值定义】
   - 年龄：0, 1, 17, 18, 65, 120, 121
   - 金额：0, 0.01, 99999.99, 100000
   - 手机号：10位, 11位, 12位, 含字母
   ```
4. **数据校验**：
   - AI生成后验证：类型正确/格式合规/关联完整/不违反业务规则

## M28: 自然语言用例转换

### 核心方法论
**非结构化 → 结构化**：人类自然语言描述 → AI解析 → 标准测试用例

### 专家操作流程
1. **支持的输入形式**：
   - 产品一句话需求："用户可以通过手机号+验证码登录"
   - 用户故事："作为普通用户，我希望...以便..."
   - 会议纪要："讨论决定登录支持3种方式..."
   - 聊天记录：开发/产品在群里的功能讨论
   - 测试要点："需要测一下订单取消的各种情况"
2. **转换Prompt模板**：
   ```
   将以下关于「{功能名称}」的自然语言描述，转换为结构化测试用例。

   【原始描述】
   {粘贴自然语言描述}

   【要求】
   1. 提取所有功能点和测试场景
   2. 按以下格式输出标准测试用例：
      | 编号 | 标题 | 前置条件 | 步骤 | 预期 | 数据 | 优先级 |
   3. 自动补充隐含的边界值和异常场景
   4. 标注你对原始描述中不明确之处的假设
   5. 列出你认为需要确认的遗漏点
   ```
3. **转换质量检查**：
   - [ ] 没有遗漏原始描述中的任何功能点
   - [ ] 补充的边界/异常场景标注为"[AI补充]"
   - [ ] AI的假设明确列出，便于人工确认
   - [ ] 用例数量合理（不过度膨胀，不过度精简）

## M29: AI测试覆盖率分析

### 核心方法论
**多维覆盖率智能审计**：代码覆盖率 → 需求覆盖率 → 风险覆盖率 → 场景覆盖率

### 专家操作流程
1. **需求覆盖率分析**：
   - 输入：需求列表 + 用例列表
   - AI逐条匹配：每条需求 → 对应的用例（可能一对多）
   - 输出：需求-用例追溯矩阵 + 未覆盖需求清单
2. **代码覆盖率分析**：
   - AI分析覆盖率报告（JaCoCo/Istanbul/coverage.py）
   - 智能识别：
     - 低覆盖但高风险模块（需优先补充）
     - 高覆盖但测试质量可疑的模块（断言不足/只测了Happy Path）
     - 死代码/不可达分支（可建议移除）
3. **场景覆盖率分析**：
   ```
   输入以下信息，请分析测试场景覆盖情况：
   【用户旅程地图/核心流程】
   {核心业务流程}
   【已有用例】
   {用例列表或概要}
   【输出】
   1. 场景覆盖矩阵（每个场景 → 覆盖的用例 → 覆盖深度）
   2. 未覆盖的场景/子场景
   3. 过度测试的场景（ROI低的重复测试）
   ```
4. **覆盖率报告模板**：
   ```
   ## 测试覆盖率分析报告
   - 代码覆盖率：行__%/分支__%/函数__%
   - 需求覆盖率：__% （__/__条需求有对应用例）
   - 风险覆盖率：P0__% P1__% P2__% P3__%
   - 场景覆盖率：基本流__% 备选流__% 异常流__%
   - 覆盖率缺口（Top 5）：
     1. ...
   - 建议补充的测试方向：
     1. ...
   ```

## M30: AI辅助测试报告

### 核心方法论
**数据聚合 → AI洞察 → 可视化 → 决策建议**

### 专家操作流程
1. **数据聚合**（向AI提供）：
   - 测试执行统计：计划/执行/通过/失败/阻塞
   - 缺陷统计：按严重度/模块/状态的分布
   - 覆盖率数据：代码/需求/风险覆盖率
   - 历史对比：与上一版本的数据对比
2. **AI生成测试报告Prompt**：
   ```
   基于以下测试数据，生成一份专业的测试报告。

   【数据】
   执行统计：{数据}
   缺陷统计：{数据}
   覆盖率：{数据}

   【报告要求】
   1. 格式：Markdown，包含以下章节：
      - 测试概览（一句话总结 + 关键指标仪表盘）
      - 测试执行情况（计划vs实际、完成率）
      - 缺陷分析（分布图数据 + 趋势判断）
      - 质量评估（是否达到发布标准，附理由）
      - 风险与建议（Top 3风险 + 对应建议）
   2. 语言风格：专业、简洁、有数据支撑
   3. 结论明确：给出"建议发布/有条件发布/不建议发布"的明确判断
   ```
3. **AI洞察要点**：
   - 缺陷收敛趋势是否正常
   - 是否有异常模块（缺陷密度异常的高）
   - 测试效率是否合理
   - 与历史版本的对比结论
4. **报告质量检查**：
   - [ ] 结论有数据支撑
   - [ ] 风险标注了严重度和建议负责人
   - [ ] 没有模糊判断（如"质量还行"）
   - [ ] 历史对比有趋势分析而非简单罗列数字
