## 零、工作流重构执行逻辑

SKILL.md定义了重构三步法的核心理念。本节提供**执行层面的操作指令**——何时触发、如何执行、如何记录。

### 何时触发重构

| 触发条件 | 示例 |
|---------|------|
| 传统工作流涉及3+角色接力 | 产品→技术→运营→财务→法务 |
| 工作流中有3+份中间文档在角色间流转 | 需求文档→技术方案→开发→测试→部署报告 |
| 30%+时间花在协调沟通上 | 评审会、进度同步、跨部门对接 |
| 返工率≥30% | 信息传递失真导致反复修改 |

### 重构执行三步法

**Step 1：拆解**——列出传统工作流的每个环节，标记类型：

| 环节类型 | 标记 | 处理 | 说明 |
|---------|------|------|------|
| 核心环节 | ✅保留 | 评估AI自动化程度 | 事情本身的逻辑步骤 |
| 校准环节 | 🔶保留 | 保留为基元内分步校准点 | 过程中纠偏的中间产出物，防止跑偏后返工 |
| 传递环节 | ❌消除 | IPO链自动传递 | 人之间传递信息的中间步骤 |
| 协调环节 | ❌消除 | 无协作即无协调 | 管理多人协作的步骤 |
| 校验环节 | ⚡精简 | 保留关键节点，去掉冗余评审 | 防止出错的审核步骤（事后检查） |
| 格式环节 | ❌消除 | 零形式开销 | 满足组织流程的形式要求 |

**校准 vs 校验**：校验是事后检查对错，校准是过程中锁定方向。校准环节的中间产出物（如功能描述、方法调用结构）不只是传递文档，更是在过程中提供纠偏锚点——省掉校准看似简化，实则把纠偏压力推到最终产出，返工成本更高。

**合并判断准则**：AI技术上能一步完成 ≠ 一步完成最好。消除环节前，除了问"这是人的局限还是事情本身的需要"，还要问"这个中间产出物是否在过程中起校准作用"。如果起校准作用，即使技术上能合并，也不应合并——应保留为基元内分步校准点。

**Step 2：消除**——去掉所有❌标记的环节和⚡标记中的冗余部分；**校准环节🔶不可消除**——中间产出物起纠偏作用的必须保留为基元内分步校准点

**Step 3：重整**——将保留的✅核心环节、🔶校准环节（作为基元内分步校准点）和⚡关键校验节点，按AI能力模型重新编排为IPO基元链

### 重构前后对比模板

| 维度 | 重构前（传统工作流） | 重构后（AI能力模型） |
|------|-------------------|-------------------|
| 环节数 | ___个 | ___个 |
| 中间文档 | ___份 | 0份 |
| 参与角色 | ___个角色 | 1人+AI |
| 协作节点 | ___个会议/评审 | ___个关键校验点 |
| 端到端耗时 | ___ | ___ |

### 重构后的典型执行形态

| 形态 | 适用场景 | 执行方式 |
|------|---------|---------|
| **单步IPO** | 标准化任务 | 描述目标→AI直接产出 |
| **简短基元链** | 有阶段的中等复杂度任务 | 2-5个IPO基元串联 |
| **IPO+人工决策** | 涉及合规/客户/品牌 | AI执行+关键节点人工介入 |

**选择原则**：能单步IPO的不用基元链，能基元链的不加人工决策。

### IPO基元

IPO基元是重构后的基本执行单元：

```
I(输入) → P(处理) → O(输出)
```

- **I 输入**：该步骤需要的信息/素材/前置条件
- **P 处理**：对输入的加工操作（标注AI自治度：⬛全自动/🟨半自动/⬜辅助）
- **O 输出**：该步骤的产出物，作为下一个基元的输入

**基元间传递**：基元1.O → 基元2.I → ...，无需中间文档，无需协调会议。

**基元内分步**：一个基元内部可以有子步骤（如需求→方案→设计→代码），子步骤之间通过中间产出物形成校准点，每步可修改再推进。基元内分步不是基元间传递——不需要跨基元边界，不需要协作，但保留过程中的纠偏能力。

**基元数约束**：≤5。超过5说明还没充分消除补偿层，需回到"消除"步骤。基元内子步骤数不限，但每个子步骤必须有校准价值——纯传递性质的子步骤应合并。

### AI自治度标注

| 标记 | 含义 | 典型场景 |
|------|------|---------|
| ⬛ 全自动 | AI独立完成，人无需介入 | 竞品分析、内容生成、数据核算 |
| 🟨 半自动 | AI完成主体，人审核关键点 | 方案设计、流程搭建、定价策略 |
| ⬜ 辅助 | 人主导，AI提供支持 | 需求采集、客户沟通、合规判断 |

**关键规则**：合规相关环节不可标注⬛全自动；任何⬛全自动环节必须有⬜辅助或🟨半自动的兜底方案。

## 一、6维能力详细映射

### S-调研能力：市场与需求侦察

| 操作类型 | 典型场景 | AI自治度 | 注意事项 |
|---------|---------|---------|---------|
| 市场调研 | 行业规模、趋势、竞品扫描 | 🟨半自动 | AI采集数据，人判断可靠性 |
| 需求采集 | 用户访谈、痛点梳理 | ⬜辅助 | 人主导沟通，AI辅助整理 |
| 竞品分析 | 功能对比、定价策略、差异化定位 | ⬛全自动 | AI可完成大部分分析 |
| 趋势监测 | 行业动态、政策变化、技术演进 | 🟨半自动 | AI监测+人判断影响 |

**常见陷阱**：信息过载迟迟不行动→设定调研时限（最多3天）；需求泛化→强制收敛到具体人群和场景。

### C-设计能力：方案设计与决策

| 操作类型 | 典型场景 | AI自治度 | 注意事项 |
|---------|---------|---------|---------|
| 技术选型 | 开发框架、AI工具、基础设施 | 🟨半自动 | 人做最终决策，AI提供对比分析 |
| 方案设计 | 架构设计、流程设计、自动化设计 | ⬛全自动 | AI编码工具可完成大部分设计 |
| 自建vs外包判断 | 哪些自己做，哪些用SaaS/外包 | 🟨半自动 | AI分析ROI，人做决策 |
| 数据分析 | 用户数据、运营数据、财务数据 | ⬛全自动 | AI可完成大部分分析 |

**设计能力决策矩阵**：

| 维度 | 自建 | SaaS/工具 | 外包 |
|------|------|----------|------|
| 核心竞争力 | ✅ 必须 | ❌ 不适合 | ❌ 不适合 |
| 非核心但有定制需求 | ⚠️ 评估ROI | ✅ 优先 | ⚠️ 备选 |
| 非核心标准流程 | ❌ 不值得 | ✅ 优先 | ✅ 可选 |
| 需要快速验证 | ⚠️ 看技术栈 | ✅ 优先 | ⚠️ 周期长 |

### A-执行能力：执行与交付

| 操作类型 | 典型场景 | AI自治度 | 注意事项 |
|---------|---------|---------|---------|
| 内容生产 | 文章、视频脚本、社交媒体内容 | ⬛全自动 | AI生产，人审核 |
| 产品交付 | 软件开发、服务交付 | 🟨半自动 | AI辅助编码，人审核关键环节 |
| 流程自动化 | 自动化工作流搭建 | ⬛全自动 | AI可搭建，人验证 |
| 客户服务 | 自动回复、FAQ、工单处理 | ⬛全自动 | 标准问题AI处理，复杂问题转人 |

**自动化优先级**：重复性高→AI优先；涉及人际沟通→人主导AI辅助；涉及创意判断→人审核AI产出。

### O-财务能力：现金流与资产

| 操作类型 | 典型场景 | AI自治度 | 注意事项 |
|---------|---------|---------|---------|
| 现金流管理 | 收支记录、runway计算、预算规划 | 🟨半自动 | AI记录计算，人做决策 |
| 税务规划 | 主体选择、税率优化、研发费用抵扣 | ⬜辅助 | 建议咨询专业会计师 |
| 成本核算 | AI工具月费、外包成本、运营成本 | ⬛全自动 | AI可完成核算 |
| 定价策略 | 服务定价、SaaS定价、阶梯定价 | 🟨半自动 | AI分析竞品，人做最终定价 |

**财务核心公式**：

```
runway(月) = 现金储备 / 月度净支出
安全runway ≥ 6个月（OPC建议12个月）
月度净支出 = 固定成本(AI工具+办公+外包) + 变动成本(税费+差旅)
目标MRR ≥ 3 × 月度固定成本（覆盖波动）
```

### I-获客能力：获客与品牌

| 操作类型 | 典型场景 | AI自治度 | 注意事项 |
|---------|---------|---------|---------|
| 内容获客 | 自媒体、SEO、社区运营 | ⬛全自动 | AI生产内容，人定策略和审核 |
| 冷启动获客 | 冷邮件、社交拓展、合作推荐 | 🟨半自动 | AI辅助写邮件，人发关系 |
| 客户沟通 | 需求沟通、提案、售后 | ⬜辅助 | 人主导，AI辅助整理 |
| 品牌建设 | 个人IP、行业认知、信任积累 | 🟨半自动 | AI辅助内容，人做品牌决策 |

**获客漏斗**（OPC标准模型）：

```
内容曝光(AI自动) → 免费内容信任建立 → 低价产品筛选 → 高客单服务变现
```

核心原则：**卖商业成果，不卖AI技术**。

### G-合规能力：风控与合规

| 操作类型 | 典型场景 | AI自治度 | 注意事项 |
|---------|---------|---------|---------|
| 主体合规 | 注册类型选择、法人独立性 | ⬜辅助 | 建议咨询律师/会计师 |
| 合同审核 | 服务合同、NDA、外包合同 | 🟨半自动 | AI初筛+人审核关键条款 |
| 数据安全 | 用户数据、隐私保护、数据跨境 | 🟨半自动 | AI辅助检查，人做合规决策 |
| 知识产权 | 商标、著作权、专利 | ⬜辅助 | 建议咨询专业代理 |

**合规底线检查清单**（任何涉及法律/财务的决策必须过此清单）：

1. ⬜ 法人财产与个人财产是否严格分离？
2. ⬜ 注册资本是否实缴？是否抽逃？
3. ⬜ 合同是否经过审核？关键条款是否理解？
4. ⬜ 用户数据是否合规采集和存储？
5. ⬜ 税务是否按期申报？是否违规节税？
6. ⬜ AI生成内容是否有侵权风险？
7. ⬜ 服务承诺是否超出能力范围？

## 二、领域校准推导

### 校准推导规则

| 规则 | 推导逻辑 | 校准结果 |
|------|---------|---------|
| 资源极限原则 | 一人时间/资金有限 | 调研和设计能力优先调用——先侦察清楚再行动，减少返工浪费 |
| 现金流敏感原则 | OPC现金流是生死线 | 财务能力权重提升，任何决策必附带现金流影响评估 |
| AI杠杆原则 | AI是OPC核心放大器 | 执行操作优先评估AI自动化可行性，获客AI辅助但人主导 |
| 垂直利基原则 | OPC必须窄而深 | 设计决策时强制收敛——「你的垂直利基是什么？」 |
| 合规底线原则 | 小组织承担完整法律责任 | 合规能力不可省略，关键节点自动嵌入合规检查 |

### OPC阶段校准矩阵

| 阶段 | 核心能力需求 | 关键变量 | 合规重点 | 自治度偏好 |
|------|------------|---------|---------|-----------|
| 0→1 探索期 | 调研+设计 | 需求验证、赛道选择 | 不要无客源就注册公司 | 🟨半自动 |
| 1→10 验证期 | 执行+获客 | 首批客户、交付跑通 | 合同审核、质量保证 | 🟨半自动 |
| 10→100 增长期 | 财务+执行 | MRR增长、产品化 | 税务合规、数据安全 | ⬛+🟨 |
| 100→1000 扩张期 | 财务+合规 | SaaS化、品牌化 | 法人独立、知识产权 | 🟨+⬜ |

## 三、IPO基元链模式详解

### OP-P1：需求验证链

```
I:想法 → [调研:市场扫描] → [设计:需求判断] → [获客:潜在客户确认] → [合规:可行性验证] → O:验证结论
```

**适用场景**：0→1阶段，有一个想法但不确定是否值得做。

**关键规则**：
- 调研阶段最多3天，超时强制进入设计阶段
- 获客确认必须找到至少3个潜在客户愿意付费
- 合规检查：需求是否真实、是否有支付意愿、是否有能力交付

### OP-P2：MVP快速交付链

```
I:已验证需求 → [设计:最小方案设计] → [执行:AI辅助快速实现] → [获客:客户反馈] → ↻(设计→执行, 满意) → O:交付归档
```

**适用场景**：1→10阶段，已验证需求，快速交付首个版本。

**关键规则**：
- 设计阶段强制使用「最小可行产品」思维——砍掉一切非必要功能
- 执行阶段优先使用AI工具辅助实现，减少手动编码
- 获客反馈循环不超过3轮，避免无限修改

### OP-P3：内容获客飞轮链

```
[执行:AI内容生产] ⇉ [获客:分发渠道1/2/3] → [调研:数据监测] → [设计:效果分析] → ↻(执行, 优化内容)
```

**适用场景**：任何阶段，OPC最主要的低成本获客方式。

**关键规则**：
- 执行阶段使用AI自动化内容生产
- 分发渠道选择：目标客户在哪里就在哪里分发
- 调研监测每周一次，设计分析每月一次
- 内容类型按垂直利基定位，不做泛内容

### OP-P4：合规风控检查链

```
[合规:主体合规] → [合规:税务合规] → [合规:合同合规] → [合规:数据安全] → O:合规档案
```

**适用场景**：任何涉及法律/财务/数据操作的决策前。

**关键规则**：
- 主体合规检查法人独立性（个人账户与公司账户分离）
- 税务合规检查（按期申报，研发费用抵扣）
- 合同合规检查（服务范围、付款条件、责任限制、知识产权）
- 数据安全检查（用户数据采集、存储、处理合规）
- 任何合规检查不通过，暂停行动，修复后再继续

### OP-P5：服务产品化链

```
I:已验证服务流程 → [设计:产品化拆解] → [设计:自动化点位识别] → [执行:AI自动化搭建] → [合规:质量验证] → O:产品归档
```

**适用场景**：10→100阶段，将已验证的人工服务转化为可复制的系统。

**关键规则**：
- 输入必须是已跑通3次以上的服务流程
- 自动化点位识别时，区分「AI可自动化」和「必须人工」
- 质量验证：产品化后的交付质量是否不低于人工服务

## 四、实例化演示

### 实例1：0→1 验证一个SaaS想法

**领域校准**：0→1探索期 → 调研+设计能力优先，合规重点「不要无客源就注册公司」

| 单元ID | 名称 | 能力调用 | 输入 | 输出 | 依赖 | 自治度 | 组合接口 |
|--------|------|---------|------|------|------|--------|----------|
| S-01 | 目标市场扫描 | 调研+行业分析 | 必选：初步想法；可选：行业偏好 | 市场数据+竞品列表 | 无 | 🟨 | →C-01 |

**链**：S-01 → C-01 → I-01 → G-01 → O-01

| O-01 | 获客系统归档 | 财务+系统管理 | 必选：C-02优化结论 | 系统文档+成本记录 | C-02 | ⬛ |

**链**：S-01 → C-01 → [A-01 → G-01] → A-02 → I-01 → C-02 → O-01

## 五、执行规则

1. **阶段优先**：先判断用户处于0→1/1→10/10→100/100→1000哪个阶段，再选择基元链模式
2. **现金流量化**：任何涉及财务的决策必须附带现金流影响评估
3. **AI优先评估**：执行操作先评估AI自动化可行性，再决定手动/外包
4. **合规不跳过**：合规检查不可标记为「跳过」或「延后」
5. **垂直利基收敛**：设计决策时如果领域过泛，强制追问「你的垂直利基是什么」
6. **成果导向表达**：获客操作时，强制检查是否在卖「AI技术」而非「商业成果」
7. **3天调研上限**：调研单次不超过3天，超时强制进入设计阶段
8. **3轮迭代上限**：OP-P2中客户反馈循环不超过3轮
9. **20%现实警示**：任何涉及OPC乐观预期时，必须同时呈现「仅20%稳定盈利」的现实数据
