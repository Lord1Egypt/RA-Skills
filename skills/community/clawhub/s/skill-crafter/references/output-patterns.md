# 输出模式与四层骨架

当技能需要产出一致的输出时，输出规范放在操作指南层，格式模板放在 references/。四层骨架决定"输出规范写在哪"，输出模式决定"怎么写"。

---

## 严格模板——操作指南直接内嵌格式

适合数据格式、固定报告结构等必须严格遵循的输出。

**在四层骨架中的位置：**

```
操作指南层：
  场景描述 → 完整操作 → 输出格式（直接内嵌）
补充说明层：
  格式冲突时的处理方式
```

**正确示例：**

```markdown
## 使用

### 生成周报

基于本周工作内容生成标准周报。

操作步骤：
1. 收集本周已完成任务和进行中任务
2. 按以下格式生成：

# 周报 - [日期范围]

## 本周完成
- [任务名]：[结果描述]

## 进行中
- [任务名]：[当前进度]%，[阻塞项]

## 下周计划
- [任务名]：[目标]

## 风险与求助
- [风险描述]：[影响范围]，[需要什么帮助]
```

---

## 灵活指引——操作指南给方向，references/ 给参考

适合需要根据内容自适应的输出（调研报告、分析文档）。

**在四层骨架中的位置：**

```
操作指南层：
  场景描述 → 完整操作 → 输出指引（推荐结构，允许调整）
references/：
  完整模板（Agent 按需读取）
补充说明层：
  结构冲突时的判断原则
```

**操作指南层的写法：**

```markdown
## 使用

### 生成调研报告

基于搜索结果生成结构化调研报告。

1. 按研究框架搜索并收集素材
2. 生成大纲（确认门）
3. 按以下结构撰写（可根据内容调整子章节）：

# [标题]

## 摘要
[核心发现浓缩]

## 背景与现状
[根据素材组织]

## 核心发现
[根据发现内容自行组织子章节]

## 建议与展望
[根据具体场景调整]

> 完整模板见 references/report-template.md
```

## 示例驱动——操作指南给输入输出对

适合输出质量依赖于具体示例的场景（提交信息、代码风格转换、文案生成）。

**在四层骨架中的位置：**

```
操作指南层：
  场景描述 → 输入/输出对（至少2组）
补充说明层：
  边界情况的处理方式
```

**正确示例：**

```markdown
## 使用

### 生成提交信息

根据代码变更生成规范的 commit message。

按以下示例的风格生成：

**示例 1：**
- 输入：添加了 JWT 用户认证
- 输出：`feat(auth): 实现基于 JWT 的认证`

**示例 2：**
- 输入：修复了报表中日期显示不正确的问题
- 输出：`fix(reports): 修正时区转换中的日期格式化`

**示例 3：**
- 输入：更新了 README 中的安装说明
- 输出：`docs(readme): 更新安装说明`
```

## 输出位置规范

无论哪种模式，操作指南中必须明确：

| 信息 | 写在哪 |
|------|--------|
| 输出格式 | 操作指南的场景示例中 |
| 输出存放路径 | 操作指南的场景示例中 |
| 同名冲突处理 | 补充说明 |
| 格式冲突时的判断 | 补充说明 |
| 完整模板（如果太长） | references/，操作指南中用引用指向 |

## 常见错误

### 错误1：把输出模板写进概述

```
❌ ## 概述
   本技能生成以下格式的报告：
   # 标题
   ## 摘要
   ## 正文
```

概述只画边界，输出格式归操作指南。

### 错误2：操作指南只说"按模板生成"

```
❌ ## 使用
   按以下模板生成报告：[模板见 references/]
```

没有场景、没有操作步骤，Agent 不知道什么情况下用什么模板。

### 错误3：严格模板和灵活指引混用

同一个 Skill 的同一类输出，要么严格要么灵活，不要一半固定一半随意。如果不同场景需要不同约束度，在操作指南中明确说明"XX 场景用严格模板，YY 场景可灵活调整"。
