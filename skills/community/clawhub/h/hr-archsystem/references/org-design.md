# 组织架构设计

## 输入与输出

**输入：** 战略目标、业务复杂度、人员规模、现有架构图、协作模式
**输出：** 《架构说明书》—— 部门划分、职类序列、汇报关系、核心决策流

---

## Step 1：战略对齐

设计架构之前，先明确：

```
1. 我们的核心战略是什么（1-3条）？
2. 关键业务能力是什么（客户价值链上的1-3个环节）？
3. 我们的竞争优势来自哪里（技术/渠道/品牌/成本）？
```

这三条决定架构的**重心**在哪里。

---

## Step 2：McKinsey 7S 现状诊断

对现有架构做7要素对齐评估：

| 要素 | 类型 | 内容 |
|------|------|------|
| Strategy | 硬 | 战略目标、资源配置优先级 |
| Structure | 硬 | 汇报关系、部门划分、层级 |
| Systems | 硬 | 流程、审批、数据系统 |
| Shared Values | 软 | 文化、价值观、行为准则 |
| Skills | 软 | 组织核心能力、差异化能力 |
| Style | 软 | 管理层决策风格 |
| Staff | 软 | 人才结构、人才密度 |

**诊断方法：** 对每个S打分（1-5），识别"断点"——7个S中断点最多的地方就是最需要改的地方。

---

## Step 3：复杂度系数模型

部门划分的决策依据：

```
复杂度系数 = f(业务复杂度, 决策频率, 协同依赖度, 监管要求)

业务复杂度：
  - 单一业务线 → 系数 1.0
  - 多业务线但同类 → 系数 1.3
  - 多元业务 → 系数 1.8

决策频率：
  - 日常运营决策 → 适合放一线
  - 战略/例外决策 → 适合放高层

协同依赖度：
  - 高内聚低耦合 → 独立部门
  - 强依赖 → 归并同类项
```

---

## Step 4：职类序列定义

组织内所有岗位归入四大序列（职类）：

| 序列 | 定义 | 典型岗位 | 晋升特点 |
|------|------|---------|---------|
| **M 管理序列** | 对人负责，对团队产出负责 | 经理/总监/VP/CEO | 以管理幅度和战略影响范围晋升 |
| **P 专业序列** | 对专业深度负责，单人不带团队 | 工程师/设计师/分析师 | 以专业深度和影响范围晋升 |
| **T 技术序列** | 对技术实现负责 | 技工/技术员/运维 | 以技术等级和资质认证晋升 |
| **S 技能操作序列** | 对操作执行负责 | 操作工/客服/助理 | 以技能熟练度和资质晋升 |

**注意：** AI时代新增 `AI` 序列——负责AI系统设计/训练/治理的岗位，可归入P或新建。

---

## Step 5：AI时代的架构新增考量

### 5.1 人机协同架构层级

```
Level 1 - 工具化:  AI作为个人效率工具，架构不变
Level 2 - 流程化:  AI嵌入业务流程，岗位边界开始模糊
Level 3 - 角色化:  AI作为协作者出现，产生"人机混合岗"
Level 4 - 系统化:  AI成为组织架构设计的一个维度
```

### 5.2 新增AI治理职能

在现有架构中增加以下AI相关职能节点（根据公司规模决定放在哪层）：

- **AI伦理审查** — 对AI应用进行合规和伦理把关
- **模型资产管理** — AI模型版本、权限、数据治理
- **人机协作体验** — AI使用体验、Prompt库管理、AI效能评估

### 5.3 决策层级扁平化机会

AI替代了部分信息中介和审批节点，可能导致：
- 某些中层管理职位被压缩
- 一线员工决策权扩大
- 组织向"哑铃型"演变（高层决策+一线执行，中间压缩）

---

## Step 6：汇报关系设计

核心原则：**一个人只有一个直接汇报对象（紧急情况例外）**

```
设计检查清单：
□ 每个部门有且只有一个负责人
□ 跨部门协作通过"产品/项目"机制而非固定汇报
□ 新增岗位先问：是否有现有岗位可以承担类似职责？
□ 管理幅度：一线主管 8-15人，中层 5-9人，高层 3-7人
□ AI渗透率高的部门，管理层级可以更扁平
```

## 触发条件

当用户提供以下内容时，触发本模块：
- 新的组织架构图（文字描述或草图）
- 部门调整、汇报关系变化
- 战略目标变化
- 问"我们应该怎么组织"类似问题
