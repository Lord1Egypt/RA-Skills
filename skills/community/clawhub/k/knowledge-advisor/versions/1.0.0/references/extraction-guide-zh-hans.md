# 萃取指南（简体中文）

从书籍和学习材料中萃取结构化知识的方法论。

## 萃取目标

萃取可操作的结构化知识，使其能够与现实情境匹配。目标不是摘要整本书，而是识别离散的、具名的、可重复使用的知识单元。

每个萃取项目必须：
- **忠于原著** — 不重新诠释
- **可操作** — 使用者能立即运用
- **足够具体** — 有明确步骤或判断标准
- **可引用** — 链接到特定章节

## 1. 框架萃取

框架是一个具名的方法、流程或模型，包含明确的步骤或组成要素。

**如何识别：**
- 作者明确命名的方法（例如：「STATE 方法」、「刺猬原则 (Hedgehog Concept)」）
- 作者规定的步骤流程
- 具有命名组件或轴线的模型（例如：2x2 矩阵）
- 编码流程的缩写

**萃取格式：**

```markdown
## [框架名称] ([English Name])

**来源：** 第[N]章，「[章节标题]」
**类型：** [沟通 | 决策 | 领导力 | 策略 | ...]
**领域：** [以逗号分隔的领域标签]

### 描述
[2-3句：这是什么，为什么重要。使用作者的语言。]

### 步骤
1. [步骤名称] — [具体做什么]
2. ...

### 应用情境
- [适用这个框架的情境]
- [另一个情境]

### 反模式（何时不该使用）
- [不适用的情境]

### 相关框架
- [[其他书籍/框架名称]]

### 关键引文
- 「[原文直接引用]」（第[N]页）
```

**规则：**
- 只萃取作者明确命名或定义的框架
- 如果是流程，必须包含所有步骤 — 不可缩减
- 应用情境至关重要 — 这些将成为搜索索引
- 关键引文使原文有据可查

## 2. 原则萃取

原则是核心教训、经验法则或格言 — 比框架更简短。

**如何识别：**
- 作者反复强调的陈述
- 经验法则（「先做X再做Y」、「绝对不要Z」）
- 具名原则（例如：「斯托克代尔悖论 (Stockdale Paradox)」）
- 作者从证据中得出的结论

**萃取格式：**

```markdown
## [原则名称] ([English Name])

**来源：** 第[N]章

### 描述
[1-2句，捕捉原则的核心。]

### 适用时机
- [情境]
```

## 3. 心智模型萃取

心智模型是一种思考问题的方式 — 不是要遵循的流程，而是观看问题的透镜。

**如何识别：**
- 「把它想成...」或「想象...」的表述
- 作者发展成推理工具的隐喻
- 框架化方式（例如：「沉默 vs. 暴力」作为两种失败模式）

**萃取格式：**

```markdown
## [模型名称] ([English Name])

**来源：** 第[N]章

### 描述
[这个模型如何框架化问题。]

### 如何使用
[何时及如何运用这个思考透镜。]
```

## 4. 反模式萃取

反模式是书中明确警告不要做的事。

**如何识别：**
- 「不要做X」、「避免Y」、「大多数人犯的错误是...」
- 具名反模式（例如：「乍看二选一 (Sucker's Choice)」）
- 作者用作警示的失败故事

**萃取格式：**

```markdown
## [反模式名称] ([English Name])

**来源：** 第[N]章

### 描述
[这个错误是什么。]

### 应该怎么做
[作者建议的替代方案。]
```

## 5. 案例研究萃取

案例研究是作者用来说明框架或原则的真实案例。

**萃取格式：**

```markdown
## [案例标题]

**来源：** 第[N]章
**相关框架：** [列表]

### 摘要
[2-3句：发生了什么。]

### 关键教训
[作者希望读者从中学到什么。]
```

**规则：**
- 只萃取能说明特定框架或原则的案例
- 每个案例必须链接到它所说明的框架
- 摘要保持事实性 — 教训是作者的，不是你的

## 6. 应用触发点对应

萃取完所有项目后，建立应用触发点索引。

**流程：**
1. 为每个框架和原则列出3-5个适用情境
2. 使用用户描述情境时会用的自然语言
3. 包含用语变化（例如：「给反馈」、「绩效面谈」、「困难对话」）
4. 按主题分组

**格式：**
```markdown
### [情境描述]
→ [框架名称]（[书名]），[原则名称]（[书名]）
```

## 7. 品质标准

好的萃取：
- 使用作者的专有名词
- 包含完整步骤（不是「详见原书」）
- 每个框架有3-5个应用触发点
- 每个框架至少有1个关键引文
- 反模式链接到相关框架
- 交叉引用知识库中其他书籍的相关框架

## 8. 常见错误

1. **过度萃取**：不是每个有趣的段落都是框架。只萃取具有名称或明确结构的项目。
2. **描述太模糊**：「当个好领导者」不是原则。「在分享结论前先听取数据」才是。
3. **遗漏步骤**：如果框架有5个步骤，就萃取全部5个。不要把5步流程缩减为3步。
4. **混淆案例与框架**：关于某位CEO的故事是案例研究，不是框架 — 除非作者从中推导出具名模型。
5. **跳过应用触发点**：没有触发点，顾问就无法配对情境与框架。这是最常被跳过但最重要的步骤。

## 9. 中英对照命名

萃取中文材料时：
- 所有内容以原文简体中文储存
- 每个框架/原则名称后加括号标注英文翻译
- 范例：「刺猬原则 (Hedgehog Concept)」
- 应用触发点使用简体中文
- 关键引文保持原文
