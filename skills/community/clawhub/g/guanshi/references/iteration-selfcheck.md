# Step 6.5：MECE 逻辑一致性自检

> 诊断后、交付前的质控步骤。问题树全量逻辑扫描，不重跑分析。

## 定位

Step 6 产出完整方案后、报告交付前，对方案内部逻辑链做全量 MECE 一致性自检。通过则交付，不通过则修补一轮后重新自检再交付。

```
诊断（Step 1-6）→ 自检 → 通过 → 交付
                     ↓
                  不通过 → 修补 → 再自检 → 仅一次 → 交付
                  再不通 → 标记"裂缝超出修补范围" → 交付+未通过清单
```

## 检查内容（四维全量扫描）

全量但只查逻辑结构，不重跑分析、不调搜索、不调专家。

| 检查项 | 全量扫描范围 | 不通过标准 |
|-------|---------|-----------|
| **MECE 互斥性** | 问题树所有节点两两对比边界 | 两个分支指向同一个结论/分析对象且未声明合并 |
| **MECE 穷尽性** | 报告五维度中所有带"判断"/"结论"标记的段落，逐一反向映射回问题树 | 报告有实质性分析但无对应问题树分支 |
| **逻辑链一致性** | 五维度中每条判断与其对应的建议段落交叉比对 | 判断说"A是护城河"，建议说"放弃A转B"且未解释 |
| **框架-结论对齐** | 框架判定表"必须"列表 ↔ 报告五维度段落，双向穷尽匹配 | "必须"框架的产出在报告中缺失，或报告大段讨论无框架支撑 |

## 迭代规则

任一检查不通过 → 仅一次修补迭代：
- 只修不通过的节点，不动其他部分
- 修完后重新跑自检
- 第二次仍不通过 → 标记交付 + 附带未通过清单

## 输出格式

```
══════════════════════════════
MECE 逻辑一致性自检

互斥性：通过 / 不通过
  [不通过时标注冲突节点]

穷尽性：通过 / 不通过
  [不通过时标注遗漏节点]

逻辑链一致性：通过 / 不通过
  [不通过时标注矛盾位置：判断X vs 建议Y]

框架-结论对齐：通过 / 不通过
  [不通过时标注：框架A(必须)在报告中缺失 / 结论B无框架支撑]

整体判定：通过（X/4）/ 不通过（X/4）
  → 不通过时执行一次修补迭代
══════════════════════════════
```

## 与宪法审计（Step 5.5）的关系

| | 宪法审计（5.5） | 逻辑自检（6.5） |
|------|:---:|:---:|
| 检查对象 | 分析过程是否严谨 | 逻辑结构是否自洽 |
| 时机 | 分析完成后（Step 5之后） | 报告完成前（Step 6之后） |
| 典型问题 | "结论只有1个数据源支撑" | "判断和那个建议互相矛盾" |
| 输出 | 内部审计意见（不写入报告） | 通过/不通过 + 修补记录 |

两者不重复，时序互补。5.5 是分析收口，6.5 是全文收口。
