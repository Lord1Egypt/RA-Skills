# v2.8.23 (2026-05-05)

**群聊直接发文件 86008 修复 — 撤回 v2.8.8 的"reply 媒体走 replyMedia"误判**

## 背景

赵博群里反复试发 zip 都不成功，trajectory + gateway log 联合复盘，**真根因不是 stream 截断也不是 86008 群权限本质问题，而是 v2.8.8 时一个错的 P0 修复**：

### v2.8.8 的错误判断

v2.8.8 把 reply 路径的媒体发送从 `wsClient.sendMediaMessage`（aibot_send_msg **主动推送通道**）改成 `wsClient.replyMedia`（aibot_respond_msg **被动回复通道**），理由是"reply 上下文应该走 reply 通道，绑定 reqId 上下文更对"。

但**实测证明这个改动让群聊场景失效**：

```
[wecom-ws] Uploading media: type=file, filename=rustdesk-deploy-sop.md, size=4342
   ↓ wsClient.replyMedia 触发
Reply ack timeout (5000ms) for reqId: aibot_upload_media_init_...
   ↓ shareFallbackOnUploadError 接管
   ↓ wsClient.sendMessage 推链接（也撞群限制）
   ↓ 降级到 mediaFailures 拼到 finalText
   ↓ Agent API sendText 兜底
[wecom-agent-api] sendText response chatId=wrGzumEQAAD... errcode=86008
   "you has no privilege to access this chat, which is created by other agent"
```

### 历史证据

gateway.log 翻了一遍，**v2.8.7 之前的主动推送通道在群聊也 work**：

| 4/30 19:03 | outbound.sendMedia → `wsClient.sendMediaMessage` | 群 `wrGzumEQAAgjV...`（同公司另一个非自创群） | ✅ "多群并行流程.png" 发出 |
| 5/3 20:14 | 同上 | DM 用户 MingYang | ✅ docx 发出 |

**结论**：`sendMediaMessage`（aibot_send_msg）的群权限模型跟 `replyStream` 文本被动回复一致——只要机器人在群里有"被 @ 触发"权限，就能主动推送媒体。`replyMedia`（aibot_respond_msg）权限更严，要 chat creator agent 才能用。

```diff
+ const body = (params.frame as { body?: { chattype?, chatid?, ... } }).body;
+ const isGroup = body?.chattype === "group";
+
+ let messageId: string;

+   if (!chatId) throw new Error("WeCom group chat reply requires body.chatid");
+   const sendResult = await params.wsClient.sendMediaMessage(chatId, sizeCheck.finalType, uploadResult.media_id);
+   messageId = sendResult?.headers?.req_id ?? `wecom-group-media-${Date.now()}`;

+   messageId = replyResult?.headers?.req_id ?? `wecom-reply-media-${Date.now()}`;

- const replyResult = await params.wsClient.replyMedia(params.frame, sizeCheck.finalType, uploadResult.media_id);
- // 原来不分场景一律 replyMedia → 群聊撞 ack timeout / 86008
```

## 后续 v2.8.22 GUIDANCE 是否要回滚

v2.8.22 的 GUIDANCE 让 LLM 优先调 `enhance_share_file` 是为了绕开当时的 86008 / stream 截断问题。现在 v2.8.23 修了根本问题，**理论上 LLM 直接 emit MEDIA: 在群聊也能 work**——但 GUIDANCE 我**不回滚**，理由：

1. `enhance_share_file` 走 structured tool call **更稳定**，永远不会撞 stream 截断
2. share-fallback 链接对 > 20MB 大文件仍然是必经之路
3. v2.8.23 让 MEDIA: 路径在群聊**真正 work** 了——LLM 偶尔 emit MEDIA: 也能成功，不会再失败

LLM 偏好用什么由 GUIDANCE 决定，但**两条路径现在都能走通**。

## 经验沉淀

**P0 修复必须有"群聊场景 + DM 场景双验证"**。v2.8.8 当时只在 DM 测试 replyMedia work，没在群里测，结果群里直接坏掉（且 share-fallback 接管让症状变成"消息发不出"而不是"明显的报错"）。今后任何动 wecom 媒体路径的修复，必须群 + DM 各跑一次集成验证。
