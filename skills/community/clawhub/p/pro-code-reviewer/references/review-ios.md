# iOS (ObjC/Swift) 审查维度

按以下 9 个维度逐一审查，每个维度独立分析。对每个疑似问题，通过多种方法（阅读上下文、搜索调用方、追踪数据流）反复验证后再确认。

## 1. 多线程安全

- 共享可变状态是否有正确的同步保护（锁、串行队列、atomic）
- Block 捕获的变量在回调时是否仍然有效
- `dispatch_async` 回调中访问的对象，其生命周期是否可控
- 是否存在竞态条件：多个线程/队列可能同时读写同一字段
- 串行队列上的操作是否有可能被外部并发调用打破串行假设

## 2. 死锁、卡顿与性能

- **同步死锁**：`dispatch_sync` 到当前队列、嵌套锁（A->B->A）、`@synchronized` 嵌套
- **异步死锁**：队列 A 等待队列 B 的结果，队列 B 又等待队列 A
- **主线程卡顿**：主线程上执行同步磁盘 I/O、同步网络请求、大量计算、`dispatch_sync` 到其他队列
- **锁粒度过大**：在锁内执行耗时操作导致其他线程长时间等待
- **dispatch_once 死锁**：singleton 初始化链路中互相依赖（A.shared -> B.shared -> A.shared）
- **高频路径重复 I/O**：循环体 / 频繁调用的方法内每次都读写磁盘
- **重复计算**：可缓存的结果每次重新计算（如循环内反复创建 DateFormatter）
- **大对象频繁创建/销毁**：热路径上反复 alloc 重型对象而未复用
- **无节制的集合增长**：数组/字典只添加不清理，随时间无限增长

## 3. 内存安全

- Block 中是否正确使用 `__weak` / `__strong`，避免循环引用
- delegate / observer 引用关系是否合理（weak vs strong）
- 是否存在 dealloc 后仍被回调的风险（悬空指针）
- 集合（NSArray/NSDictionary）中存储的对象引用是否会导致意外强持有
- C 数组/指针操作是否有越界风险
- 单例持有非单例对象是否会导致泄漏

## 4. 资源泄漏

- 未关闭的文件句柄 / 数据库连接
- 未移除的 timer / observer / notification
- `dispatch_source_t` 创建后未 cancel
- `CFRetain` 无对应 `CFRelease`
- 临时文件未清理

## 5. 逻辑正确性

- 条件判断的边界值是否正确（off-by-one, 空值, 零值）
- 新增代码对已有流程的影响面（调用方是否需要适配）
- 异步操作的时序是否有保证（先后顺序、回调是否可能不触发）
- 错误处理路径是否完整（每个分支都有合理出口）
- 防重入/幂等设计是否覆盖所有入口

## 6. 异常吞没与回调完整性

- **completion 回调遗漏**：所有执行路径（含快速返回 / 错误分支）是否都调用了 completion 回调
- `@try/@catch` 是否吞掉了不应忽略的异常（catch 后无日志、无上报、直接 return）
- 错误路径是否有日志或 APM 上报，避免静默失败
- delegate 回调方法是否在所有路径都触发（成功 + 失败 + 超时 + 取消）

## 7. 数据一致性

- 多数据源同步写入时是否保证原子性
- 缓存与真实状态是否可能不一致（写穿/写回策略）
- 状态机是否存在非法跳转（跳过中间状态、重复进入终态）
- 读取时是否使用了同一份快照（非原子地读取多个相关字段可能不一致）

## 8. API 兼容性

- 公开 API 签名变更是否破坏已有调用方（ObjC 方法名/参数调整、Swift 桥接名称变化）
- 新增方法是否可能与父类/子类的同名方法冲突
- 枚举新增值是否影响 switch/for 遍历（遗漏 case、循环范围越界）
- 协议新增 `@required` 方法是否导致现有实现类编译报错

## 9. 架构设计

- 圈复杂度是否过高（深层嵌套、过长方法）
- 是否存在重复代码可提取公共方法
- 类/模块间的调用关系和持有关系是否合理
- 引用层级是否过深（A->B->C->D->E）
- 职责划分是否清晰（单一职责原则）
- 新增 API 的命名和参数设计是否符合现有约定

**过度工程化红线（以下情况不要提出建议）：**
- 三行以内的相似代码不算"重复"
- 仅被调用一次的代码不需要提取方法
- 现有代码不在本次变更范围内，不提优化建议（除非变更直接引入了问题）
- 不要建议为假想的未来需求做扩展性设计
- 不要要求给未变更的代码加注释、文档或类型标注
- 不要建议为一次性操作创建工具类/抽象层
