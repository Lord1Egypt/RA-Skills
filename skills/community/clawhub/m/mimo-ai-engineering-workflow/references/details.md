# AI编程工程化 - 详细内容参考 v1.3.0

## 一、Superpowers方法论核心原则

### 1.1 核心理念
- **门槛(Gate)替代规则**：不是"应该做"而是"不做就不能继续"
- **强制文档**：每个阶段必须有可审查的输出物
- **测试驱动**：先红后绿，不写无测试的代码
- **AI代码更严审**：AI生成的代码不是"写完了"而是"审查完了"
- **完成优于完美**：先跑通流程，再优化细节

### 1.2 与传统开发的区别
| 维度 | 传统开发 | AI工程化开发 |
|------|----------|-------------|
| 代码来源 | 人工编写 | AI生成+人工审查 |
| 审查重点 | 逻辑正确性 | 正确性+安全性+幻觉检测 |
| 测试策略 | 可能后补 | 必须先行（TDD） |
| 文档要求 | 通常缺失 | 每个Gate强制输出 |
| 债务管理 | 容易堆积 | TODO清单+定期清理 |
| 迭代速度 | 较慢 | 更快（AI加速） |
| 质量标准 | 可能降低 | 不能降低（AI代码更易出错） |

## 四、测试策略详解

### 4.1 测试金字塔
```
        ╱╲
       ╱E2E╲        10% - 端到端测试
      ╱──────╲       验证完整用户流程
     ╱集成测试╲      20% - 模块间交互
    ╱──────────╲     验证API/数据库交互
   ╱  单元测试  ╲    70% - 函数/方法级别
  ╱──────────────╲   验证单个函数逻辑
```

### 4.2 测试用例设计原则
```
测试命名规范：test_{功能}_{场景}_{预期结果}

示例：
test_createUser_withValidData_returnsUser
test_createUser_withDuplicateEmail_returns409
test_createUser_withMissingFields_returns400
test_createUser_withInvalidEmail_returns422
```

### 4.3 测试覆盖矩阵
| 测试类型 | 覆盖范围 | 工具 | 频率 |
|----------|----------|------|------|
| 单元测试 | 函数/方法 | Jest/pytest/go test | 每次提交 |
| 集成测试 | API/模块 | Supertest/TestClient | 每次PR |
| 契约测试 | API接口 | Pact/Dredd | 每次PR |
| E2E测试 | 用户流程 | Playwright/Cypress | 每日/每次发版 |
| 性能测试 | 响应时间 | k6/JMeter | 每周/每次发版 |
| 安全测试 | 漏洞扫描 | OWASP ZAP/Snyk | 每周 |

### 4.4 AI代码测试特别注意
- AI生成的代码可能有隐藏的假设和依赖
- 测试要特别关注边界条件（AI可能忽略）
- 错误处理路径要全面覆盖（AI可能遗漏）
- 并发场景要额外测试（AI可能不考虑）
- 安全相关代码要额外审查（AI可能引入漏洞）

## 五、代码审查检查清单

### 5.1 通用检查项
- [ ] 代码是否符合项目编码规范
- [ ] 命名是否清晰、语义化
- [ ] 函数是否单一职责（≤50行）
- [ ] 是否有不必要的重复代码
- [ ] 错误处理是否完整
- [ ] 日志是否规范（级别/格式/脱敏）
- [ ] 注释是否解释"为什么"而非"做什么"

### 5.2 安全检查项
- [ ] SQL查询使用参数化/ORM
- [ ] 用户输入经过校验和清洗
- [ ] 密码使用bcrypt等安全哈希
- [ ] 敏感数据加密存储
- [ ] API有认证和授权
- [ ] CORS配置正确
- [ ] 无硬编码的密钥/密码
- [ ] 依赖无已知高危漏洞

### 5.3 性能检查项
- [ ] 数据库查询有合适的索引
- [ ] 无N+1查询问题
- [ ] 大数据集使用分页
- [ ] 适当使用缓存
- [ ] 无内存泄漏（资源正确关闭）
- [ ] 异步操作正确处理
- [ ] 无阻塞操作在主线程

### 5.4 AI代码专项检查
- [ ] 是否使用了过时的API/库
- [ ] 是否引入了不必要的依赖
- [ ] 是否有AI幻觉（不存在的函数/API）
- [ ] 错误处理是否完整（AI可能只写happy path）
- [ ] 是否有硬编码的环境相关值
- [ ] 代码风格是否与项目一致

## 六、技术债务管理

### 6.1 债务记录格式
```markdown
## 技术债务清单

| ID | 描述 | 优先级 | 影响范围 | 预计工时 | 发现日期 | 状态 |
|----|------|--------|----------|----------|----------|------|
| TD-001 | {描述} | P0 | {范围} | {工时} | {日期} | 待处理 |
| TD-002 | {描述} | P1 | {范围} | {工时} | {日期} | 待处理 |

优先级定义：
- P0：影响安全/数据完整性，立即修复
- P1：影响性能/可维护性，本周修复
- P2：代码风格/最佳实践，本月修复
- P3：可选优化，择机修复
```

### 6.2 债务清理策略
- 每周安排1-2小时专门清理技术债务
- 新功能开发时顺手修复相关的小债务（Boy Scout Rule）
- 每月进行一次债务大盘点
- 债务数量超过20个时暂停新功能开发，集中清理

## 七、CI/CD集成指南

### 7.1 GitHub Actions示例
```yaml
name: CI/CD Pipeline

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Setup
        uses: actions/setup-node@v4
        with:
          node-version: '18'
      - name: Install
        run: npm ci
      - name: Lint
        run: npm run lint
      - name: Test
        run: npm test -- --coverage
      - name: Coverage Check
        run: |
          COVERAGE=$(cat coverage/coverage-summary.json | jq '.total.lines.pct')
          if (( $(echo "$COVERAGE < 80" | bc -l) )); then
            echo "Coverage $COVERAGE% < 80%"
            exit 1
          fi

  security:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Security Scan
        run: npx audit-ci --high

  build:
    needs: [test, security]
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Build
        run: npm run build
```

### 7.2 质量门禁配置
- 代码覆盖率不低于80%
- Lint检查必须通过
- 安全扫描无高危漏洞
- 构建必须成功
- 所有测试必须通过
