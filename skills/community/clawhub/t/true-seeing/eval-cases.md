# Eval Cases — fact-checker

## Eval 1: 正常路径 — 混合事实的文章

**输入：**
```
OpenAI 于 2025 年 4 月完成了 100 亿美元的融资，估值达到 3000 亿美元。该公司由 Sam Altman 于 2015 年创立。Claude 是 OpenAI 开发的对话模型。
```

**预期行为：**
1. Stage 1 提取至少 4 条事实：
   - "OpenAI 于 2025 年 4 月完成了 100 亿美元的融资" → category: funding
   - "估值达到 3000 亿美元" → category: statistic
   - "该公司由 Sam Altman 于 2015 年创立" → category: event
   - "Claude 是 OpenAI 开发的对话模型" → category: product
2. Stage 2 前置检查 Brave Search 可用，通过。
3. 前两条通过搜索验证，生成 confirmed 或 divergent。
4. 第三条（创立事实）应为 confirmed。
5. 第四条（Claude 是 OpenAI 的）应为 Corrected（Claude 是 Anthropic 的）。
6. Stage 3 输出差异报告，包含 Verified / Corrected / Remain 三张表。

**通过标准：**
- 提取事实数量 ≥ 3
- "Claude 是 OpenAI 开发的"被识别为 divergent（entity_mismatch）
- 差异报告三张表格式正确

**失败迹象：**
- 漏掉 "Claude 是 OpenAI 开发的" 这条事实
- 没有识别出 entity_mismatch
- 报告格式缺少某张表

## Eval 2: 最小输入 — 仅含一条可验证事实

**输入：**
```
这款产品的市场反响不错，用户评价很高。
```

**预期行为：**
1. Stage 1 扫描全文，发现"市场反响不错"、"用户评价很高"均为定性描述/观点。
2. 输出空 fact_list，附注"未发现可验证事实"。
3. 流程结束，不进入 Stage 2。

**通过标准：**
- fact_list.total_facts = 0
- 附注包含"未发现可验证事实"
- 不执行搜索

**失败迹象：**
- 将"市场反响不错"标记为可验证事实
- 执行了 web_search

## Eval 3: 模糊输入 — 数值模糊的事实

**输入：**
```
字节跳动在 2024 年完成了新一轮融资，融资金额达数十亿美元，由软银领投。该公司的日活用户已突破 10 亿。
```

**预期行为：**
1. Stage 1 提取事实时，对模糊数字"数十亿美元"标记为可验证，category: funding。
2. search_query 构造时去掉模糊量词，生成 `"字节跳动 融资 2024 软银"` 而非 `"字节跳动 数十亿美元"`。
3. Stage 2 搜索成功后，如果搜索结果显示具体金额（如"20亿美元"），标记为 confirmed 或 divergent。
4. "日活用户已突破 10 亿"同样被提取和验证。

**通过标准：**
- 模糊数字被正确提取（不跳过）
- search_query 中不包含"数十亿"
- 比对时用搜索结果的具体数值与原文模糊描述做匹配

**失败迹象：**
- 跳过含模糊数字的句子
- search_query 包含"数十亿美元"
- 无法处理精确值 vs 模糊值的比对

## Eval 4: 工具不可用 — Brave Search 失败

**输入：**
```
英伟达于 2024 年 6 月完成了市值突破 3 万亿美元的里程碑。
```

**前置条件：** 模拟 web_search 工具不可用（返回错误或超时）。

**预期行为：**
1. Stage 2 前置检查：执行 `web_search` 查询 `"test"`。
2. 检测到工具不可用 → **abort**。
3. 向用户输出："web_search (Brave) 当前不可用，无法执行事实校验。建议过一段时间再试。"
4. 流程终止，不进入循环。

**通过标准：**
- 不执行任何事实搜索
- 输出明确的错误信息
- 建议用户稍后重试

**失败迹象：**
- 跳过前置检查直接开始搜索
- 错误信息不包含"Brave"或"再试"
- 标记所有事实为 unverified 而非 abort

## Eval 5: 搜索无结果 — 触发重试策略

**输入：**
```
XYZ科技有限公司在 2026 年 1 月发布了革命性的量子计算芯片 QChip-9000，算力达到 1000 Qubits。
```

**前置条件：** 这是一个虚构的产品，所有搜索均无结果。

**预期行为：**
1. Stage 1 正常提取事实。
2. Stage 2 进入循环：
   - 首次搜索无结果 → 状态保持 keywords-generated。
   - 步骤 3e 换策略重搜（英文查询 / 去掉数值 / 扩大范围）。
   - 重搜仍无结果 → 状态变为 Remain，附注"搜索重试失败"。
3. 不超过 2 轮即处理完毕（1 轮内搜完 + 重试）。

**通过标准：**
- 执行了重试策略（至少 2 种查询）
- 最终标记为 Remain
- 附注说明重试失败原因

**失败迹象：**
- 只搜索一次就标记 Remain
- 循环超过 2 轮处理同一条事实
- 标记为 unverified 而非 Remain

## Eval 6: 超出范围请求 — 语法纠错和观点验证

**输入：**
```
帮我检查这篇文章有没有语法错误，另外"AI 将取代人类"这个观点是否正确？
```

**预期行为：**
1. Skill 识别请求内容不属于事实核查范围。
2. 回复：本 Skill 专注于可验证事实的校验（数据、日期、金额、事件等），不支持语法纠错或观点验证。
3. 不执行任何搜索。

**通过标准：**
- 明确拒绝超出范围的请求
- 说明 Skill 的适用范围

**失败迹象：**
- 尝试验证观点"AI 将取代人类"
- 执行了语法纠错

## Eval 7: 大量事实 — 触发截取上限

**输入：**
一篇包含 40 条可验证事实的长文章（模拟）。

**预期行为：**
1. Stage 1 提取 40 条事实。
2. 批量上限检查：截取前 30 条。
3. 输出附注"发现 40 条事实，已截取前 30 条。如需继续，请将剩余部分作为新输入。"
4. Stage 2 分批处理：30 条 ÷ 5 = 6 批，并行处理。
5. 循环处理直到所有事实达到终态。

**通过标准：**
- 截取恰好 30 条
- 附注说明截取原因和剩余数量
- Stage 2 正确分 6 批并行

**失败迹象：**
- 处理全部 40 条（超过上限）
- 批次数量计算错误
- 无截取附注

## Eval 9: Stage 3 用户选择包含无效编号

**输入：**
```
verification_report 包含：
- divergent: id=2, id=5
- confirmed: id=1, id=3, id=4
user_selections=[2, 7]
```

**预期行为：**
1. 编号 2 在 divergent 列表中 → 正常替换。
2. 编号 7 不存在 → 提示"编号 7 不在差异列表中"，忽略。
3. 仅替换编号 2，输出修正后的文章。

**通过标准：**
- 编号 2 正确替换
- 编号 7 被识别为无效并忽略
- 输出包含无效编号的提示

**失败迹象：**
- 因编号 7 报错中断整个流程
- 编号 2 未被替换
- 编号 7 被静默忽略无提示

## Eval 10: 伪量化检测 — pseudo_quantification

**输入：**
```
Sora 的物理规律模拟准确度达到了 99.9%，是当前最接近真实物理世界的视频生成模型。
```

**预期行为：**
1. Stage 1 提取事实时，对“物理规律模拟准确度达到了 99.9%”标记 `pseudo_quantification` 标签。
2. search_query 优先搜索该指标是否有学术/行业来源，生成 `"物理规律模拟准确度" 量化标准 学术来源` 或 `"physics simulation accuracy" benchmark definition`。
3. Stage 2 搜索后发现该指标无统一定义 → 标记为 Corrected，附注“该指标无学术/行业统一定义，数值可疑”。

**通过标准：**
- 事实被标记 `pseudo_quantification`
- search_query 包含指标定义相关搜索，而非仅搜索数值
- 附注说明指标缺乏学术定义

**失败迹象：**
- 未标记 `pseudo_quantification`
- search_query 仅搜索 `"Sora 99.9%"`
- 未提及指标缺乏定义

```
verification_report 包含：

```

**预期行为：**
1. 用户回复空列表或不修改 → 输出原文不变，流程结束。
2. 不执行任何替换。
3. 不报错。

**通过标准：**
- 输出原文未修改

- 报错“用户未选择编号”
- 自动替换所有 Corrected 事实
- 流程卡住等待输入

## Eval 12: 429 Rate Limit 中断

**输入：**
```
一篇包含 10 条事实的文章，Stage 2 第 2 批次搜索时遇到全部 429 响应。
```

**预期行为：**
1. 第 1 批次（5 条）正常处理完成。
2. 第 2 批次搜索全部返回 429 → 当前批次暂停。
3. 第 1 批次的已完成结果保留。
4. 第 2 批次未完成的事实标记为 Remain，附注“因 rate limit 中断”。
5. 输出 verification_report，包含第 1 批次的 Verified/Corrected 结果和第 2 批次的 Remain 结果。

**通过标准：**
- 第 1 批次结果完整保留
- 第 2 批次事实标记 Remain 并附注 rate limit
- 流程不崩溃、不挂起
- 用户收到包含已完成部分的报告

**失败迹象：**
- 第 1 批次结果丢失
- 流程无限重试
- 无 rate limit 相关提示
- 整个流程报错终止
