## 补充说明（兜底方案与踩坑沉淀）

### 知识库依赖
- 观势依赖 `${KB_PATH}` 下的知识库（行业数据/公司档案/竞对资料/案例库等）
- 知识库初始化：运行 `guanshi-init.py` 创建骨架
- 如知识库路径不存在：降级使用通用知识和网络搜索进行分析，但在报告中明确标注"知识库缺失，分析基于公开信息，建议补充公司内部数据"
- 可通过环境变量 `GUANSHI_KB_PATH` 覆盖默认路径 `~/.hermes/strategy-knowledge/`

### 苏格拉底审计硬门控
- Step 1.5 是**前置条件**，不是可选项
- 任何一维（事实数据/多视角/时间纵深）完全缺失 = 信息不足
- 信息不足时**绝对禁止**输出分析或建议，只能问问题
- 这是观势最容易被绕过的门控——Agent 倾向"先给点建议再说"，必须严格执行 STOP

### 输出长度控制
- 复杂问题战略报告控制在 4000 字以内
- 如果内容过长，优先保留：核心结论 > 推荐方案 > 竞争分析 > 环境扫描

### 已知坑点

**坑 1**：LLM 倾向在输出中暴露思考术语（"MECE""波特五力""Step 2"）→ 输出前检查，发现即删除。

**坑 2**：S 级问题多路径推理会导致 context 膨胀 → Step 5 末尾强制 Summarizer 压缩为断言集，Step 6 只读断言集。

**坑 3**：战略建议"既要又要"式输出 → 锋利性约束强制做减法，P0 最多 2 个，P1 最多 3 个。

**坑 4**：苏格拉底审计最容易被绕过 → Agent 倾向"先给点建议再说"。但可以给一个"初步判断框架"作为开胃菜（不超过 3 句话），再要信息。

**坑 5**：用户说"具体怎么做"时 → 立刻切换到具体步骤+资源需求+时间表+监控指标，不能停留在框架层面。这是战略报告最常见的"最后一公里"失效。

**坑 6**：专家调度时 Chief 越俎代庖 → 专业深度靠专家 Agent，不靠 Chief 填死的知识库。Chief 只做界定/分解/综合/建议。

**坑 7**：ClawHub 不可用时 guanshi-init.py 自举创建专家 → 自举模板已内置完整方法论，确保离线可用。

**坑 8**：SKILL.md 超过 12KB 发布会失败 → 主文件只保留路由表和核心约束，详细内容在 references/。

**坑 9**：publish --version 与 SKILL.md frontmatter version 不一致 → 发布前必须 patch version 字段。

**坑 10**：新陈代谢 → 新增内容必须替换现有内容（一换一原则），总量保持稳定。
