# 质量检查清单

按层级和严重程度分级。P0 必须全部通过，P1 建议通过，P2 最佳实践。

---

## P0：必须通过

### 头部（description）

- [ ] description 含触发短语（`Use when user asks to` 或等效写法）
- [ ] 触发短语覆盖了用户可能的多种表达方式
- [ ] description 含不适用边界
- [ ] name 为 kebab-case，≤64 字符

### 概述

- [ ] 有一句话定位说清能力边界
- [ ] 功能范围列表只列"能做什么"，不含操作细节

### 操作指南

- [ ] 至少 2 个按场景给出的完整示例
- [ ] 不同使用场景有不同的示例（不只是一个通用模板）

### 补充说明

- [ ] 覆盖了依赖缺失的兜底方案

### 整体

- [ ] SKILL.md 中无 `[TODO` 占位符残留
- [ ] 只读各步骤标题，能串成通顺的操作链

---

## P1：建议通过

### 头部（description）

- [ ] 第一句简洁明了，一句话说清做什么
- [ ] 触发短语包含中文和英文（如果用户可能用英文）

### 概述

- [ ] 功能范围列表条目精简，每条一行
- [ ] 边界描述不含糊（不说"等"，用穷举或明确限定）

### 操作指南

- [ ] 每个场景有注释说明 + 完整命令/步骤
- [ ] 长流程有用户确认点（确认门）

### 补充说明

- [ ] 覆盖了常见错误场景和处理方式
- [ ] 给出了具体的兜底命令/方案，不只是"处理异常"

### 整体

- [ ] SKILL.md ≤ 400 行
- [ ] 超出内容拆到了 references/
- [ ] 输出格式有明确定义（格式和存放位置）

---

## P2：最佳实践

- [ ] 解释了"为什么"而不只是"必须"（让 Agent 能在边界情况自主判断）
- [ ] 补充说明中沉淀了踩过的坑
- [ ] 有至少 2 个覆盖典型场景的工作流端到端示例
- [ ] 不重复 LLM 已有的能力（技能提供结构化流程或领域知识）
- [ ] 不引用不存在的工具
