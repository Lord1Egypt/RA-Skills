# 写作风格指南

本 skill 产出长篇研究报告。语气应当清晰、有证据、有可读性：不要写成冰冷的咨询 slide，不要写成营销热文，也不要变成原始资料堆砌。

---

## 1. 核心风格原则

**证据优先**：关键事实需要可追溯来源。优先使用一手来源，无法确认的事实标注「未搜到 / 存疑」，不要编造缺失信息。

**慢思考**：解释事件、决策或转折为什么重要。不要把复杂问题压缩成「改变一切」这类口号。

**具体经验**：讨论产品体验或用户反馈时，引用具体场景，不用模糊情绪标签代替证据。

**有边界的预测**：未来推演必须写清逻辑和假设。区分「最可能」「下行情形」「上行情形」。

---

## 2. 写作特征

### 结构清楚，可读

10,000-30,000 字报告必须有清晰层级。中文报告使用中文章节编号（`一、二、三`），需要时用阿拉伯数字做子章节（`2.1`、`2.2`）。

每一节开头用一两句话说明本节回答什么问题。

### 克制使用第一人称

做判断时可以承认不确定性：

- 好：`我的判断是……`、`我倾向于认为……`、`我有点怀疑的是……`
- 避免：`显然`、`众所周知`、`毫无疑问`

第一人称可以用，但要少。报告不是日记。

### 写叙事，不写年表堆砌

发展史段落要有清楚的弧线。每个关键时刻都要回答：为什么它改变了局面？

避免写成 `2023 年 1 月发布了 A，2023 年 3 月发布了 B` 这类流水账。

### 分层推理

不要从事实直接跳到结论。优先采用：

```text
现象 → 表层解释 → 更深的问题 → 核心判断
```

核心判断段落要让读者看见结论是怎么来的。

---

## 3. 句子规则

- 中文平均句长控制在 15-25 字左右
- 长短句交替
- 避免连续 3 个长句或连续 3 个短句
- 单句不要超过 2 个分句

---

## 4. 标点

| 项目 | 规则 |
|------|------|
| 中文标点 | 中文正文使用全角标点 |
| 英文、URL、代码 | 保留原始英文标点 |
| 引号 | 中文正文使用 `「」`；英文原文引用保留原文引号 |
| 感叹号 | 尽量不用；正式报告通常为 0 |
| 破折号 | 少用；同一段不要超过 1 处 |

---

## 5. 写具体

用具体事实替换大而空的判断：

- 弱：`该公司在这一阶段实现了快速增长`
- 好：`从 2024 年中期的 1000 万美元 ARR 到 2025 年底的 10 亿美元，增长曲线几乎是垂直的`

- 弱：`用户口碑两极分化`
- 好：`GitHub issue 里最常见的抱怨是索引速度慢，但 Reddit 上被反复夸的是搜索结果相关性`

硬性黑名单和最终审核步骤见 `review-checklist.md`。
