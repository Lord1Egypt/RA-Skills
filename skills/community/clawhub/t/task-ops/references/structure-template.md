# 三层结构模板

本文件定义了领域负载物技能的标准三层结构模板。生成新技能时，将 `{变量}` 替换为实际内容。

---

## SKILL.md 模板

```markdown
# {技能中文名}知识参考库

## 定位

本技能是 **Task Ops 的领域负载物仓库**，不包含任何执行框架。只提供{领域名}的"是什么"和"长什么样"——执行全部委托Task Ops。

| 本技能提供 | Task Ops消费方式 |
|-----------|-------------|
| {领域名}任务清单+依赖拓扑 | 内容轴·清单法的任务目录 + 执行轴·管线编排的依赖输入 |
| {任务类型}要求 | 内容轴·清单法的组件清单 |
| 优秀范本（待填充） | 内容轴·样本法的样本 |

## 三层结构

```
第一层：{清单名称} + 依赖拓扑   →  references/{catalog-filename}.md
第二层：{第二层名称}            →  references/{requirements-filename}.md
第三层：{第三层名称}            →  references/exemplars.md
```

## 依赖声明

本技能**强依赖** Task Ops (task-ops)。没有Task Ops，本技能只有参考查阅能力，无法执行任何{任务类型}任务。

**加载检查流程**（每次激活时执行）：

1. 检测 `task-ops` 技能是否已安装
2. **未安装** → 自动安装 `task-ops` 技能
3. **安装成功** → 同时加载Task Ops，按本技能"使用规则"执行
4. **安装失败** → 降级为**只读参考模式**：
   - ✅ 允许：查阅{清单名称}、{第二层简称}、范本索引
   - ❌ 拒绝：任何涉及{任务类型}产出、管线编排、内容生成的任务，并提示"需先安装 Task Ops"

**任务模式判定**：

| 任务类型 | 无Task Ops | 有Task Ops |
|---------|--------|--------|
| 查阅{清单名称}/要求/范本 | ✅ 只读参考 | ✅ 完整 |
| 按{方法论}产出{成品类型} | ❌ 拒绝 | ✅ Task Ops编排执行 |
| 依赖拓扑推导{管线名称} | ❌ 拒绝 | ✅ Task Ops执行轴 |
| {合规/质量}检查点插入 | ❌ 拒绝 | ✅ Task Ops守护单元 |

## 使用规则

1. **依赖检查**：激活时按上述流程检测并安装Task Ops
2. **首次加载**：读取 `references/{catalog-filename}.md`，获取域分类、依赖拓扑、UTOS元操作映射提示
3. **按需深入**：确认目标{任务}类型后，读取 `references/{requirements-filename}.md` 获取组件清单；如需样本法，读取 `references/exemplars.md` 获取范本
4. **委托Task Ops**：将{清单名称}作为清单法输入、范本作为样本法输入、依赖拓扑作为管线编排输入，交给Task Ops执行轴+内容轴处理
5. **{用户填充说明}**

## 与Task Ops的接口

当Task Ops处理{领域名}领域任务时：

- **Step 0 三轴判定**：{领域}任务通常为{复杂度判定}+{内容类型判定} → {激活的轴}
- **Step 1 领域校准**：{领域}的R1(信息密度)={描述}+R2(创造性)={描述}+R3(交互性)={描述}+R4(规范性)={描述}+R5(迭代性)={描述} → {权重推导结论}
- **Step 2 内容轴**：清单法用本技能的{requirements-filename}；样本法用本技能的exemplars
- **Step 3 执行轴**：管线编排基于本技能的依赖拓扑自动推导元操作序列
- **Step 4 交付**：G类守护单元自动插入{质量检查点类型}

## {领域特有维度标题}

{特有维度的表格或列表说明}

## {域概览标题}

按{组织原则}组织，共{域数}域{总任务数}种{任务类型}：

| 域 | 任务数 | 典型任务 |
|----|--------|---------|
{域概览表格行}
```

## 第二层：Requirements 文件模板

> **边界说明**：以下五字段是内容轴**清单法的组件清单**，定义每个任务产出物的构成要素。它们**不替代** UTOS 能力单元Schema（单元ID/名称/元操作/输入/输出/依赖/AI自治度/组合接口等）。两者是不同层面的描述——requirements 管"产出物由什么组成"，UTOS Schema 管"任务怎么执行"。

### 每个任务的槽位定义模板

```markdown
### {ID} {任务名}
- **必选组件**: {核心要素1}、{核心要素2}、{核心要素3}...
- **可选组件**: {可选要素1}、{可选要素2}...
- **组装顺序**: {步骤1}→{步骤2}→{步骤3}→...
- **{约束类型}约束**: [待补充] 或具体约束描述
- **格式**: {输出格式}
```

### 五字段标准（每个任务的组件必须覆盖）

| 字段 | 是否必填 | 说明 |
|------|:-------:|------|
| 必选组件 | ✅ | 产出的最小必要元素集合 |
| 可选组件 | ❌ | 增强但不影响基本功能的元素 |
| 组装顺序 | ✅ | 组件填充的逻辑顺序 |
| 约束 | ⚠️ | 领域特定的质量/合规/风格限制 |
| 格式 | ✅ | 输出物格式（Markdown/Word/Excel/代码等）|

## 第三层：Exemplars 文件模板

### 结构说明

第三层采用**清单主文件 + 子文件**结构：

- `references/exemplars.md` — 范本清单主文件（按域分组的索引 + 公开标准索引 + 填充指南）
- `references/exemplars/{ID}-{name}.md` — 具体范本文件（每个范本独立一个文件）

主文件只做索引，具体范本内容存子文件。这样主文件不会因任务数多而膨胀，AI 也能按需加载单个范本。

### 主文件模板

```markdown
# {优秀范本库名称}

{领域名}的范本清单与索引。用于UTOS内容轴**样本法**。

UTOS内容轴样本法执行流程：
1. 获取样本（本文件索引指向的子文件或用户提供）
2. 分析样本四维度：结构、风格、逻辑、格式
3. 整理为「样本特征摘要」
4. 模仿产出（保持结构框架和风格，替换内容）

> **使用说明**：标注 `[待填充]` 的条目需用户补充范本后方可使用样本法。

---

## {第一个域名}

| 任务ID | 任务类型 | 范本 | 状态 |
|--------|---------|------|------|
| {X}-01 | {任务名} | [待填充] {范本描述} | 空 |
| {X}-02 | {任务名} | {公开标准名}（公开标准） | 可用 |
...

## {第二个域名}

| 任务ID | 任务类型 | 范本 | 状态 |
|--------|---------|------|------|
...

---

## 公开标准范本索引（可选）

| 标准 | 适用任务 | 来源 |
|------|---------|------|
| {标准名} | {任务ID} | {来源机构} |
...

---

## 用户填充指南

用户向本技能添加范本时：

1. **脱敏**：移除所有敏感信息（产品名、个人信息、商业机密），用占位符替换
2. **存放**：将范本文件放入 `references/exemplars/` 目录，按任务ID命名（如 `{X}-01-{task-name}.md`）
3. **注册**：在本文件对应条目中更新范本路径和状态（空→可用）
4. **特征摘要**：为每个范本撰写四维分析（结构、风格、逻辑、格式），存入同目录 `{ID}.summary.md` 文件
```

### 子文件模板

每个具体范本独立存放在 `references/exemplars/` 目录下：

```markdown
### {任务ID} {范本名称}

**对应任务**: {任务ID}

**来源**: [来源说明]

**适用场景**: [什么情况下参考此范本]

**{范本正文标题}**:
> （范本内容——对于代码类领域此处为可运行代码）

**关键设计决策**:
- **决策1**: [为什么这样设计]
- ...

**可复用要素**:
- [哪些部分可以抽象为通用模式]
```

### 范本数量建议

| 领域规模 | 最少范本数 | 推荐范围 |
|---------|----------|---------|
| 小型(<30任务) | 6 | 8-12 |
| 中型(30-60任务) | 8 | 10-15 |
| 大型(>60任务) | 10 | 15-20 |

每个主要域至少应有1-2个范本。
