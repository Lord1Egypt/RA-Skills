# 反模式（Antipatterns）

## 必须避免

### 1. 未确认直接画图

**错误做法**：用户刚说"画个架构图"，立刻动手生成，没有先分析需求。

**正确做法**：
1. 先分析用户输入（pre_think 钩子会自动校验输入）
2. 输出结构化思考结果
3. 用 AskUserQuestion 让用户确认
4. 确认后再动手

**深层原因**：跳过确认会导致生成结果不符合预期。

---

### 2. 每次都从头重做

**错误做法**：用户说"把节点颜色改成蓝色"，却重新生成整个图表。

**正确做法**：
1. 读取当前 .drawio 文件
2. 定位需要更新的部分
3. 只改需要改的部分
4. 保存为新版本（pre_iterate 钩子会自动触发备份）

**深层原因**：从头重做会丢失用户之前的调整。

---

### 3. 忽略版本管理

**错误做法**：生成图表后不初始化版本管理，导致无法回溯。

**正确做法**：
1. 首次生成后立刻 `drawio_version.py init`
2. 每次迭代更新前先 `drawio_version.py save`
3. 向用户展示版本更新信息（pre_vc 钩子会检查版本上限）

**深层原因**：没有版本管理，一旦改错无法恢复。

---

### 4. 跳过钩子校验（新增）

**错误做法**：在关键操作前忽略钩子校验结果，强行继续执行。

**错误示例**：
- pre_think 钩子报错"输入为空"，仍然继续画图
- pre_confirm 钩子报错"选项不足"，仍然用残缺选项让用户确认
- pre_iterate 钩子报错"输出目录不存在"，仍然写入文件

**正确做法**：
1. 每次钩子调用后检查返回结果
2. 遇到 abort=True 立即终止当前操作
3. 修复钩子指出的问题后再继续
4. 参考 hooks 执行结果中的 message 定位问题

**深层原因**：钩子是工作流的守护线，跳过校验等于无视约束。

---

### 5. 篡改钩子注册表（新增）

**错误做法**：直接更新 `drawio_hooks.py` 的内置钩子逻辑，绕过限制。

**正确做法**：
1. 按需 `unregister()` 特定钩子（如禁用自动预览）
2. 用 `register()` 在对应 Hook Point 插入自定义逻辑
3. 保持内置钩子的核心校验逻辑不变

**深层原因**：直接篡改会使升级时丢失自定义内容，且可能导致工作流保护缺失。
