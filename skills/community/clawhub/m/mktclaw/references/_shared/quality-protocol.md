## 🧭 使用指引

本协议定义**三个独立阶段**的统一规范：

| 阶段 | 角色 | 目标 | 触发条件 |
|------|------|------|---------|
| **Stage 1 — 筛选** | Screener | 在多个候选方案中选出最优 | Main Agent 产出 ≥2 个候选方案时 |
| **Stage 2 — 评估** | Evaluator | 对最终交付物做对抗式自检 | 所有交付物在交付前必经 |
| **Stage 3 — 合规** | Compliance | 品牌/广告法/平台红线自检 | 所有交付物在交付前必经 |

> **用户不可见**：此协议是 AI 内部质量流程，对用户透明。用户只感知到"已交付"或"AI 自检后优化版"。

## 🧬 Stage 0：创意多样性（Generate 阶段）

> 候选方案 ≥2 个时触发。解决 AI 默认从"最熟悉的解空间"采样导致的同质化问题。

### 强制维度跨越（Forced Divergence）

Generate 之前先确定差异维度，强制每个候选方案走不同路径：

| 维度 | 选项 A | 选项 B | 选项 C |
|------|------|------|------|
| 情感调性 | 温情/治愈 | 激情/热血 | 反转/戏谑 |
| 目标人群 | 核心用户 | 边缘用户 | 跨圈层 |
| 表达形式 | 故事化 | 数据型 | 互动型 |
| 创新程度 | 微创新 | 突破性 | 跨界嫁接 |
| 风险等级 | 保守 | 平衡 | 激进 |

**规则**：随机抽取 3-4 个维度，每个方案走不同组合，任意两方案相似度 ≤50%。

### 反共识约束

Generate 前列出"业内常见套路"，要求至少 1 个方案走反共识路径。反共识方案标注 ⚠️ 风险高但天花板也高，Tournament 时不能因"风险高"直接淘汰。

### 跨界嫁接

每个 Generate 过程包含 ≥1 个跨界嫁接方案（借鉴非本行业的方法/机制/表达）。

| 本行业 | 借鉴行业 | 嫁接方式 |
|-------|---------|---------|
| 美妆 | 游戏 | 抽奖盲盒/角色扮演种草 |
| 食品 | 奢侈品 | 限量发售/仪式感开箱 |
| 3C | 餐饮 | 快闪体验店/主厨式讲解 |

## 🏆 Stage 1：Tournament 成对比较协议

### 核心原则

**为什么用 Tournament 而不是打分排序**：
- 绝对打分易受 Self-Preference 影响（AI 给自己的方案打高分）
- 成对比较把"哪个更好"的判断收敛到具体维度
- 淘汰赛制自然产生唯一冠军，决策成本低

### v5.1 升级：Swiss-System（≥4 候选方案时启用）

**问题**：纯淘汰赛存在"冷门爆冷"——弱方案因对阵顺序运气好进决赛。
**解决**：Swiss-System 累计得分制 + 置信区间交叉验证。

#### Swiss-System 流程

```
候选方案 A、B、C、D... (≥4 个)

Round 1：随机配对
  A vs B → 胜者 +1 分
  C vs D → 胜者 +1 分

Round 2：同分者配对
  1分组：A vs C
  0分组：B vs D

Round 3：再次同分者配对（避免重复对阵）
  累计得分前 2 名进入 Final

Final：累计得分最高 vs 第二高 → 冠军

判定规则：
- 累计得分最高 = 冠军
- 若并列 → 启动置信区间交叉验证
```

#### 置信区间交叉验证

每场对决产出胜率 + 95% 置信区间，区间重叠则视为"势均力敌"，需加赛。

| 场次 | 方案X | 方案Y | 胜方 | 胜率(95% CI) | 是否加赛 |
|------|-------|-------|------|:----------:|:------:|
| R1 | A | B | A | 65% (55-75%) | 否 |
| R1 | C | D | C | 70% (62-78%) | 否 |
| R2 | A | C | A | 55% (45-65%) | ⚠️ 区间重叠，加赛 |
| R2 加赛 | A | C | C | 58% (50-66%) | 否 |

#### <4 候选方案时：保留纯淘汰赛

```
候选方案 A、B（仅 2 个）
→ 直接 Final：A vs B → 冠军

候选方案 A、B、C（3 个）
→ Round 1: A vs B → 胜者
→ Final: R1胜者 vs C → 冠军
```

### 通用对决维度（每个 Type Agent 必须在此基础上叠加领域维度）

| 通用维度 | 判断问题 | 权重影响 |
|---------|---------|---------|
| **需求匹配度** | 谁更精准地回应用户 Brief 的核心目标？ | 高（一票否决） |
| **差异化** | 谁在同品类/同场景中更独特、更难复制？ | 中高 |
| **可执行性** | 谁更容易在用户现实约束下落地（预算/时间/能力）？ | 中 |
| **风险可控** | 谁的隐藏假设更少、Plan B 更完善？ | 中 |

### 领域维度叠加规则

每个 Type Agent 在 `main-agent.md` 中定义**领域专属对决维度**（3-5 个），格式：

```markdown
## 领域对决维度（叠加在 quality-protocol.md 通用维度之上）

| 维度 | 判断问题 |
|------|---------|
| [领域专属维度1] | [一句话判断问题] |
| [领域专属维度2] | [一句话判断问题] |
```

### 对决记录表（统一输出格式）

```markdown
| 场次 | 方案X | 方案Y | 获胜方 | 关键理由 | 淘汰方保留价值 |
|------|-------|-------|--------|---------|--------------|
| R1 | [方案A] | [方案B] | [胜者] | [一句话] | [可融合的点] |
| R2 | [R1胜者] | [方案C] | [胜者] | [一句话] | [可融合的点] |
| Final | [R2胜者] | [方案D] | **冠军** | [最终理由] | — |
```

### 合成建议

冠军方案不一定是"完美方案"，需要从淘汰方案中**融合保留价值**：

```markdown
### 冠军方案的融合改进
- 融入 [淘汰方案X] 的 [具体元素]
- 警惕 [淘汰方案Y] 中暴露的 [风险点]
```

## ⚔️ Stage 2：Adversarial Verification 协议

### 核心原则

**Adversarial Mindset**：你不是来确认方案"很好"的，你是来**主动攻击方案、找出所有漏洞**的。

**关键偏见警告**：AI 产出的方案普遍存在的偏见（各领域有专属版本，由 Type Agent 补充）：
- 使用"安全"但平庸的通用方案
- 假设条件未被验证
- 数据基于理想场景而非历史基准
- 忽略落地成本和现实约束

### v5.1 升级：LLM-as-Judge 语义评分

**问题**：评估报告指出"断言方式仅 regex + keyword match，无法评估'方案质量'"。
**解决**：在 Adversarial 完成后，**追加 LLM-as-Judge 语义评分**。

#### LLM-as-Judge Prompt 模板

```markdown
## 评估任务

你是一位资深的 [领域] 专家。请基于以下 Brief 和候选方案，从专业角度打分。

## Brief 摘要
[Brief 内容]

## 候选方案
[方案 A / B / C...]

## 评估维度（10 分制，精确到 0.1）

| 维度 | 评分标准 |
|------|---------|
| **Brief 回应度** | 是否精准回应了 Brief 的核心目标？ |
| **专业深度** | 是否有真正的洞察而非套话？ |
| **创新性** | 是否突破行业套路？ |
| **可执行性** | 在用户现实约束下能否落地？ |
| **风险可控性** | 隐含假设是否清晰？Plan B 是否完备？ |

## 输出 JSON 格式

{
  "evaluation": {
    "方案A": {
      "brief_response": 8.5,
      "professional_depth": 7.0,
      "innovation": 6.5,
      "executability": 8.0,
      "risk_control": 7.5,
      "weighted_total": 7.4,
      "key_strength": "...",
      "key_weakness": "..."
    }
  },
  "ranking": ["方案B", "方案A"],
  "recommendation": "推荐方案B"
}
```

#### LLM-as-Judge 反偏倚机制

| 偏倚类型 | 反偏倚措施 |
|---------|----------|
| Self-Preference | 使用不同模型/人设做 Judge |
| 长度偏倚 | 评分前标准化方案长度 |
| 顺序偏倚 | 随机打乱方案顺序 |
| 中央集中偏倚 | 强制至少 1 个方案 ≥8 分或 ≤4 分 |

### 5 维度评估框架（100 分制）

| 维度 | 权重 | 评估标准 | 反偏倚检查 |
|------|------|---------|----------|
| **需求覆盖度** | 30% | 用户每个需求点都有对应交付 | 是否只覆盖了"容易写的"部分？是否避开了难做的场景？ |
| **专业深度** | 25% | 有洞察而非泛泛而谈 | 分析是否停留在表面？数据是否基于真实基准还是编造？ |
| **逻辑一致性** | 20% | 各模块自洽、推导链完整 | 推导链是否有跳步？前后假设是否冲突？ |
| **可执行性** | 15% | 含可操作步骤+时间节点+资源需求 | 时间节点是否过于乐观？资源是否可得？ |
| **创新性** | 10% | 有领域专属洞察/差异化亮点 | "创新"是否只是换了个说法？是否落入行业套路？ |

### 🔴 Red Team 通用攻击（4 项必做）

每个 Type Agent 在此基础上**叠加 2-4 项领域专属攻击**。

#### 攻击 A：假设粉碎机

```
列出方案中所有隐含假设，逐一质疑：
- 这个假设在什么条件下不成立？
- 如果假设错了，整个方案会怎样崩塌？
- 方案是否提供了 Plan B？
- 假设是否相互依赖（一个错导致连环错）？
```

#### 攻击 B：极端场景压力测试

```
把方案放到最苛刻的条件下检验：
- 预算减半时方案还能跑吗？
- 时间压缩到 1/3 时关键节点会怎么崩？
- 目标受众完全不响应时有什么早期信号？
- 竞品在 24 小时内跟进出反应策略时怎么办？
```

#### 攻击 C：成本现实检验

```
从真实落地成本角度审视：
- 隐性成本（人力/工具/学习曲线）是否被低估？
- 推荐资源（达人/媒介/工具）的真实市场价格？
- 时间成本（沟通/修改/审批）是否被忽略？
- 失败成本（方案不奏效时的补救代价）？
```

#### 攻击 D：竞品/对标撞车检测

```
将方案核心元素与以下对象比对：
- 同领域 Top10 玩家的现有做法
- 近 3 年该领域的新趋势
- 跨领域但调性相似的对标
结论：是否存在"无意抄袭"或"落入套路"？
```
