## 第1节：三层架构详解

### 1.1 三层架构全景图

```
┌─────────────────────────────────────────────────────────────────────────┐
│                          三层架构体系                                   │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│   ┌─────────────────────────────────────────────────────────────────┐   │
│   │  技能供给层（Skill供给）                                        │   │
│   │  OPC蒸馏真人专家→Skill→喂给Agent                                 │   │
│   ├─────────────────────────────────────────────────────────────────┤   │
│   │  部署层（能力交付）                                              │   │
│   │  AI Agent虚拟驻场 = 未来FDE                                      │   │
│   ├─────────────────────────────────────────────────────────────────┤   │
│   │  本体层（业务骨架）                                              │   │
│   │  Palantir/万融类平台 = 企业业务结构数字化                        │   │
│   └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 1.2 本体层详解

**核心定位**：企业业务结构的语义化表示

#### 构成要素

| 要素 | 作用 | 示例 |
|:----|:----|:-----|
| ObjectType | 核心实体定义 | 客户、订单、产品、服务 |
| LinkType | 实体间关系 | 客户-订购-产品 |
| Property | 实体属性 | 客户名称、订单金额 |
| ActionType | 可执行操作 | 创建订单、确认付款 |
| Function | 业务逻辑 | 计算折扣、风险评估 |

#### 安全与审计

- **权限链路**：谁可以看/改/删，每条记录可追溯
- **操作审计**：所有ActionType的执行记录
- **变更历史**：Property的完整修改轨迹

### 1.3 部署层详解

**核心定位**：Agent作为虚拟FDE（Forward Deployed Engineer）

#### 传统FDE vs Agent FDE

| 对比维度 | 传统真人FDE | Agent虚拟FDE |
|:--------|:-----------|:------------|
| 部署方式 | 驻场办公 | 云端/混合部署 |
| 扩展性 | 受限于人力 | 理论上无限 |
| 边际成本 | 招募+培训成本 | 接近零 |
| 知识传承 | 依赖人员流动 | 固化为Skill |
| 7x24可用 | 需轮班 | 完全支持 |
| 情绪稳定 | 有波动 | 完全稳定 |

### 1.4 技能供给层详解

**核心定位**：将人类专家能力蒸馏为可执行Skill

#### 蒸馏流程

```
真人专家
    │
    ▼ [访谈/观察]
隐性知识
    │
    ▼ [解构/显性化]
业务规则 + 决策逻辑 + 典型案例
    │
    ▼ [模板化]
标准Skill
    │
    ▼ [测试/优化]
OPC技能市场
    │
    ▼ [客户调用]
Agent执行 → 效果反馈 → Skill迭代
```

## 第2节：企业诊断评分卡（完整版）

### 2.1 评分说明

- **评分周期**：首次诊断1-2小时访谈，后续每季度复评
- **评分方式**：定量（数据）+ 定性（访谈）结合
- **权重说明**：可根据行业特点调整维度权重

### 2.2 完整评分表

#### 维度一：业务抽象度（权重：20%）

| 评分项 | 10分 | 7分 | 4分 | 1分 |
|:------|:-----|:----|:----|:----|
| 流程标准化程度 | SOP完整，覆盖>90%场景 | SOP覆盖主要流程 | 部分流程有SOP | 无SOP，纯经验驱动 |
| 交付物可量化程度 | 交付标准完全量化 | 主要交付物可量化 | 部分交付物可量化 | 交付标准模糊 |
| 客户需求收敛程度 | 需求类型<20种 | 需求类型20-50种 | 需求类型50-100种 | 需求类型>100种 |

**维度总分** = 三项平均分

#### 维度二：数据资产度（权重：20%）

| 评分项 | 10分 | 7分 | 4分 | 1分 |
|:------|:-----|:----|:----|:----|
| 数据完整度 | 全流程数据覆盖 | 主要环节有数据 | 部分环节有数据 | 数据极少 |
| 数据质量 | 数据准确率>95% | 准确率80-95% | 准确率60-80% | 准确率<60% |
| 数据可获取性 | API/系统自动采集 | 人工整理后入库 | 分散在文档/邮件 | 难以数字化 |

**维度总分** = 三项平均分

#### 维度三：客户复购度（权重：20%）

| 评分项 | 10分 | 7分 | 4分 | 1分 |
|:------|:-----|:----|:----|:----|
| 合同周期 | 年度合同/长期合同 | 半年-年度合同 | 季度合同 | 项目制 |
| 续约率 | >90% | 70-90% | 50-70% | <50% |
| 客户依赖度 | 核心决策依赖平台 | 重要决策参考平台 | 偶尔使用 | 几乎不用 |

**维度总分** = 三项平均分

#### 维度四：知识显性度（权重：20%）

| 评分项 | 10分 | 7分 | 4分 | 1分 |
|:------|:-----|:----|:----|:----|
| 文档覆盖度 | 核心业务100%文档化 | 80%以上文档化 | 50%文档化 | <50%文档化 |
| 知识传承体系 | 有正式师徒制/KM | 非正式知识分享 | 偶尔分享 | 无传承 |
| 外部可获取性 | 外部可完全复现 | 外部可部分复现 | 高度依赖内部人 | 不可复现 |

**维度总分** = 三项平均分

#### 维度五：组织可分离度（权重：20%）

| 评分项 | 10分 | 7分 | 4分 | 1分 |
|:------|:-----|:----|:----|:----|
| 人员替代成本 | 新人1周上手 | 1月上手 | 3月上手 | >6月上手 |
| 关键人集中度 | 无单一关键人 | 1-2名关键人 | 3-5名关键人 | >5名关键人 |
| 业务连续性保障 | 完整交接SOP | 有交接清单 | 非正式交接 | 无交接机制 |

**维度总分** = 三项平均分

## 第4节：商业模式设计

### 4.1 核心分成模式

#### 模式一：效果付费（Recommended）

| 参数 | 设计值 |
|:----|:------|
| 基础分成比例 | 15-25% |
| 效果定义 | 客户业务指标提升 |
| 结算周期 | 季度/年度 |
| 保底条款 | 有最低分成保障 |

**适用场景**：咨询类、方法论类Skill

#### 模式二：调用次数付费

| 参数 | 设计值 |
|:----|:------|
| 单次调用价格 | 0.1-10元（视复杂度） |
| 月度结算 | 有最低消费门槛 |
| 梯度折扣 | 调用量越大折扣越高 |

**适用场景**：工具类、查询类Skill

#### 模式三：订阅制

| 参数 | 设计值 |
|:----|:------|
| 月费区间 | 1000-50000元/月 |
| 服务范围 | 含Skill调用+人工支持 |
| 续费折扣 | 年度续约8-9折 |

**适用场景**：深度服务、高复杂度Skill

#### 模式四：混合模式

```
月费（保底）+ 效果分成（上限不封顶）
```

**适用场景**：高价值、高复杂度项目

## 第5节：与万融等中国版Palantir形态的对接方案

### 5.1 万融平台概述

**定位**：中国版Palantir形态，国内领先的本体与数据智能平台

#### 核心能力对应

| Palantir能力 | 万融对应 | 功能说明 |
|:------------|:--------|:--------|
| Ontology | 万融本体建模 | 企业业务结构数字化 |
| AIP | 万融AI引擎 | 智能决策与推理 |
| FDE | 万融实施团队 | 驻场部署与支持 |
| App Store | 万融应用市场 | Skill分发与交易 |

### 5.2 对接流程

```
┌─────────────────────────────────────────────────────────────┐
│                  OPC-万融对接流程                           │
├─────────────────────────────────────────────────────────────┤
│                                                             │

│    ├── 注册OPC导师账号                                       │
```

### 5.4 其他中国平台对接参考

| 平台名称 | 对接要点 | 特殊要求 |
|:--------|:--------|:--------|
| 某数据智能平台A | API优先 | 需通过技术认证 |

| 某AI平台C | 模型对接 | Skill需支持模型调用 |

### 5.5 多平台策略建议

1. **初期聚焦**：先选择1-2个平台深度合作，建立标杆案例
2. **中期扩展**：积累经验后扩展至更多平台
3. **标准化**：确保Skill可跨平台复用，降低适配成本
