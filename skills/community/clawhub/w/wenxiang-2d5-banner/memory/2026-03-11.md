**时间**：2026-03-11

- ✅ `knowledge/提能任务项目/大宽表设计.md` - Spark SQL 大宽表设计

**核心内容**：

#### 4 个数据表
1. `m_capacity_enhance_task` - 提能任务主表（Header）
2. `m_capacity_enhance_sdc_feedback` - 供应商 SDC 反馈表（1:N）
3. `m_capacity_enhance_verify_feedback` - 验证产能反馈表（1:N）
4. `m_capacity_enhance_bpm_record` - BPM 审批流水表（1:N）

#### 表关联关系
```
task (header)
├── sdc_feedback (1:N, task_id = enhance_task_id)
│   └── verify_feedback (1:N, sdc_feedback_id)
└── bpm_record (1:N, task_id = enhance_task_id, 取最新)
```

#### 核心枚举
- **任务状态**：0-11（待发布→提能完成）
- **审批类型**：1-SDC 无偏差/2-SDC 有偏差/3-验证产能
- **审批结果**：0-处理中/1-通过/2-驳回/3-终止

#### SQL 示例（5 个）
1. 查询所有提能任务及关联 SDC 反馈
2. 查询验证产能有偏差的任务（含偏差率计算）
3. 查询提能任务的审批流水（含最新审批结果）
4. 统计各状态的提能任务数量
5. 查询被驳回的提能任务及驳回原因

#### 大宽表设计
- 以 enhance_task 为 header
- 拼接所有 sdc_feedback 记录（1:N 展开）
- 拼接每条 sdc_feedback 对应的所有 verify_feedback 记录
- 另一侧拼接最新的 bpm_record 记录
- 字段命名规范：表前缀_原字段名（t_, s_, v_, b_）

- 短期：文件写入显式指定 `encoding='utf-8'`，改用 `Add-Content` 追加模式
- 长期：强制 UTF-8 编码规范 + 文件锁重试机制 + 日志队列序列化

## 📝 消息整合最佳实践

### 避免 TTS 轰炸和消息碎片化（2026-03-11）

**错误做法：**
```
1. "好问题！" → 发送消息 + TTS ❌
2. "找到了！" → 发送消息 + TTS ❌
3. "让我再搜索一下" → 发送消息 + TTS ❌
4. 完整报告 → 发送消息 + TTS
```

**正确做法：**
```
1-3. 静默执行所有搜索/分析（不发送任何消息）
4. 整合成完整报告 → 发送一条消息 + TTS（150 字内）
```

**核心原则：**
- ✅ 一条完整消息 > 多条零碎消息
- ✅ 静默执行所有中间步骤
- ✅ TTS 只用于最终结果（150 字内摘要）
- ✅ 飞书消息已包含完整内容 → 最后回复 NO_REPLY
