## M37: 测试环境管理

### 核心方法论
**环境即代码（Environment as Code）** + **数据隔离** + **健康检查**

### 专家操作流程
1. **环境矩阵设计**：
   | 环境 | 用途 | 数据策略 | 稳定性要求 | 部署频率 |
   |------|------|---------|-----------|---------|
   | DEV | 开发自测 | 假数据、可Reset | 低 | 每次Push |
   | TEST | 功能测试 | 脱敏真实数据 | 中 | 每日 |
   | STAGING | 预发布验证 | 脱敏生产数据 | 高 | 每版本 |
   | PERF | 性能测试 | 等量造数 | 中 | 按需 |
   | PROD-SHADOW | 生产镜像验证 | 脱敏生产流量 | 高 | 按需 |
2. **环境配置管理**：
   - 配置外部化：代码与配置分离（环境变量/配置中心）
   - 配置版本化：配置文件纳入Git管理
   - 密钥管理：使用Vault/KMS，禁止硬编码
   - Docker化：docker-compose + 多环境override文件
3. **数据管理策略**：
   - 测试数据生命周期：创建 → 使用 → 清理
   - 数据隔离：用户级/租户级/套件级隔离
   - 数据快照：关键业务数据打快照，支持快速回滚
   - 数据脱敏：生产数据导入测试环境前脱敏
4. **环境健康检查**：
   ```
   健康检查API：
   GET /health → 200 OK
   {
     "status": "healthy",
     "checks": {
       "database": {"status": "up", "latency_ms": 5},
       "redis": {"status": "up", "latency_ms": 2},
       "mq": {"status": "up"},
       "third_party_api": {"status": "up", "latency_ms": 150}
     }
   }
   ```

## M38: 探索性测试

### 核心方法论
**SBTM（Session-Based Test Management）** + **测程（Charter）** + **时间盒（Timebox）**

### 专家操作流程
1. **Charter（测试章程）设计**：
   ```
   格式：
   探索[目标区域/功能]
   使用[资源/工具/数据]
   为了发现[信息/风险/缺陷]

   示例：
   探索订单取消功能在各种时序下的行为
   使用模拟器+网络限制工具
   为了发现取消操作的竞态条件缺陷
   ```
2. **SBTM测程结构**：
   - 测程时长：60-120分钟（不可中断）
   - Charter完成判断：时间用完 或 完成Charter目标 或 发现重大缺陷需要深入
   - 测程间：休息15分钟，充电/整理笔记
3. **探索性测试笔记模板**：
   ```
   ## 测程笔记
   - Charter:
   - 日期/时间:
   - 测试人员:
   - 时间分配: 测试__% 缺陷报告__% 环境__%
   - 测试覆盖的功能/数据/场景:
   - 发现的缺陷（ID + 简述）:
   - 遇到的问题/阻碍:
   - 后续建议/新Charter想法:
   ```
4. **探索路径技巧**：
   - 地标法（Landmark）：先探索核心功能建立心智模型
   - 深度优先：一条路径走到底再回来
   - 广度优先：快速浏览所有功能，标记深入点
   - 对抗性思维：故意做"不该做的事"
   - 角色扮演：以不同用户角色（新手/专家/恶意用户）操作

## M39: 测试进度跟踪

### 核心方法论
**燃尽图** + **进度偏差分析** + **风险预警**

### 专家操作流程
1. **关键进度指标**：
   - 用例执行率 = 已执行/计划执行
   - 用例通过率 = 已通过/已执行
   - 缺陷修复率 = 已修复/已提交
   - 计划偏差 = (实际进度 - 计划进度) / 计划进度
2. **进度风险信号**：
   | 信号 | 阈值 | 行动 |
   |------|------|------|
   | 执行率偏低 | <计划的80% | 增加人力/简化用例 |
   | 通过率偏低 | <90% | 暂停新用例，集中定位失败原因 |
   | 缺陷积压 | 未解决>发现量 | 增加开发投入，降低测试节奏 |
   | 阻塞率高 | >10%用例阻塞 | 集中解决环境/依赖问题 |
   | 需求变更 | 测试中新增需求 | 评估影响面，调整计划 |
3. **每日进度报告格式**：
   ```
   ## 测试日报 YYYY-MM-DD
   - 当前阶段: 功能测试 第X天/共Y天
   - 进度: 执行率__% 通过率__%
   - 今日新增缺陷: N个 (P0:M, P1:P, P2:Q)
   - 阻塞项: (描述+负责人+预计解决时间)
   - 明日计划:
   - 风险预警: (如有)
   ```

## M40: 缺陷生命周期管理

### 核心方法论
**标准生命周期** + **缺陷报告质量** + **缺陷分类规范**

### 标准缺陷生命周期
```
发现(New) → 确认(Confirmed) → 分配(Assigned) → 修复中(In Progress) → 
修复完成(Resolved) → 验证(Verified) → 关闭(Closed)
                    ↑                    ↓
                    └──── 重开(Reopened) ←┘
```

### 专家操作流程
1. **高质量缺陷报告8要素**：
   ```
   1. 标题：模块+现象+条件（如："[订单] 使用已过期优惠券下单时金额未拦截"）
   2. 严重度：Blocker(阻塞发布)/Critical(核心功能失效)/Major(重要功能失效)
            Minor(次要功能异常)/Trivial(UI瑕疵)/Suggestion(建议)
   3. 优先级：P0(立即修复)/P1(本迭代)/P2(下迭代)/P3(排期)
   4. 环境：OS+Browser+版本+账号+数据
   5. 复现步骤：每一步精确描述（开发可直接复现）
   6. 预期结果 vs 实际结果：（含截图/录屏/日志/接口响应）
   7. 复现概率：必现/大概率(>50%)/偶现(<20%)
   8. 附件：截图+录屏+Console日志+Network HAR+测试数据
   ```
2. **缺陷严重度判定标准**：
   | 严重度 | 判定标准 |
   |--------|---------|
   | Blocker | 系统崩溃/无法继续测试/核心流程阻断/数据丢失 |
   | Critical | 核心功能完全不可用/安全漏洞/资金错误 |
   | Major | 核心功能部分不可用/数据正确性问题 |
   | Minor | 非核心功能异常/UI问题不影响使用 |
   | Trivial | 文案错误/样式微调/建议 |
3. **缺陷关闭标准**：
   - [ ] 修复代码已合并到测试分支
   - [ ] 测试人员已验证修复通过
   - [ ] 回归测试通过（修复引入的关联功能验证）
   - [ ] 相关用例已更新（如有影响）
   - [ ] 没有引入新的缺陷

## M41: 回归测试策略

### 核心方法论
**三级回归策略**：全量回归 → 选择性回归 → 冒烟回归

### 专家操作流程
1. **三级回归定义**：
   | 级别 | 范围 | 用时 | 触发条件 |
   |------|------|------|---------|
   | L1 冒烟回归 | 核心Happy Path（10-20个用例） | 15-30min | 每次构建 |
   | L2 选择回归 | 变更影响的功能+高风险模块 | 2-4h | 每个迭代/每次Release |
   | L3 全量回归 | 全部自动化用例 | 4-8h | 大版本发布前 |
2. **回归用例选择规则**：
   - 直接修改的功能：全量回归
   - 修改功能的上下游：选择回归
   - 共享模块/公共库的修改：影响面×对应功能回归
   - 仅配置/文案/样式的修改：冒烟回归
3. **回归测试执行清单**：
   - [ ] 确认环境已部署最新版本
   - [ ] 确认测试数据已准备/刷新
   - [ ] 确认依赖服务正常
   - [ ] 执行回归套件
   - [ ] 比对基线（与上一版本的执行结果对比）
   - [ ] 标注新发现的缺陷（回归缺陷）
   - [ ] 输出回归报告

## M42: 测试执行风险监控

### 核心方法论
**实时风险信号** + **预案触发机制** + **升级路径**

### 专家操作流程
1. **风险信号识别**：
   | 风险类别 | 信号 | 响应 |
   |---------|------|------|
   | 环境风险 | 环境连续3次不可用 | 触发升级→环境负责人 |
   | 质量风险 | P0缺陷密度超阈值 | 暂停迭代，质量评审 |
   | 进度风险 | 连续3天执行率<70% | 调整计划，增加资源 |
   | 依赖风险 | 第三方接口延迟>2h | 启用Mock切换 |
   | 数据风险 | 测试数据被破坏 | 触发数据恢复流程 |
2. **风险升级路径**：
   ```
   L1（4h内）→ 测试负责人 → 协调内部资源
   L2（8h内）→ 项目经理   → 调整计划/资源
   L3（24h内）→ 技术总监   → 重大决策/架构调整
   ```
3. **应急预案**：
   - 环境不可用 → 切换到备用环境/Mock模式
   - 阻塞性缺陷 → 评估Workaround + 降级方案
   - 人力不足 → 优先级重排，缩减P2/P3
   - 需求变更 → 评估影响面 → 更新测试范围
