# Improve-Architecture：架构健康

## 1. Overview（概述）

代码能跑 ≠ 架构健康。本技能在常规流程完成后执行架构层面的健康检查：发现深层耦合、执行删除测试验证可替换性、用 CONTEXT.md 中的统一术语命名，并且不重复 ADR 已经确定的架构方向。

**核心原则**：
- 深度耦合是架构腐化的根因（表层耦合不可怕，深层耦合才致命）
- 好的架构 = 每个模块可以被容易地替换
- 命名是架构的镜子 — 散乱的名字 = 散乱的架构
- ADR 决定的不要再问"要不要换个方向"

## 2. When to Use（使用条件）

**触发条件**：
- C 级任务完成后自动执行
- 项目累计修改文件 > 10 个（建议手动触发）
- 新增模块 ≥ 3 个（建议手动触发）

**可跳过**：
- A 级 / B 级任务（不需要架构级检查）
- 纯配置/数据迁移任务

## 3. Process（执行流程）

### 3.1 三步检查

```
步骤 1: 深层耦合发现（Deep Coupling Discovery）
  ├─ 扫描整个项目：哪些模块有隐藏的依赖？
  │   - A import B（显式依赖 → 正常）
  │   - A 使用 B 的内部数据结构（隐式依赖 → 🟡 警告）
  │   - A 依赖 B 的副作用而非接口（深层耦合 → 🔴 危险）
  ├─ 识别「不可替换」的模块
  │   - 删除 A → 哪些模块会报错？
  │   - 删除 B → A 能不能用 Adapter 轻松替换？
  ├─ 输出：深层耦合候选列表（编号、文件对、问题描述、方案、预估收益）
  └─ 反例 ❌：看到 50 个 import 就说耦合严重（导入数量 ≠ 耦合深度）

步骤 2: 删除测试（Deletion Tests）
  ├─ 核心思想：如果能删除一个模块而系统还能编译/运行，
  │   说明模块是可替换的（好的架构）
  ├─ 做法：
  │   1. 至少选 1 个模块
  │   2. 临时删除（或 mock 一个空实现）
  │   3. 观察：影响范围多大？
  │   4. 输出：删除影响矩阵（模块名 → 受影响文件数 → 可替换评级）
  └─ ❌ 禁止：只删不掉核心模块（如数据库驱动本身）

步骤 3: 命名统一（Terminology Alignment）
  ├─ 用 CONTEXT.md 中的术语检查代码命名
  │   - 如果 CONTEXT.md 说"Order"，代码就不能用 "Purchase" / "Transaction"
  │   - 同一个东西多个名字 = 认知负担 + 重构风险
  ├─ 输出：命名不一致项（出现位置 → 推荐统一为）
  └─ ❌ 禁止：用个人偏好替换已存在的命名约定
```

### 3.2 输出格式

```
## 架构健康报告

### 深层耦合候选
| # | 文件对 | 问题 | 方案 | 收益 | 优先级 |
|---|--------|------|------|------|--------|
| 1 | A→B  | 依赖副作用 | 提取接口 | 可测试性↑ | 🔴 |

### 删除测试结果
| 模块 | 影响文件数 | 可替换性 |
|------|-----------|---------|
| X    | 3         | 🟢 好   |

### 命名不一致
| 位置 | 当前命名 | 建议统一为 |
|------|---------|-----------|
| ...  | ...     | ...       |

### 不重复 ADR
- 已检查 ADR-{n}：{标题} → 当前代码对齐
```

## 4. Risk Scorecard（反借口表）

| # | 纪律 | 常见借口 | 反驳 |
|---|------|---------|------|
| A1 | 专注深层耦合 | "这文件 import 太多了，耦合严重" | import 数量 ≠ 耦合深度，要区分结构耦合和逻辑耦合 |
| A2 | 必须做删除测试 | "只有生产环境才知道能不能删" | 不能在测试环境验证 = 生产也验证不了 |
| A3 | 命名对齐 CONTEXT.md | "这两个词差不多，都行" | 两个名字 = 两个概念（哪怕你不这么想） |
| A4 | 不重复 ADR | "我想再确认一下这个架构方向" | ADR 决定了就执行，重新讨论 = 浪费 |

## 5. Red Flags（危险信号）

- 🚩 删除任何一个模块导致 > 10 个文件报错 — 高度耦合，需要重构
- 🚩 同一概念有 ≥ 3 种不同命名 — 领域模型混乱
- 🚩 深层耦合候选来自"核心模块"（数据库、HTTP 框架）— 这是基础依赖，不算深层耦合
- 🚩 模块没有明确的职责边界（能做任何事）— God Object 反模式
- 🚩 修改建议被 ADR 已有决定覆盖 — 浪费精力

## 6. Verification（自检清单）

**必检（≤5 项）**：
- [ ] 深层耦合候选已列出（编号、文件对、问题、方案、收益、优先级）
- [ ] 删除测试已执行（至少 1 个模块，有影响矩阵）
- [ ] 代码命名对齐 CONTEXT.md 术语（无同一概念多命名）
- [ ] 不重复 ADR 已决定事项（建议的新方向 ≠ ADR 已排除的选项）

**抽检（≤3 项）**：
- [ ] 深层耦合候选只包含逻辑耦合（不含基础框架依赖）
- [ ] 建议有优先级排序（🔴 🟡 💭）
- [ ] 每个建议可直接执行（非"建议深入研究"类模糊建议）

**免检**：是否实际执行重构、新架构的具体设计
