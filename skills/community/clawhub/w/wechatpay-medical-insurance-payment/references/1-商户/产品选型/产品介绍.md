# 商户模式产品介绍

> 来源：[商户产品介绍](https://pay.weixin.qq.com/doc/v3/merchant/4016824672.md) + [医保自费混合收款下单](https://pay.weixin.qq.com/doc/v3/merchant/4016781466.md) + 各 API 文档

## 一、产品概览

**移动医保支付** 是基于「微信医保电子凭证」的线上医保结算解决方案。用户在微信激活医保电子凭证后，可在医院/药店的小程序、公众号 H5 中直接完成挂号、门诊缴费、住院预交、互联网医院诊疗、药店购药等场景的「医保 + 自费」一键支付，无需前往线下窗口排队。

> ‼️ 本技能覆盖 **移动医保支付 2.0**（接口前缀 `/v3/med-ins/`）。1.0 版本（接口前缀 `/v3/medicalinsurance/`）不在本技能覆盖范围。请医疗行业商户尽量升级到 2.0 版本。

商户接入医保支付可以：

- 解决传统就医「三长一短」（挂号排队长、缴费排队长、取药排队长，就诊时间短）
- 一次下单，向医保局和微信支付**同时**收款（无需用户分别支付两次）
- 自动完成医保结算 + 自费收款 + 现金补充（运费等） + 现金减免（预交金等）

## 二、接入模式

医保支付从**商务签约 + 开发主体**角度区分为 3 种官方接入模式，从**接口签名 + 字段**角度可归并为 2 类技术形态：

| 接入模式 | 接口调用主体 | API 证书 | 必传字段 | 下单接口路径 | 本技能对应 skill 目录 |
| ------- | -------- | ------- | ------- | ----------- | ------------------ |
| **独立模式**（直连普通商户） | 医院/药店技术团队独立完成 | 医院 / 药店 API 证书 | `mchid` + `appid`（不传 `sub_*` 字段） | `POST /v3/med-ins/orders` | **本目录（1-商户/）** |
| **直连服务商模式** | 医院/药店申请权限，IT 服务商代开发 | 医院 / 药店 API 证书（仍归属医院） | `mchid` + `appid`（不传 `sub_*` 字段） | `POST /v3/med-ins/orders` | **本目录（1-商户/）** ⭐ |
| **间连模式** | 从业机构（银行 / 支付机构 / 普通服务商） | **服务商 API 证书** | `sp_mchid`（签名中使用）+ `sub_mchid` + `sub_appid` | `POST /v3/med-ins/orders` | [2-服务商/](../../2-服务商/产品选型/产品介绍.md) |

> ‼️ **独立模式 vs 直连服务商模式**：两者**接口签名链路与字段完全一致**（都用医院 API 证书 + `mchid` / `appid`，不传 `sub_*` 字段），区别仅在于商务上由谁开发：
> - **独立模式**：医院 IT 自主完成下单 / 调起 / 回调全链路开发
> - **直连服务商模式**：医院持有商户号和证书，但委托外部 IT 服务商代开发；IT 服务商使用医院的 API 证书签名，下单接口仍以医院 `mchid` 主体调用
>
> ‼️ **间连模式**则与上述两种完全不同：调用主体是从业机构 / 普通服务商，请求体携带 `sub_mchid` + `sub_appid`，签名使用 **服务商 API 证书**，回调路由也需按 `sub_mchid` 分发。请改读 [服务商产品介绍](../../2-服务商/产品选型/产品介绍.md)。
>
> ‼️ **本目录覆盖独立模式 + 直连服务商模式（接口侧无差异）**。已确定走这两种模式的可直接读 [开发参数与业务规则](../接入指南/开发参数与业务规则.md) 和 [接口索引](../示例代码/接口索引.md)。

## 三、典型使用场景（订单类型 `order_type`）

四大主营业务速览（占医保接入需求的 90% 以上）：

- **挂号支付**（`REG_PAY`）— 门诊挂号缴费
- **诊间缴费**（`DIAG_PAY`）— 门诊就诊间检查、检验、处方缴费
- **住院缴费**（`IN_HOSP_PAY`）— 住院预交、出院结算
- **线上问诊**（`INT_REG_PAY` / `INT_RE_DIAG_PAY` / `INT_RX_PAY`）— 互联网医院挂号 / 复诊 / 处方支付

完整订单类型枚举：

| 订单类型枚举              | 中文        | 适用业务         |
| ------------------- | --------- | ------------ |
| `REG_PAY`           | 挂号支付      | 门诊挂号缴费       |
| `DIAG_PAY`          | 诊间支付      | 门诊就诊间缴费      |
| `IN_HOSP_PAY`       | 住院费支付     | 住院预交、出院结算    |
| `PHARMACY_PAY`      | 药店支付      | 定点零售药店购药     |
| `INSURANCE_PAY`     | 保险费支付     | 商业健康险费用      |
| `INT_REG_PAY`       | 互联网医院挂号支付 | 在线问诊预约       |
| `INT_RE_DIAG_PAY`   | 互联网医院复诊支付 | 在线复诊缴费       |
| `INT_RX_PAY`        | 互联网医院处方支付 | 互联网医院开具的处方药费 |
| `MED_PAY`           | 药费支付      | 通用药品费用       |
| `COVID_EXAM_PAY`    | 新冠检测费用    | 核酸检测         |
| `COVID_ANTIGEN_PAY` | 新冠抗原检测    | 抗原检测         |


> ‼️ `order_type` 必须严格匹配实际业务场景，**不可跨场景使用**（例如：挂号场景传 `DIAG_PAY` 会触发医保局业务校验失败）。

## 四、支付类型（`mix_pay_type`）

医保自费混合下单按是否走医保 + 是否需要自费分为 4 种支付形态，前 3 种走 `/v3/med-ins/orders` 混合下单接口（由 `mix_pay_type` 决定字段约束），第 4 种走基础支付接口：

| 支付类型 | 含义 | 走混合下单？ | 金额特征 | 接口 |
| ------- | ---- | --------- | -------- | ---- |
| `CASH_AND_INSURANCE` | 医保 + 自费混合 | ✅ 是 | 医保金额 ≠ 0，微信支付金额 ≠ 0 | `POST /v3/med-ins/orders` |
| `INSURANCE_ONLY` | 纯医保 | ✅ 是 | 医保金额 ≠ 0，微信支付金额 = 0 | `POST /v3/med-ins/orders` |
| `CASH_ONLY` | 纯自费（走混合单） | ✅ 是 | 医保金额 = 0，微信支付金额 ≠ 0 | `POST /v3/med-ins/orders` |
| **（不走混合单）** | **纯自费（基础支付）** | ❌ 否 | 仅微信支付金额，无医保结算 | 直接调用 [JSAPI/小程序基础支付](https://pay.weixin.qq.com/doc/v3/merchant/4012791877.md) |

> ‼️ **`CASH_ONLY` vs 不走混合单**：两者业务上都是"纯自费"，但选哪个要看是否需要纳入医保统计 / 医保对账：
> - 选 `CASH_ONLY`：当前订单暂时不涉及医保金额，但希望与同一就诊流程的其他医保订单**走同一套混合下单 / 查单 / 退款链路**（统一对账）
> - 选 不走混合单：纯零售场景（如药店买生活用品、纯自费体检），不涉及医保对账，直接走基础支付更轻量

前 3 种走混合下单时的字段约束：

| 混合支付类型               | 必填字段                                                                                      | 禁填字段                                                                                      |
| -------------------- | ----------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------- |
| `CASH_ONLY`          | `wechat_pay_cash_fee` (>0) + `prepay_id`                                                  | `pay_order_id` / `pay_auth_no` / `geo_location` / `med_ins_order_create_time` / 4 个医保金额字段 |
| `INSURANCE_ONLY`     | `pay_order_id` / `pay_auth_no` / `geo_location` / `med_ins_order_create_time` / 4 个医保金额字段 | `wechat_pay_cash_fee` / `prepay_id`                                                       |
| `CASH_AND_INSURANCE` | 上述全部字段都必填                                                                                 | —                                                                                         |


金额校验公式（任意一种类型都必须满足）：

```text
total_fee = med_ins_gov_fee + med_ins_self_fee + med_ins_other_fee
          + wechat_pay_cash_fee + cash_reduce_detail 之和
        = med_ins_gov_fee + med_ins_self_fee + med_ins_other_fee
          + med_ins_cash_fee + cash_add_detail 之和

wechat_pay_cash_fee = med_ins_cash_fee + cash_add_detail 之和 - cash_reduce_detail 之和
```

> ‼️ 详细字段约束、金额公式与排错见 [开发参数与业务规则](../接入指南/开发参数与业务规则.md) 的「混合支付类型与金额公式」章节。

## 五、接入前提

### 5.1 资质与权限

1. **企业资质**：医院（公立/民营）、定点零售药店等具备医疗机构资质，并已与当地医保局签约
2. **微信支付商户号**：已开通微信支付，且商户号经营类目涵盖「医院/医疗服务/药店」
3. **医保城市接入**：所在城市已开通微信医保电子凭证支付能力（部分地区暂未开通）
4. **医保局对接**：医院 HIS 系统已完成与当地医保局后台的对接（费用明细上传【6201】、医保下单、医保扣款等）
5. **接入申请**：联系微信医保对接邮箱/对接群，由腾讯侧分配 **渠道号**（`channel_no`）和具体城市的 **城市 ID**（`city_id`）
6. **APIv3 升级**：商户号需开通 APIv3 + 申请 APIv3 密钥 + 商户 API 证书 + 微信支付公钥（推荐）

> ‼️ 开发参数清单、获取步骤与字段约束统一收拢到 [开发参数与业务规则](../接入指南/开发参数与业务规则.md)，本页不再重复列出。

### 5.2 与医保局的协同

医保支付不是单纯的微信支付接入，强制依赖医保局后台：

1. **费用明细上传**【6201】：商户先把每笔费用明细传到医保局后台，得到 `medOrgOrd`（即下单时的 `serial_no`）
2. **医保预结算**：调用医保局接口，得到 `pay_order_id`（医保局支付单 ID）和 `pay_auth_no`（医保局支付授权码）
3. **医保自费混合下单**：把上述参数 + 自费下单返回的 `prepay_id` 一并传给微信医保 `/v3/med-ins/orders` 接口
4. **用户授权 + 调起支付**：通过 JSAPI/小程序调起医保自费混合支付，用户在医保电子凭证页面输入支付密码
5. **回调通知**：微信医保通过 `MEDICAL_INSURANCE.SUCCESS` 事件通知商户

### 5.3 接入产物

成功接入后将具备以下能力：

1. 医保自费混合下单（`POST /v3/med-ins/orders`）→ JSAPI/小程序调起 → 接收"医保混合收款成功通知"
2. 主动查询订单状态（按 `mix_trade_no` 或 `out_trade_no`）作为回调兜底
3. 医保退款发生时，向微信医保发起「医保退款通知」（`POST /v3/med-ins/refunds/notify`）
4. 自费部分独立退款走标准退款接口（详见 [📄 排障手册](../问题排查/排障手册.md)）

## 六、全流程说明

```text
阶段一: 医保授权 (仅 INSURANCE_ONLY / CASH_AND_INSURANCE; 纯自费场景跳过)
    ├── 医院前端 → 腾讯健康授权页 → 医保局 (不经过微信医保应用系统)
    ├── 用户完成医保身份确认
    └── 返回 pay_auth_no (医保局支付授权码; 有时效, 失效需重新授权)
        ↓
阶段二: 医保预结算
    ├── HIS 携带 pay_auth_no + 费用清单 → 医保局
    ├── 返回 pay_order_id (与 pay_auth_no 须来源于同一笔医保缴费上下文)
    ├── 返回基金分账: 统筹 fund_pay_sumamt / 个账 acct_pay / 其他基金 / 现金 psn_cash_pay
    ├── 校验: medfee_sumamt = fund_pay_sumamt + psn_part_amt
    ├── 校验: psn_part_amt = preselfpay_amt + act_pay_dedc
    └── ⚠️ 费用明细变更必须重新预结算, 否则结算阶段会因金额不一致触发自动冲正
        ↓
阶段三: 自费下单 (仅 CASH_AND_INSURANCE / CASH_ONLY; 纯医保场景跳过)
    ├── 直连模式: POST /v3/pay/transactions/jsapi
    ├── 间连模式: POST /v3/pay/partner/transactions/jsapi
    │   └── 必传 sp_mchid / sub_mchid / sp_appid / sub_appid
    ├── out_trade_no = 后续混合单的 out_trade_no (必须一致)
    ├── ✅ amount.total = wechat_pay_cash_fee (现金自费部分)
    ├── ❌ 错误做法: amount.total = total_fee (含医保个账)
    │   └── 后果: 自费单 prepay_id 与混合单期望金额不一致, 触发金额校验失败
    │       (医保个账 med_ins_self_fee 由医保局从个人账户划扣, 不经过微信收银台)
    ├── notify_url: 自费支付回调地址 (HTTPS)
    ├── 返回 prepay_id
    └── V2/V3 兼容: 自费下单可走 V2 (/pay/unifiedorder), 但混合单接口仅支持 V3
        ↓
阶段四: 混合单下单
    ├── POST /v3/med-ins/orders (所有接入模式统一路径)
    ├── 三种 mix_pay_type 参数规则:
    │   ├── CASH_AND_INSURANCE: 必传 prepay_id + pay_auth_no + pay_order_id + 医保金额 + 自费金额
    │   ├── INSURANCE_ONLY:    不传 prepay_id / wechat_pay_cash_fee
    │   │                      必传 pay_auth_no + pay_order_id + 医保金额
    │   └── CASH_ONLY:         不传 pay_auth_no / pay_order_id / 医保金额
    │                          必传 prepay_id + wechat_pay_cash_fee
    └── 返回 mix_trade_no (医保自费混合订单号)
        ↓
阶段五: 拉起收银台
    ├── 公共参数: prepay_id / appid / timeStamp / nonceStr / signType / paySign + mix_trade_no
    ├── 小程序: wx.requestMedicalInsurancePay()
    ├── 公众号 H5: JSSDK requestMedicalInsurancePay (需额外传 appid)
    ├── signType 仅支持 RSA (产品文档 2026-05-14 确认)
    │   └── 自费单走 V2 时, signType / paySign 可走 V2 (MD5 / HMAC-SHA256)
    └── 纯医保 (INSURANCE_ONLY) 仅传 mixTradeNo
        (不传 package / paySign / signType / timeStamp / nonceStr)
        ↓
阶段六: 自费支付 + 医保结算 (微信医保应用系统自动触发, 商户无需驱动)
    ├── 触发链路 (三条任一即可):
    │   ├── 大单通知: eventsvr 收到自费支付成功通知 → 触发医保结算
    │   ├── 用户前端确认: 用户在 LiteApp 确认支付成功 → 触发医保结算
    │   └── 主动查单 (MQ): 下单成功后启动定时查单任务
    ├── 结算流程: 微信医保 → 腾讯健康 → 医保局 → 轮询查单 (最长 30s)
    └── 异常处理:
        ├── 30s 轮询超时: 通过 MQ 投递 UpdateUnknownMedInsOrder 消息补偿查单
        └── 结算最终失败: 自动冲正 → 撤销医保结算 + 释放冻结资金
            (冲正成功后该笔订单不可再发起支付)
        ↓
阶段七: 结果通知 (两次回调)
    ├── 第 1 次: 微信支付自费回调 → 自费下单的 notify_url (用户完成自费支付后触发)
    ├── 第 2 次: 医保混合收款成功通知 → 混合单下单的 notify_url (医保结算完成后触发)
    ├── ✅ 最终判定依据: 以第 2 次回调 mix_pay_status = MIX_PAY_SUCCESS 为准
    │   └── ⚠️ 自费支付成功 ≠ 整笔订单成功
    │       (医保结算可能失败 / 医保可能冲正, 均需退自费款)
    └── 主动查单兜底 (建议拉起后启动, 不要完全依赖回调):
        ├── 按混合单号:   GET /v3/med-ins/orders/mix-trade-no/{mix_trade_no}
        ├── 按商户订单号: GET /v3/med-ins/orders/out-trade-no/{out_trade_no}
        └── 间连模式查单时 query 参数必传 sub_mchid
```
