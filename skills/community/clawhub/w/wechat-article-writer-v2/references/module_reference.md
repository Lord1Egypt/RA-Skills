# 微信公众号排版模块完整参考

本文档列出全部 40 个 fenced block 模块的精确字段说明和写法示例。

> **关键语法**：固定使用 `::::`（4个冒号）开头，`::::`（4个冒号）结尾。开头和结尾冒号数必须一致且固定为4个。`[标题]` 可省略。

## 语法总览

| 格式 | 语法 | 适用模块数 |
|------|------|:--:|
| 元信息（纯文本） | 块内纯文本，无需 key:value | 2 个 |
| Key-Value | `key: value` 每行 | 14 个 |
| Pipe-Delimited | `字段1 \| 字段2 \| 字段3` 每行 | 13 个 |
| Special | Markdown 图片语法或混合格式 | 4 个 |
| 微信原生组件 | Key-Value | 7 个 |

---

零、元信息模块（2个）

这些模块不产生视觉输出，仅用于显式设置文章的标题和摘要信息，会被转换器提取并融入 HTML 的 `<title>` 和 `<meta description>`。

### 0. title — 文章标题

**适用场景**：显式指定文章标题，不依赖 `# 标题` 自动提取。

**写法**：

```
::::title
你的文章标题（不依赖 YAML frontmatter 或 # 标题自动提取）
::::
```

- 块内为纯文本，不需要 key:value 格式
- 放在文章开头处，优先级最高，会覆盖自动提取
- 如果未使用，转换器自动从第一个 `# 标题` 提取

---

### 0. digest — 文章摘要

**适用场景**：显式指定文章摘要，用于分享时显示。不依赖自动截取前80字。

**写法**：

```
::::digest
你的文章摘要，会在分享时显示（不依赖自动截取前80字）
::::
```

- 块内为纯文本，不需要 key:value 格式
- 放在文章开头处，优先级最高，会覆盖自动提取
- 如果未使用，转换器自动从正文第一段提取前80字

## 一、Key-Value 模块（14个）

### 1. hero — 开场 Hero

**适用场景**：任何文章的默认开头。放主标题、副标题、封面图、品牌标签。是文章最重要的门面模块。

**字段**：

| 字段 | 必填 | 说明 |
|------|:--:|------|
| eyebrow | 否 | 顶部标签，大写短文本，如 "FEATURE THEME" |
| meta | 否 | 右上角元信息，如日期 "2026.06" |
| kicker | 否 | 引导语，显示在 title 上方，全大写 |
| title | 是 | 主标题。用 `\|` 分隔时后半部分金色高亮 |
| subtitle | 否 | 副标题/描述 |
| image | 否 | 封面图 URL（推荐 800×400） |
| brand | 否 | 品牌名，显示为左下角 pill |
| tags | 否 | 标签，`\|` 分隔多个 |

**写法示例**：

```
::::hero
eyebrow: 效率工具发布
meta: 2026.06
kicker: 告别手动排版
title: 用Markdown写出 | 专业级公众号文章
subtitle: 40种高级排版模块 + 一键转换HTML，排版从2小时降到10分钟
image: https://picsum.photos/seed/hero/800/400
brand: md2wechat
tags: Markdown | 排版 | 效率
::::
```

### 2. infographic — 信息图卡

**适用场景**：把**一个核心判断**做成视觉冲击卡。适合教程核心观点、发布稿关键信息、中段转场总结。一篇文章可用 1-3 个。

**字段**：

| 字段 | 必填 | 说明 |
|------|:--:|------|
| eyebrow | 否 | 顶部标签，金色边框 pill |
| title | 是 | 标题。用 `\|` 包裹的部分会加上金色下划线+背景高亮 |
| subtitle | 否 | 副标题详细说明 |
| note | 否 | 底部备注。超过20字符自动截断为18字+… |

**title 高亮规则**：`让\|一句判断\|先被看到` → "一句判断" 被金色下划线+阴影高亮。支持多段 `|`。

**写法示例**：

```
::::infographic
eyebrow: 核心发现
title: 排版不是|视觉美化|而是|信息架构|
subtitle: 好的排版让读者在3秒内找到判断、证据和行动入口
note: 一张信息图卡只承载一个核心判断，适合教程、发布稿的中段转场
::::
```

### 3. audience-fit — 适用人群

**适用场景**：明确告诉读者"这篇适合谁，不适合谁"。放在文章前1/3位置，帮助读者快速判断是否值得读。

**字段**：

| 字段 | 必填 | 说明 |
|------|:--:|------|
| title | 否 | 模块标题 |
| fit | 是 | 适合人群，`\|` 分隔多个 |
| avoid | 是 | 不适合人群，`\|` 分隔多个 |

**写法示例**：

```
::::audience-fit
title: 这篇文章适合谁
fit: 需要稳定输出公众号内容的人 | 想用Markdown提升排版效率的人 | 对结构化写作感兴趣的内容创作者
avoid: 只发朋友圈短内容的人 | 完全不需要排版格式的场景
::::
```

### 4. verdict — 结论判断

**适用场景**：给出**结论性判断**。适合论证收尾、章节小结、"一句话说清楚"的场景。视觉上有方块装饰+金色边框。

**字段**：

| 字段 | 必填 | 说明 |
|------|:--:|------|
| eyebrow | 否 | 标签，如 "结论" "最终判断" |
| title | 是 | 结论标题，一句话说清核心判断 |
| body | 否 | 补充说明正文 |

**写法示例**：

```
::::verdict
eyebrow: 最终判断
title: 真正的护城河不是模块数量，而是结构化的内容生产系统
body: 模块解决的是单次排版效率，系统解决的是持续内容输出的一致性。两者结合才有长期价值。
::::
```

### 5. part — 章节分隔

**适用场景**：将长文章分成 P1/P2/P3 几个大段落，每个 Part 有数字圆圈+标题。

**字段**：

| 字段 | 必填 | 说明 |
|------|:--:|------|
| index | 是 | 序号，如 "01" "02" |
| title | 是 | 章节标题 |
| subtitle | 否 | 副标题，可写英文或补充信息 |

**写法示例**：

```
::::part
index: 01
title: 正文能力也要接进同一套系统
subtitle: TYPOGRAPHY · MARKDOWN · STRUCTURE
::::
```

**适用场景**：在正文中给某个小节打标签，做视觉分隔。类似"小标题+标识"的效果。

**字段**：

| 字段 | 必填 | 说明 |
|------|:--:|------|
| label | 是 | 标签文字（金色 pill） |
| title | 是 | 标题文字（H4 样式） |

**写法示例**：

```
::::label-title
label: 注意
title: 模块的任务是把内容拆成更好读的结构
::::
```

| quote | 是 | 引用正文 |
| source | 否 | 出处/作者 |
| note | 否 | 底部补充备注 |

**写法示例**：

```
::::quote

note: 适合金句、采访摘录、文章结论
::::
```

### 8. image-text — 图文双栏

**适用场景**：图片+文字并排展示。适合产品截图配说明、活动海报配文案、界面讲解等需要"图+文绑定"的场景。

**字段**：

| 字段 | 必填 | 说明 |
|------|:--:|------|
| layout | 否 | 布局方向：`right`（图右文左，默认）或 `left`（图左文右） |
| eyebrow | 否 | 标签 |
| title | 是 | 标题 |
| body | 否 | 正文描述 |
| note | 否 | 底部备注 |
| image | 是 | 图片 URL |
| alt | 否 | 图片 alt 文本 |

**写法示例**：

```
::::image-text
layout: right
eyebrow: 界面示意
title: 编辑器和预览实时同步，写完即所见
body: 双栏布局让截图和说明留在同一视线里，不适合独立展示的图片用这个模块更好。
image: https://picsum.photos/seed/imagetext/600/400
alt: 编辑器界面截图
::::
```

**适用场景**：文章末尾的总结卡片。提炼核心观点，帮助读者记住关键信息。

**字段**：

| 字段 | 必填 | 说明 |
|------|:--:|------|
| eyebrow | 否 | 标签，如 "一句话总结" |
| highlight | 是 | 高亮总结句 |

**写法示例**：

```

::::
```

### 13. subscribe — 关注/行动引导

**适用场景**：引导读者关注、分享或采取下一步行动。不限于"订阅"，任何需要双按钮引导的场景都适用。

**字段**：

| 字段 | 必填 | 说明 |
|------|:--:|------|
| label | 否 | 顶部标签 |
| title | 是 | 主标题 |
| subtitle | 否 | 副标题 |
| primary | 否 | 主按钮文字（金色按钮） |
| secondary | 否 | 次按钮文字（白底按钮） |
| note | 否 | 底部备注 |

**写法示例**：

```
::::subscribe
label: 持续更新

subtitle: 我会持续更新公众号排版、AI内容工作流和产品化复盘。
primary: 关注公众号
secondary: 转发给朋友
note: 下一篇继续拆解个人品牌模块
::::
```

### 14. cta — 行动号召

**适用场景**：文章的最终行动号召。放在最末尾，默认带有4个图标卡片（收藏 / 点赞 / 转发 / 继续探索，最后一个高亮）。可通过 `labels` 字段自定义标签文案。

**字段**：

| 字段 | 必填 | 说明 |
|------|:--:|------|
| title | 是 | CTA 文案 |
| note | 否 | 底部标注（大写样式） |
| labels | 否 | 操作按钮文案，逗号分隔，最多4个。不传则默认：`收藏,点赞,转发,继续探索` |

**写法示例**：

```
::::cta
title: 开始用这套模块搭建你的下一篇公众号文章
note: BUILD WITH STRUCTURE
::::
```

自定义标签：

```
::::cta
title: 如果对你有帮助
labels: 收藏,在看,分享给朋友,关注我
::::
```

## 三、Pipe-Delimited 模块（13个）

所有 Pipe 模块每行用 `|` 分隔字段。不同的模块列数不同，需要严格按列数填写。

### 22. cards — 卡片网格

**适用场景**：分类展示几个并列的概念/功能/步骤。自动网格布局，响应式列数。

**列结构（4列）**：`编号 | 标题 | 描述 | accent/default`

| 列 | 说明 |
|:--:|------|
| 1 | 编号/标识（如 "PART 01"） |
| 2 | 卡片标题 |
| 3 | 描述文字 |
| 4 | 样式：`accent`（金色强调卡）或 `default`（普通白卡） |

**写法示例**：

```
::::cards[高级排版模块]
PART 01 | 开场模块 | 先交代判断和阅读入口，hero让读者3秒决定是否继续读 | accent
PART 02 | 信息卡模块 | 把一句判断放大成视觉摘要，infographic适合教程和发布稿 | default
PART 03 | 证据模块 | 用数据、对比、步骤支撑结论 | default
::::
```

### 23. metrics — 数据指标

**适用场景**：用数据指标支撑论点。每个指标一张卡片，显示名称+数值+说明。适合数据报告、成果展示。

**列结构（4列）**：`指标名称 | 数值 | 说明 | accent/default`

| 列 | 说明 |
|:--:|------|
| 1 | 指标名称 |
| 2 | 指标数值（大号字体） |
| 3 | 说明文字 |
| 4 | 样式 |

**写法示例**：

```
::::metrics[关键成果]
排版效率 | +3倍 | 从2小时降到10分钟 | accent
模块数量 | 40个 | 覆盖所有公众号排版场景 | default
阅读完成率 | +38% | 结构清晰读者更愿意读完 | default
::::
```

## 五、标准 Markdown 元素

所有标准 Markdown 语法正常使用：

| 语法 | 效果 |
|------|------|
| `# H1 ~ ###### H6` | 六级标题 |
| `**粗体**` / `*斜体*` | 粗体（金色样式）/ 斜体 |
| `~~删除线~~` | 删除线 |
| `` `代码` `` | 行内代码（金色背景+边框） |
| `- 列表` / `1. 有序列表` | 列表 |
| `> 引用` | 引用块 |
| `> [!NOTE]` / `> [!TIP]` 等 | GFM 提示框 |
| ` ```语言 ``` ` | 代码块（Pygments 高亮） |
| `\| 表格 \|` | 表格 |
| `[文字](url)` | 超链接 |
| `![alt](url)` | 图片 |
| `---` | 分隔线 |
| `[^1]` | 脚注 |

## 七、编写规范

### 模块选择原则

1. **一个模块解决一个阅读任务**：不要为了好看堆叠模块，每个模块都有它的信息传达目的
2. **hero 是唯一的开头**：任何文章都应该以 hero 开头
3. **收尾要有层次**：summary → 个人品牌(author/series) → 引导(subscribe) → 行动(cta)
4. **论证模块适量**：metrics/cases/infographic 选 1-3 个，不要每种都用
5. **视觉模块控制数量**：gallery/image-steps/image-annotate 选 1-2 个即可

### 常见错误避免

| 错误 | 正确做法 |
|------|---------|
| 用6个或以上冒号 `::::::` | 固定使用4个冒号 `::::`，开头和结尾都必须是4个 |
| Pipe模块忘了最后的 accent/default | cards/metrics/people/cases/pricing 必须加第4列样式 |
| 图片用本地路径 `./img/xx.png` | 必须用完整HTTPS URL |
| faq 只写2行却加了第3列 | faq 只有2列：问题\|回答 |
| 在一个文章中 >8 个模块 | 控制 3-6 个重点模块 |
| pricing 特性不用 `/` 分 | 特性必须用 `/` 分隔多个条目 |
| image-steps 不写图片URL | 图片URL是第4列，不可省略 |
