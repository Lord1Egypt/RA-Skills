### 1.1 Esri Geometry API（独立实现，非JTS系）

```
Esri Geometry API Java    ────  不是JTS端口，独立设计
  ├── 支持地理坐标运算（geodesic buffer/distance/area）
  ├── Hadoop/Spark 兼容（分布式空间分析）
  └── REST 序列化原生集成

Esri Geometry API .NET    ────  独立C#实现
  ├── 与 ArcGIS Runtime 深度集成
  └── 支持 ArcGIS 原生格式（GDB/FGDB）
```

> **关键区别**：JTS 系列库默认在平面坐标系下运算（笛卡尔平面）；Esri Geometry API 原生支持大地测量运算（椭球面上的 true geodesic 计算）。对于 CGCS2000 高精度场景，Esri 的 geodesic 方法优于 JTS 系默认平面算法。

### 2.1 Java 生态

#### JTS (Java Topology Suite) — 几何引擎鼻祖

```
groupId:     org.locationtech.jts
artifactId:  jts-core
安装:        Maven/Gradle 依赖
仓库:        https://github.com/locationtech/jts
```

| 概念 | 类/方法 | 说明 |
|------|---------|------|
| 几何工厂 | `new GeometryFactory(new PrecisionModel(1000), 4490)` | 精度模型 + SRID |
| 读取WKT | `new WKTReader().read("POINT(120 30)")` | 文本→几何 |
| 写入WKT | `new WKTWriter().write(geom)` | 几何→文本 |
| 读取WKB | `new WKBReader().read(bytes)` | 二进制→几何 |
| 空间索引 | `new STRtree().insert(envelope, item)` | R-tree 索引 |
| 关系判断 | `geom.relate(other)` | DE-9IM 矩阵 |
| 叠加分析 | `OverlayOp.overlayOp(geom1, geom2, OverlayOp.INTERSECTION)` | 低级叠加 |
| 验证 | `geom.isValid()` | 有效性检查 |
| 面积 | `geom.getArea()` | 平面面积 |
| 长度 | `geom.getLength()` | 平面长度 |

```java
// JTS 典型工作流
import org.locationtech.jts.geom.*;
import org.locationtech.jts.io.*;

GeometryFactory gf = new GeometryFactory(new PrecisionModel(1000), 4490);
WKTReader reader = new WKTReader(gf);

// 构建几何
Point p = gf.createPoint(new Coordinate(120.0, 30.0));
Polygon aoi = (Polygon) reader.read("POLYGON((119 29, 121 29, 121 31, 119 31, 119 29))");

// 空间查询
if (aoi.contains(p)) {
    System.out.println("点落在范围内");
}

// 缓冲区（平面坐标下）
Geometry buffer = aoi.buffer(0.01); // 约1km

// 面积
double areaDeg = aoi.getArea(); // 注意：经纬度下为"度²"无实际意义
```

### 4.2 各语言 CGCS2000 处理示例

#### Python — pyproj (always_xy=True 关键)

```python
from pyproj import CRS, Transformer

crs_4490 = CRS.from_epsg(4490)
crs_4527 = CRS.from_epsg(4527)  # CGCS2000 / 3-degree Gauss-Kruger CM 120E

# 经纬度 → 投影坐标
transformer = Transformer.from_crs(crs_4490, crs_4527, always_xy=True)
x, y = transformer.transform(120.5, 30.2)

# WGS84 → CGCS2000
trans_wgs84 = Transformer.from_crs("EPSG:4326", "EPSG:4490", always_xy=True)
lon_cgcs, lat_cgcs = trans_wgs84.transform(120.0, 30.0)

# ⚠️ 常见陷阱：如果 always_xy=False (默认)，pyproj 会按 EPSG 定义的轴序(lat,lon)处理
# 这会导致经纬度颠倒！始终使用 always_xy=True
```

#### Java — GeoTools (轴序陷阱：northing/easting)

```java
import org.geotools.api.referencing.crs.CoordinateReferenceSystem;
import org.geotools.referencing.CRS;
import org.geotools.geometry.jts.JTS;
import org.geotools.api.referencing.operation.MathTransform;

// 解码 CRS
CoordinateReferenceSystem crs4490 = CRS.decode("EPSG:4490");
CoordinateReferenceSystem crs4527 = CRS.decode("EPSG:4527");
// 注意：EPSG:4490 在GeoTools中轴序为 north/east (lat, lon)!

// 坐标变换
MathTransform transform = CRS.findMathTransform(crs4490, crs4527, true);

// ⚠️ GeoTools CRS.decode("EPSG:4490") 会返回轴序 (纬度, 经度)
// 即 GeographicCRS.WGS84 风格的 northing/easting
// 使用 JTS.transform() 时要注意输入坐标顺序
```

### 4.3 轴序陷阱速查表

| 库/SDK | EPSG:4490 轴序 | 解决方法 |
|--------|:---:|------|
| **pyproj** | lat,lon (默认) | `Transformer(..., always_xy=True)` |
| **GeoTools** CRS.decode("EPSG:4490") | lat,lon | 转换时交换输入坐标 |
| **PROJ** (C API) | lat,lon | 使用 `proj_trans(..., PJ_FWD, ...)` 注意轴序 |
| **proj4js** | x,y (proj4默认) | 跟 PROJ 一致，直接使用 |
| **GDAL** ogr2ogr | 自动处理 | GDAL 3.0+ 自动处理 CRS 轴序 |
| **Shapely** | 无CRS概念 | 纯几何运算，不关心轴序 |
| **PostGIS** ST_Transform | x,y (lng,lat) | `ST_Transform(geom, 4490)` 返回 x,y 轴序 |
| **Esri** API | x,y | Esri 始终用 x,y 轴序 |

> **核心原则**：处理地理坐标时始终检查当前库的默认轴序，尤其在 Python/Java 之间互操作时必须明确轴序约定。
