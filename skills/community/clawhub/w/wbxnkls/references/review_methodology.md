# 合规审查方法论

本文档描述如何对软件代码和文档进行法律合规审查。

---

## 审查流程概览

```
输入（代码/文档）→ 初步评估 → 逐项合规检查 → 风险评级 → 生成报告
```

---

## 一、代码审查方法论

### 适用场景
- 用户提供源代码目录/仓库
- 用户提供单个代码文件
- 用户提供已部署系统的访问方式（API文档/界面）

### 审查步骤

#### Step 1：了解系统架构和数据流
- 阅读项目的 README、架构文档、API文档
- 确定系统处理的**数据类型**（个人数据/敏感个人数据/重要数据/一般数据）
- 确定系统的**用户群体**（中国用户/欧盟用户/加州居民/儿童等）
- 确定系统的**部署位置**（中国境内/境外/混合）

#### Step 2：识别个人信息处理活动
在代码中搜索以下关键词和模式：

**个人数据收集：**
- `email`, `phone`, `address`, `name`, `idCard`, `passport`, `身份证`, `手机号`
- `password`, `token`, `session`, `cookie`, `ip`, `deviceId`, `uuid`
- `location`, `gps`, `坐标`, `地址`
- `biometric`, `fingerprint`, `face`, `人脸`, `指纹`
- `health`, `medical`, `健康`, `医疗`
- `financial`, `credit`, `bank`, `金融`, `银行`

**数据存储：**
- `database`, `db`, `redis`, `mysql`, `postgres`, `mongodb`
- `save`, `insert`, `update`, `delete`, `create`
- 检查是否有加密（AES, bcrypt, scrypt, argon2）

**数据传输：**
- `http`, `https`, `tls`, `ssl`, `api`, `request`, `fetch`
- 检查是否强制HTTPS/TLS 1.2+

**日志：**
- `log`, `logger`, `console.log`, `print`
- 检查日志中是否记录敏感数据（密码、token、身份证号）

**第三方服务：**
- `sdk`, `analytics`, `tracking`, `third-party`, `第三方`
- 检查是否有Google Analytics、Facebook SDK等跨境数据传输

**用户权利接口：**
- `delete`, `export`, `download`, `update`, `consent`, `opt-out`, `withdraw`
- 检查是否实现了数据主体权利接口

#### Step 3：检查具体合规控制

| 检查项 | 检查方法 |
|--------|----------|
| 密码存储 | 是否使用bcrypt/argon2（而非MD5/SHA1明文） |
| 传输加密 | 是否强制HTTPS，TLS版本≥1.2 |
| 会话管理 | session是否设置HttpOnly/Secure/SameSite |
| 访问控制 | 是否有认证/授权机制（JWT/OAuth2/RBAC） |
| 输入验证 | 是否有SQL注入/XSS防护 |
| CSRF防护 | 是否有CSRF token |
| 安全头 | 是否有CSP/HSTS/X-Frame-Options等安全头 |
| 数据保留 | 是否有自动删除过期数据的机制 |
| 同意管理 | 是否有cookie banner/consent记录 |
| 数据泄露检测 | 是否有异常访问监控 |

#### Step 4：生成代码审查发现清单
对每个发现的问题，记录：
- 文件路径 + 行号
- 问题描述
- 违反的法规条款
- 风险等级（高/中/低）
- 修复建议

## 二、文档审查方法论

### 适用场景
- 用户提供PRD（产品需求文档）
- 用户提供隐私政策草案
- 用户提供用户协议/服务条款
- 用户提供API文档
- 用户提供数据流程图/架构图

### 审查步骤

#### Step 1：阅读文档，识别数据类型和流向
- 系统收集哪些个人数据？
- 数据存储在何处？
- 数据是否传输到境外？
- 是否与第三方共享数据？

#### Step 2：对照合规清单逐项检查

**隐私政策审查要点：**
- [ ] 是否明确身份（控制器是谁、联系方式）
- [ ] 是否列出收集的数据类型和来源
- [ ] 是否说明处理目的和法律基础
- [ ] 是否说明数据保留期限
- [ ] 是否说明数据主体权利及行使方式
- [ ] 是否说明Cookie和追踪技术
- [ ] 是否说明数据传输（含跨境）安排
- [ ] 是否说明自动决策/画像
- [ ] 是否有儿童数据处理说明（如适用）
- [ ] 是否有政策更新机制和通知方式
- [ ] 是否有历史版本存档

**用户协议审查要点：**
- [ ] 条款是否公平（无霸王条款）
- [ ] 免责声明是否合法
- [ ] 争议解决条款（管辖法院/仲裁地）是否合法
- [ ] 是否明确服务内容、费用、期限

**PRD审查要点：**
- [ ] 新功能是否涉及个人数据处理
- [ ] 是否设计了用户同意机制
- [ ] 是否设计了数据主体权利实现方式
- [ ] 是否评估了数据跨境需求
- [ ] 是否包含安全设计（security by design）

#### Step 3：生成文档审查发现清单
对每个发现的问题，记录：
- 文档位置（章节/页码）
- 问题描述
- 违反的法规条款
- 风险等级
- 修改建议（含建议措辞）

## 三、风险评级标准

| 风险等级 | 定义 | 示例 |
|----------|------|------|
| 🔴 高风险 | 直接违反法律强制性规定，可能导致行政处罚、诉讼或数据泄露 | 明文存储密码；无隐私政策；向境外传输重要数据未经安全评估 |
| 🟡 中风险 | 违反最佳实践或程序性要求，可能导致监管问询或 reputational damage | 隐私政策不完整；无数据保留政策；无DPIA |
| 🟢 低风险 | 合规改进建议，不违反强制性规定 | 安全头配置不完整；日志保留期限未明确 |

## 四、常见合规问题速查表

### 高频问题（Code Review）

| 问题 | 相关法规 | 检查方式 |
|------|----------|----------|
| 明文密码存储 | 所有 | 搜索password相关代码，检查加密方式 |
| 缺少HTTPS强制 | 所有 | 检查是否有HTTP重定向到HTTPS |
| 隐私政策缺失/不完整 | PIPL, GDPR | 检查是否有隐私政策链接和完整内容 |
| Cookie同意缺失 | GDPR, PIPL | 检查是否有Cookie Banner和Consent记录 |
| 数据主体权利接口缺失 | GDPR, PIPL, CCPA | 检查是否有DELETE/EXPORT/ACCESS API |
| 日志泄露敏感信息 | 所有 | 检查log语句是否包含password/token/身份证 |
| 第三方SDK未披露 | PIPL, GDPR | 检查隐私政策是否列出所有第三方SDK |
| 数据跨境无合规机制 | PIPL, GDPR, DSL | 检查是否有数据出境评估/合同/SCC |

### 高频问题（Document Review）

| 问题 | 相关法规 | 检查方式 |
|------|----------|----------|
| 隐私政策未包含数据处理目的 | PIPL, GDPR | 阅读隐私政策第一章 |
| 无数据主体权利行使方式说明 | PIPL, GDPR, CCPA | 检查是否有专门章节说明权利行使 |
| 用户协议含不公平条款 | 消费者保护法 | 检查免责/管辖条款 |
| 无儿童数据处理说明 | PIPL, GDPR, COPPA | 检查是否处理14岁以下数据 |
| 无数据保留期限说明 | PIPL, GDPR | 检查是否说明保留期限 |
| 无境外接收方说明 | PIPL, GDPR | 检查是否说明跨境传输 |
